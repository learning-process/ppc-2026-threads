## 1. Контекст
Предыдущие реализации показали:

- **STL**: 2.75× (ручное управление потоками)

Однако все они ограничены одним узлом. Гибридная версия (MPI + OpenMP) позволяет задействовать **несколько узлов**
(распределённая память) и **все ядра внутри каждого узла** (разделяемая память). Это необходимо для масштабирования
на очень большие матрицы (например, `n ≥ 2000`), которые не помещаются в память одного узла или требуют сотен ядер.

## 2. Постановка задачи
Полная постановка задачи приведена в `seq/report.md`.  
**Краткое напоминание:** умножить две квадратные матрицы `C = A × B` размера `n × n`, элементы `double`.  
**Baseline:** последовательная версия (O(n³)) служит эталоном корректности и производительности.

**Особенность ALL-версии:** распределение матриц по нескольким MPI-процессам с последующей параллельной 
обработкой блоков внутри каждого процесса через OpenMP.

## 3. Базовый алгоритм
**Алгоритм Фокса (гибридный):**
- MPI-процессы организуются в виртуальную топологию `√P × √P`, где `P` — число процессов.
- Матрицы `A`, `B`, `C` разбиваются на блоки размера `bs × bs`, количество блоков `q = n / bs`.
- Каждый MPI-процесс получает один блок `A[i][k]` и один блок `B[k][j]` (алгоритм Фокса требует, чтобы `P = q²`).
- Внутри каждого MPI-процесса блоки перемножаются с помощью OpenMP-параллелизации.
- По ходу алгоритма блоки `B` циклически пересылаются между процессами в строке (коммуникация MPI).
- Результат собирается на процессе 0 и собирается в итоговую матрицу.

## 4. Межпроцессная схема
**Роли rank-ов:**
- **Rank 0**: корневой процесс, выполняет разбиение исходных матриц на блоки (`DecomposeToBlocks`),
   рассылку блоков (`MPI_Scatter`), сборку результата (`MPI_Gather`) и финальную сборку матрицы (`AssembleFromBlocks`).
- **Остальные rank-ы**: получают свои блоки, участвуют в стадиях Фокса, возвращают результат.

**Распределение данных:**
- Матрицы разбиваются на `q × q` блоков, где `q = √P`.
- Каждый процесс получает ровно один блок `A` и один блок `B` (размером `bs × bs`).
- Блоки распределяются в соответствии с топологией: процесс с рангом `rank = row * q + col`
   получает блок `A[row][col]` и `B[row][col]` (но в процессе алгоритма блоки `B` пересылаются).

**Используемые MPI-вызовы и их назначение:**
- `MPI_Comm_rank` / `MPI_Comm_size`: идентификация процесса и общего числа процессов.
- `MPI_Scatter`: рассылка блоков от процесса 0 всем процессам (разные блоки разным процессам).
- `MPI_Comm_split`: создание коммуникатора для каждой строки топологии (все процессы с 
  одинаковым `row` попадают в один коммуникатор для обменов `B`).
- `MPI_Bcast`: внутри строки рассылка текущего блока `A` от процесса-источника (`root = (row + step) % q`).
- `MPI_Sendrecv_replace`: циклический сдвиг блоков `B` вдоль строки (каждый процесс отправляет свой
   `B` следующему и получает от предыдущего).
- `MPI_Gather`: сборка блоков результата от всех процессов на процессе 0.
- `MPI_Bcast`: финальная рассылка результата от процесса 0 всем процессам (для синхронизации).

**Fallback-режим:** если `P` не является полным квадратом или `n` не делится на `q`, используется 
`ExecuteFallback` (только ранг 0 выполняет `SimpleMultiply`, результат рассылается через `MPI_Bcast`).

## 5. Внутрипроцессная схема
**Какая thread-технология используется:** OpenMP.

**Как делится работа между потоками:**
- В `SimpleMultiply`: параллелизация внешних циклов `i` и `j` через `#pragma omp parallel for collapse(2)`.
- В `DecomposeToBlocks` / `AssembleFromBlocks`: параллелизация циклов по блокам `bi`, `bj` через `collapse(2)`.
- В `LocalMatrixMultiply` (ядро умножения блоков): параллелизация циклов по `i` и `j` внутри блока через `collapse(2)`.

**Как объединяются локальные результаты:**
- Внутри каждого MPI-процесса потоки OpenMP пишут в общий `local_c` (свои уникальные ячейки, гонок нет).
- После завершения параллельной области результат уже находится в `local_c` (не требует дополнительного объединения).
- Затем `local_c` отправляется на процесс 0 через `MPI_Gather`.

## 6. Детали реализации
**Файлы:**
- `all/include/ops_all.hpp` — объявление класса `SinevAMultMatrixFoxAlgorithmALL`.
- `all/src/ops_all.cpp` — реализация.

**Где стоят MPI-вызовы (с обоснованием):**
- `MPI_Scatter` (в `ScatterBlocks`): корневой процесс рассылает блоки всем. Без этого
   данные пришлось бы копировать вручную.
- `MPI_Comm_split` (в `RunImpl`): создаёт коммуникатор для каждой строки топологии. Это необходимо
   для эффективных обменов `B` внутри строки без вовлечения процессов из других строк.
- `MPI_Bcast` (в `RunFoxStages`): рассылка текущего блока `A` от процесса-источника всем процессам строки.
   Это стандартная операция алгоритма Фокса.
- `MPI_Sendrecv_replace` (в `RunFoxStages`): циклический сдвиг `B`. Используется `replace`-версия
   для экономии памяти (один буфер для приёма и отправки).
- `MPI_Gather` (в `GatherResult`): сборка блоков результата. Необходима, так как только процесс
  0 знает итоговую структуру матрицы.
- `MPI_Bcast` (в `GatherResult` и `ExecuteFallback`): рассылка результата всем процессам для синхронизации.

**Где находятся потенциальные узкие места:**
- **Балансировка нагрузки:** алгоритм Фокса требует строгого равенства `P = q²`. Если число процессов не
   является полным квадратом, включается fallback-режим (только rank 0 работает → крайне неэффективно).
- **Коммуникации:** `MPI_Bcast` и `MPI_Sendrecv_replace` на каждом из `q` шагов создают задержки. Для больших
   `q` (например, `q = 20` при `P = 400`) это 20 × (2 коммуникации на шаг) = 40 обменов.
- **Размер блока:** `bs = n / q`. Если `bs` слишком мал, overhead OpenMP и MPI доминирует. Если слишком велик,
   страдает локальность данных.
- **OpenMP-параллелизация** внутри `LocalMatrixMultiply` хорошо масштабируется, но требует, чтобы `bs` было
   достаточно большим (рекомендуется `bs ≥ 32`).

**Комментарий о цене коммуникации и синхронизации:**
- **Ускорение 5.5× на 4 процессах** превосходит OpenMP на одном узле (7.3× хуже, но 5.5× на 4 узлах — это
   уже распределённая память!). Однако в данном случае все 4 процесса на одной машине, поэтому коммуникации локальны.
- **Эффективность > 1 (1.4)** для `P = 4` указывает на суперлинейное ускорение, что может быть связано с:
  - Лучшим использованием кэша: каждый процесс работает с блоком 200×200 (40K элементов) вместо 400×400 (160K),
   что уменьшает количество кэш-промахов.
  - Уменьшением накладных расходов на OpenMP (блок 200×200 даёт 40K операций против 160K на полной матрице).
- **Падение ускорения для `P = 8` и `P = 9`** связано с fallback-режимом (работает только rank 0, 
  остальные простаивают).
- **Коммуникационные затраты:** для `P = 4`, `q = 2`, всего 2 стадии Фокса, 2 × `MPI_Bcast` и 
  2 × `MPI_Sendrecv_replace` — overhead мал.
- **Синхронизация:** `MPI_Bcast` и `MPI_Sendrecv_replace` блокируют процессы до завершения обмена,
   но при малом `q` это незаметно.

## 10. Выводы
**Когда гибридная схема оправдана:**
- **Размер задачи велик** (`n ≥ 1000`), не помещается в память одного узла.
- **Доступно несколько узлов** в кластере (распределённая память) с быстрой сетью (InfiniBand).
- **Число процессов является полным квадратом** и `n` делится на `√P`.
- **Размер блока `bs = n / √P` достаточно велик** (`bs ≥ 32`) для эффективной OpenMP-параллелизации.

**Когда гибридная схема НЕ оправдана:**
- **Малое число процессов** (`P = 2, 3, 5, 6, 7, 8, ...`), когда нельзя сформировать квадратную решётку.
- **Матрица не делится нацело** на `√P` (fallback-режим убивает производительность).
- **Размер блока слишком мал** (`bs < 16`), overhead OpenMP превышает выгоду.
- **Один узел с большим числом ядер** — достаточно чистой OpenMP-версии (проще и эффективнее).
- **Медленная сеть** — коммуникации станут узким местом.

**Итог по данной реализации:**
- ALL-версия **корректна** (все тесты пройдены).
- При оптимальных условиях (`P = 4`, `n = 400`, `bs = 200`) достигнуто ускорение **5.5×**, что лучше, 
  чем OpenMP на одном узле (хотя OpenMP показал 7.3× при 4–8 потоках, но ALL использует 4 отдельных процесса, 
что потенциально масштабируется на несколько узлов).
- **Главный недостаток:** жёсткое требование `P = q²` и `n % q = 0`. Для произвольных `P` и `n` производительность 
- падает до уровня SEQ из-за fallback-режима.
- **Главное преимущество:** возможность масштабирования на кластер. Для `n = 400` выигрыш неочевиден (один 
  узел справляется), но для `n = 4000` и `P = 16` гибридная версия будет незаменима.

**Финальный вердикт:** Гибридная версия — мощный инструмент для **крупномасштабных** задач, но на малых размерах 
(400×400) её преимущества не видны из-за коммуникационных overhead и жёстких требований к конфигурации.
 Для данной работы ALL-реализация служит демонстрацией подхода, а не рекордсменом по ускорению.
