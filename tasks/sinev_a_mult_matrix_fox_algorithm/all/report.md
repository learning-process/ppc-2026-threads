## 1. Контекст
Последовательная версия (`seq`) реализует классическое умножение матриц за O(n³). OpenMP и TBB
уже показали ускорение 5.8–7.3× и 1.6–1.7× соответственно. В данной реализации используется ручное
управление потоками через `std::thread` с динамическим распределением блоков через `std::atomic`. STL-версия
выбрана для демонстрации низкоуровневого контроля над созданием, синхронизацией и объединением потоков,
а также для сравнения с высокоуровневыми подходами (OpenMP, TBB).

## 2. Постановка задачи
Полная постановка задачи приведена в `seq/report.md`.  
**Краткое напоминание:** умножить две квадратные матрицы `C = A × B` размера `n × n`, элементы `double`.  
**Baseline:** последовательная версия (O(n³)) служит эталоном корректности и производительности.

## 3. Базовый алгоритм
Последовательная версия (`SimpleMultiply`) использует тройной вложенный цикл с оптимизацией порядка
обхода (i-k-j для лучшего использования кэша).  
**Алгоритм Фокса (STL-версия):**
- Разбить матрицы `A`, `B`, `C` на блоки размера `bs × bs`, количество блоков по каждому измерению `q = n / bs`.
- Выполнить `q` шагов (стадий) алгоритма Фокса:
  - На шаге `step` для каждого блока `(i,j)`:
    - Определить блок `A[i][k]`, где `k = (i + step) % q`.
    - Умножить блок `A[i][k]` на блок `B[k][j]` и накопить в `C[i][j]`.
- Собрать результат из блоков обратно в плоскую матрицу.

## 4. Схема распараллеливания
**Как данные делятся на диапазоны:**  
Вместо статического разбиения используется **динамическое распределение задач** через `std::atomic<size_t>`:
- Общее количество независимых единиц работы (блоков или ячеек) вычисляется как `total_blocks = q × q`.
- Атомарный счётчик `next_block` инициализируется нулём.
- Каждый поток в цикле захватывает следующий индекс через `fetch_add(1)` и обрабатывает соответствующий блок.

**Что делает каждый поток:**
- В `DecomposeToBlocks` и `AssembleFromBlocks`: преобразует блок из плоского представления в блочное и обратно.
- В `FoxStep`: для пары `(i,j)` вычисляет `k = (i + step) % q`, затем вызывает `MultiplyBlocks` для умножения блоков
  `A[i][k]` и `B[k][j]` с накоплением в `C[i][j]`.

**Где хранятся локальные результаты:**  
Результаты накапливаются непосредственно в `blocks_c` (общий вектор). Гонок нет, так как каждый поток обрабатывает
уникальные блоки `(i,j)` благодаря атомарному счётчику.

**Как и когда выполняется join:**  
После создания всех потоков (цикл `emplace_back`) выполняется отдельный цикл `thread.join()` для каждого потока.
Это гарантирует, что все потоки завершат работу до возврата из функции. Join **не** выполняется внутри
цикла создания (в отличие от антипримера из `example_threads`).

**Какая синхронизация используется и почему:**
- `std::atomic<size_t> next_block` — атомарный счётчик для динамического распределения блоков. Используется
  `fetch_add(1)` с памятью `memory_order_seq_cst` (по умолчанию), что обеспечивает строгую согласованность.
- **Нет** `mutex` или `condition_variable`, так как гонки отсутствуют (каждый блок принадлежит одному потоку).
- **Нет** `atomic` для доступа к `blocks_c`, так как каждый поток пишет в свой уникальный блок.

## 5. Детали реализации
**Файлы:**
- `stl/include/ops_stl.hpp` — объявление класса `SinevAMultMatrixFoxAlgorithmSTL`.
- `stl/src/ops_stl.cpp` — реализация.

**Карта поток → диапазон (динамическая):**
```cpp
std::atomic<size_t> next_block(0);
size_t total_blocks = q * q;

// В каждом потоке:
size_t block_idx = 0;
while ((block_idx = next_block.fetch_add(1)) < total_blocks) {
  int bi = block_idx / q;
  int bj = block_idx % q;
  // обработка блока (bi, bj)
}
```

Преимущества динамического распределения:
- Естественная балансировка нагрузки при нерегулярной вычислительной сложности.
- Автоматическая адаптация к разной скорости потоков (из-за троттлинга, прерываний ОС).

**Схема объединения локальных результатов:**
Результаты накапливаются напрямую в общий blocks_c без отдельного этапа объединения, так как блоки не пересекаются.

**Иллюстративный листинг:**

```cpp
// File: stl/src/ops_stl.cpp
void SinevAMultMatrixFoxAlgorithmSTL::FoxStep(
    const std::vector<double> &blocks_a,
    const std::vector<double> &blocks_b,
    std::vector<double> &blocks_c,
    size_t bs, int q, int step) {
  const size_t block_size = bs * bs;
  unsigned int num_threads = std::thread::hardware_concurrency();
  if (num_threads == 0) num_threads = 2;

  std::vector<std::thread> threads;
  threads.reserve(num_threads);
  std::atomic<size_t> next_cell(0);
  size_t total_cells = static_cast<size_t>(q) * static_cast<size_t>(q);

  for (unsigned int t = 0; t < num_threads; ++t) {
    threads.emplace_back([&]() {
      size_t cell_idx = 0;
      while ((cell_idx = next_cell.fetch_add(1)) < total_cells) {
        int i = cell_idx / q;
        int j = cell_idx % q;
        const int k = (i + step) % q;
        const size_t a_off = ((i * q) + k) * block_size;
        const size_t b_off = ((k * q) + j) * block_size;
        const size_t c_off = ((i * q) + j) * block_size;
        MultiplyBlocks(blocks_a, blocks_b, blocks_c, bs, a_off, b_off, c_off);
      }
    });
  }

  for (auto &thread : threads) {
    thread.join();
  }
}
```
*Пояснение*: Пул потоков создаётся один раз на каждый вызов `FoxStep` (т.е. `q` раз). Атомарный счётчик
`next_cell` распределяет ячейки `(i,j)` между потоками динамически. Это обеспечивает отличную балансировку,
но создаёт `overhead` на атомарные операции и многократное создание потоков.

**Выбор размера блока:**
В отличие от OpenMP и TBB, STL-версия использует фиксированный размер блока `bs = 64 `(с уменьшением
до 32, 16 при необходимости). Причина:
- `bs = 64` даёт блок `64×64 = 4096 элементов`, что хорошо подходит для кэш-линий и уменьшает 
  количество блоков `(q = n/64)`.
- Меньшее количество блоков снижает `overhead` на атомарные операции и управление потоками.

  - Динамическое распределение даёт идеальную балансировку.

## 9. Выводы
**Что показала ручная реализация потоков:**
- **Плюсы:** полный контроль над распределением задач, независимость от runtime-библиотек, ускорение 2.7× (лучше TBB).
- **Минусы:** высокий overhead на создание потоков, сложность кода, риск гонок, атомарные операции.
- **Почему STL оказался быстрее TBB, но медленнее OpenMP:** OpenMP имеет минимальный overhead благодаря
  компиляторной оптимизации и статическому разбиению; STL с динамическим распределением оказался эффективнее
task-модели TBB для данной задачи, но многократное создание потоков — главный недостаток.

**Итог:** Ручная реализация на `std::thread` с динамическим распределением задач показала ускорение 2.7×,
что является приемлемым результатом, но значительно уступает OpenMP (7.3×). Для регулярных задач со статической 
нагрузкой OpenMP остаётся лучшим выбором.
