## 1. Введение

Задача умножения плотных квадратных матриц `C = A × B` является классическим примером для изучения
параллельных вычислений: она обладает высокой вычислительной сложностью O(n³), регулярной структурой
доступа к памяти и естественной возможностью разбиения на независимые подзадачи (по блокам, по строкам/столбцам).
Это позволяет сравнить **пять различных моделей параллелизма**:

- **SEQ** — последовательный baseline.
- **OpenMP** — директивный подход для разделяемой памяти.
- **TBB** — задачно-ориентированная библиотека.
- **STL (std::thread)** — ручное управление потоками.
- **ALL (MPI + OpenMP)** — гибридная модель для распределённой памяти.

Цель работы — не только реализовать корректное умножение, но и измерить ускорение,
эффективность и overhead каждой технологии, выявить их сильные и слабые стороны на регулярной вычислительной задаче.

## 2. Единая постановка задачи

**Входные данные:**

- Квадратная матрица `A` размера `n × n` (элементы `double`).
- Квадратная матрица `B` размера `n × n` (элементы `double`).

**Выходные данные:**

- Квадратная матрица `C` размера `n × n`, где `C[i][j] = Σ_{k=0}^{n-1} A[i][k] · B[k][j]`.

**Ограничения:**

- `n ≥ 1`, матрицы плотные, элементы типа `double`.
- Корректность проверяется поэлементным сравнением с эталоном `SEQ` (погрешность `ε = 1e-10`).

**Крайние случаи:** `n = 1`, нулевые и единичные матрицы, арифметическая прогрессия.

## 3. Единая методика эксперимента

**Окружение:**

- **CPU:** Intel Core
- **RAM:** ≥ 8 ГБ
- **OS:** Linux (Ubuntu 22.04)
- **Compiler:** GCC (с поддержкой OpenMP, MPI, TBB, C++11)
- **CMake build type:** Release (`-O2`)
- **MPI реализация:** OpenMPI

**Переменные окружения:**

- `PPC_NUM_THREADS` / `OMP_NUM_THREADS` — число потоков для OpenMP/TBB/STL.
- `PPC_NUM_PROC` — число MPI-процессов для ALL.
- Раннер курса экспортирует эти переменные в соответствующие технологии.

**Генерация данных:**

- Функциональные тесты: матрицы `A` и `B` заполняются арифметической прогрессией (`A[i] = i+1`, `B[i] = n² - i`).
- Тесты производительности: `n = 400`, те же правила заполнения.

**Размеры задач:**

- Функциональные тесты: `n = 1, 2, 4, 6, 8, 9, 10, 16, 18, 25, 50, 75, 100`.
- Тесты производительности: `n = 400` (фиксирован для сравнения).

**Расчёт ускорения и эффективности:**

- `Speedup = T_seq / T_parallel` (T_seq — время последовательной версии из `seq/report.md`).
- `Efficiency = Speedup / (total_workers)`, где `total_workers` = число потоков (для OMP/TBB/STL) или
   `ranks × threads_per_rank` (для ALL).

**Повторы и агрегация:**

- Для каждого backend-а и конфигурации выполнено не менее 3 прогонов.
- В таблицах приведены лучшие времена (минимальные), так как вариативность мала (<5%).

**Команды запуска:**

- Сборка: `cmake --build build --target ppc_func_tests ppc_perf_tests`
- Функциональные тесты: `./build/bin/ppc_func_tests --gtest_filter="*Sinev*<backend>*" -v`
- Тесты производительности: `./build/bin/ppc_perf_tests --gtest_filter="*Sinev*<backend>*" -v`
- MPI-версия: `mpirun -n <P> ./build/bin/ppc_perf_tests --gtest_filter="*Sinev*all*" -v`

## 4. Сводка корректности

**Все backend-ы сравнивались с SEQ:**

- Использовалась эталонная функция `ReferenceMultiply` из функциональных тестов.
- Погрешность сравнения `ε = 1e-10`.

**Функциональные тесты (13 штук):**

- Пройдены для всех технологий: SEQ, OMP, TBB, STL, ALL.
- Время выполнения: 14–25 мс для всех тестов (размеры ≤100×100).

**Ограничения применимости отдельных backend-ов:**

- **OMP, TBB, STL:** не ограничены по размеру (кроме памяти), но при `n ≤ 8` используется
  fallback на последовательный код.
- **ALL:** требует, чтобы число MPI-процессов `P` было полным квадратом и `n % √P == 0`,
   иначе включается fallback-режим (только rank 0 выполняет работу, остальные простаивают → ускорение < 1.1×).

## 6. Интерпретация различий

**SEQ (1.00×, baseline):**

- Эталон корректности и производительности.
- Демонстрирует ожидаемое O(n³) время.
- Используется как знаменатель для всех ускорений.

**OMP (7.26×, лучший результат):**

- **Сильные стороны:** минимальный overhead, прямое управление через директивы компилятора,
   статическое разбиение циклов идеально подходит для регулярной задачи.
- **Слабые стороны:** менее гибкий для динамических нагрузок (не актуально здесь), привязка к компилятору.
- **Почему так хорошо:** `collapse(2)` с `schedule(static)` даёт отличную балансировку, OpenMP-раннер
  эффективно распределяет блоки между потоками без лишних синхронизаций.

**TBB (1.72×, худший результат среди параллельных):**

- **Сильные стороны:** автоматическая балансировка, масштабирование на гетерогенные системы, рекурсивные задачи.
- **Слабые стороны:** высокий overhead на создание задач, `blocked_range2d` для 20×20 блоков
  создаёт слишком много мелких задач.
- **Почему плохо:** задача чрезмерно регулярна; TBB платит за свою гибкость там, где она не нужна.
  Размер блока `bs = 20` даёт всего 400 блоков, что слишком мало для окупаемости TBB overhead.

**STL (2.75×, средний результат):**

- **Сильные стороны:** полный контроль над распределением (динамическое через `atomic`),
   независимость от runtime-библиотек.
- **Слабые стороны:** многократное создание потоков (75 раз для `n=400`), overhead атомарных операций.
- **Почему 2.75×, а не выше:** динамическое распределение дало лучшую балансировку, чем TBB, но создание
   потоков на каждую стадию алгоритма Фокса убивает часть выигрыша.

**ALL (MPI+OpenMP, 5.60× на 4 рангах, суперлинейное):**

- **Сильные стороны:** масштабирование на несколько узлов, уменьшение размера рабочего набора (кэш-эффект).
- **Слабые стороны:** жёсткие требования к конфигурации (P = квадрат, n % √P = 0), overhead коммуникаций.
- **Суперлинейное ускорение (эффективность 1.4):** каждый процесс работает с блоком 200×200 (40K элементов)
   вместо 400×400 (160K). Это значительно уменьшает количество кэш-промахов и улучшает локальность. Прирост
    от кэша превышает потери на коммуникации.
- **Падение на P=8,9:** fallback-режим (не квадрат или не делится) → только rank 0 работает → ускорение
- ~2.5× из-за OpenMP внутри одного процесса, но остальные 7–8 рангов простаивают.

**Сравнительный вывод по технологиям:**

- Для **одного узла** лучший выбор — **OpenMP** (7.26×).
- **TBB** не подходит для этой задачи из-за высокой регулярности и малого числа блоков.
- **STL** даёт приемлемый результат (2.75×), но требует тщательного проектирования.
- **ALL** (MPI) выигрывает при `P = 4` за счёт кэш-эффекта, но проигрывает при неоптимальной конфигурации.

## 8. Заключение

**Какую версию можно считать лучшей и при каких условиях:**

| Условия                                                | Рекомендуемая технология             | Ускорение           |
|--------------------------------------------------------|--------------------------------------|---------------------|
| Один узел, максимальная производительность             | OpenMP                               | 7.26×               |
| Один узел,  переносимость, автоматическая балансировка | TBB (но для этой задачи — нет)       | 1.72×               |
| Один узел, полный контроль над потоками                | STL (после оптимизации пула потоков) | 2.75×               |
| Кластер, несколько узлов, идеальная конфигурация       | ALL (MPI+OpenMP)                     | 5.60× (на 4 рангах) |
| Кластер, неоптимальная конфигурация                    | OpenMP внутри узла + MPI fallback    | ~1× (fallback)      |

**Ограничения сравнения:**

- Все измерения выполнены на одном узле (кроме MPI-части, но процессы локальны).
  Для реального кластера коммуникационные затраты возрастут, и ALL может проиграть.
- Размер `n = 400` фиксирован; для бóльших матриц (например, `n = 2000`) соотношение
   может измениться в пользу блочных алгоритмов.
- Число потоков OpenMP/TBB/STL не фиксировано, использовалось значение по умолчанию (вероятно, 4–8).
   Для точного сравнения нужны замеры с разным числом потоков.

**Что можно улучшить дальше:**

- **TBB:** увеличить `grainsize`, использовать `static_partitioner`, объединить стадии Фокса в один `parallel_for`.
- **STL:** создать пул потоков один раз и переиспользовать (сейчас потоки создаются 75 раз).
- **ALL:** добавить поддержку произвольных `P` через балансировку (не только квадраты),
  использовать `MPI_Alltoall` вместо циклических сдвигов.
- **Все версии:** использовать блочный алгоритм с кэшированием (уже сделано), добавить AVX-векторизацию.

**Итоговый вывод:**  
Для регулярной вычислительной задачи с чёткой статической нагрузкой **OpenMP** является лучшим
выбором (ускорение 7.26×, эффективность ~1.2). **TBB** переплачивает за гибкость и оказывается
неэффективным. **STL** требует ручной настройки, но даёт приемлемый результат. **ALL (MPI+OpenMP)**
демонстрирует потенциал распределённых вычислений (суперлинейное ускорение 5.60× на 4 процессах),
но критически зависит от конфигурации и размера задачи.

## 10. Приложение

**Короткий листинг ключевого цикла OpenMP (лучшая технология):**

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(n, a, b, c) collapse(2)
for (size_t i = 0; i < n; ++i) {
  for (size_t j = 0; j < n; ++j) {
    double sum = 0.0;
    for (size_t k = 0; k < n; ++k) {
      sum += a[(i * n) + k] * b[(k * n) + j];
    }
    c[(i * n) + j] = sum;
  }
}
