## 1. Контекст

Последовательная версия (`seq`) реализует классическое умножение матриц за O(n³).
В данной реализации используется OpenMP для параллельного выполнения блочного алгоритма Фокса,
который разбивает матрицы на блоки и выполняет умножение блоков с каскадным обновлением.
OpenMP выбран как простой способ параллелизации циклов по блокам с минимальными изменениями кода.

## 2. Постановка задачи

Полная постановка задачи приведена в `seq/report.md`.  
**Краткое напоминание:** умножить две квадратные матрицы `C = A × B` размера `n × n`, элементы `double`.  
**Baseline:** последовательная версия (O(n³)) служит эталоном корректности и производительности.

## 3. Базовый алгоритм

Последовательная версия (`SimpleMultiply`) использует тройной вложенный цикл «строка-столбец».  
**Алгоритм Фокса (параллельная версия):**

- Разбить матрицы `A`, `B`, `C` на блоки размера `bs × bs`, количество блоков по каждому измерению `q = n / bs`.
- Выполнить `q` шагов (стадий) алгоритма Фокса:
  - На шаге `step` для каждого блока `(i,j)`:
    - Определить блок `A[i][k]`, где `k = (i + step) % q`.
    - Умножить блок `A[i][k]` на блок `B[k][j]` и накопить в `C[i][j]` (умножение блоков — вложенными
   циклами `ii`, `kk`, `jj`).
- Собрать результат из блоков обратно в плоскую матрицу.

## 4. Схема распараллеливания

**Какая область кода параллелится:**  
Параллелизуются четыре независимых цикла `for` с помощью `#pragma omp parallel for`:

1. `SimpleMultiply` — для маленьких матриц (`n ≤ 8`) тройной цикл свёрнут в двумерный `collapse(2)`.
2. `DecomposeToBlocks` — разбиение матриц на блоки (двумерный цикл по `bi`, `bj`).
3. `AssembleFromBlocks` — сборка результата из блоков.
4. `FoxStep` — основной вычислительный шаг алгоритма Фокса.

**Переменные:**

- `shared(n, a, b, c, blocks_a, blocks_b, blocks_c, bs, q, step, block_size_bytes)` — все данные доступны всем
потокам для чтения; запись идёт в независимые элементы `c` или `blocks_c` (каждый поток пишет в свой блок).
- `default(none)` — требует явного указания атрибутов всех переменных, предотвращает случайные `shared`.

**Где нужен reduction / atomic / critical:**  
Не требуются. В `FoxStep` каждый поток обновляет свою часть `blocks_c` (блок `(i,j)` уникален для пары `(i,j)`,
гонок нет). В `SimpleMultiply` каждый поток пишет в свою уникальную ячейку `c[(i*n)+j]`. В `DecomposeToBlocks`
и `AssembleFromBlocks` потоки пишут в непересекающиеся блоки.

**Какой schedule выбран и почему:**  
Используется `schedule(static)` по умолчанию. Причина:

- Итерации циклов (по блокам) имеют одинаковую вычислительную нагрузку (размер блока `bs` фиксирован).
- `static` минимизирует overhead планирования.
- Для `collapse(2)` `static` даёт равномерное распределение итераций по потокам.

**Неявные/явные барьеры:**  

- `#pragma omp parallel for` имеет неявный барьер в конце цикла, что безопасно, так как все итерации независимы,
и результат не требуется промежуточный.
- `#pragma omp parallel` в `FoxStep` создаёт команду потоков один раз (внутри `parallel for`),
барьер в конце гарантирует, что все блоки обработаны перед следующим шагом.

- В `RunImpl` добавлена логика выбора: для маленьких матриц (`n ≤ 8`) вызывается `SimpleMultiply`
(классический алгоритм, распараллеленный через `collapse(2)`), для больших — блочный алгоритм Фокса.
- Размер блока `bs` выбирается как наибольший делитель `n`, не превышающий `√n`. Это обеспечивает хорошую
балансировку и минимизирует overhead.
  
**Иллюстративный листинг (ключевой фрагмент — параллельный шаг Фокса):**

```cpp
// File: omp/src/ops_omp.cpp
// Основной вычислительный шаг алгоритма Фокса
void SinevAMultMatrixFoxAlgorithmOMP::FoxStep(
    const std::vector<double> &blocks_a,
    const std::vector<double> &blocks_b,
    std::vector<double> &blocks_c,
    size_t bs, int q, int step) {
  const size_t block_size_bytes = bs * bs;
#pragma omp parallel for default(none) shared(blocks_a, blocks_b, blocks_c, bs, q, step, block_size_bytes) collapse(2)
  for (int i = 0; i < q; ++i) {
    for (int j = 0; j < q; ++j) {
      const int k = (i + step) % q;
      const size_t a_off = ((i * q) + k) * block_size_bytes;
      const size_t b_off = ((k * q) + j) * block_size_bytes;
      const size_t c_off = ((i * q) + j) * block_size_bytes;

      for (size_t ii = 0; ii < bs; ++ii) {
        for (size_t kk = 0; kk < bs; ++kk) {
          const double val = blocks_a[a_off + (ii * bs) + kk];
          for (size_t jj = 0; jj < bs; ++jj) {
            blocks_c[c_off + (ii * bs) + jj] += val * blocks_b[b_off + (kk * bs) + jj];
          }
        }
      }
    }
  }
}
```

*Пояснение*: директива `#pragma omp parallel for collapse(2)` параллелизует двумерный цикл по блокам `(i,j)`.
Каждый поток получает уникальную пару `(i,j)` и обрабатывает только свой блок `C[i][j]`. Гонок нет, так как индексы
c_off для разных `(i,j)` различны. default(none) заставляет явно перечислить все используемые переменные, что
предотвращает случайные `shared`. `collapse(2)` улучшает балансировку, объединяя 2 уровня циклов
в одно пространство итераций.

**Устранённые риски гонок:**

- В `FoxStep` каждый поток пишет в свой уникальный блок `blocks_c`. Проверено: разные `(i,j)` дают разные `c_off`.
- В `SimpleMultiply` каждый поток пишет в свою уникальную ячейку `c[(i*n)+j]`.
- В `DecomposeToBlocks` и `AssembleFromBlocks` — аналогично.

**Комментарий о масштабируемости и узких местах:**

- Ускорение ~6–7× — хороший результат, указывающий на эффективное использование многоядерности.
- `task_run` (7.26×) немного быстрее `pipeline` (5.80×), что объясняется разными накладными расходами внутри runner.
- **Узкие места:**
  - Выбор размера блока `bs` ≈ √n даёт `q` блоков. Для `n = 400` блоки 20×20 — хороший компромисс.
  - `collapse(2)` улучшает балансировку, но добавляет overhead.
  - Накладные расходы на разбиение и сборку блоков (`DecomposeToBlocks`, `AssembleFromBlocks`) могут быть
 заметны для маленьких `n`, но для `n = 400` они малы по сравнению с вычислениями.
- Масштабируемость ограничена размером задачи: для `n = 400` число блоков `q×q = 400`, что больше, чем число потоков,
поэтому балансировка хорошая. При увеличении числа потоков ускорение будет расти до тех пор,
пока потоков не станет больше, чем блоков.

## 9. Выводы

**OpenMP дал выигрыш** благодаря параллелизации независимых блоков в алгоритме Фокса.
Ускорение ~6–7× свидетельствует о:

- Почти линейной масштабируемости для данного размера задачи.
- Удачном выборе `collapse(2)` и `schedule(static)` для равномерного распределения нагрузки.
- Отсутствии гонок (все тесты корректности пройдены).

**Выигрыш ограничен следующими факторами:**

- Для маленьких матриц (`n ≤ 8`) распараллеливание не даёт выигрыша (используется
`SimpleMultiply` с `collapse(2)`, overhead может превышать пользу).
- Алгоритм Фокса требует дополнительной памяти под блоки (∼3·q²·bs² = 3·n²),
  что для очень больших `n` может стать проблемой.
- Эффективность сильно зависит от выбора размера блока. Текущая эвристика (`bs` — наибольший делитель ≤ √n)
   подходит для `n = 400`, но для других `n` может потребоваться настройка.
- Накладные расходы на создание параллельных регионов и синхронизацию (неявный барьер в конце `parallel for`)
   могут быть заметны при малом объёме вычислений на итерацию.

**Итог:** OpenMP-реализация значительно быстрее последовательной (ускорение >5×) и полностью корректна.
Она будет использоваться как один из эталонов для сравнения с TBB, std::thread и гибридной версией в сводном отчёте.
