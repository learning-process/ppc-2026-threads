## 1. Контекст

OpenMP-версия показывает распараллеливание задачи в модели общей памяти. В отличие от SEQ, массив разбивается на
независимые чанки, которые могут сортироваться и сливаться несколькими потоками внутри одного процесса.

## 2. Постановка задачи

На вход поступает `std::vector<int>`, на выходе должен быть `std::vector<int>` с теми же элементами, отсортированными по
неубыванию. OpenMP-версия должна давать тот же результат, что и последовательный baseline.

## 3. Базовый алгоритм

Базовая идея остается общей для всех параллельных версий: массив делится на чанки, каждый чанк сортируется независимо,
затем отсортированные части объединяются деревом попарных слияний. Для сортировки чанков используется `std::sort`, для
объединения диапазонов — `std::merge`.

## 4. Схема распараллеливания

Параллелизм реализован двумя циклами OpenMP:

- сортировка отдельных чанков выполняется через `#pragma omp parallel for`;
- операции слияния на каждом уровне дерева также выполняются через `#pragma omp parallel for`.

Используется `schedule(static)`, так как границы чанков известны заранее, а операции одного уровня дерева независимы.
Общими данными являются входной/выходной буферы и массив границ чанков. Приватными остаются индексы итераций и локальные
границы диапазонов.

Редукции (`reduction`) в данной версии не используются, потому что алгоритм не вычисляет общий скалярный результат.
Каждый поток сортирует или сливает свой непересекающийся диапазон, а общий результат формируется записью в заранее
определенные участки выходного буфера.

## 5. Детали реализации

Файлы реализации:

- `omp/include/ops_omp.hpp`;
- `omp/src/ops_omp.cpp`.

Класс `SakharovAShellButcherOMP` возвращает тип задачи `ppc::task::TypeOfTask::kOMP`.

Минимальный листинг из `omp/src/ops_omp.cpp`:

```cpp
#pragma omp parallel for default(none) shared(data, bounds, chunk_count) schedule(static)
for (int chunk = 0; chunk < chunk_count; ++chunk) {
  auto begin = data.begin() + static_cast<std::ptrdiff_t>(bounds[chunk]);
  auto end = data.begin() + static_cast<std::ptrdiff_t>(bounds[chunk + 1]);
  std::sort(begin, end);
}
```

Фрагмент показывает основную точку OpenMP-параллелизма: каждый поток получает свой непересекающийся диапазон массива.
Так как разные чанки не пересекаются, дополнительная синхронизация при сортировке не требуется.

## 6. Проверка корректности

Версия проверяется общими функциональными тестами из `tests/functional/main.cpp`. Эталонный результат строится через
`std::ranges::sort`. Проверяются массивы разной четности, отрицательные числа, дубликаты, один элемент и пустой вход. По
результатам локального запуска все 5 функциональных тестов OpenMP прошли успешно.

## 8. Результаты

В таблице приведены средние результаты perf-тестов. Для `2` и `8` потоков среднее рассчитано по трем запускам, для `4`
потоков — по двум запускам.

| Режим | Workers | Время, с | Ускорение относительно SEQ | Эффективность |
| --- | ---: | ---: | ---: | ---: |
| pipeline | 2 | 0.0141551325 | 3.17 | 1.58 |
| task_run | 2 | 0.0087188449 | 5.17 | 2.59 |
| pipeline | 4 | 0.0106660785 | 4.20 | 1.05 |
| task_run | 4 | 0.0061448581 | 7.34 | 1.84 |
| pipeline | 8 | 0.0172132381 | 2.60 | 0.33 |
| task_run | 8 | 0.0062423649 | 7.23 | 0.90 |

Эффективность для конфигураций `2` и `4` потоков местами превышает `1`, потому что ускорение считается относительно SEQ
baseline, а OpenMP-версия отличается не только числом потоков, но и организацией вычислений. В SEQ используется
сортировка Шелла с четно-нечетным слиянием и финальной досортировкой, а в OpenMP-версии локальные чанки сортируются
через `std::sort` и затем объединяются деревом `std::merge`. Поэтому метрика отражает одновременно эффект параллелизма и
более удачную структуру вычислений на данном входе.

## 10. Выводы

OpenMP-версия хорошо ускоряет основной вычислительный этап `task_run`. На малом числе потоков ускорение частично связано
не только с параллелизмом, но и с более удачной организацией сортировки чанков относительно SEQ baseline. В режиме
`pipeline` результаты сильнее зависят от накладных расходов полного жизненного цикла задачи.
