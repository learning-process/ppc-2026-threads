## 1. Контекст

oneTBB-версия использует task-based подход: работа разбивается на независимые диапазоны, а их распределение между
потоками выполняет runtime Intel oneTBB. Эта версия сравнивается с OpenMP и STL как альтернативная модель параллелизма
общей памяти.

## 2. Постановка задачи

На вход поступает `std::vector<int>`, на выходе должен быть отсортированный `std::vector<int>` с тем же набором
элементов. Результат oneTBB-версии должен совпадать с результатом последовательной версии и с эталоном
`std::ranges::sort`.

## 3. Базовый алгоритм

Массив делится на чанки. Каждый чанк сортируется независимо с помощью `std::sort`, затем чанки объединяются деревом
попарных слияний с помощью `std::merge`. Размер чанков ограничивается минимальным порогом `1 << 14`, чтобы не создавать
слишком много мелких задач.

## 4. Схема распараллеливания

Параллельные участки реализованы через:

- `tbb::parallel_for`;
- `tbb::blocked_range<std::size_t>`.

На этапе сортировки диапазон номеров чанков передается в `parallel_for`. На этапе слияния аналогично распараллеливаются
операции текущего уровня дерева. Каждая операция слияния пишет в свой непересекающийся участок выходного буфера, поэтому
блокировки не нужны. Фактическая степень параллелизма ограничивается инфраструктурой проекта через `PPC_NUM_THREADS` и
`tbb::global_control` в runner-е.

Явный `grainsize` и отдельный `partitioner` не задаются: используется стандартное разбиение `tbb::blocked_range`. Размер
исходных чанков ограничивается константой `1 << 14`, поэтому в `parallel_for` не попадает слишком много мелких задач, а
дополнительная настройка partitioner для данной реализации не требуется.

## 5. Детали реализации

Файлы реализации:

- `tbb/include/ops_tbb.hpp`;
- `tbb/src/ops_tbb.cpp`.

Класс `SakharovAShellButcherTBB` возвращает тип задачи `ppc::task::TypeOfTask::kTBB`.

Минимальный листинг из `tbb/src/ops_tbb.cpp`:

```cpp
tbb::parallel_for(tbb::blocked_range<std::size_t>(0, merge_count),
                  [&](const tbb::blocked_range<std::size_t> &range) {
  for (std::size_t merge_index = range.begin(); merge_index != range.end(); ++merge_index) {
    MergeRange(source, destination, bounds, width, merge_index);
  }
});
```

Фрагмент показывает распараллеливание уровня дерева слияний. oneTBB сам делит диапазон задач на части и распределяет их
между рабочими потоками.

## 6. Проверка корректности

Корректность проверяется общими функциональными тестами. Для каждого входа результат oneTBB-версии сравнивается с
эталоном, полученным через `std::ranges::sort`. По результатам локального запуска все 5 функциональных тестов oneTBB
прошли успешно.

## 8. Результаты

В таблице приведены средние результаты perf-тестов. Для `2` и `8` потоков среднее рассчитано по трем запускам, для `4`
потоков — по двум запускам.

| Режим | Workers | Время, с | Ускорение относительно SEQ | Эффективность |
| --- | ---: | ---: | ---: | ---: |
| pipeline | 2 | 0.0118986448 | 3.77 | 1.88 |
| task_run | 2 | 0.0120335897 | 3.75 | 1.87 |
| pipeline | 4 | 0.0078377247 | 5.72 | 1.43 |
| task_run | 4 | 0.0078923226 | 5.72 | 1.43 |
| pipeline | 8 | 0.0066771825 | 6.71 | 0.84 |
| task_run | 8 | 0.0064432303 | 7.00 | 0.88 |

Эффективность для `2` и `4` потоков выше `1` является формальным результатом расчета `speedup / workers`. Это не
означает физическое сверхлинейное ускорение только за счет потоков. Причина в том, что oneTBB-версия сравнивается с SEQ
baseline, но использует другую схему вычислений: разбиение на чанки, `std::sort` и дерево слияний. Из-за этого ускорение
включает и выигрыш от параллельного выполнения, и выигрыш от более эффективной организации сортировки.

## 10. Выводы

oneTBB-версия показала лучшее среднее время в режиме `pipeline` и стабильный рост производительности при увеличении
числа потоков. Для данной задачи модель oneTBB хорошо подходит, так как сортировка чанков и слияния на одном уровне
дерева являются независимыми задачами.
