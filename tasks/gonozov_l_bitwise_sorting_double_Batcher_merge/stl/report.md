## 1. Контекст

STL-версия (std::thread) реализует параллельную сортировку с использованием нативных потоков C++. 
Основная идея заключается в разбиении массива на чанки, количество которых равно степени двойки 
и не превышает числа доступных потоков. Каждый чанк сортируется независимо с помощью поразрядной 
сортировки, после чего выполняется итеративное параллельное слияние с использованием сети Бэтчера. 
Такой подход позволяет эффективно использовать многоядерные архитектуры за счёт независимой обработки 
блоков данных, улучшенной кэш-локальности и минимальных накладных расходов на синхронизацию.

## 2. Постановка задачи

Постановка идентична `seq/report.md`. Вход — `std::vector<double>`, выход — отсортированный 
по возрастанию вектор. Требования корректности: результат STL-версии должен полностью совпадать 
с результатом последовательного эталона на любом наборе входных данных.

## 3. Базовый алгоритм

Последовательный алгоритм описан в `seq/report.md`. Ключевые этапы:

1. Преобразование `double` → `uint64_t` с сохранением порядка.
2. Поразрядная сортировка (8 проходов, radix = 256).
3. Слияние отсортированных половин сетью Бэтчера.
4. Обратное преобразование `uint64_t` → `double`.

## 4. Схема распараллеливания

### 4.1. Декомпозиция задачи

STL-версия использует двухуровневую стратегию распараллеливания:

- **Фаза 1 (сортировка чанков)**: исходный массив дополняется до степени двойки, после чего 
разбивается на чанки. Количество чанков выбирается как наибольшая степень двойки, не превышающая 
количество доступных потоков. Каждый чанк сортируется независимо в отдельном потоке с помощью 
поразрядной сортировки.

- **Фаза 2 (параллельное слияние Бэтчера)**: выполняется итеративное слияние отсортированных чанков 
с удвоением размера блока на каждом уровне. На каждом уровне слияния все независимые блоки 
обрабатываются параллельно с использованием итеративной сети Бэтчера (odd-even merge network).

### 4.2. Используемые примитивы STL

- **`std::thread`** – создание нативных потоков для параллельного выполнения.

### 4.3. Распределение данных

**Фаза сортировки чанков**:

```cpp
std::vector<std::thread> threads;
threads.reserve(chunks_count);

for (size_t i = 0; i < chunks_count; ++i) {
    threads.emplace_back([raw_data, i, chunk_size]() {
        SortChunk(raw_data, i * chunk_size, chunk_size);
    });
}

for (auto &thread : threads) {
    thread.join();
}
```

Каждый поток создаёт локальную копию своего чанка, сортирует её и копирует обратно. 
Это исключает гонки за данные и улучшает кэш-локальность.

**Фаза слияния Бэтчера**:

```cpp
void MergeSortedChunks(double *raw_data, size_t total_size, size_t chunk_size) {
    for (size_t current_size = chunk_size; current_size < total_size; current_size *= 2) {
        size_t merges_count = total_size / (current_size * 2);
        std::vector<std::thread> threads;
        threads.reserve(merges_count);
        
        for (size_t i = 0; i < merges_count; ++i) {
            threads.emplace_back([raw_data, i, current_size]() {
                size_t start = i * 2 * current_size;
                OddEvenMergeIterative(raw_data, start, 2 * current_size);
            });
        }
        
        for (auto &thread : threads) {
            thread.join();
        }
    }
}
```

На каждом уровне слияния количество независимых блоков равно total_size / (current_size * 2). 
Все блоки на уровне обрабатываются параллельно. После завершения всех блоков уровня происходит 
синхронизация, и алгоритм переходит к следующему уровню.

### 4.4. Итеративная сеть Бэтчера

Для слияния двух отсортированных массивов используется итеративная версия четно-нечетной сети Бэтчера:

```cpp
void OddEvenMergeIterative(double *arr, size_t start, size_t n) {
    if (n <= 1) return;
    
    size_t step = n / 2;
    CompareExchangeBlocks(arr, start, step);
    
    step /= 2;
    for (; step > 0; step /= 2) {
        for (size_t i = step; i < n - step; i += step * 2) {
            CompareExchangeBlocks(arr, start + i, step);
        }
    }
}
```

Отличительные особенности: Итеративность: отсутствие рекурсии исключает риск переполнения стека. 
Регулярность: все операции сравнения-обмена имеют предсказуемую структуру. 
Эффективность: количество сравнений ~ n log² n, что оптимально для параллельной реализации.

### 4.5. Выбор количества чанков

```cpp
size_t chunks_count = 1;
while (chunks_count * 2 <= static_cast<size_t>(threads_count) && 
       chunks_count * 2 <= padded_size) {
    chunks_count *= 2;
}
```

Количество чанков всегда является степенью двойки, что обеспечивает: Сбалансированное дерево слияния 
(каждый уровень объединяет пары чанков).
Эффективное использование кэш-памяти (размер чанка обратно пропорционален количеству потоков). 
Минимальные накладные расходы на синхронизацию.

## 5. Отсутствие гонок данных

Гарантии корректности обеспечиваются следующими механизмами:
Независимые чанки на фазе сортировки:
Каждый поток работает с локальной копией своего чанка. Чанки не пересекаются и не имеют общих элементов. 
Запись результатов происходит в непересекающиеся области памяти.
Независимые блоки на фазе слияния: На каждом уровне current_size блоки [i, i+2*current_size) не пересекаются. 
Каждый поток обрабатывает свой уникальный блок. Синхронизация происходит только между уровнями.
Отсутствие разделяемых переменных: Все переменные либо локальны для потока, либо используются только для чтения. 
Нет необходимости в атомарных операциях или блокировках.

2 потока: при использовании двух потоков достигается сверхлинейное ускорение. 
Причиной сверхлинейного ускорения являеься синергия эффекта кэш-памяти (основной фактор) 
и параллельное выполнение некскольких частей программы.
Последовательная версия обрабатывает массив размером 8 МБ (1 001 000 элементов double). 
STL версия разбивает массив на 2 чанка по 4 МБ каждый. Каждый чанк полностью помещается в L3 кэш. 
В результате промахи кэша сокращаются до 10-15%, а данные остаются в кэше между проходами сортировки.
Синергетический эффект можно выразить формулой: ускорение равно параллелизму (2x) умноженному 
на кэш-эффект (примерно 2.3x) умноженному на эффективность слияния (1.04x), что даёт 4.78x, 
что соответствует полученным 4.75x.

4 потока, 8 потоков: При четырёх потоках ускорение выросло до 5.31x, а эффективность 
снизилась до 132.7 процентов. В STL реализации количество чанков ограничено степенью двойки 
и не превышает число потоков, но из-за того что на этапе сортировки создаётся только chunks_count 
потоков (2 при 2 потоках, 4 при 4 потоках), а не используется parallel_invoke с двумя задачами, 
STL версия может масштабироваться на 4 потока. При 8 потоках ускорение достигает 5.39x, 
а эффективность падает до 67.3 процентов.
