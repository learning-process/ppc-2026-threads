## 1. Контекст

Гибридная реализация MPI+TBB объединяет два уровня параллелизма:
- **MPI** — для распределения данных между процессами (межпроцессный параллелизм)
- **TBB** — для параллельной сортировки на каждом процессе (внутрипроцессный параллелизм)

Основная идея заключается в разбиении исходного массива на порции между MPI-процессами, 
независимой сортировке каждой порции с использованием TBB, последующем сборе данных 
на мастер-процесс и финальной сортировке. Такой подход позволяет эффективно использовать 
многоядерные кластеры и обрабатывать массивы, не помещающиеся в памяти одного узла.

## 2. Постановка задачи

Входные данные: неотсортированный вектор вещественных чисел (`std::vector<double>`).
Выходные данные: отсортированный по возрастанию вектор.

Требования:
- Результат гибридной версии должен быть идентичен последовательной реализации
- Алгоритм должен корректно работать как с MPI, так и без него
- Поддержка произвольного количества MPI-процессов и TBB-потоков
- Эффективное использование кэш-памяти и коммуникационной сети


## 3. Базовый алгоритм

### 3.1. Последовательная основа

Последовательный алгоритм включает следующие этапы:
1. **Преобразование double → uint64_t** с сохранением порядка:
   - Положительные числа: установка старшего бита
   - Отрицательные числа: побитовая инверсия
2. **Поразрядная сортировка (LSD Radix Sort)**:
   - 8 проходов (по 8 бит на проход)
   - Разрядность = 256
   - Устойчивая сортировка подсчётом
3. **Обратное преобразование uint64_t → double**
4. **Слияние отсортированных половин** (в последовательной версии)

### 3.2. Сеть Бэтчера (Odd-Even Merge)

Используется для слияния двух отсортированных массивов. Итеративная версия:
```cpp
void OddEvenMergeIterative(double *arr, size_t start, size_t n) {
    size_t step = n / 2;
    CompareExchangeBlocks(arr, start, step);
    step /= 2;
    for (; step > 0; step /= 2) {
        for (size_t i = step; i < n - step; i += step * 2) {
            CompareExchangeBlocks(arr, start + i, step);
        }
    }
}
```
## 4. Схема распараллеливания
### 4.1. Двухуровневая декомпозиция
Алгоритм разделён на два уровня параллелизма:
Уровень MPI (межпроцессный):
Распределение исходных данных между процессами
Сбор результатов на мастер-процесс
Рассылка отсортированного массива всем процессам
Уровень TBB (внутрипроцессный):
Разбиение локальной порции на чанки
Параллельная поразрядная сортировка чанков
Параллельное слияние Бэтчера

### 4.2. Используемые примитивы
MPI_Scatterv для распределения данных
MPI_Gatherv для	сбора результатов
MPI_Bcast для рассылки результатов
tbb::parallel_for для параллельной сортировки чанков и параллельного слияния

### 4.3. Распределение данных
text
Исходный массив: [d0, d1, d2, d3, d4, d5, d6, d7]
MPI процессов: 2W
Процесс 0: [d0, d1, d2, d3]
Процесс 1: [d4, d5, d6, d7]
Размер порции для каждого процесса вычисляется с учётом остатка:
```cpp
size_t base_count = global_n / size;
size_t remainder = global_n % size;
send_counts[i] = base_count + (i < remainder ? 1 : 0);
```
### 4.4. Локальная TBB-сортировка
На каждом MPI-процессе выполняется:
```cpp
// 1. Дополнение до степени двойки
size_t local_new_size = NextPowerOfTwo(local_n_original);
local_data.resize(local_new_size, INFINITY);

// 2. Разбиение на чанки (количество = степень двойки)
size_t num_chunks = 1;
while (num_chunks * 2 <= num_threads && num_chunks * 2 <= local_new_size) {
    num_chunks *= 2;
}

// 3. Параллельная сортировка чанков
tbb::parallel_for(0, num_chunks_int, [&](int i) {
    SortChunkTBB(raw_data, i, chunk_size);
});

// 4. Параллельное слияние Бэтчера
for (size_t cur_size = chunk_size; cur_size < local_new_size; cur_size *= 2) {
    tbb::parallel_for(0, merges_count, [&](int i) {
        OddEvenMergeIterative(raw_data, i * 2 * cur_size, 2 * cur_size);
    });
}
```
### 4.5. Сбор и финальная сортировка
После локальной сортировки все данные собираются на процессе 0:

``` cpp
if (rank == 0) {
    std::vector<double> all_data = std::move(local_data);
    for (int p = 1; p < size; ++p) {
        MPI_Recv(&recv_size, 1, MPI_INT, p, 0, ...);
        MPI_Recv(recv_data.data(), recv_size, MPI_DOUBLE, p, 1, ...);
        all_data.insert(all_data.end(), recv_data.begin(), recv_data.end());
    }
    local_data = std::move(all_data);
} else {
    MPI_Send(&my_size, 1, MPI_INT, 0, 0, ...);
    MPI_Send(local_data.data(), my_size, MPI_DOUBLE, 0, 1, ...);
}

// Финальная сортировка на процессе 0
if (rank == 0 && !local_data.empty()) {
    RadixSortDouble(local_data);
}
``` 
### 4.6. Рассылка результата
Отсортированный массив рассылается всем процессам:
```cpp
size_t total_size = local_data.size();
MPI_Bcast(&total_size, 1, MPI_UNSIGNED_LONG, 0, MPI_COMM_WORLD);

if (rank == 0) {
    MPI_Bcast(local_data.data(), total_size, MPI_DOUBLE, 0, MPI_COMM_WORLD);
    GetOutput() = std::move(local_data);
} else {
    local_data.resize(total_size);
    MPI_Bcast(local_data.data(), total_size, MPI_DOUBLE, 0, MPI_COMM_WORLD);
    GetOutput() = std::move(local_data);
}
```
## 5. Детали реализации
### 5.1. Ключевые функции
Функция	Назначение

SortableIntToDouble	Обратное преобразование
RadixSortDouble	Поразрядная сортировка
CompareExchangeBlocks	Сравнение и обмен блоков
OddEvenMergeIterative	Итеративная сеть Бэтчера
SortChunkTBB	Сортировка одного чанка
RunImpl	Основной метод (MPI + TBB)
### 5.2. Обработка граничных случаев
Пустые массивы — ранний выход
Остаток при делении — распределение между первыми процессами
Размер локальных данных < степени двойки — дополнение INFINITY
Нулевое количество потоков — установка в 1
### 5.3. Отсутствие гонок данных
Фаза распределения: каждый процесс получает непересекающуюся порцию
Фаза локальной сортировки: чанки не пересекаются
Фаза сбора: синхронные MPI-операции
Фаза рассылки: широковещательная операция

(MPI не задействован, так как процесс один). Два потока TBB работают
в рамках одного процесса, разделяя общую память.
Положительные аспекты:

Хорошая кэш-локальность благодаря разбиению массива на чанки

процессами, затем происходит сбор результатов на процесс 0
с последующей финальной сортировкой.
Положительные аспекты:
Ускорение присутствует (1.34×) — задача действительно распараллеливается
Корректная работа MPI-коммуникаций без deadlock'ов
Отрицательные аспекты (потери 33%):
Накладные расходы на MPI-коммуникации (~20%):
Рассылка исходных данных процессам
Сбор отсортированных порций на процесс 0
Широковещательная рассылка финального результата

Последовательное слияние (~5%): Процесс 0 последовательно принимает 
и сливает данные от процесса 1

Режим 2 процесса × 2 потока
Это лучшая конфигурация, использующая гибридный подход:
два MPI-процесса, каждый из которых запускает по два TBB-потока.
Суммарно задействовано 4 вычислительных ядра (или аппаратных потока).
Синергетический эффект:
MPI уменьшает объём данных на каждый процесс до ~500 000 элементов
TBB внутри каждого процесса эффективно сортирует эти уменьшенные порции
Коммуникационные накладные расходы MPI перекрываются выигрышем
от параллельной сортировки
Кэш-эффект:
Порция данных на процесс (~4 МБ) идеально помещается в L3-кэш 
В последовательной версии массив 8 МБ создаёт больше кэш-промахов 
Улучшенная локальность данных даёт дополнительный прирост производительности 
Сбалансированная загрузка:
Два MPI-процесса равномерно распределяют данные
Два TBB-потока внутри каждого процесса обеспечивают максимальную утилизацию ядер
Финальная сортировка на процессе 0 работает
с уже почти отсортированными данными
