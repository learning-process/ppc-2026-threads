## 4. Схема распараллеливания

### Параллельная сортировка половин

Два раздела выполняются параллельно

```cpp
#pragma omp parallel sections default(none) shared(left, right)
{
  #pragma omp section
  {
    RadixSortDouble(left);
  }
  #pragma omp section
  {
    RadixSortDouble(right);
  }
}
```

Параллелизуемая область: две независимые сортировки левой и правой половин массива.

Переменные и их атрибуты:
default(none) — требует явного указания атрибутов всех переменных.
shared(left, right) — оба потока работают со своими массивами (разные участки памяти), гонок нет.

### Параллельное слияние Бэтчера

Параллельное выполнение независимых итераций

```cpp
void BatcherOddEvenMergeIterative(std::vector<double> &arr, size_t n) {
  for (size_t len = 2; len <= n; len *= 2) {
    #pragma omp parallel for default(none) shared(arr, n, len)
    for (size_t i = 0; i < n; i += len) {
      MergingHalves(arr, i, len);
    }
  }
}
```

Параллелизуемая область: внешний цикл по размерам блоков len.
Итерации внутреннего цикла по начальным индексам i независимы,
так как каждый блок сливается самостоятельно.

Переменные и их атрибуты:
shared(arr, n, len) — массив arr используется всеми потоками
для чтения и записи, но в непересекающиеся диапазоны.
len и n — только для чтения.

### Синхронизация

Происходит в конце каждой parallel области (неявный барьер) -
ожидание завершения всех потоков

```cpp
    MergingHalves(arr, i, len);
  }
}
```

### Устранение рисков гонок

Проблема 1: несколько потоков одновременно обновляют массив arr.
Решение: каждый поток работает только с непересекающимися
диапазонами индексов:
В parallel sections: left и right — разные векторы.
В parallel for: каждый поток получает уникальное значение i,
и все операции MergingHalves ограничены диапазоном [i, i+len),
которые не пересекаются для разных i.

Проблема 2: одновременный доступ к общим переменным.
Решение: все переменные, кроме тех, что используются в reduction,
объявлены как shared только для чтения или с непересекающейся записью.
