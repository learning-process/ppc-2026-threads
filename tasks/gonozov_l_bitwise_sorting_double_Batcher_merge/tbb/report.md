# Поразрядная сортировка для вещественных чисел (тип double) с четно-нечетным слиянием Бэтчера

## 1. Контекст

Реализация на базе oneTBB использует двухуровневый параллелизм. 
На первом этапе массив делится на две половины, которые сортируются 
параллельно с помощью поразрядной сортировки. На втором этапе применяется 
итеративная сеть Бэтчера, где операции сравнения-обмена на каждом уровне 
выполняются параллельно. Такой подход позволяет эффективно использовать 
кэш-память и снизить накладные расходы на синхронизацию.

## 2. Постановка задачи

Постановка идентична `seq/report.md`. Вход — `std::vector<double>`, 
выход — отсортированный по возрастанию вектор. Требования корректности: 
результат TBB-версии должен полностью совпадать с результатом последовательного 
эталона на любом наборе входных данных.

## 3. Базовый алгоритм

Последовательный алгоритм описан в `seq/report.md`. Ключевые этапы:

1. Преобразование `double` → `uint64_t` с сохранением порядка.
2. Поразрядная сортировка (8 проходов, radix = 256).
3. Слияние отсортированных половин сетью Бэтчера.
4. Обратное преобразование `uint64_t` → `double`.

## 4. Схема распараллеливания

### 4.1. Декомпозиция задачи

Алгоритм разбит на две крупные параллельные фазы:

- **Фаза 1 (сортировка половин)**: исходный массив дополняется 
до степени двойки, после чего делится на две равные половины. 
Левая и правая половины сортируются одновременно с помощью `tbb::parallel_invoke`. 
Каждая половина обрабатывается последовательной поразрядной сортировкой.

- **Фаза 2 (слияние Бэтчера)**: применяется итеративная сеть Бэтчера ко всему массиву. 
На каждом уровне размер сливаемых блоков удваивается, 
а все независимые операции слияния выполняются параллельно.

### 4.2. Используемые примитивы TBB

- **`tbb::parallel_invoke`** – запускает сортировку левой и правой половины массива параллельно.
- **`tbb::parallel_for`** – выполняет итерации слияния на каждом уровне сети Бэтчера параллельно.
- **`tbb::blocked_range<size_t>`** – разбивает диапазон блоков на поддиапазоны, которые распределяются между потоками.

### 4.3. Распределение данных

**Фаза сортировки половин**:

```cpp
tbb::parallel_invoke(
    [&]() { RadixSortDouble(left); },
    [&]() { RadixSortDouble(right); }
);
```

Каждая половина массива сортируется независимо. parallel_invoke обеспечивает 
параллельное выполнение двух задач, эффективно загружая доступные потоки.

**Фаза слияния**:

```cpp
for (size_t len = 2; len <= n; len *= 2) {
    tbb::parallel_for(tbb::blocked_range<size_t>(0, n, len),
        [&](const tbb::blocked_range<size_t> &r) {
            for (size_t i = r.begin(); i < r.end(); i += len) {
                MergingHalves(arr, i, len);
            }
        });
}
```

На каждом уровне len количество независимых операций слияния равно n / len. 
Каждая операция MergingHalves обрабатывает блок размером len, выполняя серию 
сравнений-обменов между двумя подполовинками. TBB автоматически разбивает 
диапазон блоков на поддиапазоны для балансировки загрузки.

## 6. Отсутствие гонок гарантируется тем, что

На фазе сортировки левая и правая половины массива не пересекаются.
На фазе слияния каждая операция MergingHalves работает 
с независимым блоком памяти [i, i+len).
Блоки на одном уровне слияния не пересекаются, так как шаг между 
началами блоков равен len.
Отсутствуют разделяемые переменные, требующие атомарных операций или блокировок.

## 7. Результаты экспериментов

| Mode        | Processes | Threads | Time, s | Speedup | Efficiency |
|-------------|-----------|---------|---------|---------|------------|
| seq         | 1         | 1       | 0.1099  | 1.00    | N/A        |
| tbb         | 1         | 2       | 0.0667  | 1,647   | 82.38%     |
| tbb         | 1         | 4       | 0.0512  | 2,146   | 53.66%     |
| tbb         | 1         | 8       | 0.0518  | 2,121   | 26.52%     |

2 потока: При использовании двух потоков достигается высокая эффективность. 
Это объясняется тем, что фаза сортировки половин (parallel_invoke) полностью 
загружает оба доступных потока, так как левая и правая половины массива имеют 
одинаковый размер и сортируются независимо.
Фаза слияния Бэтчера также хорошо распараллеливается: на ранних уровнях сети 
количество независимых блоков велико (n/2, n/4 и т.д.), что позволяет эффективно 
распределять работу между двумя потоками. Накладные расходы на создание и синхронизацию 
задач TBB при использовании двух потоков минимальны. Потери в 17.62% связаны в основном 
с неизбежной последовательной частью алгоритма (преобразование типов, 
копирование данных, финальная обрезка массива) и накладными расходами на внутреннюю 
синхронизацию в parallel_for.

4 потока: при увеличении числа потоков до четырёх ускорение продолжает расти, 
но эффективность снижается. Основные причины:
Ограничение на фазе сортировки: parallel_invoke создаёт только две параллельные задачи. 
Это означает, что из четырёх доступных потоков только два заняты работой на первом этапе, 
а остальные два простаивают. Это создаёт узкое место и ограничивает максимальное ускорение.

Фаза слияния хотя и использует все потоки, но на поздних уровнях сети Бэтчера 
(при больших len) количество независимых блоков становится малым 
(например, при len = n/2 существует всего 2 блока). Это приводит к недозагрузке части потоков.
Конкуренция за кэш-память: при работе четырёх потоков с общим массивом возрастает 
количество промахов кэша из-за того, что потоки обращаются к различным, 
но близко расположенным областям памяти.
Накладные расходы на планировщик TBB: с ростом числа потоков увеличивается 
время на распределение задач и балансировку нагрузки.

8 потоков: Дальнейшее увеличение числа потоков до восьми не даёт прироста 
производительности — ускорение даже незначительно снижается по сравнению с четырьмя потоками. 
Закон Амдала: доля последовательного кода (включая фазу сортировки, 
которая использует только 2 потока) ограничивает максимальное ускорение. 
Для данной реализации теоретический предел ускорения не может превышать 
отношение времени работы последовательной версии к времени работы самой медленной обязательной 
последовательной части. Насыщение полезной работы: четыре потока уже достаточно полно 
загружают фазу слияния. Добавление дополнительных потоков не создаёт новой полезной работы, 
а только увеличивает накладные расходы. Возрастание накладных расходов: при восьми потоках 
планировщик TBB тратит больше времени на миграцию задач между потоками, распределение блоков 
и синхронизацию. Эти затраты начинают превышать выгоду от дополнительного распараллеливания. 
Проблемы масштабирования сети Бэтчера: общее количество независимых операций слияния 
на всех уровнях сети фиксировано и равно n-1. При увеличении числа потоков средний объём работы 
на один поток уменьшается, и накладные расходы становятся сопоставимы с полезной работой.
