## 1. Контекст

Ручная `std::thread`-версия нужна для явного управления жизненным циклом потоков и схемой распределения работы без
runtime уровня OpenMP или oneTBB. Эта реализация показывает, как та же задача может быть распараллелена средствами
стандартной библиотеки C++ с ручным контролем `create -> work -> join`.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел по схеме «сортировка Хоара с простым слиянием»: вход делится на две части,
каждая часть сортируется отдельно, затем две отсортированные последовательности объединяются линейным слиянием.
Последовательная версия из [seq/report.md](tasks/tochilin_e_hoar_sort_sim_mer/seq/report.md) используется как эталон
корректности и baseline по времени.

## 3. Базовый алгоритм

Последовательная версия делит массив на `left` и `right`, сортирует обе половины итеративным quicksort по схеме Хоара и
затем объединяет их через `std::ranges::merge`.

В `STL`-версии сохраняется та же структура:

1. вход делится на две половины;
2. каждая половина сортируется по схеме Хоара;
3. результат собирается одним простым слиянием.

Отличие в том, что внутри каждой половины крупные диапазоны распределяются между потоками вручную через `std::thread`, а
мелкие диапазоны досортировываются локальной последовательной стековой версией quicksort.

## 4. Схема распараллеливания

Как данные делятся на диапазоны:

- на верхнем уровне исходный массив делится на две половины: `left` и `right`;
- для каждой половины `QuickSortParallel(...)` поддерживает текущий фронт диапазонов `current_ranges`;
- на каждой волне диапазоны распределяются между потоками по схеме `worker_idx, worker_idx + active_workers, ...`.

Что делает каждый поток:

- получает собственный набор индексов во векторе `current_ranges`;
- для каждого диапазона вызывает `ProcessRange(...)`;
- если диапазон ещё крупный, разбивает его дальше и складывает новые поддиапазоны в локальный буфер;
- если диапазон меньше `serial_cutoff`, досортировывает его локально через `QuickSortSequential(...)`.

Где хранятся локальные результаты:

- новые поддиапазоны собираются в `next_ranges_local[worker_idx]`, то есть в отдельном векторе, принадлежащем
  конкретному потоку;
- глобальный вектор `next_ranges` формируется только после завершения всех потоков текущей волны.

Как и когда выполняется `join`:

- сначала создаются все потоки для текущей волны;
- затем выполняется проход `for (auto &worker : workers) { worker.join(); }`;
- только после завершения всех потоков объединяются локальные буферы и строится следующий фронт диапазонов;
- на верхнем уровне `RunImpl()` отдельно создаёт два потока для сортировки `left` и `right`, и `join()` вызывается
  только после запуска обеих ветвей.

Какая синхронизация используется и почему:

- `mutex`, `atomic` и `condition_variable` не используются;
- это безопасно, потому что каждый поток либо работает со своим набором диапазонов, либо пишет только в свой локальный
  `next_ranges_local`;
- общий `next_ranges` строится уже после `join()`, то есть в последовательной фазе.

Минимальный иллюстративный фрагмент:

```cpp
std::vector<std::thread> workers;
workers.reserve(2);

workers.emplace_back(
    [&]() { QuickSortParallel(left, 0, static_cast<int>(left.size()) - 1, serial_cutoff, left_workers); });
workers.emplace_back(
    [&]() { QuickSortParallel(right, 0, static_cast<int>(right.size()) - 1, serial_cutoff, right_workers); });

for (auto &worker : workers) {
  worker.join();
}
```

Этот фрагмент показывает верхнеуровневую thread-based схему: обе половины массива запускаются параллельно, а объединение
допускается только после завершения обеих ветвей.

## 5. Детали реализации

Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Карта `поток -> диапазон`:

- в `QuickSortParallel(...)` поток с номером `worker_idx` берёт диапазоны `idx = worker_idx, worker_idx +
  active_workers, ...`;
- тем самым фронт диапазонов распределяется между потоками циклически и без пересечения;
- внутри `RunImpl()` отдельные потоки верхнего уровня закреплены за `left` и `right`.

Схема объединения локальных результатов:

- каждый поток складывает новые поддиапазоны в собственный `next_ranges_local[worker_idx]`;
- после `join()` все локальные векторы последовательно конкатенируются в `next_ranges`;
- после полной сортировки обеих половин итоговый массив строится через `MergeSortedVectors(left, right)`.

## 6. Проверка корректности

Корректность STL-версии проверялась сравнением результата с эталонной сортировкой `std::ranges::sort(reference)` в
`tests/functional/main.cpp`.

Использованные сценарии:

- минимальные размеры: `OneElement`, `TwoElements`;
- типичные входы: `EightElements`, `RandomSize`, `MediumSize`, `LargeSize`;
- специальные случаи: `AlreadySorted`, `ReverseSorted`;
- крупные входы: `LongSize`, `LongLongSize`.

Характерные примеры:

- `OneElement`: вход `[42]` должен остаться `[42]`;
- `TwoElements`: вход `[2, 1]` должен стать `[1, 2]`;
- `LongLongSize`: на размере `2000` проверяется корректность работы нескольких волн ручного распределения диапазонов.

Проверка на отсутствие гонок основана на структуре записи:

- потоки не обновляют общий аккумулятор;
- вектор `next_ranges_local` разделён по индексам потока;
- итоговый `next_ranges` строится только после завершения всех потоков;
- `PostProcessingImpl()` дополнительно подтверждает корректность через `std::ranges::is_sorted(GetOutput())`.

Комментарий по накладным расходам `create/join`:

- на малом числе потоков ручная версия показывает устойчивый выигрыш;
- при росте числа потоков заметным фактором становятся волновое создание потоков в `QuickSortParallel(...)` и их
  синхронизация на каждом шаге;
- лучший результат достигается около `12` потоков, после чего дальнейший рост почти не улучшает время и начинает
  упираться в overhead управления потоками.

## 9. Выводы

Ручная реализация на `std::thread` показала, что задачу можно эффективно распараллелить без внешнего task runtime, если
явно контролировать разбиение работы и точки синхронизации. Эта версия сохраняет алгоритм Хоара и простое итоговое
слияние, при этом демонстрирует хороший выигрыш на крупных входах. Главные ограничения такого подхода - накладные
расходы на создание и завершение потоков и необходимость вручную поддерживать корректную схему распределения диапазонов
без гонок.
