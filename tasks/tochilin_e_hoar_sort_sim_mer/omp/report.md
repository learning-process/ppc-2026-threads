## 1. Контекст

В OpenMP-версии в параллельное исполнение переносится сортировка двух половин массива и дальнейшее разбиение диапазонов
внутри quicksort. Цель этой реализации - сохранить структуру исходного алгоритма и при этом использовать task-based
модель OpenMP для независимых подзадач сортировки.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел по схеме «сортировка Хоара с простым слиянием»: вход делится на две части,
каждая часть сортируется отдельно, после чего выполняется линейное слияние. Базовый эталон корректности и baseline по
времени описаны в [seq/report.md](tasks/tochilin_e_hoar_sort_sim_mer/seq/report.md).

## 3. Базовый алгоритм

Последовательная версия делит массив на `left` и `right`, сортирует обе половины итеративной реализацией quicksort по
схеме Хоара и затем объединяет их через `std::ranges::merge`. OpenMP-версия сохраняет ту же структуру данных и тот же
критерий корректности, но распараллеливает независимые ветви сортировки.

## 4. Схема распараллеливания

Параллелится участок сортировки двух половин массива и дальнейшая обработка независимых поддиапазонов quicksort внутри
`QuickSortOMP(...)`.

Переменные и область данных:

- `left` и `right` являются `shared` внутри `parallel region`, потому что задачи работают с уже выделенными половинами
  массива;
- `l`, `j`, `i`, `r`, `depth_limit` передаются в дочерние задачи как `firstprivate`, чтобы каждая задача получила
  собственные границы диапазона;
- локальные переменные внутри `Partition(...)` и внутри тела `QuickSortOMP(...)` являются приватными по области
  видимости.

Синхронизация:

- `reduction`, `atomic` и `critical` не требуются, потому что каждая задача сортирует собственный поддиапазон и не
  накапливает общий скалярный результат;
- в конце `QuickSortOMP(...)` используется `#pragma omp taskwait`, чтобы завершить все дочерние задачи перед выходом из
  текущей ветви;
- у `#pragma omp parallel` есть неявный барьер на выходе из области, поэтому после завершения `single` и всех `task` обе
  половины массива гарантированно отсортированы до вызова `MergeSortedVectors(...)`.

Выбор `schedule`:

- директива `parallel for` в реализации не используется, поэтому параметр `schedule` здесь не задаётся;
- балансировка работы выполняется через OpenMP task runtime, который сам распределяет независимые подзадачи quicksort по
  доступным потокам.

Минимальный иллюстративный фрагмент:

```cpp
#pragma omp parallel default(none) shared(left, right) num_threads(thread_count) if (thread_count > 1)
{
#pragma omp single
  {
#pragma omp task default(none) shared(left)
    QuickSortOMP(left, 0, static_cast<int>(left.size()) - 1, 3);

#pragma omp task default(none) shared(right)
    QuickSortOMP(right, 0, static_cast<int>(right.size()) - 1, 3);

#pragma omp taskwait
  }
}
```

Этот фрагмент показывает верхний уровень распараллеливания: две половины массива сортируются как независимые задачи, а
затем синхронизируются перед слиянием.

## 5. Детали реализации

Файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Относительно `SEQ` были изменены следующие участки:

- вместо `QuickSort(...)` используется `QuickSortOMP(...)`, умеющий порождать дочерние OpenMP-задачи;
- в `RunImpl()` добавлена `parallel`-область с `single`, внутри которой запускаются две верхнеуровневые задачи для
  `left` и `right`;
- число потоков берётся из `ppc::util::GetNumThreads()`, что связывает реализацию с конфигурацией `PPC_NUM_THREADS`.

Риски гонок и как они устранены:

- `left` и `right` сортируются независимо, поэтому между ними нет конкурентной записи;
- внутри одной половины новые задачи получают непересекающиеся диапазоны после `Partition(...)`;
- дочерние задачи не обновляют общий счётчик или буфер, кроме собственного участка массива;
- `taskwait` исключает преждевременный выход из ветви сортировки.

## 6. Проверка корректности

Корректность OpenMP-версии проверялась сравнением результата с эталонной сортировкой `std::ranges::sort(reference)` в
общем функциональном наборе `tests/functional/main.cpp`.

Использованные классы тестов:

- минимальные размеры: `OneElement`, `TwoElements`;
- типичные случайные входы: `RandomSize`, `MediumSize`, `LargeSize`;
- специальные случаи: `AlreadySorted`, `ReverseSorted`;
- крупные входы: `LongSize`, `LongLongSize`.

Характерные примеры:

- `OneElement`: вход `[42]` должен вернуться без изменений;
- `ReverseSorted`: массив `127, 126, ..., 1` должен превратиться в `1, 2, ..., 127`;
- `LongLongSize`: на размере `2000` проверяется корректность task-based сортировки и итогового слияния на крупном
  детерминированном входе.

Дополнительно `PostProcessingImpl()` подтверждает корректность через `std::ranges::is_sorted(GetOutput())`.

```powershell

cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release

.\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_tochilin_e_hoar_sort_sim_mer_omp_enabled*"
```

Размер задачи в performance-тесте:

- `N = 2_000_000`
- генерация входа детерминирована через `std::seed_seq{N, 0x5A17C3D2U}`
- диапазон значений: `[-10000, 10000]`

## 9. Выводы

OpenMP даёт устойчивый выигрыш на больших входах, когда ветви quicksort действительно успевают породить достаточное
число независимых задач. На этой задаче версия показывает заметное ускорение уже при `2` потоках и достигает лучшей
медианы около `12` потоков, но дальнейший рост ограничивается накладными расходами task runtime и тем, что финальное
слияние остаётся последовательным. Эта версия хорошо подходит как task-based реализация схемы Хоара, но её
масштабируемость не является линейной.
