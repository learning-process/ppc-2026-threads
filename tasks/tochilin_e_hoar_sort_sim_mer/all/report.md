## 1. Контекст

Гибридная версия требуется для объединения двух уровней параллелизма: межпроцессного распределения входного массива
через MPI и внутрипроцессной сортировки локальных фрагментов через OpenMP. Такая схема позволяет проверить, насколько
задача «сортировка Хоара с простым слиянием» масштабируется не только по потокам внутри одного процесса, но и по
нескольким MPI rank-ам.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел по схеме «сортировка Хоара с простым слиянием». Входной массив делится между
MPI-процессами, каждый процесс сортирует свой локальный фрагмент, затем локальные отсортированные куски объединяются в
общий глобальный результат. Последовательный baseline описан в
[seq/report.md](tasks/tochilin_e_hoar_sort_sim_mer/seq/report.md); сводное сравнение всех backend-ов даётся в корневом
отчёте проекта.

## 3. Базовый алгоритм

Последовательная логика остаётся той же: массив сортируется по схеме Хоара и затем представляется как одна
отсортированная последовательность. В `ALL` эта идея переносится на два уровня:

- на уровне MPI массив разбивается на непрерывные куски;
- на уровне rank-а каждый кусок сортируется локальным quicksort по схеме Хоара;
- после этого локальные результаты объединяются в один глобальный отсортированный массив.

## 4. Межпроцессная схема

Роли rank-ов:

- `rank 0` владеет исходным глобальным входом, инициирует рассылку фрагментов и в конце получает итоговый
  отсортированный массив;
- остальные rank-и получают только свой локальный кусок и участвуют в дереве глобального слияния.

Как распределяются данные:

- размер глобального массива делится между `proc_count` процессами через векторы `counts` и `displs`;
- `counts[rank]` задаёт размер локального фрагмента для данного rank-а;
- `displs[rank]` задаёт смещение этого фрагмента в глобальном массиве.

Какие MPI-вызовы используются и зачем:

- `MPI_Comm_rank` и `MPI_Comm_size` определяют номер текущего процесса и размер коммуникатора;
- `MPI_Scatterv` раздаёт непрерывные фрагменты исходного массива с `rank 0` по всем процессам;
- `MPI_Send` / `MPI_Recv` используются в дереве межпроцессного слияния: часть rank-ов отправляет уже отсортированные
  локальные данные соседним rank-ам, а принимающие rank-и выполняют локальное merge;
- `MPI_Bcast` рассылает итоговый глобальный результат всем участникам после завершения дерева слияний;
- `MPI_Barrier` стоит в конце `RunImpl()` и гарантирует, что все процессы завершили вычисление и получили согласованный
  результат перед выходом из задачи.

Минимальный иллюстративный фрагмент:

```cpp
MPI_Scatterv(data.data(), counts.data(), displs.data(), MPI_INT, local_data.data(),
             counts[static_cast<std::size_t>(rank)], MPI_INT, 0, MPI_COMM_WORLD);

local_data = MergeAcrossRanks(std::move(local_data), rank, proc_count);

if (rank == 0) {
  data = std::move(local_data);
}

MPI_Bcast(data.data(), total_size, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Barrier(MPI_COMM_WORLD);
```

Этот фрагмент показывает главный MPI-каркас версии: распределение исходного массива, межпроцессное объединение локальных
результатов и финальную рассылку общего ответа всем rank-ам.

## 5. Внутрипроцессная схема

Внутри каждого rank-а используется OpenMP.

Как делится работа между потоками:

- каждый процесс получает локальный вектор `local_data`;
- если локальный фрагмент не пуст, внутри rank-а создаётся `parallel`-область OpenMP;
- в `single`-секции запускается `QuickSortOMP(...)`, который по схеме task-based quicksort разбивает локальный диапазон
  на независимые поддиапазоны;
- дочерние OpenMP-задачи сортируют эти поддиапазоны параллельно до заданной глубины `depth_limit = 3`.

Как объединяются локальные результаты:

- внутри rank-а дополнительных merge-ов между потоками нет, потому что OpenMP-задачи сортируют непересекающиеся
  поддиапазоны одного локального массива;
- локальным результатом rank-а является полностью отсортированный `local_data`;
- объединение разных локальных фрагментов происходит уже на уровне MPI.

## 6. Детали реализации

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

Структура реализации:

- `BuildCounts(...)` и `BuildDisplacements(...)` рассчитывают разбиение глобального массива по rank-ам;
- `QuickSortOMP(...)` сортирует локальный фрагмент внутри rank-а через OpenMP task-ы;
- `MergeAcrossRanks(...)` реализует дерево межпроцессных слияний;
- `MergeSortedVectors(...)` выполняет локальное линейное слияние двух уже отсортированных векторов.

Где находятся MPI-синхронизации и обмены:

- `Scatterv` - в начале вычисления, перед локальной сортировкой;
- `Send/Recv` - внутри дерева слияний между rank-ами;
- `Broadcast` - после того как `rank 0` получил итоговый массив;
- `Barrier` - в конце `RunImpl()`.

`Gather` и `Reduce` в текущей версии не используются: после улучшения финального этапа последовательное централизованное
слияние на `root` заменено на дерево merge-ов между процессами.

Потенциальные узкие места:

- пересылка уже больших отсортированных фрагментов на поздних уровнях дерева слияний;
- последовательное локальное merge на принимающем rank-е после `MPI_Recv`;
- финальный `MPI_Bcast`, который становится заметен на больших `proc_count`.

## 7. Проверка корректности

Корректность `ALL` проверялась сравнением с тем же эталоном, что и `SEQ`: в функциональных тестах результат сравнивается
с `std::ranges::sort(reference)`. Общий функциональный набор включает все backend-ы, поэтому `ALL` проверяется в той же
инфраструктуре, что и `SEQ`, `OMP`, `TBB` и `STL`.

Использованные сценарии:

- минимальные размеры: `OneElement`, `TwoElements`;
- типичные входы: `EightElements`, `RandomSize`, `MediumSize`, `LargeSize`;
- специальные случаи: `AlreadySorted`, `ReverseSorted`;
- крупные входы: `LongSize`, `LongLongSize`.

Согласованность результата между rank-ами подтверждается тем, что:

- после дерева межпроцессных слияний итоговый массив формируется на `rank 0`;
- затем этот массив рассылается всем rank-ам через `MPI_Bcast`;
- `PostProcessingImpl()` на каждом процессе проверяет `std::ranges::is_sorted(GetOutput())`.

Функциональные тесты для `ALL` были успешно пройдены на конфигурациях `1, 2, 4, 8, 12, 16` при одинаковом числе
процессов и потоков внутри rank-а.

Комментарий о цене коммуникации и синхронизации:

- переход от `1x1` к `2x2` и `4x4` даёт выигрыш, потому что локальная сортировка ещё доминирует над MPI-overhead;
- после этого стоимость обменов в дереве merge-ов, последовательных локальных слияний на принимающих rank-ах и
  финального `MPI_Bcast` начинает перевешивать выигрыш от распараллеливания;
- поэтому конфигурации `8x8` и выше уже проигрывают baseline по времени.

## 10. Выводы

Гибридная схема оправдана тогда, когда локальный фрагмент на каждом rank-е ещё достаточно велик, чтобы выигрыш от
параллельной сортировки перекрывал цену MPI-обменов. Для этой задачи удачными оказались умеренные конфигурации `2x2` и
`4x4`. При дальнейшем росте числа процессов и потоков стоимость коммуникации, дерева межпроцессных слияний и финальной
синхронизации становится определяющим ограничением, поэтому гибридная версия перестаёт масштабироваться линейно.
