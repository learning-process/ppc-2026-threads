## 1. Контекст

Задача состоит в сортировке целочисленного массива по схеме «сортировка Хоара с простым слиянием». Последовательная
версия нужна как эталон корректности и как baseline для расчёта ускорения в `omp`, `tbb`, `stl` и `all`.

## 2. Постановка задачи

Входные данные: `InType = std::vector<int>`, то есть одномерный массив целых чисел произвольного знака.

Выходные данные: `OutType = std::vector<int>` той же длины, содержащий те же элементы, но в неубывающем порядке.

Ограничения:

- пустой вход считается недопустимым и отсекается в `ValidationImpl`;
- массив из одного элемента допустим и должен возвращаться без изменений;
- допустимы уже отсортированные данные, обратный порядок и повторяющиеся значения.

Критерий корректности: результат должен быть отсортирован по неубыванию и совпадать с эталонной сортировкой того же
входа.

## 3. Базовый алгоритм

Алгоритм состоит из трёх шагов.

1. Исходный массив делится на две половины: `left` и `right`.
2. Каждая половина сортируется итеративной реализацией quicksort по схеме Хоара.
3. Две отсортированные половины объединяются линейным слиянием в итоговый массив.

Внутри quicksort используется явный стек диапазонов `(l, r)`, а не рекурсия. Для каждого диапазона выбирается опорный
элемент `arr[(l + r) / 2]`, затем выполняется разбиение и в стек добавляются ещё не обработанные поддиапазоны.

Асимптотика по времени: `O(n log n)` в среднем. Разделение на две половины даёт две сортировки размера `n / 2` и одно
линейное слияние, что асимптотически остаётся `O(n log n)`.

Асимптотика по памяти: `O(n)`. Основной вклад дают временные векторы `left`, `right` и результирующий массив при
слиянии.

Инварианты и критерий корректности:

- после завершения `QuickSort(left, ...)` вектор `left` отсортирован;
- после завершения `QuickSort(right, ...)` вектор `right` отсортирован;
- `MergeSortedVectors(left, right)` корректно строит отсортированное объединение двух уже отсортированных
  последовательностей;
- итоговый `GetOutput()` содержит все элементы исходного массива без потерь и перестановок вне сортировки.

## 4. Детали реализации

Файлы:

- `seq/include/ops_seq.hpp`
- `seq/src/ops_seq.cpp`

Назначение стадий жизненного цикла задачи:

- `ValidationImpl` проверяет, что входной массив не пустой;
- `PreProcessingImpl` копирует вход в выходной буфер, чтобы дальнейшая сортировка не модифицировала исходные данные;
- `RunImpl` делит массив на две половины, сортирует их через `QuickSort` и объединяет результат через
  `MergeSortedVectors`;
- `PostProcessingImpl` дополнительно подтверждает корректность, проверяя `std::is_sorted(...)` для итогового массива.

Обработанные граничные случаи:

- пустой массив отклоняется до запуска алгоритма;
- массив длины `1` проходит через алгоритм корректно;
- для диапазона `low >= high` quicksort немедленно завершается;
- уже отсортированные и обратно отсортированные входы корректно обрабатываются тем же кодом без специальных веток.

Минимальный иллюстративный фрагмент:

```cpp
// File: seq/src/ops_seq.cpp
const auto mid = static_cast<std::vector<int>::difference_type>(data.size() / 2);

std::vector<int> left(data.begin(), data.begin() + mid);
std::vector<int> right(data.begin() + mid, data.end());

QuickSort(left, 0, static_cast<int>(left.size()) - 1);
QuickSort(right, 0, static_cast<int>(right.size()) - 1);

data = MergeSortedVectors(left, right);
```

Этот фрагмент показывает основную идею baseline-версии: массив делится на две части, каждая часть сортируется отдельно,
после чего выполняется одно простое слияние.

## 5. Проверка корректности

Корректность подтверждалась сравнением результата с эталоном `std::ranges::sort(reference)` в функциональных тестах из
`tests/functional/main.cpp`.

Использованные типы тестов:

- минимальные размеры: `OneElement`, `TwoElements`;
- малые и средние размеры: `EightElements`, `RandomSize`, `MediumSize`;
- специальные случаи: `AlreadySorted`, `ReverseSorted`;
- крупные входы: `LargeSize`, `LongSize`, `LongLongSize`.

Характерные примеры:

- вход `[42]` ожидаемо даёт `[42]`;
- вход `[2, 1]` ожидаемо даёт `[1, 2]`;
- вход `127, 126, ..., 1` из сценария `ReverseSorted` ожидаемо даёт `1, 2, ..., 127`;
- для сценариев `LongSize` и `LongLongSize` проверяется корректность на детерминированно сгенерированных массивах длины
  `1000` и `2000`.

Дополнительно `PostProcessingImpl` проверяет, что итоговый массив действительно отсортирован.

## 8. Выводы

Последовательная версия используется как эталон, потому что она реализует ту же самую схему алгоритма, что и остальные
backend-ы: разделение массива, сортировка по Хоару и итоговое слияние. Она полностью детерминирована, покрыта
функциональными тестами и даёт baseline-время `p = 1`, которое затем используется как знаменатель для расчёта `speedup`
и `efficiency` во всех параллельных отчётах.
