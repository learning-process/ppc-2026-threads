## 1. Контекст

TBB-версия выбрана для реализации той же задачи в task-based runtime `oneTBB`, где независимые подзадачи сортировки
могут распределяться между worker-потоками динамически. Для этой задачи TBB удобна тем, что позволяет отдельно
распараллелить верхний уровень алгоритма и внутреннюю обработку диапазонов quicksort без ручного управления очередями
задач.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел по схеме «сортировка Хоара с простым слиянием»: входной массив делится на две
части, каждая часть сортируется отдельно, затем выполняется линейное слияние отсортированных половин. Последовательный
baseline и эталон корректности описаны в [seq/report.md](tasks/tochilin_e_hoar_sort_sim_mer/seq/report.md).

## 3. Базовый алгоритм

Последовательная версия делит массив на `left` и `right`, сортирует обе половины итеративным quicksort по схеме Хоара и
объединяет их через `std::ranges::merge`. TBB-версия сохраняет ту же верхнеуровневую структуру, но использует `oneTBB`
для параллельной обработки двух половин и набора независимых диапазонов внутри quicksort.

## 4. Схема распараллеливания

В реализации используются два примитива oneTBB:

- `tbb::parallel_invoke` - для одновременного запуска сортировки `left` и `right`;
- `tbb::parallel_for` - для обработки текущего набора диапазонов quicksort внутри `QuickSortParallel(...)`.

Как определяется диапазон работы:

- на верхнем уровне массив делится на две половины;
- внутри `QuickSortParallel(...)` поддерживается вектор `current_ranges`, содержащий независимые поддиапазоны `(l, r)`;
- на каждой итерации `parallel_for` обрабатывает диапазон индексов `0 .. current_ranges.size()` и передаёт каждому
  worker-у часть этих элементов через `tbb::blocked_range<std::size_t>`.

Выбор grainsize:

- `blocked_range<std::size_t>(0, current_ranges.size())` создаётся без явного третьего аргумента;
- следовательно, grainsize выбирается runtime-ом oneTBB автоматически;
- для этой версии основной контроль крупности работы перенесён не в grainsize, а в `serial_cutoff`: если поддиапазон
  меньше порога, он досортировывается последовательно и не дробится дальше.

Partitioner:

- явный `partitioner` в коде не задаётся;
- используется поведение oneTBB по умолчанию для `parallel_for`.

Ограничение конкуренции:

- число worker-потоков ограничивается через `tbb::global_control::max_allowed_parallelism`;
- значение берётся из `ppc::util::GetNumThreads()`, поэтому runtime oneTBB подчиняется конфигурации `PPC_NUM_THREADS`;
- это именно ограничение максимально допустимого параллелизма runtime, а не просто локальная переменная с числом
  потоков.

Минимальный иллюстративный фрагмент:

```cpp
tbb::parallel_for(tbb::blocked_range<std::size_t>(0, current_ranges.size()),
                  [&](const tbb::blocked_range<std::size_t> &range) {
  auto &local_next = next_ranges_local.local();
  for (std::size_t idx = range.begin(); idx != range.end(); ++idx) {
    ProcessRange(arr, current_ranges[idx], serial_cutoff, local_next);
  }
});
```

Этот фрагмент показывает основную TBB-идею реализации: на каждой волне quicksort набор независимых диапазонов
обрабатывается как bulk-parallel работа над `blocked_range`, а новые диапазоны собираются в thread-local буферы.

## 5. Детали реализации

Файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

Структура реализации:

- `QuickSortSequential(...)` сортирует мелкие диапазоны итеративно без рекурсии;
- `ProcessRange(...)` либо дробит текущий диапазон дальше, либо переводит его в последовательную обработку;
- `QuickSortParallel(...)` хранит фронт текущих диапазонов в `current_ranges` и на каждой итерации параллельно
  обрабатывает его через `parallel_for`;
- `RunImpl()` ограничивает конкуренцию через `global_control`, запускает две верхнеуровневые задачи через
  `parallel_invoke` и затем выполняет одно линейное слияние.

Локальные аккумуляторы и память:

- для сбора новых диапазонов используется `tbb::enumerable_thread_specific<std::vector<std::pair<int, int>>>`, что
  позволяет каждому worker-у записывать результаты в собственный локальный буфер;
- общий вектор `next_ranges` формируется только после завершения `parallel_for`, когда локальные буферы последовательно
  объединяются;
- ложное разделение кэша специально не оптимизировалось отдельным allocator, но запись в `next_ranges_local.local()` уже
  уменьшает конфликтную работу между потоками.

## 6. Проверка корректности

Корректность TBB-версии проверялась тем же функциональным набором, что и у baseline: результат сравнивался с эталонной
сортировкой `std::ranges::sort(reference)` в `tests/functional/main.cpp`.

Использованные сценарии:

- минимальные размеры: `OneElement`, `TwoElements`;
- малые и средние размеры: `EightElements`, `RandomSize`, `MediumSize`;
- специальные случаи: `AlreadySorted`, `ReverseSorted`;
- крупные входы: `LargeSize`, `LongSize`, `LongLongSize`.

Характерные примеры:

- `OneElement`: вход `[42]` должен остаться `[42]`;
- `AlreadySorted`: массив `0, 1, 2, ..., 127` должен сохраниться без изменений;
- `LongLongSize`: на размере `2000` проверяется корректность параллельной обработки большого числа диапазонов quicksort.

Дополнительно `PostProcessingImpl()` подтверждает корректность через `std::ranges::is_sorted(GetOutput())`.

По результатам видно, что TBB лучше балансирует большое число независимых диапазонов, чем фиксированное грубое разбиение

каждой волны `parallel_for`, однако на крупных входах этот overhead окупается лучше, чем в baseline.

## 9. Выводы

TBB особенно удобна для этой задачи там, где quicksort естественно порождает много независимых поддиапазонов и их можно
обрабатывать как набор мелких задач. На больших входах версия показывает лучший результат среди уже реализованных
thread-backend-ов, потому что сочетает ограничение конкуренции через `global_control` и динамическое распределение
диапазонов через runtime oneTBB. Накладные расходы начинают мешать на мелких входах, где стоимость управления задачами
сравнима с полезной работой, но на основном размере `N = 2_000_000` этот backend оказался очень эффективным.
