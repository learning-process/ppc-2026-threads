## 1. Введение

В данной работе реализованы и исследованы пять версий алгоритма сортировки Шелла
с простым слиянием: последовательная (SEQ) и четыре параллельных — с использованием
OpenMP, Intel OneTBB, `std::thread` и гибридной схемы MPI + OpenMP.

---

## 2. Единая постановка задачи

Требуется отсортировать вектор целых чисел произвольного размера по неубыванию.

Условия корректности для всех реализаций:

1. Результат совпадает с `std::ranges::sort`.
2. Алгоритм корректно обрабатывает массивы, размер которых не кратен числу потоков.
3. Нет утечек памяти, гонок данных и взаимных блокировок.

---

## 3. Методология эксперимента

### Аппаратно-программная среда

- CPU: AMD Ryzen 5 5600
- RAM: 48 GB
- OS: Windows 10
- Компилятор: clang++21, сборка Release

### Параметры тестирования

- Массив из 2 000 000 псевдослучайных целых чисел.
- Медиана по 5 запускам (`--gtest_repeat=5`).
- **Speedup** = время SEQ / время параллельной версии.
- **Efficiency** = speedup / workers.
- Для ALL: workers = ranks × threads.

### Команды сборки

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
cd build/bin
```

## 6. Интерпретация результатов

**TBB и STL** показали практически одинаковые результаты и лучшее ускорение среди всех
технологий (~4.35–4.39x при 8 потоках). При равномерном делении массива work-stealing TBB
не даёт дополнительного преимущества перед статическим распределением `std::thread`.

**OMP** немного уступает TBB и STL (3.75x против 4.35–4.39x). Возможно, это связано
со спецификой реализации неявных барьеров в данной версии clang++.

**ALL (MPI + OMP)** показал ускорение 2.80x на 4 воркерах — меньше, чем чистые
SMP-реализации. Причина — накладные расходы на `MPI_Scatterv` и `MPI_Gatherv`: на одной
машине передача данных через MPI-буферы добавляет затраты без выигрыша от распределённой
памяти.

**Снижение эффективности** с ростом числа потоков — чем больше блоков, тем больше итераций у последовательного слияния.

---

## 7. Заключение

Все пять реализаций работают корректно и успешно прошли тестирование. Параллельные версии
обеспечили ускорение более чем в 4 раза. Для задач,
запускаемых на одной многоядерной машине, наилучший результат показали TBB и STL.
Применение MPI целесообразно только при запуске на нескольких физических узлах.
