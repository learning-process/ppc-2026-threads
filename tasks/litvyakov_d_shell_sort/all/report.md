## 1. Введение

В данном отчёте рассматривается гибридная реализация алгоритма, совмещающая MPI для
межпроцессного взаимодействия и OpenMP для параллелизма внутри каждого процесса.

---

## 2. Постановка задачи

Требуется корректно распределить данные между изолированными процессами MPI, выполнить
локальную сортировку на каждом узле с помощью OpenMP и собрать результат обратно на
корневом процессе.

---

## 3. Архитектура распределённых вычислений (уровень MPI)

Алгоритм состоит из нескольких этапов:

1. **Broadcast**: корневой процесс (`rank 0`) рассылает размер массива всем процессам
   через `MPI_Bcast`.
2. **Scatter**: данные распределяются по процессам через `MPI_Scatterv` с учётом
   неравномерного деления.
3. **Локальная обработка**: каждый процесс сортирует свой буфер `local_vec` — внутри
   используется OpenMP.
4. **Gather**: отсортированные фрагменты собираются на корневом узле через `MPI_Gatherv`.
5. **Финальное слияние**: корневой процесс последовательно применяет `std::inplace_merge`.
6. **Broadcast результата**: для прохождения assert-проверок результат рассылается обратно
   через `MPI_Bcast`.

---

## 4. Архитектура общей памяти (уровень OpenMP)

Внутри каждого процесса полученный буфер дополнительно делится между потоками. Используется
та же директива `#pragma omp parallel for`, что и в OMP-версии.

---

## 5. Синхронизация

- **Уровень MPI**: синхронизация обеспечивается барьерами операций
  (`MPI_Scatterv`, `MPI_Gatherv`).
- **Уровень OpenMP**: гонок данных нет, так как потоки работают с непересекающимися
  диапазонами.

---

## 6. Детали реализации

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

В отличие от систем с общей памятью, MPI требует физического копирования данных между
буферами, что создаёт дополнительные накладные расходы на коммуникацию.

---

## 7. Проверка корректности

Функциональные тесты прошли успешно на всех процессах. Для этого финальный результат
рассылается через `MPI_Bcast` на все потоки перед assert-проверками.

## 9. Выводы

Гибридная реализация работает корректно. Однако для задач с большим объёмом передаваемых
данных MPI на одном физическом хосте проигрывает чистым SMP-решениям из-за затрат на
коммуникацию. Реальный выигрыш от MPI проявляется при запуске на нескольких физических
машинах.

## Приложение

```cpp
  MPI_Scatterv(world_rank == 0 ? vec.data() : nullptr,
               sendcounts.data(), displs.data(), MPI_INT,
               local_vec.data(), sendcounts[world_rank],
               MPI_INT, 0, MPI_COMM_WORLD);

  ShellSortMerge(local_vec); // OpenMP

  MPI_Gatherv(local_vec.data(), sendcounts[world_rank], MPI_INT,
              world_rank == 0 ? vec.data() : nullptr,
              sendcounts.data(), displs.data(),
              MPI_INT, 0, MPI_COMM_WORLD);

  if (world_rank == 0) {
    for (int i = 1; i < world_size; ++i) {
      if (sendcounts[i] > 0) {
        std::inplace_merge(
            vec.begin(),
            vec.begin() + static_cast<std::ptrdiff_t>(displs[i]),
            vec.begin() + static_cast<std::ptrdiff_t>(displs[i] + sendcounts[i]));
      }
    }
  }
```

Scatter рассылает данные, каждый процесс сортирует свой блок, Gather собирает результат.
Финальное слияние выполняется только на rank 0.
