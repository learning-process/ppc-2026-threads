## 1. Введение

В данном отчёте рассматривается реализация алгоритма сортировки Шелла с использованием
OpenMP. Цель — ускорить последовательную версию за счёт параллельной сортировки независимых
фрагментов массива на многоядерном процессоре.

---

## 2. Постановка задачи

Требуется адаптировать последовательный алгоритм для многопоточной среды.

### Условия корректности

- Итоговый вектор упорядочен по неубыванию.
- Алгоритм корректно работает, если размер массива не кратен числу потоков.
- Результат совпадает с SEQ-версией.

---

## 3. Архитектура параллельного решения

Алгоритм состоит из двух фаз:

1. **Параллельная сортировка**: массив делится на N блоков по числу потоков.
   Каждый поток сортирует свой блок с помощью `BaseShellSort` независимо.
   Границы блоков вычисляются функцией `GetBounds`, которая равномерно распределяет остаток.

2. **Последовательное слияние**: после сортировки блоки сливаются с помощью
   `std::inplace_merge`. Эта фаза выполняется в одном потоке, так как зависимость
   по данным не даёт её распараллелить.

---

## 4. Схема распараллеливания

### Директива и модель данных

```cpp
#pragma omp parallel for default(none) \
    shared(vec, bounds, parts_count_t) schedule(static)
```

- **schedule(static)** — блоки распределяются равномерно, что оправдано при одинаковом
  размере сегментов.
- **shared**: вектор `vec` и массив границ `bounds` доступны всем потокам на чтение;
  запись в `vec` не конфликтует, так как диапазоны `[l, r)` у потоков не пересекаются.
- **private**: переменные `l` и `r` локальны для каждого потока.

### Синхронизация

Гонок данных нет — каждый поток пишет в свой независимый диапазон. Неявный барьер в конце
`parallel for` гарантирует, что все блоки отсортированы до начала слияния.

## 5. Детали реализации

Файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Инициализация аналогична SEQ. В `RunImpl` используется статическое планирование
OpenMP. Использование `std::inplace_merge` вместо временных копий снижает
потребление памяти, но переводит фазу слияния в однопоточный режим,
что ограничивает ускорение прибольшом числе потоков.

## 8. Результаты

Тестовые данные: массив из 2 000 000 псевдослучайных целых чисел.
Базовое время (SEQ): 0.059871 с.

| workers | time (s) | speedup | efficiency |
|---------|----------|---------|------------|
| 2       | 0.044162 | 1.36    | 0.68       |
| 4       | 0.023395 | 2.56    | 0.64       |
| 8       | 0.015960 | 3.75    | 0.47       |

Ускорение растёт до 3.75x при 8 потоках. Эффективность постепенно снижается: с ростом числа
потоков увеличивается количество блоков, а значит — и объём работы на фазе слияния, которая
остаётся однопоточной.

## Приложение

```cpp
#pragma omp parallel for default(none) \
    shared(vec, bounds, parts_count_t) schedule(static)
  for (int i = 0; i < parts_count_t; ++i) {
    const std::size_t l = bounds[i];
    const std::size_t r = bounds[i + 1];
    BaseShellSort(vec.begin() + static_cast<std::ptrdiff_t>(l),
                  vec.begin() + static_cast<std::ptrdiff_t>(r));
  }

  for (std::size_t i = 1; i < parts_count; ++i) {
    std::inplace_merge(
        vec.begin(),
        vec.begin() + static_cast<std::ptrdiff_t>(bounds[i]),
        vec.begin() + static_cast<std::ptrdiff_t>(bounds[i + 1]));
  }
```

Первый цикл — параллельная сортировка блоков. Второй цикл — последовательное слияние
в главном потоке.
