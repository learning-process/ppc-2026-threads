**Вариант:** 21.

## 1. Введение

Гибридная реализация совмещает два уровня параллелизма: MPI делит граф между процессами по вершинам, OpenMP параллелит
вычисления внутри каждого процесса. Это позволяет масштабировать алгоритм как по числу узлов кластера, так и по числу
ядер на каждом узле.

Цель — получить ускорение сверх возможностей однопроцессных параллельных реализаций (OMP, TBB, STL) за счёт
распределения данных между несколькими MPI-процессами при сохранении корректности алгоритма Дейкстры.

## 2. Постановка задачи

Задача и входные параметры идентичны SEQ-варианту: дан направленный взвешенный граф $G = (V, E)$ с неотрицательными
весами рёбер, задана вершина-источник $s$. Требуется вернуть сумму кратчайших расстояний от $s$ до всех достижимых
вершин.

Входные параметры (структура `GraphDescriptor`):

- Количество вершин $n$ (`int`), $1 \\le n \\le 10000$.
- Индекс источника `source` (`int`).
- Список рёбер `arcs` (вектор структур `Arc { NodeId from; NodeId to; Cost weight; }`).

Выходные данные (`OutType`): скалярное `std::int64_t` — сумма конечных кратчайших расстояний.

Ограничения: веса неотрицательны; недостижимые вершины (расстояние `kInf`) исключаются из суммы; переполнение при
накоплении суммы контролируется.

## 3. Последовательный алгоритм

За основу взят тот же алгоритм $O(n^2)$, что и в SEQ: линейный поиск непосещённой вершины с минимальным расстоянием
(`FindNextNode`) и последовательная релаксация смежных рёбер. Квадратичная форма выбрана намеренно — она легко
декомпозируется между процессами по вертикальным полосам вершин, тогда как heap-версия требует глобальной общей очереди
и не поддаётся простому распределению.

## 4. Схема распараллеливания

### Распределение данных

Граф реплицируется на каждый процесс: rank 0 рассылает заголовок `[n, source, arc_count]` и плоский буфер рёбер через
два вызова `MPI_Bcast`. Каждый процесс строит полный `graph_` локально из полученного буфера. Такой подход устраняет
необходимость передавать данные графа на каждой итерации.

Вершины делятся между процессами на смежные диапазоны:

$$
\\text{local_start} = r \\cdot \\lfloor n/P \\rfloor + \\min(r, n \\bmod P), \\quad \\text{local_count} = \\lfloor n/P
\\rfloor + [r < n \\bmod P]
$$

где $r$ — номер процесса, $P$ — их общее число. Остаток распределяется по одной вершине на первые $r$ процессов, что
обеспечивает балансировку при любых $n$ и $P$.

Каждый процесс хранит только `local_dist_` и `local_visited_` для своего диапазона.

### Итерация алгоритма

Каждый из $n$ шагов выполняется в три фазы:

**Фаза 1 — локальный поиск минимума (`FindLocalMin`):** OpenMP-параллельный скан `local_dist_` по своему диапазону
вершин. Каждый поток вычисляет собственный минимум, после чего итоговый минимум процесса защищен и обновляется внутри
критической секции `#pragma omp critical`. Результат — структура `NodePayload {cost, vtx}` с глобальным индексом
вершины.

**Фаза 2 — глобальная редукция (`MPI_Allreduce`):** Все процессы передают свой локальный минимум; стандартная операция
`MPI_MINLOC` со встроенным типом `MPI_2INT` выбирает глобальный pivot. Это изящное стандартное решение избавляет от
необходимости писать кастомные операции редукции через `MPI_Op_create`. После вызова pivot известен всем процессам
одновременно — без дополнительного `MPI_Bcast`.

**Фаза 3 — релаксация рёбер (`RelaxNeighbors`):** Каждый процесс параллельно (через `#pragma omp parallel for`) обходит
рёбра pivot из своей копии `graph_`. Обновляются только записи `local_dist_`, попадающие в локальный диапазон вершин
данного процесса. Запись выполняется через `std::atomic_ref<int>` c CAS-циклом — без использования мьютексов.

Поскольку граф реплицирован и фаза 3 выполняется одинаково на всех процессах (каждый обновляет только свой диапазон),
`local_dist_` остаётся консистентным без дополнительной синхронизации после каждой итерации.

```bash
PreProcessing:
  rank 0 → MPI_Bcast([n, source, arc_count])
  rank 0 → MPI_Bcast(arc_buf[arc_count × 3])
  все → строят graph_; делят [0, n) на диапазоны

RunImpl (n итераций):
  local = FindLocalMin(local_dist_, local_visited_)  // OMP parallel + critical
  MPI_Allreduce(&local, &global, 1, MPI_2INT, MPI_MINLOC)
  если global.vtx в моём диапазоне → local_visited_[...] = 1
  RelaxNeighbors(graph_[global.vtx], ...)            // OMP parallel for + CAS

PostProcessing:
  CalculateLocalSum(local_dist_)                     // OMP reduction
  MPI_Reduce(&local_sum, &global_sum, MPI_SUM, rank 0)
  rank 0 → GetOutput() = global_sum
  MPI_Bcast(global_sum) → не-root процессы
```

### Корректность консистентности

Граф реплицирован — все процессы видят одинаковые рёбра pivot. Каждый процесс обновляет только вершины своего диапазона.
После `MPI_Allreduce` pivot одновременно известен всем, поэтому состояние `local_dist_` между процессами согласовано без
явной синхронизации расстояний.

## 5. Детали реализации

Ключевые компоненты:

- `FindLocalMin`: параллельный поиск минимума в `local_dist_` с OMP. Тай-брейкинг — по меньшему глобальному индексу
  вершины, что гарантирует детерминированный выбор pivot при равных расстояниях.
- `UpdateNeighborDistance`: проверяет принадлежность целевой вершины локальному диапазону и выполняет CAS-обновление
  через `std::atomic_ref<int>`. Вызывается из `RelaxNeighbors` для каждого ребра.
- `CalculateLocalSum`: OMP `reduction(+ : local_sum)` по `local_dist_` для финального суммирования.
- `ValidationImpl`: полная проверка входных данных выполняется только на rank 0; результат рассылается через
  `MPI_Bcast(&is_valid, 1, MPI_INT, 0, ...)`. Остальные процессы на этом этапе ещё не имеют данных и возвращают
  полученное значение.
- `PostProcessingImpl`: `MPI_Bcast` итогового результата на не-root процессы — требование фреймворка, который вызывает
  `GetOutput()` у всех процессов.

Обработка крайних случаев:

- Источник попадает в диапазон конкретного процесса: `local_dist_[source_ - local_start_] = 0` устанавливается только
  тем процессом, чей диапазон содержит `source_`.
- Все вершины недостижимы: `MPI_Allreduce` возвращает `global.vtx == -1`, цикл прерывается через флаг `all_done`.
- Число процессов больше числа вершин: `local_count_` для лишних процессов равен 0, они участвуют в `MPI_Allreduce` с
  `cost = kInf`, не влияя на результат.

### Тестовые данные

**Тестирование корректности** основано на детерминированных синтетических графах малой размерности с аналитически
предсказуемым результатом суммы кратчайших путей. Используются следующие топологии:

1. Топология «Звезда» с источником в центре. Формируется функцией `MakeStarFromZero(n)`. Из центральной вершины $0$
   исходят рёбра ко всем остальным вершинам $i \\in [1, n-1]$. Вес каждого ребра равен индексу целевой вершины ($w(0, i)
   = i$).
1. Топология «Треугольник с обходным путем». Формируется функцией `MakeTriangleShortcutGraph(n)` и состоит из 3 вершин.
   Содержит три ребра: $(0 \\to 1, w=1)$, $(1 \\to 2, w=1)$ и прямой длинный путь $(0 \\to 2, w=5)$. Ожидаемый
   результат: 3 (путь $0 \\to 1 \\to 2$ имеет суммарную длину $1 + 1 = 2$, а путь $0 \\to 1$ равен $1$). Тест проверяет
   механизм релаксации рёбер и контролирует, что алгоритм успешно обновляет расстояние до вершины $2$ при нахождении
   более короткого составного пути, игнорируя прямой путь с большим весом.

**Тестирование производительности** основано на распределённом случайном разреженном графе, генерируемом функцией
`MakeRandomSparseGraph` со следующими параметрами:

- Количество вершин (`kGraphSize`): 10 000.
- Плотность (исходящая степень каждой вершины `kDegree`): 20 рёбер.
- Зерно генератора случайных чисел (`kSeed`): 42.
- Общее число рёбер в тесте: 200 000.

Эталонная сумма кратчайших расстояний вычисляется однократно последовательным алгоритмом на этапе инициализации и затем
используется для проверки результатов.

## 7. Результаты и анализ

### 7.1. Корректность

Все четыре функциональных теста (`star_n2`, `star_n4`, `star_n6`, `triangle_shortcut`) проходят при любом числе
MPI-процессов и OMP-потоков. Это подтверждает корректность распределения вершин, консистентность локальных расстояний и
корректность финальной редукции через `MPI_Reduce`.

### 7.2. Производительность

Базовое время последовательной версии SEQ (`task_run`): **0.343 с** (среднее по локальным прогонам без параллельных
директив).

Конфигурация запуска определяется как $P \\times T$, где $P$ — количество MPI-процессов, а $T$ — количество потоков
OpenMP внутри каждого процесса. Суммарное число задействованных вычислительных единиц $W = P \\cdot T$. Ускорение
вычисляется как $S = T\_{\\text{seq}} / T\_{\\text{par}}$, эффективность как $E = S / W$.

| Конфигурация $P \\times T$ | Всего воркеров ($W$) | Время, с | Ускорение ($S$) | Эффективность ($E$) | |
------------------------- | -------------------- | -------- | --------------- | ------------------- | | 1 × 1 | 1 |
0.1549 | 2.2× | 220% | | 1 × 2 | 2 | 0.4073 | 0.84× | 42% | | 1 × 4 | 4 | 0.0664 | 5.2× | 130% | | 2 × 2 | 4 | 0.0778 |
4.4× | 110% | | 4 × 1 | 4 | 0.2238 | 1.5× | 38% |

> **Замечание по эффективности > 100%:** Экстремально высокие показатели эффективности для конфигураций `1 × 1`, `1 × 4`
> и `2 × 2` вызваны тем, что базовая последовательная реализация лишена глубоких низкоуровневых оптимизаций компилятора,
> которые автоматически применяются к коду при обработке директив OpenMP (включая векторизацию и оптимизацию циклов).

### Анализ аномалий и масштабируемости

**Конфигурация 1 × 2 — Аномалия просадки:** Два потока OpenMP на одном процессе показывают явную деградацию относительно
однопоточного режима. При малом числе локальных вершин на один поток ($n/T = 5000$) накладные расходы на постоянный
вход/выход из параллельных областей и барьеры внутри критической секции `#pragma omp critical` в `FindLocalMin`
полностью перекрывают выигрыш от разделения работы. При увеличении количества потоков до 4 (`1 × 4`) объем полезной
работы на ядро возрастает, что окупает оверхед и выводит конфигурацию в абсолютные лидеры по времени (**0.0664 с**).

**Конфигурация 4 × 1 — Аномалия горизонтального масштабирования:** Распределение вычислений на 4 MPI-процесса с 1
потоком демонстрирует низкое ускорение (всего 1.5× при 4 воркерах). Причины кроются в природе самого алгоритма Дейкстры:

1. Алгоритм строго итеративен и выполняет ровно $n = 10000$ шагов. На каждом шаге происходит коллективная синхронизация
   `MPI_Allreduce`. На локальной машине накладные расходы ОС на межпроцессное взаимодействие (IPC) суммируются 10 000
   раз.
1. При разбиении на 4 процесса каждый из них обрабатывает всего 2500 вершин. Локальный скан и релаксация происходят
   мгновенно, из-за чего коммуникационный оверхед сети начинает доминировать над полезными вычислениями.

**Конфигурация 2 × 2 — Оптимальный гибридный баланс:** Демонстрирует качественную работу гибридной схемы. За счет
сокращения числа MPI-процессов до 2 накладные расходы на `MPI_Allreduce` снижаются вдвое по сравнению с конфигурацией
`4 × 1`, а подключение 2 потоков OpenMP внутри каждого процесса позволяет эффективно утилизировать все 4 доступных
вычислительных ядра процессора, достигая ускорения **4.4×**.

## 8. Заключение

Гибридная ALL-реализация алгоритма Дейкстры полностью корректна, но имеет выраженную специфику масштабирования. Из-за
жесткой итерационной структуры алгоритма, требующей $n = 10000$ последовательных вызовов `MPI_Allreduce` для поиска
глобального минимума, чистый межпроцессный параллелизм (`4 × 1`) малоэффективен на графах средней размерности.

Наилучшую производительность демонстрируют конфигурации с упором на внутрипроцессный параллелизм (`1 × 4` и гибридный
`2 × 2`), где накладные расходы на синхронизацию потоков через общую память значительно ниже латентности межпроцессного
взаимодействия. Для эффективного горизонтального масштабирования на множество MPI-узлов реального кластера требуется
существенное увеличение размерности графа ($n \\gg 10000$), при котором вычислительная сложность релаксации ребер начнет
превалировать над задержками сетевых коммуникаций.
