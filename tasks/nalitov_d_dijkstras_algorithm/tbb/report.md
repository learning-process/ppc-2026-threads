## 1. Введение

Внутрипроцессная параллельная реализация алгоритма Дейкстры на базе библиотеки **oneAPI TBB** использует парадигму
параллелизма задач (Task-based Parallelism). В отличие от классического жесткого разделения итераций между потоками, TBB
оперирует концепцией диапазонов (`tbb::blocked_range`) и рекурсивного расщепления задач, автоматически балансируя
нагрузку между ядрами процессора с помощью планировщика типа Work-stealing (кража задач).

Данный подход призван повысить эффективность утилизации процессорного времени на гетерогенных нагрузках и разреженных
структурах за счет динамического перераспределения подзадач в рамках единого адресного пространства (Shared Memory).

## 2. Постановка задачи

Входные параметры, ограничения и цель вычислений полностью соответствуют спецификации интерфейса `BaseTask`:

- **Вход:** Дескриптор направленного графа $G = (V, E)$ с неотрицательными весами рёбер и выделенная вершина-источник
  $s$. Число вершин $n \\le 10000$.
- **Выход (`OutType`):** Скалярный `std::int64_t` — сумма кратчайших путей от источника до всех достижимых вершин графа.

Алгоритмическая сложность параллельной структуры спроектирована как $O(n^2)$.

## 3. Последовательный алгоритм

Последовательная базовая логика опирается на итерационный выбор вершины с минимальным расстоянием (`kInf` для
недостижимых) среди подмножества еще не посещенных вершин, ее фиксацию (`visited = 1`) и последующую релаксацию смежных
с ней путей. Классический последовательный подход на массивах совершает $n$ шагов внешнего цикла, выполняя полный
просмотр векторов на каждой итерации.

## 4. Схема распараллеливания

Параллельный алгоритм TBB декомпозирует итерационные блоки на три изолированные фазы, управляемые через generic-шаблоны
библиотеки:

1. **Параллельная редукция для поиска минимума (`SelectBestVertex`):** Нахождение глобально лучшей вершины на каждой
   итерации реализуется через `tbb::parallel_reduce`. Функция делит интервал вершин `[0, n)` на поддиапазоны
   `tbb::blocked_range<int>`. Каждый поток вычисляет локальный минимум (`FindLocalBestVertex`), после чего TBB
   производит их иерархическое слияние с помощью метода `Merge` структуры `BestVertex` (с сохранением детерминизма по ID
   вершины).
1. **Параллельное распределение релаксации (`RelaxNeighbors`):** Обход смежных ребер зафиксированной вершины-лидера
   выполняется через `tbb::parallel_for`. Вектор соседей `neighbors` оборачивается в `tbb::blocked_range<std::size_t>`,
   распределяя релаксацию путей между свободными воркерами планировщика TBB.
1. **Атомарное обновление без взаимных блокировок:** Для предотвращения состояния гонки (Data Race) при одновременной
   релаксации смежных вершин из разных подзадач, вектор расстояний построен на базе потокобезопасного типа
   `std::vector<std::atomic<Cost>>`. Обновление данных выполняется внутри функции `RelaxEdge` через неблокирующий цикл
   Lock-free CAS (`compare_exchange_weak`) с расслабленной моделью консистентности `std::memory_order_relaxed`.

```bash
PreProcessing:
  Параллельное заполнение списков смежности через tbb::parallel_for
  с использованием атомарного row_next[from].fetch_add для резервирования мест.

RunImpl (Внешний последовательный цикл на n итераций):
  pivot = tbb::parallel_reduce(...) -> Нахождение оптимальной вершины
  Если pivot.vtx == -1 или cost == kInf → выход (все компоненты обойдены)
  visited_[pivot.vtx] = 1
  tbb::parallel_for(...)            -> Распределенная релаксация ребер через CAS

PostProcessing:
  tbb::parallel_reduce(...)         -> Агрегация финальной суммы длин путей std::plus<>()

```

## 5. Детали реализации

- **`PreProcessingImpl` (Параллельный препроцессинг):** Вместо последовательного построения графа, сначала за один
  проход вычисляются исходящие степени вершин. Далее, используя `tbb::parallel_for`, ребра безопасно и параллельно
  раскладываются по динамическим векторам `graph_`. Потоки конкурируют за ячейки памяти через атомарный инкремент
  индекса `fetch_add`, что полностью ликвидирует необходимость блокировок структуры графа.
- **Контейнер `std::atomic`:** Использование `std::vector<std::atomic<Cost>>` напрямую в TBB-версии заменяет логику
  `std::atomic_ref`, применявшуюся в OpenMP, гарантируя нативную кроссплатформенную потокобезопасность на уровне
  элементов массива при чтении и записи весов.
- **`SumDistances`:** Финальный расчет результата переведен на параллельные рельсы посредством `tbb::parallel_reduce` с
  передачей стандартного бинарного функтора сложения `std::plus<>()` для результирующих порций данных.

### Тестовые данные

**Тестирование корректности** основано на детерминированных синтетических графах малой размерности с аналитически
предсказуемым результатом суммы кратчайших путей. Используются следующие топологии:

1. Топология «Звезда» с источником в центре. Формируется функцией `MakeStarFromZero(n)`. Из центральной вершины $0$
   исходят рёбра ко всем остальным вершинам $i \\in [1, n-1]$. Вес каждого ребра равен индексу целевой вершины ($w(0, i)
   = i$).
1. Топология «Треугольник с обходным путем». Формируется функцией `MakeTriangleShortcutGraph(n)` и состоит из 3 вершин.
   Содержит три ребра: $(0 \\to 1, w=1)$, $(1 \\to 2, w=1)$ и прямой длинный путь $(0 \\to 2, w=5)$. Ожидаемый
   результат: 3 (путь $0 \\to 1 \\to 2$ имеет суммарную длину $1 + 1 = 2$, а путь $0 \\to 1$ равен $1$). Тест проверяет
   механизм релаксации рёбер и контролирует, что алгоритм успешно обновляет расстояние до вершины $2$ при нахождении
   более короткого составного пути, игнорируя прямой путь с большим весом.

**Тестирование производительности** основано на распределённом случайном разреженном графе, генерируемом функцией
`MakeRandomSparseGraph` со следующими параметрами:

- Количество вершин (`kGraphSize`): 10 000.
- Плотность (исходящая степень каждой вершины `kDegree`): 20 рёбер.
- Зерно генератора случайных чисел (`kSeed`): 42.
- Общее число рёбер в тесте: 200 000.

Эталонная сумма кратчайших расстояний вычисляется однократно последовательным алгоритмом на этапе инициализации и затем
используется для проверки результатов.

## 7. Результаты и анализ

### 7.1. Корректность

Программа успешно выдержала все функциональные тесты. Использование `std::atomic` исключило порчу данных при
параллельной релаксации рёбер. Логика разрешения коллизий в предикате `IsBetterThan` гарантирует строго
детерминированный выбор ведущей вершины во время параллельного редукционного спуска, независимо от характера разбиения
диапазонов планировщиком TBB.

### 7.2. Производительность

Сравнение показателей выполнения функции `task_run` для последовательного кода и различных моделей параллелизма на

### Анализ масштабируемости и эффективности TBB

Реализация продемонстрировала высокую общую производительность, выполнив расчёт за 0.0516 секунды, что соответствует
ускорению в 6.85 раз по сравнению с классическим последовательным кодом.

1. **Преимущество над STL:** Решение на базе TBB существенно обошло встроенный параллелизм `std::execution::par` из STL
   (0.1080 с). Это связано с тем, что `tbb::parallel_reduce` и `tbb::parallel_for` работают поверх оптимизированного
   пула потоков с автоматическим размером зерна (grain size). Архитектура TBB минимизирует накладные расходы на создание
   подзадач, тогда как стандартная библиотека STL порождает избыточный оверхед на частых синхронизациях мелких
   диапазонов во внутреннем цикле Дейкстры.
1. **Сравнение с OpenMP:** TBB уступил решению на OpenMP (0.0341 с на 4 потоках). Разница объясняется спецификой
   планирования:

- В OpenMP использовалось жесткое статическое распределение циклов (`schedule(static)`), которое идеально ложится на
  регулярные плотные структуры данных и имеет практически нулевой оверхед во время итераций.
- TBB использует динамический планировщик и механизм _Work-stealing_. На каждой из 10 000 итераций внешнего цикла
  алгоритма Дейкстры вызов `parallel_reduce` и `parallel_for` заново инициирует логику разбиения задач в очереди
  воркеров. Для графа скромного размера ($10,000$ элементов) затраты планировщика TBB на управление задачами и
  балансировку нагрузки оказываются чуть выше, чем явные легковесные директивы компилятора OpenMP.

Разработанная программа на базе oneAPI TBB успешно решает задачу поиска кратчайших путей, демонстрируя высокую степень
масштабируемости (ускорение ×6.85). Применение Task-based подхода доказало свою высокую эффективность по сравнению со
стандартными параллельными алгоритмами STL. Реализация подтверждает, что комбинация алгоритмов редукции TBB с атомарными

многоядерных архитектурах.
