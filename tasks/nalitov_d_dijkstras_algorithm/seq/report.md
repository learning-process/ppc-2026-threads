## 1. Введение

Алгоритм Дейкстры — фундаментальный метод теории графов для нахождения кратчайших путей от одной вершины-источника до
всех остальных верифицируемых вершин во взвешенных графах с неотрицательными весами рёбер. Имеет широкое практическое
применение в сетевой маршрутизации, логистике и геоинформационных системах.

Цель работы — построение стабильной, оптимизированной последовательной реализации алгоритма Дейкстры. Относительно
времени выполнения последовательной реализации производится расчёт метрик ускорения и эффективности параллельных версий.

## 2. Постановка задачи

Задача формулируется следующим образом: дан направленный взвешенный граф $G = (V, E)$, где $V$ — множество вершин, $E$ —
множество ориентированных рёбер. Каждому ребру $(u, v) \\in E$ сопоставлен вес $w(u, v) \\ge 0$. Требуется найти
кратчайшие расстояния от заданной стартовой вершины $s \\in V$ до всех вершин $v \\in V$, достижимых из $s$.

Входные параметры (структура `GraphDescriptor`):

- Количество вершин $n$ (`int`), где $1 \\le n \\le 10000$.
- Индекс исходной вершины-источника `source` (`int`).
- Динамический список рёбер `arcs` (вектор структур `Arc { NodeId from; NodeId to; Cost weight; }`).

Выходные данные (`OutType`):

- Скалярное значение типа `std::int64_t`, представляющее собой арифметическую сумму длин всех кратчайших путей от
  источника до всех достижимых вершин графа.

Ограничения:

- Веса рёбер строго неотрицательны ($w \\ge 0$).
- Если вершина недостижима из источника, её расстояние остаётся равным маркеру бесконечности (`kInf`), и она исключается
  из финальной агрегации.
- Реализована строгая проверка на переполнение стандартных типов данных при сложении в процессе релаксации и при
  накоплении итоговой суммы путей.

## 3. Последовательный алгоритм

Последовательная версия использует классический алгоритм Дейкстры с квадратичной временной сложностью $O(|V|^2 + |E|)$.
Данный подход выбран намеренно для обеспечения единой вычислительной схемы с параллельными реализациями (OMP, TBB, STL),
что позволяет корректно и прозрачно измерять чистый фактор ускорения, достигаемый за счёт параллелизма, а не за счёт
изменения алгоритмической структуры.

Пошаговое описание алгоритма:

**1. Инициализация**: Создаётся вектор оптимальных расстояний `best` размера $|V|$, заполненный значением бесконечности
`kInf` (определено как `std::numeric_limits<int>::max() / 2` для предотвращения переполнений). Для стартовой вершины
устанавливается `best[source] = 0`. Также инициализируется вектор `visited` размера $|V|$ нулевыми значениями.

**2. Основной вычислительный цикл**: Выполняется ровно $|V|$ итераций для последовательного разбора каждой вершины
графа.

- **Выбор вершины (`FindNextNode`)**: Производится линейное сканирование векторов `best` и `visited`. Находится
  непосещённый узел `u` с минимальным текущим расстоянием `best[u]`. Временная сложность этого шага составляет $O(|V|)$.
  Если доступных вершин не осталось (возвращено значение -1), цикл прерывается.
- **Фиксация состояния**: Выбранная вершина `u` помечается как посещённая (`visited[u] = 1`).
- **Релаксация рёбер (`RelaxNeighborsSeq`)**: Происходит последовательный обход всех смежных рёбер, исходящих из вершины
  `u`, которые хранятся в таблице списков смежности `OutgoingTable`. Для каждого соседа `v` с весом ребра `w`
  проверяется условие: если `best[u] + w < best[v]`, то значение `best[v]` обновляется.

**3. Агрегация**: Осуществляется финальный проход по вектору путей `best`, отсеивание недостижимых узлов (`kInf`) и
безопасное суммирование конечных расстояний.

Асимптотическая сложность по памяти составляет $O(|V| + |E|)$ для хранения списков смежности графа и векторов состояний.

## 5. Детали реализации

Ключевые внутренние компоненты:

- `FindNextNode`: функция последовательного поиска непосещённого узла с минимальным расстоянием. Выполняет полный
  линейный перебор за $O(|V|)$ без использования сторонних контейнеров, минимизируя накладные расходы на работу с
  динамической памятью.
- `RelaxNeighborsSeq`: функция итеративной релаксации смежных узлов для выбранной вершины на основе списков смежности.
- `CheckedSum`: функция безопасного сложения, предотвращающая неопределённое поведение (UB) при потенциальном выходе за
  границы `std::numeric_limits<std::int64_t>::max()`.
- `FindShortestPaths`: изолированная функция, инкапсулирующая вычисления и возвращающая заполненный вектор расстояний.

Обработка крайних случаев:

- Граф из одной изолированной вершины: циклы релаксации завершаются мгновенно, возвращается корректная сумма 0.
- Несвязные графы: вершины, до которых нет путей, корректно обрабатываются проверкой `if (d == kInf) continue;` в
  функции `AccumulateFiniteDistances` и не ломают общую сумму.

### Тестовые данные

**Тестирование корректности** основано на детерминированных синтетических графах малой размерности с аналитически
предсказуемым результатом суммы кратчайших путей. Используются следующие топологии:

1. Топология «Звезда» с источником в центре. Формируется функцией `MakeStarFromZero(n)`. Из центральной вершины $0$
   исходят рёбра ко всем остальным вершинам $i \\in [1, n-1]$. Вес каждого ребра равен индексу целевой вершины ($w(0, i)
   = i$).
1. Топология «Треугольник с обходным путем». Формируется функцией `MakeTriangleShortcutGraph(n)` и состоит из 3 вершин.
   Содержит три ребра: $(0 \\to 1, w=1)$, $(1 \\to 2, w=1)$ и прямой длинный путь $(0 \\to 2, w=5)$. Ожидаемый
   результат: 3 (путь $0 \\to 1 \\to 2$ имеет суммарную длину $1 + 1 = 2$, а путь $0 \\to 1$ равен $1$). Тест проверяет
   механизм релаксации рёбер и контролирует, что алгоритм успешно обновляет расстояние до вершины $2$ при нахождении
   более короткого составного пути, игнорируя прямой путь с большим весом.

**Тестирование производительности** основано на распределённом случайном разреженном графе, генерируемом функцией
`MakeRandomSparseGraph` со следующими параметрами:

- Количество вершин (`kGraphSize`): 10 000.
- Плотность (исходящая степень каждой вершины `kDegree`): 20 рёбер.
- Зерно генератора случайных чисел (`kSeed`): 42.
- Общее число рёбер в тесте: 200 000.

Эталонная сумма кратчайших расстояний вычисляется однократно последовательным алгоритмом на этапе инициализации и затем
используется для проверки результатов.

### 7.2. Производительность

Замеры времени проводились для двух стандартных режимов запуска тестового фреймворка: прямого выполнения вычислительного
ядра (`task_run`) и полного конвейера с учётом инициализации и валидации (`task_pipeline`). В силу однопоточности
реализации, показатели не изменяются при масштабировании пула потоков.

| Mode | Count (Threads) | Time, s | Speedup | | ------------- | --------------- | ------- | ------- | | task_run | 1 |
0.3535 | 1.00 | | task_pipeline | 1 | 0.3535 | 1.00 |

Основную долю времени работы в режиме `task_run` занимает итеративный квадратичный поиск минимального узла через
`FindNextNode` и проведение 200 000 операций условной релаксации рёбер. Разница между `task_pipeline` и `task_run`
незначительна, что указывает на крайне низкие накладные расходы на этапе `PreProcessingImpl` при построении списков
смежности.
