## 1. Контекст

Вычисление многомерных интегралов по регулярным сеткам обладает высокой степенью независимости вычислений в различных
пространственных подобластях. Это делает задачу идеальным кандидатом для распараллеливания по технологиям Fork-Join,
таким как OpenMP. Перенос вычислений на параллельные потоки позволяет кратно сократить время интегрирования на
многоядерных процессорах.

---

## 2. Постановка задачи

Постановка задачи полностью идентична последовательной версии, однако геометрия вычислений адаптируется под параллельную
 архитектуру:

* **Разделение данных:** Отрезок интегрирования по первому измерению (`limits[0]`) делится на $P$ равных частей, где $P
= \text{num\_threads}$.
* **Синхронизация результатов:** Локально вычисленные потоками суммы интегрирования собираются в общую переменную
посредством встроенной редукции OpenMP.

---

## 3. Базовый алгоритм

В основу параллельной реализации заложен принцип геометрического декомпозирования пространства интегрирования по
**первой (внешней) координатной оси**.

1. **Деление отрезка:** Вычисляется базовая длина подобласти для одного потока: `delta = (limits[0].second -
limits[0].first) / num_threads`.
2. **Коррекция границ:** Чтобы потоки не пересекались и не создавали вычислительных пустот из-за погрешностей типа
`double`, границы локальных отрезков `left_border` и `right_border` выравниваются по узлам сетки через вспомогательную
функцию `customRound()`.
3. **Параллельный итеративный DFS:** Каждый поток независимо выполняет обход закрепленной за ним подсетки с
использованием структуры `std::stack` (аналогично последовательной версии, но со смещенными границами первого
измерения).
4. **Свертка:** Интегральные вклады потоков суммируются.

## 4. Детали реализации

* **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Специфика OpenMP сосредоточена в методе `RunImpl()`. Директива параллелизации спроектирована с жестким контролем области
 видимости переменных (`default(none)`):

```cpp
#pragma omp parallel num_threads(num_threads) default(none) \
            shared(limits, simpson_factor, actual_step, delta, function) \
            reduction(+:res)
{
  int tid = omp_get_thread_num();
  double left_border, right_border;
  
  // Вычисление локальных границ для каждого потока с выравниванием по сетке
  if (tid != 0) {
    left_border = customRound(limits[0].first + delta * tid, actual_step[0]);
  } else {
    left_border = limits[0].first;
  }

  if (tid != num_threads - 1) {
    right_border = customRound(limits[0].first + delta * (tid + 1), actual_step[0]);
  } else {
    right_border = limits[0].second;
  }

  // Каждый поток считает интеграл на своем выделенном интервале [left_border, right_border]
  res += count(left_border, right_border, simpson_factor, limits, actual_step, function);
}
I_res = res;

```

### Анализ директив OpenMP

* `num_threads(num_threads)`: Задаёт фиксированное количество потоков, полученное из тестового каркаса
(`ppc::util::GetNumThreads()`).
* `default(none)`: Обязывает явно указывать режим доступа ко всем переменным вне параллельного блока, предотвращая
скрытые гонки данных (data races).
* `shared(...)`: Передаёт векторы шагов, лимитов и объект подынтегральной функции по ссылке во все потоки на чтение.
* `reduction(+:res)`: Создаёт приватную копию переменной `res` для каждого потока, инициализирует её нулём, а при
выходе из блока безопасно складывает (атомарно) промежуточные результаты в общую переменную.

## 5. Проверка корректности

Параллельный backend тестируется с помощью `Google Test` на тех же наборах математических функций, что и
последовательный. Дополнительно проверяется:

* Инвариантность вычисленного интеграла при изменении числа потоков от 1 до 24.
* Отсутствие эффекта «потерянных узлов» на стыках границ потоков (верификация корректности работы `customRound`).

## 7. Результаты

Замеры производительности выполнены на той же вычислительной задаче, что и для последовательной версии. Для демонстрации
 масштабируемости тесты запущены на различном числе физических/логических ядер процессора ($2, 4, 8, 12, 16, 24$).
 Базовое время последовательной версии взято из отчета `seq` ($T_1 = 1.360$ с для `task_run`).

Замечание к результатам: График ускорения демонстрирует линейный рост вплоть до 12 потоков (соответствует числу
физических ядер процессора Xeon). При выходе за пределы 12 потоков включается технология Hyper-Threading (до 24
логических процессоров). Так как алгоритм утилизирует интенсивные вычисления и частые аллокации памяти внутри стеков,
логические ядра делят общие исполнительные блоки физического ядра, из-за чего эффективность ($Efficiency$) падает до
~0.57 на 24 потоках, что полностью согласуется с архитектурой процессора.
