## 1. Введение

Задача численного интегрирования функций высокой размерности по регулярным сеткам методом Симпсона
(составным методом парабол) обладает высокой вычислительной сложностью («проклятие размерности»),
но при этом имеет колоссальный потенциал для распараллеливания.
Итерационный обход пространственной координатной сетки содержит полностью независимые ветви вычислений,
а финальная сборка результата сводится к единственной операции редукции промежуточных сумм.
Это делает задачу идеальным полигоном для исследования и сравнения накладных расходов различных параллельных
runtime-сред (OpenMP, oneTBB, std::thread, MPI) в изоляции от алгоритмической сложности синхронизации.

## 2. Единая постановка задачи

**Вход:**

* Подынтегральная функция `std::function<double(const std::vector<double>&)>`.
* Границы интегрирования `std::vector<std::pair<double, double>> limits` для каждого измерения.
* Базовый шаг интегрирования `double h` (используется для расчета количества шагов по каждой координатной оси;
 если число шагов получается нечетным, оно инкрементируется в большую сторону для соблюдения требований метода
 Симпсона).

**Выход:** приближённое значение многомерного интеграла (`double`).

**Ограничения:** вектор `limits` не пуст, функция определена и задана, `h <= 0.01` (для обеспечения сходимости).

**Критерий корректности:** математическая эквивалентность результатов последовательной и всех параллельных версий
«бит-в-бит» с учетом погрешностей плавающей точки.

## 3. Единая методика эксперимента

**Окружение:**

* **CPU:** Intel(R) Xeon(R) CPU E5-2650 v4 @ 2.20GHz (**Cores:** 12, **Threads:** 24)
* **RAM:** 32Gb *(укажите объем вашей ОЗУ, если он отличается)*
* **OS:** Microsoft Windows 10 Pro (Сборка 18363) / Среда выполнения тестов курса
* **Compiler:** MSVC 19.43.34808.0 (Release)

**Переменные окружения / Параметры каркаса:**

* `Количество потоков/воркеров (Workers)`: 2, 4, 8, 12, 16, 24
* `Конфигурация MPI процессов (для ALL)`: 2, 3, 4 процесса (ранга)

**Тестовая задача:** Вычисление многомерного интеграла высокой плотности расчетной сетки. Время последовательного
эталона (Baseline) для обычного запуска зафиксировано на уровне **1.360 с**.

**Расчёт метрик:**

* Speedup = T(seq) / T(parallel)
* Efficiency = Speedup / NumProcs $\times$ 100%
* Для ALL: NumProcs = Ranks $\times$ Threads/rank (Общее число воркеров)

**Повторы:** Каждая конфигурация запускалась в рамках встроенного перформанс-каркаса курса. Временные показатели
стабильны между прогонами (разброс < 3%).

## 4. Сводка корректности

Все разработанные backend-ы (SEQ, OMP, TBB, STL, ALL) проходят верификацию в рамках фреймворка `Google Test`. Набор
тестов включает интеграцию константных, тригонометрических и многомерных полиномиальных функций на регулярных отрезках.
Результаты работы параллельных версий полностью совпадают с последовательным решением.

Ограничения применимости: ALL-версия требует гибридной среды запуска (`mpiexec` + поддержка многопоточности); при
нехватке аппаратных потоков подзадачи эффективно уплотняются планировщиком операционной системы.

## 6. Интерпретация различий

**SEQ** — Последовательный baseline (1.360 с). Производит честный полный обход многомерной сетки с использованием
`std::stack`. Время выполнения стабильно во всех прогонах, что гарантирует объективность вычисления метрик ускорения
параллельных версий.

**OMP** — Демонстрирует отличную масштабируемость на физических ядрах ($Speedup = 10.88\times$ при 12 потоках).
Декомпозиция внешней оси с использованием встроенной директивы редукции `reduction(+:res)` работает с минимальным
overhead. При выходе за пределы 12 физических ядер процессора (включение Hyper-Threading на 16 и 24 потоках)
эффективность падает до 57.8%, поскольку логические потоки начинают конкурировать за математические исполнительные блоки
 (АЛУ) ядра процессора Xeon.

**TBB** — Показывает результаты, близкие к OpenMP ($Speedup = 13.08\times$ на 24 воркерах). Высокоуровневая абстракция
`tbb::parallel_reduce` с диапазоном `tbb::blocked_range` эффективно балансирует нагрузку. Мизерное отставание от OMP и
STL обусловлено внутренними инфраструктурными расходами oneTBB на поддержку диспетчеризации задач и механизмов
work-stealing (которые при равномерной сетке простаивают, но создают небольшой overhead).

**STL** — Продемонстрировал наилучшее абсолютное время выполнения на физических ядрах ($T_{12} = 0.121$ с, $Speedup =
11.24\times$, $Efficiency = 93.7\%$). Объясняется тем, что потоки `std::thread` сопоставляются напрямую с потоками ОС.
Ручное крупноблочное деление и изоляция вывода (запись каждого потока строго в свою ячейку массива `results[i]`)
полностью устранили ложное разделение кэш-линий (False Sharing) и гонки данных без использования тяжелых мьютексов.
Сборка результатов через `std::accumulate` выполняется только после барьера `t.join()`.

**ALL** — Реализует гибридную модель (MPI для межпроцессного разделения + OpenMP для внутрипроцессных потоков).
Результаты сопоставимы с чистым OpenMP. Поскольку тесты проводятся на одном физическом сервере, выигрыша от
распределенной памяти нет, а вызов коллективной операции синхронизации и сборки `MPI_Reduce` на мастер-ранге порождает
небольшие накладные расходы. Эффективность гибридной схемы полностью раскроется при переносе вычислений на многоузловой
кластер (многосерверную стойку).

## 8. Заключение

**Лучшая версия:** На исследуемой 12-ядерной (24-поточной) архитектуре в рамках одного вычислительного узла наилучшую
скорость показал backend **STL (`std::thread`)**, вплотную за ним следуют **OMP** и **ALL**. Выбор между ними зависит от
 контекста: `OMP` — наиболее лаконичный код с минимумом изменений, `STL` не имеет сторонних зависимостей, а `ALL`
незаменим для масштабирования на суперкомпьютерные кластеры.

**Ограничения сравнения:**

* Тестирование велось на одной физической ноде с процессором архитектуры Intel Xeon Broadwell, поведение алгоритма на
иных архитектурах (например, NUMA с несколькими сокетами) может потребовать настройки привязки потоков к ядрам (Thread
Affinity).
* Итеративный DFS алгоритм со `std::stack` создает локальные аллокации памяти, интенсивность которых упирается в
пропускную способность шины памяти на больших пулах потоков.
