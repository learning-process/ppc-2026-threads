## 1. Контекст

Использование стандартных потоков `std::thread` предоставляет прямой низкоуровневый интерфейс к потокам операционной
системы (POSIX Threads в Linux или Windows Threads в Windows). В отличие от высокоуровневых абстракций (OpenMP, TBB),
программист здесь вручную управляет жизненным циклом потоков, стратегией декомпозиции и синхронизацией памяти. Это
минимизирует скрытые накладные расходы рантайма, но требует жесткого контроля за состоянием гонок данных (data race).

---

## 2. Постановка задачи

Вычислительное ядро производит численный расчет кратного интеграла. Параллельная структура адаптирована под
низкоуровневое управление:

* **Распределение нагрузки:** Область интегрирования по первому измерению разделяется на $P$ изолированных непрерывных
интервалов, где $P = \text{num\_threads}$.
* **Хранение результатов:** Во избежание взаимной блокировки и гонок за общую переменную, каждый поток записывает
вычисленный интеграл подобласти в выделенный элемент независимого вектора результатов.

---

## 3. Базовый алгоритм

Для обеспечения истинного параллелизма и максимального быстродействия без использования примитивов синхронизации
(мьютексов, атомиков) реализована классическая схема декомпозиции данных:

1. Вычисляется размер подобласти `delta` по первой оси координат.
2. Инициализируется вектор `results` размером `num_threads` для локальных сумм.
3. В цикле порождается `num_threads` объектов `std::thread`. Каждый поток получает свой уникальный логический номер `i`,
 на основе которого вычисляет локальные границы `left` и `right` с помощью функции стыковки сетки `CustomRound`.
4. Функция потока запускает тяжелый итеративный обход подсетки (`OuntNtIntegral`), сохраняя итог в `results[i]`.
5. Главный поток дожидается завершения всех воркеров (`join()`), после чего последовательно суммирует частичные ответы.

---

## 4. Детали реализации

* **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Ключевой фрагмент управления потоками и сборки данных в методе `RunImpl()`:

```cpp
std::vector<double> results(num_threads, 0.0);
std::vector<std::thread> threads;

// Фаза 1: Инициализация и параллельный запуск всех потоков
for (int i = 0; i < num_threads; ++i) {
  threads.emplace_back([this, i, num_threads, delta, &limits, &actual_step, &function, simpson_factor, &results]() {
    double left = limits[0].first;
    double right = limits[0].second;

    // Расчет изолированных непересекающихся границ для потока i
    if (i != 0) {
      left = CustomRound(limits[0].first + delta * i, actual_step[0]);
    }
    if (i != num_threads - 1) {
      right = CustomRound(limits[0].second - delta * (num_threads - i - 1), actual_step[0]);
    }

    // Запись строго в свою ячейку памяти для предотвращения Data Race
    results[i] = OuntNtIntegral(left, right, simpson_factor, limits, actual_step, function);
  });
}

// Фаза 2: Барьерная синхронизация (ожидание выполнения всех расчетов)
for (auto &t : threads) {
  t.join();
}

// Фаза 3: Финальный редукционный сбор данных в главном потоке
I_res_ = std::accumulate(results.begin(), results.end(), 0.0);

```

### Архитектурные особенности STL-реализации

* **Разделение фаз Запуска и Ожидания:** Цикл вызова `join()` гарантированно вынесен за пределы цикла создания потоков.
Это обеспечивает одновременное (параллельное) выполнение вычислений всеми воркерами.
* **Изоляция памяти (`results[i]`):** Запись локальных ответов происходит по уникальным индексам. Данный подход
устраняет необходимость использования блокировок (`std::mutex`), сохраняя производительность на уровне последовательного
 кода в рамках одного потока.
* `std::accumulate`: Выполняет последовательную редукцию над вектором результатов в контексте главного потока после
барьера `join()`, гарантируя детерминизм плавающей точки.

## 5. Проверка корректности

Корректность функционирования многопоточного кода проверяется с помощью автоматических тестов `Google Test`:

* Тесты проверяют стабильность результатов вычисления интеграла при варьировании числа потоков (`num_threads` от 1 до
24).
* Сравнивается абсолютная погрешность со значениями аналитических первообразных, а также проводится сверка «бит-в-бит»
с результатами версий `seq`, `omp` и `tbb`.

## 8. Выводы

Низкоуровневое распараллеливание средствами `std::thread` показало максимальную вычислительную эффективность за счет
прямого сопоставления подзадач потокам ОС. Архитектурное решение с разделением массива вывода (`results[i]`) позволило
полностью отказаться от блокировок во время выполнения циклов интегрирования. Алгоритм полностью готов к использованию и
 выступает наиболее производительным решением для локальной вычислительной ноды.
