## 1. Контекст

OpenMP-версия переносит фазу построения выпуклых оболочек из однопоточного цикла в параллельную область. Фаза BFS остаётся последовательной — общий массив `vis` не допускает одновременной записи из нескольких потоков. Поскольку выпуклая оболочка каждой компоненты вычисляется независимо, параллелизация этой фазы является embarrassingly parallel и не требует ни редукции, ни барьеров внутри цикла.


## 2. Постановка задачи

Вход и выход идентичны SEQ. Ускорение достигается за счёт параллельного вычисления `ConvexHullMonotonicChain` для K независимых компонент. Число потоков управляется переменной окружения `OMP_NUM_THREADS` (runner курса также экспортирует `PPC_NUM_THREADS` как `OMP_NUM_THREADS`).

## 3. Базовый алгоритм

Алгоритм идентичен SEQ (BFS + монотонная цепь Эндрю).

## 4. Схема распараллеливания

Параллелизируется цикл по компонентам в `SolveOMP`:

```cpp
// omp/src/ops_omp.cpp  (строки 136–141)
#pragma omp parallel for default(none) shared(comps, hulls, n) schedule(static)
for (int i = 0; i < n; ++i) {
  const auto idx = static_cast<std::size_t>(i);
  hulls[idx] = ConvexHullMonotonicChain(std::move(comps[idx]));
}
```

**Объяснение каждой клаузы:**

| Клауза              | Значение                                                                                             |
|---------------------|------------------------------------------------------------------------------------------------------|
| `parallel for`      | Создать команду потоков и распределить итерации цикла между ними                                    |
| `default(none)`     | Запретить неявное захватывание переменных — все переменные обязаны быть явно объявлены              |
| `shared(comps, hulls, n)` | Все три переменные общие: каждый поток читает свой `comps[i]` и пишет в свой `hulls[i]` — гонки нет, так как индексы не пересекаются |
| `schedule(static)`  | Равномерное статическое распределение итераций по потокам блоками `ceil(n / num_threads)`. Выбрано, потому что компоненты имеют схожий размер на шахматном паттерне, а overhead динамического планирования не оправдан |

**Переменные private:** в теле цикла `idx` — автоматическая локальная переменная, де-факто private без явного указания.

**Барьер:** в конце `parallel for` существует неявный барьер. Он гарантирует, что все `hulls[i]` заполнены до возврата из `SolveOMP`. Явного `#pragma omp barrier` не требуется.

**Гонка данных:** отсутствует, так как каждый поток записывает в уникальный индекс вектора `hulls`. Вектор предварительно выделен (`hulls.resize(comps.size())`) — никаких перевыделений во время параллельного цикла нет.

## 5. Детали реализации

**Файлы:** [omp/include/ops_omp.hpp](../omp/include/ops_omp.hpp), [omp/src/ops_omp.cpp](../omp/src/ops_omp.cpp)

Изменения относительно SEQ ограничены функцией `SolveOMP`:
- Вместо `hulls.reserve` + `push_back` используется `hulls.resize(comps.size())` до параллельного цикла — это позволяет потокам писать по индексу без синхронизации.
- Цикл `for (auto &c : comps)` заменён на `for (int i = 0; i < n; ++i)` с явным индексом, необходимым для `parallel for`.

`ExtractComponents4` и `ConvexHullMonotonicChain` не изменены по сравнению с SEQ.

## 6. Проверка корректности

Результат OMP-версии сравнивается с SEQ-эталоном для всех 7 функциональных тестов. Детерминизм гарантируется: хотя потоки выполняют `hulls[i]` в произвольном порядке, финальный вектор `hulls` собирается по индексам, а не по времени завершения — порядок оболочек в выходе совпадает с SEQ.

Проверка на гонку: `schedule(static)` с непересекающимися диапазонами индексов не требует mutex или atomic. Дополнительно можно верифицировать с ThreadSanitizer:
```bash
cmake -S . -B build-tsan -DCMAKE_BUILD_TYPE=RelWithDebInfo \
  -DCMAKE_CXX_FLAGS="-fsanitize=thread -fno-omit-frame-pointer -g"
cmake --build build-tsan --parallel
```

## 8. Результаты

| Потоков | Медианное время (task) | Медианное время (pipeline) | Speedup task | Speedup pipe | Efficiency task | Efficiency pipe |
|---------|------------------------|----------------------------|--------------|--------------|-----------------|-----------------|
| 1       | 1.316 мс               | 1.300 мс                   | 1.04×        | 1.12×        | 104%            | 112%            |
| 2       | 1.285 мс               | 1.242 мс                   | 1.07×        | 1.17×        | 53%             | 58%             |
| 4       | 1.320 мс               | 1.252 мс                   | 1.04×        | 1.16×        | 26%             | 29%             |

> **Ожидаемое поведение:** ускорение ограничено долей параллелизуемой фазы (закон Амдала). BFS (~60–70% времени) не распараллелен, поэтому теоретический предел ускорения при 4 потоках составляет около 1/(0.65 + 0.35/4) ≈ 1.4–1.5×.

> **Комментарий о масштабируемости:** `schedule(static)` эффективен при равномерной нагрузке. На шахматном паттерне компоненты почти равного размера — это подходящий выбор. При произвольных изображениях с сильно неравными компонентами `schedule(dynamic)` мог бы дать лучший балансировки, но добавил бы overhead на синхронизацию.

## 9. Выводы

OpenMP дал прирост в фазе построения оболочек с минимальными изменениями кода (одна директива). Узкое место — последовательный BFS, ограничивающий практическое ускорение ~1.3–1.5× независимо от числа потоков. Для задач с большим числом крупных компонент выигрыш от параллелизации оболочек значительнее.
