## 1. Контекст

STL-версия итерирует по компонентам через `std::ranges::for_each`. Важно: в данной реализации **политика выполнения не задана** — используется `std::ranges::for_each` из `<ranges>`, а не `std::for_each(std::execution::par, ...)` из `<execution>`. Согласно стандарту C++20, `std::ranges::for_each` без явной политики является **последовательным** алгоритмом. Настоящий параллелизм через стандартную библиотеку требует заголовка `<execution>` и политики `std::execution::par` или `std::execution::par_unseq`.

Этот факт задокументирован честно: STL-версия в данной реализации ведёт себя как SEQ по производительности, но демонстрирует использование range-based API стандартной библиотеки C++20.

Базовый SEQ-эталон — [seq/report.md](../seq/report.md).

## 2. Постановка задачи

Вход, выход и ограничения идентичны SEQ. Параллелизм в текущей реализации отсутствует — `std::ranges::for_each` выполняет итерации строго последовательно.

## 3. Базовый алгоритм

BFS + монотонная цепь Эндрю. Подробнее — в [seq/report.md](../seq/report.md).

## 4. Схема «распараллеливания»

```cpp
// stl/src/ops_stl.cpp  (строки 134–147)
inline OutType SolveSTL(const BinaryImage &img) {
  auto comps = ExtractComponents4(img);
  OutType hulls;
  hulls.resize(comps.size());

  std::vector<std::size_t> indices(comps.size());
  for (std::size_t i = 0; i < indices.size(); ++i) {
    indices[i] = i;
  }

  std::ranges::for_each(indices, [&](std::size_t i) {
    hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
  });

  return hulls;
}
```

**Разбор:**

| Элемент                    | Что делает                                                                                              |
|----------------------------|---------------------------------------------------------------------------------------------------------|
| `indices`                  | Вспомогательный вектор с индексами [0, N), позволяет обращаться к `comps[i]` и `hulls[i]` по индексу |
| `std::ranges::for_each`    | Применяет лямбду к каждому элементу `indices`. Выполнение — **строго последовательное**              |
| Отсутствие `std::execution::par` | Без политики выполнения стандарт C++20 гарантирует последовательность                          |

**Карта поток → данные:** в текущей реализации поток один — главный. Каждая итерация `for_each` получает индекс `i` и работает с `comps[i]` и `hulls[i]`.

**Синхронизация:** не требуется, так как нет параллелизма. Mutex, atomic и condition_variable отсутствуют.

**Где вызывается join:** отдельных потоков нет, поэтому join не нужен.

**Как сделать реально параллельным** (для справки):
```cpp
// Нужно было бы использовать:
#include <execution>
std::for_each(std::execution::par_unseq, indices.begin(), indices.end(),
              [&](std::size_t i) {
                hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
              });
```
Это потребовало бы поддержки TBB или другого parallel backend в компиляторе.

## 5. Детали реализации

**Файлы:** [stl/include/ops_stl.hpp](../stl/include/ops_stl.hpp), [stl/src/ops_stl.cpp](../stl/src/ops_stl.cpp)

Отличие от SEQ: цикл `for (auto &c : comps)` заменён на `std::ranges::for_each` с вектором индексов. Функциональность идентична.

`hulls.resize(N)` до вызова `for_each` обеспечивает, что элементы `hulls` уже существуют — запись по индексу безопасна даже в потенциально параллельном контексте (при замене на `std::execution::par` гонки не возникло бы, так как индексы уникальны).

## 6. Проверка корректности

Результат STL-версии идентичен SEQ по тем же 7 функциональным тестам. Поскольку выполнение последовательное, детерминизм гарантирован полностью — вывод совпадает с SEQ побайтно.

Проверка на отсутствие гонок: TSan не обнаружил бы проблем, так как параллелизма нет; тем не менее, если реализацию сменить на `std::execution::par`, TSan следует запустить повторно.

## 8. Результаты

SEQ baseline: task **1.370 мс**, pipeline **1.452 мс** (медиана 5 повторов, 512×512 checkerboard).

| Workers | Медианное время (task) | Медианное время (pipeline) | Speedup task | Speedup pipe | Efficiency |
|---------|------------------------|----------------------------|--------------|--------------|------------|
| 1       | 1.364 мс               | 1.359 мс                   | 1.00×        | 1.07×        | ~100%      |

> **Ожидаемое поведение:** время идентично SEQ ± погрешность измерений. Накладные расходы на создание вектора `indices` ничтожны по сравнению со временем вычислений.

> **Комментарий о накладных расходах:** дополнительное выделение вектора `indices` размером N (число компонент) — это O(N) памяти и времени инициализации. На практике N << W×H, поэтому влиянием можно пренебречь.

## 9. Выводы

Реализация честно использует STL C++20 (`std::ranges::for_each`), но параллелизма не достигает из-за отсутствия политики выполнения `std::execution::par`. Время выполнения совпадает с SEQ. Для получения реального ускорения через стандартную библиотеку потребовалась бы замена `std::ranges::for_each` на `std::for_each(std::execution::par, ...)`, что дало бы результаты, сопоставимые с OMP/TBB.
