## 1. Контекст

STL-версия итерирует по компонентам через `std::ranges::for_each`. Важно: политика выполнения
**не задана** — используется `std::ranges::for_each` из `<ranges>`, а не
`std::for_each(std::execution::par, ...)` из `<execution>`. Согласно стандарту C++20,
`std::ranges::for_each` без явной политики является **последовательным** алгоритмом.

Этот факт задокументирован честно: STL-версия ведёт себя как SEQ по производительности,
но демонстрирует использование range-based API стандартной библиотеки C++20.

## 2. Постановка задачи

Вход, выход и ограничения идентичны SEQ. Параллелизм в текущей реализации отсутствует —
`std::ranges::for_each` выполняет итерации строго последовательно.

## 3. Базовый алгоритм

BFS + монотонная цепь Эндрю.

## 4. Схема «распараллеливания»

```cpp
// stl/src/ops_stl.cpp  (строки 134–147)
inline OutType SolveSTL(const BinaryImage &img) {
  auto comps = ExtractComponents4(img);
  OutType hulls;
  hulls.resize(comps.size());

  std::vector<std::size_t> indices(comps.size());
  for (std::size_t i = 0; i < indices.size(); ++i) {
    indices[i] = i;
  }

  std::ranges::for_each(indices, [&](std::size_t i) {
    hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
  });

  return hulls;
}
```

**Разбор:**

| Элемент                          | Что делает                                              |
|----------------------------------|---------------------------------------------------------|
| `indices`                        | Вспомогательный вектор [0, N) для доступа по индексу   |
| `std::ranges::for_each`          | Применяет лямбду последовательно к каждому элементу    |
| Отсутствие `std::execution::par` | Без политики — строго последовательный алгоритм (C++20) |

**Карта поток → данные:** поток один — главный. Каждая итерация `for_each` получает
индекс `i` и работает с `comps[i]` и `hulls[i]`.

**Синхронизация:** не требуется, так как нет параллелизма.

**Где вызывается join:** отдельных потоков нет, поэтому join не нужен.

**Как сделать реально параллельным** (для справки):

```cpp
// Нужно было бы использовать:
#include <execution>
std::for_each(std::execution::par_unseq, indices.begin(), indices.end(),
              [&](std::size_t i) {
                hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
              });
```

Это потребовало бы поддержки TBB или другого parallel backend в компиляторе.

## 5. Детали реализации

**Файлы:** [stl/include/ops_stl.hpp](../stl/include/ops_stl.hpp),
[stl/src/ops_stl.cpp](../stl/src/ops_stl.cpp)

Отличие от SEQ: цикл `for (auto &c : comps)` заменён на `std::ranges::for_each`
с вектором индексов. Функциональность идентична.

`hulls.resize(N)` до вызова `for_each` обеспечивает, что элементы `hulls` уже существуют —
запись по индексу безопасна даже в потенциально параллельном контексте.

## 6. Проверка корректности

Результат STL-версии идентичен SEQ по тем же 7 функциональным тестам. Поскольку выполнение
последовательное, детерминизм гарантирован полностью — вывод совпадает с SEQ побайтно.

Проверка на отсутствие гонок: TSan не обнаружил бы проблем, так как параллелизма нет;
тем не менее, если реализацию сменить на `std::execution::par`, TSan следует запустить повторно.

> **Комментарий о накладных расходах:** дополнительное выделение вектора `indices` — O(N)

## 9. Выводы

Реализация использует STL C++20 (`std::ranges::for_each`), но параллелизма не достигает
из-за отсутствия политики `std::execution::par`. Время выполнения совпадает с SEQ.
Для реального ускорения потребовалась бы замена на `std::for_each(std::execution::par, ...)`.
