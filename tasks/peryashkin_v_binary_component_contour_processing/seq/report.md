## 1. Контекст

Задача — обработка бинарного изображения: нахождение всех связных компонент (4-связность) и построение выпуклой оболочки для каждой из них. Последовательная версия служит эталоном корректности и baseline для измерения ускорения параллельных версий. Именно здесь фиксируются все инварианты задачи, которые параллельные реализации обязаны сохранять.

## 2. Постановка задачи

**Вход (`InType = BinaryImage`):**
- `width`, `height` — размеры изображения (оба > 0)
- `data` — плоский массив `uint8_t` размером `width * height`, элементы: 0 (фон) или 1 (объект)

**Выход (`OutType = vector<vector<Point>>`):**
- Вектор выпуклых оболочек. Каждая оболочка — упорядоченный список вершин в виде точек `{x, y}`.

**Ограничения и граничные случаи:**
- Пустое изображение (width ≤ 0 или height ≤ 0) — `ValidationImpl` возвращает `false`.
- `data.size() ≠ width * height` — `ValidationImpl` возвращает `false`.
- Изображение без единичных пикселей — результат: пустой вектор.
- Одиночный пиксель — оболочка из одной точки.
- Линия пикселей — оболочка из двух точек.

## 3. Базовый алгоритм

### Этап 1: Извлечение компонент (`ExtractComponents4`)

BFS с 4-связностью по всему изображению. Общий массив `vis` предотвращает повторное посещение пикселей. Каждая компонента — вектор точек, собранный одним вызовом `BfsComponent4`.

Сложность: **O(W × H)** — каждый пиксель посещается не более одного раза.

### Этап 2: Выпуклая оболочка (`ConvexHullMonotonicChain`)

Алгоритм монотонной цепи Эндрю. Для каждой компоненты:
1. Сортировка точек лексикографически по (x, y): **O(M log M)**
2. Удаление дубликатов: **O(M)**
3. Построение нижней цепи (слева направо, поворот > 0): **O(M)**
4. Построение верхней цепи (справа налево): **O(M)**
5. Объединение цепей: **O(M)**

Итоговая сложность для одной компоненты из M точек: **O(M log M)**.  
Суммарная по всем K компонентам: **O(sum_k M_k × log M_k) ≤ O(N log N)**, где N = W × H.

**Инварианты:**
- Ориентация: используется векторное произведение `Cross(o, a, b)` — добавляем точку только при левом повороте (Cross > 0).
- Результат — выпуклый многоугольник в порядке обхода CCW (counter-clockwise), без коллинеарных точек.
- Для компоненты из 1 точки — оболочка из 1 точки; для 2 различных точек — оболочка из 2 точек.

### Конвейер задачи (SEQ)

```cpp
// seq/src/ops_seq.cpp
inline OutType SolveSEQ(const BinaryImage &img) {
  auto comps = ExtractComponents4(img);   // O(W*H) — BFS, последовательно
  OutType hulls;
  hulls.reserve(comps.size());
  for (auto &c : comps) {
    hulls.push_back(ConvexHullMonotonicChain(std::move(c)));  // O(M log M)
  }
  return hulls;
}
```

`ExtractComponents4` является последовательным узким местом во всех версиях: общий массив `vis` делает BFS непригодным для прямой параллелизации. Этап построения оболочек — embarrassingly parallel, поскольку компоненты независимы.

## 4. Детали реализации

**Файлы:** [seq/include/ops_seq.hpp](../seq/include/ops_seq.hpp), [seq/src/ops_seq.cpp](../seq/src/ops_seq.cpp)

| Метод               | Действие                                                                |
|---------------------|-------------------------------------------------------------------------|
| `ValidationImpl`    | Проверяет `width > 0`, `height > 0`, `data.size() == width * height`   |
| `PreProcessingImpl` | Очищает `local_out_`                                                    |
| `RunImpl`           | Вызывает `SolveSEQ`, результат сохраняет в `local_out_`                |
| `PostProcessingImpl`| Копирует `local_out_` в `GetOutput()`                                  |

**Структура данных `BinaryImage`:**
```cpp
struct BinaryImage { int width; int height; std::vector<uint8_t> data; };
```
Индексация: `data[y * width + x]`.

## 5. Проверка корректности

Результат проверяется набором функциональных тестов (`tests/functional/main.cpp`):

| Тест              | Вход                          | Ожидаемый результат               |
|-------------------|-------------------------------|-----------------------------------|
| `empty`           | 5×4, все нули                 | Пустой вектор оболочек            |
| `point`           | 5×5, единица в (2,2)          | 1 оболочка из 1 точки             |
| `line`            | Горизонтальная линия пикселей | 1 оболочка из 2 точек             |
| `square`          | Квадрат 3×3                   | 1 оболочка из 4 угловых вершин    |
| `two_components`  | Два квадрата 2×2              | 2 оболочки по 4 вершины каждая    |
| `hole`            | Прямоугольный контур          | 1 оболочка по внешнему периметру  |
| `touch_border`    | Прямоугольник у края          | 1 оболочка, касающаяся границы    |

Корректность оценивается сравнением выхода SEQ с эталонными значениями. Все параллельные версии должны давать идентичный результат (порядок оболочек и вершин совпадает, так как компоненты обнаруживаются детерминированно).

## 8. Выводы

Последовательная реализация выбрана как baseline по двум причинам:
1. **Детерминизм:** порядок компонент и порядок вершин в оболочках однозначно определены алгоритмом обхода и сортировкой, что позволяет точно сравнивать выход любого backend с эталоном.
2. **Узкое место:** BFS принципиально последователен из-за общего массива `vis`. Параллелизовать имеет смысл только фазу построения оболочек (embarrassingly parallel), что и делают все остальные версии.
