## 1. Контекст

TBB-версия реализует ту же схему распараллеливания, что и OMP, но через задачно-ориентированную
модель oneTBB: вместо директив компилятора используется `tbb::parallel_for` с `blocked_range`.
Это даёт runtime-контроль над распределением итераций и позволяет TBB самостоятельно выбирать
оптимальный grain size через work-stealing планировщик.

## 2. Постановка задачи

Вход, выход и ограничения идентичны SEQ. Число worker-потоков TBB ограничивается через
`PPC_NUM_THREADS` (runner устанавливает `tbb::global_control::max_allowed_parallelism`).

## 3. Базовый алгоритм

BFS + монотонная цепь Эндрю.

## 4. Схема распараллеливания

```cpp
// tbb/src/ops_tbb.cpp  (строки 142–147)
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<std::size_t>(0, comps.size()),
    [&](const oneapi::tbb::blocked_range<std::size_t> &range) {
      for (std::size_t i = range.begin(); i != range.end(); ++i) {
        hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
      }
    });
```

**Разбор компонентов вызова:**

| Элемент                       | Назначение                                                        |
|-------------------------------|-------------------------------------------------------------------|
| `blocked_range<size_t>(0, N)` | Линейный диапазон [0, N). TBB рекурсивно делит на подзадачи       |
| `grainsize` (не задан явно)   | Используется дефолт (1); TBB находит оптимум через work-stealing  |
| Партиционер                   | `auto_partitioner` — адаптивно делит с учётом объёма работы       |
| Лямбда `[&](...)`             | Захватывает `comps` и `hulls` по ссылке; работает с поддиапазоном |

**Контроль конкуренции:** число worker-потоков ограничено через `global_control`:

```cpp
// устанавливается runner-ом курса
tbb::global_control gc(tbb::global_control::max_allowed_parallelism, PPC_NUM_THREADS);
```

Согласно документации oneTBB, `max_allowed_parallelism` задаёт максимум worker-потоков,
а не гарантированное количество — scheduler может использовать меньше.

**Отсутствие гонок:** каждый вызов лямбды работает с уникальным диапазоном индексов.
Записи в `hulls[i]` не перекрываются. Вектор `hulls` предварительно выделен через `resize`.

## 5. Детали реализации

**Файлы:** [tbb/include/ops_tbb.hpp](../tbb/include/ops_tbb.hpp),
[tbb/src/ops_tbb.cpp](../tbb/src/ops_tbb.cpp)

Изменения относительно SEQ: только функция `SolveTBB` — `parallel_for` вместо
последовательного цикла. `ExtractComponents4` и `ConvexHullMonotonicChain` — без изменений.

Память: `hulls.resize(N)` создаёт N пустых векторов заранее. Каждый поток присваивает
`hulls[i]` результат `ConvexHullMonotonicChain` — это перемещение (`std::move`), не копия.

## 6. Проверка корректности

TBB-версия проверяется теми же 7 функциональными тестами. `parallel_for` гарантирует
выполнение всех итераций до возврата, поэтому вектор `hulls` полностью заполнен к моменту
сравнения с SEQ-эталоном. Порядок оболочек детерминирован (соответствует порядку компонент
из BFS), несмотря на произвольный порядок выполнения итераций TBB.

> чтобы амортизировать этот overhead.

## 9. Выводы

TBB удобнее OMP для задач с нерегулярным распределением нагрузки: work-stealing автоматически
балансирует потоки при разных размерах компонент. На данной задаче разница с OMP минимальна.
Узкое место то же: последовательный BFS ограничивает ускорение законом Амдала.
