## 1. Контекст

TBB-версия реализует ту же схему распараллеливания, что и OMP, но через задачно-ориентированную модель oneTBB: вместо директив компилятора используется `tbb::parallel_for` с `blocked_range`. Это даёт runtime-контроль над распределением итераций и позволяет TBB самостоятельно выбирать оптимальный grain size через work-stealing планировщик.

Базовый SEQ-эталон — [seq/report.md](../seq/report.md).

## 2. Постановка задачи

Вход, выход и ограничения идентичны SEQ. Число worker-потоков TBB ограничивается через `PPC_NUM_THREADS` (runner курса устанавливает `tbb::global_control::max_allowed_parallelism`).

## 3. Базовый алгоритм

BFS + монотонная цепь Эндрю. Подробнее — в [seq/report.md](../seq/report.md).

## 4. Схема распараллеливания

```cpp
// tbb/src/ops_tbb.cpp  (строки 142–147)
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<std::size_t>(0, comps.size()),
    [&](const oneapi::tbb::blocked_range<std::size_t> &range) {
      for (std::size_t i = range.begin(); i != range.end(); ++i) {
        hulls[i] = ConvexHullMonotonicChain(std::move(comps[i]));
      }
    });
```

**Разбор компонентов вызова:**

| Элемент                          | Назначение                                                                                                         |
|----------------------------------|--------------------------------------------------------------------------------------------------------------------|
| `blocked_range<size_t>(0, N)`    | Описывает линейный диапазон индексов [0, N). TBB рекурсивно делит его на подзадачи                               |
| `grainsize` (не задан явно)      | Используется значение по умолчанию (1), что позволяет TBB автоматически находить оптимальный размер чанка через work-stealing. При явном grainsize G: каждый чанк обрабатывает G компонент |
| Партиционер                      | По умолчанию `auto_partitioner` — адаптивно делит диапазон, учитывая объём работы и число потоков               |
| Лямбда `[&](...)`                | Захватывает `comps` и `hulls` по ссылке; каждый вызов лямбды работает с независимым поддиапазоном индексов       |

**Контроль конкуренции:** число worker-потоков ограничено через `global_control`:
```cpp
// устанавливается runner-ом курса
tbb::global_control gc(tbb::global_control::max_allowed_parallelism, PPC_NUM_THREADS);
```
Согласно документации oneTBB, `max_allowed_parallelism` задаёт максимум worker-потоков, а не гарантированное количество — scheduler может использовать меньше при малом объёме работы.

**Отсутствие гонок:** как и в OMP, каждый вызов лямбды работает с уникальным диапазоном индексов. Записи в `hulls[i]` не перекрываются. Вектор `hulls` предварительно выделен через `resize` до вызова `parallel_for`.

## 5. Детали реализации

**Файлы:** [tbb/include/ops_tbb.hpp](../tbb/include/ops_tbb.hpp), [tbb/src/ops_tbb.cpp](../tbb/src/ops_tbb.cpp)

Изменения относительно SEQ: только функция `SolveTBB` — `parallel_for` вместо последовательного цикла. `ExtractComponents4` и `ConvexHullMonotonicChain` — без изменений.

Память: `hulls.resize(N)` создаёт N пустых векторов заранее. Каждый поток затем присваивает `hulls[i]` результат `ConvexHullMonotonicChain`, который возвращает по значению — это перемещение (`std::move`), а не копирование.

## 6. Проверка корректности

TBB-версия проверяется теми же 7 функциональными тестами. Так как `parallel_for` гарантирует выполнение всех итераций до возврата, выходной вектор `hulls` полностью заполнен к моменту сравнения с SEQ-эталоном. Порядок оболочек детерминирован (соответствует порядку компонент из BFS), несмотря на произвольный порядок выполнения итераций TBB.

## 8. Результаты

SEQ baseline: task **1.370 мс**, pipeline **1.452 мс** (медиана 5 повторов, 512×512 checkerboard).

| Workers | Медианное время (task) | Медианное время (pipeline) | Speedup task | Speedup pipe | Efficiency task | Efficiency pipe |
|---------|------------------------|----------------------------|--------------|--------------|-----------------|-----------------|
| 1       | 1.402 мс               | 1.469 мс                   | 0.98×        | 0.99×        | 98%             | 99%             |
| 2       | 1.036 мс               | 1.022 мс                   | 1.32×        | 1.42×        | 66%             | 71%             |
| 4       | 1.032 мс               | 0.910 мс                   | 1.33×        | 1.60×        | 33%             | 40%             |

> **Комментарий о балансировке и overhead:** `auto_partitioner` при малом числе компонент может создавать overhead на делении диапазона. На шахматном паттерне 512×512 число компонент достаточно велико (~тысячи), чтобы амортизировать этот overhead. При очень малом числе крупных компонент рекомендуется задать `grainsize=1` явно с `simple_partitioner`, чтобы форсировать равномерное распределение.

## 9. Выводы

TBB удобнее OMP для задач с нерегулярным распределением нагрузки: work-stealing автоматически балансирует потоки при разных размерах компонент. На данной задаче (компоненты примерно одинакового размера) разница с OMP минимальна. Узкое место то же, что и в OMP: последовательный BFS ограничивает теоретическое ускорение законом Амдала.
