## 1. Контекст

Ручная реализация на `std::thread` демонстрирует низкоуровневое управление
параллелизмом без зависимости от внешних библиотек (OpenMP, TBB). Это позволяет
явно контролировать создание потоков, разбиение данных и сбор результатов,
а также понять накладные расходы, которые в высокоуровневых фреймворках скрыты.

## 2. Постановка задачи

Вычислить приближённое значение многомерного определённого интеграла методом
средних прямоугольников. Входные данные: подынтегральная функция, границы
интегрирования по каждому измерению, число разбиений `n`.

## 3. Базовый алгоритм

Последовательная версия строит равномерную сетку из `n^dim` ячеек, перебирает их
с помощью инкрементального многомерного индекса (`NextIndex`), вычисляет значение
функции в центре каждой ячейки, суммирует результаты и умножает на объём ячейки.

## 4. Схема распараллеливания

**Разбиение данных на диапазоны.** Итерации по внешнему измерению (`num_splits`
значений) делятся между потоками. Число потоков определяется через
`std::thread::hardware_concurrency()`. Формула разбиения:

```text
chunk = num_splits / actual_threads
rem   = num_splits % actual_threads
start = tid * chunk + min(tid, rem)
end   = start + chunk + (tid < rem ? 1 : 0)
```

Первые `rem` потоков получают на одну итерацию больше. Диапазоны непрерывны и
не пересекаются.

**Что делает каждый поток.** Поток вызывает функцию `ThreadWorker`, которая
создаёт локальные буферы `idx` и `x_vec` (один раз), затем в цикле `[start, end)`
вызывает `ComputeSlice` для каждого среза внешнего измерения, накапливая
`thread_sum`.

**Локальные результаты.** Каждый поток пишет только в свой элемент массивов:

- `partial_sums[tid]` — частичная сумма (тип `double`)
- `error_flags[tid]` — флаг ошибки (тип `int`)

Массивы предаллоцированы в главном потоке до создания рабочих потоков. Потоки
не обращаются к чужим элементам — гонки исключены по построению.

**Порядок create → work → join.** Все потоки создаются в первом цикле, работают
параллельно, затем во втором цикле главный поток выполняет `join()` для каждого:

```cpp
// Создание — все потоки запускаются до первого join
for (int tid = 0; tid < actual_threads; ++tid) {
  threads[tid] = std::thread([...]() {
    ThreadWorker(...);
  });
}

// Join — только после создания всех потоков
for (auto &th : threads) {
  th.join();
}
```

Это принципиально: `join()` внутри цикла создания сериализовал бы работу.

**Синхронизация.** Отсутствует полностью — нет `atomic`, `mutex`,
`condition_variable`. Каждый поток работает с независимым диапазоном итераций и
пишет в свой элемент массива. Сбор результатов происходит в главном потоке
после `join()` всех рабочих потоков.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

**Изменения относительно SEQ:**

1. Цикл `while (true)` с `NextIndex` заменён на `num_splits` итераций по
   внешнему измерению, распределённых между `std::thread`.
2. Тело потока вынесено в функцию `ThreadWorker` для снижения когнитивной
   сложности (требование clang-tidy: threshold 15) и устранения проблемы
   с `std::vector<bool>` (proxy-объект несовместим с `std::ref`).
3. Лямбда захватывает ссылки на `partial_sums[tid]` и `error_flags[tid]`
   через init-capture (`&partial_sum = partial_sums[tid]`) и передаёт их
   в `ThreadWorker` — без `std::ref`/`std::cref`.
4. `ComputeSlice` — общая функция с OMP и TBB версиями.

**Карта поток → диапазон** (пример для `n = 10000`, 4 потока):

| tid | start | end   | итераций |
|-----|-------|-------|----------|
| 0   | 0     | 2500  | 2500     |
| 1   | 2500  | 5000  | 2500     |
| 2   | 5000  | 7500  | 2500     |
| 3   | 7500  | 10000 | 2500     |

**Схема объединения.** После `join()` главный поток проверяет `error_flags`
и суммирует `partial_sums` в простом цикле:

```cpp
for (int tid = 0; tid < actual_threads; ++tid) {
  if (error_flags[tid] != 0) return false;
}
double sum = 0.0;
for (int tid = 0; tid < actual_threads; ++tid) {
  sum += partial_sums[tid];
}
```

## 6. Проверка корректности

Результат STL-версии сравнивается с аналитическим значением интеграла с точностью
`eps = 1e-3` (тот же критерий, что и в SEQ). Набор функциональных тестов
идентичен SEQ: константная, линейная, квадратичная, тригонометрическая и
экспоненциальная функции. Все тесты пройдены.

**Отсутствие гонок** гарантируется тем, что каждый поток пишет только в свой
элемент `partial_sums[tid]` и `error_flags[tid]`. Чтение общих данных (`f`, `brd`,
`sz`) безопасно, так как они не модифицируются после создания потоков.

**Узкие места реализации:**

- Ограничение Docker-контейнера — контейнер имеет доступ только к 4 ядрам из 12
  доступных на хост-машине. При 8 потоках потоки конкурируют за 4 ядра, что
  приводит к переключениям контекста и падению эффективности до 47%.
- Накладные расходы create/join — создание объектов `std::thread` и их `join()`
  занимает порядка десятков микросекунд, что пренебрежимо мало на фоне ~1.4 с
  вычислений. Аллокация буферов `idx` и `x_vec` внутри `ThreadWorker` происходит
  один раз на поток.
- Вызов `std::function` — функция передаётся через указатель, компилятор не может
  встроить её напрямую. При `n = 10000` и `dim = 2` это 10⁸ вызовов через
  косвенный указатель.

## 9. Выводы

При 4 потоках STL даёт ускорение 3.76× на task_run (эффективность 94%) —
оптимальный результат для 4 ядер, доступных в Docker-контейнере. При 2 потоках
наблюдается сверхлинейное ускорение (3.71×, эффективность 185%), что объясняется
лучшим использованием кэша при разделении данных на две части. При 8 потоках
эффективность падает до 47%: потоки конкурируют за 4 доступных ядра контейнера.
Результаты сопоставимы с OMP и TBB, что подтверждает: для задач с равномерной
нагрузкой высокоуровневые фреймворки не дают принципиального
преимущества перед ручным управлением потоками через `std::thread`.
