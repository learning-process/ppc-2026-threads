## 1. Введение

Задача вычисления многомерного определённого интеграла методом средних
прямоугольников хорошо подходит для сравнения моделей параллелизма: основной
вычислительный цикл содержит `n^dim` полностью независимых итераций (для 2D
с `n = 10000` это 10⁸ вызовов функции), результат — скалярная сумма, а обмен
данными между потоками/процессами сводится к единственной операции редукции.
Это позволяет изолировать влияние параллельного runtime от алгоритмической
сложности синхронизации.

## 2. Единая постановка задачи

**Вход:**

- Подынтегральная функция `std::function<double(const std::vector<double>&)>`.
- Границы интегрирования `std::vector<std::pair<double, double>>` для каждого
  измерения.
- Число разбиений `int n` по каждой оси.

**Выход:** приближённое значение интеграла (`double`).

**Ограничения:** границы конечны, левая < правой, `n > 0`, функция задана,
значения функции конечны.

**Критерий корректности:** `|result - analytical| ≤ 1e-3`.

## 3. Единая методика эксперимента

**Окружение:**

- **CPU:** AMD Ryzen 9 9900X3D (**Cores:** 12, **Threads:** 24)
- **RAM:** 32Gb
- **OS:** Linux
- **Compiler:** gcc 14
- **build type:** Release

**Переменные окружения:**

- `OMP_NUM_THREADS` / `PPC_NUM_THREADS`: 2, 4, 8
- `PPC_NUM_PROC`: 2, 4 (для ALL)

**Тестовая задача:** `sin(x₁)·sin(x₂)` на `[0, π] × [0, π]`, `n = 10000`,
аналитическое значение = 4.0.

**Расчёт метрик:**

- Speedup = T(seq) / T(parallel)
- Efficiency = Speedup / NumProcs × 100%
- Для ALL: NumProcs = Ranks × Threads/rank

**Повторы:** каждая конфигурация запускалась в трёх прогонах (2, 4, 8 потоков).
Время из вывода фреймворка (значение после двоеточия, в секундах). Значения
стабильны между прогонами (разброс < 3%).

## 4. Сводка корректности

Все backend-ы (SEQ, OMP, TBB, STL, ALL) сравниваются с аналитическим значением
интеграла с точностью `eps = 1e-3`. Набор функциональных тестов единый:
константная, линейная, квадратичная, тригонометрическая и экспоненциальная
функции. Все тесты пройдены для всех backend-ов.

Ограничения применимости: ALL-версия требует запуска под MPI (`mpirun`); при
запуске без MPI работает как чистый OpenMP (один rank).

## 6. Интерпретация различий

**SEQ** — baseline 1.360 с (task_run). Вся работа в одном потоке, инкрементальный
обход через `NextIndex`. Стабильное время между прогонами подтверждает
надёжность baseline.

**OMP** — при 4 потоках ускорение 3.74× (эффективность 93%). Предаллокация
буферов по `omp_get_max_threads()` исключает аллокации в горячем цикле.
`schedule(static)` оптимален для равномерной нагрузки. Сверхлинейное ускорение
при 2 потоках (3.68×) объясняется кэш-эффектом. При 8 потоках на 4 ядрах
эффективность падает до 47%.

**TBB** — при 4 worker-ах ускорение 3.79× (эффективность 95%). `parallel_reduce`
с `blocked_range` и автоматическим grainsize. Аллокация буферов внутри лямбды
амортизируется по итерациям range. Work-stealing не срабатывает (нагрузка
равномерна), но и не создаёт overhead. Результат сопоставим с OMP.

**STL** — при 4 потоках ускорение 3.76× (эффективность 94%). Ручное разбиение
данных, `ThreadWorker` как отдельная функция, независимые `partial_sums[tid]`.
Отсутствие синхронизации (нет mutex/atomic). `join()` вызывается после запуска
всех потоков. Результат сопоставим с OMP и TBB — для равномерной нагрузки
высокоуровневые фреймворки не дают преимущества.

**ALL** — лучшее абсолютное время (~0.366 с) при 2R×4T и 4R×4T, сопоставимо
с чистыми backend-ами. Однако эффективность в пересчёте на total_workers
(23–46%) значительно ниже, так как все workers делят 4 ядра контейнера.
Конфигурация 4R×2T (0.898 с) хуже 2R×4T (0.367 с) при одинаковых 8 workers:
4 MPI-процесса на 4 ядрах оставляют каждому rank по одному ядру, и 2
OpenMP-потока внутри одного ядра конкурируют. Гибридная схема оправдана на
распределённых системах с выделенными ядрами на каждый rank.

## 8. Заключение

**Лучшая версия:** для данной задачи на одном узле с 4 ядрами все три чистых
backend-а (OMP, TBB, STL) дают практически одинаковый результат (~0.36 с,
ускорение ~3.7×, эффективность ~94%). Выбор между ними — вопрос удобства:
OMP — минимум изменений кода, TBB — автоматическая балансировка, STL — нет
внешних зависимостей.

**Ограничения сравнения:**

- Docker-контейнер ограничен 4 ядрами — масштабируемость за пределами 4 workers
  не проверялась на реальном железе.
- Подынтегральная функция `sin·sin` вычислительно лёгкая — на тяжёлых функциях
  соотношение backend-ов может измениться.
- `std::function` не инлайнится — при замене на шаблонную функцию абсолютное
  время уменьшится для всех backend-ов.

**Возможные улучшения:**

- Шаблонная подынтегральная функция вместо `std::function` для инлайнинга.
- Тестирование на реальном кластере для ALL-версии.
- Адаптивный grainsize в TBB для неравномерных функций.

## 10. Приложение

**Ядро вычислений (общее для OMP, TBB, STL, ALL):**

```cpp
bool ComputeSlice(const std::function<double(const std::vector<double> &)> &f,
                  const std::vector<std::pair<double, double>> &brd,
                  const std::vector<double> &h,
                  std::size_t outer_dim, int64_t inner_cells,
                  int num_splits, int outer,
                  std::vector<int> &idx, std::vector<double> &x,
                  double &local_sum) {
  for (std::size_t i = 0; i < outer_dim; ++i) idx[i] = 0;
  idx[outer_dim] = outer;
  x[outer_dim] = brd[outer_dim].first
               + ((static_cast<double>(outer) + 0.5) * h[outer_dim]);

  for (int64_t cell = 0; cell < inner_cells; ++cell) {
    for (std::size_t i = 0; i < outer_dim; ++i)
      x[i] = brd[i].first + ((static_cast<double>(idx[i]) + 0.5) * h[i]);

    const double fx = f(x);
    if (!std::isfinite(fx)) return false;
    local_sum += fx;

    for (std::size_t pos = 0; pos < outer_dim; ++pos) {
      if (++idx[pos] < num_splits) break;
      idx[pos] = 0;
    }
  }
  return true;
}
```
