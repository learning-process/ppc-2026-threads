## 1. Контекст

Гибридная версия объединяет два уровня параллелизма: MPI распределяет работу
между процессами (потенциально на разных узлах кластера), а OpenMP параллелит
вычисления внутри каждого процесса на несколько потоков. Это позволяет
масштабироваться как по числу узлов, так и по числу ядер внутри узла.
Для задачи численного интегрирования, где `n^dim` ячеек полностью независимы,
двухуровневая схема естественна и не требует обмена промежуточными данными.

## 2. Постановка задачи

Вычислить приближённое значение многомерного определённого интеграла методом
средних прямоугольников. Входные данные: подынтегральная функция, границы
интегрирования по каждому измерению, число разбиений `n`.

## 3. Базовый алгоритм

Последовательная версия строит равномерную сетку из `n^dim` ячеек, перебирает их
с помощью инкрементального многомерного индекса (`NextIndex`), вычисляет значение
функции в центре каждой ячейки, суммирует результаты и умножает на объём ячейки.

## 4. Межпроцессная схема

**Роли rank-ов.** Все rank-и равноправны — нет выделенного «мастера». Каждый
процесс самостоятельно вычисляет границы своего диапазона по внешнему измерению
и выполняет интегрирование на своём подмножестве срезов.

**Распределение данных.** Итерации по внешнему измерению (`num_splits` значений)
делятся между `num_procs` процессами:

```text
mpi_chunk = num_splits / num_procs
mpi_rem   = num_splits % num_procs
mpi_start = rank * mpi_chunk + min(rank, mpi_rem)
mpi_end   = mpi_start + mpi_chunk + (rank < mpi_rem ? 1 : 0)
```

Первые `mpi_rem` процессов получают на одну итерацию больше. Входные данные
(функция, границы, `n`) доступны всем rank-ам через общий вход задачи —
`Scatter`/`Broadcast` входных данных не требуется.

**MPI-вызовы:**

- `MPI_Comm_rank(MPI_COMM_WORLD, &rank)` — каждый процесс узнаёт свой ранг
  для вычисления своего диапазона `[mpi_start, mpi_end)`.
- `MPI_Comm_size(MPI_COMM_WORLD, &num_procs)` — общее число процессов для
  формулы разбиения.
- `MPI_Allreduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, ...)` —
  суммирование частичных интегралов со всех процессов. Используется именно
  `Allreduce` (а не `Reduce`), чтобы результат был доступен на всех rank-ах
  без дополнительного `Broadcast`.
- `ppc::util::IsUnderMpirun()` — проверка, запущен ли процесс под MPI. Если
  нет — MPI-вызовы пропускаются, работает только OpenMP-уровень.

**Барьеры.** Явный `MPI_Barrier` не используется. `MPI_Allreduce` содержит
неявную синхронизацию: каждый процесс блокируется до тех пор, пока все процессы
не предоставят свои локальные суммы.

## 5. Внутрипроцессная схема

**Thread-технология:** OpenMP.

**Разделение работы.** Внутри каждого rank-а диапазон `[mpi_start, mpi_end)`
параллелится по потокам через `#pragma omp parallel for`:

```cpp
#pragma omp parallel for schedule(static) num_threads(max_threads) \
    reduction(+ : local_sum) shared(error_flag) default(none)      \
    shared(funcx, brd, h_vec, outer_dim, inner_cells, num_splits, mpi_start, mpi_end, all_idx, all_x)
for (int outer = mpi_start; outer < mpi_end; ++outer) {
  // ...
  ComputeSlice(funcx, brd, h_vec, ..., outer, all_idx[tid], all_x[tid], slice_sum);
  local_sum += slice_sum;
}
```

**Буферы:** `all_idx` и `all_x` предаллоцированы по `max_threads` до
параллельного региона. Каждый поток обращается к `all_idx[tid]`/`all_x[tid]` —
гонки исключены.

**Объединение локальных результатов:**

1. Внутри процесса: `reduction(+ : local_sum)` в OpenMP складывает частичные
   суммы потоков.
2. Между процессами: `MPI_Allreduce` с `MPI_SUM` суммирует `local_sum` всех
   rank-ов в `global_sum`.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

**Расположение MPI-вызовов:**

- `MPI_Comm_rank` / `MPI_Comm_size` — в начале `RunImpl()`, до вычислений.
- `MPI_Allreduce` — после OpenMP-цикла, перед записью результата.

**Потенциальные узкие места:**

- `MPI_Allreduce` — блокирующий вызов, время ожидания определяется самым
  медленным rank-ом. При равномерном разбиении и одинаковой нагрузке задержка
  минимальна.
- Аллокация `all_idx`/`all_x` — выполняется один раз на rank, стоимость
  амортизируется по `10⁸ / (num_procs)` вызовов функции.
- `std::function` — вызов через виртуальный указатель, не инлайнится.

## 7. Проверка корректности

Результат ALL-версии сравнивается с аналитическим значением интеграла с точностью
`eps = 1e-3` (тот же критерий, что и в SEQ). Набор функциональных тестов
идентичен SEQ: константная, линейная, квадратичная, тригонометрическая и
экспоненциальная функции. Все тесты пройдены.

**Согласованность между rank-ами:** `MPI_Allreduce` гарантирует, что все
rank-и получают одинаковый `global_sum`. Результат `GetOutput()` идентичен
на всех процессах.

**Узкие места реализации:**

- Ограничение Docker-контейнера — контейнер имеет доступ только к 4 ядрам из 12
  доступных на хост-машине. Лучшее абсолютное время (~0.37 с) достигается при
  конфигурациях 2×4 и 4×4, где суммарное число worker-ов уже превышает число
  ядер, но каждый rank получает достаточно потоков для полной загрузки.
- Oversubscription — при 4 ranks × 8 threads = 32 workers на 4 ядрах расходы
  на переключение контекста между MPI-процессами и OpenMP-потоками растут,
  а абсолютное время практически не улучшается.
- Конфигурация 4×2 (8 workers) показывает худший результат (0.898 с), чем 2×4
  (8 workers, 0.367 с): при 4 MPI-процессах на 4 ядрах каждый rank получает
  ~1 ядро, и 2 OpenMP-потока внутри одного ядра только мешают друг другу.
  При 2 rank-ах каждый получает ~2 ядра и 4 потока эффективно их загружают.

## 10. Выводы

Лучший абсолютный результат — 0.366–0.367 с — достигается при конфигурациях
2 ranks × 4 threads и 4 ranks × 4 threads, что сопоставимо с чистым OMP
(0.364 с), TBB (0.359 с) и STL (0.362 с). Однако эффективность в пересчёте
на total_workers составляет 23–46%, так как все workers делят 4 ядра
контейнера. Ключевое наблюдение: число потоков на rank важнее числа rank-ов.
Конфигурация 2×4 (8 workers) и 4×4 (16 workers) дают одинаковое время,
а 4×2 (8 workers) — вдвое хуже. Гибридная схема MPI + OpenMP оправдана на
распределённых системах, где каждый rank работает на отдельном узле с
собственными ядрами. На одном узле с 4 ядрами чистый OMP/TBB/STL дают тот же
результат при лучшей эффективности (93–96%).
