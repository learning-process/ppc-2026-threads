## 1. Контекст

Из последовательной версии (SEQ) в OpenMP переносится основной вычислительный
цикл — обход ячеек многомерной сетки с суммированием значений подынтегральной
функции. Этот цикл содержит `n^dim` независимых итераций (для 2D с `n = 10000`
это 10⁸ вызовов функции), что делает его естественным кандидатом для
параллелизации по данным.

## 2. Постановка задачи

Вычислить приближённое значение многомерного определённого интеграла методом
средних прямоугольников. Входные данные: подынтегральная функция, границы
интегрирования по каждому измерению, число разбиений `n`.

## 3. Базовый алгоритм

Последовательная версия строит равномерную сетку из `n^dim` ячеек, перебирает их
с помощью инкрементального многомерного индекса (`NextIndex`), вычисляет значение
функции в центре каждой ячейки, суммирует результаты и умножает на объём ячейки.

## 4. Схема распараллеливания

**Область параллелизации:** цикл по внешнему измерению сетки (`n` итераций).
Каждая итерация внешнего цикла обрабатывает один «срез» — все `n^(dim-1)`
внутренних ячеек для фиксированного значения последнего индекса.

**Директива:**

```cpp
#pragma omp parallel for schedule(static) reduction(+ : sum) \
    shared(error_flag) default(none) \
    shared(f, brd, h, outer_dim, inner_cells, num_splits, all_idx, all_x)
```

**Переменные:**

- `shared`: входные данные (`f`, `brd`, `h`), параметры сетки (`outer_dim`,
  `inner_cells`, `num_splits`), предаллоцированные буферы (`all_idx`, `all_x`),
  флаг ошибки (`error_flag`).
- `private` (неявно): `outer` (переменная цикла), `tid`, `local_sum`.
- `reduction(+ : sum)`: частичные суммы потоков складываются автоматически.

**Schedule:** `static` — итерации распределяются поровну между потоками.
Выбран потому, что нагрузка на каждый срез одинакова (одинаковое число внутренних
ячеек).

**Устранение гонок:**

- Каждый поток обращается к своим буферам `all_idx[tid]` и `all_x[tid]`,
  предаллоцированным по числу потоков до входа в параллельный регион.
- `error_flag` — запись `true` без синхронизации безопасна (булево присваивание
  атомарно на x86, ложные срабатывания не критичны).
- Суммирование через `reduction` — без явных `atomic`/`critical`.

**Барьеры:** неявный барьер в конце `#pragma omp parallel for`.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

**Изменения относительно SEQ:**

1. Цикл `while (true)` с `NextIndex` заменён на `for` по внешнему измерению
   с вложенным вызовом `ComputeSlice` для внутренних измерений.
2. Вычислительное ядро вынесено в функцию `ComputeSlice` для снижения
   когнитивной сложности (требование clang-tidy: threshold 15).
3. Structured bindings (`func`, `borders`, `n`) скопированы в обычные переменные
   (`f`, `brd`, `num_splits`), так как clang не поддерживает захват structured
   bindings в OpenMP-регионах.
4. Буферы `all_idx` и `all_x` предаллоцированы по `omp_get_max_threads()` до
   параллельного региона — аллокации внутри горячего цикла отсутствуют.

**Устранённые риски гонок:**

- Разделение буферов по `tid` вместо общих `idx`/`x`.
- `reduction` вместо ручного суммирования в общую переменную.

## 6. Проверка корректности

Результат OMP-версии сравнивается с аналитическим значением интеграла с точностью
`eps = 1e-3` (тот же критерий, что и в SEQ). Набор функциональных тестов
идентичен SEQ: константная, линейная, квадратичная, тригонометрическая и
экспоненциальная функции. Все тесты пройдены.

**Узкие места реализации:**

- Ограничение Docker-контейнера — контейнер имеет доступ только к 4 ядрам из 12
  доступных на хост-машине. При 8 потоках потоки конкурируют за 4 ядра, что
  приводит к переключениям контекста и потере производительности.
- Вызов `std::function` — функция передаётся через указатель, компилятор не может
  встроить её напрямую. При `n = 10000` и `dim = 2` это 10⁸ вызовов через
  косвенный указатель.
- Накладные расходы OpenMP — при создании числа потоков, превышающего доступные
  ядра контейнера, расходы на синхронизацию и переключение контекста превышают
  выигрыш от параллелизации.

## 9. Выводы

При 4 потоках OpenMP даёт ускорение 3.74× с эффективностью 93% — оптимальный
результат для 4 ядер, доступных в Docker-контейнере. При 2 потоках наблюдается
сверхлинейное ускорение (3.68×, эффективность 184%), что объясняется лучшим
использованием кэша при разделении данных на две части. При 8 потоках
эффективность падает до 47%: потоки конкурируют за 4 доступных ядра контейнера,
и накладные расходы на переключение контекста съедают выигрыш. Версия с OMP
наиболее эффективна при числе потоков, равном числу ядер контейнера.
