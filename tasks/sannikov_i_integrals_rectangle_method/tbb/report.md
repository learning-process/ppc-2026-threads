## 1. Контекст

Задача вычисления многомерного интеграла методом прямоугольников сводится к
суммированию значений функции в центрах `n^dim` ячеек — классическая операция
редукции по большому диапазону. TBB выбрана как альтернатива OpenMP, поскольку
`tbb::parallel_reduce` предоставляет задачно-ориентированную декомпозицию с
автоматическим work-stealing, что позволяет эффективно балансировать нагрузку
без ручного управления потоками.

## 2. Постановка задачи

Вычислить приближённое значение многомерного определённого интеграла методом
средних прямоугольников. Входные данные: подынтегральная функция, границы
интегрирования по каждому измерению, число разбиений `n`.

## 3. Базовый алгоритм

Последовательная версия строит равномерную сетку из `n^dim` ячеек, перебирает их
с помощью инкрементального многомерного индекса (`NextIndex`), вычисляет значение
функции в центре каждой ячейки, суммирует результаты и умножает на объём ячейки.

## 4. Схема распараллеливания

**Примитив:** `tbb::parallel_reduce` — выбран потому, что задача сводится к
суммированию (редукции) частичных результатов. В отличие от `parallel_for`,
`parallel_reduce` позволяет накапливать частичную сумму внутри каждой подзадачи
и автоматически объединять результаты через `std::plus<>`.

**Диапазон работы:** `tbb::blocked_range<int>(0, num_splits)` — итерации по
внешнему измерению сетки. Каждая итерация `outer` обрабатывает один «срез» из
`n^(dim-1)` внутренних ячеек через вызов `ComputeSlice`.

**Grainsize:** оставлен по умолчанию (автоматический). TBB runtime самостоятельно
разбивает `blocked_range` пополам до достижения порога. При `n = 10000` итераций
внешнего цикла и стоимости каждой итерации ~10000 вызовов функции (для 2D)
автоматический grainsize обеспечивает достаточно крупные чанки, чтобы overhead
от создания/кражи задач был пренебрежимо мал.

**Partitioner:** по умолчанию (`auto_partitioner`). Он рекурсивно делит диапазон
и прекращает деление, когда дальнейшее дробление не улучшает балансировку. Для
данной задачи нагрузка на каждый срез одинакова, поэтому `auto_partitioner`
работает оптимально.

**Контроль конкуренции:** фактическое число worker-потоков ограничивается через
`tbb::global_control::max_allowed_parallelism`, которое задаётся фреймворком
проекта (PPC_NUM_THREADS). Это ограничивает именно число worker-потоков в TBB
arena, а не число системных потоков.

```cpp
double sum = tbb::parallel_reduce(
    tbb::blocked_range<int>(0, num_splits),
    0.0,
    [&](const tbb::blocked_range<int> &range, double partial_sum) -> double {
      std::vector<int> idx(dim, 0);
      std::vector<double> x(dim);

      for (int outer = range.begin(); outer < range.end(); ++outer) {
        double local_sum = 0.0;
        ComputeSlice(f, brd, h, outer_dim, inner_cells,
                     num_splits, outer, idx, x, local_sum);
        partial_sum += local_sum;
      }
      return partial_sum;
    },
    std::plus<>());
```

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

**Изменения относительно SEQ:**

1. Цикл `while (true)` с `NextIndex` заменён на `tbb::parallel_reduce` по
   внешнему измерению с вложенным вызовом `ComputeSlice`.
2. Вычислительное ядро вынесено в функцию `ComputeSlice` (общая с OMP-версией)
   для снижения когнитивной сложности.
3. Structured bindings скопированы в обычные переменные для корректного захвата
   в лямбде.

**Локальные аккумуляторы:** каждая подзадача (вызов лямбды) создаёт локальные
`idx` и `x` и накапливает `partial_sum`. При `blocked_range` размером в
несколько сотен итераций аллокация двух небольших векторов (размер `dim`)
происходит один раз на подзадачу и амортизируется по всем итерациям внутри range.

**Объединение результатов:** `std::plus<>()` передаётся как reduce-операция —
TBB вызывает её для объединения `partial_sum` из разных подзадач.

**Обработка ошибок:** если `ComputeSlice` обнаруживает не-конечное значение
функции, выставляется shared-флаг `error_flag`, остальные подзадачи досрочно
завершаются.

## 6. Проверка корректности

Результат TBB-версии сравнивается с аналитическим значением интеграла с точностью
`eps = 1e-3` (тот же критерий, что и в SEQ). Набор функциональных тестов
идентичен SEQ: константная, линейная, квадратичная, тригонометрическая и
экспоненциальная функции. Все тесты пройдены.

## 8. Результаты

**task_run**:

| Mode | Workers | Time, s | Speedup | Efficiency |
|------|---------|---------|---------|------------|
| seq  | 1       | 1.360   | 1.00    | N/A        |
| tbb  | 2       | 0.688   | 1.98    | 99%        |
| tbb  | 4       | 0.359   | 3.79    | 95%        |
| tbb  | 8       | 0.368   | 3.70    | 46%        |

**task_pipeline**:

| Mode | Workers | Time, s | Speedup | Efficiency |
|------|---------|---------|---------|------------|
| seq  | 1       | 1.364   | 1.00    | N/A        |
| tbb  | 2       | 0.680   | 2.01    | 100%       |
| tbb  | 4       | 0.357   | 3.82    | 96%        |
| tbb  | 8       | 0.366   | 3.73    | 47%        |

**Узкие места реализации:**

- Ограничение Docker-контейнера — контейнер имеет доступ только к 4 ядрам из 12
  доступных на хост-машине. При 8 worker-ах потоки конкурируют за 4 ядра, что
  приводит к переключениям контекста и падению эффективности до 46–47%.
- Аллокация `std::vector<int>` и `std::vector<double>` размера `dim` при каждом
  вызове лямбды `parallel_reduce`. При автоматическом grainsize каждая подзадача
  обрабатывает достаточно итераций, чтобы стоимость аллокации была пренебрежима.
- Балансировка нагрузки — нагрузка на каждый срез одинакова (`n^(dim-1)` вызовов
  функции), поэтому work-stealing практически не срабатывает, задачи
  распределяются равномерно при первичном разбиении `blocked_range`.

## 9. Выводы

При 4 worker-ах TBB даёт ускорение 3.79× на task_run (эффективность 95%) и
3.82× на pipeline (эффективность 96%) — оптимальный результат для 4 ядер,
доступных в Docker-контейнере. При 2 worker-ах ускорение близко к идеальному
(~2×, эффективность ~100%). При 8 worker-ах эффективность падает до 46–47%:
потоки конкурируют за 4 доступных ядра контейнера. Для данной задачи с
равномерной нагрузкой TBB и OpenMP показывают сопоставимые результаты.
`parallel_reduce` с `std::plus` — наиболее естественный примитив, так как задача
сводится к суммированию независимых вкладов.
