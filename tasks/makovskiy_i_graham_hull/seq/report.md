## 2. Постановка задачи

- Вход: `std::vector<Point>` (двумерные точки с `double x, y`).
- Выход: вершины выпуклой оболочки в порядке обхода против часовой стрелки,
  начиная с самой нижней (при равенстве - самой левой) точки.
- Ограничения:
  - `n < 3` - выход равен входу (выпуклая оболочка совпадает с самим множеством).
  - Если после фильтрации коллинеарных точек осталось `< 3` вершин - выход и есть
    результат (вырожденный случай - отрезок или одиночная точка).
- Толерантность по cross product / координатам - `1e-9`.

## 3. Базовый алгоритм

Реализован классический Graham scan:

1. Найти базовую точку `p0` с минимальным `y` (при равенстве - минимальным `x`)
   (`FindMinPointIndex`).
2. Переставить её на нулевую позицию.
3. Отсортировать остальные точки по полярному углу относительно `p0`. Сравнение
   через знак векторного произведения `CrossProduct(p0, a, b)`; при коллинеарности -
   по квадрату расстояния (ближайшая первой).
4. `FilterPoints`: схлопнуть подряд идущие коллинеарные точки относительно `p0` -
   оставить только дальнюю в каждой группе с одинаковым углом.
5. `BuildHull`: проход стеком - пока поворот `(hull[-2], hull[-1], next)` не строго
   левый (`cross <= 1e-9`), снимаем вершину; затем кладём `next`.

Сложность по времени: $O(n \log n)$ - доминирует сортировка. Поиск минимума,
фильтрация и стековый проход - линейные по `n`.

Сложность по памяти: $O(n)$ - буфер отсортированных точек плюс стек оболочки.

Инвариант: после построения каждая тройка соседних вершин в `hull` образует
строго левый поворот (`cross > 1e-9`), что и есть критерий выпуклости.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

- `ValidationImpl` возвращает `true` - алгоритм безопасен для любого корректно
  сконструированного `std::vector<Point>`, включая пустой и вырожденные наборы.
- `PreProcessingImpl` пуст: вход уже копируется в `GetInput()` через конструктор,
  никаких преобразований данных не требуется.
- `RunImpl` обрабатывает граничные случаи (`n < 3`, `filtered.size() < 3`) до
  основного прохода и возвращает результат напрямую.
- `PostProcessingImpl` пуст - результат уже лежит в `GetOutput()` после `Run`.

Ключевой фрагмент стекового прохода:

```cpp
// File: seq/src/ops_seq.cpp
std::vector<Point> BuildHull(const std::vector<Point> &filtered) {
  std::vector<Point> hull;
  hull.push_back(filtered[0]);
  hull.push_back(filtered[1]);
  hull.push_back(filtered[2]);
  for (size_t i = 3; i < filtered.size(); ++i) {
    while (hull.size() > 1 &&
           CrossProduct(hull[hull.size() - 2], hull.back(), filtered[i]) <= 1e-9) {
      hull.pop_back();
    }
    hull.push_back(filtered[i]);
  }
  return hull;
}
```

Условие `<= 1e-9` отсекает не только правые повороты, но и коллинеарные тройки -
последние нежелательны как промежуточные вершины оболочки. Цикл `while`
амортизированно стоит $O(n)$ суммарно: каждая точка кладётся и удаляется
не более одного раза.

## 5. Проверка корректности

Метод сравнения: тест-каркас курса (`ppc::util::BaseRunFuncTests`) вызывает
`CheckTestOutputData` после `Run`/`PostProcessing`. В данной задаче проверка
делает сравнение по размеру выходной оболочки (см.
`tests/functional/main.cpp`) - для всех 9 кейсов известное ожидаемое количество вершин.
Этого достаточно, потому что на использованных входах размер оболочки
однозначно определяет правильность ответа (квадрат -> 4 угла, треугольник ->
3 вершины, вырожденный отрезок -> 2 точки и т.п.).

Функциональные тесты (`tests/functional/main.cpp`) покрывают:

| ID | Кейс | Ожидаемый размер оболочки |
| ---: | ------ | --------------------------- |
| 1 | квадрат с внутренней точкой | 4 |
| 2 | треугольник | 3 |
| 3 | коллинеарные точки на рёбрах квадрата | 4 |
| 4 | пустой массив | 0 |
| 5 | две точки | 2 |
| 6 | диагональ (5 коллинеарных) | 2 |
| 7 | вертикаль (5 коллинеарных) | 2 |
| 8 | отрицательные координаты с внутренней точкой | 4 |
| 9 | сетка 60x55 = 3300 точек | 4 (уголки) |

Кейсы 4-7 проверяют именно вырожденные ситуации (пусто, отрезок,
коллинеарная линия), которые в текущей реализации выходят на ранние возвраты.
Кейс 9 проверяет, что алгоритм держит увеличенный объём и корректно
обрабатывает коллинеарность внутри плотной сетки.

Все 9 тестов пройдены SEQ-реализацией (gtest filter `*MakovskiyI*seq_enabled*`).

SEQ-версия получилась компактной и без побочных эффектов, поэтому цифры
OMP/TBB/STL/ALL дальше можно читать именно как ускорение параллельного

Параллелить в этой задаче имеет смысл прежде всего сортировку, на неё
приходится около 80% времени. `FindMin` и `Filter` тоже линейные, но их
вклад в общее время мал. Стековый проход `BuildHull` параллельный
напрямую не сделать: итерация `i` зависит от состояния стека после
итерации `i-1`, и обойти это можно только сменой алгоритма на
divide-and-conquer. В рамках курса я оставил этот шаг последовательным
во всех реализациях.
