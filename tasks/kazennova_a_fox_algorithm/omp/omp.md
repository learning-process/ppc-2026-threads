## 1. Контекст

Параллельная реализация умножения плотных квадратных матриц на основе
алгоритма Фокса с использованием OpenMP. Последовательная версия для
размера 700×700 была очень медленной, поэтому
распараллеливание необходимо. OpenMP позволяет эффективно распараллелить
независимые блочные умножения на общей памяти.

## 2. Постановка задачи

Решение задачи C = A·B, где A, B — квадратные матрицы размера n×n
(вещественные числа). Ограничения: n делится на выбранный размер блока.
Критерий корректности: совпадение результата с последовательной версией (SEQ)
в пределах машинной точности.
SEQ-версия служит baseline для производительности и верификации.

## 3. Базовый алгоритм

Алгоритм Фокса (блочное умножение с циклической передачей блоков):

1. Разбить A и B на блоки размера bs×bs, q = n / bs.
2. Для каждого шага step = 0 … q-1:
  Каждый блок C_ij накапливает произведение A_i,k · B_k,j,
  где k = (i+step) mod q.
3. Собрать C из блоков.

время O(n^3), память O(n^2).

## 4. Схема распараллеливания

Параллельная область: внутренние циклы по i и j.
В реализации директива
`#pragma omp parallel for default(none) shared(step, block_elements)`
применена к циклу по i. Каждый поток выполняет вложенный цикл по j,
вычисляет индексы блоков A_i,k, B_k,j и вызывает MultiplyBlock для
накопления результата в C_i,j.

Анализ переменных:

- shared(step, block_elements) – все потоки читают одни и те же значения.
- private по умолчанию: i, j, k, индексы блоков.
- Массив c_blocks_ записывается по непересекающимся индексам
(каждый поток обрабатывает свою уникальную
пару (ij)), поэтому reduction или critical не нужны.

Выбор schedule: по умолчанию schedule(static). Итерации цикла по i
независимы и имеют одинаковую вычислительную нагрузку (каждая итерация
умножает q блоков), поэтому статическое распределение оптимально.

Синхронизация:

- Неявный барьер в конце parallel for гарантирует завершение всех
блоков текущего шага перед переходом к следующему step.
- Отсутствие гонок данных проверено с помощью ThreadSanitizer.

Ключевой фрагмент (матрично-векторное умножение):

```cpp
#pragma omp parallel for default(none) shared(step, block_elements)
for (int i = 0; i < block_count_; ++i) {
    for (int j = 0; j < block_count_; ++j) {
        int k = (i + step) % block_count_;
        size_t a_idx = ((i * block_count_) + k) * block_elements;
        size_t b_idx = ((k * block_count_) + j) * block_elements;
        size_t c_idx = ((i * block_count_) + j) * block_elements;
        MultiplyBlock(a_idx, b_idx, c_idx, block_size_);
    }
}
```

## 5. Детали реализации

Файлы:

- kazennova_a_fox_algorithm/omp/include/ops_omp.hpp
- kazennova_a_fox_algorithm/omp/src/ops_omp.cpp

Основные компоненты:

- ChooseBlockSize(n) – выбирает наибольший делитель n, не превышающий √n, как размер блока.
- DecomposeMatrix / AssembleMatrix – перевод матрицы в блочное
  представление и обратно.
- MultiplyBlock – умножение двух блоков bs×bs с накоплением в блок результата.
- RunImpl() – реализует алгоритм Фокса с OpenMP-параллелизмом
  (именно здесь находится директива parallel for).

Изменения относительно SEQ:

- В цикле по step внутренние циклы по i и j заменены на один цикл по i с директивой parallel for.
- Все остальные части (разбиение на блоки, сборка результата) остались без изменений.

- Прогон с ThreadSanitizer – гонок данных не обнаружено.

## 9. Выводы

OpenMP-реализация алгоритма Фокса корректна и даёт ускорение до 2.7× на 2 потоках.
При 4 потоках эффективность падает до 47% из-за накладных расходов на синхронизацию
и малого размера блоков (31×31). Для масштабирования нужны более крупные матрицы
или динамическое распределение итераций.
