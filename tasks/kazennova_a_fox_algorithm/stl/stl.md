## 1. Контекст

Параллельная реализация умножения плотных квадратных матриц на основе
алгоритма Фокса с ручным управлением потоками через `std::thread`.
В отличие от высокоуровневых библиотек (OpenMP, TBB), здесь потоки создаются
явно, работа распределяется через атомарный счётчик, а синхронизация
отсутствует благодаря независимости обрабатываемых блоков.

## 2. Постановка задачи

Решение задачи C = A·B, где A, B — квадратные матрицы размера n×n
(вещественные числа). Реализация с блоком 64 и std::min не требует деления без остатка.
Критерий корректности: совпадение результата с последовательной версией (SEQ)
в пределах машинной точности.
SEQ-версия служит baseline для производительности и верификации.

## 3. Базовый алгоритм

Алгоритм Фокса (блочное умножение с циклической передачей блоков):

1. Разбить A и B на блоки размера bs×bs.
   Число блоков: q_i = ceil(n/bs), q_j, q_k.
2. Для каждого блока результата C_{i,j}:
   - Для каждого k = 0 … q_k-1:
     - Взять блок A_{i,k} и блок B_{k,j}.
     - Выполнить умножение блоков с накоплением в C_{i,j}.
3. Результат накапливается в C в процессе умножения блоков.

время O(n^3), память O(n^2).

## 4. Схема распараллеливания

Архитектура: пул потоков, создаваемый на каждый вызов RunImpl().
Число потоков равно ppc::util::GetNumThreads() (или hardware_concurrency()).
Потоки не переиспользуются между вызовами.

Разбиение данных: все блоки результата (i,j) обрабатываются независимо.
Для распределения работы используется атомарный счётчик next_block_idx,
инкрементируемый каждым потоком. Каждый поток захватывает индекс блока,
загружает соответствующие блоки A и B для всех k и выполняет умножение.

Синхронизация:

- std::atomic обеспечивает безопасное распределение индексов без блокировок.
- Запись в разные элементы матрицы C ведётся по непересекающимся индексам
  (каждый поток отвечает за свои блоки), поэтому гонки данных отсутствуют.
- std::thread::join() в конце дожидается завершения всех потоков.

Отличие от наивного std::thread: динамическое распределение блоков через
атомарный счётчик автоматически балансирует нагрузку даже при неравномерной
обработке блоков (например, из-за неполных последних блоков). Это эффективнее
статического разбиения, но создание потоков на каждом вызове RunImpl()
даёт дополнительные накладные расходы.

## 5. Детали реализации

Файлы:

- kazennova_a_fox_algorithm/stl/include/ops_stl.hpp
- kazennova_a_fox_algorithm/stl/src/ops_stl.cpp

Основные компоненты:

- kBlockSize = 64 – фиксированный размер блока.
- GetBlock() – извлекает блок из матрицы с учётом границ (последний блок
  может быть неполным).
- MultiplyBlock() – умножает два блока и добавляет результат в нужное
  место матрицы C. Внутри – тройной вложенный цикл по i, j, kk.
- RunImpl() – создаёт потоки, распределяет блоки через атомарный счётчик
  next_block_idx. Каждый поток обрабатывает блоки (i,j) последовательно:
  для каждого k загружает блоки A и B и вызывает MultiplyBlock.

Изменения относительно SEQ:

- Вместо тройного вложенного цикла по строкам и столбцам – блочная структура.
- Параллелизм на уровне независимых блоков (i,j).
- Ручное управление потоками и синхронизация через std::atomic.
- Потоки не переиспользуются между вызовами (создаются заново).

## 6. Проверка корректности

Корректность проверена следующими способами:

- Функциональные тесты для размеров от 1 до 128, включая граничные случаи.
- Сравнение с SEQ на случайных матрицах размера 700 при 2 и 4 потоках
- Проверка на размерах, не кратных 64 (например, n = 700) – корректная
  обработка неполных блоков через min() в GetBlock и MultiplyBlock.

Все тесты пройдены, гонок данных не обнаружено.

Анализ:

- Ускорение на 2 и 4 потоках не превышает 1.3×, что значительно ниже
  ожидаемого. Причина – накладные расходы на создание потоков при каждом
  вызове `RunImpl()` и атомарный счётчик, а также малый размер задачи
  (n=700), при котором число блоков (11×11 = 121) недостаточно для
  эффективной параллельной загрузки.
- Блочная структура даёт небольшой выигрыш в локальности данных,
  но не компенсирует накладные расходы на управление потоками.

## 9. Выводы

STL-реализация алгоритма Фокса с ручным управлением потоками и атомарным
счётчиком корректна, но на размере матрицы 700×700 даёт ускорение не выше
1.3×.
Основные причины низкой эффективности:

- создание потоков при каждом вызове `RunImpl()` (вместо переиспользуемого пула);
- малый размер задачи (всего 121 блок), не позволяющий окупить накладные расходы;
- атомарный счётчик вносит дополнительную задержку при распределении блоков.
