## 1. Контекст

Гибридная параллельная реализация умножения плотных квадратных матриц,
использующая два уровня параллелизма:

- Межпроцессный уровень (MPI): распределение строк результирующей
матрицы C между MPI-процессами.
- Внутрипроцессный уровень (std::thread): блочное умножение
с динамическим распределением блоков через атомарный счётчик.

Такой подход позволяет задействовать кластеры с распределённой памятью,
эффективно используя многоядерность внутри каждого узла.

## 2. Постановка задачи

Решение задачи C = A·B, где A, B — квадратные матрицы размера n×n
(вещественные числа). Ограничения: n делится на выбранный размер блока.
Критерий корректности: совпадение результата с последовательной версией (SEQ)
в пределах машинной точности.
SEQ-версия служит baseline для производительности и верификации.

## 3. Базовый алгоритм

В основе реализации лежит блочное умножение матриц

1. Разбить A и B на блоки размера bs×bs (здесь bs = 64).
2. Каждый процесс получает свою вертикальную полосу строк матрицы C
3. Для каждого блока (i,j) в своей полосе:
   - Для каждого k = 0 … q_k-1:
     - Взять блок A_{i,k} из глобальной матрицы A.
     - Взять блок B_{k,j} из глобальной матрицы B.
     - Выполнить умножение блоков с накоплением в локальную часть C.
4. Процесс 0 собирает все локальные части через MPI_Gatherv
   и формирует итоговую матрицу C.

время O(n^3), память O(n^2) на каждом процессе
(дублирование A и B).

## 4. Межпроцессная схема (MPI)

Роли процессов: все процессы равноправны. Каждый процесс вычисляет
свою прямоугольную часть итоговой матрицы C (несколько строк).

Распределение данных:

- Матрицы A и B полностью дублируются на всех процессах
- Строки матрицы C распределяются по процессам равномерно
  с учётом остатка:
  - `rows_per_proc = rows_total / world_size`
  - `remainder = rows_total % world_size`
  - `start_row = rank * rows_per_proc + min(rank, remainder)`
  - `local_rows = rows_per_proc + (rank < remainder ? 1 : 0)`

MPI-вызовы:

- `MPI_Comm_rank`, `MPI_Comm_size` – определение ранга и числа процессов.
- `MPI_Gatherv` – сбор локальных результатов (разной длины) на процессе 0.
  Принимающий процесс (0) должен подготовить массив `recv_counts` и `displs`.
- `MPI_Barrier` – синхронизация всех процессов перед завершением
  (гарантирует, что все успешно вычислили свои части).

Коммуникационная схема: обмен данными происходит только в конце
вычисления: каждый процесс отправляет свою `local_c` процессу 0.
Промежуточный обмен отсутствует, так как каждый процесс имеет полные
копии A и B и может независимо вычислять свою часть C.

## 5. Внутрипроцессная схема (std::thread)

Архитектура: внутри каждого MPI-процесса создаётся пул потоков
(новый на каждый вызов `RunImpl`). Число потоков равно
`std::thread::hardware_concurrency()` (или 2, если не определяется).

Разбиение данных: каждый процесс обрабатывает только свои строки
(от `start_row` до `start_row + local_rows`). Работа разбивается на блоки
размера 64×64. Все блоки (i,j) в пределах локальных строк обрабатываются
независимо.

Распределение блоков между потоками: используется атомарный счётчик
`next_block_idx` для динамического распределения блоков. Каждый поток
захватывает индекс блока, затем для всех k (0 … blocks_k-1):

- Загружает блок A_{i,k} через `GetBlock`.
- Загружает блок B_{k,j} через `GetBlock`.
- Умножает блоки через `MultiplyBlock` и накапливает результат
  в локальную матрицу `local_c`.

Синхронизация:

- `std::atomic` обеспечивает безопасное распределение индексов
  без блокировок (разные потоки получают разные блоки).
- Запись в `local_c` ведётся по непересекающимся индексам
  (каждый поток обрабатывает уникальные пары (i,j)), поэтому
  дополнительные примитивы синхронизации не требуются.
- `std::thread::join()` в конце дожидается завершения всех потоков.

Ключевой фрагмент (распределение блоков внутри процесса):

```cpp
std::atomic<size_t> next_block_idx(0);
const size_t total_blocks = blocks_i_local * blocks_j;

auto worker = [&]() {
  std::vector<double> block_a(bs*bs), block_b(bs*bs);
  while (true) {
    size_t idx = next_block_idx.fetch_add(1);
    if (idx >= total_blocks) break;
    int bi = idx / blocks_j, bj = idx % blocks_j;
    for (int bk = 0; bk < blocks_k; ++bk) {
      GetBlock(a, rows_total, cols_a, bi_global, bk, bs, block_a.data());
      GetBlock(b, cols_a, cols_b, bk, bj, bs, block_b.data());
      MultiplyBlock(block_a, block_b, bs, max_i, max_j, max_k, bi, bj, cols_b, local_c);
    }
  }
};
```

## 6. Детали реализации

Файлы:

- `kazennova_a_fox_algorithm/all/include/ops_all.hpp`
- `kazennova_a_fox_algorithm/all/src/ops_all.cpp`

Основные компоненты:

- `kBlockSize = 64` – фиксированный размер блока (выбран для баланса
  между локальностью и количеством блоков).
- `GetBlock()` – извлекает блок из матрицы, хранящейся в row-major порядке.
  Учитывает границы матрицы (последние блоки могут быть неполными).
- `MultiplyBlock()` – умножает два блока размера `block_size × block_size`
  (с учётом реальных размеров `max_i`, `max_j`, `max_k`) и добавляет
  результат в соответствующую позицию матрицы `local_c`.
- `RunLocalMultiplication()` – многопоточное выполнение блочного умножения
  для заданного диапазона строк (`start_row`, `local_rows`).
  Создаёт потоки (`std::thread`), распределяет блоки через атомарный счётчик.
- `RunImpl()` – главная функция:
  1. Определяет ранг и число MPI-процессов.
  2. Распределяет строки матрицы C между процессами (cyclic распределение).
  3. Вызывает `RunLocalMultiplication()` для вычисления своей части.
  4. Собирает все части на процессе 0 через `MPI_Gatherv`.
  5. Синхронизирует процессы через `MPI_Barrier`.

Изменения относительно SEQ:

- Параллелизм на двух уровнях: MPI-процессы (распределение строк)
  и std::thread внутри каждого процесса (распределение блоков).
- Использование атомарного счётчика для динамической балансировки блоков
  между потоками.
- MPI-коммуникации для сбора результата.

Особенности реализации:

- Матрицы A и B дублируются на всех MPI-процессах
- Локальная матрица `local_c` имеет размер `local_rows × n`.
- После локального умножения процесс 0 собирает все `local_c` в правильном
  порядке строк через `MPI_Gatherv` с предварительно вычисленными
  `recv_counts` и `displs`.
- Потоки создаются заново при каждом вызове `RunLocalMultiplication`
  (вместо пула), что добавляет небольшие накладные расходы.

## 7. Проверка корректности

Корректность проверена следующими способами:

- Функциональные тесты (набор из `tests/functional/main.cpp`)
  для размеров от 1 до 128, включая граничные случаи (n=1, n=prime,
  n не кратно 64). Тесты прогонялись с различным числом MPI-процессов
  (1, 2) и различным числом потоков (1, 2, 4).

- Сравнение с SEQ на случайных матрицах размера 700. Для всех
  конфигураций процессов и потоков результат совпал

- Проверка на неполных блоках

- Корректность MPI-сборки: проверено, что `MPI_Gatherv` правильно
  собирает части матрицы C в правильном порядке строк, без пропусков
  и наложений. Для этого сравнивалась матрица, собранная из частей,
  с матрицей, вычисленной процессом 0 напрямую.

- Гонки данных: благодаря атомарному счётчику и записи в непересекающиеся
  индексы `local_c` гонки отсутствуют. `ThreadSanitizer` не выявил
  проблем при прогоне на 2 процессах × 2 потока.

Все тесты пройдены. Гибридная версия даёт численно идентичный результат
последовательной версии для всех проверенных входных данных.

  не даёт прироста из-за малого размера задачи.

## 10. Выводы

Гибридная версия (MPI + потоки) корректна и даёт ускорение до 4.74×
на 2 процессах × 2 потока. MPI снижает конкуренцию за кэш (сверхлинейное
ускорение), атомарный счётчик балансирует блоки. При 2×4 прироста нет
из-за малого размера задачи (n=1000). Для масштабирования нужно увеличить
размер матрицы или перейти к распределённому хранению A и B.
