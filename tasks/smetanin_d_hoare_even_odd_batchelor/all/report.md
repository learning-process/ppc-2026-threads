# Сортировка Хоара с нечётно-чётным слиянием Бэтчера - ALL

- **Студент:** Сметанин Дмитрий Владимирович, группа 3823Б1ПР3
- **Технология:** ALL (MPI + `std::thread`)
- **Вариант:** 14

---

## 1. Контекст

Гибридная версия объединяет распределение массива между MPI-процессами, локальную сортировку куска с
OpenMP (та же схема `omp task`, что в OMP-версии), сборку и финальную упорядоченность на корне и
проверку использования потоковых API на ранге 0.

---

## 2. Постановка задачи

**Вход / выход.** `std::vector<int>` - глобально отсортированный по неубыванию вектор той же длины у всех
процессов после коллективных операций.

Базовое описание локальной сортировки Хоара + odd-even merge - см. в SEQ.

---

## 3. Базовый алгоритм (локально)

На каждом ранге после `Scatterv` локальный кусок сортируется тем же `HoarSortBatcherOMPImpl`, что в
OMP-версии (с порогом `kTaskCutoff` и последовательным `HoarSortBatcherSeq` на малых отрезках).

---

## 4. Межпроцессная схема

**Коммуникатор.** `MPI_COMM_WORLD`.

**Раскладка.** `BuildScatterLayout(n, comm_size, counts, displs)` - почти равномерное разбиение `n`
элементов между процессами (первые `n % comm_size` процессов получают на один элемент больше).

**Раздача.** `MPI_Scatterv` с корня (`rank == 0`): полный массив режется по `counts` и `displs`.

**Сборка.** `MPI_Gatherv` локальных отсортированных кусков на ранг 0 в буфер `gathered` длины `n`.

**Слияние на корне.** `KWayMergeChunks` — поочерёдное извлечение минимума из лобов отрезков через
`std::priority_queue` (k-way merge по отсортированным кускам). После слияния выполняется
`std::ranges::sort(data)` на полном массиве на ранге 0 для гарантии глобального порядка.

**Трансляция результата.** `MPI_Bcast(data.data(), n, MPI_INT, 0, MPI_COMM_WORLD)` - у всех рангов
одинаковый финальный вектор для проверки тестами.

**Барьеры.** Явный `MPI_Barrier` используется для согласования в ветках раннего выхода (`n <= 1`,
ошибка счётчиков потоков на ранге 0); основной алгоритм опирается на коллективные вызовы как на точки
синхронизации.

---

## 5. Внутрипроцессная схема

**Локальная сортировка.**

```cpp
// File: all/src/ops_all.cpp
#pragma omp parallel default(none) shared(local, local_n)
#pragma omp single
HoarSortBatcherOMPImpl(local, 0, local_n - 1);
```

Переменная `local_n` явно указана в `shared` вместе с `local` для совместимости с `default(none)` на
Clang/OpenMP.

**Демонстрационный блок на `rank == 0`.** После `Bcast` на ранге 0 запускаются `std::thread` и
`tbb::parallel_for`, считающие число «рабочих» единиц; если суммы не совпадают с
`ppc::util::GetNumThreads()`, возвращается `false` (с барьером).

---

## 6. Детали реализации

**Файлы.** `all/include/ops_all.hpp`, `all/src/ops_all.cpp` — класс `SmetaninDHoarSortALL`;
вспомогательные `BuildScatterLayout`, `KWayMergeChunks`, реализации partition/merge и
`HoarSortBatcherOMPImpl` в анонимном namespace.

**Ключевой фрагмент k-way merge (инициализация очереди):**

```cpp
// File: all/src/ops_all.cpp
pq.push(Item{.val = gathered[static_cast<size_t>(dsp)], .chunk = i, .next_j = 1});
```

Индексация при продвижении по куску учитывает расширение типов:
`idx = static_cast<size_t>(dsp) + static_cast<size_t>(x.next_j)` во избежание предупреждений
анализатора.

## 7. Проверка корректности

**Функциональные тесты.** `SmetaninDRunFuncTests`: при запуске под MPI вход синхронизируется
(`MPI_Bcast` из ранга 0); ожидаемый результат - полная сортировка, сравнение с `expected_`.

**Перф-тесты.** `SmetaninDRunPerfTests`: массив размера **1 000 000**, проверка
`std::ranges::is_sorted`.

**Запуск ALL под MPI.** Использовал runner с `--running-type=processes` и задавал `PPC_NUM_PROC`; для
потоков — `PPC_NUM_THREADS`.

## 8. Экспериментальная среда

- **Процессор:** AMD Ryzen 5 5600H with Radeon Graphics
- **ОЗУ:** 16 ГБ
- **ОС:** Microsoft Windows 10
- **Компилятор:** MSVC 14.43 (Visual Studio 2022 Community)
- **Сборка:** Release
- **Open MPI:** MS-MPI (`mpiexec`)
- **oneTBB:** 2022.3.0
- **OpenMP:** OpenMP в составе MSVC

**Baseline.** Время SEQ на том же перф-входе: **0,493 с** (`task_run`; см. в SEQ).

**Эффективность.** Для гибрида удобно нормировать на `P × T` (число процессов × потоков на процесс):

`efficiency = (T_seq / T_all) / (P · T) · 100%`.

**Команды запуска.**

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel

export PPC_NUM_PROC=2
export PPC_NUM_THREADS=4
scripts/run_tests.py --running-type=processes --counts 2 4

scripts/run_tests.py --running-type=performance
```

## 9. Результаты

- **P=1, T=1** (P×T=1): время 1,060 с; ускорение 0,47; эффективность 46,5%.
- **P=1, T=2** (P×T=2): время 0,824 с; ускорение 0,60; эффективность 29,9%.
- **P=1, T=4** (P×T=4): время 0,814 с; ускорение 0,61; эффективность 15,1%.
- **P=1, T=8** (P×T=8): время 1,681 с; ускорение 0,29; эффективность 3,7%.
- **P=2, T=1** (P×T=2): время 0,842 с; ускорение 0,59; эффективность 29,2%.
- **P=2, T=2** (P×T=4): время 0,805 с; ускорение 0,61; эффективность 15,3%.
- **P=2, T=4** (P×T=8): время 0,792 с; ускорение 0,62; эффективность 7,8%.
- **P=4, T=1** (P×T=4): время 0,814 с; ускорение 0,61; эффективность 15,1%.
- **P=4, T=2** (P×T=8): время 0,808 с; ускорение 0,61; эффективность 7,6%.

*(Режим `task_run`, среднее по 5 прогонам; ускорение и эффективность относительно **T_seq = 0,493 с**:
speedup = T_seq/T_all, efficiency = speedup/(P·T).)*

На одном узле гибрид **медленнее чистого SEQ**: доминируют `Scatterv`/`Gatherv`/`Bcast` и финальный
`std::ranges::sort` на корне. Лучший измеренный случай — **P=2, T=4** (**0,792 с**), но всё же хуже
локального OMP (**0,258 с** при T=4).

## 10. Выводы

ALL подтверждает корректность конвейера MPI + локальный OMP + слияние, но для массива **10⁶** на
ноутбуке с MS-MPI выигрыша по времени над последовательным или поточным OMP нет. Имеет смысл
наращивать полезность версии на кластере или при больших **n**, когда распределение памяти оправдывает
коммуникации.
