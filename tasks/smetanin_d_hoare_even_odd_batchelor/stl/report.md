# Сортировка Хоара с нечётно-чётным слиянием Бэтчера - STL

- **Студент:** Сметанин Дмитрий Владимирович, группа 3823Б1ПР3
- **Технология:** STL (`std::thread`)
- **Вариант:** 14

---

## 1. Контекст

Реализация на **`std::thread`** без OpenMP/TBB: параллелизм ограничен одним дополнительным потоком на
верхнем уровне рекурсивной процедуры (после первого разбиения один поток сортирует правую половину,
основной - левую); обе половины досортировываются тем же последовательным `HoarSortBatcherSeq`, что в
SEQ. При `PPC_NUM_THREADS ≤ 1` или на коротких отрезках работает только последовательная версия.

---

## 2. Постановка задачи

Вход / выход - `std::vector<int>`, сортировка по неубыванию. Детали алгоритма Хоара и odd-even merge -
в SEQ.

---

## 3. Базовый алгоритм

Используется общий последовательный `HoarSortBatcherSeq` (стек + partition + `OddEvenMerge`).

---

## 4. Схема распараллеливования

**Число потоков.** `num_threads = max(1, ppc::util::GetNumThreads())`.

**Порог.** `kTaskCutoff = 1000`; если отрезок короче порога или поток один - вызывается полный
`HoarSortBatcherSeq` на `[lo, hi]`.

**Разделение после partition.** После `HoarePartition` и `OddEvenMerge` на `[lo, hi]`:

- поток `right` выполняет `HoarSortBatcherSeq(arr, p + 1, hi)`;
- основной поток выполняет `HoarSortBatcherSeq(arr, lo, p)`;
- затем `right.join()`.

**Синхронизация.** Общие `mutex`/`atomic` не используются: подотрезки `[lo, p]` и `[p+1, hi]` не
пересекаются по индексам при последующей стековой обработке внутри `HoarSortBatcherSeq`.

**Важно.** Углублённое параллельное ветвление (как в OMP-версии с `omp task`) здесь не делается -
только один уровень разделения потоком.

---

## 5. Детали реализации

**Файлы.** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp` - класс `SmetaninDHoarSortSTL`, функция
`HoarSortBatcherSTLImpl`.

**Порядок create / join.** Сначала создаётся `std::thread right(...)`, затем работа в текущем потоке,
затем `right.join()` - типичный безопасный порядок для одного вспомогательного потока.

## 7. Экспериментальная среда

- **Процессор:** AMD Ryzen 5 5600H with Radeon Graphics
- **ОЗУ:** 16 ГБ
- **ОС:** Microsoft Windows 10
- **Компилятор:** MSVC 14.43 (Visual Studio 2022 Community)

**Baseline.** Время SEQ на перф-тесте: **0,493 с** (`task_run`; см. в SEQ).

**Эффективность.** `efficiency = (T_seq / T_stl) / threads · 100%`.

**Команды запуска.**

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel

export PPC_NUM_THREADS=4
scripts/run_tests.py --running-type=threads --counts 1 2 4
scripts/run_tests.py --running-type=performance
```

## 8. Результаты

- **1 поток:** 0,511 с; ускорение 0,96; эффективность 96,3%.
- **2 потока:** 0,293 с; ускорение 1,68; эффективность 84,0%.
- **4 потока:** 0,297 с; ускорение 1,66; эффективность 41,4%.
- **8 потоков:** 0,297 с; ускорение 1,66; эффективность 20,8%.

*(Режим `task_run`, среднее по 5 прогонам; ускорение от **T_seq = 0,493 с**.)*

Начиная с **2 потоков**, время почти не улучшается при росте `PPC_NUM_THREADS`: используется только
**один** дополнительный `std::thread` на первом расколе, дальнейшая работа — последовательное ядро.

---

## 9. Выводы

STL-версия достигает **~×1,68** к SEQ уже при 2 потоках и дальше не ускоряется при увеличении числа
доступных потоков - это ожидаемо для схемы «один вспомогательный поток + два последовательных
`HoarSortBatcherSeq`».
