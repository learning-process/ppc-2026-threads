## 1. Контекст

В данном отчёте рассматривается реализация параллельного фильтра Гаусса с использованием низкоуровневых примитивов стандартной библиотеки C++ (`std::thread`). В отличие от высокоуровневых фреймворков (OMP, TBB), здесь управление жизненным циклом потоков и распределение нагрузки на данные выполняется вручную.

## 2. Постановка задачи

Основная цель — реализовать честное «вертикальное разбиение» изображения. Требуется разделить матрицу на независимые вертикальные полосы, назначить каждую полосу отдельному потоку и обеспечить корректную сборку результата. SEQ-версия используется как эталон.

## 3. Базовый алгоритм

Алгоритм применяет ядро Гаусса 3x3 к каждому пикселю. Поскольку значение нового пикселя зависит только от старых значений в его окрестности, задача идеально подходит для параллелизма по данным.

## 4. Схема распараллеливания

В данной версии реализована декомпозиция по ширине изображения:

- **Разбиение данных:** Общее количество столбцов `width` делится на число доступных потоков `num_threads`.
- **Балансировка:** Для корректной обработки случаев, когда ширина не кратна числу потоков, используется классическая формула распределения остатка: первым `remainder` потокам достаётся на один столбец больше (`cols_per_thread + 1`).
- **Синхронизация и потокобезопасность:** Реализация полностью свободна от блокировок (`lock-free`). Каждый поток записывает результаты в свой непересекающийся диапазон столбцов выходного вектора. Это исключает состояние гонки (data race).
- **Управление потоками:** Потоки создаются в цикле, после чего основной поток дожидается их завершения.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Важным моментом является порядок запуска и ожидания потоков. Чтобы избежать сериализации (выполнения потоков по очереди), вызов `.join()` вынесен в отдельный цикл после того, как все воркеры уже запущены:

```cpp
// File: stl/src/ops_stl.cpp
for (int thread_idx = 0; thread_idx < actual_threads_count; ++thread_idx) {
  int end_column = start_column + cols_per_thread + (thread_idx < remainder_cols ? 1 : 0);
  threads.emplace_back([&, start_column, end_column]() {
    for (int col_idx = start_column; col_idx < end_column; ++col_idx) {
      for (int row_idx = 0; row_idx < height; ++row_idx) {
        // Вычисление и запись в result
      }
    }
  });
  start_column = end_column;
}

// 2. Ожидание завершения всех воркеров
for (auto &thread_obj : threads) {
  thread_obj.join();
}
```

Такая структура гарантирует, что все ядра процессора будут загружены работой одновременно.

## 9. Выводы

Ручное управление потоками через `std::thread` позволяет полностью контролировать логику декомпозиции, но требует тщательного расчета индексов. Реализация доказала свою работоспособность, хотя и выявила ограничения «вертикального разбиения» для памяти-зависимых задач.(горизонтальная лучше из-за того как хранится к кэш памяте)
