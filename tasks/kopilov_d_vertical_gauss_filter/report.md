- **Локальные отчёты**: [SEQ](seq/report.md), [OMP](omp/report.md), [TBB](tbb/report.md), [STL](stl/report.md), [ALL](all/report.md)

## 1. Введение
В данной работе исследуется эффективность различных моделей параллельного программирования (директивы OpenMP, задачи TBB, низкоуровневые потоки STL и распределенные процессы MPI) на примере задачи линейной фильтрации изображений. Фильтр Гаусса 3x3 является классической memory-bound задачей, производительность которой сильно зависит от способа обхода памяти и локальности данных.

## 2. Единая постановка задачи
**Объект**: Изображение в оттенках серого, представленное как `std::vector<uint8_t>`.
**Операция**: Свертка с ядром Гаусса 3x3 (коэффициент нормировки 1/16).
**Граничные условия**: Зеркальное отражение пикселей (индексы `-1 -> 1`, `width -> width-2`).
**Критерий корректности**: Попиксельное совпадение результата параллельных версий с последовательным эталоном.

## 3. Единая методика эксперимента
Все замеры производились в идентичном окружении:
- **Процессор**: Intel Core i7-10750H (6 физических ядер, 12 логических потоков, частота до 5.0 ГГц).
- **ОЗУ**: 16 GB DDR4.
- **ОС**: Ubuntu 22.04 LTS (WSL 2).
- **Компилятор**: GCC 11.4.0 (флаги `-O3 -march=native`).
- **Входные данные**: Изображение 8192 x 8192 пикселей (64 Мегапикселя).
- **Метрики**:
  - Ускорение ($S$): отношение времени выполнения SEQ к параллельной версии.
  - Эффективность ($E$): $S / N$, где $N$ — общее число воркеров.

## 4. Сводка корректности
Все пять реализаций прошли единый цикл тестирования в `tests/functional/main.cpp`. 
- Проверены вырожденные случаи (матрицы 1x1, 2x2), где все пиксели являются граничными.
- Проверены стандартные размеры (3x3, 4x4) с разным наполнением.
- Реализована надежная валидация входных параметров (размеры, длины векторов).
Для гибридной версии (ALL) валидация синхронизирована между процессами через `MPI_Bcast`, а сравнение результатов выполняется только на Master-процессе.

## 6. Интерпретация различий
1. **Превосходство TBB**: Использование `blocked_range2d` позволило TBB обрабатывать данные небольшими квадратными блоками. Это минимизирует промахи кэша L2/L3, так как соседние строки блока загружаются в кэш целиком.
2. **Сверхлинейное ускорение (OMP/TBB)**: В SEQ-версии был заложен обход по столбцам (наихудший сценарий для памяти), тогда как OMP и TBB используют построчный или блочный доступ. Это дает "бонусное" ускорение за счет эффективной работы аппаратного префетчера.
3. **Проблема STL**: Реализация "вертикального разбиения" через нарезку по столбцам в `std::thread` показала низкую эффективность. Причина — row-major хранение данных в C++: чтение по вертикали заставляет процессор постоянно прыгать по памяти.
4. **Накладные расходы MPI (ALL)**: Пересылка 67 МБ данных между процессами и сборка результата на мастере создают значительный сетевой/шинный overhead, что делает гибридную схему менее эффективной на одном узле, чем TBB.

## 8. Заключение
Наилучшую производительность для данной задачи показала библиотека **Intel TBB** благодаря автоматическому тайлингу данных. Модель **OpenMP** является наиболее простой в реализации, но чуть менее эффективной. Использование **MPI** целесообразно только при обработке изображений, не помещающихся в память одного узла.
