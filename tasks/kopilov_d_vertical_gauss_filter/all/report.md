## 1. Контекст

Гибридная реализация сочетает две модели параллелизма: распределенную память (MPI) для масштабирования на кластерные системы и общую память (TBB) для эффективного использования многоядерности каждого отдельного узла. Это наиболее сложная и производительная конфигурация для крупномасштабных задач обработки данных.

## 2. Постановка задачи

Требуется реализовать фильтр Гаусса 3x3, обеспечив корректное взаимодействие между процессами. Основная сложность — расчет пикселей на границах полос, для которых требуются данные, находящиеся в памяти соседних MPI-процессов. Итоговый результат должен быть собран на Master-процессе и совпадать с `seq/report.md`.

## 3. Базовый алгоритм

Математическая модель фильтрации (свертка 3x3) остается неизменной. Однако алгоритм дополняется фазами коммуникации (рассылка, обмен границами, сборка).

## 4. Межпроцессная схема (MPI)

Реализована модель «вертикального разбиения» на уровне процессов:

- **Роли:** Процесс с рангом 0 (Master) выполнеят функции диспетчера (чтение входных данных, распределение полос, сборка результата). Остальные процессы (Workers) выполняют вычисления.
- **Распределение (Scatter):** Master рассчитывает смещения и количество столбцов для каждого процесса. Рассылка данных выполняется вручную через `MPI_Send/MPI_Recv` (из-за особенностей вертикальной нарезки row-major данных).
- **Обмен тенями (Halo Exchange):** Поскольку для фильтра 3x3 каждому пикселю нужны значения соседей, реализован обмен «призрачными» столбцами (halo cells). Каждый процесс отправляет свои крайние столбцы и принимает соседние с помощью функции `MPI_Sendrecv`, что исключеат риск взаимной блокировки (deadlock).
- **Синхронизация:** Использован `MPI_Bcast` для распространения результатов валидации и размеров изображения между всеми участниками.

## 5. Внутрипроцессная схема (TBB)

После того как каждый MPI-процесс получил свою полосу данных и обменялся границами с соседями, запускается внутренний параллелизм:

- **Примитив:** `tbb::parallel_for`.
- **Диапазон:** Локальная полоса обрабатывается с использованием двумерного диапазона `tbb::blocked_range2d`, что обеспечивает оптимальный тайлинг и локальность кэша внутри узла.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

В коде реализована жесткая декомпозиция для снижения сложности:

```cpp
// File: all/src/ops_all.cpp
if (world_rank == 0) {
    MasterDistribute(world_size, global_width, global_height, counts, displs, GetInput().data.data(), l_in.data(), lw_halo);
} else {
    WorkerReceive(local_w, global_height, l_in.data(), lw_halo);
}

ExchangeHalo(world_rank, world_size, global_height, local_w, lw_halo, l_in, counts);
ComputeTBB(local_w, global_height, lw_halo, l_in.data(), l_out.data());
```

Такая структура позволяет четко разделить фазы передачи данных, синхронизации границ и непосредственных вычислений.

## 7. Проверка корректности

Корректность проверялась на тестах. Гибридная версия успешно проходит тесты 1x1, 2x2 и 4x4. Проверка `MPI_Comm_rank != 0` в `CheckTestOutputData` гарантирует, что тесты завершаются успешно, так как результат валидируется только на процессе-мастере, который обладает полными данными.

## 8. Экспериментальная среда

Конфигурация задается парой `ranks × threads`:

- **PPC_NUM_PROC**: число процессов MPI.
- **PPC_NUM_THREADS**: число потоков TBB внутри каждого процесса.
- **Окружение**: Intel Core i7-10750H (12 логических ядер), Ubuntu 22.04.

**Команды для запуска:**

```bash
# Сборка с поддержкой MPI
cmake -S . -B build -D USE_FUNC_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel

# Запуск на 2 процессах по 2 потока в каждом
export PPC_NUM_PROC=2
export PPC_NUM_THREADS=2
scripts/run_tests.py --running-type=processes --counts 2
```

*Примеачние: Гибридная версия показывает себя хуже на одной машине, так как создаваласб под масштабные вычисления с использованием mpi, поэтому с увеличение кол-во машин, эффективность будет расти быстрее. Это связано со значительными накладными расходами на копирование памяти при Scatter/Gather и пересылку строк через сокеты MPI. Однако конфигурация 2x4 оказалась быстрее, чем 4x1, что подтверждает преимущество использования TBB для внутриузловых вычислений.*
