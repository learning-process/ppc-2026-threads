## 1. Контекст
Данный отчёт описывает перенос базового алгоритма Гауссовой фильтрации на технологию OpenMP. Цель реализации — продемонстрировать эффективность распараллеливания на системах с общей памятью (UMA/NUMA) с минимальными изменениями в исходном коде за счёт директив компилятора.

## 2. Постановка задачи
Необходимо распараллелить вычисление свертки изображения с ядром 3x3. Входные и выходные данные, а также ограничения полностью идентичны последовательной версии. Эталонным временем (`baseline`) для расчета ускорения считается результат из `seq/report.md`.

## 3. Базовый алгоритм
В последовательной версии алгоритм проходит по каждому пикселю, собирает взвешенную сумму окрестности (с учетом зеркалирования границ) и делит результат на 16. Вычисления для каждого пикселя абсолютно независимы друг от друга.

## 4. Схема распараллеливания
Так как расчет каждого пикселя не зависит от остальных, была применена стратегия параллелизма по данным (Data Parallelism).
- **Область распараллеливания:** Директива `#pragma omp parallel for` применена к внешнему циклу обхода изображения по строкам (`j < height`). 
- **Разделение данных:**
  - `shared`: Исходный вектор `source_image`, вектор результата `destination_image`, константы размеров `width` и `height`, а также ядро `kGaussKernel` и делитель `kDivisor`. Все потоки безопасно читают эти данные одновременно.
  - `private` (неявно): Переменные циклов `j`, `i`, `kernel_y`, `kernel_x` и накопитель `pixel_sum` объявляются локально внутри цикла, поэтому каждый поток получает свою независимую копию в стеке.
- **Синхронизация и Reduction:** Конструкция `reduction` или атомарные операции (`atomic`, `critical`) **не требуются**, так как каждый поток записывает результат строго в уникальный индекс массива `destination_image`. Отсутствие блокировок обеспечивает максимальную масштабируемость.
- **Schedule:** Используется `schedule(static)` (по умолчанию в OpenMP для циклов без указания политики). Поскольку вычислительная нагрузка на каждый пиксель (ровно 9 итераций свертки) абсолютно идентична, статическое равномерное распределение итераций между потоками является оптимальным и не требует накладных расходов на динамическую балансировку.
- **Барьеры:** В конце области `parallel for` присутствует неявный барьер (implicit barrier). Основной поток гарантированно дождется завершения записи всех строк перед переходом к `PostProcessingImpl`.

## 5. Детали реализации
Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Ключевым изменением относительно `SEQ` стало добавление прагмы с атрибутом `default(none)`. Это правило хорошего тона в OpenMP, которое отключает автоматическое угадывание областей видимости переменных и заставляет программиста явно прописать, какие данные являются общими (`shared`). Это архитектурно исключает случайное возникновение состояний гонки (race conditions). Количество потоков задается явно через вызов `omp_set_num_threads`.

## 6. Проверка корректности
Корректность алгоритма проверялась сравнением с последовательной версией `SEQ` в рамках `tests/functional/main.cpp`. 
Так как потоки лишь читают общую память и пишут в непересекающиеся адреса, гонки за данными (data races) исключены by design, что подтверждается успешным прохождением тестов на вырожденных (1x1, 2x2) и нормальных матрицах.

OpenMP продемонстрировал превосходные результаты при минимальном вмешательстве в код. Статическое расписание (`static schedule`) и построчный обход матрицы обеспечили высокую утилизацию кэша и практически линейное масштабирование вплоть до упирания в физическое число ядер и пропускную способность памяти шины при 8 потоках.

## Приложение (фрагмент кода)

```cpp
// File: omp/src/ops_omp.cpp
  int num_threads = ppc::util::GetNumThreads();
  omp_set_num_threads(num_threads);

// Явное указание атрибутов доступа. Нет reduction, так как индексы записи уникальны
#pragma omp parallel for default(none) shared(source_image, destination_image, width, height, kGaussKernel, kDivisor)
  for (int j = 0; j < height; ++j) {
    for (int i = 0; i < width; ++i) {
      int pixel_sum = 0;
      // ... вычисление свертки
      destination_image[(j * width) + i] = static_cast<uint8_t>(pixel_sum / kDivisor);
    }
  }
```
Фрагмент демонстрирует границу распараллеливания: параллелится внешний цикл по строкам (`j`). Переменные защиты и синхронизации отсутствуют, что делает критическую секцию свободной от блокировок. Неявный барьер в конце цикла обеспечивает корректность завершения фазы вычислений перед выдачей результата.
