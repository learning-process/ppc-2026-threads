## 1. Контекст

Отчет описывает процесс распараллеливания алгоритма фильтрации Гаусса с использованием стандарта OpenMP.
Основная цель — утилизировать многоядерность процессора на системах с общей памятью, сохраняя простоту
исходного кода за счет применения прагм компилятора.

## 2. Постановка задачи

Требуется ускорить вычисление свертки 3x3, распределив обработку строк изображения между потоками.
Условия по обработке границ (зеркалирование) остаются такими же, как в SEQ-версии. Результаты
всех потоков должны корректно объединяться в итоговой матрице.

## 3. Базовый алгоритм

Алгоритм выполняет проход по пикселям изображения, вычисляя для каждого новое значение на основе
его окрестности. Вычисления для каждого пикселя детерминированы и не зависят от результатов
соседних итераций, что делает задачу идеально подходящей для параллельного цикла.

## 4. Схема распараллеливания

Для реализации выбран метод декомпозиции по данным:

- **Область параллелизма**: Директива `#pragma omp parallel for` применена к внешнему циклу по строкам.
- **Атрибуты доступа**:
  - `shared`: входное и выходное изображения, размеры матрицы, ядро фильтра.
  - `private` (неявно): счетчики циклов `i`, `j`, `kernel_x`, `kernel_y` и сумма `pixel_sum`.
- **Синхронизация**: Использование `atomic` или `critical` не требуется, так как каждый поток
  записывает результат в уникальный индекс выходного вектора. Это исключает возникновение
  состояний гонки (data race).
- **Schedule**: Используется стандартное статическое распределение (`static`). Так как вычислительный
  вес каждой строки одинаков, это минимизирует накладные расходы на управление очередью задач.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

В реализации используется правило `default(none)`, которое обязывает явно указывать права доступа
для каждой переменной. Это стандарт безопасности, предотвращающий случайное использование общих данных.

```cpp
// Фрагмент из omp/src/ops_omp.cpp
#pragma omp parallel for default(none) \
    shared(source_image, destination_image, width, height, kGaussKernel, kDivisor)
for (int j = 0; j < height; ++j) {
  for (int i = 0; i < width; ++i) {
    int pixel_sum = 0;
    // Вложенные циклы свертки...
    destination_image[(j * width) + i] = static_cast<uint8_t>(pixel_sum / kDivisor);
  }
}
```

## 6. Проверка корректности

Правильность работы OpenMP-версии подтверждена через `tests/functional/main.cpp`. Сравнение
проводилось с последовательным эталоном. Тесты показали, что результат не зависит от
количества задействованных потоков (проверено на 1, 2, 4 и 8 потоках).

## 8. Результаты

Замеры производительности для изображения 8K.

| Backend | Mode | Size | Workers | Median Time (ms) | Speedup | Efficiency |
|---------|------|------|---------|------------------|---------|------------|
| SEQ     | perf | 8K   | 1       | 420.5            | 1.00x   | 100.0%     |
| OMP     | perf | 8K   | 2       | 155.7            | 2.70x   | 135.0%     |
| OMP     | perf | 8K   | 4       | 85.8             | 4.90x   | 122.5%     |
| OMP     | perf | 8K   | 8       | 55.3             | 7.60x   | 95.0%      |

*Примечание: Сверхлинейное ускорение (эффективность > 100%) на малом числе потоков связано
с тем, что OMP версия обходит память построчно (row-major), что на порядки эффективнее
для кэша процессора, чем вертикальный обход в SEQ.*
