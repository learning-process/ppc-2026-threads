## 1. Контекст

В данной работе решается задача умножения разреженных матриц в формате CCS с комплексными числами. Эта задача выбрана для
сравнения разных моделей параллелизма, потому что:

1) Вычисления независимы - столбцы результирующей матрицы вычисляются независимо, что позволяет легко распараллелить задачу
2) Универсальность - задача решается пятью способами: последовательно (SEQ), с помощью OpenMP, TBB, STL и гибридно (MPI+OpenMP)

Сравнение этих подходов позволяет понять, какая технология лучше подходит для разных условий: простоты реализации,
производительности и масштабируемости.

## 2. Единая постановка задачи

Входные данные
Две разреженные матрицы A и B в формате CCS:

- values - вектор комплексных чисел (ненулевые элементы)
- row_indices - вектор номеров строк для каждого элемента
- col_ptrs - вектор указателей на начало каждого столбца (размер cols + 1)
- rows, cols, nnz - размеры и количество ненулевых элементов

Выходные данные
Матрица C = A × B в том же формате CCS.

Ограничения

- количество столбцов A должно равняться количеству строк B
- все размеры матриц должны быть положительными
- формат CCS должен быть корректным: col_ptrs[0] == 0, размер col_ptrs == cols + 1, nnz == values.size()

Результат считается корректным, если выполняются все условия:

- размеры rows и cols совпадают с ожидаемыми
- количество ненулевых элементов nnz совпадает
- массивы col_ptrs и row_indices полностью идентичны эталону
- значения элементов отличаются не более чем на eps = 10^-10

nnz, массивы col_ptrs и row_indices полностью идентичны, а значения элементов отличаются не более 
чем на eps = 10^-10

## 6. Интерпретация различий

### SEQ (baseline)

Последовательная версия показывает базовое время выполнения 1953 мс. От этого значения рассчитывается ускорение всех параллельных
версий.

### OMP

Плюсы: простой код (одна директива #pragma omp parallel for), отсутствие лишних накладных расходов, хорошее ускорение (3.94 на 4
потоках). Слабые стороны: при schedule(static) возможен дисбаланс, если столбцы сильно различаются по количеству ненулевых
элементов.

### TBB

TBB автоматически разбивает диапазон на блоки. grainsize не задавался - использовалось значение по умолчанию. Это создало
дополнительные накладные расходы на управление задачами. Результат: ускорение 2.95 на 4 потоках (эффективность 73.8%) - хуже, чем
у OMP. Вывод: автоматическая балансировка TBB не всегда даёт выигрыш

### STL

Ручное управление через std::async дало лучший speedup (4.16) и сверхлинейную эффективность (104% на 4 потоках, 214% на 2
потоках). Минус - сложный код: нужно самому делить диапазоны, запускать задачи, ждать через future.get(). Высокий риск ошибок.

### ALL (MPI+OpenMP)

Цена коммуникации оказалась высокой. На 4 потоках ускорение всего 2.74 (эффективность 69%), а при 2×4 (8 потоков) — 2.60
(эффективность 33%).
Гибрид имеет смысл только при работе на нескольких компьютерах или при очень больших объёмах данных.

## 8. Заключение

Лучшая версия получилась OpenMP. Она даёт отличное ускорение (3.94), проста в реализации и не имеет лишних накладных расходов.
STL чуть быстрее (4.16), но код сложнее.

Ограничения: замеры только на одном процессоре, при одной плотности (1%) и одном размере матриц (1500×1500). На других данных
результаты могут отличаться.

Можно было бы улучшить добавив schedule(dynamic) для OMP, подобрав оптимальный grainsize для TBB и реализовать динамическую
балансировку для STL.
