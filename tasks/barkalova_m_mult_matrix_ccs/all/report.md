## 1. Контекст
Гибридная версия сочетает два уровня параллелизма: MPI + OpenMP. MPI распределяет столбцы результирующей матрицы между процессами, а каждый процесс внутри себя использует OpenMP для параллельной обработки своих столбцов. Такой подход позволяет задействовать ресурсы как на уровне кластера (несколько узлов), так и на уровне многоядерного процессора.


## 2. Постановка задачи
Даны две разреженные матрицы A и B в формате CCS (Compressed Column Storage) с комплексными элементами. Требуется вычислить их произведение C = A × B и представить результат также в формате CCS.
Условие корректности: количество столбцов A должно равняться количеству строк B

Последовательная версия (`seq/report.md`) служит эталоном корректности

## 3. Базовый алгоритм
Последовательная версия (SEQ) реализует следующий алгоритм:
1. Транспонирование матрицы A -> Aт (также в формате CCS). Это позволяет эффективно обходить строки A при вычислении скалярных произведений.
2. Для каждого столбца j матрицы B:
 - Для каждой строки i матрицы A:
      Вычислить скалярное произведение i-й строки A и j-го столбца B
      Если результат не ноль - сохранить в результат
3. Формирование выходной матрицы C в формате CCS.


## 4. Межпроцессная схема
Процесс с рангом 0 является главным. Он:
1. Получает входные матрицы
2. Рассылает их всем процессам через `BroadcastMatrix()`
3. Собирает результаты от всех процессов через `GatherResults()`
4. Формирует итоговую матрицу
Остальные процессы (ранги 1..size-1) только вычисляют свою часть и отправляют результаты процессу 0.
После завершения вычислений процесс 0 рассылает итоговую матрицу всем остальным, чтобы `GetOutput()` у всех процессов содержал одинаковый результат.

Столбцы результирующей матрицы C распределяются между процессами поровну:
```cpp
// File: `all/src/ops_all.cpp`
int cols_per_process = total_cols / size;
int remainder = total_cols % size;

int start_col = (rank * cols_per_process) + std::min(rank, remainder);
int local_cols = cols_per_process + (rank < remainder ? 1 : 0);
```
Каждый процесс получает непрерывный диапазон столбцов. Первые remainder процессов получают на один столбец больше.

В работе использовались такие MPI-вызовы:
MPI_Bcast - для рассылки матриц A и B от процесса 0 всем остальным и в конце рассылка результата всем процессам
MPI_Allreduce - суммирование количества ненулевых элементов по столбцам от всех процессов
MPI_Gather - сбор локальных nnz от всех процессов на процесс 0 для расчёта смещений перед Gatherv
MPI_Gatherv - сбор индексов строк и значений от всех процессов на процесс 0 (разные объёмы данных)


## 5. Внутрипроцессная схема
Внутри каждого процесса распараллеливание выполняется через OpenMP:
```cpp
// File: `all/src/ops_all.cpp`
#pragma omp parallel for schedule(static) default(none) shared(start_col, local_cols, at, b, col_rows, col_vals)
for (int j_local = 0; j_local < local_cols; ++j_local) {
    int global_col = start_col + j_local;
    // обработка столбца
}
```
`schedule(static)` — равномерное распределение, так как нагрузка на столбцы примерно одинаковая
`default(none)` — явное указание атрибутов переменных для безопасности

Каждый поток внутри процесса обрабатывает свою группу столбцов и записывает результаты в `col_rows[j_local]` и `col_vals[j_local]`. Гонок данных нет, так как разные потоки работают с разными индексами.

### Объединение локальных результатов
Внутри процесса объединение не требуется, так как потоки пишут в разные индексы. После завершения OpenMP-цикла локальные векторы собираются в линейные массивы через `LocVectors()` и отправляются процессу 0 через MPI.

## 7. Проверка корректности
Результаты гибридной версии сравниваются с эталонной последовательной реализацией. Совпадение считается достигнутым, если совпадают размеры матриц, nnz, массивы col_ptrs и row_indices, а значения элементов отличаются не более чем на eps = 10^-10.

Используется 6 предопределённых тестов, покрывающих диагональные, прямоугольные и разреженные матрицы, а также комплексные и действительные числа.
Все тесты пройдены успешно для конфигураций с 1, 2 и 4 процессами.

Замер производительности проводился на матрице 1500×1500, плотность 1%, случайно сгенерированные ненулевые элементы со значениями от 0.1 до 10.0

| 1 × 2               | 2     |   725        | 2.69    | 135%        |

Ускорение во всех конфигурациях примерно одинаковое — около 2.7x. Увеличение числа потоков с 2 до 8 почти не меняет результат. Это значит, что накладные расходы на MPI-передачи и синхронизацию съедают потенциальный выигрыш. Добавление второго процесса не помогает. Для данной задачи на одном компьютере проще и выгоднее использовать обычную OpenMP-версию.


## 10. Выводы
Гибридная схема оправдана только для очень больших задач, которые не помещаются в память одного компьютера. В этом случае MPI распределяет данные между несколькими узлами, а OpenMP внутри узла использует все ядра.

Передача данных между процессами и синхронизация создают лишние накладные расходы. Для задачи 1500×1500 чистая OpenMP-версия работает быстрее и проще в реализации.
