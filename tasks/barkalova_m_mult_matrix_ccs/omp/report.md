## 1. Контекст

В данной работе реализовано последовательное умножение двух разреженных матриц в формате CCS (Compressed Column Storage) с
комплексными элементами.
Последовательная версия (SEQ) реализует алгоритм умножения матриц в формате CCS (Compressed Column Storage) с транспонированием
первой матрицы для эффективного доступа по строкам. Данная OpenMP-реализация распараллеливает обработку столбцов результирующей
матрицы, что позволяет получить ускорение на многопроцессорных системах.

## 2. Постановка задачи

Даны две разреженные матрицы A и B в формате CCS (Compressed Column Storage) с комплексными элементами. Требуется вычислить их
произведение C = A × B и представить результат также в формате CCS.
Условие корректности: количество столбцов A должно равняться количеству строк B

Подробная постановка уже дана в `seq/report.md`

## 3. Базовый алгоритм

Последовательная версия (SEQ) реализует следующий алгоритм:

1. Транспонирование матрицы A -> Aт (также в формате CCS). Это позволяет эффективно обходить строки A при вычислении скалярных
произведений.
2. Для каждого столбца j матрицы B:

- Для каждой строки i матрицы A:
      Вычислить скалярное произведение i-й строки A и j-го столбца B
      Если результат не ноль - сохранить в результат

3. Формирование выходной матрицы C в формате CCS.

Асимптотика:
По времени: O(nnz_A × nnz_B) в худшем случае (при плотных матрицах)
По памяти: O(nnz_A + nnz_B + nnz_C)

## 4. Схема распараллеливания

Распараллелен внешний цикл по столбцам результирующей матрицы C:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for schedule(static) default(none) shared(at, b, col_rows, col_vals)
for (int j = 0; j < c.cols; j++) {
    // вычисление одного столбца результата
}
```

Обоснование:

- столбцы C независимы - вычисления для разных j не пересекаются, так как каждый столбец формируется отдельно
- schedule(static) выбран, потому что нагрузка на столбцы примерно одинаковая - итерации цикла делятся на примерно равные блоки, и каждый блок заранее закрепляется за определённым потоком. static подходит благодаря предсказуемой нагрузке и отсутствию лишней синхронизации
- default(none) - требует явного указания атрибутов всех переменных, что повышает безопасность и читаемость кода

Синхронизация:

- гонок данных нет - каждый поток пишет только в col_rows[j] и col_vals[j] со своим уникальным j
- гритические секции не нужны - все операции над общими данными - только чтение
- неявный барьер - в конце parallel for все потоки синхронизируются перед продолжением

После параллельной обработки результаты из col_rows и col_vals последовательно объединяются в финальную матрицу C.

## 5. Детали реализации

Файлы
`omp/include/ops_omp.hpp` - объявление класса BarkalovaMMultMatrixCcsOMP
`omp/src/ops_omp.cpp` - реализация

Алгоритм транспонирования матрицы A (TransponirMatr), функция вычисления скалярного произведения (ComputeScalarProduct), проверка
на ноль (IsNonZero), валидация входных данных и постобработка результата - не изменились

Отличия
Основной цикл вычислений:
SEQ - последовательный цикл по столбцам:

```cpp
// File: `seq/src/ops_seq.cpp`
for (int j = 0; j < c.cols; j++) {
    ProcessColumn(at, b, j, values, rows, nz);
}
```

OMP - распараллеленный цикл с локальными буферами:

```cpp
// File: `omp/src/ops_omp.cpp`
#pragma omp parallel for schedule(static) default(none) shared(at, b, c, col_rows, col_vals)
for (int j = 0; j < c.cols; j++) {
    std::vector<int> rows;
    std::vector<Complex> vals;
    // ... вычисления ...
    col_rows[j] = std::move(rows);
    col_vals[j] = std::move(vals);
}
```

Хранение промежуточных результатов:
SEQ - один общий вектор values, один общий вектор rows, счётчик nz
OMP - массив векторов col_rows[c.cols] и col_vals[c.cols] — для каждого столбца свой

Сбор результата:
SEQ - результаты добавляются сразу в финальные векторы
OMP - сначала накапливаются в раздельные массивы, затем объединяются

Гонки данных отсутствуют. Критические секции не нужны, так как внутри параллельной области нет операций записи в общие переменные.

## 6. Проверка корректности

Результаты OpenMP-версии сравниваются с эталонной последовательной реализацией (SEQ). Совпадение считается достигнутым, если выполняются все условия из метода CheckTestOutputData():

- размеры матриц rows и cols совпадают
- количество ненулевых элементов nnz совпадает
- массивы col_ptrs и row_indices одинаковы
- значения элементов отличаются не более чем на eps = 10^-10

Использовано 6 тестов, покрывающих квадратные и прямоугольные матрицы, диагональные и разреженные матрицы, комплексные и 
вещественные числа.
например:
умножение диагональных матриц 2×2 с комплексными элементами
умножение прямоугольных матриц 2×2 и 2×3
умножение разреженных матриц 3×3 (4 ненулевых элемента)
Все тесты пройдены успешно
