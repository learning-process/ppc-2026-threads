## 1. Контекст

В данном отчете описывается реализация параллельного умножения разреженных матриц с использованием стандартной
библиотеки C++ (`std::thread`). В отличие от высокоуровневых OMP и TBB, использование `std::thread` требует ручного
управления жизненным циклом потоков и самостоятельного разбиения данных на блоки. Это позволяет лучше понять
низкоуровневые механизмы многопоточности и оценить накладные расходы на создание потоков.

## 2. Постановка задачи

Требуется реализовать умножение матриц `A` и `B` в формате CCS. Последовательная версия (SEQ) считается эталоном.
Ограничения на входные данные и критерии корректности остаются неизменными.

## 3. Базовый алгоритм

Алгоритм вычисляет результирующую матрицу по столбцам. Для каждого столбца матрицы `B` происходит умножение его
ненулевых элементов на соответствующие столбцы матрицы `A`. Результат накапливается в векторе, фильтруется от нулей и
сохраняется в итоговую структуру.

## 4. Схема распараллеливания

- **Разбиение данных:** Используется статическое разбиение (static chunking). Общее количество столбцов матрицы `B`
делится на количество доступных аппаратных потоков (`std::thread::hardware_concurrency()`). Остаток от деления
равномерно распределяется между первыми потоками.
- **Работа потока:** Каждый поток получает свой диапазон `[start, end)` и обрабатывает столбцы строго внутри этого
диапазона.
- **Локальные результаты:** Как и в предыдущих версиях, потоки пишут результаты в независимые двумерные векторы
`local_row_indices` и `local_values` по индексу столбца `j`. Это гарантирует отсутствие пересечений по памяти.
- **Синхронизация:** Поскольку каждый поток работает со своим уникальным набором индексов и пишет в заранее выделенные
независимые ячейки памяти, использование `std::mutex` или `std::atomic` не требуется.
- **Объединение (Join):** Сначала в цикле создаются все объекты `std::thread`, и только после этого в отдельном цикле
вызывается `join()`. Это критически важно для обеспечения реального параллелизма (избежания сериализации).

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Вся логика управления потоками вынесена в шаблонную функцию `RunParallelChunks`. Ниже приведен фрагмент, демонстрирующий
правильный порядок создания и ожидания потоков:

```cpp
// File: stl/src/ops_stl.cpp
std::vector<std::thread> threads;
threads.reserve(static_cast<size_t>(num_threads));

int current_start = 0;
for (int i = 0; i < num_threads; ++i) {
  int current_end = current_start + chunk_size + (i < remainder ? 1 : 0);
  if (current_start < current_end) {
    // Запуск потока с передачей диапазона работы
    threads.emplace_back(worker_func, current_start, current_end);
  }
  current_start = current_end;
}

// Ожидание завершения всех потоков в отдельном цикле
for (auto &t : threads) {
  if (t.joinable()) {
    t.join();
  }
}
```

В методе `RunImpl` эта функция вызывается дважды: первый раз для параллельного вычисления столбцов, второй раз для
параллельного копирования данных в итоговые одномерные массивы после последовательного расчета `col_ptrs`. Аккумулятор
выделяется один раз на поток внутри лямбда-выражения, что минимизирует работу с памятью.

## 6. Проверка корректности

Корректность проверялась сравнением с эталоном SEQ. Функциональные тесты успешно пройдены. Отсутствие гонок данных (race
conditions) обеспечивается архитектурой решения: потоки не имеют общих точек записи.

## 8. Результаты

Ниже приведены медианные значения времени выполнения для 4 потоков в сравнении с базовой версией (p=1).

 | Размер матриц | Плотность | Работников (p) | Время работы (мс) | Ускорение | Эффективность |
 | --- | --- | --- | --- | --- | --- |
 | 500x500 | 10% | 4 | 4 | 3.00x | 75.0% |
 | 1000x1000 | 5% | 4 | 21 | 3.09x | 77.2% |
 | 2000x2000 | 5% | 4 | 95 | 3.26x | 81.5% |

Ускорение присутствует и масштабируется, однако оно немного ниже, чем у TBB или OMP с `schedule(dynamic)`. Это связано с
тем, что статическое разбиение (static chunking) плохо справляется с неравномерной нагрузкой. Если одному потоку
достанется блок столбцов с большим количеством ненулевых элементов, остальные потоки завершат работу раньше и будут
простаивать в ожидании `join()`.

## 9. Выводы

Ручная реализация на базе `std::thread` показала свою работоспособность и дала ощутимое ускорение. Главный вывод
заключается в том, что для задач с неравномерной вычислительной сложностью (таких как обработка разреженных матриц)
статическое распределение диапазонов является неоптимальным. Для достижения максимальной производительности на
`std::thread` потребовалось бы реализовать собственный пул потоков с динамической очередью задач, что "из коробки" уже
предоставляют TBB и OMP.
