## 1. Контекст

В данном отчете рассматривается реализация умножения разреженных матриц с использованием TBB. В отличие от OMP, который
опирается на директивы компилятора, TBB предоставляет задачно-ориентированную модель параллелизма (task-based
parallelism). Это позволяет более гибко управлять балансировкой нагрузки во время выполнения (runtime), что особенно
актуально для разреженных матриц с неравномерным распределением данных.

## 2. Постановка задачи

Необходимо выполнить умножение матриц `A` и `B`, заданных в формате CCS. Последовательная версия (SEQ) выступает в роли
эталона корректности и базовой метрики производительности. Ограничения на размерности и формат данных остаются прежними.

## 3. Базовый алгоритм

Алгоритм последовательно обходит столбцы матрицы `B`. Для каждого ненулевого элемента столбца извлекается
соответствующий столбец матрицы `A`, умножается на скаляр и прибавляется к плотному вектору-аккумулятору. После
обработки столбца результат фильтруется от нулей и сохраняется.

## 4. Схема распараллеливания

- **Примитив:** Для распараллеливания цикла по столбцам матрицы `B` используется алгоритм `tbb::parallel_for`.
- **Диапазон работы:** Задается через `tbb::blocked_range<int>(0, matrix_b.cols_count)`.
- **Grainsize и Partitioner:** Размер зерна (grainsize) явно не указан, поэтому используется `auto_partitioner` по
умолчанию. Для задачи умножения разреженных матриц это оптимальный выбор: планировщик TBB сам делит диапазон на
подзадачи (chunks) и применяет механизм "кражи задач" (work-stealing), если один из потоков освобождается раньше других
из-за разной плотности столбцов.
- **Управление памятью:** Чтобы избежать гонок данных, результаты вычислений пишутся в независимые двумерные векторы
`local_row_indices` и `local_values` по уникальному индексу столбца `j`.
- **Ограничение конкуренции:** Контролируется тестовым фреймворком через переменную окружения `PPC_NUM_THREADS`, которая
инициализирует планировщик TBB нужным числом рабочих потоков.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Ключевой особенностью реализации является оптимизация выделения памяти под аккумулятор. Вектор `col_accumulator`
создается не для каждого столбца отдельно, а один раз на весь блок (chunk), который обрабатывает поток:

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::parallel_for(tbb::blocked_range<int>(0, matrix_b.cols_count), [&](const tbb::blocked_range<int> &range) {
  // Аккумулятор выделяется один раз на весь range (блок итераций)
  std::vector<double> col_accumulator(matrix_a.rows_count, 0.0);

  for (int j = range.begin(); j != range.end(); ++j) {
    ProcessColumn(j, matrix_a, matrix_b, col_accumulator, local_row_indices[j], local_values[j]);
  }
});
```

Вызов вспомогательной функции `ProcessColumn` инкапсулирует математическую логику. Внутри `ProcessColumn` аккумулятор
очищается нулями после записи результатов столбца, что позволяет переиспользовать его на следующей итерации цикла `j`
без повторной аллокации памяти.

После первого `parallel_for` выполняется последовательный подсчет префиксных сумм для массива `col_ptrs`, а затем
запускается второй `parallel_for` для копирования данных в итоговые одномерные массивы.

## 6. Проверка корректности

Корректность проверялась путем сравнения с эталонной версией SEQ. Функциональные тесты успешно пройдены на граничных
случаях: пустые матрицы, единичные матрицы, матрицы с высокой степенью разреженности. Использование независимых буферов
для каждого столбца гарантирует отсутствие состояния гонки.

## 9. Выводы

Использование TBB оказалось крайне эффективным для данной задачи. Автоматический партиционировщик (`auto_partitioner`)
успешно распределил неравномерную нагрузку между потоками. Архитектурное решение с выделением локального аккумулятора на
уровне блока (range), а не отдельной итерации, позволило минимизировать затраты на работу с памятью, что сделало
реализацию быстрой и масштабируемой.
