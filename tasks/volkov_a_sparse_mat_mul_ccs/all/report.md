## 1. Контекст

В данном отчете описывается гибридная реализация умножения разреженных матриц. Использование комбинации MPI и OMP
позволяет эффективно задействовать вычислительные ресурсы кластерных систем: MPI отвечает за распределение работы между
узлами (процессами) с раздельной памятью, а OMP распараллеливает вычисления внутри каждого узла, используя общую память
многоядерного процессора.

## 2. Постановка задачи

Требуется реализовать умножение матриц `A` и `B` в формате CCS с использованием гибридной модели параллелизма.
Последовательная версия (SEQ) выступает эталоном. Ограничения на входные данные остаются неизменными. Дополнительное
требование: итоговый результат должен быть доступен на нулевом процессе (root), а в рамках тестов курса - корректно
возвращаться фреймворку.

## 3. Базовый алгоритм

Основа алгоритма не меняется: вычисление результирующей матрицы происходит по столбцам. Ненулевые элементы столбца
матрицы `B` умножаются на соответствующие столбцы матрицы `A`, результаты накапливаются, фильтруются от нулей и
упаковываются в формат CCS.

## 4. Межпроцессная схема

- **Роли процессов:** Процесс с рангом 0 (root) является управляющим. Он валидирует входные данные и собирает итоговый
результат. Остальные процессы выступают в роли рабочих (workers).
- **Распределение данных:** Нулевой процесс рассылает исходные матрицы `A` и `B` всем остальным процессам с помощью
`MPI_Bcast`. Общее количество столбцов матрицы `B` делится между процессами статически (каждый получает свой блок
`chunk` и возможный остаток `rem`).
- **Сборка результатов:** Поскольку MPI требует передачи данных в непрерывных участках памяти, рабочие процессы
"выравнивают" (flatten) свои двумерные векторы с результатами в одномерные массивы (`my_rows_flat`, `my_vals_flat`) и
отправляют их нулевому процессу через `MPI_Send`. Нулевой процесс принимает данные через `MPI_Recv` и восстанавливает
двумерную структуру.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса полученный диапазон столбцов `[start_col, end_col)` обрабатывается параллельно с помощью
OMP.

- Используется директива `#pragma omp parallel` с созданием потоколокального аккумулятора `col_accumulator`.
- Цикл по столбцам распределяется между потоками с помощью `#pragma omp for schedule(dynamic)`, что обеспечивает
балансировку нагрузки при работе с разреженными данными.
- Нулевой процесс также использует OMP (`schedule(static)`) на финальном этапе для параллельного копирования собранных
данных в итоговые одномерные массивы матрицы `C`.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Ключевой фрагмент кода демонстрирует гибридную природу вычислений: сначала определяется диапазон для процесса, а затем
он обрабатывается потоками:

```cpp
// File: all/src/ops_all.cpp
int start_col = (world_rank * chunk) + std::min(world_rank, rem);
int end_col = start_col + chunk + (world_rank < rem ? 1 : 0);
int my_cols_count = end_col - start_col;

// ... выделение памяти под локальные результаты ...

// Внутрипроцессное распараллеливание (OMP)
#pragma omp parallel default(none) shared(matrix_a, matrix_b, start_col, end_col, local_row_indices, local_values)
{
  std::vector<double> col_accumulator(matrix_a.rows_count, 0.0);
#pragma omp for schedule(dynamic)
  for (int j = start_col; j < end_col; ++j) {
    int local_j = j - start_col;
    ProcessColumnOmp(j, matrix_a, matrix_b, col_accumulator, local_row_indices[local_j], local_values[local_j]);
  }
}
```

После этого блока рабочие процессы вызывают функцию `FlattenLocalData` для подготовки данных к отправке по сети (или
через разделяемую память в случае запуска на одной машине). В методе `PostProcessingImpl` нулевой процесс делает
финальный `MPI_Bcast` матрицы `C`, чтобы все процессы имели консистентное состояние для проверяющего фреймворка.

## 7. Проверка корректности

Корректность проверялась сравнением с эталонной версией SEQ. Функциональные тесты запускались в конфигурациях 2 и 4
процессов. Проверена согласованность результатов: нулевой процесс успешно собирает данные от воркеров, смещения
(`offset`) рассчитываются верно, потери ненулевых элементов не происходит.

## 8. Экспериментальная среда

- CPU: Intel Core i5-10400F (6 ядер, 12 потоков)
- RAM: 16 GB

## 10. Выводы

Гибридная схема (MPI + OMP) успешно реализована и выдает корректные результаты. Она оправдана при запуске на
вычислительных кластерах, где матрицы не помещаются в память одного узла или требуется распределить колоссальную
вычислительную нагрузку. Однако при запуске на локальном ПК (в рамках одного процессора) затраты на межпроцессную
коммуникацию и упаковку/распаковку данных (flattening) снижают общую эффективность масштабирования по сравнению с
легковесными потоковыми моделями.
