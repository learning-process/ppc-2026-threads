# Умножение разреженных матриц (формат CCS)

## 1. Введение

Данный проект посвящен исследованию различных моделей параллельного программирования на примере классической задачи
вычислительной линейной алгебры - умножения разреженных матриц. Использование формата CCS (Compressed Column Storage)
делает задачу интересной для распараллеливания: вычисление каждого столбца результирующей матрицы не зависит от других,
однако неравномерное распределение ненулевых элементов создает проблемы с балансировкой нагрузки. Цель работы - сравнить
эффективность технологий OMP, TBB, `std::thread` и MPI на одной и той же аппаратной платформе.

## 2. Единая постановка задачи

На вход подаются две разреженные матрицы `A` и `B`, хранящиеся в формате CCS (массивы `values`, `row_indices`,
`col_ptrs`). Требуется вычислить матрицу `C = A * B` и также представить ее в формате CCS.
Обязательное условие корректности: количество столбцов матрицы `A` должно совпадать с количеством строк матрицы `B`.
Элементы, по модулю меньшие `1e-10`, считаются нулями и не включаются в итоговую структуру.

## 3. Единая методика эксперимента

Все замеры производились в строго одинаковых условиях для исключения влияния внешних факторов:

- **Аппаратная среда:** CPU Intel Core i5-10400F (6 ядер, 12 потоков), 16 GB RAM.
- **Программная среда:** Windows 11 Pro, запуск через Docker 4.74.0 (контейнер курса), компилятор GCC, тип сборки
`Release`.
- **Метрики:** Ускорение (Speedup) рассчитывалось как отношение времени работы последовательной версии (SEQ) к времени
параллельной. Эффективность (Efficiency) - как отношение ускорения к числу вычислительных единиц (workers).
- **Вычислительные единицы:** Для OMP, TBB и STL использовалось 4 потока (`PPC_NUM_THREADS=4`). Для гибридной версии ALL
использовалась конфигурация из 2 процессов по 4 потока в каждом (`PPC_NUM_PROC=2`, итого 8 работников).
- **Набор данных:** Использовались случайно сгенерированные матрицы размеров 500x500 (плотность 10%), 1000x1000
(плотность 5%) и 2000x2000 (плотность 5%).

## 4. Сводка корректности

Все пять реализаций успешно прошли единый набор функциональных тестов. Корректность подтверждена на крайних случаях
(умножение на нулевую матрицу, умножение на единичную матрицу) и на случайных данных. Во всех параллельных версиях (OMP,
TBB, STL, ALL) проблема состояния гонки (race condition) была решена архитектурно: каждый поток записывал результаты
строго в свой независимый двумерный буфер, что исключило необходимость использования мьютексов и атомарных операций в
вычислительном ядре.

## 6. Интерпретация различий

Сравнение технологий выявило следующие закономерности:

1. **TBB показал лучший результат.** Задачно-ориентированная модель и `auto_partitioner` идеально справились с
неравномерной плотностью разреженных матриц, динамически перераспределяя нагрузку между потоками без простоев.
2. **OMP оказался почти наравне с TBB.** Использование директивы `schedule(dynamic)` позволило достичь отличной
балансировки. Небольшое отставание от TBB находится в пределах погрешности измерений и накладных расходов на старт
параллельной секции.
3. **STL показал наименьшее ускорение среди потоковых моделей.** Причина кроется в ручном статическом разбиении данных
(static chunking). Если одному потоку доставался блок столбцов с большим числом ненулевых элементов, остальные потоки
простаивали в ожидании `join()`.
4. **ALL (MPI+OMP) проиграл на локальной машине.** Несмотря на использование 8 работников, гибридная версия оказалась
медленнее 4-поточных реализаций. Это методически верный результат: накладные расходы на сериализацию данных
(flattening), пересылку матриц через `MPI_Bcast` и сборку через `MPI_Send/Recv` внутри одного физического узла превышают
выгоду от дополнительных потоков. Технология ALL предназначена для распределенных кластеров, а не для локальных ПК.


## 8. Заключение

В ходе выполнения работы алгоритм умножения разреженных матриц был успешно адаптирован под различные парадигмы
параллельного программирования. Наилучшим выбором для систем с общей памятью (shared memory) признана TBB,
предоставляющая наиболее удобный и эффективный механизм динамической балансировки нагрузки. Использование MPI оправдано
только в случае выхода за рамки одного вычислительного узла.
