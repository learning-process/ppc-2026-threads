## 1. Контекст

В OMP-версии задача параллелизации решается через разбиение входного массива на равные непересекающиеся подмассивы,
которые каждый поток сортирует независимо с помощью того же алгоритма, что и в SEQ. После завершения параллельной
сортировки отсортированные чанки объединяются последовательным слиянием (Merge). Такой подход является прямой
параллельной надстройкой над SEQ-эталоном и позволяет значительно повысить эффективность алгоритма, задействуя
возможности многопоточных систем.

## 2. Постановка задачи

**Краткая постановка задачи:** необходимо отсортировать массив целых чисел в порядке неубывания.

Baseline для сравнения — [`seq/report.md`](../seq/report.md) c временем выполнения $T_{\text{seq}}$ при $p = 1$ на том же
наборе данных и в том же окружении.

## 3. Базовый алгоритм

Последовательная основа — итеративный алгоритм сортировки Хоара, включающий в себя выбор опорного элемента из середины,
разбиение массива относительно опорного элемента и итеративную сортировку каждого подмассива с явным стеком подмассивов.

## 4. Схема распараллеливания

**Область параллелизма.** Параллелизации подвергается только фаза сортировки: массив длины $N$ делится на $p$
непересекающихся чанков, каждый из которых сортируется одним потоком независимо. Фаза слияния $p$ отсортированных
чанков выполняется последовательно.

**Разбиение данных.** Размер каждого чанка равен $⌊\frac{N}{p}⌋$; последний чанк поглощает остаток $N\mod p$. Поток с
номером $t$ отвечает за диапазон индексов $[t \cdot ⌊\frac{N}{p}⌋, (t+1) \cdot ⌊\frac{N}{p}⌋ − 1]$, а поток $p−1$ — за
$[(p−1) \cdot ⌊\frac{N}{p}⌋, n−1]$. Диапазоны строго непересекаются.

**Разбор директивы.**

- `#pragma omp parallel` — открывает параллельную область; каждый поток выполняет весь тело блока.
- `default(none)` — запрещает неявное разделение переменных; все переменные должны быть перечислены явно. Это защищает
от случайных гонок на переменных, забытых в списке атрибутов.
- `shared(data, borders, num_threads)` — вектор `data`, массив границ `borders` и счётчик `num_threads` разделяются
между потоками. Гонки на `data` нет, поскольку каждый поток пишет только в свой непересекающийся диапазон.
- `num_threads(num_threads)` — явно задаёт число потоков.
- `reduction` не применяется: результат накапливается напрямую в разделённый, но неконкурентно записываемый вектор
`data`. Каждый поток имеет собственный непересекающийся поддиапазон, поэтому атомарные операции и критические секции не
нужны.
- `schedule` не указан явно, поэтому распределение по потокам определяется по умолчанию как статическое. Каждый поток
получает ровно один чанк.
- `parallel for` не используется намеренно: итерации неравномерны по стоимости (алгоритм на разных чанках может работать
разное время), поэтому явное статическое разбиение через `borders` даёт более предсказуемое поведение.

**Синхронизация.** В конце параллельной области присутствует неявный барьер OpenMP: ни один поток не приступает к фазе
слияния, пока все потоки не завершат сортировку своего чанка. Явных `critical`, `atomic` и `barrier` директив не
требуется, поскольку гонки исключены на уровне разбиения данных.

**Число потоков.** Число потоков берётся из переменной окружения `PPC_NUM_THREADS`, которую инфраструктура курса
экспортирует также как `OMP_NUM_THREADS`.

**Обработка малых массивов.** Если $N < 2p$, параллельное разбиение нецелесообразно: чанки вырождаются в пустые или
одноэлементные. В этом случае весь массив сортируется однопоточно без создания параллельной секции.

**Ограничение ускорения.** Последовательное слияние $p$ чанков выполняется за $O(N \cdot p)$ шагов и является
последовательной частью. При росте $p$ эта часть становится доминирующей и ограничивает практическое ускорение ниже
теоретического линейного предела.

## 5. Детали реализации

**Файлы:** [`omp/include/ops_omp.hpp`](include/ops_omp.hpp), [`omp/src/ops_omp.cpp`](src/ops_omp.cpp)

**Что изменилось относительно SEQ:**

- [`Quicksort`](src/ops_omp.cpp#L77): расширен параметрами `low`, `high`, что позволяет запускать метод на произвольном
диапазоне.
- Добавлен метод [`Merge(data, left, mid, right)`](src/ops_omp.cpp#L97) $-$ стандартное слияние двух упорядоченных
подмассивов `[left..mid]` и `[mid+1..right]` через временный буфер размером `right − left + 1` с обратной записью
результата в `data[left..right]`.
- Метод [`RunImpl`](src/ops_omp.cpp#L123) переработан: строит массив границ чанков `borders`, запускает параллельную
секцию, а после неявного барьера последовательно выполняет `num_threads − 1` слияний, каждый раз расширяя
отсортированный префикс на один чанк.

**Параллельная секция:**

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel default(none) shared(data, borders, num_threads) num_threads(num_threads)
{
  int tid = omp_get_thread_num();
  if (tid < num_threads) {
    Quicksort(data, borders[tid], borders[tid + 1] - 1);
  }
}
```

> Примечание: `default(none)` запрещает неявное разделение переменных — это предотвращает случайные гонки на
> незадекларированных переменных. `shared(data, borders, num_threads)` декларирует три переменные с общим доступом.
> Гонки на `data` нет, так как диапазоны записи непересекаются. `num_threads(num_threads)` явно фиксирует число потоков.
> `tid` объявлен внутри блока и является `private` автоматически.

**Последовательное слияние:**

```cpp
// File: omp/src/ops_omp.cpp
for (int i = 1; i < num_threads; i++) {
  Merge(data, 0, borders[i] - 1, borders[i + 1] - 1);
}
```

> Примечание: На каждой итерации `i` накопленный отсортированный префикс `[0, borders[i] − 1]` сливается с чанком
> `[borders[i], borders[i+1] − 1]`. После `num_threads − 1` слияний весь массив отсортирован.

**Устранённые риски гонок.** Гонка по записи в `data` исключена структурой разбиения: каждый поток пишет только в свой
диапазон `[borders[tid], borders[tid+1]−1]`. Массив `borders` только читается внутри параллельной секции. `num_threads`
константен на всём протяжении `RunImpl`. Дополнительных `mutex`, `atomic` или `critical` секций не требуется.

### Масштабируемость и узкие места

На диапазоне $p$ от $1$ до $8$ алгоритм масштабируется положительно: время сокращается 0.67 с до 0.21–0.215 с. Однако
эффективность падает со 100% при $p = 1$ до 38–39% при $p = 8$, что свидетельствует о нарастающей доле последовательной
работы. При $p = 16$ наблюдается деградация: время возрастает до 0.298 с, а ускорение снижается с максимального
3.14 до 2.27–2.28. Таким образом, последовательная фаза слияния занимает фиксированную долю работы $O(N \cdot p)$, и
при достаточно большом $p$ она полностью поглощает параллельный выигрыш.

Основное узкое место — **последовательное слияние чанков**: на каждом из $p − 1$ шагов цикла слияния обрабатывается
весь накопленный префикс массива, суммарная стоимость составляет $O(N \cdot p)$. При $p = 16$ эта часть уже доминирует
над параллельной фазой сортировки. Вторичное ограничение — **неравномерность нагрузки**: `Quicksort` на разных чанках
завершается в разное время, и быстрые потоки простаивают у неявного барьера в ожидании медленных. Распределение чанков
статическое и равномерное по числу элементов, но не по реальной стоимости сортировки — это также снижает эффективность
при $p > 4$.

## 9. Выводы

Использование OpenMP оправдано в диапазоне $p$ от $2$ до $8$, при которых параллельная фаза сортировки независимых
чанков доминирует над накладными расходами, и алгоритм приносит реальный выигрыш по времени. Практически оптимальное
значение достигается при $p = 8$ (speedup ≈ 3.14, время ≈ 0.215 с), после которого дальнейшее увеличение числа потоков
контрпродуктивно. Выигрыш ограничен двумя взаимосвязанными причинами: последовательное слияние с растущей стоимостью
$O(N \cdot p)$ сужает параллельную долю, а статическое равномерное разбиение не компенсирует неравномерность стоимости
сортировки на разных чанках, оставляя быстрые потоки простаивать у барьера.
