## 1. Контекст
STL-версия реализует ту же декомпозицию «сортировка чанков параллельно — слияние последовательно», что и OMP и TBB, но без каких-либо сторонних библиотек параллелизма: каждый поток создаётся явно через `std::thread`, а синхронизация сводится к `join`. Это делает схему максимально прозрачной: нет планировщика задач, нет неявных барьеров, нет директив компилятора — только явное создание потоков, явное ожидание их завершения и явная связка потоков с диапазонами данных.

## 2. Постановка задачи
**Краткая постановка задачи:** необходимо отсортировать массив целых чисел в порядке неубывания.
 
Baseline для сравнения — [`seq/report.md`](../seq/report.md) c временем выполнения $T_{seq}$ при $p = 1$ на том же наборе данных и в том же окружении.

## 3. Базовый алгоритм
Последовательная основа — итеративный алгоритм сортировки Хоара, включающий в себя выбор опорного элемента из середины, разбиение массива относительно опорного элемента и итеративную сортировку каждого подмассива с явным стеком подмассивов.

## 4. Схема распараллеливания
**Ручное разбиение диапазонов.** Массив длины $N$ делится на $p$ чанков: размер каждого равен $⌊\frac{N}{p}⌋$, последний чанк поглощает остаток $N \mod p$. Чанк $i$ охватывает индексы $[i \cdot ⌊\frac{N}{p}⌋, (i+1) \cdot ⌊\frac{N}{p}⌋ − 1]$. Границы диапазонов вычисляются заранее и фиксируются отдельно в памяти до запуска потоков. Каждый поток получает ровно один чанк — явная, статическая карта "поток $-$ блок данных".
 
**Связка потоков с блоками данных.** Поток `i` создаётся с лямбдой, захватывающей `data`, `borders` и значение `i` по значению. Лямбда вызывает метод сортировки на диапазоне `[borders[i], borders[i+1]-1]`. Таким образом, каждый поток однозначно знает свой диапазон до начала выполнения, и этот диапазон не меняется в процессе работы — балансировки нагрузки нет.
 
**Локальные буферы.** Локальный стек разбиений создаётся внутри метода сортировки на стеке каждого потока и не разделяется между потоками. Временный буфер в методе слияния также локален. Таким образом, никакие промежуточные структуры не являются разделяемыми.
 
**`join` и синхронизация.** После запуска всех $p$ потоков главный поток последовательно вызывает `t.join()` для каждого из них. `join` блокирует вызывающий поток до завершения соответствующего рабочего потока. Только после того, как все `join` завершены, начинается фаза последовательного слияния. Гонок нет: каждый поток пишет только в свой диапазон `[borders[i], borders[i+1]−1]`, `borders` только читается. `atomic` и `mutex` не требуются.
 
**Число потоков.** Число потоков определяется через `std::thread::hardware_concurrency()` — стандартный способ запросить число аппаратных потоков у ОС. Если `hardware_concurrency()` возвращает 0 (платформа не поддерживает запрос), число потоков принудительно устанавливается равным 2. Это принципиальное отличие от OMP и TBB: число потоков не берётся из `PPC_NUM_THREADS`, а определяется аппаратно.
 
**Деградация при малых данных.** Если $n \lt 2p$, чанки вырождаются. В этом случае весь массив сортируется однопоточно без создания `std::thread`.
 
**Ограничение ускорения.** Последовательное слияние $p$ чанков стоит $O(N \cdot p)$ и является последовательной частью. Дополнительно, отсутствие балансировки нагрузки (статическое разбиение, отсутствует механизм "work stealing") означает, что при неравномерной стоимости сортировки чанков быстрые потоки простаивают, ожидая медленных у `join`.

## 5. Детали реализации
**Файлы:** [`stl/include/ops_stl.hpp`](include/ops_stl.hpp), [`stl/src/ops_stl.cpp`](src/ops_stl.cpp)

**Изменения относительно SEQ.** 
- [`Quicksort`](src/ops_tbb.cpp#L80): расширен параметрами `low`, `high`, что позволяет запускать метод на произвольном диапазоне.
- Добавлен метод [`Merge(data, left, mid, right)`](src/ops_tbb.cpp#L100) $-$ стандартное слияние двух упорядоченных подмассивов `[left..mid]` и `[mid+1..right]` через временный буфер размером `right − left + 1` с обратной записью результата в `data[left..right]`.
- В [`RunImpl`](src/ops_tbb.cpp#L126) добавлены: определение числа потоков через `ppc::util::GetNumThreads()`, вектор `threads`, вектор границ `borders`, цикл запуска потоков, цикл `join` и цикл последовательного слияния.

**Запуск потоков:**
 
```cpp
// File: stl/src/ops_stl.cpp
for (int i = 0; i < num_threads; i++) {
  threads.emplace_back([&data, &borders, i]() {
    rozenberg_a_quicksort_simple_merge::RozenbergAQuicksortSimpleMergeSTL::Quicksort(
        data, borders[i], borders[i + 1] - 1);
  });
}
```
 
> Примечание: Лямбда захватывает `data` и `borders` по ссылке, а `i` — по значению. Захват `i` по значению принципиален: если бы `i` захватывался по ссылке, к моменту выполнения лямбды значение `i` в главном потоке уже изменилось бы. `data` захвачен по ссылке — все потоки работают с одним вектором, но в непересекающихся диапазонах. `borders` захвачен по ссылке и используется только для чтения.
 
**Ожидание завершения:**
 
```cpp
// File: stl/src/ops_stl.cpp
for (auto &t : threads) {
  if (t.joinable()) {
    t.join();
  }
}
```
 
> Примечание: Проверка `joinable()` перед `join()` защищает от двойного join на уже завершённом потоке. После этого цикла все чанки гарантированно отсортированы, и главный поток приступает к слиянию.
 
**Определение числа потоков:**
 
```cpp
// File: stl/src/ops_stl.cpp
int num_threads = ppc::util::GetNumThreads();
```
 
> Примечание: `GetNumThreads()` читает переменную окружения `PPC_NUM_THREADS`, обеспечивая единое управление числом потоков для всех backend-ов в инфраструктуре курса.
 
**Последовательное слияние:**
 
```cpp
// File: stl/src/ops_stl.cpp
for (int i = 1; i < num_threads; i++) {
  Merge(data, 0, borders[i] - 1, borders[i + 1] - 1);
}
```
 
> Примечание: На каждой итерации `i` накопленный отсортированный префикс `[0, borders[i] − 1]` сливается с чанком `[borders[i], borders[i+1] − 1]`. После `p − 1` слияний весь массив отсортирован.
 
**Устранённые риски гонок.** Каждый поток пишет только в диапазон `[borders[i], borders[i+1]−1]` — диапазоны непересекаются. `borders` только читается внутри потоков. Локальный стек `Quicksort` и буфер `Merge` создаются на стеке каждого потока и не разделяются. `atomic` и `mutex` отсутствуют намеренно — они не нужны при данной структуре доступа к данным.
