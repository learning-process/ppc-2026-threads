## 1. Контекст

В данной реализации комбинируются технологии OpenMP и MPI. Гибридная версия добавляет второй уровень параллелизма поверх
OMP-схемы: массив делится между процессами MPI, каждый процесс независимо сортирует свою часть с помощью OpenMP-потоков,
после чего ранг 0 собирает и сливает отсортированные части. Такая схема оправдана, когда данные не помещаются в память
одного узла или когда желательно задействовать несколько физических машин. На одном узле гибридная версия дополнительно
нагружает коммуникационный стек MPI, что создаёт издержки, отсутствующие в чисто поточных backend-ах.

## 2. Постановка задачи

**Краткая постановка задачи:** необходимо отсортировать массив целых чисел в порядке неубывания.

Baseline для сравнения — [`seq/report.md`](../seq/report.md) c временем выполнения $T_{\text{seq}}$ при $p = 1$ на том же
наборе данных и в том же окружении.

## 3. Базовый алгоритм

Последовательная основа — итеративный алгоритм сортировки Хоара, включающий в себя выбор опорного элемента из середины,
разбиение массива относительно опорного элемента и итеративную сортировку каждого подмассива с явным стеком подмассивов.

## 4. Межпроцессная схема

**Роли рангов.** Ранг 0 является координатором: он владеет входными данными, раздаёт чанки остальным рангам, получает от
них отсортированные части и выполняет финальное слияние. Ранги $[1..\text{size} − 1]$ являются рабочими: каждый получает
свой чанк, сортирует его и отправляет обратно рангу 0. Ранг 0 также получает свой чанк для локальной сортировки. Результат
всей сортировки присутствует только в памяти ранга 0.

**Распределение данных.** Размер массива $N$ рассылается всем рангам через `MPI_Bcast`. Затем массив делится на $size$
чанков: чанк ранга $r$ содержит $⌊\frac{N}{\text{size}}⌋$ элементов, первые $N \mod \text{size}$ рангов получают на один
элемент больше.

**MPI-вызовы и их роль.**

- `MPI_Bcast` — ранг 0 передаёт длину массива всем рангам до рассылки данных. Без данного вызова рабочие ранги не знают
размер своего буфера.
- `MPI_Scatterv` — ранг 0 рассылает непрерывные чанки разной длины каждому рангу. `Scatterv` в отличие от `Scatter`
поддерживает неравномерные размеры чанков (остаток от деления), что необходимо при $N \mod \text{size} \ne 0$.
- `MPI_Send` / `MPI_Recv` — после локальной сортировки ранги $[1..{\text{size}} − 1]$ отправляют результат рангу 0 блокирующим
`MPI_Send`; ранг 0 принимает их последовательно блокирующим `MPI_Recv`. Последовательный приём упрощает слияние: каждый
принятый буфер немедленно сливается с накопленным префиксом.

**Синхронизация между уровнями.** Явного `MPI_Barrier` нет: синхронизация обеспечивается семантикой блокирующих
коммуникаций. `MPI_Scatterv` гарантирует, что все ранги получили данные до начала локальной сортировки. `MPI_Send` /
`MPI_Recv` гарантируют, что ранг 0 приступает к слиянию только после полного получения очередного чанка.

## 5. Внутрипроцессная схема

**Технология потоков.** Внутри каждого ранга параллелизм реализован через OpenMP. Функция локальной сортировки получает
локальный чанк ранга и применяет к нему похожую декомпозицию, что и в OMP-версии: делит чанк на $t$ подчанков, где $t$
$-$ число OMP-потоков, сортирует их параллельно директивой `#pragma omp parallel for`, затем последовательно сливает.

**Деление работы между потоками.** Размер локального чанка ранга $N_{\text{local}}$ делится на $t$ подчанков по
$⌊\frac{N_{\text{local}}}{t}⌋$ элементов. Каждый OpenMP-поток сортирует один подчанк независимо; диапазоны непересекаются.

**Объединение локальных результатов.** После завершения OMP-директивы (неявный барьер OpenMP) $t-1$ отсортированных
подчанков объединяются последовательным слиянием через — аналогично OMP-версии. Результат — полностью отсортированный
локальный чанк ранга.

**Деградация.** Если $N_{\text{local}} \le t$, деление на подчанки нецелесообразно: весь локальный чанк сортируется однопоточно.

## 6. Детали реализации

**Файлы:** [`all/include/ops_all.hpp`](include/ops_all.hpp), [`all/src/ops_all.cpp`](src/ops_all.cpp)

**Изменения относительно SEQ.**

- [`Quicksort`](src/ops_all.cpp#L78) расширен параметрами `low` и `high`.
- Добавлены [`ThreadMerge`](src/ops_all.cpp#L115) (слияние в рамках одного ранга, идентично [`Merge`](../omp/src/ops_omp.cpp#L97)
из OMP-версии) и `Merge` — новая перегрузка, принимающая два `const InType&` и возвращающая новый вектор. Используется
рангом 0 для последовательного слияния чанков от рабочих рангов.
- Добавлена функция [`ThreadQuicksort`](src/ops_all.cpp#L141), инкапсулирующая внутрипроцессный OpenMP-параллелизм.
- [`RunImpl`](src/ops_all.cpp#L167) переработан: добавлены все MPI-вызовы, ветвление по рангу и финальное слияние на
ранге 0.

**Рассылка и сбор данных:**

```cpp
// File: all/src/ops_all.cpp
MPI_Bcast(&n, 1, MPI_INT, 0, MPI_COMM_WORLD);
// ...
MPI_Scatterv(rank == 0 ? GetInput().data() : nullptr,
             send_counts.data(), displs.data(), MPI_INT,
             local_data.data(), send_counts[rank], MPI_INT,
             0, MPI_COMM_WORLD);
```

> Примечание: На ранге 0 определяется общий размер входного массива, который затем рассылается остальным рангам.
> Ранг 0 передаёт `GetInput().data()` как буфер рассылки; остальные ранги передают `nullptr` — стандартная
> идиома `MPI_Scatterv`. `send_counts[r]` и `displs[r]` предварительно вычислены на всех рангах из `n` и `size`, без
> дополнительных коммуникаций.

**Внутрипроцессная параллельная сортировка (`ThreadQuicksort`):**

```cpp
// File: all/src/ops_all.cpp
#pragma omp parallel for default(none) shared(local_data, thr_borders, num_threads) num_threads(num_threads)
for (int i = 0; i < num_threads; i++) {
  Quicksort(local_data, thr_borders[i], thr_borders[i + 1] - 1);
}
for (int i = 1; i < num_threads; i++) {
  ThreadMerge(local_data, 0, thr_borders[i] - 1, thr_borders[i + 1] - 1);
}
```

> Примечание: `parallel for` распределяет итерации по потокам OpenMP. `default(none)` и `shared` обеспечивают явное
> декларирование атрибутов переменных — гонок нет, так как каждый поток пишет только в свой подчанк. После неявного
> барьера `parallel for` подчанки сливаются последовательно.

**Отправка результата рабочими рангами и финальное слияние на ранге 0:**

```cpp
// File: all/src/ops_all.cpp
if (rank != 0) {
  MPI_Send(local_data.data(), static_cast<int>(local_data.size()), MPI_INT, 0, 0, MPI_COMM_WORLD);
```

> Примечание: Ранги `[1..size−1]` отправляют отсортированный локальный чанк рангу 0 блокирующим `MPI_Send` с тегом 0.
> Вызов блокирует рабочий ранг до тех пор, пока буфер не может быть повторно использован (стандартная семантика MPI).
> После этого рабочий ранг завершает `RunImpl`.

**Финальное слияние на ранге 0:**

```cpp
// File: all/src/ops_all.cpp
InType total_data = local_data;
for (int i = 1; i < size; ++i) {
  InType recv_buf(send_counts[i]);
  MPI_Recv(recv_buf.data(), send_counts[i], MPI_INT, i, 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
  auto merged = Merge(total_data, recv_buf);
  total_data.swap(merged);
}
GetOutput() = total_data;
```

> Примечание: Ранг 0 принимает чанки по одному в порядке рангов `[1..size−1]` и сливает с накопленным результатом.
> `Merge` возвращает новый вектор, `swap` избегает копирования. Порядок приёма фиксирован (по рангу `i`), что исключает
> неопределенное поведение при сборе.

**Потенциальные узкие места.** Финальное слияние на ранге 0 является полностью последовательным и выполняется за
$O(N \cdot \text{size})$ операций — то же ограничение, что и в OMP. Последовательный `MPI_Recv` в цикле означает, что
ранг 0
ожидает ранги строго по порядку: если ранг `i` завершает позже других, остальные уже готовые чанки ожидают приёма.
Внутрипроцессное слияние подчанков OpenMP также последовательно.

## 7. Проверка корректности

Корректность проверяется аналогично [SEQ-версии](../seq/report.md#5-проверка-корректности). Дополнительно результат
должен совпадать с SEQ при любой конфигурации числа рангов и потоков.
Согласованность между рангами проверяется косвенно: поскольку результат собирается только на ранге 0, тест запускается
на ранге 0 и сравнивается с эталоном — если хотя бы один рабочий ранг прислал некорректно отсортированный чанк,
итоговое слияние на ранге 0 даст неверный результат, и тест упадёт.

## 10. Выводы

Гибридная схема оправдывает себя уже при 2 рангах:
конфигурация `2×4` (8 работников) превосходит любой чисто поточный
backend при том же числе потоков. Оптимальная конфигурация на данном
оборудовании — `4×4` (task speedup 4.13). При дальнейшем росте числа
рангов (`8×1`, `8×2`) прирост ускорения замедляется из-за накладных расходов
на `MPI_Send`/`MPI_Recv` и последовательного слияния на ранге 0.
