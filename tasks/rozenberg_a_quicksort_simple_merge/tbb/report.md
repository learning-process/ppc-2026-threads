## 1. Контекст

TBB-версия воспроизводит ту же декомпозицию, что и OMP-версия: массив делится на равные чанки, каждый сортируется
независимо, затем чанки объединяются последовательным слиянием. Отличие от OMP заключается в механизме параллелизма:
вместо явных потоков с ручным управлением используется планировщик задач TBB, который распределяет итерации по рабочим
потокам пула.

## 2. Постановка задачи

**Краткая постановка задачи:** необходимо отсортировать массив целых чисел в порядке неубывания.

Baseline для сравнения — [`seq/report.md`](../seq/report.md) c временем выполнения $T_{\text{seq}}$ при $p = 1$ на том же
наборе данных и в том же окружении.

## 3. Базовый алгоритм

Последовательная основа — итеративный алгоритм сортировки Хоара, включающий в себя выбор опорного элемента из середины,
разбиение массива относительно опорного элемента и итеративную сортировку каждого подмассива с явным стеком подмассивов.

## 4. Схема распараллеливания

**Примитив параллелизма.** Параллельная фаза реализована через `tbb::parallel_for` — алгоритмический шаблон TBB,
принимающий итерационное пространство и тело в виде лямбды. `parallel_for` не привязан к конкретным потокам: он
порождает задачи, которые планировщик TBB распределяет по пулу рабочих потоков динамически.

**`blocked_range` и `grainsize`.** Итерационное пространство задаётся через `tbb::blocked_range` как диапазон
целочисленных индексов чанков от $0$ до $p−1$. `grainsize` явно не задан, поэтому используется значение по умолчанию
равное 1. Это означает, что планировщик вправе выдать каждому потоку ровно по одному индексу-чанку, не объединяя их в
более крупные блоки. При небольшом значении $p$ и тяжёлых задачах накладные расходы на дробление несущественны, поэтому
использование `grainsize = 1` в данном случае оправдано.

**Partitioner.** Partitioner явно не указан, используется `tbb::auto_partitioner` по умолчанию. `auto_partitioner`
адаптивно дробит диапазон в зависимости от реальной загрузки потоков: при равномерной нагрузке он ведёт себя как
статический, при неравномерной — включает механизм **кражи задач** (work stealing), позволяя свободному потоку
захватить подзадачу из очереди занятого.

**Разбиение данных.** Массив длины $N$ делится на $p$ чанков: размер каждого равен $⌊\frac{N}{p}⌋$, последний чанк
поглощает остаток $N \mod p$. Чанк $i$ охватывает индексы $[i \cdot ⌊\frac{N}{p}⌋, (i+1) \cdot ⌊\frac{N}{p}⌋ − 1]$.
Диапазоны строго непересекаются, что исключает конкурентную запись в `data` без каких-либо примитивов синхронизации.

**Ограничение конкуренции.** Максимальное число потоков пула ограничивается через
`tbb::global_control::max_allowed_parallelism` значением `ppc::util::GetNumThreads()` (переменная окружения
`PPC_NUM_THREADS`). `global_control` действует как RAII-объект на всё время жизни вызывающего алгоритм метода и
гарантирует, что TBB не создаст сверхлимитных потоков. Это эквивалент `num_threads(p)` в OMP, но реализованный на уровне
всего планировщика пула, а не отдельной директивы.

**Синхронизация.** `tbb::parallel_for` является **блокирующим** вызовом: он возвращает управление вызывающему потоку
только после того, как все задачи диапазона завершены. Явных барьеров, мьютексов и атомарных операций не требуется.
Фаза последовательного слияния, следующая за `parallel_for`, гарантированно видит полностью отсортированные чанки.

**Деградация при малых данных.** Если $N \lt 2p$, параллельное разбиение нецелесообразно: чанки вырождаются. В этом
случае весь массив сортируется однопоточно без запуска `parallel_for`.

**Ограничение ускорения.** Последовательное слияние $p$ чанков стоит $O(N \cdot p)$ и является последовательной частью.
`auto_partitioner` с механизмом work stealing устраняет простой быстрых потоков у барьера, но не сокращает стоимость
самого слияния.

## 5. Детали реализации

**Файлы:** [`tbb/include/ops_tbb.hpp`](include/ops_tbb.hpp), [`tbb/src/ops_tbb.cpp`](src/ops_tbb.cpp)

**Изменения относительно SEQ.**

- [`Quicksort`](src/ops_tbb.cpp#L80): расширен параметрами `low`, `high`, что позволяет запускать метод на произвольном
диапазоне.
- Добавлен метод [`Merge(data, left, mid, right)`](src/ops_tbb.cpp#L100) $-$ стандартное слияние двух упорядоченных
подмассивов `[left..mid]` и `[mid+1..right]` через временный буфер размером `right − left + 1` с обратной записью
результата в `data[left..right]`.
- В [`RunImpl`](src/ops_tbb.cpp#L126) добавлены: построение массива границ чанков `borders`, объект `tbb::global_control`
для ограничения числа потоков пула, вызов `tbb::parallel_for` для параллельной сортировки чанков и цикл последовательного
слияния.

**Ограничение числа потоков:**

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::global_control control(tbb::global_control::max_allowed_parallelism, num_threads);
```

> Примечание: `global_control` действует как RAII-объект: ограничение активно на всём времени жизни переменной `control`
> в `RunImpl` и снимается при её разрушении.

**Параллельная фаза:**

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::parallel_for(tbb::blocked_range<int>(0, num_threads), [&](const tbb::blocked_range<int> &range) {
  for (int i = range.begin(); i != range.end(); i++) {
    Quicksort(data, borders[i], borders[i + 1] - 1);
  }
});
```

> Примечание: `tbb::blocked_range<int>(0, num_threads)` задаёт итерационное пространство из `num_threads` индексов чанков.
> TBB вправе разбить его на несколько подзадач и распределить по потокам пула. Лямбда захватывает `data` и `borders` по
> ссылке (`[&]`); гонок нет, поскольку каждый индекс `i` однозначно определяет непересекающийся диапазон записи.
> `parallel_for` блокирует вызывающий поток до завершения всех подзадач — явного барьера не требуется.

**Последовательное слияние:**

```cpp
// File: tbb/src/ops_tbb.cpp
for (int i = 1; i < num_threads; i++) {
  Merge(data, 0, borders[i] - 1, borders[i + 1] - 1);
}
```

> Примечание: На каждой итерации `i` накопленный отсортированный префикс `[0, borders[i] − 1]` сливается с чанком
> `[borders[i], borders[i+1] − 1]`. После `num_threads − 1` слияний весь массив отсортирован.

**Устранённые риски гонок.** Каждый индекс `i` внутри лямбды отображается в непересекающийся диапазон
`[borders[i], borders[i+1]−1]`, что исключает конкурентную запись в `data`. Массив `borders` только читается. `data`
захвачен по ссылке, но TBB не требует атомарности при отсутствии совместных записей.

**Балансировка нагрузки:** Поскольку исходный массив разделяется на $p$ равных чанков до начала вычислений
(на основе индексов в векторе границ `borders`), алгоритм демонстрирует уязвимость к дисбалансу данных.
Конструкция `tbb::parallel_for` со стандартным разделителем (`auto_partitioner`) пытается сгладить этот дисбаланс за
счет автоматического дробления пространства итераций. Однако, поскольку количество задач верхнего уровня жестко
ограничено числом $p$ (количеством чанков), у планировщика TBB отсутствует глубинное пространство для маневра и
применения алгоритма кражи задач (*work-stealing*). Потоки, завершившие сортировку своих "легких" чанков, вынуждены
простаивать на барьерной синхронизации в ожидании потоков, обрабатывающих «тяжелые» участки данных.

**Накладные расходы (Overhead):** При росте числа рабочих до $p=8$ эффективность падает до 40.3%.
На этом этапе накладные расходы начинают доминировать. Они складываются из:

- *Кэш-инвалидации и false sharing:* При параллельном обращении нескольких потоков к соседним областям памяти единого
  вектора `data` подсистемы памяти процессора тратят такты на синхронизацию строк кэша.
- *Затраты на слияние:* Финальная фаза объединения отсортированных чанков выполняется последовательно. Чем больше
  потоков, тем больше итераций слияния требуется выполнить (количество шагов слияния линейно растет как $p-1$).
  При $p=16$ время последовательного слияния начинает поглощать выигрыш от параллельной сортировки.

## 9. Выводы

TBB при $p=8$ показывает ускорение в 3.22–3.27 раза против 3.09–3.14 у OMP на тех же
данных благодаря работе work stealing при неравномерной нагрузке. При
$p=16$ TBB (2.74–2.79) лучше OMP (2.27–2.28), что говорит
о меньших накладных расходах на барьер. При этом оба ограничены $O(N \cdot p)$
слиянием.
