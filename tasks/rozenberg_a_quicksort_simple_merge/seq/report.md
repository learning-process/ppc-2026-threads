## 1. Контекст

Сортировка Хоара с простым слиянием является одним из самых эффективных и известных методов сортировки массивов.

Последовательная версия алгоритма сортировки Хоара разработана в качестве эталонного решения для последующего сравнения
с параллельными реализациями. Она обеспечивает гарантированно корректный результат и используется для измерения
базового времени выполнения, необходимого для расчета ускорения и эффективности, с которыми сопоставляются результаты
всех параллельных реализаций в рамках функционального и производительного тестирования.

## 2. Постановка задачи

### Входные данные

- Неотсортированный массив $\text{data}$ целых чисел произвольного размера $N$.

### Выходные данные

- Массив $\text{data}_{\text{sorted}}$, содержащий все $N$ элементов массива $data$, упорядоченный по неубыванию.

### Ограничения и крайние случаи

- Массив $\text{data}$ не должен быть пустым ($N>0$)
  - Алгоритм должен прерываться на этапе валидации, если на вход подается пустой массив ($N=0$).
- Элементы массива $data$ $-$ целые числа без дополнительных ограничений на диапозон значений

### Крайние случаи

- Массив из одного элемента $-$ корректно возвращается без каких-либо перестановок.
- Массив из двух элементов $-$ одно разбиение, максимум одна перестановка.
- Уже отсортированный массив $-$ алгоритм отрабатывает за $O(N^2)$ в худшем случае.
- Массив с повторяющимися элементами $-$ алгоритм корректно обрабатывает благодаря схеме Хоара.

## 3. Базовый алгоритм

В основе алгоритма лежит классический принцип "разделяй и властвуй", реализованный итеративно для исключения рисков
переполнения стека вызовов в случае рекурсивной реализации.

Алгоритм состоит из трех шагов:

### 1. Выбор опорного элемента (Pivot Selection)

Выбирается элемент из массива, называемый **опорным** ($\text{pivot}$).

Для предотвращения деградации производительности на уже отсортированных массивах (худший случай $O(N^2)$ при выборе
крайних элементов) в данной реализации в качестве опорного элемента выбирается **центральный элемент**:
$$\text{pivot} = \text{data}\left[\text{left} + \frac{\text{right} - \text{left}}{2}\right],$$
где $\text{left}$ $-$ нижняя граница рассматриваемого массива, $\text{right}$ $-$ верхняя граница.

### 2. Разбиение (Partition). Схема разделения Хоара

Элементы в массиве перераспределяются таким образом, что элементы, меньшие опорного, помещаются перед ним, а большие
или равные $-$ после.

В данной реализации в качестве алгоритма разбиения используется схема Хоара, которая более эффективна на практике, чем
традиционная схема Ломуто, так как выполняет в среднем в три раза меньше обменов (swaps) элементов.

1. Инициализируются два индекса: $i$ (начало диапазона) и $j$ (конец диапозона).
2. Индекс $i$ увеличивается, пока $\text{data}[i] < \text{pivot}$.
3. Индекс $j$ уменьшается, пока $\text{data}[j] > \text{pivot}$.
4. Если $i \le j$, элементы меняются местами, индексы сдвигаются.
5. Процесс повторяется, пока $i$ не станет больше $j$. Функция возвращает пару новых границ.

### 3. Итеративный подход

Первые два шага применяются рекурсивно к двум подмассивам слева и справа от опорного элемента. Шаги не применяются для
массивов, в которых только один элемент.

В данной реализации рекурсия заменена на явное использование стека, где хранятся границы еще не обработанных подмассивов.

- В самом начале стек инициализируется парой $(0, N - 1)$.
- Пока стек не пуст, извлекается пара $(\text{left}, \text{right})$. Если $l \lt r$, то для этой пары выполняются первые
два шага алгоритма.
- Результатом разбиения Хоара является новая пара индексов, которая помещается в стек.

К моменту, когда стек становится пустым, массив уже отсортирован.

**Асимптотика по времени:** $O(N\log N)$ в среднем, $O(N^2)$ в худшем случае

**Асимптотика по памяти:** $O(\log N)$ дополнительной памяти (стек), $O(N)$ для хранения копии входного массива.

**Инварианты:**

- В каждый момент стек содержит только непересекающиеся подотрезки исходного массива.
- После разбиения верно следующее: $\forall k \in [l, j]: \text{data}[k] \le \text{pivot}$ и
$\forall k \in [i, r]: \text{data}[k] \ge \text{pivot}$.
- При завершении алгоритма все элементы массива перераспределены в порядке неубывания.

## 4. Детали реализации

**Файлы:** [`seq/include/ops_seq.hpp`](include/ops_seq.hpp), [`seq/src/ops_seq.cpp`](src/ops_seq.cpp)

Класс [`RozenbergAQuicksortSimpleMergeSEQ`](include/ops_seq.hpp#L11) наследует `BaseTask` и переопределяет четыре
метода жизненного цикла:

**[`ValidationImpl()`](src/ops_seq.cpp#L24)** $-$ проверяет, что входной вектор не пуст и выходной вектор пуст.
Отсечение пустого ввода принципиально, так как алгоритм предполагает обращение к `data[0]` и `data[N-1]`.

**[`PreProcessingImpl()`](src/ops_seq.cpp#L28)** $-$ резервирует память под выходной вектор того же размера, что и
входной (`resize`). Возвращает `true`, только если размеры совпали $-$ неявная проверка успешности аллокации.

**[`RunImpl()`](src/ops_seq.cpp#L94)** $-$ делает локальную копию входного вектора, запускает
[`Quicksort`](src/ops_seq.cpp#L74) над этой копией и присваивает результат выходному вектору. Копирование гарантирует,
что входные данные не модифицируются.

**[`PostProcessingImpl()`](src/ops_seq.cpp#L101)** $-$ не выполняет дополнительных действий, возвращает `true`.

**Ключевой фрагмент $-$ функция [`Partition`](src/ops_seq.cpp#L33):**

```cpp
// File: seq/src/ops_seq.cpp
std::pair<int, int> RozenbergAQuicksortSimpleMergeSEQ::Partition(
    InType &data, int left, int right) {
  const int pivot = data[left + ((right - left) / 2)];
  int i = left;
  int j = right;
  while (i <= j) {
    while (data[i] < pivot) i++;
    while (data[j] > pivot) j--;
    if (i <= j) {
      std::swap(data[i], data[j]);
      i++;
      j--;
    }
  }
  return {i, j};
}
```

> Примечание: опорный элемент берется из середины `[left..right]`, а не из начала $-$ это предотвращает деградацию на
> уже отсортированных данных. Два указателя `i` и `j` движутся навстречу: `i` ищет элемент, который не меньше `pivot`,
> `j` ищет элемент, который не больше. При `i <= j` происходит обмен, после которого `i` и `j` расходятся на единицу.
> Функция возвращает `{i, j}` $-$ граничные позиции двух подмассивов, которые предстоит отсортировать рекурсивно (или
> итерационно через стек).

**Фрагмент управления стеком ([`PushSubarrays`](src/ops_seq.cpp#L55)):**

```cpp
// File: seq/src/ops_seq.cpp
void RozenbergAQuicksortSimpleMergeSEQ::PushSubarrays(
    std::stack<std::pair<int, int>> &stack,
    int left, int right, int i, int j) {
  if (j - left > right - i) {
    if (left < j) stack.emplace(left, j);
    if (i < right) stack.emplace(i, right);
  } else {
    if (i < right) stack.emplace(i, right);
    if (left < j) stack.emplace(left, j);
  }
}
```

> Примечание: больший из двух подмассивов кладется в стек первым (то есть обрабатывается позже), а меньший $-$ вторым
> (обрабатывается следующим). Это классическая оптимизация, ограничивающая глубину стека $O(\log N)$ в среднем даже для
> итеративной реализации.

**Обработка граничных и исключительных случаев:**

- Массив длиной $N=0$ $-$ заблокирован `ValidationImpl`.
- Массив длиной $N=1$ $-$ ранний возврат в начале `Quicksort` (`data.size() < 2`).
- Все элементы одинаковы $-$ оба указателя движутся к центру, лишних обменов нет.

## 5. Проверка корректности

Для проверки корректности работы алгоритма используется фреймворк функциональных тестов
[`tests/functional/main.cpp`](../tests/functional/main.cpp). Входные данные и эталон для проверки результата содержатся
в `.txt` в файлах в [`data`](../data/), эти данные считываются в `tests/functional/main.cpp`. Каждый отдельный файл $-$
отдельный тест.

**Структура тестового файла:**

- 1-я строка: число $N$ $-$ число элементов массива
- 2-я строка: неотсортированный массив `data` $-$ входной массив алгоритма
- 3-я строка: отсортированный в порядке неубывания массив из элементов `data` $-$ эталон для сравнения с результатом алгоритма

Набор функциональных тестов охватывает следующие сценарии:

1. [`basic_test.txt`](../data/basic_test.txt) $-$ стандартный тест для проверки общего принципа работы алгоритма,
содержит массив из 10 элементов без особых условий на их значения
2. [`duplicate_test.txt`](../data/duplicate_test.txt) $-$ проверяет случай, когда входной массив содержит исключительно
дублирующиеся элементы
3. [`inorder_test.txt`](../data/inorder_test.txt) $-$ проверяет случай, когда на вход принимается массив,
отсортированный по неубыванию
4. [`mixed_sign_test.txt`](../data/mixed_sign_test.txt) $-$ принимает на вход массив, содержащий 3 элемента: два числа,
равных по модулю, но разных по знаку, и 0
5. [`negative_values_test.txt`](../data/negative_values_test.txt) $-$ проверяет случай, когда все элементы входного
массива отрицательные
6. [`random_data_test.txt`](../data/random_data_test.txt) $-$ тест, принимающий на вход массив из 50 чисел из отрезка
$(-100, 100)$. Проверяет корректность алгоритма на массивах среднего размера.
7. [`random_data_test_2.txt`](../data/random_data_test_2.txt) $-$ тест, принимающий на вход массив из 500 чисел из
отрезка $(-1000, 1000)$. Проверяет корректность алгоритма на массивах большого размера.
8. [`reverse_order_test.txt`](../data/reverse_order_test.txt) $-$ проверяет случай, когда на вход принимается массив,
отсортированный по невозрастанию
9. [`single_element_test.txt`](../data/single_element_test.txt) $-$ проверяет случай, когда массив состоит из одного элемента

## 8. Выводы

Последовательная версия реализует итеративный алгоритм сортировки Хоара с опорным из середины и стеком подмассивов.

Реализация:

- корректно обрабатывает все предусмотренные крайние случаи;
- не использует рекурсию, что устраняет риск переполнения стека при больших $N$;
- оптимизирует порядок обработки подмассивов (`PushSubarrays`) для контроля глубины стека;
- воспроизводимо проверяется функциональными тестами через сравнение с эталоном.

Именно эта реализация используется как **эталонный baseline**: ее время $T_{\text{seq}}$ при $p = 1$ служит знаменателем
для вычисления
ускорения (speedup) и эффективности (efficiency) во всех последующих параллельных версиях. Любые изменения алгоритма в
параллельных backend-ах обязаны давать результат, тождественный результату данной последовательной версии на тех же
входных данных.
