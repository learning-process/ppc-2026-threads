## 1. Введение

Быстрая сортировка Хоара — один из наиболее широко применяемых алгоритмов
сортировки с асимптотикой $O(N \log N)$ в среднем. Задача хорошо подходит для
сравнения моделей параллелизма по нескольким причинам: вычислительное ядро
(сортировка независимых подмассивов) допускает декомпозицию "разделяй и
властвуй" с минимальными зависимостями между частями, а финальное слияние
образует последовательную фазу с явно измеримой стоимостью. Это позволяет
наблюдать как параллельный выигрыш, так и его деградацию при увеличении числа
работников.

## 2. Единая постановка задачи

### Входные данные

- Неотсортированный массив $\text{data}$ целых чисел произвольного размера $N$.

### Выходные данные

- Массив $\text{data}_{\text{sorted}}$, содержащий все $N$ элементов массива $data$, упорядоченный по неубыванию.

### Ограничения и крайние случаи

- Массив $\text{data}$ не должен быть пустым ($N>0$)
  - Алгоритм должен прерываться на этапе валидации, если на вход подается пустой массив ($N=0$).
- Элементы массива $\text{data}$ $-$ целые числа без дополнительных ограничений на диапозон значений

### Критерий корректности

Выходной вектор должен поэлементно совпадать с результатом,
указанным в тестовом файле (для функциональных тестов)
или результатом `std::ranges::sort`, применённого к той же копии входного вектора
(для тестов производительности).

### Функциональные тесты. Чтение данных

Для проверки корректности работы алгоритма используется фреймворк функциональных тестов
[`tests/functional/main.cpp`](tests/functional/main.cpp). Входные данные и эталон для проверки результата содержатся
в `.txt` в файлах в [`data`](data/), эти данные считываются в `tests/functional/main.cpp`. Каждый отдельный файл $-$
отдельный тест.

**Структура тестового файла:**

- 1-я строка: число $N$ $-$ число элементов массива
- 2-я строка: неотсортированный массив `data` $-$ входной массив алгоритма
- 3-я строка: отсортированный в порядке неубывания массив из элементов `data` $-$ эталон для сравнения с результатом алгоритма

Набор функциональных тестов охватывает следующие сценарии:

1. [`basic_test.txt`](data/basic_test.txt) $-$ стандартный тест для проверки общего принципа работы алгоритма,
содержит массив из 10 элементов без особых условий на их значения
2. [`duplicate_test.txt`](data/duplicate_test.txt) $-$ проверяет случай, когда входной массив содержит исключительно
дублирующиеся элементы
3. [`inorder_test.txt`](data/inorder_test.txt) $-$ проверяет случай, когда на вход принимается массив,
отсортированный по неубыванию
4. [`mixed_sign_test.txt`](data/mixed_sign_test.txt) $-$ принимает на вход массив, содержащий 3 элемента: два числа,
равных по модулю, но разных по знаку, и 0
5. [`negative_values_test.txt`](data/negative_values_test.txt) $-$ проверяет случай, когда все элементы входного
массива отрицательные
6. [`random_data_test.txt`](data/random_data_test.txt) $-$ тест, принимающий на вход массив из 50 чисел из отрезка
$(-100, 100)$. Проверяет корректность алгоритма на массивах среднего размера.
7. [`random_data_test_2.txt`](data/random_data_test_2.txt) $-$ тест, принимающий на вход массив из 500 чисел из
отрезка $(-1000, 1000)$. Проверяет корректность алгоритма на массивах большого размера.
8. [`reverse_order_test.txt`](data/reverse_order_test.txt) $-$ проверяет случай, когда на вход принимается массив,
отсортированный по невозрастанию
9. [`single_element_test.txt`](data/single_element_test.txt) $-$ проверяет случай, когда массив состоит из одного элемента

### Тесты производительности. Генерация данных

Данные для тестов производительности генерируются внутри [`tests/performance/main.cpp`](../tests/performance/main.cpp)
с помощью генератора псевдослучайных чисел с фиксированным сидом. Полученный в результате алгоритма массив поэлементно
сравнивается с результатом `std::ranges::sort` над входным массивом.

### Нормировка результатов

$$\text{speedup}(p) = \frac{T_{\text{seq}}}{T(p)}, \quad \text{efficiency}(p) =
\frac{\text{speedup}(p)}{p}$$

Для ALL: $p = \text{ranks} \times \text{threads\_per\_rank}$.

Значение $T_{\text{seq}}$ фиксируется однократно на том же окружении и
используется как знаменатель для всех backend-ов.

### Агрегирование

Функциональные тесты и тесты производительности запускаются раздельно,
результаты не смешиваются.

## 4. Сводка корректности

Все пять backend-ов проверяются поэлементным сравнением выходного вектора с эталоном из файла или результатом `std::ranges::sort`.
Дополнительно результаты при $p \gt 1$ сравниваются с результатом SEQ на тех же входных данных.

**Ограничения применимости:** В гибридной реализации результат присутствует только в памяти ранга 0,
корректность рабочих рангов проверяется транзитивно через результат слияния.

## 6. Интерпретация различий

**SEQ.** Итеративная сортировка Хоара без рекурсии, с опорным из середины. Служит
единственным знаменателем для всех ускорений. Время при $N = 100000000$ отражает
стоимость чистой сортировки без каких-либо преимуществ и накладных расходов параллелизма.

**OMP.** Статическое разбиение на чанки через ручной массив `borders`, где каждый
поток сортирует один чанк. Слияние последовательное, выполняется с асимптотикой $O(N·p)$.
Показывает хорошее ускорение до $p = 8$, после чего стоимость слияния начинает доминировать.
Нет балансировки нагрузки: при неравномерной стоимости чанков быстрые потоки простаивают у
неявного барьера. Это главное структурное ограничение OMP-версии на данной
задаче.

**TBB.** Декомпозиция та же, что у OMP, но параллельная фаза выражена через
`parallel_for` с `auto_partitioner`. Механизм work stealing автоматически
перераспределяет подзадачи между потоками при неравномерной нагрузке, устраняя
простой у барьера. При равномерных входных данных разница с OMP минимальна,
однако на тестах производительности, где данные неравномерны, преимущество TBB более выражено,
особенно ярко эта разница выражена при $p=16$.
Слияние остаётся последовательным — то же узкое место, что у OMP.

**STL.** Явное создание `std::thread` для каждого чанка и явный `join`.
Декомпозиция и слияние идентичны OMP. Балансировки нет — аналогично OMP.
Накладные расходы на создание потоков при каждом вызове `RunImpl` выше, чем у
OMP (пул потоков) и TBB (пул задач), что может проявиться при многократных
вызовах или малых $N$. При однократном запуске на большом $N$ разница с остальными
поточными backend-ами мала: слегка превосходит OMP при $p=8$ и значительно превосходит при $p=16$,
однако разница с TBB менее выраженная.

**ALL.** Два уровня параллелизма: MPI делит массив между рангами, OpenMP
сортирует локальный чанк каждого ранга многопоточно. Добавляет коммуникационные
издержки (`MPI_Scatterv`, `MPI_Send` / `MPI_Recv`) поверх уже существующего
последовательного слияния, однако они не проявляются при $\text{ranks} < 8$.
Максимальное ускорение проявляется при $\text{ranks} = 4$, достигая пикового значения среди всех реализаций
$\text{efficiency}_{\text{max}}=4.13$.

## 8. Заключение

На данной задаче все поточные backend-ы (OMP, TBB, STL) показывают схожее
поведение: хорошее ускорение до $p = 8$, деградация при $p = 16$ из-за
последовательного слияния $O(N \cdot p)$. TBB выигрывает у OMP и незначительно у STL благодаря work stealing.

Среди всех реализаций гибридная показывает наилучшие результаты ускорения, достигая пикового значения 4.13
при конфигурации `4 × 4`. Однако с ростом числа рангов прирост значения ускорения замедляется, что свидетельствует
о росте коммуникационных издержек.

**Что можно улучшить.** Основное узкое место всех версий — последовательное
слияние. Более оптимизированный алгоритм слияния позволил бы приблизиться к линейному ускорению и
устранить доминирующую последовательную фазу.

### Формула эффективности для ALL

При сравнении конфигураций ALL с поточными backend-ами эффективность нормируется
по суммарному числу работников:

$$\text{efficiency}_{\text{all}}(r, t) = \frac{T_{\text{seq}}}{T_{\text{all}}(r,
t) \cdot r \cdot t}$$

где $r$ — число MPI-рангов, $t$ — число OpenMP-потоков на ранг.
