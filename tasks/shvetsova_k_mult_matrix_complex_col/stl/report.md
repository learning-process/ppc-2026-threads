## 1. Контекст

Данная версия реализует параллельное умножение разреженных комплексных матриц в
формате **CCS (Compressed Column Storage)** с использованием стандартной
библиотеки C++ и механизма **`std::thread`**.

Цель реализации — ускорить вычисление произведения матриц путём ручного
распределения независимых вычислений между потоками.

В отличие от OpenMP и TBB, где распределением задач управляет runtime, в
реализации на `std::thread` управление потоками выполняется вручную:

- определяется число потоков;
- вычисляется диапазон работы каждого потока;
- запускаются worker-потоки;
- выполняется синхронизация через `join()`.

Последовательная версия (`SEQ`) используется как baseline для проверки
корректности и оценки ускорения.

---

## 2. Постановка задачи

На вход подаются две разреженные комплексные матрицы:

\[ A \in \mathbb{C}^{m \times n} \]

\[ B \in \mathbb{C}^{n \times k} \]

в формате **CCS**.

Требуется вычислить:

\[ C = A \times B \]

где результирующая матрица также представляется в формате CCS.

Условие корректности умножения:

\[ A.cols = B.rows \]

Как и в других версиях, метод `ValidationImpl()` всегда возвращает `true`,
поэтому входная валидация отсутствует.

Подробное описание математической постановки и структуры хранения приведено в
`seq/report.md`.

---

## 3. Базовый алгоритм

Вычислительная логика полностью наследует последовательную реализацию.

Алгоритм:

1. выполняет обход столбцов матрицы `B`;
2. для каждого ненулевого элемента выбирает соответствующий столбец `A`;
3. выполняет накопление результата;
4. преобразует dense-представление обратно в CCS.

Для каждого ненулевого элемента:

\[ B(k, i) \]

выполняется накопление:

\[ C(:, i) += A(:, k)\cdot B(k, i) \]

Ключевое отличие STL-версии состоит в том, что вычисление столбцов
распределяется вручную между потоками.

---

## 4. Схема распараллеливания

### Стратегия распараллеливания

Параллелизация выполнена **по диапазонам столбцов матрицы `B`**.

Каждый поток получает непрерывный диапазон:

```text
[start_col, end_col)
```

и вычисляет принадлежащие ему столбцы результирующей матрицы.

Схема работы:

```text
thread 0 → columns [0 ... k1)
thread 1 → columns [k1 ... k2)
thread 2 → columns [k2 ... k3)
...
```

Такое разбиение выбрано потому, что столбцы результата независимы и не требуют
обмена данными.

---

### Определение числа потоков

Число потоков определяется автоматически:

```cpp
std::thread::hardware_concurrency()
```

```cpp
int num_threads =
    std::thread::hardware_concurrency();
```

Если определить число потоков не удалось, используется fallback:

```cpp
num_threads = 4;
```

Это обеспечивает переносимость реализации.

---

### Разделение диапазонов

Диапазоны распределяются равномерно.

Для каждого потока вычисляется:

```cpp
cols_per_thread =
    matrix_b.cols / num_threads;
```

Остаток:

```cpp
remainder =
    matrix_b.cols % num_threads;
```

распределяется между первыми потоками:

```cpp
(thread_idx < remainder ? 1 : 0)
```

Такой подход позволяет:

- минимизировать дисбаланс нагрузки;
- избежать простаивания потоков;
- максимально равномерно распределить столбцы.

---

### Локальные данные потока

Каждый поток использует собственный dense-буфер:

```cpp
std::vector<std::complex<double>>
column_c_local(matrix_a.rows,
               {0.0, 0.0});
```

Буфер:

- полностью локален;
- переиспользуется внутри потока;
- исключает совместную запись.

Для вычисления столбца используется отдельная функция:

```cpp
ComputeColumnTask(...)
```

которая выполняет:

- накопление произведения;
- фильтрацию нулевых элементов;
- запись sparse-результата.

---

### Синхронизация

Дополнительная синхронизация отсутствует.

Не используются:

- `mutex`;
- `atomic`;
- `condition_variable`.

Причина — каждый поток записывает только в:

```cpp
columns_c[i]
```

где `i` соответствует уникальному столбцу.

После запуска всех worker-потоков выполняется:

```cpp
join()
```

```cpp
for (auto &th : threads) {
    if (th.joinable()) {
        th.join();
    }
}
```

Синхронизация через `join()` гарантирует завершение всех вычислений перед
сборкой итоговой матрицы.

## 5. Детали реализации

### Основные файлы

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

---

### PreProcessingImpl

На этапе подготовки:

- извлекаются матрицы `A` и `B`;
- задаются размеры результата;
- очищаются структуры хранения:
  - `row_ind`;
  - `values`;
  - `col_ptr`.

Размер результирующей матрицы:

\[ rows(C)=rows(A) \]

\[ cols(C)=cols(B) \]

---

### RunImpl

Основная вычислительная логика вынесена в функцию:

```cpp
ComputeColumnTask(...)
```

Она выполняет вычисление одного столбца результата.

Внутри потока:

1. dense-буфер очищается:

```cpp
std::ranges::fill(...)
```

1. выполняется накопление произведения;

2. производится компрессия результата в sparse-формат.

После завершения вычислений всех потоков выполняется последовательная сборка
результата:

```cpp
for (int i = 0; i < matrix_b.cols; i++) {
    matrix_c.row_ind.insert(...);
    matrix_c.values.insert(...);
    matrix_c.col_ptr.push_back(...);
}
```

Это позволяет избежать конкурентной записи в итоговую структуру `matrix_c`.

## 6. Проверка корректности

Корректность STL-версии проверялась сравнением с последовательной реализацией
(`SEQ`).

Проверялись:

- малые размеры матриц;
- случайные sparse-матрицы;
- комплексные значения;
- корректность размеров результата;
- совпадение содержимого матриц;
- устойчивость результата между повторными запусками.

### Особенности performance-тестов

Во встроенных performance-тестах используется:

```cpp
const int size = 1000;
const int nnz_per_col = 20;
```

Для размера:

\[ 1000 \times 1000 \]

ускорение выражено слабо, поскольку накладные расходы на:

- создание потоков;
- запуск worker-задач;
- синхронизацию через `join()`

частично компенсируют выигрыш от распараллеливания.

Для более показательных измерений дополнительно использовалась увеличенная
задача:

\[ size = 4000 \]

где объём вычислений становится существенно больше.

### Сравнение с baseline (SEQ)

Результаты для размера:

\[ size = 4000 \]

| Mode     | SEQ (s) | STL (s) | Speedup |
| -------- | ------- | ------- | ------- |
| task     | 0.753   | 0.538   | 1.40×   |
| pipeline | 0.736   | 0.520   | 1.42×   |

### Интерпретация результатов

На размере:

\[ size = 4000 \]

реализация на `std::thread` показывает устойчивое ускорение относительно
последовательной версии.

Недостатки:

- ручное управление потоками создаёт дополнительный overhead;
- отсутствует scheduler runtime;
- стоимость `create + join` выше;
- балансировка нагрузки менее гибкая.

При этом выбранная схема разбиения диапазонов позволила получить устойчивый
выигрыш благодаря:

- независимости вычислений;
- отсутствию блокировок;
- локальным буферам потоков;
- равномерному распределению столбцов.

## 9. Выводы

Реализована параллельная версия умножения комплексных разреженных матриц в
формате CCS с использованием `std::thread`.

Распараллеливание выполнено путём ручного разделения диапазонов столбцов между
потоками.

Преимущества реализации:

- полный контроль над потоками;
- отсутствие гонок данных;
- отсутствие `mutex` и `atomic`;
- предсказуемая схема распределения работы.

Ограничения:

- overhead создания потоков;
- менее гибкое распределение нагрузки по сравнению с runtime TBB;
- слабый выигрыш на малых размерах (`size = 1000`);
- последовательный этап сборки результата;
- отсутствие входной валидации.

Эксперименты показали, что использование `std::thread` оправдано при увеличении
размера задачи (`size = 4000`), когда вычислительная нагрузка становится
достаточно большой для компенсации затрат на управление потоками.
