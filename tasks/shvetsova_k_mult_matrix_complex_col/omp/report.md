## 1. Контекст

Данная версия реализует параллельное умножение разреженных комплексных матриц в
формате **CCS (Compressed Column Storage)** с использованием технологии
**OpenMP**.

Цель реализации — ускорить вычисление произведения матриц за счёт
распараллеливания независимых вычислений по столбцам результирующей матрицы. В
отличие от последовательной версии (`SEQ`), где столбцы вычисляются
последовательно, в OMP-версии разные столбцы распределяются между потоками.

Последовательная версия (`seq/report.md`) рассматривается как baseline для
проверки корректности и оценки ускорения.

---

## 2. Постановка задачи

На вход подаются две разреженные комплексные матрицы:

\[ A \in \mathbb{C}^{m \times n} \]

\[ B \in \mathbb{C}^{n \times k} \]

в формате **CCS**.

Требуется вычислить:

\[ C = A \times B \]

и вернуть результат также в формате CCS.

Условие корректности умножения:

\[ A.cols = B.rows \]

---

## 3. Базовый алгоритм

Алгоритм наследует вычислительную схему последовательной версии.

Основная идея:

- обход столбцов матрицы `B`;
- обработка ненулевых элементов каждого столбца;
- накопление результата во временном dense-векторе;
- преобразование результата обратно в CCS.

Для каждого ненулевого элемента:

\[ B(k, i) \]

выбирается соответствующий столбец `k` матрицы `A`, после чего выполняется
накопление:

\[ C(:, i) += A(:, k)\cdot B(k, i) \]

Ключевое отличие OMP-версии состоит в том, что вычисления разных столбцов
результирующей матрицы выполняются параллельно.

---

## 4. Схема распараллеливания

### Граница распараллеливания

Параллелизация выполнена **по столбцам матрицы `B`**.

Каждый поток вычисляет независимый набор столбцов результирующей матрицы `C`.

Ключевая параллельная область:

```cpp
#pragma omp parallel default(none) shared(matrix_a, matrix_b, columns_c)
{
    std::vector<std::complex<double>> column_c(matrix_a.rows, {0.0, 0.0});

#pragma omp for
    for (int i = 0; i < matrix_b.cols; i++) {
        ...
    }
}
```

Такой подход выбран потому, что столбцы результата вычисляются независимо и не
требуют синхронизации между потоками.

### Shared / Private данные

#### Shared

Общие данные (`shared`):

- `matrix_a`
- `matrix_b`
- `columns_c`

Матрицы только читаются, поэтому гонки отсутствуют.

`columns_c` является общим контейнером, однако каждый поток записывает **только
в свой индекс `columns_c[i]`**, что устраняет конфликт записи.

#### Private

Локальные данные потока:

```cpp
std::vector<std::complex<double>> column_c(matrix_a.rows);
```

Каждый поток имеет собственный dense-буфер для накопления столбца результата.

Это позволяет избежать:

- гонок записи;
- необходимости использования `critical`;
- применения `mutex` или `atomic`.

### Синхронизация

Дополнительная синхронизация не используется.

Не применяются:

- `critical`;
- `atomic`;
- `reduction`.

Причина — независимость вычислений столбцов.

После завершения `omp for` действует **неявный барьер OpenMP**, гарантирующий
завершение вычислений всеми потоками перед сборкой результирующей матрицы.

### Schedule

В реализации используется стандартное распределение OpenMP:

```cpp
#pragma omp for
```

Явный `schedule(...)` не задан, поэтому применяется политика по умолчанию
(`static`).

Такой выбор подходит для задачи, поскольку:

- столбцы имеют близкое число ненулевых элементов;
- вычислительная нагрузка распределяется относительно равномерно;
- overhead динамического планирования не требуется.

## 5. Детали реализации (OMP)

Реализация OpenMP-версии построена как расширение последовательного алгоритма,
где ключевое отличие заключается в распараллеливании обработки столбцов матрицы
**B**. Архитектура кода сохраняет последовательную структуру pipeline, но
вычислительная часть выполняется параллельно внутри одной директивы
`omp parallel for`.

### 5.1 Общая структура решения

В основе реализации лежит та же логика, что и в SEQ-версии:

- обход столбцов матрицы **B**;
- обработка ненулевых элементов каждого столбца;
- накопление результата в dense-векторе;
- преобразование результата в CCS-формат.

Отличие состоит в том, что каждый поток обрабатывает свою часть столбцов
независимо от других.

### 5.2 Ключевая вычислительная функция

Основная вычислительная логика не изменилась и вынесена в общую функцию
обработки столбца:

- выполняется обход ненулевых элементов столбца `B`;
- для каждого элемента выбирается соответствующий столбец `A`;
- происходит накопление значений в временный dense-вектор;
- затем выполняется фильтрация ненулевых элементов.

Таким образом, OpenMP-версия переиспользует вычислительное ядро SEQ, изменяя
только способ распределения работы.

### 5.3 Модель данных и память

Для каждого потока используется локальный dense-вектор:

- размер: `rows(A)`;
- инициализация: нулевые комплексные значения;
- назначение: накопление значений одного или нескольких столбцов.

Каждый поток также формирует собственную структуру:

- `SparseColumn` — хранит результат одного столбца;
  - `rows` — индексы ненулевых элементов;
  - `vals` — соответствующие значения.

Важно, что запись в итоговую матрицу `C` происходит **после завершения
параллельной секции**, что исключает конкуренцию потоков.

### 5.4 Особенности реализации

Ключевые свойства реализации:

- отсутствие общей записи во время вычислений;
- переиспользование dense-вектора внутри потока;
- отсутствие динамического аллокационного давления внутри цикла;
- детерминированность результата при фиксированном числе потоков.

## 6. Сложность реализации

### 6.1 Временная сложность

Пусть:

- `cols(B)` — число столбцов матрицы B;
- `nnz(B)` — число ненулевых элементов B;
- `nnz(A)` — число ненулевых элементов A.

Тогда вычислительная сложность аналогична SEQ-версии:

\[ O\left(\sum*{b \in nnz(B)} nnz(A*{column(b)})\right) \]

Однако в параллельной версии достигается уменьшение реального времени выполнения
за счёт распределения столбцов между потоками:

\[ T*p \approx \frac{T_1}{p} + T*{overhead} \]

где:

- `p` — число потоков;
- `T_overhead` — накладные расходы OpenMP.

В худшем случае:

\[ O(n^3) \]

### 6.2 Пространственная сложность

Дополнительная память:

\[ O(rows(A)) \]

для каждого потока используется независимый dense-вектор.

Дополнительно:

- временные структуры `SparseColumn`;
- итоговые массивы CCS результата.

Общая память линейно зависит от числа ненулевых элементов результата:

\[ O(nnz(C)) \]

## 8. Вывод

OpenMP-реализация умножения разреженных комплексных матриц в формате CCS
демонстрирует корректное и предсказуемое масштабирование при увеличении числа
потоков.

Основные свойства реализации:

- распараллеливание по независимым столбцам;
- отсутствие синхронизационных примитивов;
- сохранение структуры SEQ-алгоритма;
- корректность относительно baseline.

Ключевые ограничения:

- неполная линейная масштабируемость;
- рост overhead при увеличении числа потоков;
- ограничение пропускной способности памяти при работе с dense-буферами.
