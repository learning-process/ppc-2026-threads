## 1. Контекст

Данная версия реализует параллельное умножение разреженных комплексных матриц в формате **CCS (Compressed Column Storage)** с использованием библиотеки **oneTBB (Threading Building Blocks)**.

Основная цель реализации — ускорить вычисление произведения разреженных матриц за счёт автоматического распределения независимых вычислений между worker-потоками runtime TBB.

OneTBB использует **task-based модель**, позволяющую runtime самостоятельно балансировать нагрузку между потоками.

Последовательная версия (`SEQ`) используется как baseline для проверки корректности и оценки ускорения.

---

## 2. Постановка задачи

На вход подаются две разреженные комплексные матрицы:

\[
A \in \mathbb{C}^{m \times n}
\]

\[
B \in \mathbb{C}^{n \times k}
\]

в формате **CCS (Compressed Column Storage)**.

Необходимо вычислить:

\[
C = A \times B
\]

где результирующая матрица также хранится в формате CCS.

Условие корректности умножения:

\[
A.cols = B.rows
\]

---

## 3. Базовый алгоритм

TBB-версия использует тот же вычислительный алгоритм, что и последовательная реализация.

Основная идея:

- обход столбцов матрицы `B`;
- обработка ненулевых элементов каждого столбца;
- поиск соответствующих столбцов матрицы `A`;
- накопление промежуточного результата;
- преобразование dense-представления обратно в CCS.

Для каждого ненулевого элемента:

\[
B(k, i)
\]

выполняется:

\[
C(:, i) += A(:, k)\cdot B(k, i)
\]

Ключевое отличие от `SEQ` состоит в том, что вычисление столбцов результирующей матрицы распределяется между worker-потоками oneTBB.

---

## 4. Схема распараллеливания

### Выбранный примитив

Для распараллеливания использован:

```cpp
tbb::parallel_for
```

Такой выбор обусловлен тем, что вычисление каждого столбца результата является независимой задачей.

Параллельная область:

```cpp
tbb::parallel_for(
    tbb::blocked_range<int>(0, matrix_b.cols, grain_size),
    [&](const tbb::blocked_range<int>& r) {
        ...
    },
    tbb::static_partitioner()
);
```

---

### Декомпозиция задачи

Параллелизация выполнена **по диапазонам столбцов матрицы `B`**.

Каждый worker получает диапазон:

```cpp
blocked_range<int>
```

и вычисляет соответствующие столбцы результата.

Внутри диапазона выполняется:

```cpp
for (int i = r.begin(); i != r.end(); ++i)
```

что позволяет независимо обрабатывать столбцы без необходимости синхронизации.

Такое разбиение естественно соответствует структуре алгоритма, поскольку:

- столбцы результата независимы;
- отсутствуют конфликты записи;
- нет необходимости обмена данными между потоками.

---

### blocked_range и grain size

Используется:

```cpp
size_t grain_size = 100;
```

и диапазон:

```cpp
tbb::blocked_range<int>(
    0,
    matrix_b.cols,
    grain_size
)
```

Параметр `grain_size` определяет минимальный размер блока задач, который runtime может передать worker-потоку.

Выбор значения:

```text
grain_size = 100
```

представляет компромисс между:

- слишком мелкими задачами (высокий scheduling overhead);
- слишком крупными задачами (снижение балансировки нагрузки).

Для данной задачи нагрузка по столбцам распределена относительно равномерно, поэтому фиксированный размер блока оказался достаточным.

---

### Partitioner

Используется:

```cpp
tbb::static_partitioner()
```

Статическое разбиение выбрано потому, что:

- число ненулевых элементов в столбцах близко;
- вычислительная стоимость столбцов примерно одинакова;
- динамический scheduling не даёт заметного выигрыша.

Статическое распределение позволяет уменьшить runtime-overhead.

---

### Локальные данные потока

Для каждого worker создаётся локальный dense-буфер:

```cpp
std::vector<std::complex<double>>
column_c_local(matrix_a.rows, {0.0, 0.0});
```

Буфер:

- не разделяется между потоками;
- переиспользуется внутри диапазона;
- устраняет гонки записи.

Накопление результата выполняется локально:

```cpp
column_c[row] += tmp_val * val_a;
```

## 5. Детали реализации

### Основные файлы

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`



---

### PreProcessingImpl

На этапе подготовки:

- извлекаются матрицы `A` и `B`;
- задаются размеры результата;
- очищаются структуры хранения:
  - `row_ind`;
  - `values`;
  - `col_ptr`.

Результирующая матрица получает размер:

\[
rows(C)=rows(A)
\]

\[
cols(C)=cols(B)
\]

---

### RunImpl

Основная логика вычислений вынесена в lambda-функцию:

```cpp
compute_column
```

которая выполняет вычисление одного столбца результата.

Для каждого столбца:

1. dense-вектор очищается:

```cpp
std::ranges::fill(...)
```

2. выполняется накопление произведения;

3. результат компрессируется обратно в sparse-формат.

Ненулевые элементы сохраняются в:

```cpp
columns_c[i]
```

После завершения `parallel_for` выполняется последовательная сборка итоговой CCS-матрицы:

```cpp
for (int i = 0; i < matrix_b.cols; i++) {
    matrix_c.row_ind.insert(...);
    matrix_c.values.insert(...);
    matrix_c.col_ptr.push_back(...);
}
```

Такой подход позволяет избежать конкурентной записи в общий объект результата.

---

### PostProcessingImpl

Постобработка отсутствует:

```cpp
bool PostProcessingImpl() {
    return true;
}
```

## 8. Результаты

### Особенности performance-тестов

Встроенный performance-тест проекта использует:

```cpp
const int size = 1000;
const int nnz_per_col = 20;
```

Для матриц размера **1000×1000** ускорение выражено слабо, поскольку вычислительная нагрузка недостаточно велика для компенсации накладных расходов runtime TBB.

Для оценки масштабируемости дополнительно были проведены измерения на увеличенном размере:

\[
size = 4000
\]

Именно на таком размере наблюдается заметный выигрыш от параллельного выполнения.

### Сравнение с baseline (SEQ)

Результаты для увеличенного размера матриц (`size = 4000`):

| Технология | Ranks (MPI) | Threads per Rank | Всего ресурсов | Время, сек | Ускорение | Эффективность |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: |
| **SEQ**  | 1 | 1 | 1 | 0.746 | 1.00 | 100.0% |
| **TBB**  | 1 | 2 | 2 | 0.439 | 1.70 | 84.9% |

### Интерпретация результатов

На размере:

\[
size = 4000
\]

TBB показывает устойчивое ускорение относительно последовательной версии.

Причины выигрыша:

- независимость вычислений столбцов;
- эффективное распределение нагрузки runtime TBB;
- повторное использование локального dense-буфера;
- низкие накладные расходы статического partitioning.

При этом ускорение остаётся нелинейным, поскольку часть алгоритма остаётся последовательной:

- финальная сборка `matrix_c`;
- операции вставки в `std::vector`;

## 9. Выводы

Реализована параллельная версия умножения комплексных разреженных матриц в формате CCS с использованием oneTBB.
Для распараллеливания использовался:

```cpp
parallel_for + blocked_range
```

с параметрами:

- `grain_size = 100`;
- `static_partitioner`.

Преимущества реализации:

- отсутствие гонок данных;
- независимая обработка столбцов;
- эффективное распределение задач runtime TBB;
- заметное ускорение на больших объёмах данных.

Ограничения:

- слабый выигрыш на малых размерах (`size = 1000`);
- наличие последовательного этапа сборки результата;

Эксперименты показали, что применение TBB оправдано для больших sparse-матриц (`size = 4000`), где вычислительная нагрузка достаточно велика для компенсации runtime-overhead.
