## 1. Введение

В работе реализована и исследована параллельная версия алгоритма свёртки
изображения с ядром Гаусса 3×3 (размытие). Задача хорошо подходит для сравнения
моделей параллелизма: каждый выходной пиксель вычисляется независимо на основе
небольшой окрестности (3×3), что даёт естественную параллельность.
Однако краевые эффекты и доступ к памяти создают ограничения.
Сравниваются последовательная (SEQ), OpenMP, oneTBB, std::thread
и гибридная MPI+OpenMP (all) реализации.

## 2. Единая постановка задачи

- **Вход**: изображение в оттенках серого, заданное целочисленной матрицей
  `width` × `height` (значения 0–255). Данные хранятся в одномерном векторе по строкам.
- **Выход**: изображение тех же размеров, каждый пиксель — результат свёртки
  с ядром Гаусса (нормированным):

```text
[1 2 1]
[2 4 2] / 16
[1 2 1]
```

- **Краевые условия**: пиксели за границами заменяются ближайшим допустимым (clamp).
- **Критерий корректности**: поэлементное равенство с эталонной
  последовательной реализацией на всех функциональных тестах.

### Функциональные тесты

Используется фреймворк `ppc_func_tests`, генерирующий изображения 3×3, 5×5, 7×7, 10×10, 16×16
с константным, градиентным, случайным и шахматным заполнением.
Результат сравнивается с эталонной функцией `ReferenceFilter`.

### Тесты производительности

Изображение генерируется внутри `tests/performance/main.cpp` с фиксированным размером
512×512, 1024×1024, 2048×2048. Значения пикселей — случайные (равномерное распределение 0–255).
Время замеряется только для `RunImpl` (`task_run`). Каждая конфигурация запускается 5 раз,
приведена медиана.

### Нормировка результатов

$$\text{speedup}(p) = \frac{T_{\text{seq}}}{T(p)}, \quad
\text{efficiency}(p) = \frac{\text{speedup}(p)}{p}$$
Для all: $p = \text{ranks} \times \text{threads\_per\_rank}$.

## 4. Сводка корректности
Все пять backend-ов проходят функциональные тесты.
Результаты при $p>1$ идентичны SEQ. Для all корректность проверяется
на любом процессе (все получают одинаковый результат).

## 6. Интерпретация различий

**SEQ.** Простая последовательная свёртка.
Служит единственным знаменателем для всех ускорений. Время растёт пропорционально площади.

**OMP.** Распараллеливание по строкам с помощью `#pragma omp parallel for schedule(static)`.
Каждый поток обрабатывает непрерывную полосу строк – хорошая локальность кэша.
Показывает ускорение до 2.66× на 4 потоках для 1024×1024. При 8 потоках ускорение падает.

**TBB.** Использует `tbb::parallel_for` с разбиением по столбцам (блоки 64).
При 1 потоке медленнее SEQ из-за overhead планировщика. Масштабирование хуже, чем у OMP,
но при 8 потоках на 2048×2048 показывает ускорение 1.78×.

**STL.** Ручное управление потоками (`std::thread`) с разбиением по строкам.
Показывает наилучшее ускорение (2.59× на 4 потоках для 2048×2048) благодаря минимальным
накладным расходам и оптимальному использованию кэша. Эффективность 65% на 4 потоках –
лучший показатель.

**all (MPI+OpenMP, широковещательная).** Каждый процесс получает полную копию изображения
через `MPI_Bcast` и выполняет фильтрацию с использованием OpenMP.
Наилучшее ускорение (0.83) достигается при 1×2 (один MPI-процесс, 2 потока) –
близко к OpenMP, но с оверхедом на коммуникации. Увеличение числа процессов
(2 и 4) приводит к резкому падению производительности из-за полного дублирования
вычислений. Данная реализация служит учебным примером гибридного программирования,
но не даёт выигрыша в производительности.

## 7. Репродуцируемость

### Сборка
```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --target ppc_perf_tests
```

### Запуск (пример для OMP, 4 потока, размер 1024)
```bash
# Изменить размер в tasks/.../tests/performance/main.cpp (const int size = 1024)
cd build/bin
export PPC_NUM_THREADS=4
./ppc_perf_tests --gtest_filter="*omp*"
```

### Запуск для all
```bash
export PPC_NUM_THREADS=2
mpirun --allow-run-as-root -np 2 ./build/bin/ppc_perf_tests --gtest_filter="*all*"
```

## 8. Заключение
На данной задаче **наиболее эффективной оказалась реализация на std::thread** (STL)
благодаря минимальным накладным расходам и оптимальному разбиению по строкам.
OpenMP также даёт хорошие результаты при 2–4 потоках. TBB уступает в производительности
из-за overhead планировщика и разбиения по столбцам, но удобен для более сложных
нерегулярных задач. Гибридная MPI+OpenMP версия (all) в данной реализации не даёт
выигрыша и предназначена для демонстрации комбинирования технологий.
