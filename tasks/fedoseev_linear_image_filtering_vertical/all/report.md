## 1. Контекст

Гибридная версия `all` сочетает межпроцессный параллелизм (MPI) и внутрипроцессный (OpenMP).
Каждый MPI-процесс получает полную копию изображения через широковещательную рассылку,
после чего независимо выполняет фильтрацию, распараллеливая циклы по строкам с помощью
OpenMP. Такая модель не даёт выигрыша в производительности, но служит демонстрацией
комбинирования двух технологий.

## 2. Постановка задачи

См. `seq/report.md`.

## 3. Базовый алгоритм

Последовательная свёртка (как в SEQ) с добавлением MPI-коммуникаций и OpenMP-директивы.

## 4. Межпроцессная схема

**Роли рангов.** Все ранги равноправны. Ранг 0 изначально владеет изображением,
остальные ранги получают его через широковещание.

**MPI-вызовы:**

- `MPI_Bcast` – рассылка размеров (`w`, `h`) и всех пикселей изображения от ранга 0
  всем остальным.
- Отсутствуют `Scatterv`, `Gatherv`, `Sendrecv` – данные не делятся,
  каждый процесс работает с полным массивом.

**Синхронизация.** `MPI_Bcast` является коллективной операцией
и синхронизирует все процессы.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса используется **OpenMP** для распараллеливания фильтрации
по строкам. Количество потоков задаётся через `PPC_NUM_THREADS`
(одинаково для всех процессов). Директива `#pragma omp parallel for` применяется
к внешнему циклу по строкам.

## 6. Детали реализации

**Файлы:** [`all/include/ops_all.hpp`](include/ops_all.hpp),
[`all/src/ops_all.cpp`](src/ops_all.cpp)

**Алгоритм:**

1. Ранг 0 читает входные данные.
2. `MPI_Bcast` размеров и массива данных всем рангам.
3. Каждый ранг независимо вычисляет свёртку, используя OpenMP
   для распараллеливания по строкам.
4. Каждый ранг сохраняет результат в свой `GetOutput()`.

**Ключевые фрагменты кода:**

```cpp
// Файл: all/src/ops_all.cpp

// Широковещание размеров
MPI_Bcast(&w, 1, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(&h, 1, MPI_INT, 0, MPI_COMM_WORLD);
if (rank != 0) {
    img.resize(w * h);
}
MPI_Bcast(img.data(), img.size(), MPI_INT, 0, MPI_COMM_WORLD);

// Параллельная фильтрация с OpenMP
#pragma omp parallel for default(none) shared(img, result, w, h, kernel, kernel_sum)
for (int row = 0; row < h; ++row) {
    for (int col = 0; col < w; ++col) {
        int sum = 0;
        for (int ky = -1; ky <= 1; ++ky)
            for (int kx = -1; kx <= 1; ++kx) {
                int px = std::clamp(col + kx, 0, w - 1);
                int py = std::clamp(row + ky, 0, h - 1);
                sum += img[py * w + px] * kernel.at(ky+1).at(kx+1);
            }
        result[row * w + col] = sum / 16;
    }
}
```

**Наблюдения:**

- Наилучшее ускорение (0.83) достигается при конфигурации 1×2 (один MPI-процесс, 2 потока) –
  близко к OpenMP-версии, но с дополнительным оверхедом на `MPI_Bcast`.
- Увеличение числа MPI-процессов свыше 1 приводит к резкому падению производительности
  (например, 4×1 в 3–4 раза медленнее SEQ), поскольку каждый процесс выполняет всю работу,
  а коммуникации добавляют накладные расходы.
- Гибридная модель на одном узле неэффективна из-за полного дублирования вычислений.

## 10. Выводы

Данная реализация `all` является демонстрационной – она показывает,
как комбинировать MPI и OpenMP, но не преследует цели ускорения.
Реальное ускорение можно получить только при распределении работы
между процессами (например, деление изображения на блоки).
В рамках данной задачи чисто потоковые версии (OMP, STL) остаются
наиболее эффективными.
