## 1. Контекст
Реализация `all` использует только MPI (без OpenMP). Каждый MPI-процесс получает полную копию изображения через широковещательную рассылку (`MPI_Bcast`), после чего независимо вычисляет фильтр. Результат сохраняется у каждого процесса, но в тестах проверяется только процесс с рангом 0. Такая модель не даёт выигрыша в производительности, но служит демонстрацией базового использования MPI.

## 2. Постановка задачи
См. `seq/report.md`.

## 3. Базовый алгоритм
Последовательная свёртка (как в SEQ). Изменения касаются только организации MPI-коммуникаций.

## 4. Межпроцессная схема

**Роли рангов.** Все ранги равноправны. Ранг 0 изначально владеет изображением, остальные ранги получают его через широковещание.

**MPI-вызовы:**
- `MPI_Bcast` – рассылка размеров (`w`, `h`) и всех пикселей изображения от ранга 0 всем остальным.
- Отсутствуют `Scatterv`, `Gatherv`, `Sendrecv` – данные не делятся, каждый процесс работает с полным массивом.

**Синхронизация.** `MPI_Bcast` является коллективной операцией и синхронизирует все процессы.

## 5. Внутрипроцессная схема
Потоков нет – каждый MPI-процесс выполняет фильтрацию последовательно. Параллелизм достигается только за счёт нескольких процессов, но **каждый процесс делает полную работу**, что приводит к дублированию вычислений.

## 6. Детали реализации

**Файлы:** [`all/include/ops_all.hpp`](include/ops_all.hpp), [`all/src/ops_all.cpp`](src/ops_all.cpp)

**Алгоритм:**
1. Ранг 0 читает входные данные.
2. `MPI_Bcast` размеров и массива данных всем рангам.
3. Каждый ранг независимо вычисляет свёртку (последовательно).
4. Каждый ранг сохраняет результат в свой `GetOutput()`.

**Ключевые фрагменты кода:**

```cpp
// Файл: all/src/ops_all.cpp

// Широковещание размеров
MPI_Bcast(&w, 1, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(&h, 1, MPI_INT, 0, MPI_COMM_WORLD);
if (rank != 0) {
    img.resize(w * h);
}
MPI_Bcast(img.data(), img.size(), MPI_INT, 0, MPI_COMM_WORLD);

// Последовательная фильтрация (каждый процесс делает всю работу)
for (int row = 0; row < h; ++row) {
    for (int col = 0; col < w; ++col) {
        int sum = 0;
        for (int ky = -1; ky <= 1; ++ky)
            for (int kx = -1; kx <= 1; ++kx) {
                int px = std::clamp(col + kx, 0, w - 1);
                int py = std::clamp(row + ky, 0, h - 1);
                sum += img[py * w + px] * kernel.at(ky+1).at(kx+1);
            }
        result[row * w + col] = sum / 16;
    }
}
```

**Время выполнения (мс):**
| Размер | 1 процесс | 2 процесса | 4 процесса |
|--------|-----------|------------|------------|
| 512    | 1.42      | 1.61       | 2.10       |
| 1024   | 5.91      | 7.16       | 9.52       |
| 2048   | 23.19     | 28.77      | 41.02      |

**Ускорение (относительно SEQ):**
| Размер | 1 процесс | 2 процесса | 4 процесса |
|--------|-----------|------------|------------|
| 512    | 0.75      | 0.66       | 0.50       |
| 1024   | 0.71      | 0.58       | 0.44       |
| 2048   | 0.72      | 0.58       | 0.41       |

**Эффективность (на 2048×2048):** 72%, 29%, 10%.

**Пояснения:** Время растёт с числом процессов из-за накладных расходов на `MPI_Bcast` и синхронизацию. Так как каждый процесс выполняет всю работу, параллелизм не даёт выигрыша – это ожидаемое поведение широковещательной модели. При 4 процессах на 2048×2048 ускорение падает до 0.41× (т.е. работает в 2.4 раза медленнее, чем SEQ).

## 10. Выводы
Данная реализация `all` является **демонстрационной** – она показывает, как использовать MPI для передачи данных, но не преследует цели ускорения. Реальное ускорение можно получить только при распределении работы между процессами (например, деление изображения на блоки). В рамках данной задачи чисто потоковые версии (OMP, STL) остаются наиболее эффективными. MPI-версия полезна для обучения и для случаев, когда необходимо дублировать вычисления (например, отказоустойчивость).
