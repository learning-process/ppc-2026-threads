## 1. Контекст

OpenMP-версия переносит вычислительное ядро `RunImpl` из последовательной
реализации в параллельную область с минимальными изменениями кода.
Поскольку каждый внутренний пиксель обрабатывается независимо,
декомпозиция по строкам не требует синхронизации записи и хорошо ложится
на модель `#pragma omp parallel for`.
Baseline для сравнения - `seq/report.md`.

## 2. Постановка задачи

Вход, выход и ограничения идентичны SEQ-версии: `Image{width, height, data}`

- `std::vector<int>` размером `width * height`.
Корректность подтверждается полным совпадением с SEQ на всём тестовом наборе.

## 3. Базовый алгоритм

Подробно описан в `seq/report.md`. Кратко: RGB - grayscale - двойной цикл
по внутренним пикселям - свёртка с ядром Собеля 3×3 -
`magnitude = sqrt(Gx² + Gy²)`.

## 4. Схема распараллеливания

Параллелится **внешний цикл по строкам** в `RunImpl`. Директива:

```cpp
// File: omp/src/ops_omp.cpp - RunImpl
#pragma omp parallel for default(none) \
    shared(output, gray, gx_kernel, gy_kernel) \
    firstprivate(h, w) \
    schedule(static)
for (int row = 1; row < h - 1; ++row) {
  for (int col = 1; col < w - 1; ++col) {
    // вычисление gx, gy, magnitude - полностью локально
    output[(row * w) + col] = magnitude;
  }
}
```

**Атрибуты переменных:**

| Переменная              | Атрибут        | Обоснование                                      |
| ----------------------- | -------------- | ------------------------------------------------ |
| `output`                | `shared`       | Каждый поток пишет в непересекающиеся строки     |
| `gray`                  | `shared`       | Только чтение                                    |
| `gx_kernel`, `gy_kernel`| `shared`       | Только чтение                                    |
| `h`, `w`                | `firstprivate` | Копируются в поток, исключают запись через `none`|
| `gx`, `gy`, `magnitude` | неявно private | Объявлены внутри цикла                           |

`default(none)` требует явного объявления каждой переменной -
это предотвращает скрытые ошибки при рефакторинге.

**Schedule:** `schedule(static)` делит строки равными блоками по числу потоков.
Это оправдано, поскольку стоимость обработки каждой строки одинакова
(фиксированное ядро 3×3), и нет необходимости в динамическом балансировании.

**Барьер:** в конце `parallel for` существует неявный барьер.
Он гарантирует, что все строки записаны в `output` до возврата из `RunImpl`.
Дополнительной синхронизации не требуется.

**Число потоков:** задаётся переменной окружения `OMP_NUM_THREADS` /
`PPC_NUM_THREADS`. В тестах использовалось значение 4.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Относительно SEQ изменён только `RunImpl`: добавлена директива
`#pragma omp parallel for` и вынесены локальные переменные `h`, `w` перед
циклом для явного `firstprivate`.
`ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` идентичны SEQ.

Гонка данных невозможна: строки `[1, h-2]` разбиваются `schedule(static)`
на непересекающиеся диапазоны, каждый поток пишет только в «свои» строки
`output`, а `gray` - read-only.

## 6. Проверка корректности

OMP-версия прошла все три функциональных теста
(ConstantImage, VerticalEdge, HorizontalEdge)
с результатами, побайтово совпадающими с SEQ.
Дополнительно проверена устойчивость при `OMP_NUM_THREADS=1` и
`OMP_NUM_THREADS=4`.

## 8. Результаты

Ускорение считается как `T_seq_task_run / T_omp_task_run` по режиму `task_run`.
Эффективность `E = speedup / threads`.

| size | SEQ task_run, с | OMP task_run, с | Speedup | Efficiency |
| ---- | --------------- | --------------- | ------- | ---------- |
| 512  | 0.00738         | 0.00605         | 1.22x   | 30.5%      |
| 1024 | 0.03471         | 0.01992         | 1.74x   | 43.5%      |
| 2048 | 0.11435         | 0.10180         | 1.12x   | 28.1%      |

| size | SEQ pipeline, с | OMP pipeline, с | Speedup |
| ---- | --------------- | --------------- | ------- |
| 512  | 0.00851         | 0.00751         | 1.13x   |
| 1024 | 0.03374         | 0.02719         | 1.24x   |
| 2048 | 0.12496         | 0.10881         | 1.15x   |

На размере 1024 достигается наилучшее ускорение `task_run`.
Низкая эффективность (~30-44%) объясняется накладными расходами на создание
parallel-региона и memory-bound характером задачи.
На 2048 ускорение падает из-за давления на LLC при доступе к большому
массиву `grayscale_`.

## 9. Выводы

OpenMP даёт умеренное ускорение (до ~1.74x при 4 потоках),
реализуется минимальной правкой SEQ-кода и не требует ручного управления
потоками. Узкое место - memory bandwidth, а не вычисления.
Для данной задачи `schedule(static)` оптимален; динамическое расписание
добавило бы overhead без выигрыша в балансировании.
