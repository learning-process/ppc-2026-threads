## 1. Контекст

oneTBB (Intel Threading Building Blocks) предоставляет высокоуровневые абстракции задачного параллелизма с автоматическим управлением пулом потоков и балансировщиком нагрузки. В отличие от OMP, поток управления явно не задаётся: программист описывает диапазон работы и операцию над ним, а runtime сам решает, как разбить диапазон на чанки. Baseline - `seq/report.md`.

## 2. Постановка задачи

Идентична SEQ-версии: `Image{width, height, data}` - `std::vector<int>`. Полное совпадение с SEQ на всём тестовом наборе.

## 3. Базовый алгоритм

Подробно описан в `seq/report.md`. RGB - grayscale - свёртка ядром Собеля 3×3 по внутренним пикселям - `magnitude = sqrt(Gx² + Gy²)`.

## 4. Схема распараллеливания

Использован примитив `tbb::parallel_for` с диапазоном `tbb::blocked_range<int>` по строкам изображения:

```cpp
// File: tbb/src/ops_tbb.cpp - RunImpl
tbb::parallel_for(
    tbb::blocked_range<int>(1, h - 1),
    [&](const tbb::blocked_range<int>& range) {
      for (int row = range.begin(); row != range.end(); ++row) {
        for (int col = 1; col < w - 1; ++col) {
          // вычисление gx, gy, magnitude
          output[(row * w) + col] = magnitude;
        }
      }
    });
```

**Диапазон работы:** `blocked_range<int>(1, h-1)` охватывает все внутренние строки. Каждый чанк - непрерывный поддиапазон строк.

**Grain size:** не задан явно, используется значение по умолчанию (`grainsize = 1`). TBB runtime выбирает фактический размер чанка динамически через work-stealing scheduler, адаптируясь к числу доступных потоков. Для данной задачи, где стоимость каждой строки равномерна, явный grainsize не критичен; при желании можно задать `blocked_range<int>(1, h-1, h/num_threads)` для уменьшения overhead при большом числе потоков.

**Partitioner:** используется `auto_partitioner` (по умолчанию) - он разбивает диапазон рекурсивно, пока чанки не становятся меньше grainsize. Это обеспечивает хорошее балансирование без явной настройки.

**Контроль конкуренции:** число рабочих потоков задаётся через `PPC_NUM_THREADS`. TBB использует `global_control::max_allowed_parallelism`, которое устанавливается фреймворком курса.

**Синхронизация:** `parallel_for` блокирует вызывающий поток до завершения всех задач - неявный барьер по завершении. Записи в `output` безопасны: чанки по строкам не пересекаются.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Изменения относительно SEQ минимальны: `RunImpl` оборачивает внешний цикл по строкам в `tbb::parallel_for`. Ядра `gx_kernel` и `gy_kernel` объявлены как `const` до лямбды и захватываются по ссылке - они неизменны, поэтому гонок нет. Локальные переменные `gx`, `gy`, `magnitude` объявлены внутри лямбды и полностью приватны для каждого потока.

`ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` идентичны SEQ.

## 8. Результаты

| size | SEQ task_run, с | TBB task_run, с | Speedup | Efficiency |
|------|----------------|----------------|---------|-----------|
| 512  | 0.00738        | 0.00951        | 0.78×   | 19.4%     |
| 1024 | 0.03471        | 0.03209        | 1.08×   | 27.1%     |
| 2048 | 0.11435        | 0.11893        | 0.96×   | 24.1%     |

| size | SEQ pipeline, с | TBB pipeline, с | Speedup |
|------|----------------|----------------|---------|
| 512  | 0.00851        | 0.01235        | 0.69×   |
| 1024 | 0.03374        | 0.04789        | 0.70×   |
| 2048 | 0.12496        | 0.14248        | 0.88×   |

TBB на данной задаче не опередил SEQ и уступил OMP. Причина - overhead work-stealing планировщика заметен при относительно лёгком вычислении (memory-bound задача с малой долей арифметики). При `pipeline`-режиме TBB дополнительно проигрывает из-за инициализации runtime при каждом запуске теста. TBB раскрывает своё преимущество при нерегулярных и вычислительно тяжёлых задачах.

## 9. Выводы

oneTBB удобен для задач с нерегулярным или неизвестным заранее распределением нагрузки - тогда work-stealing даёт реальный выигрыш. Для регулярной memory-bound задачи вроде фильтра Собеля накладные расходы runtime перевешивают выгоду от автоматического балансирования. При необходимости явный `grainsize = h/num_threads` и `simple_partitioner` снизили бы overhead, но полностью устранить его не позволят.
