## 1. Контекст

Оператор Собеля - классический алгоритм выделения границ на изображениях, применяемый в компьютерном зрении и обработке сигналов. Последовательная реализация служит эталоном корректности: все параллельные версии должны воспроизводить её результат побайтово. Именно по времени SEQ вычисляется ускорение во всех остальных отчётах.

## 2. Постановка задачи

**Вход (`InType = Image`):**
- `int width, height` - размеры изображения в пикселях;
- `std::vector<Pixel> data` - пиксели в порядке row-major, каждый пиксель `{uint8_t r, g, b}`.

**Ограничения:** `width > 2`, `height > 2`, `data.size() == width * height`.

**Выход (`OutType = std::vector<int>`):** массив размером `width * height`, где `output[row * width + col]` - целочисленная величина градиента для пикселя `(row, col)`. Граничные пиксели (строки 0 и `height-1`, столбцы 0 и `width-1`) остаются равными 0.

**Крайние случаи, блокируемые в `ValidationImpl`:**
- изображение 2×N, N×2 или меньше (нет ни одного внутреннего пикселя);
- несоответствие `data.size()` и `width * height`.

## 3. Базовый алгоритм

1. Преобразовать RGB в полутоновое изображение по формуле яркости:
   `gray = 0.299·R + 0.587·G + 0.114·B`.
2. Для каждого внутреннего пикселя `(row, col)`, где `1 ≤ row ≤ height−2`, `1 ≤ col ≤ width−2`, применить свёртку с ядрами Собеля 3×3:

```
Gx = [[-1, 0, 1],    Gy = [[-1, -2, -1],
      [-2, 0, 2],          [ 0,  0,  0],
      [-1, 0, 1]]          [ 1,  2,  1]]
```

3. Вычислить магнитуду: `magnitude = (int)sqrt(Gx² + Gy²)`, записать в `output[row*width + col]`.

**Асимптотика по времени:** O(W·H) - каждый внутренний пиксель обрабатывается за константное время (фиксированное ядро 3×3).

**Асимптотика по памяти:** O(W·H) - массив полутонового изображения и выходной массив.

**Инварианты и критерий корректности:** при одинаковом входе выход детерминирован и не зависит от порядка обхода пикселей, поскольку операции над каждым пикселем независимы.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

- **`ValidationImpl`** - проверяет `width > 2 && height > 2 && data.size() == width * height`. При несоблюдении задача не запускается.
- **`PreProcessingImpl`** - сохраняет `width_` и `height_`, выполняет конвертацию RGB-grayscale в `grayscale_`, инициализирует выходной буфер нулями.
- **`RunImpl`** - двойной цикл по строкам и столбцам внутренней области; для каждого пикселя считает `gx`, `gy` через вложенный обход 3×3-окрестности, записывает `magnitude`.
- **`PostProcessingImpl`** - пустой (результат уже в `GetOutput()`).

Ключевой фрагмент (`seq/src/ops_seq.cpp`):

```cpp
// File: seq/src/ops_seq.cpp  - RunImpl
for (int row = 1; row < height_ - 1; ++row) {
  for (int col = 1; col < width_ - 1; ++col) {
    int gx = 0, gy = 0;
    for (int ky = -1; ky <= 1; ++ky)
      for (int kx = -1; kx <= 1; ++kx) {
        int pixel = grayscale_[(row + ky) * width_ + (col + kx)];
        gx += pixel * gx_kernel[ky+1][kx+1];
        gy += pixel * gy_kernel[ky+1][kx+1];
      }
    GetOutput()[row * width_ + col] =
        static_cast<int>(std::sqrt((double)(gx*gx + gy*gy)));
  }
}
```

Оба ядра объявлены как `const std::array<std::array<int,3>,3>` - это гарантирует их размещение на стеке и отсутствие ложного разделения кэша при параллелизации в других версиях.

## 5. Проверка корректности

Функциональные тесты (`tests/functional/main.cpp`) охватывают три сценария:

| test_id | Описание | Ожидаемый результат |
|---------|----------|---------------------|
| 0 | Константное изображение 4×4 (все пиксели одного цвета) | Все градиенты == 0 |
| 1 | Вертикальный перепад яркости (левая половина чёрная, правая белая) | Ненулевые градиенты на внутренних пикселях границы: позиции (1,1),(1,2),(2,1),(2,2) == 1020 |
| 2 | Горизонтальный перепад (верхняя половина чёрная, нижняя белая) | Аналогично: (1,1),(1,2),(2,1),(2,2) == 1020 |

Значение 1020 объясняется тем, что при перепаде 0-255 на однопиксельной границе компонента градиента достигает 4·255 = 1020 (для ядра Gx при ky=0: `(-2)*0 + 2*255 = 510`; суммарно по строкам: 1020), а вторая компонента равна 0.

Корректность всех параллельных версий проверяется идентичным набором тестов - разногласий выявлено не было.

## 7. Результаты

Тест производительности запускается в двух режимах: `pipeline` (полный конвейер Validation - PreProcessing - Run - PostProcessing) и `task_run` (только `RunImpl`). Размер задачи - квадратное изображение `size × size` с горизонтальным градиентом яркости.

| size | pipeline, с | task_run, с |
|------|------------|------------|
| 512  | 0.00851    | 0.00738    |
| 1024 | 0.03374    | 0.03471    |
| 2048 | 0.12496    | 0.11435    |

Доминирующий вклад вносит `RunImpl`: вычисление ведётся за O(W·H) итераций с обращением к случайным позициям `grayscale_`, что создаёт давление на кэш при больших размерах. Именно эти числа используются как знаменатель для ускорения во всех параллельных отчётах.
