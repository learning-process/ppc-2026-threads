## 1. Введение

Оператор Собеля - классический метод выделения границ на изображениях,
основанный на приближении градиента яркости через свёртку с двумя ядрами 3×3
(по осям X и Y). Задача вычислительно регулярна: каждый внутренний пиксель
обрабатывается независимо с фиксированным числом операций.
Это делает её удобным полигоном для сравнения технологий параллелизации:
потенциально высокий параллелизм данных сталкивается с реальным ограничением -
memory bandwidth, а не арифметика.

В проекте реализовано пять версий: SEQ (эталон), OMP, TBB, STL и ALL.

## 2. Единая постановка задачи

**Вход (`InType = Image`):** `{int width, int height, std::vector<Pixel> data}`,
где `Pixel = {uint8_t r, g, b}`. Пиксели в row-major порядке.

**Выход (`OutType = std::vector<int>`):** размер `width * height`.
Элемент `output[row*width + col]` - целочисленная магнитуда градиента.
Граничные пиксели равны 0.

**Ограничения:** `width > 2`, `height > 2`, `data.size() == width * height`.

**Критерий корректности:** результат побайтово совпадает с SEQ-версией
на одинаковом входе.

## 3. Единая методика эксперимента

| Параметр                       | Значение                             |
| ------------------------------ | ------------------------------------ |
| CPU                            | AMD Ryzen 3 3100 (4 физических ядра) |
| ОС                             | Windows 10                           |
| Компилятор                     | MSVC (Release)                       |
| CMake build type               | Release                              |
| `PPC_NUM_THREADS`              | 4                                    |
| MPI-процессов (`mpiexec -n`)   | 4                                    |
| Конфигурация ALL               | 4 ranks x 4 OMP-потока               |

**Генерация входных данных:** квадратное изображение `size × size`
с горизонтальным линейным градиентом: `value = (255 * col) / (size - 1)`,
`R = G = B = value`.

**Размеры задач:** 512×512, 1024×1024, 2048×2048.

**Режимы производительности:** `task_run` (только `RunImpl`)
и `pipeline` (полный конвейер).

**Формула ускорения:** `S = T_seq_task_run / T_backend_task_run`.

**Формула эффективности:** `E = S / workers`.
Для ALL: `workers = ranks x threads_per_rank = 16`.

## 6. Интерпретация различий

**SEQ** демонстрирует линейный рост времени с площадью изображения - O(W·H),
что ожидаемо. Используется как знаменатель ускорения.

**OMP** показывает лучшую эффективность среди потоковых версий (до 43.5%
при 4 потоках, size=1024) благодаря минимальным накладным расходам `parallel for`
и хорошей локальности кэша при `schedule(static)`.
На size=2048 ускорение падает до 1.12x - признак насыщения memory bandwidth.

**TBB** на данной задаче уступает SEQ при size=512.
Work-stealing scheduler добавляет overhead, который заметен при регулярной
и лёгкой вычислительной нагрузке.
TBB раскрывает преимущества при нерегулярных задачах.

**STL** достигает ускорения 1.57-1.59x при 8 потоках на size=1024-2048.
Высокий overhead создания потоков делает его невыгодным на size=512
в `pipeline`-режиме. Ручное разбиение диапазона корректно,
`join` вызывается вне цикла создания потоков - что обеспечивает реальный
параллелизм.

**ALL** показывает наилучшее абсолютное ускорение (3.21x при size=512,
3.14x при size=2048), опережая все чисто потоковые версии.
Двухуровневый параллелизм: 4 MPI-процесса делят строки крупными блоками,
4 OMP-потока внутри каждого параллелят более мелкие блоки.

Близкая эффективность у всех технологий (~17-44%) указывает на то,
что задача memory-bound: добавление потоков конкурирует за одну шину памяти,
а не за свободные ALU.

## 8. Заключение

Для задачи фильтрации Собеля на изображениях размером 512-2048:

- **Лучшая производительность:** ALL (MPI + OpenMP) - до 3.21x ускорения
  при 16 рабочих единицах.
- **Лучшее соотношение ускорение/простота реализации:** OMP - минимальные
  правки SEQ, ускорение до 1.74x.
- **TBB неэффективен** на данной регулярной задаче.
- **STL** занимает промежуточное место: выше TBB при больших задачах,
  но требует больше кода.

Главное ограничение полученных чисел - memory-bound характер алгоритма:
дальнейшее увеличение числа потоков без роста пропускной способности памяти
не даст линейного ускорения.
