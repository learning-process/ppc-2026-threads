## 1. Введение

Оператор Собеля - классический метод выделения границ на изображениях, основанный на приближении градиента яркости через свёртку с двумя ядрами 3×3 (по осям X и Y). Задача вычислительно регулярна: каждый внутренний пиксель обрабатывается независимо с фиксированным числом операций. Это делает её удобным полигоном для сравнения технологий параллелизации: потенциально высокий параллелизм данных сталкивается с реальным ограничением - memory bandwidth, а не арифметика.

В проекте реализовано пять версий: SEQ (эталон), OMP, TBB, STL (`std::thread`) и ALL (MPI + OpenMP).

## 2. Единая постановка задачи

**Вход (`InType = Image`):** `{int width, int height, std::vector<Pixel> data}`, где `Pixel = {uint8_t r, g, b}`. Пиксели в row-major порядке.

**Выход (`OutType = std::vector<int>`):** размер `width * height`. Элемент `output[row*width + col]` - целочисленная магнитуда градиента. Граничные пиксели равны 0.

**Ограничения:** `width > 2`, `height > 2`, `data.size() == width * height`.

**Критерий корректности:** результат побайтово совпадает с SEQ-версией на одинаковом входе.

| ОС | Windows 10 |

| CMake build type | Release |

**Генерация входных данных (perf-тесты):** квадратное изображение `size × size` с горизонтальным линейным градиентом: `value = (255 * col) / (size - 1)`, `R = G = B = value`.

**Размеры задач:** 512×512, 1024×1024, 2048×2048.

**Режимы производительности:** `task_run` (только `RunImpl`) и `pipeline` (полный конвейер). Здесь и далее ускорение считается по `task_run`, эффективность - относительно этого же режима.

**Формула ускорения:** `S = T_seq_task_run / T_backend_task_run`.

**Формула эффективности:** `E = S / workers`, где `workers` - число рабочих единиц данной технологии (для ALL: `ranks × threads_per_rank = 16`).

**Агрегация:** каждая метрика - результат одного запуска пакета из 10 тестов фреймворком курса. Числа отражают типичную нагрузку без специальных мер.

**Команда запуска:**
```bash
mpiexec -n 4 .\build\bin\ppc_perf_tests.exe --gtest_filter="*KrykovE*"
```

## 6. Интерпретация различий

**SEQ** демонстрирует линейный рост времени с площадью изображения (≈O(W·H)), что ожидаемо. Именно он используется как знаменатель ускорения.

**OMP** показывает лучшую эффективность среди потоковых версий (до 43.5% при 4 потоках, size=1024) благодаря минимальным накладным расходам `parallel for` и хорошей локальности кэша при `schedule(static)`. На size=2048 ускорение падает до 1.12× - признак насыщения memory bandwidth при 4 потоках.

**TBB** на данной задаче уступает SEQ при size=512 и незначительно опережает его при 1024. Work-stealing scheduler добавляет overhead, который заметен при регулярной и лёгкой вычислительной нагрузке. TBB раскрывает преимущества при нерегулярных задачах - данная задача для него не оптимальна.

**STL** достигает ускорения 1.57–1.59× при 8 потоках на size=1024–2048. Высокий overhead создания потоков (`emplace_back`) делает его невыгодным на size=512 в `pipeline`-режиме. Ручное разбиение диапазона корректно, `join` вызывается вне цикла создания потоков - что обеспечивает реальный параллелизм.

**ALL** показывает наилучшее абсолютное ускорение (3.21× при size=512, 3.14× при size=2048), опережая все чисто потоковые версии. Это объясняется двухуровневым параллелизмом: 4 MPI-процесса делят строки крупными блоками, 4 OMP-потока внутри каждого параллелят более мелкие блоки. Несмотря на `MPI_Gatherv` + `MPI_Bcast`, суммарный коммуникационный overhead при данных размерах изображений (4–16 МБ) оказывается меньше выигрыша от распределения вычислений.

Близкая эффективность у OMP, STL и ALL (~17–44%) при разном числе рабочих единиц указывает на то, что задача memory-bound: добавление потоков конкурирует за одну шину памяти, а не за свободные ALU. Это типичное ограничение для алгоритмов с малой вычислительной интенсивностью (≈18 операций на 9 чтений для ядра 3×3).

## 8. Заключение

Для задачи фильтрации Собеля на изображениях размером 512–2048:

- **Лучшая производительность:** ALL (MPI + OpenMP) - до 3.21× ускорения при 16 рабочих единицах.
- **Лучшее соотношение ускорение/простота реализации:** OMP - минимальные правки SEQ, ускорение до 1.74×.
- **TBB неэффективен** на данной регулярной задаче: overhead runtime перевешивает выигрыш при 4 потоках.
- **STL** занимает промежуточное место: выше TBB при больших задачах, но требует больше кода.

Главное ограничение полученных чисел - memory-bound характер алгоритма: дальнейшее увеличение числа потоков без роста пропускной способности памяти не даст линейного ускорения. Переход на тайловую обработку с учётом кэш-иерархии или использование SIMD-инструкций для свёртки мог бы кардинально изменить ситуацию.
