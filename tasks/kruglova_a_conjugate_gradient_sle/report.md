## 1. Введение

Метод сопряженных градиентов является итерационным методом решения СЛАУ с симметричной положительно определенной
матрицей. Сложность метода составляет $O(k \cdot n^2)$, где $k$ — количество итераций. Основная вычислительная нагрузка
приходится на операцию умножения матрицы на вектор, что делает данный алгоритм отличным кандидатом для параллелизации.

## 2. Постановка задачи

### Входные данные

- Квадратная симметричная положительно определенная матрица коэффициентов $A$ размера $n \times n$ (представлена как
одномерный вектор типа double)
- Вектор правой части $b$ размера $n$ (тип double)
- Размерность системы $n$ (целое число)

### Выходные данные

- Вектор неизвестных $x$ размера $n$ (тип double), являющийся решением системы $Ax = b$

### Критерий корректности

Выходной вектор $x$ считается корректным решением СЛАУ, если норма вектора невязки $r = b - Ax$ не превышает заданного
порогового значения точности $\epsilon$:

$$\|r\|_2 = \sqrt{\sum_{i=0}^{n-1} \left(b_i - \sum_{j=0}^{n-1} A_{ij}x_j\right)^2} < \epsilon, \quad \text{где }
\epsilon = 10^{-8}$$

Это условие гарантирует, что найденное приближение $x$ с высокой точностью удовлетворяет исходной системе уравнений.

## 3. Методика эксперимента

### Окружение

- **CPU:** AMD Ryzen 5 5500U @ 2.10 GHz (6 ядер, 12 потоков)
- **RAM:** 16 ГБ

- **Build type:** Release

### Генерация данных

**Входные данные:** Система линейных уравнений $Ax = b$ генерируется таким образом, чтобы матрица $A$ была симметричной
и положительно определенной (для обеспечения сходимости метода). Элементы матрицы формируются с использованием
генератора случайных чисел `std::mt19937_64` с последующим обеспечением диагонального преобладания.

**Объем данных для perf-тестов:** размер системы $n = 1000$ (что соответствует $10^6$ элементов в матрице $A$).
Итерационный процесс продолжается до достижения невязки $\|r\|_2 < 10^{-8}$.

### Метрики эффективности

Для оценки параллельных реализаций используются следующие показатели, где $T_{seq}$ — время выполнения последовательной
версии, а $T_{p}$ — время параллельной версии на $N$ исполнителях (потоках или процессах):

$$S = \frac{T_{seq}}{T_{p}}, \quad E = \frac{S}{N}$$

За итоговое время выполнения каждой реализации принимается минимальное значение из трех последовательных запусков для
исключения влияния фоновых процессов операционной системы.

## 4. Сводка корректности

Корректность всех параллельных реализаций (OMP, TBB, STL, ALL) проверялась путем сравнения результатов работы алгоритма
с эталонной последовательной версией (SEQ) и контроля достижения заданной точности на наборе из 7 функциональных тестов.

**Size1_1:** Система из одного уравнения (базовая проверка корректности размерности)

**Identity_3:** Система с единичной матрицей (проверка корректности работы алгоритма при отсутствии смешивания данных)

**Diagonal_5:** Система с диагональной матрицей (проверка на разреженных структурах)

**Size10_10:** Система малого размера (проверка сходимости на типичных данных)

**Size50_50:** Система среднего размера (проверка стабильности итерационного процесса)

**OddSize_101:** Система нечетной размерности (проверка корректности работы с остатками при распределении данных между
потоками/процессами)

**ConvergenceTest:** Массив идентичных элементов / плохо обусловленная система (проверка устойчивости метода к делению
на малые величины и сходимости при наличии вырожденности)

### Критерий успеха

Во всех случаях достижение нормы невязки $\|r\|_2 < 10^{-8}$ за количество итераций, не превышающее $2n$. Сравнение с
последовательным эталоном показало идентичность результатов с точностью до машинного эпсилон (разница не превышает
$10^{-12}$).

### 6. Интерпретация различий

**SEQ:** Служит базовой метрикой производительности (baseline). Время выполнения стабильно, так как отсутствуют
накладные расходы на синхронизацию потоков.

**OMP, TBB, STL:** Демонстрируют незначительное ускорение (до 1.13x). Это связано с тем, что на данной размерности
задачи ($n=1000$) накладные расходы на создание потоков, передачу управления планировщику (в TBB) и синхронизацию памяти
сравнимы с временем выполнения самих вычислений (умножения матрицы на вектор).

**ALL (MPI + OMP):** Гибридная версия показала суперлинейное ускорение (4.18x) на 4 процессах. Это является важным
феноменом: при делении матрицы $A$ на блоки, части матрицы каждого процесса начинают эффективно помещаться в локальную
кэш-память L2/L3 процессора, что резко снижает время доступа к данным по сравнению с последовательной версией,
работающей с общей памятью (где данные не кэшируются столь эффективно).

## 7. Репродуцируемость

Для воспроизведения результатов необходимо выполнить следующие шаги:

1. **Сборка проекта:**

    ```powershell
    cmake -B build -DCMAKE_BUILD_TYPE=Release
    cmake --build build --config Release
    ```

2. **Запуск функциональных тестов:**

    ```powershell
    .\build\bin\ppc_func_tests.exe
    ```

3. **Получение основных замеров:**

    ```powershell
    $env:OMP_NUM_THREADS=6
    .\build\bin\ppc_perf_tests.exe --gtest_filter="*kruglova*omp*"

    $env:TBB_NUM_THREADS=6
    .\build\bin\ppc_perf_tests.exe --gtest_filter="*kruglova*tbb*"

    $env:STL_NUM_THREADS=6
    .\build\bin\ppc_perf_tests.exe --gtest_filter="*kruglova*stl*"

    mpiexec -np 4 .\build\bin\ppc_perf_tests.exe --gtest_filter="*kruglova*all*"
    ```
