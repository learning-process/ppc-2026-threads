## 1. Контекст

Данный отчет описывает реализацию метода сопряженных градиентов для решения систем линейных алгебраических уравнений
(СЛАУ) с использованием стандартной библиотеки потоков C++ (std::thread).

В отличие от высокоуровневых абстракций OpenMP и TBB, использование std::thread требует ручного управления жизненным
циклом потоков (создание, запуск, синхронизация через join), самостоятельного расчета границ данных для каждого потока и
явной организации локальных буферов.
Эта версия необходима для глубокого понимания низкоуровневых механизмов многопоточности и оценки чистых накладных
расходов на управление потоками операционной системы.

## 2. Постановка задачи

Входные и выходные параметры, а также математические инварианты соответствуют эталонной последовательной версии:

- **Вход:** одномерный вектор вещественных чисел типа double размера $n \times n$ (матрица A), вектор правой части b
размера $n$, размерность системы $n$
- **Выход:** вектор решений x размера $n$
- **Критерий корректности:** норма вектора невязки должна быть строго меньше заданного допуска: $\sqrt{\sum_{i=0}^{n-1}
r_i^2} < 10^{-8}$

**Задача распараллеливания:** реализовать равномерное ручное деление вычислительной нагрузки между заданным количеством
потоков std::thread, минимизировать использование блокировок (mutex) за счет применения локальных аккумуляторов и
обеспечить строгую синхронизацию перед обновлением глобальных состояний.

## 3. Схема распараллеливания

Поскольку метод сопряженных градиентов требует барьерной синхронизации на каждом шаге (для вычисления параметров
$\alpha$ и $\beta$), потоки должны быть синхронизированы перед переходом к следующей фазе итерации.

**Декомпозиция данных:** Итерационное пространство $n$ делится на непрерывные блоки (chunks). Размер блока для потока
вычисляется как `n / num_threads`, а остаток от деления `n % num_threads` распределяется по одному элементу между
первыми потоками для балансировки нагрузки.

**Локальные вычисления:** Каждый поток вычисляет свою часть вектора $ap$ (умножение подматрицы на вектор $p$), а также
локальные суммы для скалярных произведений $r^T \cdot r$ и $p^T \cdot ap$.

**Синхронизация и редукция:** Вместо использования тяжелых мьютексов (std::mutex), каждый поток записывает свою
локальную сумму в выделенную ячейку разделяемого массива `local_sums[thread_id]`. После вызова `join()` для всех
потоков, главный поток линейно суммирует массив локальных результатов и обновляет векторы.

## 4. Детали реализации

Основной паттерн управления потоками включает создание вектора `std::vector<std::thread>`, запуск вычислений и
обязательный барьер `join()`.

Иллюстративный пример ручного распределения работы:

```cpp
// Вычисление произведения матрицы на вектор с использованием std::thread
std::vector<std::thread> threads;
int chunk = n / num_threads;
int remainder = n % num_threads;

for (int t = 0; t < num_threads; ++t) {
  int begin = t * chunk + std::min(t, remainder);
  int end = begin + chunk + (t < remainder ? 1 : 0);
  
  threads.emplace_back([&a, &p, &ap, n, begin, end]() {
    for (int i = begin; i < end; ++i) {
      ap[i] = 0.0;
      for (int j = 0; j < n; ++j) {
        ap[i] += a[(i * n) + j] * p[j];
      }
    }
  });
}

// Точка явной синхронизации (барьер)
for (auto& th : threads) {
  th.join();
}

```

В данной схеме отсутствует риск состояний гонки (data races), так как каждый поток производит запись только в свой
изолированный непересекающийся диапазон `[begin, end)` массива `ap`.

## 8. Выводы

Анализ результатов ручной реализации std::thread выявляет несколько важных феноменов:

**Сверхлинейное ускорение на 2 потоках:** Переход от последовательной версии к 2 потокам дал аномально высокое ускорение
($S \approx 4.36$). Это классическое проявление эффекта кэширования (cache effect).
При разделении матрицы на 2 блока, рабочие наборы данных (working sets) для каждого потока стали помещаться в более
быстрые уровни кэш-памяти процессора (L2/L3), что кардинально снизило количество медленных обращений к оперативной
памяти (cache misses), характерных для последовательной версии.

**Предел масштабируемости (Memory Bound):** При дальнейшем увеличении числа потоков до 4 и 6 время работы остается
практически неизменным (119 ms и 121 ms соответственно). Эффективность падает. Это означает, что алгоритм достиг предела
пропускной способности шины памяти (Memory Bandwidth).
Процессор способен вычислять быстрее, но ожидание доставки данных из RAM становится главным "узким местом".

**Накладные расходы join:** Незначительное замедление при переходе от 2 к 6 потокам (с 117 ms до 121 ms) обусловлено
накладными расходами операционной системы на создание потоков (`emplace_back`) и ожидание их завершения (`join`) на
каждой итерации цикла.
 Для дальнейшей оптимизации STL-версии необходимо реализовать шаблон пула потоков (Thread Pool), который позволил бы
избежать постоянного пересоздания системных потоков выполнения.
