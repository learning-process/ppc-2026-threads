## 1. Контекст

Данная реализация является развитием последовательной (SEQ) версии алгоритма решения систем линейных алгебраических
уравнений (СЛАУ) методом сопряженных градиентов. Использование стандарта OpenMP позволяет задействовать ресурсы
многоядерных процессоров (в рамках архитектуры общей памяти) путем распараллеливания наиболее ресурсоемких циклов.

Основная цель реализации — продемонстрировать корректное разделение данных между потоками и измерить реальное ускорение
($S$) и эффективность ($E$) относительно базовой линии (baseline), полученной в последовательной версии.

## 2. Постановка задачи

Входные и выходные данные, а также математические ограничения полностью идентичны последовательной версии:

- **Вход:** матрица коэффициентов A (развернутая в одномерный вектор размером $n \times n$), вектор правой части b
(размером $n$), размерность системы $n$
- **Выход:** вектор решений x (размером $n$)
- **Критерий корректности:** $\sqrt{\sum_{i=0}^{n-1} r_i^2} < 10^{-8}$

**Задача распараллеливания:** минимизировать время выполнения одной итерации метода за счет распределения независимых
вычислительных блоков (перемножение матрицы на вектор, скалярные произведения, обновление векторов) между рабочими
потоками OpenMP.

## 3. Базовый алгоритм и стратегия параллелизма

Математически алгоритм остается неизменным. Ускорение достигается за счет параллельного выполнения независимых итераций
циклов (for) на каждом шаге метода:

- **Matrix-Vector Multiplication ($A \cdot p$):** Вычисление каждой $i$-й компоненты результирующего вектора полностью
независимо от других. Внешний цикл по $i$ распараллеливается.

- **Скалярные произведения ($r^T \cdot r$, $p^T \cdot ap$):** Операции требуют вычисления локальных сумм в каждом потоке
с последующей глобальной редукцией (сложением) результатов в одну переменную.

- **Обновление векторов ($x$, $r$, $p$):** Пересчет $i$-й компоненты вектора зависит только от $i$-х компонент других
векторов, что делает эти циклы идеально распараллеливаемыми (embarrassingly parallel).

## 4. Детали реализации

В файле `ops_omp.cpp` для управления потоками используются директивы `#pragma omp parallel for`. Для предотвращения
состояний гонки (race conditions) и обеспечения детерминированности строго контролируются области видимости переменных.

### Ключевые атрибуты OpenMP, использованные в коде

- **`default(none)`**: обязывает явно указывать класс памяти для каждой используемой переменной внутри параллельной
секции, что исключает неявное и потенциально ошибочное разделение

- **`shared(...)`**: применяется к массивам (a, p, ap, r, x) и константам (n, alpha), так как потоки обращаются к
непересекающимся участкам памяти или только читают данные

- **`reduction(+ : ...)`**: используется для безопасного накопления скалярных произведений (rsold, p_ap). OpenMP
автоматически создает приватные копии переменной для каждого потока, а по завершении цикла суммирует их в глобальную
разделяемую переменную

### Примеры из реализации

```cpp
// Умножение матрицы на вектор
namespace {
void MatrixVectorMultiply(const std::vector<double> &a, const std::vector<double> &p, 
                          std::vector<double> &ap, int n) {
#pragma omp parallel for default(none) shared(a, p, ap, n)
  for (int i = 0; i < n; ++i) {
    ap[i] = 0.0;
    for (int j = 0; j < n; ++j) {
      ap[i] += a[(static_cast<size_t>(i) * n) + j] * p[j];
    }
  }
}
}  // namespace

// Вычисление скалярного произведения
double rsold = 0.0;
#pragma omp parallel for default(none) shared(r, n) reduction(+ : rsold)
for (int i = 0; i < n; ++i) {
  rsold += r[i] * r[i];
}

// ... внутри основного итерационного цикла ...
double p_ap = 0.0;
#pragma omp parallel for default(none) shared(p, ap, n) reduction(+ : p_ap)
for (int i = 0; i < n; ++i) {
  p_ap += p[i] * ap[i];
}

// Обновление векторов
#pragma omp parallel for default(none) shared(x, r, ap, p, alpha, n)
for (int i = 0; i < n; ++i) {
  x[i] += alpha * p[i];
  r[i] -= alpha * ap[i];
}

```

Корректность параллельной версии подтверждена на том же наборе функциональных тестов (SleSolverTests), что и
последовательная версия.

Все 6 тестов (Size1_1, Identity_3, Diagonal_5, Size10_10, Size50_50, OddSize_101) успешно пройдены при запуске с
конфигурациями 2, 4 и 6 потоков. Это подтверждает, что директивы редукции и распределения итераций отрабатывают
корректно, в том числе на матрицах нечетного размера, где работа делится между потоками неравномерно.

## 7. Результаты

Согласно логам `ppc_perf_tests`, последовательная и параллельная реализации продемонстрировали следующие результаты на
запусках производительности (режим pipeline):

| Backend | Потоки (p) | Время (ms) | Ускорение (S) | Эффективность (E) |
|---------|------------|------------|---------------|-------------------|
| SEQ     | 1          | 510        | 1.00          | 1.00              |
| OMP     | 2          | 544        | 0.94          | 0.47              |
| OMP     | 4          | 319        | 1.60          | 0.40              |
| OMP     | 6          | 305        | 1.67          | 0.28              |

### Формулы расчета

- **Ускорение:** $S = T_1 / T_p$, где $T_1$ — время выполнения последовательной версии, $T_p$ — время выполнения на $p$
потоках
- **Эффективность:** $E = S / p$, где $p$ — число потоков

## 8. Выводы

Реализация алгоритма с использованием OpenMP полностью функциональна. Анализ результатов производительности позволяет
сделать следующие выводы:

**Накладные расходы (Overhead):** На 2 потоках наблюдается деградация производительности (время выросло до 544 ms, $S <
1$). Это классический случай, когда накладные расходы на создание пула потоков OpenMP, синхронизацию барьеров в конце
каждого `#pragma omp parallel for` и операции редукции превышают выигрыш от распараллеливания на малых или средних
объемах данных.

**Масштабируемость:** На 4 потоках получено ускорение в 1.6 раза, на 6 потоках - в 1.67 раза. Рост производительности
присутствует, но далек от линейного (эффективность падает до 0.28 на 6 потоках).

**Узкое место (Memory Bound):** Метод сопряженных градиентов (в частности, операция умножения матрицы на вектор)
характеризуется низким отношением числа арифметических операций к количеству обращений в память. Основным ограничивающим
фактором (bottleneck) при увеличении числа потоков становится пропускная способность шины памяти (Memory Bandwidth),
из-за чего процессорные ядра простаивают в ожидании данных. Расширение числа потоков с 4 до 6 дает крайне незначительный
прирост, что типично для алгоритмов данного класса на локальных ПК.
