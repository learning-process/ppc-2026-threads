## 1. Контекст

Реализация параллелизма средствами стандартной библиотеки C++ (модель `std::thread`) представляет собой низкоуровневый
подход к многопоточности «вручную» (explicit multithreading). В отличие от высокоуровневых абстракций (OpenMP или Intel
oneTBB), данный бэкенд не использует скрытый runtime-планировщик для динамического деления итераций, а возлагает задачу
декомпозиции данных, распределения нагрузки и жизненного цикла потоков непосредственно на код разработчика.

## 2. Постановка задачи

Необходимо распараллелить вычисление пространственной свёртки матрицы пикселей изображения Ultra HD 8K. На низком уровне
требуется разделить итерационное пространство по строкам кадра, обеспечив:

- Детерминированное и равномерное распределение строк матрицы между заданным числом потоков $p$.
- Корректную обработку остатка от деления (когда высота изображения не делится нацело на количество потоков).
- Запуск всех вычислительных потоков строго до начала процедуры ожидания их завершения (`join`).

## 3. Схема распараллеливания и управление потоками

В данной реализации применена одномерная блочная декомпозиция данных по строкам (row-wise slicing):

1. **Равномерное разделение (Grid Slicing)**: Исходная высота изображения `height` делится на количество доступных
   потоков `num_threads`. Базовое количество строк на один поток определяется как `height / num_threads`. Оставшиеся
   строки (`height % num_threads`) распределяются по одной среди первых потоков-воркеров. Такой подход гарантирует
   сбалансированную нагрузку с максимальной разницей в одну строку между потоками.
2. **Жизненный цикл потоков**: Все объекты `std::thread` создаются внутри последовательного цикла и сразу же начинают
   независимое выполнение метода `ProcessRows()`. Потоки аккумулируются в динамическом векторе
   `std::vector<std::thread>`.
3. **Барьерная синхронизация через `join`**: Вызовы метода `.join()` для всех созданных потоков вынесены в отдельный
   изолированный цикл, который начинает работу **строго после** того, как все потоки были успешно запущены. Это
   критически важно, так как вызов `join` сразу после создания потока привёл бы к фактической сериализации
   (последовательному выполнению) алгоритма.

## 4. Детали реализации

Файлы проекта: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Ниже представлен управляющий код бэкенда из метода `RunImpl()`:

```cpp
// File: stl/src/ops_stl.cpp
bool NikolaevDBlockLinearImageFilteringSTL::RunImpl() {
  const int width = std::get<0>(GetInput());
  const int height = std::get<1>(GetInput());
  const auto &src = std::get<2>(GetInput());

  auto &dst = GetOutput();
  dst.assign(src.size(), 0);

  int num_threads = ppc::util::GetNumThreads();
  if (num_threads == 0) {
    num_threads = 4;
  }
  num_threads = std::min(height, num_threads);

  std::vector<std::thread> threads;
  threads.reserve(num_threads);

  int rows_per_thread = height / num_threads;
  int remainder = height % num_threads;

  int start_row = 0;
  // Шаг 1: Конвейерный запуск всех потоков без блокировок
  for (int i = 0; i < num_threads; ++i) {
    int end_row = start_row + rows_per_thread + (i < remainder ? 1 : 0);
    threads.emplace_back(
        [this, &src, width, height, start_row, end_row]() { ProcessRows(src, width, height, start_row, end_row); });

    start_row = end_row;
  }

  // Шаг 2: Ожидание завершения всей группы потоков
  for (auto &t : threads) {
    if (t.joinable()) {
      t.join();
    }
  }

  return true;
}
```

Вычислительная функция `ProcessRows` производит запись в общий вектор `dst`. Каждый поток оперирует строго внутри своего
диапазона строк от `start_row` до `end_row`. Поскольку интервалы памяти `((ny * width + nx) * 3)` для разных потоков
гарантированно не пересекаются и алгоритм выполняется без синхронизирующих мьютексов.

- Относительное ускорение: $S(p) = T(1) / T(p)$

## 8. Выводы и анализ

1. **Анализ базовой точки ($p = 1$)**: Однопоточная версия STL показала время $\approx 0.505$ сек, что практически
   идентично чистому последовательному коду `SEQ` ($\approx 0.497$ сек). Минимальный оверхед (менее **2%**) объясняется
   отсутствием тяжеловесных надстроек и runtime-библиотек. Программа создаёт ровно один системный поток, выполняющий
   прямолинейный проход по строкам матрицы, что делает этот бэкенд самым легковесным при $p = 1$.
2. **Оценка масштабируемости**: Код демонстрирует отличную линейную масштабируемость на малом количестве ядер. При
   $p = 2$ ускорение достигает $1.90\times - 1.94\times$ с эффективностью до **96.9%**. Ручное распределение непрерывных
   и протяженных блоков строк (размером более чем по 1000 строк на поток при $p = 2, 4$) обеспечивает прекрасную
   локальность данных, так как каждый поток последовательно читает и пишет в свой крупный непрерывный кусок ОЗУ.
3. **Ограничения шины памяти (6 ядер)**: На 6 потоках время выполнения сокращается до рекордных **0.113 сек**,
   обеспечивая относительное ускорение в $4.47\times$. Эффективность снижается до **74.5%**. Так как вычислительные
   потоки `std::thread` запущены параллельно и не имеют интеллектуального планировщика (как `auto_partitioner` в TBB,
   оптимизирующего блоки под размер кэша), они начинают одновременно и агрессивно насыщать шину памяти ОЗУ операциями
   чтения-записи тяжелых массивов данных. Это предел пропускной способности памяти (Memory Bound).
