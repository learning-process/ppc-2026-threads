## 1. Контекст

Гибридная реализация сочетает две дополняющие друг друга концепции параллельного программирования. На макроуровне (между
процессами) используется интерфейс передачи сообщений MPI, позволяющий декомпозировать задачу для выполнения в
распределенных системах или изолированных адресных пространствах. На микроуровне (внутри каждого отдельного процесса
MPI) применяется библиотека Intel oneTBB, которая эффективно утилизирует доступные вычислительные ядра
многопроцессорного узла через динамическое распределение легковесных задач по потокам. Такой подход минимизирует
накладные расходы на межпроцессное взаимодействие и максимизирует локальность кэш-памяти.

## 2. Постановка задачи

Вычислительное ядро производит пространственную блочную фильтрацию изображения Ultra HD 8K. При распределении строк
матрицы между процессами возникает проблема граничных пикселей: для расчёта свёртки Гаусса $3 \times 3$ на границе
раздела процессов необходимы данные из соседних областей.

**Ключевые требования к гибридной обработке:**

- Реализация корректного разбиения координатной сетки с помощью структуры `GridSlicing`.
- Организация обмена граничными элементами (т. н. гало-зон или ghost cells) при распределении данных.
- Исключение избыточных синхронизаций и обеспечение параллельной обработки тайлов размера $32 \times 32$ внутри
  локальной памяти каждого ранга.

## 3. Схема распараллеливания и MPI-синхронизация

Алгоритм распределения вычислительной нагрузки состоит из трех последовательных этапов:

1. **Инициализация и распределение данных (`ScatterWithHalo`)**: Главный процесс (Rank 0) вычисляет схему нарезки
   изображения `GridSlicing`. Высота изображения делится на макро-блоки по строкам. На стыках этих блоков формируются
   гало-зоны (`halo_top` и `halo_bottom`) шириной в 1 строку, необходимые для корректного применения трафарета оператора
   фильтрации. Передача подмассивов с учетом перекрытий осуществляется коллективной операцией `MPI_Scatterv`.
2. **Внутриузловой параллелизм (`RunLocalTBB`)**: Каждый процесс MPI, получив свою часть кадра и гало-строки, запускает
   `tbb::parallel_for` над двумерным пространством локальных блоков `blocked_range2d`. Внутри диапазона изображение
   нарезается на тайлы размером $32 \times 32$ (`kBlockSize = 32`). Если блок лежит внутри области, вызывается
   оптимизированный метод `ProcessFullBlock`, если на границе кадра — `ProcessPartBlock` с проверкой выхода за
   физические индексы.
3. **Сбор результатов и синхронизация (`GatherAndBroadcast`)**: По завершении локальных циклов TBB, очищенные от
   гало-зон результирующие строки собираются на главном процессе с помощью `MPI_Gatherv`. Затем, согласно архитектурным
   требованиям тестов производительности, результирующий буфер рассылается обратно всем процессам через `MPI_Bcast`.

## 4. Детали реализации

Файлы проекта: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Критически важная часть логики гибридного ядра сосредоточена в методе `RunImpl()`:

```cpp
// File: all/src/ops_all.cpp
bool NikolaevDBlockLinearImageFilteringALL::RunImpl() {
  int rank = 0;
  int world_size = 1;
  MPI_Comm_rank(MPI_COMM_WORLD, &rank);
  MPI_Comm_size(MPI_COMM_WORLD, &world_size);

  std::array<int, 2> dims{};
  if (rank == 0) {
    dims[0] = std::get<0>(GetInput());
    dims[1] = std::get<1>(GetInput());
  }
  MPI_Bcast(dims.data(), 2, MPI_INT, 0, MPI_COMM_WORLD);
  const int width = dims[0];
  const int height = dims[1];

  // Построение сетки распределения строк
  const GridSlicing dist = BuildGridSlicing(rank, world_size, height);

  // Выделение памяти под локальный входной буфер (с учетом гало-зон)
  std::vector<uint8_t> local_input(static_cast<size_t>(dist.buffer_height) * width * 3);
  const uint8_t *full_image = (rank == 0) ? std::get<2>(GetInput()).data() : nullptr;

  // Распределенная отправка данных процессам
  ScatterWithHalo(rank, world_size, width, height, dist, full_image, local_input);

  std::vector<uint8_t> local_output(static_cast<size_t>(dist.rows_per_proc[rank]) * width * 3);

  // Локальный многопоточный расчет блоками 32x32 через TBB
  RunLocalTBB(rank, world_size, width, height, dist, local_input, local_output);

  // Консолидация данных и широковещательная рассылка
  std::vector<uint8_t> result;
  GatherAndBroadcast(world_size, width, height, dist, local_output, result);

  GetOutput() = std::move(result);
  return true;
}
```

## 5. Проверка корректности

Гибридный backend `ALL` успешно протестирован на общем наборе функциональных тестов (`tests/functional/main.cpp`).
Манипуляции с индексами в `GridSlicing` при добавлении виртуальных строк гало-зон не нарушают общую математическую
корректность алгоритма свёртки, что подтверждено успешным прохождением граничных тестов (`1pixel`, `2x1` и др.) в
многопроцессорной среде.

## 7. Результаты

В качестве рабочей нагрузки использовалась фильтрация кадра Ultra HD 8K ($7680 \times 4320$ пикселей, ~100 МБ данных).

Расчёт метрик параллельной масштабируемости производится относительно базовой точки минимального параллелизма —
конфигурации из **1 процесса и 1 потока** ($T(1 \times 1)$):

- Суммарное число исполнителей (Workers): $W = \text{PPC\_NUM\_PROC} \times \text{PPC\_NUM\_THREADS}$
- Относительное ускорение: $S(W) = T(1 \times 1) / T(W)$
- Относительная эффективность: $E(W) = S(W) / W \times 100\%$

### Режим `pipeline`

|  Конфигурация ($R \times T$)  | Всего воркеров ($W$) | Время выполнения (сек) | Ускорение $S(W)$ | Эффективность $E(W)$ |
| :---------------------------: | :------------------: | :--------------------: | :--------------: | :------------------: |
| **1 Proc $\times$ 1 Thread**  |          1           |        1.692183        |       1.00       |        100.0%        |
| **2 Proc $\times$ 2 Threads** |          4           |        0.629731        |       2.69       |        67.2%         |

### Режим `task`

|  Конфигурация ($R \times T$)  | Всего воркеров ($W$) | Время выполнения (сек) | Ускорение $S(W)$ | Эффективность $E(W)$ |
| :---------------------------: | :------------------: | :--------------------: | :--------------: | :------------------: |
| **1 Proc $\times$ 1 Thread**  |          1           |        1.162563        |       1.00       |        100.0%        |
| **2 Proc $\times$ 2 Threads** |          4           |        0.618392        |       1.88       |        47.0%         |

## 8. Выводы и анализ

1. **Анализ базовой точки ($1 \times 1$)**: В конфигурации из одного процесса и одного потока гибридный код
   демонстрирует высокое время выполнения ($1.692$ сек в pipeline и $1.163$ сек в task). Это существенно больше, чем у
   изолированных бэкендов OMP или TBB. Данный оверхед вызван необходимостью выполнения полного цикла распределения:
   сериализация метаданных в `GridSlicing`, выделение памяти под промежуточные локальные буферы, выполнение локальных
   MPI-копирований внутри `ScatterWithHalo` и `GatherAndBroadcast` (даже если ранг всего один), а также инициализация
   инфраструктуры двумерных поддиапазонов TBB.
2. **Оценка масштабируемости ($2 \times 2$)**: Переход к конфигурации из 2 процессов по 2 потока в каждом (суммарно 4
   вычислительных контекста) приводит к резкому сокращению времени выполнения. В режиме `pipeline` время падает до
   $0.629$ сек, обеспечивая ускорение в **$2.69\times$** с относительной эффективностью **$67.2\%$**. В режиме `task`
   ускорение составляет **$1.88\times$** с эффективностью **$47.0\%$**.
3. **Факторы снижения эффективности**: Падение эффективности при переходе на конфигурацию $2 \times 2$ обусловлено двумя
   факторами:
    - _Коммуникационный оверхед_: Операции `MPI_Scatterv`, `MPI_Gatherv` и в особенности `MPI_Bcast` выполняют
      физическое копирование и пересылку больших объемов данных (~100 МБ) через механизмы межпроцессного взаимодействия
      (IPC) Docker-контейнера, что сильно нагружает подсистему памяти.
    - _Конкуренция за ресурсы (Memory Bound)_: Два независимых процесса MPI одновременно порождают потоки TBB, которые
      агрессивно конкурируют за общую шину RAM и общую кэш-память L3 процессора. Тем не менее, блочная нарезка кадра на
      тайлы $32 \times 32$ частично нивелирует эти задержки, позволяя алгоритму успешно масштабироваться и сокращать
      общее время обработки кадра.
