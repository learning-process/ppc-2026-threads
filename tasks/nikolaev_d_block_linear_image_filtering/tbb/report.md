## 1. Контекст

Реализация на базе библиотеки Intel oneTBB представляет собой событийно-ориентированный параллелизм, основанный на
декомпозиции пространства задач (task-based parallelism) и алгоритме кражи задач (work-stealing). В отличие от жесткого
распределения итераций в OpenMP, TBB оперирует абстракциями диапазонов и интеллектуально распределяет нагрузку в
динамике, подстраиваясь под аппаратный контекст и минимизируя простои ядер процессора.

## 2. Постановка задачи

Вычислительное ядро производит пространственную свёртку изображения Ultra HD 8K. Основная задача TBB-реализации —
разбить плоскую двумерную сетку пикселей кадра на оптимальные подблоки (тайлы) и обработать их параллельно, предотвращая
конфликты доступа в памяти и обеспечивая максимальную локальность данных в процессорных кэшах.

## 3. Схема распараллеливания и контроль конкуренции

Параллельная обработка данных реализована с помощью высокоуровневого шаблона `tbb::parallel_for` и двумерного
итерационного пространства:

1. **Многомерный диапазон `tbb::blocked_range2d<int>`**: Задаёт итерационное пространство в виде прямоугольной матрицы
   от `0` до `height` по вертикали и от `0` до `width` по горизонтали. TBB рекурсивно делит (бисекция) этот диапазон
   пополам по обеим координатным осям до тех пор, пока полученные подблоки не станут оптимальными для обработки одним
   потоком. Это автоматически повышает пространственную локальность данных (кэш-дружественность), поскольку процессор
   обрабатывает компактные прямоугольные блоки изображения, а не длинные изолированные строки.
2. **Размер подблока (grainsize) и Partitioner**: В вызове `tbb::parallel_for` не задаются жесткие значения `grainsize`.
   Это означает, что библиотека использует автоматический разделитель по умолчанию — `tbb::auto_partitioner`. Он
   динамически оценивает накладные расходы на создание задач и текущую загрузку ядер, прекращая деление блоков, когда
   дальнейшее дробление перестает приносить выигрыш в производительности.
3. **Управление числом потоков**: Ограничение уровня параллелизма (concurrency control) контролируется тестовым
   фреймворком через переменную окружения `PPC_NUM_THREADS`. Внутри каркаса эти данные транслируются в объект управления
   ресурсами `tbb::global_control`, жестко лимитирующий максимальное количество одновременно работающих воркеров (worker
   threads) в пуле TBB.

## 4. Детали реализации

Файлы проекта: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Основной вычислительный фрагмент, инкапсулирующий вызов параллельного цикла и лямбда-функцию обработки блоков,
представлен ниже:

```cpp
// File: tbb/src/ops_tbb.cpp
bool NikolaevDBlockLinearImageFilteringTBB::RunImpl() {
  const int width = std::get<0>(GetInput());
  const int height = std::get<1>(GetInput());
  const auto &src = std::get<2>(GetInput());

  auto &dst = GetOutput();
  dst.assign(src.size(), 0);

  tbb::parallel_for(tbb::blocked_range2d<int>(0, height, 0, width), [&](const tbb::blocked_range2d<int> &r) {
    for (int ny = r.rows().begin(); ny < r.rows().end(); ++ny) {
      for (int nx = r.cols().begin(); nx < r.cols().end(); ++nx) {
        for (int ch = 0; ch < 3; ++ch) {
          dst[((ny * width + nx) * 3) + ch] = ApplyKernel(src, width, height, nx, ny, ch);
        }
      }
    }
  });

  return true;
}
```

Математическая логика свёртки вынесена в изолированный метод `ApplyKernel()`, который вызывается потоками независимо.
Поскольку запись результатов производится строго по уникальным индексам выходного вектора `dst`, соответствующим текущим
`(ny, nx)` из поддиапазона `r`, межпоточная синхронизация и барьеры внутри лямбда-выражения полностью отсутствуют,
исключая блокировки (lock-free обработка).

## 7. Результаты

В качестве входной нагрузки использовался тестовый прогон на изображении Ultra HD 8K ($7680 \times 4320$ пикселей, объем
сырых данных ~100 МБ).

В соответствии с методологией, базовой точкой отсчёта эффективности ($100\%$) принято время работы параллельного
TBB-бэкенда на одном активном потоке ($T(1)$):

- Относительное ускорение: $S(p) = T(1) / T(p)$
- Эффективность масштабирования: $E(p) = S(p) / p \times 100\%$

## 8. Выводы и анализ

1. **Анализ оверхеда на одном потоке**: При $p = 1$ время работы TBB-версии составляет $\approx 0.95$ сек, что заметно
   медленнее как последовательного кода `SEQ` ($\approx 0.49$ сек), так и однопоточного `OMP` ($\approx 0.76$ сек).
   Столь высокая накладная стоимость вызвана сложной инфраструктурой oneTBB: развертыванием планировщика задач (task
   scheduler), динамическим рекурсивным разбиением `blocked_range2d` через `auto_partitioner`, а также вызовом
   внутренней логики инкапсулированного метода `ApplyKernel` на каждом пикселе.
2. **Оценка масштабируемости**: Несмотря на стартовый оверхед, относительные показатели масштабируемости TBB
   превосходны. При переходе с 1 на 2 потока ускорение составляет $1.96\times$ (эффективность $98\%$). На 4 потоках
   эффективность удерживается выше $93\%$, что доказывает высокую сбалансированность пула задач TBB и отсутствие
   простоев ядер.
3. **Эффективность при полной загрузке (6 ядер)**: На 6 физических ядрах относительное ускорение достигает максимального
   значения $\approx 4.54\times$, а эффективность опускается до $\approx 75.5\%$. Как и в случае с OpenMP, главным
   ограничивающим фактором здесь становится шина памяти (Memory Bound). Параллельное чтение и запись в
   неструктурированный массив размером 100 МБ шестью независимыми воркерами перегружает каналы обмена данных с RAM. Тем
   не менее, за счет разбиения диапазона на двумерные суб-блоки (`blocked_range2d`) вместо одномерных строк, TBB
   эффективнее утилизирует кэш-память процессора, обеспечивая стабильное снижение времени выполнения вплоть до полной
   загрузки всех физических ядер.
