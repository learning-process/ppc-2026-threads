## 1. Контекст

Параллельная реализация с использованием стандарта OpenMP направлена на ускорение базового алгоритма фильтрации за счёт
распределения независимых итераций циклов по потокам ОС внутри одного многоядерного процессора. Так как пиксели
результирующего изображения вычисляются независимо друг от друга и запись производится в непересекающиеся области
памяти, задача обладает идеальным внутренним параллелизмом (embarrassingly parallel), что делает её классическим
кандидатом для оптимизации через директивы компилятора.

## 2. Постановка задачи

Вычислительное ядро должно параллельно обработать три цветовых канала изображения Ultra HD 8K. Стратегия
распараллеливания заключается в декомпозиции двумерного пространства сетки пикселей.

**Ключевые требования к параллельной обработке:**

- Отсутствие состояний гонки (race conditions) при обращении к исходному буферу `src` и разделяемому выходному вектору
  `dst`.
- Минимизация накладных расходов на создание и диспетчеризацию потоков.
- Обеспечение строгой инкапсуляции данных (data-sharing attributes) для предотвращения логических ошибок.

## 3. Схема распараллеливания

Распределение вычислительной нагрузки реализовано на уровне строк (`ny`) и столбцов (`nx`) матрицы изображения с
использованием явного схлопывания итерационного пространства:

1. **Директива `collapse(2)`**: Объединяет циклы по высоте (`height`) и ширине (`width`) в один плоский виртуальный цикл
   размером $W \times H$. Это увеличивает гранулярность задачи и позволяет OpenMP более равномерно нарезать итерации
   между потоками, что особенно важно при больших разрешениях кадра.
2. **Планировщик `schedule(static)`**: Распределяет сгенерированные итерации на равные блоки между потоками на этапе
   компиляции/запуска. Так как вычислительная сложность обработки каждого пикселя абсолютно идентична (ровно 9 итераций
   метода `GetPixel` на один цветовой канал), статический планировщик является наиболее энергоэффективным и исключает
   накладные расходы динамической диспетчеризации (runtime overhead).

## 4. Детали реализации

Файлы проекта: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Критический участок кода, реализующий многопоточную обработку, локализован в методе `RunImpl()`:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for collapse(2) schedule(static) default(none) shared(width, height, src, dst, kernel, sum)
for (int ny = 0; ny < height; ++ny) {
  for (int nx = 0; nx < width; ++nx) {
    for (int ch = 0; ch < 3; ++ch) {
      int acc = 0;
      for (int ky = -1; ky <= 1; ++ky) {
        for (int kx = -1; kx <= 1; ++kx) {
          acc += GetPixel(src, width, height, nx + kx, ny + ky, ch) * kernel.at(ky + 1).at(kx + 1);
        }
      }

      int res = (acc + 8) / sum;
      dst[((ny * width + nx) * 3) + ch] = static_cast<uint8_t>(std::clamp(res, 0, 255));
    }
  }
}
```

### Анализ атрибутов доступа к данным (Data-Sharing)

- **`default(none)`**: Принудительно обязывает разработчика явно указать область видимости для каждой внешней
  переменной, исключая неявное заведение разделяемых переменных и снижая риск случайных Race Conditions.
- **`shared(...)`**:
  - Переменные `width`, `height`, `sum` и объект маски `kernel` — неизменяемые константы, безопасные для совместного
      чтения.
  - Вектор `src` — исходное изображение, открытое только на чтение.
  - Вектор `dst` — выходной буфер. Потоки пишут в него параллельно, но поскольку индекс записи
      `((ny * width + nx) * 3) + ch` уникален для каждой комбинации `(ny, nx)`, потоки никогда не перезаписывают данные
      друг друга. Синхронизирующие примитивы (`critical`, `atomic`, `barrier`) не требуются.
- **Локальные переменные (Implicitly `private`)**: Переменные `ch`, `acc`, `ky`, `kx`, `res`, объявленные внутри
  параллельного блока, автоматически размещаются в персональных стеках каждого потока, предотвращая разрушение
  промежуточных результатов свёртки.

## 7. Результаты

В качестве нагрузки использовался эталонный тест производительности курса: фильтрация кадра Ultra HD 8K
($7680 \times 4320$ пикселей, ~100 МБ данных).

Расчёт метрик параллельной масштабируемости производится относительно времени работы OpenMP на одном активном потоке
($T(1)$):

- Относительное ускорение: $S(p) = T(1) / T(p)$
- Эффективность масштабирования: $E(p) = S(p) / p \times 100\%$

## 8. Выводы и анализ

1. **Анализ базовой точки ($p = 1$)**: Согласно требованиям, время выполнения на одном потоке OpenMP ($0.766$ сек в
   режиме pipeline) принято за базовую единицу с эффективностью $100\%$. Стоит отметить, что это время превышает
   показатели чистого последовательного кода `SEQ` ($\approx 0.497$ сек). Данный оверхед обусловлен инициализацией
   внутренних структур библиотеки OpenMP, созданием пула потоков runtime-окружением и трансформацией адресной арифметики
   многомерных вложенных циклов из-за применения директивы `collapse(2)`.
2. **Оценка масштабируемости**: Относительные показатели демонстрируют высокую степень масштабируемости алгоритма.
   Переход от 1 потока к 2 даёт практически линейное ускорение ($1.95\times$ для pipeline и $1.98\times$ для task), а
   эффективность удерживается на уровне $97.7\% - 99.0\%$.
3. **Падение эффективности на 6 ядрах**: При задействовании всех 6 физических ядер процессора абсолютное относительное
   ускорение составляет $\approx 4.74\times$, а эффективность масштабирования снижается до $\approx 79\%$. Данное
   падение является классическим проявлением ограничения пропускной способности памяти (Memory Bound). На 6 параллельных
   потоках вычислительные конвейеры ядер начинают конкурировать за кэш-память L3 и общую шину RAM при одновременном
   чтении и записи тяжелого массива размером 100 МБ. Физический предел скорости обмена данными с памятью препятствует
   достижению идеального линейного ускорения $6.0\times$. Расширение пула потоков далее ($p > 6$) на виртуальные
   SMT-потоки нецелесообразно, так как они усилят задержки на шине памяти без прироста производительности.
