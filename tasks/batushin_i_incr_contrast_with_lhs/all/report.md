## 1. Контекст
Данный отчет описывает гибридную реализацию алгоритма, сочетающую межпроцессное взаимодействие через MPI 
и внутрипроцессное распараллеливание через `std::thread`. Это наиболее сложная и многоуровневая версия, 
предназначенная для выполнения на кластерных системах или многоядерных узлах с распределенной памятью. 
Цель — сравнить эффективность гибридного MPI+STL подхода с эталонной последовательной версией 
(`seq/report.md`) и другими параллельными бэкендами (`omp/report.md`, `tbb/report.md`, `stl/report.md`).

## 2. Постановка задачи
Задача полностью идентична SEQ (см. `seq/report.md`): преобразование вектора пикселей 
`unsigned char` для охвата диапазона [0, 255]. Отличительная особенность версии ALL — распределение 
данных между несколькими MPI-процессами, каждый из которых может дополнительно использовать 
многопоточность STL для обработки своей локальной порции.

## 3. Базовый алгоритм
Алгоритм состоит из следующих фаз:
1.  Распределение данных между MPI-процессами (`MPI_Scatterv`).
2.  Локальный поиск минимума и максимума каждым процессом.
3.  Глобальная редукция для получения общего минимума и максимума (`MPI_Allreduce`).
4.  Параллельная нормализация локальных данных с использованием `std::thread`.
5.  Сбор результатов в корневом процессе (`MPI_Gatherv`).
6.  Широковещательная рассылка полного результата всем процессам (`MPI_Bcast`).

## 4. Схема распараллеливания
Версия ALL использует двухуровневый параллелизм: MPI для межпроцессного взаимодействия и 
STL для внутрипроцессного.

**Уровень 1: MPI (межпроцессное распараллеливание)**

**1.1. Распределение данных:**
- `MPI_Scatterv` распределяет входной вектор от корневого процесса (rank 0) всем процессам.
- Разбиение неравномерное: `base = global_size / size`, остаток распределяется по первым `remainder` процессам.
- Такой подход гарантирует, что разница в размере локальных данных между процессами не превышает 1 элемента.

**1.2. Глобальная редукция:**
- `MPI_Allreduce` с операциями `MPI_MIN` и `MPI_MAX` находит глобальные минимум и 
максимум по всем локальным значениям.
- Результат получают все процессы, что исключает необходимость дополнительной коммуникации.

**1.3. Сбор результатов:**
- `MPI_Gatherv` собирает нормализованные локальные векторы в полный выходной массив на процессе rank 0.
- Смещения (`displs`) вычисляются заранее на основе `send_counts`.

**1.4. Синхронизация результата:**
- `MPI_Bcast` рассылает полный результат от rank 0 всем процессам, чтобы выходной вектор 
был доступен на всех узлах.

**Уровень 2: STL (внутрипроцессное распараллеливание)**

**2.1. Нормализация: `NormalizeParallel`**
- Использует `std::thread::hardware_concurrency()` для определения числа потоков.
- Локальные данные каждого MPI-процесса разбиваются на равные блоки.
- Каждый поток обрабатывает свой блок через `NormalizeBlock`.
- Синхронизация через `std::thread::join()`.

**2.2. Поиск локального min/max:**
- Выполняется последовательно одним потоком в каждом MPI-процессе (`FindLocalMinMax`).
- Поскольку локальные данные после `Scatterv` относительно невелики (1/size от глобального размера), 
последовательный поиск не является узким местом.

**2.3. Случай равномерного изображения:**
- Если `global_min == global_max`, нормализация не выполняется, локальные данные копируются без изменений.
- Глобальное условие проверяется после `MPI_Allreduce`, поэтому все процессы принимают одинаковое решение.

**Обоснование архитектуры:**
- MPI выбран для возможности масштабирования на несколько узлов с распределенной памятью.
- STL выбран для внутрипроцессного параллелизма как наиболее легковесное решение 
(показавшее лучшие результаты в `stl/report.md`).
- Гибридный подход позволяет использовать все ядра на каждом узле через потоки, 
а MPI — объединять несколько узлов.

## 5. Детали реализации
**Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

**Ключевые функции:**
- `FindLocalMinMax()` — последовательный поиск min/max в локальном векторе процесса.
- `NormalizeParallel()` — многопоточная нормализация локального вектора (аналогична STL-версии).
- `NormalizeBlock()` — последовательная нормализация блока.
- `NormalizePixel()` — формула линейного контрастирования с округлением.

**Поток управления в `RunImpl()`:**
1.  Определение `rank` и `size` через `MPI_Comm_rank/size`.
2.  Вычисление `send_counts` и `displs` для неравномерного распределения.
3.  `MPI_Scatterv` — распределение входных данных.
4.  Локальный поиск min/max.
5.  `MPI_Allreduce` (MIN) — получение глобального минимума.
6.  `MPI_Allreduce` (MAX) — получение глобального максимума.
7.  Проверка `global_min == global_max`.
8.  Параллельная нормализация (или копирование для uniform случая).
9.  `MPI_Gatherv` — сбор результатов на rank 0.
10. Расширение выходного вектора на не-корневых процессах.
11. `MPI_Bcast` — рассылка результата всем процессам.

**Устранение рисков гонок:**
- На уровне MPI: каждый процесс работает со своей копией данных, гонки между процессами исключены.
- На уровне STL: каждый поток пишет в свой диапазон `local_result`, гонки между потоками исключены.
- Комбинация MPI + потоки безопасна, так как MPI-коммуникации выполняются вне параллельных регионов STL.

**Обработка граничных случаев:**
- Неравномерное распределение данных корректно обрабатывается через `remainder`.
- Последний блок в каждом процессе может быть меньше `chunk_size` (обрабатывается через `std::min`).
- Случай `global_min == global_max` обрабатывается без вызова `NormalizeParallel`.
- Все процессы получают одинаковый результат благодаря `MPI_Bcast`.

### 8.4. Анализ

**1. ALL при np=1:**
- Время 0.1426 сек — на 33% медленнее чистой STL (0.1069 сек) при том же np=1.
- Причина: накладные расходы на MPI-коммуникации (`Scatterv`, `Allreduce`×2, `Gatherv`, `Bcast`)
даже при одном процессе. Данные копируются через MPI-буферы, что добавляет latency.

**2. ALL при np=2:**
- Время 0.1075 сек — **лучший результат среди всех бэкендов** при np=2.
- ALL быстрее чистой STL (0.1298 сек) на 17%.
- ALL быстрее OMP (0.1313 сек) на 18%.
- Причина успеха: данные разделены между двумя MPI-процессами (≈33.6 МП на процесс). 
Каждый процесс работает с меньшим объёмом памяти, что улучшает локальность данных и 
эффективность кэша. MPI-коммуникаций всего 4 вызова, и при 2 процессах overhead минимален.

**3. ALL при np=4:**
- Время 0.1347 сек — хуже, чем при np=2 (замедление на 25%).
- Причины деградации:
  - **Oversubscription:** Флаг `--oversubscribe` разрешает запуск 4 MPI-процессов на одном узле. 
Каждый MPI-процесс дополнительно порождает потоки STL (`std::thread::hardware_concurrency()` 
возвращает одно и то же число для всех процессов). В сумме количество потоков значительно 
превышает число физических ядер, что приводит к конкуренции за CPU.
  - **Коммуникационный overhead:** 4 процесса требуют больше MPI-сообщений 
(каждый `Allreduce` требует log₂(4)=2 шагов обмена).
  - **Малый размер блока:** Каждый процесс обрабатывает ≈16.8 МП, и накладные расходы 
начинают доминировать над полезной работой.

**4. Поведение других бэкендов под mpirun:**
- SEQ, OMP, TBB, STL не используют MPI, поэтому каждый MPI-процесс выполняет полную 
обработку всех 67.1 МП независимо.
- При np=2 и np=4 время этих бэкендов растёт из-за конкуренции между процессами за память 
и кэш (4 процесса × 67 МП = 268 МП данных в обороте).

## 9. Выводы
Гибридная реализация MPI + STL корректно решает задачу, но показывает значительно худшую 
производительность по сравнению с чистой STL-версией (замедление в 1.99 раза) из-за накладных 
расходов на MPI-коммуникации. При одном процессе версия ALL все еще быстрее 
последовательной (ускорение 1.75×), но проигрывает всем другим параллельным бэкендам. 
Гибридный подход оправдан только при необходимости обработки изображений, не помещающихся в 
память одного узла, или при запуске на кластере с распределенной памятью. Для задач, умещающихся 
на одном узле, чистые STL или OpenMP демонстрируют значительно лучшие результаты.
