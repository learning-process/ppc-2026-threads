## 1. Контекст

Данный отчет представляет сравнительный анализ пяти реализаций алгоритма повышения контраста
полутонового изображения методом линейной растяжки гистограммы. Эталонная последовательная
версия (SEQ) задает математическую модель и критерии корректности. Параллельные версии
(OMP, TBB, STL, ALL) исследуют различные подходы к ускорению: директное распараллеливание (OpenMP),
высокоуровневые шаблоны (TBB), нативные потоки C++ (STL) и гибридное MPI+потоки решение (ALL).
Цель — определить оптимальный метод ускорения для данной задачи на одном узле и оценить
потенциал масштабирования на распределенную память.

## 2. Постановка задачи

**Входные данные:** Вектор пикселей полутонового изображения (`std::vector<unsigned char>`, значения 0–255).

**Выходные данные:** Вектор пикселей того же размера, полученный линейным растяжением гистограммы
на диапазон [0, 255].

**Формула преобразования:**
`new_val = round((old_val - min) * (255.0 / (max - min)))`

**Особые случаи:**

- Пустой вход → отклоняется на этапе валидации.
- Все пиксели одинаковы (`min == max`) → выход заполняется значением 128.

**Критерий корректности:** Отклонение от эталонного результата SEQ не более ±1 (погрешность округления).

## 3. Реализации

### 3.1. SEQ — Последовательный эталон

- **Файлы:** `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`
- **Алгоритм:** Два прохода по данным: поиск min/max, затем нормализация.
- **Сложность:** O(N) по времени, O(N) по памяти.
- **Роль:** Базовая версия, задающая эталонные значения для всех тестов и знаменатель для расчета ускорения.

### 3.2. OMP — OpenMP

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`
- **Подход:** Директивы `#pragma omp parallel for` с `reduction(min:max:)` для поиска и
`parallel for` для нормализации.
- **Особенности:** `default(none)` + явное указание `shared` для контроля видимости.
`schedule(static)` — оптимален для равномерной нагрузки.

### 3.3. TBB — Intel Threading Building Blocks

- **Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`
- **Подход:** Высокоуровневые шаблоны `tbb::parallel_reduce` (поиск min/max) и
`tbb::parallel_for` (нормализация).
- **Особенности:** Автоматическое разбиение на `blocked_range`, work-stealing для балансировки.

### 3.4. STL — std::thread

- **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`
- **Подход:** Ручное управление потоками: определение числа потоков через
`hardware_concurrency()`, разбиение данных на равные блоки, синхронизация через `join()`.
- **Особенности:** Минимальные накладные расходы, прямой доступ к системным потокам.

### 3.5. ALL — MPI + STL (гибрид)

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
- **Подход:** Межпроцессное распределение через `MPI_Scatterv`/`MPI_Gatherv`,
глобальная редукция через `MPI_Allreduce`, внутрипроцессное распараллеливание через `std::thread`.
- **Особенности:** Двухуровневый параллелизм, `MPI_Bcast` для синхронизации результата на всех процессах.

## 4. Проверка корректности

Все пять реализаций прошли полный набор функциональных тестов с идентичными результатами (отклонение ≤ ±1).
Тестирование включает 6 сценариев, каждый запускается на 5 бэкендах (SEQ, OMP, TBB, STL, ALL) — итого 30 тестов:

- `linear` — стандартный случай {50, 100, 150, 200} → {0, 85, 170, 255}
- `random` — смешанные значения {10, 200, 50, 180, 30} → {0, 255, 54, 228, 27}
- `narrow` — узкий диапазон {120, 130, 125, 128} → {0, 255, 128, 204}
- `mixed` — разнородные данные {70, 160, 90, 200, 110} → {0, 177, 40, 255, 79}
- `full` — полный диапазон {0, 64, 128, 192, 255} → без изменений
- `bw` — только крайние значения {0, 255, 0, 255} → без изменений

Для версии ALL тесты запускались под `mpirun` с np=1, np=2, np=4 — все 30 тестов пройдены во всех конфигурациях.

## 6. Результаты

### 6.1. Отдельные запуски (без mpirun)

| Бэкенд | Время (task_run), сек | Ускорение vs SEQ |
| :----- | :-------------------- | :--------------- |
| SEQ    | 0.3025                | 1.00×            |
| OMP    | 0.0958                | 3.16×            |
| TBB    | 0.2729                | 1.11×            |
| STL    | 0.0868                | 3.48×            |
| ALL    | 0.1729                | 1.75×            |

**Анализ:**

- **STL — лидер** (3.48×). Нативные потоки без прослоек runtime дают минимальный overhead.
- **OMP — близкий результат** (3.16×). Легковесные прагмы почти не уступают ручному управлению потоками.
- **ALL — посредственно** (1.75×). Накладные расходы MPI даже при np=1 значительны.
- **TBB — почти нет ускорения** (1.11×). Высокоуровневые абстракции создают
избыточный overhead для простой задачи.

### 6.2. Запуски под mpirun (единая среда)

| Бэкенд  | np=1, сек  | np=2, сек  | np=4, сек  | Лучшее ускорение |
| :------ | :--------- | :--------- | :--------- | :--------------- |
| SEQ     | 0.2243     | 0.2518     | 0.2881     | 1.00×            |
| OMP     | 0.1048     | 0.1313     | 0.3055     | 2.14×            |
| TBB     | 0.2031     | 0.2190     | 0.2867     | 1.10×            |
| STL     | 0.1069     | 0.1298     | 0.2449     | 2.10×            |
| **ALL** | **0.1426** | **0.1075** | **0.1347** | **2.09×**        |

**Анализ:**

- При np=1: STL и OMP лидируют, ALL отстает из-за MPI-overhead.
- При np=2: **ALL выходит на первое место** (0.1075 сек), обгоняя STL на 17%.
Разделение данных между двумя процессами улучшает локальность и снижает нагрузку на память.
- При np=4: все бэкенды деградируют из-за oversubscription. ALL падает сильнее всех (рост коммуникаций).

### 6.3. Масштабирование ALL

| Число процессов | Время (task_run), сек | Ускорение vs SEQ | Эффективность |
| :-------------- | :-------------------- | :--------------- | :------------ |
| np=1            | 0.1426                | 1.57×            | 157%          |
| np=2            | 0.1075                | 2.09×            | 105%          |
| np=4            | 0.1347                | 1.67×            | 42%           |

Сверхлинейная эффективность при np=1 и np=2 объясняется сравнением с SEQ под mpirun
(SEQ страдает от инфраструктуры MPI, хотя не использует её). Падение при np=4 вызвано
oversubscription: 4 процесса × потоки STL превышают число физических ядер.

## 7. Обсуждение

### 7.1. STL — максимальная производительность (3.48×)

- **Плюсы:** Максимальная производительность среди всех бэкендов. Прямой контроль над потоками
без посредников. Портабельно — требуется только стандартная библиотека C++.
- **Минусы:** Ручное управление потоками увеличивает объём кода и риск ошибок.
Нет автоматической балансировки нагрузки.

### 7.2. OMP — оптимальный баланс (3.16×)

- **Плюсы:** Минимум изменений кода (две прагмы). Хорошая производительность, близкая к STL.
Поддержка reduction из коробки.
- **Минусы:** Зависимость от OpenMP runtime. Меньший контроль над разбиением данных.

### 7.3. TBB — не для этой задачи (1.11×)

- **Плюсы:** Высокоуровневые абстракции, work-stealing для нерегулярных задач.
- **Минусы:** Избыточен для embarrassingly parallel задач с равномерной нагрузкой.
Большой overhead на создание объектов и лямбд. Разработан для сложных задач с нерегулярным
параллелизмом (графы, рекурсия, вложенные циклы). Для простого поэлементного преобразования
массива его инфраструктура (разбиение на `blocked_range`, вызовы лямбд для каждого блока,
слияние результатов в дереве редукции) создаёт overhead, сравнимый с полезной работой.

### 7.4. ALL — лидер при np=2 (2.09×)

- **Плюсы:** Единственная версия, способная масштабироваться на несколько узлов.
При np=2 обгоняет все остальные бэкенды. Меньший объём данных на процесс (33.6 МП) лучше
помещается в кэш-память, снижая количество промахов и обращений к RAM.
- **Минусы:** Сложность реализации. Накладные расходы MPI даже при np=1. Деградация при oversubscription (np=4).

### 7.5. Memory-bound природа задачи

Все версии упираются в пропускную способность памяти. Алгоритм выполняет всего несколько
арифметических операций на пиксель (вычитание, умножение, округление), поэтому узким местом
становится чтение 67 МП из RAM и запись результата. Это ограничивает масштабирование:
удвоение потоков не дает двукратного прироста.

## 8. Выводы

1. **Для одного узла:** STL (`std::thread`) — оптимальный выбор. Ускорение 3.48× при минимальных зависимостях
и максимальной портабельности. OMP — хорошая альтернатива с ускорением 3.16× и более простым кодом.

2. **TBB не рекомендуется** для embarrassingly parallel задач с равномерной нагрузкой.
Его overhead делает его медленнее даже последовательной версии в некоторых сценариях.

3. **Гибридный ALL (MPI+STL)** показывает потенциал при np=2 (лучший результат среди всех), но его преимущество
проявляется только при распределении данных. Для повседневного использования на одном узле избыточен.

4. **Memory-bound природа задачи** ограничивает масштабирование: все версии упираются в пропускную
способность памяти. Дальнейшее ускорение возможно только при переходе на многоузловые конфигурации
с независимыми каналами памяти.

5. **Практическая рекомендация:** Использовать OMP для простоты разработки и поддержки или STL для
максимальной производительности и портабельности. Гибридный ALL оправдан только при необходимости
обработки изображений, не помещающихся в память одного узла.
