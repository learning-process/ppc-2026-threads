## 1. Контекст

Данная работа является частью проекта по реализации алгоритма повышения контраста полутонового изображения.
Последовательная версия (SEQ) служит эталонной базой для проверки корректности и измерения ускорения параллельных
версий (OpenMP, TBB, STL). Здесь реализован строгий baseline, определяющий математику преобразования, обработку
граничных условий и структуру данных, которая будет использоваться во всех последующих реализациях.

## 2. Постановка задачи

**Цель:** Преобразовать входное полутоновое изображение так, чтобы его гистограмма заняла весь доступный
динамический диапазон [0, 255].

**Входные данные (`InType`):**

- Контейнер `std::vector<unsigned char>` (пиксели со значениями от 0 до 255).
- Размер вектора произвольный.
- Входные данные не должны быть пустыми (проверяется в `ValidationImpl`).

**Выходные данные (`OutType`):**

- Контейнер `std::vector<unsigned char>` того же размера, что и входной.
- Каждый пиксель вычисляется по формуле линейного контрастирования:
  `new_val = (old_val - min) * (255 / (max - min))`
- Результат округляется до ближайшего целого и ограничивается диапазоном [0, 255].

**Ограничения и крайние случаи:**

- Если все пиксели изображения имеют одно и то же значение (`min == max`),
гистограмму растянуть невозможно. В этом случае выходное изображение заполняется значением 128 (серый),
что соответствует равномерной засветке без
потери данных.

## 3. Базовый алгоритм

Алгоритм состоит из двух проходов по данным:

1. **Поиск минимального и максимального значений** яркости среди всех пикселей входного вектора.
2. **Линейное преобразование** каждого пикселя по формуле `(pixel - min) * scale_factor`,
   где `scale_factor = 255.0 / (max - min)`, с последующим округлением.

**Асимптотика по времени:** O(N), где N — количество пикселей.
Выполняется ровно два полных прохода по массиву данных.
**Асимптотика по памяти:** O(N) для хранения входного и выходного массивов.
**Инварианты:**

- Размер выходного вектора всегда равен размеру входного.
- Выходные значения всегда находятся в диапазоне [0, 255].
- Если `min == max`, выходные данные принимают значение 128.

## 4. Детали реализации

**Файлы:** `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

**Жизненный цикл задачи:**

- `ValidationImpl()`: Проверяет, что входной вектор не пуст.
- `PreProcessingImpl()`: Пустая, оставлена для совместимости с фреймворком.
- `RunImpl()`: Содержит основную логику.
    1. Вызов `FindMinMax(source)`.
    2. Если `min == max`, вызывает `FillUniformImage(destination, size)`.
    3. Иначе вычисляет `scale_coefficient`.
    4. В цикле `for` вызывает `NormalizePixel` для каждого элемента.
- `PostProcessingImpl()`: Пустая.

**Обработка граничных случаев:**

- Случай пустого входа блокируется на этапе валидации.
- Случай постоянного изображения (`min == max`) обрабатывается отдельной
веткой во избежание деления на ноль.

## 5. Проверка корректности

Корректность проверяется путём сравнения с предварительно рассчитанными ожидаемыми
значениями для каждого тестового набора. Допускается отклонение не более чем на ±1
из-за особенностей округления.

**Используемые функциональные тесты (6 сценариев):**

- `linear` (case 1): стандартный случай {50, 100, 150, 200} → {0, 85, 170, 255}
- `random` (case 2): смешанный набор {10, 200, 50, 180, 30} → {0, 255, 54, 228, 27}
- `narrow` (case 3): узкий диапазон {120, 130, 125, 128} → {0, 255, 128, 204}
- `mixed` (case 4): разнородные данные {70, 160, 90, 200, 110} → {0, 177, 40, 255, 79}
- `full` (case 5): полный диапазон {0, 64, 128, 192, 255} → без изменений
- `bw` (case 7): только крайние значения {0, 255, 0, 255} → без изменений

Все 6 тестов пройдены успешно.
