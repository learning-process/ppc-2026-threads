## 1. Контекст
Данный отчет описывает реализацию алгоритма повышения контраста с помощью библиотеки Intel TBB. 
В отличие от OpenMP, где используется директный подход к распараллеливанию, 
TBB предоставляет высокоуровневые шаблоны параллельных алгоритмов
 (`parallel_for`, `parallel_reduce`) и автоматическое разбиение
  диапазонов итераций с балансировкой нагрузки. Цель — сравнить эффективность TBB
   с эталонной последовательной версией (`seq/report.md`) и версией OpenMP (`omp/report.md`).

## 2. Постановка задачи
Задача полностью идентична SEQ (см. `seq/report.md`): 
преобразование вектора пикселей `unsigned char` для охвата диапазона [0, 255]. 
Ключевое требование — сохранение математической точности (отклонение не более ±1 от эталона SEQ) 
при параллельном выполнении.

## 3. Базовый алгоритм
Алгоритм состоит из двух фаз:
1.  Поиск глобального минимума и максимума по всем пикселям.
2.  Поэлементная нормализация: `new_val = (old_val - min) * (255.0 / (max - min))`.
В параллельной версии TBB обе фазы реализованы через высокоуровневые шаблоны библиотеки.

## 4. Схема распараллеливания
В TBB применен декларативный подход с использованием двух ключевых алгоритмических шаблонов.

**1. Поиск минимума/максимума: `parallel_reduce`**
- **Конструкция:** `tbb::parallel_reduce(tbb::blocked_range<size_t>(0, data.size()), 
  initial_value, body_lambda, join_lambda)`.
- **Механизм:** Диапазон автоматически разбивается на блоки (`blocked_range`). 
Каждый поток обрабатывает свой блок независимо, вычисляя локальную пару `(min, max)`. 
Затем результаты попарно объединяются через `join_lambda` по схеме дерева редукции.
- **Преимущество:** Отсутствие явных блокировок или критических секций. 
Вся синхронизация скрыта внутри `parallel_reduce`, что исключает риск гонок данных.

**2. Нормализация пикселей: `parallel_for`**
- **Конструкция:** `tbb::parallel_for(tbb::blocked_range<size_t>(0, source.size()), body_lambda)`.
- **Механизм:** Итерации равномерно распределяются по потокам с автоматической балансировкой.
TBB использует work-stealing: если поток завершил свой блок раньше других, он "крадет" 
часть работы у занятого потока.
- **Преимущество:** Адаптивная балансировка нагрузки без необходимости 
вручную подбирать размер блока (chunk size).

**3. Равномерная заливка: `parallel_for`**
- **Конструкция:** Аналогична нормализации, вызывается при `min == max`.
- **Назначение:** Параллельное заполнение вектора константным значением 128.

**Обоснование выбора `blocked_range`:** В отличие от OpenMP с его статическим расписанием по умолчанию, 
TBB автоматически подбирает размер блока для оптимального соотношения накладных расходов и балансировки.

**Параметры разбиения:**
- `blocked_range` используется без явного указания grainsize, 
поэтому TBB применяет автоматический partitioner (`auto_partitioner`), который сам определяет 
оптимальный размер блока на основе числа потоков и объёма данных.
- При необходимости grainsize может быть задан вручную: `tbb::blocked_range<size_t>(0, size, grainsize)`.

## 5. Детали реализации
**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

**Изменения относительно SEQ:**
- Функция `FindMinMax` переписана с использованием `tbb::parallel_reduce`.
- Основной цикл нормализации заменен на вызов `NormalizeImage`, внутри которого `tbb::parallel_for`.
- Функция `FillUniformImage` также распараллелена через `tbb::parallel_for`.

**Устранение рисков гонок:**
- В `parallel_reduce` каждый поток работает с локальной копией аккумулятора `local`, 
которая затем безопасно объединяется через `join_lambda`. Гонки исключены на уровне шаблона.
- В `parallel_for` каждый поток пишет в уникальный диапазон индексов `destination`, гонка отсутствует.

**Обработка граничных случаев:**
- Случай пустого входа блокируется в `ValidationImpl`.
- Случай `min == max` обрабатывается отдельной веткой с параллельной заливкой.

## 8. Результаты
Замеры производились для задачи размером 8192x8192 (67.1 МП).

| Реализация | Время (task_run), сек | Ускорение (Speedup) | Относительно OMP |
| :--------- | :-------------------- | :------------------ | :--------------- |
| SEQ        | 0.3025                | 1.00                | —                |
| OMP        | 0.0958                | 3.16                | 1.00×            |
| TBB        | 0.2729                | 1.11                | 0.35×            |

**Анализ:**
- **Ускорение S(TBB) = 0.3025 / 0.2729 ≈ 1.11**. Прирост относительно SEQ минимален.
- **Сравнение с OMP:** TBB показал время в 2.85 раза хуже, чем OpenMP (0.2729 / 0.0958 ≈ 2.85).
- **Вероятная причина:** Накладные расходы на создание и объединение блоков `blocked_range`, 
а также работа lambda-функций в TBB могут создавать значительный overhead при таком объеме данных. 
Версия OMP использует более легковесные прагмы, которые компилируются непосредственно в инструкции OpenMP runtime. 
Также возможно, что в TBB-версии неоптимально выбран размер блока по умолчанию для данной задачи.

**Комментарий о масштабируемости:** Низкая производительность TBB на данной задаче может быть связана с тем, 
что `parallel_reduce` создает множество промежуточных объектов `std::pair` и вызывает лямбды объединения. 
Это дополнительные накладные расходы, которые становятся заметны на memory-bound задачах, 
где время доступа к памяти доминирует над вычислениями.

## 9. Выводы
Реализация TBB корректно решает поставленную задачу, 
но показывает значительно худшую производительность по сравнению с OpenMP (ускорение 1.11× против 3.16×). 
Это объясняется более высокими накладными расходами на инфраструктуру TBB (блоки, лямбды, work-stealing), 
которые не окупаются на данной вычислительно легкой, но memory-bound задаче. 
TBB может быть предпочтительнее для более сложных алгоритмов с нерегулярным параллелизмом или 
сильной зависимостью от балансировки нагрузки, но для простого поэлементного преобразования 
массива OpenMP демонстрирует лучшие результаты.
