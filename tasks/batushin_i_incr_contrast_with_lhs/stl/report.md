## 1. Контекст
Данный отчет описывает реализацию алгоритма повышения контраста с использованием только 
стандартных средств C++ (`std::thread`). В отличие от OpenMP и TBB, которые полагаются на 
внешние библиотеки и runtime-системы, STL-версия использует "ручное" управление потоками: явное создание, 
разбиение данных на блоки и синхронизацию через `join`. 
Цель — сравнить эффективность нативных потоков C++ с эталонной последовательной версией (`seq/report.md`) 
и другими параллельными бэкендами (`omp/report.md`, `tbb/report.md`).

## 2. Постановка задачи
Задача полностью идентична SEQ (см. `seq/report.md`): преобразование вектора пикселей 
`unsigned char` для охвата диапазона [0, 255]. Ключевое требование — сохранение 
математической точности (отклонение не более ±1 от эталона SEQ) при параллельном выполнении.

## 3. Базовый алгоритм
Алгоритм состоит из двух фаз:
1.  Поиск глобального минимума и максимума по всем пикселям.
2.  Поэлементная нормализация: `new_val = (old_val - min) * (255.0 / (max - min))`.
В STL-версии обе фазы реализованы через явное создание потоков и ручное разбиение данных на блоки.

## 4. Схема распараллеливания
В STL-версии применен классический подход "разделяй и властвуй" с ручным управлением потоками.

**1. Определение числа потоков:**
- `GetNumThreads()` вызывает `std::thread::hardware_concurrency()` для получения числа аппаратных потоков.
- Если значение не определено (возвращает 0), используется 1 поток.

**2. Разбиение данных:**
- `ComputeChunkSize()` вычисляет размер блока: `data_size / num_threads`, минимум 1 элемент.
- `ComputeNumBlocks()` определяет количество блоков с округлением вверх.
- Такое равномерное разбиение минимизирует дисбаланс нагрузки между потоками.

**3. Поиск минимума/максимума: `FindMinMaxParallel`**
- Основной массив разбивается на равные блоки.
- Для каждого блока создается отдельный поток `std::thread`, который выполняет `FindMinMaxInBlock`.
- Каждый поток пишет результат в свою ячейку вектора `block_results`, 
что исключает гонки (запись в непересекающиеся области памяти).
- После завершения всех потоков (`join`) вызывается `MergeMinMaxResults` для объединения 
локальных результатов в глобальный минимум/максимум.

**4. Нормализация пикселей: `NormalizeImageParallel`**
- Данные разбиваются на блоки аналогичным образом.
- Каждый поток выполняет `NormalizeBlock` над своим диапазоном индексов.
- Каждый поток читает из `source` и пишет в `destination` по непересекающимся индексам, гонки отсутствуют.

**5. Равномерная заливка: `FillUniformImageParallel`**
- Аналогичная схема разбиения на блоки.
- Каждый поток заполняет свой диапазон константным значением 128.

**Синхронизация:**
- Единственный механизм синхронизации — `std::thread::join()`.
- Неявный барьер: главный поток ожидает завершения всех рабочих потоков перед переходом 
к следующей фазе (merge результатов или возврат из функции).

## 5. Детали реализации
**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

**Структура данных:**
- `MinMaxResult` — простая структура для хранения пары `(min_val, max_val)` с инициализаторами
по умолчанию (255 и 0 соответственно для корректного поиска минимума/максимума).
- `std::vector<MinMaxResult>` — вектор для хранения результатов каждого блока.

**Ключевые функции:**
- `GetNumThreads()` — получение числа потоков.
- `ComputeChunkSize()` — вычисление размера блока.
- `ComputeNumBlocks()` — вычисление количества блоков.
- `FindMinMaxInBlock()` — последовательный поиск min/max в заданном диапазоне.
- `MergeMinMaxResults()` — последовательное объединение результатов блоков.
- `NormalizeBlock()` — последовательная нормализация заданного диапазона.
- `FillUniformBlock()` — последовательная заливка заданного диапазона.

**Изменения относительно SEQ:**
- Полностью заменен механизм итерации: вместо одного цикла — разбиение на блоки и создание потоков.
- Добавлена инфраструктура для управления потоками (создание, join, разбиение).

**Устранение рисков гонок:**
- Каждый поток работает со своим диапазоном индексов.
- Результаты блоков пишутся в разные элементы вектора `block_results`.
- Никакие два потока не пишут в одну и ту же область памяти.
- Мьютексы и атомарные операции не требуются.

**Обработка граничных случаев:**
- Последний блок может быть меньше остальных (обрабатывается через
`std::min(start + chunk_size, data_size)`).
- Случай пустого входа блокируется в `ValidationImpl`.
- Случай одного потока корректно обрабатывается (chunk_size = data_size, num_blocks = 1).
- Случай `min == max` обрабатывается отдельной веткой.

## 8. Результаты
Замеры производились для задачи размером 8192x8192 (67.1 МП).

| Реализация | Время (task_run), сек | Ускорение (Speedup) | Относительно OMP |
| :--------- | :-------------------- | :------------------ | :--------------- |
| SEQ        | 0.3025                | 1.00                | —                |
| OMP        | 0.0958                | 3.16                | 1.00×            |
| STL        | 0.0868                | 3.48                | 1.10×            |
| TBB        | 0.2729                | 1.11                | 0.35×            |

**Анализ:**
- **Ускорение S(STL) = 0.3025 / 0.0868 ≈ 3.48**. Это лучший результат среди всех параллельных реализаций.
- **Сравнение с OMP:** STL показал время на 9.4% лучше, чем OpenMP (0.0868 / 0.0958 ≈ 0.906).
- **Сравнение с TBB:** STL быстрее TBB в 3.14 раза (0.2729 / 0.0868 ≈ 3.14).

**Причины высокой производительности STL:**
- Минимальные накладные расходы: `std::thread` напрямую отображается на потоки ОС 
без дополнительных прослоек runtime.
- Простое статическое разбиение на равные блоки идеально подходит для данной 
вычислительно однородной задачи.
- Отсутствие избыточных операций: в отличие от TBB нет создания промежуточных объектов и вложенных лямбд,
в отличие от OMP нет накладных расходов на работу runtime-библиотеки OpenMP.
- Вектор `block_results` для min/max минимален по размеру (равен числу потоков), 
что делает фазу слияния практически бесплатной.

**Комментарий о масштабируемости:** Эффективность составила 87% при 4 потоках (3.48 / 4 = 0.87). 
Это хороший показатель для memory-bound задачи. 
Дальнейшее масштабирование ограничено пропускной способностью памяти.

## 9. Выводы
Реализация на STL (`std::thread`) показала лучшую производительность 
среди всех параллельных версий (ускорение 3.48×). Это объясняется минимальными накладными 
расходами при создании и управлении потоками, а также простотой и прозрачностью кода без 
дополнительных абстракций. "Ручное" управление потоками в данном случае оказалось эффективнее 
как высокоуровневых шаблонов TBB, так и директив OpenMP. STL-версия демонстрирует, что для 
простых параллельных задач с равномерной нагрузкой нативные средства C++ могут быть оптимальным выбором.
