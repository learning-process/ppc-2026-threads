## 1. Контекст

ALL-версия является гибридной: она объединяет MPI, OpenMP, `std::thread` и oneTBB.
Ее задача не просто повторить одну из потоковых реализаций, а показать два уровня
параллелизма: распределение данных между MPI-rank-ами и внутрипроцессную обработку
локального фрагмента.

---

## 2. Постановка задачи

Нужно отсортировать `std::vector<double>` по неубыванию так, чтобы результат был
корректен и в обычном однопроцессном запуске, и при запуске через `mpiexec`.

Требования для ALL:

1. Процесс `rank 0` распределяет входные данные между процессами.
2. Каждый rank сортирует свой локальный фрагмент.
3. Процесс `rank 0` собирает локально отсортированные блоки, выполняет итоговое слияние и
   рассылает результат обратно.
4. Все rank-и получают одинаковый итоговый вектор, чтобы проверки GoogleTest были
   корректны на каждом процессе.

---

## 3. Базовый алгоритм

Локальная сортировка использует тот же radix sort для `double`, что и остальные версии:
битовое представление переводится в сортируемый ключ, после чего выполняются 8 проходов
по байтам. Слияние блоков выполнено через четно-нечетную сеть Бэтчера с padding до
степени двойки.

---

## 4. Межпроцессная схема

MPI используется как внешний уровень:

1. `MPI_Bcast` передает общий размер входа.
2. `MPI_Scatterv` распределяет части массива между rank-ами.
3. Каждый rank вызывает локальную сортировку.
4. `MPI_Gather` собирает размеры локальных результатов.
5. `MPI_Gatherv` собирает данные на rank 0.
6. Процесс `rank 0` сливает блоки.
7. `MPI_Bcast` рассылает итоговый результат всем rank-ам.

Короткий фрагмент из `all/src/ops_all.cpp`:

```cpp
MPI_Scatterv(context.rank == 0 ? input.data() : nullptr, counts.data(), displacements.data(), MPI_DOUBLE,
             local_data.data(), counts[static_cast<size_t>(context.rank)], MPI_DOUBLE, 0, MPI_COMM_WORLD);

MPI_Gatherv(local_sorted.data(), local_count, MPI_DOUBLE, context.rank == 0 ? gathered_data.data() : nullptr,
            counts.data(), context.rank == 0 ? displacements.data() : nullptr, MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

`Scatterv/Gatherv` выбраны потому, что размер входа может не делиться ровно на число
rank-ов. Массивы `counts` и `displacements` задают реальное разбиение.

---

## 5. Внутрипроцессная схема

Внутри каждого rank-а используются три потоковые технологии:

- OpenMP сортирует локальные блоки через `parallel for schedule(static)`;
- `std::thread` распределяет независимые пары блоков на раундах слияния;
- TBB ускоряет крупные compare-exchange фазы внутри сети Бэтчера.

Число внутрипроцессных рабочих потоков задается `PPC_NUM_THREADS`. Для TBB дополнительно
создается `tbb::global_control`, чтобы ограничить конкуренцию тем же значением.

## 6. Детали реализации

Основные файлы:

- `tasks/gusev_d_double_sort_even_odd_batcher_all/all/include/ops_all.hpp`
- `tasks/gusev_d_double_sort_even_odd_batcher_all/all/src/ops_all.cpp`

`GetMpiContext()` проверяет, инициализирован ли MPI. Если программа запущена без
`mpiexec`, контекст считается неактивным и ALL работает как обычная локальная гибридная
реализация. Это позволяет использовать одни и те же тесты для одиночного запуска и для
MPI-запуска.

Передача исключений из потоков выполняется через `std::exception_ptr` и `std::mutex`.
Для MPI-count-ов есть проверка `ToMpiCount`, чтобы размер не выходил за пределы `int`,
которые требуются большинству MPI-вызовов.

## 7. Проверка корректности

Функциональные тесты покрывают пустой вход, один элемент, отсортированные и обратные
массивы, нечетные размеры, повторы, экстремальные значения, перенос последнего блока и
сохранение входного снимка. Результат сравнивается с `std::ranges::sort`.

Фактический локальный запуск от 04.06.2026:

```bash
PPC_NUM_THREADS=4 PPC_NUM_PROC=1 ./build_all_verify/bin/ppc_func_tests.exe \
  --gtest_filter="*gusev_d_double_sort_even_odd_batcher_all*"
```

Результат: 23 теста из 1 набора, все пройдены.

| `PPC_NUM_PROC=1` | Случайный | task_run | 4 | 0.004872 |

## 10. Выводы

ALL-версия корректно объединяет распределенный и потоковый уровни. На локальной машине
`mpiexec -n 2` немного ускорил случайный `task_run`, но эффективность ниже, чем у чисто
потоковых схем, потому что появляются MPI-обмены и дополнительное итоговое слияние на
rank 0. Гибридная версия оправдана как демонстрация нескольких моделей параллелизма; для
практического ускорения ей нужны более крупные входы или запуск на нескольких узлах.
