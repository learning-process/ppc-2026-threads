## 1. Контекст

STL-версия использует только стандартные средства C++: `std::thread`, `std::mutex` и
`std::exception_ptr`. Эта реализация показывает ручное управление потоками без OpenMP и
без планировщика TBB. Поэтому в отчете важно фиксировать не только алгоритм, но и
жизненный цикл рабочих потоков.

---

## 2. Постановка задачи

Нужно отсортировать `std::vector<double>` по неубыванию, распределив работу между
потоками стандартной библиотеки. Итог должен совпадать с `std::ranges::sort`, а все
потоки должны быть корректно завершены через `join()`.

Дополнительные требования:

1. Не запускать лишние рабочие потоки сверх доступного аппаратного параллелизма.
2. Передавать исключения из рабочих потоков в основной поток.
3. Не допускать записи разных потоков в один и тот же блок результата.

---

## 3. Базовый алгоритм

Локальная сортировка блоков использует тот же radix sort для `double`, что и SEQ.
Слияние в STL-версии реализовано через элементы `MergeItem` с признаком padding:
размер объединяемых частей доводится до степени двойки, после чего выполняется сеть
compare-exchange четно-нечетного слияния Бэтчера.

---

## 4. Схема распараллеливания

Работа распределяется функцией `RunThreadedByIndex`: каждый worker получает индексы
`thread_index, thread_index + thread_count, ...`. Один и тот же механизм используется
для сортировки блоков и для слияния пар блоков.

Короткий фрагмент из `stl/src/ops_stl.cpp`:

```cpp
for (size_t thread_index = 0; thread_index < thread_count; ++thread_index) {
  workers.emplace_back([&, thread_index] {
    RunThreadChunk(thread_index, work_count, thread_count, function, worker_exception, exception_mutex);
  });
}

JoinWorkersAndRethrow(workers, worker_exception);
```

`join()` вызывается после запуска всех потоков, поэтому работа действительно идет
параллельно. Исключение сохраняется через `std::exception_ptr`, а доступ к нему защищен
`std::mutex`.

## 5. Детали реализации

Основные файлы:

- `tasks/gusev_d_double_sort_even_odd_batcher_stl/stl/include/ops_stl.hpp`
- `tasks/gusev_d_double_sort_even_odd_batcher_stl/stl/src/ops_stl.cpp`

`GetSafeParallelism()` ограничивает запрошенное число потоков значением
`std::thread::hardware_concurrency()`, если оно известно. Это важно для CI, где
extended-тесты могут задавать большое `PPC_NUM_THREADS`.

`MergeBatcherEvenOdd()` использует padding-элементы, которые считаются больше реальных
значений. После сети compare-exchange реальные элементы извлекаются обратно в итоговый
вектор.

## 6. Проверка корректности

Функциональные тесты проверяют пустой вход, один элемент, отсортированные и обратные
массивы, нечетные размеры, плотные повторы, экстремальные значения, вход меньше числа
потоков, нечетное число блоков, перенос последнего блока при параллельном слиянии,
повторный `Run()` до `PostProcessing()` и сохранение входного снимка.

## 9. Выводы

STL-версия дает максимальный контроль над рабочими потоками, но требует вручную решать
задачи, которые OpenMP и TBB берут на себя: разбиение индексов, обработку исключений,
завершение потоков и ограничение параллелизма. На текущем perf-наборе версия проходит
оба режима `pipeline` и `task_run`; для более строгой оценки масштабируемости нужно
отдельно расширять тест производительности так, чтобы он не фиксировал число потоков внутри
самого теста.
