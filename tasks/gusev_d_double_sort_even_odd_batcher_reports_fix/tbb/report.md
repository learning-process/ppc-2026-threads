## 1. Контекст

TBB-версия использует Intel oneTBB как среду выполнения задач. В отличие от OMP, здесь явно
задаются диапазоны работы через `blocked_range`, а планировщик TBB сам распределяет
задачи между рабочими потоками. Для данной сортировки это удобно на двух уровнях:
сортировка независимых блоков и обработка независимых compare-exchange операций.

---

## 2. Постановка задачи

Нужно отсортировать `std::vector<double>` по неубыванию, используя oneTBB для общей
памяти. Результат должен совпадать с `std::ranges::sort` при любом допустимом значении
`PPC_NUM_THREADS`.

Дополнительные требования:

1. Ограничить конкуренцию TBB значением из тестовой инфраструктуры.
2. Не создавать слишком мелкие TBB-задачи на малых массивах.
3. Сохранять корректность слияния при нечетном числе блоков.

---

## 3. Базовый алгоритм

Как и в SEQ, каждое значение `double` преобразуется в сортируемый 64-битный ключ и
сортируется LSD radix sort по 8 байтам. После этого блоки объединяются четно-нечетным
слиянием Бэтчера. Различие TBB-версии состоит в том, что части слияния и compare-exchange
могут выполняться параллельно.

---

## 4. Схема распараллеливания

TBB используется в следующих местах:

- `tbb::parallel_for` для сортировки блоков входного массива;
- `tbb::parallel_for` для обработки независимых пар блоков;
- `tbb::parallel_for` для крупных фаз compare-exchange;
- `tbb::parallel_invoke` для независимых ветвей split/merge parity-групп;
- `tbb::global_control` для ограничения числа рабочих потоков.

Короткий фрагмент из `tbb/src/ops_tbb.cpp`:

```cpp
const auto parallelism = static_cast<size_t>(std::max(1, ppc::util::GetNumThreads()));
const tbb::global_control control(tbb::global_control::max_allowed_parallelism, parallelism);

auto blocks = MakeSortedBlocks(input_data_, parallelism);
result_data_ = MergeBlocks(std::move(blocks));
```

`global_control` делает запуск воспроизводимым относительно `PPC_NUM_THREADS`. Внутри
`MakeSortedBlocks` используется `tbb::blocked_range<size_t>(0, block_count)`. Пороги
`kMinParallelElements` и `kMinParallelPairs` оставляют малые случаи последовательными,
чтобы среда выполнения не тратила время на слишком мелкие задачи.

---

## 5. Детали реализации

Основные файлы:

- `tasks/gusev_d_double_sort_even_odd_batcher_tbb/tbb/include/ops_tbb.hpp`
- `tasks/gusev_d_double_sort_even_odd_batcher_tbb/tbb/src/ops_tbb.cpp`

`RunImpl()` обрабатывает пустой вход отдельно, затем получает число потоков через
`ppc::util::GetNumThreads()`, создает `tbb::global_control`, строит отсортированные
блоки и сливает их до одного результата. При нечетном числе блоков последний блок
переносится в следующий раунд без изменения.

## 6. Проверка корректности

Функциональные тесты проверяют пустой и единичный входы, отсортированные и обратные
массивы, повторы, случайные размеры, экстремальные значения, вход меньше числа потоков,
нечетное число блоков, повторный `Run()` до `PostProcessing()` и сохранение входного
снимка.

Фактический локальный запуск от 04.06.2026:

```bash
PPC_NUM_THREADS=4 ./build_tbb_verify_only/bin/ppc_func_tests.exe \
  --gtest_filter="*gusev_d_double_sort_even_odd_batcher_tbb*"
```

Результат: 22 теста из 1 набора, все пройдены.

```bash
  --gtest_filter="*gusev_d_double_sort_even_odd_batcher_tbb*"
```

В этой ветке TBB-тесты производительности используют ручной замер стадии `Run()`: подготовка
задачи выполняется до таймера, затем измеряется основная сортировка и после нее
проверяется результат. Поэтому таблицы ниже являются `task_run`-подобными. Отдельный
`pipeline` performance-кейс в этой ветке не выделен; полный порядок вызовов pipeline
проверяется функциональными тестами.

## 9. Выводы

TBB-версия демонстрирует задачно-ориентированную модель параллелизма и контроль
конкуренции через `global_control`. Для выбранного размера входа накладные расходы TBB
оказались больше полезной параллельной работы. Версия методически полезна тем, что
показывает, где в алгоритме есть независимые диапазоны, но для ускорения нужны более
крупные входы или более грубая гранулярность задач.
