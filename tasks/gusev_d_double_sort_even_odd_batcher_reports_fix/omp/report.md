## 1. Контекст

OMP-версия переносит базовый алгоритм на модель общей памяти. OpenMP используется там,
где работа естественно раскладывается на независимые диапазоны: сортировка блоков
исходного массива и обработка пар блоков на одном раунде слияния.

---

## 2. Постановка задачи

Нужно отсортировать `std::vector<double>` по неубыванию, используя несколько OpenMP
потоков. Результат не должен зависеть от числа потоков и должен совпадать с эталоном
`std::ranges::sort`.

Дополнительные требования для OMP:

1. Блоки должны покрывать входной массив без пропусков и пересечений.
2. Запись разных потоков должна идти в разные блоки или разные элементы массива
   следующего раунда.
3. Исключение внутри OpenMP-области должно быть сохранено и повторно выброшено после
   завершения параллельного участка.

---

## 3. Базовый алгоритм

Алгоритм повторяет SEQ-версию: каждое значение `double` переводится в сортируемый
64-битный ключ, затем блок сортируется LSD radix sort по 8 байтам. После локальной
сортировки блоки объединяются четно-нечетным слиянием Бэтчера.

---

## 4. Схема распараллеливания

Входной массив разбивается на `block_count`, где число блоков ограничено
`omp_get_max_threads()` и размером входа. OpenMP применяется в двух местах:

- `MakeSortedBlocks`: параллельная сортировка независимых блоков;
- `MergeBlockPairs`: параллельное слияние независимых пар блоков одного раунда.

Короткий фрагмент из `omp/src/ops_omp.cpp`:

```cpp
#pragma omp parallel for default(none) shared(input, blocks, exception, has_exception) \
    firstprivate(block_count, signed_block_count, total_size) schedule(static) if (block_count > 1)
for (std::ptrdiff_t block = 0; block < signed_block_count; ++block) {
  ExecuteWithExceptionCapture(has_exception, exception, [&]() {
    const auto index = static_cast<size_t>(block);
    FillAndSortBlock(input, blocks[index], GetBlockRange(index, block_count, total_size));
  });
}
```

`schedule(static)` выбран из-за заранее известных границ блоков. В `shared` находятся
общие контейнеры и объект исключения, а индексы и размеры передаются как `firstprivate`.
Каждый поток пишет только в свой `blocks[index]`, поэтому гонки данных на результате
сортировки блоков отсутствуют.

## 5. Детали реализации

Основные файлы:

- `tasks/gusev_d_double_sort_even_odd_batcher_omp/omp/include/ops_omp.hpp`
- `tasks/gusev_d_double_sort_even_odd_batcher_omp/omp/src/ops_omp.cpp`

`ValidationImpl()` проверяет, что выходной буфер еще не подготовлен. `PreProcessingImpl()`
копирует вход в `input_data_`, а `PostProcessingImpl()` переносит `result_data_` в
выход задачи. В `RunImpl()` пустой вход обрабатывается отдельно; непустой вход проходит
через `MakeSortedBlocks` и `MergeBlocks`.

Для исключений используется `std::exception_ptr` и `std::atomic_bool`. Запись первого
исключения защищена OpenMP `critical`, после параллельной области вызывается
`RethrowIfCaptured`.

## 6. Проверка корректности

Функциональные тесты проверяют пустой вход, один элемент, отсортированные и обратно
отсортированные массивы, повторы, случайные входы, экстремальные значения, нечетное
число блоков, сохранение входного снимка после `PreProcessing` и корректность порядка
вызовов pipeline.

Фактический локальный запуск от 04.06.2026:

```bash
PPC_NUM_THREADS=4 ./build_omp_verify/bin/ppc_func_tests.exe \
  --gtest_filter="*gusev_d_double_sort_even_odd_batcher_omp*"
```

Результат: 21 тест из 1 набора, все пройдены.

Команда performance-запуска:

```bash
PPC_NUM_THREADS=4 ./build_omp_verify/bin/ppc_perf_tests.exe \
  --gtest_filter="*gusev_d_double_sort_even_odd_batcher_omp*"
```

В этой ветке OMP-тесты производительности оформлены как ручной замер стадии `Run()`: до
таймера выполняются `Validation()` и `PreProcessing()`, затем измеряется только
основная сортировка, после чего проверяется `PostProcessing()`. В терминологии отчета
это соответствует `task_run`-подобному режиму. Отдельный `pipeline` performance-кейс
в данной ветке не выделен, но корректность порядка вызовов полного pipeline покрыта
функциональными тестами.

## 9. Выводы

OMP-реализация корректно показывает декомпозицию по блокам и парам слияния. На небольшом
тестовом размере основная ценность версии - проверка модели распараллеливания и
синхронизации, а не максимальное ускорение. Для больших массивов ожидается лучший
баланс между полезной работой и накладными расходами OpenMP.
