## 1. Контекст

Данный отчёт описывает реализацию поразрядной сортировки с использованием OpenMP
для параллелизма на разделяемой памяти. OpenMP-версия использует битовое
преобразование (`DoubleToSortable`), которое позволяет единообразно обрабатывать
отрицательные и положительные числа, упрощая алгоритм по сравнению с
последовательной версией.

## 2. Постановка задачи

- **Входные данные:** Вектор чисел `double` произвольной длины `N`.
- **Выходные данные:** Отсортированный вектор.
- **Ограничения:** Вектор не пуст. Все значения `double` обрабатываются
  корректно благодаря битовому преобразованию.

## 3. Базовый алгоритм (последовательный)

Последовательная версия (из `seq/report.md`) обрабатывает отрицательные и
положительные числа раздельно. OpenMP-версия использует единый путь за счёт
преобразования double в sortable unsigned integer.

## 4. Схема распараллеливания

**4.1 Битовое преобразование:** Функция `DoubleToSortable` преобразует double в
64-битное целое: для положительных чисел — инвертирует знаковый бит, для
отрицательных — инвертирует все биты. Это обеспечивает правильный порядок при
целочисленной сортировке без раздельной обработки.

**4.2 Трёхфазный параллельный проход подсчёта:**

- **Фаза 1 — параллельное построение гистограмм:** OpenMP `parallel for` с
  распределением итераций `schedule(static)`. Каждый поток строит локальную
  гистограмму (256 buckets). Переменные: `shared(input, local_hist)`,
  `private(shift)`. Гонок нет, так как каждый поток пишет в свою память.

- **Фаза 2 — последовательное агрегирование префиксных сумм:** Выполняется одним
  потоком. Размер данных: `256 × num_threads`, что пренебрежимо мало.

- **Фаза 3 — параллельное рассеивание (scatter):** OpenMP `parallel for` с
  `schedule(static)`. Каждый поток использует свои указатели позиций для записи
  в выходной массив. Отсутствие гонок гарантируется тем, что потоки записывают в
  непересекающиеся диапазоны.

**4.3 Выбор schedule:** Используется `schedule(static)` во всех фазах, так как
вычислительная нагрузка на каждой итерации равномерна (обработка одного
элемента). Это исключает накладные расходы на динамическое планирование.

**4.4 Синхронизация:** Неявный барьер в конце каждой `parallel` области.
Дополнительные явные барьеры не требуются.

## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`,
  `omp/src/sorter_omp.cpp`
- **Изменения относительно SEQ:** Полностью переработан алгоритм: убрано
  разделение на `negatives/non_negatives`, добавлено битовое преобразование, все
  циклы сортировки подсчётом распараллелены через OpenMP.
- **Управление потоками:** Число потоков задаётся через `PPC_NUM_THREADS`,
  которое также экспортируется как `OMP_NUM_THREADS`.

**Пример ключевого фрагмента (параллельная гистограмма):**

```cpp
// omp/src/sorter_omp.cpp
#pragma omp parallel default(none) shared(src, dst, shift, n, local_hists) private(byte)
{
    int tid = omp_get_thread_num();
    std::vector<size_t>& local_hist = local_hists[tid];

    #pragma omp for schedule(static)
    for (size_t i = 0; i < n; ++i) {
        uint64_t val = src[i];
        byte = (val >> shift) & 0xFF;
        local_hist[byte]++;
    }
}
```

## 6. Проверка корректности

- Сравнение с последовательной версией (`seq`) на всех функциональных тестах.
  Расхождений не обнаружено.
- Дополнительная проверка: результат идентичен при `OMP_NUM_THREADS=1` и при
  запуске `seq`.
- Тесты на граничные случаи (один элемент, дубликаты, уже отсортированные
  массивы, `+inf`, `-inf`, `NaN`).
- ThreadSanitizer не обнаружил гонок.

## 7. Экспериментальная среда

- **Окружение:** Intel Core i7-1165G7, 8 аппаратных потоков, Ubuntu 22.04
  (WSL2), GCC 14.2.0, Release
- **Переменные:** `export PPC_NUM_THREADS=8` (он же `OMP_NUM_THREADS`)
- **Команда запуска:** `scripts/run_tests.py --running-type=performance`

| N           | Время (мс) | Speedup vs SEQ | Эффективность (на 8 потоках) |

| 1 000       | 4          | 4.00x          | 50.0%                        |

**Анализ:** Ускорение растёт с размером задачи и достигает 4.86x на 100M
элементов при эффективности 60.8%. Ограничивающий фактор — пропускная
способность памяти, так как каждый проход читает и записывает весь массив.
Аппроксимация: `T_omp(ms) ≈ 0.00011 × N + 3.85`.

## 9. Выводы

OpenMP-реализация обеспечивает хорошее ускорение (до 4.86x) для больших
массивов. Упрощение за счёт битового преобразования снижает сложность кода по
сравнению с SEQ. Основные ограничения: накладные расходы на динамическую
диспетчеризацию (даже со `static`) и неявные барьеры.
