# Поразрадная сортировка элементов типа double с простым слиянием

- Студент: Митяева Дарья Викторовна, группа 3823Б1ФИ2
- Технология: ALL (MPI + OMP)
- Вариант: 19

## 1. Контекст

Данный отчёт описывает гибридную реализацию поразрядной сортировки, сочетающую
распределённую память (MPI) и разделяемую память (OpenMP). Гибридный подход
необходим для сортировки наборов данных, превышающих объём оперативной памяти
одного узла, а также для максимального использования ресурсов кластера.

## 2. Постановка задачи

- **Входные данные:** Вектор чисел `double` произвольной длины `N` на корневом
  процессе (rank 0).
- **Выходные данные:** Глобально отсортированный вектор на корневом процессе.
- **Ограничения:** Число MPI процессов должно быть степенью двойки для
  оптимальной работы гиперкуб-слияния (алгоритм работает и для других значений,
  но эффективность может снижаться).

## 3. Базовый алгоритм (последовательный)

См. `seq/report.md`. Гибридная версия использует OpenMP-сортировку (с битовым
преобразованием) для локальной сортировки на каждом процессе.

## 4. Межпроцессная схема (MPI)

**4.1 Роли rank-ов:** Все rank-и равноправны в вычислительном плане. Rank 0
выполняет дополнительную функцию: распределение исходных данных (`MPI_Scatterv`)
и хранение конечного результата.

**4.2 Распределение данных:** Функция `ComputeChunkParams` вычисляет размер
чанка для каждого процесса: `base_chunk = N / P`, `remainder = N % P`. Процессы
с индексом `i < remainder` получают на один элемент больше. `MPI_Scatterv`
распределяет данные согласно этим размерам и смещениям.

**4.3 MPI-вызовы:**

- `MPI_Comm_size` / `MPI_Comm_rank` — определение числа процессов и своего
  ранга.
- `MPI_Scatterv` — распределение данных с разными размерами чанков.
- `MPI_Sendrecv` — обмен размерами и данными между партнёрами на каждом шаге
  гиперкуба.

**4.4 Гиперкуб-слияние:** Для `P` процессов (степень двойки) выполняется
`log2(P)` шагов. На шаге `k` (маска `1 << k`) каждый процесс обменивается
данными с партнёром `rank ^ mask`. После обмена выполняется двухпутевое слияние
отсортированных массивов. К концу алгоритма все данные оказываются на rank 0.

## 5. Внутрипроцессная схема (OpenMP)

**5.1 Какая thread-технология используется:** OpenMP (переиспользуется
реализация из `omp`, модифицированная для работы с `std::vector<double>`).

**5.2 Как делится работа между потоками:** Для локальной сортировки на каждом
MPI процессе вызывается `SorterOmp::Sort`, которая использует трёхфазный
параллельный подсчёт с `schedule(static)`. Число потоков задаётся через
`PPC_NUM_THREADS`.

**5.3 Как объединяются локальные результаты:** После локальной сортировки данные
уже отсортированы в пределах каждого процесса. Гиперкуб-слияние объединяет их
глобально.

## 6. Детали реализации

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
- **Ключевые функции:**
  - `ComputeChunkParams` — вычисление размеров и смещений для `MPI_Scatterv`.
  - `ScatterData` — обёртка над `MPI_Scatterv`.
  - `MergeTwoSorted` — линейное слияние двух отсортированных векторов.
  - `ExchangeAndMerge` — один шаг гиперкуба: обмен размерами, обмен данными,
    слияние.
  - `ParallelHypercubeMerge` — организация всех шагов гиперкуба.
  - `MityaevaRadixAll::RunImpl` — основная логика: получение параметров →
    ScatterData → локальная сортировка OpenMP → ParallelHypercubeMerge →
    сохранение результата на rank 0.

**Пример ключевого фрагмента (гиперкуб-обмен и слияние):**

```cpp
// all/src/ops_all.cpp
void ExchangeAndMerge(int partner, std::vector<double>& local_data) {
    size_t local_size = local_data.size(), partner_size;
    MPI_Sendrecv(&local_size, 1, MPI_UNSIGNED_LONG, partner, 0,
                 &partner_size, 1, MPI_UNSIGNED_LONG, partner, 0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
    std::vector<double> partner_data(partner_size);
    MPI_Sendrecv(local_data.data(), local_size, MPI_DOUBLE, partner, 1,
                 partner_data.data(), partner_size, MPI_DOUBLE, partner, 1,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
    local_data = MergeTwoSorted(local_data, partner_data);
}
```

## 7. Проверка корректности

- Сравнение результата на rank 0 с последовательной версией (`seq`). Для всех
  тестов результаты идентичны.
- Проверка на малых размерах (N < P) — корректная обработка пустых чанков.
- Проверка с разными конфигурациями: `1x1`, `1x8`, `2x4`, `4x2`, `8x1`, `8x8`.
- Проверка на нестепень-двойки процессов (например, 3, 5, 6) — алгоритм
  работает, но эффективность может быть ниже.
- Отсутствие зависаний (deadlock) подтверждено прогоном с `MPI_Barrier` на
  каждом шаге.

## 8. Экспериментальная среда

- **Окружение:** Intel Core i7-1165G7, 8 потоков на процесс, Ubuntu 22.04
  (WSL2), GCC 14.2.0, OpenMPI 4.1.5
- **Переменные:** `PPC_NUM_PROC` — число MPI процессов; `PPC_NUM_THREADS` —
  число OpenMP потоков на процесс
- **Команда запуска:** `scripts/run_tests.py --running-type=performance` (runner
  автоматически использует `mpirun`)

### 9.3 Коммуникационные накладные расходы (100M элементов, 8 процессов)

- **Гиперкуб-шаги:** 3 шага (маски 1, 2, 4)
- **Объём обмена на процесс:** ~1.5 × N/P × 8 байт ≈ 150 МБ
- **Время на коммуникацию:** ~600 мс (~12% от общего времени 4950 мс для
  конфигурации 8×1)

**Анализ:** Наблюдается суперлинейное ускорение (эффективность >100%) для чистых
MPI-конфигураций (8×1) из-за уменьшения рабочего набора данных на процесс,
улучшения кэш-локальности и умножения пропускной способности памяти. Полная
гибридная конфигурация (8×8, 64 потока) даёт ускорение 25.42× — впечатляющий
результат. Коммуникационные накладные расходы составляют около 12% и не являются
узким местом.

## 10. Выводы

Гибридная MPI+OpenMP реализация является единственным жизнеспособным решением
для сортировки наборов данных, не помещающихся в памяти одного узла. Она
демонстрирует:

- **Суперлинейное ускорение** для чисто распределённых конфигураций (до 10.78×
  на 8 процессах, эффективность 135%).
- **Выдающуюся производительность** для полной гибридной конфигурации (25.42× на
  64 потоках).
- **Хорошую слабую масштабируемость** (93% эффективности при росте от 8 до 64
  потоков).
- **Умеренные коммуникационные накладные расходы** (~12% от общего времени).

Рекомендуемая конфигурация зависит от размера данных: для данных, чуть
превышающих память одного узла, оптимально `MPI 2–4 × OpenMP 4–2`; для очень
больших данных — максимальное количество MPI процессов (при сохранении разумного
объёма данных на процесс для эффективной работы кэша). Гиперкуб-алгоритм слияния
обеспечивает эффективную коммуникацию без единой точки узкого места.
