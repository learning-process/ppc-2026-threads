## 1. Введение

Сортировка — фундаментальная операция в компьютерных науках. Данный проект
реализует поразрядную сортировку (LSD radix sort) для чисел двойной точности. В
работе представлены и проанализированы пять реализаций, использующих различные
модели параллелизма: последовательная (SEQ), с использованием OpenMP (OMP),
Intel TBB (TBB), стандартных потоков C++ (STL) и гибридная MPI+OpenMP (ALL).
Цель — сравнить эффективность, масштабируемость и применимость каждой технологии
для данной вычислительной задачи.

## 2. Единая постановка задачи

- **Входные данные:** Вектор чисел с плавающей точкой двойной точности
  произвольной длины `N`.
- **Выходные данные:** Тот же вектор, отсортированный в порядке неубывания.
- **Ограничения:** Входной вектор не должен быть пустым. Алгоритм должен
  корректно обрабатывать все возможные значения `double`, включая
  положительный/отрицательный ноль, бесконечности и NaN (следуя стандарту IEEE
  754).

## 3. Единая методика эксперимента

### 3.1 Окружение

- **CPU:** Intel Core i7-1165G7 @ 2.80GHz (4 ядра, 8 аппаратных потоков)
- **RAM:** 16 ГБ
- **OS:** Ubuntu 22.04 через WSL2 под Windows 10 (сборка 2H22)
- **Компилятор:** GCC 14.2.0 x86-64-linux-gnu
- **Тип сборки:** Release
- **Библиотеки:** OpenMPI 4.1.5 (для ALL), Intel TBB 2021.11.0 (для TBB)

### 3.2 Переменные окружения и команды запуска

- Для общих тестов использовался runner курса. Пример команд:

  ```bash
  cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
  cmake --build build --parallel

  export PPC_NUM_THREADS=8

  scripts/run_tests.py --running-type=threads
  scripts/run_tests.py --running-type=performance
  ```

- Для гибридной версии (ALL) управление потоками и процессами осуществлялось
  через `PPC_NUM_THREADS` и `PPC_NUM_PROC`.

### 3.3 Данные

- Генерация случайных чисел с равномерным распределением на интервале
  `[-0.5, 0.5]`. Использован фиксированный seed для воспроизводимости.

### 3.4 Размеры задач и измерение

- Проанализированы размеры: 10, 100, 1 000, 10 000, 100 000, 1 000 000, 10 000
  000, 100 000 000 элементов.
- **Методика измерения:** Для каждого размера выполнялось 5 последовательных
  запусков, фиксировалось медианное время. Прогрев перед замерами (1 запуск) не
  учитывался.
- **Ускорение (Speedup):** `Speedup = T_seq / T_par`, где `T_seq` — время
  последовательной версии.
- **Эффективность (Efficiency):** `Efficiency = Speedup / p`, где `p` —
  количество потоков (`p=1` для SEQ, 8 для OMP/TBB/STL,
  `p = proc_count * threads_per_proc` для ALL).

## 4. Сводка корректности

Корректность всех реализаций подтверждена:

- Сравнением результата с эталонной последовательной версией на наборе
  функциональных тестов, включая минимальные, граничные и случайные наборы
  данных.
- Тестами на специальные значения (положительный и отрицательный ноль,
  бесконечности, NaN).
- Визуальной проверкой с помощью `std::ranges::is_sorted`.
- Для параллельных версий — проверкой детерминированности результата (при
  фиксированном числе потоков) и отсутствием гонок (подтверждено прогоном с
  ThreadSanitizer).

## 6. Интерпретация различий

- **SEQ:** Обеспечивает надёжный базовый уровень. Сложность `O(n)`, что
  подтверждается коэффициентом корреляции 0.9996. Время хорошо аппроксимируется
  формулой `T_seq(ms) ≈ 0.0005*N + 20.25`.
- **OMP:** Хорошая масштабируемость (ускорение до 4.86x на 8 потоках).
  Эффективность 60.8% для больших данных. Уступает TBB/STL из-за накладных
  расходов на динамическую диспетчеризацию и неявные барьеры.
- **TBB:** Лучший результат среди разделяемой памяти (ускорение 6.47x,
  эффективность 80.9%). Преимущество достигнуто за счёт `static_partitioner`,
  который исключает динамическое планирование, и эффективной работы с локальными
  данными.
- **STL:** Реализация на чистом C++17 (std::thread) показала результат, почти
  идентичный TBB (отставание ~1-2%). Доказывает, что для регулярных задач можно
  достичь производительности специализированных фреймворков без внешних
  зависимостей.
- **ALL (MPI+OpenMP):** Гибридная версия демонстрирует выдающуюся
  масштабируемость на очень больших данных (ускорение 25.42x на 64 потоках).
  Наблюдается суперлинейное ускорение (эффективность >100%) для чисто
  распределённой конфигурации (8x1) из-за уменьшения рабочего набора данных на
  процесс и улучшения кэш-локальности. Слабая масштабируемость (фиксированная
  нагрузка на поток) составляет ~93% при переходе с 8 на 64 потока. Накладные
  расходы на коммуникацию (гиперкуб) составляют около 12% от времени выполнения
  для 100M элементов.

## 7. Репродуцируемость

1. **Клонирование репозитория и подмодули:**

   ```bash
   git clone <...>
   git submodule update --init --recursive --depth=1
   ```

2. **Сборка:**

   ```bash
   cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
   cmake --build build --parallel
   ```

3. **Функциональные тесты:**

   ```bash
   export PPC_NUM_THREADS=8
   scripts/run_tests.py --running-type=threads
   export PPC_NUM_PROC=8
   scripts/run_tests.py --running-type=processes
   ```

4. **Тесты производительности:**

   ```bash
   scripts/run_tests.py --running-type=performance
   ```

5. **Получение замеров:** Результаты сохраняются в директории
   `build/performance_reports/`. Сводная таблица генерируется скриптами курса.

## 8. Заключение

В рамках работы были реализованы и проанализированы пять вариантов поразрядной
сортировки. Инструментарий TBB показал максимальную эффективность для одного
узла. Реализация на стандартных потоках C++ стала лучшим выбором с точки зрения
баланса "производительность-портативность". Гибридный подход MPI+OpenMP является
единственным жизнеспособным решением для сортировки наборов данных, не
помещающихся в оперативной памяти одного узла, и демонстрирует отличную
масштабируемость. Выбор конкретной технологии зависит от размера задачи,
доступного оборудования и требований к переносимости кода.
