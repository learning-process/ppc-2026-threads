## 1. Контекст

Гибридная реализация, сочетающая технологии **MPI** и **OpenMP**, является
необходимым этапом для достижения максимальной масштабируемости алгоритма в
средах высокопроизводительных вычислений (HPC). В то время как чистый
параллелизм на общей памяти (OpenMP/TBB) ограничен ресурсами одного
вычислительного узла, а чистый MPI создает избыточные накладные расходы при
запуске отдельного процесса на каждое ядро, гибридный подход предлагает
двухуровневую иерархию управления ресурсами.

Для задачи многомерного интегрирования, обладающей идеальным параллелизмом,
выбор гибридной схемы обусловлен следующими факторами:

1. **Минимизация коммуникационных затрат**: В схеме ALL технология MPI
   используется для крупноблочного распределения итерационного пространства
   между узлами кластера. Это позволяет сократить количество участников в
   коллективных операциях (`MPI_Bcast`, `MPI_Reduce`) по сравнению с запуском
   MPI на каждом ядре, что критически важно для уменьшения времени
   синхронизации.
2. **Эффективное использование общей памяти**: Внутри каждого MPI-процесса
   технология OpenMP берет на себя «мелкозернистое» распараллеливание
   вычислений. Это позволяет потокам работать с общим объектом интегрируемой
   функции и вектором шагов сетки без необходимости их дублирования или
   пересылки по сети, что значительно экономит пропускную способность памяти.
3. **Иерархическая декомпозиция**: Линеаризация $N$-мерной сетки в одномерный
   диапазон $[0, Total)$ позволяет элегантно реализовать вложенное разделение
   нагрузки. Сначала диапазон делится на «порции» для процессов, а затем каждая
   порция динамически нарезается на «чанки» для потоков.

## 3. Базовый алгоритм

Используется итеративный метод средних прямоугольников на линеаризованной сетке.
Многомерная сетка из $\prod n_i$ элементов обходится в одном цикле. На каждой
итерации восстанавливается вектор координат центра текущего гиперпрямоугольника,
вычисляется значение функции и аккумулируется в общую сумму.

## 4. Межпроцессная схема

На верхнем уровне иерархии параллелизма используется технология MPI для
распределения вычислительной нагрузки между узлами системы. Используется
классическая парадигма **Master-Worker**.

- **Роли rank-ов:**
  - **Master (`rank == 0`)**: владеет исходными параметрами интегрирования
    (объектом `IntegrationData`), вычисляет количество итераций для каждого
    процесса с учетом остатка и выполняет финальную сборку результата.
  - **Workers (`rank > 0`)**: принимают параметры сетки, вычисляют локальные
    суммы на своих участках и отправляют их обратно мастеру.
- **Распределение данных:**
  - В данной задаче передача самих расчетных точек не требуется, так как каждый
    процесс способен вычислить их самостоятельно. Поэтому Master-процесс
    рассылает только **параметры задачи** (границы и количество шагов) всем
    остальным участникам.
  - **`MPI_Bcast`**: используется в методе `DistributeData`. Сначала
    транслируется количество размерностей (`dims`), затем — покомпонентно
    границы (`limits`) и шаги (`n_steps`). Покомпонентная рассылка (вместо
    рассылки структуры целиком) выбрана для обеспечения переносимости между
    системами с разным выравниванием памяти (alignment).
- **Декомпозиция (Partitioning):**
  - Итерационное пространство $[0, Total)$ делится на крупные блоки. Каждый
    процесс вычисляет свои переменные `my_proc_start` и `my_proc_count`.
  - Для обеспечения **Load Balancing** используется логика распределения остатка
    (`remainder`): процессы с рангом меньше остатка получают на одну итерацию
    больше. Это предотвращает простои при некратных размерах сетки.
- **Синхронизация и агрегация:**
  - **`MPI_Reduce`**: используется для сбора результатов. Каждый процесс
    (включая Master) передает свою `local_sum`, рассчитанную внутри
    OpenMP-секции, а MPI выполняет операцию `MPI_SUM`. Итоговый гиперобъем
    доступен только на корневом процессе.
  - Явные барьеры (`MPI_Barrier`) не требуются, так как коллективные операции
    `Bcast` и `Reduce` обеспечивают необходимую логическую синхронизацию между
    процессами.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса (rank) для задействования вычислительных ядер
процессора используется технология **OpenMP**. Реализована иерархическая схема
распределения нагрузки, где порция данных процесса (`my_proc_count`)
дополнительно дробится между потоками.

- **Используемая технология:** Прямое управление потоками через библиотеку
  OpenMP. Количество потоков задается программно через `omp_set_num_threads()`
  на основе значения, полученного из окружения фреймворка
  (`ppc::util::GetNumThreads()`).
- **Деление работы между потоками:**
  - В отличие от стандартного использования `#pragma omp for`, в данной
    реализации применена **ручная декомпозиция диапазона** внутри параллельной
    области.
  - Каждый поток, используя свой идентификатор `tid` и общее число потоков
    `t_count`, вычисляет границы своего поддиапазона (`my_thread_start` и
    `my_thread_count`).
  - **Балансировка нагрузки (Load Balancing):** На уровне потоков также
    реализована логика распределения остатка (`thread_remainder`). Это
    гарантирует, что даже если количество итераций процесса не кратно числу
    потоков, вычислительная нагрузка будет распределена максимально равномерно
    (разница между потоками составит не более 1 итерации).
- **Работа потоков:**
  - Каждый поток независимо вызывает метод `CalculateChunkSum`.
  - **Изоляция данных:** Для обеспечения потокобезопасности векторы координат
    (`coords`) и текущих индексов (`current_indices`) создаются локально внутри
    метода `CalculateChunkSum`. Это гарантирует, что каждый поток работает в
    своем собственном стеке и не мешает другим потокам, исключая возникновение
    состояния гонки (data race) при вычислении функции.
- **Объединение локальных результатов:**
  - Для агрегации результатов используется механизм редукции OpenMP:
    **`reduction(+ : local_sum)`**.
  - Данный подход позволяет каждому потоку накапливать результат в своей
    локальной копии переменной `local_sum`. Итоговое сложение этих копий в общую
    переменную процесса происходит один раз в момент выхода из параллельного
    региона, что исключает накладные расходы на блокировки (`mutex`) или
    атомарные операции внутри циклов.
- **Синхронизация:** Неявный барьер в конце директивы `#pragma omp parallel`
  гарантирует, что процесс перейдет к этапу межпроцессной сборки (`MPI_Reduce`)
  только тогда, когда все его внутренние потоки завершат свои вычисления.

## 6. Детали реализации

Файлы: `all/include/rect_method_all.hpp`, `all/src/rect_method_all.cpp`

В гибридной реализации ключевым аспектом является координация между
распределенной (MPI) и общей (OpenMP) памятью. Основная логика сосредоточена в
методах распределения данных и иерархической декомпозиции в `RunImpl`.

### Покомпонентная рассылка параметров (MPI)

```cpp
// File: all/src/ops_all.cpp (в методе DistributeData)
if (rank != 0) {
  local_input_.limits.resize(dims);
  local_input_.n_steps.resize(dims);
}
for (size_t i = 0; i < dims; ++i) {
  MPI_Bcast(&local_input_.limits[i].first, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Bcast(&local_input_.limits[i].second, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Bcast(&local_input_.n_steps[i], 1, MPI_INT, 0, MPI_COMM_WORLD);
}
```

_Пояснение:_ Так как в тестах фреймворка данные изначально присутствуют только
на процессе с `rank 0`, реализован метод `DistributeData`. Рассылка параметров
интегрирования выполняется через коллективную операцию **`MPI_Bcast`**.
Использован покомпонентный подход (отдельная пересылка границ и шагов для каждой
размерности) вместо передачи всей структуры. Это гарантирует корректность данных
на системах с разным выравниванием памяти и предотвращает передачу
несериализуемого объекта `std::function`, который инициализируется локально на
каждом процессе в `SetUp`.

### Иерархическая декомпозиция и редукция

```cpp
// Уровень MPI: деление итераций между процессами
size_t my_proc_count = proc_chunk + (std::cmp_less(rank, proc_remainder) ? 1 : 0);

// Уровень OpenMP: деление порции процесса между потоками
#pragma omp parallel default(none) shared(h, dims, my_proc_start, my_proc_count, func, n_steps, limits) reduction(+ : local_sum)
{
  // ... расчет локальных границ my_thread_start ...
  local_sum += CalculateChunkSum(my_thread_start, my_thread_start + my_thread_count, h, limits, n_steps, func);
}

// Глобальная агрегация
MPI_Reduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
```

_Пояснение:_ Реализован двухуровневый сбор данных. На первом уровне потоки
внутри каждого процесса суммируют свои результаты в переменную `local_sum`,
используя механизм редукции OpenMP. На втором уровне Master-процесс собирает эти
промежуточные суммы со всех узлов через **`MPI_Reduce`**. Использование
блокирующей коллективной операции `MPI_Reduce` в конце `RunImpl` служит
естественной точкой синхронизации: мастер не приступит к расчету финального
гиперобъема (`global_sum * d_v`), пока не получит данные от всех «воркеров».

### Потенциальные узкие места

В ходе анализа архитектуры были выявлены следующие факторы, ограничивающие
производительность:

1. **Нагрузка на подсистему памяти**: Каждый поток внутри вызова
   `CalculateChunkSum` динамически создает векторы `coords` и `current_indices`.
   При огромном количестве итераций и большом числе потоков это может привести к
   конкуренции за аллокатор кучи.
2. **Вычислительная интенсивность**: Основное время тратится на вызов
   `std::function`. Хотя она инкапсулирует математику, накладные расходы на
   косвенный вызов функции (indirect call) могут быть заметны на очень мелких
   сетках.
3. **Коммуникационный оверхед**: В данной задаче объем передаваемых данных мал
   (только параметры и одна итоговая сумма), однако при запуске на сотнях
   процессов время ожидания в `MPI_Reduce` может стать доминирующим, если
   вычислительная нагрузка на каждый процесс будет слишком мала (согласно закону
   Амдала).

## 7. Проверка корректности

Корректность гибридной реализации подтверждена серией функциональных тестов,
охватывающих размерности от 1D до 5D.

- **Сравнение с эталоном:** Итоговые значения, полученные версией ALL,
  сопоставлялись с результатами последовательной реализации (SEQ). Допустимое
  отклонение было установлено на уровне $\epsilon = 10^{-4}$, что позволило
  подтвердить математическую эквивалентность гибридного алгоритма
  последовательному и чисто многопоточным версиям (OMP, TBB, STL).
- **Согласованность между рангами:** В методе `CheckTestOutputData` реализована
  специализированная логика проверки для MPI-среды. Сравнение фактического
  результата с ожидаемым выполняется **только на корневом процессе** (`rank ==
  0`), так как именно он владеет финальным результатом после операции
  `MPI_Reduce`. Рабочие процессы (`rank != 0`) автоматически возвращают `true`,
  что предотвращает ложные срабатывания тестов на узлах, не участвующих в
  финальной записи результата.
- **Устойчивость к конфигурации:** Тесты успешно пройдены на различных
  конфигурациях $Ranks \times Threads$. Использование `MPI_Bcast` для трансляции
  параметров сетки (`limits`, `n_steps`) гарантирует, что все процессы работают
  в едином вычислительном пространстве, обеспечивая детерминированность
  результата независимо от количества задействованных узлов.
- **Крайние случаи:** Алгоритм корректно обрабатывает специфические сценарии:
  - **Малые размеры задачи:** Благодаря иерархическому распределению нагрузки с
    учетом остатка (`remainder`), код сохраняет работоспособность даже в
    случаях, когда число итераций $n$ меньше суммарного количества ядер или
    процессов.
  - **Интервалы нулевой длины:** Валидация и вычислительное ядро успешно
    обрабатывают случаи $a = b$, возвращая нулевой гиперобъем без аварийного
    завершения.
- **Изоляция и защита:** Применение `ppc::util::IsUnderMpirun()` в методах
  валидации и проверки исключает некорректные вызовы MPI-функций при запуске
  тестов в однопроцессном (поточном) режиме, что гарантирует стабильность
  CI-конвейера на всех этапах проверки.

   вычислительной мощностью и затратами на коммуникацию.
3. **Преимущество гибридности над «чистыми» методами:** Сравнение конфигураций с
   общим числом воркеров равным 6 (2x3, 3x2, 6x1) наглядно показывает
   преимущество гибридной схемы. Конфигурация **3x2** оказалась быстрее «чистого
   MPI» (6x1) на **25%** и быстрее схемы с преобладанием потоков (2x3) на
   **26%**. Это доказывает, что иерархическое разделение (3 процесса по 2
   потока) лучше сохраняет локальность данных и снижает нагрузку на планировщик
   OpenMP по сравнению с «плоскими» моделями.
4. **Деградация в режиме SMT:** При использовании 12 воркеров наиболее
   эффективно показала себя схема **3x4**. Запуск 6 потоков внутри всего 2
   процессов (2x6) привел к катастрофическому падению эффективности до 23.6%,
   что подтверждает теорию о том, что большое количество потоков в одном
   MPI-процессе при выполнении математически интенсивных задач (FPU-bound)
   создает критические задержки при доступе к общим ресурсам ядра.

## 10. Выводы

Реализация гибридной схемы (MPI + OpenMP) для задачи многомерного интегрирования
позволила достичь наилучших показателей производительности, эффективно объединив
распределенную и общую модели памяти.

Гибридная схема **оправдана**:

- **При высокой вычислительной интенсивности**: Благодаря большому числу
  итераций ($30^5$), полезная нагрузка на ядра процессора значительно превышает
  затраты на MPI-коммуникации. Это позволило гибридной версии стать абсолютным
  лидером по скорости (658.12 мс), обогнав чисто многопоточные реализации
  (OMP/TBB/STL) более чем в 2 раза при использовании того же количества
  физических ядер.
- **Для достижения оптимального баланса ресурсов**: Эксперимент показал, что
  конфигурация **3 процесса × 2 потока** эффективнее «плоских» моделей.
  Разделение на 3 процесса снижает накладные расходы OpenMP на управление
  большим пулом потоков, а использование 2 потоков внутри каждого процесса
  позволяет им эффективно делить кэш-память L1/L2, не создавая избыточных
  очередей к блокам плавающей точки (FPU).
- **При необходимости масштабирования на кластеры**: Архитектура ALL позволяет
  выйти за пределы одного системного блока, используя MPI для связи между узлами
  и OpenMP для параллелизма внутри многоядерных процессоров каждого узла.

Гибридная схема **не оправдана**:

- **При неверном выборе пропорции Ranks/Threads**: Замеры показали, что
  конфигурация с избыточным числом потоков внутри процесса (**2 ранга × 6
  потоков**) приводит к деградации производительности (956.98 мс). В задачах,
  сильно нагружающих математические сопроцессоры (как вычисление синусов),
  конкуренция большого числа потоков за ресурсы одного процесса становится
  «бутылочным горлышком».
- **На малых объемах сетки**: Если общее число итераций невелико, время на
  инициализацию MPI-среды и покомпонентную рассылку параметров через `MPI_Bcast`
  сделает гибридную версию медленнее простой последовательной реализации.
- **При дефиците физических ядер**: В условиях данной задачи использование SMT
  (12 воркеров на 6 ядрах) не дает линейного прироста, так как вычислительные
  блоки процессора уже максимально загружены основными потоками.
