## 1. Введение

В данной работе проводится сравнительный анализ эффективности пяти парадигм
параллельного программирования (последовательная версия, OpenMP, oneTBB, STL
Threads и гибридная модель MPI+OMP) на примере алгоритма многомерного
интегрирования методом средних прямоугольников. Данная задача идеально подходит
для сравнительного анализа, так как обладает «идеальным параллелизмом»:
вычисление значения функции в каждой ячейке $N$-мерной сетки полностью
независимо от других. Это позволяет исследовать чистую масштабируемость каждой
технологии, оценить накладные расходы различных рантайм-систем на управление
потоками и проверить эффективность механизмов редукции (сборки итоговой суммы) в
условиях высокой вычислительной интенсивности.

## 2. Единая постановка задачи

- **Вход:**
  - Структура `IntegrationData`, включающая:
    - Вектор пар вещественных чисел `limits`, задающий границы интегрирования
      $[a_i, b_i]$ для каждой размерности.
    - Вектор целых чисел `n_steps`, определяющий количество шагов разбиения по
      каждой оси.
    - Объект `std::function`, представляющий подынтегральную функцию $f(x_1,
      \dots, x_N)$.
- **Выход:** Число типа `double` — вычисленное значение гиперобъема под
  поверхностью функции на заданном гиперкубе.
- **Ограничения:**
  - Алгоритм должен поддерживать произвольную размерность пространства $N \ge
    1$.
  - Количество шагов по каждой размерности должно быть строго положительным
    ($n_i > 0$).
  - Реализация обязана обеспечивать автоматическую балансировку нагрузки (Load
    Balancing) в случаях, когда общее число элементарных ячеек сетки не кратно
    количеству задействованных потоков или процессов.
- **Критерий корректности:** Абсолютное расхождение между результатом
  параллельных реализаций и аналитическим эталоном (либо результатом
  последовательной версии) не должно превышать установленную погрешность
  $\epsilon = 10^{-4}$.

## 3. Единая методика эксперимента

**Окружение:**

- **CPU:** AMD Ryzen 5 7500F, 6 ядер, 12 потоков
- **RAM:** 32 GB
- **OS:** Ubuntu 24.04.4 LTS (Docker)
- **Compiler:** GCC 13.3.0
- **Build type:** Release

**Управление экспериментом:**

- **Конфигурация параллелизма:**
  - Число рабочих потоков для технологий OMP, TBB, STL и внутрипроцессного
    уровня ALL задавалось через переменную окружения **`PPC_NUM_THREADS`**. Это
    значение считывалось фреймворком и использовалось для настройки пула
    потоков.
  - Количество процессов для гибридной реализации (ALL) задавалось
    непосредственно при запуске через параметр **`-n` утилиты `mpirun`**.
  - Итоговая вычислительная мощность для гибридной версии рассчитывалась как
    произведение параметров, заданных в `mpirun -n` и `PPC_NUM_THREADS`.
- **Генерация данных:** В тестах производительности использовалась фиксированная
  5D-функция $\sum_{i=1}^{5} \sin(x_i)$. Границы интегрирования для всех
  измерений были установлены в диапазоне $[0, 1]$.
- **Размер задачи:** Сравнение технологий проводилось на сетке с количеством
  шагов **30** по каждой из 5 осей. Общее число элементарных итераций составило
  $30^5 = \mathbf{24\,300\,000}$. Данный объем нагрузки обеспечивает время
  выполнения более 0.5 секунд, что позволяет получить статистически значимые
  результаты.

**Расчет метрик:**

- **Базовое время ($T_1$):** За эталон (baseline) принято медианное значение
  времени работы в режиме `task_run` последовательной версии (SEQ).
- **Ускорение ($S$):** Рассчитывается по формуле $S = T_1 / T_p$, где $T_p$ —
  медианное время выполнения параллельной реализации.
- **Эффективность ($E$):** Рассчитывается по формуле $E = (S / P) \times 100\%$.
- **Рабочая единица ($P$):**
  - Для технологий OMP, TBB, STL — число задействованных потоков.
  - Для гибридной технологии ALL — общее число вычислителей ($P = \text{ranks}
    \times \text{threads}$).

**Агрегация результатов:** Для каждой технологической конфигурации проводилось
10 повторных запусков. В итоговые таблицы заносилось **медианное значение**
времени выполнения. Перед основными замерами выполнялся один «прогревочный»
прогон для инициализации ресурсов рантайм-библиотек (особенно важно для TBB и
MPI). Дополнительные средства стабилизации (вроде фиксации частоты CPU) не
применялись для оценки работы алгоритмов в стандартных условиях планировщика ОС.

## 4. Сводка корректности

Корректность всех параллельных backend-ов (`omp`, `tbb`, `stl`, `all`)
проверялась путем автоматизированного сравнения полученных результатов с
эталонной последовательной версией (`seq`).

- **Методика:** Функциональные тесты запускались через единый фреймворк Google
  Test. Сравнение результатов вычислений выполнялось с абсолютной погрешностью
  $\epsilon = 10^{-4}$.
- **Тестовое покрытие:** Все технологии детерминированно прошли тесты в
  пространствах от 1D до 5D, включая проверку краевых случаев (интервалы
  интегрирования нулевой длины, сетки из одного элемента).
- **Ограничения:** Разработанные реализации универсальны и не имеют
  алгоритмических ограничений применимости. Единственным инфраструктурным
  требованием является наличие менеджера процессов MPI для запуска гибридной
  версии (ALL).

## 6. Интерпретация различий

Анализ сводных данных позволяет сделать выводы о пригодности различных моделей
параллелизма для задачи многомерного интегрирования.

1. **SEQ (Базовый уровень):** Последовательная версия фиксирует «чистую»
   стоимость вычислений на одном ядре. Высокое время выполнения (2850.57 мс)
   подтверждает экспоненциальную сложность алгоритма.

2. **OMP (Простота и стабильность):** OpenMP показал хорошую масштабируемость,
   достигнув ускорения **3.73x** на 6 физических ядрах. Сильной стороной
   является минимальное количество кода, однако слабой стороной в данной задаче
   выступает фиксированный runtime-оверхед на управление пулом потоков.
   Поскольку работа в каждой ячейке сетки одинакова, статическое расписание OMP
   работает эффективно, но не дает такого гибкого управления, как TBB.

3. **TBB (Динамическое планирование):** Библиотека Intel TBB продемонстрировала
   высокую эффективность (**79.9%** на 6 воркерах). Основную роль здесь сыграл
   автоматический выбор `grainsize` и работа `auto_partitioner`. Рекурсивное
   деление диапазона `blocked_range` позволило TBB лучше адаптироваться к
   кэш-памяти процессора. TBB оказался эффективнее OMP, что доказывает
   превосходство динамического планировщика (task-stealing) даже в задачах с
   равномерной нагрузкой.

4. **STL (Цена ручного управления):** Ручная реализация на `std::thread` стала
   **лидером по скорости** среди чисто многопоточных версий на 6 ядрах
   (ускорение **4.96x**). «Цена» ручного управления в данном случае оказалась
   отрицательной: отсутствие сложного рантайма (как в TBB) и максимально простая
   логика распределения остатка (`remainder`) позволили потокам работать с
   минимальными задержками. Это доказывает, что для идеально сбалансированных
   задач ручное деление диапазонов является наиболее эффективным.

5. **ALL (Иерархический выигрыш):** Гибридная версия стала абсолютным чемпионом
   по абсолютному времени (464.67 мс на 12 воркерах). При сравнении конфигураций
   на 6 физических ядрах (3x2, 2x3, 6x1) лидером стала схема **3 процесса × 2
   потока (ALL 3x2)**. Она обошла «чистый» MPI (6x1) на 25%. Это подтверждает,
   что иерархическое разделение (MPI для крупных блоков, OMP для вычислений
   внутри) обеспечивает лучшую локальность данных в L1/L2 кэшах и снижает
   конкуренцию за шину памяти.

6. **Аппаратные ограничения (SMT):** В данных видно понижение эффективности при
   переходе от 6 к 12 воркерам. Это обусловлено архитектурой AMD Ryzen 5 7500F:
   после заполнения 6 физических ядер в работу вступают логические потоки (SMT).
   Так как вычисление интеграла ($\sin, \cos$) сильно нагружает блоки плавающей
   точки (FPU), два потока на одном ядре начинают конфликтовать за одни и те же
   математические блоки, что приводит к падению эффективности почти в 2 раза.

## 7. Репродуцируемость

Все замеры проводились в изолированной среде Docker. Для воспроизведения
результатов необходимо собрать проект и использовать соответствующие команды
запуска для каждой технологии.

**Сборка проекта:**

```bash
# Конфигурация Release-версии
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
# Компиляция
cmake --build build --parallel
```

**Запуск функциональных тестов:** Для проверки корректности потоковых версий
используется прямой запуск. Для гибридной версии используется менеджер процессов
MPI.

```bash
# Для SEQ, OMP, TBB, STL
PPC_NUM_THREADS=4 ./build/bin/ppc_func_tests --gtest_filter="MultidimensionalIntegration/*"

# Для гибридной технологии ALL
PPC_NUM_THREADS=2 mpirun -n 2 --allow-run-as-root ./build/bin/ppc_func_tests --gtest_filter="MultidimensionalIntegration/*"
```

**Получение замеров производительности:** Для исключения накладных расходов от
MPI-рантайма замеры потоковых технологий проводились без использования `mpirun`.

```bash
# 1. Замеры для потоковых технологий (SEQ, OMP, TBB, STL)
# Пример запуска на 6 потоках:
PPC_NUM_THREADS=6 ./build/bin/ppc_perf_tests --gtest_filter="MultidimensionalIntegrationPerf/*"

# 2. Замеры для гибридной технологии (ALL)
# Пример для конфигурации 3 процесса по 2 потока (всего 6 рабочих единиц):
PPC_NUM_THREADS=2 mpirun -n 3 --allow-run-as-root ./build/bin/ppc_perf_tests --gtest_filter="MultidimensionalIntegrationPerf/*all_enabled*"

# 3. Замеры в режиме SMT (Hyper-Threading)
# Запуск на 12 логических потоках:
PPC_NUM_THREADS=12 ./build/bin/ppc_perf_tests --gtest_filter="MultidimensionalIntegrationPerf/*"
```

## 8. Заключение

Наилучшую абсолютную скорость выполнения показала гибридная реализация (**ALL**)
в конфигурации **3 процесса × 2 потока**, что подтверждает эффективность
иерархического параллелизма при полной утилизации физических ядер процессора.
Среди чисто многопоточных технологий лидером стала **STL-версия**,
продемонстрировав минимальные накладные расходы на управление потоками в задаче
с идеально сбалансированной нагрузкой. Сравнительный анализ ограничен спецификой
алгоритма: вычисление тригонометрических функций сильно нагружает блоки
плавающей точки (**FPU**), что делает использование логических потоков (SMT/12
воркеров) малоэффективным из-за аппаратной конкуренции за общие вычислительные
модули ядра. Дальнейшее улучшение производительности возможно за счёт внедрения
векторных инструкций (**SIMD**) для вычисления значений функции в блоке точек, а
также перехода к адаптивным сеткам интегрирования, позволяющим снизить общее
количество итераций без потери точности.
