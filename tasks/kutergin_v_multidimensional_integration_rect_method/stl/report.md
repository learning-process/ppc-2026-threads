## 1. Контекст

Реализация алгоритма многомерного интегрирования на базе стандартной библиотеки
потоков C++ (**STL Threads**) является важным этапом исследования параллельных
технологий. В то время как OpenMP и TBB предоставляют высокоуровневые абстракции
и автоматизированное управление ресурсами, использование примитивов
`std::thread` требует от разработчика самостоятельного проектирования
архитектуры параллельного исполнения.

Для данной задачи, обладающей **идеальным параллелизмом** (независимость
вычислений в каждой ячейке сетки), ручная реализация на потоках STL обоснована
следующими факторами:

1. **Минимальный уровень абстракции (Low Overhead)**: В отличие от TBB, который
   содержит сложный планировщик задач (runtime), или OpenMP, полагающегося на
   магию компилятора, `std::thread` является прямой оберткой над системными
   вызовами ОС (например, pthreads в Linux). Это позволяет минимизировать
   накладные расходы на управление параллелизмом и замерить «чистую»
   производительность алгоритма.
2. **Явная реализация паттерна Fork-Join**: Данная версия служит для
   демонстрации фундаментальных механизмов многопоточности. Программист обязан
   вручную реализовать декомпозицию данных, управлять жизненным циклом каждого
   «воркера» (создание и ожидание через `join`) и проектировать схему агрегации
   результатов без использования автоматизированных редукций.
3. **Ручная балансировка нагрузки (Load Balancing)**: Поскольку итерационное
   пространство линеаризовано в одномерный диапазон $[0, Total)$, задача требует
   применения математических формул для честного разделения всех итераций между
   потоками. Это позволяет реализовать наиболее эффективную для данной задачи
   статическую схему распределения, учитывающую остаток от деления и
   гарантирующую отсутствие простаивающих ядер.
4. **Портируемость и стандарт**: Реализация на STL гарантирует работу кода на
   любой платформе, поддерживающей стандарт C++11 и выше, без необходимости
   установки дополнительных библиотек или специфических компиляторов.

## 3. Базовый алгоритм

Используется итеративный метод средних прямоугольников на линеаризованной сетке.
Многомерная сетка из $\prod n_i$ элементов обходится в одном цикле. На каждой
итерации восстанавливается вектор координат центра текущего гиперпрямоугольника,
вычисляется значение функции и аккумулируется в общую сумму.

## 4. Схема распараллеливания

Параллелизм в данной реализации основан на ручном управлении жизненным циклом
потоков стандартной библиотеки C++ (`std::thread`) и применении паттерна
**Fork-Join**.

**Анализ механизмов и декомпозиции:**

- **Разбиение данных (Range Partitioning):** Как и в предыдущих версиях,
  $N$-мерное итерационное пространство линеаризуется в одномерный диапазон $[0,
  Total)$. Этот диапазон делится на $P$ частей (где $P$ — количество потоков из
  `ppc::util::GetNumThreads()`) вручную.
- **Балансировка нагрузки (Load Balancing):** Реализована статическая схема
  распределения итераций с учётом остатка. Для этого вычисляется базовый размер
  блока `chunk = Total / P` и остаток `remainder = Total % P`. Первые
  `remainder` потоков получают на одну итерацию больше. Границы вычисляются по
  формулам:
  - `start = i * chunk + min(i, remainder)`
  - `end = start + chunk + (i < remainder ? 1 : 0)` Это гарантирует отсутствие
    «дыр» в вычислениях и максимально равномерную загрузку ядер.
- **Задачи потоков:** Каждый поток запускается с лямбда-выражением, которое
  вызывает метод `CalculateChunkSum` для назначенного ему диапазона. Внутри
  этого метода каждый поток создает свой собственный вектор координат `coords` и
  вектор индексов `current_indices`. Эти структуры данных располагаются в стеке
  потока, что обеспечивает **полную изоляцию данных** и отсутствие необходимости
  в синхронизации (mutex/atomic) на этапе вычислений.
- **Хранение локальных результатов:** Для сбора данных используется
  промежуточный вектор `std::vector<double> partial_sums` размером $P$. Каждый
  поток имеет исключительное право записи в свою ячейку `partial_sums[i]`. Такой
  подход («массив частичных результатов») исключает состояние гонки и
  минимизирует вероятность возникновения эффекта **false sharing**, так как
  запись происходит редко (один раз в конце работы потока).
- **Схема синхронизации и Join:** Программа реализует строгий барьер между фазой
  вычислений и фазой агрегации. Сначала запускается цикл создания всех объектов
  `std::thread` (фаза **Fork**). После того как все потоки запущены, выполняется
  второй цикл, вызывающий метод **`join()`** для каждого потока (фаза **Join**).
  Разделение этих циклов критически важно: если вызывать `join()` сразу после
  создания потока, выполнение станет последовательным.
- **Финальная агрегация:** После того как все вызовы `join()` завершены (все
  потоки гарантированно закончили работу), главный поток выполняет
  последовательное суммирование значений из вектора `partial_sums` с помощью
  алгоритма `std::accumulate`.

## 5. Детали реализации

Файлы: `stl/include/rect_method_stl.hpp`, `stl/src/rect_method_stl.cpp`

Реализация на базе `std::thread` требует наиболее детального описания механизмов
управления памятью и жизненным циклом потоков, так как в ней отсутствуют
встроенные средства автоматического распределения и редукции данных.

### Ручное распределение диапазона и создание потоков

```cpp
// Фрагмент из RunImpl
for (int i = 0; i < num_threads; ++i) {
  size_t start = (i * chunk) + std::min(static_cast<size_t>(i), remainder);
  size_t end = start + chunk + (std::cmp_less(static_cast<size_t>(i), remainder) ? 1 : 0);

  if (start < end) {
    threads.emplace_back(
        [this, start, end, &h, &partial_sums, i]() { 
            partial_sums[i] = CalculateChunkSum(start, end, h); 
        });
  }
}
```

_Пояснение:_ В цикле создаются `num_threads` объектов `std::thread`. Карта
**поток -> диапазон** реализуется через вычисление уникальных индексов `start` и
`end` для каждой итерации цикла. Критически важным является способ захвата
переменных в лямбда-выражении:

- **Захват по значению (`i`, `start`, `end`)**: Индексы копируются в контекст
  потока. Если бы `i` была захвачена по ссылке, к моменту старта потока значение
  `i` могло бы измениться в основном цикле, что привело бы к обращению к
  неверному диапазону или выходу за границы вектора `partial_sums`.
- **Захват по ссылке (`&h`, `&partial_sums`)**: Массивы данных и результатов
  разделяются между всеми потоками без копирования для экономии памяти и
  времени.

### Схема объединения локальных результатов

```cpp
// Ожидание завершения работы
for (auto &t : threads) {
  if (t.joinable()) t.join();
}

// Финальная агрегация
double total_sum = std::accumulate(partial_sums.begin(), partial_sums.end(), 0.0);
```

_Пояснение:_ Схема объединения построена по принципу **неблокирующего
накопления**. Вместо использования мьютекса внутри вычислительной функции (что
привело бы к огромным очередям при 24 млн итераций), каждый поток записывает
свой итог в индивидуальный слот массива `partial_sums[i]`. Синхронизация
обеспечивается барьером в виде цикла `join()`. Только после того как все потоки
завершили запись, главный поток безопасно суммирует промежуточные значения
методом `std::accumulate`.

## 6. Проверка корректности

Корректность реализации на базе `std::thread` подтверждена сравнением с
результатами последовательного эталона (SEQ) во всех функциональных тестах
(1D–5D). Полученные значения совпадают в пределах заданной погрешности
$10^{-4}$. Проверка при различном количестве потоков (от 1 до 12) показала
полную детерминированность результатов. Отсутствие гонок данных (data race)
гарантируется архитектурой решения: каждый поток работает в строго изолированном
диапазоне итераций и записывает результат в персональную ячейку вектора
`partial_sums`. Дополнительная изоляция обеспечена локализацией векторов
координат и индексов внутри стека каждого потока.

_Примечание к результатам:_

1. **Минимальные накладные расходы на малых конфигурациях:** На одном потоке
   STL-версия показала результат (2839.64 мс), максимально близкий к
   последовательному эталону. Это подтверждает гипотезу о том, что `std::thread`
   является максимально «легкой» оберткой над системными вызовами ОС, не
   вносящей существенных задержек при отсутствии реального параллелизма.
2. **Масштабируемость на физических ядрах:** Наблюдается стабильный рост
   производительности при увеличении числа потоков до 6. Особый интерес вызывает
   рост эффективности при переходе с 4 на 6 потоков ($72.8\% \rightarrow
   78.7\%$). Это свидетельствует о том, что ручное деление итерационного
   пространства на крупные чанки (Block Partitioning) позволяет потокам
   эффективно использовать кэш-память L3 процессора Ryzen 5 7500F, минимизируя
   простои при чтении общих данных (объекта функции).
3. **Влияние создания и завершения потоков:** В отличие от OpenMP, где потоки
   переиспользуются из пула, здесь каждый запуск создавал потоки «с нуля».
   Однако для длительной задачи (~0.5–2.8 сек) время системных вызовов
   `clone/join` оказалось ничтожным и не повлияло на общую динамику ускорения.
4. **Лимит SMT (12 потоков):** При задействовании логических ядер ускорение
   практически остановилось ($574.87 \rightarrow 548.18$ мс). Поскольку
   вычисление многомерного интеграла — это задача с интенсивным использованием
   блоков плавающей точки (FPU), два потока на одном физическом ядре начинают
   конкурировать за одни и те же математические модули, что делает дальнейшее
   масштабирование неэффективным.

## 9. Выводы

Ручная реализация через примитивы **`std::thread`** продемонстрировала, что
алгоритм многомерного интегрирования обладает высокой степенью параллелизма и
эффективно масштабируется на физических ядрах процессора. Отсутствие
высокоуровневых абстракций планирования (как в TBB) позволило минимизировать
накладные расходы, что подтверждается минимальной задержкой исполнения в
однопоточном режиме относительно последовательного эталона.

Эксперимент подтвердил, что при использовании паттерна **Fork-Join** и
корректном ручном расчете диапазонов (**Load Balancing**) можно добиться
производительности, сопоставимой с промышленными фреймворками (OMP/TBB). Однако
данный подход является наиболее трудоёмким, так как требует от разработчика
самостоятельной реализации механизмов распределения нагрузки, управления
жизненным циклом «воркеров» и проектирования бесконфликтной схемы агрегации
результатов. Результаты на 12 потоках наглядно показали, что для математически
интенсивных задач (FPU-bound) лимитирующим фактором становится не способ
управления потоками, а архитектурные ограничения аппаратных ресурсов ядра.
