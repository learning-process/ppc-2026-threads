## 1. Контекст

Перенос алгоритма из последовательной версии (`SEQ`) в OpenMP необходим для
эффективного использования вычислительных ресурсов современных многоядерных
систем. Численное интегрирование в многомерных пространствах сталкивается с
проблемой экспоненциального роста итераций при увеличении числа измерений, что
делает последовательный расчет на больших сетках практически невозможным.

Поскольку метод средних прямоугольников является задачей с **идеальным
параллелизмом** (вычисление значения функции в конкретной ячейке сетки полностью
независимо от других), основной задачей при переходе к OpenMP становится выбор
оптимальной схемы декомпозиции. Вместо использования вложенных параллельных
циклов по каждой размерности, которые создали бы избыточную нагрузку на
планировщик потоков, в данной реализации сохранена стратегия **линеаризации
итерационного пространства**. Всё $N$-мерное пространство «сплющивается» в
единый одномерный диапазон. Это позволяет:

1. Равномерно распределить нагрузку между потоками независимо от глубины
   вложенности интеграла.
2. Использовать эффективный механизм редукции для атомарного накопления итоговой
   суммы без использования дорогостоящих блокировок.
3. Минимизировать накладные расходы на создание и синхронизацию параллельных
   областей.

## 2. Постановка задачи

Требуется вычислить определенный интеграл многомерной функции, заданной
структурой `IntegrationData`, методом средних прямоугольников. В качестве
эталона математической точности и базового времени выполнения ($T_1$) для
последующего расчета ускорения ($S$) и эффективности ($E$) используется
последовательная реализация алгоритма, описанная в `seq/report.md`.

## 3. Базовый алгоритм

Используется итеративный метод средних прямоугольников на линеаризованной сетке.
Многомерная сетка из $\prod n_i$ элементов обходится в одном цикле. На каждой
итерации восстанавливается вектор координат центра текущего гиперпрямоугольника,
вычисляется значение функции и аккумулируется в общую сумму.

## 4. Схема распараллеливания

Алгоритм распараллеливания основан на «сплющивании» $N$-мерной расчетной сетки в
одномерное итерационное пространство и последующем распределении нагрузки между
потоками.

**Анализ директив и атрибутов данных:**

- **Область параллелизма:** Параллельная область создается один раз в методе
  `RunImpl` с помощью директивы `#pragma omp parallel`. Внутри этой области
  каждый поток самостоятельно определяет свой участок работы, что минимизирует
  накладные расходы на вход/выход из параллельных секций.
- **Декомпозиция задач:** Реализована **ручная статическая балансировка нагрузки
  (Load Balancing)**. Общее количество итераций `total_iterations` делится на
  количество потоков. Остаток от деления распределяется между первыми потоками
  (каждый получает на одну итерацию больше). Это гарантирует максимально
  равномерную загрузку ядер процессора.
- **`default(none)`**: Используется для обеспечения строгой дисциплины
  программирования. Все переменные, используемые внутри параллельного блока,
  должны быть явно классифицированы, что исключает случайные гонки данных.
- **`shared(total_iterations, h)`**: Переменные, описывающие параметры сетки,
  являются общими. Вектор шагов `h` и общее число итераций доступны всем потокам
  только для чтения. Также общим является указатель `this`, через который
  осуществляется доступ к полям класса и методам `local_input_` и
  `CalculateChunkSum`.
- **`private`**: Переменные, отвечающие за границы чанка (`tid`, `threads`,
  `my_start`, `my_count`), объявляются внутри блока, что делает их локальными
  для каждого потока. Векторы `coords` и `current_indices` внутри
  вспомогательной функции `CalculateChunkSum` также являются локальными
  (стековыми) для каждого потока, что критически важно для обеспечения
  **потокобезопасности (Thread Safety)** при многомерном обходе.
- **`reduction(+ : total_sum)`**: Используется для безопасного накопления
  результатов вычислений от всех потоков. Механизм редукции эффективнее
  использования `atomic` или `critical`, так как каждый поток работает со своей
  локальной копией суммы, а финальное сложение происходит только один раз при
  выходе из параллельной области.
- **`schedule`**: Поскольку применена ручная декомпозиция диапазона внутри общей
  параллельной области (а не через `omp for`), управление расписанием
  выполняется программно. Это эквивалентно наиболее эффективному для данной
  задачи режиму `static`, так как объем вычислений в каждой итерации (вызов
  функции) примерно одинаков.
- **Барьеры:** В коде присутствует **неявный барьер** в конце блока `#pragma omp
  parallel`. Основной поток (master) ожидает завершения работы всех потоков и
  выполнения всех операций редукции, прежде чем умножить накопленную сумму на
  элементарный объем `d_v` и завершить выполнение.

## 5. Детали реализации

Файлы: `omp/include/rect_method_omp.hpp`, `omp/src/rect_method_omp.cpp`

Реализация OpenMP-версии сохраняет математическую логику метода средних
прямоугольников, но претерпела значительные архитектурные изменения для
обеспечения эффективности и безопасности в многопоточной среде.

### Декомпозиция и устранение вложенности

В отличие от последовательной версии, где все вычисления находились внутри
одного метода, в OMP-версии вычислительное ядро вынесено в отдельный приватный
метод `CalculateChunkSum`. В основном методе `RunImpl` осталась только логика
распределения потоков:

```cpp
// Фрагмент из RunImpl
#pragma omp parallel default(none) shared(total_iterations, h) reduction(+ : total_sum)
{
  size_t tid = omp_get_thread_num();    // индекс потока
  int threads = omp_get_num_threads();  // число потоков
  size_t chunk = total_iterations / threads;
  size_t remainder = total_iterations % threads;

  // вычисление стартового индекса и количество работы для текущего потока
  size_t my_start = (tid * chunk) + std::min(tid, remainder);
  size_t my_count = chunk + (tid < remainder ? 1 : 0);

  total_sum += CalculateChunkSum(my_start, my_count, h);
}
```

Это изменение позволило:

1. **Снизить когнитивную сложность**: Логика распределения потоков отделена от
   логики обхода $N$-мерной сетки.
2. **Изолировать контекст**: Каждый поток работает в своем вызове функции со
   своим стеком.

### Восстановление многомерного индекса

Так как потоки получают произвольные диапазоны из «плоского» итерационного
пространства $[0, Total)$, возникла необходимость в один момент времени (в
начале работы потока) восстановить текущие индексы для всех измерений.

```cpp
// Фрагмент из CalculateChunkSum
size_t temp_idx = start_idx;
for (int dim = static_cast<int>(dims) - 1; dim >= 0; --dim) {
  current_indices[dim] = static_cast<int>(temp_idx % local_input_.n_steps[dim]);
  temp_idx /= local_input_.n_steps[dim];
}
```

_Пояснение:_ Этот участок кода является ключевым отличием от SEQ. Он позволяет
любому потоку мгновенно «прыгнуть» в нужную точку гиперпространства и начать
вычисления, не проходя все предыдущие итерации.

### Устранение рисков гонок данных (Data Race)

В процессе разработки были выявлены и устранены следующие риски:

1. **Общие векторы координат**: В SEQ версии использовался один вектор `coords`.
   Если бы мы сделали его `shared`, потоки перезаписывали бы координаты друг
   друга, что привело бы к хаосу в вычислениях.
    - **Решение**: Векторы `coords` и `current_indices` объявляются **внутри**
      метода `CalculateChunkSum`. Поскольку этот метод вызывается каждым потоком
      отдельно, эти объекты создаются в локальном стеке потока, что
      автоматически делает их `private` и гарантирует потокобезопасность.

2. **Накопление суммы**: Несколько потоков не могут одновременно писать в одну
   переменную `res_`.
    - **Решение**: Использование механизма `reduction(+ : total_sum)`. OpenMP
      создает локальную копию `total_sum` для каждого потока, а итоговое
      суммирование выполняет атомарно при выходе из параллельной области.

3. **Константный доступ**: Доступ к объекту функции `local_input_.func` и
   вектору шагов `h` (помечен как `shared`) осуществляется только для чтения,
   что является потокобезопасным по стандарту C++.

## 8. Результаты

В таблице приведено чистое время выполнения алгоритма (`task_run`) при различных
конфигурациях потоков для вычисления 5D-интеграла (сетка $30^5 \approx 24.3$ млн
итераций). Базовое время ($T_1$) для расчёта метрик взято из SEQ-версии
(**2712.62 мс**).

| Backend | Mode     | Size   | Workers | Median Time (ms) | Speedup vs SEQ | Efficiency |
| ------- | -------- | ------ | ------- | ---------------- | -------------- | ---------- |
| OMP     | task_run | $30^5$ | 1       | 2567.59          | 1.06x          | 105.6%     |
| OMP     | task_run | $30^5$ | 2       | 1439.48          | 1.88x          | 94.2%      |
| OMP     | task_run | $30^5$ | 4       | 847.63           | 3.20x          | 80.0%      |
| OMP     | task_run | $30^5$ | 6       | 764.96           | 3.55x          | 59.1%      |
| OMP     | task_run | $30^5$ | 12      | 421.39           | 6.44x          | 53.6%      |

_Примечание к результатам:_

1. **Сверхэффективность на 1 потоке (105.6%):** OMP-версия на одном потоке
   отработала быстрее, чем SEQ-версия. Это обусловлено архитектурными
   особенностями: в OMP-реализации вычислительное ядро вынесено в метод
   `CalculateChunkSum`, что при определенных настройках компилятора GCC 14
   приводит к более агрессивной оптимизации циклов и более эффективному
   использованию регистров процессора по сравнению с монолитным `RunImpl` в SEQ.
2. **Масштабируемость:** Алгоритм демонстрирует уверенный рост
   производительности. При переходе от 1 к 4 потокам наблюдается почти линейное
   ускорение.
3. **Узкие места и SMT:** При использовании 6 потоков (соответствует числу
   физических ядер Ryzen 5 7500F) эффективность начинает снижаться (59.1%). Это
   связано с тем, что задача является вычислительно-интенсивной и активно
   использует блоки FPU (плавающей точки). Однако при задействовании всех 12
   логических потоков (технология SMT/Hyper-Threading) время выполнения падает
   почти в два раза по сравнению с 6 потоками ($764.96 \rightarrow 421.39$ мс).
   Это свидетельствует о том, что накладные расходы на вызов `std::function` и
   создание локальных векторов внутри `CalculateChunkSum` позволяют процессору
   эффективно переключаться между потоками на одном ядре, скрывая задержки
   доступа к памяти.

## 9. Выводы

OpenMP дал существенный выигрыш в скорости благодаря возможности параллельного
вычисления независимых значений функции во всех узлах многомерной сетки.
Использование редукции (`reduction`) позволило эффективно собрать итоговую сумму
без использования дорогостоящих блокировок и критических секций. Однако выигрыш
ограничен накладными расходами на инфраструктуру C++: частые вызовы
`std::function` и постоянное выделение памяти под локальные векторы координат
внутри `CalculateChunkSum` создают нагрузку на подсистему памяти. При увеличении
числа рабочих потоков эти издержки начинают конкурировать с выгодой от
параллелизма, что приводит к постепенному снижению эффективности на каждое
задействованное ядро.
