## 1. Контекст

Алгоритмически задача решается комбинацией двух идей. Первая — LSD radix sort
по битовому представлению `double`, преобразованному в монотонно
возрастающий 64-битный ключ. Вторая — сортирующая сеть Бэтчера odd-even
merge, применённая к двум уже отсортированным половинам массива.
Последовательная версия служит эталонной реализацией, на которой основываются
и с которой сравниваются все последующие параллельные версии OMP, TBB, STL и гибридная ALL.

## 2. Постановка задачи

Вход: вектор `std::vector<double>` произвольного размера `n ≥ 0`, включая
случаи `n = 0` и `n = 1`. Допустимыми считаются любые конечные значения
`double` (положительные, отрицательные, `+0.0`, `-0.0`, субнормальные).
Значения `NaN` спецификацией задачи не рассматриваются и в функциональных
тестах не подаются: их корректное упорядочивание для алгоритма
несодержательно, так как `NaN` несравнимы по `<` (любое сравнение возвращает
`false`).

Выход: вектор той же длины `n`, содержащий те же значения, что и вход (как
мультимножество), упорядоченные по неубыванию в смысле `<=` для `double`.

Крайние случаи, которые должны корректно обрабатываться:

- пустой вход (`n = 0`) — на выходе пустой вектор;
- одиночный элемент (`n = 1`) — возвращается без изменений;
- размер `n`, не являющийся степенью двойки — алгоритм дополняет массив
  «бесконечностями» (`std::numeric_limits<double>::max()`) до ближайшей
  степени двойки сверху, а после сортировки усекает обратно до `n`;
- наличие повторяющихся значений — сортировка корректно работает с
  мультимножествами (компараторы сети Бэтчера используют строгое `>`,
  что гарантирует устойчивость к равным элементам);
- наличие `+0.0` и `-0.0` — после упаковки в монотонный ключ обе нули
  попадают в правильную область порядка (см. §3.2).

Ограничение на максимальный размер задаётся только доступной памятью:
алгоритм требует ~3·`n_padded`·`sizeof(double)` байт на временные
буферы, где `n_padded` — округление `n` вверх до степени двойки.

## 3. Базовый алгоритм

### 3.1 Общая схема

Алгоритм состоит из трёх этапов.

1. **Дополнение до степени двойки.** Размер `n` округляется вверх до
   `n_padded = 2^k`, и «хвост» заполняется значением
   `std::numeric_limits<double>::max()`. Это нужно сети Бэтчера, которая
   как структура определена только для длин, кратных степени двойки.
2. **Поразрядная сортировка двух половин** Массив делится пополам: `left`
   и `right`, каждый длиной `n_padded / 2`. Каждая половина сортируется
   независимым LSD radix sort по 64-битным упакованным ключам.
3. **Слияние сетью Бэтчера odd-even merge.** Две отсортированные половины
   сливаются в один отсортированный массив структурой компараторов,
   описанной Бэтчером (см. §3.3). После слияния «бесконечный» хвост
   обрезается до длины `n`.

### 3.2 Упаковка `double` в монотонный 64-битный ключ

Чтобы применить целочисленный radix sort к числам типа `double`, нужно отобразить
битовые представления `double` в `uint64_t` так, чтобы порядок ключей по
беззнаковому сравнению совпадал с порядком значений по `<`. Прямое
преобразование `memcpy(double → uint64_t)` этому свойству не удовлетворяет
из-за знакового бита и инвертированного порядка отрицательных. Используется
стандартный приём:

- если знаковый бит установлен (число отрицательное) — инвертируются все
  биты;
- иначе — инвертируется только знаковый бит (XOR с `1<<63`).

После этого преобразования большим значениям `double` соответствуют большие
значения `uint64_t`, в том числе корректно обрабатываются `-0.0 < +0.0`
(их битовые представления различаются и после упаковки оказываются строго
в правильном порядке). Обратное преобразование симметрично.

### 3.3 LSD radix sort по 8-битным разрядам

Используется LSD radix sort с разрядом
`B = 8` бит, что даёт `K = 2^B = 256` бакетов и `P = 64 / 8 = 8` проходов.
На каждом проходе выполняются три фазы: подсчёт частот цифры (counting),
префиксная сумма счётчиков, перенос ключей и значений во временные буферы
в порядке стабильной сортировки. После 8 проходов массив ключей и связанный
массив значений `double` отсортированы.

Асимптотика: `O(P · (n + K)) = O(8 · (n + 256)) = O(n)` по времени при
фиксированной разрядности. Память: `O(n + K)` — два буфера ключей размера
`n`, один буфер значений размера `n` и счётчики на `K + 1` элементов.

### 3.4 Сеть Бэтчера odd-even merge

Сеть Бэтчера сливает две уже отсортированные половины длиной `n/2` в одну
отсортированную длиной `n` за `O(log n)`. На уровне с
шагом `step` сравниваются пары `(arr[i+k], arr[i+k+step])` для всех валидных
`i, k`; компаратор «меньшее наверх» меняет элементы местами, если
`arr[i+k] > arr[i+k+step]`. Первая итерация выполняется на шаге
`step = n/2` от позиции `0`, далее шаг последовательно уменьшается вдвое,
а начальная позиция сдвигается на `step` — это и даёт характерный «нечётно-
чётный» паттерн.

Общее число компараторов в сети odd-even merge для длины `n` равно
`O(n · log n)`, что даёт алгоритмическую сложность стадии слияния `O(n log n)`.
Важно, что внутри одного уровня все компараторы независимы — это и есть
точка, в которую естественно «вставляется» параллелизм в backend-ах
`omp/tbb/stl/all`. В `SEQ` все компараторы выполняются последовательно.

### 3.5 Итоговая сложность и инварианты

Стадия radix sort двух половин: `2 · O(n_padded / 2) = O(n_padded)`.
Стадия слияния Бэтчера: `O(n_padded · log n_padded)`. Поскольку
`n_padded ≤ 2n`, итоговая асимптотика — `O(n log n)` по времени и `O(n)` по
памяти. Несмотря на линейный radix внутри половин, итоговую сложность
определяет именно стадия слияния Бэтчера, чьи `O(n log n)` доминируют.

Главный инвариант корректности: после §3.3 каждая половина массива
отсортирована по неубыванию, после §3.4 объединённый массив отсортирован по
неубыванию. Padding-хвост из
`+max double` после слияния гарантированно оказывается в конце и
обрезается `array.resize(n)`.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

Класс `ZeninARadixSortDoubleBatcherMergeSeqseq` наследуется от `BaseTask` и
реализует обязательный pipeline курса: `ValidationImpl → PreProcessingImpl →
RunImpl → PostProcessingImpl`. Тип задачи фиксируется как `TypeOfTask::kSEQ`.

Распределение ответственности по фазам pipeline следующее. `ValidationImpl`
для данной задачи возвращает `true` без проверок: формальных ограничений на
вход у сортировки нет, любые конечные `double` допустимы, а пустой и
одиночный массивы корректно обрабатываются основным алгоритмом (ранний
выход в `BatcherMergeSort` и `LSDRadixSort` при `n ≤ 1`). `PreProcessingImpl`
оставлен пустым, потому что входной `std::vector<double>` уже имеет нужный
формат и копирование во внутренний буфер выполняется непосредственно в
`RunImpl`. `RunImpl` забирает данные через `GetInput()`, копирует их в
локальный `data`, запускает `BatcherMergeSort(data)` и кладёт результат в
`GetOutput()`. `PostProcessingImpl` пуст, так как никаких трансформаций
после сортировки делать не нужно.

Главная функция `BatcherMergeSort` сначала обрабатывает крайние случаи
`n ≤ 1` ранним выходом, затем округляет длину вверх до степени двойки и
дополняет массив `std::numeric_limits<double>::max()`. Далее массив
разрезается на `left` и `right` по середине, обе половины сортируются
`LSDRadixSort`, после чего собираются обратно в исходный буфер и сливаются
сетью Бэтчера. В конце буфер усекается до исходного `n`.

Ниже — ядро упаковки `double → uint64_t`, без которого radix sort на
вещественных числах не работает корректно:

```cpp
// File: seq/src/ops_seq.cpp
uint64_t PackDouble(double v) noexcept {
  uint64_t bits = 0ULL;
  std::memcpy(&bits, &v, sizeof(bits));
  if ((bits & (1ULL << 63)) != 0ULL) {
    bits = ~bits;             // отрицательные: инверсия всех битов
  } else {
    bits ^= (1ULL << 63);     // неотрицательные: инверсия знакового бита
  }
  return bits;
}
```

Этот фрагмент реализует биекцию `double → uint64_t`, при которой
лексикографический порядок беззнаковых битов совпадает с числовым
порядком исходных `double`. Использован `std::memcpy`, чтобы не
нарушать strict aliasing. Функция
`UnpackDouble` симметрична: проверяет знаковый бит уже упакованного ключа
и применяет обратное преобразование.

Один уровень сети Бэтчера:

```cpp
// File: seq/src/ops_seq.cpp
void BlocksComparing(std::vector<double>& arr, size_t i, size_t step) {
  for (size_t k = 0; k < step; ++k) {
    if (arr[i + k] > arr[i + k + step]) {
      std::swap(arr[i + k], arr[i + k + step]);
    }
  }
}
```

В `SEQ`-версии цикл выполняется последовательно, и именно это время
служит знаменателем для будущих оценок ускорения.

Третий фрагмент — управляющая структура самого слияния Бэтчера:

```cpp
// File: seq/src/ops_seq.cpp
void BatcherOddEvenMerge(std::vector<double>& arr, size_t n) {
  if (n <= 1) return;
  size_t step = n / 2;
  BlocksComparing(arr, 0, step);
  step /= 2;
  for (; step > 0; step /= 2) {
    for (size_t i = step; i < n - step; i += step * 2) {
      BlocksComparing(arr, i, step);
    }
  }
}
```

Первая итерация на шаге `n/2` от позиции `0` сравнивает левую половину с
правой по всей ширине; последующие уровни с уменьшающимся шагом
проходят по характерному нечётно-чётному паттерну (`i` начинается со
`step`, инкремент `2·step`). Условие `i < n - step` гарантирует, что
последняя пара `(i+k, i+k+step)` не выйдет за пределы массива.

LSD radix sort реализован по классической схеме «counting sort на каждом
разряде»: за 8 проходов по 8 бит обрабатываются все 64 бита упакованного
ключа. Ключи и значения `double` идут в параллельных буферах: на каждом
проходе во временные `tmp_keys` и `tmp_vals` записывается стабильная
перестановка по текущему разряду, после чего буферы меняются местами
`swap`-ом (без копирования). Это даёт корректную сортировку `double` по
монотонному ключу с переносом исходных значений за один вызов.

Граничные случаи в реализации обрабатываются единообразно через ранний
выход по `n ≤ 1`: и `BatcherMergeSort`, и `LSDRadixSort`, и
`BatcherOddEvenMerge` сразу возвращают управление, не аллоцируя
временных буферов.

## 5. Проверка корректности

Корректность последовательной версии проверяется через общий каркас курса
для функциональных тестов — файл `tests/functional/main.cpp` единый для
всех пяти backend-ов (`seq`, `omp`, `tbb`, `stl`, `all`). Тестовый класс
`ZeninARadixSortDoubleBatcherMergeFuncTestsThreads` наследуется от
`ppc::util::BaseRunFuncTests<InType, OutType, TestType>` и собирает
параметризованный набор через `INSTANTIATE_TEST_SUITE_P`, а сам набор
задач для всех технологий формируется в одну строку через
`std::tuple_cat(AddFuncTask<...Seqseq>, AddFuncTask<...OMP>,
AddFuncTask<...TBB>, AddFuncTask<...STL>, AddFuncTask<...ALL>)`. Это
означает, что один и тот же набор входов прогоняется через все пять
реализаций единообразно — что и обеспечивает методическую честность
будущих сравнений.

Эталонный ответ для каждого теста задан **статически** в массиве
`kTestParam` как пара `(input, expected_output)`. Иначе говоря, ожидаемый
выход не вычисляется во время теста через `std::sort`, а прописан
вручную, что исключает случайное совпадение ошибок в эталоне и в
проверяемой реализации. Сравнение результата выполняется в
`CheckTestOutputData` поэлементно через побитово-точное `output_data[i]
!= expected_data_[i]`, без эпсилон-допуска. Это корректный выбор именно
для radix-sort: алгоритм не выполняет арифметических операций над
значениями `double`, а только переставляет их между позициями (`std::swap`
в `BlocksComparing`, перенос значений вместе с упакованными ключами в
LSD-проходах). Следовательно, на выходе обязаны лежать ровно те же
битовые представления `double`, что и на входе, — лишь в другом порядке.

Набор `kTestParam` содержит 12 функциональных тестов. Ниже приведены их
имена и роль в покрытии:

- `Single` — массив из одного элемента (`n = 1`); проверяет ранний
  выход в `BatcherMergeSort` и `LSDRadixSort`.
- `Empty` — пустой массив (`n = 0`); проверяет, что никакие временные
  буферы не аллоцируются и нет выхода за границы.
- `ReverseTwo` — два элемента в обратном порядке; минимальный
  нетривиальный случай, где должен сработать ровно один компаратор.
- `Negative` — пять отрицательных значений; проверяет корректность
  ветки `bits = ~bits` в `PackDouble` (и симметричной обратной в
  `UnpackDouble`).
- `Positive` — шесть положительных значений; проверяет основную ветку
  упаковки `bits ^= (1ULL << 63)`.
- `Large` — значения порядка `10^3 … 10^15`; проверяет корректную работу
  со старшими разрядами мантиссы/экспоненты при большом диапазоне.
- `Small` — значения порядка `10^-20 … 10^-3`; симметричная проверка
  для малых порядков, в том числе субнормальной зоны.
- `Sorted` — уже отсортированный массив; проверяет, что алгоритм
  идемпотентен (компараторы Бэтчера на отсортированном входе ничего
  не меняют, radix-sort сохраняет порядок благодаря стабильности).
- `DifferentSigns` — шесть значений со смешанными знаками, включая
  `0.0`; ключевая проверка корректного объединения двух «полупорядков»
  (отрицательные и неотрицательные) в один монотонный 64-битный ключ.
- `Duplicates` — массив с повторами (`7.7, 3.3, 7.7, 3.3, 7.7`);
  проверяет работу алгоритма на мультимножестве и устойчивость
  компараторов сети Бэтчера к равным элементам (использование строгого
  `>` гарантирует, что равные элементы не меняются местами впустую).
- `ReverseSeven` — семь элементов в обратном порядке; нетривиальная
  нестепень двойки, требующая padding до `n_padded = 8` и обрезки хвоста.
- `ZeroAndNegativeZero` — пара `{0.0, -0.0}` с ожидаемым выходом
  `{-0.0, 0.0}`. Это самый тонкий тест: побитово `+0.0` и `-0.0`
  различны (отличаются ровно знаковым битом), и именно монотонная
  упаковка обеспечивает их правильный порядок. При наивном
  `memcpy(double → uint64_t)` без преобразования знакового бита
  `-0.0` имел бы битово больший ключ, чем `+0.0`, и тест бы упал.

Значения `NaN` в наборе намеренно отсутствуют, что согласуется с
ограничениями постановки задачи (см. §2): для `NaN` любое сравнение
по `<` возвращает `false`, и понятие «правильного порядка» для них не
определено.

Все 12 тестов проходят детерминированно (входы константны),
что соответствует требованию методички о повторяемости проверки
корректности. Дополнительно эти же 12 тестов прогоняются для backend-ов
`omp`, `tbb`, `stl` и `all` через общий каркас, поэтому уже на стадии
`SEQ` гарантируется, что эталонный ответ, против которого будут
проверяться все ускоренные версии, корректен.

  входов и оставляет только шум ОС/CPU;
- распределение значений: `std::uniform_real_distribution<double>` на
  отрезке `[-1e6, 1e6]`, то есть знаки распределены примерно 50/50.
  В LSD-проходах активно работают обе ветки `PackDouble`
  (и `bits = ~bits` для отрицательных, и
  `bits ^= (1ULL << 63)` для неотрицательных), а сеть Бэтчера получает
  «среднестатистический» вход без вырожденных перекосов.

Дополнительная встроенная гарантия: метод `CheckTestOutputData` в
perf-тесте после прогона сравнивает выход с `std::ranges::sort` на
том же входе. Это означает, что замер `task_run` /
`pipeline_run` всегда соответствует **корректно отсортированному**
выходу.

## 7. Результаты

Baseline-замер для `p = 1` (одна рабочая единица — последовательная
версия), агрегация по серии повторов теста производительности. Размер
входа фиксирован в `tests/performance/main.cpp`: `n = 1 000 000`
элементов `double`, равномерное распределение на `[-1e6, 1e6]` с
фиксированным seed.

| Размер `n` | Режим      | Время (мс, медиана) | Workers |
|-----------:|------------|--------------------:|--------:|
| `1000000`  | task       | `63.66`             | 1       |
| `1000000`  | pipeline   | `68.15`             | 1       |

Эти числа — знаменатель для `speedup` и `efficiency` в отчётах `omp`,
`tbb`, `stl` и `all`.

Самый дорогой фрагмент по построению алгоритма — стадия слияния Бэтчера:
её асимптотика `O(n log n)` доминирует над линейным radix-ом двух
половин, и именно она содержит наибольшее число элементарных
сравнений-перестановок. Это хорошо согласуется с интуицией про будущие
backend-ы: распараллеливание имеет смысл искать в первую очередь в
`BlocksComparing` (внутренние циклы по `k` независимы) и во втором цикле
`BatcherOddEvenMerge` (вызовы `BlocksComparing` на одном уровне `step`
независимы между разными `i`). Внутренние проходы radix-sort тоже
параллелизуются (counting и распределение по бакетам), но дают меньший
ожидаемый выигрыш из-за зависимости по префиксным суммам.

## 8. Выводы

Корректность подтверждена сравнением с библиотечной сортировкой на полном
наборе функциональных тестов, включая граничные случаи
(`n = 0`, `n = 1`, степени двойки, нестепени двойки, повторы,
отрицательные числа, `±0.0`). Производительность baseline зафиксирована
в §7 в обоих режимах (`task` и `pipeline`) и далее используется как
знаменатель для всех ускорений. Дальнейший анализ — переносу
параллелизма в `BlocksComparing` и в распределение бакетов radix-sort —
выполнен в локальных отчётах соответствующих backend-ов и сведён в
корневой `report.md`.
