## 1. Контекст

Цель данной версии — перенести алгоритм, описанный и обоснованный в
`seq/report.md`, на OpenMP. Сам алгоритм при этом не меняется: те же три
этапа — padding до степени двойки, LSD radix sort двух половин,
финальное слияние сетью Бэтчера — выполняются над теми же данными и в
том же порядке. Меняется только то, **где** каждая стадия выполняется:
последовательно главным потоком или параллельно набором рабочих
потоков OpenMP.

## 2. Постановка задачи

Постановка идентична `seq/report.md`: на вход подаётся
`std::vector<double>` произвольной длины `n ≥ 0`, допустимы любые
конечные значения (положительные, отрицательные, `+0.0`, `-0.0`,
субнормальные), значения `NaN` не рассматриваются. На выходе — вектор
той же длины, содержащий те же значения, упорядоченные по неубыванию.
Эталоном корректности служит результат `SEQ` (а в perf-тесте —
дополнительно `std::ranges::sort`), и любой OMP-прогон обязан давать
побитово идентичный результат.

## 3. Базовый алгоритм

1. размер `n` округляется вверх до `n_padded = 2^k`, хвост заполняется
   `std::numeric_limits<double>::max()`;
2. массив разрезается пополам на `left` и `right`, каждая половина
   сортируется LSD radix-ом по 64-битному монотонно упакованному ключу;
3. две отсортированные половины сливаются сетью Бэтчера odd-even merge.

Асимптотика: `O(n log n)` по времени, `O(n)` по памяти. В OMP-
версии асимптотика не меняется — параллелизуется работа внутри
постоянных по форме стадий.

## 4. Схема распараллеливания

В реализации применены **два независимых уровня OpenMP-параллелизма**,
покрывающие разные стадии алгоритма.

### 4.1 Уровень 1: `parallel sections` для двух половин radix-sort

Стадия «сортировка двух половин» по построению алгоритма состоит из
двух полностью независимых задач: `LSDRadixSort(left)` и
`LSDRadixSort(right)` работают над непересекающимися векторами и не
имеют ни общих данных, ни синхронизаций. Это
**task-level parallelism**, для которого в OpenMP естественен примитив
`sections`:

```cpp
// File: omp/src/ops_omp.cpp (RunImpl)
#pragma omp parallel sections default(none) shared(left, right)
{
#pragma omp section
  { LSDRadixSort(left); }
#pragma omp section
  { LSDRadixSort(right); }
}
```

Разбор директивы:

- `parallel sections` создаёт команду потоков и сразу же распределяет
  по ним перечисленные `section`-блоки. Каждый блок выполняется ровно
  одним потоком; если потоков больше двух, лишние потоки в этой
  секции бездействуют.
- `default(none)` — обязательное методически дисциплинирующее
  требование: любая переменная, используемая внутри параллельного
  региона, должна быть явно классифицирована.
- `shared(left, right)` — оба вектора передаются по ссылке и должны
  быть видимы обоим потокам. Гонки между ними нет, потому что
  `section`-1 пишет только в `left`, а `section`-2 — только в
  `right`; их `data()` указывают на разные области памяти. Внутри
  каждой секции потоку видна **его собственная** копия итераторов и
  локальных переменных `LSDRadixSort` (включая `keys`, `tmp_keys`,
  `tmp_vals`, `cnt`, `pass`, `shift`, `i`, `pos`, `d`) — все они
  объявлены как локальные переменные внутри функции.
- `reduction` не нужен: алгоритм не имеет аккумулятора, а лишь
  переставляет элементы между позициями.
- `atomic`/`critical` не нужны: непересечение записей гарантировано
  разрезанием массива пополам.

В конце `parallel sections` действует **неявный барьер**: управление не
возвращается главному потоку, пока обе секции не завершатся. Следующая
стадия — слияние Бэтчера — обязана видеть
обе половины уже отсортированными, иначе теорема Бэтчера об
odd-even merge для двух отсортированных последовательностей не
применима. Барьер не отключал через `nowait` именно по этой причине.

### 4.2 Уровень 2: `parallel for` для уровней сети Бэтчера

Сеть Бэтчера odd-even merge устроена так, что компараторы одного
уровня шага `step` взаимно независимы по индексам, то есть **по
построению** допускают параллельное выполнение. Этот факт прямо
используется:

```cpp
// File: omp/src/ops_omp.cpp (BatcherOddEvenMerge)
for (size_t po = n / 2; po > 0; po >>= 1) {
  if (po == n / 2) {
#pragma omp parallel for shared(arr, po) default(none)
    for (size_t i = 0; i < po; ++i) {
      BlocksComparing(arr, i, i + po);
    }
  } else {
#pragma omp parallel for shared(arr, n, po) default(none)
    for (size_t i = po; i < n - po; i += 2 * po) {
      for (size_t j = 0; j < po; ++j) {
        BlocksComparing(arr, i + j, i + j + po);
      }
    }
  }
}
```

Структура устроена в два сценария.

**Первая итерация (`po == n / 2`).** На первом уровне сети Бэтчера
сравниваются пары «левая половина против правой» — `n/2` независимых
компараторов с индексами `(i, i + n/2)`. Параллельный цикл идёт по
самим компараторам: каждая итерация делает ровно одно сравнение через
`BlocksComparing(arr, i, i + po)`. На `n = 10^6` это даёт `5·10^5`
независимых итераций, что более чем достаточно для эффективного
распределения по 4–8 потокам.

**Остальные уровни (`po < n / 2`).** На последующих уровнях работа
организована блоками: для каждого начального индекса `i ∈ {po, 3·po,
5·po, …}` нужно сделать `po` смежных компараторов
`(i+j, i+j+po)`. Параллелизуется **внешний** цикл по `i`, а внутренний
по `j` оставлен последовательным внутри потока. Это сознательный
выбор по нескольким причинам. Во-первых, число блоков `i` на каждом уровне
шага `po` равно `(n - po) / (2·po) ≈ n / (2·po) − 1`, что уменьшается
с уровнем; параллелить здесь именно блоки даёт OpenMP-runtime разумно
крупный chunk для распределения. Во-вторых, смежные пары
`(i+j, i+j+po)` для соседних `j` располагаются в смежных кэш-линиях,
поэтому последовательное прохождение по `j` внутри одного потока
обеспечивает **локальность доступа к памяти** и эффективное
использование L1/L2 кэша. Если бы параллелили внутренний цикл, в
кэше каждого потока оказывались бы разрозненные элементы, что дало
бы лишние промахи.

Разбор директив:

- `default(none)` — явная классификация
  всех переменных.
- `shared(arr, po)` / `shared(arr, n, po)` — массив `arr` шарится,
  потому что компараторы пишут в него; `po` и `n` —
  параметры цикла. Переменная цикла `i` в OpenMP-`parallel for`
  **по умолчанию `private`**,
  поэтому её отдельно объявлять не нужно. Внутренняя `j` — обычная
  локальная переменная в теле цикла, она автоматически private для
  каждого потока, так как её стек-фрейм принадлежит итерации.
- `reduction` не используется и не нужен: компараторы только
  переставляют пары элементов через `std::swap`, аккумулятора нет.
- `atomic`/`critical` не нужны: записи в `arr` идут по
  непересекающимся индексам внутри одного уровня. Для индексов
  `i_1, i_2` (разных значений `i` во внешнем цикле) пары
  `[i_1, i_1 + 2·po)` и `[i_2, i_2 + 2·po)` не пересекаются — это и
  есть ключевое свойство сети Бэтчера, обеспечивающее
  thread-safety без всякой синхронизации.
- `schedule` явно **не задан**, что соответствует поведению
  `schedule(static)` по умолчанию. Для данной нагрузки это оправдано:
  все итерации (компараторы) делают примерно одинаковую работу — одно
  сравнение и одну условную перестановку, — поэтому статическое
  равномерное разбиение даёт оптимальную балансировку без runtime-
  overhead-а на динамическое распределение чанков. `schedule(dynamic)`
  или `guided` здесь бессмысленны, так как нет сильного разброса
  времени между итерациями.

Между уровнями шага `po` стоит **неявный барьер** в конце каждого
`parallel for`: следующая итерация внешнего цикла по `po >>= 1`
зависит от того, что предыдущий уровень компараторов уже завершился
полностью. Поэтому `nowait` не применялся — это была бы ошибка
корректности, нарушающая структуру сети Бэтчера. Каждая итерация
внешнего цикла создаёт **новый параллельный регион** (на каждом уровне
заново), что вносит накладные расходы на форк/джойн команды потоков;
это потенциальное узкое место, и оно обсуждается в §8.

### 4.3 Что НЕ распараллелено

Сознательно оставлены последовательными:

- **Внутренние проходы LSD radix-sort** (counting, prefix-sum,
  распределение по бакетам). Параллелизация counting-фазы потребовала
  бы либо приватных счётчиков на 256 ячеек × число потоков с
  последующей merge-фазой, либо `atomic increment` на каждый элемент;
  и то и другое сравнимо или дороже самой работы при `n_half =
  5·10^5`. Параллелизм оставлен на уровне sections.
- **`PackDouble` / `UnpackDouble` циклы** на входе и выходе LSD.
  Это `O(n)` без зависимостей и теоретически параллелится через
  `parallel for`, но при размере `5·10^5` и стоимости одной итерации
  в ~3 такта overhead на запуск параллельного региона сопоставим с
  выигрышем. Оставлено последовательным внутри своей секции, что
  означает, что эти циклы всё равно идут параллельно с
  соответствующими циклами другой секции — то есть фактически уже
  работают на двух потоках через section-параллелизм.
- **Финальная обрезка `data.resize(original_size)`** и копирования
  между `data ↔ left/right` — это короткие операции, не критичные
  по времени.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Класс `ZeninARadixSortDoubleBatcherMergeOMP` наследуется от `BaseTask`
и устанавливает `TypeOfTask::kOMP`. Pipeline-методы распределены так
же, как в SEQ: `ValidationImpl` и `PreProcessingImpl` возвращают
`true` без действий, `PostProcessingImpl` пуст, а вся содержательная
работа сосредоточена в `RunImpl`.

Относительно SEQ-версии изменения структурно минимальны:

1. **`RunImpl` вынесен из вспомогательной функции `BatcherMergeSort`
   обратно в тело самого метода.** В SEQ был отдельный статический
   метод `BatcherMergeSort`, в OMP его роль играет содержимое
   `RunImpl`. Логика та же: ранний выход для `n ≤ 1`, padding до
   степени двойки, разрезание на `left/right`, две сортировки, копия
   обратно, слияние Бэтчера, обрезка до `n`. Различие чисто
   стилистическое — `parallel sections` удобнее видеть в основном
   потоке выполнения, а не «внутри статической функции».

2. **`LSDRadixSort` идентичен SEQ.** Никаких OMP-директив внутри
   radix-sort не добавлено по причинам, описанным в §4.3. Это
   позволяет утверждать, что результат сортировки половины битовой
   побайтово идентичен результату SEQ — что критично для
   функциональных тестов с побитовым сравнением `!=`.

3. **`BatcherOddEvenMerge` перестроен.** В SEQ он состоял из одной
   итерации на шаге `n/2` и затем цикла по уменьшающемуся шагу с
   вложенным циклом по `i`. В OMP внешний цикл также идёт по `po =
   n/2, n/4, …, 1`, но внутри первой итерации используется упрощённая
   форма `parallel for` сразу по компараторам (диапазон `[0, n/2)`),
   а в последующих — стандартная форма «параллельно по блокам, внутри
   блока последовательно». Содержательно сеть Бэтчера та же самая,
   меняется лишь форма обхода компараторного слоя — каждая пара
   `(i+j, i+j+po)` сравнивается ровно один раз на своём уровне.

4. **`BlocksComparing` в OMP упрощён** относительно SEQ: вместо
   цикла внутри функции она теперь делает один-единственный
   компаратор `if (arr[i] > arr[j]) std::swap(...)`. Это согласовано
   с новой формой `BatcherOddEvenMerge`, где цикл по `j` стал внешним
   относительно вызова `BlocksComparing`, а не внутренним.

5. **Параллельный регион `sections` поверх двух radix-sort-ов**
   обсуждался в §4.1.

Граничные случаи обработаны явно в самом начале `RunImpl`:

```cpp
// File: omp/src/ops_omp.cpp (RunImpl)
if (original_size <= 1) {
  GetOutput() = data;
  return true;
}
```

Это устраняет необходимость защищать `parallel sections` и
`parallel for` от случаев `n = 0` и `n = 1`, при которых корректность
индексов в `BlocksComparing` нарушилась бы (для `n = 1` было бы
`po = 0`, и циклы вырождаются в пустые — но ранний выход чище и
дешевле).

Потокобезопасность алгоритма гарантируется тремя свойствами,
проверяемыми по построению:

- **На уровне `sections`** — непересечение векторов `left` и `right`
  как объектов памяти; никакие OMP-потоки не пишут в одну и ту же
  кэш-линию.
- **Внутри `parallel for` по компараторам** — непересечение пар
  индексов `(i+j, i+j+po)` для разных значений `i` на одном уровне
  сети Бэтчера. Это аксиоматическое свойство сети, и именно поэтому
  она вообще параллелизуется.
- **Между уровнями `po`** — неявные барьеры в конце каждого
  `parallel for` и `parallel sections` гарантируют, что следующая
  стадия видит полностью обновлённый массив.

## 6. Проверка корректности

Корректность OMP-версии проверяется тем же общим каркасом курса,
описанным в `seq/report.md`, §5. Тот же файл `tests/functional/main.cpp`
прогоняет тот же набор `kTestParam` из 12 тестов (`Single`, `Empty`,
`ReverseTwo`, `Negative`, `Positive`, `Large`, `Small`, `Sorted`,
`DifferentSigns`, `Duplicates`, `ReverseSeven`, `ZeroAndNegativeZero`)
дополнительно для `ZeninARadixSortDoubleBatcherMergeOMP` через
`AddFuncTask<...OMP>` в общем `tuple_cat`. Сравнение выполняется
побитово через `!=` без эпсилон-допуска, что для алгоритма, не
выполняющего арифметических операций, методически корректно (см.
обоснование в §5 `seq/report.md`).

Все 12 функциональных тестов проходят детерминированно. Из них
особо важные для OMP-версии — те, что проверяют корректность точек
синхронизации:

- `ReverseSeven` — нестепень двойки `n = 7` → `n_padded = 8`,
  половины по 4 элемента. После `parallel sections` обе половины
  должны быть отсортированы; если бы в `LSDRadixSort` была гонка
  (которой нет, но это страховка), `BatcherOddEvenMerge` получил бы
  неотсортированный вход и выдал бы некорректный результат.
- `Duplicates` — повторяющиеся значения. Если бы в `BlocksComparing`
  использовалось `>=` вместо `>`, равные элементы лишний раз
  переставлялись бы между потоками; на побитовом сравнении это могло
  бы дать ложно-корректный ответ, но фактически устойчиво воспроизводит
  правильность выбора `>`.
- `DifferentSigns` — смешанные знаки. Проверяет согласованность работы
  обеих секций при том, что в одной половине больше отрицательных, а
  в другой больше положительных.

Дополнительно, поскольку Perf-тест (`tests/performance/main.cpp`,
`CheckTestOutputData`) сравнивает выход с `std::ranges::sort` на
`n = 1 000 000`, успешный прогон `pipeline_run` / `task_run` означает,
что OMP-версия корректна и на крупном детерминированном входе с
фиксированным seed-ом 1 000 000 — то есть на том же массиве, который
будут сортировать и все остальные backend-ы.

Проверка на устойчивость результата при разном числе потоков (1, 2, 4,
8) выполнена дополнительно через переменную окружения
`PPC_NUM_THREADS` (см. §7); во всех случаях выход побитово совпадает,
что подтверждает отсутствие зависимости результата от планирования
потоков.

## 8. Результаты

Замеры выполнялись для каждого режима (`task` = `task_run`, `pipeline`
= `pipeline_run`) при числе потоков `p ∈ {1, 2, 4, 8}`. Время — медиана
по серии повторов perf-каркаса курса, переведённая из секунд в
миллисекунды. Ускорение `S(p) = T_SEQ / T_OMP(p)`, эффективность
`E(p) = S(p) / p`. В качестве знаменателя `T_SEQ` используется baseline
из `seq/report.md`, §7.

Самые вероятные узкие места OMP-версии (по структурному анализу):

1. **`parallel sections` ограничен двумя секциями.** Это известное
   методическое ограничение sections-параллелизма: при наличии
   `OMP_NUM_THREADS > 2` лишние потоки на этой стадии простаивают.
2. **Создание нового параллельного региона на каждом уровне сети
   Бэтчера.** Внешний цикл по `po >>= 1` делает `log₂(n_padded)` ≈ 20
   итераций для `n = 10^6`, и каждая запускает свой `parallel for`. На
   мелких уровнях (когда `po` становится малым и число блоков
   уменьшается) overhead на форк/джойн команды может оказаться
   сопоставим с полезной работой. Один из возможных способов
   обхода — обернуть весь `BatcherOddEvenMerge` в один `#pragma omp
   parallel` с внутренними `#pragma omp for` и `#pragma omp single`
   для управления внешним циклом; это не реализовано здесь ради
   простоты, но указано как путь будущей оптимизации.
3. **Memory-bound характер сети Бэтчера на крупных уровнях.** При
   `po = n/2` каждый из `n/2` компараторов читает и пишет в две
   удалённые ячейки массива размера `n · sizeof(double) = 8 МБ`, что
   не помещается в L2-кэш современных CPU и периодически промахивается
   в L3 и DRAM. Это ставит верхний предел на достижимое ускорение
   независимо от числа потоков.

## 9. Выводы

OMP-версия реализована минимально-инвазивно относительно SEQ-эталона:
структура алгоритма, ядро LSD radix-sort и сеть компараторов Бэтчера
не изменены. Параллелизм добавлен ровно в тех двух местах, где он
существует структурно: между двумя независимыми сортировками половин
(`parallel sections`) и между независимыми компараторами одного уровня
сети Бэтчера (`parallel for`). Все директивы используют `default(none)`
с явным указанием `shared`, что соответствует методическим требованиям
курса к OMP-коду; гонки исключены по построению (непересечение
индексов), `reduction`/`atomic`/`critical` не нужны и не используются;
`schedule` оставлен по умолчанию (`static`), что соответствует
сбалансированной нагрузке итераций.

Известные ограничения масштабируемости — sections-параллелизм
ограничен двумя потоками на стадии radix-sort, и сеть Бэтчера на
крупных уровнях упирается в пропускную способность памяти — описаны
прямо в §8 и являются структурной ценой выбранного крупнозернистого
подхода. Численные подтверждения этих ожиданий (или их опровержение)
заполняются конкретными значениями в таблицах §8 по результатам
прогонов на используемой машине; интерпретация и сравнение с другими
backend-ами (`tbb`, `stl`, `all`) — задача корневого `report.md`.
