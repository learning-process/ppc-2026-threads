## 1. Контекст

TBB-версия переносит тот же алгоритм (LSD radix-sort двух половин +
сеть Бэтчера odd-even merge), что и в SEQ/OMP, на task-based runtime
oneTBB. Проверяется как этот же алгоритм ведёт себя на
runtime, который сам управляет распределением задач по worker-потокам
через work-stealing, в отличие от статической команды потоков OpenMP.
В частности, интересно, насколько хорошо `auto_partitioner` справится с
неоднородной по структуре нагрузкой сети Бэтчера, и можно ли получить
сопоставимый с OMP результат.

## 2. Постановка задачи

Идентична `seq/report.md`, §2: сортировка `std::vector<double>`
произвольной длины `n ≥ 0`, любые конечные значения допустимы (включая
`±0.0` и субнормальные), `NaN` не подаётся. Эталон корректности — выход
SEQ-версии и `std::ranges::sort` в perf-тесте; OMP-, TBB-, STL-, ALL-
версии обязаны давать побитово идентичный результат.

## 3. Базовый алгоритм

Кратко: padding до степени двойки → независимая сортировка двух половин
LSD radix-ом по монотонно упакованному `uint64_t`-ключу → слияние двух
отсортированных половин сетью Бэтчера odd-even merge → обрезка
padding-хвоста. Асимптотика: `O(n log n)` по времени, `O(n)` по памяти.
Полное описание — в `seq/report.md`, §3.

## 4. Схема распараллеливания

В TBB-версии параллелизм добавлен в двух местах, причём в этих
местах применены разные TBB-примитивы, подобранные под характер
нагрузки. Третий потенциальный участок — внутренние уровни сети
Бэтчера — оставлен последовательным; это известное ограничение
текущей реализации, и оно явно обсуждается в §4.3 ниже.

### 4.1 `tbb::parallel_invoke` для двух radix-sort-ов

Стадия «отсортировать две половины» структурно соответствует
fork-join из двух независимых задач. Для такого паттерна в oneTBB
есть специально предназначенный примитив `parallel_invoke`, который
запускает указанные функторы как параллельные задачи и возвращает
управление только после завершения всех из них:

```cpp
// File: tbb/src/ops_tbb.cpp (RunImpl)
tbb::parallel_invoke(
    [&]() { LSDRadixSort(left);  },
    [&]() { LSDRadixSort(right); });
```

Разбор:

- `parallel_invoke` — task-level примитив TBB, спроектированный именно
  для запуска фиксированного небольшого числа независимых функторов
  (от 2 до 10). Семантически он эквивалентен `#pragma omp parallel
  sections` в OMP-версии, но реализован через task-scheduler oneTBB,
  что даёт два технических отличия: задачи попадают в общий пул
  worker-потоков и могут быть украдены work-stealing-ом любым
  свободным потоком, и накладные расходы на создание задачи (две
  пары `tbb::task::allocate` + `tbb::task::spawn`) обычно ниже, чем
  на форк команды OMP-потоков.
- Лямбды захватывают `left` и `right` по ссылке (`[&]`); это
  безопасно, потому что: (а) оба вектора живут на стеке `RunImpl`
  до конца `parallel_invoke`;
  (б) каждый функтор пишет только в свой вектор, записи в
  разные объекты памяти не пересекаются — race conditions исключены
  по построению.
- Неявная синхронизация: `parallel_invoke` блокирует
  вызывающий поток до завершения обеих задач. Это та же гарантия,
  что давал неявный барьер в `#pragma omp parallel sections` в
  OMP-версии. Она критически важна: следующая стадия (слияние Бэтчера)
  обязана видеть обе половины уже отсортированными, иначе
  теорема Бэтчера об odd-even merge для двух отсортированных
  последовательностей не применима.

Структурное ограничение, общее с OMP-`sections`: число параллельных
задач здесь равно двум, что для TBB-runtime означает, что на этой
стадии будут заняты не более двух worker-потоков. Лишние worker-ы (если
их в арене больше двух) бездействуют. Параллелизация counting-фазы внутри
LSD-сортировки требует синхронизации по 256-ячеечному `cnt` на каждом из
8 проходов, что для половины из `5·10^5` элементов даёт неблагоприятное соотношение
полезной работы и overhead-а.

### 4.2 `tbb::parallel_for` с `blocked_range` для первого уровня сети Бэтчера

Первая итерация, на которой
сравниваются пары `(i, i + n/2)` для `i ∈ [0, n/2)`. Это `5·10^5`
независимых компараторов на `n = 10^6`. Здесь применён классический
`parallel_for` поверх `blocked_range`:

```cpp
// File: tbb/src/ops_tbb.cpp (BatcherOddEvenMerge)
if (po == n / 2) {
  tbb::parallel_for(
      tbb::blocked_range<size_t>(0, po),
      [&](const tbb::blocked_range<size_t> &r) {
        for (size_t i = r.begin(); i < r.end(); ++i) {
          BlocksComparing(arr, i, i + po);
        }
      });
}
```

Разбор:

- **Диапазон работы:** `tbb::blocked_range<size_t>(0, po)`. Этот
  range-объект описывает рекурсивно делимый одномерный диапазон
  индексов, который TBB-партиционер автоматически разбивает на
  подзадачи. В отличие от OpenMP-`parallel for`, где итерации
  раздаются по фиксированному `schedule`, blocked_range позволяет
  runtime-у **сам решать**, на сколько чанков делить, и
  перебалансировать их через work-stealing.
- **Grain size:** не задан явно, что соответствует значению по
  умолчанию `1`. Однако фактический размер
  чанка определяется не grain size непосредственно, а связкой
  «grain size + partitioner»: `auto_partitioner` использует grain size
  только как нижнюю границу и подбирает фактический «разумный»
  размер на основе работы work-stealing-механизма.
- **Partitioner:** не задан явно, что в TBB означает использование
  **`auto_partitioner`** по умолчанию. `auto_partitioner` начинает с
  крупных чанков и делит их
  дальше только если другие worker-ы простаивают и могут украсть
  работу. Для данной нагрузки это удобно, потому что: (а) все
  компараторы дают примерно одинаковую работу (одно `if` + условный
  `swap`), и крупные чанки не вызывают дисбаланса; (б) при
  `n/2 = 5·10^5` итераций и грубо ~6–8 worker-ов TBB разобьёт диапазон
  на ~16–64 чанка, что более чем достаточно для эффективного
  распределения, и при этом overhead на создание задач остаётся
  малым.
- **Захват по ссылке** `[&]`: `arr` и `po` шарятся между функторами
  через ссылку. Гонки между итерациями исключены по построению
  сети Бэтчера: пары `(i, i + po)` и `(i', i' + po)` для разных
  `i, i' ∈ [0, po)` не пересекаются (разница не меньше 1 и не равна
  `po`).
- **Никакой явной синхронизации внутри лямбды.** Это и есть смысл
  «безопасного по построению» дизайна сети Бэтчера — `mutex`,
  `atomic` и `concurrent_vector` здесь не нужны.

`parallel_for` блокирует вызывающий поток до завершения всех чанков,
поэтому к моменту перехода к следующему уровню `po >>= 1` все
компараторы текущего уровня гарантированно отработали.

### 4.3 Что не распараллелено

Здесь TBB-версия отличается от OMP-версии

**Внутренние уровни сети Бэтчера (`po < n / 2`)** в текущей реализации
**не распараллелены**: для них стоит обычный последовательный
`for`-цикл:

```cpp
// File: tbb/src/ops_tbb.cpp (BatcherOddEvenMerge), ветка else
} else {
  for (size_t i = po; i < n - po; i += 2 * po) {
    for (size_t j = 0; j < po; ++j) {
      BlocksComparing(arr, i + j, i + j + po);
    }
  }
}
```

Сеть Бэтчера для длины `n_padded` имеет `log₂(n_padded)` уровней
компараторов. Для `n_padded = 2^{20} = 1 048 576` это 20 уровней.
Параллелится в TBB-версии **ровно один из них — самый первый**.
На последующих уровнях число блоков по `i` сокращается
геометрически: на уровне с шагом `po = n_padded / 4` блоков уже два,
на `po = n_padded / 8` — четыре, и так далее. Суммарное число
компараторов на всех «внутренних» уровнях равно `O(n_padded · log
n_padded / 2)` — формально это половина всей работы слияния.

Особенности:

- Первый уровень действительно содержит наибольший **отдельный** массив
  независимых компараторов (`n/2 = 5·10^5` на `n = 10^6`). Именно для
  него `auto_partitioner` имеет смысл — там много работы, которую можно
  крупно разделить.
- На более глубоких уровнях каждый отдельный «блок» из `po` компараторов
  имеет хорошую кэш-локальность (смежные индексы), и распараллеливание
  внутри блока ломало бы эту локальность; а внешнее распараллеливание
  по блокам даёт мало работы — например, на уровне `po = n/4` всего
  3 блока, что недостаточно для нагрузки 4–8 worker-ов.
- Половина всей работы слияния оказывается последовательной. Закон
  Амдала даёт верхнюю границу ускорения этой стадии в районе `~2`
  даже при бесконечном числе потоков.
- В OMP-версии та же раскладка распараллелена через `#pragma omp
  parallel for` по внешнему `i`, и это работает (число блоков на
  «средних» уровнях ещё достаточно велико — несколько десятков на
  уровнях около `po = 2^{14}…2^{17}`).

Это — место для будущей оптимизации (например, обернуть
внутренние уровни в `tbb::parallel_for` по блокам с явно заданным
`grainsize ≥ po`, либо использовать `parallel_for` с разделением на
«фазы»: при больших `po` параллелить по блокам, при малых — оставлять
последовательно). В текущей версии этот шаг сознательно не сделан;
численные следствия описаны в §8.

**Внутренние проходы LSD radix-sort** (counting, prefix-sum,
распределение по бакетам) также оставлены последовательными — по тем
же причинам, что и в OMP-версии (см. `omp/report.md`, §4.3).

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Класс `ZeninARadixSortDoubleBatcherMergeTBB` наследуется от `BaseTask` и
устанавливает `TypeOfTask::kTBB`. Pipeline-методы распределены так же,
как в SEQ/OMP: `ValidationImpl`, `PreProcessingImpl`,
`PostProcessingImpl` пустые и возвращают `true`; вся работа — в
`RunImpl`.

Изменения относительно OMP-версии **минимальны** и точечны:

1. `#pragma omp parallel sections` заменён на `tbb::parallel_invoke`
   с двумя лямбдами. Семантика идентична: обе задачи выполняются
   параллельно, управление возвращается после завершения обеих.
2. Первый уровень сети Бэтчера переведён с `#pragma omp parallel for`
   на `tbb::parallel_for(blocked_range, lambda)`. Семантика
   эквивалентна, но партиционирование диапазона теперь адаптивное
   (`auto_partitioner`), а не статическое.
3. Внутренние уровни сети Бэтчера возвращены к последовательной
   форме SEQ-версии (см. §4.3 и обсуждение последствий).
4. `LSDRadixSort`, `PackDouble`, `UnpackDouble`, `BlocksComparing`
   битово идентичны OMP-версии. Это гарантирует, что для одного и
   того же входа SEQ, OMP и TBB выдают побитово одинаковый
   результат — что и проверяется в общих функциональных тестах.

Граничный случай `n ≤ 1` обработан явным ранним выходом в начале
`RunImpl`, как и в OMP, что устраняет необходимость защищать
`parallel_invoke`/`parallel_for` от вырожденных диапазонов.

Потокобезопасность гарантируется тремя свойствами, которые
выполняются по построению:

- На уровне `parallel_invoke` — непересечение объектов памяти `left` и
  `right` (это разные `std::vector`, лежащие в разных областях кучи);
- Внутри `parallel_for` — непересечение пар индексов
  `(i, i + n/2)` для разных `i` (свойство первого уровня сети Бэтчера);
- Между уровнями `po` — блокирующий характер `parallel_for`
  гарантирует, что следующий уровень видит полностью обновлённый
  массив. Аналогично — `parallel_invoke` блокирует переход к стадии
  слияния.

## 6. Проверка корректности

Корректность TBB-версии проверяется тем же общим каркасом,
описанным в `seq/report.md`, §5 и `omp/report.md`, §6. Файл
`tests/functional/main.cpp` через `AddFuncTask<...TBB>` в общем
`tuple_cat` прогоняет те же 12 тестов из `kTestParam` (`Single`,
`Empty`, `ReverseTwo`, `Negative`, `Positive`, `Large`, `Small`,
`Sorted`, `DifferentSigns`, `Duplicates`, `ReverseSeven`,
`ZeroAndNegativeZero`) дополнительно и для TBB-задачи. Сравнение —
побитово через `!=`, без эпсилон-допуска; для radix-sort это
методически корректно (см. §5 `seq/report.md`).

Все 12 тестов проходят детерминированно. Особо важные для TBB-версии:

- `ReverseSeven` — `n = 7`, `n_padded = 8`, две половины по 4 элемента.
  Проверяет, что `parallel_invoke` действительно дожидается обоих
  `LSDRadixSort` перед запуском слияния (если бы блокировки не
  было, слияние получило бы частично отсортированные половины).
- `Duplicates` — повторяющиеся значения; проверяет, что строгое `>`
  в `BlocksComparing` и адаптивное разбиение `auto_partitioner` не
  приводят к лишним перестановкам равных элементов между чанками.
- `ZeroAndNegativeZero` — самый тонкий тест на корректность
  `PackDouble`/`UnpackDouble`; в TBB-версии эти функции битово
  идентичны SEQ-версии, и тест на это и опирается.

Дополнительно perf-тест (`tests/performance/main.cpp`,
`CheckTestOutputData`) сравнивает выход TBB на `n = 1 000 000` с
`std::ranges::sort` на том же входе — поэтому если бы существовала
гонка, проявляющаяся только на больших размерах, perf-тест упал бы
до того, как время было бы напечатано.

Устойчивость результата при разном числе worker-ов проверена через
`PPC_NUM_THREADS = 1, 2, 4, 8`: во всех случаях выход побитово
совпадает, что подтверждает отсутствие зависимости результата от
конкретного плана work-stealing.

## 8. Результаты

Замеры выполнялись для каждого режима (`task_run`, `pipeline_run`) при
числе worker-ов `p ∈ {1, 2, 4, 8}`. Время — медиана по серии повторов
perf-каркаса, переведённая из секунд в миллисекунды. Ускорение
`S(p) = T_SEQ / T_TBB(p)`, эффективность `E(p) = S(p) / p`.
Знаменатель `T_SEQ` — baseline из `seq/report.md`, §7.

Режим `task_run`:

| `p` (threads) | `T_TBB` (мс) | Speedup `S` | Efficiency `E` |
|--------------:|-------------:|------------:|---------------:|
|       1       |   `65.92`    |   `0.966`   |     `0.966`    |
|       2       |   `49.84`    |   `1.277`   |     `0.639`    |
|       4       |   `47.80`    |   `1.332`   |     `0.333`    |
|       8       |   `48.94`    |   `1.300`   |     `0.163`    |

Режим `pipeline_run`:

| `p` (threads) | `T_TBB` (мс) | Speedup `S` | Efficiency `E` |
|--------------:|-------------:|------------:|---------------:|
|       1       |    `64.66`   |   `1.054`   |    `1.054`     |
|       2       |    `50.10`   |   `1.360`   |    `0.680`     |
|       4       |    `48.31`   |   `1.410`   |    `0.352`     |
|       8       |    `48.51`   |   `1.404`   |    `0.176`     |

Размер `n` фиксирован: `1 000 000` для всех строк.

Ожидаемое поведение масштабируемости:

- **Переход `p: 1 → 2`** — основной выигрыш на стадии radix-sort через
  `parallel_invoke` (две независимые задачи распределяются на два
  worker-а). Аналогично OMP-версии.
- **Переход `p: 2 → 4`** — выигрыш только на первом уровне сети
  Бэтчера (`parallel_for` имеет достаточно работы для 4 worker-ов:
  500 000 итераций). Однако последующие 19 уровней сети остаются
  последовательными (§4.3), что **существенно ограничивает** общий
  выигрыш стадии слияния — в этом TBB-версия структурно проигрывает
  OMP-версии, где последовательная часть меньше.
- **Переход `p: 4 → 8`** — ожидается слабый рост или плато по тем же
  причинам: последовательная часть слияния доминирует, плюс
  накладывается memory-bound характер первого уровня (большие
  компараторы `(i, i + n/2)` промахиваются мимо L2/L3).

**Главное узкое место TBB-версии относительно OMP** — последовательные
внутренние уровни сети Бэтчера (см. §4.3). По закону Амдала, если
параллельная часть составляет долю `α` от полного времени, то
максимально достижимое ускорение при `p → ∞` равно `1 / (1 − α)`.
В TBB-версии параллельная часть — это два radix-sort-а
(~`O(n/2)` каждый) плюс один уровень сети Бэтчера из ~20. То есть `α`
заметно меньше, чем в OMP-версии, где параллельны все уровни сети.
Поэтому, по прогнозу, при `p ≥ 4` TBB-версия должна показывать
заметно меньший `speedup`, чем OMP-версия — и именно для подтверждения
этого тезиса полезны замеры.

Второстепенные узкие места, общие с OMP:

- `parallel_invoke` структурно ограничен двумя параллельными
  задачами; лишние worker-ы простаивают на стадии radix-sort.
- Memory-bound характер первого уровня Бэтчера: на `n = 10^6` массив
  занимает 8 МБ, чтения и записи попадают в L3/DRAM, что ставит
  верхний потолок ускорения независимо от числа потоков.

Положительная сторона TBB-runtime, которую стоит ожидать в численных
данных: благодаря work-stealing и `auto_partitioner`-у накладные
расходы на запуск параллельной задачи в TBB обычно ниже, чем на
форк/джойн команды потоков OpenMP. Это может проявиться на сравнении
**абсолютного времени** при `p = 2`: TBB-версия за счёт более
дешёвого `parallel_invoke` может оказаться чуть быстрее OMP-версии,
несмотря на меньшую долю параллельной работы в среднем.

## 9. Выводы

TBB-версия реализована минимально-инвазивно относительно SEQ/OMP:
ядро алгоритма не изменено, параллелизм добавлен через два
TBB-примитива — `parallel_invoke` для двух radix-sort-ов и
`parallel_for` поверх `blocked_range` для первого, наиболее «жирного»
уровня сети Бэтчера. Для обоих участков `grainsize` не задан, что
вместе с дефолтным `auto_partitioner` даёт адаптивное разбиение
диапазона — это структурно оправдано для нагрузки с однородными
итерациями (компараторы делают примерно одинаковую работу). Конкуренция
управляется через `PPC_NUM_THREADS` → `tbb::global_control` в каркасе
курса.

Главное методическое отличие от OMP-версии и одновременно главное
ограничение TBB-версии — **внутренние уровни сети Бэтчера оставлены
последовательными** (§4.3). Это сознательный выбор реализации, но он
имеет численные последствия: по закону Амдала достижимый speedup при
`p ≥ 4` ограничен сильнее, чем у OMP-версии, где параллельны все
уровни. Это направление для следующей итерации — обернуть внутренние
уровни в дополнительные `parallel_for` по блокам `i` (с явно заданным
`grainsize` порядка `po` или больше, чтобы избежать дробления внутри
кэш-локального блока).
