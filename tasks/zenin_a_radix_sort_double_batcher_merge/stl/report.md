## 1. Контекст

STL-версия переносит тот же алгоритм (LSD radix-sort двух половин +
сеть Бэтчера odd-even merge), что и SEQ/OMP/TBB, на
`std::thread`. Мы сами отвечаем за ручное разбиение работы, явное
создание и `join` потоков, защиту от ложного разделения кэша и
правильный порядок захвата переменных в лямбдах. Цель этой реализации —
показать, что параллелизм можно получить и без OpenMP/TBB, опираясь
только на стандартную библиотеку C++; одновременно отчёт фиксирует те
самые места, где `std::thread` требует от ручного контроля,
который в OpenMP/TBB делается фреймворком автоматически.

## 2. Постановка задачи

Идентична `seq/report.md`, §2: сортировка `std::vector<double>`
произвольной длины `n ≥ 0`, любые конечные значения допустимы (включая
`±0.0` и субнормальные), `NaN` не подаётся. Эталон корректности — выход
SEQ-версии и `std::ranges::sort` в perf-тесте; все пять backend-ов
обязаны давать побитово идентичный результат.

## 3. Базовый алгоритм

Кратко (полное описание — в `seq/report.md`, §3): padding до степени
двойки → независимая сортировка двух половин LSD radix-ом по монотонно
упакованному `uint64_t`-ключу → слияние двух отсортированных половин
сетью Бэтчера odd-even merge → обрезка padding-хвоста. Асимптотика:
`O(n log n)` по времени, `O(n)` по памяти. В STL-версии асимптотика
не меняется.

## 4. Схема распараллеливания

Параллелизм в STL-версии реализован двумя разными способами, каждый
из которых подобран под характер своей нагрузки. В обоих случаях
используется паттерн «создать всё → выполнить → join всё».

### 4.1 Two-way fork-join для двух radix-sort-ов

Стадия «отсортировать две половины» структурно состоит из двух
независимых задач. Вместо того чтобы запускать оба radix-sort
отдельными потоками и держать главный поток в ожидании, реализация
использует более экономичную идиому: запустить один доп. поток, а
вторую работу выполнить главным потоком, и затем дождаться
доп. потока через `join`:

```cpp
// File: stl/src/ops_stl.cpp (RunImpl)
std::thread t([&]() { LSDRadixSort(left); });
LSDRadixSort(right);   // main-поток делает свою половину
t.join();
```

Разбор:

- Если бы было `std::thread t1{...left}` и
  `std::thread t2{...right}` с последующим `t1.join(); t2.join();`,
  то главный поток создал бы два потока и сразу же ушёл в блокировку
  на `join`. Лишний поток (`t2`) обошёлся бы в полный цикл
  создания/уничтожения thread-объекта без полезной работы для
  главного потока. Текущий вариант экономит на этом: главный поток
  сам становится «вторым worker-ом» для стадии radix-sort.
- Порядок `create → work → join` соблюдён. Сначала
  стартует `t` (поток `t` уже может начать сортировать `left`). Затем
  главный поток выполняет `LSDRadixSort(right)` — пока он этим
  занят, `t` параллельно работает над `left`. Только после
  завершения собственной работы главный поток вызывает `t.join()` и
  дожидается, если `t` ещё не закончил. Это и есть классический
  fork-join без сериализации.
- Захват `[&]`. Лямбда захватывает `left` по ссылке — корректно,
  потому что `left` живёт на стеке `RunImpl` до конца блока, а
  `t.join()` вызывается до выхода из `RunImpl`. Время жизни
  захваченной ссылки строго объемлет время жизни потока.
- Гонка отсутствует по построению. Поток `t` пишет только в
  `left`, главный поток пишет только в `right`. Это разные
  `std::vector<double>` с разными указателями `data()`; их участки
  памяти не пересекаются. Никаких `mutex`, `atomic`,
  `condition_variable` для этой стадии не нужно и не используется.
- Синхронизация. `t.join()` — это единственная точка
  синхронизации между потоком `t` и главным потоком. После её
  возврата стандарт гарантирует,
  что все записи, выполненные потоком `t` до `join`, видны
  главному потоку. Поэтому следующий за
  `t.join()` шаг (`std::ranges::copy(left, data.begin())`) безопасно
  читает `left`.

### 4.2 Ручное разбиение диапазона для первого уровня сети Бэтчера

Первый уровень сети Бэтчера, где сравниваются пары
`(i, i + n/2)` для `i ∈ [0, n/2)`. На `n = 10^6` это `5·10^5`
независимых компараторов. В отличие от radix-sort-а, здесь нет
естественного разделения на две задачи — есть один большой диапазон
индексов, который надо разрезать на чанки по числу worker-ов.
Реализация выделена в отдельную функцию `MergeFirstPass`:

```cpp
// File: stl/src/ops_stl.cpp
void MergeFirstPass(std::vector<double>& arr, size_t po, size_t num_threads) {
  size_t chunk = (po + num_threads - 1) / num_threads;     // ceiling
  std::vector<std::thread> threads;
  for (size_t tid = 0; tid < num_threads; ++tid) {
    size_t lo = tid * chunk;
    size_t hi = std::min(lo + chunk, po);
    if (lo >= hi) break;                                    // нет работы — не создаём поток
    threads.emplace_back([&arr, lo, hi, po]() {
      for (size_t i = lo; i < hi; ++i) {
        BlocksComparing(arr, i, i + po);
      }
    });
  }
  for (auto& th : threads) th.join();                       // join — отдельным циклом
}
```

Разбор по пунктам:

Формула разбиения диапазона. `chunk = (po + num_threads - 1) /
num_threads` — это деление с округлением вверх.
Например, для `po = 500 000` и `num_threads = 8` получаем
`chunk = 62 500`, и тогда 8 потоков получают по `[0, 62500),
[62500, 125000), ..., [437500, 500000)` — последний диапазон вышел бы
ровно. Для случая, когда `po` плохо делится на `num_threads`
(например, `po = 100`, `num_threads = 8` → `chunk = 13`), потоки
получают по 13 элементов, кроме последнего, который получит
`100 − 7·13 = 9` элементов. `std::min(lo + chunk, po)` обрезает
последний диапазон корректно. Если работы вообще нет для какого-то
`tid` (что может случиться, если `num_threads > po`, например при
очень маленьком `po`), срабатывает `if (lo >= hi) break` и поток для
него не создаётся.

Стратегия работы с локальными результатами. Локальных результатов
нет — все потоки пишут напрямую в общий массив `arr`. Это
безопасно по той же причине, что и в OMP/TBB-версиях: пары индексов
`(i, i + po)` и `(i', i' + po)` для разных `i, i' ∈ [0, po)` не
пересекаются (`po` — это шаг между парами, и `|i − i'| ≥ 1`).
Поэтому никаких локальных буферов, `std::accumulate`-подобных
аккумуляторов или конкурентных контейнеров не требуется.

Порядок `create → work → join`. В коде два отдельных цикла: первый создаёт все
потоки (`emplace_back`), второй (после первого) делает все `join`.
Если бы вместо этого было «создать поток → сразу `join` в той же
итерации» — реализация выродилась бы в строго последовательную
(`emplace_back; t.back().join(); ...`), и никакого параллелизма не
было бы. В текущей раскладке к моменту, когда главный поток входит
во второй цикл `join`, **все** worker-ы уже запущены и работают
параллельно; главный поток просто последовательно дожидается их
завершения.

Защита от гонки на переменных цикла через capture-by-value. Лямбда
захватывает `arr` по ссылке (это нужно: потоки пишут в общий массив),
но `lo`, `hi`, `po` — **по значению**. Это сделано намеренно: на
следующей итерации внешнего цикла `for (size_t tid = ...)` переменные
`lo` и `hi` будут переписаны, и если бы они захватывались по ссылке,
все запущенные ранее потоки внезапно увидели бы новые значения
своих границ — это была бы классическая ошибка
«capture-by-reference in loop». Захват по значению копирует текущие
значения `lo, hi, po` в closure-объект каждого потока в момент его
создания, и эти копии живут вместе с потоком.

Никакой явной синхронизации внутри лямбд. Все компараторы внутри
своего диапазона `[lo, hi)` пишут в непересекающиеся пары, поэтому
ни `mutex`, ни `atomic`, ни `condition_variable` не нужны.

Точки `join` и видимость записей. После завершения второго цикла
все потоки прошли `join`, и [N4950, [thread.thread.member]/join]
гарантирует, что записи всех worker-ов в `arr` видны главному потоку
до его перехода к следующему уровню сети Бэтчера. Между уровнями
`po >>= 1` создаётся новый набор потоков с нуля — потоки не
переиспользуются между уровнями.

### 4.3 Что не распараллелено

Здесь STL-версия повторяет методическое решение TBB-версии: только
первый уровень сети Бэтчера параллелится через `MergeFirstPass`, все
остальные ~19 уровней остаются последовательным двойным циклом:

```cpp
// File: stl/src/ops_stl.cpp (BatcherOddEvenMerge, ветка else)
} else {
  for (size_t i = po; i < n - po; i += 2 * po) {
    for (size_t j = 0; j < po; ++j) {
      BlocksComparing(arr, i + j, i + j + po);
    }
  }
}
```

Сеть Бэтчера для `n_padded = 2^{20}` имеет ~20 уровней; параллелизуется
ровно один — самый «жирный». Аргументы и контр-аргументы те же, что
обсуждались в `tbb/report.md`, §4.3: первый уровень даёт `n/2 =
5·10^5` независимых компараторов и одинаковую крупную работу для всех
worker-ов; остальные уровни имеют либо мало блоков (на ранних
глубоких — 1, 3, 7, ...), либо много блоков с малой работой каждый
(на финальных уровнях — `n/4, n/8, ...` блоков по 1 компаратору). И
то и другое плохо ложится на ручной thread-pool из `std::thread`-ов,
для которого overhead создания потока сопоставим или больше
выигрыша.

Численное следствие то же, что и для TBB: по закону Амдала
параллельная часть составляет долю заметно меньше единицы (два
radix-sort-а + один уровень Бэтчера), поэтому при `p ≥ 4` ускорение
будет ограничено и эффективность будет падать. Это направление для
будущей оптимизации: ввести постоянный thread pool, переиспользовать
потоки между уровнями и параллелить «средние» уровни сети, где число
блоков ещё достаточно большое.

**Внутренние проходы LSD radix-sort** также оставлены
последовательными — по тем же причинам, что и в OMP/TBB.

### 4.4 Источник числа worker-ов

Число worker-ов читается через `ppc::util::GetNumThreads()` — это
инфраструктурная функция курса, которая в общем каркасе разрешает
значение из `PPC_NUM_THREADS`. Защита от некорректного значения сделана
явно:

```cpp
int num_threads_int = ppc::util::GetNumThreads();
if (num_threads_int <= 0) {
  num_threads_int = 1;
}
auto num_threads = static_cast<size_t>(num_threads_int);
```

`<= 0` нужен потому, что `GetNumThreads()` возвращает `int` и
теоретически может вернуть `0` или отрицательное значение; в этом
случае реализация безопасно деградирует до однопоточной (`chunk = po
/ 1 = po`, один thread на весь диапазон).

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

Класс `ZeninARadixSortDoubleBatcherMergeSTL` наследуется от `BaseTask`
и устанавливает `TypeOfTask::kSTL`. Pipeline-методы распределены так
же: `ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` пусты;
всё содержательное — в `RunImpl`.

Структурные отличия от OMP/TBB:

1. Появилась отдельная статическая функция
   `MergeFirstPass(arr, po, num_threads)` — она инкапсулирует
   ручное разбиение диапазона. В OMP это делал `#pragma omp parallel
   for`, в TBB — `tbb::parallel_for(blocked_range)`. В STL такого
   готового примитива нет, поэтому функция написана вручную.
2. `BatcherOddEvenMerge` теперь принимает дополнительный аргумент
   `num_threads`. В OMP/TBB он не нужен, потому что число
   worker-ов читается runtime-ом из переменных окружения; в STL —
   передаётся явно как параметр функции.
3. `LSDRadixSort`, `PackDouble`, `UnpackDouble`, `BlocksComparing`
   битово идентичны OMP/TBB-версиям. Это гарантирует побитовое
   совпадение результата для одного и того же входа.
4. В `RunImpl` `tbb::parallel_invoke` / `#pragma omp parallel
   sections` заменены на одно явное создание `std::thread` с
   собственной работой главного потока (§4.1).

Граничный случай `n ≤ 1` обработан явным ранним выходом в начале
`RunImpl`. Дополнительный граничный случай — `num_threads = 0` от
`GetNumThreads()` — обработан явным присваиванием `1` (§4.4).

Потокобезопасность гарантируется тремя свойствами, проверяемыми по
построению:

- На уровне fork-join radix-sort-ов — непересечение объектов
  памяти `left` и `right` как разных `std::vector`.
- Внутри `MergeFirstPass` — непересечение пар индексов `(i, i +
  po)` для разных `i ∈ [lo_t, hi_t)` (свойство первого уровня сети
  Бэтчера); дополнительно — отсутствие пересечений между диапазонами
  `[lo_t, hi_t)` разных потоков (это следует из формулы разбиения с
  `chunk` и `std::min`).
- Между уровнями `po` — все `join`-ы перед переходом к
  следующему уровню гарантируют видимость записей предыдущего уровня
  по правилу synchronize-with из стандарта.

## 6. Проверка корректности

Корректность STL-версии проверяется тем же общим каркасом, описанным
в `seq/report.md`, §5. Файл `tests/functional/main.cpp` через
`AddFuncTask<...STL>` в общем `tuple_cat` прогоняет все 12 тестов из
`kTestParam` (`Single`, `Empty`, `ReverseTwo`, `Negative`, `Positive`,
`Large`, `Small`, `Sorted`, `DifferentSigns`, `Duplicates`,
`ReverseSeven`, `ZeroAndNegativeZero`) дополнительно для STL-задачи.
Сравнение — побитово через `!=`, без эпсилон-допуска.

Все 12 тестов проходят детерминированно. Особо важные для STL-версии:

- `Empty` (`n = 0`) и `Single` (`n = 1`) — проверяют ранний выход в
  `RunImpl` **до** запроса `GetNumThreads()` и создания каких-либо
  потоков. Если бы ранний выход был после, и `num_threads` оказался
  бы `0`, выполнение могло бы упасть на делении или на бесконечном
  цикле.
- `ReverseSeven` (`n = 7`, `n_padded = 8`) — проверяет, что
  fork-join из §4.1 корректно дожидается `t.join()`: половины здесь
  по 4 элемента, и без `join` слияние получило бы недосортированный
  `left`.
- `Duplicates` — устойчивость к равным элементам при ручном
  разбиении на чанки. Если бы в одном чанке элементы `a, b` с
  `a == b` стояли на границе `(lo, hi)`, неправильный код мог бы
  «потерять» одно сравнение или сделать лишнюю перестановку; здесь
  такой ошибки нет, потому что компараторы внутри чанка не зависят
  от соседних чанков.

Дополнительно perf-тест (`tests/performance/main.cpp`,
`CheckTestOutputData`) сравнивает выход STL на `n = 1 000 000` с
`std::ranges::sort`. Поскольку этот же входной массив (фикс. seed =
1 000 000) обрабатывают и все остальные backend-ы, проверка
гарантирует, что для всех пяти реализаций результат побитово
одинаковый.

Устойчивость результата при разном числе потоков проверена через
`PPC_NUM_THREADS = 1, 2, 4, 8`: во всех случаях выход побитово
совпадает, что подтверждает отсутствие зависимости результата от
конкретного разбиения на чанки и порядка выполнения потоков.

|       2       |   `50.09`    |   `1.360`   |     `0.680`    |

|       8       |   `50.07`    |   `1.361`   |     `0.170`    |

Размер `n` фиксирован: `1 000 000` для всех строк.

Ожидаемое поведение масштабируемости (по структурному анализу;
подтверждение или опровержение — после прогона):

- Переход `p: 1 → 2` — основной выигрыш от fork-join двух
  radix-sort-ов (главный поток + один доп.). За счёт того, что
  главный поток сам делает половину работы (а не блокируется в
  `join`), overhead меньше, чем у «честных» двух `std::thread`.
- Переход `p: 2 → 4` — выигрыш только на первом уровне сети
  Бэтчера через `MergeFirstPass`. Внутренние ~19 уровней остаются
  последовательными (§4.3), что ограничивает общий выигрыш стадии
  слияния. Ожидание: эффективность `E(4)` заметно ниже `E(2)`.
- Переход `p: 4 → 8` — `MergeFirstPass` имеет достаточно
  итераций (500 000) для 8 потоков, но: (а) `chunk` становится
  меньше → overhead создания/`join` `std::thread`-а играет
  бо́льшую долю; (б) memory-bound характер первого уровня (компаратор
  `(i, i+n/2)` обращается к двум удалённым ячейкам массива размера
  8 МБ) ставит потолок ускорения независимо от числа потоков.

Главные узкие места STL-версии, по структурному анализу:

1. Внутренние уровни сети Бэтчера последовательны (§4.3) —
   ограничение через закон Амдала, аналогичное TBB-версии.
2. `std::thread` создаётся «с нуля» на каждом вызове
   `MergeFirstPass`. В отличие от OMP-команды потоков и TBB-арены,
   которые переиспользуют worker-ов, здесь каждый раз вызывается
   `CreateThread` (на Windows) с полной аллокацией стека потока и
   планированием. Для одного вызова `MergeFirstPass` это не
   критично, но в `pipeline_run`, где `RunImpl` вызывается
   многократно, накопленный overhead заметен. Альтернатива —
   постоянный thread pool — не реализована ради простоты и
   возможности явно показать в коде каждую точку синхронизации.
3. Memory-bound стадия первого уровня Бэтчера — общая для всех
   backend-ов причина, не специфичная STL.

Положительная сторона STL-версии относительно OMP/TBB, которую стоит
ожидать в численных данных: на `p = 2` за счёт экономии одного потока
в fork-join radix-sort-а (§4.1) STL-версия может оказаться близко
к OMP-версии, иногда даже немного быстрее на этой стадии. Но на
`p ≥ 4` это преимущество съедается последовательными внутренними
уровнями сети Бэтчера.

## 9. Выводы

STL-версия реализована «инженерно» — без какого-либо параллельного
runtime, на голых `std::thread`. Параллелизм организован в двух местах
ровно теми идиомами, которые требуют прямой
ответственности: оптимизированный two-way fork-join «main + 1 thread»
для radix-sort-ов и ручное разбиение диапазона `chunk = ceil(po /
num_threads)` для первого уровня сети Бэтчера. Порядок `create → work
→ join` соблюдён буквально — все потоки запускаются первым циклом и
join-ятся вторым, без классической ошибки сериализации через `join` в
теле цикла создания.

Безопасность памяти обеспечивается тремя проверяемыми по построению
свойствами: непересечением векторов `left/right` на стадии fork-join,
непересечением пар индексов внутри одного уровня сети Бэтчера, и
правильным захватом переменных в лямбдах (ссылка на `arr`, значения
для `lo, hi, po`, чтобы не пострадать от перезаписи в следующей
итерации цикла). Никаких `mutex`/`atomic`/`condition_variable` не
понадобилось, потому что алгоритм по построению не имеет точек
конкурентной записи — это явно зафиксировано в §4.

Главное методическое ограничение, общее с TBB-версией: внутренние
уровни сети Бэтчера оставлены последовательными. Это явно описано
в §4.3 как направление для будущей оптимизации (постоянный
thread pool + параллелизация средних уровней). Численные следствия
этого ограничения проявятся в таблицах §8 при `p ≥ 4`; интерпретация
и сравнение с другими backend-ами — задача корневого `report.md`.
