## 1. Контекст

ALL-версия — гибридная реализация, объединяющая два уровня
параллелизма. Внешний (межпроцессный) уровень обеспечивает MPI:
рабочая нагрузка распределяется по `num_procs` процессам через
`MPI_Scatter`, собирается обратно через `MPI_Gather` и финально
рассылается всем через `MPI_Bcast`. Внутренний (внутрипроцессный)
уровень обеспечивает oneTBB: внутри каждого процесса локальная
сортировка чанка использует `tbb::parallel_invoke` для двух
независимых radix-sort-ов половин чанка.

Мотивация для гибридной схемы: padding до степени
двойки удобно делится на `num_procs` равных чанков, локальная
сортировка чанка структурно делится надвое (это то, что мы уже делали
в SEQ/OMP/TBB/STL), а финальное слияние выполняется сетью Бэтчера
последовательными попарными merge-операциями. Полученная конфигурация
`ranks × threads` позволяет, в идеале, использовать как
многопроцессорные узлы кластера (через MPI), так и многоядерность
каждого узла (через TBB) одновременно.

## 2. Постановка задачи

Идентична `seq/report.md`, §2. Дополнение, специфичное для гибридной
версии: входные данные доступны только на процессе с `rank == 0`.
Все остальные процессы
получают свою порцию работы через коллективные операции MPI и
возвращают результат на rank 0; после `RunImpl` финальный
отсортированный массив должен быть доступен на всех процессах
(это требование `CheckTestOutputData`, который вызывается каждым
процессом и сравнивает свой `GetOutput()` с эталоном).

## 3. Базовый алгоритм

Структура остаётся прежней, но в гибридной версии она перестраивается
под две оси параллелизма:

1. **Padding до степени двойки**, делящейся на `num_procs`. В отличие
   от SEQ/OMP/TBB/STL, где `pow2` — это просто ближайшая степень
   двойки сверху, здесь к этому условию добавляется требование
   делимости на число процессов (см. §4.1, шаг 2).
2. **Распределение `num_procs` отсортированных чанков** размера
   `chunk_size = pow2 / num_procs` через `MPI_Scatter`. Каждый процесс
   получает свой чанк.
3. **Локальная сортировка чанка** через `SortChunk`. Внутри —
   деление чанка на две половины, `tbb::parallel_invoke` для двух
   LSD radix-sort-ов, сборка обратно и одно слияние Бэтчера.
4. **Сбор отсортированных чанков** через `MPI_Gather` на rank 0.
5. **Финальное попарное слияние** соседних чанков сетью Бэтчера —
   `FinalMerge`, выполняется последовательно на rank 0.
6. **Рассылка финального результата** всем процессам через
   `MPI_Bcast`.

Асимптотика по работе остаётся `O(n log n)` (та же, что и для всех
предыдущих версий). Асимптотика по времени при идеальном
распараллеливании: `O(n / num_procs · log(chunk_size))` на стадии
локальной сортировки + `O(n log num_procs)` на стадии финального
слияния (последовательное на rank 0) + накладные расходы на MPI-
коммуникации `O(n)`.

## 4. Схема распараллеливания

ALL-версия — единственная из пяти реализаций, где параллелизм
организован **на двух уровнях одновременно**, и поэтому требует
отдельного описания каждого уровня и точек их стыковки.

### 4.1 MPI-уровень

Все процессы симметричны в том смысле, что каждый вызывает
`RunImpl()` и проходит через одну и ту же последовательность
коллективных операций. При этом rank 0 играет особую роль:

- именно на rank 0 (по конвенции perf-каркаса) приходит исходный
  вход через `GetInput()` → `local_data_`;
- именно rank 0 владеет полным padded-массивом размера `pow2` до
  scatter-а и после gather-а;
- именно rank 0 выполняет последовательный `FinalMerge` после gather-а.

Полный конвейер MPI-операций выглядит так.

**Шаг 1. Рассылка размера задачи.** Только rank 0 знает
`original_size`, поэтому первое действие — пробросить это значение
всем:

```cpp
size_t original_size = 0;
if (rank == 0) original_size = local_data_.size();
MPI_Bcast(&original_size, 1, MPI_UNSIGNED_LONG, 0, MPI_COMM_WORLD);
if (original_size == 0) return true;
```

После этого все процессы знают размер и могут принять единое решение
по последующим шагам, без рассинхронизации. Если массив пуст —
выход без коммуникаций (`return true` на всех процессах одновременно).

**Шаг 2. Fallback при `num_procs`, не являющемся степенью двойки.**
Сеть Бэтчера odd-even merge корректна именно для длин,
являющихся степенями двойки, и финальное слияние `FinalMerge`
устроено как набор попарных merge-ов `2·chunk_size, 4·chunk_size,
8·chunk_size, …`, что требует, чтобы число чанков было степенью
двойки (`num_procs` ∈ {1, 2, 4, 8, …}). Если это не так, реализация
**сознательно отказывается от распараллеливания** и переходит к
простому fallback-сценарию:

```cpp
if (!procs_is_pow2) {
  if (rank == 0) LSDRadixSort(local_data_);
  if (rank != 0) local_data_.resize(original_size);
  MPI_Bcast(local_data_.data(), static_cast<int>(original_size),
            MPI_DOUBLE, 0, MPI_COMM_WORLD);
  return true;
}
```

Вместо того чтобы изобретать
нерегулярные сети слияния для `num_procs ∈ {3, 5, 6, 7}` и рисковать
корректностью, реализация явно деградирует до однопроцессорной
сортировки на rank 0, после чего рассылает результат всем. Цена —
полная потеря параллелизма для не-степеней двойки. Это явно влияет
на интерпретацию результатов: при `num_procs = 3` ускорение
относительно SEQ должно быть около единицы (даже хуже, из-за
дополнительного `MPI_Bcast` на 1 млн элементов), и так и должно быть
по построению.

**Шаг 3. Согласование `pow2`, делящегося на `num_procs`.** При
поддерживаемом числе процессов вычисляется минимальное `pow2 ≥
original_size`, такое что `pow2 % num_procs == 0`. Поскольку
`num_procs` сам степень двойки, последнее условие эквивалентно
`pow2 ≥ num_procs`, и второй цикл `while (pow2 % num_procs != 0)
pow2 <<= 1` срабатывает только если `num_procs > pow2`. Это
гарантирует, что `chunk_size = pow2 / num_procs` тоже **степень
двойки**, что критически важно для корректности сети Бэтчера на
шагах 5–6:

```cpp
size_t pow2 = 1;
while (pow2 < original_size) pow2 <<= 1;
while (pow2 % static_cast<size_t>(num_procs) != 0) pow2 <<= 1;
MPI_Bcast(&pow2, 1, MPI_UNSIGNED_LONG, 0, MPI_COMM_WORLD);
```

Это значение рассылается всем процессам через `Bcast`, чтобы все
участники последующего `Scatter` знали размер чанка.

**Шаг 4. Распределение работы через `MPI_Scatter`.** Rank 0 расширяет
свой `local_data_` до размера `pow2`, заполняя хвост
`std::numeric_limits<double>::max()` (тот же padding, что в SEQ —
гарантированно уходит в конец после сортировки). Затем `MPI_Scatter`
раздаёт каждому процессу свой непрерывный чанк размера `chunk_size`:

```cpp
MPI_Scatter(local_data_.data(), chunk_size, MPI_DOUBLE,
            chunk.data(),       chunk_size, MPI_DOUBLE,
            0, MPI_COMM_WORLD);
```

Это коллективная блокирующая операция: после её возврата у каждого
процесса в `chunk` лежит его кусок данных. Никаких явных
`MPI_Barrier` после `Scatter` не нужно — `Scatter` сам по себе
является точкой синхронизации.

**Шаг 5. Локальная сортировка чанка** — описана в §4.2 (TBB-уровень).

**Шаг 6. Сбор результатов через `MPI_Gather`.** После того как каждый
процесс отсортировал свой чанк, rank 0 собирает их обратно в
`local_data_`:

```cpp
MPI_Gather(chunk.data(),        chunk_size, MPI_DOUBLE,
           local_data_.data(),  chunk_size, MPI_DOUBLE,
           0, MPI_COMM_WORLD);
```

После `Gather` на rank 0 лежит массив длины `pow2`, состоящий из
`num_procs` подряд идущих **отсортированных** блоков длины
`chunk_size`. Этот массив пока **не** глобально отсортирован — он
лишь «локально упорядочен по чанкам».

**Шаг 7. Финальное слияние `FinalMerge` (только на rank 0).**
Реализация выполняет каскад попарных Batcher-merge-ов соседних
блоков:

```cpp
// File: all/src/ops_all.cpp (FinalMerge)
for (size_t size = chunk_size; size < pow2; size *= 2) {
  size_t merges_count = pow2 / (size * 2);
  for (size_t i = 0; i < merges_count; ++i) {
    size_t lo = i * 2 * size;
    std::vector<double> block(local_data_.begin() + lo,
                              local_data_.begin() + lo + (2 * size));
    BatcherOddEvenMerge(block, 2 * size);
    std::ranges::copy(block, local_data_.begin() + lo);
  }
}
```

На каждой итерации внешнего цикла `size` удваивается: сначала
сливаются пары чанков размера `chunk_size` в блоки `2·chunk_size`,
потом эти блоки попарно сливаются в блоки `4·chunk_size`, и так
далее до полного покрытия `pow2`. Внешний цикл делает
`log₂(num_procs)` итераций. Корректность каждого `BatcherOddEvenMerge`
гарантирована тем, что: (а) оба «полу-блока» уже отсортированы
(по индукции — на первом уровне это чанки после `SortChunk`, на
последующих — результаты предыдущих слияний); (б) их длина
`size` — степень двойки (по построению `pow2` и `num_procs`).

**Это — главное узкое место ALL-версии.** `FinalMerge`
выполняется **последовательно одним процессом** (rank 0), пока все
остальные `num_procs − 1` процессов простаивают. Полное время
финального слияния — `O(n · log num_procs)` в работе. По мере роста
`num_procs` доля этой последовательной стадии **растёт**, что
напрямую ограничивает масштабируемость по закону Амдала
(подробнее — в §8).

**Шаг 8. Рассылка результата через `MPI_Bcast`.** После того как rank
0 закончил `FinalMerge` и обрезал массив до `original_size`, финальный
отсортированный массив рассылается всем процессам:

```cpp
if (rank != 0) local_data_.resize(original_size);
MPI_Bcast(local_data_.data(), static_cast<int>(original_size),
          MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

Это нужно потому, что perf-каркас курса вызывает
`CheckTestOutputData` на **каждом** процессе после `RunImpl`, и
каждый процесс должен иметь корректный отсортированный массив. Цена
этого шага — пересылка `n · sizeof(double) · num_procs` байт через
сеть/shared memory MPI (для `n = 10^6` это ~8 МБ × `num_procs`).

### 4.2 TBB-уровень (внутрипроцессный)

Внутри каждого процесса локальная сортировка чанка делается через
`SortChunk`. Здесь применён тот же приём, что и в чистой TBB-версии:
чанк делится пополам, две половины сортируются параллельно через
`tbb::parallel_invoke`, после чего объединяются одним вызовом сети
Бэтчера:

```cpp
// File: all/src/ops_all.cpp (SortChunk)
void SortChunk(std::vector<double>& chunk, size_t chunk_size) {
  size_t half = chunk_size / 2;
  if (half > 0) {
    std::vector<double> left(chunk.begin(), chunk.begin() + half);
    std::vector<double> right(chunk.begin() + half, chunk.end());
    tbb::parallel_invoke(
        [&]() { LSDRadixSort(left); },
        [&]() { LSDRadixSort(right); });
    std::ranges::copy(left,  chunk.begin());
    std::ranges::copy(right, chunk.begin() + half);
    BatcherOddEvenMerge(chunk, chunk_size);
  } else {
    LSDRadixSort(chunk);   // chunk_size == 1: тривиально
  }
}
```

Разбор:

- `parallel_invoke` запускает обе LSD-сортировки как независимые
  задачи TBB-runtime, и блокирует вызывающий поток до их
  завершения. Внутри одного MPI-процесса worker-потоки TBB
  работают в общей памяти и могут безопасно писать в `left`/`right`,
  поскольку эти векторы не пересекаются.
- `BatcherOddEvenMerge` внутри чанка — **последовательный** (см.
  `seq/report.md`, §3.4). В отличие от чистой TBB-версии, здесь нет
  `parallel_for` для первого уровня сети Бэтчера. Это сознательное
  упрощение: для типичного размера чанка `chunk_size = pow2 /
  num_procs (при`num_procs = 4` это ~256 K элементов) основная
  доля работы уходит на radix-sort двух половин, а слияние Бэтчера
  составляет меньшую часть.

### 4.3 Что не распараллелено

Сознательно последовательны:

- **`FinalMerge` на rank 0** (см. §4.1, шаг 7). Это самое
  значимое последовательное место, ограничивающее ускорение при
  росте `num_procs`.
- **`BatcherOddEvenMerge` внутри `SortChunk`** (см. §4.2). На уровне
  одного процесса.
- **Внутренние проходы LSD radix-sort** — как и во всех остальных
  версиях.

## 5. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.

Класс `ZeninARadixSortDoubleBatcherMergeALL` наследуется от `BaseTask`
и устанавливает `TypeOfTask::kALL`. Структура pipeline-методов
**меняется** относительно других backend-ов:

- `ValidationImpl` возвращает `true` без проверок (как и везде).
- `PreProcessingImpl` теперь не пустой: `local_data_ = GetInput()`.
  Это сделано для того, чтобы рабочий буфер был полем класса, а не
  локальной переменной `RunImpl`, — потому что между процессами он
  ведёт себя по-разному (rank 0 загружает полный массив, остальные
  получают чанк), и удобнее держать его как `local_data_`.
- `RunImpl` выполняет всю MPI-логику (§4.1).
- `PostProcessingImpl` теперь тоже не пустой: `GetOutput() =
  local_data_`. После `MPI_Bcast` на шаге 8 в `local_data_` каждого
  процесса лежит финальный отсортированный массив; `PostProcessingImpl`
  переносит его в выходное поле задачи.

`LSDRadixSort`, `PackDouble`, `UnpackDouble`, `BlocksComparing`,
`BatcherOddEvenMerge` **битово идентичны** SEQ-версии (а не OMP/TBB-
вариантам с одиночным компаратором — здесь восстановлен SEQ-вариант
с циклом внутри `BlocksComparing`). Это даёт побитово одинаковый
результат сортировки на каждом отдельном процессе.

Граничные случаи обработаны следующим образом:

- **`original_size == 0`** — ранний выход на всех процессах сразу
  после `MPI_Bcast(&original_size, ...)`. Никаких scatter/gather/bcast
  данных, никаких аллокаций.
- **`num_procs == 1`** — `procs_is_pow2 = (num_procs > 1) && ...`
  возвращает `false` (т.к. `num_procs > 1` ложно), и срабатывает
  fallback-ветка из §4.1, шаг 2. То есть при единственном процессе
  алгоритм сводится к чистому `LSDRadixSort(local_data_)` без всякой
  межпроцессной коммуникации (Bcast на единственный процесс —
  no-op).
- **`num_procs` не степень двойки** — fallback на rank 0 + Bcast
  результата.

Потокобезопасность гарантируется тремя уровнями свойств:

- **На MPI-уровне** — все коллективные операции (`Bcast`, `Scatter`,
  `Gather`) блокирующие, и каждый процесс работает только со своими
  локальными буферами между ними. Гонок между процессами нет по
  построению.
- **На TBB-уровне внутри `SortChunk`** — `parallel_invoke` пишет в
  непересекающиеся векторы `left`/`right`; гонок внутри процесса нет.
- **Между MPI-операциями и TBB-операциями** — последовательны по
  потоку управления (`Scatter → SortChunk → Gather`). TBB-задачи
  запускаются только между `Scatter` и `Gather`, никакой MPI-вызов
  параллельно с TBB-задачей не выполняется. Это снимает необходимость
  использовать `MPI_THREAD_MULTIPLE`.

## 6. Проверка корректности

Корректность ALL-версии проверяется тем же общим каркасом курса. Файл
`tests/functional/main.cpp` через `AddFuncTask<...ALL>` в общем
`tuple_cat` прогоняет все 12 тестов из `kTestParam` (`Single`,
`Empty`, `ReverseTwo`, `Negative`, `Positive`, `Large`, `Small`,
`Sorted`, `DifferentSigns`, `Duplicates`, `ReverseSeven`,
`ZeroAndNegativeZero`). Поскольку `CheckTestOutputData` вызывается
на каждом MPI-процессе, проверка фактически означает: после
`MPI_Bcast` шага 8 **каждый** процесс имеет побитово одинаковый
корректный результат.

Особо важные тесты для ALL-версии:

- `Empty` (`n = 0`) — проверяет ранний выход на всех процессах
  одновременно после `MPI_Bcast(&original_size)`. Если бы хотя бы
  один процесс пропустил выход, последующий `MPI_Scatter` подвис бы
  навсегда (deadlock).
- `Single` (`n = 1`) — проверяет работу fallback-ветки
  (`procs_is_pow2 == false` при `num_procs = 1` или `num_procs > 1`,
  не являющемся степенью двойки): rank 0 сортирует тривиальный
  массив, остальные просто принимают `Bcast`.
- `ReverseSeven` (`n = 7`) — самый нетривиальный размер.
  Padding-логика: при `num_procs = 2` нужна `pow2 ≥ 7` и `pow2 % 2 ==
  0`, минимально `pow2 = 8`. Каждый процесс получает чанк длины 4,
  сортирует его, отдаёт. На rank 0 после Gather лежат два
  отсортированных чанка по 4 элемента; `FinalMerge` сливает их в
  один блок длины 8, рассылает всем, обрезает до 7.
- `Duplicates` — устойчивость в `FinalMerge`: повторяющиеся значения
  могут оказаться в одном чанке или быть разбиты между чанками после
  Scatter, и сеть Бэтчера должна корректно их слить.
- `ZeroAndNegativeZero` — корректность `PackDouble`/`UnpackDouble`
  при работе на каждом из процессов независимо. Эти функции
  битово идентичны SEQ-версии, поэтому корректность наследуется.

Дополнительно perf-тест на `n = 1 000 000` сравнивает выход ALL с
`std::ranges::sort`. Поскольку проверка делается на каждом процессе,
этот тест одновременно проверяет: (а) корректность алгоритма на
большом размере; (б) что финальный `MPI_Bcast` действительно
рассылает корректные данные всем участникам коммуникатора.

Устойчивость результата при разных конфигурациях `ranks × threads`
проверена прогоном с `(num_procs, PPC_NUM_THREADS) ∈ {(1, 1), (1, 2),
(1, 4), (2, 1), (2, 2), (2, 4), (4, 1), (4, 2)}`: во всех случаях
выход побитово совпадает.

## 7. Экспериментальная среда

Замеры выполнялись на следующей конфигурации:

- OS: Windows `<10>`, архитектура x86_64
- CPU: `<Intel(R) Core(TM) i5-10400F CPU @ 2.90GHz>`
- RAM: `<32 GB>`
- Компилятор: MSVC, стандарт `/std:c++20`
- Build type: `Release` (`/O2`), без санитайзеров
- CMake: `<3.30.4>`, генератор `<Visual Studio 17 2022>`

Число потоков задаётся переменной окружения `PPC_NUM_THREADS`. Замеры
проводились для значений `1, 2, 4, 8` потоков.

Конфигурация параллелизма в гибридной версии задаётся **двумя
параметрами**:

- `num_proc` — число MPI-процессов, задаётся аргументом
  `mpiexec -n <num_procs>`.
- `PPC_NUM_THREADS` — число TBB-worker-ов **внутри каждого процесса**,
  задаётся переменной окружения.

Полное число «работающих единиц» — это **произведение**
`workers_total = PPC_NUM_THREADS × PPC_NUM_THREADS`, при условии, что
суммарно оно не превышает число логических потоков машины (иначе
процессы и потоки начнут конкурировать за CPU и эффективность
обрушится).

Меры стабилизации — те же, что для предыдущих версий. Дополнительно
для гибридной версии важно: при `num_procs > 1` все процессы запускаются
**на одной машине** (нет кластера), поэтому MPI-коммуникации идут
через shared memory backend MS-MPI, что заметно дешевле сети, но всё
равно дороже прямого доступа к памяти внутри одного процесса.

Команды запуска (PowerShell):

```powershell
# Функциональные тесты ALL
python scripts/run_tests.py --running-type processes --counts <число процессов>"
 
# Тесты производительности ALL: 4 процесса × 2 TBB-потока на процесс
$env:PPC_NUM_THREADS = "2"
python scripts/run_tests.py --running-type performance --counts 4
```

Параметры perf-нагрузки те же: `n = 1 000 000` элементов `double`,
равномерное распределение на `[-1e6, 1e6]`, фиксированный seed =
1 000 000.

## 8. Результаты

Замеры выполнялись для каждого режима (`task_run`, `pipeline_run`)
при различных конфигурациях `ranks × threads`. Время — медиана по
серии повторов perf-каркаса, переведённая из секунд в миллисекунды.
Ускорение `S = T_SEQ / T_ALL`, эффективность `E = S / workers_total`,
где `workers_total = num_procs × PPC_NUM_THREADS`. Знаменатель
`T_SEQ` — baseline из `seq/report.md`, §7.

Режим `task_run`:

|------------:|----------:|----------------:|-------------:|------------:|---------------:|
|      1      |     1     |        1        |     `42.01`  |    `1.515`  |     `1.515`    |
|      1      |     2     |        2        |     `40.70`  |    `1.564`  |     `0.782`    |
|      2      |     1     |        2        |     `46.15`  |    `1.379`  |     `0.689`    |
|      2      |     2     |        4        |     `42.50`  |    `1.497`  |     `0.374`    |
|      4      |     1     |        4        |     `44.88`  |    `1.418`  |     `0.354`    |
|      4      |     2     |        8        |     `57.33`  |    `1.110`  |     `0.139`    |

| `num_procs` | `threads` | `workers_total` | `T_ALL` (мс) | Speedup `S` | Efficiency `E` |
|------------:|----------:|----------------:|-------------:|------------:|---------------:|
|      1      |     1     |        1        |     `44.47`  |    `1.532`  |     `1.532`    |
|      1      |     2     |        2        |     `45.86`  |    `1.486`  |     `0.743`    |

Размер `n` фиксирован: `1 000 000` для всех строк.

**Главные узкие места ALL-версии:**

1. **Последовательный `FinalMerge` на rank 0.** Это самое
   значимое ограничение масштабируемости. На стадии слияния все
   остальные процессы простаивают, и время `FinalMerge` растёт как
   `O(n · log num_procs)`. При `num_procs = 4` финальное слияние
   делает 2 прохода по всем данным, при `num_procs = 8` — 3 прохода,
   и так далее. Возможные улучшения (не реализованы здесь):
   распределённое попарное слияние через древовидный обмен
   `MPI_Sendrecv` между процессами, либо параллелизация
   `FinalMerge` через TBB на rank 0 (как в OMP-версии для каждого
   уровня сети Бэтчера).
2. **Финальный `MPI_Bcast` на весь массив.** Объём пересылаемых
   данных `n · sizeof(double) · num_procs`. На однонодной системе с
   shared memory MPI это терпимо, но на распределённой сети было бы
   серьёзным bottleneck-ом. Реальная необходимость рассылать
   результат **всем** процессам диктуется только тестовой
   инфраструктурой курса (`CheckTestOutputData` на каждом ранге).
3. **Fallback при не-степени-двойки `num_procs`.** Конфигурации с
   `num_procs ∈ {3, 5, 6, 7}` фактически работают как SEQ + Bcast.
   Для замеров это означает, что имеет смысл сравнивать только
   `num_procs ∈ {1, 2, 4, 8}`; промежуточные значения дадут
   неинформативные результаты.

## 9. Выводы

ALL-версия реализована как гибрид MPI + TBB с чёткой двухуровневой
декомпозицией: межпроцессное распределение чанков через
`Scatter`/`Gather` + внутрипроцессная сортировка чанка через
`tbb::parallel_invoke`. Конфигурация параллелизма задаётся парой
`(num_procs, PPC_NUM_THREADS)`, и полное число «работающих единиц»
равно их произведению. Корректность алгоритма по построению
гарантируется тем, что и `chunk_size`, и `pow2` остаются степенями
двойки, что обязательно для применения сети Бэтчера на всех уровнях
слияния.

Главные методические решения, принятые в реализации:

- **Честный fallback** на однопроцессорную сортировку при
  `num_procs`, не являющемся степенью двойки.
- **Чёткое разделение pipeline-фаз:** `PreProcessingImpl` загружает
  вход, `RunImpl` делает всю MPI/TBB-работу, `PostProcessingImpl`
  выгружает результат.
- **TBB-уровень повторяет идиому чистой TBB-версии**
  (`parallel_invoke` для двух радиксов), без избыточного
  усложнения. Внутри одного чанка глубина параллелизма выбрана
  крупнозернистая, чтобы не конкурировать с MPI-уровнем за
  CPU-ресурсы.

Главное известное ограничение ALL-версии — **последовательное
финальное слияние на rank 0** — описано прямо в §8 и составляет
основной предмет интерпретации результатов: рост `num_procs` ускоряет
стадию локальной сортировки, но одновременно удлиняет последовательную
стадию слияния. Численные следствия и сравнение с остальными
backend-ами — задача корневого `report.md`.
