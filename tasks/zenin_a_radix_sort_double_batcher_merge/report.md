## 1. Введение

Задача — отсортировать массив `std::vector<double>` за пять
технологически разных реализаций: `seq`, `omp`, `tbb`, `stl`, `all`.
Алгоритмически — LSD-радикс по битовому представлению `double`,
монотонно упакованному в `uint64_t`, с финальным попарным слиянием
двух отсортированных половин сетью Бэтчера odd-even merge. Этот выбор
не случаен: и LSD-радикс, и сеть Бэтчера структурно содержат
независимые подзадачи и независимые компараторы соответственно, что
делает их удобным полигоном для сравнения моделей параллелизма с
разным характером накладных расходов — статической команды потоков
(OpenMP), task-scheduler-а с work-stealing (oneTBB), ручного
управления потоками (`std::thread`) и гибрида межпроцессного MPI с
внутрипроцессным TBB.

Цель корневого отчёта — связать пять локальных отчётов в одну
доказательную историю. Каждая отдельная реализация подробно
обсуждается в своём локальном `report.md`; здесь, на верхнем уровне,
выносится общий baseline, единая методика, агрегированные результаты
и — главное — интерпретация различий.

## 2. Единая постановка задачи

Все пять реализаций решают **одну и ту же задачу**:

- **Вход:** `std::vector<double>` произвольной длины `n ≥ 0`. Допустимы
  любые конечные значения `double`, включая `±0.0`, отрицательные и
  субнормальные. Значения `NaN` не подаются и не рассматриваются.
- **Выход:** вектор той же длины `n`, содержащий те же значения как
  мультимножество, упорядоченные по неубыванию.
- **Критерий корректности:** для каждого функционального теста выход
  побитово (через `!=`, без эпсилон-допуска) совпадает с заранее
  заданным эталоном `kTestParam[i].second`. Для perf-теста — выход
  побитово совпадает с `std::ranges::sort` на том же входе.
- **Ограничения:** размер `n` ограничен только доступной памятью —
  алгоритм требует ~3·`n_padded`·`sizeof(double)` байт временных
  буферов, где `n_padded` — округление `n` вверх до степени двойки.

Подробная постановка с обсуждением граничных случаев
(`n = 0`, `n = 1`, нестепени двойки, `±0.0`, повторы) — в
seq/report.md, §2. Поскольку все backend-ы делают
одно и то же по семантике, постановка не повторяется в локальных
отчётах OMP/TBB/STL/ALL: они ссылаются обратно на SEQ.

## 3. Единая методика эксперимента

Окружение и методика идентичны для всех пяти backend-ов.

Замеры выполнялись на следующей конфигурации:

- OS: Windows `<10>`, архитектура x86_64
- CPU: `<Intel(R) Core(TM) i5-10400F CPU @ 2.90GHz>`
- RAM: `<32 GB>`
- Компилятор: MSVC, стандарт `/std:c++20`
- Build type: `Release` (`/O2`), без санитайзеров
- CMake: `<3.30.4>`, генератор `<Visual Studio 17 2022>`

**Источник входных данных.** Один и тот же файл
`tests/performance/main.cpp` формирует вход для всех пяти backend-ов:

- размер `n = 1 000 000` элементов `double`, зафиксирован как
  `kCount_ = 1'000'000`;
- генератор `std::mt19937` с seed-ом, равным `kCount_` — то есть
  **фиксированным `1 000 000`**;
- значения из `std::uniform_real_distribution<double>` на отрезке
  `[-1e6, 1e6]`.

Это означает, что все пять реализаций сортируют **побитово
одинаковый** массив.

**Управление параллелизмом:**

- `PPC_NUM_THREADS` — переменная окружения, читаемая каркасом курса.
  Для OMP пробрасывается в `OMP_NUM_THREADS`, для TBB — в
  `tbb::global_control::max_allowed_parallelism`, для STL — в
  `ppc::util::GetNumThreads()`.
- Число MPI-процессов задаётся аргументом `mpiexec -n <num_procs>`
  и используется только в ALL.

**Размеры задач и конфигурации.** Для всех потоковых backend-ов
сравнивались конфигурации `p ∈ {1, 2, 4, 8}`. Для ALL дополнительно
сравнивались гибридные конфигурации `(num_procs, threads) ∈ {(1,1),
(1,2), (2,1), (2,2), (4,1), (4,2)}`. Размер `n` фиксирован
`1 000 000` для всех замеров.

**Метрики:**

- `T_SEQ` — медианное время SEQ-версии, baseline.
- `T_X(p)` — медианное время backend-а `X` при `p` потоках/worker-ах.
- `Speedup`: `S(p) = T_SEQ / T_X(p)`.
- `Efficiency`: `E(p) = S(p) / p`. Для ALL знаменатель —
  `workers_total = num_procs × threads`, что соответствует общему
  числу «рабочих единиц».

## 4. Сводка корректности

Корректность всех пяти backend-ов проверяется одним и тем же общим
каркасом — `tests/functional/main.cpp`, набор `kTestParam` из 12
тестов: `Single`, `Empty`, `ReverseTwo`, `Negative`, `Positive`,
`Large`, `Small`, `Sorted`, `DifferentSigns`, `Duplicates`,
`ReverseSeven`, `ZeroAndNegativeZero`. Эталон в каждом тесте задан
**статически** как пара `(input, expected_output)`; сравнение — побитово
через `!=`, без эпсилон-допуска. Для radix-sort, который не выполняет
арифметики, а лишь переставляет битовые представления `double`, такое
сравнение корректно.

Все 12 тестов проходят для всех пяти backend-ов. Дополнительно
`tests/performance/main.cpp` сравнивает выход каждого backend-а на
`n = 1 000 000` с `std::ranges::sort` — то есть успешное измерение
времени одновременно является доказательством корректности на крупном
размере.

**Ограничения применимости.** Только у одного backend-а есть
методическое ограничение — у ALL: при `num_procs`, не являющемся
степенью двойки (`{3, 5, 6, 7, ...}`), реализация сознательно
переходит к **fallback на однопроцессорную сортировку** (`LSDRadixSort`
на rank 0 + `MPI_Bcast` результата), потому что финальное слияние
`FinalMerge` устроено как каскад попарных Batcher-merge-ов и требует,
чтобы число чанков было степенью двойки. Это значит, что для ALL
осмысленно сравнивать только `num_procs ∈ {1, 2, 4, 8, ...}`;
промежуточные конфигурации алгоритмически вырождаются.

Никаких других ограничений у backend-ов нет: SEQ, OMP, TBB, STL
работают для любых `n ≥ 0` и любых конечных входных значений.

## 6. Интерпретация различий

### 6.1 Общая картина

Главное наблюдение, видное по всем таблицам §5: **никакая
параллельная реализация не дотягивает даже до speedup = 2 на 8
потоках**. Максимум — OMP в режиме pipeline на 4 потоках достигает
`S = 1.578`, что соответствует эффективности `E = 0.394`. Это значит,
что на данной задаче и на данной машине алгоритм фундаментально
недостаточно параллелизуется.

Структурные причины ограниченной масштабируемости:

1. **Параллельная часть алгоритма мала.** Внутренние проходы
   LSD-радикса оставлены последовательными во всех backend-ах
   (см. локальные отчёты, §4.3 у каждого) — параллелизация
   counting-фазы требовала бы синхронизации по 256-ячеечному `cnt` на
   каждом из 8 проходов, что для половины из 5·10⁵ элементов даёт
   неблагоприятное соотношение работы и overhead. По закону Амдала
   это сразу ограничивает достижимый speedup.

2. **Memory-bound характер первого уровня сети Бэтчера.** На
   `n = 10⁶` массив занимает `8 МБ`, что не помещается в L2-кэш
   современных CPU. Компараторы первого уровня `(i, i+n/2)`
   обращаются к двум удалённым ячейкам — это гарантированный промах
   мимо L2 и часто мимо L3. Пропускная способность памяти становится
   bottleneck-ом независимо от числа потоков.

3. **TBB и STL дополнительно ограничены сознательным упрощением:**
   внутренние уровни сети Бэтчера у них оставлены последовательными
   (tbb/report.md, stl/report.md). Параллелится только первый,
   самый «жирный» уровень. В OMP параллельны все уровни.

### 6.2

OMP, STL и даже ALL
при единственном worker выполняются заметно быстрее, чем SEQ.

| backend | T(p=1) task | T_SEQ task | T(p=1) / T_SEQ |
|---------|------------:|-----------:|---------------:|
|   OMP   |    41.11    |     63.66  |      0.646     |
|   TBB   |    65.92    |     63.66  |      1.036     |
|   STL   |    50.20    |     63.66  |      0.789     |
|ALL(1,1) |    42.01    |     63.66  |      0.660     |

В терминах «ускорения от параллелизма» это бессмысленно: при `p = 1`
ни OMP, ни STL ничего не параллелят. Тем не менее они быстрее SEQ.

Объяснение лежит не в параллелизме, а в **разной структуре кода
backend-ов**. В SEQ-версии функция `BlocksComparing` устроена как
**внутренний цикл** по `k`, а `BatcherOddEvenMerge` строится из
вызовов этой функции:

```cpp
// SEQ
void BlocksComparing(arr, i, step) {
  for (size_t k = 0; k < step; ++k)
    if (arr[i+k] > arr[i+k+step]) swap(arr[i+k], arr[i+k+step]);
}
```

В OMP, TBB и STL та же функция упрощена до **одного компаратора**, а
цикл по `k` поднят в вызывающий код (`BatcherOddEvenMerge` или лямбду
параллельного цикла):

```cpp
// OMP / TBB / STL
void BlocksComparing(arr, i, j) {
  if (arr[i] > arr[j]) swap(arr[i], arr[j]);
}
```

Эта переписанная форма даёт компилятору MSVC больше возможностей для
оптимизации: упрощается inline, индексация (`arr[i+k]`, `arr[i+k+step]`
с переменным `step`) превращается в `arr[i]`, `arr[j]` с готовыми
индексами, исчезает зависимость по `step` внутри цикла. На больших
тестах это даёт стабильно 20–35% выигрыша просто от рефакторинга,
без всякого параллелизма.

### 6.3 OMP

В абсолютном времени OMP — **лидер**: 41.11 мс на p=1 в task-режиме.
Это объясняется сочетанием двух факторов: (а) переписанный
`BlocksComparing` (как у STL); (б) самая полная схема параллелизации
из всех — параллелятся **все** уровни сети Бэтчера через `parallel
for`, а не только первый, как в TBB/STL (см.
omp/report.md, §4.2).

Однако с ростом числа потоков OMP **не масштабируется**: 41.11 →
43.73 → 46.46 → 45.00. Время **растёт** при добавлении потоков.
Причина — создание нового параллельного региона на каждом уровне
сети Бэтчера (внешний цикл по `po >>= 1` запускает свой
`parallel for` ~20 раз для `n = 10⁶`). Для мелких уровней, где `po`
становится малым и работа в каждом блоке мизерна, overhead
форка/джойна команды потоков сопоставим с полезной работой, и
параллелизм работает в минус.

Это — структурный диагноз, описанный в omp/report.md,
§8 как «потенциальное узкое место». Числа в §5 это подтверждают.

### 6.4 TBB

TBB — единственный backend, где данные §5 однозначно показывают
ускорение именно от добавления потоков:

| `p` | T_TBB(p) task | T_TBB(p) / T_TBB(1) |
|----:|--------------:|--------------------:|
|  1  |     65.92     |       1.000         |
|  2  |     49.84     | 0.756 (ускор. 1.32) |
|  4  |     47.80     |       0.725         |
|  8  |     48.94     |       0.742         |

От p=1 к p=2 — реальный спад на 24% (`T(2)/T(1) = 0.756`), что
соответствует чистому speedup ≈ 1.32 внутри TBB-семейства. Это
именно тот эффект, которого мы ожидаем от параллелизма, без примеси
рефакторинга кода. Дальше — плато (4 и 8 потоков дают то же, что 2):
это потому, что **только первый уровень сети Бэтчера параллелится в
TBB**, а внутренние ~19 уровней последовательные (см.
tbb/report.md, §4.3). По закону Амдала
последовательная часть быстро доминирует.

Если бы TBB-версия параллелила и внутренние уровни (как OMP), её
абсолютное время на p=4 могло бы оказаться лучшим из всех; это
очевидное направление улучшения, явно зафиксированное в
tbb/report.md, §9.

### 6.5 STL

STL показывает узкий разброс: 49.2–50.2 мс независимо от числа
потоков. Это означает, что параллелизм через `std::thread` на этой
задаче работает «вничью»: выигрыш от распределения работы между
потоками примерно равен накладным расходам на создание/завершение
`std::thread`-ов на каждом вызове `MergeFirstPass`.

Структурная причина — отсутствие thread pool: каждый раз
`MergeFirstPass` создаёт `num_threads` новых потоков «с нуля» через
`CreateThread` под капотом MSVC (stl/report.md,
§8, пункт 2). У OMP и TBB этой проблемы нет — там команда worker-ов
переиспользуется. Это известная цена ручного управления потоками.

При этом сам код STL-версии методически образцовый: правильный
порядок `create → work → join`, безопасный захват `[&arr, lo, hi, po]`
(не by-reference!), деление с округлением вверх и `if (lo >= hi)
break` (см.stl/report.md, §4.2). Низкий speedup —
не следствие ошибок в коде, а следствие структурного ограничения
самой технологии `std::thread` без пула.

### 6.6 ALL

ALL — самый поучительный случай для интерпретации. Если упорядочить
конфигурации task-режима по `workers_total`, видна **немонотонная**
картина:

| `workers_total` | конфигурация | T_ALL task |
|----------------:|--------------|-----------:|
|        1        |    (1, 1)    |   42.01    |
|        2        |    (1, 2)    |   40.70    |
|        2        |    (2, 1)    |   46.15    |
|        4        |    (2, 2)    |   42.50    |
|        4        |    (4, 1)    |   44.88    |
|        8        |    (4, 2)    |   57.33    |

Несколько наблюдений:

1. **(1, 1) и (1, 2) — самые быстрые.** Это **fallback-ветка** ALL-
   версии: при `num_procs = 1` срабатывает условие
   `procs_is_pow2 = false` (`num_procs > 1` ложно), и алгоритм
   деградирует до `LSDRadixSort(local_data_)` + `MPI_Bcast` на
   единственный процесс (no-op). TBB при этом не задействован, и
   `(1, 2)` отличается от `(1, 1)` только шумом измерения. Это
   подтверждает обсуждение в all/report.md, §8.
2. **(2, 1) хуже, чем (1, 1):** добавление второго MPI-процесса
   **замедляет** работу с 42.01 до 46.15 мс. Причина — накладные
   расходы коммуникаций `MPI_Scatter` + `MPI_Gather` +
   `MPI_Bcast` для 8 МБ данных на `num_procs = 2` стоят дороже, чем
   выигрыш от распараллеливания сортировки на два процесса. Это
   классический эффект «MPI на однонодной системе не окупается на
   маленьких задачах».
3. **(2, 2) лучше, чем (2, 1):** добавление TBB-уровня поверх MPI
   возвращает время к 42.50 мс. Внутрипроцессный параллелизм
   `tbb::parallel_invoke` действительно помогает, но в сумме лишь
   компенсирует overhead MPI.
4. **(4, 2) — резкая деградация до 57.33 мс.** При максимальной
   нагрузке проявляются сразу несколько накладок: (а) MPI делает 4
   процесса, и `FinalMerge` на rank 0 выполняет 2 каскадных уровня
   попарных слияний (см. all/report.md, §4.1, шаг 7);
   (б) `workers_total = 8` сравним с числом логических потоков
   машины, и процессы с TBB-потоками внутри начинают **конкурировать
   за CPU**, что обычно даёт чистое ухудшение.

### 6.7 task vs pipeline

Разница между режимами `task_run` и `pipeline_run` мала (в пределах
~5%) для всех backend-ов, что ожидаемо: pipeline просто прогоняет
полный pipeline (`Validation → PreProc → Run → PostProc`) много раз и
агрегирует медиану. Какого-либо систематического перекоса в пользу
одного из режимов нет — это подтверждает, что `Pre/PostProc` у
большинства backend-ов пустые, и накладные расходы pipeline-режима
обусловлены только многократным выделением буфера в `RunImpl`.
Незначительное исключение — ALL в pipeline-режиме, где (2,1), (4,1) и
(4,2) **заметно медленнее** соответствующего task-режима (54.04 vs
46.15, 61.62 vs 44.88, 61.94 vs 57.33): причина — повторяющиеся
вызовы MPI-коллективов в каждой итерации pipeline, чей overhead
накапливается.

## 7. Заключение

В абсолютном времени на `n = 10⁶`
лидирует **OMP**, но это
лидерство в основном объясняется переписанным `BlocksComparing` и
полной параллелизацией всех уровней сети Бэтчера. STL и ALL(1,1)
показывают близкие результаты благодаря тому же рефакторингу. TBB
— единственный backend, демонстрирующий чистое ускорение от
параллелизма (1.32×), но из-за упрощённой схемы (параллельный
только первый уровень Бэтчера) проигрывает OMP в абсолютных числах.
**ALL** на однонодной системе с `n = 10⁶` неконкурентоспособна: цена
MPI-коммуникаций превышает выигрыш от распределения работы.

**Рекомендация по выбору технологии для этой задачи:**

- На одной машине и при `n` до ~10⁷ — **OpenMP**: простой синтаксис,
  лучшее абсолютное время, полная параллелизация без отдельной
  библиотеки.
- Для случаев, когда нужен work-stealing на неоднородной нагрузке —
  **TBB** (но требует доработки: параллелизация внутренних уровней
  Бэтчера).
- На многонодных кластерах с `n ≥ 10⁸` — **ALL** с увеличенной долей
  локальной работы и распределённым `FinalMerge` (текущая реализация
  с последовательным `FinalMerge` на rank 0 неэффективна для больших
  `num_procs`).

**Ограничения данного сравнения:**

1. **Эффект рефакторинга `BlocksComparing` смешан с эффектом
   параллелизма** (см. §6.2). Для честного разделения нужен
   отдельный эксперимент «SEQ с упрощённым BlocksComparing», не
   предусмотренный инфраструктурой курса.
2. **Размер `n = 10⁶` фиксирован каркасом.** При больших размерах
   относительная стоимость накладных расходов снизилась бы, и
   масштабируемость TBB/OMP/STL/ALL могла бы выглядеть иначе.
3. **Машина однонодная**, что для ALL особенно неблагоприятно:
   MPI-коммуникации идут через shared-memory backend MS-MPI, но
   всё равно стоят дороже прямого доступа к памяти внутри одного
   процесса.

**Что можно улучшить дальше:**

1. **Параллелизация внутренних уровней сети Бэтчера в TBB и STL** —
   обернуть в `tbb::parallel_for` / руками-`std::thread` цикл по
   блокам `i` с явным `grainsize ≥ po`. Это должно поднять TBB-
   масштабируемость до уровня OMP.
2. **Постоянный thread pool в STL** вместо создания
   `std::thread`-ов на каждый вызов `MergeFirstPass`. Это
   убрало бы основную накладную расходную статью STL.
3. **Распределённый `FinalMerge` в ALL** — попарное слияние через
   `MPI_Sendrecv` между процессами вместо последовательного на
   rank 0. На текущих размерах эффект может быть небольшим, но на
   `n ≥ 10⁸` это решающий шаг.
4. **Параллелизация LSD-радикса** — counting-фаза с приватными
   счётчиками + reduction по бакетам. Не реализовано во всех
   backend-ах из-за неблагоприятной экономики на малых половинах
   массива; имеет смысл при больших `n`.
