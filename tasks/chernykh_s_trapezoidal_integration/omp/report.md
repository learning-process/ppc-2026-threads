## 1. Контекст

Задача состоит в вычислении многомерных интегралов с использованием метода трапеций с использованием OpenMP. Необходимо сравнить время выполнения алгоритма с последовательной версией и исследовать масштабируемость алгоритма.

## 2. Постановка задачи
### Входные данные:
- `limits`: Вектор пар [a_i, b_i] - границы интегрирования по каждому измерению. 
- `steps`: Вектор целых положительных чисел `n_i`, задающий количество шагов разбиения по каждому измерению.
- `func`: Функция, интеграл которой нужно посчитать, передается в виде лямбда выражения.
### Выходные данные:
Значение типа double - результат счета.

### Ограничения и крайние случаи:
- Входные вектора `limits` и `steps` должны совпадать по размеру и не должны быть пустыми.
- Входной вектор `steps` должен содержать числа строго больше нуля.

## 3. Базовый алгоритм

Последовательный алгоритм использует один вектор `counters` для многомерного шага внутри цикла `while`. Каждая итерация зависит от состояния счетчиков предыдущей, что создает явную циклическую зависимость по данным и препятствует прямому распараллеливанию через `#pragma omp parallel for`.

## 4. Схема распараллеливания

Чтобы обойти ограничение последовательного алгоритма, в OMP-версии применен переход к плоскому линейному пространству итераций.

### Параллельный участок кода
```cpp
#pragma omp parallel default(none) shared(input, dims, total_points) reduction(+ : total_sum)
{
    std::vector<std::size_t> local_counters(dims);  // Локальный вектор индексов потока
    std::vector<double> local_point(dims);          // Локальный вектор координат точки

#pragma omp for schedule(static)
    for (int64_t j = 0; j < total_points; j++) {
        int64_t temp_j = j;
        for (std::size_t i = 0; i < dims; i++) {
            int64_t point_in_dims = static_cast<int64_t>(input.steps[i]) + 1;
            local_counters[i] = static_cast<std::size_t>(temp_j % point_in_dims);
            temp_j /= point_in_dims;
        }
        double weight = CalculatePointAndWeight(input, local_counters, local_point);
        total_sum += input.func(local_point) * weight;
    }
}
```

### Область параллелизма
Параллельная область открывается `#pragma omp parallel` один раз на весь блок, чтобы выделить память под локальные векторы `local_counters` и `local_point` один раз для каждого потока.
Внутри региона директива `#pragma omp for` разделяет только сам вычислительный цикл. Это убирает накладные расходы на постоянное выделение и освобождение памяти на каждой итерации.

### Разделение данных
* **shared** — `input`, `dims`, `total_points` (доступны только на чтение).
* **private (локальные)** — `local_counters`, `local_point`, `j`, `temp_j`, `point_in_dims`, `weight` (создаются внутри параллельной области и являются индивидуальными для каждого потока).

### Синхронизация
* Использован механизм редукции `reduction(+: total_sum)`. Для устранения гонки данных при обновлении общей суммы.
* Критические секции и атомики отсутствуют.

### Барьеры
* Неявный барьер в конце области `#pragma omp parallel`.

### Распределение работы
* Используется статическое планирование `schedule(static)`. Так как вычислительная сложность каждой итерации неизменна, при старте OpenMP делит `total_points` на равные блоки и распределяет их по потокам.

## 5. Детали реализации

### Файлы:

- `omp/src/ops_omp.cpp`

### Особенности реализации
Главным изменением относительно SEQ стал отказ от последовательного инкремента многомерного счетчика и переход к математическому восстановлению индексов:
``` cpp
int64_t temp_j = j;
for (std::size_t i = 0; i < dims; i++) {
    int64_t point_in_dims = static_cast<int64_t>(input.steps[i]) + 1;
    local_counters[i] = static_cast<std::size_t>(temp_j % point_in_dims);
    temp_j /= point_in_dims;
}
```
Данный расчет локален для каждой итерации `j`, не зависит от состояния `j-1`, что делает итерации независимыми.

Расхождения в выходных данных между параллельной и последовательной реализациями отсутствуют: результаты вычислений полностью совпадают в пределах допустимой погрешности 10^-3. Дополнительно была проверена устойчивость и детерминированность результатов при запуске алгоритма на 1, 2, 4 и более потоках, что подтверждает корректность логики счёта и полное отсутствие состояний гонки.
