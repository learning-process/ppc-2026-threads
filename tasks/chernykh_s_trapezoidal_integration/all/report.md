## 1. Контекст

Гибридная версия (MPI + OpenMP) требуется для реализации иерархического параллелизма, который позволяет эффективно
масштабировать алгоритм численного интегрирования как на многоядерные архитектуры, так и на кластерные системы с
распределенной памятью.

## 2. Постановка задачи

### Входные данные

- `limits`: Вектор пар [a_i, b_i] - границы интегрирования по каждому измерению.
- `steps`: Вектор целых положительных чисел `n_i`, задающий количество шагов разбиения по каждому измерению.
- `func`: Функция, интеграл которой нужно посчитать, передается в виде лямбда выражения.

### Выходные данные

Значение типа double - результат счета.

### Ограничения и крайние случаи

- Входные вектора `limits` и `steps` должны совпадать по размеру и не должны быть пустыми.
- Входной вектор `steps` должен содержать числа строго больше нуля.

## 3. Базовый алгоритм

Последовательный алгоритм выполняет обход многомерной сетки с помощью вложенных счётчиков `counters`, обновляемых в
едином цикле. Каждая итерация зависит от состояния счетчиков на предыдущем шаге, что создает циклическую зависимость по
данным и препятствует прямому распараллеливанию. Для обеспечения параллельного выполнения алгоритм был трансформирован:
многомерное пространство индексов отображается в линейный диапазон `[0, total_points)`, что позволяет независимо
вычислять многомерные координаты на основе текущего линейного индекса.

## 4. Межпроцессная схема

### Роли rank-ов

- Ранг 0 (Master) - определяет общее количество итераций `total_points`, вычисляет границы поддиапазонов `borders` для
всех rank-ов.
- Остальные ранги (rank = 1, 2, ..., size-1) - получают необходимые параметры задачи, вычисляют локальную сумму на своём
поддиапазоне и участвуют в финальной агрегации результата.

### Распределение данных и используемые MPI-вызовы

1) Параметры задачи `dims`, `steps`, `limits`, `func_id` распространяются от Master ранга ко всем остальным с помощью
`MPI_Bcast`.
Фрагмент кода с использованием MPI_Bcast:

``` cpp
 MPI_Bcast(&dims, 1, MPI_UINT64_T, 0, MPI_COMM_WORLD);
 MPI_Bcast(input.steps.data(), static_cast<int>(dims), MPI_INT, 0, MPI_COMM_WORLD);
 MPI_Bcast(input.limits.data(), static_cast<int>(dims * 2), MPI_DOUBLE, 0, MPI_COMM_WORLD);
 MPI_Bcast(&func_id, 1, MPI_INT, 0, MPI_COMM_WORLD);
```

2) Массив `borders`, содержащий пары `{start, end}` для каждого процесса, передаётся через `MPI_Scatter`. Каждый процесс
получает свою индивидуальную пару границ, что исключает пересечение рабочих областей.
Фрагмент кода с использованием MPI_Scatter:

```cpp
MPI_Scatter(send_ptr, 2, MPI_LONG_LONG, recv_ptr, 2, MPI_LONG_LONG, 0, MPI_COMM_WORLD);
```

3) `MPI_Allreduce` (с операцией `MPI_SUM`) - используется для глобальной редукции — каждый процесс передает свою
`local_sum`, а после выполнения операции все процессы получают итоговую `global_sum`.
Фрагмент кода с использованием MPI_Allreduce:

```cpp
MPI_Allreduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, MPI_COMM_WORLD);
```

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса для параллельного выполнения вычислений используется OpenMP.

Работа между потоками делится внутри функции `OnProcessCalculate`. Использование директивы `#pragma omp parallel for` с
параметром `schedule(static)` обеспечивает равномерное и предсказуемое распределение итераций поддиапазона, полученного
от MPI, между потоками процесса. Каждый поток внутри ранга имеет собственные локальные переменные `local_counters` и
`local_point`, что гарантирует отсутствие состояния гонки при вычислении координат текущей итерации.

Объединение результатов реализовано с помощью механизма редукции: `#pragma omp parallel for reduction(+ : total_sum)`.
Это наиболее эффективный подход: каждый поток накапливает частичную сумму в своей локальной копии переменной
`total_sum`, а по завершении работы параллельной секции OpenMP автоматически выполняет безопасное суммирование всех
локальных значений в общую переменную процесса. Это исключает необходимость использования критических секций или
атомарных операций, обеспечивая максимальную скорость выполнения.

### Фрагмент кода внутрипроцессной работы

```cpp
double ChernykhSTrapezoidalIntegrationALL::OnProcessCalculate(const IntegrationInType &input, std::size_t dims,
                                                              int64_t start, int64_t end) {
  double total_sum = 0.0;
#pragma omp parallel default(none) shared(input, dims, start, end) reduction(+ : total_sum)
  {
    std::vector<std::size_t> local_counters(dims);
    std::vector<double> local_point(dims);

#pragma omp for schedule(static)
    for (int64_t j = start; j < end; j++) {
      int64_t temp_j = j;
      for (int i = static_cast<int>(dims) - 1; i >= 0; i--) {
        int64_t point_in_dims = static_cast<int64_t>(input.steps[static_cast<std::size_t>(i)]) + 1;
        local_counters[static_cast<std::size_t>(i)] = static_cast<std::size_t>(temp_j % point_in_dims);
        temp_j /= point_in_dims;
      }
      double weight = CalculatePointAndWeight(input, local_counters, local_point);
      total_sum += input.func(local_point) * weight;
    }
  }
  return total_sum;
}
```

## 6. Детали реализации

### Файлы

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

### Синхронизация и коммуникации

- `MPI_Bcast`: Используется для рассылки размерности `dims`, массивов границ `limits` и `steps`, а также идентификатора
целевой функции (`func_id`) на все рабочие процессы.
- `MPI_Scatter`: Применяется для распределения уникальных вычислительных диапазонов `borders` между рангами.
- `MPI_Allreduce`: Выполняет глобальное суммирование локальных результатов каждого процесса в финальную переменную
`global_sum`.

### Узкие места

- Использование множественных `MPI_Bcast` для доставки метаданных задачи при малом числе итераций может создавать
задержки, сопоставимые со временем вычислений.
- Внутри `OnProcessCalculate` каждый поток OpenMP пересчитывает многомерные координаты (деление и взятие остатка) для
каждого `j` из своего диапазона, из-за чего накладные расходы на вычисление индекса могут начать доминировать над
полезной работой, что негативно сказывается на производительности.

### Особенности

Из-за обобщенной реализации входных данных для тестов, а именно использовании лямбда функций, в реализации была
применена схема идентификации функций через ID. Ранг 0 анализирует входные данные, определяет тип целевой функции с
помощью вспомогательных методов `DetectFunctionId`, `DetectDim1`, `DetectDim3` и передает его на другие процессы. После
получения ID каждый процесс восстанавливает лямбду локально через `switch`-конструкцию. Необходимость подобной
реализации обуславливается тем, что лямбда функцию напрямую нельзя передать через MPI.

## 7. Проверка корректности

- Результат, полученный после `MPI_Allreduce`, сравнивается с результатом последовательной версии (SEQ). Используется
стандартная для численного интегрирования точность сравнения 10^-3.
- Гибридная версия (`ALL`) проходит тот же набор функциональных тестов (`kTestParam`), что и `SEQ`, `OMP`, `TBB` и `STL`
версии. Это гарантирует, что иерархический параллелизм не вносит искажений в математический результат.

Все тесты пройдены для конфигураций 1×1, 1×2, 1x4, 2×1, 2×2, 2x4, 4x1, 4x2, 4x4.

## 10. Выводы

### Когда гибридная схема оправдана

- Гибридная версия демонстрирует максимальное ускорение 5.01x на оптимальной конфигурации (4 ранга × 3 потока). Это
наглядно доказывает преимущество иерархического параллелизма над чисто MPI-реализациями при работе на многоядерных
узлах.
- Использование OpenMP-потоков внутри MPI-ранга позволяет сократить количество процессов используемых для вычислений.
Это снижает накладные расходы на вызов `MPI_Reduce` и `MPI_Scatter`, сохраняя при этом высокую загрузку ядер процессора.

### Когда гибридная схема неэффективна

- При чрезмерном увеличении числа OpenMP-потоков (например, 6 потоков на ранг) эффективность резко падает (до 0.24) из-
за борьбы за ресурсы кэша и контекстных переключений. В данном случае программная сложность гибридной модели не
окупается приростом скорости.
- На малых задачах или при использовании конфигураций с малым числом воркеров (1х1, 1х2) накладные расходы на MPI-среду
делают гибридную модель медленнее, чем последовательный код (скорость 0.86x от SEQ).
