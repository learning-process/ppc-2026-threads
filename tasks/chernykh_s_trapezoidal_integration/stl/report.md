## 1. Контекст

Ручная thread-based версия нужна, чтобы обеспечить максимальный контроль над созданием, жизненным циклом и
распределением ресурсов потоков вручную.

## 2. Постановка задачи

### Входные данные

- `limits`: Вектор пар [a_i, b_i] - границы интегрирования по каждому измерению.
- `steps`: Вектор целых положительных чисел `n_i`, задающий количество шагов разбиения по каждому измерению.
- `func`: Функция, интеграл которой нужно посчитать, передается в виде лямбда выражения.

### Выходные данные

Значение типа double - результат счета.

### Ограничения и крайние случаи

- Входные вектора `limits` и `steps` должны совпадать по размеру и не должны быть пустыми.
- Входной вектор `steps` должен содержать числа строго больше нуля.

## 3. Базовый алгоритм

Последовательный алгоритм осуществляет обход многомерной сетки в рамках одного цикла `while` с использованием единого
вектора счётчиков `counters`. Каждая текущая итерация шага жёстко зависит от состояния счётчиков на предыдущей итерации,
 что создаёт циклическую зависимость по данным. Такая структура алгоритма делает невозможным его прямое
распараллеливание с помощью `std::thread`, так как низкоуровневые потоки не могут вычислять свои многомерные координаты
изолированно друг от друга и требуют полной независимости обрабатываемых диапазонов памяти для безопасного запуска без
блокировок.

## 4. Схема распараллеливания

### Деление данных на диапазоны

Выполнен переход от вложенных зависимых циклов к плоскому линейному пространству итераций общего размера `total_point`
(64 * 10^6 узлов). Пространство итераций делится между потоками статически на этапе инициализации.
Вычисляется базовая порция итераций на поток (`points_per_thread = total_point / num_threads`), а остаток от деления
(`remainder = total_point % num_threads`) распределяется по одной дополнительной итерации между первыми потоками. Это
обеспечивает равномерное распределение нагрузки и отсутствие простоя потоков. Границы диапазонов каждого потока `i`
фиксируются в векторе пар `borders[i]`.

### Фрагмент кода деления данных

``` cpp
  int64_t points_per_thread = total_point / num_threads;
  int64_t remainder = total_point % num_threads;

  int64_t start = 0;
  for (int i = 0; i < num_threads; i++) {
    borders[i].first = start;
    start += points_per_thread;

    if (i < remainder) {
      start++;
    }
    borders[i].second = start;
  }
```

### Что делает каждый поток

Каждый поток выполняет функцию `MemberWorker`, которая принимает свой уникальный поддиапазон `[start, end)`. Внутри
функции поток переводит текущий линейный индекс `i` в многомерные координаты `point`. После этого поток вычисляет
значение функции, применяет весовой коэффициент метода трапеций и накапливает локальную сумму в регистровую переменную
`sum`.

### Где хранятся локальные результаты

Каждый поток записывает итоговое значение своей локальной суммы строго в свой `result[i]`. Это исключает необходимость в
 общей критической секции и предотвращает состояние гонки.

### Как и когда выполняется join

После запуска всех потоков вызывается метод `join()` для каждого потока, для того чтобы главный поток дождался
завершения всех вычислений, прежде чем вычислить финальное суммирование результатов из вектора result.

### Синхронизация

Синхронизация не используется , потому что:

- Входные данные (`input`, `h`, `dims`) доступны потокам только для чтения.
- Каждый поток пишет результаты исключительно в свой собственный, заранее выделенный слот вектора `result`.

## 5. Детали реализации

### Файлы

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

### Фрагмент кода создания потоков и вызова join()

``` cpp
// Создание и запуск потоков
  for (int i = 0; i < num_threads; ++i) {
    threads[i] = std::thread(work_function, borders[i].first, borders[i].second, std::ref(result[i]));
  }

// Ожидание завершения всех потоков (join)
  for (int i = 0; i < num_threads; ++i) {
    threads[i].join();
  } 
```

Распределение вычислительного пространства реализовано через вектор `borders`, где каждый элемент `borders[i]` хранит
пару `{start, end}`.
При инициализации общего количества точек `total_point`:

1. Вычисляется базовая порция `points_per_thread` для всех потоков.
2. Остаток `remainder` распределяется по 1 дополнительной итерации на первые потоки.
Это обеспечивает равномерную нагрузку: каждый из `num_threads` потоков получает уникальный непересекающийся интервал
индексов.

### Особенности

- Потоки не пишут в общую переменную `total_sum` напрямую. Вместо этого в `RunImpl` создается вектор `result` размера
`num_threads`.
- Каждый поток `tid` получает ссылку на свой слот `result[tid]` и выполняет вычисления независимо, накапливая сумму в
локальной переменной `sum`. По завершении своего диапазона итераций, поток однократно записывает локальную сумму в свой
слот вектора результатов.
- Создание потоков `create` полностью завершается до начала фазы ожидания `join`, что исключает последовательное
выполнение потоков и обеспечивает их полноценную параллельную работу.

## 6. Проверка корректности

Тестирование параллельной версии STL проводилось на аналогичном последовательной версии (SEQ) наборе функциональных
тестов (`Linear_1D`, `Sum_2D`, `Constant_3D`, `Parabola_1D`, `Sin_1D`, `Zero_Range`), включая проверку на нулевой
отрезок интегрирования, разные размерности пространств и виды самих функций интегрирования. Расхождения в выходных
данных между параллельной и последовательной реализациями отсутствуют: результаты вычислений полностью совпадают в
пределах допустимой погрешности 10^-3. Дополнительно была проверена устойчивость и детерминированность результатов при
запуске алгоритма на 1, 2, 4 и более потоках, что подтверждает корректность логики счёта и полное отсутствие состояний
гонки.
