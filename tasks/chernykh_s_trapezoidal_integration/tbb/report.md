## 1. Контекст

Для данной задачи выбрана TBB-версия, так как вычисление многомерного интеграла на большой сетке эффективно разбивается
на множество независимых подзадач.
С помощью TBB можно позволяет динамически балансировать нагрузку между ядрами процессора в зависимости от текущей
загруженности операционной системы, снижая накладные расходы по сравнению с жестким распределением итераций.

## 2. Постановка задачи

### Входные данные

- `limits`: Вектор пар [a_i, b_i] - границы интегрирования по каждому измерению.
- `steps`: Вектор целых положительных чисел `n_i`, задающий количество шагов разбиения по каждому измерению.
- `func`: Функция, интеграл которой нужно посчитать, передается в виде лямбда выражения.

### Выходные данные

Значение типа double - результат счета.

### Ограничения и крайние случаи

- Входные вектора `limits` и `steps` должны совпадать по размеру и не должны быть пустыми.
- Входной вектор `steps` должен содержать числа строго больше нуля.

## 3. Базовый алгоритм

Последовательный алгоритм использует один вектор `counters` для многомерного шага внутри цикла `while`. Каждая следующая
 итерация обхода сетки жестко зависит от состояния счетчиков на предыдущем шаге. Данная циклическая зависимость по
данным делает невозможным параллельное выполнение этого цикла в исходном виде, так как потоки не могут вычислять свои
координаты изолированно друг от друга.

## 4. Схема распараллеливания

Чтобы применить высокоуровневые шаблоны TBB, выполнен переход от зависимого цикла обхода многомерных счетчиков к
плоскому линейному пространству итераций. Общее число узлов сетки `total_points` вычисляется предварительно как
произведение `steps[i] + 1` по всем измерениям.

### Параллельный участок кода

```cpp
total_sum = tbb::parallel_reduce(
    tbb::blocked_range<int64_t>(0, total_points), 
    0.0, 
    body, 
    std::plus<>()
);
```

### Лямбда функция

```cpp
auto body = [&](const tbb::blocked_range<int64_t> &r, double local_sum) -> double {
    std::vector<double> local_point(dims);
    std::vector<size_t> local_counters(dims);

    for (int64_t j = r.begin(); j < r.end(); j++) {
      int64_t temp_j = j;
      for (size_t i = 0; i < input.steps.size(); i++) {
        local_counters[i] = temp_j % (input.steps[i] + 1);
        temp_j /= input.steps[i] + 1;
      }
      double weight = CalculatePointAndWeight(input, local_counters, local_point);
      local_sum += input.func(local_point) * weight;
    }
    
    return local_sum;
  };
```

Выбран `tbb::parallel_reduce`, т.к. нужно суммировать результаты вычислений всех подзадач без использования явных
блокировок.

`tbb::blocked_range<int64_t>(0, total_points)` используется один диапазон, который автоматически разбивается TBB блоки.
Параметр `grainsize` не указан, чтобы TBB автоматически подбирал оптимальный размер блока.
`partitioner` используется по умолчанию, чтобы минимизировать накладные расходы на создание задач, если все ядра
загружены.
ОneTBB считывает количество потоков, заданное через переменную среды `PPC_NUM_THREADS`, и автоматически распределяет
работу между ними.
Лямбда-функция `body` захватывает все необходимые данные по ссылке (`[&]`), что дает доступ к `input` и `dims` внутри
тела цикла. Векторы `local_point` и `local_counters` выделяются внутри лямбды один раз на весь поддиапазон `r`,
предотвращая гонку данных. Финальное слияние сумм выполняется автоматически через встроенный бинарный функтор
`std::plus<>()`.

## 5. Детали реализации

### Файлы

- `tbb/include/ops_tbb.hpp``
- `tbb/src/ops_tbb.cpp``

### Изменения относительно SEQ

1. Вместо использования последовательного while по всем измерениям был использован запуск `tbb::parallel_reduce` на
плоском одномерном пространстве итераций `total_points`.
2. Внутри лямбда-функции каждая задача рассчитывает свои многомерные координаты `local_counters` из линейного индекса
`j` с помощью операций деления и остатка от деления. Это сделало итерации полностью независимыми друг от друга.

### Локальные аккумуляторы и объединение результатов

- Каждый поток обрабатывает свой выделенный поддиапазон итераций `[r.begin(), r.end())`.
- Внутри лямбды для каждого поддиапазона инициализируется локальный аккумулятор `local_sum`, переданный по значению.
Потоки накапливают промежуточные суммы изолированно, что полностью исключает гонки данных.
- Финальное объединение результатов происходит автоматически, TBB с помощью `std::plus<>()`. По мере завершения работы
параллельных задач их локальные аккумуляторы попарно суммируются, формируя итоговое значение `total_sum` на выходе из
`tbb::parallel_reduce`.

### Работа с памятью

- Проблема с разделением кэш-линий отсутствует, т.к. все потоки работают со своими локальными переменными `local_sum`,
`local_point` и `local_counters`.
- Координаты точек вычисляются

## 6. Проверка корректности

Тестирование параллельной версии TBB проводилось на аналогичном последовательной версии (SEQ) наборе функциональных
тестов (`Linear_1D`, `Sum_2D`, `Constant_3D`, `Parabola_1D`, `Sin_1D`, `Zero_Range`), включая проверку на нулевой
отрезок интегрирования, разные размерности пространств и виды самих функций интегрирования. Расхождения в выходных
данных между параллельной и последовательной реализациями отсутствуют: результаты вычислений полностью совпадают в
пределах допустимой погрешности 10^-3. Дополнительно была проверена устойчивость и детерминированность результатов при
запуске алгоритма на 1, 2, 4 и более потоках, что подтверждает корректность логики счёта и полное отсутствие состояний
гонки.

- На физических ядрах (до 6 потоков) алгоритм `tbb::parallel_reduce` показывает хорошую эффективность от 67% до 83%
процентов. При переходе на 12 потоков эффективность упала до 46%.

- Механизм work stealing балансирует нагрузку между потоками, сглаживая микрозадержки операционной системы.
