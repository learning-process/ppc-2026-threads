## 1. Контекст

Задача — та же, что в [`seq/report.md`](../seq/report.md): оценка
многомерного интеграла методом Монте-Карло по последовательности Кронекера.
В OpenMP-версии распараллелен **внешний цикл по индексу выборки** \(i\):
итерации полностью независимы по чтению глобальных данных
(`lower_bounds_`, `upper_bounds_`, `alpha`), а единственный
write-after-read — это аккумулятор `sum`, для которого OpenMP даёт явный
`reduction(+:sum)`.

## 2. Постановка задачи

Краткое определение — то же, что в SEQ:

\[I \approx V \cdot \frac{1}{n}\sum_{i=0}^{n-1} f(x^{(i)}),\quad
  x_d^{(i)} = \mathrm{lower}_d + (\mathrm{upper}_d-\mathrm{lower}_d)\,\{0.5+(i+1)\alpha_d\}.\]

Тип задачи — `TypeOfTask::kOMP`; класс — `ShilinNMonteCarloIntegrationOMP`.
Валидация и препроцессинг повторяют SEQ.

## 3. Базовый алгоритм

В OMP-варианте формула \(t_d^{(i)} = \{0.5+(i+1)\alpha_d\}\) переписана в
**замкнутом виде** (через `std::floor`), а не итеративно. Это сделано
специально, чтобы убрать межитерационную зависимость по `current[]`,
которая была в SEQ: теперь точку для индекса \(i\) можно вычислить
независимо от соседних индексов, и весь цикл становится
embarrassingly parallel.

## 4. Схема распараллеливания

Параллельный регион — один; в нём поток создаёт **локальный** буфер `point`
(privatе по природе, потому что объявлен внутри блока `parallel`), а
аккумулятор `sum` редуцируется через `reduction(+:sum)`.

```46:85:tasks/shilin_n_monte_carlo_integration/omp/src/ops_omp.cpp
bool ShilinNMonteCarloIntegrationOMP::RunImpl() {
  auto dimensions = static_cast<int>(lower_bounds_.size());

  const std::vector<double> alpha = { /* sqrt(2)..sqrt(29) */ };

  double sum = 0.0;

#pragma omp parallel reduction(+ : sum) default(none) shared(dimensions, alpha)
  {
    std::vector<double> point(dimensions);
#pragma omp for
    for (int i = 0; i < num_points_; ++i) {
      for (int di = 0; di < dimensions; ++di) {
        double val = 0.5 + (static_cast<double>(i + 1) * alpha[di]);
        double current = val - std::floor(val);
        point[di] = lower_bounds_[di] + ((upper_bounds_[di] - lower_bounds_[di]) * current);
      }
      sum += IntegrandFunction::Evaluate(func_type_, point);
    }
  }
```

Расшифровка директив:

- **`#pragma omp parallel`** — стартует параллельный регион; число потоков
  определяется средой OpenMP. Курсовой runner экспортирует
  `OMP_NUM_THREADS` из `PPC_NUM_THREADS`, поэтому фактическое число потоков
  равно тому, что задано в `PPC_NUM_THREADS` при `--running-type=threads`.
- **`reduction(+ : sum)`** — каждый поток получает приватную копию `sum`,
  накапливает её, а в конце региона значения суммируются в
  глобальную переменную; гонок по записи нет, и порядок суммирования не
  фиксирован, но допустим, потому что задача численная и математически
  ассоциативная по конечному вкладу (аналитическая ошибка стандартного
  Монте-Карло \(O(1/\sqrt n)\) не зависит от порядка обхода).
- **`default(none)`** — заставляет явно перечислять переменные в
  `shared`/`private`/`reduction`. Это требование принято курсом и
  предотвращает неявные гонки. Поля класса (`lower_bounds_`, `upper_bounds_`,
  `num_points_`, `func_type_`) корректно видны из лямбды через `this`,
  поэтому их перечислять в `shared` не нужно — достаточно `dimensions` и
  `alpha`, которые объявлены локальной переменной/вектором в `RunImpl`.
- **`#pragma omp for`** без явного `schedule` — компилятор выбирает
  `schedule(static)` как реализационно дефолтное; для embarrassingly parallel
  `for` с одинаковой стоимостью итераций это правильный выбор:
  балансировка не нужна, а статическое разбиение даёт минимальные
  накладные расходы.
- **Барьер** — неявно стоит в конце `omp for` и в конце `omp parallel`;
  это гарантирует, что главная нить не прочитает `sum` до того, как
  все рабочие нити закончат своё накопление и редукция выполнится.

`std::vector<double> point(dimensions)` объявлен **внутри** блока
`#pragma omp parallel` — в OpenMP это **firstprivate-by-construction**:
у каждого потока свой `point`, никакой ложной разделяемости (false sharing)
по этому буферу нет.

**Наблюдения.**

1. До 8 потоков — почти линейный рост, \(E \ge 86\%\) в `task_run`.
   Это подтверждает, что цикл действительно embarrassingly parallel и
   накладные расходы создания/синхронизации параллельной области малы по
   сравнению с полезной работой.
2. **Sweet spot при `T = 12`.** \(S \approx 9{,}3\), \(E \approx 77\%\) —
   это **практический потолок** OpenMP на M4 Max: ровно 12 потоков
   используют ровно 12 P-ядер (E-ядра не задействуются), все потоки
   заканчивают работу синхронно, и неявный барьер `omp for` ничего не
   ждёт. Это лучшая точка по `S` среди всех конфигураций OpenMP.
3. **На `T = 16` производительность даже хуже, чем на `T = 12`**:
   \(S = 8{,}54\) против \(9{,}26\) при `T=12`. Причина та же, что в
   прошлой версии отчёта: 4 E-ядра подключаются, но они в ~2 раза
   медленнее P-ядер, поэтому весь регион ждёт «отстающих» на барьере
   `omp for`. Эффективность падает до 53% не потому, что код плох, а
   потому, что считается \(E = S/16\), хотя «полных» ядер у задачи 12.
4. `task_run` и `pipeline` дают практически одинаковый результат при
   любом \(T\) — для лёгкой задачи Монте-Карло разница между этими
   режимами укладывается в шум.

## 9. Выводы

- OpenMP-версия даёт **ускорение, близкое к линейному**, до 8 потоков и
  достигает абсолютного максимума при `T = 12` (\(S \approx 9{,}3\),
  \(E \approx 77\%\)) — это «потолок» OpenMP на M4 Max.
- На `T = 16` ускорение **ухудшается** относительно `T = 12`, а не
  растёт: подключение 4 медленных E-ядер замедляет неявный барьер
  `omp for`. Это особенность железа Apple Silicon, а не кода; на
  однотипных x86-ядрах та же программа должна сохранять более ровную
  эффективность.
- В предметной задаче (Монте-Карло) OpenMP — действительно «самый
  дешёвый» способ распараллеливания, как и подсказывал преподаватель:
  одна директива `#pragma omp parallel for reduction(+:sum)` даёт почти
  оптимальное ускорение без ручного управления потоками.
