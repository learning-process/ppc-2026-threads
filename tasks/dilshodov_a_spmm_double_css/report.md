## 1. Введение

Задача — умножение разреженных квадратных матриц `C = A · B` с элементами типа `double`,
хранящихся в столбцовом разреженном формате CCS (Compressed Column Storage).
Реализованы пять backend-ов (SEQ, OMP, TBB, STL, ALL), функциональные и производительные
тесты — единые для всех версий. Эталоном корректности и baseline для ускорения служит
SEQ-реализация.

Задача удобна для сравнения моделей параллелизма по двум причинам:

1. Внешний цикл — по столбцам B — естественно даёт независимые единицы работы,
   что подходит и data-parallel моделям (OMP/TBB/STL), и task-distributed (MPI/ALL).
2. Внутри столбца идёт sparse accumulator с произвольным паттерном доступа к памяти —
   это нагрузка memory-bound, и каждая модель проявляет свои сильные и слабые стороны
   в управлении кешем и потоками.

## 2. Единая постановка задачи

**Вход:** `A` (m × k), `B` (k × n), обе в CCS:

- `col_ptrs[i+1] - col_ptrs[i]` = число ненулей в столбце `i`;
- `row_indices[col_ptrs[i] .. col_ptrs[i+1])` строго возрастают;
- `values[i]` — соответствующее значение типа `double`.

**Выход:** `C` (m × n) в том же CCS-формате.

**Ограничения:**

- `A.cols == B.rows`;
- структурная корректность входа (см. [seq/report.md §2](seq/report.md));
- численная фильтрация: значения с `|v| < 10⁻¹⁰` отбрасываются как численные нули.

**Критерий корректности:** сравнение с эталоном — побитное совпадение `col_ptrs`
и `row_indices`, разница значений < `10⁻⁶` (perf-тесты) или `10⁻⁸` (функциональные).

### 3.3 Данные

Все perf-замеры на одинаковом входе ([tests/performance/main.cpp](tests/performance/main.cpp)):

- Две банд-матрицы 2000 × 2000;
- Радиус ленты ±50 (то есть ~100 ненулей на колонку);
- Значения — случайные `double ∈ [1, 2]`, генерируемые `std::mt19937(42)`
  (фиксированный seed → воспроизводимость);
- Результирующая матрица содержит ~10⁵ ненулей.

Эталон считается локально функцией `ReferenceSpMM` — тот же sparse-accumulator алгоритм,
что в SEQ, но без фильтрации нулей (поскольку для положительных значений в `[1,2]`
нулей в произведении не возникает).

### 3.5 Режимы измерения и повторы

Инфраструктура `ppc::util::BaseRunPerfTests` запускает каждый тест в двух режимах:

- **pipeline:** многократный прогон с измерением среднего времени всего конвейера
  (Validation+PreProcessing+Run+PostProcessing);
- **task_run:** многократный прогон только фазы `Run` (чистое вычисление).

В отчётах используется `task_run` как основной показатель — он ближе к «времени самого
алгоритма» без overhead инфраструктуры.

## 4. Сводка корректности

| Backend | Func tests (3 шт) | Perf test (2000×2000) |
| --- | --- | --- |
| SEQ | ✅ | ✅ |
| OMP | ✅ | ✅ |
| TBB | ✅ | ✅ |
| STL | ✅ | ✅ |
| ALL | SKIPPED без `mpirun` | ✅ |

Эталон формируется через плотное умножение в функциональных тестах и через `ReferenceSpMM`
(последовательный sparse accumulator) в perf-тестах. Все параллельные версии дают побитно
идентичные `col_ptrs`/`row_indices` и значения с точностью до `1e-8`.

Структурные гарантии отсутствия гонок:

- **OMP/TBB:** разделяемая `column_results` адресуется уникальным индексом потока в каждой
  итерации; thread-local буферы изолированы;
- **STL:** статическое разбиение, каждый поток пишет только в свои `j`,
  `join` после всех `emplace_back`;
- **ALL:** локальные буферы на каждом ранге; MPI-коллективы детерминированы.

## 6. Интерпретация различий

### SEQ

Базовая реализация на `std::map<int,double>` — простая и читаемая, но проигрывает всем
параллельным версиям даже на одном потоке. Главный недостаток — лог-фактор от красно-чёрного
дерева на каждой вставке. SEQ нужен только как эталон.

### OMP

Самое короткое и понятное распараллеливание. `#pragma omp parallel default(none)` +
`#pragma omp for schedule(dynamic)` дают ≈25× ускорения на 8 потоках.
Эффективность падает с 80% до 40% — типично для memory-bound нагрузки на SMT.

### TBB

Чуть больше overhead на одном потоке (out-of-box `task_arena`), но **лучше всех
масштабируется**: эффективность 88%/74%/53% против 80%/69%/40% у OMP. Work-stealing oneTBB
справляется с неравномерной нагрузкой и SMT-конкуренцией успешнее, чем
`schedule(dynamic)` OpenMP.

### STL

**Самый быстрый однопоточный** (нет runtime-инфраструктуры) и **самый быстрый на 8 потоках**
в абсолютном выражении (0.0063 с). Но T=2 показал аномалию — масштабирование не сработало,
вероятно из-за планировщика Docker/WSL2. Низкая адаптивность к неравномерной нагрузке —
цена ручного разбиения без work-stealing.

### ALL

Без `mpirun` (1 процесс) ведёт себя как чуть-более-сложная OMP-версия — те же ~30× ускорения.
Под `mpirun` лучшая конфигурация `2×2` даёт 21× относительно SEQ, но **хуже** чистого
OMP/TBB/STL на 8 потоках. На одноузловой системе MPI-уровень избыточен: коммуникация через
`Allgatherv` стоит ощутимо больше, чем shared-memory работа OpenMP.

Конфигурация `4×2` ярко иллюстрирует это: при том же `total_workers=8`, что и чистый OMP@8
(0.0077 с), время гибрида 0.0130 с — почти вдвое хуже. Цена `Allgatherv` между 4 рангами
с ~400K ненулей вычета доминирует.

## 8. Заключение

**Какую версию считать «лучшей» — зависит от контекста:**

- Для одноузлового запуска с фиксированным числом ядер и предсказуемой нагрузкой —
  **TBB** (наилучшая эффективность, минимум кода для управления потоками);
- Для абсолютного минимума времени на текущей системе —
  **STL** (0.0063 с, ×31), но с оговоркой нестабильности T=2;
- Для read-only-понятности и минимальной зависимости —
  **OMP** (одна директива, ясный код);
- Для кластера / multi-node —
  **ALL** (единственный способ выйти за пределы одного узла).

**Ограничения сравнения:**

1. Замеры в Docker под WSL2 — есть шум планировщика.
   На bare-metal Linux числа были бы стабильнее.
2. Один размер задачи (2000×2000, банд ±50). На других паттернах разреженности или больших
   размерах относительные позиции backend-ов могут поменяться.
3. AMD Ryzen 5 5600 — одна NUMA-нода, что делает ALL менее эффективным; на NUMA-системах
   с несколькими сокетами гибридная схема выглядела бы выгоднее.
4. ALL хранит полный вход на каждом ранге — не масштабируется по памяти.
   Production-вариант делал бы scatter/broadcast явно.

**Возможные улучшения:**

- Параллельный `parallel_scan` для построения финального `col_ptrs`
  (ожидаемый эффект — менее 5%);
- Кеширование `acc/marker` между перфтестами для уменьшения аллокаций
  (видимый эффект в режиме task_run);
- Для ALL — заменить `Allgatherv` на `Gather` с последующим `Bcast`,
  если потребитель — только один процесс.

### 10.3 Замечание о методике

perf-тестов всех backend-ов и всех чисел потоков занимает менее минуты.
