## 1. Контекст

STL-реализация решает ту же задачу `C = A · B`, что и OMP/TBB,
но без использования внешней библиотеки параллелизма — только `std::thread` из стандарта C++.
Эта версия наиболее «инженерная»: она требует явного управления жизненным циклом потоков
и ручного разбиения работы.

## 2. Постановка задачи

Без изменений относительно [seq/report.md](../seq/report.md): два разреженных матричных
операнда в CCS, на выходе их произведение в том же формате.

## 3. Базовый алгоритм

Тот же sparse-accumulator паттерн что в OMP/TBB. Различие — только в способе диспетчеризации
работы между потоками.

## 4. Схема распараллеливания

- **Декомпозиция данных:** диапазон столбцов B `[0, cols_B)` делится на `num_threads`
  поддиапазонов **статически** по формуле:

  ```text
  j_start_t = (t * cols_B) / num_threads
  j_end_t   = ((t + 1) * cols_B) / num_threads
  ```

  Такое разбиение гарантирует, что каждый столбец достанется ровно одному потоку,
  а остаток распределится равномерно (округление в большую сторону у первых потоков).
- **Что делает поток:** функция-воркер `Worker(j_start, j_end, ...)` создаёт локальные
  `acc`, `marker`, `used_rows` и в цикле обрабатывает свой диапазон столбцов.
  Результаты пишутся в `column_results[j]` для своего `j`.
- **Локальные результаты:** не нужны отдельные «локальные buffer-ы для слияния» —
  каждая колонка независима, потоки пишут в разные ячейки `column_results`.
  Финальная склейка делается уже последовательно после `join`.
- **Число потоков:** берётся из `ppc::util::GetNumThreads()`, обычно равно `PPC_NUM_THREADS`.
  Дополнительно ограничивается сверху числом столбцов, чтобы не создавать пустые потоки.
- **Синхронизация:** только `join()` после запуска **всех** потоков (важно: не внутри цикла
  создания, иначе работа сериализуется — это типовая ошибка в учебном `example_threads/stl`).
  `atomic`, `mutex`, `condition_variable` не используются — гонок по построению нет.

Ключевой фрагмент:

```cpp
// File: stl/src/ops_stl.cpp
int num_threads = ppc::util::GetNumThreads();
if (num_threads > rhs.cols_count) {
  num_threads = std::max(1, rhs.cols_count);
}

std::vector<std::thread> threads;
threads.reserve(num_threads);
for (int tid = 0; tid < num_threads; ++tid) {
  const int j_start = (tid * rhs.cols_count) / num_threads;
  const int j_end   = ((tid + 1) * rhs.cols_count) / num_threads;
  if (j_start >= j_end) continue;
  threads.emplace_back(Worker, j_start, j_end,
                       std::cref(lhs), std::cref(rhs),
                       std::ref(column_results));
}
for (auto &th : threads) th.join();
```

Замечания по аргументам `std::cref` / `std::ref`:

- `lhs`/`rhs` передаются как `const&` через `std::cref`, иначе `std::thread` по умолчанию
  скопировал бы матрицы целиком в каждый поток;
- `column_results` — через `std::ref`, потому что в неё пишут
  (но в разные индексы, поэтому гонки нет).

## 5. Детали реализации

**Файлы:** [stl/include/ops_stl.hpp](include/ops_stl.hpp), [stl/src/ops_stl.cpp](src/ops_stl.cpp).

Карта «поток → диапазон столбцов» при `num_threads=4` и `cols_B=2000`:

| Thread | j_start | j_end | Number of columns |
| ---: | ---: | ---: | ---: |
| 0 | 0 | 500 | 500 |
| 1 | 500 | 1000 | 500 |
| 2 | 1000 | 1500 | 500 |
| 3 | 1500 | 2000 | 500 |

Воркер хранит локальные `acc(m, 0.0)`, `marker(m, -1)`, `used_rows` на стеке —
нет аллокаций между итерациями.

Схема объединения локальных результатов: после `join` всех потоков делается последовательный
проход `BuildOutputFromColumns` (тот же, что в OMP/TBB).

## 6. Проверка корректности

Все три функциональных теста проходят. Гонки исключены архитектурно:

- `column_results[j]` — каждый поток пишет только в свои `j ∈ [j_start, j_end)`;
- локальные `acc`, `marker`, `used_rows` живут на стеке воркера;
- `lhs`, `rhs` только читаются;
- `join` стоит после всех `emplace_back`, поэтому потоки действительно работают параллельно
  (если бы `join` был внутри цикла — мы получили бы сериальное исполнение).

## 8. Результаты

| Threads | STL, task_run | STL, pipeline | Speedup vs SEQ@1 | Speedup vs STL@1 | Efficiency vs STL@1 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 1 | 0.0213 | 0.0241 | 9.17 | 1.00 | 100% |
| 2 | 0.0215 | 0.0151 | 9.09 | 0.99 | 50% |
| 4 | 0.0075 | 0.0110 | 26.05 | 2.84 | 71% |
| 8 | 0.0063 | 0.0100 | 31.02 | 3.38 | 42% |

**Комментарий.**

- STL@1 — **самая быстрая** среди однопоточных версий (0.0213 с), даже быстрее OMP@1 (0.0246 с)
  и TBB@1 (0.0288 с). Объясняется отсутствием runtime-инфраструктуры: один поток просто
  вызывает функцию напрямую без накладных расходов на task arena или parallel-регион.
- На 2 потоках наблюдается **аномалия**: время не уменьшилось (0.0215 ≈ 0.0213).
  Возможная причина — шум измерений в Docker под WSL2 или планировщик, не использующий
  второй CPU. На pipeline-режиме (0.0151 c) поведение более ожидаемое — Speedup ≈ 1.42.
- На 4 и 8 потоках масштабирование возвращается в норму. На 8 потоках STL —
  **самый быстрый** из чистых thread-backend-ов (0.0063 с против 0.0068 у TBB и 0.0077 у OMP).
- Эффективность относительно STL@1: 50%/71%/42% — нестабильно по сравнению с TBB.

Аномалия T=2 в `task_run` объясняется тем, что инфраструктура `task_run` в perf-тестах
запускает задачу один раз, а `pipeline` — несколько; в первом случае случайный пик
планировщика виден целиком. Это типичная цена ручного управления потоками
без work-stealing рантайма.

## 9. Выводы

`std::thread` показывает себя как «низкоуровневый, но конкурентоспособный» инструмент:
минимальный overhead на одном потоке, отличный итоговый результат на 8 потоках,
но плохая адаптивность при неравномерной нагрузке
(видно по «пилообразному» масштабированию T=1→2→4).

Преимущества: нет внешних зависимостей, поведение полностью предсказуемо.
Недостатки: ручное разбиение, нет work-stealing — если одна часть диапазона тяжелее другой,
потоки простаивают; именно поэтому при банд-матрицах со слабо неравномерной плотностью
эффект меньше, чем у TBB.

Замеры в Docker под WSL2 имеют дополнительный источник шума — вложенный планировщик;
в реальной нативной Linux-системе показатели должны быть более гладкими.
