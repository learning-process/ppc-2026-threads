## 1. Контекст

Реализация переносит цикл по столбцам результата `C = A · B` на OpenMP.
Дополнительно по сравнению с SEQ заменён словарь-аккумулятор `std::map<int,double>`
на массив-аккумулятор с маркерами (sparse accumulator, SPA) —
это даёт алгоритмическое ускорение даже на одном потоке, поверх которого добавляется
параллельный выигрыш.

## 2. Постановка задачи

Та же, что в [seq/report.md](../seq/report.md): на входе две разреженные матрицы в CCS,
на выходе их произведение в том же формате.
Все инварианты структуры и численная точность совпадают.

## 3. Базовый алгоритм

Внешний цикл — по столбцам `j` матрицы `B`, каждый столбец независим от других,
поэтому именно по ним проходит граница распараллеливания.
Внутри столбца применяется тот же sparse-accumulator паттерн:

```text
для каждого j:
    used_rows = []
    для каждого ненулевого (k, B[k][j]):
        для каждого ненулевого (i, A[i][k]):
            если marker[i] != j:   # ещё не трогали в этом столбце
                marker[i] = j
                acc[i] = A[i][k]*B[k][j]
                used_rows.push_back(i)
            иначе:
                acc[i] += A[i][k]*B[k][j]
    отсортировать used_rows, отбросить численные нули, записать в C[:, j]
```

**Асимптотика:** `O(flop / P)` времени и `O(m · P)` памяти на аккумуляторы
(по одному комплекту `acc`/`marker` на поток).

## 4. Схема распараллеливания

- **Область распараллеливания:** внешний `for j` по столбцам B.
- **Декомпозиция данных:** каждый поток получает свой набор столбцов `j` и обрабатывает
  их полностью независимо. Никаких разделяемых аккумуляторов нет.
- **`shared` переменные:** `lhs`, `rhs`, `column_results` — только чтение и запись в свои индексы.
- **`private`/локальные:** буферы `acc`, `marker`, `used_rows` объявлены **внутри**
  `#pragma omp parallel` блока, поэтому автоматически приватны.
- **Reduction:** не используется — результирующие столбцы пишутся в `column_results[j]`,
  где `j` уникален для потока в данной итерации.
- **Schedule:** `schedule(dynamic)`. Выбран потому что число ненулей на колонку
  (и, следовательно, время обработки) сильно варьируется в реальных разреженных матрицах.
  Статическое разбиение может оставить часть потоков без работы,
  если плотные колонки сконцентрированы в одном диапазоне.
- **Синхронизация:** только неявный барьер в конце `omp for`. После него рабочий поток
  выходит из `parallel`-региона; объединение `column_results[]` в окончательный CCS-выход
  делается уже последовательно в `BuildOutputFromColumns`. Это нужно потому что в CCS-формате
  `col_ptrs` обязан быть монотонной префиксной суммой, что является inherently sequential
  шагом, но он линеен по числу столбцов и недорог.

Ключевой фрагмент:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel default(none) shared(lhs, rhs, column_results)
{
  std::vector<double> acc(lhs.rows_count, 0.0);
  std::vector<int>    marker(lhs.rows_count, -1);
  std::vector<int>    used_rows;
  used_rows.reserve(256);

#pragma omp for schedule(dynamic)
  for (int rhs_col = 0; rhs_col < rhs.cols_count; ++rhs_col) {
    auto &col = column_results[rhs_col];
    AccumulateColumnProduct(lhs, rhs, rhs_col, acc, marker, used_rows, col);
  }
}
```

Расшифровка директивы:

- `default(none)` обязывает явно перечислить все shared-объекты —
  защита от случайного захвата лишних переменных;
- `shared(lhs, rhs, column_results)` — два входа только читаются,
  в `column_results[j]` каждый поток пишет в свою уникальную ячейку без гонки;
- буферы внутри `parallel` — приватны без явного `private(...)`;
- `schedule(dynamic)` — динамическое разбирание итераций для устойчивости
  к неравномерной плотности столбцов.

## 5. Детали реализации

**Файлы:** [omp/include/ops_omp.hpp](include/ops_omp.hpp), [omp/src/ops_omp.cpp](src/ops_omp.cpp).

Относительно SEQ изменено:

1. `std::map` заменён массивом-аккумулятором `acc[m]` + массивом маркеров `marker[m]`,
   инициализированных в `-1`. Маркер сравнивается с текущим номером столбца, что заменяет
   «проверку существования ключа» — без перезаписи всего массива нулями между колонками.
2. `used_rows` хранит список тронутых строк, чтобы по окончании итерации не сканировать весь
   `acc`. После обработки колонки `used_rows` сортируется (`std::ranges::sort`) — это нужно
   для соблюдения инварианта возрастания `row_indices` в CCS.
3. Финальная сборка вынесена в `BuildOutputFromColumns`, которая последовательно проходит
   `column_results` и формирует общий `col_ptrs`, `row_indices`, `values`.
   Это сериальная часть, ограничивающая масштабируемость по закону Амдала.

Точка гонки исключена структурно: каждый поток пишет только в `column_results[j]` для своего `j`,
и потом сериальный шаг лишь читает уже завершённые данные.

## 6. Проверка корректности

Функциональные тесты проходят для всех трёх параметров
(`TwoByTwoBasic`, `ThreeByThreeSparse`, `RectangularCheck`) —
выход побайтно совпадает с эталоном, посчитанным из плотного умножения.
Производительный тест дополнительно сверяет результат с локально посчитанным SEQ-эталоном
на матрицах 2000×2000 — все три прогона (pipeline, task_run × 4 значения PPC_NUM_THREADS)
прошли успешно.

Отсутствие гонок проверено логически: разделяемая `column_results` адресуется по уникальному
индексу потока в каждой итерации; локальные `acc`/`marker`/`used_rows` приватны.
Дополнительная гарантия — `default(none)` отсёк бы любой shared-захват за пределами
явного списка.

## 8. Результаты

Время `task_run` (медиана по нескольким прогонам инфраструктуры perf-теста, секунды):

| Threads | OMP, task_run | OMP, pipeline | Speedup vs SEQ@1 | Speedup vs OMP@1 | Efficiency vs OMP@1 |
| ---: | ---: | ---: | ---: | ---: | ---: |
| 1 | 0.0246 | 0.0251 | 7.94 | 1.00 | 100% |
| 2 | 0.0153 | 0.0215 | 12.77 | 1.61 | 80% |
| 4 | 0.0089 | 0.0134 | 21.96 | 2.76 | 69% |
| 8 | 0.0077 | 0.0109 | 25.38 | 3.20 | 40% |

(SEQ@1 = 0.1954 с — baseline из [seq/report.md](../seq/report.md).)

**Комментарий о масштабируемости.**

- Базовое ускорение OMP@1 относительно SEQ — 7.94 — почти полностью алгоритмическое:
  sparse accumulator уже на одном потоке устраняет лог-фактор от `std::map`.
- Параллельный speedup чистого OMP относительно OMP@1: 1.61 → 2.76 → 3.20
  для 2/4/8 потоков на 6 физических ядрах. Эффективность падает с 80% до 40%.
- Падение на 8 потоках объясняется тем, что SMT-потоки делят функциональные блоки ядра;
  в memory-bound нагрузке (а sparse-mult сильно зависит от latency обращений к `acc`/`marker`)
  гипертрединг даёт ограниченный выигрыш.
- Сериальная часть `BuildOutputFromColumns` (один проход по колонкам, аллокация
  `row_indices`/`values`) и кеш-промахи на `acc`/`marker` размером `m=2000` — основные
  ограничители.

## 9. Выводы

OpenMP даёт самое прямолинейное распараллеливание задачи: одна директива
`#pragma omp parallel` + `#pragma omp for schedule(dynamic)`,
тривиальное приватирование буферов, нулевые требования к синхронизации помимо неявного барьера.
Алгоритмический выигрыш SPA + параллельный по столбцам даёт суммарное ускорение ≈25×
относительно SEQ на 8 потоках.

Ограничения: эффективность падает с ростом числа потоков из-за memory-bound характера
и сериальной фазы сборки результата. Если бы матрица была плотнее или столбцы — длиннее,
эффективность держалась бы выше; на текущем профиле задачи `dynamic`-расписание оправдывает
себя для устойчивости к неравномерной плотности.
