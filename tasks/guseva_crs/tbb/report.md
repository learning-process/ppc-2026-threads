## 1. Контекст

Данный отчёт описывает реализацию умножения разреженных матриц в формате CRS с
использованием стандартных потоков C++ (`std::thread`). В отличие от OpenMP и
TBB, STL требует явного управления потоками — их создания, распределения работы
и синхронизации. Эта реализация демонстрирует, что можно достичь
производительности, сравнимой со специализированными фреймворками, используя
только стандартную библиотеку.

## 2. Постановка задачи

- **Вход:** Матрицы A и B в CRS формате.
- **Выход:** C = A × B в CRS формате.
- **Ограничения:** A.ncols == B.nrows. |value| < 10⁻⁵ → элемент считается
  нулевым.

## 3. Базовый алгоритм (последовательный)

См. `seq/report.md`. STL-версия распределяет строки матрицы A между потоками.

## 4. Схема распараллеливания

**4.1 Как данные делятся на диапазоны:** Статическое распределение строк.
Функция `ComputeRanges` вычисляет для каждого потока начальный и конечный
индексы строк.

**4.2 Что делает каждый поток:** Поток получает диапазон `[start_row, end_row)`.
Для каждой строки в этом диапазоне поток выполняет полное умножение:
инициализирует маркерный массив, вычисляет скалярные произведения со всеми
столбцами Bᵀ, добавляет ненулевые результаты в локальный буфер.

**4.3 Где хранятся локальные результаты:** Каждый поток строит локальные векторы
`local_columns`, `local_values` и `local_row_index` для своих строк. После
завершения всех потоков агрегируется общий результат.

**4.4 Как и когда выполняется join:** Все потоки создаются в цикле и сохраняются
в `std::vector<std::thread>`. После цикла создания выполняется второй цикл:
`for (auto& th : threads) th.join();`. Это гарантирует, что все потоки
завершились до начала агрегации.

**4.5 Какая синхронизация используется:** Синхронизация не требуется на этапе
вычислений (каждый поток пишет только в свои структуры). Единственная точка
синхронизации — `join()`. Атомарные операции и мьютексы не используются.

**4.6 Порог последовательного выполнения:** Если число строк меньше
4\*num_threads, последовательная версия выполняется без создания потоков
(избегаем overhead на малых задачах).

## 5. Детали реализации

- **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`,
  `stl/include/multiplier_stl.hpp`
- **Функция распределения:** `ComputeRanges` — делит строки на `num_threads`
  блоков, остаток распределяется первым потокам.
- **Лямбда-функция потока:** Каждый поток выполняет `ProcessRows` для своего
  диапазона.

**Ключевой фрагмент (создание потоков и join):**

```cpp
// Файл: stl/include/multiplier_stl.hpp
std::vector<std::thread> threads;
std::vector<LocalResult> local_results(num_threads);

for (size_t t = 0; t < num_threads; ++t) {
    size_t start = ranges[t].first;
    size_t end = ranges[t].second;
    threads.emplace_back([&, t, start, end]() {
        ProcessRows(a, bt, start, end, local_results[t]);
    });
}
for (auto& th : threads) {
    th.join();
}
// Агрегация local_results в финальный результат
```

## 6. Проверка корректности

- **Сравнение с SEQ:** Результат идентичен на всех функциональных тестах.
- **Проверка детерминизма:** При фиксированном числе потоков и фиксированных
  диапазонах результат воспроизводим.
- **Отсутствие гонок:** Каждый поток работает со своим `LocalResult`. Маркерный
  массив `temp` — локальная переменная внутри `ProcessRows`.
- **ThreadSanitizer:** `-fsanitize=thread` не выявил проблем.

## 9. Выводы

STL-реализация на `std::thread` достигает производительности, практически
идентичной TBB и OpenMP (разница ≤0.6%), при этом не требуя никаких внешних
зависимостей (кроме стандартной библиотеки C++17). Явное управление потоками
(`create → work → join`) добавляет объём кода, но даёт полный контроль над
распределением строк. Рекомендуется для максимально портативных решений.
