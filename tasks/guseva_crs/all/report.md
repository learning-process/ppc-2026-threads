## 1. Контекст

Данный отчёт описывает гибридную реализацию умножения разреженных матриц в
формате CRS с использованием MPI (для распределённой памяти) и OpenMP (для
разделяемой памяти). Гибридный подход необходим для сортировки наборов данных,
не помещающихся в оперативной памяти одного узла, и позволяет эффективно
использовать кластерные конфигурации.

## 2. Постановка задачи

- **Вход:** Матрицы A и B в CRS формате. На root-процессе (rank 0) хранятся
  полные матрицы.
- **Выход:** C = A × B в CRS формате. Результат собирается на root-процессе и
  рассылается всем процессам.
- **Ограничения:** A.ncols == B.nrows. |value| < 10⁻⁵ → элемент считается
  нулевым.

## 3. Базовый алгоритм (последовательный)

См. `seq/report.md`. Гибридная версия дополняется распределением строк A между
MPI процессами и OpenMP-параллелизацией внутри каждого процесса.

## 4. Межпроцессная схема (MPI)

**4.1 Роли rank-ов:** Все rank-и равноправны в вычислительном смысле. Rank 0
выполняет функции: (1) хранение исходных матриц, (2) распределение строк A между
процессами, (3) сборка результата, (4) рассылка результата всем процессам.

**4.2 Распределение данных:** Блоковое распределение строк A. Формула:

```text
base = a.nrows / num_procs
rem = a.nrows % num_procs
local_nrows = base + (rank < rem ? 1 : 0)
start_row = rank * base + min(rank, rem)
```

**4.3 MPI-вызовы:**

- `MPI_Comm_rank` / `MPI_Comm_size` — определение ранга и числа процессов
- `MPI_Send` / `MPI_Recv` — отправка локальных результатов от не-root процессов
  на root
- `MPI_Bcast` — рассылка готовой матрицы C от root всем процессам
- `MPI_Barrier` — опционально для синхронизации перед сборкой

**4.4 Коммуникационная схема:**

1. Root-процесс распределяет строки A между процессами (локальная копия A для
   каждого процесса)
2. Матрица B (транспонированная) реплицируется на все процессы
3. Каждый процесс вычисляет свои строки результата (с использованием OpenMP
   внутри)
4. Не-root процессы отправляют локальные результаты root-процессу
5. Root-процесс собирает результат и рассылает всем через `MPI_Bcast`

## 5. Внутрипроцессная схема (OpenMP)

**5.1 Какая thread-технология используется:** OpenMP. Внутри каждого MPI
процесса локальные строки обрабатываются параллельно.

**5.2 Как делится работа между потоками:**
`#pragma omp parallel for schedule(static)` распределяет `local_nrows` строк
между OpenMP-потоками.

**5.3 Как объединяются локальные результаты:** Каждый OpenMP-поток строит свой
`LocalResult` (колонки, значения, row_ptrs для своих строк). После завершения
`parallel for` эти части объединяются в `LocalProcessResult`. Затем MPI
отправляет `LocalProcessResult` на root.

## 6. Детали реализации

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`,
  `all/include/multiplier_all.hpp`
- **Структура `LocalProcessResult`:** `columns`, `values`, `row_index` —
  локальные результаты для процесса.
- **Ключевые функции:**
  - `ComputeRowDistribution` — вычисление `start_row` и `local_nrows` для
    каждого процесса
  - `ProcessLocalRows` — обработка локальных строк с OpenMP
  - `SendLocalResult` / `ReceiveLocalResult` — MPI-коммуникация

**Ключевой фрагмент (гибридный цикл):**

```cpp
// Файл: all/include/multiplier_all.hpp
int rank, size;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

// Распределение строк
size_t local_nrows = ComputeLocalRows(rank, size, a.nrows, start_row);

// OpenMP-параллелизация внутри процесса
#pragma omp parallel default(none) shared(a, bt, local_results, start_row, local_nrows) private(i)
{
    #pragma omp for schedule(static)
    for (size_t i = 0; i < local_nrows; ++i) {
        size_t global_row = start_row + i;
        std::vector<int> temp(a.ncols, -1);
        // ... умножение строки global_row
    }
}
// MPI-коммуникация (только после завершения OpenMP)
if (rank == 0) {
    // собрать результаты от всех процессов
} else {
    MPI_Send(...);
}
MPI_Bcast(final_result, ...);
```

## 7. Проверка корректности

- **Сравнение с SEQ:** Результат на root-процессе идентичен последовательному на
  всех тестах.
- **Проверка распределения строк:** Сумма `local_nrows` по всем процессам равна
  `a.nrows`, строки не пересекаются.
- **MPI-коммуникация:** Проверка отсутствия deadlock-ов (симметричные
  `Send`/`Recv`).
- **OpenMP + MPI:** Отсутствие гонок (каждый процесс имеет свою память, внутри
  процесса — thread-local `temp`).

### 9.3 Анализ

- **Лучшая гибридная конфигурация:** 2×4 (2 MPI процесса × 4 OpenMP потока) —
  overhead всего 0.87%
- **Ухудшение с ростом числа процессов:** overhead растёт из-за увеличения числа
  MPI сообщений и сериализации/десериализации данных
- **MPI vs OpenMP внутри одного узла:** OpenMP (1×8) быстрее из-за отсутствия
  коммуникационных накладных расходов
- **Когда ALL необходим:** Когда матрица A не помещается в памяти одного узла —
  распределение строк позволяет обрабатывать данные на нескольких узлах

## 10. Выводы

Гибридная MPI+OpenMP реализация демонстрирует overhead всего 0.9–1.8% по
сравнению с pure OpenMP на одном узле, что является приемлемой ценой за
возможность распределённых вычислений. Оптимальная конфигурация на одном узле —
2×4 (минимизирует число MPI процессов, сохраняя возможность использовать все
ядра). Рекомендуется для:

1. Матриц, не помещающихся в памяти одного узла
2. Кластерных сред с несколькими узлами
3. NUMA-архитектур (один процесс на NUMA-узел)

Для данных, помещающихся в память одного узла, pure OpenMP/TBB/STL
предпочтительнее из-за меньшей сложности кода и отсутствия коммуникационных
overhead.
