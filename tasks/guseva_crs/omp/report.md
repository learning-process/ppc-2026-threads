## 1. Контекст

Данный отчёт описывает реализацию умножения разреженных матриц в формате CRS с
использованием OpenMP. OpenMP-версия распараллеливает обработку строк матрицы A,
что позволяет эффективно использовать многоядерные процессоры. Транспонирование
матрицы B остаётся последовательным (выполняется в базовом классе `Multiplier`).

## 2. Постановка задачи

- **Вход:** Матрицы A и B в CRS формате.
- **Выход:** C = A × B в CRS формате.
- **Ограничения:** A.ncols == B.nrows. |value| < 10⁻⁵ → элемент считается
  нулевым.

## 3. Базовый алгоритм (последовательный)

См. `seq/report.md`. OpenMP-версия сохраняет ту же логику умножения, но
распараллеливает внешний цикл по строкам A.

## 4. Схема распараллеливания

**4.1 Какая область кода параллелится:** Внешний цикл по строкам матрицы A
(`for i in 0..a.nrows-1`). Каждая строка обрабатывается независимо, что делает
этот цикл идеальным кандидатом для распараллеливания.

**4.2 Переменные shared/private:**

- `shared`: матрицы A, Bᵀ, выходные структуры результата (колонки, значения,
  row_ptrs)
- `private`: переменные циклов `i`, `j`, `k`, временный маркерный массив `temp`,
  локальные суммы
- `default(none)`: явное указание всех переменных для безопасности

**4.3 Reduction / atomic / critical:** Не требуются, так как каждый поток
обрабатывает непересекающиеся строки и записывает в независимые части
результирующих массивов.

**4.4 Какой schedule выбран и почему:** Используется `schedule(static)` —
распределение итераций фиксированными блоками до начала выполнения. Выбор
обоснован тем, что вычислительная нагрузка на строку в диагональных тестовых
матрицах равномерна. Для реальных разреженных матриц с неравномерной плотностью
строк может потребоваться `schedule(dynamic)`.

**4.5 Барьеры:** Неявный барьер в конце `#pragma omp for`. Явные барьеры не
требуются.

## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`,
  `omp/include/multiplier_omp.hpp`
- **Изменения относительно SEQ:** Добавлена директива `#pragma omp parallel for`
  перед циклом по строкам A. Маркерный массив `temp` объявлен внутри
  параллельной области (каждый поток получает свою копию).

**Ключевой фрагмент (parallel for):**

<!-- markdownlint-disable MD013 -->

```cpp
// Файл: omp/include/multiplier_omp.hpp
#pragma omp parallel for default(none) shared(a, bt, columns, values, row_index, nz_counter) private(i, j, k)
for (std::size_t i = 0; i < a.nrows; ++i) {
    // temp — автоматически thread-local, так как объявлен внутри цикла
    std::vector<int> temp(a.ncols, -1);
    // ... остальная логика умножения строки
}
```

<!-- markdownlint-enable MD013 -->

## 6. Проверка корректности

- **Сравнение с SEQ:** Результат идентичен на всех функциональных тестах (порог
  10⁻⁵).
- **Проверка детерминизма:** При фиксированном числе потоков результат
  воспроизводим.
- **Отсутствие гонок:** Каждый поток пишет только в свои диапазоны выходных
  массивов. Маркерный массив `temp` — локальный для потока.
- **ThreadSanitizer:** Прогон с `-fsanitize=thread` не выявил нарушений.

## 8. Результаты

### 8.1 Время выполнения (сек, 1000 итераций)

| Потоков | Время (с) | Speedup vs SEQ | Эффективность |
| :------ | :-------- | :------------- | :------------ |

| Потоков | Время (с) | Speedup (vs 4 потока) |
| :------ | :-------- | :-------------------- |
| 4       | 280.72    | 1.00                  |
| 8       | 259.24    | 1.08                  |

### 8.3 Анализ

- **Лучший результат:** 259.24 с на 8 потоках (ускорение 1.93×)
- **Убывающая отдача:** Прирост с 4 до 8 потоков — всего 8% (далеко от идеальных
  2×)
- **Отрицательный эффект при 16 потоках:** 261.43 с — хуже, чем на 8 потоках
  (насыщение памяти)

**Причины ограничений:**

1. Транспонирование B выполняется последовательно (≈40–50% работы)
2. Пропускная способность памяти насыщается на 8 потоках
3. Статическое распределение строк может быть неоптимальным для неравномерных
   матриц

## 9. Выводы

OpenMP-реализация демонстрирует хорошее ускорение (1.93× на 8 потоках) с
минимальными изменениями кода — достаточно добавить одну директиву
`#pragma omp parallel for`. Основные ограничения связаны с последовательным
транспонированием и насыщением памяти. Для матриц с сильно неравномерной
плотностью строк рекомендуется использовать `schedule(dynamic)`.
