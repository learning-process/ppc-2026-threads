# Вычисление многомерных интегралов по методу Симпсона — ALL

- **Student**: Овсянников Никита
- **Technology**: ALL (Hybrid: MPI + OMP + TBB + std::thread)
- **Variant**: 11

## 1. Контекст

Данная реализация является наиболее сложной и объединяет все изученные технологии
параллельного программирования. Целью является создание масштабируемого решения,
способного работать как на распределенных системах (кластерах), так и эффективно
использовать многоядерность внутри каждого вычислительного узла.

## 2. Постановка задачи

Краткая постановка совпадает с базовой (SEQ). Технология ALL требует обеспечения
согласованности результатов между всеми процессами MPI и потоками внутри них. В
качестве эталона используется последовательная версия (SEQ) со временем 0.023865 с.

## 3. Базовый алгоритм

Алгоритм реализует двумерную схему Симпсона. В гибридной версии вычисления
распределяются иерархически: сначала между процессами, затем между ядрами.

## 4. Межпроцессная схема (MPI)

- **Разделение:** Внешний цикл по строкам сетки ($nx + 1$) делится между процессами
  MPI. Каждый процесс вычисляет локальный диапазон итераций `[my_start, my_end)`.
- **Сборка:** По завершении расчетов используется функция `MPI_Reduce` с флагом
  `MPI_SUM` для накопления частичных сумм на процессе с рангом 0.
- **Синхронизация:** Использована функция `MPI_Bcast` для рассылки итогового
  значения всем процессам, что необходимо для корректной работы GTest-валидации
  на каждом узле.

## 5. Внутрипроцессная схема (OMP + TBB)

- **OpenMP:** Внутри локального диапазона MPI-процесса строки распределяются между
  потоками с помощью директивы `#pragma omp parallel for` с редукцией.
- **TBB:** Для каждой конкретной строки вычисление внутреннего интеграла (цикл по $j$)
  выполняется через `tbb::parallel_reduce`, что демонстрирует возможность
  вложенного параллелизма.
- **std::thread:** Добавлен вспомогательный блок с `std::atomic` счетчиком для
  валидации работы стандартных потоков в гибридном окружении.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.
Для обеспечения корректности на любом количестве процессов реализована формула:
`my_start = rank * rows_per_rank + min(rank, remainder)`. Это гарантирует
равномерную загрузку при любом размере сетки.

## 7. Проверка корректности

Корректность подтверждена функциональными тестами в MPI-окружении (запуск через
`mpiexec`). Все проверки на согласованность данных (`CheckTestOutputData`) пройдены.

## 8. Экспериментальная среда

- **CPU**: 13th Gen Intel(R) Core(TM) i5-13500H.
- **Конфигурация**: 2 процесса MPI, каждый использует до 8 потоков OpenMP/TBB.
- **Compiler**: MSVC v143.
- **Команды запуска**: `mpiexec -n 2 ./build/bin/Release/ppc_perf_tests.exe`.

## 9. Результаты

Замеры на сетке 2000x2000:

| Mode | Ranks x Threads | Time (s) | Speedup |
|------|-----------------|----------|---------|
| SEQ  | 1 x 1           | 0.0238   | 1.00    |
| ALL  | 2 x 8           | 0.0325   | 0.73    |

## 10. Выводы

Гибридная схема продемонстрировала стабильную работу и отсутствие конфликтов между
различными runtime-библиотеками (MPI, OMP, TBB). Наблюдаемое локально замедление
относительно SEQ связано с высокими накладными расходами на межпроцессное
взаимодействие (IPC) в ОС Windows при малом общем времени вычислений.
