## 1. Контекст
Данная реализация переносит базовый алгоритм поразрядной сортировки (SEQ) в многопоточную среду с использованием библиотеки Intel oneTBB (Threading Building Blocks). В отличие от OMP, TBB реализует парадигму пула потоков и управления задачами (Task-based parallelism), что может обеспечить более эффективное распределение нагрузки на уровне планировщика (work-stealing).

## 2. Постановка задачи
Задача идентична последовательной версии: упорядочить вектор целых чисел `std::vector<int>`, корректно обрабатывая отрицательные числа и дубликаты. SEQ-версия служит эталоном корректности (функциональные тесты) и базовой линией для расчета метрик ускорения и эффективности.

## 3. Базовый алгоритм
Базовым алгоритмом выступает LSD Radix Sort со сдвигом значений для поддержки отрицательных чисел. Асимптотика базового последовательного алгоритма составляет $O(d \cdot (n + b))$.

## 4. Схема распараллеливания
В реализации TBB используется явная декомпозиция данных с последующим слиянием, аналогичная OMP:
1. **Предварительное разбиение:** Входной массив делится на `num_threads` независимых блоков (chunks). Разбиение выполняется статически: размер базового блока `n / num_threads`, остаток распределяется равномерно.
2. **Параллельная фаза (`tbb::parallel_for`):** Создается задача для независимой сортировки каждого блока. Планировщик TBB распределяет эти задачи между потоками в пуле.
   * *Оптимизация:* Как и в OMP, внутри параллельной фазы используется поразрядная сортировка по основанию 256 (сдвиг `shift += 8`), что уменьшает константу проходов по сравнению с SEQ-версией.
3. **Синхронизация:** Примитив `tbb::parallel_for` обладает неявным барьером. Главный поток дожидается завершения обработки всех блоков.
4. **Последовательное слияние:** Отсортированные блоки последовательно сливаются в единый массив с помощью `std::ranges::merge`.

## 5. Детали реализации
Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Ключевой фрагмент распараллеливания в `RunImpl`:

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::parallel_for(0, num_threads, [&](int i) { RadixSort(local_data[i]); });

std::vector<int> result = std::move(local_data[0]);
for (int i = 1; i < num_threads; ++i) {
  std::vector<int> temp;
  temp.reserve(result.size() + local_data[i].size());
  std::ranges::merge(result, local_data[i], std::back_inserter(temp));
  result = std::move(temp);
}
```

*Пояснение к реализации:* Для распределения нагрузки используется самый простой вариант `tbb::parallel_for` с диапазоном от `0` до `num_threads`. В данном случае мы не используем `tbb::blocked_range`, так как ручное разбиение массива `local_data` на `num_threads` блоков уже выполнено на предыдущем шаге. TBB-планировщик просто назначает выполнение лямбда-функции для каждого индекса `i` доступным рабочим потокам.

## 8. Результаты
Замеры проводились на массиве в 5 000 000 элементов. Baseline (T1) взят из отчета SEQ.
Ускорение ($S = T_1 / T_p$), Эффективность ($E = S / p$).

**Режим: task_run**
| Потоки (p) | Время (секунды) | Ускорение (S) | Эффективность (E) |
|---|---|---|---|
| 1 (SEQ) | 0.111403 | 1.00 | 100% |
| 4 (TBB) | 0.022852 | ~4.87 | ~121% |
| 8 (TBB) | 0.025567 | ~4.35 | ~54% |

**Режим: pipeline**
| Потоки (p) | Время (секунды) | Ускорение (S) | Эффективность (E) |
|---|---|---|---|
| 1 (SEQ) | 0.110838 | 1.00 | 100% |
| 4 (TBB) | 0.035920 | ~3.08 | ~77% |
| 8 (TBB) | 0.046624 | ~2.37 | ~29% |

*Примечание:* Сверхлинейное ускорение на 4 потоках (эффективность > 100% в режиме `task_run`) вызвано переходом в параллельной функции на сортировку по основанию 256 (вместо основания 10 в SEQ-версии), что существенно сокращает константу алгоритма.

## 9. Выводы
Библиотека oneTBB показала отличные результаты на 4 потоках (ускорение в ~4.87 раза для task), обогнав даже реализацию OpenMP (0.022s против 0.026s). 

Однако, как и в случае с OpenMP, при увеличении числа потоков до 8 наблюдается **падение производительности** (время увеличилось с 0.022s до 0.025s). Причины остаются теми же:
1. Закон Амдала: Последовательное слияние блоков (`std::ranges::merge`) становится доминирующей статьей расходов при увеличении числа подмассивов.
2. Пропускная способность памяти (Memory Bound): Radix Sort требует интенсивного обмена данными с памятью (создание буферов, перестановка элементов), и 8 потоков создают конкуренцию за кэш и шину памяти.

Вывод: TBB отлично справляется с задачей, но для выбранного алгоритма слияния оптимальным является использование 4 потоков.
