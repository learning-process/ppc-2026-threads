## 1. Контекст

Реализация переносит базовый алгоритм в многопоточную среду с использованием Intel oneTBB.
В отличие от OMP, TBB реализует парадигму пула потоков и управления задачами (work-stealing).

## 2. Постановка задачи

Задача идентична последовательной версии: упорядочить вектор целых чисел. SEQ-версия
служит эталоном корректности и базовой линией для расчета метрик ускорения.

## 3. Базовый алгоритм

Базовым алгоритмом выступает LSD Radix Sort со сдвигом значений для отрицательных чисел.

## 4. Схема распараллеливания

Используется явная декомпозиция данных с последующим слиянием:

1. **Предварительное разбиение:** Входной массив делится на `num_threads` независимых блоков.
2. **Параллельная фаза:** Создается задача для независимой сортировки каждого блока.
Внутри применяется оптимизированная сортировка по основанию 256.
3. **Синхронизация:** Примитив `tbb::parallel_for` обладает неявным барьером.
4. **Последовательное слияние:** Отсортированные блоки сливаются в единый массив.

## 5. Детали реализации

```cpp
tbb::parallel_for(0, num_threads, [&](int i) { RadixSort(local_data[i]); });
```

*Пояснение к реализации:* Для распределения нагрузки используется упрощенный вариант
`tbb::parallel_for`. Мы намеренно не используем `tbb::blocked_range`, `grainsize` и
встроенные partitioner-ы, так как ручное макро-разбиение массива на `num_threads` блоков
уже выполнено. Планировщику TBB остается лишь назначить выполнение лямбда-функции
доступным потокам. Контроль конкуренции обеспечивается ограничением среды выполнения.

## 6. Проверка корректности

Функциональные тесты Google Test успешно пройдены для всех граничных сценариев.
Сортировка отрабатывает детерминированно при любом заданном количестве потоков.

## 9. Выводы

Библиотека oneTBB показала лучшие результаты на 4 потоках, обогнав реализацию OpenMP.
Однако при увеличении числа потоков до 8 наблюдается падение производительности из-за
затрат на последовательное слияние и конкуренции за шину памяти. Оптимально 4 потока.
