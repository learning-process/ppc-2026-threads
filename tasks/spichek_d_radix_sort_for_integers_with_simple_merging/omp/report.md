## 1. Контекст
Данная реализация переносит базовый алгоритм поразрядной сортировки (SEQ) в многопоточную среду с использованием директив OpenMP. Цель — ускорить обработку больших массивов данных за счет распределения независимых блоков данных между потоками с последующим слиянием локальных результатов.

## 2. Постановка задачи
Задача остается идентичной последовательной версии: упорядочить вектор целых чисел `std::vector<int>` с учетом наличия отрицательных значений и возможных дубликатов. Последовательная версия (SEQ) выступает в роли эталона для проверки корректности (базовых ответов) и расчета метрик ускорения.

## 3. Базовый алгоритм
В последовательной версии алгоритм выполняет LSD Radix Sort над всем массивом целиком, используя предварительный сдвиг для отрицательных чисел. Асимптотика последовательного эталона составляет $O(d \cdot (n + b))$. 

## 4. Схема распараллеливания
В OMP-реализации применена модель декомпозиции данных (Data Parallelism) с последующим простым слиянием:
1. **Разбиение:** Исходный массив делится на `num_threads` независимых блоков (chunks). Размер базового блока вычисляется как `n / num_threads`, остаток `n % num_threads` распределяется по первым потокам.
2. **Локальная сортировка (Параллельная фаза):** Каждый поток независимо выполняет `RadixSort` над своим подмассивом. 
   * *Важное алгоритмическое улучшение:* В отличие от SEQ (основание 10), в локальной сортировке OMP применена поразрядная сортировка по основанию 256 (сдвиг `shift += 8` и маска `& 255`). Это радикально уменьшает количество проходов по данным (константу $d$).
3. **Синхронизация:** В конце параллельного региона `#pragma omp parallel for` присутствует неявный барьер. Главный поток дожидается завершения локальных сортировок всеми рабочими потоками.
4. **Слияние (Последовательная фаза):** Главный поток последовательно объединяет отсортированные блоки `local_data` в итоговый массив с помощью `std::ranges::merge`.

## 5. Детали реализации
Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Ключевой фрагмент распараллеливания в `RunImpl`:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for num_threads(num_threads) default(none) shared(local_data, num_threads)
for (int i = 0; i < num_threads; ++i) {
  RadixSort(local_data[i]);
}

std::vector<int> result = std::move(local_data[0]);
for (int i = 1; i < num_threads; ++i) {
  std::vector<int> temp;
  temp.reserve(result.size() + local_data[i].size());
  std::ranges::merge(result, local_data[i], std::back_inserter(temp));
  result = std::move(temp);
}
```

*Пояснение к реализации:* Директива `default(none)` обязывает явно указывать атрибуты всех переменных. Массив `local_data` и переменная `num_threads` объявлены как `shared`. Состояния гонки (Data Race) не возникает, так как каждый поток с номером `i` (который является приватным по правилам цикла `for`) обращается строго к своему выделенному вектору `local_data[i]`. `schedule` явно не указан, применяется статическое распределение итераций по умолчанию, что идеально подходит, так как количество итераций равно количеству потоков.

## 8. Результаты
Замеры проводились на массиве в 5 000 000 элементов. Baseline (T1) взят из отчета SEQ.
Ускорение ($S = T_1 / T_p$), Эффективность ($E = S / p$).

**Режим: task_run**
| Потоки (p) | Время (секунды) | Ускорение (S) | Эффективность (E) |
|---|---|---|---|
| 1 (SEQ) | 0.111403 | 1.00 | 100% |
| 4 (OMP) | 0.026521 | ~4.20 | ~105% |
| 8 (OMP) | 0.028397 | ~3.92 | ~49% |

**Режим: pipeline**
| Потоки (p) | Время (секунды) | Ускорение (S) | Эффективность (E) |
|---|---|---|---|
| 1 (SEQ) | 0.110838 | 1.00 | 100% |
| 4 (OMP) | 0.036411 | ~3.04 | ~76% |
| 8 (OMP) | 0.043710 | ~2.53 | ~31% |

*Примечание:* Сверхлинейное ускорение (>100% эффективности) на 4 потоках в режиме `task_run` объясняется тем, что в OMP-версии локальная сортировка выполняется по основанию 256, в то время как SEQ-версия использует основание 10. Алгоритмическая оптимизация наложилась на многопоточность.

## 9. Выводы
Применение OpenMP дало существенный выигрыш производительности (в ~4 раза на 4 потоках для task).
Однако при переходе от 4 к 8 потокам наблюдается **деградация производительности** (время увеличилось с 0.026 до 0.028). Это обусловлено двумя факторами:
1. Закон Амдала: Последовательная фаза слияния блоков (`std::ranges::merge`) становится узким местом. Чем больше потоков, тем больше блоков нужно последовательно слить главному потоку.
2. Пропускная способность памяти (Memory Bandwidth): Процессор упирается в скорость чтения/записи оперативной памяти при одновременной работе 8 рабочих потоков, так как сортировка массивов сильно зависит от памяти.

Таким образом, для данной декомпозиции оптимальным числом рабочих потоков является 4.
