## 1. Контекст
Данная реализация (ALL) представляет собой гибридную модель распараллеливания, сочетающую межпроцессное взаимодействие (MPI) и внутрипроцессную многопоточность (oneTBB). Гибридный подход используется в распределенных системах, где MPI управляет распределением между узлами, а потоковые библиотеки максимально утилизируют ядра внутри каждого узла.

## 2. Постановка задачи
Задача состоит в сортировке массива `std::vector<int>` по неубыванию с учетом обработки отрицательных чисел и дубликатов. Последовательная версия (SEQ) служит эталоном корректности и базой (baseline) для вычисления метрик ускорения.

## 3. Базовый алгоритм
В основе лежит поразрядная сортировка (LSD Radix Sort). Асимптотика базового последовательного алгоритма составляет $O(d \cdot (n + b))$. Внутри параллельных веток (как и в чистом TBB) применена оптимизация: основание системы счисления увеличено до 256 (битовые сдвиги `shift += 8`).

## 4. Межпроцессная схема
В текущей реализации выбрана архитектура с явным выделением главного процесса (Master-Only Work):
1. **Роли рангов:** Ветвление осуществляется по `rank == 0`. Процесс с рангом 0 выступает в роли главного и единственного рабочего узла. Остальные процессы (`rank > 0`) не получают фрагментов данных и остаются в состоянии ожидания.
2. **Отсутствие обменов:** Так как вся нагрузка переложена на нулевой ранг, сложные MPI-коммуникации (`MPI_Scatter`, `MPI_Gather`) не используются.
3. **Синхронизация:** В конце метода `RunImpl` установлен общий барьер `MPI_Barrier(MPI_COMM_WORLD)`, который гарантирует, что все процессы покинут функцию одновременно после того, как главный ранг завершит вычисления.

## 5. Внутрипроцессная схема
Вся фактическая параллельная работа выполняется внутри процесса `rank == 0` с использованием библиотеки TBB:
1. Массив делится на `num_threads` независимых блоков с помощью вспомогательного метода `SplitData`.
2. Запускается пул потоков `tbb::parallel_for`, где каждый поток выполняет сортировку `RadixSort` над своим подмассивом.
3. Отсортированные блоки последовательно сливаются в единый массив с помощью функции `MergeData`.

## 6. Детали реализации
Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Ключевой гибридный фрагмент в `RunImpl`:

```cpp
// File: all/src/ops_all.cpp
int rank = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);

if (rank == 0 && !GetOutput().empty()) {
  int num_threads = ppc::util::GetNumThreads();
  // ... (проверка на необходимость распараллеливания) ...
  std::vector<std::vector<int>> local_data = SplitData(GetOutput(), num_threads);
  tbb::parallel_for(0, num_threads, [&](int i) { RadixSort(local_data[i]); });
  GetOutput() = MergeData(local_data);
}

MPI_Barrier(MPI_COMM_WORLD);
```

*Пояснение к реализации:* Данный код строго изолирует фазы валидации, препроцессинга и вычислений на нулевом узле. Это исключает риски состояния гонки (Data Race) и конфликтов памяти между процессами, однако означает, что вычислительная мощность узлов `rank > 0` не утилизируется, они просто блокируются на барьере до окончания сортировки.

Замеры проводились на массиве в 5 000 000 элементов. 
Ускорение ($S = T_1 / T_p$), Эффективность ($E = S / threads$). Поскольку активны только потоки нулевого ранга, эффективность рассчитывалась относительно числа потоков (`num_threads`), а не общего количества всех процессов в системе.

## 10. Выводы
Гибридная реализация ALL продемонстрировала **лучшее абсолютное время выполнения задачи** из всех рассмотренных версий (0.0202s в режиме task_run на конфигурации 2×4). Это достигнуто за счет быстрой внутрипроцессной декомпозиции средствами TBB и эффективной сортировки по основанию 256.

Однако архитектурная особенность данной реализации — выполнение всей работы исключительно на `rank = 0` — делает наличие дополнительных MPI-процессов избыточным накладным расходом. Увеличение конфигурации до 4×8 привело к деградации времени (до 0.0255s). Основные причины:
1. Закон Амдала и узкое горлышко (Bottleneck) при последовательном слиянии `MergeData` одним потоком.
2. Процессы `rank > 0` создают дополнительный системный overhead, простаивая на `MPI_Barrier`.
3. Упор в пропускную способность памяти при одновременной работе 8 потоков на одном узле (rank 0).

Таким образом, реализация успешна как внутрипроцессный TBB-акселератор, но для истинного масштабирования на кластерах требуется внедрение распределения данных (`MPI_Scatterv`) между рангами перед локальной потоковой обработкой.
