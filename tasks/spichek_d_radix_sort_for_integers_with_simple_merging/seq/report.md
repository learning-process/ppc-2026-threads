## 1. Контекст
Данная реализация представляет собой базовую (последовательную) версию поразрядной сортировки для вектора целых чисел. Она служит эталоном корректности и базовой линией производительности, относительно которой будет измеряться ускорение и эффективность параллельных реализаций (OMP, TBB, STL, ALL).

## 2. Постановка задачи
Необходимо отсортировать вектор целых чисел (`std::vector<int>`). Алгоритм должен корректно обрабатывать:
* Положительные и отрицательные числа.
* Дублирующиеся значения.
* Уже отсортированные (в прямом и обратном порядке) массивы.
* Массивы, состоящие из одинаковых элементов.
* Пустые векторы.

Входные данные (`InType`) и выходные данные (`OutType`) представляют собой `std::vector<int>`.

## 3. Базовый алгоритм
В качестве базового алгоритма выбрана поразрядная сортировка (LSD Radix Sort) по основанию 10 с использованием сортировки подсчетом (Counting Sort) для каждого разряда.
Пошаговое описание:
1.  Поиск минимального элемента. Если в массиве присутствуют отрицательные числа, весь массив сдвигается на величину модуля минимального элемента, чтобы все числа стали неотрицательными.
2.  Поиск максимального элемента для определения количества разрядов.
3.  Итерация по каждому десятичному разряду (единицы, десятки, сотни и т.д.). Для каждого разряда выполняется устойчивая сортировка подсчетом (Counting Sort).
4.  Если на первом шаге применялся сдвиг, выполняется обратный сдвиг всех элементов для восстановления исходных значений.

* Асимптотика по времени: $O(d \cdot (n + b))$, где $n$ — размер массива, $b$ — основание системы счисления ($b=10$), $d$ — максимальное количество цифр в числе. Фактически алгоритм работает за линейное время $O(n)$, так как $d$ и $b$ — константы для типа `int`.
* Асимптотика по памяти: $O(n + b)$ для выделения памяти под выходной буфер (`output`) и массив счетчиков (`count`).

Инвариантом и критерием корректности является полная отсортированность выходного массива по неубыванию (проверяется через `std::ranges::is_sorted`).

## 4. Детали реализации
Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

* **ValidationImpl**: Проверяет, что входной вектор не пуст.
* **PreProcessingImpl**: Копирует входные данные в выходной буфер, с которым дальше будет вестись работа.
* **RunImpl**: Содержит проверку на пустоту выходного буфера и вызывает основной метод `RadixSort`.
* **PostProcessingImpl**: Осуществляет финальную проверку результата с помощью `std::ranges::is_sorted`.

Ключевой фрагмент обработки отрицательных чисел в `RadixSort`:

```cpp
// File: seq/src/ops_seq.cpp
int min_val = *std::ranges::min_element(data);
if (min_val < 0) {
  for (auto &x : data) {
    x -= min_val; // Сдвиг значений в положительную область
  }
}
```

*Пояснение к реализации:* Сдвиг значений позволяет корректно использовать поразрядную сортировку (LSD) без риска получения отрицательных индексов при вычислении остатка от деления вида `(x / exp) % 10`. Метод устойчивой сортировки подсчетом гарантирует сохранение относительного порядка равных элементов, что является обязательным условием правильной работы Radix Sort. В конце метода выполняется обратный сдвиг значений на исходную величину `min_val`.

## 5. Проверка корректности
Соответствие результатов эталону проверяется с помощью автоматических тестов в рамках тестового фреймворка Google Test (`tests/functional/main.cpp`). 
Функциональные тесты покрывают следующие пограничные сценарии:
* `RunSortStandard`: Сортировка массива случайных чисел в диапазоне от -10 000 до 10 000.
* `RunSortWithNegativeValues`: Массивы, полностью состоящие из отрицательных элементов.
* `RunSortWithAllSameValues`: Работа с идентичными дублирующимися элементами (заполнение значениями 42).
* `RunSortWithSortedInput`: Тестирование на уже отсортированном массиве.
* `RunSortWithReversedInput`: Тестирование на массиве, отсортированном в обратном порядке (по убыванию).

Все тесты выполняются на векторах малого (100 элементов), среднего (1000 элементов) и большого (5000 элементов) размеров.

В качестве базовой рабочей единицы взят один последовательный поток выполнения ($p = 1$). Измерения проводились для двух стандартных режимов каркаса производительности курса: `task` (прямой последовательный запуск задачи) и `pipeline` (конвейерная обработка данных).

| Режим запуска | Размер задачи (N) | Рабочие единицы (p) | Время выполнения (секунды) |
|---|---|---|---|
| **pipeline** | 5 000 000 | 1 | 0.1108388901 |
| **task_run** | 5 000 000 | 1 | 0.1114030361 |

*Самым вычислительно и ресурсоемким фрагментом алгоритма является внутреннее поразрядное перераспределение элементов массива. На каждой итерации по разряду (`exp *= 10`) алгоритм выполняет полное копирование 5 000 000 элементов во временный буфер `output` и осуществляет частые обращения к оперативной памяти для подсчета индексов в массиве `count`.*
