# Подробное описание гибридной ALL-реализации

## 1. Назначение ALL-версии

ALL-версия является комбинированной реализацией задачи умножения разреженных
матриц. Она использует общий тип задачи `kALL` из инфраструктуры курса и
сочетает межпроцессную синхронизацию MPI с потоковым параллелизмом внутри
вычисления.

Вычисляется произведение `C = A * B`, где элементы матриц имеют тип `double`.
Входные матрицы и результат хранятся в формате CCS с массивами `values`,
`row_indices` и `col_ptrs`.

## 2. Формат CCS

В формате CCS ненулевые элементы сгруппированы по столбцам. Для столбца `j`
диапазон элементов задается границами:

```text
col_ptrs[j] .. col_ptrs[j + 1]
```

Массив `values` хранит значения ненулевых элементов, а `row_indices` хранит
номера строк. Такой формат подходит для реализации, потому что результат
строится и объединяется по столбцам.

## 3. Проверка входных данных

Перед вычислением проверяется корректность CCS-представления обеих матриц:

- размер `col_ptrs` должен быть равен `cols + 1`;
- массивы `values` и `row_indices` должны иметь одинаковый размер;
- первый элемент `col_ptrs` должен быть равен нулю;
- последний элемент `col_ptrs` должен совпадать с числом ненулевых элементов;
- значения `col_ptrs` должны быть неубывающими;
- индексы строк должны находиться в диапазоне `[0, rows)`.

Также проверяется условие совместимости размеров: `A.cols == B.rows`.

## 4. Базовый алгоритм умножения

Алгоритм строит результат по столбцам. Для текущего столбца матрицы `B`
создается временный аккумулятор длины `A.rows`. Если найден ненулевой элемент
`B(k, j)`, то столбец `k` матрицы `A` добавляет вклад:

```text
C(i, j) += A(i, k) * B(k, j)
```

После обработки столбца ненулевые значения аккумулятора записываются в
`values` и `row_indices`, а массив `col_ptrs` фиксирует границы столбцов
результата.

## 5. Роль MPI-процессов

В реализации используется MPI-среда, характерная для задач типа ALL. Каждый
MPI-процесс получает те же входные данные от тестовой инфраструктуры и строит
полный результат локально. Такой подход упрощает проверку корректности, потому
что каждый ранг может вернуть одинаковое CCS-представление результата.

Работа между MPI-рангами в данной версии не делится по разным частям матрицы.
MPI используется как часть комбинированного режима и для синхронизации перед
измеряемым вычислением и после него.

## 6. Потоковый параллелизм внутри задачи

Внутри каждого MPI-процесса вычисление делится по диапазонам столбцов
результата. Число диапазонов выбирается на основе `PPC_NUM_THREADS` и числа
столбцов матрицы `B`.

OpenMP используется для подготовки независимых границ диапазонов. Затем
oneTBB выполняет параллельную обработку этих диапазонов. Каждая рабочая задача
строит локальную частичную матрицу CCS для своего набора столбцов.

## 7. Синхронизация

В начале и конце метода вычисления вызывается `MPI_Barrier`. Первый барьер
нужен, чтобы процессы входили в измеряемую часть согласованно. Второй барьер
нужен, чтобы все процессы завершили вычисление до выхода из этапа `RunImpl`.

Такая синхронизация особенно важна для производительных тестов, где время
измеряется в режимах `pipeline` и `task_run`.

## 8. Отсутствие гонок и порядок результата

Входные матрицы `A` и `B` только читаются. Каждая потоковая задача записывает
результат в собственную частичную матрицу CCS. Итоговые массивы `values`,
`row_indices` и `col_ptrs` не изменяются во время параллельного участка.

После завершения потоковых задач частичные матрицы объединяются
последовательно, в порядке возрастания номеров диапазонов. Поэтому порядок
столбцов и элементов в итоговом CCS-представлении остается детерминированным.

## 9. Настройка числа процессов и потоков

Для локального запуска используются переменные инфраструктуры курса:

- `PPC_NUM_PROC` — количество MPI-процессов;
- `PPC_NUM_THREADS` — количество потоков внутри процесса.

Локальные тесты выполнялись с запуском через `mpiexec -n 2`. Если точные
характеристики оборудования неизвестны, окружение описывается как локальное
окружение.

## 10. Производительные режимы

Производительные тесты используют два режима:

- `pipeline` — полный жизненный цикл задачи, включая проверку и подготовку;
- `task_run` — основной вычислительный этап после подготовки данных.

Численные таблицы времени не приводятся, потому что значения зависят от
локального окружения, параметров сборки, числа процессов и числа потоков.

## 11. Проверка корректности

Функциональные тесты сравнивают результат с заранее заданным CCS-представлением.
Проверяются размеры результата, массив `col_ptrs`, индексы `row_indices` и
значения `values`. Для `double` используется допустимая погрешность.

Также производительный тест дополнительно сравнивает плотное представление
результата с эталонным плотным умножением.

## 12. Команды запуска

Команды для локальной проверки ALL-реализации:

```powershell
cmake --build build --config Release --parallel
$env:PPC_NUM_PROC="2"
$env:PPC_NUM_THREADS="4"
mpiexec -n 2 .\build\bin\ppc_func_tests.exe --gtest_filter="*safaryan_a_sparse_matrix_mult_crs_all*"
mpiexec -n 2 .\build\bin\ppc_perf_tests.exe --gtest_filter="*safaryan_a_sparse_matrix_mult_crs_all*"
```

## 13. Выводы

Гибридная ALL-версия сохраняет CCS-формат данных и объединяет MPI-синхронизацию
с потоковым распараллеливанием по диапазонам столбцов. Локальные частичные
матрицы предотвращают гонки данных, а последовательное объединение сохраняет
детерминированный результат.
