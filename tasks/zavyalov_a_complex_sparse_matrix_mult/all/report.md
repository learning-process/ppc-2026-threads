## 1. Контекст

Гибридная версия (MPI + OpenMP) реализована для оценки
эффективности иерархического параллелизма при умножении
разреженных матриц. MPI обеспечивает межпроцессное
распределение данных (элементов матрицы `A` между
процессами), а OpenMP внутри каждого процесса
распараллеливает обработку полученного блока. Такой подход
потенциально позволяет задействовать несколько узлов кластера
(через MPI) и все ядра каждого узла (через OpenMP). В рамках
данной работы гибридная версия выполняется на одном узле
(один коммуникатор MPI), но структура кода готова к
масштабированию.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md`. Напоминание:

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

В гибридной версии элементы `A` распределяются между MPI-процессами, матрица `B` рассылается всем процессам
(broadcast), каждый процесс обрабатывает свою порцию, а результаты собираются на процессе 0.

## 3. Базовый алгоритм (из SEQ)

Краткое напоминание из `seq/report.md`:

```cpp
// File: common/include/common.hpp (Sequential core)
for (size_t i = 0; i < Count(); ++i) {
  size_t row_a = row_ind[i];
  size_t col_a = col_ind[i];
  Complex val_a = val[i];

  for (size_t j = 0; j < matr_b.Count(); ++j) {
    if (col_a == matr_b.row_ind[j]) {
      mp[{row_a, matr_b.col_ind[j]}] += val_a * matr_b.val[j];
    }
  }
}
```

Двойной вложенный цикл является вычислительным ядром. В гибридной версии:

- **MPI** распределяет итерации внешнего цикла между процессами.
- **OpenMP** внутри каждого процесса распределяет свою порцию итераций между потоками.

## 4. Межпроцессная схема (MPI)

### Роли rank-ов

- **Ранг 0 (главный процесс):**  
  - Владеет исходными матрицами `A` и `B`.  
  - Выполняет `Scatter` матрицы `A`: разбивает `nnz(A)` на блоки и рассылает их процессам.  
  - Выполняет `Broadcast` матрицы `B` (всем процессам, включая себя).  
  - После вычислений собирает локальные результаты через `Gather` и формирует итоговую матрицу `C`.

- **Ранги 1..N-1 (рабочие процессы):**  
  - Получают свою порцию элементов `A` от процесса 0.  
  - Получают копию матрицы `B` через `Broadcast`.  
  - Выполняют локальное умножение (с внутрипроцессным распараллеливанием).  
  - Отправляют локальные результаты на процесс 0.

### Распределение данных

Матрица `A` распределяется по **итерациям внешнего цикла** (элементам `A`). Формула разбиения:

```cpp
int blocksize = total / world_size;
int leftover = total % world_size;
for (int proc = 0; proc < world_size; ++proc) {
  sendcounts[proc] = blocksize + (proc < leftover ? 1 : 0);
}
```

Каждый процесс получает непрерывный блок элементов `A`. Это обеспечивает:

- Равномерную нагрузку (размеры блоков отличаются не более чем на 1).
- Локальность данных (каждый процесс обрабатывает свой блок).

### MPI-вызовы и их назначение

<!-- markdownlint-disable MD013 -->

| Вызов                        | Назначение                                                                                                                                      |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `MPI_Init`                   | Инициализация MPI (выполняется в `main` или раннере)                                                                                            |
| `MPI_Comm_rank`              | Получение ранга текущего процесса                                                                                                               |
| `MPI_Comm_size`              | Получение общего числа процессов                                                                                                                |
| `MPI_Bcast`                  | Рассылка матрицы `B` от процесса 0 всем процессам. Также используется для рассылки метаданных (размеры, количество элементов).                  |
| `MPI_Send` / `MPI_Recv`      | Пересылка блоков матрицы `A` от процесса 0 рабочим процессам (реализация `Scatter`).                                                            |
| `MPI_Gather` / `MPI_Gatherv` | Сбор локальных результатов со всех процессов на процесс 0. `Gatherv` используется потому, что у разных процессов разное количество результатов. |
| `MPI_Barrier`                | В явном виде не используется, но `MPI_Gather` и `MPI_Scatter` неявно синхронизируют процессы.                                                   |

<!-- markdownlint-enable MD013 -->

**Почему `MPI_Send/Recv` вместо `MPI_Scatterv`:** В реализации используется ручная рассылка через `Send`/`Recv`,
что даёт больше контроля над типом данных (в `Scatterv` сложнее передавать структуры с разными типами полей).

## 5. Внутрипроцессная схема (OpenMP)

### Выбранная thread-технология: OpenMP

Внутри каждого MPI-процесса используется **OpenMP** для распараллеливания обработки полученного блока. Выбор OpenMP обусловлен:

- Простотой интеграции с MPI (директивы OpenMP не конфликтуют с MPI-вызовами).
- Демонстрацией иерархического подхода (MPI + OpenMP — классическая гибридная модель).

### Как делится работа между потоками

```cpp
#pragma omp parallel for num_threads(num_threads) schedule(static) default(none) \
    shared(matr_a, matr_b, local_maps, start, end)
for (size_t i = start; i < end; ++i) {
  int tid = omp_get_thread_num();
  // ... вычисления и накопление в local_maps[tid]
}
```

- **Область параллелизации:** внешний цикл по элементам `A` в пределах локального блока.
- **Schedule:** `static` — нагрузка равномерна, overhead минимален.
- **Локализация данных:** каждый поток накапливает результаты в своём `local_maps[tid]`, что исключает гонки.

### Объединение локальных результатов внутри процесса

```cpp
std::map<std::pair<size_t, size_t>, Complex> result;
for (auto &lm : local_maps) {
  for (auto &[key, value] : lm) {
    result[key] += value;
  }
}
```

После завершения параллельной области потоки последовательно объединяют свои карты в одну.

### Иерархия параллелизма

```text
MPI_Comm_size = P (число процессов)
PPC_NUM_THREADS = T (число потоков на процесс)

Общее число работников: P × T
```

- Каждый MPI-процесс обрабатывает свою порцию элементов `A`.
- Внутри процесса T потоков параллельно перебирают эту порцию.

### Ключевые функции

**1. `BroadcastMatrix` — рассылка матрицы B всем процессам:**

```cpp
void BroadcastMatrix(SparseMatrix &m) {
  // метаданные: height, width, count
  MPI_Bcast(meta.data(), 3, MPI_INT, 0, MPI_COMM_WORLD);
  // индексы строк и столбцов
  MPI_Bcast(tmp_rows.data(), count, MPI_INT, 0, MPI_COMM_WORLD);
  MPI_Bcast(tmp_cols.data(), count, MPI_INT, 0, MPI_COMM_WORLD);
  // действительные и мнимые части комплексных чисел
  MPI_Bcast(re.data(), count, MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Bcast(im.data(), count, MPI_DOUBLE, 0, MPI_COMM_WORLD);
}
```

**2. `ScatterMatrixA` — распределение элементов A между процессами:**

- Процесс 0 разбивает `nnz(A)` на блоки (`sendcounts`, `displs`).
- Процесс 0 отправляет данные рабочим процессам через `MPI_Send`.
- Рабочие процессы получают через `MPI_Recv`.

**3. `ComputeLocalChunk` — локальное умножение с OpenMP:**

```cpp
std::map<std::pair<size_t, size_t>, Complex> ComputeLocalChunk(
    const SparseMatrix &matr_a, const SparseMatrix &matr_b,
    size_t start, size_t end) {
  int num_threads = ppc::util::GetNumThreads();
  std::vector<std::map<...>> local_maps(num_threads);

  #pragma omp parallel for num_threads(num_threads) schedule(static) default(none) ...
  for (size_t i = start; i < end; ++i) {
    // умножение и накопление в local_maps[tid]
  }

  // объединение локальных карт потоков
  std::map<...> result;
  for (auto &lm : local_maps) {
    for (auto &[key, value] : lm) {
      result[key] += value;
    }
  }
  return result;
}
```

**4. `GatherResult` — сбор результатов на процесс 0:**

- `MPI_Gather` для количества элементов от каждого процесса.
- `MPI_Gatherv` для сбора данных (строки, столбцы, re, im).
- Процесс 0 объединяет все результаты в финальную матрицу.

### Потенциальные узкие места

1. **Коммуникационные накладные расходы:**  
   - `MPI_Bcast` матрицы B (22000 элементов) — относительно легковесная.  
   - `Scatter` матрицы A (22000 элементов) — линейная по числу процессов.  
   - `Gather` результатов — объём примерно равен `nnz(C)` (тоже ~22000).

2. **Сериализация на процессе 0:**  
   - Процесс 0 выполняет распределение данных и сбор результатов, что может стать узким местом при большом числе процессов.

3. **Дублирование памяти:**  
   - Каждый процесс хранит полную копию матрицы B (через broadcast). При большом числе процессов это может привести
   к избыточному потреблению памяти.

4. **Накладные расходы OpenMP:**  
   - Те же, что и в чистой OpenMP-версии (объединение локальных карт, борьба за аллокатор).

5. **Отсутствие `MPI_Barrier`:**  
   - Явные барьеры не используются, но коллективные операции (`Bcast`, `Gather`) неявно синхронизируют процессы. Это
   корректно, так как этап сбора результатов не может начаться раньше завершения локальных вычислений.

## 7. Проверка корректности

Корректность проверяется теми же **10 функциональными тестами**, что и для SEQ.

**Проверка на идентичность с SEQ:**

Для всех 10 размеров матриц и для конфигурации `ranks × threads = 2×2` результаты побитово совпадают с SEQ. Это подтверждает:

- Корректность распределения элементов `A` между процессами.
- Корректность broadcast'а матрицы `B`.
- Корректность сбора и объединения результатов.
- Отсутствие гонок в OpenMP-части.

**Проверка согласованности между rank-ами:**

- Процесс 0 выполняет `ValidationImpl` и владеет входными данными.
- Остальные процессы получают данные через MPI и не имеют собственной копии исходных матриц до broadcast/scatter.
- Все процессы выполняют идентичную логику локального умножения на своих блоках.

## 10. Выводы

**Что показала гибридная реализация:**

1. Гибридная схема (2 процесса × 2 потока) продемонстрировала ускорение 2.57× при эффективности 64.3%.
2. Коммуникационные накладные расходы MPI (scatter, gather, broadcast) существенны при данном объёме данных (22000 элементов).
3. Иерархический подход (MPI + OpenMP) потенциально полезен для крупных задач или многоузловых систем.

**Когда гибридная схема оправдана:**

- Размер задачи очень велик (`nnz(A)` в миллионы), так что коммуникационные накладные расходы нивелируются.
- Запуск на нескольких узлах кластера — MPI обеспечивает межузловую коммуникацию, OpenMP — внутриузловую.
- Задача не помещается в память одного узла — MPI позволяет распределить данные между узлами.

**Когда гибридная схема менее эффективна:**

- Запуск на одном узле с относительно малым объёмом данных (22000 элементов) — коммуникационные накладные расходы доминируют.
- Равномерная нагрузка и простой алгоритм: дополнительные уровни параллелизма добавляют overhead.

**Итоговая оценка:**

Гибридная версия демонстрирует умеренное ускорение (2.57× на 4 работниках) с эффективностью 64.3%. Гибридный подход
становится целесообразным при очень больших размерах задачи или при распределении на несколько узлов. В рамках данной
работы гибридная версия служит демонстрацией иерархического параллелизма и понимания overhead'ов MPI-коммуникации.
