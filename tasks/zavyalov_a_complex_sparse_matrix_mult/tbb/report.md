## 1. Контекст

TBB-версия реализована для оценки задачно-ориентированного
подхода (oneTBB) при умножении разреженных матриц. TBB
предлагает высокоуровневые алгоритмы параллельной обработки,
автоматическое распределение задач по потокам и встроенные
механизмы локального хранения данных. Базовая логика
полностью сохранена, изменена лишь схема распределения
вычислений между потоками.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md`.

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

## 3. Базовый алгоритм (из SEQ)

Краткое напоминание из `seq/report.md`:

```cpp
// File: common/include/common.hpp (Sequential core)
for (size_t i = 0; i < Count(); ++i) {
  size_t row_a = row_ind[i];
  size_t col_a = col_ind[i];
  Complex val_a = val[i];

  for (size_t j = 0; j < matr_b.Count(); ++j) {
    if (col_a == matr_b.row_ind[j]) {
      mp[{row_a, matr_b.col_ind[j]}] += val_a * matr_b.val[j];
    }
  }
}
```

Двойной вложенный цикл является вычислительным ядром.
Асимптотика по времени: O(nnz_A · nnz_B). В TBB-версии
внешний цикл распределяется между потоками с помощью
`parallel_for`.

## 4. Схема распараллеливания

### Выбранный примитив: `tbb::parallel_for`

Использован `tbb::parallel_for` — наиболее подходящий
примитив для независимого перебора диапазона итераций.
Альтернативы (`parallel_reduce`, `task_group`) не
использовались, так как:

- `parallel_reduce` предназначен для свёртки с ассоциативной операцией, но редукция для `std::map` не определена.
- `task_group` избыточен для простого распараллеливания цикла.

### Диапазон работы

```cpp
tbb::blocked_range<size_t>(0, matr_a.Count())
```

Диапазон охватывает все индексы ненулевых элементов матрицы
`A` (от 0 до `nnz(A)`). Каждый поток получает непрерывный
поддиапазон (блок) индексов.

### Grainsize

В реализации **явный grainsize не задан** — используется
значение по умолчанию. TBB автоматически выбирает размер
чанка, основываясь на:

- Количестве итераций (`nnz(A) = 22000`)
- Числе доступных потоков
- Эвристиках для минимизации накладных расходов

**Почему это приемлемо:** Для `nnz(A) = 22000` автоматический
выбор grainsize даёт достаточно крупные блоки (сотни-тысячи
итераций на блок), что минимизирует overhead на распределение
задач. Вручную подбирать grainsize не требуется, так как
нагрузка между итерациями равномерна (каждая итерация
перебирает все `nnz(B)`).

### Partitioner

Используется **partitioner по умолчанию** (`auto_partitioner`
в старых версиях или эквивалент в oneTBB). TBB автоматически
выбирает стратегию распределения итераций, адаптируясь к
нагрузке и числу потоков.

### Контроль конкуренции

```cpp
int num_threads = ppc::util::GetNumThreads();
```

Число потоков задаётся через `PPC_NUM_THREADS`. TBB
использует `global_control::max_allowed_parallelism` для
ограничения числа worker-потоков (runner курса настраивает
это автоматически). TBB создаёт пул потоков один раз, и все
вызовы `parallel_for` переиспользуют его, что уменьшает
overhead на создание потоков.

### Локализация данных: `tbb::enumerable_thread_specific`

Ключевое отличие от последовательной версии — использование `enumerable_thread_specific`:

```cpp
tbb::enumerable_thread_specific<std::map<std::pair<size_t, size_t>, Complex>> local_maps;
```

- Каждый поток автоматически получает свой экземпляр `std::map`.
- Доступ к локальной карте через `local_maps.local()`.
- После завершения `parallel_for` все локальные карты объединяются в одну.

Этот подход:

- **Исключает гонки данных** — потоки не конкурируют за общую структуру.
- **Упрощает код** — не нужно вручную создавать вектор карт и индексировать по `tid`.
- **Уменьшает false sharing** — карты разных потоков физически разделены.

### Барьеры и синхронизация

`parallel_for` неявно синхронизирует все потоки по
завершении: после выхода из параллельной области все
локальные карты заполнены и можно безопасно их объединять.
Явные барьеры не требуются.

## 5. Детали реализации

**Файлы:**

- `tbb/include/ops_tbb.hpp` — объявление класса `ZavyalovAComplSparseMatrMultTBB`.
- `tbb/src/ops_tbb.cpp` — реализация с TBB-примитивами.

**Структура пайплайна:** полностью повторяет `seq`:

- `ValidationImpl` — проверка совместимости.
- `PreProcessingImpl` / `PostProcessingImpl` — без действий.
- `RunImpl` — вызывает `MultiplicateWithTbb()`.

**Ключевой фрагмент — вычислительное ядро с TBB:**

```cpp
// File: tbb/src/ops_tbb.cpp, MultiplicateWithTbb()

tbb::enumerable_thread_specific<std::map<std::pair<size_t, size_t>, Complex>> local_maps;

tbb::parallel_for(tbb::blocked_range<size_t>(0, matr_a.Count()),
    [&](const tbb::blocked_range<size_t>& range) {
        auto& my_map = local_maps.local();
        for (size_t i = range.begin(); i != range.end(); ++i) {
            size_t row_a = matr_a.row_ind[i];
            size_t col_a = matr_a.col_ind[i];
            Complex val_a = matr_a.val[i];

            for (size_t j = 0; j < matr_b.Count(); ++j) {
                if (col_a == matr_b.row_ind[j]) {
                    my_map[{row_a, matr_b.col_ind[j]}] += val_a * matr_b.val[j];
                }
            }
        }
    });

// Объединение локальных результатов
std::map<std::pair<size_t, size_t>, Complex> mp;
for (auto& lm : local_maps) {
    for (auto& [key, value] : lm) {
        mp[key] += value;
    }
}
```

**Важные изменения относительно SEQ:**

1. **`tbb::enumerable_thread_specific`** — поток-локальное
хранилище для карт, автоматически создающее отдельный
экземпляр на поток.
2. **`tbb::parallel_for` + `blocked_range`** — распределение итераций внешнего цикла.
3. **Лямбда-функция** — тело параллельной секции.
4. **Постобработка** — последовательное слияние всех локальных карт.

### Гонки данных

Потенциальная проблема с записью в общий `std::map` исключена
за счёт `enumerable_thread_specific` — каждый поток работает
со своим экземпляром карты. Это ключевое решение, делающее
реализацию корректной без дополнительной синхронизации.

*Измерения: 10 повторов на каждой конфигурации.*

### Анализ масштабируемости

**Сильное масштабирование:**

- **1 -> 2 потока:** Ускорение 1.71× (эффективность 85.5%).
Накладные расходы на разбиение диапазона и распределение
задач заметны, но ускорение остаётся хорошим.
- **2 -> 4 потока:** Ускорение 1.85× (с 87 до 47 мс).
Эффективность снижается до 79.3%, что типично для задач с
динамическими структурами данных.

**Эффективность:**

- На 2 потоках эффективность 85.5% - хороший результат.
- На 4 потоках эффективность 79.3% - приемлемая, показывает наличие накладных расходов при удвоении числа потоков.

**Предполагаемые узкие места:**

1. *Накладные расходы на автоматическое разбиение диапазона:*
TBB runtime тратит время на распределение итераций между потоками.
2. *Объединение локальных карт:* Слияние 4 карт по ~20000 элементов каждая занимает измеримое время.
3. *`std::map` как внутренняя структура:* Логарифмическая
сложность вставки и динамическое выделение узлов остаются
узким местом, независимо от технологии распараллеливания.
4. *Разреженность:* Доля успешных вставок мала (~0.005% от
всех проверок). TBB не помогает сократить количество проверок
`col_a == row_b`.

**Балансировка нагрузки:**

TBB автоматически распределяет итерации, и благодаря
равномерной нагрузке (каждая итерация обрабатывает все `nnz(B)
`) дисбаланс минимален. Заметного простоя потоков не
наблюдалось.

**Сравнение с теоретическим максимумом:**

Идеальное ускорение на 4 потоках: `149 / 4 = 37.3 мс`.
Реальное: 47 мс. Потери ~10 мс (~21% от реального времени)
приходятся на:

- Overhead TBB runtime (разбиение диапазона, распределение задач).
- Объединение локальных карт.
- Борьбу потоков за аллокатор памяти при работе `std::map`.

## 9. Выводы

**Что показала TBB-реализация:**

1. **Хорошее ускорение на 4 потоках:** 3.17× (эффективность 79.3%).
2. **Удобство `enumerable_thread_specific`:** Код чище, чем при ручном управлении вектором локальных карт.
3. **Отсутствие гонок:** Поток-локальные карты полностью исключают contention.
4. **Детерминизм:** Результаты воспроизводимы при фиксированном числе потоков.

**Когда TBB эффективен для этой задачи:**

- Размер `nnz(A)` достаточно велик (≥10000), чтобы overhead на разбиение диапазона окупался.
- Нагрузка между итерациями равномерна — TBB автоматически распределяет блоки.
- Количество потоков невелико (2–4), так как при дальнейшем
росте накладные расходы на объединение локальных карт
становятся заметнее.

**Ограничения:**

- TBB не помогает сократить количество проверок `col_a == row_b` — сложность остаётся O(nnz_A · nnz_B).
- Автоматическое разбиение диапазона вносит небольшой, но измеримый overhead.
- `std::map` (древесная структура) даёт логарифмические накладные расходы и борьбу потоков за аллокатор памяти.

**Итоговая оценка:**

TBB-версия демонстрирует **хорошее ускорение** (3.17× на 4
потоках) с **приемлемой эффективностью** (79.3%). Основным
узким местом является не модель распараллеливания, а
внутреннее устройство `std::map`. TBB предоставляет удобный и
безопасный способ распараллеливания, требующий минимальных изменений в коде по сравнению с последовательной версией.
