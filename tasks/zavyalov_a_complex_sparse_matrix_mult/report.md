## 1. Введение

Задача умножения разреженных матриц с комплексными элементами
в формате CCS выбрана для сравнения различных моделей
параллельного программирования: директивной (OpenMP),
задачно-ориентированной (oneTBB), низкоуровневой ручной
(std::thread) и гибридной (MPI + OpenMP). Задача интересна
тем, что разреженная структура данных создаёт нерегулярный
доступ к памяти, а комплексная арифметика добавляет
вычислительную нагрузку. Сравнение позволяет оценить,
насколько каждая технология справляется с нерегулярностью
данных и каковы накладные расходы на управление параллелизмом.

## 2. Единая постановка задачи

**Входные данные:** две разреженные матрицы A (height ×
width) и B (height' × width') в формате CCS (три вектора:
значения, индексы строк, индексы столбцов).

**Выходные данные:** разреженная матрица C = A × B в формате CCS.

**Ограничения:**

- Умножение определено только если A.width == B.height.
- Нулевые элементы не хранятся.
- Комплексные числа умножаются по правилу (a+bi)(c+di) = (ac-bd) + (ad+bc)i.

**Критерий корректности:** результат каждой параллельной
реализации должен побитово совпадать с результатом
последовательной версии.

### Генерация данных

Для функциональных тестов используются детерминированно
сгенерированные матрицы 10 различных размеров (от 1x1 до
9x3). Для тестов производительности — матрицы размером 11000
x 11000 с ровно 2 ненулевыми элементами на строку (nnz =
22000 для каждой матрицы).

### Размеры задач для производительности

- Матрицы: 11000 x 11000
- nnz(A) = 22000, nnz(B) = 22000
- Доля заполнения: ~0.018%

### Метрики

**Ускорение (Speedup):** Speedup = T_seq / T_parallel, где T_seq = 149 мс (медиана SEQ)

**Эффективность (Efficiency):** Efficiency = Speedup /
total_workers, где total_workers — число потоков (для OMP,
TBB, STL) или ranks x threads (для ALL)

### Процедура измерений

- 10 повторов для каждой конфигурации
- Фиксируется медиана, минимум и максимум
- Измеряется время выполнения task_run (один вызов оператора умножения)

## 6. Интерпретация различий

### SEQ (базовый эталон)

Последовательная версия выполняет двойной вложенный цикл за
149 мс. Эталон стабилен и детерминирован. Основная
вычислительная нагрузка приходится на проверку col_a == row_b
и вставку в std::map.

### OpenMP (2.81x на 4 потоках, эффективность 70.3%)

OpenMP показал хорошее ускорение на 2 потоках (1.84x,
эффективность 92.2%), но при переходе на 4 потока
эффективность упала до 70.3%. Причина — использование
std::map для локальных результатов: каждый поток создаёт свою
карту, и при слиянии 4 карт возникают накладные расходы на
объединение и борьба за аллокатор памяти.

### TBB (3.17x на 4 потоках, эффективность 79.3%)

TBB оказался эффективнее OpenMP благодаря
enumerable_thread_specific, который более эффективно
управляет поток-локальными данными и уменьшает contention.
Автоматическое разбиение диапазона и переиспользование пула
потоков снижают накладные расходы.

### STL (3.39x на 4 потоках, эффективность 84.8%)

STL показал наилучшее ускорение. Ручное управление потоками
позволило оптимально разбить диапазоны, минимизировать
накладные расходы на создание потоков и избежать лишних
абстракций. Цена — сложность кода.

### ALL (2.57x на 2x2, эффективность 64.3%)

Гибридная версия показала наименьшее ускорение. Причины:
коммуникационные overhead'ы MPI (Bcast, Send/Recv, Gather),
дублирование памяти (каждый процесс хранит копию B),
сериализация на процессе 0. Гибридный подход оправдан только
при очень больших размерах задачи или на нескольких узлах кластера.

## 8. Заключение

### Какая версия лучше и при каких условиях

<!-- markdownlint-disable MD013 -->

| Условия                              | Рекомендуемая версия | Ускорение         |
| ------------------------------------ | -------------------- | ----------------- |
| Максимальная производительность      | STL                  | 3.39x             |
| Баланс производительности и простоты | TBB                  | 3.17x             |
| Быстрая разработка                   | OpenMP               | 2.81x             |
| Распределение на несколько узлов     | ALL (MPI+OpenMP)     | 2.57x (на 1 узле) |

<!-- markdownlint-enable MD013 -->

### Ограничения сравнения

1. Разреженность фиксирована (2 элемента на строку).
2. Размер задачи фиксирован (11000 x 11000).
3. Оборудование: Intel Core i5 14600kf (6 P-ядер).
4. ALL тестировалась на одном узле.

### Что можно улучшить

1. Заменить std::map на std::unordered_map для уменьшения логарифмического фактора.
2. Использовать std::vector + sort + merge для промежуточных результатов.
3. Динамическое планирование для неравномерной нагрузки.
4. Асинхронные MPI-операции для наложения коммуникации на вычисления.

### Итоговая оценка

На данной задаче STL показал наилучшую производительность (3.
39x), TBB — лучший баланс между производительностью и
простотой (3.17x), OpenMP — самый простой способ
распараллеливания (2.81x). Основным узким местом всех
реализаций является не модель распараллеливания, а внутреннее
устройство std::map.
