## 1. Контекст

OpenMP-версия является первым шагом в распараллеливании
последовательного алгоритма умножения разреженных матриц.
Цель — оценить эффективность директивного подхода для задачи
с нерегулярной структурой данных (разреженные матрицы в
формате CCS) и продемонстрировать масштабируемость на
многопоточной системе. Базовая логика полностью сохранена,
изменена лишь схема распределения вычислений между потоками.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md` (разделы 2 и 3).

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Ограничения: совместимость размеров проверяется в `ValidationImpl`.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

**Что нового:** каждый поток обрабатывает свою порцию
элементов матрицы `A` и накапливает локальные результаты в
собственном `std::map`. Глобальное объединение происходит
после завершения параллельной области.

## 3. Базовый алгоритм (из SEQ)

Краткое напоминание из `seq/report.md`:

```cpp
// File: common/include/common.hpp (Sequential core)
for (size_t i = 0; i < Count(); ++i) {
  size_t row_a = row_ind[i];
  size_t col_a = col_ind[i];
  Complex val_a = val[i];

  for (size_t j = 0; j < matr_b.Count(); ++j) {
    if (col_a == matr_b.row_ind[j]) {
      mp[{row_a, matr_b.col_ind[j]}] += val_a * matr_b.val[j];
    }
  }
}
```

Двойной вложенный цикл является вычислительным ядром.
Асимптотика по времени: O(nnz_A · nnz_B). В OpenMP-версии
внешний цикл распределяется между потоками.

## 4. Схема распараллеливания

### Выбранная область параллелизации

Параллелится **внешний цикл** по ненулевым элементам матрицы
`A` (`for (size_t i = 0; i < matr_a.Count(); ++i)`). Это решение обусловлено тем, что:

1. Каждая итерация внешнего цикла обрабатывает один элемент `A` (столбец `col_a`).
2. Итерации внешнего цикла независимы по чтению (из `matr_a`
и `matr_b` читаются только константные данные).
3. Запись в общий аккумулятор `mp` происходит из разных
итераций, что создает гонку — следовательно, требуется либо
синхронизация, либо локализация данных.

### Переменные: shared / private / reduction

```cpp
#pragma omp parallel for num_threads(num_threads) schedule(static) default(none) shared(matr_a, matr_b, local_maps)
```

<!-- markdownlint-disable MD013 -->

| Переменная                       | Атрибут                   | Обоснование                                                                |
| -------------------------------- | ------------------------- | -------------------------------------------------------------------------- |
| `matr_a`, `matr_b`               | `shared`                  | Только чтение, гонок нет                                                   |
| `local_maps`                     | `shared`                  | Каждый поток пишет в свой элемент `local_maps[tid]`, поэтому конфликта нет |
| `i`                              | `private` (автоматически) | Индекс цикла, уникален для каждой итерации                                 |
| `row_a`, `col_a`, `val_a`, `tid` | `private`                 | Локальные переменные внутри итерации                                       |
| `j`                              | `private`                 | Индекс внутреннего цикла                                                   |
| `num_threads`                    | не участвует в директиве  | Константа, определена до входа                                             |

<!-- markdownlint-enable MD013 -->

`default(none)` используется для явного контроля — компилятор не делает предположений, все переменные перечислены явно.

### Почему не `reduction`?

Классическая `reduction` для `std::map` не поддерживается
OpenMP (редукция определена только для скалярных типов и
некоторых контейнеров C++, но не для ассоциативных).
Альтернатива — **локализация данных**: каждый поток
накапливает результаты в своём `local_maps[tid]`, а после
параллельной области результаты объединяются в один
глобальный `mp`.

### Выбор `schedule(static)`

```cpp
schedule(static)
```

- **Характер нагрузки:** Все итерации внешнего цикла имеют
одинаковую вычислительную сложность — каждая итерация
перебирает все `nnz(B)` (22000 элементов) и выполняет
проверку `col_a == row_b`.
- **Static schedule:** Итерации распределяются блоками между
потоками на этапе компиляции/запуска. Нет оверхеда на
динамическое планирование.
- **Альтернативы:** `dynamic` ввёл бы дополнительный оверхед
без выигрыша (нагрузка равномерна). `guided` также избыточен.
- **Размер чанка:** По умолчанию `schedule(static)` делит
итерации на блоки размера `count / num_threads`. Для `nnz(A)
= 22000` и, например, 4 потоков — каждый получает 5500
итераций.

### Барьеры

В конце `parallel for` существует **неявный барьер** — все
потоки ждут завершения самого медленного, прежде чем
продолжить выполнение (объединение результатов в `mp`). Это
корректно, так как объединение должно происходить после полной обработки всех итераций.

### Число потоков

```cpp
int num_threads = ppc::util::GetNumThreads();
```

Число потоков задаётся через переменную окружения
`PPC_NUM_THREADS` (runner курса также экспортирует её как
`OMP_NUM_THREADS`). Это обеспечивает единообразие между всеми потоковыми backend-ами.

## 5. Детали реализации

**Файлы:**

- `omp/include/ops_omp.hpp` — объявление класса `ZavyalovAComplSparseMatrMultOMP`.
- `omp/src/ops_omp.cpp` — реализация с OpenMP-директивами.

**Структура пайплайна:** полностью повторяет `seq`:

- `ValidationImpl` — проверка совместимости.
- `PreProcessingImpl` / `PostProcessingImpl` — без действий.
- `RunImpl` — вызывает `MultiplicateWithOmp()`.

**Ключевой фрагмент — вычислительное ядро с OpenMP:**

```cpp
// File: omp/src/ops_omp.cpp, MultiplicateWithOmp()

int num_threads = ppc::util::GetNumThreads();
std::vector<std::map<std::pair<size_t, size_t>, Complex>> local_maps(num_threads);

#pragma omp parallel for num_threads(num_threads) schedule(static) default(none) shared(matr_a, matr_b, local_maps)
for (size_t i = 0; i < matr_a.Count(); ++i) {
  int tid = omp_get_thread_num();
  size_t row_a = matr_a.row_ind[i];
  size_t col_a = matr_a.col_ind[i];
  Complex val_a = matr_a.val[i];

  for (size_t j = 0; j < matr_b.Count(); ++j) {
    if (col_a == matr_b.row_ind[j]) {
      local_maps[tid][{row_a, matr_b.col_ind[j]}] += val_a * matr_b.val[j];
    }
  }
}

// Объединение локальных результатов
std::map<std::pair<size_t, size_t>, Complex> mp;
for (auto &lm : local_maps) {
  for (auto &[key, value] : lm) {
    mp[key] += value;
  }
}
```

**Важные изменения относительно SEQ:**

1. **Вектор локальных карт** `local_maps` размера
`num_threads` — каждый поток накапливает результаты в своей
карте, избегая гонок.
2. **`omp_get_thread_num()`** — идентификатор потока для индексации `local_maps`.
3. **Явная передача `num_threads`** в директиву `parallel for`.
4. **Постобработка** — последовательное слияние `num_threads` карт в одну.

### Гонки данных

**Потенциальная проблема:** Запись в `local_maps[tid]` безопасна, т.к. каждый поток обращается только к своему
элементу вектора. Однако `std::map` не является
потокобезопасным для операций вставки/изменения из разных
потоков, даже если ключи разные. Но поскольку **разные потоки
работают с разными объектами `std::map`** (индексы `tid`
различны), гонок нет. Это ключевое решение, делающее
реализацию корректной без `mutex` или `critical`.

## 6. Проверка корректности

Корректность проверяется теми же **10 функциональными
тестами**, что и для SEQ (см. `seq/report.md`, раздел 5).
Дополнительные проверки:

**Проверка детерминизма:**

OpenMP-реализация **детерминирована** при фиксированном числе потоков:

- `schedule(static)` гарантирует одинаковое распределение
итераций между потоками при каждом запуске.
- Порядок слияния карт в `mp` последователен (цикл по
`local_maps` от 0 до `num_threads-1`), что даёт одинаковый
порядок итераций в результирующих векторах `val`, `row_ind`,
`col_ind`.

**Проверка на идентичность с SEQ:**

Для всех 10 размеров матриц и для `num_threads = 1, 2, 4` результаты побитово совпадают с SEQ. Это подтверждает:

- Корректность распараллеливания.
- Отсутствие гонок (иначе результат был бы нестабильным или неверным).
- Правильную обработку комплексной арифметики.

**Валидация через функциональные тесты:** все тесты пройдены для всех конфигураций потоков.

### Анализ масштабируемости

**Сильное масштабирование:**

- **1 -> 2 потока:** Ускорение 1.84× (эффективность 92.2%) —
отличный результат, близкий к линейному.
- **2 -> 4 потока:** Ускорение 1.53× (с 81 до 53 мс).
Эффективность снижается до 70.3%, что указывает на появление
накладных расходов при удвоении числа потоков.

**Предполагаемые узкие места:**

1. *Накладные расходы на локальные карты:* Каждый поток
создаёт и обслуживает свой `std::map`. При 4 потоках
суммарные затраты на выделение памяти и балансировку деревьев
начинают конкурировать за глобальную кучу.
2. *Объединение локальных карт:* Слияние 4 карт по ~20000
элементов каждая занимает измеримое время.
3. *Память и кэш:* `std::map` реализован как красно-черное
дерево с динамическими узлами. При параллельной работе
нескольких потоков узлы распределяются по куче, что приводит
к кэш-промахам и борьбе потоков за аллокатор памяти.
4. *Разреженность:* Доля успешных вставок мала (~0.005% от
всех проверок), но OpenMP не помогает сократить количество
проверок — только распределяет их.

**Сравнение с теоретическим максимумом:**

Идеальное ускорение на 4 потоках: `149 / 4 = 37.3 мс`.
Реальное (медиана): 53 мс. Потери ~16 мс (~30% от реального
времени) приходятся на:

- Оверхед OpenMP runtime (создание потоков, распределение итераций).
- Накладные расходы на локальные карты и их последующее объединение.
- Кэш-промахи и борьбу потоков за аллокатор памяти при выделении узлов `std::map`.

## 9. Выводы

**Что показала OpenMP-реализация:**

1. **Хорошая масштабируемость на 2 потоках:** эффективность 92.2% — почти идеальный результат.
2. **Умеренное ускорение на 4 потоках:** ускорение 2.81× при
эффективности 70.3% — приемлемый результат для задачи с
разреженными матрицами и динамическими структурами данных.
3. **Простота реализации:** добавление распараллеливания потребовало минимальных изменений относительно SEQ.
4. **Детерминизм:** `schedule(static)` и последовательное объединение гарантируют воспроизводимость результата.
5. **Отсутствие гонок:** локализация данных через `local_maps[tid]` безопасна.

**Когда OpenMP эффективен для этой задачи:**

- Размер `nnz(A)` достаточно велик (≥10000), чтобы оверхед окупался.
- Количество потоков невелико (2–4), так как при дальнейшем
росте накладные расходы на локальные структуры данных
начинают доминировать.

**Итоговая оценка:**

OpenMP-версия демонстрирует **умеренное ускорение** (2.81× на
4 потоках) с **приемлемой эффективностью** (70.3%). Основным
узким местом является не модель распараллеливания, а
внутреннее устройство `std::map`.
