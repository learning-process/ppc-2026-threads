## 1. Контекст

Реализация на `std::thread` представляет собой низкоуровневый подход к распараллеливанию,
где управление потоками, разбиение данных и объединение результатов выполняется вручную.
Этот вариант важен для понимания накладных расходов на создание и синхронизацию потоков.

## 2. Постановка задачи

Полная постановка задачи приведена в отчёте для SEQ-версии [seq/report.md](../seq/report.md).  
Кратко:

- **Вход:** вектор `uint8_t` длины N (пиксели полутонового изображения).
- **Выход:** вектор `uint8_t` той же длины после линейного растяжения диапазона [min, max] → [0, 255].

STL-реализация должна давать **идентичный** SEQ результат при любом количестве потоков.

## 3. Базовый алгоритм

Последовательная версия (SEQ):

1. Найти min и max среди всех пикселей (один линейный проход).
2. Если min == max → вернуть копию входных данных.
3. Вычислить `scale = 255.0 / (max - min)`.
4. Для каждого пикселя: `out[i] = round((input[i] - min) * scale)`.
5. Вернуть выходной вектор.

Сложность: O(N) времени, O(N) памяти.

## 4. Схема распараллеливания

В реализации на базе STL применяется подход **явного разделения данных** и ручное управление временем жизни потоков.
Алгоритм организован в две фазы, между которыми действует барьерная синхронизация.

### Деление на диапазоны

Для сбалансированной загрузки вычислительных ядер исходный массив данных разбивается на блоки (чанки) одинакового размера:

- **Количество потоков ($p$):** определяется вызовом `std::thread::hardware_concurrency()`.
- **Размер блока ($chunk\_size$):** вычисляется как $N / p$, где $N$ — общее количество пикселей.
- **Обработка остатка:** если $N$ не делится нацело на $p$,
  последний поток получает все оставшиеся пиксели вплоть до индекса $N$.
  Это обеспечивает полный обход данных без выхода за допустимые границы.

### Работа потоков

Каждый поток независимо обрабатывает выделенный ему диапазон индексов от `start` до `end`:

1. **В первой фазе (Редукция):** Поток проходит по своему участку и находит локальные экстремумы яркости.
2. **Во второй фазе (Трансформация):** Поток применяет формулу линейной растяжки к каждому пикселю своего диапазона.

### Хранение локальных результатов и синхронизация

- **Локальные результаты:** Минимумы и максимумы каждого потока сохраняются в отдельные векторы `loc_mins` и `loc_maxs`
  по индексу потока.
- **Отсутствие Race Condition:** Поскольку каждый поток пишет строго в свою ячейку памяти, конкуренция за данные
  исключена на уровне архитектуры алгоритма.
- **Синхронизация (Join):** барьер реализован методом `thread.join()`.
  Главный поток ожидает завершения всех рабочих потоков после первой фазы, затем вычисляет глобальные `min` и `max`.
  Только после этого стартует вторая фаза.

### Обоснование выбранной синхронизации

Использование `join()` выполняет здесь роль **барьера**.
Вторая фаза (трансформация) напрямую зависит от результатов первой (глобальные $min$ и $max$).
Вызов `join()` гарантирует, что к моменту начала второй фазы все потоки первой фазы завершили запись своих данных,
и глобальные экстремумы готовы к использованию.
Это наиболее простой и надёжный способ синхронизации для двухстадийных алгоритмов в STL.

### Карта «Поток - Диапазон»

Для обеспечения равномерной загрузки вычислительных ядер используется статическая декомпозиция данных.
Входной вектор
размером $N$ делится на $p$ частей, где $p$ соответствует количеству доступных аппаратных потоков
(`std::thread::hardware_concurrency()`).

**Расчет параметров:**

- `thread_count_` = количество ядер.
- `chunk_size_` = `size_ / thread_count_`.

**Распределение индексов:** Каждый поток $i$ (где $i \in [0, p-1]$) получает свой диапазон $[start_i, end_i)$.

- $start_i = i \times chunk\_size\_$.
- $end_i = (i + 1) \times chunk\_size\_$.

**Обработка остатка:** для последнего потока ($i = p-1$) верхняя граница принудительно устанавливается в `size_`.
Это гарантирует корректную обработку «хвоста» вектора, если размер данных не кратен количеству потоков.

### Схема объединения локальных результатов (Reduction)

Поиск глобальных экстремумов ($min$ и $max$) реализован по схеме бесконфликтной записи,
что исключает состояние гонки и избавляет от накладных расходов на мьютексы.

- **Локальная фаза:** В методе `FindLocalMinMax` создаются два временных вектора `loc_mins` и `loc_maxs` размером
  `thread_count_`. Каждый поток записывает найденный локальный минимум и максимум строго в ячейку по своему индексу $i$.
- **Глобальная фаза:** После завершения всех потоков (`thread.join()`), основной поток выполняет финальную редукцию
  (поиск минимума среди локальных минимумов и максимума среди локальных максимумов) с помощью `std::ranges::min_element`
  и `max_element`.

### Жизненный цикл потоков

В методе `RunImpl` пул потоков создается и синхронизируется дважды:

1. **Первый барьер:** Вызов `FindLocalMinMax`. Потоки завершаются через `join`, чтобы гарантировать готовность глобальных
   значений `min` и `max`.

2. **Второй барьер:** Вызов `ApplyLinearStretching`. Потоки выполняют трансформацию пикселей.
   Синхронизация через `join` в конце
   гарантирует, что метод `RunImpl` вернет управление только после полной готовности выходных данных.

## 9. Выводы

Ручная реализация на `std::thread` показала:

1. **Эффективность ручного управления:** `std::thread` со статической декомпозицией данных показал высокую эффективность
   (ускорение 4.33× на 8 потоках), подтверждая применимость низкоуровневых средств C++ для обработки данных.

2. **Корректность:** Схема бесконфликтной записи (локальные массивы `loc_mins`/`loc_maxs`)
   исключила race conditions без мьютексов, обеспечив побитовую идентичность результату SEQ.

3. **Аппаратные ограничения:** Масштабируемость ограничена **Memory Bound** (низкая вычислительная плотность)
   и эффектом Hyper-Threading (эффективность падает до 0.54 на 8 потоках).

4. **Сложность реализации:** Потребовалась ручная обработка жизненного цикла потоков и граничных условий
   (обработка «хвоста» вектора), но это дало полный контроль над нагрузкой и минимизировало неявные накладные расходы.

**Заключение:** STL Threads — оптимальное решение при отсутствии сторонних зависимостей.
Для данной задачи и конфигурации наилучший выбор — 4 потока (ускорение 3.34×, эффективность 0.83).
