## 1. Контекст

Последовательная версия алгоритма выполняет два независимых прохода по массиву пикселей:
поиск min/max и преобразование значений.
Оба прохода являются **data-parallel**, что делает задачу идеальным кандидатом для распараллеливания с помощью **oneTBB**.

Для реализации выбрана **Intel oneTBB** — библиотека параллельного программирования на основе задач,
которая автоматически управляет пулом потоков и балансировкой нагрузки.
TBB предоставляет более высокий уровень абстракции по сравнению с OpenMP: декомпозиция на задачи,
рекурсивное разбиение диапазонов, автоматическая настройка количества чанков.
Наиболее интересными аспектами применения TBB в данной задаче являются
**разбиение диапазонов** и **динамическое распределение задач**.

Реализация сохраняет структуру SEQ-кода, заменяя два основных цикла на TBB-примитивы.

## 2. Постановка задачи

Полная постановка задачи приведена в отчёте для SEQ-версии [seq/report.md](../seq/report.md).  
Кратко:

- **Вход:** вектор `uint8_t` длины N (пиксели полутонового изображения).
- **Выход:** вектор `uint8_t` той же длины после линейного растяжения диапазона [min, max] → [0, 255].

TBB-реализация должна давать **идентичный** SEQ результат при любом количестве потоков.

## 3. Базовый алгоритм

Последовательная версия (SEQ):

1. Найти min и max среди всех пикселей (один линейный проход).
2. Если min == max → вернуть копию входных данных.
3. Вычислить `scale = 255.0 / (max - min)`.
4. Для каждого пикселя: `out[i] = round((input[i] - min) * scale)`.
5. Вернуть выходной вектор.

Сложность: O(N) времени, O(N) памяти.

## 4. Схема распараллеливания

```cpp
  auto [min_pixel, max_pixel] = tbb::parallel_reduce(tbb::blocked_range<size_t>(1, size), MinMax{input[0], input[0]}, [&](const auto &range, MinMax init) -> MinMax {
    for (size_t i = range.begin(); i != range.end(); ++i) {
      init.first = std::min(init.first, input[i]);
      init.second = std::max(init.second, input[i]);
    }
    return init;
  }, [](const MinMax &a, const MinMax &b) -> MinMax {
    return {std::min(a.first, b.first), std::max(a.second, b.second)};
  });
```

### Выбранные примитивы

- **Для поиска min/max:** `tbb::parallel_reduce` — параллельная редукция с накоплением промежуточных результатов.
- **Для преобразования пикселей:** `tbb::parallel_for` — независимое применение операции к каждому элементу.
- **Для копирования при константном изображении:** `tbb::parallel_for` с `simple_partitioner`.

### Диапазон работы

Используется `tbb::blocked_range<size_t>(start, end, grainsize)`.
Диапазон представляет собой полуинтервал [begin, end).
TBB автоматически рекурсивно разбивает его на более мелкие поддиапазоны до достижения размера grainsize.

### Размер grainsize

- **parallel_reduce (поиск min/max):** grainsize не указан явно. Используется значение по умолчанию,
  которое TBB подбирает эмпирически, обычно около 1–8 итераций на задачу.
  Это обеспечивает хороший баланс между параллелизмом и накладными расходами на создание задач.
- **parallel_for (преобразование):** также используется grainsize по умолчанию.
  Для массива из 10 миллионов элементов TBB создаст сотни или тысячи задач,
  что обеспечивает эффективную балансировку нагрузки.
- **parallel_for (копирование при константном изображении):** используется `tbb::simple_partitioner()`,
  который разбивает диапазон на максимально мелкие задачи (grainsize ≈ 1). Это оправдано,
  так как операция копирования очень лёгкая, и требуется максимальное распараллеливание.

### partitioner

| Примитив                        | partitioner                       | Обоснование                                                                                                                      |
| ------------------------------- | --------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `parallel_reduce`               | `auto_partitioner` (по умолчанию) | Адаптивно подстраивается под загрузку системы                                                                                    |
| `parallel_for` (преобразование) | `static_partitioner`              | Диапазон известен, итерации одинаковой сложности. static_partitioner минимизирует накладные расходы на динамическое планирование |
| `parallel_for` (копирование)    | `simple_partitioner`              | Максимальное измельчение для равномерной загрузки при очень лёгких операциях                                                     |

## 5. Детали реализации

### Файлы

- `tbb/include/ops_tbb.hpp` — заголовок класса `PylaevaSIncContrastImgByLshTBB`.
- `tbb/src/ops_tbb.cpp` — реализация с использованием TBB.

### Локальные аккумуляторы и объединение результатов

Для `parallel_reduce` определена структура `MinMax` как `std::pair<uint8_t, uint8_t>`.
Локальный аккумулятор `init` инициализируется первым элементом массива.
Тело редукции для каждого поддиапазона обновляет локальные min и max.
Функция объединения (combine) принимает два результата и возвращает пару из минимального и максимального значений.

### Работа с памятью

Кэш-зависимости не являются проблемой, так как доступ к массиву линейный,
а TBB автоматически распределяет поддиапазоны так,
чтобы соседние индексы с высокой вероятностью обрабатывались одним потоком.
Это минимизирует ложное разделение кэша (false sharing).
Для данной задачи не требуется использования `cache_aligned_allocator` или `scalable_allocator`,
так как каждый поток работает с непрерывными блоками памяти, и конфликтов кэш-линий не возникает.

### Особые случаи

- Пустой вход → `ValidationImpl()` возвращает `false`.
- Константное изображение (min == max) → выполняется `parallel_for` с `simple_partitioner`
  для параллельного копирования входных данных в выходной массив.
- Диапазон начинается с индекса 1 в `parallel_reduce`,
  так как элемент с индексом 0 используется для инициализации.

### **Анализ результатов**

- **Падение эффективности** с 0.56 (2 потока) до 0.29 (8 потоков)
  указывает на возрастающие накладные расходы при увеличении числа потоков.
- **Основной «узкий» участок** — оперативная память.
  При 8 потоках пропускная способность памяти становится лимитирующим фактором,
  так как все потоки одновременно читают входной массив и пишут в выходной.
- **Накладные расходы TBB** включают создание и синхронизацию задач в `parallel_reduce`,
  а также вызовы комбинирующей функции.
  Для задачи с O(N) арифметики и O(N) обращений к памяти эти затраты заметны, особенно на малых размерах задач.

## 9. Выводы

### **Применимость TBB**

В данной задаче TBB обеспечил ускорение до 2.33× на 8 потоках.
Основное ограничение — пропускная способность памяти, а не накладные расходы библиотеки.

### **Когда TBB удобен**

- Регулярные data-parallel задачи с автоматической балансировкой (`parallel_for`, `parallel_reduce`)
- Кроссплатформенная разработка без привязки к конкретному компилятору
- Сценарии с неоднородной нагрузкой (благодаря work-stealing)

### **Когда накладные расходы TBB мешают**

- **Малые объёмы данных** (< 10⁵ элементов) — затраты на создание задач и редукцию превышают выгоду от распараллеливания
  (в тестах с N=8 ускорение отсутствовало)
- **Очень лёгкие итерации** — даже `simple_partitioner` не окупает издержки планировщика
- **Системы реального времени** — непредсказуемые задержки из-за кражи задач

### **Итог**

TBB — эффективный инструмент для распараллеливания задач с регулярными итерациями на крупных объёмах данных.
В данной реализации ключевым фактором успеха стал правильный выбор partitioner'а:
`static_partitioner` для однородного преобразования и `simple_partitioner` для лёгкого копирования.
