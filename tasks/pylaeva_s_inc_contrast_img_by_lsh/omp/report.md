## 1. Контекст

Последовательная версия (SEQ) алгоритма линейной растяжки гистограммы выполняет два независимых прохода по массиву пикселей: поиск минимума/максимума и непосредственно преобразование значений. Для данных проходов по массивам обработка каждого элемента не зависит от других. Это делает алгоритм идеальным кандидатом для распараллеливания с помощью OpenMP. В данную реализацию переносятся оба цикла с минимальными изменениями кода.

## 2. Постановка задачи

Полная постановка задачи приведена в отчёте для SEQ-версии (`seq/report.md`).  
Кратко:

- **Вход:** вектор `uint8_t` длины N (пиксели полутонового изображения).
- **Выход:** вектор `uint8_t` той же длины после линейного растяжения диапазона [min, max] → [0, 255].

OpenMP-реализация должна давать **идентичный** SEQ результат при любом количестве потоков.

## 3. Базовый алгоритм

Последовательная версия (SEQ):

1. Найти min и max среди всех пикселей (один линейный проход).
2. Если min == max → вернуть копию входных данных.
3. Вычислить `scale = 255.0 / (max - min)`.
4. Для каждого пикселя: `out[i] = round((input[i] - min) * scale)`.
5. Вернуть выходной вектор.

Сложность: O(N) времени, O(N) памяти.

## 4. Схема распараллеливания

```cpp
#pragma omp parallel for default(none) shared(input, input_size) reduction(min : min_pixel) reduction(max : max_pixel)
  for (size_t i = 0; i < input_size; ++i) {
    min_pixel = std::min(min_pixel, input[i]);
    max_pixel = std::max(max_pixel, input[i]);
  }

#pragma omp parallel for default(none) shared(input, output, min_pixel, input_size, scale)
  for (size_t i = 0; i < input.size(); ++i) {
    output[i] = static_cast<uint8_t>(std::round(static_cast<float>(input[i] - min_pixel) * scale));
  }

  return true;
}
```

### Параллелизуемые области

Первый цикл — поиск минимального и максимального значений пикселей. Второй цикл — линейное преобразование каждого пикселя. Обе области являются независимыми по данным: каждая итерация обрабатывает свой элемент, и результат не зависит от порядка выполнения.

### Переменные и их атрибуты

В первом цикле переменные input и input_size объявлены как shared, так как все потоки только читают эти данные. Переменные min_pixel и max_pixel используют reduction с операциями min и max соответственно. Это позволяет каждому потоку иметь локальную копию, а после завершения цикла все локальные значения объединяются с помощью соответствующих операций.

Во втором цикле все переменные объявлены как shared. Переменная output является shared, но конфликтов не возникает, потому что каждый поток пишет в уникальные индексы выходного массива. Остальные переменные (input, min_pixel, scale) используются только для чтения.

### Синхронизация

Reduction используется в первом цикле для корректного объединения локальных минимумов и максимумов из каждого потока. Операции atomic и critical не требуются, так как во втором цикле отсутствуют гонки данных. В конце каждой параллельной области по умолчанию присутствует неявный барьер, который ожидает завершения всех потоков перед продолжением выполнения программы.

### Выбор schedule

Используется schedule(static), который является поведением по умолчанию. Такой выбор обоснован тем, что доступ к памяти линейный и равномерный, все итерации имеют одинаковую вычислительную сложность. Static-разбиение минимизирует накладные расходы на планирование, и нет необходимости в динамическом перераспределении работы.

## 5. Детали реализации

### Файлы

- `omp/include/ops_omp.hpp` — объявление класса `PylaevaSIncContrastImgByLshOMP`.
- `omp/src/ops_omp.cpp` — реализация методов.

### Изменения относительно SEQ

В последовательной версии для поиска минимума и максимума использовался вызов std::ranges::minmax. В OpenMP-версии этот вызов заменён на ручной цикл с директивой parallel for и reduction. Цикл преобразования также получил директиву parallel for.

### Устранение рисков гонок

Первая проблема — несколько потоков одновременно обновляют общие переменные min_pixel и max_pixel. Она решается с помощью reduction: каждый поток работает с локальной копией, а финальное объединение происходит автоматически.

Вторая проблема — одновременная запись в output разными потоками. Она решается тем, что каждый поток пишет только в свои индексы на основе статического распределения.
