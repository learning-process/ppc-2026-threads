## 1. Введение

Линейная растяжка гистограммы — классическая задача обработки изображений,
идеально подходящая для сравнения различных моделей параллелизма.
Алгоритм выполняется за два независимых линейных прохода по массиву пикселей:

- поиск глобальных min / max;
- преобразование каждого пикселя по формуле `out[i] = round((in[i] - min) * 255/(max-min))`.

Оба этапа обладают **идеальной параллелизуемостью** по данным: обработка каждого элемента не зависит от других.
Это позволяет сосредоточиться на сравнении накладных расходов, балансировки и удобства реализации.

## 2. Единая постановка задачи

### Входные данные

- Вектор `uint8_t` произвольной длины \( N \) — пиксели полутонового изображения (значения от 0 до 255).

### Выходные данные

- Вектор `uint8_t` той же длины — обработанное изображение после линейной растяжки.

### Ограничения

- Размер изображения \( N >= 1 \).
- Значения пикселей — целые беззнаковые 8 бит.

### Инварианты и критерий корректности

- Инвариант: сохраняется порядок пикселей (монотонное преобразование).
- Критерий: выходной диапазон всегда [0, 255], если только вход не был константным.

### Подготовка данных

Заполнение данных для `performance` тестов:

```cpp
void SetUp() override {
    input_data_.resize(kCount_);

    // Создаем градиент: значения от 0 до 255
    for (size_t i = 0; i < kCount_; ++i) {
      input_data_[i] = static_cast<uint8_t>((i * 255) / kCount_);
```

### Метрики

- **Speedup** = `T_seq / T_par`.
- **Efficiency** = `Speedup / total_workers` (total_workers = число потоков × число процессов для MPI).
- База для `T_seq` — замер SEQ-версии.

## 6. Интерпретация различий

### 6.1. SEQ — роль baseline (эталона)

Последовательная версия выполняет алгоритм за два линейных прохода по массиву: поиск min/max и преобразование.
Её характеристики:

| Характеристика           | Значение                                    |
| ------------------------ | ------------------------------------------- |
| Вычислительная сложность | O(N)                                        |
| Память                   | O(N)                                        |
| Время (N=10⁷)            | 0.130615 с                                  |
| Затратная операция       | `std::round` + умножение с плавающей точкой |

**Роль SEQ в анализе:**

21. **База для расчёта ускорения** — `Speedup = T_seq / T_par`.
    Именно SEQ даёт точку отсчёта для оценки эффективности параллелизации.

22. **Эталон корректности** — выходные данные всех параллельных версий должны побитово совпадать с SEQ.

### 6.2. OMP — сильные и слабые стороны на данной задаче

#### Сильные стороны

| Аспект                            | Проявление в задаче                                                       |
| --------------------------------- | ------------------------------------------------------------------------- |
| **Минимальные накладные расходы** | Ускорение 5.02× на 8 потоках — лучшее среди всех технологий на одном узле |
| **Простота реализации**           | Достаточно двух директив `#pragma omp parallel for` с `reduction`         |
| **Эффективная редукция**          | `reduction(min:...)` выполняется без явных барьеров и критических секций  |
| **Статическое планирование**      | Идеально подходит для равномерной вычислительной нагрузки                 |
| **Локальность данных**            | Каждый поток обрабатывает непрерывный блок памяти (cache-friendly)        |

#### Слабые стороны

### 6.3. TBB — роль grain size и runtime

В реализации на oneTBB ключевым параметром, определяющим баланс между параллелизмом и накладными расходами,
является **grain size** — минимальный размер блока данных, который выделяется одной задаче.

- **Влияние grain size:**
  - В `parallel_reduce` (поиск min/max) grain size не задавался явно,
      TBB подбирает его автоматически (обычно 1–8 итераций на задачу).
      Это даёт хорошую балансировку, но увеличивает число задач и накладные расходы.
  - В `parallel_for` для преобразования использовался `static_partitioner`,
      который минимизирует динамическое планирование — оптимально для однородной нагрузки.
  - Для копирования при константном изображении применён `simple_partitioner` (grain size ≈ 1),
      что максимально дробит задачу, но оправдано из-за крайне лёгких операций.

- **Роль runtime (планировщика задач):**
  - TBB использует **work-stealing**: потоки, завершившие свои задачи, «воруют» задачи у других.
      Это автоматически балансирует нагрузку при неравномерном распределении данных.
  - Однако для идеально регулярной задачи (градиент) планировщик вносит
      **ненужные накладные расходы** на управление задачами и кражу,
      что видно по падению эффективности с 0.57 (2 потока) до 0.29 (8 потоков).
  - В отличие от OpenMP со статическим планированием, TBB на 8 потоках дал ускорение только 2.32× против 5.02× у OpenMP.

**Вывод:** TBB удобен для **нерегулярных задач** с непредсказуемой нагрузкой,
но для **регулярных data-parallel алгоритмов** (как линейная растяжка) его динамический runtime создаёт
избыточные накладные расходы, снижая эффективность по сравнению с OpenMP.

### 6.4. STL

Реализация на `std::thread` требует **явного управления** всеми аспектами параллелизма,
что даёт как преимущества, так и недостатки.

**Цена ручного управления:** Увеличенный объем кода , риск ошибок, отсутствие динамической балансировки.

**Выгода:** Минимальные накладные расходы

### 6.5. ALL (MPI+OpenMP) — цена коммуникации и выигрыш гибридности

Гибридная версия сочетает распределённую память (MPI) и общую память (OpenMP),
что позволяет использовать кластерные узлы максимально эффективно,
но платой являются **коммуникационные накладные расходы**.

**Цена коммуникации:**

- Операции `MPI_Scatterv` / `MPI_Allgatherv` требуют передачи всего массива данных через коммуникационную сеть,
  что создаёт существенную нагрузку на межпроцессное взаимодействие.
- Два вызова `MPI_Allreduce` вынуждают все процессы ожидать завершения самого медленного из них,
  что приводит к общему простою и снижению масштабируемости.
- Даже в пределах одного вычислительного узла межпроцессное взаимодействие порождает
  избыточные копирования данных между адресными пространствами процессов.

**Когда ALL оправдан:**

- Данные не помещаются в оперативную память одного узла.
- Использование реального многоузлового кластера с высокоскоростной коммуникационной сетью (InfiniBand).
- Решение задач с высокой вычислительной сложностью на один пиксель,
  где накладные расходы на коммуникацию становятся незаметными на фоне интенсивных вычислений.

## 8. Заключение

### Лучшая версия и условия применения

- **Для одной машины (до 4 физических ядер):** лучшую производительность показывает **OpenMP**
  (ускорение до 5.02× на 8 потоках) при минимальных накладных расходах и простоте реализации.
  Оптимальная конфигурация — 2 потока (эффективность 96%) или 4 потока (баланс скорости и ресурсов).
- **Для многоузловых кластеров / крупных данных (≥10⁷ пикселей):** предпочтительна
  **гибридная схема MPI+OpenMP** (ускорение до 5.19× на 16 воркерах).
  Оптимальная конфигурация — 2 процесса × 4 потока (эффективность 95%).
- **При отсутствии сторонних зависимостей:**
  **STL threads** даёт хорошее ускорение (4.33× на 8 потоках) и полный контроль над разбиением.

### Ограничения сравнения

- Замеры выполнены на **одном узле** (ноутбук 4 ядра / 8 потоков) — выводы для больших кластеров могут отличаться.
- Размер задачи фиксирован (10⁷ пикселей) — для малых изображений (<10⁵)
  все параллельные версии могут проигрывать SEQ из-за накладных расходов.
- Данные представляют **идеально сбалансированный градиент** — реальные изображения
  с неравномерной гистограммой могут изменить баланс.
- Используется один компилятор (MinGW‑GCC) — результаты для MSVC или Clang/ICC могут различаться.

### Направления улучшений

- Применить **векторизацию (AVX2/AVX-512)** — для memory-bound задач это может дать дополнительный прирост 2–4×.
- Для гибридной версии заменить `MPI_Allreduce` на асинхронный вариант
  или совместить две редукции (min и max) в одно сообщение.
- В TBB-реализации попробовать `affinity_partitioner` для улучшения локальности данных на больших потоках.
