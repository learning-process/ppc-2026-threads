## 1. Контекст

Гибридная версия (MPI + OpenMP) объединяет два уровня параллелизма: **межпроцессный** (распределённая память через MPI)
и **внутрипроцессный** (общая память через OpenMP). Такая иерархическая схема необходима для эффективного использования
современных кластерных систем, где каждый вычислительный узел содержит многоядерный процессор.

- **MPI** обеспечивает распределение данных между независимыми процессами, которые могут выполняться на разных узлах кластера,
  а также глобальную синхронизацию и обмен результатами.
- **OpenMP** внутри каждого процесса распараллеливает обработку локальной части данных на всех доступных ядрах узла.

## 2. Постановка задачи

Полная постановка задачи приведена в отчёте для SEQ-версии [seq/report.md](../seq/report.md).

Кратко:

- **Вход:** вектор `uint8_t` длины N (пиксели полутонового изображения).
- **Выход:** вектор `uint8_t` той же длины после линейного растяжения диапазона [min, max] → [0, 255].

**Особенность ALL-версии:** входные данные распределяются между MPI-процессами, каждый процесс обрабатывает свою часть
с использованием потоков OpenMP, затем результаты собираются на всех процессах (каждый процесс получает полный результат).

## 3. Базовый алгоритм

Последовательная версия (SEQ):

1. Найти min и max среди всех пикселей (один линейный проход).
2. Если min == max → вернуть копию входных данных.
3. Вычислить `scale = 255.0 / (max - min)`.
4. Для каждого пикселя: `out[i] = round((input[i] - min) * scale)`.
5. Вернуть выходной вектор.

Сложность: O(N) времени, O(N) памяти.

## 4. Межпроцессная схема

### Роли процессов (рангов)

Используется **однопрограммный многопоточный (SPMD)** подход с явным ветвлением по рангу:

| Ранг         | Роль                                                                                                                                                                                      |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **0**        | Главный процесс. Выполняет валидацию входных данных, вычисляет параметры распределения (`recv_counts`, `recv_displs`), рассылает их всем процессам, а также распределяет исходные данные. |
| **1 .. P-1** | Рабочие процессы. Получают свою часть данных, выполняют вычисления, участвуют в глобальной редукции.                                                                                      |

### Коммуникатор

Все процессы используют стандартный коммуникатор `MPI_COMM_WORLD`, включающий все запущенные MPI-процессы.

### Распределение данных (Scatterv)

Поскольку размер входных данных N может не делиться нацело на количество процессов P,
используется **векторная версия** `MPI_Scatterv`, позволяющая передавать блоки разного размера разным процессам.

**Алгоритм распределения:**

1. Процесс 0 вычисляет базовый размер чанка: `base_chunk = N / P`
2. Остаток от деления: `remainder = N % P`
3. Первые `remainder` процессов получают `base_chunk + 1` элементов, остальные — `base_chunk` элементов.
4. Массивы `recv_counts[]` (количество элементов для каждого процесса)
   и `recv_displs[]` (смещения в исходном массиве) заполняются на процессе 0.
5. Через `MPI_Bcast` эти массивы рассылаются всем процессам.
6. `MPI_Scatterv` распределяет данные из `input` процесса 0 по локальным буферам `local_data_` всех процессов.

### Используемые MPI-вызовы

| Вызов                 | Назначение                                                                                    | Место вызова                                   |
| --------------------- | --------------------------------------------------------------------------------------------- | ---------------------------------------------- |
| `MPI_Comm_rank`       | Получение ранга текущего процесса                                                             | Validation, PreProcessing, Run, PostProcessing |
| `MPI_Comm_size`       | Получение общего числа процессов                                                              | PreProcessing                                  |
| `MPI_Bcast` (bool)    | Синхронизация результата валидации — все процессы получают статус проверки                    | ValidationImpl                                 |
| `MPI_Bcast` (size)    | Рассылка размера входных данных всем процессам                                                | PreProcessing                                  |
| `MPI_Bcast` (counts)  | Рассылка массива `recv_counts`                                                                | PreProcessing                                  |
| `MPI_Bcast` (displs)  | Рассылка массива `recv_displs`                                                                | PreProcessing                                  |
| `MPI_Scatterv`        | Распределение данных от процесса 0 ко всем                                                    | PreProcessing                                  |
| `MPI_Allreduce` (MIN) | Глобальное нахождение минимума — все процессы получают глобальный min                         | RunImpl                                        |
| `MPI_Allreduce` (MAX) | Глобальное нахождение максимума — все процессы получают глобальный max                        | RunImpl                                        |
| `MPI_Allgatherv`      | Сбор локальных результатов от всех процессов — каждый процесс получает полный выходной вектор | PostProcessing                                 |

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса для обработки локальной части данных используется **OpenMP**.
Выбор OpenMP для гибридной версии обусловлен:

- Простотой интеграции с MPI (оба используют модель памяти с раздельными адресными пространствами процессов)
- Эффективностью на общей памяти одного узла
- Минимальными накладными расходами на создание потоков (по сравнению с `std::thread`)
- Возможностью использовать директивы `reduction` для простого объединения локальных результатов

### Распараллеливаемые области

В методе `RunImpl()` распараллеливаются два цикла с помощью директив OpenMP.

#### Поиск локальных минимума и максимума

```cpp
uint8_t loc_min_v = 255;
uint8_t loc_max_v = 0;

#pragma omp parallel for default(none) reduction(min : loc_min_v) reduction(max : loc_max_v)
for (size_t i = 0; i < local_size_; i++) {
    loc_min_v = std::min(loc_min_v, local_data_[i]);
    loc_max_v = std::max(loc_max_v, local_data_[i]);
}
```

## Принцип распределения итераций

Итерации цикла делятся на непрерывные блоки (статическое планирование по умолчанию):

- Каждый поток получает примерно равное количество итераций
- Размер блока ≈ `local_size_ / num_threads`

## Объединение локальных результатов

Для объединения результатов используется **reduction**:

| Переменная  | Операция | Назначение                                                                      |
| ----------- | -------- | ------------------------------------------------------------------------------- |
| `loc_min_v` | `min`    | Каждый поток находит свой локальный минимум, затем все объединяются через `min` |
| `loc_max_v` | `max`    | Каждый поток находит свой локальный максимум, затем объединение через `max`     |

### Механизм работы reduction

1. Каждый поток получает приватную копию переменной
2. Поток обрабатывает свои итерации, обновляя приватную копию
3. После завершения цикла все приватные значения объединяются глобальной операцией
4. Результат записывается в исходную переменную

**Преимущество:** отсутствие гонок данных без использования `atomic` или `critical`, что обеспечивает максимальную производительность.

## Второй параллельный цикл (преобразование)

```cpp
#pragma omp parallel for default(none) shared(global_min, scale)
for (size_t i = 0; i < local_size_; i++) {
    int pixel = local_data_[i];
    int result = static_cast<int>(std::lround((pixel - global_min) * scale));
    local_out_[i] = static_cast<uint8_t>(std::clamp(result, 0, 255));
}
```

- shared(global_min, scale) — переменные только читаются, гонок нет
- Каждый поток пишет в уникальные индексы local*out* — синхронизация не требуется
- Неявный барьер в конце гарантирует завершение всех потоков перед продолжением

### Анализ результатов

1. **Эффективность гибридной схемы:** Использование MPI+OpenMP позволило сократить время обработки
   изображения с **0.1306 с** до **0.0252 с**.
   Максимальное достигнутое ускорение составило **5.19×** при использовании 16 работников
   (8 процессов × 2 потока).

2. **Хорошая масштабируемость до 8 работников:** При увеличении общего числа работников с 2 до 8
   ускорение выросло с 2.18× до 4.19×.
   Это свидетельствует о том, что гибридная версия эффективно использует доступные вычислительные
   ресурсы в этом диапазоне.

3. **Насыщение после 8 работников:** Дальнейшее увеличение числа работников с 8 до 16 дало прирост
   ускорения лишь с 4.19× до 5.19×.
   Добавление новых воркеров становится всё менее эффективным из-за растущей доли коммуникационных
   затрат.

4. **Предпочтительность увеличения потоков над процессами:** Конфигурация 2 процесса × 8 потоков
   (16 работников) показала ускорение 4.25×,
   что сопоставимо с 4 процессами × 4 потока (4.22×).
   Увеличение числа потоков на процесс эффективнее добавления новых MPI-процессов благодаря меньшим
   накладным расходам на коммуникацию.

5. **Основное узкое место:** При малом числе работников (≤ 8) узким местом являются вычисления,
   которые хорошо распараллеливаются.
   При большом числе работников (≥ 16) доминирующим фактором становятся коммуникационные затраты на
   `MPI_Allreduce` и `MPI_Allgatherv`,
   а также снижение эффективности из-за диспропорции между вычислительной и коммуникационной
   нагрузкой.

6. **Оптимальная конфигурация:** Лучший баланс между ускорением (5.19×) и эффективностью (32.5%)
   достигнут на конфигурации 8 процессов × 2 потока.
   Для практического использования рекомендована конфигурация 2 процесса × 4 потока
   (ускорение 3.81×, эффективность 95.2%) как наиболее эффективное использование ресурсов.

## 10. Выводы

### Когда гибридная схема оправдана

- **Крупные изображения** (≥ 10⁷ пикселей) — коммуникационные затраты перекрываются выигрышем от
  параллелизации
- **Кластеры с многоядерными узлами** (≥ 8 ядер на узел) — эффективно использовать оба уровня
  параллелизма
- **Слабо связанные задачи** — алгоритм с одним глобальным обменом (min/max) хорошо ложится на
  гибридную модель

### Когда гибридная схема не оправдана

- **Малые изображения** (< 10⁵ пикселей) — накладные расходы на MPI превышают выигрыш
- **Одноядерные узлы** — OpenMP не даёт ускорения, остаются только MPI-накладные расходы
- **Плотные коммуникации** — если требуется частый обмен данными между процессами, чисто MPI может
  быть быстрее

### Итог

Гибридная схема MPI+OpenMP показала **ускорение до 5.2×** на 16 воркерах (8 процессов × 2 потока).
Однако эффективность падает с 95% (2 воркера) до 32% (16 воркеров) из-за растущей доли
коммуникации.
**Оптимальная конфигурация** для данной задачи: 2 процесса × 4 потока (эффективность 95%),
что соответствует архитектуре узла (4 физических ядра с Hyper-Threading).

Рекомендация: гибридная модель оправдана только при решении **крупных задач** на
**многоядерных кластерах**.
Для небольших изображений или отладки предпочтительнее последовательная или чисто OpenMP-версия.
