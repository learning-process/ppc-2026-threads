## 1. Контекст
В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших наборов точек целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от результата предыдущего шага.

В отличие от OpenMP и TBB, которые предоставляют высокоуровневые абстракции, версия на `std::thread` требует ручного управления потоками. Это позволяет получить полный контроль над созданием, синхронизацией и жизненным циклом потоков, но также переносит на разработчика ответственность за корректное разделение данных, предотвращение гонок и минимизацию накладных расходов.

Цель данной реализации — продемонстрировать возможность распараллеливания алгоритма Грэхема с использованием базовых инструментов стандартной библиотеки C++ и оценить накладные расходы такого подхода.

## 2. Постановка задачи
Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости. Входными данными является вектор пар координат, выходными — вектор точек, образующих выпуклую оболочку и упорядоченных против часовой стрелки.

Полная постановка задачи, описание входных/выходных данных и крайних случаев приведены в отчёте по последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

STL-версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом достигая ускорения за счёт использования многоядерной архитектуры.

## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в [`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:
1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания
Распараллеливанию подверглись первые два этапа алгоритма:
1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.1. Разделение данных
Разделение данных на `P` потоков выполняется по принципу блочного распределения. Весь массив точек делится на `P` непрерывных, непересекающихся блоков. Для равномерного распределения нагрузки размеры блоков делаются максимально близкими.

Если общее число точек `N` не делится нацело на `P`, то первые `N % P` блоков получают по одному дополнительному элементу. Таким образом, каждый поток получает для обработки свой уникальный диапазон индексов.

### 4.2. Поиск опорной точки
1. **Локальный поиск**: Каждый поток ищет локальную опорную точку в своём блоке данных.
2. **Хранение локальных результатов**: Индекс найденной локальной опорной точки каждый поток записывает в общиий массив `local_idx` в ячейку, соответствующую своему `tid`. Это исключает гонку данных, так как каждый поток пишет в уникальную область памяти.
3. **Синхронизация**: После завершения работы всех потоков (неявный барьер) главный поток последовательно обходит массив `local_idx` и находит глобальную опорную точку среди локальных кандидатов.

### 4.3. Параллельная сортировка
Реализована параллельная сортировка слиянием.
1. **Локальная сортировка**: Каждый поток сортирует свой блок данных с помощью `std::sort`.
2. **Слияние**: Выполняется итеративное попарное слияние отсортированных блоков по принципу дерева. На каждом уровне дерева слияния (`size = 1, 2, 4, ...`) запускается новый набор потоков, каждый из которых выполняет `std::inplace_merge` для своей пары блоков.

### 4.4. Управление потоками и синхронизация
- **Жизненный цикл**: Для каждого параллельного этапа (поиск опорной точки, локальная сортировка, каждый уровень слияния) создаётся свой набор потоков `std::jthread`.
- **Синхронизация**: Явные примитивы синхронизации (`mutex`, `atomic`) не используются. Синхронизация достигается за счёт барьеров, которые реализуются с помощью RAII-семантики `std::jthread`. Потоки для одного этапа создаются, выполняют свою работу, а в конце области видимости их деструкторы автоматически вызывают `join()`, что останавливает главный поток до завершения всех рабочих потоков. Это гарантирует, что, например, все локальные сортировки завершатся до начала первого этапа слияния.

Такой подход `create → work → join` для каждого этапа приводит к значительным накладным расходам по сравнению с использованием постоянного пула потоков (как в OpenMP или TBB).

### Поиск опорной точки
В программной реализации для сбора локальных результатов применяется общий вектор `local_idx`. Выделение независимых ячеек памяти по индексу потока (`tid`) исключает состояние гонки и позволяет отказаться от использования примитивов синхронизации, таких как `std::atomic` или `std::mutex`.

```cpp
// stl/src/ops_stl.cpp
std::vector<size_t> local_idx(threads, 0);
{
  std::vector<std::jthread> workers(threads);

  for (int tid = 0; tid < threads; tid++) {
    workers.emplace_back([&, tid]() {
      size_t begin = start[tid];
      size_t end = start[tid + 1];

      size_t local = begin;
      for (size_t i = begin + 1; i < end; i++) {
        if (pts[i].second < pts[local].second ||
            (pts[i].second == pts[local].second && pts[i].first < pts[local].first)) {
          local = i;
        }
      }
      local_idx[tid] = local; // Бесконфликтная запись результата
    });
  }
} // Неявный вызов join() в деструкторах std::jthread
```

Контроль жизненного цикла потоков (`create → work → join`) осуществляется за счёт ограничения области видимости вектора `workers`. При выходе из блока `{ ... }` деструкторы `std::jthread` автоматически блокируют вызывающий поток до завершения работы всех рабочих потоков, формируя барьер синхронизации перед этапом последовательной редукции.

### Сортировка точек по полярному углу
Процесс параллельной сортировки программно разделен на независимую локальную сортировку подмассивов и их последующее параллельное слияние.

```cpp
// stl/src/ops_stl.cpp
// 1. Локальная сортировка
{
  std::vector<std::jthread> workers(threads);
  for (int tid = 0; tid < threads; tid++) {
    workers.emplace_back([&, tid]() {
      std::sort(data.begin() + start[tid], data.begin() + start[tid + 1],
                [&](const auto &a, const auto &b) { return AngleCmp(a, b, pivot); });
    });
  }
}

// 2. Слияние отсортированных сегментов
for (int size = 1; size < threads; size *= 2) {
  std::vector<std::jthread> workers(threads);

  for (int i = 0; i < threads; i += 2 * size) {
    if (i + size >= threads) { continue; }

    workers.emplace_back([&, i, size]() {
      int left = start[i];
      int mid = start[i + size];
      int right = start[std::min(i + (2 * size), threads)];

      std::inplace_merge(data.begin() + left, data.begin() + mid, data.begin() + right,
                         [&](const auto &a, const auto &b) { return AngleCmp(a, b, pivot); });
    });
  }
}
```

На этапе слияния внутренний цикл распределяет непересекающиеся диапазоны `[left, right)` между потоками для стандартной функции `std::inplace_merge`. 

Вектор `workers` создается заново на каждой итерации внешнего цикла (на каждом уровне дерева слияния). Это гарантирует строгую барьерную синхронизацию: переход к следующему уровню укрупнения блоков начинается только после того, как все потоки завершат текущее слияние и будут корректно уничтожены.

## 6. Проверка корректности
Корректность работы STL-реализации проверяется путем прямого сравнения её выходного вектора с результатом, полученным от SEQ-версии. Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный результат.

Дополнительные гарантии:
- При `PPC_NUM_THREADS=1` результат работы STL-версии полностью идентичен SEQ-версии.
- При различных значениях `PPC_NUM_THREADS` (2, 4, 6, ...) выходные векторы по-прежнему совпадают между собой и с эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

### Анализ масштабируемости
Анализ выполнялся на основе сценария `task_run`.
- **Ускорение**: Пиковое ускорение ~3.0x достигается на 8 потоках. Рост производительности наблюдается до 8 потоков, что близко к количеству физических ядер (6).
- **Эффективность**: Эффективность монотонно снижается с ростом числа потоков, что является типичным для ручного управления. На 2 потоках она составляет ~76%, но падает до ~37% на 8 потоках.
- **Накладные расходы**: После 8 потоков производительность начинает деградировать. Это связано с высокими накладными расходами на создание и уничтожение потоков (`create/join`). В данной реализации потоки создаются заново для каждого из этапов: поиск опоры, локальная сортировка и для каждого из `log(P)` уровней слияния. Это значительно увеличивает общее время по сравнению с OMP/TBB, использующими внутренний пул потоков.

## 9. Выводы
Ручная реализация с помощью `std::thread` показала, что алгоритм Грэхема можно успешно распараллелить, достигнув пикового ускорения в 3 раза.

Однако эта версия значительно уступает в производительности и простоте реализации версиям на OMP и TBB. Основные причины:
1. **Высокие накладные расходы**: Постоянное создание и уничтожение потоков для каждого этапа алгоритма является главным узким местом, ограничивающим масштабируемость.
2. **Сложность кода**: Ручное управление разделением данных, жизненным циклом потоков и синхронизацией требует значительно больше кода и повышает вероятность ошибок.

Для данной задачи, где используются стандартные паттерны параллелизма (map-reduce для поиска опоры, параллельная сортировка), использование высокоуровневых библиотек, таких как TBB, является более предпочтительным. Они скрывают сложность управления потоками и обеспечивают лучшую производительность за счёт использования оптимизированных пулов потоков и планировщиков.
