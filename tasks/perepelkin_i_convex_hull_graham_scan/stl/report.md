## 1. Контекст

Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости
с использованием алгоритма «Проход Грэхема».

В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших
наборов данных целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным.
Это связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом
шаге зависит от результата предыдущего шага.

В отличие от OpenMP и TBB, которые предоставляют высокоуровневые абстракции, версия на `std::thread`
(или `std::jthread`) требует ручного управления потоками. Это позволяет получить полный контроль над созданием,
синхронизацией и жизненным циклом потоков, но также переносит на разработчика ответственность за корректное разделение
данных, предотвращение гонок и минимизацию накладных расходов.

Данная реализация используется для демонстрации низкоуровневого подхода к распараллеливанию
и анализа эффективности в сравнении с более высокоуровневыми решениями.

## 2. Постановка задачи

Полная постановка задачи, описание входных/выходных данных, ограничений и крайних случаев приведены
в отчёте по последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- **Входные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий координаты точек на плоскости.
- **Выходные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий точки,
  образующие выпуклую оболочку, упорядоченные против часовой стрелки.

STL-версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом достигая
ускорения за счёт использования многоядерной архитектуры.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в
[`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:

1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания

Распараллеливанию подверглись первые два этапа алгоритма:

1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.1. Разделение данных

```cpp
// File: stl/src/ops_stl.cpp
size_t base = total_size / threads;
size_t rem = total_size % threads;

size_t offset = 0;
for (int i = 0; i < threads; i++) {
  start[i] = static_cast<int>(offset);

  size_t extra = std::cmp_less(i, rem) ? 1 : 0;
  offset += base + extra;
}

start[threads] = static_cast<int>(total_size);
```

Разделение данных на `threads` потоков выполняется по принципу блочного распределения. Весь массив точек делится
на `threads` непрерывных, непересекающихся блоков. Для равномерного распределения нагрузки размеры блоков
делаются максимально близкими.

Если общее число точек `total_size` не делится нацело на `threads`, то первые `total_size % threads` блоков получают
по одному дополнительному элементу. Таким образом, каждый поток получает для обработки свой уникальный
диапазон индексов.

### 4.2. Поиск опорной точки

```cpp
// File: stl/src/ops_stl.cpp
std::vector<size_t> local_idx(threads, 0);
{
  std::vector<std::jthread> workers(threads);

  for (int tid = 0; tid < threads; tid++) {
    workers.emplace_back([&, tid]() {
      // Поиск локальной опорной точки
      local_idx[tid] = local;
    });
  }
}
// Последовательный поиск глобального минимума по local_idx
```

1. **Локальный поиск**: Каждый поток ищет локальную опорную точку в своём блоке данных.
2. **Хранение локальных результатов**: Индекс найденной локальной опорной точки каждый поток записывает в общий
   массив `local_idx` в ячейку, соответствующую своему `tid`. Это исключает гонку данных, так как каждый поток
   пишет в уникальную область памяти.
3. **Синхронизация**: После завершения работы всех потоков (неявный барьер) главный поток последовательно обходит
   массив `local_idx` и находит глобальную опорную точку среди локальных кандидатов.

### 4.3. Параллельная сортировка

Реализована параллельная сортировка слиянием.

```cpp
// File: stl/src/ops_stl.cpp
// Локальная сортировка
{
  std::vector<std::jthread> workers(threads);

  for (int tid = 0; tid < threads; tid++) {
    workers.emplace_back([&, tid]() {
      std::sort(...);
    });
  }
}

// Слияние отсортированных сегментов
for (int size = 1; size < threads; size *= 2) {
  std::vector<std::jthread> workers(threads);

  for (int i = 0; i < threads; i += 2 * size) {
    if (i + size >= threads) {
      continue;
    }

    workers.emplace_back([&, i, size]() {
      // ...
      std::inplace_merge(...);
    });
  }
}
```

1. **Локальная сортировка**: Каждый поток сортирует свой блок данных с помощью `std::sort`.
2. **Слияние**: Выполняется итеративное попарное слияние отсортированных блоков по принципу дерева. На каждом уровне
   слияния (`size = 1, 2, 4, ...`) запускается новый набор потоков, каждый из которых выполняет `std::inplace_merge`
   для своей пары блоков.

### 4.4. Управление потоками и синхронизация

- **Жизненный цикл**: Для каждого параллельного этапа (поиск опорной точки, локальная сортировка, каждый уровень
  слияния) создаётся свой набор потоков `std::jthread`.
- **Синхронизация**: Явные примитивы синхронизации (`mutex`, `atomic`) не используются. Синхронизация достигается
  за счёт барьеров, которые реализуются с помощью RAII-семантики `std::jthread`. Потоки для одного этапа создаются,
  выполняют свою работу, а в конце области видимости их деструкторы автоматически вызывают `join()`, что останавливает
  главный поток до завершения всех рабочих потоков. Это гарантирует, что, например, все локальные сортировки
  завершатся до начала первого этапа слияния.

Такой подход `create → work → join` для каждого этапа приводит к значительным накладным расходам по сравнению
с использованием постоянного пула потоков.

## 5. Детали реализации

### Файлы

- `stl/include/ops_stl.hpp` – объявление класса `PerepelkinIConvexHullGrahamScanSTL`.
- `stl/src/ops_stl.cpp` – реализация методов класса.

### Карта "поток -> диапазон"

Разделение данных между потоками выполняется по принципу блочного распределения. Начальные и конечные индексы
элементов для каждого потока рассчитываются перед началом параллельных вычислений.

Для этого общее количество точек делится на число потоков, определяя базовый размер блока. Остаток от деления
распределяется по одному элементу между первыми потоками. Начало диапазона для каждого потока сохраняется
в векторе `start` размером `threads + 1`.

Каждый рабочий поток с уникальным идентификатором `tid` считывает свой рабочий диапазон
как `[start[tid], start[tid + 1])`. Такой подход гарантирует:

- **Непересекаемость**: каждый поток работает с собственной выделенной непрерывной областью памяти, что полностью
  исключает возможность конфликтов доступа к данным без применения блокировок.
- **Равномерность**: размер задач для разных потоков отличается максимум на один элемент, что обеспечивает
  сбалансированную загрузку всех ядер.

### Схема объединения локальных результатов

- **Поиск опорной точки**: каждый рабочий поток находит индекс локальной опорной точки на своем диапазоне и записывает
  его в соответствующую ячейку общего вектора `local_idx` по своему индексу `tid`. После завершения работы потоков
  и прохождения неявного барьера синхронизации главный поток в последовательном цикле обходит вектор `local_idx`
  и находит среди локальных кандидатов глобальную опорную точку.
- **Сортировка по полярному углу (дерево слияния)**: объединение отсортированных частей массива в единый упорядоченный
  вектор выполняется попарно по схеме восходящего двоичного дерева слияния. На каждом уровне слияния с шагом `size`
  рабочие потоки объединяют соседние пары сегментов с использованием стандартного метода `std::inplace_merge`.
  Синхронизация между шагами слияния обеспечивается за счет RAII-семантики вектора потоков `std::jthread`.

## 6. Проверка корректности

Корректность работы STL-реализации проверяется путем прямого сравнения её выходного вектора с результатом, полученным
от SEQ-версии. Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный результат.

### Набор тестов и характерные случаи

Используется тот же набор валидационных и функциональных тестов, что и для SEQ-версии (подробное описание приведено
в [seq/report.md](../seq/report.md#5-проверка-корректности)). Тесты покрывают все основные сценарии и граничные случаи:

- Валидация на пустом входном векторе.
- Вырожденные случаи (1 или 2 точки на входе).
- Наборы с коллинеарными точками.
- Наборы с повторяющимися и полностью совпадающими точками.

### Дополнительные проверки

- При `PPC_NUM_THREADS=1` результат работы STL-версии полностью идентичен SEQ-версии.
- При различных значениях `PPC_NUM_THREADS` (2, 4, 6, ...) выходные векторы по-прежнему совпадают между собой
  и с эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

Успешное прохождение всех тестов показывает, что STL-версия соблюдает инварианты базового алгоритма.

### Анализ масштабируемости

Анализ выполнялся на основе сценария `task_run`.

- **Ускорение**: Стабильный рост наблюдается вплоть до 8 потоков (3.014x). Дальнейшее увеличение числа рабочих потоков
  за пределы количества физических ядер процессора (6) приводит к прекращению роста производительности и деградации
  времени выполнения.
- **Эффективность**: Монотонно снижается с 76.22% на 2 потоках до 11.37% на 24 потоках из-за высокой
  доли последовательной части алгоритма и накладных расходов.
- **Накладные расходы**: Снижение скорости вычислений при большом числе потоков связано с ручным созданием и уничтожением
  потоков (`create/join`) для каждого этапа (поиск опоры, локальная сортировка, уровни слияния). Отсутствие
  переиспользования потоков через готовый пул значительно увеличивает издержки параллелизма по сравнению
  с версиями на OpenMP и TBB.

## 9. Выводы

Разработка параллельной версии алгоритма с использованием `std::jthread` позволила успешно решить задачу масштабирования
вычислений, обеспечив 3-кратное ускорение относительно последовательного решения (SEQ).

При сравнении с высокоуровневыми технологиями ручная реализация на стандартных потоках показала следующее:

- **По сравнению с oneTBB**: STL-версия значительно уступает как по пиковому ускорению (3.014x против 4.020x), так
  и по общей масштабируемости из-за отсутствия планировщика задач и высоких накладных расходов.
- **По сравнению с OpenMP**: STL-версия в целом уступает по уровню абстракции и производительности при превышении числа
  физических ядер процессора, однако на конфигурации с 8 потоками её пиковая производительность оказалась незначительно
  выше (ускорение STL составило 3.014x против 2.964x у OpenMP).

**Недостатки ручного управления потоками:**

1. **Высокие накладные расходы**: Постоянное создание и уничтожение потоков для каждого этапа алгоритма является
   узким местом, ограничивающим масштабируемость.
2. **Сложность кода**: Ручное управление разделением данных, жизненным циклом потоков и синхронизацией требует
   значительно больше кода и повышает вероятность ошибок.

Для данной задачи, где используются стандартные паттерны параллелизма (map-reduce для поиска опорной точки, параллельная
сортировка), использование высокоуровневых библиотек, таких как TBB, является более предпочтительным. Они скрывают
сложность управления потоками и обеспечивают лучшую производительность за счёт использования оптимизированных пулов
потоков и планировщиков.
