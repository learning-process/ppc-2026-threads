## 1. Контекст

Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости с использованием
алгоритма «Проход Грэхема».

В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших наборов
данных целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это
связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от
результата предыдущего шага.

Библиотека oneTBB выбрана для этой задачи, поскольку она предоставляет мощные высокоуровневые примитивы (такие как
`tbb::parallel_reduce` и `tbb::parallel_sort`). Это позволяет разработчику сфокусироваться на логике алгоритма,
делегируя низкоуровневое управление пулом потоков, балансировкой нагрузки и синхронизацией самой библиотеке.

## 2. Постановка задачи

Полная постановка задачи, описание входных/выходных данных, ограничений и крайних случаев приведены в отчёте по
последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- **Входные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий координаты точек на плоскости.
- **Выходные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий точки, образующие выпуклую
  оболочку, упорядоченные против часовой стрелки.

TBB-версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом достигая ускорения за счёт
использования многоядерной архитектуры.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в
[`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:

1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания

Распараллеливанию подверглись первые два этапа алгоритма:

1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.1 Примитивы и диапазоны

1. **Поиск опорной точки**: Используется примитив `tbb::parallel_reduce`. Диапазон работы задается с помощью объекта
   `tbb::blocked_range`. Внутри этого диапазона происходит поиск индекса точки с минимальной y-координатой (а при
   равенстве — с минимальной x). Класс `blocked_range` позволяет планировщику TBB рекурсивно дробить одномерный интервал
   на независимые подзадачи для параллельной обработки.

2. **Сортировка точек по полярному углу**: Используется высокоуровневый алгоритм `tbb::parallel_sort`. Диапазон задается
   стандартными итераторами. Алгоритм `tbb::parallel_sort` скрывает детали реализации и самостоятельно управляет
   разделением этих данных на отрезки для параллельной локальной сортировки с последующим слиянием.

### 4.2 Grainsize и Partitioner

- **Grainsize**: Явно не задается. При использовании `tbb::blocked_range` с двумя аргументами применяется размер зерна
  по умолчанию (равный 1). Библиотека TBB использует эвристики для автоматического объединения итераций в задачи
  оптимального размера.
- **Partitioner**: Явно не специфицирован, поэтому TBB применяет `auto_partitioner` по умолчанию. Он динамически
  регулирует разбиение диапазона, обеспечивая балансировку нагрузки между потоками с помощью механизма "work-stealing" и
  минимизируя накладные расходы на создание избыточного числа задач.

### 4.3 Ограничение конкуренции

Количество рабочих потоков строго контролируется объектом `tbb::global_control`, инициализируемым в начале метода
`RunImpl`:

```cpp
tbb::global_control gc(tbb::global_control::max_allowed_parallelism, ppc::util::GetNumThreads());
```

Это позволяет ограничивать максимальный уровень параллелизма в соответствии с переменной окружения, заданной тестовым
фреймворком.

## 5. Детали реализации

### Файлы

- `tbb/include/ops_tbb.hpp` – объявление класса `PerepelkinIConvexHullGrahamScanTBB`.
- `tbb/src/ops_tbb.cpp` – реализация методов класса.

### Поиск опорной точки

Для вычисления глобального минимума применяется `tbb::parallel_reduce`.

```cpp
// File: tbb/src/ops_tbb.cpp
size_t pivot = tbb::parallel_reduce(
    tbb::blocked_range<size_t>(1, pts.size()),
    static_cast<size_t>(0), // Начальное значение аккумулятора (локальный индекс)
    [&](const tbb::blocked_range<size_t> &r, size_t local_idx) {
      for (size_t i = r.begin(); i != r.end(); i++) {
        local_idx = better(local_idx, i); // Поиск локального минимума в блоке
      }
      return local_idx;
    },
    [&](size_t a, size_t b) {
      return better(a, b); // Объединение (редукция) результатов двух потоков
    }
);
```

**Локальные аккумуляторы**: Каждый поток инициализирует свой аккумулятор `local_idx` нулем и обновляет его при проходе
по своему блоку `r`. На этапе **объединения результатов** вызывается лямбда функция, возвращающая "лучший" индекс из
двух локальных результатов, найденных разными потоками. Гонки данных исключены архитектурой `parallel_reduce`.

### Сортировка точек по полярному углу

Используется встроенный `tbb::parallel_sort`.

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::parallel_sort(data.begin(), data.end(), [&](const auto &a, const auto &b) {
  return AngleCmp(a, b, pivot);
});
```

В отличие от OMP-версии, где параллельная сортировка слиянием реализовывалась вручную с неявными барьерами,
`tbb::parallel_sort` сам управляет разбиением массива и слиянием отсортированных частей (что значительно снижает объем
кода) и эффективнее работает с кэшем.

## 6. Проверка корректности

Корректность работы TBB-реализации проверяется путем прямого сравнения её выходного вектора с результатом, полученным от
SEQ-версии. Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный результат.

### Набор тестов и характерные случаи

Используется тот же набор валидационных и функциональных тестов, что и для SEQ-версии (подробное описание приведено в
[seq/report.md](../seq/report.md#5-проверка-корректности)). Тесты покрывают все основные сценарии и граничные случаи:

- Валидация на пустом входном векторе.
- Вырожденные случаи (1 или 2 точки на входе).
- Наборы с коллинеарными точками.
- Наборы с повторяющимися и полностью совпадающими точками.

### Дополнительные проверки

- При `PPC_NUM_THREADS=1` результат работы TBB-версии полностью идентичен SEQ-версии.
- При различных значениях `PPC_NUM_THREADS` (2, 4, 6, ...) выходные векторы по-прежнему совпадают между собой и с
  эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

Успешное прохождение всех тестов показывает, что TBB-версия соблюдает инварианты базового алгоритма.

- **Метрики:**
  - Среднее время выполнения по серии из 100 запусков.
  - Ускорение относительно последовательной версии: `speedup = time_seq / time_par`.

- **Сценарии измерения:**

### Анализ масштабируемости

Анализ выполнялся на основе сценария `task_run`.

- **Ускорение**: Стабильный рост наблюдается вплоть до 8 потоков (3.853x). Пиковое ускорение составляет 4.02x на 24
  потоках без деградации времени выполнения при избытке рабочих потоков.
- **Эффективность**: Монотонно снижается с 89.33% на 2 потоках до 16.75% на 24 потоках из-за фиксированного времени
  последовательной части алгоритма.
- **Балансировка (balancing)**: Планировщик задач TBB (с механизмом work-stealing) отлично балансирует нагрузку без
  необходимости ручного подбора параметров, динамически перераспределяя отрезки массива при сортировке.
- **Накладные расходы (overhead)**: При превышении числа задейстованных физических ядер накладные расходы на управление
  пулом задач TBB и переключение контекста ОС начинают оказывать большее влияние, снижая общую эффективность.

По сравнению с OMP, реализация на TBB достигла большего пикового ускорения (4.02x против 2.964x) за счёт
высокооптимизированного `tbb::parallel_sort` и отсутствия избыточных неявных барьеров синхронизации.

## 9. Выводы

Разработка параллельной версии алгоритма с использованием библиотеки oneTBB позволила успешно решить задачу
масштабирования вычислений, обеспечив 4-кратное ускорение относительно последовательного решения (SEQ). Библиотека TBB
продемонстрировала высокую эффективность, превзойдя подход OpenMP как по уровню абстракции и простоте кода, так и по
итоговой производительности на многоядерной системе.

**Случаи, когда TBB удобнее:**

- Для параллелизации стандартных вычислительных шаблонов: высокоуровневые методы (такие как `tbb::parallel_sort` и
  `tbb::parallel_reduce`) позволяют существенно сократить объем кода по сравнению с ручной реализацией аналогичных
  операций.
- При необходимости автоматической балансировки нагрузки: планировщик TBB эффективно распределяет вычислительные задачи
  между потоками, освобождая разработчика от ручного подбора параметров распределения.

**Ограничения и накладные расходы:**

- На малых объемах входных данных издержки на создание задач и управление планировщиком TBB превышают полезное время
  вычислений.
- Наличие строго последовательной части (финальное построение оболочки) жестко ограничивает максимальное теоретическое
  ускорение.
