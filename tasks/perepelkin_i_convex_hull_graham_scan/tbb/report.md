## 1. Контекст
В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших наборов точек целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от результата предыдущего шага.

Библиотека Intel TBB выбрана для этой задачи, поскольку она предоставляет мощные высокоуровневые примитивы (такие как `tbb::parallel_reduce` и `tbb::parallel_sort`). Это позволяет разработчику сфокусироваться на логике алгоритма, делегируя низкоуровневое управление пулом потоков, балансировкой нагрузки (work-stealing) и синхронизацией самой библиотеке.

## 2. Постановка задачи
Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости. Входными данными является вектор пар координат, выходными — вектор точек, образующих выпуклую оболочку и упорядоченных против часовой стрелки.

Полная постановка задачи, описание входных/выходных данных и крайних случаев приведены в отчёте по последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

TBB-версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом достигая ускорения за счёт использования многоядерной архитектуры.

## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в [`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:
1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания
Распараллеливанию подверглись первые два этапа алгоритма:
1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.1 Примитивы и диапазоны
1. **Поиск опорной точки**: Используется примитив `tbb::parallel_reduce`. Диапазон работы задается с помощью объекта `tbb::blocked_range`. Внутри этого диапазона происходит поиск индекса точки с минимальной y-координатой (а при равенстве — с минимальной x). Класс `blocked_range` позволяет планировщику TBB рекурсивно дробить одномерный интервал на независимые подзадачи для параллельной обработки.

2. **Сортировка точек по полярному углу**: Используется высокоуровневый алгоритм `tbb::parallel_sort`. Диапазон задается стандартными итераторами. Алгоритм `tbb::parallel_sort` скрывает детали реализации и самостоятельно управляет разделением этих данных на отрезки для параллельной локальной сортировки с последующим слиянием.

### 4.2 Grainsize и Partitioner
- **Grainsize**: Явно не задается. При использовании `tbb::blocked_range` с двумя аргументами применяется размер зерна по умолчанию (равный 1). Библиотека TBB использует эвристики для автоматического объединения итераций в задачи оптимального размера.
- **Partitioner**: Явно не специфицирован, поэтому TBB применяет `auto_partitioner` по умолчанию. Он динамически регулирует разбиение диапазона, обеспечивая балансировку нагрузки между потоками с помощью механизма "work-stealing" и минимизируя накладные расходы на создание избыточного числа задач.

### 4.3 Ограничение конкуренции
Количество рабочих потоков строго контролируется объектом `tbb::global_control`, инициализируемым в начале метода `RunImpl`:
```cpp
tbb::global_control gc(tbb::global_control::max_allowed_parallelism, ppc::util::GetNumThreads());
```
Это позволяет ограничивать максимальный уровень параллелизма в соответствии с переменной окружения, заданной тестовым фреймворком.

## 5. Детали реализации
### Файлы
- `tbb/include/ops_tbb.hpp` – объявление класса `PerepelkinIConvexHullGrahamScanTBB`.
- `tbb/src/ops_tbb.cpp` – реализация  методов класса.

### Поиск опорной точки
Для вычисления глобального минимума применяется `tbb::parallel_reduce`.

```cpp
size_t pivot = tbb::parallel_reduce(
    tbb::blocked_range<size_t>(1, pts.size()), 
    static_cast<size_t>(0), // Начальное значение аккумулятора (локальный индекс)
    [&](const tbb::blocked_range<size_t> &r, size_t local_idx) {
      for (size_t i = r.begin(); i != r.end(); i++) {
        local_idx = better(local_idx, i); // Поиск локального минимума в блоке
      }
      return local_idx;
    }, 
    [&](size_t a, size_t b) { 
      return better(a, b); // Объединение (редукция) результатов двух потоков
    }
);
```
**Локальные аккумуляторы**: каждый поток инициализирует свой аккумулятор `local_idx` нулем и обновляет его при проходе по своему блоку `r`. На этапе **объединения результатов** (четвертый аргумент) вызывается лямбда функция, возвращающая "лучший" индекс из двух локальных результатов, найденных разными потоками. Гонки данных исключены архитектурой `parallel_reduce`.

### Сортировка точек по полярному углу
Используется встроенный `tbb::parallel_sort`.
```cpp
tbb::parallel_sort(data.begin(), data.end(), [&](const auto &a, const auto &b) { 
  return AngleCmp(a, b, pivot); 
});
```
В отличие от OMP-версии, где параллельная сортировка слиянием реализовывалась вручную с неявными барьерами, `tbb::parallel_sort` сам управляет разбиением массива и слиянием отсортированных частей (что значительно снижает объем кода) и эффективнее работает с кэшем.

## 6. Проверка корректности
### Сравнение с SEQ
Корректность работы TBB-реализации проверяется путем прямого сравнения её выходного вектора с результатом, полученным от SEQ-версии. Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный результат.

Дополнительные гарантии:
- При `PPC_NUM_THREADS=1` результат работы TBB-версии полностью идентичен SEQ-версии.
- При различных значениях `PPC_NUM_THREADS` (2, 4, 6, ...) выходные векторы по-прежнему совпадают между собой и с эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

### Набор тестов и характерные случаи
Используется тот же набор валидационных и функциональных тестов, что и для SEQ-версии (подробное описание приведено в [seq/report.md](../seq/report.md#5-проверка-корректности)). Тесты покрывают все основные сценарии и граничные случаи:
- Валидация на пустом входном векторе.
- Вырожденные случаи (1 или 2 точки на входе).
- Наборы с коллинеарными точками.
- Наборы с повторяющимися и полностью совпадающими точками.

Успешное прохождение всех тестов доказывает, что TBB-версия строго соблюдает инварианты базового алгоритма.

### Анализ масштабируемости
Анализ выполнялся на основе сценария `task_run`.
- **Ускорение**: Стабильный рост наблюдается вплоть до 6 потоков (~3.3x). Пиковое ускорение составляет ~4.02x на 24 потоках без деградации времени выполнения при избытке рабочих потоков.
- **Эффективность**: Монотонно снижается с ~89.3% на 2 потоках до ~16.7% на 24 потоках из-за фиксированного времени последовательной части алгоритма.
- **Балансировка (balancing)**: Планировщик задач TBB (с механизмом work-stealing) отлично балансирует нагрузку без необходимости ручного подбора параметров, динамически перераспределяя отрезки массива при сортировке.
- **Накладные расходы (overhead)**: При превышении числа задейстованных физических ядер накладные расходы на управление пулом задач TBB и переключение контекста ОС начинают оказывать большее влияние, снижая общую эффективность.

По сравнению с OMP, реализация на TBB достигла большего ускорения (~4.02x против ~2.96x) за счёт высокооптимизированного `tbb::parallel_sort` и отсутствия избыточных неявных барьеров синхронизации.

## 9. Выводы
Использование Intel TBB позволило добиться 4-кратного ускорения работы алгоритма, что является отличным результатом и демонстрирует лучшую масштабируемость по сравнению с OMP-версией.

**Когда TBB удобнее:**
- Для задач, требующих стандартных паттернов: высокоуровневые алгоритмы (такие как `tbb::parallel_sort`, `tbb::parallel_reduce`) позволяют избежать написания объемного и сложного кода (например, ручного слияния массивов в OMP).
- Когда важна автоматическая балансировка: встроенный планировщик сам эффективно управляет пулом потоков и размером задач, избавляя от ручного подбора параметров распределения.

**Когда overhead мешает:**
- На небольших наборах данных, где накладные расходы на создание пула задач превышают время самой полезной работы.
- При наличии строгих зависимостей по данным (как на этапе построения оболочки), принципиально ограничивающих степень параллелизма.
- При упоре в пропускную способность подсистемы памяти во время интенсивной сортировки миллионов элементов, что нивелирует пользу от добавления новых потоков.
