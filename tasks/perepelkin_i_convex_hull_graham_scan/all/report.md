## 1. Контекст

Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости с использованием
алгоритма «Проход Грэхема».

В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших наборов
данных целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это
связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от
результата предыдущего шага.

Гибридная версия (MPI + TBB) позволяет эффективно масштабировать вычисления: MPI отвечает за распределение данных и
межпроцессное взаимодействие между узлами кластера, а технология oneTBB обеспечивает многопоточную обработку внутри
каждого отдельного узла, максимально утилизируя доступные ядра.

## 2. Постановка задачи

Полная постановка задачи, описание входных/выходных данных, ограничений и крайних случаев приведены в отчёте по
последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- **Входные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий координаты точек на плоскости.
- **Выходные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий точки, образующие выпуклую
  оболочку, упорядоченные против часовой стрелки.

Гибридная версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом обеспечивая масштабируемость
как по числу распределённых MPI-процессов, так и по числу потоков TBB внутри каждого процесса.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в
[`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:

1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

В гибридной версии распараллеливаются первые два этапа, а третий — выполняется последовательно на главном процессе из-за
сильной зависимости по данным.

## 4. Межпроцессная схема

### 4.1. Роли процессов

**Главный процесс (rank 0):**

- Хранит исходный массив точек.
- Выполняет распределение данных между всеми процессами.
- После локальной обработки всеми процессами собирает промежуточные результаты (локальные опорные точки и
  отсортированные блоки).
- Выполняет построение выпуклой оболочки и рассылает финальный результат.

**Рабочие процессы (rank 1 ... size-1):**

- Получают свой уникальный блок данных.
- Находят в нем локальную опорную точку и отправляют её главному процессу.
- Сортируют свой блок относительно глобальной опорной точки.
- Отправляют отсортированную часть главному процессу.

### 4.2. Распределение данных

Исходный массив равномерно делится между процессами. Для идеального статического распределения исходный массив
дополняется фиктивными точками, чтобы на каждый процесс приходилось одинаковое количество элементов. Массив
распределяется с помощью вызова `MPI_Scatterv`.

### 4.3. Используемые MPI-вызовы

- `MPI_Bcast` – используется для рассылки:
  - исходного и выровненного размеров массива от главного процесса всем рабочим;
  - глобальной опорной точки;
  - финального результата.
- `MPI_Scatterv` – делит входной массив точек на блоки и отправляет их всем процессам.
- `MPI_Gather` и `MPI_Gatherv` – используются для сбора локальных опорных точек и отсортированных блоков на главном
  процессе.
- `MPI_Allreduce` – используется с операцией `MPI_MAX` для определения ранга процесса, которому принадлежит глобальная
  опорная точка (для её удаления из локального массива перед сортировкой).

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса вычисления распараллеливаются с помощью технологии oneTBB.

### 5.1. Поиск локальной опорной точки

Используется примитив `tbb::parallel_reduce` с диапазоном `tbb::blocked_range`. Планировщик автоматически разбивает
одномерный диапазон точек на поддиапазоны и распределяет их между доступными рабочими потоками, каждый из которых
независимо находит локальный минимум (по y-координате, а при равенстве — по x-координате). После завершения вычислений в
потоках полученные локальные минимумы объединяются операцией редукции.

### 5.2. Параллельная сортировка точек

Для сортировки локальных блоков используется встроенный алгоритм `tbb::parallel_sort`. Он самостоятельно управляет
разбиением массива и балансировкой нагрузки на потоки внутри отдельного процесса, обеспечивая эффективную сортировку
точек по полярному углу относительно глобальной опорной точки.

### 5.3. Слияние отсортированных блоков

После того как все отсортированные блоки собраны на главном процессе, их слияние выполняется по принципу дерева. Вызовы
слияния соседних блоков оборачиваются в `tbb::parallel_invoke`. Это позволяет загрузить многопоточностью главный процесс
на этапе агрегации промежуточных результатов.

### 6.2. Использование MPI-вызовов

Ниже приведены ключевые фрагменты кода с вызовами MPI-функций, их назначение и место в логике алгоритма.

#### MPI_Type_contiguous, MPI_Type_commit, MPI_Type_free

```cpp
// File: all/src/ops_all.cpp
MPI_Type_contiguous(2, MPI_DOUBLE, &MPI_POINT_);
MPI_Type_commit(&MPI_POINT_);
// ... использование MPI_POINT_ для пересылки точек ...
MPI_Type_free(&MPI_POINT_);
```

Создаёт непрерывный пользовательский тип `MPI_POINT_` из двух элементов `MPI_DOUBLE`. Это необходимо для удобной
пересылки объектов `std::pair<double, double>` стандартными средствами MPI.

#### MPI_Bcast

```cpp
// File: all/src/ops_all.cpp
// Рассылка размеров
MPI_Bcast(&original_size, 1, MPI_UNSIGNED_LONG_LONG, 0, MPI_COMM_WORLD);
MPI_Bcast(&padded_size, 1, MPI_UNSIGNED_LONG_LONG, 0, MPI_COMM_WORLD);

// Рассылка глобальной опорной точки
MPI_Bcast(&global_pivot, 1, MPI_POINT_, 0, MPI_COMM_WORLD);

// Рассылка финального результата
MPI_Bcast(&size, 1, MPI_UNSIGNED_LONG_LONG, 0, MPI_COMM_WORLD);
if (size > 0) {
  MPI_Bcast(GetOutput().data(), static_cast<int>(size), MPI_POINT_, 0, MPI_COMM_WORLD);
}
```

Применяется на нескольких этапах для широковещательной рассылки данных от главного процесса всем остальным:

- В начале работы рассылается исходный и выровненный размеры массива точек.
- В середине алгоритма рассылается найденная глобальная опорная точка для независимой сортировки блоков.
- В конце работы рассылается финальный размер и массив точек построенной выпуклой оболочки.

#### MPI_Scatterv

```cpp
// File: all/src/ops_all.cpp
MPI_Scatterv(padded_input.data(), counts.data(), displs.data(), MPI_POINT_, local_data.data(), local_size, MPI_POINT_, 0, MPI_COMM_WORLD);
```

Распределяет исходный массив точек на равные блоки между всеми процессами с учётом смещений.

#### MPI_Gather, MPI_Gatherv

```cpp
// File: all/src/ops_all.cpp
// Сбор локальных опорных точек
MPI_Gather(&local_pivot, 1, MPI_POINT_, pivots.data(), 1, MPI_POINT_, 0, MPI_COMM_WORLD);

// Сбор актуальных размеров отсортированных блоков
MPI_Gather(&local_size, 1, MPI_INT, real_counts.data(), 1, MPI_INT, 0, MPI_COMM_WORLD);

// Сбор отсортированных блоков
MPI_Gatherv(local_data.data(), local_size, MPI_POINT_, gathered.data(), real_counts.data(), real_displs.data(), MPI_POINT_, 0, MPI_COMM_WORLD);
```

Операции сбора данных используются для агрегации информации на главном процессе:

- Сначала собираются локальные опорные точки со всех процессов для выбора из них единой глобальной.
- Затем собираются актуальные размеры локальных блоков (размер мог измениться у одного из процессов после удаления
  глобальной опорной точки).
- Наконец, собираются отсортированные локальные блоки в единый буфер для последующего параллельного слияния.

#### MPI_Allreduce

```cpp
// File: all/src/ops_all.cpp
MPI_Allreduce(&owner_rank, &pivot_rank, 1, MPI_INT, MPI_MAX, MPI_COMM_WORLD);
```

Глобальная редукция используется для поиска процесса, которому принадлежит глобальная опорная точка. Найденный
процесс-владелец удаляет её из своего локального массива, чтобы избежать появления дубликатов при последующем слиянии
блоков.

### 6.3. Потенциально узкие места

1. **Передача данных по сети**: Использование `MPI_Scatterv` и `MPI_Gatherv` может стать узким местом на больших объёмах
   данных, так как передаются все элементы массива точек.
2. **Слияние на одном процессе**: Слияние отсортированных блоков выполняется только на нулевом процессе. Хотя оно
   реализовано с использованием потоков TBB, остальные процессы простаивают на этом этапе.
3. **Последовательное построение оболочки**: Финальный проход выполняется строго последовательно на нулевом процессе
   из-за сильной зависимости по данным, что ограничивает максимальное теоретическое ускорение.

## 7. Проверка корректности

Корректность работы гибридной реализации проверяется путем прямого сравнения её выходного вектора с результатом,
полученным от SEQ-версии. Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный
результат.

### Набор тестов и характерные случаи

Используется тот же набор валидационных и функциональных тестов, что и для SEQ-версии (подробное описание приведено в
[seq/report.md](../seq/report.md#5-проверка-корректности)). Тесты покрывают все основные сценарии и граничные случаи:

- Валидация на пустом входном векторе.
- Вырожденные случаи (1 или 2 точки на входе).
- Наборы с коллинеарными точками.
- Наборы с повторяющимися и полностью совпадающими точками.

### Дополнительные проверки

- При `PPC_NUM_THREADS=1` результат работы гибридной версии полностью идентичен SEQ-версии.
- При различных конфигурациях `PPC_NUM_PROC` и `PPC_NUM_THREADS` выходные векторы стабильно совпадают между собой и с
  эталоном, что подтверждает корректность межпроцессных MPI-обменов и отсутствие гонок данных на уровне потоков TBB.

Успешное прохождение всех тестов показывает, что гибридная версия соблюдает инварианты базового алгоритма.

### Анализ масштабируемости

Анализ выполнялся на основе сценария `task_run`.

- **Ускорение**: Пиковое ускорение `~2.86x` (время `~0.096 с`) достигается при использовании конфигураций: `1x6`, `2x6`
  и `4x6`. Дальнейшее увеличение суммарного числа задействованных потоков и процессов не приводит к соразмерному росту
  скорости. Напротив, наблюдается стагнация производительности, обусловленная исчерпанием доступных аппаратных ресурсов
  (физических ядер) и нарастающими издержками на пересылки и синхронизацию.
- **Эффективность**: Выше всего эффективность при малом числе рабочих потоков (104.6% для `1x2`, 72.2% для `2x1`), после
  чего она монотонно падает. Существенная деградация (ниже 20%) наблюдается при попытках использовать более 12 рабочих
  единиц, что обусловлено аппаратными ограничениями: процессор располагает лишь 12 логическими ядрами, и дальнейшее
  увеличение числа рабочих потоков приводит к избыточной конкуренции за ресурсы.
- **Накладные расходы**: Конфигурация с одним MPI-процессом (`1xN`) неизменно работает на уровне или быстрее
  многопроцессных аналогов с тем же общим числом рабочих единиц. Масштабирование по MPI-процессам сильно ограничивается
  накладными расходами гибридной архитектуры, требующей постоянного взаимодействия процессов. Значительное время уходит
  как на пересылку больших объёмов данных (`MPI_Scatterv`, `MPI_Gatherv`), так и на неявную барьерную синхронизацию при
  совместном поиске и рассылке глобальной опорной точки (`MPI_Allreduce`, `MPI_Bcast`).
- **Дисбаланс нагрузки**: Дополнительным фактором снижения производительности многопроцессных конфигураций является
  финальное слияние отсортированных блоков. Оно выполняется исключительно силами нулевого процесса, в то время как все
  остальные MPI-процессы простаивают, ожидая завершения работы мастера.

## 10. Выводы

Гибридная схема (MPI + TBB) в полной мере оправдана только в случаях распределенных вычислений на нескольких узлах
кластера, когда объем данных слишком велик для размещения в оперативной памяти одного узла, и когда вычислительная
сложность задачи значительно превышает стоимость пересылки данных.

Для вычислений на одном физическом узле (как в случае текущей тестовой среды) использование MPI вносит избыточные
накладные расходы на копирование и передачу данных между процессами, а также на слияние результатов. В таких условиях
более предпочтительным является использование чистых потоковых технологий с разделяемой памятью (например, TBB), что
наглядно подтверждается результатами экспериментов: наивысшая производительность в комбинации с оптимальной
эффективностью масштабирования достигается при запуске одного MPI-процесса с использованием всех доступных ядер
процессора.
