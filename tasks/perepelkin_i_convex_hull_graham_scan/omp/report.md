## 1. Контекст
В последовательной версии (SEQ) все вычисления выполняются в одном потоке на одном ядре. Для ускорения обработки больших наборов точек целесообразно распараллелить два основных этапа:
- Поиск опорной точки.
- Сортировка точек по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от результата предыдущего шага.

Технология OpenMP выбрана для распараллеливания, так как она позволяет с помощью простых директив эффективно организовать параллельное выполнение циклов на общей памяти, что хорошо подходит для данной задачи.

## 2. Постановка задачи
Решается задача построения выпуклой оболочки для заданного множества точек на плоскости. Входными данными является вектор пар координат, выходными — вектор точек, образующих выпуклую оболочку.

Полная постановка задачи, описание входных/выходных данных и крайних случаев приведены в отчёте по последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

OMP-версия должна давать результат, идентичный SEQ-версии, и обеспечивать ускорение за счёт параллельной обработки данных.

## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание приведено в [`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и включает 3 основных этапа:
1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания
### 4.1 Области кода
Распараллеливанию подверглись следующие части алгоритма:
1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.2 Переменные и синхронизация
#### 4.2.1 Поиск опорной точки
```cpp
#pragma omp parallel default(none) shared(pts, local_idx) num_threads(threads)
{
  int tid = omp_get_thread_num();
  size_t local = 0;

#pragma omp for nowait
  for (size_t i = 1; i < pts.size(); i++) {
    // ... find local pivot ...
  }
  local_idx[tid] = local;
}
// Sequential reduction
```
- **Схема**: Каждый поток находит индекс локальной опорной точки в своей части данных, после чего главный поток находит глобальную опорную точку среди локальных результатов, тем самым выполняя последовательную редукцию.
- **Параметры директивы `parallel`**:
  - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной области, что помогает избежать случайных гонок данных.
  - `shared(pts, local_idx)`: `pts` (входные точки) используется всеми потоками только для чтения. `local_idx` — общий массив для записи локальных результатов; каждый поток пишет в свою уникальную ячейку по индексу `tid`, что исключает гонку данных.
  - `private`: `tid`, `local` и итератор цикла `i` явно объявлены внутри блока и являются приватными для каждого потока.
  - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
- **Синхронизация**: Вместо `reduction` используется ручное сведение результатов. После параллельной секции следует неявный барьер, гарантирующий, что все потоки записали свои локальные индексы в `local_idx` перед началом последовательного сведения.

#### 4.2.2 Параллельная сортировка
Реализована параллельная сортировка слиянием.
1.  **Локальная сортировка**:
    ```cpp
    #pragma omp parallel default(none) shared(data, start, pivot) num_threads(threads)
    {
      int tid = omp_get_thread_num();
      std::sort(data.begin() + start[tid], data.begin() + start[tid + 1], ...);
    }
    ```
    - **Схема**: Массив точек делится на части. Каждый поток параллельно сортирует свой блок данных (границы заданы в массиве `start`) по полярному углу относительно опорной точки `pivot`.
    - **Параметры директивы `parallel`**:
      - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной области, что помогает избежать случайных гонок данных.
      - `shared(data, start, pivot)`: `data` — общий вектор точек, в котором каждый поток обрабатывает свой блок данных. `start` и `pivot` используются всеми потоками только для чтения.
      - `private`: `tid` явно объявлен внутри блока и является приватным для каждого потока.
      - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
    - **Синхронизация**: Гонки данных отсутствуют, так как каждый поток работает строго со своим изолированным блоком данных. Неявный барьер на выходе из параллельной области гарантирует завершение локальной сортировки всех блоков перед началом их слияния.

2.  **Слияние отсортированных сегментов**:
    ```cpp
    for (int size = 1; size < threads; size *= 2) {
    #pragma omp parallel for default(none) shared(data, start, threads, size, pivot) num_threads(threads)
      for (int i = 0; i < threads; i += 2 * size) {
        if (i + size >= threads) {
          continue;
        }

        int left = start[i];
        int mid = start[i + size];
        int right = start[std::min(i + (2 * size), threads)];

        std::inplace_merge(...);
      }
    }
    ```
    - **Схема**: Итеративное попарное слияние блоков по принципу дерева. Внешний цикл управляет шагом слияния и выполняется последовательно на главном потоке, а внутренний цикл распределяет пары блоков данных для слияния между потоками.
    - **Параметры директивы `parallel for`**:
      - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной области, что помогает избежать случайных гонок данных.
      - `shared(data, start, threads, size, pivot)`: `data` — общий вектор для выполнения `std::inplace_merge` над соответствующими парами блоков данных. Переменные структуры дерева `threads`, `size`, инвариант `pivot` и границы разбиения `start` используются всеми потоками только для чтения.
      - `private`: итератор цикла `i`, а также локальные переменные границ слияния `left`, `mid` и `right` явно объявлены внутри блока и являются приватными для каждого потока.
      - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
    - **Синхронизация**: на каждом уровне дерева слияния потоки обрабатывают разные пары блоков данных, что исключает коллизии памяти. Неявный барьер в конце директивы `parallel for` гарантирует полное завершение текущего уровня слияния всеми потоками перед тем, как внешний последовательный цикл увеличит шаг `size`.

### 4.3 Выбор `schedule`
Во всех параллельных циклах `schedule` явно не указан, что означает использование `static` по умолчанию в большинстве реализаций OpenMP. При статическом распределении итерации делятся на непрерывные блоки примерно одинакового размера, каждый поток получает один такой блок.

Это оптимальный выбор, так как:
- **Поиск опорной точки и локальная сортировка**: нагрузка на потоки распределяется равномерно, так как объемы данных для обработки примерно одинаковы.
- **Слияние**: количество пар для слияния на каждом шаге также распределяется равномерно.

Статическое расписание минимизирует накладные расходы на распределение итераций.

### Устранённые риски гонок
- **При поиске опорной точки**: гонка за право записи в глобальную переменную-кандидат устранена путем использования локальных переменных в каждом потоке и записи результатов в выделенные ячейки общего массива `local_idx`.
- **При сортировке**: на этапе локальной сортировки потоки работают с непересекающимися участками памяти. На этапе слияния `parallel for` также распределяет пары блоков для слияния между потоками.

## 6. Проверка корректности
### Сравнение с SEQ
Корректность работы OMP-реализации проверяется путем прямого сравнения её выходного вектора с результатом, полученным от последовательной версии (SEQ). Такое сравнение возможно, поскольку обе версии выдают одинаково (канонически) упорядоченный результат.

Дополнительные гарантии:
- При `PPC_NUM_THREADS=1` результат работы OMP-версии полностью идентичен SEQ-версии.
- При различных значениях `PPC_NUM_THREADS` (2, 4, 6, ...) выходные векторы по-прежнему совпадают между собой и с эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

### Набор тестов и характерные случаи
Используется тот же набор валидационных и функциональных тестов, что и для SEQ-версии (подробное описание приведено в [seq/report.md](../seq/report.md#5-проверка-корректности)). Тесты покрывают все основные сценарии и граничные случаи:

### Анализ масштабируемости
Анализ выполнялся на основе сценария `task_run`.
- **Ускорение**: Стабильный рост производительности наблюдается вплоть до 6 потоков, что обусловлено количеством физических ядер тестового процессора. Пиковое ускорение 2.964x достигается на 8 потоках, где полезный вклад вносит технология Hyper-Threading. При достижении аппаратного предела логических ядер (12 потоков) прирост останавливается из-за конкуренции за ресурсы, а избыточное выделение потоков (24) приводит к деградации (ускорение снижается до 2.544x) из-за высоких накладных расходов.
- **Эффективность**: Эффективность монотонно снижается с ростом числа задействованных рабочих единиц. На 2 потоках она держится на высоком уровне (~81.6%), так как накладные расходы еще невелики, однако к 6 потокам падает до ~42.8%, а на 12 — до ~24.5%. Такое поведение связано с тем, что с каждым новым потоком доля последовательного кода и суммарное время простоя на барьерах синхронизации (при слиянии массивов) начинают оказывать всё большее непропорциональное влияние на общее время работы.

### Узкие места
1.  Наиболее значимым ограничением является наличие **последовательной части** — финального построения оболочки. Ее время выполнения не уменьшается с ростом числа потоков и ограничивает максимальное теоретическое ускорение.
2.  **Накладные расходы**: Параллельная сортировка слиянием требует многократной синхронизации (неявных барьеров) между потоками на каждом уровне слияния, что вносит существенные накладные расходы.
3.  **Доступ к памяти**: С ростом числа потоков увеличивается конкуренция за доступ к общей памяти и кэшам процессора, что также снижает эффективность.

## 9. Выводы
OMP-версия даёт существенный выигрыш по времени (ускорение до 2.964x на 8 потоках) по сравнению с последовательной версией. Однако при дальнейшем увеличении числа потоков наблюдается деградация производительности, так как возникает упор в память: пропускной способности оперативной памяти и кэша процессора становится недостаточно для эффективного параллельного обмена большими объемами данных при сортировке.

**Выигрыш достигается:**
- При использовании числа потоков, не превышающего количество логических и близкого к числу физических ядер (оптимально 6–8 потоков для тестовой системы).
- На больших наборах точек, где этап сортировки с асимптотической сложностью O(N logN) занимает подавляющую часть времени и эффективно перекрывает накладные расходы на работу с потоками.

**Ограничения подхода:**
- **Наличие последовательного участка**: финальный этап построения оболочки остаётся строго последовательным из-за сильной зависимости по данным, что формирует узкое место и жестко лимитирует максимальное теоретическое ускорение.
- **Упор в подсистему памяти**: интенсивная работа с массивами координат при слиянии ведёт к сильной конкуренции потоков за доступ к шине памяти и вызывает частые промахи в кэше.
- **Синхронизация**: алгоритм параллельной сортировки требует многократных неявных барьеров на каждом уровне объединения подмассивов, что влечёт дополнительные накладные расходы.
