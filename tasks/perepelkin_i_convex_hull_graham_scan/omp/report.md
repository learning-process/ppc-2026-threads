## 1. Контекст

Решается задача построения выпуклой оболочки для заданного множества точек на двумерной плоскости с использованием
алгоритма «Проход Грэхема».

В последовательной версии (SEQ) все вычисления выполняются в одном потоке. Для ускорения обработки больших наборов
данных целесообразно распараллелить этапы поиска опорной точки и сортировки массива по полярному углу.

При этом финальный этап — построение выпуклой оболочки по отсортированным точкам — остаётся последовательным. Это
связано с сильной зависимостью по данным: решение о добавлении или удалении точки из оболочки на каждом шаге зависит от
результата предыдущего шага.

Технология OpenMP выбрана для распараллеливания, так как она позволяет с помощью простых директив эффективно
организовать параллельное выполнение циклов на общей памяти, что хорошо подходит для данной задачи.

## 2. Постановка задачи

Полная постановка задачи, описание входных/выходных данных, ограничений и крайних случаев приведены в отчёте по
последовательной версии: [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- **Входные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий координаты точек на плоскости.
- **Выходные данные:** `std::vector<std::pair<double, double>>` – вектор, содержащий точки, образующие выпуклую
  оболочку, упорядоченные против часовой стрелки.

OMP-версия должна обеспечивать результат, полностью идентичный SEQ-версии, при этом достигая ускорения за счёт
использования многоядерной архитектуры.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ-версией (подробное описание в
[`seq/report.md`](../seq/report.md#3-базовый-алгоритм)) и состоит из 3 основных этапов:

1. Поиск опорной точки.
2. Сортировка остальных точек по полярному углу относительно опорной.
3. Последовательное построение оболочки по отсортированным точкам.

## 4. Схема распараллеливания

### 4.1 Области кода

Распараллеливанию подверглись первые два этапа алгоритма:

1. **Поиск опорной точки**.
2. **Сортировка точек по полярному углу**.

### 4.2 Переменные и синхронизация

#### 4.2.1 Поиск опорной точки

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel default(none) shared(pts, local_idx) num_threads(threads)
{
  int tid = omp_get_thread_num();
  size_t local = 0;

#pragma omp for nowait
  for (size_t i = 1; i < pts.size(); i++) {
    // Поиск локальной опорной точки
  }
  local_idx[tid] = local;
}
```

- **Схема**: Каждый поток находит индекс локальной опорной точки в своей части данных, после чего главный поток находит
  глобальную опорную точку среди локальных результатов, тем самым выполняя последовательную редукцию.
- **Параметры директивы `parallel`**:
  - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной
      области, что помогает избежать случайных гонок данных.
  - `shared(pts, local_idx)`: `pts` (входные точки) используется всеми потоками только для чтения. `local_idx` — общий
      массив для записи локальных результатов; каждый поток пишет в свою уникальную ячейку по индексу `tid`, что
      исключает гонку данных.
  - `private`: `tid`, `local` и итератор цикла `i` явно объявлены внутри блока и являются приватными для каждого
      потока.
  - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
- **Параметры директивы `for`**:
  - `nowait`: отключает неявный барьер в конце цикла. Это позволяет потокам, завершившим свою часть итераций, сразу
      перейти к записи локального индекса в `local_idx`, не дожидаясь остальных.
- **Синхронизация**: Вместо `reduction` используется ручное сведение результатов. После параллельной секции следует
  неявный барьер, гарантирующий, что все потоки записали свои локальные индексы в `local_idx` перед началом
  последовательного сведения.

#### 4.2.2 Сортировка точек по полярному углу

Реализована параллельная сортировка слиянием.

1. **Локальная сортировка**:

    ```cpp
    // File: omp/src/ops_omp.cpp
    #pragma omp parallel default(none) shared(data, start, pivot) num_threads(threads)
    {
      int tid = omp_get_thread_num();
      std::sort(data.begin() + start[tid], data.begin() + start[tid + 1], ...);
    }
    ```

    - **Схема**: Массив точек делится на части. Каждый поток параллельно сортирует свой блок данных (границы заданы в
      массиве `start`) по полярному углу относительно опорной точки `pivot`.
    - **Параметры директивы `parallel`**:
        - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной
          области, что помогает избежать случайных гонок данных.
        - `shared(data, start, pivot)`: `data` — общий вектор точек, в котором каждый поток обрабатывает свой блок
          данных. `start` и `pivot` используются всеми потоками только для чтения.
        - `private`: `tid` явно объявлен внутри блока и является приватным для каждого потока.
        - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
    - **Синхронизация**: Гонки данных отсутствуют, так как каждый поток работает строго со своим изолированным блоком
      данных. Неявный барьер на выходе из параллельной области гарантирует завершение локальной сортировки всех блоков
      перед началом их слияния.

2. **Слияние отсортированных сегментов**:

    ```cpp
    // File: omp/src/ops_omp.cpp
    for (int size = 1; size < threads; size *= 2) {
    #pragma omp parallel for default(none) shared(data, start, threads, size, pivot) num_threads(threads)
      for (int i = 0; i < threads; i += 2 * size) {
        if (i + size >= threads) {
          continue;
        }

        int left = start[i];
        int mid = start[i + size];
        int right = start[std::min(i + (2 * size), threads)];

        std::inplace_merge(...);
      }
    }
    ```

    - **Схема**: Итеративное попарное слияние блоков по принципу дерева. Внешний цикл управляет шагом слияния и
      выполняется последовательно на главном потоке, а внутренний цикл распределяет пары блоков данных для слияния между
      потоками.
    - **Параметры директивы `parallel for`**:
        - `default(none)`: обязывает явно указывать область видимости для всех внешних переменных внутри параллельной
          области, что помогает избежать случайных гонок данных.
        - `shared(data, start, threads, size, pivot)`: `data` — общий вектор для выполнения `std::inplace_merge` над
          соответствующими парами блоков данных. Переменные структуры дерева `threads`, `size`, инвариант `pivot` и
          границы разбиения `start` используются всеми потоками только для чтения.
        - `private`: итератор цикла `i`, а также локальные переменные границ слияния `left`, `mid` и `right` явно
          объявлены внутри блока и являются приватными для каждого потока.
        - `num_threads(threads)`: явно задает количество параллельных потоков, создаваемых для выполнения данной секции.
    - **Синхронизация**: на каждом уровне дерева слияния потоки обрабатывают разные пары блоков данных, что исключает
      коллизии памяти. Неявный барьер в конце директивы `parallel for` гарантирует полное завершение текущего уровня
      слияния всеми потоками перед тем, как внешний последовательный цикл увеличит шаг `size`.

### 4.3 Выбор `schedule`

Во всех параллельных циклах `schedule` явно не указан, что означает использование `static` по умолчанию в большинстве
реализаций OpenMP. При статическом распределении итерации делятся на непрерывные блоки примерно одинакового размера,
каждый поток получает один такой блок.

Это оптимальный выбор, так как:

- **Поиск опорной точки и локальная сортировка**: нагрузка на потоки распределяется равномерно, так как объемы данных
  для обработки примерно одинаковы.
- **Слияние**: количество пар для слияния на каждом шаге также распределяется равномерно.

Статическое расписание минимизирует накладные расходы на распределение итераций.

### 4.4 Барьеры

- **Неявный барьер в конце блоков `omp parallel`**: гарантирует, что все потоки завершили свою работу (поиск локальной
  опорной точки или локальную сортировку) перед тем, как главный поток перейдет к последующим этапам (последовательной
  редукции или многопоточному слиянию).
- **Отключение неявного барьера (`nowait`)**: применено в цикле `omp for` при поиске опорной точки. Это устраняет
  ожидание завершения работы остальных потоков в конце цикла, позволяя сразу выполнить запись локального результата в
  массив `local_idx`.
- Явные барьеры (`#pragma omp barrier`) в коде не используются, так как корректная синхронизация обеспечивается неявными
  барьерами на границах параллельных областей.

- `omp/src/ops_omp.cpp` – реализация методов класса.

### Изменения относительно SEQ

| Этап                | SEQ-реализация             | OMP-реализация                                    |
| ------------------- | -------------------------- | ------------------------------------------------- |
| Поиск опорной точки | `std::ranges::min_element` | Параллельный поиск с последовательной редукцией   |
| Сортировка точек    | `std::ranges::sort`        | Параллельная сортировка слиянием                  |
| Построение оболочки | Последовательный цикл      | Последовательный цикл, аналогичный SEQ-реализации |

### Устранение рисков гонок

- **При поиске опорной точки**: гонка за право записи в глобальную переменную-кандидат устранена путем использования
  локальных переменных в каждом потоке и записи результатов в выделенные ячейки общего массива `local_idx`.
- **При сортировке**: на этапе локальной сортировки потоки работают с непересекающимися участками памяти. На этапе
  слияния `parallel for` также распределяет пары блоков для слияния между потоками.

  эталоном, что подтверждает отсутствие гонок данных и детерминизм параллельного алгоритма.

Успешное прохождение всех тестов показывает, что OMP-версия соблюдает инварианты базового алгоритма.

### Анализ масштабируемости

Анализ выполнялся на основе сценария `task_run`.

- **Динамика ускорения:** Стабильный рост производительности продолжается вплоть до 8 потоков. Именно на 8 потоках
  достигается пиковое ускорение в 2.964x, что обеспечивается совместной работой 6 физических ядер и использованием
  технологии Hyper-Threading.
- **Предел масштабируемости:** При дальнейшем увеличении числа потоков до 10–12 прирост производительности
  останавливается из-за исчерпания аппаратных ресурсов и возрастающей конкуренции за общую память, а также наличия
  строго последовательной части алгоритма.
- **Деградация:** Создание большего числа потоков, чем доступно логических ядер, ожидаемо приводит к деградации
  производительности. Ускорение снижается до 2.544x из-за высоких накладных расходов на переключение контекста ОС и
  синхронизацию.
- **Падение эффективности:** Показатель эффективности монотонно убывает с ростом числа рабочих потоков. На 2 потоках она
  остается на высоком уровне ~81.6%, на пиковых 8 потоках снижается до ~37.1%, а на 12 потоках падает до ~24.5%. \
  Столь резкое падение эффективности обусловлено присутствием в алгоритме существенной строго последовательной части
  (финальное построение оболочки). Чем больше потоков задействовано, тем сильнее эта последовательная часть и простои на
  барьерах синхронизации ограничивают общий прирост производительности.

### Узкие места

1. Наиболее значимым ограничением является наличие **последовательной части** — финального построения оболочки. Ее время
   выполнения не уменьшается с ростом числа потоков и ограничивает максимальное теоретическое ускорение.
2. **Накладные расходы:** параллельная сортировка слиянием требует многократной синхронизации (неявных барьеров) между
   потоками на каждом уровне слияния, что вносит существенные накладные расходы.
3. **Доступ к памяти:** с ростом числа потоков увеличивается конкуренция за доступ к общей памяти и кэшам процессора,
   что также снижает эффективность.

## 9. Выводы

OMP-версия даёт существенный выигрыш по времени (ускорение до 2.964x на 8 потоках) по сравнению с последовательной
версией. Однако при дальнейшем увеличении числа потоков масштабируемость ограничивается и наблюдается деградация
производительности. Это обусловлено двумя факторами:

- Алгоритм содержит существенную строго последовательную часть (финальное построение оболочки), что ограничивает
  максимальное ускорение.
- Возникает упор в подсистему памяти — основное время уходит на перемещение данных между оперативной памятью и кэшем
  процессора, вместо фактических вычислений.

**Выигрыш использования OpenMP достигается:**

- При использовании числа потоков, не превышающего количество логических и близкого к числу физических ядер (оптимально
  6–8 потоков для тестовой системы).
- При равномерном распределении нагрузки между потоками.
- При правильном подобранном размере входных данных, когда ещё не достигается упор в подсистему памяти.

**Ограничения подхода:**

- При малых объемах входных данных накладные расходы на создание и синхронизацию потоков превышают выигрыш от
  распараллеливания.
- Наличие строго последовательной части (финальное построение оболочки) жестко ограничивает максимальное теоретическое
  ускорение.
