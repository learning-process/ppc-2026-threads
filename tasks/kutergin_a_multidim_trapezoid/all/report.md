## 1. Контекст

Гибридная реализация, сочетающая технологии **MPI** и **OpenMP**, является необходимым этапом для
достижения максимальной масштабируемости алгоритма в средах высокопроизводительных вычислений
(HPC). В то время как чистый параллелизм на общей памяти (OpenMP/TBB) ограничен ресурсами одного
вычислительного узла, а чистый MPI создает избыточные накладные расходы при запуске отдельного
процесса на каждое ядро, гибридный подход предлагает двухуровневую иерархию управления ресурсами.

Для задачи многомерного интегрирования, обладающей идеальным параллелизмом (независимость расчетов
во всех узлах регулярной сетки), выбор гибридной схемы обусловлен следующими факторами:

1. **Минимизация коммуникационных затрат**: В схеме ALL технология MPI используется для
крупноблочного распределения итерационного пространства между узлами кластера. Это позволяет
сократить количество участников в коллективных операциях (`MPI_Bcast`, `MPI_Reduce`) по сравнению с
запуском MPI на каждом ядре, что критически важно для уменьшения времени синхронизации.
2. **Эффективное использование общей памяти**: Внутри каждого MPI-процесса технология OpenMP берет
на себя «мелкозернистое» распараллеливание вычислений. Это позволяет потокам работать с общим
объектом интегрируемой функции и вектором шагов сетки без необходимости их дублирования или
пересылки по сети, что значительно экономит пропускную способность памяти.
3. **Иерархическая декомпозиция**: Линеаризация $N$-мерной сетки в одномерный диапазон $[0, Total)$
позволяет элегантно реализовать вложенное разделение нагрузки. Сначала диапазон делится на «порции»
для процессов, а затем каждая порция динамически нарезается на «чанки» для потоков.

## 2. Постановка задачи

Требуется вычислить определенный интеграл многомерной функции методом трапеций на регулярной
расчетной сетке с использованием гибридной технологии параллелизма ALL (MPI + OpenMP). В качестве
эталона математической точности и базового времени выполнения ($T_1$) для последующего расчета
ускорения ($S$) и эффективности ($E$) используется последовательная реализация алгоритма (`SEQ`).

## 3. Базовый алгоритм

Используется многомерный метод трапеций на линеаризованной сетке. Общее число расчетных точек
(узлов) по всем осям составляет $Total = (n + 1)^d$, где $n$ — количество интервалов разбиения, а
$d$ — размерность пространства. Весь одномерный диапазон $[0, Total)$ обходится параллельно с
использованием двухуровневой схемы декомпозиции. На каждой итерации выделенного блока из линейного
индекса восстанавливается многомерный вектор индексов, рассчитываются геометрические координаты
текущей точки, определяется ее мультипликативный вес (в зависимости от положения на границах
расчетного гиперкуба) и вычисляется значение функции.

## 4. Межпроцессная схема

На верхнем уровне иерархии параллелизма используется технология MPI для распределения
вычислительной нагрузки между узлами системы. Используется классическая парадигма **Master-Worker**.

- **Роли rank-ов:**
  - **Master (`rank == 0`)**: владеет исходными параметрами интегрирования, вычисляет количество
  итераций для каждого процесса с учетом остатка и выполняет финальную сборку результата.
  - **Workers (`rank > 0`)**: принимают параметры сетки, вычисляют локальные суммы на своих
  участках с использованием OpenMP и участвуют в редукции.
- **Распределение данных:**
  - В данной задаче передача самих расчетных точек не требуется, так как каждый процесс способен
  вычислить их самостоятельно. Поэтому Master-процесс рассылает только параметры задачи всем
  остальным участникам с помощью `MPI_Bcast`. Рассылка выполняется покомпонентно, что гарантирует
  переносимость между системами с разным выравниванием памяти.
- **Декомпозиция (Partitioning):**
  - Итерационное пространство $[0, total\_nodes)$ делится на крупные блоки. Каждый процесс
  вычисляет свои переменные `my_proc_start` и `my_proc_count`. Для обеспечения **Load Balancing**
  используется логика распределения остатка (`proc_remainder`): процессы с рангом меньше остатка
  получают на одну итерацию больше. Это предотвращает простои при некратных размерах сетки.
- **Синхронизация и агрегация:**
  - **`MPI_Reduce`**: используется для сбора результатов. Каждый процесс передает свою сумму
  `local_sum`, рассчитанную внутри OpenMP-секции, а MPI выполняет операцию `MPI_SUM`. Итоговый
  результат масштабируется на объем элементарной ячейки `cell_volume` на корневом процессе.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса для задействования вычислительных ядер процессора используется
технология **OpenMP**. Реализована иерархическая схема распределения нагрузки, где порция данных
процесса (`my_proc_count`) дополнительно дробится между потоками.

- **Используемая технология:** Прямое управление потоками через библиотеку OpenMP. Количество
потоков задается программно через `omp_set_num_threads()` на основе значения, полученного из
окружения фреймворка (`ppc::util::GetNumThreads()`).
- **Деление работы между потоками:**
- В данной реализации применена **ручная декомпозиция диапазона** внутри параллельной области.
Каждый поток, используя свой идентификатор `tid` и общее число потоков `t_count`, вычисляет границы
своего поддиапазона (`my_thread_start` и `my_thread_count`). На уровне потоков также реализована
логика распределения остатка (`thread_remainder`), что гарантирует максимально равномерную загрузку
ядер.
- **Работа потоков:**
  - Каждый поток независимо вызывает метод `CalculateChunkSum`. Для обеспечения потокобезопасности
  вектор координат `coords` создается локально внутри метода `CalculateChunkSum`. Это гарантирует,
  что каждый поток работает в своем собственном стеке, исключая гонки данных.
- **Объединение локальных результатов:**
  - Для агрегации результатов используется механизм редукции OpenMP: **`reduction(+ : local_sum)
  `**. Данный подход позволяет каждому потоку накопить результат в своей локальной копии переменной
  `local_sum`. Итоговое сложение этих копий в общую переменную процесса происходит один раз в
  момент выхода из параллельного региона.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

В гибридной реализации ключевым аспектом является координация между распределенной (MPI) и общей
(OpenMP) памятью. Основная логика сосредоточена в методах распределения данных и иерархической
декомпозиции в `RunImpl`.

### Покомпонентная рассылка параметров (MPI)

Рассылка параметров интегрирования выполняется через коллективную операцию `MPI_Bcast` в методе
`DistributeData`:

```cpp
int dims_io = static_cast<int>(dims);
MPI_Bcast(&dims_io, 1, MPI_INT, 0, MPI_COMM_WORLD);
dims = static_cast<size_t>(dims_io);

auto &limits = std::get<1>(local_input_);
auto &n_steps = std::get<2>(local_input_);

if (rank != 0) {
  limits.resize(dims);
}

for (size_t i = 0; i < dims; ++i) {
  MPI_Bcast(&limits[i].first, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
  MPI_Bcast(&limits[i].second, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
}
MPI_Bcast(&n_steps, 1, MPI_INT, 0, MPI_COMM_WORLD);
```

### Иерархическая декомпозиция и редукция

Двухуровневый сбор данных сочетает в себе параллельную область OpenMP с редукцией и последующую
глобальную межпроцессную сборку через `MPI_Reduce` внутри метода `RunImpl`:

```cpp
#pragma omp parallel default(none) shared(h, dims, my_proc_start, my_proc_count, func, n_steps, 
limits) \
    reduction(+ : local_sum)
{
  int tid = omp_get_thread_num();
  int t_count = omp_get_num_threads();

  size_t thread_chunk = my_proc_count / t_count;
  size_t thread_remainder = my_proc_count % t_count;

  size_t my_thread_count = thread_chunk + (std::cmp_less(tid, thread_remainder) ? 1 : 0);
  size_t my_thread_start =
      my_proc_start + (tid * thread_chunk) + std::min(static_cast<size_t>(tid), thread_remainder);

  local_sum += CalculateChunkSum(my_thread_start, my_thread_start + my_thread_count, h, limits, 
  n_steps, func);
}
```

### Распаковка индексов и расчет весов узлов сетки

В изолированном методе `CalculateChunkSum` реализован расчет пространственных координат и
мультипликативного веса для каждой расчетной точки по методу трапеций:

```cpp
for (size_t i = start_idx; i < end_idx; ++i) {
  double weight = 1.0;
  size_t temp_idx = i;

  for (size_t dim = 0; dim < dims; ++dim) {
    const int nodes_in_dim = n_steps + 1;
    const int coord_idx = static_cast<int>(temp_idx % nodes_in_dim);
    temp_idx /= nodes_in_dim;

    coords[dim] = limits[dim].first + (static_cast<double>(coord_idx) * h[dim]);

    if (coord_idx == 0 || coord_idx == n_steps) {
      weight *= 0.5;
    }
  }
  chunk_sum += weight * func(coords);
}
```

## 7. Проверка корректности

Корректность гибридной реализации подтверждена серией функциональных тестов, охватывающих
размерности от 1D до 3D.

- **Сравнение с эталоном:** Итоговые значения, полученные версией ALL, сопоставлялись с
результатами последовательной реализации (SEQ). Допустимое отклонение было установлено на уровне
$\epsilon = 10^{-3}$, что позволило подтвердить математическую эквивалентность гибридного алгоритма.
- **Согласованность между рангами:** Сравнение фактического результата с ожидаемым выполняется на
корневом процессе (`rank == 0`), так как именно он владеет финальным результатом после операции
`MPI_Reduce`. На этапе `PostProcessingImpl` значение `res_` транслируется обратно всем процессам
для обеспечения корректности завершения тестов.
- **Изоляция и защита:** Применение `ppc::util::IsUnderMpirun()` в методах валидации и проверки
исключает некорректные вызовы MPI-функций при запуске тестов в однопроцессном режиме, что
гарантирует стабильность конвейера.

### Анализ результатов

1. **Кумулятивный накладной расход (1×1):** Конфигурация из одного процесса и одного потока
показала худшее время во всей серии тестов — **45.10 мс** ($S = 0.59x$). Это следствие
одновременной инициализации подсистем связи MS-MPI и параллельных регионов OpenMP. На малых объемах
сетки этот суммарный оверхед «съедает» базовую скорость вычислений.
2. **Сравнение «чистого» OpenMP и «чистого» MPI (1×2 vs 2×1):** При использовании двух рабочих
единиц производительность идентична: 23.95 мс для OpenMP и 24.20 мс для MPI. Это доказывает
симметричность декомпозиции, однако эффективность упирается в малый масштаб задачи: накладные
расходы на создание потоков (`#pragma omp parallel`) примерно равны расходам на локальный обмен
сообщениями внутри разделяемой памяти (Shared Memory IPC в MS-MPI).
3. **Эффект взаимной блокировки (2×2):** Конфигурация 2 MPI-процесса по 2 потока в каждом показала
сильное замедление относительно ожидаемых результатов (**23.81 мс**, $S = 1.11x$). Когда 4
параллельных воркера жестко привязаны к физическим ядрам, но логически разделены границами
независимых процессов, их рантаймы начинают конкурировать за шину памяти без возможности
динамического перераспределения задач. Чистая многопроцессность (4×1 — **21.15 мс**) в данном
случае оказывается эффективнее.
4. **Оптимальная точка масштабирования (2×4):** Максимальное ускорение зафиксировано на
конфигурации из двух процессов по 4 потока (**15.97 мс**, $S = 1.65x$). Здесь OpenMP за счет работы
на четырех логических потоках частично нивелирует задержки планировщика внутри каждого
MPI-процесса. Переход к конфигурации 4×2 (**16.29 мс**) не дает преимущества из-за утяжеления
MPI-инфраструктуры (увеличение числа участников в `MPI_Reduce`).

многомерных интегралов методом трапеций на процессоре Intel Core i5-8300H позволило сделать выводы

Гибридная параллельная схема **оправдана**:

- **При масштабировании под SMT-архитектуры**: Использование схемы «крупных» узлов (2 MPI-процесса,
внутри которых разворачиваются по 4 OpenMP-потока) привело к лучшему для данной сборки результату в
**15.97 мс**. Это подтверждает классическое правило проектирования гибридных систем: количество
MPI-процессов предпочтительно минимизировать до физических сокетов или NUMA-узлов, отдавая
внутреннее распараллеливание на откуп легковесным потокам.

Гибридная параллельная схема **не оправдана**:

- **На ультракоротких расчетных сетках**: Если время последовательного выполнения задачи
исчисляется десятками миллисекунд (26.43 мс), развертывание гибридной инфраструктуры приводит к
падению эффективности ($E$) до 20–30%. Внутренний оверхед межпроцессной сборки данных
(`MPI_Reduce`) и барьерной синхронизации OpenMP перекрывает выигрыш от деления итераций.
- **В сравнении с чистыми абстракциями**: По чистой скорости гибридный вариант уступил
оптимизированным реализациям на C++ STL (11.18 мс) и Intel TBB (14.87 мс). Схема MPI+OpenMP слишком
тяжеловесна для работы в рамках одного физического компьютера (без реальной сетевой сети Infiniband/
Ethernet) и раскрывает свой потенциал исключительно на крупномасштабных промышленных кластерах при
объемах вычислений, исчисляемых минутами или часами.
