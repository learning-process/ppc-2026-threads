## 1. Контекст

Перенос алгоритма из последовательной версии (`SEQ`) в реализацию на базе библиотеки **Intel
Threading Building Blocks (TBB)** обусловлен необходимостью эффективного масштабирования вычислений
на современных многоядерных архитектурах. Численное интегрирование в многомерных пространствах
сталкивается с проблемой экспоненциального роста количества расчетных узлов при увеличении числа
измерений (проклятие размерности), что делает последовательный расчет неэффективным.

Поскольку вычисление значения функции в каждом узле регулярной сетки полностью независимо от других
узлов, задача обладает **идеальным параллелизмом**. Библиотека Intel TBB является наиболее
подходящим инструментом для данной задачи по следующим причинам:

1. **Задачно-ориентированная модель**: В отличие от потоковых моделей, TBB оперирует абстракцией
логических задач. Это позволяет встроенному планировщику (task scheduler) использовать алгоритм
**work-stealing** (кража задач), обеспечивая гибкую динамическую балансировку нагрузки между ядрами.
2. **Декларативная редукция**: Использование высокоуровневого примитива **`tbb::parallel_reduce`**
идеально ложится на математическую суть алгоритма. Нам не требуется вручную управлять
синхронизацией; библиотека гарантирует потокобезопасную сборку частичных сумм на уровне рантайма.
3. **Рекурсивное разбиение диапазонов**: Вместо вложенных циклов в данной реализации сохранена
стратегия **линеаризации итерационного пространства** в одномерный диапазон индексов типа `size_t`.
TBB через механизм **`tbb::blocked_range`** выполняет рекурсивное деление этого диапазона,
автоматически адаптируя размер порций работы под конкретную архитектуру процессора.

## 2. Постановка задачи

Требуется вычислить определенный интеграл многомерной функции методом трапеций на регулярной сетке
с использованием возможностей библиотеки Intel TBB. В качестве эталона математической точности и
базового времени выполнения ($T_1$) для последующего расчета ускорения ($S$) и эффективности ($E$)
используется последовательная реализация алгоритма (`SEQ`).

## 3. Базовый алгоритм

Используется многомерный метод трапеций на линеаризованной сетке. Общее число расчетных точек
(узлов) по всем измерениям составляет $Total = (n + 1)^d$, где $n$ — количество интервалов
разбиения, а $d$ — размерность пространства. Весь одномерный диапазон $[0, Total)$ обходится
параллельно. На каждой итерации из линейного индекса восстанавливается многомерный вектор индексов,
рассчитываются геометрические координаты текущей точки, определяется ее мультипликативный вес
(граничные точки получают вес $0.5$) и вычисляется значение функции. В конце выполнения общая
накопленная сумма масштабируется на элементарный объем ячейки сетки.

## 4. Схема распараллеливания

Параллелизм в данной реализации построен на базе задачно-ориентированной модели Intel TBB с явным
разделением итерационного пространства через диапазоны.

**Анализ механизмов и примитивов:**

- **Выбор примитива:** Для реализации алгоритма выбран шаблон **`tbb::parallel_reduce`**. Этот
выбор является наиболее обоснованным, так как задача сводится к суммированию независимых значений.
Шаблон автоматически создает локальные аккумуляторы для каждой подзадачи и объединяет их с помощью
заданного функционала слияния.
- **Диапазон работы (`blocked_range`):** В качестве итерационного пространства используется объект
**`tbb::blocked_range<size_t>`** от $0$ до `total_points`. Это позволяет планировщику TBB
рекурсивно дробить диапазон индексов на непересекающиеся блоки.
- **Grainsize (Размер гранулы):** Параметр `grainsize` оставлен по умолчанию. Это позволяет
библиотеке использовать встроенные эвристики (`auto_partitioner`) для определения оптимального
размера подзадач, гарантируя, что накладные расходы на создание задачи не превысят выигрыш от
параллельного исполнения.
- **Синхронизация и сборка:** Объединение промежуточных результатов выполняется с помощью
стандартного функтора **`std::plus<>()`**. Потокобезопасность гарантируется тем, что каждый поток
работает со своей изолированной копией `local_sum` внутри лямбда-функции, а финальная редукция
выполняется рантаймом TBB в точках синхронизации рекурсивного дерева задач.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

### Реализация вычислительного ядра

Вся логика декомпозиции, рекурсивного деления и расчетов сосредоточена в методе `RunImpl()` с
использованием лямбда-выражения:

```cpp
res_ = tbb::parallel_reduce(tbb::blocked_range<size_t>(0, total_points), 0.0,
                            [=](const tbb::blocked_range<size_t> &r, double local_sum) {
  std::vector<double> point(dim);

  for (size_t i = r.begin(); i < r.end(); ++i) {
    double weight = 1.0;
    size_t temp_idx = i;

    // Восстановление многомерного индекса и координат узла сетки
    for (int dim_idx = 0; dim_idx < dim; ++dim_idx) {
      int coord_idx = static_cast<int>(temp_idx % (n + 1));
      temp_idx /= (n + 1);

      point[dim_idx] = borders[dim_idx].first + (coord_idx * h[dim_idx]);

      // Расчет мультипликативного веса точки по методу трапеций
      if (coord_idx == 0 || coord_idx == n) {
        weight *= 0.5;
      }
    }
    local_sum += weight * func(point);
  }
  return local_sum;
}, std::plus<>()) * cell_volume;
```

**Архитектурные особенности и потокобезопасность:**

- Изоляция локальной памяти: Локальный вектор координат point объявляется внутри лямбда-функции
обработки диапазона tbb::blocked_range. Он создается один раз на вызов диапазона и переиспользуется
внутри вложенного цикла for, что защищает алгоритм от постоянных аллокаций памяти в куче и
полностью предотвращает гонки данных (Data Race).
- Контекст захвата [=]: Лямбда-выражение захватывает параметры задачи (n, dim, h, borders, func)
по значению. Объект функции func и векторы параметров используются только для чтения, что является
абсолютно потокобезопасным по стандарту C++.
- Мгновенное позиционирование: Восстановление индексов через операции деления (/) и остатка (%)
позволяет любому рабочему потоку TBB взять произвольный поддиапазон r и сразу начать расчеты с
нужной точки пространства, что необходимо для корректной работы алгоритма work-stealing.

## 6. Проверка корректности

### Анализ результатов

1. **Значительный оверхед на одном потоке:** При запуске на 1 потоке TBB демонстрирует существенное
замедление относительно последовательной версии (46.43 мс против 26.43 мс, $S = 0.57x$). Это
объясняется «тяжеловесностью» внутренней инфраструктуры Intel TBB (инициализация пула потоков,
создание графа задач и работа менеджера памяти), критически заметной на коротких вычислительных
задачах с общим временем счета менее 50 мс.
2. **Эффективное масштабирование на физических ядрах:** При переходе на 2 и 4 потока алгоритм
преодолевает начальный оверхед и демонстрирует уверенный рост производительности. На 4 потоках
время сокращается до **15.57 мс** ($S = 1.70x$). Снижение эффективности до 42.5% вызвано тем, что
накладные расходы планировщика на управление задачами продолжают отсекать часть полезного времени
на малом объеме расчетной сетки.
3. **Успешная утилизация режима SMT (8 потоков):** В отличие от OpenMP, который на 8 потоках
показал деградацию, Intel TBB на конфигурации из 8 логических воркеров достиг абсолютного пика
скорости — **14.87 мс** ($S = 1.78x$). Несмотря на вычислительную интенсивность задачи (FPU-bound),
динамический планировщик TBB смог сбалансировать нагрузку на логических ядрах без критических
блокировок.

## 9. Выводы

Реализация параллельной схемы на базе Intel TBB для расчета многомерных интегралов методом трапеций
выявила ключевые архитектурные отличия этой технологии от классического OpenMP на процессоре Intel
Core i5-8300H.

Параллельная схема на TBB **оправдана**:

- **Для достижения абсолютного максимума скорости**: Конфигурация на 8 потоках показала наилучшее
время выполнения (14.87 мс) среди всех протестированных версий. Алгоритм кражи задач
(**Work-Stealing**), лежащий в основе планировщика TBB, доказал свою высокую эффективность: вместо
жесткого закрепления итераций за потоками ОС, TBB гибко перераспределяет задачи в реальном времени,
что минимизирует простои ядер при совместном использовании блоков плавающей точки (FPU) в режиме
Hyper-Threading.
- **На полноценных многоядерных конфигурациях (от 2 потоков и выше)**: Библиотека успешно решает
задачу декомпозиции расчетной сетки и обеспечивает стабильное ускорение, полностью исключая race
conditions благодаря локализации векторов координат.

Параллельная схема на TBB **не оправдана**:

- **Для последовательного или однопоточного выполнения**: Внутренние накладные расходы на
развертывание пула потоков TBB слишком велики для ультракоротких задач. Если алгоритм запускается
на одном ядре, чистая последовательная версия (SEQ) оказывается быстрее почти в 2 раза.
- **При жестких требованиях к эффективности утилизации ресурсов ($E$)**: Из-за малого объема
вычислительной сетки планировщик TBB тратит заметную часть времени на координацию потоков, что
снижает общую эффективность параллелизма до 22.3% на максимуме потоков. Для раскрытия полной
эффективности TBB требуется кратное увеличение объема интегрируемого пространства.
