## 1. Введение

В данной работе проводится сравнительный анализ эффективности пяти парадигм параллельного программирования
(последовательная версия, OpenMP, Intel TBB, C++ STL Execution Policies и гибридная модель MPI+OpenMP)
на примере алгоритма многомерного интегрирования методом трапеций.
Данная задача обладает «идеальным параллелизмом»: вычисление значения функции в каждой ячейке
$N$-мерной сетки полностью независимо от других.
Это позволяет исследовать чистую масштабируемость каждой технологии, оценить
накладные расходы различных рантайм-систем на управление потоками
и проверить эффективность механизмов редукции в
условиях высокой вычислительной интенсивности.

## 2. Единая постановка задачи

- **Вход:**
  - Структура `IntegrationData`, включающая:
    - Вектор пар вещественных чисел `limits`, задающий границы интегрирования $[a_i, b_i]$ для каждой размерности.
    - Вектор целых чисел `n_steps`, определяющий количество шагов разбиения по каждой оси.
    - Объект `std::function`, представляющий подынтегральную функцию $f(x_1, \dots, x_N)$.
- **Выход:** Число типа `double` — вычисленное значение гиперобъема под поверхностью
функции на заданном гиперкубе методом трапеций.
- **Ограничения:**
  - Алгоритм должен поддерживать произвольную размерность пространства $N \ge 1$.
  - Количество шагов по каждой размерности должно быть строго положительным ($n_i > 0$).
  - Реализация обязана обеспечивать автоматическую декомпозицию диапазонов в случаях,
   когда общее число элементарных ячеек сетки не кратно количеству задействованных потоков или процессов.
- **Критерий корректности:** Абсолютное расхождение между результатом параллельных реализаций и
аналитическим эталоном не должно превышать установленную погрешность $\epsilon = 10^{-4}$.

## 3. Единая методика эксперимента

**Окружение:**

- **CPU:** Intel Core i5-8300H (4 физических ядра, 8 логических потоков)
- **RAM:** 12 GB
- **OS:** Windows 10 / MSYS2 Environment
- **Compiler:** GCC / Clang via MSYS2
- **Build type:** Release

**Управление экспериментом:**

- **Конфигурация параллелизма:**
  - Число рабочих потоков для технологий OMP, TBB, STL и внутрипроцессного уровня ALL задавалось
  через переменную окружения **`PPC_NUM_THREADS`**.
  - Количество процессов для гибридной реализации (ALL) задавалось при запуске через параметр
  **`-n` утилиты `mpiexec`**.
- **Размер задачи:** Тесты производительности выполнялись на базе класса
`KuterginATrapezoidPerfTest` для многомерной сетки. Время замерялось во встроенных режимах
тестирования производительности (`task_run`).

**Расчет метрик:**

- **Базовое время ($T_1$):** За эталон (baseline) принято чистое время работы в режиме `task_run`
последовательной версии (SEQ), составившее **26.43 мс**.
- **Ускорение ($S$):** Рассчитывается по формуле $S = T_1 / T_p$, где $T_p$ — время выполнения
параллельной реализации.
- **Эффективность ($E$):** Рассчитывается по формуле $E = (S / P) \times 100\%$.

## 6. Интерпретация различий

Анализ сводных данных позволяет сделать важные выводы о поведении различных моделей параллелизма на
архитектуре с 4 физическими ядрами и SMT (Intel Core i5-8300H).

1. **SEQ (Базовый уровень):** Последовательная версия фиксирует «чистую» стоимость вычислений на
одном ядре без каких-либо накладных расходов параллельных рантаймов (26.43 мс).
2. **OMP (Накладные расходы и ограничения SMT):** OpenMP показал пик на 4 физических ядрах (17.26
мс, $S = 1.53x$). Переход на 8 потоков привел к жесткой деградации до 32.36 мс. Потоки OpenMP
привязаны к логическим ядрам ОС и начинают непрерывно блокировать друг друга на FPU-интенсивных
операциях вычисления весов трапеций.
3. **TBB (Превосходство алгоритма Work-Stealing):** Intel TBB показал стабильное масштабирование и
единственный смог эффективно утилизировать режим SMT (8 потоков), выдав время **14.87 мс** ($S = 1.
78x$). Алгоритм кражи задач (Work-Stealing) балансирует очереди на логических ядрах в реальном
времени, предотвращая глубокие простои конвейера CPU. Однако на 1 потоке из-за тяжеловесной
структуры управления TBB показал худший результат (46.43 мс).
4. **STL (Парадокс абстракции):** Реализация на базе параллельных политик STL
(`std::execution::par`) продемонстрировала **наилучшую чистую скорость** на 4 ядрах — **11.18 мс**
($S = 2.36x$). Встроенные оптимизации компилятора MSYS2 смогли идеально линеаризовать и
векторизовать обход сетки итераторов. Тем не менее, STL полностью игнорирует ограничения среды
(`PPC_NUM_THREADS=1` выдал сразу 11.80 мс), работая как черный ящик.
5. **ALL (Гибридный компромисс):** Гибридная версия MPI+OpenMP показала сильный начальный
кумулятивный оверхед рантаймов на малых масштабах (1x1 — 45.10 мс). Пик производительности
гибридной схемы достигнут на конфигурации **2 процесса × 4 потока** (**15.97 мс**, $S = 1.65x$),
где легковесные потоки OpenMP эффективно сгладили накладные расходы внутри каждого из двух
MPI-процессов.

## 7. Репродуцируемость

Запуски и компиляция осуществляются в среде Windows 10 через командную строку `cmd.exe`:

```cmd
:: Сборка проекта
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel

:: Запуск последовательной версии
build\bin\ppc_perf_tests.exe --gtest_filter=*Kutergin*seq*

:: Запуск OpenMP-версии (Пример на 4 потоках)
set PPC_NUM_THREADS=4 && build\bin\ppc_perf_tests.exe --gtest_filter=*Kutergin*omp*

:: Запуск TBB-версии (Пример на 8 потоках)
set PPC_NUM_THREADS=8 && build\bin\ppc_perf_tests.exe --gtest_filter=*Kutergin*tbb*

:: Запуск STL-версии
build\bin\ppc_perf_tests.exe --gtest_filter=*Kutergin*stl*

:: Запуск гибридной ALL-версии (Пример: 2 процесса по 4 потока)
set PPC_NUM_THREADS=4 && mpiexec -n 2 build\bin\ppc_perf_tests.exe --gtest_filter=*Kutergin*all*
```

## 8. Заключение

Сравнительный анализ выявил, что на коротких временах счета (~26 мс) выбор технологии критически
зависит от оверхеда рантайм-системы. Наилучшее абсолютное время выполнения на физических ядрах
показала STL-версия (11.18 мс) за счет агрессивного инлайнинга компилятора, однако она не поддается
внешнему контролю распределения потоков. Среди управляемых систем Intel TBB подтвердил свое
превосходство над OpenMP на логических ядрах (SMT), успешно утилизировав 8 потоков за счет
планировщика Work-Stealing. Гибридный подход MPI+OpenMP оправдывает себя на конфигурации 2х4 (15.97
мс), минимизируя межпроцессное взаимодействие, но является избыточным для локального запуска на
одном вычислительном узле без увеличения глобального объема расчетной сетки.
