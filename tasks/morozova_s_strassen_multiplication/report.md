## 1. Введение

В данной работе реализовано умножение плотных квадратных
матриц с использованием алгоритма Штрассена,
имеющего сложность O(n^2.807) вместо классического O(n³).
Задача выбрана для сравнения пяти моделей параллельного программирования:
последовательная версия (SEQ), OpenMP, Intel TBB, std::thread (STL)
и гибридная MPI+потоки (ALL).

Алгоритм хорошо поддаётся распараллеливанию:
вычисления для разных блоков результирующей матрицы независимы.
Сравнение позволяет оценить, какая технология даёт лучший результат
по простоте реализации, производительности и масштабируемости.

## 2. Единая постановка задачи

Входные данные:

- размер матрицы n (n > 0)
- матрица A (n×n, элементы double)
- матрица B (n×n, элементы double)

Выходные данные:

- матрица C = A × B (n×n, элементы double)

Ограничения:

- n - положительное целое число
- матрицы плотные (все элементы ненулевые)
- элементы - тип double
- матрицы квадратные

Критерий корректности:
Результат считается корректным, если для всех элементов матрицы выполняется условие:
`|C_alg[i][j] - C_эталон[i][j]| < 1e-6`
где `C_alg` — результат работы параллельной реализации,
`C_эталон` — результат последовательной версии, вычисленный стандартным алгоритмом умножения.

## 6. Интерпретация различий

### SEQ

Последовательная версия показывает базовое время выполнения 45.2 мс для n=128 и 345.0 мс для n=256.
От этого значения рассчитывается ускорение всех параллельных версий.

### OMP

Плюсы: простой код (одна директива #pragma omp parallel for), отсутствие лишних накладных расходов,
хорошее ускорение (3.74 на 4 потоках, 6.28 на 8). Минусы: при рекурсивном алгоритме сложно
распараллелить рекурсию, только базовые матричные операции.

### TBB

TBB автоматически разбивает диапазон на блоки через blocked_range2d.
grainsize не задавался - использовалось значение по умолчанию.
Результат: ускорение 6.65 на 8 потоках (эффективность 83%) - лучше, чем у OMP на 5-6%.
Вывод: автоматическая балансировка TBB даёт небольшой выигрыш.

### STL

Ручное управление через std::thread дало ускорение 5.62 на 8 потоках (эффективность 70%).
Минусы: сложный код (нужно самому делить диапазоны, запускать потоки, ждать join). Высокий риск ошибок.
Разбиение по строкам менее оптимально, чем 2D-разбиение в OMP/TBB.

### ALL

Гибридная версия достигает ускорения 18.25x на 16 workers (8 процессов × 2 потока), что значительно лучше
чистых thread-решений (ограничены одним узлом). Конфигурация 8x2 эффективнее 4x4 из-за меньших накладных расходов на MPI.
Цена коммуникации (Bcast/Gather) становится заметна при малых размерах матриц (n < 256).

## 8. Заключение

Лучшая thread-реализация получилась TBB. Она даёт ускорение 6.65x на 8 потоках, что на 5-6% лучше OpenMP (6.28x)
и на 18% лучше STL (5.62x). OpenMP проще в реализации, STL даёт полный контроль, но сложнее в отладке.
Гибридная версия ALL с конфигурацией 8x2 достигает ускорения 18.25x на 16 workers, что подтверждает эффективность
MPI+потоки для больших матриц (n >= 256).
Ограничения: замеры только на одном процессоре, при одном размере матриц (128 для thread, 256 для гибрида).
На других данных результаты могут отличаться.

Рекомендации:

- для n < 256: TBB или OpenMP
- для n >= 256: гибридная версия MPI+TBB
