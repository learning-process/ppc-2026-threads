## 1. Introduction

Данная работа посвящена реализации параллельного алгоритма поразрядной сортировки (Radix Sort) для чисел
с плавающей точкой двойной точности (`double`) с последующим объединением блоков с помощью четно-нечетного слияния
Бэтчера (Odd-Even Batcher Merge). В этой версии для распараллеливания используется стандартная библиотека C++
(`std::thread` / STL алгоритмы).

## 2. Problem Statement

**Задача:** Отсортировать массив элементов типа `double`.
**Ограничения:**

- Данные типа `double` нельзя напрямую сортировать классической поразрядной сортировкой из-за стандарта IEEE 754.
- Необходимо использовать четно-нечетное слияние Бэтчера, требующее, чтобы количество сливаемых блоков было
  степенью двойки.
- Запрещено использовать сторонние библиотеки (OMP, TBB, MPI). Разрешены только возможности стандарта C++11 и выше.

## 3. Baseline Algorithm (Sequential)

Последовательный алгоритм состоит из двух этапов:

1. **Локальная сортировка (LSD Radix Sort):** Массив чисел `double` побитово преобразуется в `uint64_t`. Знаковый
   бит инвертируется (для сохранения естественного порядка отрицательных и положительных чисел). Применяется
   8-проходная (побайтовая) сортировка подсчетом. Затем числа конвертируются обратно.
2. **Слияние Бэтчера:** Массив делится на логические блоки, которые попарно сливаются итеративным алгоритмом.

## 4. Parallelization Scheme (STL Threads)

**Ручное управление потоками:**

- **Создание пула потоков:** Вектор объектов `std::thread` создается динамически на основе требуемого количества
  потоков (`ppc::util::GetNumThreads()`).
- **Разделение данных:** Размер массива делится на количество потоков. Каждый поток получает индексы начала и
  конца своего диапазона для локальной поразрядной сортировки.
- **Синхронизация:** После запуска локальных сортировок главный поток ожидает завершения воркеров с помощью
  метода `.join()`.
- **Слияние Бэтчера:** Для каждой итерации сети слияния Бэтчера заново порождаются потоки (или используются
  параллельные алгоритмы `std::for_each` с `std::execution::par`), распределяющие между собой операции `CompareExchange`.

## 5. Implementation Details

- **Жизненный цикл потоков:** Порождение и уничтожение `std::thread` — дорогая операция ОС. Для минимизации
  накладных расходов стараются сгруппировать независимые слияния Бэтчера в более крупные задачи.
- **Безопасность памяти:** Для передачи данных в `std::thread` используются захваты по ссылке (`[&]`) сырых
  указателей массива. Это позволяет избежать копирования больших объемов данных при запуске потоков.
- **Совместимость:** Решение полностью кроссплатформенное и не требует флагов компиляции кроме `-pthread`.

## 7. Results and Discussion

### 7.1 Correctness

Корректность реализации STL подтверждена прохождением набора функциональных тестов (`ppc_func_tests`). Механизм
`join()` обеспечивает надежную барьерную синхронизацию, не допуская начала следующего этапа Бэтчера до окончания
предыдущего.

### 7.2 Performance

Время измерялось метрикой `task_run` (чистое время работы алгоритма).
Базовое время последовательной версии (SEQ) на 1 потоке: **0.01237 s**.

| Mode | Count | Time, s | Speedup | Efficiency |
|------|-------|---------|---------|------------|
| seq  | 1     | 0.01237 | 1.00    | 100.0%     |
| omp  | 4     | 0.00831 | 1.48    | 37.0%      |
| tbb  | 4     | 0.00764 | 1.61    | 40.2%      |
| tbb  | 8     | 0.00664 | 1.86    | 23.2%      |
| stl  | 4     | 0.00995 | 1.24    | 31.0%      |

**Анализ результатов:**
Реализация через STL показала наименьшее ускорение среди всех многопоточных версий (1.24x на 4 потоках). Это
связано с тем, что прямое использование `std::thread` приводит к значительным системным накладным расходам на
каждое создание и уничтожение потока. В отличие от OMP или TBB, здесь отсутствует встроенный интеллектуальный
пул потоков (thread pool) и механизм work-stealing, что делает STL-версию неэффективной на коротких задачах.

## 8. Conclusions

Технология STL `std::thread` позволяет получить полный контроль над распараллеливанием и избежать внешних
зависимостей, однако требует написания значительного количества boilerplate-кода. Из-за отсутствия встроенных
механизмов пулирования задач (в стандарте C++ до C++26), производительность этой версии уступает
специализированным библиотекам.
