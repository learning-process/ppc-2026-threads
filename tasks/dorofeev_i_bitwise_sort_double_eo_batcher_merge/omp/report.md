## 1. Introduction

Данная работа посвящена реализации параллельного алгоритма поразрядной сортировки (Radix Sort) для чисел
с плавающей точкой двойной точности (`double`) с последующим объединением блоков с помощью четно-нечетного слияния
Бэтчера (Odd-Even Batcher Merge). Цель — распараллелить вычисления в рамках одного узла (разделяемая память) с
использованием директив OpenMP.

## 2. Problem Statement

**Задача:** Отсортировать массив элементов типа `double`.
**Ограничения:**

- Данные типа `double` нельзя напрямую сортировать классической поразрядной сортировкой из-за стандарта IEEE 754.
- Необходимо использовать четно-нечетное слияние Бэтчера, требующее, чтобы количество сливаемых блоков было
  степенью двойки.
- Параллелизация должна осуществляться исключительно средствами технологии OpenMP.

## 3. Baseline Algorithm (Sequential)

Последовательный алгоритм состоит из двух этапов:

1. **Локальная сортировка (LSD Radix Sort):** Массив чисел `double` побитово преобразуется в `uint64_t`. Знаковый
   бит инвертируется (для сохранения естественного порядка отрицательных и положительных чисел). Применяется
   8-проходная (побайтовая) сортировка подсчетом. Затем числа конвертируются обратно.
2. **Слияние Бэтчера:** Массив делится на логические блоки, которые попарно сливаются итеративным алгоритмом.

## 4. Parallelization Scheme (OpenMP)

**Декомпозиция данных и задач:**

- **Локальная сортировка:** Исходный массив (дополненный до степени двойки) логически делится на независимые чанки
  (chunks). С помощью директивы `#pragma omp parallel for` потоки параллельно применяют алгоритм `RadixSortDouble`
  к своим участкам массива.
- **Слияние Бэтчера:** После локальной сортировки начинается фаза слияния. Внешний цикл итерируется по размерам
  сливаемых блоков, а внутренний цикл (распараллеленный через `#pragma omp parallel for`) распределяет пары
  сливаемых блоков между потоками OpenMP.

## 5. Implementation Details

- **Управление потоками:** Количество используемых потоков задается через внутреннюю утилиту `ppc::util::GetNumThreads()`.
  Если переменная среды не задана, используется 1 поток как fallback-вариант.
- **Барьерная синхронизация:** Использование директивы `parallel for` автоматически обеспечивает неявную барьерную
  синхронизацию в конце каждого этапа слияния, что критически важно для корректной работы сети Бэтчера
  (результат предыдущего шага должен быть полностью готов до начала следующего).
- **Разделение переменных:** Все временные переменные внутри блоков `parallel for` делаются локальными для каждого
  потока, чтобы избежать состояния гонки (race condition) при сортировке подсчетом.

## 7. Results and Discussion

### 7.1 Correctness

Корректность OpenMP-реализации подтверждена прохождением набора функциональных тестов (`ppc_func_tests`). Код
успешно справляется с разделением массива между потоками и корректным итеративным слиянием без возникновения
race conditions.

### 7.2 Performance

Время измерялось метрикой `task_run` (чистое время работы алгоритма).
Базовое время последовательной версии (SEQ) на 1 потоке: **0.01237 s**.

| Mode | Count | Time, s | Speedup | Efficiency |
|------|-------|---------|---------|------------|
| seq  | 1     | 0.01237 | 1.00    | 100.0%     |
| omp  | 4     | 0.00831 | 1.48    | 37.0%      |
| tbb  | 4     | 0.00764 | 1.61    | 40.2%      |
| tbb  | 8     | 0.00664 | 1.86    | 23.2%      |
| stl  | 4     | 0.00995 | 1.24    | 31.0%      |

**Анализ результатов:**
Использование OpenMP на 4 потоках позволило получить ускорение в 1.48 раза. Данный результат показывает, что на
малых объемах данных накладные расходы на порождение пула потоков OpenMP (fork-join overhead) отнимают
существенную часть времени вычислений. Эффективность составляет 37%, что является типичным показателем для
мелких массивов.

Алгоритм был успешно распараллелен средствами OpenMP. Переход от последовательного кода потребовал минимального
