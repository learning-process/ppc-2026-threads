## 1. Контекст

Исходная задача — применение оператора Собеля к полутоновому изображению для выделения
границ объектов. Последовательная версия служит эталоном корректности и базой для измерения
ускорения параллельных реализаций.

Оператор Собеля вычисляет градиент интенсивности изображения в каждой точке, применяя
свёртку с двумя ядрами 3×3: горизонтальным `Gx` и вертикальным `Gy`. Магнитуда градиента
характеризует резкость перехода интенсивности и используется как мера выраженности границы.

ALL-реализация использует двухуровневый параллелизм: на верхнем уровне строки изображения
распределяются между MPI-процессами через `MPI_Scatterv`, на нижнем уровне каждый процесс
обрабатывает свой фрагмент параллельно через `std::thread`. Такой подход позволяет
масштабироваться как по числу узлов кластера, так и по числу ядер на каждом узле.

## 2. Постановка задачи

**Входные данные**: структура `ImgContainer`, содержащая:

- `width`, `height` — размеры изображения;
- `pixels` — вектор значений пикселей в диапазоне `[0, 255]`.

Поддерживаются форматы `.png`, `.jpg`, `.bmp` (загрузка через `stb_image` с автоматическим
преобразованием в полутоновое по формуле `0.299*R + 0.587*G + 0.114*B`), а также текстовый
формат `.txt` (первая строка — размеры, далее — значения пикселей).

**Выходные данные**: `std::vector<int>` — вектор магнитуд градиента в диапазоне `[0, 255]`,
где каждый элемент соответствует пикселю исходного изображения.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги:

1. Для каждого внутреннего пикселя `(row, col)` (граничные пиксели остаются нулевыми)
   считываются указатели на три соседние строки: `prev`, `curr`, `next`.

2. Вычисляются горизонтальный и вертикальный компоненты градиента:

```bash
Gx = -prev[col-1] + prev[col+1] - 2*curr[col-1] + 2*curr[col+1]
     - next[col-1] + next[col+1]

Gy = -prev[col-1] - 2*prev[col] - prev[col+1] + next[col-1]
     + 2*next[col] + next[col+1]
```

3. Магнитуда вычисляется по быстрому целочисленному приближению:

```bash
magnitude = (max(|Gx|, |Gy|) * 123 + min(|Gx|, |Gy|) * 51) >> 7
```

4. Результат ограничивается диапазоном `[0, 255]` и записывается в выходной вектор.

Временная сложность SEQ: O(width × height). Граничные пиксели остаются нулевыми.

## 4. Межпроцессная схема (MPI)

**Распределение строк.** Внутренние строки изображения (`inner_rows = h - 2`) делятся
между `mpi_size` процессами равномерно с учётом остатка. Для каждого процесса `i`
вычисляются `row_counts[i]` и `row_offsets[i]`:

```cpp
row_counts[i] = (inner_rows / mpi_size)
              + (i < inner_rows % mpi_size ? 1 : 0);
row_offsets[i] = (i == 0) ? 0
               : row_offsets[i-1] + row_counts[i-1];
```

**Ghost rows (строки-призраки).** Для корректного вычисления ядра Собеля на граничных
строках каждого фрагмента каждый процесс получает свой диапазон с одной дополнительной
строкой сверху и снизу. Это реализуется через `send_counts[i] = (row_counts[i] + 2) * w`
и `send_displs[i] = row_offsets[i] * w` при вызове `MPI_Scatterv`.

**Используемые MPI-вызовы:**

- `MPI_Bcast` — рассылка размеров изображения (`w`, `h`) от процесса 0 всем.
- `MPI_Scatterv` — раздача фрагментов изображения (с ghost rows) каждому процессу.
- `MPI_Gatherv` — сбор обработанных строк на процессе 0.
- `MPI_Bcast` — рассылка финального результата всем процессам.

**Результаты** собираются на процессе 0 через `MPI_Gatherv` со смещением `+1` строку
(чтобы пропустить ghost row при записи в выходной вектор), после чего рассылаются всем
процессам через `MPI_Bcast`.

## 5. Внутрипроцессная схема (STL threads)

Каждый MPI-процесс обрабатывает свой фрагмент (`local_inner` строк) параллельно через
`std::thread`. Число потоков определяется `ppc::util::GetNumThreads()`. Диапазон строк
делится равномерно между потоками:

```cpp
void RunThreads(const int *lb, int *lo, int w,
                int local_inner, int num_stl) {
  std::vector<std::thread> threads;
  threads.reserve(static_cast<size_t>(num_stl));

  for (int tid = 0; tid < num_stl; ++tid) {
    threads.emplace_back(
      [lb, lo, w, local_inner, num_stl, tid]() {
        const int chunk = local_inner / num_stl;
        const int row_begin = tid * chunk;
        const int row_end = (tid == num_stl - 1)
                          ? local_inner : row_begin + chunk;
        ProcessRowsALL(lb, lo, w, row_begin, row_end);
      });
  }
  for (auto &th : threads) th.join();
}
```

Каждый поток записывает результат в независимый диапазон выходного буфера `local_out`,
что полностью исключает гонки данных без мьютексов.

## 6. Детали реализации

**Файлы реализации:**

- `all/include/ops_all.hpp` — объявление класса `ShakirovaESobelEdgeDetectionALL`,
  наследующего `BaseTask`.
- `all/src/ops_all.cpp` — реализация методов, включая MPI-логику в `RunImpl()` и
  вспомогательные функции `ProcessRowsALL`, `ComputeDistribution`,
  `ComputeScatterParams`, `RunThreads`.

**Разделение ответственности:**

- `ComputeDistribution` — вычисляет `row_counts` и `row_offsets` для каждого процесса.
- `ComputeScatterParams` — вычисляет `send_counts` и `send_displs` с учётом ghost rows.
- `ProcessRowsALL` — применяет оператор Собеля к диапазону строк локального буфера.
- `RunThreads` — создаёт STL-потоки и распределяет строки между ними.

**Отличия от STL-версии:**

1. Добавлен MPI-уровень: каждый процесс обрабатывает только свой фрагмент изображения.
2. Введены ghost rows для корректной свёртки на границах фрагментов.
3. Локальный буфер `local_buf` содержит `(local_inner + 2) * w` элементов (с ghost rows),
   а `local_out` — только `local_inner * w` результирующих элементов.
4. Индексация в `ProcessRowsALL` ведётся относительно начала локального буфера: `row=0`
   соответствует первой внутренней строке фрагмента, `prev` указывает на ghost row сверху.

## 9. Результаты производительности

Производительность измерялась с помощью `ppc_perf_tests` при одном MPI-процессе
(`mpirun -n 1`) и разном числе STL-потоков, задаваемом через `PPC_NUM_THREADS`.
Ускорение рассчитано по метрике `task_run` относительно однопоточного запуска (0.1565 с).

| Потоки (PPC\_NUM\_THREADS) | task\_run (сек) | pipeline (сек) | Ускорение | Эффективность |
| -------------------------- | --------------- | -------------- | --------- | ------------- |
| 1                          | 0.1565          | 0.1895         | 1.00x     | 100%          |
| 2                          | 0.1323          | 0.1608         | 1.18x     | 59%           |
| 4                          | 0.0949          | 0.1418         | 1.65x     | 41%           |
| 8                          | 0.0981          | 0.1627         | 1.60x     | 20%           |

В отличие от чистых STL и TBB-версий, ALL-реализация работает с изображениями, переданными
через `MPI_Scatterv`, что значительно увеличивает объём данных на задачу. Это объясняет
существенно большее абсолютное время (0.16 с против 0.02–0.04 с) и более выраженное
ускорение при распараллеливании.

Наилучший результат достигается на 4 потоках (0.0949 с, ускорение 1.65x). При переходе
к 8 потокам время незначительно возрастает (0.0981 с) — задача исчерпывает параллелизм
раньше, чем число потоков достигает числа физических ядер. Это типичное поведение для
memory-bound задач: оператор Собеля выполняет 9 операций чтения и 1 запись на пиксель при
минимальных арифметических вычислениях, что упирается в пропускную способность памяти.

Метрика `pipeline` стабильно выше `task_run` на ~0.033 с — это накладные расходы MPI-
инфраструктуры (`Scatterv`, `Gatherv`, `Bcast`) и инициализации задачи, не зависящие от
числа потоков.

## 10. Выводы

ALL-реализация успешно объединяет два уровня параллелизма: MPI для распределения строк
между процессами и STL-потоки для параллельной обработки внутри каждого процесса. Ключевым
архитектурным решением является передача ghost rows через `MPI_Scatterv`, что обеспечивает
корректное применение ядра Собеля на границах фрагментов без дополнительных обменов.

Разбиение кода на специализированные функции (`ComputeDistribution`, `ComputeScatterParams`,
`ProcessRowsALL`, `RunThreads`) улучшает читаемость и тестируемость каждого уровня
независимо. Захват переменных по значению в STL-лямбдах исключает случайные гонки данных.

Ускорение 1.65x на 4 потоках при одном MPI-процессе подтверждает эффективность
внутрипроцессного параллелизма. При наличии нескольких MPI-процессов на разных узлах
кластера гибридная схема масштабируется линейно по числу процессов, что делает её
предпочтительным выбором для обработки изображений большого размера на кластерных системах.
