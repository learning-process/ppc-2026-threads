## 1. Введение

Задача обнаружения границ (рёбер) на изображении — одна из базовых задач
компьютерного зрения. Оператор Собеля вычисляет приближённую производную яркости
в двух направлениях: горизонтальном и вертикальном. Резкие перепады яркости
соответствуют границам объектов. Последовательная реализация служит эталоном
корректности и базой для измерения ускорения параллельных версий.

В данном отчёте обобщаются результаты пяти реализаций:

- **SEQ** — последовательная версия, эталон корректности.
- **OMP** — распараллеливание через директивы OpenMP (общая память).
- **TBB** — задачно-ориентированное распараллеливание через `tbb::parallel_for`.
- **STL** — ручное управление потоками с помощью `std::thread`.
- **ALL** — гибрид MPI + STL threads, двухуровневый параллелизм.

Задача хорошо поддаётся распараллеливанию: строки изображения полностью независимы
друг от друга и не требуют синхронизации при записи в выходной вектор.

## 2. Единая постановка задачи

**Входные данные**: контейнер `ImgContainer`, содержащий:

- `width` — ширина изображения в пикселях;
- `height` — высота изображения в пикселях;
- `pixels` — вектор значений пикселей в диапазоне `[0, 255]`.

Контейнер `ImgContainer` унифицирует загрузку данных из любых поддерживаемых
форматов. Цветные изображения `.png`, `.jpg`, `.bmp` загружаются через `stb_image`
и автоматически преобразуются в полутоновые по формуле:

```bash
gray = 0.299 * R + 0.587 * G + 0.114 * B
```

Текстовый формат `.txt` читается напрямую: первая строка содержит `width height`,
далее — значения пикселей. Таким образом, матричные и растровые тесты поступают
в обработчик в одном и том же виде — через поле `pixels` контейнера.

**Выходные данные**: `std::vector<int>` — вектор магнитуд градиента в диапазоне
`[0, 255]`, размером `width * height`.

**Ограничения**: `width > 2`, `height > 2`, все значения пикселей принадлежат
`[0, 255]`. Граничные пиксели остаются нулевыми: для них невозможно применить
полное ядро 3×3.

**Критерий корректности**: побайтовое совпадение выходного вектора параллельной
версии с результатом SEQ-версии на всём наборе функциональных тестов.

## 4. Базовый алгоритм

Алгоритм применяет оператор Собеля к каждому внутреннему пикселю изображения.

Для каждого пикселя `(row, col)`, где `1 <= row < height-1`
и `1 <= col < width-1`, вычисляется свёртка с двумя ядрами 3×3:

```bash
Gx = [[-1, 0, 1],      Gy = [[-1, -2, -1],
      [-2, 0, 2],             [ 0,  0,  0],
      [-1, 0, 1]]             [ 1,  2,  1]]
```

SEQ использует точную формулу магнитуды:

```bash
magnitude = sqrt(Gx^2 + Gy^2)
```

OMP-, STL-, TBB- и ALL-реализации применяют быстрое целочисленное приближение:

```bash
magnitude = (max(|Gx|, |Gy|) * 123 + min(|Gx|, |Gy|) * 51) >> 7
```

Результат ограничивается диапазоном `[0, 255]` и записывается в выходной вектор.

**Временная сложность**: O(width × height). **Память**: O(width × height).

## 6. Детали реализации каждой версии

### SEQ

Класс `ShakirovaESobelEdgeDetectionSEQ`. Обход пикселей в лексикографическом
порядке (строка за строкой). Ядра Собеля объявлены как
`std::array<std::array<int, 3>, 3>` и хранятся локально, что исключает лишние
аллокации. Магнитуда вычисляется через `std::sqrt` с последующим `std::clamp`.

### OMP

Класс `ShakirovaESobelEdgeDetectionOMP`. Внешний цикл по строкам распараллеливается
директивой `#pragma omp parallel for`. Ключевые решения:

- `shared(out, inp)` — `inp` является сырым указателем на `input_`, доступным
  только на чтение; гонки исключены.
- Три указателя `prev/curr/next` вычисляются однократно для каждой строки,
  открывая возможности для автовекторизации компилятором.
- Целочисленное приближение магнитуды вместо `sqrt` устраняет операции
  с плавающей точкой на каждый пиксель.
  
### TBB

Класс `ShakirovaESobelEdgeDetectionTBB`. Использует `tbb::parallel_for` с диапазоном
`tbb::blocked_range<int>(1, h - 1, 8)`, где `8` — grain size, выбранный
эмпирически как баланс между накладными расходами планировщика и локальностью
данных. Захват переменных в лямбду выполняется по значению `[inp, out, w]`, что
исключает случайное разделяемое состояние. Сырые указатели на строки
(`prev`, `curr`, `next`) вычисляются однократно для каждой строки, открывая
возможности для автовекторизации компилятором.

### STL

Класс `ShakirovaESobelEdgeDetectionSTL`. Вычислительная логика вынесена в функцию
`ProcessRows`. Диапазон внутренних строк делится на `num_t` непрерывных фрагментов,
каждый обрабатывается отдельным `std::thread`. Последний поток получает оставшиеся
строки. Гонки данных исключены без мьютексов: потоки пишут в непересекающиеся
диапазоны выходного вектора. Единственная синхронизация — `th.join()`.

### ALL

Класс `ShakirovaESobelEdgeDetectionALL`. Двухуровневый параллелизм:

- **MPI-уровень**: внутренние строки распределяются между процессами через
  `MPI_Scatterv`. Каждый процесс получает свой фрагмент с ghost rows (одна
  дополнительная строка сверху и снизу) для корректного применения ядра Собеля
  на границах. Результаты собираются через `MPI_Gatherv` и рассылаются
  через `MPI_Bcast`.
- **STL-уровень**: каждый MPI-процесс обрабатывает свой фрагмент параллельно
  через `std::thread` (функция `RunThreads`).

Специализированные функции `ComputeDistribution`, `ComputeScatterParams`,
`ProcessRowsALL` и `RunThreads` разделяют ответственность и упрощают тестирование
каждого уровня независимо.

## 8. Интерпретация различий

**SEQ** обеспечивает корректный эталонный baseline. Время растёт линейно с числом
пикселей, многоядерность архитектуры не используется.

**OMP** при `n = 4` показывает ускорение 2.69x относительно SEQ. Прирост достигается
за счёт двух факторов: параллелизма OpenMP по строкам изображения и замены `sqrt`
на целочисленное приближение магнитуды, что устраняет дорогостоящие операции
с плавающей точкой. Использование указателей `prev/curr/next` вместо индексных
вычислений предоставляет компилятору возможности для автовекторизации внутреннего цикла.

**STL** обеспечивает ускорение ~1.2x при 2 потоках и полный контроль над
разбиением данных. Накладные расходы на создание и завершение `std::thread`-потоков
становятся сопоставимы с полезной работой на небольших изображениях. На 4 и 8
потоках время практически не меняется: задача упирается в пропускную способность
памяти, а не в число вычислительных ядер.

**TBB** показывает схожее ускорение (~1.18x на 2 потоках), при этом не требует
ручного управления диапазонами. Планировщик TBB с work-stealing и явным grain size
обеспечивает хорошую локальность кэша. Результаты на 2, 4 и 8 потоках
практически одинаковые (~0.034 с), что подтверждает memory-bound характер задачи:
оператор Собеля выполняет 9 операций чтения и 1 запись на пиксель при минимальных
арифметических вычислениях.

**ALL** при 4 потоках на один MPI-процесс показывает наилучший результат среди всех
версий: 0.0949 с, ускорение 1.65x относительно однопоточного запуска. Существенно
большее абсолютное время по сравнению с STL и TBB объясняется накладными расходами
MPI-инфраструктуры (`MPI_Scatterv`, `MPI_Gatherv`, `MPI_Bcast`). При наличии
нескольких MPI-процессов на разных узлах кластера гибридная схема масштабируется
линейно по числу процессов.

Метрика `pipeline` стабильно превышает `task_run` на постоянную величину,
не зависящую от числа потоков — это накладные расходы инфраструктуры `ppc`
на инициализацию и финализацию задачи.

## 9. Заключение

Все пять реализаций корректно выделяют рёбра на изображениях произвольного размера
и формата. Шесть функциональных тестов успешно проходят при любом числе потоков
и MPI-процессов. Унифицированный контейнер `ImgContainer` обеспечивает единый
интерфейс для матричных (`.txt`) и растровых (`.png`, `.jpg`, `.bmp`) входных
данных без изменений на уровне алгоритма.

Основным ограничением всех реализаций остаётся memory-bound характер задачи:
при малых изображениях накладные расходы параллельной инфраструктуры нивелируют
прирост от параллелизма.

Рекомендации по выбору реализации:

Для одноузловых вычислений с умеренным числом потоков ≤ 8 оптимальным выбором
является **TBB** — за счёт автоматической балансировки нагрузки и хорошей
локальности кэша.

**STL** оправдан при необходимости полного контроля над потоками без зависимости
от сторонних библиотек.

**OMP** обеспечивает наименьший объём дополнительного кода при наличии OpenMP
в проекте.

**Гибридная версия ALL** предпочтительна при обработке изображений большого
размера на кластерных системах с несколькими узлами.
