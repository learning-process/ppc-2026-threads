## 1. Контекст

Исходная задача — применение оператора Собеля к полутоновому изображению для выделения
границ объектов. Последовательная версия служит эталоном корректности и базой для измерения
ускорения параллельных реализаций.

Оператор Собеля вычисляет градиент интенсивности изображения в каждой точке, применяя
свёртку с двумя ядрами 3×3: горизонтальным `Gx` и вертикальным `Gy`. Магнитуда градиента
характеризует резкость перехода интенсивности и используется как мера выраженности границы.

STL-реализация использует ручное управление потоками на основе `std::thread`. В отличие от
TBB, где планировщик автоматически распределяет задачи через work-stealing, здесь диапазон
строк явно делится между потоками, каждый из которых обрабатывает непрерывный фрагмент
без координации с другими. Такой подход даёт полный контроль над разбиением данных и
исключает накладные расходы планировщика.

## 2. Постановка задачи

**Входные данные**: структура `ImgContainer`, содержащая:

- `width`, `height` — размеры изображения;
- `pixels` — вектор значений пикселей в диапазоне `[0, 255]`.

Поддерживаются форматы `.png`, `.jpg`, `.bmp` (загрузка через `stb_image` с автоматическим
преобразованием в полутоновое по формуле `0.299*R + 0.587*G + 0.114*B`), а также текстовый
формат `.txt` (первая строка — размеры, далее — значения пикселей).

**Выходные данные**: `std::vector<int>` — вектор магнитуд градиента в диапазоне `[0, 255]`,
где каждый элемент соответствует пикселю исходного изображения.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги:

1. Для каждого внутреннего пикселя `(row, col)` (граничные пиксели остаются нулевыми)
   считываются указатели на три соседние строки: `prev`, `curr`, `next`.

2. Вычисляются горизонтальный и вертикальный компоненты градиента по развёрнутым ядрам
   Собеля 3×3:

```bash
Gx = -prev[col-1] + prev[col+1] - 2*curr[col-1] + 2*curr[col+1]
     - next[col-1] + next[col+1]

Gy = -prev[col-1] - 2*prev[col] - prev[col+1] + next[col-1]
     + 2*next[col] + next[col+1]
```

3. Магнитуда градиента вычисляется по быстрому целочисленному приближению:

```bash
magnitude = (max(|Gx|, |Gy|) * 123 + min(|Gx|, |Gy|) * 51) >> 7
```

4. Результат ограничивается диапазоном `[0, 255]` и записывается в выходной вектор.

Временная сложность SEQ: O(width × height). Граничные пиксели не обрабатываются и остаются
нулевыми, поскольку для них невозможно применить полное ядро 3×3.

## 4. Схема распараллеливания

**Разбиение данных.** Внутренние строки изображения (от 1 до `h - 2` включительно)
являются естественной единицей параллелизма: каждая строка читает данные из соседних строк,
но не изменяет их, что полностью исключает зависимости между потоками.

Общее количество внутренних строк `inner_rows = h - 2` делится на `num_t` непрерывных
фрагментов примерно равного размера. Для каждого потока с идентификатором `tid`
вычисляются границы его диапазона:

```cpp
const int chunk = inner_rows / num_t;
const int row_begin = (tid * chunk) + 1;
const int row_end = (tid == num_t - 1)
                  ? (inner_rows + 1) : (row_begin + chunk);
```

Последний поток получает все оставшиеся строки, что гарантирует полное покрытие диапазона
без пропусков и перекрытий.

**Создание и завершение потоков.** Потоки создаются через `threads.emplace_back(...)`.
После запуска всех потоков вызывается `th.join()` для каждого из них, что гарантирует
завершение всех вычислений перед тем, как основной поток продолжит выполнение.

**Синхронизация.** Каждый поток записывает результат в независимый непересекающийся
диапазон выходного вектора `out`. Гонки данных полностью исключены без применения мьютексов,
атомарных операций или барьеров. Единственная синхронизация — неявный барьер через `join`.

**Число потоков.** Определяется вызовом `ppc::util::GetNumThreads()`, что позволяет
управлять параллелизмом через переменную окружения `PPC_NUM_THREADS` без изменения кода.

## 5. Детали реализации

**Файлы реализации:**

- `stl/include/ops_stl.hpp` — объявление класса `ShakirovaESobelEdgeDetectionSTL`,
  наследующего `BaseTask`.
- `stl/src/ops_stl.cpp` — реализация методов, включая `RunImpl()` с ручным управлением
  потоками и вспомогательную функцию `ProcessRows`.

**Вычислительная логика** вынесена в отдельную функцию `ProcessRows`, что позволяет
чётко разделить код параллельной инфраструктуры и код обработки пикселей:

```cpp
void ProcessRows(const int *inp, int *out,
                 int w, int row_begin, int row_end) {
  for (int row = row_begin; row < row_end; ++row) {
    const int *prev = inp + (row - 1) * w;
    const int *curr = inp + row * w;
    const int *next = inp + (row + 1) * w;
    int *out_row = out + row * w;

    for (int col = 1; col < w - 1; ++col) {
      const int gx = -prev[col-1] + prev[col+1]
          - 2*curr[col-1] + 2*curr[col+1]
          - next[col-1] + next[col+1];
      const int gy = -prev[col-1] - 2*prev[col] - prev[col+1]
          + next[col-1] + 2*next[col] + next[col+1];

      const int agx = std::abs(gx);
      const int agy = std::abs(gy);
      const int magnitude =
          ((std::max(agx, agy) * 123)
           + (std::min(agx, agy) * 51)) >> 7;
      out_row[col] = magnitude > 255 ? 255 : magnitude;
    }
  }
}
```

**Ключевой фрагмент `RunImpl()`:**

```cpp
const int num_t = ppc::util::GetNumThreads();
std::vector<std::thread> threads;
threads.reserve(static_cast<size_t>(num_t));

for (int tid = 0; tid < num_t; ++tid) {
  threads.emplace_back([inp, out, w, inner_rows, num_t, tid]() {
    const int chunk = inner_rows / num_t;
    const int row_begin = (tid * chunk) + 1;
    const int row_end = (tid == num_t - 1)
                      ? (inner_rows + 1) : (row_begin + chunk);
    ProcessRows(inp, out, w, row_begin, row_end);
  });
}

for (auto &th : threads) th.join();
```

**Отличия от последовательной версии (SEQ):**

1. Цикл по строкам заменён явным созданием потоков с ручным разбиением диапазона.
2. Вычислительная логика вынесена в `ProcessRows` для чистоты разделения ответственности.
3. Лямбда захватывает переменные по значению (`[inp, out, w, inner_rows, num_t, tid]`),
   что исключает случайное разделяемое состояние.

## 8. Результаты производительности

Производительность измерялась с помощью `ppc_perf_tests` при одном MPI-процессе
(`mpirun -n 1`) и разном числе STL-потоков, задаваемом через `PPC_NUM_THREADS`.
Ускорение рассчитано по метрике `task_run` относительно однопоточного запуска (0.0210 с).

| Потоки (PPC\_NUM\_THREADS) | task\_run (сек) | pipeline (сек) | Ускорение | Эффективность |
| -------------------------- | --------------- | -------------- | --------- | ------------- |
| 1                          | 0.0210          | 0.0321         | 1.00x     | 100%          |
| 2                          | 0.0173          | 0.0265         | 1.21x     | 61%           |
| 4                          | 0.0182          | 0.0258         | 1.15x     | 29%           |
| 8                          | 0.0178          | 0.0276         | 1.18x     | 15%           |

Ускорение реальное, но скромное — порядка 1.2x. Это объясняется тем, что тестовые
изображения относительно небольшие: при времени однопоточного выполнения ~0.021 с накладные
расходы на создание `std::thread`-потоков и синхронизацию через `join` становятся
сопоставимы с полезной вычислительной работой. Для более крупных изображений (HD, 4K) те же
накладные расходы составят незначительную долю, и ускорение будет ближе к линейному.

Эффективность использования ядер падает с ростом числа потоков, что типично для задач с
высоким отношением накладных расходов к объёму работы. На 2 потоках эффективность 61% —
наилучший результат в данной конфигурации. При 4 и 8 потоках время практически не меняется:
задача уже целиком умещается в кэш одного ядра, и дополнительные потоки не дают выигрыша.

Метрика `pipeline` стабильно выше `task_run` на ~0.009–0.010 с независимо от числа потоков —
это постоянные накладные расходы инфраструктуры `ppc` на инициализацию и финализацию
задачи, не связанные с параллельностью вычислений.

## 9. Выводы

STL-реализация обеспечивает полный контроль над разбиением данных и управлением потоками.
Выбранная схема — явное деление строк изображения между `std::thread`-потоками — естественно
отражает структуру задачи: строки независимы и не требуют синхронизации при записи в
выходной вектор.

Вынесение вычислительной логики в `ProcessRows` улучшает читаемость кода и упрощает
тестирование отдельных диапазонов. Захват переменных по значению в лямбде исключает
случайные гонки через разделяемые ссылки. Все функциональные тесты пройдены корректно при
любом числе потоков, что подтверждает надёжность реализации.

По сравнению с TBB-версией STL-подход требует явного управления диапазонами, но выигрывает
в отсутствии зависимости от сторонних библиотек и полной прозрачности потоковой модели.
