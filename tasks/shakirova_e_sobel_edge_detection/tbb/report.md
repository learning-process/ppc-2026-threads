## 1. Контекст

Исходная задача — применение оператора Собеля к полутоновому изображению для выделения границ
объектов. Последовательная версия служит эталоном корректности и базой для измерения ускорения
параллельных реализаций.

Оператор Собеля вычисляет градиент интенсивности изображения в каждой точке, применяя свёртку
с двумя ядрами 3×3: горизонтальным `Gx` и вертикальным `Gy`. Магнитуда градиента характеризует
резкость перехода интенсивности и используется как мера выраженности границы.

TBB-реализация использует задачно-ориентированную модель: строки изображения распределяются
между рабочими потоками с помощью механизма work-stealing планировщика TBB. Такой подход
позволяет эффективно задействовать все доступные вычислительные ресурсы без ручного управления
потоками.

## 2. Постановка задачи

**Входные данные**: структура `ImgContainer`, содержащая:

- `width`, `height` — размеры изображения;
- `pixels` — вектор значений пикселей в диапазоне `[0, 255]`.

Поддерживаются форматы `.png`, `.jpg`, `.bmp` (загрузка через `stb_image` с автоматическим
преобразованием в полутоновое по формуле `0.299*R + 0.587*G + 0.114*B`), а также текстовый
формат `.txt` (первая строка — размеры, далее — значения пикселей).

**Выходные данные**: `std::vector<int>` — вектор магнитуд градиента в диапазоне `[0, 255]`,
где каждый элемент соответствует пикселю исходного изображения.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги:

1. Для каждого внутреннего пикселя `(row, col)` (граничные пиксели остаются нулевыми)
   считываются указатели на три соседние строки: `prev`, `curr`, `next`.

2. Вычисляются горизонтальный и вертикальный компоненты градиента по развёрнутым ядрам
   Собеля 3×3:

```bash
Gx = -prev[col-1] + prev[col+1] - 2*curr[col-1] + 2*curr[col+1]
     - next[col-1] + next[col+1]

Gy = -prev[col-1] - 2*prev[col] - prev[col+1] + next[col-1]
     + 2*next[col] + next[col+1]
```

3. Магнитуда градиента вычисляется по быстрому целочисленному приближению:

```bash
magnitude = (max(|Gx|, |Gy|) * 123 + min(|Gx|, |Gy|) * 51) >> 7
```

4. Результат ограничивается диапазоном `[0, 255]` и записывается в выходной вектор.

Временная сложность SEQ: O(width × height). Граничные пиксели не обрабатываются и остаются
нулевыми, поскольку для них невозможно применить полное ядро 3×3.

## 4. Схема распараллеливания

**Выбранный примитив.** Для распараллеливания выбран `tbb::parallel_for`. Внешний цикл по
строкам изображения является естественной единицей параллелизма: строки независимы друг от
друга (каждая строка читает данные из соседних строк, но не изменяет их), что исключает
необходимость синхронизации.

**Диапазон работы.** Диапазон строк задаётся как `tbb::blocked_range<int>(1, h - 1, 8)`,
где `1` и `h - 1` — границы внутренних строк (граничные строки пропускаются), а `8` —
размер зерна (grain size).

**Grain size.** Явно задан `grainsize = 8` строк. Это значение выбрано эмпирически: при
слишком малом зерне накладные расходы планировщика TBB доминируют над полезной работой; при
слишком большом зерне возможен дисбаланс нагрузки. Значение 8 обеспечивает хороший компромисс
для типичных изображений и соответствует кратности кэш-линии процессора, что улучшает
локальность данных.

**Локальность кэша.** Внутри лямбды для каждой строки `row` однократно вычисляются сырые
указатели `prev`, `curr`, `next` на строки изображения. Это позволяет компилятору выполнить
автовекторизацию внутреннего цикла и минимизирует промахи кэша, поскольку данные обрабатываются
последовательно.

**Управление конкуренцией.** Все потоки работают только с непересекающимися диапазонами
выходного вектора `out`, а входные данные (`inp`) используются только для чтения. Гонки
данных полностью исключены без применения явных блокировок.

## 5. Детали реализации

**Файлы реализации:**

- `tbb/include/ops_tbb.hpp` — объявление класса
  `ShakirovaESobelEdgeDetectionTBB`, наследующего `BaseTask`.
- `tbb/src/ops_tbb.cpp` — реализация методов, включая `RunImpl()` с вызовом
  `tbb::parallel_for`.

**Ключевой фрагмент `RunImpl()`:**

```cpp
tbb::parallel_for(
    tbb::blocked_range<int>(1, h - 1, 8),
    [inp, out, w](const tbb::blocked_range<int>& r) {
        for (int row = r.begin(); row < r.end(); ++row) {
            const int* prev = inp + (row - 1) * w;
            const int* curr = inp + row * w;
            const int* next = inp + (row + 1) * w;
            int* out_row = out + row * w;

            for (int col = 1; col < w - 1; ++col) {
                const int gx = -prev[col-1] + prev[col+1]
                    - 2*curr[col-1] + 2*curr[col+1]
                    - next[col-1] + next[col+1];
                const int gy = -prev[col-1] - 2*prev[col] - prev[col+1]
                    + next[col-1] + 2*next[col] + next[col+1];

                const int agx = std::abs(gx);
                const int agy = std::abs(gy);
                const int magnitude =
                    ((std::max(agx, agy) * 123)
                     + (std::min(agx, agy) * 51)) >> 7;
                out_row[col] = magnitude > 255 ? 255 : magnitude;
            }
        }
    }
);
```

**Отличия от последовательной версии (SEQ):**

1. Цикл по строкам заменён вызовом `tbb::parallel_for`, строки обрабатываются параллельно.
2. Введён явный `grainsize = 8` для оптимальной балансировки накладных расходов.
3. Захват переменных в лямбду выполняется по значению (`[inp, out, w]`), что исключает
   случайное разделяемое состояние и помогает компилятору оптимизировать код.

## 8. Результаты производительности

Производительность измерялась с помощью `ppc_perf_tests` при одном MPI-процессе
(`mpirun -n 1`) и разном числе TBB-потоков, задаваемом через `PPC_NUM_THREADS`.
Ускорение рассчитано по метрике `task_run` относительно однопоточного запуска (0.0398 с).

| Потоки (PPC\_NUM\_THREADS) | task\_run (сек) | pipeline (сек) | Ускорение | Эффективность |
| -------------------------- | --------------- | -------------- | --------- | ------------- |
| 1                          | 0.0398          | 0.0483         | 1.00x     | 100%          |
| 2                          | 0.0338          | 0.0435         | 1.18x     | 59%           |
| 4                          | 0.0341          | 0.0424         | 1.17x     | 29%           |
| 8                          | 0.0342          | 0.0432         | 1.16x     | 15%           |

Ускорение реальное, но ограниченное — порядка 1.18x на 2 потоках. Это объясняется тем, что
тестовые изображения относительно небольшие: при времени однопоточного выполнения ~0.040 с
накладные расходы TBB на инициализацию пула потоков, разбиение `blocked_range` и объединение
результатов становятся сопоставимы с полезной работой. Для изображений большего размера
(HD, 4K) накладные расходы составят незначительную долю и ускорение будет существенно выше.

Характерная особенность TBB-результатов — время `task_run` на 2, 4 и 8 потоках практически
одинаковое (~0.034 с). Это означает, что задача уже на 2 потоках достигает предела,
определяемого пропускной способностью памяти и кэша, а не числом вычислительных ядер.
Такое поведение типично для задач с высокой долей операций чтения/записи относительно
объёма вычислений (оператор Собеля — именно такая задача: 9 операций чтения, 2 свёртки,
1 запись на пиксель).

Метрика `pipeline` стабильно выше `task_run` на ~0.009 с независимо от числа потоков —
это постоянные накладные расходы инфраструктуры `ppc`, не связанные с параллельностью.

## 9. Выводы

Для TBB-реализации успешно произведено распараллеливание оператора Собеля с использованием
задачно-ориентированной модели. Выбранный примитив `tbb::parallel_for` естественно отражает
структуру задачи: строки изображения полностью независимы и обрабатываются без синхронизации.
Явный `grainsize = 8` обеспечивает оптимальный баланс между накладными расходами планировщика
и равномерностью нагрузки.

Использование сырых указателей на строки внутри лямбды улучшает локальность кэша и открывает
возможности для автовекторизации компилятором. Все функциональные тесты пройдены корректно при
любом числе потоков, что подтверждает отсутствие гонок данных в реализации.
