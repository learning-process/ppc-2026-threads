## 1. Контекст

Это версия без OpenMP-директив и примитивов TBB. Всё распараллеливание сделано через `std::thread`,
а синхронизация между двумя фазами сделана через `std::barrier`. Цель — показать,
что та же задача решается чистой стандартной библиотекой, и заодно зафиксировать цену ручного управления потоками.

---

## 2. Постановка задачи

**Вход.** `std::vector<uint8_t>` — полутоновый кадр, пиксели идут подряд (строка за строкой).
Допустимы значения от 0 до 255 включительно. Пустой вектор — ошибка.

**Выход.** `std::vector<uint8_t>` той же длины. Контраст растянут на полный диапазон 0…255.

**Что нужно сделать.**

1. Найти по всему кадру минимальное `min_el` и максимальное `max_el` значение пикселя.
2. Если `min_el == max_el`, кадр однотонный — выход совпадает с входом.
3. Иначе каждый пиксель пересчитывается по формуле линейной растяжки.
   В коде для этого заранее строится `lut` на 256 значений: значения ниже `min_el` переходят в 0,
   значения выше `max_el` переходят в 255, остальные считаются как `(in - min_el) * 255 / (max_el - min_el)`.

**Ограничения и крайние случаи.** Пустой вход блокируется в `ValidationImpl`.
Однотонный кадр требует побайтового совпадения выхода с входом.
После растяжки минимум и максимум на выходе должны выйти к краям диапазона (на тестах — не хуже 1 и 254).

---

## 3. Базовый алгоритм

Два линейных прохода по массиву. Первый — поиск min и max; второй — пересчёт пикселей по формуле растяжки.
На однотонном кадре второй проход заменяется копированием. Подробное описание лежит в [seq/report.md](../seq/report.md).

---

## 4. Схема распараллеливания

Число потоков берётся из `ppc::util::GetNumThreads()`, но используется не всегда напрямую.
Сначала выбирается реальное число потоков через `ChooseThreadCount`.
На коротких массивах (длина < 100 000) или при одном потоке весь алгоритм остаётся последовательным.
Также число потоков ограничивается размером данных и минимальным размером блока.

**Общий принцип.** В параллельной ветке потоки создаются один раз.
Каждый поток сначала ищет локальные `min` и `max` на своём диапазоне.
Потом все потоки доходят до `std::barrier`.
В completion-функции барьера один раз считается общий `min/max` и строится `lut`.
После барьера те же самые потоки сразу обрабатывают свои диапазоны выхода.

**Поиск min и max.** Массив режется на почти равные блоки по числу потоков.
Каждый поток проходит **свой** диапазон и пишет результат в **свою** ячейку предвыделенного вектора `locals`.
В одной ячейке хранится и минимум, и максимум этого потока: `LocalMinMax`.
После того как все потоки дошли до барьера, completion-функция проходит по `locals` и получает общий минимум и общий максимум.

**Однотонный кадр.** Если общий минимум равен общему максимуму, в completion-функции выставляется флаг `copy_only`.
После барьера каждый поток копирует свой диапазон из входа в выход. Отдельный второй запуск потоков для этого не нужен.

**Растяжка через LUT.** `lut` на 256 значений строится один раз в completion-функции барьера.
Для этого используется целочисленная формула.
После барьера каждый поток читает пиксели из своего диапазона и пишет результат в соответствующий диапазон `output`.

**Карта поток → диапазон.** Для `t = 0..thread_count-1`:
`begin = n * t / thread_count`, `end = n * (t + 1) / thread_count`.
Так остаток от деления распределяется между потоками равномернее, а не отдаётся только последнему потоку.

**Порядок `create → barrier → join`.** В параллельной ветке сначала **в одном цикле**
все потоки кладутся в `std::vector<std::thread>`.
Внутри каждого потока есть две части работы: поиск локального `min/max`, затем ожидание на `std::barrier`,
затем применение результата к своему диапазону.
После создания всех потоков главный поток вызывает `join()` для каждого из них.

**Синхронизация.** `mutex`, `atomic` и `condition_variable` не используются. Между фазами стоит `std::barrier`.
До барьера каждый поток пишет только в свою ячейку `locals`.
В completion-функции барьера один поток строит общий результат.
После барьера каждый поток пишет только в свой диапазон `output`. Гонок по индексам нет.

**Ключевой фрагмент `RunImpl`:**

```cpp
// File: stl/src/ops_stl.cpp
bool BatkovFContrastEnhLinHistStretchSTL::RunImpl() {
  const auto &input = GetInput();
  auto &output = GetOutput();

  const size_t requested_threads = static_cast<size_t>(std::max(1, ppc::util::GetNumThreads()));
  StretchContrast(input, output, requested_threads);

  return true;
}
```

**Ключевой фрагмент параллельной части:**

```cpp
// File: stl/src/ops_stl.cpp
auto completion = [&locals, &lut, &copy_only]() noexcept {
  const LocalMinMax global = CombineLocalMinMax(locals);
  copy_only = global.min_val == global.max_val;

  if (!copy_only) {
    lut = BuildStretchLut(global.min_val, global.max_val);
  }
};

std::barrier<decltype(completion)> barrier(
    static_cast<std::ptrdiff_t>(thread_count), completion);

for (size_t thread_index = 0; thread_index < thread_count; ++thread_index) {
  threads.emplace_back([&, thread_index]() {
    const Range range = MakeRange(size, thread_index, thread_count);
    locals.at(thread_index) = FindLocalMinMax(input, range);
    barrier.arrive_and_wait();
    ApplyResultRange(input, output, range, copy_only, lut);
  });
}
```

Здесь видна вся идея STL-версии: потоки создаются один раз, а две фазы работы разделяются барьером.
Каждый поток работает с замыканием на `input`, `output`, `locals`, `lut` и индекс своего поддиапазона.

## 5. Детали реализации

**Файлы:**

- `stl/include/ops_stl.hpp` — класс `BatkovFContrastEnhLinHistStretchSTL`.
- `stl/src/ops_stl.cpp` — функции `ChooseThreadCount`, `MakeRange`, `BuildStretchLut`, `FindLocalMinMax`,
  `CombineLocalMinMax`, `ApplyLutRange`, `StretchSequential`, `StretchParallel`, `StretchContrast`
  в анонимном namespace, плюс реализация пайплайна PPC.

**Что изменилось относительно SEQ.**

- Поиск min/max — ручное разбиение диапазона на блоки и параллельная редукция через локальные min/max.
- Вместо счёта формулы для каждого пикселя строится `lut` на 256 значений.
- Поиск `min/max` и применение `lut` выполняются в одном наборе потоков.
- Между двумя фазами используется `std::barrier`.
- Для маленьких входов и для одного потока используется последовательная ветка.

**Где могут возникнуть гонки.** Принципиально нет: каждая запись идёт в индекс или диапазон,
который не использует другой поток. Гонок на чтении `input` и `lut` тоже нет — это только чтение.
`lut` и `copy_only` записываются до выхода потоков из барьера, а читаются уже после него.

## 6. Проверка корректности

**Где лежат тесты.** `tests/functional/main.cpp` — функциональные тесты (класс `BatkovFRunFuncTestsThreads`).
`tests/performance/main.cpp` — перф-тесты (класс `BatkovFRunPerfTestThreads`). Оба набора прогоняют общий пайплайн PPC:
`Validation` → `PreProcessing` → `Run` → `PostProcessing`, затем проверяют выход.

**Что прогоняется в функциональных тестах.** Случайные слабоконтрастные квадратные кадры размером 100×100, 500×500,
1000×1000 и 2000×2000. Отдельный сценарий `degenerate_uniform` — кадр 256×256 из одинакового серого:
выход обязан побайтово совпасть со входом.

**Критерий корректности.** Либо побайтовое равенство (для однотонного кадра),
либо `min_out ≤ 1` и `max_out ≥ 254` после растяжки.

**Что проверяется в перф-тестах.** Тот же по смыслу критерий выхода,
но на одном детерминированном квадрате со стороной **25000** пикселей.

**Сравнение с SEQ.** Результат строится по тем же границам `min_el` и `max_el`.
В STL-версии пересчёт вынесен в `lut`, поэтому во втором проходе не выполняется деление для каждого пикселя.
Для однотонного кадра выход побайтово совпадает с входом.

| Потоки | Время, с | Ускорение | Эффективность |

| 1      | 0,458    | 1,66      | 166,4 %       |

На одном потоке STL уже быстрее SEQ, потому что используется LUT и пересчёт каждого пикселя становится проще.
При переходе к двум потокам видно резкое ускорение: время уменьшается с 0,458 с до 0,105 с.
Дальше рост сохраняется, но становится менее выраженным: на 4, 6 и 8 потоках время улучшается уже постепенно.
Это ожидаемо для алгоритма с двумя линейными проходами по памяти. Получили достаточно хороший резульат,
все благодаря тому, что потоки создаются один раз, а синхронизация между поиском min/max и растяжкой выполняется через std::barrier.

## 9. Выводы

По результатам видно, что даже на одном потоке STL быстрее SEQ за счёт использования LUT.
Максимальное ускорение достигается на 8 потоках и составляет 14,94 раза относительно SEQ.
При этом после 4 потоков прирост становится меньше,
так как алгоритм в основном ограничен скоростью чтения и записи памяти.
Такая версия остаётся простой, использует только стандартную библиотеку C++ и хорошо показывает,
как можно эффективно распараллелить задачу без OpenMP и TBB.
