## 1. Контекст

Это «инженерная» версия: никаких OpenMP-директив и никаких примитивов TBB.
Всё распараллеливание сделано руками через `std::thread`. Цель — показать,
что та же задача решается чистой стандартной библиотекой, и заодно зафиксировать цену ручного управления потоками.

---

## 2. Постановка задачи

**Вход.** `std::vector<uint8_t>` — полутоновый кадр, пиксели идут подряд (строка за строкой).
Допустимы значения от 0 до 255 включительно. Пустой вектор — ошибка.

**Выход.** `std::vector<uint8_t>` той же длины. Контраст растянут на полный диапазон 0…255.

**Что нужно сделать.**

1. Найти по всему кадру минимальное `min_el` и максимальное `max_el` значение пикселя.
2. Если `min_el == max_el`, кадр однотонный — выход совпадает с входом.
3. Иначе по формуле `out = clamp(a*in + b, 0, 255)` каждый пиксель пересчитывается,
где `a = 255 / (max_el − min_el)`, `b = −a · min_el`.

**Ограничения и крайние случаи.** Пустой вход блокируется в `ValidationImpl`.
Однотонный кадр требует побайтового совпадения выхода с входом.
После растяжки минимум и максимум на выходе должны выйти к краям диапазона (на тестах — не хуже 1 и 254).

---

## 3. Базовый алгоритм

Два линейных прохода по массиву. Первый — поиск min и max; второй — пересчёт пикселей по формуле растяжки.
На однотонном кадре второй проход заменяется копированием. Подробное описание лежит в [seq/report.md](../seq/report.md).

---

## 4. Схема распараллеливания

Число потоков: `max(1, ppc::util::GetNumThreads())`. На коротких массивах (длина < 100 000)
или при одном потоке оба прохода остаются последовательными.

**Поиск min и max.** Массив режется на почти равные блоки по числу потоков;
последний поток добирает хвост (`block = n / num_threads`).
Каждый поток проходит **свой** диапазон и пишет результат в **свою** ячейку двух предвыделенных векторов:
`mins[thread_index]` и `maxs[thread_index]`. После того как все потоки завершились,
общий минимум — это `min_element(mins)`, общий максимум — `max_element(maxs)`.

**Однотонный кадр.** Параллельный регион не запускается; вход копируется в выход через `std::ranges::copy`.

**Растяжка через LUT.** В одном потоке (главном) считается `lut` на 256 значений в `float` (как в TBB-версии,
только во `float`, а не в `double` — это не влияет на результат,
потому что итог всё равно идёт через `clamp` к `uint8_t`). Затем массив снова режется на те же по логике диапазоны
и тем же набором потоков пишется `output[i] = lut[input[i]]`.

**Карта поток → диапазон.** Для `t = 0..num_threads-1`:
`begin = t * block`, `end = (t == num_threads-1) ? n : begin + block`.
Так последний поток забирает остаток от деления.

**Порядок `create → join`.** В обоих параллельных участках сначала **в одном цикле** все потоки кладутся
в `std::vector<std::thread>`, и только **во втором цикле** у всех вызывается `join()`.
Это важно: вызывать `join` сразу в цикле создания нельзя — это сериализует работу. Здесь такой ошибки нет.

**Синхронизация.** Ни `mutex`, ни `atomic`, ни `condition_variable` не используются.
Это специально: в первом проходе каждый поток пишет в **разные** ячейки `mins`/`maxs`,
во втором — в **разные** ячейки `output`. Никаких пересечений по индексам нет, поэтому защита не нужна.

**Ключевой фрагмент `RunImpl` (растяжка):**

```cpp
// File: stl/src/ops_stl.cpp
std::vector<std::thread> threads;
threads.reserve(num_threads);
for (size_t thread_index = 0; thread_index < num_threads; ++thread_index) {
  const size_t begin = thread_index * block;
  const size_t end = (thread_index == num_threads - 1) ? n : begin + block;
  threads.emplace_back([&, begin, end]() {
    for (size_t i = begin; i < end; ++i) {
      output[i] = lut.at(input[i]);
    }
  });
}
for (auto &th : threads) {
  th.join();
}
```

Здесь видна вся идея STL-версии: сначала создаются все потоки, потом — общий `join`.
Каждый поток работает с замыканием на `input`, `output`, `lut` и индексами своего поддиапазона.

## 5. Детали реализации

**Файлы:**

- `stl/include/ops_stl.hpp` — класс `BatkovFContrastEnhLinHistStretchSTL`.
- `stl/src/ops_stl.cpp` — функции `FindMinMax`, `FindMinMaxParallel`, `BuildStretchLut`, `ApplyStretchParallel`
в анонимном namespace, плюс реализация пайплайна PPC.

**Что изменилось относительно SEQ.**

- Поиск min/max — ручное разбиение диапазона на блоки и параллельная редукция
  через локальные min/max (на одной ячейке на поток).
- Вместо счёта в `double` для каждого пикселя строится `lut` на 256 значений.
- Второй проход — ручное разбиение тем же набором потоков.

**Где могут возникнуть гонки.** Принципиально нет: каждая запись идёт в индекс, который не использует другой поток.
Гонок на чтении `input` и `lut` тоже нет — это только чтение.

## 6. Проверка корректности

**Где лежат тесты.** `tests/functional/main.cpp` — функциональные тесты (класс `BatkovFRunFuncTestsThreads`).
`tests/performance/main.cpp` — перф-тесты (класс `BatkovFRunPerfTestThreads`).
Оба набора прогоняют общий пайплайн PPC: `Validation` → `PreProcessing` → `Run` → `PostProcessing`, затем проверяют выход.

**Что прогоняется в функциональных тестах.** Случайные слабоконтрастные квадратные кадры размером 100×100, 500×500,
1000×1000 и 2000×2000. Отдельный сценарий `degenerate_uniform` — кадр 256×256 из одинакового серого:
выход обязан побайтово совпасть со входом.

**Критерий корректности.** Либо побайтовое равенство (для однотонного кадра),
либо `min_out ≤ 1` и `max_out ≥ 254` после растяжки.

**Что проверяется в перф-тестах.** Тот же по смыслу критерий выхода,
но на одном детерминированном квадрате со стороной **25000** пикселей.

**Сравнение с SEQ.** LUT построен по тем же `a` и `b` (только во `float`,
что для байтового результата неотличимо от `double` после `clamp`).
Результат побайтово совпадает с SEQ.

## 9. Выводы

Ручное разбиение диапазонов и два набора потоков дают заметное ускорение без OpenMP и без TBB.
На восьми потоках время хуже, чем у TBB на той же машине: у STL нет планировщика TBB
и нет такого же сочетания `parallel_reduce` и крупного зерна. Зато код предельно простой,
легко читается и хорошо подходит как пример «как параллелить руками без рантайма».
