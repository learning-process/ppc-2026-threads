## 1. Контекст

Это версия задачи на oneTBB. Она интересна по двум причинам: для поиска min/max используется не директива OpenMP,
а `parallel_reduce`; для второго прохода вместо счёта в `double` для каждого пикселя строится **LUT**
на 256 заранее посчитанных значений, а сам обход кадра идёт через `parallel_for` с `blocked_range` и явным `static_partitioner`.

---

## 2. Постановка задачи

**Вход.** `std::vector<uint8_t>` — полутоновый кадр, пиксели идут подряд (строка за строкой).
Допустимы значения от 0 до 255 включительно. Пустой вектор — ошибка.

**Выход.** `std::vector<uint8_t>` той же длины. Контраст растянут на полный диапазон 0…255.

**Что нужно сделать.**

1. Найти по всему кадру минимальное `min_el` и максимальное `max_el` значение пикселя.
2. Если `min_el == max_el`, кадр однотонный — выход совпадает с входом.
3. Иначе по формуле `out = clamp(a*in + b, 0, 255)` каждый пиксель пересчитывается, где `a = 255 / (max_el − min_el)`,
`b = −a · min_el`.

**Ограничения и крайние случаи.** Пустой вход блокируется в `ValidationImpl`.
Однотонный кадр требует побайтового совпадения выхода с входом. После растяжки минимум и максимум на выходе
должны выйти к краям диапазона (на тестах — не хуже 1 и 254).

---

## 3. Базовый алгоритм

Два линейных прохода по массиву. Первый — поиск min и max; второй — пересчёт пикселей по формуле растяжки.
На однотонном кадре второй проход заменяется копированием. Подробное описание лежит в [seq/report.md](../seq/report.md).

---

## 4. Схема распараллеливания

Версия использует два разных примитива oneTBB на двух разных проходах.
Для коротких массивов (длина < 100 000) первый проход остаётся последовательным.

**Поиск min и max.** Для длинных массивов — `oneapi::tbb::parallel_reduce` по `blocked_range<size_t>(0, n)`.
Каждый чанк держит свою пару `(min, max)`; объединение пар делает функция-объединитель:
`(min_a, max_a) ⊕ (min_b, max_b) → (min(min_a,min_b), max(max_a,max_b))`.
Стартовое значение редукции — `(uint8_t::max, uint8_t::min)`, чтобы любой первый реальный пиксель уменьшил `min`
и увеличил `max`. Размер чанка не задан явно — берётся стандартное разбиение `blocked_range`.

**Однотонный кадр.** Параллельный регион не запускается; вход копируется в выход через `std::ranges::copy`.

**Растяжка через LUT.** В одном потоке заполняется массив `lut` на 256 значений в `double` (та же формула, что в SEQ,
но всего 256 раз — это копейки). После этого `oneapi::tbb::parallel_for` обходит `blocked_range<size_t>(0, n, kGrain)`
с зерном `kGrain = 1 << 17` (131072 пикселя) и `static_partitioner`.
Внутри тела для каждого `i` берётся байт из `input`, по нему — значение из `lut`, и оно пишется в `output`.

**Почему такой grainsize.** 131072 пикселя на чанк — это компромисс. С одной стороны, чанк должен быть крупным,
чтобы накладные расходы планировщика были маленькими по сравнению с полезной работой.
С другой — чанков должно остаться достаточно, чтобы balancing не страдал. На кадре 25000×25000
при 8 потоках чанков получается несколько тысяч на поток.

**Почему `static_partitioner`.** Работа на каждом байте одинаковая — одна выборка из LUT и одна запись.
Балансировка через voracious-стиль не нужна: статическое разбиение даёт более предсказуемое
и воспроизводимое время от запуска к запуску.

**Где нет гонок.** Каждый поток пишет в свой подынтервал `output`; чтения `input` и `lut` — только чтения.
Гонок на `min`/`max` нет благодаря механизму `parallel_reduce`.

**Откуда берётся число потоков.** Через `PPC_NUM_THREADS` runner курса задаёт ограничение конкуренции.
В коде версии явный `global_control::max_allowed_parallelism` не выставляется,
но рантайм TBB читает соответствующие настройки окружения, выставленные средой PPC.

**Ключевой фрагмент `RunImpl`:**

```cpp
// File: tbb/src/ops_tbb.cpp
constexpr size_t kGrain = 1 << 17;
oneapi::tbb::static_partitioner part;

oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<size_t>(0, input.size(), kGrain),
    [&](const oneapi::tbb::blocked_range<size_t> &r) {
      const uint8_t *src = input.data() + r.begin();
      uint8_t *dst = output.data() + r.begin();
      for (size_t i = 0; i < r.size(); ++i) {
        dst[i] = lut.at(src[i]);
      }
    },
    part);
```

Внутри лямбды — указатели на свой подынтервал и обычный обход. Никаких атомиков и блокировок не нужно:
каждый поток пишет в непересекающийся участок `output`.

## 5. Детали реализации

**Файлы:**

- `tbb/include/ops_tbb.hpp` — класс `BatkovFContrastEnhLinHistStretchTBB`.
- `tbb/src/ops_tbb.cpp` — функции `FindMinMax`, `FindMinMaxParallel` в анонимном namespace и реализация пайплайна PPC.

**Что изменилось относительно SEQ.**

- Поиск min/max — через `parallel_reduce` (для длинных массивов).
- Вместо счёта в `double` для каждого пикселя строится `lut` на 256 значений.
- Второй проход — `parallel_for` с `blocked_range`, явным grainsize и `static_partitioner`.

**Локальные данные.** В `parallel_reduce` каждый поток работает со своей парой (min, max).
В `parallel_for` каждый поток работает со своим участком `[r.begin(), r.end())`. Общий `lut` только читается.

**Где могут возникнуть узкие места.** На больших кадрах задача упирается в память: одно чтение из `input`,
одно чтение из `lut[256]`, одна запись в `output`. LUT почти всегда лежит в L1-кэше,
но сама пропускная способность DRAM на длинных кадрах становится узким местом раньше, чем CPU.

## 6. Проверка корректности

**Где лежат тесты.** `tests/functional/main.cpp` — функциональные тесты (класс `BatkovFRunFuncTestsThreads`).
`tests/performance/main.cpp` — перф-тесты (класс `BatkovFRunPerfTestThreads`).
Оба набора прогоняют общий пайплайн PPC: `Validation` → `PreProcessing` → `Run` → `PostProcessing`, затем проверяют выход.

**Что прогоняется в функциональных тестах.** Случайные слабоконтрастные квадратные кадры размером 100×100, 500×500,
1000×1000 и 2000×2000. Отдельный сценарий `degenerate_uniform` — кадр 256×256 из одинакового серого:
выход обязан побайтово совпасть со входом.

**Критерий корректности.** Либо побайтовое равенство (для однотонного кадра),
либо `min_out ≤ 1` и `max_out ≥ 254` после растяжки.

**Что проверяется в перф-тестах.** Тот же по смыслу критерий выхода,
но на одном детерминированном квадрате со стороной **25000** пикселей.

**Сравнение с SEQ.** Так как LUT строится по точно тем же `a` и `b`, что в SEQ,
разница только в порядке вычислений (256 значений посчитаны заранее, а не на лету).
Результат побайтово совпадает с SEQ.

## 9. Выводы

На этом железе и этом кадре TBB даёт самое малое время среди потоковых вариантов.
Сочетание `parallel_reduce` для min/max, LUT и `parallel_for` с крупным зерном
плюс `static_partitioner` хорошо подходит к задаче «два линейных прохода по большому массиву»:
 оно даёт большой выигрыш уже на одном потоке (за счёт LUT) и предсказуемо масштабируется до 8 потоков.
