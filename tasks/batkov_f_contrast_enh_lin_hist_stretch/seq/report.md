## 2. Постановка задачи

**Вход.** `std::vector<uint8_t>` — полутоновый кадр, пиксели идут подряд (строка за строкой).
Допустимы значения от 0 до 255 включительно. Пустой вектор — ошибка.

**Выход.** `std::vector<uint8_t>` той же длины. Контраст растянут на полный диапазон 0…255.

**Что нужно сделать.**

1. Найти по всему кадру минимальное `min_el` и максимальное `max_el` значение пикселя.
2. Если `min_el == max_el`, кадр однотонный — выход совпадает с входом.
3. Иначе по формуле `out = clamp(a*in + b, 0, 255)` каждый пиксель пересчитывается,
где `a = 255 / (max_el − min_el)`, `b = −a · min_el`.

**Ограничения и крайние случаи.** Пустой вход блокируется в `ValidationImpl`.
Однотонный кадр требует побайтового совпадения выхода с входом.
После растяжки минимум и максимум на выходе должны выйти к краям диапазона (на тестах — не хуже 1 и 254).

---

## 3. Базовый алгоритм

Алгоритм состоит из двух последовательных линейных проходов по массиву.

1. Один проход `std::ranges::minmax_element` — получаем `min_el`, `max_el`.
2. Если максимум больше минимума, считаются коэффициенты `a` и `b` в `double`.
Затем второй цикл пересчитывает каждый пиксель и пишет результат через `std::clamp` в выходной вектор.
3. Если минимум равен максимуму, делается `std::ranges::copy` входа в выход.

**Сложность по времени:** O(N), где N — число пикселей. Два прохода по массиву.
**Сложность по памяти:** O(N) — отдельный выходной буфер той же длины.

## 4. Детали реализации

**Файлы:**

- `seq/include/ops_seq.hpp` — объявление класса `BatkovFContrastEnhLinHistStretchSEQ`.
- `seq/src/ops_seq.cpp` — реализация. Класс наследует `BaseTask` и переопределяет четыре метода пайплайна PPC.

**Что делают методы пайплайна.**

- `ValidationImpl` — возвращает `false`, если вход пустой.
- `PreProcessingImpl` — делает `output.resize(input.size())`.
- `RunImpl` — основной алгоритм (см. фрагмент ниже).
- `PostProcessingImpl` — проверяет, что выход не пустой.

**Ключевой фрагмент `RunImpl`:**

```cpp
// File: seq/src/ops_seq.cpp
auto [min_it, max_it] = std::ranges::minmax_element(GetInput());
uint8_t min_el = *min_it;
uint8_t max_el = *max_it;

if (max_el > min_el) {
  double a = 255.0 / (max_el - min_el);
  double b = -a * min_el;
  for (size_t i = 0; i < input.size(); ++i) {
    double new_pixel = (a * static_cast<double>(input[i])) + b;
    output[i] = static_cast<uint8_t>(std::clamp(new_pixel, 0.0, 255.0));
  }
} else {
  std::ranges::copy(input, output.begin());
}
```

В этом фрагменте видно сразу обе ветки: «обычный кадр» и «однотонный кадр».
Все вычисления второго прохода идут в `double`;
LUT здесь намеренно не используется — это делает версию максимально простой и читаемой как эталон.

## 5. Проверка корректности

**Где лежат тесты.** `tests/functional/main.cpp` — функциональные тесты (класс `BatkovFRunFuncTestsThreads`).
`tests/performance/main.cpp` — перф-тесты (класс `BatkovFRunPerfTestThreads`).
Оба набора прогоняют общий пайплайн PPC: `Validation` → `PreProcessing` → `Run` → `PostProcessing`, затем проверяют выход.

**Что прогоняется в функциональных тестах.** Случайные слабоконтрастные квадратные кадры размером 100×100, 500×500,
1000×1000 и 2000×2000. Отдельный сценарий `degenerate_uniform` — кадр 256×256 из одинакового серого:
выход обязан побайтово совпасть со входом.

**Критерий корректности.** Либо побайтовое равенство (для однотонного кадра),
либо `min_out ≤ 1` и `max_out ≥ 254` после растяжки.

**Что проверяется в перф-тестах.** Тот же по смыслу критерий выхода,
но на одном детерминированном квадрате со стороной **25000** пикселей (константа `kImageSize`).

## 8. Выводы

SEQ задаёт смысл задачи и эталонное время. Алгоритм состоит из двух линейных проходов по массиву,
обе ветки (растяжка и копирование) обрабатываются в одном `RunImpl`.
Эффективность для одной строки таблицы не считается, потому что параметра «число потоков» нет.
Все остальные отчёты используют именно это число — **0,76217 с** — как baseline для ускорения.
