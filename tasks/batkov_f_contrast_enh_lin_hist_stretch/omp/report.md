## 1. Контекст

Это первая параллельная версия. Вычислительное ядро остаётся таким же, как в SEQ,
но оба линейных прохода по кадру (поиск min/max и растяжка) переносятся в OpenMP-регионы.
Цель — получить понятный baseline по ускорению через директивы `#pragma omp parallel for`.

---

## 2. Постановка задачи

**Вход.** `std::vector<uint8_t>` — полутоновый кадр, пиксели идут подряд (строка за строкой).
Допустимы значения от 0 до 255 включительно. Пустой вектор — ошибка.

**Выход.** `std::vector<uint8_t>` той же длины. Контраст растянут на полный диапазон 0…255.

**Что нужно сделать.**

1. Найти по всему кадру минимальное `min_el` и максимальное `max_el` значение пикселя.
2. Если `min_el == max_el`, кадр однотонный — выход совпадает с входом.
3. Иначе по формуле `out = clamp(a*in + b, 0, 255)` каждый пиксель пересчитывается,
где `a = 255 / (max_el − min_el)`, `b = −a · min_el`.

**Ограничения и крайние случаи.** Пустой вход блокируется в `ValidationImpl`.
Однотонный кадр требует побайтового совпадения выхода с входом.
После растяжки минимум и максимум на выходе должны выйти к краям диапазона (на тестах — не хуже 1 и 254).

---

## 3. Базовый алгоритм

Два линейных прохода по массиву. Первый — поиск min и max; второй — пересчёт пикселей по формуле растяжки.
На однотонном кадре второй проход заменяется копированием. Подробное описание лежит в [seq/report.md](../seq/report.md).

---

## 4. Схема распараллеливания

Параллельные регионы — два, на оба линейных прохода кадра.
Для коротких массивов оба прохода остаются последовательными (порог 100 000 пикселей).

**Поиск min и max.** Для длин не меньше 100 000 — один цикл `#pragma omp parallel for` с двумя редукциями:
`reduction(min : min_el)` и `reduction(max : max_el)`.
У каждого потока свои локальные min и max; в конце региона OpenMP сам собирает общий результат.
Список переменных задан через `default(none)`, массив входа и его длина помечены как `shared`.
Schedule не указан явно — берётся стандартный для рантайма.

**Растяжка.** Второй регион — `#pragma omp parallel for` по индексам.
Внутри тело то же, что в SEQ: пересчёт в `double` и `clamp`.
Переменные `a` и `b` помечены как `shared`, гонок на них нет — они только читаются.

**Однотонный кадр.** Параллельный регион не запускается; делается обычный `std::ranges::copy`.

**Барьеры.** В конце каждого `parallel for` есть неявный барьер: второй цикл не начнётся, пока не закончится первый.
Это безопасно: `output` использует `min_el` и `max_el`, которые получены в первом регионе.

**Откуда берётся число потоков.** Runner курса экспортирует `PPC_NUM_THREADS` и через него же `OMP_NUM_THREADS`.
То есть число потоков задаётся снаружи — никаких `omp_set_num_threads` в коде нет.

**Ключевой фрагмент `RunImpl`:**

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(input, input_size) \
                         reduction(min : min_el) reduction(max : max_el)
for (int64_t i = 0; i < input_size; ++i) {
  min_el = std::min(input[i], min_el);
  max_el = std::max(input[i], max_el);
}
```

Здесь `default(none)` обязывает явно описать каждую переменную (это ловит лишние «утечки» переменных в регион).
`min_el` и `max_el` приватны для каждого потока через `reduction`, в конце региона сводятся к одному значению.
Гонки на `min_el`/`max_el` исключены самим механизмом редукции.

## 5. Детали реализации

**Файлы:**

- `omp/include/ops_omp.hpp` — класс `BatkovFContrastEnhLinHistStretchOMP`.
- `omp/src/ops_omp.cpp` — функции `FindMinMax` и `FindMinMaxParallel` в анонимном namespace, плюс реализация пайплайна PPC.

**Что изменилось относительно SEQ.**

- Поиск min/max вынесен в отдельную функцию `FindMinMax` с порогом параллелизма (для коротких массивов работает `std::ranges::minmax_element`).
- Цикл растяжки помечен `#pragma omp parallel for`.
- Сами формулы (`a`, `b`, `clamp(a*x + b, 0, 255)`) и тип данных в формуле (`double`) не менялись.

**Где могут возникнуть гонки.** На записи `output[i]` гонок нет:
каждый поток работает со своим уникальным индексом `i`. На чтении `input[i]` гонок нет:
чтение одного и того же массива не модифицирует данные. На редукции min/max гонок нет:
рантайм OpenMP сам синхронизирует частичные результаты.

## 6. Проверка корректности

**Где лежат тесты.** `tests/functional/main.cpp` — функциональные тесты (класс `BatkovFRunFuncTestsThreads`).
`tests/performance/main.cpp` — перф-тесты (класс `BatkovFRunPerfTestThreads`).
Оба набора прогоняют общий пайплайн PPC: `Validation` → `PreProcessing` → `Run` → `PostProcessing`, затем проверяют выход.

**Что прогоняется в функциональных тестах.** Случайные слабоконтрастные квадратные кадры размером 100×100, 500×500,
1000×1000 и 2000×2000. Отдельный сценарий `degenerate_uniform` — кадр 256×256 из одинакового серого:
выход обязан побайтово совпасть со входом.

**Критерий корректности.** Либо побайтовое равенство (для однотонного кадра),
либо `min_out ≤ 1` и `max_out ≥ 254` после растяжки.

**Что проверяется в перф-тестах.** Тот же по смыслу критерий выхода,
но на одном детерминированном квадрате со стороной **25000** пикселей.

**Сравнение с SEQ.** Поскольку формула одинаковая и редукция корректно сводит min/max,
результат побайтово совпадает с SEQ при любом числе потоков.

**Baseline для ускорения.** Время SEQ на том же кадре производительности: **0,76217 с**.
Ускорение в таблице ниже — отношение **0,76217** к времени OMP при данном числе потоков.

**Что такое «эффективность» в этой таблице.** Ускорение из строки, делённое на число потоков из той же строки, в процентах.

**Команды запуска.**

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel

export PPC_NUM_THREADS=4
scripts/run_tests.py --running-type=threads --counts 1 2 4
scripts/run_tests.py --running-type=performance
```

Runner подставляет `OMP_NUM_THREADS` равным `PPC_NUM_THREADS`. Лимиты времени задаются `PPC_TASK_MAX_TIME` и `PPC_PERF_MAX_TIME`.

типичную для задач с двумя линейными проходами.

## 9. Выводы

OpenMP даёт устойчивый рост скорости с числом потоков на этом кадре. К восьми потокам
выигрыш около шести раз к baseline. Дальше масштабироваться мешают память
и накладные расходы двух больших параллельных регионов подряд. Версия удачна как простой baseline
для всех остальных потоковых отчётов: код минимально отличается от SEQ,
а корректность гарантируется механикой `reduction`.
