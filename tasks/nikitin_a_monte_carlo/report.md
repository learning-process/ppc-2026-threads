## 1. Введение

Метод Монте-Карло — это вычислительный алгоритм, использующий случайную выборку для аппроксимации значений многомерных интегралов. Данная работа посвящена сравнению пяти различных подходов к параллельной реализации этого метода: последовательная версия (SEQ), OpenMP, oneTBB, средства стандартной библиотеки C++ (STL) и гибридная версия MPI+OpenMP (ALL). Цель работы — оценить эффективность каждого подхода на задаче численного интегрирования.

## 2. Постановка задачи

Вычисление определённого интеграла вида:

$$I = \int_{a_1}^{b_1} \int_{a_2}^{b_2} ... \int_{a_d}^{b_d} f(x_1, x_2, ..., x_d) \, dx_1 dx_2 ... dx_d$$

**Входные данные:**
- lower_bounds — вектор нижних пределов интегрирования по каждому измерению
- upper_bounds — вектор верхних пределов интегрирования по каждому измерению
- num_points — количество точек выборки
- func_type — тип интегрируемой функции (константа, линейная, произведение, квадратичная, экспонента)

**Выходные данные:**
- result — приближённое значение интеграла (вещественное число)

**Ограничения:**
- Размерности lower_bounds и upper_bounds совпадают и не равны нулю
- Для каждого измерения lower_bounds[i] < upper_bounds[i]
- num_points > 0

## 3. Базовый алгоритм (последовательная версия)

Алгоритм использует метод Монте-Карло с квазислучайной последовательностью Кронекера:

1. Вычисляется объём области интегрирования: V = prod_{i=1}^{d} (b_i - a_i)
2. Генерируется num_points точек в единичном гиперкубе [0,1)^d с помощью последовательности Кронекера
3. Каждая точка масштабируется в реальную область интегрирования: x_j = a_j + u_j * (b_j - a_j)
4. Для каждой точки вычисляется значение подынтегральной функции f(x)
5. Вычисляется среднее арифметическое значений функции
6. Результат интеграла: I ≈ V * (1/n) * sum_{i=1}^{n} f(x_i)

**Сложность по времени:** O(n * d), где n — количество точек, d — размерность

**Сложность по памяти:** O(d)

**Инварианты:** Результат сходится к точному значению интеграла при увеличении num_points.

## 4. Схема распараллеливания

### 4.1 SEQ
Распараллеливание отсутствует. Все вычисления выполняются в одном потоке. Служит эталоном для сравнения.

### 4.2 OpenMP
Распараллеливание реализовано с помощью директивы `#pragma omp parallel for`. Основной цикл по точкам выборки разделяется между потоками. Переменная sum объявлена как reduction(+). Выбран статический тип распределения итераций (schedule(static)). Директива default(none) требует явного указания атрибутов всех переменных.

### 4.3 TBB
Распараллеливание реализовано с помощью алгоритма `tbb::parallel_reduce`. Диапазон итераций [0, num_points) разбивается на поддиапазоны, которые распределяются между потоками. Каждый поток вычисляет локальную сумму, после чего результаты объединяются. Размер зёрна и планировщик используются по умолчанию.

### 4.4 STL
Распараллеливание реализовано с помощью ручного разбиения диапазона и асинхронных задач (`std::async` с политикой `std::launch::async`). Диапазон индексов разбивается на непрерывные поддиапазоны, количество которых равно числу потоков. Каждая задача возвращает результат через `std::future`. Суммирование выполняется в главном потоке после получения всех результатов.

### 4.5 ALL (гибрид MPI+OpenMP)
Реализация использует два уровня параллелизма:
- **Межпроцессный уровень (MPI):** Диапазон индексов распределяется между MPI-процессами. Процесс с рангом 0 выполняет сбор результатов через MPI_Reduce и рассылку финального результата через MPI_Bcast. MPI_Barrier обеспечивает глобальную синхронизацию.
- **Внутрипроцессный уровень (OpenMP):** Внутри каждого MPI-процесса цикл по точкам поддиапазона распараллеливается с помощью OpenMP (аналогично версии OMP).

## 5. Детали реализации

**Структура кода:**
- `common/include/common.hpp` — базовые типы (InType, OutType, FunctionType)
- `seq/` — последовательная версия
- `omp/` — версия с OpenMP
- `tbb/` — версия с oneTBB
- `stl/` — версия с std::async
- `all/` — гибридная версия MPI+OpenMP

**Общие компоненты:**
- `EvaluateFunction` — вычисление значения тестовой функции
- `KroneckerSequence` — генерация квазислучайной последовательности
- `ValidationImpl` — проверка корректности входных данных

**Особенности каждой реализации:**

| Технология | Ключевой примитив | Синхронизация |
|------------|-------------------|---------------|
| SEQ | последовательный цикл | отсутствует |
| OMP | #pragma omp parallel for | reduction, неявный барьер |
| TBB | tbb::parallel_reduce | редукция, автоматическое объединение |
| STL | std::async + std::future | future::get() |
| ALL | MPI + OpenMP | MPI_Reduce, MPI_Bcast, MPI_Barrier, reduction |

## 7. Результаты и обсуждение

### 7.1 Корректность

Корректность проверялась сравнением результатов всех параллельных реализаций с последовательной версией. Для каждого типа тестовой функции выполнялись прогоны с одинаковыми входными данными. Расхождения в выходных данных не наблюдались в пределах вычислительной погрешности. Дополнительно была проверена устойчивость результатов при различном количестве потоков/процессов.

### 7.2 Производительность

**Результаты замеров для 4D линейной функции:**

| Технология | Кол-во | Время, с | Ускорение | Эффективность |
|------------|--------|----------|-----------|---------------|
| seq        | 1      | 1.178    | 1.00      | Н/Д           |
| omp        | 4      | 0.618    | 1.91      | 47.8%         |
| tbb        | 4      | 0.618    | 1.91      | 47.8%         |
| stl        | 4      | 0.631    | 1.87      | 46.8%         |
| all        | 4      | 0.538    | 2.19      | 54.8%         |
| all        | 8      | 0.481    | 2.45      | 30.6%         |
| all        | 32     | 0.521    | 2.26      | 7.1%          |

**Анализ результатов:**

- **SEQ (1.178 с):** Базовое время выполнения, используемое как эталон для расчёта ускорения.

- **OpenMP (1.91x при 4 потоках):** Показал хорошее ускорение, но эффективность снижается до 47.8% из-за накладных расходов на создание потоков и синхронизацию редукции.

- **TBB (1.91x при 4 потоках):** Результаты практически идентичны OpenMP. Автоматическое разбиение диапазонов TBB эффективно справляется с равномерной вычислительной нагрузкой.

- **STL (1.87x при 4 потоках):** Несколько ниже, чем у OpenMP и TBB, что объясняется накладными расходами на создание асинхронных задач и управление future.

- **ALL (2.19x при 1×4, 2.45x при 2×4, 2.26x при 4×8):** Гибридная версия показывает наилучшее ускорение при небольшом количестве MPI-процессов (2×4 = 8 работников). При увеличении до 32 работников (4×8) ускорение не растёт, а эффективность резко падает до 7.1% из-за коммуникационных накладных расходов MPI.

**Узкие места и ограничения масштабируемости:**

1. **Вычислительная нагрузка равномерна**, поэтому статическое распределение итераций даёт хорошие результаты для всех реализаций.

2. **Накладные расходы на синхронизацию** наиболее заметны в STL (создание задач) и ALL (MPI-коммуникации).

3. **Предел масштабируемости ALL** достигается при 8 работниках, дальнейшее увеличение числа процессов не даёт выигрыша из-за доминирования коммуникационных затрат над вычислительными.

## 8. Выводы

В ходе работы были реализованы и протестированы пять версий алгоритма Монте-Карло для численного интегрирования:

1. **SEQ** — последовательная версия, служащая эталоном корректности.

2. **OpenMP** — простая в реализации директива parallel for с редукцией даёт хорошее ускорение (1.91x) с эффективностью около 48%.

3. **TBB** — показывает результаты, аналогичные OpenMP, но обеспечивает более гибкое управление планировщиком.

4. **STL** — ручное управление потоками через std::async и std::future даёт немного меньшее ускорение (1.87x) из-за дополнительных накладных расходов.

5. **ALL (MPI+OpenMP)** — гибридная версия обеспечивает наилучшее абсолютное ускорение (2.45x при 8 работниках), но дальнейшее увеличение числа работников приводит к снижению эффективности из-за коммуникационных затрат MPI.

**Рекомендации по выбору технологии:**
- Для одноузловых вычислений на многоядерной системе оптимальны OpenMP или TBB.
- Для кластерных систем с большим количеством узлов предпочтительна чистая MPI-реализация.
- Гибридная версия ALL эффективна при умеренном количестве MPI-процессов (2–4) с несколькими потоками на процесс.
