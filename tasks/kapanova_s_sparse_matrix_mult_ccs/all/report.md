## 1. Контекст

Гибридная версия (ALL) сочетает межпроцессный параллелизм (MPI) и внутрипроцессный (OpenMP), что позволяет
задействовать несколько процессов, каждый из которых использует несколько потоков. Эта версия позволяет исследовать
накладные расходы на коммуникацию (`MPI_Gatherv`, `MPI_Bcast`) и изучить двухуровневый параллелизм вместо чисто
потоковых версий.

В ALL-версию из SEQ перенесён внешний цикл по столбцам матрицы `B`. Столбцы распределяются между MPI-процессами на
основе оценки вычислительной стоимости, а внутри процесса обрабатываются OpenMP-потоками. Результаты собираются на
нулевом процессе, формируется итоговая матрица и рассылается всем.

## 2. Постановка задачи

Задача: перемножить две разреженные матрицы `A` и `B` с элементами типа double в формате CCS с использованием MPI и
OpenMP. Ограничения, крайние случаи и критерии корректности совпадают с `seq/report.md`.
Последовательная версия используется как эталон корректности и baseline для расчёта ускорения.

**Цели:**

- Уменьшить время выполнения по сравнению с SEQ.
- Исследовать накладные расходы гибридной схемы (MPI + OpenMP).
- создание реализации, эффективно использующей технологии `OpenMP` и `MPI` совместно
- масштабирование алгоритма на уровне процессов и потоков

## 3. Базовый алгоритм

Последовательный алгоритм (см. `seq/report.md`, раздел 3) проходит по столбцам матрицы `B`. Для каждого столбца `j`
перебираются его ненулевые элементы. Для каждого такого элемента с индексом строки `k` и значением `b_val`
обрабатывается столбец `k` матрицы `A`: значения `a_val` умножаются на `b_val` и накапливаются во временном векторе
`accum` по индексу строки `i`. Затем измененные строки сортируются, ненулевые значения и их индексы записываются в
результирующую матрицу `C` в формате CCS, а `accum` сбрасывается для следующего столбца.

## 4. Межпроцессная схема

**Роли rank-ов:**

- **Rank 0 (корневой):** вычисляет сбалансированное распределение столбцов, рассылает диапазоны, собирает результаты
от всех процессов, строит итоговую CCS-матрицу и рассылает её.
- **Остальные rank-и:** получают свой диапазон столбцов, выполняют умножение, отправляют результаты на rank 0.

**Как распределяются данные:**
`ComputeBalancedRanges` на rank 0 оценивает стоимость каждого столбца — количество операций умножения (проход по
ненулевым элементам `B` и соответствующим столбцам `A`). Столбцы распределяются так, чтобы суммарная стоимость у
каждого процесса была примерно одинаковой. Это балансирует нагрузку лучше, чем простое деление поровну.

**MPI-вызовы:**

1. **`MPI_Bcast` (ranges):** rank 0 рассылает массив `ranges` всем процессам. Каждый процесс узнаёт свой диапазон
`[ranges[rank], ranges[rank+1])`.
2. **`MPI_Gather` (local_nnz):** каждый процесс сообщает rank 0, сколько ненулевых элементов он вычислил. Это нужно
для подготовки буферов под `Gatherv`.
3. **`MPI_Gatherv` (rows, cols, vals):** rank 0 собирает от всех процессов массивы индексов строк, индексов столбцов
и значений. Каждый процесс отправляет только свои данные.
4. **`MPI_Bcast` (результат):** rank 0 формирует итоговую матрицу `C` и через `BroadcastResult` рассылает её всем
процессам. Это гарантирует, что все процессы имеют одинаковый результат.

## 5. Внутрипроцессная схема

**Thread-technology:** OpenMP (`#pragma omp parallel` + `#pragma omp for`).

**Как делится работа между потоками:**
Внутри `ComputeLocalColumns` параллельный цикл `#pragma omp for schedule(guided, 32)` распределяет столбцы из
диапазона процесса между потоками. `schedule(guided)` похож на `dynamic`, но начинает с больших блоков и уменьшает их
— это снижает накладные расходы.

Каждый поток имеет свои локальные массивы (`accum`, `used`, `active`, `thr_rows`, `thr_cols`, `thr_vals`) и
обрабатывает назначенные ему столбцы независимо.

**Как объединяются локальные результаты:**
После обработки своих столбцов каждый поток добавляет результаты в общие векторы `send_rows`, `send_cols`,
`send_vals` внутри критической секции `#pragma omp critical`. Это единственное место синхронизации между потоками
внутри процесса.

## 6. Детали реализации

Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp

**ValidationImpl, PreProcessingImpl, PostProcessingImpl** — идентичные тем что в SEQ-версии (см. `seq/report.md`
раздел 4).

**Где используются MPI-вызовы:**

- `MPI_Bcast(ranges)` — после вычисления распределения на rank 0.
- `MPI_Gather(local_nnz)` — сбор размеров данных от всех процессов.
- `MPI_Gatherv` × 3 — сбор индексов строк, столбцов и значений.
- `MPI_Bcast` × 3 (в `BroadcastResult`) — рассылка `nnz`, `cols`, `col_ptrs`, `row_indices`, `values`.

**Где используются OpenMP-директивы:**

- `ComputeBalancedRanges`: `#pragma omp parallel for reduction(+ : total_cost) schedule(guided)` — параллельная
оценка стоимости столбцов.
- `ComputeLocalColumns`: `#pragma omp parallel` + `#pragma omp for schedule(guided, 32) nowait` + `#pragma omp
critical` — параллельная

**Потенциальные узкие места:**

1. **`MPI_Gatherv` на rank 0:** все процессы отправляют данные одному — rank 0 становится узким местом при большом
числе процессов.
2. **`#pragma omp critical`:** все потоки внутри процесса сливают результаты через одну критическую секцию — при
большом числе потоков возможна конкуренция.
3. **`BroadcastResult`:** рассылка полной матрицы всем процессам — избыточно, если результат нужен только на rank 0.
4. **Вычисление ranges на rank 0:** выполняется одним процессом перед рассылкой — последовательный участок.

**Ключевой фрагмент — гибридный параллелизм:**

```cpp
// all/src/ops_all.cpp — внутрипроцессная обработка
#pragma omp parallel default(none) shared(a, b, start, local_cols, send_rows, send_cols, send_vals)
  {
    std::vector**double** accum(a.rows, 0.0);
    std::vector**char** used(static_cast**size_t**(a.rows), 0);
    std::vector**size_t** active(static_cast**size_t**(a.rows));
    std::vector**MpiU64** thr_rows;
    std::vector**MpiU64** thr_cols;
    std::vector**double** thr_vals;
    int active_count = 0;

#pragma omp for schedule(guided, 32) nowait
    for (size_t j = 0; j < local_cols; ++j) {
      active_count = 0;
      ProcessColumn(start + j, a, b, thr_rows, thr_cols, thr_vals,
                    accum.data(), used.data(), active.data(), active_count);
    }
#pragma omp critical
    {
      send_rows.insert(send_rows.end(), thr_rows.begin(), thr_rows.end());
      send_cols.insert(send_cols.end(), thr_cols.begin(), thr_cols.end());
      send_vals.insert(send_vals.end(), thr_vals.begin(), thr_vals.end());
    }
  }
```

каждый поток имеет приватные accum, used, active. #pragma omp for schedule(guided, 32) распределяет столбцы. nowait
убирает неявный барьер в конце цикла. Слияние через critical — узкое место при большом числе потоков.

## 9. Результаты

Тесты производительности запущены в обоих режимах (`task_run` и `pipeline`), все выполнилось в пределах лимитов.
Ускорение считается по `task_run`. Эффективность — относительно общего числа работников `ranks × threads_per_rank`.

| Backend | Ranks | Thr/rank | Workers | Время task (с) | Ускорение | Эффективность |
| --------- | ------- | ---------- | --------- | --------------- | ----------- | --------------- |
| SEQ | 1 | 1 | 1 | 0.506 | 1.00 | 100% |
| ALL | 1 | 4 | 4 | 1.562 | 0.32 | 8% |
| ALL | 2 | 2 | 4 | 2.040 | 0.25 | 6% |
| ALL | 2 | 4 | 8 | 1.980 | 0.26 | 3% |
| ALL | 4 | 2 | 8 | 2.282 | 0.22 | 3% |

Время в режиме `pipeline`: 1p×4t — 1.734 с, 2p×2t — 2.071 с, 2p×4t — 1.934 с, 4p×2t — 2.264 с.

**Комментарий о цене коммуникации и синхронизации:**

ALL-версия существенно уступает чисто потоковым реализациям при том же числе работников. Ускорение меньше 1 во всех
конфигурациях — гибридная схема медленнее последовательной. Основные причины:

1. **`MPI_Gatherv` на rank 0:** все процессы отправляют данные одному — это последовательное узкое место.
2. **`#pragma omp critical`:** все потоки внутри процесса сливают результаты через одну критическую секцию.
3. **`BroadcastResult`:** рассылка полной матрицы всем процессам добавляет время.
4. **Вычисление ranges на rank 0:** последовательный участок перед параллельной работой.

Лучшая конфигурация — 1 процесс × 4 потока (1.562 с), где нет межпроцессной коммуникации, но даже она в 3 раза
медленнее SEQ.

## 10. Выводы

Гибридная версия (MPI + OpenMP) на данной задаче и оборудовании не показала преимуществ перед чисто потоковыми
реализациями. Максимальное ускорение 0.32× получено при 1 процессе × 4 потока — это по сути чистая OpenMP-версия
внутри ALL.

Накладные расходы на MPI-коммуникацию (`Gatherv`, `Bcast`) и синхронизацию (`critical`) превышают выигрыш от
распределения работы между процессами. При 2 процессах × 2 потока время оказалось даже хуже последовательного (0.25×).

Гибридная схема была бы оправдана в случаях, которые не тестировались в данной работе:

- матрицы не помещаются в память одного процесса;
- запуск на нескольких узлах кластера, где каждый процесс использует свою память;
- объём вычислений настолько велик, что коммуникационные издержки пренебрежимо малы по сравнению с вычислениями.

Для данной задачи (10000×10000, плотность 0.5%) на одном компьютере чисто потоковые версии, особенно TBB, оказались
эффективнее.

Полученные числа нельзя механически переносить на другие машины, поскольку замеры выполнялись на Apple M1 (macOS,
arm64), а поведение backend-а чувствительно к архитектуре процессора и объёму кэш-памяти.
