## 1. Контекст

Последовательная версия нужна как базовая реализация оператора Собеля. Она выполняет весь алгоритм без
параллельного программирования: изображение обрабатывается обычными циклами по строкам и столбцам.

Эта версия используется как эталон корректности и как baseline для расчёта ускорения
остальных backend-ов: `OMP`, `TBB`, `STL` и `ALL`.

Если последовательная версия работает неправильно, то сравнивать с ней параллельные реализации уже нельзя.
Поэтому в `SEQ` сначала фиксируется общая логика задачи, формат входных и выходных данных, ограничения и
ожидаемый результат.

## 2. Постановка задачи

На вход подаётся RGB-изображение. В коде оно хранится в структуре `Image`:

```cpp
struct Image {
  int width = 0;
  int height = 0;
  std::vector<Pixel> data;
};
```

Каждый пиксель хранит три компоненты цвета:

```cpp
struct Pixel {
  uint8_t r = 0;
  uint8_t g = 0;
  uint8_t b = 0;
};
```

Массив `data` хранит изображение в одномерном виде. Его размер должен быть равен:

```text
width * height
```

Результатом работы является массив целых чисел:

```cpp
using OutType = std::vector<int>;
```

Каждый элемент выходного массива содержит величину градиента для соответствующего пикселя.

Оператор Собеля использует окно `3×3`, поэтому крайние строки и столбцы не пересчитываются.
Для них в выходном массиве остаются нули.

Перед запуском проверяется, что:

- ширина изображения больше нуля;
- высота изображения больше нуля;
- количество пикселей равно `width * height`;
- выходной массив перед запуском пустой.

Если размер изображения меньше `3×3`, внутренних пикселей для обработки нет. В этом случае алгоритм
завершает работу без вычисления градиентов, а выходной массив остаётся заполненным нулями.

## 3. Базовый алгоритм

Алгоритм работает по внутренним пикселям изображения. Для каждого такого пикселя берётся его окрестность `3×3`.

Сначала RGB-пиксели переводятся в оттенки серого по формуле:

```text
gray = (77 * r + 150 * g + 29 * b) >> 8
```

После этого считаются два значения:

```text
Gx — изменение яркости по горизонтали
Gy — изменение яркости по вертикали
```

Итоговое значение записывается как:

```text
|Gx| + |Gy|
```

Порядок работы:

1. Проверить корректность входного изображения.
2. Создать выходной массив размера `width * height`.
3. Пройти по всем внутренним пикселям.
4. Для каждого внутреннего пикселя взять соседей из окна `3×3`.
5. Перевести соседние пиксели в оттенки серого.
6. Посчитать `Gx`, `Gy` и итоговую величину градиента.
7. Записать результат в соответствующую позицию выходного массива.

Асимптотика по времени:

```text
O(width * height)
```

Каждый внутренний пиксель обрабатывается один раз, а размер окна всегда фиксированный — `3×3`.

Асимптотика по памяти:

```text
O(width * height)
```

Основная дополнительная память нужна для выходного массива.

Основные инварианты:

- размер выходного массива равен `width * height`;
- граничные пиксели остаются равными нулю;
- для каждого внутреннего пикселя используется одна и та же формула Собеля;
- входное изображение во время вычислений не изменяется.

## 4. Детали реализации

Файлы реализации:

```text
seq/include/ops_seq.hpp
seq/src/ops_seq.cpp
```

Класс последовательной версии:

```cpp
BalchunayteZSobelOpSEQ
```

Он наследуется от общего типа задачи и реализует стандартный pipeline:

```text
ValidationImpl → PreProcessingImpl → RunImpl → PostProcessingImpl
```

`ValidationImpl` проверяет входные данные. Если ширина или высота меньше либо равны нулю,
либо размер массива пикселей не совпадает с `width * height`, задача считается некорректной.

`PreProcessingImpl` создаёт выходной массив и заполняет его нулями. Это важно, потому что границы
изображения не обрабатываются и должны остаться нулевыми.

`RunImpl` выполняет основную работу: проходит по внутренним пикселям изображения и считает оператор Собеля.

`PostProcessingImpl` в этой задаче не выполняет дополнительной обработки и просто возвращает успешное завершение.

Основной фрагмент последовательной обработки:

```cpp
// File: seq/src/ops_seq.cpp
for (int row_index = 1; row_index < image_height - 1; ++row_index) {
  for (int col_index = 1; col_index < image_width - 1; ++col_index) {
    // read 3x3 neighbourhood
    // compute Gx and Gy
    // write |Gx| + |Gy| to output
  }
}
```

Здесь видно, что последовательная версия обрабатывает пиксели строго по порядку: сначала строки,
затем столбцы внутри каждой строки.

## 5. Проверка корректности

Корректность `SEQ` проверялась функциональными тестами. Эти же тесты дальше используются для остальных backend-ов.

Используются три тестовых случая:

| Тест             | Что проверяет          | Ожидаемый результат                                 |
|------------------|------------------------|-----------------------------------------------------|
| `ConstantImage`  | Однотонное изображение | Все значения равны нулю                             |
| `VerticalEdge`   | Вертикальная граница   | Внутренние пиксели на границе имеют значение `1020` |
| `HorizontalEdge` | Горизонтальная граница | Внутренние пиксели на границе имеют значение `1020` |

Во всех функциональных тестах используется изображение размером `4×4`.

Пример смысла тестов:

- если всё изображение одного цвета, границ нет;
- если левая часть изображения тёмная, а правая светлая, появляется вертикальная граница;
- если верхняя часть изображения тёмная, а нижняя светлая, появляется горизонтальная граница.

Команда запуска функционального теста для `SEQ`:

```powershell
$env:PPC_NUM_THREADS="8"
$env:OMP_NUM_THREADS="8"

.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_seq*
```

Для `SEQ` переменные окружения с количеством потоков не влияют на сам алгоритм, потому что версия
последовательная. Они выставлялись для единообразия с общим запуском тестов.

## 7. Результаты

Результаты последовательной версии:

| Backend | Mode       |    Size | Workers |      Time, s | Комментарий                         |
|---------|------------|--------:|--------:|-------------:|-------------------------------------|
| SEQ     | `task_run` | 512×512 |       1 | 0.0107273000 | Время основной вычислительной части |
| SEQ     | `pipeline` | 512×512 |       1 | 0.0202614000 | Время полного конвейера задачи      |

В дальнейшем значение `task_run = 0.0107273000` используется как baseline для расчёта ускорения
параллельных версий в режиме `task_run`.

Значение `pipeline = 0.0202614000` используется как baseline для расчёта ускорения в режиме `pipeline`.

Самая дорогая часть последовательной версии находится в `RunImpl`, потому что именно там выполняется
проход по внутренним пикселям изображения и вычисляется оператор Собеля. Остальные этапы pipeline проще:
валидация проверяет размеры, а предобработка создаёт выходной массив.

## 8. Выводы

Последовательная версия реализует основную логику оператора Собеля без параллелизма. Она нужна как эталон
для проверки корректности и как базовая точка сравнения производительности.

На размере `512×512` время `SEQ` в режиме `task_run` составило `0.0107273000` секунды. Именно это значение
используется как знаменатель при расчёте ускорения для `OMP`, `TBB`, `STL` и `ALL`.

Эта версия проще остальных по структуре, но не использует возможности многоядерного процессора.
Поэтому для ускорения задачи дальше были реализованы параллельные backend-и.
