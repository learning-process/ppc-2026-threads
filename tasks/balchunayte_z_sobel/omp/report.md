## 1. Контекст

Версия `OMP` основана на последовательной реализации `SEQ`, но основной цикл обработки изображения
распараллелен с помощью OpenMP.

В задаче Собеля это удобно, потому что разные строки изображения можно обрабатывать независимо.
Для каждого внутреннего пикселя используется только его локальная окрестность `3×3`, а результат
записывается в свою позицию выходного массива.

Поэтому OpenMP здесь применяется не для изменения самого алгоритма, а для ускорения прохода по пикселям изображения.

## 2. Постановка задачи

На вход подаётся RGB-изображение, которое хранится в структуре `Image`:

```cpp
struct Image {
  int width = 0;
  int height = 0;
  std::vector<Pixel> data;
};
```

Результатом является массив целых чисел:

```cpp
using OutType = std::vector<int>;
```

Каждый элемент выходного массива содержит величину градиента для соответствующего пикселя.

Как и в `SEQ`, обрабатываются только внутренние пиксели изображения. Граничные строки и столбцы
остаются равными нулю, потому что для них нельзя применить окно `3×3`.

Последовательная версия `SEQ` используется как baseline:

- для проверки корректности результата;
- для расчёта ускорения;
- для сравнения времени выполнения.

## 3. Базовый алгоритм

Алгоритм вычисляет оператор Собеля для каждого внутреннего пикселя изображения.

Для пикселя берётся окрестность `3×3`. Соседние RGB-пиксели переводятся в оттенки серого по формуле:

```text
gray = (77 * r + 150 * g + 29 * b) >> 8
```

Затем считаются два градиента:

```text
Gx - изменение яркости по горизонтали
Gy - изменение яркости по вертикали
```

В выходной массив записывается значение:

```text
|Gx| + |Gy|
```

В версии `OMP` сама формула не меняется. Отличие от `SEQ` только в том, что внешний цикл
по строкам выполняется параллельно.

## 4. Схема распараллеливания

В `OMP` распараллелен внешний цикл по строкам изображения:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(input_image, output_data, image_width, image_height, image_width_size) \
    schedule(static)
for (int row_index = 1; row_index < image_height - 1; ++row_index) {
  for (int col_index = 1; col_index < image_width - 1; ++col_index) {
    // compute Sobel for one pixel
  }
}
```

Параллелится именно цикл по `row_index`. Это означает, что разные потоки получают разные строки изображения.

### 4.1. Shared и private данные

В директиве явно указаны общие данные:

```cpp
shared(input_image, output_data, image_width, image_height, image_width_size)
```

Они являются `shared`, потому что нужны всем потокам:

- `input_image` - общее входное изображение, только читается;
- `output_data` - общий выходной массив, в который записываются результаты;
- `image_width`, `image_height`, `image_width_size` - общие параметры изображения.

Переменная `row_index` является переменной цикла `parallel for`, поэтому OpenMP делает её приватной для каждого потока.

Переменная `col_index` и остальные локальные переменные создаются внутри тела цикла,
поэтому у каждого потока есть свои экземпляры этих переменных.

### 4.2. Почему нет reduction

В этой задаче не нужен `reduction`, потому что потоки не считают общую сумму и не обновляют одну общую переменную.

Каждый поток вычисляет значения для своей строки и записывает результат в отдельные элементы массива `output_data`.

### 4.3. Почему не нужны atomic и critical

`atomic` и `critical` тоже не нужны, потому что нет ситуации, где несколько потоков одновременно
записывают в один и тот же элемент.

Индекс выходного массива зависит от `row_index` и `col_index`:

```cpp
output_data[output_index] = magnitude;
```

Так как строки распределены между потоками, каждый поток работает со своим набором индексов.
Поэтому гонки данных при записи результата не возникает.

### 4.4. Почему выбран schedule(static)

Используется расписание:

```cpp
schedule(static)
```

Оно подходит для этой задачи, потому что нагрузка по строкам почти одинаковая.
Для каждой внутренней строки выполняется одинаковый набор действий: пройти по внутренним столбцам
и посчитать оператор Собеля.

`static` заранее распределяет строки между потоками. Это уменьшает накладные расходы на планирование,
потому что runtime не нужно постоянно решать, какой поток получит следующую порцию работы.

### 4.5. Барьер

В конце `parallel for` есть неявный барьер OpenMP. Он нужен, чтобы все потоки завершили обработку
своих строк до выхода из `RunImpl`.

В этой задаче барьер безопасен и не мешает корректности, потому что после параллельного цикла
результат уже должен быть полностью записан в выходной массив.

## 5. Детали реализации

Файлы реализации:

```text
omp/include/ops_omp.hpp
omp/src/ops_omp.cpp
```

Класс OpenMP-версии:

```cpp
BalchunayteZSobelOpOMP
```

Он использует такой же pipeline, как и `SEQ`:

```text
ValidationImpl → PreProcessingImpl → RunImpl → PostProcessingImpl
```

`ValidationImpl` проверяет корректность входного изображения:

- ширина больше нуля;
- высота больше нуля;
- размер массива пикселей равен `width * height`;
- выходной массив перед запуском пустой.

`PreProcessingImpl` создаёт выходной массив и заполняет его нулями.

`RunImpl` выполняет основное вычисление. Именно в этом методе используется
OpenMP-директива для распараллеливания цикла по строкам.

`PostProcessingImpl` не выполняет дополнительной обработки.

Главное отличие от `SEQ` находится в `RunImpl`: вместо обычного внешнего цикла используется `#pragma omp parallel for`.

Риски гонок устранены за счёт того, что:

- входное изображение только читается;
- каждая строка обрабатывается независимо;
- каждый поток записывает результат в свои элементы выходного массива;
- общей накапливаемой переменной нет.

## 6. Проверка корректности

Корректность `OMP` проверялась теми же функциональными тестами, что и `SEQ`.

Используются три тестовых случая:

| Тест             | Что проверяет          | Ожидаемый результат                                 |
|------------------|------------------------|-----------------------------------------------------|
| `ConstantImage`  | Однотонное изображение | Все значения равны нулю                             |
| `VerticalEdge`   | Вертикальная граница   | Внутренние пиксели на границе имеют значение `1020` |
| `HorizontalEdge` | Горизонтальная граница | Внутренние пиксели на границе имеют значение `1020` |

Для тестов используется изображение размером `4×4`.

Команда запуска функционального теста для `OMP`:

```powershell
$env:PPC_NUM_THREADS="8"
$env:OMP_NUM_THREADS="8"

.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_omp*
```

Также `OMP` запускался вместе с другими потоковыми backend-ами:

```powershell
.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_seq*:*balchunayte_z_sobel_omp*:
*balchunayte_z_sobel_tbb*:*balchunayte_z_sobel_stl*
```

В общем запуске для `SEQ`, `OMP`, `TBB`, `STL` прошло:

```text
[  PASSED  ] 12 tests.
```

Это подтверждает, что `OMP` выдаёт тот же результат, что и базовая последовательная логика.

## 8. Результаты

Результаты `OMP` сравнивались с `SEQ`.

Baseline `SEQ`:

| Mode       |  SEQ time, s |
|------------|-------------:|
| `task_run` | 0.0107273000 |
| `pipeline` | 0.0202614000 |

Результаты `OMP`:

| Backend | Mode       |    Size | Threads |      Time, s | Speedup vs SEQ | Efficiency |
|---------|------------|--------:|--------:|-------------:|---------------:|-----------:|
| OMP     | `task_run` | 512×512 |       8 | 0.0034280800 |           3.13 |     39.12% |
| OMP     | `pipeline` | 512×512 |       8 | 0.0047181800 |           4.29 |     53.68% |

В режиме `task_run` OpenMP-версия ускорилась примерно в `3.13` раза относительно `SEQ`.

В режиме `pipeline` ускорение получилось примерно `4.29` раза.

Ускорение не является линейным относительно 8 потоков. Это нормальный результат: часть времени уходит на
создание и управление потоками, а также на общие накладные расходы runtime.
Кроме того, размер изображения `512×512` не очень большой, поэтому полезной работы не всегда
достаточно для полной загрузки всех потоков.

## 9. Выводы

OpenMP хорошо подошёл для этой задачи, потому что оператор Собеля удобно распараллеливается по строкам изображения.

Главные причины хорошего результата:

- строки можно обрабатывать независимо;
- входное изображение только читается;
- каждый поток пишет в свой участок выходного массива;
- не нужны `reduction`, `atomic`, `critical` и `mutex`;
- `schedule(static)` подходит для равномерной нагрузки по строкам.

В режиме `task_run` версия `OMP` показала лучшее время среди всех backend-ов.
Это делает её одной из самых удачных реализаций для данной задачи.

Ограничение результата связано с тем, что ускорение не может быть полностью линейным:
есть накладные расходы OpenMP и часть работы, которая остаётся общей для всего pipeline.
