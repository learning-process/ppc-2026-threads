## 1. Контекст

Версия `STL` основана на последовательной реализации `SEQ`, но основной проход по изображению выполняется
вручную через `std::thread`.

Эта версия нужна, чтобы показать прямое управление потоками: самостоятельно определить число потоков,
разделить строки изображения на диапазоны, создать потоки, выполнить работу и затем дождаться их
завершения через `join`.

В отличие от `OMP` и `TBB`, здесь нет runtime-библиотеки, которая автоматически распределяет работу.
Вся схема разбиения данных задаётся в коде.

## 2. Постановка задачи

На вход подаётся RGB-изображение:

```cpp
struct Image {
  int width = 0;
  int height = 0;
  std::vector<Pixel> data;
};
```

Каждый пиксель содержит три компоненты цвета:

```cpp
struct Pixel {
  uint8_t r = 0;
  uint8_t g = 0;
  uint8_t b = 0;
};
```

Результат работы:

```cpp
using OutType = std::vector<int>;
```

Каждый элемент выходного массива содержит величину градиента для соответствующего пикселя.

Как и в `SEQ`, обрабатываются только внутренние пиксели изображения. Граничные строки и столбцы остаются равными нулю.

Последовательная версия `SEQ` используется как эталон:

- для проверки корректности;
- для сравнения времени выполнения;
- для расчёта ускорения версии `STL`.

## 3. Базовый алгоритм

Последовательная версия проходит по внутренним пикселям изображения и для каждого пикселя применяет оператор Собеля.

Для каждого пикселя берётся окрестность `3×3`. RGB-пиксели переводятся в оттенки серого:

```text
gray = (77 * r + 150 * g + 29 * b) >> 8
```

Затем вычисляются два градиента:

```text
Gx — изменение яркости по горизонтали
Gy — изменение яркости по вертикали
```

В выходной массив записывается:

```text
|Gx| + |Gy|
```

В версии `STL` формула Собеля не меняется. Отличие только в том, что строки изображения делятся между
несколькими потоками.

## 4. Схема распараллеливания

В `STL`-версии используется ручное разбиение внутренних строк изображения между потоками.

Количество внутренних строк:

```text
inner_row_count = height - 2
```

Именно эти строки нужно обработать, потому что первая и последняя строки являются граничными.

Количество потоков определяется через:

```cpp
std::thread::hardware_concurrency()
```

Если значение равно нулю, используется один поток. Затем число потоков ограничивается количеством внутренних строк:

```text
actual_thread_count = min(hardware_thread_count, inner_row_count)
```

Это нужно, чтобы не создавать больше потоков, чем есть строк для обработки.

### 4.1. Разбиение диапазона строк

Строки делятся между потоками почти равномерно.

Основная идея:

```text
base_rows = inner_row_count / actual_thread_count
extra_rows = inner_row_count % actual_thread_count
```

Каждый поток получает `base_rows` строк. Первые `extra_rows` потоков получают ещё по одной дополнительной строке.

Такой подход позволяет распределить остаток без перекоса нагрузки.

Пример:

```text
inner_row_count = 10
actual_thread_count = 3

base_rows = 3
extra_rows = 1
```

Тогда диапазоны будут такими:

```text
thread 0: 4 строки
thread 1: 3 строки
thread 2: 3 строки
```

### 4.2. Что делает каждый поток

Каждый поток получает свой полуинтервал строк:

```text
[start_row; end_row)
```

После этого поток последовательно проходит по своим строкам и столбцам внутри каждой строки.

Основной смысл работы потока:

```cpp
// File: stl/src/ops_stl.cpp
void ProcessRows(const Image& input_image, OutType& output_data, int start_row, int end_row) {
  for (int row_index = start_row; row_index < end_row; ++row_index) {
    for (int col_index = 1; col_index < input_image.width - 1; ++col_index) {
      // compute Sobel for one pixel
    }
  }
}
```

Каждый поток использует одну и ту же функцию обработки строк, но получает свой диапазон.

### 4.3. Локальные результаты

Отдельные локальные буферы в этой задаче не нужны.

Причина в том, что результат для каждого пикселя сразу записывается в общий выходной массив `output_data`,
но каждый поток пишет только в свой диапазон строк.

То есть потоки не записывают в одни и те же элементы массива.

### 4.4. Порядок create → work → join

Потоки создаются в отдельном цикле:

```cpp
// File: stl/src/ops_stl.cpp
std::vector<std::thread> worker_threads;

worker_threads.emplace_back(
    ProcessRows,
    std::cref(input_image),
    std::ref(output_data),
    start_row,
    end_row);
```

После создания всех потоков выполняется отдельный цикл ожидания:

```cpp
for (auto& worker_thread : worker_threads) {
  worker_thread.join();
}
```

Это важный момент. `join()` вызывается не внутри цикла создания потоков, а после запуска всех потоков.

Поэтому работа действительно выполняется параллельно:

```text
create all threads → work in parallel → join all threads
```

Если бы `join()` вызывался сразу после создания каждого потока, выполнение фактически стало бы
последовательным: программа создала бы поток, дождалась его завершения и только потом создала следующий.

В данной реализации эта ошибка не допущена.

### 4.5. Синхронизация

В версии `STL` не используются:

```text
mutex
atomic
condition_variable
```

Они не нужны, потому что:

- входное изображение только читается;
- выходной массив общий, но диапазоны записи не пересекаются;
- общей переменной-аккумулятора нет;
- каждый поток работает со своим набором строк.

Потокобезопасность обеспечивается не блокировками, а правильным разбиением данных.

## 5. Детали реализации

Файлы реализации:

```text
stl/include/ops_stl.hpp
stl/src/ops_stl.cpp
```

Класс STL-версии:

```cpp
BalchunayteZSobelOpSTL
```

Он использует стандартный pipeline:

```text
ValidationImpl → PreProcessingImpl → RunImpl → PostProcessingImpl
```

`ValidationImpl` проверяет:

- ширину изображения;
- высоту изображения;
- соответствие размера `data` значению `width * height`;
- что выходной массив перед запуском пустой.

`PreProcessingImpl` создаёт выходной массив и заполняет его нулями.

`RunImpl` выполняет основное вычисление. В этом методе:

1. определяется количество внутренних строк;
2. определяется возможное количество потоков;
3. строки делятся на диапазоны;
4. создаются объекты `std::thread`;
5. каждый поток обрабатывает свой диапазон строк;
6. после запуска всех потоков вызывается `join()`.

`PostProcessingImpl` не выполняет дополнительной обработки.

Карта распределения работы имеет вид:

```text
thread 0 → [start_0; end_0)
thread 1 → [start_1; end_1)
...
thread k → [start_k; end_k)
```

Диапазоны не пересекаются, поэтому объединение результатов не требуется. Каждый поток сразу записывает
итоговые значения в нужные позиции общего выходного массива.

## 6. Проверка корректности

Корректность `STL` проверялась теми же функциональными тестами, что и `SEQ`.

Используются три основных теста:

| Тест             | Что проверяет          | Ожидаемый результат                                 |
|------------------|------------------------|-----------------------------------------------------|
| `ConstantImage`  | Однотонное изображение | Все значения равны нулю                             |
| `VerticalEdge`   | Вертикальная граница   | Внутренние пиксели на границе имеют значение `1020` |
| `HorizontalEdge` | Горизонтальная граница | Внутренние пиксели на границе имеют значение `1020` |

Во всех этих тестах используется изображение размером `4×4`.

Крайние случаи обрабатываются так же, как в `SEQ`:

- изображение с некорректной шириной или высотой не проходит валидацию;
- изображение с неправильным размером массива `data` не проходит валидацию;
- изображение меньше `3×3` не имеет внутренних пикселей, поэтому результат остаётся нулевым.

Команда запуска функционального теста для `STL`:

```powershell
$env:PPC_NUM_THREADS="8"
$env:OMP_NUM_THREADS="8"

.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_stl*
```

Также `STL` запускался вместе с другими потоковыми backend-ами:

```powershell
.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_seq*:*balchunayte_z_sobel_omp*:
*balchunayte_z_sobel_tbb*:*balchunayte_z_sobel_stl*
```

В общем запуске для `SEQ`, `OMP`, `TBB`, `STL` прошло:

```text
[  PASSED  ] 12 tests.
```

Это подтверждает, что ручное разбиение строк между потоками не изменило результат относительно базовой логики.

## 9. Выводы

Версия `STL` показывает ручной способ распараллеливания задачи через `std::thread`.

Главные особенности реализации:

- строки изображения делятся между потоками вручную;
- каждый поток получает свой диапазон строк;
- потоки создаются сначала все вместе;
- `join()` вызывается отдельным циклом после создания всех потоков;
- синхронизация через `mutex` или `atomic` не требуется;
- корректность обеспечивается непересекающимися диапазонами записи.

На изображении `512×512` версия `STL` дала ускорение относительно `SEQ`. В режиме `pipeline` она показала
лучший результат среди всех backend-ов, а в режиме `task_run` уступила `OMP` и `TBB`.

Основное ограничение этой версии - ручное управление потоками. Программист сам отвечает за разбиение
данных, запуск потоков и правильный вызов `join()`. Поэтому такая реализация более инженерная и менее
удобная, чем `OMP` или `TBB`, но она хорошо показывает, как именно устроено распараллеливание на уровне
стандартной библиотеки C++.
