## 1. Контекст

Версия `ALL` предназначена для проверки гибридного подхода,
где в одной реализации используются несколько технологий параллельного программирования.

В данной задаче применяются:

- `MPI` - для запуска нескольких процессов и синхронизации между ними;
- `OpenMP` - для обработки части строк внутри процесса;
- `std::thread` - для ручной потоковой обработки части строк;
- `oneTBB` - для задачного распараллеливания части строк.

Главная цель этой версии - показать иерархическую схему параллелизма:
сначала есть уровень MPI-процессов, а внутри процесса используются потоковые технологии.

При этом важно отметить, что в данной реализации изображение дополнительно
не распределяется между MPI rank-ами. MPI используется как часть гибридной схемы запуска и синхронизации,
но основная работа по строкам выполняется внутри каждого процесса.

## 2. Постановка задачи

На вход подаётся RGB-изображение:

```cpp
struct Image {
  int width = 0;
  int height = 0;
  std::vector<Pixel> data;
};
```

Каждый пиксель содержит три цветовые компоненты:

```cpp
struct Pixel {
  uint8_t r = 0;
  uint8_t g = 0;
  uint8_t b = 0;
};
```

Результат работы:

```cpp
using OutType = std::vector<int>;
```

Каждый элемент выходного массива содержит величину градиента для соответствующего пикселя.

Последовательная версия `SEQ` используется как baseline:

- для проверки корректности;
- для расчёта ускорения;
- для сравнения времени выполнения.

## 3. Базовый алгоритм

Базовый алгоритм совпадает с `SEQ`.

Для каждого внутреннего пикселя изображения берётся окрестность `3×3`. RGB-пиксели переводятся в оттенки серого:

```text
gray = (77 * r + 150 * g + 29 * b) >> 8
```

Затем вычисляются два значения:

```text
Gx — горизонтальный градиент
Gy — вертикальный градиент
```

В выходной массив записывается:

```text
|Gx| + |Gy|
```

Граничные строки и столбцы не обрабатываются и остаются равными нулю.

В версии `ALL` формула Собеля не меняется. Отличие находится только в способе организации параллельного выполнения.

## 4. Межпроцессная схема

На межпроцессном уровне используется MPI.

В коде вызывается:

```cpp
// File: all/src/ops_all.cpp
int rank = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
```

`MPI_Comm_rank` получает номер текущего процесса внутри коммуникатора `MPI_COMM_WORLD`.

В данной реализации у rank-ов нет сложного разделения ролей вида “master-worker”.
Также не выполняется отдельное распределение изображения между MPI-процессами через `Scatter`,
`Broadcast`, `Gather` или `Reduce`.

То есть схема не является полноценным распределением изображения между процессами.
Каждый процесс запускается в рамках общей MPI-конфигурации, а основное разбиение работы выполняется
внутри процесса между потоковыми технологиями.

В конце работы используется барьер:

```cpp
MPI_Barrier(MPI_COMM_WORLD);
```

`MPI_Barrier` блокирует каждый процесс до тех пор, пока все процессы из `MPI_COMM_WORLD` не дойдут до этой точки.

В данной задаче барьер нужен для согласованного завершения гибридной части работы.
Он гарантирует, что все MPI-процессы дошли до конца вычислительного участка перед выходом из `RunImpl`.

## 5. Внутрипроцессная схема

Внутри процесса строки изображения делятся между тремя потоковыми технологиями:

- первая часть строк обрабатывается через `OpenMP`;
- вторая часть строк обрабатывается через `std::thread`;
- третья часть строк обрабатывается через `oneTBB`.

Обрабатываются только внутренние строки изображения:

```text
[1; height - 1)
```

Первая и последняя строки не включаются, потому что для них нельзя применить окно `3×3`.

Общая идея разбиения:

```text
внутренние строки изображения
        |
        |--> часть 1: OpenMP
        |--> часть 2: std::thread
        |--> часть 3: oneTBB
```

Каждая технология получает свой диапазон строк. Диапазоны не пересекаются,
поэтому разные части реализации не записывают результат в одни и те же элементы выходного массива.

### 5.1. OpenMP-часть

OpenMP используется для обработки одной части строк.

Внутри этой части применяется параллельный цикл по строкам. Нагрузка по строкам равномерная,
поэтому такой способ подходит для оператора Собеля.

### 5.2. std::thread-часть

`std::thread` используется для ручной обработки отдельного диапазона строк.

В этой части потоки создаются явно, получают свои поддиапазоны строк и затем завершаются через `join`.

Важно, что `join` должен выполняться после создания потоков, а не сразу после каждого `emplace_back`.
Иначе работа фактически стала бы последовательной.

### 5.3. TBB-часть

oneTBB используется для обработки ещё одного диапазона строк через `parallel_for`.

TBB самостоятельно распределяет итерации внутри переданного диапазона. Явное объединение результатов не
требуется, потому что каждый пиксель записывается сразу в свою позицию выходного массива.

### 5.4. Объединение результатов

Отдельной операции объединения локальных результатов нет.

Причина в том, что все технологии записывают результат в общий выходной массив, но работают с разными диапазонами строк.

Поэтому схема объединения выглядит так:

```text
OpenMP пишет свою часть output_data
std::thread пишет свою часть output_data
TBB пишет свою часть output_data
```

После завершения всех частей выходной массив уже содержит полный результат.

## 6. Детали реализации

Файлы реализации:

```text
all/include/ops_all.hpp
all/src/ops_all.cpp
```

Класс гибридной версии:

```cpp
BalchunayteZSobelOpALL
```

Он использует стандартный pipeline:

```text
ValidationImpl → PreProcessingImpl → RunImpl → PostProcessingImpl
```

`ValidationImpl` проверяет:

- ширину изображения;
- высоту изображения;
- соответствие размера `data` значению `width * height`;
- что выходной массив перед запуском пустой.

`PreProcessingImpl` создаёт выходной массив и заполняет его нулями.

`RunImpl` выполняет основную гибридную обработку:

1. получает MPI rank через `MPI_Comm_rank`;
2. определяет внутренний диапазон строк;
3. делит строки между OpenMP, `std::thread` и TBB;
4. запускает обработку каждой части;
5. выполняет `MPI_Barrier`.

`PostProcessingImpl` не выполняет дополнительной обработки.

В реализации используются следующие основные фрагменты:

```cpp
// File: all/src/ops_all.cpp
int rank = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
```

```cpp
// File: all/src/ops_all.cpp
ProcessRowsOMP(...);
ProcessRowsSTL(...);
ProcessRowsTBB(...);
```

```cpp
// File: all/src/ops_all.cpp
MPI_Barrier(MPI_COMM_WORLD);
```

Возможное узкое место этой версии - накладные расходы гибридной схемы. Для изображения `512×512`
объём вычислений не очень большой, поэтому расходы на MPI-запуск, барьер и несколько способов
организации потоков становятся заметными.

Также важно, что отсутствие настоящего распределения изображения между rank-ами ограничивает пользу от
MPI-уровня. Поэтому результат `ALL` корректен, но не обязан быть быстрее чистых потоковых версий.

Корректность `ALL` проверялась функциональными тестами отдельно через `mpiexec`.

Использовались те же логические случаи, что и для остальных backend-ов:

| Тест             | Что проверяет          | Ожидаемый результат                                 |

| `ConstantImage`  | Однотонное изображение | Все значения равны нулю                             |
| `VerticalEdge`   | Вертикальная граница   | Внутренние пиксели на границе имеют значение `1020` |
| `HorizontalEdge` | Горизонтальная граница | Внутренние пиксели на границе имеют значение `1020` |

Команда запуска функционального теста для `ALL`:

```powershell
$env:PPC_NUM_THREADS="4"
$env:OMP_NUM_THREADS="4"
$env:PPC_NUM_PROC="2"

mpiexec -n 2 .\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_all*
```

По результату запуска:

```text
[  PASSED  ] 3 tests.
```

Это подтверждает, что гибридная версия выдаёт корректный результат на проверочных изображениях.

Отсутствие гонок обеспечивается тем, что:

- входное изображение только читается;
- строки делятся на непересекающиеся диапазоны;
- каждая технология пишет только в свой участок выходного массива;
- общей переменной-аккумулятора нет.

## 10. Выводы

Версия `ALL` показывает гибридную организацию вычислений, где используются MPI, OpenMP, `std::thread` и oneTBB.

Главные особенности реализации:

- MPI используется для запуска нескольких процессов и барьерной синхронизации;
- внутри процесса строки делятся между OpenMP, `std::thread` и TBB;
- результат записывается в общий выходной массив по непересекающимся диапазонам строк;
- `Reduce`, `Broadcast`, `Gather` не используются, потому что изображение дополнительно
не распределяется между rank-ами;
- корректность обеспечивается независимостью диапазонов записи.

На изображении `512×512` гибридная версия работает корректно, но не показывает преимущества перед чистыми
потоковыми backend-ами. Для такого размера задачи расходы на гибридную организацию заметны по сравнению с
полезной вычислительной работой.

Гибридная схема была бы более оправдана для более крупных данных или для реализации, где изображение
действительно распределяется между MPI-процессами, а затем внутри каждого процесса дополнительно
обрабатывается потоками.
