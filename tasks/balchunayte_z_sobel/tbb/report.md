## 1. Контекст

Версия `TBB` основана на последовательной реализации `SEQ`, но основной проход по изображению выполняется через oneTBB.

Для этой задачи подходит `parallel_for`, потому что вычисление оператора Собеля для разных строк
изображения можно выполнять независимо. Каждый поток читает входное изображение и записывает результат
только в свой участок выходного массива.

В отличие от `SEQ`, где строки обрабатываются строго последовательно, `TBB` передаёт выполнение итераций
runtime-библиотеке oneTBB. Она сама распределяет работу между доступными worker-потоками.

## 2. Постановка задачи

На вход подаётся RGB-изображение:

```cpp
struct Image {
  int width = 0;
  int height = 0;
  std::vector<Pixel> data;
};
```

Каждый пиксель содержит три цветовые компоненты:

```cpp
struct Pixel {
  uint8_t r = 0;
  uint8_t g = 0;
  uint8_t b = 0;
};
```

Выходной результат:

```cpp
using OutType = std::vector<int>;
```

Каждый элемент выходного массива содержит величину градиента для соответствующего пикселя.

Как и в `SEQ`, обрабатываются только внутренние пиксели изображения. Граничные строки и столбцы остаются равными нулю.

Последовательная версия `SEQ` используется как baseline:

- для проверки корректности;
- для сравнения времени выполнения;
- для расчёта ускорения `TBB`.

## 3. Базовый алгоритм

Последовательная версия проходит по внутренним пикселям изображения и для каждого пикселя применяет оператор Собеля.

RGB-пиксели переводятся в оттенки серого:

```text
gray = (77 * r + 150 * g + 29 * b) >> 8
```

Затем по окрестности `3×3` вычисляются два значения:

```text
Gx — горизонтальный градиент
Gy — вертикальный градиент
```

В выходной массив записывается:

```text
|Gx| + |Gy|
```

В версии `TBB` сама формула не меняется. Изменяется только способ выполнения внешнего цикла по строкам.

## 4. Схема распараллеливания

В реализации выбран примитив:

```cpp
oneapi::tbb::parallel_for
```

Он подходит для задачи, где есть набор независимых итераций. В данной работе такой набор - это внутренние
строки изображения.

Основной фрагмент:

```cpp
// File: tbb/src/ops_tbb.cpp
oneapi::tbb::parallel_for(1, image_height - 1, [&](int row_index) {
  for (int col_index = 1; col_index < image_width - 1; ++col_index) {
    // compute Sobel for one pixel
  }
});
```

Параллельный диапазон:

```text
[1; image_height - 1)
```

То есть TBB обрабатывает строки от первой внутренней строки до последней внутренней строки.
Нулевая строка и последняя строка не включаются, потому что для них нельзя применить окно `3×3`.

### 4.1. Диапазон работы

Единицей работы является строка изображения.

Для каждой строки выполняется внутренний последовательный цикл по столбцам:

```cpp
for (int col_index = 1; col_index < image_width - 1; ++col_index)
```

Такое разбиение удобно, потому что:

- строки независимы друг от друга;
- каждая строка имеет примерно одинаковый объём работы;
- запись результата идёт в разные элементы выходного массива;
- не требуется синхронизация между строками.

### 4.2. Grainsize

В этой реализации явно не используется `blocked_range` и явно не задаётся `grainsize`.

Используется простая форма `parallel_for(start, end, lambda)`. В этом случае разбиение диапазона на задачи
выполняет oneTBB по своему стандартному механизму для данного overload.

Это решение подходит для данной задачи, потому что нагрузка по строкам равномерная. Каждая строка содержит
одинаковое количество внутренних пикселей, и для каждого пикселя выполняется одна и та же формула Собеля.

Если бы строки имели сильно разную сложность, тогда имело бы смысл вручную использовать `blocked_range` и
подбирать `grainsize`. В данной задаче это не требуется.

### 4.3. Partitioner

Partitioner явно не задаётся.

Это означает, что используется стандартное поведение oneTBB для выбранной формы `parallel_for`.

В данной задаче это допустимо, потому что:

- диапазон строк простой;
- работа по строкам равномерная;
- нет общей операции объединения результатов;
- нет необходимости вручную управлять балансировкой.

### 4.4. Ограничение конкуренции

В коде задачи явно не создаётся `oneapi::tbb::global_control`.

Ограничение уровня параллелизма задавалось через тестовое окружение курса:

```powershell
$env:PPC_NUM_THREADS="8"
```

На компьютере было доступно 8 логических процессоров, поэтому для сравнения использовалось 8 workers.

Важно, что для TBB это не означает ручное создание ровно 8 потоков в коде задачи. Это ограничение уровня
параллелизма на уровне runtime и тестового окружения.

### 4.5. Синхронизация

В этой задаче не нужны `mutex`, `atomic` или `critical`.

Причина в том, что каждая итерация по строке записывает результат в свои индексы выходного массива:

```cpp
output_data[output_index] = magnitude;
```

Входное изображение только читается. Общей переменной-аккумулятора нет. Поэтому `parallel_reduce` тоже не требуется.

## 5. Детали реализации

Файлы реализации:

```text
tbb/include/ops_tbb.hpp
tbb/src/ops_tbb.cpp
```

Класс TBB-версии:

```cpp
BalchunayteZSobelOpTBB
```

Как и остальные backend-и, он использует общий pipeline:

```text
ValidationImpl → PreProcessingImpl → RunImpl → PostProcessingImpl
```

`ValidationImpl` проверяет входные данные:

- ширина изображения больше нуля;
- высота изображения больше нуля;
- размер массива `data` равен `width * height`;
- выходной массив перед запуском пустой.

`PreProcessingImpl` создаёт выходной массив и заполняет его нулями.

`RunImpl` выполняет основное вычисление. Именно здесь используется `oneapi::tbb::parallel_for`.

`PostProcessingImpl` не выполняет дополнительной обработки.

Главное отличие от `SEQ` находится в `RunImpl`: внешний цикл по строкам передан в `parallel_for`.

Локальные аккумуляторы не используются, потому что задача не считает общую сумму или максимум.
Для каждого пикселя сразу вычисляется отдельное значение и записывается в выходной массив.

Работа с памятью остаётся простой:

- `input_image` используется только для чтения;
- `output_data` используется для записи;
- каждый поток пишет в непересекающийся набор индексов.

## 6. Проверка корректности

Корректность `TBB` проверялась теми же функциональными тестами, что и `SEQ`.

Используются три основных случая:

| Тест             | Что проверяет          | Ожидаемый результат                                 |
|------------------|------------------------|-----------------------------------------------------|
| `ConstantImage`  | Однотонное изображение | Все значения равны нулю                             |
| `VerticalEdge`   | Вертикальная граница   | Внутренние пиксели на границе имеют значение `1020` |
| `HorizontalEdge` | Горизонтальная граница | Внутренние пиксели на границе имеют значение `1020` |

Тесты выполнялись на изображениях размером `4×4`.

Команда запуска функционального теста для `TBB`:

```powershell
$env:PPC_NUM_THREADS="8"
$env:OMP_NUM_THREADS="8"

.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_tbb*
```

Также `TBB` запускался вместе с другими backend-ами:

```powershell
.\build\bin\ppc_func_tests.exe --gtest_filter=*balchunayte_z_sobel_seq*:*balchunayte_z_sobel_omp*:
*balchunayte_z_sobel_tbb*:*balchunayte_z_sobel_stl*
```

В общем запуске для `SEQ`, `OMP`, `TBB`, `STL` прошло:

```text
[  PASSED  ] 12 tests.
```

Это подтверждает, что `TBB` сохраняет ту же логику результата, что и последовательная версия.

## 8. Результаты

Baseline `SEQ`:

| Mode       |  SEQ time, s |
|------------|-------------:|
| `task_run` | 0.0107273000 |
| `pipeline` | 0.0202614000 |

Результаты `TBB`:

| Backend | Mode       |    Size | Workers |      Time, s | Speedup vs SEQ | Efficiency |
|---------|------------|--------:|--------:|-------------:|---------------:|-----------:|
| TBB     | `task_run` | 512×512 |       8 | 0.0038740800 |           2.77 |     34.61% |
| TBB     | `pipeline` | 512×512 |       8 | 0.0044883200 |           4.51 |     56.43% |

В режиме `task_run` версия `TBB` ускорилась примерно в `2.77` раза относительно `SEQ`.

В режиме `pipeline` ускорение составило примерно `4.51` раза.

Результат показывает, что oneTBB хорошо справляется с равномерным разбиением работы по строкам.
При этом ускорение не является линейным относительно 8 workers. Часть времени уходит на работу runtime,
разбиение диапазона и управление задачами.

## 9. Выводы

TBB-версия хорошо подходит для этой задачи, потому что вычисление Собеля естественно делится на
независимые строки изображения.

В реализации используется `oneapi::tbb::parallel_for`. Это позволяет передать распределение итераций
runtime-библиотеке oneTBB и не создавать потоки вручную.

Основные причины корректности:

- входное изображение только читается;
- строки изображения обрабатываются независимо;
- каждый worker записывает результат в свой участок выходного массива;
- общей переменной для накопления результата нет;
- синхронизация через `mutex`, `atomic` или `critical` не требуется.

На изображении `512×512` версия `TBB` дала заметное ускорение относительно `SEQ`. В режиме `pipeline` она
оказалась немного быстрее `OMP`, а в режиме `task_run` немного медленнее `OMP`.

Ограничение результата связано с накладными расходами oneTBB runtime. Для небольших изображений эти
расходы могут быть заметны, а для более крупных входных данных TBB обычно имеет больше возможностей для
эффективного распределения работы.
