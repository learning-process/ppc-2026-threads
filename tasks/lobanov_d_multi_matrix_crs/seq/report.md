## 1. Контекст

Последовательная реализация является фундаментом проекта.
Она задаёт эталонный результат вычислений и служит точкой отсчёта для измерения ускорения всех параллельных версий.
Код написан без использования средств синхронизации, полностью детерминирован и максимально прозрачен для верификации.

## 2. Постановка задачи

Входные данные — две разреженные матрицы `A` и `B` в формате CRS (Compressed Row Storage):

- `row_count`, `column_count` – размеры;
- `row_pointer_data` – смещения начала строк (длина `row_count + 1`);
- `column_index_data` – индексы столбцов ненулевых элементов;
- `value_data` – значения элементов (тип `double`).

Выходные данные — матрица `C = A × B` в том же формате.  
Условие корректности: `A.column_count == B.row_count`. Элементы, по модулю меньшие `1e-15`, отбрасываются.

## 3. Базовый алгоритм

1. **Транспонирование** матрицы `B` – для быстрого доступа к столбцам `B` как к строкам `Bᵀ`.
2. **Основной цикл** – для каждой строки `i` матрицы `A` и каждой строки `j`
матрицы `Bᵀ` вычисляется скалярное произведение методом двух указателей.
3. **Формирование CRS результата** – ненулевые суммы сохраняются в `value_data`
и `column_index_data`, массив `row_pointer_data` строится накоплением.  

**Сложность:**  

- Транспонирование: `O(nnz(B) + rows_B)`.  
- Умножение: в худшем случае `O(rows_A · cols_B · max(row_nnz(A), row_nnz(B)))`.  
- Память: `O(nnz(C) + rows_A)` плюс временная память для `Bᵀ`.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

Класс `LobanovMultyMatrixSEQ` наследует `BaseTask` и реализует:

- `ValidationImpl()` – проверка согласованности размеров и положительности.
- `PreProcessingImpl()` – инициализация выходной структуры.
- `RunImpl()` – вызов `PerformMatrixMultiplication()`, где выполняется транспонирование и двойной цикл.  
- `PostProcessingImpl()` – финальная проверка целостности `row_pointer_data`.  

Вспомогательная функция `MultiplyRowByMatrix` изолирует логику обработки одной строки:  
использует временные массивы `temp_row` и `temp_col_markers` для накопления произведений без повторных аллокаций.

**Ключевой фрагмент:**

```cpp
// seq/src/ops_seq.cpp – фрагмент MultiplyRowByMatrix
for (int b_ptr = matrix_b.row_pointer_data[b_row];
     b_ptr < matrix_b.row_pointer_data[b_row+1]; ++b_ptr) {
    int b_col = matrix_b.column_index_data[b_ptr];
    double b_val = matrix_b.value_data[b_ptr];
    if (temp_col_markers[b_col] != row_index) {
        temp_col_markers[b_col] = row_index;
        temp_row[b_col] = a_val * b_val;
    } else {
        temp_row[b_col] += a_val * b_val;
    }
}
```

## 7. Основные выводы

1. **Роль baseline:** время `T_seq = 389.7 мс` будет использовано
   как знаменатель при расчёте ускорения для всех параллельных реализаций
   (`speedup = T_seq / T_par`).

2. **Независимость строк:** вычисление каждой строки результирующей матрицы `C`
   не зависит от других строк. Это делает задачу идеальным кандидатом
   для распараллеливания внешнего цикла по строкам.

3. **Узкое место:** профилирование подтверждает, что основное время (≈96%)
   уходит на вложенный цикл скалярных произведений.
   Любое распараллеливание должно быть направлено именно на этот участок.

4. **Эффективность памяти:** переиспользование временных буферов `temp_row`
   и `temp_marker` для каждой строки минимизирует количество аллокаций
   и положительно влияет на производительность.

5. **Фундамент для параллельных версий:** данная реализация служит эталоном
   корректности для всех последующих реализаций
   (OpenMP, TBB, STL, гибридной MPI+OpenMP).
