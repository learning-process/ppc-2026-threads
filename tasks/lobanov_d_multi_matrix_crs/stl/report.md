## 1. Контекст

Реализация на `std::thread` из стандартной библиотеки C++ даёт полный контроль
над жизненным циклом потоков, разбиением данных и синхронизацией.
В отличие от высокоуровневых решений (OpenMP, TBB), где управление параллелизмом
частично или полностью автоматизировано, здесь разработчик самостоятельно
реализует декомпозицию задачи, распределение нагрузки и координацию потоков.
Эта версия демонстрирует «ручное» управление параллелизмом на низком уровне
и позволяет оценить накладные расходы системных вызовов, а также понять
фундаментальные принципы многопоточного программирования.

## 2. Постановка задачи

**Входные данные:** две разреженные матрицы `A` и `B` в CRS-формате
(Compressed Row Storage). Каждая описывается:

- `row_count`, `column_count` – размерности;
- `row_pointer_data` – массив смещений начала строк;
- `column_index_data` – индексы столбцов ненулевых элементов;
- `value_data` – значения элементов (тип `double`).

**Выходные данные:** матрица `C = A × B` в том же CRS-формате.

**Ограничения:** `A.column_count == B.row_count`, все размерности положительны.
Элементы, по модулю меньшие `1e-12`, отбрасываются.

**Базовая линия (baseline):** последовательная реализация SEQ с временем
выполнения **389.7 мс**.

## 3. Базовый алгоритм

Используется тот же алгоритм, что и в SEQ:

1. **Транспонирование** матрицы `B` → получение `B^T` в CRS-формате.
2. **Вычисление скалярных произведений** – для каждой строки `i` матрицы `A`
   и каждой строки `j` матрицы `B^T` методом двух указателей.
3. **Формирование выходной CRS** – накопление ненулевых элементов в
   результирующие массивы.

Отличие от SEQ заключается в ручном распараллеливании внешнего цикла
(по строкам `A`) между созданными потоками.

## 4. Схема распараллеливания

### 4.1 Разбиение данных на диапазоны

Применяется **статическая блочная декомпозиция** строк матрицы `A`.
Пространство итераций (строки) делится на непрерывные непересекающиеся блоки,
которые распределяются между потоками.

**Алгоритм разбиения:**

```cpp
int rows = A.row_count;
int num_threads = ppc::util::GetNumThreads();
int chunk = rows / num_threads;
int remainder = rows % num_threads;
int start = 0;

std::vector<std::thread> threads;
std::vector<std::vector<double>> temp_values(rows);
std::vector<std::vector<int>> temp_cols(rows);

for (int t = 0; t < num_threads; ++t) {
    int end = start + chunk + (t < remainder ? 1 : 0);
    if (start < end) {
        threads.emplace_back([&, start, end] {
            for (int i = start; i < end; ++i) {
                // Вычисление строки i
                std::vector<double> local_vals;
                std::vector<int> local_cols;

                for (int j = 0; j < B.column_count; ++j) {
                    double sum = DotProduct(A, i, Bt, j);
                    if (std::abs(sum) > 1e-12) {
                        local_vals.push_back(sum);
                        local_cols.push_back(j);
                    }
                }

                temp_values[i] = std::move(local_vals);
                temp_cols[i] = std::move(local_cols);
            }
        });
    }
    start = end;
}
```
Каждый поток создаёт локальные векторы local_vals и local_cols на своей стековой памяти.   
После вычисления строки векторы перемещаются (std::move) в глобальный массив по уникальному индексу i.   
Такой подход:    

- Исключает необходимость в мьютексах или атомарных операциях  
- Минимизирует копирование данных  
- Гарантирует потокобезопасность за счёт архитектуры  

```cpp 
// 1. Создание и запуск всех потоков
for (int t = 0; t < num_threads; ++t) {
    threads.emplace_back(worker_function, ...);
}

// 2. Ожидание завершения всех потоков (барьер)
for (auto& thread : threads) {
    thread.join();
}

// 3. Последовательная сборка результата (только после join)
AssembleResult(temp_values, temp_cols, result);
```

```cpp
void AssembleResult(const std::vector<std::vector<double>>& temp_values,
                    const std::vector<std::vector<int>>& temp_cols,
                    CompressedRowMatrix& result) {
    int rows = temp_values.size();
    result.row_pointer_data.resize(rows + 1, 0);
    
    // Шаг 1: вычисление префиксных сумм (row_pointers)
    for (int i = 0; i < rows; ++i) {
        result.row_pointer_data[i + 1] = result.row_pointer_data[i] + temp_values[i].size();
    }
    
    // Шаг 2: выделение памяти под результирующие массивы
    int total_nnz = result.row_pointer_data[rows];
    result.value_data.reserve(total_nnz);
    result.column_index_data.reserve(total_nnz);
    
    // Шаг 3: копирование данных из двумерной структуры в плоские массивы
    for (int i = 0; i < rows; ++i) {
        result.value_data.insert(result.value_data.end(), 
                                 temp_values[i].begin(), 
                                 temp_values[i].end());
        result.column_index_data.insert(result.column_index_data.end(), 
                                        temp_cols[i].begin(), 
                                        temp_cols[i].end());
    }
    
    result.non_zero_count = total_nnz;
}
```

## 8. Выводы

Ручная реализация с использованием `std::thread` продемонстрировала следующие
результаты:

1. **Ответственность за декомпозицию:** Разработчик должен самостоятельно
   реализовать разбиение данных, учитывая остатки от деления и граничные случаи.
   В отличие от `auto_partitioner` из TBB, STL не прощает ошибок в расчёте
   диапазонов.

2. **Цена системных вызовов:** Создание `std::thread` – дорогая операция.
   Для задач малого объёма или при многократном вызове `RunImpl` использование
   STL без пула потоков нецелесообразно.

3. **Бесшовная потокобезопасность:** Отказ от примитивов синхронизации в пользу
   разнесения записи по независимым буферам памяти – правильное архитектурное
   решение. Алгоритм не тратит время на ожидание мьютексов.
