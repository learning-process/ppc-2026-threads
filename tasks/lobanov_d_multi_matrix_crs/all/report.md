## 1. Контекст

Гибридная модель программирования объединяет два уровня параллелизма: межпроцессный (MPI) и внутрипроцессный (OpenMP). Такой подход позволяет эффективно использовать как многоядерные процессоры в рамках одного вычислительного узла, так и масштабироваться на кластеры, состоящие из множества узлов. MPI отвечает за распределение данных и обмен сообщениями между процессами, а OpenMP распараллеливает вычисления внутри каждого процесса на все доступные ядра. Данная реализация демонстрирует, как можно совместить эти две технологии для достижения максимальной производительности на задаче умножения разреженных матриц.

## 2. Постановка задачи

**Входные данные:** две разреженные матрицы `A` и `B` в CRS-формате (Compressed Row Storage). Каждая описывается:
- `row_count`, `column_count` – размерности;
- `row_pointer_data` – массив смещений начала строк;
- `column_index_data` – индексы столбцов ненулевых элементов;
- `value_data` – значения элементов (тип `double`).

**Выходные данные:** матрица `C = A × B` в том же CRS-формате.

**Ограничения:** `A.column_count == B.row_count`, все размерности положительны. Элементы, по модулю меньшие `1e-12`, отбрасываются.

**Базовая линия (baseline):** последовательная реализация SEQ с временем выполнения **389.7 мс**.

## 3. Базовый алгоритм

Используется тот же алгоритм, что и в SEQ:

1. **Транспонирование** матрицы `B` → получение `B^T` в CRS-формате (выполняется только на процессе 0).
2. **Вычисление скалярных произведений** – для каждой строки `i` матрицы `A` и каждой строки `j` матрицы `B^T` методом двух указателей.
3. **Формирование выходной CRS** – накопление ненулевых элементов в результирующие массивы.

## 4. Межпроцессная схема (MPI)

### 4.1 Модель Master-Worker

Используется модель **Master-Worker** (или SPMD с выделенным мастером):

- **Процесс 0 (Master):**
  - Читает входные данные из фреймворка (`GetInput()`).
  - Выполняет транспонирование матрицы `B`.
  - Рассылает данные всем процессам.
  - Собирает локальные результаты.
  - Формирует итоговую выходную матрицу (`GetOutput()`).

- **Процессы 1..P-1 (Workers):**
  - Получают копии матриц.
  - Выполняют вычисления для своей порции строк.
  - Отправляют результаты мастеру.

### 4.2 Распределение данных

В отличие от чистого MPI, где данные обычно разделяются между процессами, в данной гибридной реализации применена стратегия **полного дублирования данных (Full Replication)**:

```cpp
void DistributeSparseMatrix(CompressedRowMatrix& mat, int root, int rows, int cols) {
    int rank;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    
    // Шаг 1: рассылка размеров
    std::array<int, 2> sizes = {0, 0};
    if (rank == root) {
        sizes[0] = mat.row_pointer_data.size();
        sizes[1] = mat.value_data.size();
    }
    MPI_Bcast(sizes.data(), 2, MPI_INT, root, MPI_COMM_WORLD);
    
    // Шаг 2: подготовка памяти на рабочих процессах
    if (rank != root) {
        mat.row_pointer_data.resize(sizes[0]);
        mat.value_data.resize(sizes[1]);
        mat.column_index_data.resize(sizes[1]);
    }
    
    // Шаг 3: рассылка данных
    if (sizes[0] > 0) {
        MPI_Bcast(mat.row_pointer_data.data(), sizes[0], MPI_INT, root, MPI_COMM_WORLD);
    }
    if (sizes[1] > 0) {
        MPI_Bcast(mat.value_data.data(), sizes[1], MPI_DOUBLE, root, MPI_COMM_WORLD);
        MPI_Bcast(mat.column_index_data.data(), sizes[1], MPI_INT, root, MPI_COMM_WORLD);
    }
    
    mat.row_count = rows;
    mat.column_count = cols;
}
```

Изоляция: до OMP-секции создаются std::vector<std::vector<double>> row_vals(local_rows).  

Безопасная запись: внутри цикла каждый поток записывает в row_vals[i] с уникальным индексом i.  

Неявный барьер: после #pragma omp parallel for все потоки синхронизируются.  

Объединение:     главный поток процесса «сплющивает» двумерную структуру в плоские буферы.

2. Влияние топологии (8 воркеров, 4×2 vs 2×4):
Разница в 1.5 раза объясняется:  

Меньшая конкуренция за кэш: 4 процесса имеют изолированные адресные пространства, каждый использует свою часть L3.  

Меньшая нагрузка на планировщик OpenMP: 2 потока на процесс проще балансировать, чем 4.  

Лучшая локальность данных: каждый процесс обрабатывает меньше строк, данные лучше помещаются в кэш.
