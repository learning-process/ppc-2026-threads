## 1. Контекст

Гибридная версия (`ALL`) предназначена для демонстрации работы в распределенных системах или на узлах с NUMA-архитектурой.
Алгоритм сочетает два уровня параллелизма:

1. **Межпроцессный (MPI):** Распределение исходного массива между процессами для независимой обработки.
2. **Внутрипроцессный (TBB):** Использование библиотеки oneTBB для параллельной локальной сортировки и, что важно,
для параллельного слияния отсортированных блоков на главном процессе.

В отличие от чисто shared-memory решений, здесь данные физически копируются между процессами,
что позволяет изолировать кэш-память каждого ядра/процесса на этапе сортировки.

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив `std::vector<int>` (на root-процессе).
- **Выходные данные:** Отсортированный массив на всех процессах (синхронизированный через `MPI_Bcast`).
- **Цель:** Реализовать корректную сортировку в гибридной среде, минимизируя время коммуникации и эффективно используя
многоядерность узлов.
- **Эталон:** Последовательная версия (`SEQ`).

## 3. Базовый алгоритм

Локальная обработка данных выполняется с помощью **LSD Radix Sort**.

- **Обработка знака:** Перед сортировкой применяется инверсия старшего бита (`XOR INT32_MIN`),
после сортировки — обратная инверсия. Это вынесено в отдельную функцию `InvertSignBit`.
- **Локальная сортировка:** Функция `RadixSortLocal` выполняет 4 прохода подсчета с использованием временного буфера.

## 4. Схема распараллеливания

Алгоритм следует схеме **Master-Worker** с последующим централизованным слиянием.

### Этап 1: Распределение данных (MPI Scatter)

Исходный массив на `rank 0` разбивается на `world_size` блоков. Размеры блоков вычисляются так,
чтобы разница между ними не превышала 1 элемент.

- Используется `MPI_Scatterv` для передачи данных всем процессам (включая root).
- Если запуск не под MPI (`IsUnderMpirun() == false`), данные просто копируются локально.

### Этап 2: Локальная сортировка (TBB / Sequential)

Каждый процесс независимо сортирует свой локальный блок `local_data`.

### Этап 3: Сбор данных (MPI Gather)

Отсортированные локальные блоки собираются обратно на `rank 0` с помощью `MPI_Gatherv`.
На этом этапе массив на `rank 0` представляет собой конкатенацию `world_size` отсортированных подмассивов.

### Этап 4: Глобальное слияние (TBB Parallel Merge)

Только на `rank 0` выполняется финальное слияние всех блоков в один отсортированный массив.
Для ускорения этого этапа используется **`tbb::parallel_for`**:

```cpp
void ParallelMerge(std::vector<int> &arr, const std::vector<int> &offsets, int num_blocks) {
  for (int step = 1; step < num_blocks; step *= 2) {
    tbb::parallel_for(0, num_blocks, [&, step](int i) {
      if (i % (2 * step) == 0 && i + step < num_blocks) {
        // ... вычисление границ и std::inplace_merge ...
      }
    });
  }
}
```

Это позволяет задействовать все ядра узла, на котором запущен Master-процесс, для быстрого объединения результатов.

### Этап 5: Синхронизация результата (MPI Broadcast)

Полностью отсортированный массив рассылается всем остальным процессам через `MPI_Bcast`,
чтобы состояние `GetOutput()` было идентичным на всех рангах.

## 5. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

### Управление памятью и MPI

- **`DistributeData`:** Выделяет `local_data` точно под размер полученного куска. Это экономит память на worker-узлах.
- **`GatherData`:** На `rank 0` выделяется полный массив `output`. На worker-узлах выходной буфер не используется для
приема.
- **Обработка пустых данных:** Добавлены проверки `if (n == 0)` и `if (local_size == 0)`,
чтобы избежать ошибок MPI при передаче нулевых буферов.

### Инверсия знака

Выполняется локально на каждом процессе до и после сортировки:

```cpp
constexpr int32_t kSignMask = INT32_MIN;
InvertSignBit(local_data, kSignMask);
RadixSortLocal(local_data.begin(), local_data.end());
InvertSignBit(local_data, kSignMask);
```

Это гарантирует, что обмен данными через MPI происходит уже с корректными знаковыми значениями (или что знак
восстанавливается до сборки). *В данном коде инверсия делается до отправки обратно,
поэтому в `GatherData` уходят правильные числа.*

### Параллельное слияние на TBB

Использование `tbb::parallel_for` вместо `omp parallel for` требует захвата переменных по ссылке `[&]`.

- **Фильтрация задач:** Условие `if (i % (2 * step) == 0 ...)` внутри лямбды означает,что часть задач TBB будет
"пустыми". Это компромисс ради простоты кода, позволяющий использовать единый диапазон `[0, num_blocks)`.
- **Независимость:** Операции `inplace_merge` для разных пар блоков на одном шаге `step` независимы,
что безопасно для параллельного выполнения.

## 6. Проверка корректности

- **Функциональные тесты:** Сравнение результата на `rank 0` с `std::ranges::sort`.
- **MPI-корректность:** Проверка того, что все процессы получают одинаковый отсортированный массив после `Bcast`.
- **Граничные случаи:** Корректная работа при `world_size > n` (некоторые процессы получат пустые блоки),
а также при `n=0`.

### Анализ результатов

1. **Накладные расходы MPI:** При малом числе рангов (1-2) время коммуникации (`Scatterv`/`Gatherv` копированием 40 МБ
данных) существенно влияет на общее время.
2. **Эффективность слияния:** Использование TBB для финального слияния на `rank 0` позволяет быстро объединить
результаты. Без этого слияние стало бы "бутылочным горлышком" на одном потоке.
3. **Масштабируемость:** Ускорение ограничено скоростью сборки данных на мастере и пропускной способностью памяти при
слиянии. Дальнейшее увеличение рангов свыше числа физических ядер не даст прироста из-за конкуренции за ресурсы узла.

## 9. Выводы

1. **Гибридный подход оправдан** для больших данных, где объем вычислений (сортировка) значительно превышает объем
коммуникаций. Разделение массива на части позволяет каждому процессу работать со своим куском кэш-памяти,
улучшая локальность данных.
2. **Роль TBB:** Использование `tbb::parallel_for` для финального слияния критически важно для производительности
Master-узла. Последовательное слияние 8+ блоков заняло бы значительную часть времени.
3. **Ограничения MPI:** Для массивов размером 10M элементов накладные расходы на передачу данных через MPI (даже в
рамках одного узла через shared memory backend MPI) заметны. Алгоритм станет более эффективным на массивах от 100M+
элементов или в настоящем кластере.
4. **Корректность:** Схема с полной синхронизацией через `Bcast` гарантирует консистентность данных на всех процессах,
что важно для дальнейших этапов распределенных вычислений.

Данная реализация демонстрирует успешную интеграцию MPI для распределения данных и TBB для параллельной обработки и
слияния, обеспечивая модульность и масштабируемость алгоритма.
