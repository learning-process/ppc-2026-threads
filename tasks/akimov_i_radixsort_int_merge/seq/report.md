## 1. Контекст

В данном отчете рассматривается базовая последовательная реализация алгоритма поразрядной сортировки (`Radix Sort`).
Данная версия реализует подход LSD (Least Significant Digit) с предварительной трансформацией данных для корректной
работы со знаковыми типами `int`.

Эта реализация служит эталоном (`SEQ`) для измерения ускорения в параллельных версиях (`OMP`, `TBB`, `STL`, `ALL`) и
гарантирует математическую корректность алгоритма перед распараллеливанием.

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив целых чисел `std::vector<int>`.
- **Выходные данные:** Тот же массив, отсортированный по неубыванию.
- **Ограничения:**
  - Тип данных: `int` (32 бита).
  - Алгоритм должен корректно обрабатывать весь диапазон значений, включая `INT_MIN` и `INT_MAX`.
  - Пустые массивы считаются невалидными входными данными согласно реализации `ValidationImpl`.

## 3. Базовый алгоритм

Выбран алгоритм **LSD Radix Sort** (сортировка от младшего разряда к старшему) с основанием 256 (побайтовая
сортировка).

### Особенности обработки знаковых чисел

Стандартная поразрядная сортировка работает с беззнаковыми числами. Для корректной сортировки `signed int` используется
метод **инверсии старшего бита**:

1. Перед началом сортировки каждый элемент массива подвергается операции XOR с маской `0x80000000` (`INT32_MIN`).
Это меняет порядок следования отрицательных и положительных чисел так, что они выстраиваются в единый линейный порядок,
пригодный для сортировки как беззнаковые величины.
2. Выполняется 4 прохода устойчивой сортировки подсчетом (по одному на каждый байт `int`).
3. После завершения всех проходов применяется обратная операция XOR с той же маской,
чтобы восстановить исходные значения битов, но уже в отсортированном порядке.

- **Асимптотика по времени:** $O(d \cdot (N + K))$, где $N$ — размер массива, $d=4$ (количество байт),
$K=256$ (размер алфавита/корзины). Фактически $O(N)$.
- **Асимптотика по памяти:** $O(N)$ для временного буфера `temp` и $O(K)$ для массива частот `count`.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

Класс `AkimovIRadixSortIntMergeSEQ` наследуется от `BaseTask`.

### Основные методы

- **`ValidationImpl()`**:
  Проверяет, что входной вектор не пуст.

  ```cpp
  return !GetInput().empty();
  ```

- **`PreProcessingImpl()`**:
  Копирует входные данные в выходной буфер для работы алгоритма.

  ```cpp
  GetOutput() = GetInput();
  ```

- **`RunImpl()`**:
  Реализует основную логику сортировки:
  1. **Инверсия знака:** Проход по всему массиву с применением `x ^= INT32_MIN`.
  2. **Цикл по байтам:** Выполняется 4 итерации (`byte_pos` от 0 до 3).
     - Подсчет частот встречаемости каждого байта (0–255).
     - Преобразование частот в префиксные суммы (cumulative count) для определения позиций.
     - Распределение элементов во временный массив `temp` в обратном порядке (для сохранения устойчивости сортировки).
     - Обмен содержимого основного массива и временного (`arr.swap(temp)`).
  3. **Обратная инверсия:** Повторный проход с `x ^= INT32_MIN` для восстановления знаков.

- **`PostProcessingImpl()`**:
  Возвращает `true`, так как результат уже готов в `GetOutput()`.

### Ключевой фрагмент кода (RunImpl)

```cpp
constexpr int32_t kSignMask = INT32_MIN;  // 0x80000000

// 1. Инверсия старшего бита для корректной работы со знаком
for (int &x : arr) {
  x ^= kSignMask;
}

const int num_bytes = static_cast<int>(sizeof(int));
std::vector<int> temp(arr.size());

// 2. Поразрядная сортировка
for (int byte_pos = 0; byte_pos < num_bytes; ++byte_pos) {
  std::vector<int> count(256, 0);
  
  // Подсчет
  for (int x : arr) {
    uint8_t byte = (x >> (byte_pos * 8)) & 0xFF;
    ++count[byte];
  }

  // Префиксные суммы
  for (int i = 1; i < 256; ++i) {
    count[i] += count[i - 1];
  }

  // Распределение (stable sort)
  for (int i = static_cast<int>(arr.size()) - 1; i >= 0; --i) {
    uint8_t byte = (arr[i] >> (byte_pos * 8)) & 0xFF;
    temp[--count[byte]] = arr[i];
  }

  arr.swap(temp);
}

// 3. Возврат знаков
for (int &x : arr) {
  x ^= kSignMask;
}
```

## 5. Проверка корректности

Корректность проверяется путем сравнения результата работы алгоритма с `std::ranges::sort` (эталонная сортировка из
STL).

**Тестовые сценарии (из `tests/functional/main.cpp`):**
Генерация случайных массивов размером 10, 100 и 1000 элементов со значениями в диапазоне `[-1000, 1000]`.

**Проверяемые свойства:**

1. **Устойчивость к знакам:** Благодаря инверсии `INT32_MIN`, отрицательные числа корректно располагаются перед
положительными.
2. **Граничные значения:** Алгоритм корректно обрабатывает `INT_MIN` и `INT_MAX`,
так как они просто интерпретируются как большие беззнаковые значения после инверсии.
3. **Дубликаты:** Устойчивая сортировка подсчетом сохраняет относительный порядок равных элементов.

### Анализ затрат времени

Основное время выполнения складывается из следующих операций:

1. **Инверсия знака (Pre/Post-processing):** 2 прохода по 10 млн элементов (~10-12 мс).
Операция XOR очень быстрая и хорошо векторизуется.
2. **Сортировка (4 прохода):**
   - **Подсчет частот:** Быстрый линейный проход.
   - **Распределение (Scatter):** Самая ресурсоемкая часть. Запись во временный массив `temp` по косвенным адресам
   приводит к промахам кэша данных (L1/L2 cache misses). На архитектуре Zen 3+ (6800HS) контроллер памяти эффективен,
   но латентность остается главным ограничителем скорости.
3. **Выделение памяти:** Вектор `temp` выделяется один раз. Если бы он выделялся внутри цикла,
время было бы значительно выше из-за системных вызовов `malloc/new`.

## 8. Выводы

1. **Корректность:** Реализованная SEQ-версия успешно проходит все функциональные тесты.
Использование маски `INT32_MIN` через XOR является эффективным и стандартным способом адаптации Radix Sort для знаковых
целых чисел без ветвлений внутри внутреннего цикла сортировки.
2. **Эталонность:** Данная версия не содержит накладных расходов на синхронизацию потоков,
что делает её идеальным базисом ($T_1$) для расчета ускорения параллельных версий ($S = T_1 / T_p$).
3. **Производительность:** На процессоре AMD Ryzen 7 6800HS последовательная версия демонстрирует высокую скорость
обработки (~215 мс для 10M элементов). Это создает высокую планку для параллельных версий: чтобы получить заметное
ускорение, параллельные реализации должны эффективно использовать все 8 ядер и компенсировать накладные расходы на
создание потоков и синхронизацию.

Эта реализация готова к использованию в качестве основы для параллельных модификаций (OpenMP, TBB, STL Algorithms).
