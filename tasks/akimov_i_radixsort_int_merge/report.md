## 1. Введение

В данной работе исследуется эффективность различных парадигм параллельного программирования (OpenMP, oneTBB, STL-потоки
и гибридная модель MPI+TBB) на примере алгоритма поразрядной сортировки (LSD Radix Sort). Алгоритм отлично подходит для
сравнительного анализа, так как сочетает в себе идеально распараллеливаемую фазу (локальная побайтовая сортировка
независимых блоков) и фазу с жесткой зависимостью по данным (итеративное древовидное слияние).

## 2. Единая постановка задачи

- **Вход:** Неотсортированный массив 32-битных знаковых целых чисел `std::vector<int>`.
- **Выход:** Тот же массив, отсортированный по неубыванию.
- **Ограничения:** Алгоритм должен корректно обрабатывать отрицательные числа. Для этого на этапе обработки старшего
байта применяется инверсия знакового бита операцией `XOR 128` (или `INT32_MIN`).
- **Критерий корректности:** Строгое совпадение результирующего массива с результатом работы эталонной функции
`std::ranges::sort`.

## 3. Единая методика эксперимента

**Окружение:**

- **CPU:** AMD Ryzen 7 6800HS, 8 ядер, 16 потоков
- **RAM:** 32 GB DDR5
- **OS:** Ubuntu (Linux)
- **Compiler:** GCC (Release, -O3)
- **Build type:** Release

- **Данные:** Для функциональных тестов генерируются массивы различных конфигураций (пустые, отсортированные,
`INT_MAX/MIN`). Для тестов производительности используется вектор из $N = 10\,000\,000$ элементов,
заполняемый псевдослучайными числами.
- **Расчет метрик:**
  - Базовое время ($T_1$) берется из медианного значения `task_run` последовательной (SEQ) версии (~215.40 мс).
  - Ускорение ($S$) считается по формуле $S = T_1 / T_p$, где $T_p$ — медианное время параллельной версии.
  - Эффективность ($E$) считается по формуле $E = (S / P) \times 100\%$.
  - Рабочая единица ($P$) для потоковых версий равна числу потоков.
  Для гибридной версии (ALL) $P = \text{ranks} \times \text{threads}$.
- **Агрегация:** Для каждой конфигурации выполнено несколько повторов. В качестве итогового результата в таблицу
занесено медианное значение.

## 4. Сводка корректности

Корректность всех backend-ов (`omp`, `tbb`, `stl`, `all`) проверялась сравнением с результатами `seq`.
Функциональные тесты запускались через общий фреймворк Google Test. Все технологии детерминированно прошли тесты на
граничные случаи (пустые массивы, реверсивные массивы, массивы из одного элемента, наличие отрицательных чисел).
Ограничений применимости у разработанных реализаций не выявлено.

## 6. Интерпретация различий

Анализ полученных таблиц позволяет сделать ряд важных архитектурных выводов о поведении алгоритма в различных средах
исполнения.

1. **Сопоставимость SEQ и параллельных версий при $P=1$:** Время выполнения на 1 потоке/процессе для всех версий
практически идентично (~215-225 мс). Небольшой разброс (до 5%) обусловлен накладными расходами на инициализацию сред
параллелизма (MPI, TBB, OMP) и различиями в аллокаторах памяти.

2. **Поведение shared-memory backend-ов (OMP, TBB, STL):**
   - Все три технологии показывают схожую динамику с высоким ускорением до 8 потоков (физические ядра).
   - Пик производительности достигается на **8 потоках**: OMP (~5.6x), TBB (~5.5x), STL (~5.1x).
   - При переходе к **16 потокам** (SMT) наблюдается деградация эффективности из-за конкуренции за пропускную
   способность памяти, что характерно для memory-bound задач.
   - **STL** показывает чуть худшие результаты на высоких нагрузках из-за отсутствия пула потоков и затрат на создание
   `std::thread` на каждом шаге слияния.

3. **Эффективность гибридной схемы (ALL - MPI + TBB):**
   - Гибридная версия демонстрирует хорошее ускорение (3.91x на 8 процессах), но уступает чистым shared-memory решениям
   (OMP/TBB дают ~5.5x на тех же 8 ядрах).
   - Это связано с тем, что на одноузловой системе накладные расходы на межпроцессное взаимодействие
   (MPI_Scatterv/Gatherv) и копирование данных между адресными пространствами процессов превышают выгоду от изоляции
   кэшей.
   - Тем не менее, эффективность на 8 процессах остается высокой (48.9%),
   что подтверждает работоспособность гибридной модели.

4. **Аппаратные лимиты (Ryzen 7 6800HS):**
   - Процессор имеет 8 физических ядер.
   - Использование SMT (16 потоков/процессов) неэффективно для данного алгоритма из-за ограничений памяти.

## 7. Репродуцируемость

**Сборка проекта:**

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

**Функциональное тестирование:**

```bash
mpirun -n 4 ./build/bin/ppc_func_tests --gtest_filter="*akimov_i_radixsort*"
```

**Бенчмаркинг:**

```bash
# Для потоковых технологий (SEQ, OMP, TBB, STL)

mpirun -n 1 ./build/bin/ppc_perf_tests --gtest_filter="*akimov_i_radixsort_omp*"

# Для гибридной технологии (ALL)
export PPC_NUM_THREADS=2
mpirun -n 4 ./build/bin/ppc_perf_tests --gtest_filter="*akimov_i_radixsort_all*"
```

## 8. Заключение

Наилучшую абсолютную производительность на одноузловой системе (Ryzen 7 6800HS) показали версии **OMP** и **TBB**,
достигнув ускорения ~5.5x на 8 физических ядрах. Гибридная версия (**ALL**) показала меньшее ускорение (~3.9x) из-за
накладных расходов MPI на локальной машине, но продемонстрировала хорошую эффективность и масштабируемость,
что делает её перспективной для распределенных кластерных систем.

Версия на **STL threads** оказалась наименее эффективной на больших количествах потоков из-за высоких затрат на
управление потоками ОС.

Ключевым выводом является то, что для алгоритмов типа Radix Sort с этапом слияния масштабирование ограничено пропускной
способностью памяти. Использование SMT контрпродуктивно. Для дальнейшего улучшения производительности требуется
оптимизация этапа слияния.
