## 1. Контекст

Данная реализация демонстрирует использование низкоуровневых примитивов многопоточности стандартной библиотеки C++ (`<thread>`). В отличие от OpenMP или TBB, где планировщик управляет пулом потоков и распределением задач, здесь программист вручную создает системные потоки для каждой фазы алгоритма и явно управляет их синхронизацией через `join()`.

Этот подход дает полный контроль над жизненным циклом потоков, но требует осторожности при управлении ресурсами, чтобы избежать избыточных накладных расходов на создание и уничтожение потоков ОС.

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив `std::vector<int>`.
- **Выходные данные:** Отсортированный по неубыванию массив.
- **Цель:** Реализовать параллельную версию алгоритма, используя только средства STL, без сторонних библиотек.
- **Эталон:** Последовательная версия (`SEQ`) используется для проверки корректности и расчета ускорения.

## 3. Базовый алгоритм

Локальная сортировка блоков выполняется с помощью **LSD Radix Sort**.
- **Обработка знака:** В функции `CountingSortStep` применяется инверсия старшего бита (`XOR 128`) для корректной сортировки знаковых `int`.
- **Оптимизация памяти:** В `RadixSortLocal` временный буфер `temp` выделяется один раз. Чередование направлений сортировки позволяет избежать лишнего копирования данных.

## 4. Схема распараллеливания

Алгоритм разделен на две фазы, в каждой из которых создается новая группа потоков.

### Фаза 1: Параллельная локальная сортировка
Массив делится на `num_threads` блоков. Для каждого блока создается отдельный `std::thread`, который выполняет `RadixSortLocal`.

```cpp
{
    std::vector<std::thread> threads;
    threads.reserve(static_cast<size_t>(num_threads));
    for (int i = 0; i < num_threads; ++i) {
        threads.emplace_back([&arr, &offsets, i]() {
            auto begin = arr.begin() + offsets[i];
            auto end = arr.begin() + offsets[i + 1];
            RadixSortLocal(begin, end);
        });
    }
    for (auto &t : threads) {
        t.join();
    }
}
```
- **Изоляция данных:** Каждый поток получает свой индекс `i` и работает только с диапазоном `[offsets[i], offsets[i+1])`.
- **Синхронизация:** Цикл `join()` гарантирует, что все потоки завершат сортировку своих блоков перед переходом к следующей фазе. Область видимости `{}` обеспечивает уничтожение вектора потоков после использования.

### Фаза 2: Параллельное древовидное слияние
Слияние происходит итеративно. На каждом шаге `step` создается **новая группа потоков** для выполнения независимых операций слияния пар блоков.

```cpp
void MergeBlocks(std::vector<int> &arr, const std::vector<int> &offsets, int num_threads) {
  for (int step = 1; step < num_threads; step *= 2) {
    std::vector<std::thread> threads;
    // Расчет количества необходимых потоков на этом уровне
    int reserve_size = (num_threads + (2 * step) - 1) / (2 * step);
    threads.reserve(static_cast<size_t>(reserve_size));
    
    for (int i = 0; i < num_threads; i += 2 * step) {
      if (i + step < num_threads) {
        threads.emplace_back([&arr, &offsets, i, step, num_threads]() {
          auto begin = arr.begin() + offsets[i];
          auto middle = arr.begin() + offsets[i + step];
          int end_idx = std::min(i + (2 * step), num_threads);
          auto end = arr.begin() + offsets[end_idx];
          std::inplace_merge(begin, middle, end);
        });
      }
    }
    for (auto &t : threads) {
      t.join();
    }
  }
}
```
- **Динамическое создание потоков:** На каждом уровне дерева слияния (`step = 1, 2, 4...`) создается новый вектор потоков. Количество потоков уменьшается с каждым шагом (на первом шаге ~N/2, на втором ~N/4 и т.д.).
- **Захват переменных:** Лямбда захватывает `i`, `step`, `num_threads` по значению, а `arr` и `offsets` по ссылке. Это критично, так как переменные цикла меняются, а данные общие.
- **Барьер:** `join()` всех потоков текущего уровня служит барьером перед увеличением `step`.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

### Управление ресурсами
В отличие от OpenMP/TBB, где потоки переиспользуются из пула, данная реализация создает и уничтожает системные потоки на каждом этапе.
- **Фаза 1:** Создается `num_threads` потоков.
- **Фаза 2:** Создается суммарно около `num_threads` потоков за все итерации цикла `step` (так как $N/2 + N/4 + ... \approx N$).
Это может приводить к значительным накладным расходам на системные вызовы `clone`/`pthread_create`, особенно на больших количествах ядер.

### Локальная сортировка
Функция `RadixSortLocal` идентична последовательной версии. Выделение `temp` внутри каждого потока снижает конкуренцию за глобальный аллокатор, но увеличивает общее потребление памяти (Peak Memory Usage $\approx P \cdot N/P = N$, плюс накладные расходы стека потоков).

## 6. Проверка корректности

Корректность проверяется сравнением с `std::ranges::sort`.
- **Отсутствие гонок:** Гарантируется тем, что на любом этапе потоки работают с непересекающимися диапазонами памяти.
- **Синхронизация:** Явные вызовы `join()` обеспечивают строгий порядок выполнения фаз. Данные не будут читаться для слияния, пока не закончится их сортировка.

### Анализ результатов
1. **1 поток:** Время близко к SEQ, небольшой оверхед на структуру кода.
2. **2-4 потока:** Хорошее ускорение. Накладные расходы на создание нескольких потоков компенсируются выигрышем от параллельной сортировки.
3. **8 потоков:** Ускорение ~5.1x. Эффективность снижается, так как время слияния начинает доминировать, а создание 8+ потоков на каждом шаге слияния добавляет задержки.
4. **16 потоков:** Деградация эффективности. Создание 16 потоков, их планирование ОС и последующее уничтожение занимает заметное время. Кроме того, гиперпоточность не помогает в memory-bound задаче слияния.

## 9. Выводы

1. **Контроль vs Удобство:** Реализация на `std::thread` требует ручного управления жизненным циклом потоков. Отсутствие пула потоков приводит к тому, что на каждом шаге алгоритма происходят системные вызовы создания потоков, что является узким местом по сравнению с OMP/TBB.
2. **Масштабируемость:** Алгоритм масштабируется до числа физических ядер, но дальнейшее увеличение потоков неэффективно из-за двух факторов:
   - Накладные расходы на создание/уничтожение потоков ОС.
   - Ограничения пропускной способности памяти при слиянии.
3. **Корректность:** Использование `join()` и изоляция диапазонов памяти гарантируют отсутствие гонок данных.
4. **Рекомендации:** Для продакшн-кода предпочтительнее использовать OpenMP или TBB, которые используют пулы потоков. STL-версия полезна для понимания низкоуровневых механизмов параллелизма или в средах, где запрещены сторонние библиотеки.
