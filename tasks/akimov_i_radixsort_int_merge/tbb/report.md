## 1. Контекст

Использование библиотеки **oneTBB** позволяет перейти от ручного управления потоками к задачно-ориентированной модели
(task-based parallelism). Планировщик TBB автоматически распределяет задачи по доступным ядрам,
используя механизм work-stealing для балансировки нагрузки.

В данной реализации алгоритм сохраняет структуру «разделяй и властвуй»: массив разбивается на блоки,
которые сортируются параллельно, а затем объединяются. Однако, в отличие от OpenMP,
где используются директивы компилятора, здесь параллелизм выражается через высокоуровневые примитивы библиотеки,
такие как `tbb::parallel_for`.

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив `std::vector<int>`.
- **Выходные данные:** Отсортированный по неубыванию массив.
- **Цель:** Реализовать параллельную версию, использующую возможности планировщика TBB для эффективного распределения
вычислений локальной сортировки и этапов слияния.
- **Эталон:** Последовательная версия (`SEQ`) используется для проверки корректности и расчета ускорения.

## 3. Базовый алгоритм

Локальная сортировка блоков выполняется с помощью **LSD Radix Sort**.

- **Обработка знака:** В функции `CountingSortStep` применяется инверсия старшего бита (`XOR 128`) для корректной
сортировки знаковых `int`.
- **Оптимизация памяти:** В `RadixSortLocal` временный буфер `temp` выделяется один раз. Чередование направлений
сортировки (из `arr` в `temp` и обратно) позволяет избежать лишнего копирования данных после завершения 4-х проходов.

## 4. Схема распараллеливания

Алгоритм состоит из двух фаз, обе реализованы с использованием примитива `tbb::parallel_for`.

### Фаза 1: Параллельная локальная сортировка

Массив делится на `num_threads` блоков. Границы блоков записываются в вектор `offsets`.
Для запуска параллельной сортировки используется упрощенный синтаксис `tbb::parallel_for`:

```cpp
tbb::parallel_for(0, num_threads, [&](int i) {
    auto begin = arr.begin() + offsets[i];
    auto end = arr.begin() + offsets[i + 1];
    RadixSortLocal(begin, end);
});
```

- **Диапазон:** `[0, num_threads)`.
- **Лямбда-выражение:** Принимает индекс блока `i`. Каждый поток (или задача TBB) получает свой уникальный индекс и
обрабатывает соответствующий диапазон памяти.
- **Безопасность:** Так как диапазоны `[offsets[i], offsets[i+1])` не пересекаются, гонки данных при записи исключены.

### Фаза 2: Параллельное древовидное слияние

Слияние отсортированных блоков происходит итеративно. Внешний цикл по уровню дерева (`step`) выполняется
последовательно, так как каждый следующий уровень зависит от результатов предыдущего. Внутри каждого уровня слияния распараллеливаются:

```cpp
for (int step = 1; step < num_threads; step *= 2) {
    tbb::parallel_for(0, num_threads, [&](int i) {
        // Фильтр: обрабатываем только начало пары блоков
        if (i % (2 * step) == 0 && i + step < num_threads) {
            auto begin = arr.begin() + offsets[i];
            auto middle = arr.begin() + offsets[i + step];
            int end_idx = std::min(i + (2 * step), num_threads);
            auto end = arr.begin() + offsets[end_idx];
            
            std::inplace_merge(begin, middle, end);
        }
    });
}
```

- **Стратегия фильтрации:** В отличие от OpenMP, где можно использовать шаг цикла `i += 2*step`,
`tbb::parallel_for` итерирует по непрерывному диапазону `[0, num_threads)`.
Поэтому внутри лямбды добавлена проверка`if (i % (2 * step) == 0 ...)`.
- **Избыточные итерации:** Часть задач (где условие `if` ложно) завершается мгновенно. Это небольшой накладной расход,
но он позволяет сохранить простой и единый интерфейс запуска задач для всех уровней дерева.
- **Синхронизация:** `tbb::parallel_for` содержит неявный барьер. Все задачи текущего уровня `step` должны завершиться
перед переходом к следующему шагу внешнего цикла.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

### Адаптивность

Как и в OMP-версии, реализована проверка на размер массива:

```cpp
if (n < num_threads * 100) {
    num_threads = 1;
}
```

Это предотвращает деградацию производительности на малых данных из-за оверхеда планировщика TBB.

### Локальная сортировка (`RadixSortLocal`)

Реализация полностью идентична последовательной версии, что обеспечивает высокую эффективность кэша L1/L2 внутри
каждой задачи. Выделение `std::vector<int> temp(n)` происходит в стеке/куче потока, выполнившего задачу,
что снижает конкуренцию за глобальный аллокатор.

### Использование `std::inplace_merge`

Стандартная функция STL выбрана для слияния благодаря своей устойчивости и оптимизированной реализации (часто
использует временный буфер и линейное время $O(N)$ при наличии памяти).

## 6. Проверка корректности

Корректность проверяется сравнением с `std::ranges::sort`.

- **Тесты:** Случайные данные, отрицательные числа, граничные значения `INT_MIN`/`INT_MAX`.
- **Отсутствие гонок:** Разделение памяти через `offsets` гарантирует, что разные задачи TBB пишут в разные участки
массива `arr`.
- **Детерминированность:** Несмотря на то, что порядок выполнения задач внутри `parallel_for` не определен,
результат слияния детерминирован, так как операция `merge` применяется к строго определенным соседним блокам.

### Анализ результатов

1. **1 поток:** Накладные расходы TBB минимальны, время близко к SEQ.
2. **2-4 потока:** Высокая эффективность. Планировщик TBB эффективно распределяет тяжеловесные задачи локальной
сортировки.
3. **8 потоков:** Ускорение ~5.5x. Эффективность снижается из-за затрат на слияние.
`std::inplace_merge` требует интенсивного обмена данными между ядрами, что ограничивает масштабируемость.
4. **16 потоков:** Деградация. Гиперпоточность не дает преимущества в memory-bound задаче слияния,
а избыточные итерации в `parallel_for` (где `if` ложно) создают дополнительный холостой пробег задач.

## 9. Выводы

1. **Удобство API:** oneTBB позволил реализовать параллельное слияние лаконично, используя лямбда-выражения.
Отсутствие необходимости явно управлять барьерами (они скрыты внутри `parallel_for`) упрощает код.
2. **Специфика реализации слияния:** Использование непрерывного диапазона `[0, num_threads)` с внутренней проверкой
`if (i % ...)` является компромиссом. Это проще в коде, чем создание кастомных range-объектов, но приводит к созданию
"пустых" задач на каждом шаге слияния. На больших массивах этот оверхед ничтожен по сравнению со временем работы
`inplace_merge`.
3. **Производительность:** TBB показывает результаты, аналогичные OpenMP. Главным ограничителем остается не технология
параллелизма, а алгоритмическая сложность фазы слияния и пропускная способность памяти.
4. **Рекомендации:** Для дальнейшей оптимизации в TBB можно было бы использовать `tbb::parallel_invoke` для
рекурсивного слияния (divide-and-conquer merge), что более естественно для task-based модели, чем итеративный цикл с
`parallel_for`. Однако текущая реализация успешно демонстрирует применимость TBB для гибридных алгоритмов сортировки.
