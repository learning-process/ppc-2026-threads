## 1. Контекст

Данный отчет описывает параллельную реализацию сортировки целых чисел с использованием OpenMP.
Алгоритм сочетает в себе эффективность поразрядной сортировки (Radix Sort) для локальных участков данных и надежность
стандартного алгоритма слияния (`std::inplace_merge`) для объединения результатов.

Стратегия «разделяй и властвуй» выбрана потому, что прямое распараллеливание внутренней логики Radix Sort (подсчет
префиксных сумм) сложно из-за зависимостей по данным. Вместо этого массив делится на независимые блоки,
которые сортируются параллельно, а затем объединяются.

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив `std::vector<int>`.
- **Выходные данные:** Отсортированный по неубыванию массив.
- **Цель:** Реализовать параллельную версию, использующую несколько потоков для ускорения сортировки больших массивов,
сохраняя корректность обработки знаковых чисел.
- **Эталон:** Последовательная версия (`SEQ`) используется для проверки корректности и расчета ускорения.

## 3. Базовый алгоритм

Локальная сортировка каждого блока выполняется с помощью **LSD Radix Sort** (поразрядная сортировка от младшего байта к
старшему).

### Обработка знаковых чисел

В функции `CountingSortStep` реализована инверсия старшего бита для корректной сортировки `signed int`:

```cpp
if (byte_index == sizeof(int) - 1) {
  byte_val ^= 128; // Инверсия знакового бита
}
```

Это позволяет интерпретировать отрицательные числа как большие беззнаковые значения на старшем разряде,
обеспечивая правильный порядок после сортировки.

### Оптимизация памяти

В функции `RadixSortLocal` временный буфер `temp` выделяется **один раз**. Чередование источников и приемников данных
на четных/нечетных шагах цикла по байтам позволяет избежать лишнего копирования массива в конце работы.

## 4. Схема распараллеливания

Алгоритм состоит из двух основных фаз, разделенных барьерами синхронизации OpenMP.

### Фаза 1: Параллельная локальная сортировка

Исходный массив делится на `num_threads` непрерывных блоков. Границы блоков вычисляются заранее и сохраняются в векторе
`offsets`.

```cpp
#pragma omp parallel num_threads(num_threads) default(none) shared(offsets, arr)
{
  int tid = omp_get_thread_num();
  auto begin = arr.begin() + offsets[tid];
  auto end = arr.begin() + offsets[tid + 1];
  
  // Каждый поток сортирует только свой блок независимо
  RadixSortLocal(begin, end);
}
```

- **Безопасность:** Потоки работают с непересекающимися диапазонами памяти `[offsets[tid], offsets[tid+1])`.
Гонки данных отсутствуют.
- **Локальность:** Каждый поток использует свой стек для выделения `temp` внутри `RadixSortLocal`,
что снижает конкуренцию за аллокатор памяти.

### Фаза 2: Параллельное древовидное слияние (Tree Merge)

После локальной сортировки массив представляет собой набор отсортированных блоков. Они сливаются итеративно по схеме
двоичного дерева.

```cpp
for (int step = 1; step < num_threads; step *= 2) {
  #pragma omp parallel for num_threads(num_threads) default(none) shared(step, num_threads, offsets, arr)
  for (int i = 0; i < num_threads; i += 2 * step) {
    if (i + step < num_threads) {
      // Определение границ двух соседних блоков
      auto begin = arr.begin() + offsets[i];
      auto middle = arr.begin() + offsets[i + step];
      int end_idx = std::min(i + (2 * step), num_threads);
      auto end = arr.begin() + offsets[end_idx];
      
      // Слияние двух отсортированных половин
      std::inplace_merge(begin, middle, end);
    }
  }
}
```

- **Логика:** На шаге `step=1` сливаются пары соседних блоков (0-1, 2-3...). На шаге `step=2` сливаются результаты
предыдущего шага (0..3, 4..7...) и так далее, пока все блоки не сольются в один.
- **Параллелизм:** Внутренний цикл `for (int i ...)` распараллеливается. Операции слияния разных пар независимы,
поэтому они могут выполняться одновременно разными потоками.
- **Синхронизация:** Директива `#pragma omp parallel for` содержит неявный барьер в конце. Это гарантирует,
что все слияния текущего уровня завершатся перед началом следующего уровня (`step *= 2`).

### Анализ директив OpenMP

- `default(none)`: Требует явного указания `shared` переменных, что предотвращает случайные гонки данных.
- `shared(offsets, arr)`: Массив данных и границы блоков общие для всех потоков.
- Отсутствие `atomic/critical`: В коде нет разделяемых счетчиков, требующих атомарности.
Доступ к памяти строго разграничен индексами.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

### Адаптивность к размеру задачи

В начале `RunImpl` проверяется размер массива:

```cpp
if (n < num_threads * 100) {
  num_threads = 1;
}
```

Если массив слишком мал, накладные расходы на создание потоков и слияние превысят выгоду от параллелизма.
В этом случае алгоритм переключается в последовательный режим.

### Функция `CountingSortStep`

Реализует один проход сортировки подсчетом:

1. Подсчет частот байтов в локальном диапазоне.
2. Вычисление префиксных сумм (позиций).
3. Распределение элементов в выходной итератор.
Использует `std::array<size_t, 256>` для быстрых стековых операций.

### Функция `RadixSortLocal`

Выполняет 4 прохода `CountingSortStep`. Использует двойную буферизацию через `temp` вектор,
меняя направление сортировки (`begin->temp` или `temp->begin`) в зависимости от четности номера байта.

## 6. Проверка корректности

Корректность подтверждается функциональными тестами, сравнивающими результат с `std::ranges::sort`.

- **Граничные случаи:** Пустые массивы (отсекаются валидацией), массивы из 1 элемента, наличие `INT_MIN`/`INT_MAX`.
- **Знаковые числа:** Благодаря инверсии бита знака в `CountingSortStep`,
отрицательные числа корректно встают перед положительными.
- **Устойчивость слияния:** `std::inplace_merge` является устойчивой операцией, что сохраняет относительный порядок
равных элементов, если он был сохранен на этапе локальной сортировки (Radix Sort устойчив).

### Анализ результатов

1. **1 поток:** Время близко к SEQ. Небольшой оверхед связан с проверками OpenMP и структурой кода.
2. **2-4 потока:** Высокая эффективность (>85%). Локальная сортировка доминирует во времени выполнения,
и она идеально параллелится.
3. **8 потоков:** Ускорение ~5.6x. Эффективность падает до ~70%,
так как вклад фазы слияния (`std::inplace_merge`) становится значительным.
Слияние требует интенсивной работы с памятью и создает конкуренцию за кэш и шину данных между ядрами.
4. **16 потоков:** Деградация производительности по сравнению с 8 потоками.
Гиперпоточность (SMT) не помогает в задачах, ограниченных пропускной способностью памяти (memory-bound),
какими являются операции слияния больших массивов. Логические ядра конкурируют за ресурсы физических.

## 9. Выводы

1. **Эффективность гибридного подхода:** Комбинация быстрого локального Radix Sort и стандартного `inplace_merge`
позволила достичь значительного ускорения на многоядерных CPU.
2. **Масштабируемость:** Алгоритм хорошо масштабируется до количества физических ядер (8 для Ryzen 7 6800HS).
Дальнейшее увеличение потоков неэффективно из-за насыщения канала памяти на этапе слияния.
3. **Оптимизация памяти:** Однократное выделение буфера `temp` в каждом потоке при локальной сортировке минимизировало
накладные расходы на аллокацию.
4. **Корректность:** Использование `default(none)` и строгое разделение диапазонов по `offsets` гарантировало
отсутствие гонок данных. Инверсия знакового бита обеспечила правильную работу с отрицательными числами.

Данная реализация демонстрирует хороший баланс между сложностью кода и производительностью,
подходя для задач сортировки больших массивов целых чисел на современных многоядерных процессорах.
