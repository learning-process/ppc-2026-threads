## 1. Контекст

Данная реализация является гибридной и использует двухуровневый параллелизм: межпроцессный на основе MPI
(Message Passing Interface) ивнутрипроцессный с использованием OpenMP, TBB или `std::thread`.
Цель — исследовать возможность масштабирования алгоритма на кластерныхсистемах, объединяя несколько
вычислительных узлов.

## 2. Постановка задачи

Полностью совпадает с постановкой в `seq/report.md`. Корректность проверяется относительно `SEQ`.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с последовательной версией. Гибридная версия реализует распределенный
алгоритм Фокса с использованием MPI.

## 4. Межпроцессная схема (MPI)

- **Коммуникатор:** Используется `MPI_COMM_WORLD`.
- **Роли процессов:** Все процессы участвуют в вычислениях, каждый отвечает за вычисление своей "плитки"
  результирующей матрицы.
- **Распределение данных:** Матрица A распределяется по процессам по блокам строк, матрица B — по
  блокам столбцов. Алгоритм Фокса подразумевает на каждом этапе пересылку блоков.
- **MPI-вызовы:**
  - `MPI_Comm_rank`, `MPI_Comm_size`: для определения идентификатора и общего числа процессов.
  - `MPI_Bcast`: для рассылки блоков матрицы A в пределах кольца процессов.
  - `MPI_Send` / `MPI_Recv`: для пересылки блоков матрицы B.
  - `MPI_Barrier`: для синхронизации процессов в конце каждого этапа, гарантируя, что все необходимые
  данные были получены перед началом следующего этапа.
- **Синхронизация:** Явные барьеры после каждого этапа алгоритма Фокса необходимы для поддержания корректности
  распределенного алгоритма.

## 5. Внутрипроцессная схема

- **Технология:** Внутри каждого MPI-процесса используется одна из технологий: OpenMP, TBB или `std::thread`
  (выбирается в зависимости от блока, см. `ops_all.cpp`).
- **Распараллеливание:** Умножение локальных блоков матриц, за которые отвечает процесс, выполняется параллельно
  с использованием выбранной внутрипроцессной технологии. Для больших матриц (n > 256) используются
  `tbb::parallel_for`, для средних — OpenMP, для маленьких — `std::thread` или последовательный код.
- **Объединение результатов:** Каждый процесс самостоятельно вычисляет свою часть результирующей матрицы,
  используя локальные умножения блоков. Глобального объединения не требуется, так как результат распределен.

### Листинг ключевого фрагмента (гибридная логика)

```cpp
void FoxAlgorithmAdaptive(const DenseMatrix &a, const DenseMatrix &b, DenseMatrix &result, int block_size) {
  // ... проверки ...
  if (num_blocks <= 8) {
    HandleSmallNumBlocks(a, b, result, num_blocks, block_size); // OpenMP
  } else if (num_blocks <= 32) {
    HandleMediumNumBlocks(a, b, result, num_blocks, block_size); // TBB
  } else {
    HandleLargeNumBlocks(a, b, result, num_blocks, block_size); // STL threads
  }
}```

Здесь демонстрируется, как внутри одного MPI-процесса выбор технологии параллелизма адаптируется под размер 
задачи (количество блоков).

## 6. Детали реализации

*   **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
*   **Класс:** `YakimovIMultOfDenseMatricesFoxAlgorithmAll`
*   **Особенности:** Код является комбинацией подходов из `seq`, `omp`, `tbb` и `stl`. Выбор конкретной 
thread-технологии зависит от размера задачи для максимальной эффективности.
*   **Узкие места:** Основными узкими местами являются межпроцессные коммуникации (`MPI_Bcast`, `MPI_Send/Recv`) 
и глобальная синхронизация (`MPI_Barrier`), время которых может расти с увеличением числа процессов.

## 7. Проверка корректности

Корректность проверена для конфигураций с 1, 2 и 4 MPI-процессами. Для каждого процесса использовалось по 2 потока. 
Результат на всех функциональных тестах совпадал с `SEQ`.

## 8. Экспериментальная среда

## 10. Выводы

Гибридная реализация успешно работает и позволяет задействовать несколько MPI-процессов. Однако, из-за высоких накладных расходов на 
коммуникацию, для задач, умещающихся в памяти одного узла, использование чисто потоковых версий (OpenMP/TBB) оказывается более 
эффективным. Гибридный подход становится выгодным при решении задач очень большого размера, когда матрицы не помещаются в памяти одного 
узла, и необходимо объединять ресурсы нескольких машин.
