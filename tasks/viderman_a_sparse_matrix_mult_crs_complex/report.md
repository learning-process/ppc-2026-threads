## 1. Введение

В работе исследуется умножение двух разреженных комплексных матриц в
формате CRS (Compressed Row Storage). Формат удобен тем, что хранит только
ненулевые элементы, а значит позволяет не тратить время и память на
обработку явных нулей. Последовательная версия используется как образец
корректности и как базовая точка сравнения для всех параллельных
реализаций.

Задача хорошо подходит для сравнения технологий параллелизма, потому что
строки результата строятся независимо. Это позволяет распараллеливать
обработку строк между потоками или процессами, а затем собирать итоговую
матрицу в CRS-формате. При этом полезная работа на одну строку остаётся
небольшой, поэтому накладные расходы runtime и синхронизации заметно
влияют на итоговое время.

В проекте реализованы пять версий одной задачи. `SEQ` выполняет расчёт
последовательно, `OMP` использует OpenMP, `TBB` строится на
`oneapi::tbb::parallel_for`, `STL` делит строки между `std::jthread`, а
`ALL` сочетает MPI-распределение строк между процессами с
OpenMP-параллелизмом внутри каждого процесса.

## 2. Постановка задачи

На вход подаются две матрицы `A` и `B` в CRS-формате. Требуется вычислить
`C = A * B` и вернуть результат также в CRS. В ходе вычисления значения,
по модулю не превосходящие `EPSILON`, отбрасываются, чтобы не засорять
результат численно незначимыми элементами.

Для всех реализаций одинаковы и критерий корректности, и набор тестов.
Результат должен совпадать с `SEQ`, сохранять корректную CRS-структуру,
не содержать явных нулей, иметь отсортированные индексы столбцов и
корректные указатели строк.

## 3. Единая методика эксперимента

Замеры выполнялись на машине с процессором Intel(R) Core(TM) i5-9300H CPU
@ 2.40GHz, 16 GB RAM и Windows 10 в `Release`-сборке Visual Studio
Community 2022 Release - amd64. Входные данные для перф-тестов формируются
синтетически в [`tests/performance/main.cpp`](tests/performance/main.cpp):
используются полосовая, диагональная, разреженно-рассеянная,
блочно-диагональная и прямоугольная полосовая матрицы.

Внутри каркаса курса один `task_run` уже усредняется по 5 внутренним
запускам через `Perf::CommonRun`. Чтобы уменьшить влияние шума ещё
сильнее, для отчёта дополнительно использовалась серия из 5 внешних
запусков `BandMatrixPerf` при `PPC_NUM_THREADS=4`, а для `ALL`
дополнительно задавался `PPC_NUM_PROC=2` и использовался
`mpiexec -n 2`.

Для сравнения backend-ов в таблице ниже используется медиана серии по
`task_run`. Этот режим удобен тем, что он меньше зависит от накладных
расходов на полный pipeline, но при этом сохраняет те же входные данные
и ту же логику вычисления.

## 4. Проверка корректности

После пересборки Release-бинаря все функциональные тесты и все
перф-тесты проходят. Функциональные тесты покрывают несовместимые
размеры, некорректные индексы столбцов, ошибки в `row_ptr`, случаи с
пустыми матрицами, единичными элементами, диагональными матрицами,
сокращением к нулю и проверкой корректности CRS-выхода. Для `ALL`
функциональные тесты запускаются под `mpiexec`, потому что эта версия
требует MPI-окружения.

Корректность всех пяти backend-ов подтверждается сравнением с `SEQ`:
после пересборки расхождений в выходных данных не наблюдается.

Лучшее абсолютное время на этом наборе данных показывает `SEQ`. Из
параллельных single-node версий наиболее удачно выглядит `TBB`, но и он
не обгоняет эталон. `OMP` и `STL` проигрывают сильнее, а `ALL`
дополнительно платит за MPI-коммуникации и сборку результата между
процессами.

## 6. Выводы

На текущем наборе входных данных полезная работа на одну строку слишком
мала, чтобы окупить накладные расходы параллелизма. Поэтому
последовательная версия остаётся самой быстрой, а параллельные backend-ы
удобнее рассматривать как корректные учебные реализации разных моделей
распараллеливания.

Для этой задачи особенно хорошо видна разница между моделями: `TBB`
даёт наиболее мягкий runtime-overhead среди single-node версий, `OMP`
проще по коду, `STL` требует ручного управления потоками, а `ALL`
показывает цену гибридной MPI+OpenMP-схемы.

## 7. Команды запуска

```powershell
cmake -S . -B build
cmake --build build --config Release --target ppc_func_tests ppc_perf_tests -- /m:1

$env:PPC_NUM_THREADS='4'
& .\build\bin\ppc_perf_tests.exe

$env:PPC_NUM_THREADS='4'
$env:PPC_NUM_PROC='2'
& mpiexec -n 2 .\build\bin\ppc_func_tests.exe
& mpiexec -n 2 .\build\bin\ppc_perf_tests.exe --gtest_filter='BandMatrixPerf/*all_enabled*'
```
