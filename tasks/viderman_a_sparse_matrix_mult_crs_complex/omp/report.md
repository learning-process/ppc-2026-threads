# Умножение разреженных матриц в формате CRS (комплексные числа) - OMP

**Студент:** Видерман Александра Геннадьевна, группа 3823Б1ПР2.  
**Технология:** OMP.  
**Вариант:** 6.

## 1. Контекст

OpenMP-версия нужна для распараллеливания вычислений на машине с
общей памятью. Для этой задачи она естественно ложится на обработку
строк, потому что каждая строка результата считается независимо.

## 2. Постановка задачи

Вход и выход совпадают с `SEQ`: нужно перемножить две CRS-матрицы
комплексных чисел и вернуть CRS-результат без явных нулей. Корректность
проверяется сравнением с последовательным эталоном и общим
функциональным набором тестов.

## 3. Схема распараллеливания

Основная идея состоит в том, чтобы делить набор строк матрицы `A`
между потоками OpenMP. Каждый поток обрабатывает свой диапазон строк,
использует локальные буферы для накопления промежуточных значений и не
мешает другим потокам до финальной сборки результата.

Такой подход удобен тем, что не требует сложной синхронизации внутри
самой арифметики. Сопоставление строк и независимость вычислений
позволяют ограничиться разделением диапазона и финальным объединением.

## 4. Детали реализации

В коде используется `#pragma omp parallel for` для разбиения строк и
стандартные локальные контейнеры для накопления результата. На выходе
каждая строка фильтруется по `EPSILON`, затем добавляется в итоговую
CRS-структуру. Такой способ сохраняет исходную логику алгоритма, но
меняет только способ распределения работы.

Важный момент: параллелизм здесь не меняет математическую схему, а
только способ её исполнения. Поэтому и верификация, и структура выходных
данных остаются такими же, как в `SEQ`.

## 5. Проверка корректности

Функциональные тесты покрывают некорректные размеры, пустые и граничные
матрицы, корректность CRS-структуры, случаи с явными нулями и
совпадение с эталоном `SEQ`. Тесты проходят без ошибок.

## 6. Экспериментальная среда

Замеры выполнялись на Intel(R) Core(TM) i5-9300H CPU @ 2.40GHz,
16 GB RAM, Windows 10, в `Release`-сборке Visual Studio Community 2022
Release - amd64. Управление числом потоков задавалось через
`PPC_NUM_THREADS=4`.

## 7. Результаты

На `BandMatrixPerf` получена серия из пяти запусков `task_run`:

| Серия из 5 запусков, s | Медиана, s | Speedup относительно `SEQ` |
| --- | ---: | ---: |
| 0.00060112, 0.00048022, 0.00047432, 0.00041434, 0.00048222 | 0.00048022 | 0.26 |

На текущем наборе входов OpenMP даёт корректный результат, но не
ускоряет вычисление. Накладные расходы на параллельную область и
распределение строк оказываются выше полезной работы на одну строку.

## 8. Выводы

OpenMP-версия корректна и удобна как учебная реализация параллелизма
по строкам. На этой задаче она показывает, что правильная декомпозиция
ещё не гарантирует ускорения: если работа слишком мелкая, управление
потоками начинает доминировать.
