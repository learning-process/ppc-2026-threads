# Умножение разреженных матриц в формате CRS (комплексные числа) - SEQ

**Студент:** Видерман Александра Геннадьевна, группа 3823Б1ПР2.  
**Технология:** SEQ.  
**Вариант:** 6.

## 1. Контекст

Последовательная версия нужна как эталон корректности и как базовая
линия для сравнения всех остальных backend-ов. В этой реализации нет
потоков, MPI и других примитивов синхронизации, поэтому она лучше всего
показывает собственную стоимость алгоритма без накладных расходов
runtime.

## 2. Постановка задачи

На вход подаются две разреженные комплексные матрицы в формате CRS.
Требуется вычислить произведение `C = A * B`, вернуть результат тоже в
CRS и отфильтровать элементы, по модулю не превосходящие `EPSILON`.

Критерий корректности для `SEQ` совпадает с остальными версиями:
результат должен быть валиден как CRS-матрица, не содержать явных нулей,
иметь отсортированные индексы столбцов и совпадать с эталоном по
значениям.

## 3. Базовый алгоритм

Алгоритм обрабатывает строки матрицы `A` по очереди. Для каждой строки
создаётся локальный накопитель длины `B.cols`, затем для каждого
ненулевого элемента строки пробегается соответствующая строка матрицы
`B`. Частичные произведения суммируются в локальном буфере, после чего
ненулевые элементы собираются в итоговую строку результата.

Сложность определяется не плотностью матриц, а количеством ненулевых
элементов в строках. Для CRS это естественный выбор, потому что он
позволяет пропускать нули и не тратить время на полный просмотр матрицы.

## 4. Детали реализации

SEQ-версия реализует полный pipeline без параллельных примитивов.
Валидация проверяет корректность CRS-входа и согласованность размеров,
preprocessing подготавливает внутренние структуры, `Run` выполняет само
умножение, а postprocessing приводит результат к итоговому виду.

Важная особенность этой версии в том, что она не создаёт дополнительных
буферов ради синхронизации и не платит за переключение потоков. Именно
поэтому она остаётся хорошим baseline для сравнения с более сложными
вариантами.

## 5. Проверка корректности

Функциональные тесты сравнивают `SEQ` с эталонными значениями на
наборах, где покрыты несовместимые размеры, пустые и граничные матрицы,
диагональные случаи, свёртка к нулю, единичные элементы и проверка
корректности CRS-структуры. После пересборки `SEQ` проходит весь набор
тестов.

## 6. Экспериментальная среда

Замеры проводились на Intel(R) Core(TM) i5-9300H CPU @ 2.40GHz,
16 GB RAM, Windows 10, в `Release`-сборке Visual Studio Community 2022
Release - amd64. Для перф-тестов использовался `BandMatrixPerf` из
[`tests/performance/main.cpp`](../tests/performance/main.cpp). Для
отчёта дополнительно снималась серия из 5 внешних запусков при
`PPC_NUM_THREADS=4`.

## 7. Результаты

На `BandMatrixPerf` получена серия из пяти запусков `task_run`:

- `0.00014670`
- `0.00012000`
- `0.00010176`
- `0.00014614`
- `0.00012460`

Медиана серии равна `0.00012460`. Это baseline без накладных расходов
параллелизма.

В этой задаче последовательная реализация показывает лучшее абсолютное
время. Это ожидаемо, потому что полезная работа на одну строку
невелика, а разреженность входов не даёт параллельным runtime-ам
окупить свои накладные расходы.

## 8. Выводы

SEQ-версия остаётся эталоном для корректности и производительности.
Она проще всего по коду, не требует синхронизации и даёт максимально
предсказуемый результат на текущем наборе входов.
