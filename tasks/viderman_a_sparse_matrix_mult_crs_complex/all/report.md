# Умножение разреженных матриц в формате CRS (комплексные числа) - ALL

**Студент:** Видерман Александра Геннадьевна, группа 3823Б1ПР2.  
**Технология:** ALL.  
**Вариант:** 6.

## 1. Контекст

Гибридная версия сочетает два уровня параллелизма сразу: между
MPI-процессами и внутри каждого процесса через OpenMP. Для CRS-задачи
это выглядит естественно, потому что строки результата можно
распределять между процессами, а затем дополнительно распараллелить
локальную обработку строк потоками.

## 2. Постановка задачи

Вход и выход совпадают с остальными backend-ами: нужно перемножить две
CRS-матрицы комплексных чисел и вернуть CRS-результат без явных нулей.
Корректность подтверждается общим функциональным набором тестов и
сравнением с `SEQ`.

## 3. Межпроцессная и внутрипроцессная схема

Сначала `A` разбивается между MPI-рангами. `rank 0` подготавливает
данные, рассылает нужные блоки и затем собирает частичные результаты
обратно. После этого каждый процесс работает со своим локальным блоком
строк, а внутренняя обработка строк выполняется OpenMP-потоками.

Такое разделение удобно для гибридной схемы: MPI отвечает за
распределение данных между процессами, а OpenMP - за локальную
утилизацию ядер внутри процесса. Именно из-за этого `ALL`
методически сложнее остальных версий: кроме арифметики нужно
безошибочно организовать и scatter/broadcast, и gather, и локальное
вычисление.

## 4. Детали реализации

В реализации используются `ValidationImpl`, `PreProcessingImpl`,
`RunImpl` и `PostProcessingImpl`. На этапе preprocessing данные
рассылаются между процессами, локальная матрица `A` собирается на
каждом ранге, затем выполняется локальное умножение, после чего
результаты собираются на `rank 0` и рассылаются обратно в
согласованном виде.

После пересборки Release-бинаря `ALL` стал доступен и в
функциональных, и в производительных тестах, поэтому здесь уже можно
привести реальные числа, а не описание ограничения старого бинаря.

## 5. Проверка корректности

Функциональные тесты `ALL` проходят в MPI-окружении и покрывают
несовместимые размеры, некорректные индексы, ошибки CRS-структуры,
пустые и граничные матрицы, случаи с явными нулями и совпадение с
`SEQ`. Это важно и для MPI-части, и для OpenMP-части, потому что
ошибка может появиться как при распределении данных, так и на этапе
локального перемножения.

## 6. Экспериментальная среда

Замеры выполнялись на Intel(R) Core(TM) i5-9300H CPU @ 2.40GHz,
16 GB RAM, Windows 10, в `Release`-сборке Visual Studio Community 2022
Release - amd64. Для `ALL` использовались `PPC_NUM_THREADS=4` и
`PPC_NUM_PROC=2`, запуск - через `mpiexec -n 2`.

## 7. Результаты

На `BandMatrixPerf` получена серия из пяти внешних запусков `task_run`:

| Серия | Медиана, s | Speedup относительно `SEQ` |
| --- | ---: | ---: |
| 0.00072082, 0.00057276, 0.00053278, 0.00072764, 0.00055570 | 0.00057276 | 0.22 |

Гибридная схема корректна и проходит тесты, но на текущем наборе
данных она не даёт ускорения относительно `SEQ`. Для такой задачи
объём вычислений на одну строку слишком мал, чтобы окупить
MPI-коммуникации и сборку результатов между процессами.

## 8. Выводы

`ALL` полезен как учебная демонстрация гибридного параллелизма. На
этой CRS-задаче он показывает, что даже корректная MPI+OpenMP-схема не
обязана ускорять вычисление, если вход слишком мелкий и
коммуникационные издержки доминируют над полезной работой.
