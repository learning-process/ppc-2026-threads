## 1. Контекст

STL-версия реализует ту же блочную свёртку, что OMP и TBB, но с использованием `std::thread`
вместо runtime-библиотек параллельных вычислений. Здесь нет автоматического распределения
работы - всё делается вручную: явное создание потоков, явное разбиение работы, явный join.

Эта версия показывает, какую часть выигрыша обеспечивает сама блочная декомпозиция,
а какую - библиотека поверх неё.

## 2. Постановка задачи

Постановка идентична seq-версии (см. `seq/report.md`): RGB-изображение `width × height`
на входе, изображение того же размера после свёртки 3×3 с ядром Гаусса на выходе.
Корректность определяется побайтовым равенством с эталоном seq.

## 3. Базовый алгоритм

Свёртка для каждого пикселя по каждому каналу:

```cpp
sum = sum of {kr=-1..1, kc=-1..1} img[row+kr, col+kc, c] * K[1+kr, 1+kc]
result = clamp((sum + 8) / 16, 0, 255)
```

Сложность: `O(width × height)`.

## 4. Схема распараллеливания

**Разбиение данных.** Число потоков `N` берётся из `ppc::util::GetNumThreads()`,
которое читает `PPC_NUM_THREADS`. Работа распределяется между всеми `N` потоками по формуле:

```cpp
left_border = (num_blocks * tid) / N
right_border = (num_blocks * (tid + 1)) / N
```

Эта формула гарантирует, что:

- Каждый ряд блоков обрабатывается ровно одним потоком.
- Распределение почти равномерное: разница между потоками не больше одного блока.
- Сумма диапазонов всех потоков покрывает весь массив без пропусков и пересечений.

**Локальные результаты не нужны.** Каждый поток пишет напрямую в общий `result_picture`
по индексам своих блоков, которые гарантированно не пересекаются с блоками других
потоков. Это устраняет необходимость в локальных буферах и последующем объединении.

**Синхронизация (`atomic`, `mutex`, `condition_variable`) не используется.** Гонки
данных невозможны по построению: каждый выходной пиксель пишется ровно одним
потоком и ровно один раз. Входной буфер разделяется между всеми потоками
без защиты - это безопасно, поскольку никто не пишет во вход.

**Порядок create → work → join.** Сначала все `N` потоков создаются в одном цикле
`emplace_back`, потом отдельным циклом вызывается `join` для каждого:

```cpp
for (int tid = 0; tid < num_threads; tid++) {
  threads.emplace_back(processing, tid);
}
for (auto &th : threads) {
  th.join();
}
```

**Нижне-правый угол** обрабатывается главным потоком уже после `join` - там
работы мало (до 31×31 пикселей), накладные расходы на ещё один параллельный
вызов не окупились бы.

## 5. Детали реализации

Файлы:

- `tasks/romanov_a_gauss_block/stl/include/ops_stl.hpp`
- `tasks/romanov_a_gauss_block/stl/src/ops_stl.cpp`

Структура одного `RunImpl`: создание `num_threads` потоков, каждый выполняет лямбду
`processing(tid)`, join всех потоков, последовательная обработка нижне-правого
угла.

Лямбда `processing` захватывает по ссылке `[&]` все нужные переменные: размеры
изображения, входной и выходной буферы, число потоков и количество блоков.
Параметр `current_part` передаётся по значению - это локальный идентификатор
потока, по которому он вычисляет свои границы работы.

**Карта поток → диапазон:** для потока `tid` область работы определяется как
непрерывный диапазон рядов блоков `[left_border_r, right_border_r)` для основной
зоны и правого хвоста, и непрерывный диапазон колонок блоков `[left_border_l, right_border_l)`
для нижнего хвоста.

## 6. Проверка корректности

Используется тот же набор из 8 функциональных тестов, что в seq, OMP и TBB версиях.
`CheckTestOutputData` сравнивает результат STL-версии с эталоном побайтово.

Тесты прогонялись при разных значениях `PPC_NUM_THREADS` (1, 2, 4, 6, 12) -
расхождений с seq не зафиксировано. Поскольку синхронизации между потоками
нет, отсутствие гонок гарантировано архитектурно: каждый поток работает только
со своей областью выходного буфера.

**Команда запуска func-тестов:**

```bash
./build/bin/ppc_func_tests --gtest_filter=*romanov_a_gauss_block*stl*
```

## 8. Результаты

| Threads | Время, с (task_run) | Speedup vs STL(1) | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.3247 | 1.00× | 100% |
| 2 | 0.1831 | 1.77× | 89% |
| 4 | 0.1449 | 2.24× | 56% |
| 6 | 0.1295 | 2.51× | 42% |
| 12 | 0.0953 | 3.41× | 28% |

Speedup рассчитан относительно STL на одном потоке (`T_stl(1) = 0.325 с`).

**Масштабируемость.** На 2 потоках эффективность 89% - почти линейная. На 4 потоках
эффективность падает до 56%, далее снижается с увеличением числа потоков. Картина
качественно похожа на OMP и TBB: те же физические ограничения (L3-кэш на 4 ядрах,
все физические ядра на 6, Hyper-Threading на 12).

**Накладные расходы create/join.** На каждом вызове `RunImpl` создаётся
`num_threads` новых `std::thread` и сразу же уничтожаются после `join`.
В отличие от OMP и TBB, здесь нет пула потоков, который можно переиспользовать.
Это видно на сравнении с другими реализациями при одинаковом числе потоков:
на 12 потоках STL даёт 0.0953 с против 0.0907 с у OMP и 0.0943 с у TBB.
Разница небольшая (около 5%), потому что время на создание потока (\~50 микросекунд)
относительно мало по сравнению с временем самой свёртки (~\300 миллисекунд на
одном потоке). На задачах с короткими `RunImpl` эта разница была бы существеннее.

## 9. Выводы

Ручная реализация на `std::thread` показывает, что значительная часть выигрыша
от параллелизации в данной задаче определяется не runtime-библиотекой, а самой
блочной декомпозицией. Архитектурное устранение гонок (непересекающиеся блоки
записи) позволяет обойтись без какой-либо синхронизации - нет ни одного `mutex`,
`atomic` или barrier-объекта.

Главный недостаток — отсутствие пула потоков. При каждом вызове RunImpl
создаётся num_threads новых объектов std::thread, которые завершают работу
и уничтожаются после join. В OMP и TBB потоки переиспользуются между
вызовами параллельной секции, что снижает накладные расходы. На текущем
размере задачи (8K) затраты на создание и завершение потоков незначительны
относительно времени самой свёртки, но для задач с более короткими
параллельными секциями STL уступал бы OMP и TBB заметнее.
