## 1. Контекст

TBB-версия - параллельная реализация той же блочной свёртки, что и OMP, но с
использованием oneTBB вместо OpenMP. Главное отличие в модели исполнения: вместо
явных директив компилятора используется task-based параллелизм через `parallel_for`,
а распределением работы между потоками управляет TBB runtime.

Алгоритмическая идея та же: изображение режется на блоки 32×32, каждый блок
обрабатывается независимо.

## 2. Постановка задачи

Постановка идентична seq-версии (см. `seq/report.md`): RGB-изображение `width × height`
на входе, изображение того же размера после свёртки 3×3 с ядром Гаусса на выходе.
Корректность проверяется побайтовым равенством с эталоном seq.

## 3. Базовый алгоритм

Свёртка для каждого пикселя по каждому каналу:

```cpp
sum = sum of {kr=-1..1, kc=-1..1} img[row+kr, col+kc, c] * K[1+kr, 1+kc]
result = clamp((sum + 8) / 16, 0, 255)
```

Сложность: `O(width × height)`.

## 4. Схема распараллеливания

**Выбранный примитив - `tbb::parallel_for`** с явным `tbb::blocked_range<int>`.
Используется именно `parallel_for`, а не `parallel_reduce` или `task_group`,
потому что задача состоит из независимых единиц работы без необходимости агрегации
результата: каждый блок пишет в свой непересекающийся регион выходного изображения.

**Диапазон работы.** Изображение делится на три зоны, как и в OMP-версии:

1. Сетка полных блоков 32×32 - основная масса работы.
2. Правый и нижний хвосты - полосы, которые не помещаются в полные блоки.
3. Нижний правый угол - пересечение хвостов, размер до 31×31.

Для каждой зоны вызывается отдельный `parallel_for`. Диапазон выражается через
индексы рядов или колонок блоков, а не через пиксели:

```cpp
tbb::parallel_for(
    tbb::blocked_range<int>(0, num_row_blocks, 1),
    [&](const tbb::blocked_range<int> &r) {
      for (int bi = r.begin(); bi != r.end(); ++bi) {
        for (int bj = 0; bj < num_col_blocks; ++bj) {
          ProcessFullBlock(initial_picture, result_picture, width, height,
                           bi * kBlockSize, bj * kBlockSize);
        }
      }
    });
```

**Grainsize = 1.** Третий параметр `blocked_range` задаёт минимальный размер чанка,
который TBB не будет дробить дальше. Значение 1 означает «допустимо отдать одному
worker-потоку даже один ряд блоков». Поскольку обработка одного ряда блоков -
это уже значительная работа (`num_col_blocks * 32 * 32` пикселей), накладные
расходы на запуск задачи окупаются.

**Partitioner - по умолчанию (`auto_partitioner`).** TBB самостоятельно решает,
как именно разбить диапазон между worker-потоками, и может перераспределять работу
динамически через work-stealing. Это полезно, если разные ряды блоков будут
обрабатываться неравномерно (например, из-за пропусков кэша на разных участках
изображения). Для данной задачи нагрузка по блокам почти однородна, и преимущество
auto_partitioner - небольшое, но overhead тоже минимален.

**Ограничение конкуренции.** В коде нет явного `global_control::max_allowed_parallelism`.
Число worker-потоков задаётся инфраструктурой курса через переменную окружения
`PPC_NUM_THREADS` (это видно по тому, что замеры с разным `PPC_NUM_THREADS` дают
разное время - см. раздел "Результаты"). При запуске вне runner-а TBB использует
число потоков по умолчанию, равное числу аппаратных потоков машины.

**Синхронизация.** Между тремя `parallel_for` существует неявная барьерная
синхронизация: вызов `parallel_for` блокирует поток вызова до завершения всех
дочерних задач. Это гарантирует, что обработка полных блоков завершится до начала
обработки хвостов, хотя в данной задаче эти зоны записывают в непересекающиеся
регионы и могли бы работать одновременно. Последовательность из трёх `parallel_for`
делает код прозрачным и безопасным.

**Гонки данных отсутствуют** по той же причине, что в OMP: каждый блок обрабатывается
ровно одним потоком, и блоки записывают в непересекающиеся пиксели выходного
изображения.

## 5. Детали реализации

Файлы:

- `tasks/romanov_a_gauss_block/tbb/include/ops_tbb.hpp`
- `tasks/romanov_a_gauss_block/tbb/src/ops_tbb.cpp`

Структура `RunImpl` идентична OMP-версии, отличается только способ распараллеливания.
Функции `ProcessFullBlock`, `ProcessPartBlock`, `ApplyKernel` совпадают по логике
с OMP-версией.

Локальные аккумуляторы или ручное объединение результатов не требуются:
каждый поток пишет напрямую в `result_picture` по индексам своего блока,
без конкуренции с другими потоками.

`cache_aligned_allocator` или `scalable_allocator` не используются. В данной
задаче нет частых аллокаций маленьких объектов внутри параллельной секции
и нет проблем с false sharing: размер блока 32×32×3 = 3072 байта значительно
больше типичной кэш линии (64 байта), и соседние блоки в разных потоках
не пересекаются по кэш линиям на границах.

## 8. Результаты

| Threads | Время, с (task_run) | Speedup vs TBB(1) | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.3300 | 1.00× | 100% |
| 2 | 0.2069 | 1.60× | 80% |
| 4 | 0.1300 | 2.54× | 63% |
| 6 | 0.1164 | 2.83× | 47% |
| 12 | 0.0943 | 3.50× | 29% |

Speedup рассчитан относительно TBB на одном потоке (`T_tbb(1) = 0.330 с`).

**Масштабируемость.** Картина похожа на OMP: близкая к линейной эффективность
на 2 потоках (80%), постепенное падение при увеличении числа потоков, и заметное
снижение эффективности на Hyper-Threading. На 4 потоках TBB обрабатывает изображение
быстрее OMP (0.130 vs 0.141), что может быть связано с work-stealing - если один
worker завершает свои ряды раньше других, TBB перераспределяет нагрузку. На 12
потоках разница с OMP пренебрежимо мала: оба runtime упираются в ограничения
аппаратуры (L3-кэш и конкуренция Hyper-Threading).

**Накладные расходы.** На 1 потоке TBB немного быстрее OMP (0.330 vs 0.341) - task-based
runtime TBB при отсутствии конкуренции работает почти без накладных расходов.

## 9. Выводы

TBB даёт результаты, сравнимые с OMP: 3.50× ускорения на 12 потоках. Преимущество
TBB в данной задаче не радикальное - большая часть выигрыша достаётся уже от
самой блочной декомпозиции, а runtime просто эффективно распределяет независимые
задачи. Auto_partitioner и work-stealing TBB могут давать небольшой выигрыш
на промежуточных значениях числа потоков (4-6), где есть смысл балансировать
работу динамически.

При увеличении числа потоков сверх числа физических ядер эффективность падает -
здесь TBB ничем не отличается от OMP: упираемся в общие ресурсы памяти и
конкуренцию за исполнительные блоки на одном физическом ядре.
