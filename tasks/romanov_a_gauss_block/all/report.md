## 1. Контекст

ALL-версия объединяет два уровня параллелизма: межпроцессный (MPI) и
внутрипроцессный (`std::thread`). Изображение распределяется между несколькими
MPI-процессами по горизонтальным полосам, а внутри каждого процесса полоса
обрабатывается потоками - так же, как в STL-версии.

Гибридная схема позволяет использовать модели параллелизма с разными свойствами
одновременно: MPI подходит для крупных независимых блоков работы и потенциально
для распределённых машин, а потоки эффективны на общей памяти внутри одного
узла.

## 2. Постановка задачи

Постановка идентична seq-версии (см. `seq/report.md`): RGB-изображение
`width × height` на входе, изображение того же размера после свёртки 3×3 с
ядром Гаусса на выходе. Корректность определяется побайтовым равенством с
эталоном seq.

## 3. Базовый алгоритм

Свёртка для каждого пикселя по каждому каналу:

```cpp
sum = sum of {kr=-1..1, kc=-1..1} img[row+kr, col+kc, c] * K[1+kr, 1+kc]
result = clamp((sum + 8) / 16, 0, 255)
```

Сложность: `O(width × height)`.

## 4. Межпроцессная схема

**Роли rank-ов.** Все процессы равноправны в плане вычислений: каждый
обрабатывает свою полосу изображения. Rank 0 дополнительно хранит исходное
изображение до распределения и получает результат после сборки.

**Распределение данных.** Изображение делится на полосы по рядам блоков
размером 32 пикселя по вертикали. Если общее число рядов блоков не делится
на число процессов нацело, остаток размазывается по первым процессам. Если
высота изображения не делится на 32 пикселя нацело, нижний хвост целиком
отдаётся последнему процессу.

Для корректной обработки границ между полосами
каждому процессу передаётся не только его полоса, но и одна дополнительная
строка пикселей сверху и одна снизу (если у процесса есть соседи в этих
направлениях). Это необходимо, поскольку ядро 3×3 для пикселя на границе
полосы должно читать соседние пиксели, которые лежат в полосе соседнего
процесса.

Передача реализована через `MPI_Scatterv` с пересекающимися смещениями: счётчики
и смещения подобраны так, что куски, передаваемые соседним процессам,
перекрываются на одну строку. Это позволяет избежать ручных `MPI_Send`/`MPI_Recv`
для обмена граничными строками.

**Используемые MPI-вызовы:**

- `MPI_Bcast` для размеров изображения (`width`, `height`) - все процессы
  должны знать размер задачи перед построением распределения.
- `MPI_Scatterv` для рассылки полос с отступами. Пересекающиеся смещения отправляют
  каждому процессу его полосу плюс граничные строки соседей.
- `MPI_Gatherv` для сборки результата на rank 0. В отличие от scatter, здесь
  смещения не пересекаются - каждый процесс пишет только свою полосу.
- `MPI_Bcast` финального результата на все процессы. Это необходимо потому,
  что тестирующая инфраструктура курса вызывает `CheckTestOutputData` на
  каждом процессе и ожидает валидный `GetOutput()` везде, а не только на
  rank 0.

**Барьерная синхронизация.** Явных `MPI_Barrier` в коде нет. Все коллективные
операции (`Bcast`, `Scatterv`, `Gatherv`) сами по себе содержат
неявную синхронизацию: процесс не может продолжить выполнение, пока все
участники не вошли в операцию.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса полоса обрабатывается потоками `std::thread` по
той же схеме, что в STL-версии. Число потоков на процесс задаётся через
`ppc::util::GetNumThreads()`, которое читает `PPC_NUM_THREADS`.

**Разбиение работы между потоками.** Полоса процесса делится на три зоны:

1. Сетка полных блоков 32×32 - основная работа.
2. Правый хвост - полоса справа шириной `width % 32`.
3. Нижний хвост - обрабатывается только последним MPI-процессом (только у
   него есть `height % 32` строк остатка).

Каждый поток получает свою долю рядов блоков по формуле
`(num_blocks * tid) / num_threads`, как в STL-версии. Ряды блоков
не пересекаются, поэтому потоки пишут в разные пиксели выходного буфера.

**Объединение результатов.** Локальные результаты потоков не нужны: каждый
поток сразу пишет в общий `local_output` процесса. После завершения всех
потоков процесса локальный результат собирается из всех процессов через
`MPI_Gatherv`.

**Синхронизация между потоками** одного процесса не требуется по той же
причине, что в STL: каждый блок обрабатывается ровно одним потоком,
блоки не пересекаются по выходным пикселям.

## 6. Детали реализации

Файлы:

- `tasks/romanov_a_gauss_block/all/include/ops_all.hpp`
- `tasks/romanov_a_gauss_block/all/src/ops_all.cpp`

Логика разбита на пять функций:

- `BuildDistribution` - формирует распределение полос между процессами.
- `ScatterWithHalo` - выполняет `MPI_Scatterv` с пересекающимися смещениями.
- `ProcessThreadShare` - обработка одной долей работы одним потоком.
- `RunLocal` - создание потоков и обработка локальной полосы.
- `GatherAndBroadcast` - `MPI_Gatherv` собирает результат на rank 0, затем
  `MPI_Bcast` рассылает его всем.

**Потенциальные узкие места:**

- **Копирование данных.** При каждом вызове `RunImpl` выполняется
  scatter (\~99 МБ для 8K) + gather (\~99 МБ) + broadcast (\~99 МБ). На
  одном узле это всё локальные `memcpy` в общей памяти, но всё равно
  занимает заметное время.
- **Нижний хвост на одном процессе.** Если `height % 32 ≠ 0`, нижний хвост
  целиком отдаётся последнему процессу. При больших размерах хвоста и малом
  числе процессов это может привести к неравномерной нагрузке.

## 7. Проверка корректности

Используется тот же набор из 8 функциональных тестов, что в seq, OMP, TBB, STL
версиях. `CheckTestOutputData` сравнивает результат ALL-версии с эталоном
побайтово.

Тесты прогонялись при разных значениях `PPC_NUM_THREADS` и `-np` (1, 2, 4 процессов;
1, 4, 12 потоков на процесс). Расхождений с seq не зафиксировано. Поскольку
финальный результат рассылается через `MPI_Bcast`, согласованность между
процессами гарантируется автоматически: на всех рангах `GetOutput()` содержит
одинаковые данные.

**Команда запуска func-тестов:**

```bash
mpirun --allow-run-as-root --oversubscribe -np 4 \
  ./build/bin/ppc_func_tests --gtest_filter=*romanov_a_gauss_block*all*
```

Эффективность считается как `speedup / total_workers`, где `total_workers = ranks × threads`.

**Цена MPI-инфраструктуры.** Уже на конфигурации `(1, 1)` ALL медленнее seq
(0.41 с против 0.37 с). Эта разница объясняется накладными расходами на
коллективные операции даже в случае одного процесса: scatter, gather и broadcast
выполняются хоть и с тривиальной коммуникацией, но включают копирование
~300 МБ данных через буферы MPI.

**Поведение при росте числа потоков на одном процессе.** Конфигурации
`(1, 4)` и `(1, 12)` дают почти одинаковое время (~0.30 с), что заметно
хуже масштабируемости STL-версии при том же числе потоков (0.145 с и 0.095 с
соответственно). Причина - фиксированный накладной расход на копирование
данных в локальные буферы и обратно, который не зависит от числа потоков.
При одном процессе вся работа по копированию остаётся на главном потоке,
и параллельные вычисления не могут перекрыть эту фазу.

**Поведение при росте числа процессов.** Конфигурации с несколькими процессами
показывают лучший результат, чем с одним: ALL(6, 2) и ALL(4, 3) обходят
ALL(1, 12). Это связано с тем, что копирование данных в `local_input`
теперь распределяется между процессами - каждый процесс копирует только
свою полосу, и эти копирования происходят параллельно.

**Лучшая конфигурация - `(4, 3)` с 12 рабочими единицами:** 0.190 с. Она
обходит варианты с большим числом процессов (`(6, 2)`) и меньшим числом
потоков (`(2, 6)`).

**Эффективность невысока** на всех конфигурациях с большим числом workers.
Это ожидаемо для гибридной схемы: значительная часть времени уходит не на
вычисления, а на накладные расходы MPI и копирование буферов. Сравнение с
иными реализациями (OMP, TBB, STL) в этом смысле будет некорректным без
учёта того, что ALL делает дополнительную работу.

## 10. Выводы

ALL даёт лучшее время на 12 рабочих единицах (0.190 с) среди всех гибридных
конфигураций, но всё же уступает чистым потоковым реализациям (OMP - 0.091 с,
TBB - 0.094 с, STL - 0.095 с на тех же 12 потоках).

Главная причина - накладные расходы MPI на копирование данных. На одном узле
с общей памятью MPI не имеет преимуществ перед чистыми потоками, и его
участие добавляет работу без выигрыша.

Гибридная схема была бы оправдана в одном из двух случаев:

1. Машина состоит из нескольких узлов с собственной памятью (кластер).
   Тогда MPI становится единственным способом задействовать ресурсы соседних
   узлов, а потоки внутри узла используют локальную память.
2. Задача значительно крупнее, чем помещается в RAM одного узла.

В рамках одной машины наилучший результат показывают полностью потоковые
реализации; гибридная версия здесь интересна скорее как демонстрация работы
с коллективными операциями MPI, чем как способ ускорения.
