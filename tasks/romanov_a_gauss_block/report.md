## 1. Введение

Работа содержит реализацию и сравнительный анализ алгоритма линейной фильтрации
RGB-изображения сглаживающим ядром Гаусса 3×3. Для одного и того же алгоритма
выполнены пять реализаций: последовательная (SEQ) и четыре параллельные -
OpenMP (OMP), oneTBB (TBB), `std::thread` (STL) и гибридная MPI + `std::thread` (ALL).

Задача подходит для сравнения моделей параллелизма по следующим причинам:

- Каждый выходной пиксель вычисляется независимо от других выходных пикселей,
  что делает алгоритм идеально подходящим для распараллеливания без необходимости
  синхронизации записи.
- Объём работы фиксирован и заранее известен - `width × height × 3 × 9` умножений
  на один прогон. Это упрощает анализ распределения нагрузки.
- Алгоритм имеет компактную окрестность зависимости (3×3 пикселя), что делает
  его удобным для блочной декомпозиции.

## 2. Единая постановка задачи

**Вход.** RGB-изображение размера `width × height`, представленное в виде
плоского массива байтов длины `width × height × 3` (порядок каналов R, G, B).

**Выход.** Изображение того же размера и формата, в котором каждый пиксель получен
свёрткой с ядром Гаусса 3×3 (нормировка 1/16, веса `1 2 1 / 2 4 2 / 1 2 1`).

**Граничные случаи.** На границах изображения часть ядра выходит за пределы массива;
такие позиции пропускаются, делитель остаётся равным 16.

**Критерий корректности.** Результат любой параллельной версии считается корректным,
если он совпадает с результатом SEQ-версии на одинаковом входе.

## 3. Единая методика эксперимента

### Окружение

- **CPU:** Intel Core i5-10400F @ 2.90 GHz, 6 ядер / 12 потоков (Hyper-Threading)
- **RAM:** 32 GB
- **OS контейнера:** Ubuntu 24.04.3 внутри Docker (Host: Windows 10)
- **Compiler:** g++ 13.3.0
- **Build type:** Release
- **MPI:** Open MPI

### Размер задачи и генерация данных

Размер изображения для perf-тестов зафиксирован в коде `tests/performance/main.cpp`
и равен `7680 × 4320` пикселей (8K). Входное изображение генерируется в `SetUp`
методом `std::uniform_int_distribution<int>(0, 255)` с `std::mt19937` -
случайное равномерное распределение байтов по всем трём каналам.

### Управление числом потоков и процессов

- `PPC_NUM_THREADS` - число потоков на процесс. STL и TBB читают эту переменную
  через `ppc::util::GetNumThreads()`; OMP - через `OMP_NUM_THREADS`.
- `-np N` для `mpirun` - число MPI-процессов. Используется только в ALL-версии.
- Общее число рабочих единиц для ALL - `ranks × threads`.

### Формулы

```cpp
speedup(p)    = T_seq / T(p)
efficiency(p) = speedup(p) / workers(p)
```

Для ALL `workers = ranks × threads_per_rank`. Для остальных кодов
`workers = threads`.

### Агрегация результатов

Для SEQ замер выполнялся 5 раз, в таблицу занесена медиана. Для остальных
реализаций приведены значения одиночных прогонов на каждой конфигурации.
Дополнительных мер стабилизации (закрепление за ядром, фиксация частоты) не
применялось.

## 6. Интерпретация различий

### SEQ

Baseline для всех остальных реализаций. Простой тройной цикл по строкам,
столбцам и каналам, без каких-либо специальных оптимизаций. Время 0.367 с
на изображении 7680×4320 - это около 2.7 млрд умножений и столько же сложений.

### OMP

Показывает наилучший результат среди всех реализаций: 4.0x на 12 потоках.
Схема параллелизации минимальна - три директивы `#pragma omp parallel for` с
`schedule(static)`. OpenMP runtime использует пул потоков, который переиспользуется
между параллельными секциями, что даёт небольшое преимущество над STL на коротких
параллельных секциях. Эффективность на 2 потоках близка к идеальной (94%), но
падает с ростом числа потоков из-за конкуренции за общий кэш и переход на
Hyper-Threading сверх 6 потоков.

### TBB

Результат близок к OMP: 3.89x на 12 потоках. Использует `tbb::parallel_for`
с `blocked_range` и `auto_partitioner`. Преимущество work-stealing проявляется
на промежуточном числе потоков (4 потока): TBB обрабатывает быстрее OMP
(0.130 с против 0.141 с), что может говорить о небольшом выигрыше от
динамической балансировки. На 12 потоках разница с OMP пренебрежимо мала -
обе реализации упираются в аппаратные ограничения.

### STL

Ручная реализация на `std::thread` показывает результат, сравнимый с OMP и TBB
(3.85x на 12 потоках). Это подтверждает гипотезу, что значительная часть выигрыша
обеспечивается самой блочной декомпозицией, а не выбором runtime. STL немного
отстаёт от OMP (0.095 с против 0.091 с) из-за отсутствия пула потоков: каждый
вызов `RunImpl` создаёт `num_threads` новых объектов `std::thread`. На текущем
размере задачи эта разница составляет около 5%.

### ALL

Гибридная версия даёт лучший результат на конфигурации `(4 × 3)`: 0.190 с,
что соответствует 1.93× относительно SEQ. На той же общей численности рабочих
единиц (12) ALL заметно отстаёт от чистых потоковых реализаций. Причины:

- На каждом вызове `RunImpl` выполняются три коллективные операции (Scatterv,
  Gatherv, Bcast), которые включают копирование ~300 МБ данных через буферы MPI.
  На одной машине с общей памятью эта работа избыточна.
- При запуске с одним процессом (`-np 1`) всё копирование выполняется
  последовательно главным потоком, и параллельные вычисления не могут перекрыть
  эту фазу - отсюда плохой результат `(1 × 12)`: 0.301 с.
- При запуске с несколькими процессами копирование распределяется между ними,
  и результат улучшается.

Гибридная схема была бы оправдана на машине из нескольких узлов с раздельной
памятью или для задач, не помещающихся в RAM одного узла. В рамках одной машины
её преимущества не проявляются.

## 7. Репродуцируемость

### Сборка

```bash
cmake -S . -B build \
  -D USE_FUNC_TESTS=ON \
  -D USE_PERF_TESTS=ON \
  -D CMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

### Функциональные тесты

Запуск всех версий одной командой:

```bash
mpirun --allow-run-as-root --oversubscribe -np 4 \
  ./build/bin/ppc_func_tests --gtest_filter=*romanov_a_gauss_block*
```

### Perf-тесты

```bash
# SEQ
./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*seq*

# OMP (число потоков задаётся OMP_NUM_THREADS)
OMP_NUM_THREADS=<N> \
  ./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*omp*

# TBB и STL (число потоков задаётся PPC_NUM_THREADS)
PPC_NUM_THREADS=<N> \
  ./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*tbb*
PPC_NUM_THREADS=<N> \
  ./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*stl*

# ALL (число процессов через -np, потоков через PPC_NUM_THREADS)
PPC_NUM_THREADS=<T> \
  mpirun --allow-run-as-root --oversubscribe -np <R> \
  ./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*all*
```

### Запуск через скрипт курса

```bash
# Threads-режим: seq, omp, tbb, stl
OMPI_ALLOW_RUN_AS_ROOT=1 OMPI_ALLOW_RUN_AS_ROOT_CONFIRM=1 \
PPC_NUM_THREADS=4 PPC_NUM_PROC=1 \
python3 scripts/run_tests.py --running-type=threads --counts 1 2 3 4

# Processes-режим: all
OMPI_ALLOW_RUN_AS_ROOT=1 OMPI_ALLOW_RUN_AS_ROOT_CONFIRM=1 \
PPC_NUM_THREADS=4 PPC_NUM_PROC=4 \
python3 scripts/run_tests.py --running-type=processes --counts 1 2 4 \
  --additional-mpi-args="--oversubscribe"

# Performance
OMPI_ALLOW_RUN_AS_ROOT=1 OMPI_ALLOW_RUN_AS_ROOT_CONFIRM=1 \
PPC_NUM_THREADS=4 PPC_NUM_PROC=4 \
python3 scripts/run_tests.py --running-type=performance \
  --additional-mpi-args="--oversubscribe"
```

## 8. Заключение

Реализованы и протестированы пять версий блочного алгоритма линейной фильтрации
изображения ядром Гаусса 3×3. На изображении 7680×4320 получены следующие
основные результаты:

1. **Наилучшая абсолютная производительность** - OMP с 12 потоками
   (0.091 с, ускорение 4.04x относительно SEQ). TBB и STL показывают
   близкие результаты (3.89x и 3.85x соответственно), разница между этими
   тремя реализациями невелика - около 5%.
2. **Основной источник ускорения** - блочная декомпозиция, а не выбор
   фреймворка. Это подтверждается тем, что ручная реализация на `std::thread`
   практически не уступает OMP и TBB.
3. **Гибридная версия ALL** на одной машине проигрывает чистым потоковым
   реализациям в два раза (1.93x против 3.85-4.04x). Причина - накладные
   расходы на копирование данных через буферы MPI при коллективных операциях.

**Ограничения исследования:**

- Замеры выполнялись без явных мер стабилизации и в
  среде Docker под Windows.
- Использовалось одно фиксированное разрешение (8K). Поведение на меньших
  и больших размерах изображения может отличаться, особенно для ALL, где
  накладные расходы MPI становятся более или менее существенными в зависимости
  от объёма данных.
- Для OMP, TBB, STL и ALL приведены результаты одиночных прогонов, а не
  медианы - это не позволяет различить близкие значения внутри статистической
  погрешности.

**Возможные направления улучшения:**

- Замеры по нескольким размерам изображения (например, 1080p, 4K, 8K) показали
  бы зависимость ускорения от объёма работы и точку выхода на плато для
  каждой реализации.
- В ALL-версии можно было бы отказаться от финального `MPI_Bcast` результата,
  если бы тестовая инфраструктура курса допускала возвращение пустого
  `GetOutput()` на не-нулевых рангах. Это сократило бы один из трёх крупных
  обменов данных.
