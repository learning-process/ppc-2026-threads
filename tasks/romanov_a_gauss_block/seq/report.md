## 1. Контекст

Задача заключается в линейной фильтрации цветного изображения с применением ядра Гаусса 3×3.
Последовательная версия играет ключевую роль: на ней фиксируется корректность алгоритма
и базовое время выполнения, относительно которого считаются ускорение и эффективность
всех параллельных реализаций.

## 2. Постановка задачи

**Вход.** RGB-изображение размера `width × height`, представленное в виде плоского массива
байтов длины `width × height × 3` (порядок каналов R, G, B). Тип входа -
`std::tuple<int, int, std::vector<uint8_t>>`.

**Выход.** Изображение того же размера и формата, в котором каждый пиксель получен
сглаживанием соответствующего пикселя входа ядром Гаусса 3×3. Тип выхода -
`std::vector<uint8_t>`.

**Используемое ядро** (нормировано):

```
1/16 * | 1  2  1 |
       | 2  4  2 |
       | 1  2  1 |
```

**Ограничения.** Размер вектора пикселей должен в точности равняться `width * height * 3`;
несоответствие отклоняется в `ValidationImpl`. Каналы изображения обрабатываются независимо
друг от друга.

**Граничные случаи.** На границах изображения часть ядра выходит за пределы массива.
Такие позиции просто пропускаются - отсутствующие пиксели в сумму не входят, делитель
при этом остаётся равным 16.

## 3. Базовый алгоритм

Для каждого пикселя `(row, col)` и каждого канала `c ∈ {0, 1, 2}` вычисляется свёртка
с ядром Гаусса:

```
sum(row, col, c) = sum of {kr ∈ -1..1} sum of {kc ∈ -1..1}
                   img[row+kr, col+kc, c] * K[1+kr, 1+kc]
result(row, col, c) = clamp((sum + 8) / 16, 0, 255)
```

Прибавление `8` перед делением реализует округление к ближайшему целому при делении на 16.
`clamp` предотвращает выход за диапазон `uint8_t`.

**Сложность.**

- По времени: `O(width × height × 3 × 9) = O(width × height)`
- По памяти: `O(width × height)` - один входной и один выходной буфер.

**Главная структура данных** - линейный `std::vector<uint8_t>` с упаковкой
`[row][col][channel]`. Индексация: `((row * width) + col) * 3 + channel`. Это
обеспечивает последовательный доступ к памяти при проходе по изображению, что
важно для кэша процессора.

**Инварианты.**

- Размеры выходного буфера фиксируются равными размерам входного перед началом обработки.
- Каждый пиксель выхода зависит только от пикселей входа; запись в выход и чтение из входа
  не пересекаются. Это и делает алгоритм пригодным для последующего распараллеливания
  без синхронизации записи.
- Эталонная корректность определяется побитовым равенством результата эталонному выходу
  (см. раздел "Проверка корректности").

## 4. Детали реализации

Файлы:

- `tasks/romanov_a_gauss_block/seq/include/ops_seq.hpp` - объявление класса
  `RomanovAGaussBlockSEQ`, наследника `BaseTask` с `TypeOfTask = kSEQ`.
- `tasks/romanov_a_gauss_block/seq/src/ops_seq.cpp` - реализация четырёх этапов жизненного
  цикла задачи.

**`ValidationImpl`** проверяет одну инвариантную связь между полями входа:

```cpp
return std::get<0>(GetInput()) * std::get<1>(GetInput()) * 3 ==
       static_cast<int>(std::get<2>(GetInput()).size());
```

Случаи, когда указанные `width × height` не согласуются с реальным размером буфера
пикселей, отклоняются на этом этапе и не доходят до `RunImpl`.

**`PreProcessingImpl`** возвращает `true`. Никакая дополнительная подготовка не требуется:
входной буфер уже находится в формате, пригодном для свёртки, а размер выходного буфера
аллоцируется внутри `RunImpl`.

**`RunImpl`** содержит тройной цикл по `row`, `col`, `channel`. Свёртка вынесена в
отдельную функцию `ApplyKernel`, которая суммирует произведения соседних пикселей на
веса ядра, аккуратно обходя позиции за границей изображения. Ниже - иллюстративный
фрагмент тела цикла, отвечающий за единственный пиксель:

```cpp
// File: tasks/romanov_a_gauss_block/seq/src/ops_seq.cpp
for (int row = 0; row < height; ++row) {
  for (int col = 0; col < width; ++col) {
    for (int channel = 0; channel < 3; ++channel) {
      int sum = ApplyKernel(initial_picture, row, col, channel, width, height, kernel);
      int result_value = (sum + 8) / 16;
      result_value = std::clamp(result_value, 0, 255);
      auto idx = ((static_cast<size_t>(row) * width + col) * 3) + channel;
      result_picture[idx] = static_cast<uint8_t>(result_value);
    }
  }
}
```

Порядок итераций `row → col → channel` соответствует порядку хранения данных в памяти
и обеспечивает последовательный (cache-friendly) доступ к входному и выходному буферам.

**`PostProcessingImpl`** возвращает `true`. Готовый результат уже записан в выходной
буфер задачи в конце `RunImpl`, дополнительной обработки не требуется.

## 5. Проверка корректности

Функциональные тесты находятся в `tasks/romanov_a_gauss_block/tests/functional/main.cpp`.
Используется один шаблонный класс `RomanovARunFuncTestsThreads`, который параметризован
типом задачи и набором тестовых случаев. Метод `CheckTestOutputData` сравнивает
полученный выход с заранее посчитанным эталонным результатом побайтовым равенством:
`expected_result_ == output_data`.

Набор тестов покрывает следующие случаи:

| Тест | Размер | Что проверяет |
|---|---|---|
| Test1 | 1×1 | Один пиксель, все границы за пределами. Тривиальный угловой случай. |
| Test2 | 2×1 | Узкая полоса, проверка обработки соседей по горизонтали. |
| Test3 | 1×2 | Узкая полоса, проверка обработки соседей по вертикали. |
| Test4 | 2×2 | Минимальный двумерный случай со всеми каналами. |
| Test5 | 3×3 | Центральный пиксель имеет всю окрестность; угловые и краевые - обрезаются. |
| Test6 | 2×3 | Прямоугольное изображение с градиентом. |
| Test7 | 4×1 | Полностью чёрная строка - проверка устойчивости при нулевом входе. |
| Test8 | 100×100 | Большой случай (3·100·100 = 30 000 пикселей нулей), проверка инвариантов аллокации и записи. |

## 7. Результаты

| Размер | Время, с (медиана из 5) |
|---|---:|
| 7680×4320 | 0.3668 |

Самой дорогой частью алгоритма является сама тройная свёртка в `RunImpl`: на каждом
пикселе выполняется ~27 умножений, ~27 сложений и одна операция деления, что при
99.5 миллионах пикселей даёт около **2.7 миллиарда умножений** на каждый прогон.
Доступы к входному буферу частично теряют локальность при переходе между соседними
строками - окно ядра охватывает три строки одновременно, что увеличивает загрузку
на кэш процессора.

## 8. Выводы

Последовательная реализация устроена максимально просто: один проход по изображению,
один линейный буфер на вход и один на выход, без специальных структур данных или
оптимизаций. Эта простота - то, что делает её пригодной как эталон. Любое отклонение
параллельной версии от seq при побайтовом сравнении выявляется функциональными
тестами, а время `T_seq` даёт честную точку отсчёта для масштабируемости.

Параллельные реализации (OMP, TBB, STL, ALL) надстраиваются над тем же логическим
алгоритмом - каждая своим способом разбивает изображение на независимые куски,
не меняя самой свёртки. Это позволяет в каждом следующем отчёте сосредоточиться
на схеме распараллеливания, а не на алгоритме как таковом.
