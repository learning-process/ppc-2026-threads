## 1. Контекст

OMP-версия параллелизует последовательную реализацию из `seq/report.md`. В отличие
от seq, изображение здесь обрабатывается **блочно**: разбивается на квадратные блоки
32×32 пикселя, и блоки независимо распределяются между потоками OpenMP.

Блочная декомпозиция повышает локальность доступа к памяти: каждый поток обрабатывает
компактную область изображения, которая помещается в L1/L2-кэш ядра.

## 2. Постановка задачи

Постановка идентична seq-версии (см. `seq/report.md`): на вход подаётся RGB-изображение
`width × height`, на выход - изображение того же размера после применения свёртки 3×3
с ядром Гаусса. Корректность определяется побайтовым равенством с эталоном seq.

## 3. Базовый алгоритм

Каждый пиксель `(row, col)` для каждого канала `c` пересчитывается как:

```cpp
sum = Σ_{kr=-1..1, kc=-1..1} img[row+kr, col+kc, c] * K[1+kr, 1+kc]
result = clamp((sum + 8) / 16, 0, 255)
```

Сложность та же: `O(width × height)`.

## 4. Схема распараллеливания

Изображение делится на три зоны:

1. **Сетка полных блоков 32×32** - основная масса работы, покрывает большую часть изображения.
2. **Правый и нижний хвосты** - полосы шириной/высотой `width % 32` и `height % 32`,
   которые не помещаются в полные блоки.
3. **Нижне-правый угол** - пересечение хвостов, размер до 31×31.

**Распараллеливаются три внешних цикла**, по одному на каждую зону:

```cpp
#pragma omp parallel for schedule(static) default(none) \
    shared(initial_picture, result_picture, width, height)
for (int start_row = 0; start_row < (height + 1 - kBlockSize); start_row += kBlockSize) {
  for (int start_col = 0; start_col < (width + 1 - kBlockSize); start_col += kBlockSize) {
    ProcessFullBlock(initial_picture, result_picture, width, height, start_row, start_col);
  }
}
```

**Атрибуты переменных:**

- `default(none)` - обязывает явно описать каждую переменную, защищает от случайного
  захвата.
- `shared(initial_picture, result_picture, width, height)` - все потоки читают из
  общего входного буфера и пишут в общий выходной буфер. Гонки не возникает,
  поскольку **разные потоки пишут в непересекающиеся блоки** (каждый блок 32×32
  обрабатывается ровно одним потоком, и блоки не пересекаются по выходным пикселям).
- `start_row`, `start_col` - переменные цикла, автоматически становятся `private`
  для каждого потока по правилам OpenMP.

**Reduction / atomic / critical не нужны** - операция записи в выходной пиксель
не зависит от других пикселей, и каждый пиксель записывается ровно один раз.
Это прямое следствие инварианта seq-версии: запись в выход и чтение из входа
не пересекаются.

**`schedule(static)`** выбран потому что нагрузка по блокам одинакова: каждый блок
обрабатывает ровно `32 × 32 × 3 = 3072` пикселя, и время на блок практически одинаковое.
Динамическое распределение здесь дало бы лишние накладные расходы на запросы итераций без
выигрыша в балансировке. Со `static` каждому потоку выдаётся непрерывный диапазон
блоков, что также улучшает локальность кэша.

**Неявный барьер в конце `parallel for`** обеспечивает, что все блоки одной зоны
полностью обработаны до начала следующей зоны. Это необходимо: хотя зоны пишут
в непересекающиеся регионы изображения, последовательность из трёх отдельных
parallel-регионов делает код проще и безопаснее, чем одна большая директива.

Нижне-правый угол обрабатывается **последовательно** одним вызовом `ProcessPartBlock` -
работы там мало (до 31×31 пикселей), и распараллеливание этого фрагмента не имеет смысла.

## 5. Детали реализации

Файлы:

- `tasks/romanov_a_gauss_block/omp/include/ops_omp.hpp`
- `tasks/romanov_a_gauss_block/omp/src/ops_omp.cpp`

Относительно seq добавлены:

- Функции `ProcessFullBlock` и `ProcessPartBlock`, инкапсулирующие обработку одного блока.
- Три `#pragma omp parallel for` для трёх зон обработки.
- Финальная последовательная обработка нижне-правого угла.

`ApplyKernel` совпадает с seq-версией и работает потокобезопасно: читает только из
общего входа, возвращает локальную сумму.

**Риски гонок** устранены архитектурно: декомпозиция гарантирует, что каждый выходной
пиксель пишется ровно одним потоком и ровно один раз. Никакой синхронизации записи
не требуется.

## 6. Проверка корректности

Используется тот же набор из 8 функциональных тестов, что описан в `seq/report.md`.
Метод `CheckTestOutputData` сравнивает выход OMP-версии с эталоном побайтово.

независима.

Дополнительно тесты прогонялись при разных значениях `OMP_NUM_THREADS` (1, 2, 4, 6, 12) -

**Управление числом потоков.** Поскольку директивы `#pragma omp parallel for` не
содержат явного `num_threads(...)`, OpenMP runtime использует значение переменной
окружения `OMP_NUM_THREADS`. При запуске через `scripts/run_tests.py` runner курса
автоматически экспортирует `OMP_NUM_THREADS = PPC_NUM_THREADS`; при прямом запуске
бинарника нужно задавать `OMP_NUM_THREADS` напрямую.

**Команда запуска:**

```bash
OMP_NUM_THREADS=<N> \
  ./build/bin/ppc_perf_tests --gtest_filter=*romanov_a_gauss_block*omp*
```

## 8. Результаты

| Threads | Время, с (task_run) | Speedup vs OMP(1) | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.3410 | 1.00× | 100% |
| 2 | 0.1958 | 1.74× | 87% |
| 4 | 0.1411 | 2.42× | 60% |
| 6 | 0.1225 | 2.78× | 46% |
| 12 | 0.0907 | 3.76× | 31% |

Speedup рассчитан относительно OMP-версии на одном потоке (`T_omp(1) = 0.341 с`),
а не относительно seq, чтобы изолировать эффект параллелизации от накладных расходов
самого OpenMP runtime. Сравнение с seq будет в корневом отчёте.

**Масштабируемость.** При переходе от 1 к 2 потокам эффективность близка к идеальной
(87%) - это ожидаемо для задачи c большим число вычислений без синхронизации. На 4 потоках
эффективность падает до 60%, что соответствует началу конкуренции за общий L3-кэш
(12 МБ на 4 ядра вместо отдельных 12 МБ на ядро). На 6 потоках используются все
физические ядра, эффективность ~46% - ожидаемое поведение, когда задействованы все
физические ядра. Переход от 6 к 12 потокам - это использование Hyper-Threading: ускорение
продолжает расти (3.76× против 2.78×), но эффективность падает до 31%, поскольку
два логических потока на одном физическом ядре конкурируют за ALU и L1.

**Узкое место** - нагрузка на L3-кэш. Каждый поток обрабатывает блок 32×32, что
занимает `32 × 32 × 3 = 3072 байта` входа и столько же выхода. Само окно ядра
тянет ещё три строки соседних пикселей. При нескольких потоках, работающих над
разными частями изображения, совокупный рабочий набор быстро превышает L1-кэш
и начинает упираться в L3 и память.

## 9. Выводы

OpenMP даёт значительный выигрыш на этой задаче: 3.76× на полной загрузке
(12 потоков) и 2.78× на чистых физических ядрах (6 потоков). При этом сама
схема распараллеливания получилась простой - три `#pragma omp parallel for`
со `schedule(static)`, без явной синхронизации и без `reduction`.

Главное ограничение - пропускная способность памяти и L3-кэш. На малых числах потоков
эффективность близка к 90%, но падает с ростом числа потоков из-за конкуренции
за общие ресурсы памяти. Hyper-Threading даёт прирост, но небольшой -
это характерно для задач, где основное время уходит на вычисления, а не на
ожидание памяти.
