# Линейная фильтрация изображений (блочное разбиение). Ядро Гаусса 3x3

- **Студент**: Вдовин Артемий Николаевич
- **Группа**: 3823Б1ПР4
- **Вариант**: 26
- **Технологии**: SEQ, OMP, TBB, STL, ALL

## 1. Введение

Фильтрация изображений — одна из наиболее ресурсоемких задач в области обработки мультимедиа. Фильтр Гаусса 3x3
является классическим примером локальной операции, где значение каждого пикселя зависит от его ближайшего
окружения. В данной работе исследуются и сравниваются различные подходы к распараллеливанию этого алгоритма:
от многопоточности в общей памяти (OMP, TBB, STL) до гибридных схем для распределенных систем (MPI + OpenMP).

## 2. Единая постановка задачи

Задано трехканальное изображение произвольного размера. Требуется применить к нему фильтр Гаусса с ядром 3x3.

- **Вход**: ширина `W`, высота `H`.
- **Ядро**: фиксированная матрица 3x3 с коэффициентами, сумма которых равна 16.
- **Обработка границ**: использование метода `clamp` (повторение крайних пикселей).
- **Результат**: контрольная сумма, вычисляемая как среднее арифметическое значение всех цветовых каналов.
- **Цель**: достижение максимального ускорения при сохранении корректности вычислений.

## 3. Единая методика эксперимента

- **Окружение**: MacBook Air M2 (8 ядер), 16 GB RAM, macOS Tahoe 26.4.1, компилятор Apple clang 17.0.6.
- **Данные**: Изображения генерировались программно с фиксированным заполнением для повторяемости.
- **Метрики**:
  - Ускорение (`Speedup`): `T_seq / T_par`.
  - Эффективность (`Efficiency`): `(Speedup / p) * 100%`.
- **Параметры**: Замеры проводились для числа потоков/процессов `p` в диапазоне от 1 до 6.

## 4. Сводка корректности

Все реализованные параллельные версии успешно прошли функциональное тестирование с использованием
`Google Test`. Проверка включала:

- Изображения минимального размера (3x3).
- Изображения большого разрешения (до 5000x5000).
- Проверку на отсутствие гонок данных и корректность обработки границ.
Контрольные суммы во всех версиях полностью совпали с последовательным эталоном.

## 6. Интерпретация различий

- **SEQ**: показывает базовый уровень производительности. Время выполнения линейно зависит от объема данных.
- **OMP**: демонстрирует высокое ускорение уже на одном потоке. Это связано с тем, что прагмы OpenMP
  позволяют компилятору эффективно векторизовать внутренние циклы свертки.
- **TBB и STL**: показывают практически идеальное линейное ускорение до 4 потоков. Эффективность выше 90%
  свидетельствует о хорошей масштабируемости алгоритма.
- **ALL (MPI+OMP)**: гибридная схема показывает отличные результаты, сопоставимые с OMP. Однако при увеличении
  числа процессов время начинает расти из-за накладных расходов на `MPI_Allgatherv`.

## 8. Заключение

Наиболее эффективной технологией для данной задачи на многоядерной системе оказалась OpenMP. Она обеспечивает
не только параллелизм на уровне ядер, но и помогает компилятору выполнять векторизацию. Библиотека TBB и
стандартные потоки STL также показали отличную масштабируемость. Гибридная схема MPI+OpenMP оправдана только
при обработке изображений сверхвысокого разрешения.
