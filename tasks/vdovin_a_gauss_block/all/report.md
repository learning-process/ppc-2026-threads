## 1. Контекст

Гибридное параллельное программирование, объединяющее MPI и OpenMP, является стандартом для
высокопроизводительных вычислений (HPC). MPI используется для взаимодействия между узлами (или процессами)
в распределенной памяти, а OpenMP — для распараллеливания вычислений внутри каждого процесса в общей
памяти. В данной работе такая схема применяется для масштабирования фильтрации изображений.

## 2. Постановка задачи

Задача заключается в реализации фильтра Гаусса 3x3, который может эффективно работать как на одном
многоядерном узле, так и на кластере. Необходимо обеспечить корректную передачу данных между процессами
и эффективное использование потоков внутри каждого процесса.

## 3. Базовый алгоритм

Алгоритм реализует свертку с ядром 3x3. В гибридной версии изображение должно быть доступно всем
вычислителям. Каждый вычислитель отвечает за свой горизонтальный участок изображения.

## 4. Схема распараллеливания

Реализована двухуровневая схема параллелизма:

1. **Уровень процессов (MPI)**:
    - Процесс с рангом 0 инициализирует изображение.
    - Размеры изображения и данные пикселей рассылаются всем процессам с помощью `MPI_Bcast`.
    - Общее количество строк `height` делится между процессами. Каждый процесс вычисляет свой `row_start`
      и `local_rows`.
    - После завершения вычислений результаты собираются со всех процессов с помощью `MPI_Allgatherv`,
      что позволяет каждому процессу иметь полную копию отфильтрованного изображения.
2. **Уровень потоков (OpenMP)**:
    - Внутри каждого MPI-процесса цикл по локальным строкам распараллеливается с помощью
      `#pragma omp parallel for`.
    - Это позволяет задействовать все ядра процессора, выделенные данному MPI-рангу.
    - Используется статическое планирование для минимизации накладных расходов.

## 5. Детали реализации

Основные фрагменты кода:

```cpp
bool VdovinAGaussBlockALL::RunImpl() {
  // MPI инициализация и Bcast ...

  int rows_per_proc = height_ / size;
  int extra = height_ % size;
  // расчет смещений displs и количеств send_counts ...

  int local_rows = rows_per_proc + (rank < extra ? 1 : 0);
  int local_size = local_rows * width_ * kChannels;
  std::vector<uint8_t> local_output(local_size, 0);

#pragma omp parallel for schedule(static) default(none) shared(local_output, local_rows, row_start)
  for (int ry = 0; ry < local_rows; ry++) {
    int py = row_start + ry;
    for (int px = 0; px < width_; px++) {
      for (int ch = 0; ch < kChannels; ch++) {
        local_output[(((ry * width_) + px) * kChannels) + ch] =
            ComputePixelChannel(input_image_, width_, height_, py, px, ch);
      }
    }
  }

  MPI_Allgatherv(local_output.data(), local_size, MPI_UNSIGNED_CHAR, output_image_.data(), send_counts.data(),
                 displs.data(), MPI_UNSIGNED_CHAR, MPI_COMM_WORLD);
  return true;
}
```

## 9. Выводы

Гибридная схема MPI+OpenMP показала отличную производительность. Однако при увеличении числа процессов
накладные расходы на сетевые взаимодействия начинают перекрывать выигрыш от параллельных вычислений,
что приводит к снижению эффективности на больших конфигурациях.
