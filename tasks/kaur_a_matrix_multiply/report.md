## 1. Введение
Задача умножения разреженных матриц выбрана как вычислительно плотная операция с нерегулярным доступом к памяти. Формат CCS (Compressed Column Storage) позволяет эффективно хранить матрицы с большим количеством нулей, но создаёт трудности для распараллеливания из-за косвенной адресации. В работе реализованы и сравнены пять технологий: последовательная (SEQ), OpenMP (OMP), oneTBB (TBB), std::thread (STL) и гибридная MPI + потоки (ALL). Элементы матриц — комплексные числа.

## 2. Единая постановка задачи
- **Вход**: две разреженные матрицы A (height × width) и B (height × width) в формате CCS.
- **Выход**: матрица C = A × B в формате CCS.
- **Ограничения**: A.width == B.height (проверяется в ValidationImpl).
- **Критерий корректности**: поэлементное сравнение результата с последовательной эталонной реализацией через метод Compare класса CCSMatrix.

## 3. Единая методика эксперимента
- **CPU**: Intel Core i7-12700H (14 ядер, 20 потоков)
- **RAM**: 32 GB DDR5
- **OS**: Ubuntu 22.04 (контейнер Docker)
- **Compiler**: GCC 14.2.0
- **Build type**: Release
- **Переменные окружения**: PPC_NUM_THREADS для OMP/STL/TBB, PPC_NUM_PROC + PPC_NUM_THREADS для ALL
- **Входные данные**: генерируются в SetUp тестов — матрицы с разреженностью ~2 ненулевых элемента на строку, значения заполняются по формуле (i² × rows % 50) + i·(j² × cols % 50)
- **Размеры задач**: функциональные тесты — 4×5×6, 6×5×4, 4×6×5, 5×6×4, 6×6×5, 6×5×6, 5×6×6, 8×8×8, 9×9×9, 10×10×10; performance тесты — 14000×14000
- **Speedup**: T_seq / T_parallel
- **Efficiency**: Speedup / num_workers
- **Число повторов**: не менее 5, результаты агрегируются медианой

## 4. Сводка корректности
Все backend-ы сравнивались с SEQ на полном наборе функциональных тестов (10 различных размеров, включая граничные случаи). Расхождения не обнаружены. Тесты производительности подтвердили корректность на матрицах 14000×14000. ALL-версия дополнительно проверена на согласованность результата между rank-ами через MPI.

## 6. Интерпретация различий
- **SEQ** — показывает baseline. Основное время тратится на двойной цикл по ненулевым элементам. Сложность O(nnz_left × nnz_right).
- **OMP** — даёт ускорение за счёт параллельного прохода по элементам левой матрицы. Применение schedule(static) обеспечивает равномерное разбиение. Узкое место — слияние локальных map-буферов после параллельной секции.
- **TBB** — использование blocked_range2d с grain size 64×256 позволяет эффективно использовать кэш. tbb::enumerable_thread_specific автоматически создаёт локальные копии буферов.
- **STL** — ручное управление потоками показывает результаты, близкие к OMP, но требует явного управления локальными буферами и join. Накладные расходы на создание потоков становятся заметны на малых размерах.
- **ALL** — MPI-коммуникации (Bcast + Scatter + Gather) добавляют overhead, который окупается только на больших размерах. Внутрипроцессное распараллеливание через std::thread аналогично STL-версии.
