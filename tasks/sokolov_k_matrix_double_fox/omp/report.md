## 1. Контекст

OpenMP-версия алгоритма Фокса использует директивы компилятора для автоматического
распараллеливания циклов. Технология OpenMP позволяет с минимальными изменениями кода получить
параллельную версию, используя системный пул потоков.

## 2. Постановка задачи

Задача остается той же, что и в последовательной версии: умножение квадратных матриц A и B
размером n x n методом Фокса. Подробная постановка приведена в seq/report.md. Базовая
последовательная версия используется как эталон для проверки корректности и расчета ускорения.

## 3. Базовый алгоритм

Алгоритм Фокса выполняет блочное умножение матриц за q шагов, где q - количество блоков по
каждому измерению. На каждом шаге для каждой блочной строки выполняется умножение соответствующих
блоков матриц A и B с накоплением результата в матрице C.

## 4. Схема распараллеливания

### 4.1. Область параллелизма

Основная область распараллеливания - внешний цикл по блочным строкам в функции FoxStepOmp.
Блочные строки независимы между собой на каждом шаге, что позволяет безопасно распределять их
по потокам.

### 4.2. Директива OpenMP

```cpp
#pragma omp parallel for default(none) shared(a, b, c, bs, q, step, bsq) schedule(static)
for (int i = 0; i < q; i++) {
    int k = (i + step) % q;
    for (int j = 0; j < q; j++) {
        MultiplyBlocksLocal(a, ((i * q) + k) * bsq,
                          b, ((k * q) + j) * bsq,
                          c, ((i * q) + j) * bsq, bs);
    }
}
```

### 4.3. Атрибуты переменных

- **shared:** a, b, c, bs, q, step, bsq - эти переменные читаются всеми потоками и не изменяются во время параллельной секции
- **private:** i, k, j - переменные цикла, каждый поток имеет свою копию
- **default(none):** требует явного указания атрибутов для всех переменных, что предотвращает ошибки

### 4.4. Выбор schedule

Используется **schedule(static)**, так как:

- Все блочные строки требуют примерно одинакового времени обработки
- Статическое распределение минимизирует накладные расходы на планирование
- Отсутствует дисбаланс нагрузки между итерациями

### 4.5. Синхронизация

- **Неявный барьер:** в конце каждой параллельной секции OpenMP автоматически вставляет барьер,
  что гарантирует завершение всех потоков перед переходом к следующему шагу Фокса
- **Отсутствие гонок:** каждый поток записывает в свой уникальный блок матрицы C, поэтому
  reduction и критические секции не требуются

### 4.6. Дополнительное распараллеливание

Также распараллелены фазы декомпозиции и сборки матриц, что дает дополнительный выигрыш на
этапах подготовки и постобработки данных.

## 5. Детали реализации

**Файлы:**

- omp/include/ops_omp.hpp - объявление класса SokolovKMatrixDoubleFoxOMP
- omp/src/ops_omp.cpp - реализация с OpenMP-директивами

**Изменения относительно SEQ:**

- Добавлена директива `#pragma omp parallel for` в функцию FoxStepOmp
- Добавлено распараллеливание циклов декомпозиции и сборки
- Функция MultiplyBlocks переименована в MultiplyBlocksLocal для подчеркивания локальности
  вычислений

**Устранение рисков гонок:**

- Каждый поток обрабатывает свою блочную строку i
- Запись в матрицу C происходит в блок C\[i\]\[j\], где i уникален для каждого потока
- Чтение из матриц A и B безопасно, так как они не изменяются

## 6. Проверка корректности

**Сравнение с SEQ:** результаты OpenMP-версии побитово совпадают с последовательной версией для всех тестовых случаев.

**Набор тестов:** 12 функциональных тестов для матриц размером от 1x1 до 100x100.

**Характерные случаи:**

- n = 1: один блок, один поток выполняет всю работу
- n = 20: q = 4, работа распределяется между потоками равномерно
- n = 400: q = 20, достаточно работы для эффективной загрузки всех потоков

Все тесты проходят успешно, что подтверждает корректность параллельной реализации.


где n - желаемое количество потоков (1, 3, 5, 7).

**Комментарий о масштабируемости:**

OpenMP показывает стабильное ускорение около 5.3-5.6x независимо от значения PPC_NUM_THREADS.
Это объясняется тем, что OpenMP-runtime использует системный пул потоков и может задействовать все
доступные ядра процессора даже при PPC_NUM_THREADS=1. Эффективность выше 100% при малом числе
потоков связана с тем, что runtime фактически использует больше ядер, чем указано в переменной
окружения.

**Узкие места:**

- Барьер синхронизации между шагами Фокса ограничивает масштабируемость
- При q = 20 и 7 потоках возникает небольшой дисбаланс нагрузки
- Накладные расходы на создание параллельных секций становятся заметными при малых размерах
  задачи

## 9. Выводы

OpenMP обеспечивает простую и эффективную параллелизацию алгоритма Фокса. Достигнуто ускорение
около 5.3-5.6x на матрице 400x400, что близко к теоретическому максимуму для данного процессора с
учетом накладных расходов на синхронизацию. Технология хорошо подходит для задач с регулярной
структурой параллелизма и равномерной нагрузкой. Ограничением является необходимость барьера между
шагами алгоритма, что не позволяет достичь линейного ускорения при большом числе потоков.
