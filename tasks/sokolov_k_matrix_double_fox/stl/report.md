## 1. Контекст

STL-версия использует стандартную библиотеку потоков C++ для ручного
управления параллелизмом. В отличие от OpenMP и TBB, здесь требуется явное
создание потоков, разбиение данных и синхронизация через join.

## 2. Постановка задачи

Задача остается той же: умножение квадратных матриц A и B размером n x n методом
Фокса. Подробная постановка приведена в seq/report.md. Последовательная версия служит
эталоном для проверки корректности.

## 3. Базовый алгоритм

Алгоритм Фокса выполняет q шагов блочного умножения. На каждом шаге для каждой
блочной строки выполняется умножение соответствующих блоков матриц A и B.

## 4. Схема распараллеливания

### 4.1. Разбиение данных

Блочные строки статически распределяются между потоками. Каждый поток получает
непрерывный диапазон строк [row_begin, row_end).

**Формула разбиения:**

```cpp
int num_threads = std::min(PPC_NUM_THREADS, q);
int rows_per = q / num_threads;
int extra = q % num_threads;

for (int idx = 0; idx < num_threads; ++idx) {
    int row_begin = idx * rows_per + std::min(idx, extra);
    int row_count = rows_per + (idx < extra ? 1 : 0);
    int row_end = row_begin + row_count;
    // создание потока для диапазона [row_begin, row_end)
}
```

**Пример:** при q = 20 и num_threads = 3:

- Поток 0: строки [0, 7)
- Поток 1: строки [7, 14)
- Поток 2: строки [14, 20)

### 4.2. Рабочая функция потока

Каждый поток выполняет функцию FoxWorker, которая обрабатывает все q шагов Фокса
для своего диапазона строк:

```cpp
void FoxWorker(const std::vector<double>& a,
               const std::vector<double>& b,
               std::vector<double>& c,
               int bs, int q,
               int row_begin, int row_end) {
    int bsq = bs * bs;
    for (int step = 0; step < q; step++) {
        for (int i = row_begin; i < row_end; i++) {
            int k = (i + step) % q;
            for (int j = 0; j < q; j++) {
                MultiplyBlocks(a, ((i * q) + k) * bsq,
                             b, ((k * q) + j) * bsq,
                             c, ((i * q) + j) * bsq, bs);
            }
        }
    }
}
```

### 4.3. Локальные результаты

Локальные буферы не требуются, так как каждый поток записывает результат напрямую в свои
блоки матрицы C. Блоки C\[i\]\[j\], где i принадлежит диапазону потока, не пересекаются
между потоками.

### 4.4. Создание и завершение потоков

```cpp
std::vector<std::thread> threads;
threads.reserve(num_threads);

for (int idx = 0; idx < num_threads; ++idx) {
    threads.emplace_back(FoxWorker, std::cref(a), std::cref(b),
                        std::ref(c), bs, q, row_begin, row_end);
}

for (auto& th : threads) {
    th.join();
}
```

**Важно:** join выполняется после создания всех потоков, что обеспечивает истинный
параллелизм. Вызов join внутри цикла создания привел бы к последовательному выполнению.

### 4.5. Синхронизация

**Отсутствие явной синхронизации:** потоки работают с непересекающимися диапазонами
блочных строк, поэтому atomic, mutex и condition_variable не требуются.

**Неявная синхронизация:** join гарантирует, что все потоки завершили работу перед
возвратом из RunImpl.

**Отличие от OpenMP/TBB:** каждый поток самостоятельно проходит все q шагов Фокса для
своих строк. Это устраняет необходимость барьера между шагами, но требует, чтобы все
потоки имели доступ ко всем блокам матриц A и B на всех шагах.

## 6. Проверка корректности

**Сравнение с SEQ:** результаты STL-версии идентичны последовательной версии для всех
тестовых случаев.

**Крайние случаи:**

- n = 1: q = 1, создается один поток, который выполняет всю работу
- n = 400, num_threads = 1: один поток обрабатывает все 20 строк
- n = 400, num_threads = 7: семь потоков делят 20 строк

**Проверка на отсутствие гонок:** каждый поток i записывает только в блоки
C\[i\]\[j\], где i принадлежит его диапазону. Чтение из A и B безопасно, так как эти
матрицы не изменяются. Гонки данных отсутствуют.

Все 12 функциональных тестов проходят успешно.

**Комментарий о накладных расходах create/join:**

При одном потоке накладные расходы на создание и join минимальны, время практически
совпадает с последовательной версией. При увеличении числа потоков до 3 эффективность
составляет около 92%, что говорит о небольших накладных расходах.

При 5 потоках эффективность падает до 85-89% из-за:

- Статического разбиения: при q = 20 и 5 потоках каждый поток получает 4 строки, что
  может приводить к небольшому дисбалансу
- Накладных расходов на создание и уничтожение потоков

При 7 потоках эффективность 75-79%, что связано с:

- Превышением числа физических ядер (6 ядер, 7 потоков)
- Неравномерным распределением: 20 строк на 7 потоков дает 6 потоков по 3 строки и 1
  поток с 2 строками

**Преимущество перед OpenMP/TBB:** отсутствие барьера между шагами Фокса позволяет
каждому потоку работать независимо, что дает преимущество при большом числе потоков.

**Недостаток:** статическое разбиение менее гибко, чем динамическая балансировка в TBB,
что проявляется при 5 потоках.

## 9. Выводы

STL-версия с ручным управлением потоками показывает хорошую масштабируемость и
сопоставимую с TBB производительность. Отсутствие барьеров между шагами Фокса дает
преимущество при большом числе потоков, но статическое разбиение работы уступает
динамической балансировке при нечетном делении.
