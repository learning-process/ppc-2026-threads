## 1. Контекст

Intel Threading Building Blocks предоставляет задачно-ориентированную модель параллелизма с
динамическим планировщиком. TBB-версия алгоритма Фокса использует примитив parallel_for для
распределения работы по блочным строкам между потоками.

## 2. Постановка задачи

Задача остается той же: умножение квадратных матриц A и B размером n x n методом Фокса. Подробная
постановка приведена в seq/report.md. Последовательная версия используется как baseline для оценки
ускорения.

## 3. Базовый алгоритм

Алгоритм Фокса выполняет q шагов блочного умножения матриц. На каждом шаге для каждой блочной
строи выполняется умножение соответствующих блоков с накоплением результата.

## 4. Схема распараллеливания

### 4.1. Выбор примитива

Используется **parallel_for** с blocked_range для распределения итераций цикла по
блочным строкам между рабочими потоками.

### 4.2. Реализация

```cpp
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<int>(0, q, 1),
    [&](const oneapi::tbb::blocked_range<int>& r) {
        for (int i = r.begin(); i != r.end(); ++i) {
            int k = (i + step) % q;
            for (int j = 0; j < q; j++) {
                MultiplyBlocks(a, ((i * q) + k) * bsq,
                             b, ((k * q) + j) * bsq,
                             c, ((i * q) + j) * bsq, bs);
            }
        }
    });
```

### 4.3. Диапазон работы

**blocked_range\<int\>(0, q, 1)** определяет:

- Начало диапазона: 0
- Конец диапазона: q (количество блочных строк)
- Grain size: 1 (минимальный размер подзадачи)

### 4.4. Grain size

Выбран **grain size = 1**, так как:

- Каждая блочная строка требует значительного времени обработки (q умножений блоков размером
  bs x bs)
- При q = 20 и grain size = 1 создается достаточно подзадач для эффективной балансировки нагрузки
- Накладные расходы на создание задач незначительны по сравнению с временем вычислений

### 4.5. Partitioner

Используется **auto_partitioner** по умолчанию. Он автоматически выбирает стратегию разбиения
диапазона на основе характеристик системы и размера задачи. Auto_partitioner обеспечивает хороший
баланс между накладными расходами и равномерностью распределения нагрузки.

### 4.6. Ограничение конкуренции

Число рабочих потоков контролируется через переменную окружения PPC_NUM_THREADS. TBB-runtime
создает пул потоков соответствующего размера и динамически распределяет задачи между ними.

```cpp
if (const char* env_p = std::getenv("PPC_NUM_THREADS")) {
    int num_threads = std::stoi(env_p);
    oneapi::tbb::global_control gc(
        oneapi::tbb::global_control::max_allowed_parallelism,
        num_threads);
    // выполнение параллельных операций
}
```

## 5. Детали реализации

**Файлы:**

- tbb/include/ops_tbb.hpp - объявление класса SokolovKMatrixDoubleFoxTBB
- tbb/src/ops_tbb.cpp - реализация с использованием TBB

**Локальные аккумуляторы:** не требуются, так как каждый поток записывает в свой
уникальный блок матрицы C.

**Объединение результатов:** не требуется, результат формируется напрямую в общей
матрице C без необходимости финальной редукции.

**Работа с памятью:** используются стандартные векторы `std::vector<double>` для
хранения блочных представлений матриц. TBB-аллокаторы не применяются, так как паттерн
доступа к памяти не создает проблем с false sharing.

**Комментарий о balancing и overhead:**

TBB демонстрирует хорошую масштабируемость благодаря динамическому планировщику. При
одном потоке накладные расходы минимальны, время практически совпадает с
последовательной версией. При увеличении числа потоков до 5 эффективность остается
высокой (93-96%), что говорит о хорошей балансировке нагрузки.

При 7 потоках эффективность падает до 77-78% из-за:

- Ограниченного числа физических ядер процессора (6 ядер)
- Накладных расходов на переключение контекста
- Барьера синхронизации между шагами Фокса

Динамический планировщик TBB эффективно справляется с небольшим дисбалансом нагрузки,
возникающим при нечетном делении 20 блочных строк на число потоков.

## 9. Выводы

TBB обеспечивает эффективное распараллеливание алгоритма Фокса с хорошей масштабируемостью
до 5 потоков. Динамический планировщик автоматически балансирует нагрузку между потоками,
что особенно полезно при нечетном делении работы. Накладные расходы TBB-runtime минимальны
при одном потоке и остаются приемлемыми при увеличении числа потоков. Технология удобна
для задач с регулярной структурой параллелизма, где не требуется ручное управление
потоками и синхронизацией.
