## 1. Контекст

Гибридная версия сочетает распределенную память через MPI и общую память через OpenMP.
Это позволяет эффективно использовать многоузловые кластеры, где MPI распределяет
работу между узлами, а OpenMP распараллеливает вычисления внутри каждого узла.

## 2. Постановка задачи

Задача остается той же: умножение квадратных матриц A и B размером n x n методом
Фокса. Подробная постановка приведена в seq/report.md и корневом report.md.
Последовательная версия служит baseline для оценки ускорения.

## 3. Базовый алгоритм

Алгоритм Фокса выполняет q шагов блочного умножения матриц. На каждом шаге для каждой
блочной строки выполняется умножение соответствующих блоков.

## 4. Межпроцессная схема

### 4.1. Роли rank-ов

- **Rank 0:** главный процесс, выполняет подготовку данных, рассылку параметров и
  матриц, сбор результатов и финальный бродкаст
- **Rank 1..n-1:** рабочие процессы, получают данные, вычисляют свою часть результата,
  отправляют результат обратно в rank 0

### 4.2. Распределение данных

**Инициализация:**

```cpp
int rank = -1, num_procs = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &num_procs);
```

**Бродкаст параметров:** rank 0 рассылает n, block_size, q, total всем процессам через
MPI_Bcast.

**Бродкаст матриц:** rank 0 рассылает полные блочные массивы A и B всем процессам через
MPI_Bcast. Это упрощает реализацию и допустимо для матриц размером 400x400, так как объем
данных невелик.

**Распределение блочных строк:**

```cpp
int rows_per = q / num_procs;
int leftover = q % num_procs;
int my_row_start = rank * rows_per + std::min(rank, leftover);
int my_row_count = rows_per + (rank < leftover ? 1 : 0);
```

**Пример:** при q = 20 и num_procs = 3:

- Rank 0: строки [0, 7)
- Rank 1: строки [7, 14)
- Rank 2: строки [14, 20)

### 4.3. MPI-вызовы

**MPI_Bcast:** используется для рассылки параметров, матриц A, B и финального результата
C.

```cpp
MPI_Bcast(&n, 1, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(blocks_a.data(), total, MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

**MPI_Send/MPI_Recv:** используются для сбора результатов от рабочих процессов в rank 0.

```cpp
// в rank != 0:
MPI_Send(blocks_c.data(), total, MPI_DOUBLE, 0, 0, MPI_COMM_WORLD);

// в rank 0:
MPI_Recv(blocks_c.data(), total, MPI_DOUBLE, src, 0, MPI_COMM_WORLD, &status);
```

**MPI_Barrier:** используется для синхронизации всех процессов перед завершением.

```cpp
MPI_Barrier(MPI_COMM_WORLD);
```

## 5. Внутрипроцессная схема

### 5.1. Технология внутри rank-а

Внутри каждого MPI-процесса используется **OpenMP** для распараллеливания цикла по
блочным строкам.

### 5.2. Разделение работы

Каждый процесс выполняет функцию FoxStepMpiOmp для своего диапазона строк
[my_row_start, my_row_start + my_row_count):

```cpp
void FoxStepMpiOmp(const std::vector<double>& a,
                   const std::vector<double>& b,
                   std::vector<double>& c,
                   int bs, int q, int step,
                   int row_start, int row_end) {
    int bsq = bs * bs;
#pragma omp parallel for default(none) \
    shared(a, b, c, bs, q, step, bsq, row_start, row_end) \
    schedule(static)
    for (int i = row_start; i < row_end; i++) {
        int k = (i + step) % q;
        for (int j = 0; j < q; j++) {
            MultiplyBlocksLocal(a, ((i * q) + k) * bsq,
                              b, ((k * q) + j) * bsq,
                              c, ((i * q) + j) * bsq, bs);
        }
    }
}
```

### 5.3. Объединение локальных результатов

Локальные результаты не требуют объединения внутри процесса, так как каждый OpenMP-поток
записывает в свой уникальный блок матрицы C.

## 6. Детали реализации

**Файлы:**

- all/include/ops_all.hpp - объявление класса SokolovKMatrixDoubleFoxALL
- all/src/ops_all.cpp - гибридная реализация MPI + OpenMP

**MPI_Barrier:** вызывается в конце RunImpl для гарантии, что все процессы завершили
вычисления перед возвратом.

**Сбор результатов (GatherResults):**

```cpp
if (rank != 0) {
    MPI_Send(blocks_c.data(), total, MPI_DOUBLE, 0, 0, MPI_COMM_WORLD);
} else {
    for (int src = 1; src < num_procs; src++) {
        MPI_Recv(blocks_c.data(), total, MPI_DOUBLE, src, 0,
                 MPI_COMM_WORLD, &status);
    }
}
```

**Финальный бродкаст:** rank 0 рассылает собранный результат всем процессам, чтобы каждый
процесс мог выполнить PostProcessing.

**Потенциальные узкие места:**

- Бродкаст полных матриц A и B создает коммуникационные накладные расходы
- Сбор результатов через Send/Recv последовательный, что может стать узким местом при
  большом числе процессов
- Барьер синхронизации между шагами Фокса ограничивает масштабируемость

## 7. Проверка корректности

**Сравнение с SEQ:** результаты гибридной версии идентичны последовательной версии для
всех тестовых случаев.

**Сравнение с чистыми backend-ами:** результаты совпадают с OMP, TBB и STL версиями.

**Тесты на малых и средних размерах:** протестированы размеры от 1x1 до 100x100.

**Проверка согласованности между rank-ами:** после финального бродкаста все процессы
имеют идентичную матрицу C, что проверяется в PostProcessing.

Все 12 функциональных тестов проходят успешно для различных конфигураций
MPI-процессов.

**Комментарий о цене коммуникации и синхронизации:**

При n=1 MPI-процессе достигается наилучший результат среди всех технологий (16.74x в
task_run). Это объясняется тем, что весь внутренний OpenMP-параллелизм доступен без
коммуникационных накладных расходов MPI.

При увеличении числа MPI-процессов на одной машине они начинают конкурировать за ядра
процессора:

- При n=3: эффективность все еще высокая (517%), так как 3 процесса могут разместиться
  на 6 ядрах
- При n=5: эффективность падает до 167%, начинается конкуренция за ресурсы
- При n=7: эффективность 66%, сильная конкуренция за ядра и накладные расходы на
  переключение контекста

**Коммуникационные накладные расходы:**

- Бродкаст матриц A и B размером 400x400 занимает значительное время
- Последовательный сбор результатов через Send/Recv создает узкое место
- Барьер синхронизации между шагами Фокса ограничивает параллелизм

**Оптимальная конфигурация:** для одноузловой системы оптимально использовать n=1
MPI-процесс с полным OpenMP-параллелизмом. Гибридная схема выигрывает на многоузловых
кластерах, где каждый узел имеет свою локальную память.

накладных расходов.
