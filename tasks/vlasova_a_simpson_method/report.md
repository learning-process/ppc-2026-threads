## 1. Введение

Задача численного интегрирования функций многих переменных является фундаментальной для
широкого круга научных и инженерных приложений, включая вычислительную физику, финансовое моделирование 
и машинное обучение. В то время как аналитическое вычисление многомерных интегралов возможно лишь для ограниченного 
класса функций, численные методы предоставляют универсальный инструмент для получения приближенного значения.

В данной работе рассматривается реализация составного метода Симпсона, обобщенного на многомерный случай. 
Целью является создание корректной и вычислительно эффективной реализации для интегрирования функций на 
гиперпрямоугольной области с использованием различных технологий параллельного программирования: OpenMP, 
Intel TBB, стандартной библиотеки STL (std::thread) и гибридной MPI+TBB версии. Последовательная версия 
служит эталоном для сравнения корректности и производительности.

Данная задача хорошо подходит для сравнения разных моделей параллелизма, поскольку:
- Вычислительная нагрузка легко разделяется на независимые подзадачи (узлы сетки).
- Отсутствует необходимость в сложной синхронизации между потоками.
- Объем вычислений легко масштабируется изменением количества шагов разбиения `n`.

## 2. Единая постановка задачи

Требуется вычислить приближенное значение многомерного определенного интеграла 
от заданной функции на гиперпрямоугольной области.

**Входные данные:**
- Вектор `a` типа `std::vector<double>` — нижние пределы интегрирования по каждому измерению.
- Вектор `b` типа `std::vector<double>` — верхние пределы интегрирования по каждому измерению.
- Вектор `n` типа `std::vector<int>` — количество подинтервалов разбиения 
по каждому измерению (каждое значение должно быть четным и положительным).
- Объект `func` типа `std::function<double(const std::vector<double>&)>` — подынтегральная функция.

**Выходные данные:**
- Значение типа `double` — приближенное значение интеграла.

**Ограничения:**
- Размерности векторов `a`, `b` и `n` должны быть одинаковыми и больше нуля.
- Для каждого измерения `i` должно выполняться `a[i] < b[i]`.
- Каждый элемент `n[i]` должен быть четным положительным числом.

**Критерий корректности:**
Результат каждой параллельной реализации считается корректным, если абсолютная погрешность
относительно последовательной версии не превышает `1e-3` на всех функциональных тестах.

## 3. Единая методика эксперимента

**Аппаратное обеспечение:**
- Процессор: Intel(R) Core(TM) Ultra 9 285H (2.90 GHz)
- Оперативная память: 32 GB
- ОС: Windows 11

**Инструментарий:**
- Компилятор: clang version 21.1.6
- Тип сборки: Release
- Библиотеки: OpenMP, Intel TBB, MPI, STL

**Переменные окружения:**
- `PPC_NUM_THREADS` — задает количество потоков для OpenMP, TBB и STL.
- `OMP_NUM_THREADS` — экспортируется runner'ом из `PPC_NUM_THREADS`.
- `PPC_NUM_PROC` — задает количество MPI-процессов.

**Тестовые данные:**
Функциональные тесты используют 22 предопределенных набора параметров, включающих:
- Размерности: 1D, 2D, 3D.
- Типы функций: константные, линейные, квадратичные, кубические, функции-произведения.
- Сетки разбиения: от минимальных (`n = {2}`) до подробных (`n = {50, 50, 50}`).
- Симметричные интервалы интегрирования для проверки точности.

Тесты производительности используют трехмерную задачу интегрирования 
функции Гаусса в кубе `[-2, 2]^3` с разбиением `n = {200, 200, 200}`, 
что соответствует более чем 8 миллионам узлов сетки.

**Методика измерения:**
- Для каждого backend-а и конфигурации выполнено по 2 замера в режимах `pipeline` и `task_run`.
- Время измеряется в секундах.
- Ускорение вычисляется как `Speedup = Time_seq / Time_parallel`.
- Эффективность вычисляется как `Efficiency = Speedup / Workers` 
(где Workers — количество потоков для OMP/TBB/STL, количество MPI-процессов для ALL).

## 6. Интерпретация различий

### 6.1 SEQ (базовый эталон)
Последовательная версия выполняет вычисление 8-миллионного трехмерного интеграла примерно за 0.42 секунды. 
Этот результат служит знаменателем для вычисления ускорения всех параллельных версий. 
Основное узкое место — экспоненциальный рост вычислительной сложности с увеличением размерности задачи, 
что делает последовательную версию непригодной для задач высокой размерности.

### 6.2 OMP (OpenMP)
OpenMP показал наилучший результат среди всех параллельных версий: ускорение **6.74x** на 20 потоках.
Эффективность 33-34% обусловлена:
- Накладными расходами на создание потоков.
- Преобразованием глобального индекса в многомерный (в отличие от эффективного метода `Nextindex()` в SEQ).
- Статическим планированием `schedule(static)`, которое минимизирует управляющие накладные расходы.

**Вывод:** OpenMP является наиболее эффективной технологией для данной задачи 
благодаря низким накладным расходам и эффективной редукционной операции.

### 6.3 TBB (Intel TBB)
TBB показал результат хуже последовательной версии (ускорение 0.79x). Причины:
- Высокие накладные расходы библиотеки TBB на создание задач и управление пулом потоков.
- Динамическое планирование с кражей задач (work-stealing) добавляет дополнительные накладные расходы.
- Для данной задачи объем вычислений на итерацию слишком мал, чтобы окупить накладные расходы TBB.

**Вывод:** TBB не подходит для задач с очень мелкими итерациями. 
Библиотека показывает себя лучше на задачах с более крупными вычислительными блоками.

### 6.4 STL (std::thread)
STL также показал результат хуже последовательной версии (ускорение 0.82x). Причины:
- Ручное создание и управление 20 потоками требует значительного времени.
- Отсутствие встроенной редукции — требуется отдельный массив `partial_sums` и финальное суммирование.
- `join()` всех потоков создает дополнительную синхронизационную задержку.

**Вывод:** Ручное управление потоками через `std::thread` редко дает 
выигрыш на задачах с мелкими итерациями из-за высоких накладных расходов. 
OpenMP предоставляет более высокоуровневую и оптимизированную абстракцию.

### 6.5 ALL (MPI + TBB)
Гибридная версия показала **отличную масштабируемость**:
- На 4 MPI-процессах: ускорение 3.19x, эффективность 80%.
- На 8 MPI-процессах: ускорение 5.08x, эффективность 64%.
- На 12 MPI-процессах: ускорение 6.41x, эффективность 53%.

**Анализ:**
- Эффективность снижается с ростом числа процессов из-за увеличения 
доли коммуникационных затрат (`MPI_Allreduce`).
- При 12 процессах ALL достигает ускорения, близкого к OpenMP на 20 потоках (6.41x против 6.74x).
- Преимущество ALL: возможность распределения данных между несколькими узлами, 
что позволяет решать задачи, не помещающиеся в память одного узла.

**Вывод:** Гибридная схема наиболее универсальна, но имеет смысл при достаточно большом размере задачи, 
когда коммуникационные затраты окупаются выигрышем от параллелизации.

## 8. Заключение

В рамках данной работы были успешно реализованы и протестированы пять версий 
алгоритма вычисления многомерных интегралов составным методом Симпсона с использованием 
различных технологий параллельного программирования.

**Основные выводы:**

1. **OpenMP** показал наилучшую производительность (ускорение 6.74x) 
благодаря низким накладным расходам и эффективной редукционной операции. 
Рекомендуется для данной задачи при работе на одном узле.

2. **TBB и STL** не дали выигрыша в производительности из-за высоких накладных расходов 
при малом объеме вычислений на итерацию. Они не рекомендуются для данного класса задач.

3. **Гибридная версия ALL (MPI+TBB)** показала отличную масштабируемость 
(ускорение 6.41x на 12 процессах) и является единственным решением, позволяющим масштабироваться
на несколько вычислительных узлов. Рекомендуется для очень больших задач,
не помещающихся в память одного узла.

**Ограничения сравнения:**
- Замеры выполнены на одном конкретном аппаратном обеспечении.
- Результаты могут отличаться на других процессорах с другой архитектурой.
- Для задач с большим объемом вычислений на итерацию выводы о TBB и STL могут измениться.

**Возможные улучшения:**
- Использование `schedule(dynamic)` в OpenMP для лучшей балансировки при неравномерной нагрузке.
- Увеличение `grainsize` в TBB для уменьшения накладных расходов.
- Использование `MPI_Scatter`/`MPI_Gather` вместо ручного распределения диапазонов в ALL версии.
