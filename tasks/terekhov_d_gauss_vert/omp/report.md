## 1. Контекст

Данная реализация переносит вычислительно трудоёмкую операцию свёртки изображения на
параллельную основу с использованием технологии OpenMP. Основная цель — ускорение обработки
за счёт распределения вертикальных полос изображения между потоками процессора.

## 2. Постановка задачи

Краткая постановка задачи совпадает с последовательной версией. Используется изображение размером
10 млн пикселей. В качестве базового времени (baseline) для расчёта ускорения принято
время работы SEQ-версии: **0.0809 сек**.

## 3. Базовый алгоритм

Алгоритм повторяет последовательную версию: подготовка дополненного изображения (padding) и
последующая свёртка каждого пикселя с ядром Гаусса 3x3. Основное изменение касается этапа
выполнения свёртки, который теперь выполняется параллельно.

## 4. Схема распараллеливания

Для распараллеливания используется директива `#pragma omp parallel for`.

- **Область распараллеливания**: Цикл в функции `SolveOMP`, итерирующийся по
количеству вертикальных полос (`num_bands = 4`).
- **Атрибут `default(none)`**: Используется для обеспечения явного контроля над
всеми переменными в параллельной области, что соответствует требованиям безопасности и методическим рекомендациям.
- **Атрибут `shared`**: Входное изображение `padded_image` и выходной объект `output`
являются общими. Потоки не конфликтуют при записи, так как каждый работает со своей
вертикальной полосой (непересекающиеся диапазоны индекса `col`).
- **Планировщик `schedule(static)`**: Выбран ввиду того, что вычислительная сложность
обработки каждой полосы изображения идентична. Это минимизирует накладные расходы на балансировку нагрузки.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`
Основные компоненты:

- `PreProcessingImpl`: Подготовка данных выполняется последовательно (формирование зеркальной рамки).
- `RunImpl`: Вызывает `SolveOMP`, где сосредоточена параллельная логика.

Листинг ключевого фрагмента:

```cpp
#pragma omp parallel for default(none) \
    shared(output, padded_image, padded_width, width, height, band_width, num_bands) schedule(static)
for (int band = 0; band < num_bands; ++band) {
  ProcessBand(output, padded_image, padded_width, width, height, band, band_width, num_bands);
}
```

## 6. Проверка корректности

Результаты OMP-версии сравниваются с SEQ-версией в рамках функциональных тестов `GaussFilterTestsOMP`.
Проверяется детерминированность результата и отсутствие гонок данных (data races) при различных
конфигурациях потоков.

## 7. Экспериментальная среда

- CPU: Intel Core i5-10400F (6 ядер, 12 потоков).
- RAM: 32 GB DDR4.
- OS: Windows 10 (Docker env).
- Переменные: `PPC_NUM_THREADS` задаёт количество потоков.
- Команда запуска: mpirun -n 1 ./build/bin/ppc_perf_tests --gtest_filter=*OMP*

## 9. Выводы

Применение OpenMP позволило получить выигрыш в производительности относительно чисто последовательного
кода. Однако при увеличении числа потоков с 2 до 4 наблюдается снижение эффективности. Основной причиной
является фиксированное количество полос в реализации (`num_bands = 4`), что ограничивает потенциал
масштабируемости, а также доминирование накладных расходов на создание параллельной области над полезной
нагрузкой при малом объёме вычислений на один поток.
