## 1. Контекст

Реализация на базе библиотеки Intel oneTBB (Threading Building Blocks) использует высокоуровневые
абстракции для распределения вычислений. В отличие от OpenMP, TBB ориентирована на задачи (tasks),
а не на потоки, что позволяет библиотеке динамически балансировать нагрузку.

## 2. Постановка задачи

Задача фильтрации изображения 10 млн пикселей. Результаты TBB-версии сравниваются с
последовательным эталоном (SEQ), время выполнения которого составляет **0.0809 сек**.

## 3. Базовый алгоритм

Алгоритм свёртки идентичен последовательной версии. Изображение дополняется зеркальной рамкой
(padding), после чего к нему применяется маска Гаусса 3x3. Распараллеливание осуществляется
на уровне обработки вертикальных полос изображения.

## 4. Схема распараллеливания

- **Примитив**: `oneapi::tbb::parallel_for`.
- **Диапазон**: `oneapi::tbb::blocked_range<int>(0, num_bands)`.
Диапазон итераций охватывает 4 вертикальные полосы изображения.
- **Grainsize**: Используется значение по умолчанию. Поскольку общее
количество задач мало (всего 4 полосы), `grainsize` не оказывает существенного влияния на декомпозицию.
- **Диспетчеризация**: TBB runtime автоматически распределяет задачи по потокам из внутреннего пула.
- **Синхронизация**: Не требуется, так как каждый вызов `ProcessBand` работает
с изолированным участком памяти выходного изображения.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`
Основные компоненты:

- `PreProcessingImpl`: Последовательная подготовка `padded_image_`.
- `RunImpl`: Запуск `SolveTBB`, содержащей параллельную структуру.
- `ValidationImpl` и `PostProcessingImpl`: Проверка целостности данных до и после вычислений.

Листинг ключевого фрагмента:

```cpp
oneapi::tbb::parallel_for(oneapi::tbb::blocked_range<int>(0, num_bands),
                          [&](const oneapi::tbb::blocked_range<int> &range) {
  for (int band = range.begin(); band != range.end(); ++band) {
    ProcessBand(output, padded_image, padded_width, width, height, band, band_width, num_bands);
  }
});
```

## 6. Проверка корректности

Корректность подтверждается функциональными тестами `GaussFilterTestsTBB`. Проверяется
идентичность выходных данных результатам SEQ-версии для различных размеров входных данных.

## 7. Экспериментальная среда

- CPU: Intel Core i5-10400F (6 ядер, 12 потоков).
- RAM: 32 GB DDR4.
- OS: Windows 10 (Docker env).
- Переменные: `PPC_NUM_THREADS` задаёт количество потоков.
- Команда запуска: mpirun -n 1 ./build/bin/ppc_perf_tests --gtest_filter=*TBB*

## 9. Выводы

Результаты показывают, что версия TBB в текущей реализации демонстрирует значительные накладные
расходы (overhead) и не достигает ускорения относительно SEQ. Основным сдерживающим фактором является
декомпозиция всего на 4 задачи (`num_bands = 4`), что избыточно для механизмов планирования TBB при
такой малой вычислительной нагрузке. Для данной задачи и объёма данных (10 Мп) накладные расходы на
создание и диспетчеризацию задач в TBB превышают выигрыш от параллельного исполнения.
