## 1. Введение

Решается задача блочной фильтрации изображения ядром Гаусса 3×3.
Фильтрация Гаусса применяется для размытия изображения, подавления шума,
уменьшения детализации. Блочная обработка позволяет обрабатывать большие
изображения и естественно распараллеливается.

Сравниваются четыре модели параллелизма: OpenMP, TBB, STL (std::thread),
а также гибридная MPI+threads (ALL). Последовательная версия (SEQ)
служит эталоном корректности и производительности.

## 2. Единая постановка задачи

**Входные данные:**

- width (int) — ширина изображения в пикселях
- height (int) — высота изображения в пикселях
- channels (int) — число цветовых каналов (1 — серое, 3 — RGB)
- block_size (int) — размер блока (фиксирован 64)
- image_data (vector<uint8_t>) — пиксели в формате row-major, каналы interleaved

**Выходные данные:**

- vector<uint8_t> — отфильтрованное изображение той же размерности

**Ядро Гаусса 3×3 (нормированное):**

    1/16  2/16  1/16
    2/16  4/16  2/16
    1/16  2/16  1/16

**Ограничения:**

- Вектор входных данных не должен быть пустым
- Размер изображения не менее 1×1
- Границы изображения обрабатываются симметричным отражением (clamp)

**Крайние случаи:**

- Изображение 1×1 — фильтр не применяется, выход равен входу
- Блоки у правой и нижней границы меньше 64×64 — обрабатываются корректно

## 3. Единая методика эксперимента

**Окружение:**

- CPU: Intel Core i5-11400H
- RAM: 16 ГБ
- OS: Windows 10
- Компилятор: MSVC 2022 (Release, x64)
- Размер задачи: 2048×2048×3 пикселя

**Переменные окружения:**

- OpenMP: OMP_NUM_THREADS
- TBB/STL/ALL: PPC_NUM_THREADS
- MPI: количество процессов через -n

**Способ получения данных:**
Тесты производительности запускались через ppc_perf_tests.exe
с фильтром по соответствующему backend-у. Использовался режим task_run.

**Формулы:**

- Speedup = T_seq / T_par
- Efficiency = Speedup / (число потоков)

**Протокол измерений:**
Каждое измерение выполнялось после прогрева (один пробный запуск,
не учитываемый в результатах). Для каждой конфигурации проведено
5 повторных запусков, в таблице приведена медиана. Базовое время SEQ (медиана) = 0.196 с.

## 6. Интерпретация различий

**SEQ:**
Базовый уровень — 0.196 с. Отсутствие накладных расходов.

**OpenMP:**

- При 1 потоке в 3 раза медленнее SEQ из-за оверхеда на создание
  параллельных областей.
- Максимальное ускорение — 2.09× на 8 потоках.
- Эффективность падает с ростом числа потоков.

**TBB:**

- Ускорение 4.13× на 8 потоках — лучше, чем OpenMP.
- Work stealing помогает балансировать нагрузку.

**STL:**

- Лучшее ускорение — 4.62× на 8 потоках.
- При 1 потоке почти равен SEQ (0.220 vs 0.214).
- Почти линейное ускорение до 4 потоков.

**ALL (MPI+threads):**

- Худшие результаты из-за оверхеда MPI.
- Лучшая конфигурация — 4×2 (0.140 с, speedup 1.40×).
- MPI не нужен для этой задачи на одной машине.

## 7. Репродуцируемость

**Команды сборки:**

    cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
    cmake --build build --config Release --parallel

**Команды запуска тестов (пример для STL 8 потоков):**
    $env:PPC_NUM_THREADS=8; .\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_moskaev_v_lin_filt_block_gauss_3_stl_enabled"
**Команды для ALL (4 процесса × 2 потока):**
    $env:PPC_NUM_THREADS=2; mpiexec -n 4 .\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_moskaev_v_lin_filt_block_gauss_3_all_enabled"

## 8. Заключение

**Лучшая технология:** STL с ускорением 4.62× на 8 потоках.

**Второй:** TBB с ускорением 4.13×.

**OpenMP** отстаёт (2.09×) из-за оверхеда.

**ALL (MPI+threads)** неэффективен на одной машине.

**Рекомендация:** для данной задачи на одном компьютере использовать STL или TBB.
