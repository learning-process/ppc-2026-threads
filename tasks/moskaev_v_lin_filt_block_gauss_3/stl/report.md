## 1. Контекст

STL-реализация блочной фильтрации изображения ядром Гаусса 3×3
с использованием std::thread. Потоки создаются вручную, каждый поток
обрабатывает свою часть блоков. Цель — оценить эффективность
низкоуровневого управления потоками.

## 2. Постановка задачи

Полностью совпадает с последовательной версией (см. seq/report.md).
Вход — изображение, выход — отфильтрованное изображение.
Корректность проверяется сравнением с SEQ.

## 3. Базовый алгоритм

Тот же блочный алгоритм, что в SEQ: разбиение на блоки 64×64,
копирование с отступами, свёртка с ядром Гаусса, запись результата.

## 4. Схема распараллеливания

**Как данные делятся на диапазоны:**
Общее количество блоков (total_blocks = blocks_x * blocks_y)
делится между потоками непрерывными диапазонами.

**Фрагмент кода разбиения:**

```cpp
int total_blocks = blocks_y * blocks_x;
for (int tid = 0; tid < num_threads; ++tid) {
    int start = (tid * total_blocks) / num_threads;
    int end = ((tid + 1) * total_blocks) / num_threads;
    threads[tid] = std::thread([&, start, end] {
        for (int idx = start; idx < end; ++idx) {
            int by = idx / blocks_x;
            int bx = idx % blocks_x;
            // обработка блока (bx, by)
        }
    });
}
for (auto &th : threads) th.join();
```

**Что делает каждый поток:**
Каждый поток обрабатывает свой диапазон блоков:
для каждого блока выполняет копирование с отступами, фильтрацию, запись результата.

**Где хранятся локальные результаты:**
Каждый поток пишет напрямую в общий выходной вектор, но в разные позиции (разные блоки).
Гонок нет, так как блоки не пересекаются.

**Как и когда выполняется join:**
После создания всех потоков вызывается join() для каждого, чтобы дождаться завершения всех.

**Какая синхронизация используется и почему:**
Синхронизация не требуется, так как:

- Каждый поток обрабатывает свой набор блоков
- Блоки не перекрываются
- Запись в выходной вектор происходит в разные позиции

## 5. Детали реализации

**Файлы:** stl/include/ops_stl.hpp, stl/src/ops_stl.cpp

**Ключевые особенности:**

- Количество потоков определяется через std::thread::hardware_concurrency()
- При создании потоков используется захват переменных по ссылке [&]
- Функция ApplyGaussianFilterToBlock не распараллелена внутри, только внешний цикл по блокам

**Фрагмент кода создания потоков:**

```cpp
const int num_threads = ppc::util::GetNumThreads();
std::vector<std::thread> threads(num_threads);
for (int tid = 0; tid < num_threads; ++tid) {
    int block_start = (tid * total_blocks) / num_threads;
    int block_end = ((tid + 1) * total_blocks) / num_threads;
    threads[tid] = std::thread([&, block_start, block_end]() {
        for (int linear_idx = block_start; linear_idx < block_end; ++linear_idx) {
            // обработка блока
        }
    });
}
for (auto &th : threads) th.join();
```

**Особенности:**

- Статическое распределение блоков (не адаптивное)
- Нет work stealing
- Потоки создаются один раз на всё изображение

## 6. Проверка корректности

Сравнение с SEQ. Функциональные тесты (из tests/functional/main.cpp):

- Тест 1: 2×2 серое, вход [100,150,200,250] → выход [138,163,188,213]
- Тест 2: 3×3 серое, вход [1,2,3,4,5,6,7,8,9] → выход [2,3,4,4,5,6,7,7,8]
- Тест 3: 2×2 RGB, вход 12 чисел → выход 12 чисел
- Тест 4: 1×1 серое, вход [255] → выход [255]

Все тесты пройдены для 1, 2, 4, 8 потоков.

**Проверка на отсутствие гонок:**
Гонки данных отсутствуют, так как:

- Каждый поток обрабатывает свой диапазон блоков (непересекающиеся индексы)
- Запись в выходной вектор происходит в разные позиции
- При создании потоков используется захват по ссылке, но все общие данные только читаются,
кроме выходного вектора, куда каждый пишет в свои уникальные позиции.
Дополнительная синхронизация (mutex, atomic) не требуется.

## 8. Результаты

| Потоков | Время, с | Speedup | Efficiency |
|---------|----------|---------|------------|
| 1       | 0.220    | 0.97    | 97%        |
| 2       | 0.159    | 1.35    | 67%        |
| 4       | 0.074    | 2.89    | 72%        |
| 8       | 0.044    | 4.86    | 61%        |

**Комментарий о масштабируемости и узких местах:**

- STL показывает лучшее ускорение среди всех технологий — 4.86× на 8 потоках.
- При 1 потоке STL почти не уступает SEQ (0.220 против 0.214) — оверхед минимален.
- Ускорение почти линейное до 4 потоков (2.89× на 4 потоках).
- На 8 потоках эффективность падает до 61% из-за насыщения ядер (6 физических ядер).
- Отсутствие work stealing не является проблемой, так как блоки одинакового размера.

## 9. Выводы

STL даёт максимальное ускорение 4.86× на 8 потоках.
Это лучший результат среди всех реализаций.
Простота реализации и эффективность делают std::thread хорошим выбором для данной задачи.
