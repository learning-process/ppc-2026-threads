## 1. Контекст

В данной версии задачи распараллеливание реализуется вручную средствами `std::thread`.

Цель STL-версии — показать низкоуровневую thread-based модель параллелизма путем ручного создания потоков,
явного разбиения данных, управленияем жизненным циклом потоков.

Последовательная версия из `seq/report.md`используется как baseline для оценки ускорения.

## 2. Постановка задачи

Требуется распределить блоки массива между потоками средствами стандартной библиотеки,
обеспечив корректность и отсутствие гонок данных.
Входные, выходные данные и корректность результата аналогичны последовательной версии.

## 3. Базовый алгоритм

Структура та же, что в других параллельных версиях:

1. Вычисление числа блоков и их границ.
2. Создание пула потоков, каждый сортирует свой блок.
3. Ожидание завершения всех потоков (`.join()`).
4. Последовательное слияние блоков в главном потоке.

## 4. Схема распараллеливания

### **Разбиение данных**

Массив разбивается на диапазоны одинакового размера: `block_size = (n + num_threads - 1) / num_threads`
Для каждого потока вычисляются:

`start = tid * block_size`  и  
`end = std::min(start + block_size, n)`.
Каждый поток работает только со своим диапазоном элементов.

### **Работа потоков**

Создание потоков: `std::vector<std::thread> threads(num_threads)`

Каждый поток:

1. копирует собственный диапазон во временный буфер;
2. выполняет локальный Shell Sort;
3. сохраняет результат в: `blocks[tid]`

### **Локальные результаты**

Для хранения результатов используется: `std::vector<std::vector<int>> blocks`
Каждый поток записывает данные только в собственный элемент массива,
это позволяет избежать совместной записи в одну область памяти.

### **Синхронизация**

Дополнительная синхронизация (mutex, atomic, condition_variable) не используется,
потому что  потоки работают с независимыми диапазонами, shared-записи отсутствуют,
каждый поток изменяет только свой `blocks[tid]`. Гонки данных отсутствуют.

### **Join**

После создания всех потоков выполняется:

```bash
for (auto &th : threads) {
  th.join();
}
```

join() вызывается только после завершения цикла создания потоков.
Если бы join() выполнялся внутри цикла создания,
потоки запускались бы последовательно и параллелизм был бы потерян.

После завершения всех потоков выполняется последовательное слияние.
Слияние остаётся последовательным этапом алгоритма.

## 5. Детали реализации

Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

### **Карта поток → диапазон**

Для потока tid:  `[start, end)`
где:

```bash
start = tid * block_size
end = min(start + block_size, n)
```

### **Объединение результатов**

После join() локальные результаты объединяются в `result + blocks[i]`
через последовательный `std::merge`.

## 6. Проверка корректности

Результаты проверялись сравнением с SEQ на том же наборе тестов.
Повторные запуски с разным числом потоков дали одинаковый результат.

Гонки данных отсутствуют, так как каждый поток использует собственный локальный буфер,
и запись выполняется только в `blocks[tid]`.
Между потоками нет конкурентной записи в один объект.

## 8. Результаты

| workers  | time (s)  | speedup   | efficiency    |
| -------- | --------- | --------- | ------------- |
| 1        | 2.709     | 1.00      | 1.0           |
| 2        | 2.149     | 1.26      | 0.63          |
| 4        | 2.106     | 1.29      | 0.32          |

Параллельная STL-версия показывает умеренное ускорение относительно последовательной реализации.
Наибольший выигрыш достигается при использовании 2–4 потоков, однако рост производительности
оказывается ограниченным. При увеличении числа потоков эффективность заметно снижается.

Основные причины ограничения масштабируемости:

- последовательный этап std::merge;
- накладные расходы на создание потоков (std::thread);
- затраты на join();
- выделение локальных временных буферов.

Для данной реализации overhead ручного управления потоками начинает заметно влиять на производительность уже при 4 потоках.

## 9. Выводы

Параллельное выполнение локальной сортировки даёт ускорение относительно SEQ-версии,
однако масштабируемость остаётся ограниченной. Результаты показывают, что данная реализация
эффективна при небольшом числе потоков, но дальнейшее увеличение степени параллелизма даёт
всё меньший выигрыш из-за роста overhead.

## Приложения

```bash
for (size_t tid = 0; tid < num_threads; ++tid) {
    threads[tid] = std::thread([&, tid]() {
      size_t start = tid * block_size;
      size_t end = std::min(start + block_size, n);

      if (start >= n) {
        return;
      }

      std::vector<int> local;
      local.reserve(end - start);

      for (size_t i = start; i < end; ++i) {
        local.push_back(input[i]);
      }

      ShellSort(local);

      blocks[tid] = std::move(local);
    });
  }

  for (auto &th : threads) {
    th.join();
  }
```
