## 1. Введение

В данной работе рассматривается задача сортировки массива целых чисел алгоритмом Shell Sort
с последующим слиянием отсортированных блоков.

Алгоритм содержит как вычислительно затратную часть (сортировка), так и этапы объединения данных,
что позволяет исследовать влияние различных типов параллелизма и коммуникационных накладных расходов.

## 2. Единая постановка задачи

Требуется отсортировать вектор целых чисел произвольного размера по неубыванию.

Входные данные: Вектор целых чисел произвольной длины (включая пустой вектор).
Выходные данные: Вектор тех же целых чисел, отсортированный в порядке неубывания (по возрастанию).

Результат считается корректным, если:

- элементы не изменены;
- массив отсортирован по неубыванию;
- результат совпадает с `std::sort`.

## 3. Единая методика эксперимента

### **Окружение:**

- CPU: Intel Core i7-1255U
- Память: 16,0 ГБ DDR4  
- OS: Windows 10  
- Компилятор: MSVC  
- Сборка: Release  

### **Параметры тестирования**

- Массив из 4 000 000 псевдослучайных целых чисел.
- Медиана по 5 запускам (`--gtest_repeat=5`).
- **Speedup** = время SEQ / время параллельной версии.
- **Efficiency** = speedup / workers.
- Для ALL: workers = ranks × threads.

### Сборка

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

Команды запуска тестов находятся в локальных отчётах:

- `seq/report.md`
- `omp/report.md`
- `tbb/report.md`
- `stl/report.md`
- `all/report.md`

## 4. Сводка корректности

Все реализации прошли функциональные тесты. Для ALL дополнительно реализована рассылка

## 6. Интерпретация различий

SEQ — Последовательная версия используется как baseline. Время 2.709 s определяет нормировку speedup.
OMP — Демонстрирует лучший результат среди shared-memory моделей при 2 потоках.
      При 4 workers эффективность падает из-за роста overhead и конкуренции за память.
TBB — Более стабильная, но менее эффективная на данной задаче модель.
      Выигрывает за счёт scheduler и балансировки задач, ограничение — стоимость локального merge
      и отсутствие идеального grain tuning.
STL — Показывает минимальное ускорение. По причинам:
        - ручное управление потоками (create/join overhead);
        - отсутствие эффективного thread pool;
        - стоимость merge сопоставима с вычислениями.
ALL — Дает наилучший абсолютный результат.
        - при 2 × 1 достигается максимальный speedup (4.23), включая суперлинейный эффект;
        - при 2 × 2 наблюдается рост overhead из-за MPI-коммуникаций и финального merge на rank 0;
        - эффективность зависит от выбранной нормировки (workers vs ranks×threads),
          поэтому значения > 1 интерпретируются как эффект декомпозиции задачи и кэш-локальности.

Наблюдается снижение эффективности с ростом числа потоков — чем больше блоков,
тем больше итераций у последовательного слияния.

## 7. Заключение

Результаты показывают, что производительность существенно зависит не только от числа вычислительных потоков,
но и от архитектуры параллелизма.

Итоговые выводы по каждой технологии:

STL — наименее эффективная модель из-за overhead управления потоками;
OMP — хороший баланс простоты и производительности;
TBB — стабильный runtime, но ограничен стоимостью merge;
ALL — лучшая производительность за счёт иерархического параллелизма,
    а суперлинейные значения speedup/efficiency объясняются эффектами разбиения данных и кэширования.
    Применение MPI целесообразно только при запуске на нескольких физических узлах.
