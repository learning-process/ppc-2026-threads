## 1. Контекст

Гибридная версия объединяет *MPI* и *OpenMP* для реализации двухуровневого параллелизма:
межпроцессного и внутрипроцессного.

*MPI* используется для распределения данных между процессами,
а *OpenMP* — для параллельной сортировки внутри каждого rank-а.

Такой подход позволяет одновременно использовать распределённую память между процессами
и многопоточность внутри каждого процесса.

SEQ-версия используется как baseline для оценки ускорения.

## 2. Постановка задачи

Требуется реализовать гибридную сортировку массива целых чисел с использованием:

1. MPI для распределения данных;
2. OpenMP для внутрипроцессного распараллеливания.

Входные, выходные данные и корректность результата аналогичны последовательной версии.
SEQ-версия считается эталоном корректности.

## 3. Базовый алгоритм

Алгоритм работы гибридной версии состоит из нескольких этапов:

1. Процесс с rank 0 хранит исходный массив.
2. MPI распределяет данные между процессами.
3. Каждый процесс:
    - получает собственный блок данных;
    - дополнительно делит его между OpenMP-потоками;
    - сортирует локальные части алгоритмом ShellSort.
4. Внутри процесса локальные части объединяются через merge.
5. MPI собирает локальные результаты на rank 0.
6. Rank 0 выполняет финальное слияние всех отсортированных блоков.

## 4. Межпроцессная схема

### Роли rank-ов

`Rank 0` отвечает за:

- хранение исходного массива;
- распределение данных;
- сбор локальных результатов;
- финальное слияние;
- рассылку итогового результата.

`Остальные rank-и` выполняют:

- получение локального блока;
- локальную сортировку;
- отправку результата обратно.

### Распределение данных

Размеры блоков вычисляются функцией: `CalculateCountsAndDisplacements(...)`

Для каждого процесса определяются: `sendcounts` и `displacements`.
Это позволяет распределять массив даже при неравномерном делении
размера данных на число процессов.

`MPI_Comm_rank(MPI_COMM_WORLD, &rank)` определяет номер процесса в коммуникаторе.

`MPI_Comm_size(MPI_COMM_WORLD, &size)` получает число процессов.

`MPI_Bcast` Используется для рассылки размера глобального массива,
sendcounts, displacements, итогового результата.

`MPI_Scatterv(...)` Используется для распределения частей массива между процессами
с учётом различных размеров блоков.

`MPI_Gatherv` Используется для сбора размеров локальных результатов и сбора отсортированных блоков на rank 0.

Синхронизация обеспечивается коллективными операциями (MPI_Bcast, MPI_Scatterv, MPI_Gather, MPI_Gatherv),
так как данные операции блокируют выполнение до завершения обмена, дополнительный барьер не требуется.

## 5. Внутрипроцессная схема

Используемая технология внутри rank-а: **openMP**. Внутри каждого процесса разделение
происходит через `#pragma omp parallel for`.

### Разделение работы между потоками

Каждый rank дополнительно делит свой локальный массив на `parts` частей.

Для каждого потока вычисляются границы: `borders[i]` и `borders[i + 1]`.

Каждый поток:

- копирует собственный диапазон;
- выполняет локальный Shell Sort;
- сохраняет результат в: `buffers[i]`

### Объединение результатов

После завершения OpenMP-сортировки `MergeLocalBuffers(buffers)` объединяет
локальные части внутри rank-а.
После `MPI_Gatherv` rank 0 дополнительно объединяет результаты всех процессов.

Каждый поток работает только со своим локальным буфером, поэтому гонки данных отсутствуют.
После `parallel for` *OpenMP* создаёт неявный барьер синхронизации.

## 6. Детали реализации

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

### Основные этапы

1. Рассылка размеров `MPI_Bcast(...)`
2. Распределение данных `MPI_Scatterv(...)`
3. OpenMP-сортировка  `#pragma omp parallel for`
4. Сбор результатов `MPI_Gatherv(...)`
5. Финальное merge `MergeLocalBuffers(chunks)`

### Потенциальные узкие места

- последовательное финальное merge на rank 0;
- коммуникационные затраты MPI;
- дополнительные копирования данных;
- дисбаланс размеров блоков при малых массивах.

## 7. Проверка корректности

Для проверки функционала использовались те же тесты, что и для последовательной версии.

### Проверка согласованности

## 9. Результаты

| processes × threads| workers  | time (s)  | speedup   | efficiency    |
| ------------------ | -------- | --------- | --------- | ------------- |
|  seq               | 1        | 2.709     | 1.00      | 1.0           |
|  2 x 1             | 2        | 0.639     | 4.23      | 2.12          |
|  2 x 2             | 4        | 0.594     | 4.56      | 0.55          |
|  2 x 3             | 6        | 0.756     | 3.58      | 0.55          |

Гибридная версия MPI + OpenMP даёт значительное ускорение относительно SEQ.

Лучший результат достигается при конфигурации 2 × 2 (0.594 c).

Основные наблюдения:

- при 2 × 1 наблюдается суперлинейный speedup (4.23), что объясняется уменьшением размера
    локальных задач и улучшением кэш-локальности;
- при увеличении числа потоков внутри процесса (2 × 2 → 2 × 3) производительность падает;
- эффективность ограничивается накладными расходами OpenMP и MPI-обменами.

## 10. Выводы

Гибридная схема эффективно использует два уровня параллелизма: MPI для распределения данных и
OpenMP для локальной обработки.

Однако масштабируемость ограничена затратами на коммуникации MPI, overhead OpenMP при росте числа потоков,
последовательным merge на rank 0.

Оптимальной конфигурацией для данной реализации является 2 × 2, где достигается лучший баланс между
вычислениями и накладными расходами.

## Приложения

### MPI-распределение

```bash
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

MPI_Bcast(&global_size, 1, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(sendcounts.data(), size, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(displs.data(), size, MPI_INT, 0, MPI_COMM_WORLD);

MPI_Scatterv(input_ptr, sendcounts.data(), displs.data(), MPI_INT,
             local_data.data(), local_size, MPI_INT, 0, MPI_COMM_WORLD);
```

### OpenMP часть

```bash
#pragma omp parallel for default(none) shared(local_data, buffers, borders, parts)
for (size_t i = 0; i < parts; ++i) {
  std::vector<int> temp(
      local_data.begin() + borders[i],
      local_data.begin() + borders[i + 1]);

  ShellSort(temp);
  buffers[i] = std::move(temp);
}
```
