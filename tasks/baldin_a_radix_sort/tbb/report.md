## 1. Контекст

Использование Intel Threading Building Blocks (oneTBB) переводит модель распараллеливания от ручного управления потоками
(как в `std::thread` или OpenMP) к задачно-ориентированной архитектуре (task-based parallelism). Планировщик TBB берет
на себя распределение задач по рабочим потокам (worker threads) и балансировку нагрузки через механизм "кражи задач"
(work-stealing). Для задачи поразрядной сортировки со слиянием это позволяет лаконично описать параллельные фазы
алгоритма с помощью лямбда-выражений.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел `std::vector<int>`. В качестве эталона корректности и базового времени
($T_1$) для расчета ускорения используется последовательная реализация, описанная в `seq/report.md`.

## 3. Базовый алгоритм

Последовательная версия реализует побайтовый LSD Radix Sort (4 прохода). Для корректной сортировки отрицательных чисел
на последнем проходе применяется побитовая инверсия знакового бита операцией `XOR 128`.

## 4. Схема распараллеливания

Алгоритм разделен на две фазы. Для обеих фаз выбран примитив **`tbb::parallel_for`**, однако архитектура применения TBB
имеет специфическую особенность: мы не отдаем фреймворку сырой массив для автоматического дробления.

1. **Ограничение числа задач:** Чтобы итеративное дерево слияния работало эффективно, нам нужно строго фиксированное
   количество начальных блоков. Поэтому исходный массив заранее разбивается на `num_chunks` частей, где `num_chunks`
   равно аппаратному числу потоков (`ppc::util::GetNumThreads()`).
2. **Диапазон работы (`blocked_range`):** В `tbb::parallel_for` передается диапазон не по элементам массива `[0, N)`, а
   по индексам вычисленных блоков: `tbb::blocked_range<int>(0, num_chunks)`.
3. **Grainsize:** Оставлен по умолчанию (равен 1). Поскольку диапазон задач очень мал (от 0 до количества ядер),
   дальнейшее дробление диапазона планировщиком не требуется. Каждая итерация `blocked_range` — это уже крупная
   вычислительная задача по сортировке $N/P$ элементов.
4. **Partitioner:** Оставлен `auto_partitioner` (по умолчанию), так как при малом размере диапазона и тяжеловесных
   задачах внутри планировщик TBB корректно распределит индексы блоков по пулу потоков без необходимости ручной
   балансировки.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Важной деталью реализации является адаптация цикла слияния под требования TBB. Примитив `tbb::blocked_range` требует
непрерывного шага итерации (+1), в то время как в классическом дереве слияний шаг меняется ($2, 4, 8 \dots$).

### Древовидное слияние с адаптацией индексов

```cpp
// File: tbb/src/ops_tbb.cpp
for (int step = 1; step < num_chunks; step *= 2) {
  int num_merges = (num_chunks + (2 * step) - 1) / (2 * step);

  tbb::parallel_for(tbb::blocked_range<int>(0, num_merges),
    [&](const tbb::blocked_range<int> &r) {
      for (int m_idx = r.begin(); m_idx != r.end(); m_idx++) {
        int i = m_idx * (2 * step); // Восстановление реального сдвига

        if (i + step < num_chunks) {
          auto begin = out.begin() + offsets[i];
          auto middle = out.begin() + offsets[i + step];
          int end_idx = std::min(i + (2 * step), num_chunks);
          auto end = out.begin() + offsets[end_idx];

          std::inplace_merge(begin, middle, end);
        }
      }
  });
}
```

Внешний цикл отвечает за уровни дерева и выполняется последовательно. Внутри высчитывается точное количество слияний на
текущем уровне (`num_merges`). `tbb::parallel_for` итерируется по непрерывному диапазону индексов слияний
`[0, num_merges)`. Внутри лямбда-выражения рабочий поток TBB получает индекс задачи `m_idx` и математически
пересчитывает его в реальный сдвиг `i` для массива `offsets`. Это позволяет удовлетворить архитектурные требования TBB,
сохранив оригинальную логику слияния соседних кусков через `std::inplace_merge`.

### Оптимизация работы с памятью (буфер temp)

```cpp
// Внутри функции RadixSortLocal
std::vector<int> temp(n);
for (size_t i = 0; i < sizeof(int); i++) {
  if (i % 2 == 0) {
    CountingSortStep(begin, end, temp.begin(), i);
  } else {
    CountingSortStep(temp.begin(), temp.end(), begin, i);
  }
}
```

Дополнительно сохранена оптимизация памяти: буфер `temp` выделяется локально внутри функции `RadixSortLocal` один раз на
каждую задачу, что исключает ложное разделение и снижает нагрузку на аллокатор.

## 6. Проверка корректности

Сравнение результатов выполняется с эталонным `std::ranges::sort`. Тестовый фреймворк успешно подтверждает прохождение
функциональных тестов (включая пустые массивы и массивы со смешанными знаками). Поскольку каждый рабочий поток в
`tbb::parallel_for` модифицирует строго непересекающиеся диапазоны в общем массиве `out`, состояние гонки (data race)
концептуально исключено.

_Примечание к результатам:_

1. **Сверхэффективность на 1 потоке (>100%):** Время на 1 потоке (80.82 мс) лучше базового SEQ (124.81 мс). Это связано
   с оптимизацией памяти в TBB-версии (однократное выделение буфера `temp`), в то время как в SEQ память динамически
   перевыделяется 4 раза внутри цикла.
2. **Масштабируемость:** Overhead от создания пула задач TBB минимален (результаты близки к OMP). Ускорение алгоритма
   достигает пика на 2 потоках, после чего падает. Это обусловлено законом Амдала: последовательная компонента алгоритма
   (ожидание на верхних ярусах дерева слияния `std::inplace_merge`) не поддается распараллеливанию. Рост числа workers
   лишь дробит массив на большее количество мелких блоков, увеличивая глубину дерева слияния.

## 9. Выводы

Архитектура oneTBB позволила лаконично реализовать параллельные фазы алгоритма через лямбда-выражения без необходимости
явной работы с потоками ОС. Однако специфика алгоритма Radix Sort со слиянием (необходимость фиксированного дерева
объединений) не позволяет в полной мере использовать главное преимущество TBB - рекурсивное динамическое разбиение
данных и work-stealing. Мы были вынуждены искусственно ограничить число задач на старте константой аппаратных ядер,
сведя работу планировщика TBB к поведению, аналогичному статическому расписанию (`schedule(static)`) в OpenMP.
