## 1. Контекст

Гибридная версия требуется для максимальной утилизации современных вычислительных кластеров. Чистый MPI создает
избыточные накладные расходы на коммуникацию, если запускать по одному процессу на каждое ядро многоядерного узла.
Чистый OpenMP ограничен рамками одной общей памяти. Связка MPI + OpenMP решает эту проблему двухуровневой иерархией: MPI
распределяет крупные блоки данных между сетевыми узлами, а OMP эффективно сортирует эти блоки внутри каждого узла,
используя общую память.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел `std::vector<int>`. Последовательная версия (SEQ) служит эталоном
корректности. Базовое время ($T_1$) для расчета ускорения берется из SEQ-версии для обеспечения методически честного
сравнения накладных расходов гибридной архитектуры.

## 3. Базовый алгоритм

Основой служит LSD Radix Sort (4 побайтовых прохода) с инверсией знакового бита (`XOR 128`) на последнем этапе для
поддержки отрицательных чисел.

## 4. Межпроцессная схема

На уровне процессов (MPI) используется парадигма Master-Worker.

- **Роли rank-ов:** Нулевой процесс (`rank == 0`) выступает в роли Master. Он владеет исходным массивом, вычисляет
  размеры блоков для каждого узла с учетом остатка от деления и собирает итоговый результат. Процессы `rank > 0`
  выступают в роли Workers.
- **Распределение данных:** Передача данных от Master к Workers осуществляется через коллективный вызов `MPI_Scatterv`.
  Обратный сбор отсортированных блоков — через `MPI_Gatherv`. Использование векторных версий (`*v`) необходимо, так как
  размер массива не всегда делится нацело на число процессов.
- **Синхронизация:** Явные вызовы `MPI_Barrier` в коде отсутствуют за ненадобностью. Коллективные операции `Scatterv` и
  `Gatherv` являются блокирующими, что обеспечивает естественную семантическую синхронизацию между фазами алгоритма.

## 5. Внутрипроцессная схема

Внутри каждого `rank`-а для распараллеливания используется OpenMP.

- Данные (`local_data_`), полученные процессом от MPI, логически делятся на `num_threads` диапазонов.
- **Работа потоков:** В параллельной секции `#pragma omp parallel` каждый поток локально сортирует свой диапазон с
  помощью `RadixSortLocal`.
- **Объединение:** После локальной сортировки потоки внутри процесса выполняют древовидное слияние
  (`std::inplace_merge`) своих участков. Неявные барьеры в конце директив OpenMP гарантируют, что поток не начнет
  слияние до завершения сортировки.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

### Межпроцессный обмен (MPI)

```cpp
// File: all/src/ops_all.cpp (в методе DistributeData)
if (is_mpi) {
  MPI_Scatterv(rank == 0 ? GetOutput().data() : nullptr,
               counts_.data(), displs_.data(), MPI_INT,
               local_data_.data(), counts_[rank], MPI_INT,
               0, MPI_COMM_WORLD);
} else {
  local_data_ = GetOutput();
}
```

_Пояснение:_ `MPI_Scatterv` отправляет каждому процессу (включая нулевой) порцию данных размером `counts_[rank]`. Важно,
что память под `local_data_` выделяется строго по размеру куска. Если программа запущена без `mpirun`, массив просто
копируется в локальный буфер, что позволяет переиспользовать дальнейшую OpenMP логику без изменений.

### Глобальное слияние на Master-узле

```cpp
// File: all/src/ops_all.cpp (в методе GlobalMerge)
for (int step = 1; step < size; step *= 2) {
#pragma omp parallel for num_threads(num_threads) default(none) shared(step, size, global_offsets, out)
  for (int i = 0; i < size; i += 2 * step) {
    if (i + step < size) {
      auto begin = out.begin() + global_offsets[i];
      auto middle = out.begin() + global_offsets[i + step];
      int end_idx = std::min(i + (2 * step), size);
      auto end = out.begin() + global_offsets[end_idx];

      std::inplace_merge(begin, middle, end);
    }
  }
}
```

_Пояснение:_ После того как `MPI_Gatherv` собрал отсортированные MPI-куски в массив `out` на нулевом процессе, массив
состоит из `size` (число процессов) отсортированных последовательностей. Нулевой процесс выполняет финальное древовидное
слияние этих кусков. Для устранения узкого места этот цикл также распараллелен средствами OpenMP, что позволяет
задействовать все ядра Master-узла на завершающем этапе.

## 7. Проверка корректности

Сравнение выполняется с эталонным `std::ranges::sort` на нулевом процессе. Рабочие процессы (`rank != 0`) обходят
проверку, так как их итоговый массив пуст. Алгоритм устойчив к крайним случаям: добавлен ранний выход `if (n == 0)` и
`if (local_n == 0)`, что предотвращает вызов MPI-функций с нулевыми буферами и исключает зависания в коммуникаторах.
Изоляция адресных пространств процессов MPI гарантирует отсутствие межпроцессных гонок данных.

## 10. Выводы

Гибридная схема (MPI + OpenMP) **оправдана**:

- **При обработке больших массивов данных** (как в тесте на 10М элементов). Использование MPI позволило физически
  разделить массив на части, что снизило нагрузку на шину памяти и улучшило работу кэш-памяти. Благодаря этому гибридная
  версия стала лидером, обогнав чистый OpenMP почти в 1.7 раза (45.07 мс против 76.56 мс на 6 ядрах).
- **Для масштабирования на кластеры**, где узлы не имеют общей памяти.

Гибридная схема **не оправдана**:

- **На малых объемах данных**, так как время на пересылку сообщений (`MPI_Scatterv`/`MPI_Gatherv`) превысит время самой
  сортировки.
- **При превышении числа физических ядер**. Замеры показали, что при $P=8$ (использование Hyper-Threading)
  производительность начинает падать, так как два процесса MPI начинают конкурировать за ресурсы одного ядра и его кэша.
