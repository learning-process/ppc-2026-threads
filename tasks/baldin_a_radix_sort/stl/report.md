## 1. Контекст

Ручная thread-based версия необходима для демонстрации умения работать с примитивами многопоточности стандартной
библиотеки C++ (STL). В отличие от OpenMP, где компилятор берет на себя управление пулом потоков и синхронизацией, здесь
программист обязан самостоятельно распределить данные, запустить асинхронные задачи и расставить точки барьерной
синхронизации для обеспечения корректности выполнения иерархического алгоритма.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел `std::vector<int>`. В качестве эталона корректности и базового времени
($T_1$) для расчета ускорения используется последовательная реализация, описанная в `seq/report.md`.

## 3. Базовый алгоритм

Базовый алгоритм представляет собой 4 побайтовых прохода устойчивой сортировки подсчетом (LSD Radix Sort). Для поддержки
отрицательных чисел на последнем проходе применяется побитовая инверсия знакового бита операцией `XOR 128`.

## 4. Схема распараллеливания

Алгоритм разделен на фазу локальной сортировки и фазу древовидного слияния. Вместо `std::thread`, требующих ручного
управления `join`, выбрана абстракция более высокого уровня: `std::async` с политикой `std::launch::async`, которая
принудительно создает и запускает новый системный поток, и объекты `std::future`.

- **Разбиение данных:** Массив делится на `num_chunks` блоков. Массив `offsets` хранит границы каждого блока (карта
  `поток -> диапазон`).
- **Задачи потоков:** В первой фазе каждый поток локально сортирует свой независимый блок. Во второй фазе каждый поток
  сливает (`std::inplace_merge`) пару соседних блоков.
- **Хранение локальных результатов:** Локальные результаты записываются "на месте" (in-place) в общий вектор `out`.
- **Синхронизация:** Явные мьютексы (`std::mutex`) или атомарные переменные (`std::atomic`) отсутствуют. Отсутствие
  гонок обеспечивается тем, что на любой итерации потоки работают со строго непересекающимися диапазонами памяти
  `[local_begin, local_end)`.
- **Схема барьера (замена join):** Вызов `f.get()` для каждого объекта `std::future` в векторе используется как жесткий
  барьер. Код не переходит к следующему этапу или следующему ярусу дерева слияния, пока все асинхронные потоки текущего
  этапа не завершат работу.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

### Создание потоков и барьерная синхронизация

```cpp
// File: stl/src/ops_stl.cpp
std::vector<std::future<void>> futures;
futures.reserve(num_chunks);

for (int tid = 0; tid < num_chunks; ++tid) {
  futures.push_back(std::async(std::launch::async, [tid, &out, &offsets]() {
    auto local_begin = out.begin() + offsets[tid];
    auto local_end = out.begin() + offsets[tid + 1];
    RadixSortLocal(local_begin, local_end);
  }));
}

for (auto &f : futures) {
  f.get();
}
```

_Пояснение:_ В цикле создаются `num_chunks` асинхронных задач. Общие структуры данных `out` и `offsets` захватываются по
ссылке (`&`), чтобы избежать тяжелого копирования, а индекс потока `tid` - по значению. Если захватить `tid` по ссылке,
к моменту старта нового потока цикл `for` может успеть инкрементировать `tid`, что приведет к гонке данных и
некорректному вычислению диапазонов (`local_begin`/`local_end`). После запуска всех задач вызывается второй цикл с
`f.get()`.

### Оптимизация работы с памятью (буфер temp)

```cpp
// Внутри функции RadixSortLocal
std::vector<int> temp(n);
for (size_t i = 0; i < sizeof(int); i++) {
  if (i % 2 == 0) {
    CountingSortStep(begin, end, temp.begin(), i);
  } else {
    CountingSortStep(temp.begin(), temp.end(), begin, i);
  }
}
```

Дополнительно сохранена оптимизация памяти: буфер `temp` выделяется локально внутри функции `RadixSortLocal` один раз на
каждую задачу, что исключает ложное разделение и снижает нагрузку на аллокатор.

## 6. Проверка корректности

Сравнение результатов выполняется с эталонным алгоритмом. Алгоритм стабильно проходит функциональные тесты, включая
крайние случаи (пустые массивы, уже отсортированные данные, `INT_MAX/MIN`). Проверка на отсутствие гонок концептуальна:
благодаря предвычисленной карте границ `offsets`, потоки не пишут в одни и те же ячейки памяти до момента финального
слияния. Иерархия `get()` исключает пересечение работы разных фаз.

_Примечание к результатам:_

1. **Сверхэффективность на 1 потоке (>100%):** Время на 1 потоке (81.77 мс) лучше базового SEQ (124.81 мс). Это связано
   с оптимизацией памяти в STL-версии (однократное выделение буфера `temp`), в то время как в SEQ память динамически
   перевыделяется 4 раза внутри цикла.
2. **Накладные расходы STL:** Пик ускорения (1.75x) наблюдается на 2 потоках. В отличие от OMP, который использует пул
   заранее созданных спящих потоков, `std::launch::async` может требовать от ОС создания нового системного потока при
   каждом вызове. Это создает более высокие накладные расходы на `create`/`join` системных ресурсов, что особенно
   заметно на 12 потоках (85.93 мс против 81.15 у TBB).

## 9. Выводы

Ручная реализация подтвердила, что архитектура "разделяй и властвуй" успешно ложится на примитивы STL. Применение
`std::async` и `std::future` позволило избежать ручного управления `std::thread` и построить механизм барьерной
синхронизации. Однако ручное создание системных потоков на каждом этапе алгоритма генерирует больший overhead по
сравнению с декларативными фреймворками (OMP/TBB), что ограничивает эффективность масштабирования при увеличении числа
ядер.
