## 1. Контекст

Перенос алгоритма из последовательной версии (`SEQ`) в OpenMP необходим для утилизации многоядерных процессоров.
Поскольку базовая сортировка подсчетом (`Counting Sort`) имеет строгую зависимость по данным (расчет префиксных сумм и
запись в один массив), прямое распараллеливание циклов внутри нее неэффективно. Поэтому для OMP-версии выбрана стратегия
"разделяй и властвуй": массив физически бьется на независимые куски, сортируется локально, а затем собирается через
параллельное древовидное слияние.

## 2. Постановка задачи

Требуется отсортировать массив целых чисел `std::vector<int>`. В качестве эталона корректности и базового времени
($T_1$) для расчета ускорения используется последовательная реализация, описанная в `seq/report.md`.

## 3. Базовый алгоритм

Последовательная версия использует LSD Radix Sort: выполняется 4 прохода побайтовой сортировки подсчетом. Отрицательные
числа корректно обрабатываются за счет инверсии старшего (знакового) бита операцией `XOR 128` на последнем проходе.

## 4. Схема распараллеливания

Алгоритм разделен на два этапа, между которыми существует жесткая синхронизация:

1. **Локальная сортировка:** Исходный массив логически делится на `num_threads` участков. Создается параллельная область
   `#pragma omp parallel`, где каждый поток выполняет последовательный `Radix Sort` только над своим участком.
2. **Древовидное слияние (Tree Merge):** Выполняется итеративный цикл, на каждом шаге которого пары соседних
   отсортированных участков сливаются с помощью `std::inplace_merge`. Внутренний цикл распараллелен через
   `#pragma omp parallel for`.

**Анализ директив и атрибутов данных:**

- `default(none)`: Форсирует явное указание классов памяти для всех переменных из внешнего скоупа, предотвращая скрытые
  гонки данных.
- `shared(num_threads, offsets, out, step)`: Массив данных `out` и массив границ `offsets` являются общими. Состояние
  гонки (Data Race) не возникает, так как на любом этапе каждый поток гарантированно работает с непересекающимся
  диапазоном индексов `[begin, end)`.
- `private`: Переменные `tid`, `begin`, `middle`, `end` объявляются непосредственно внутри параллельных блоков, поэтому
  они автоматически становятся локальными (`private`) для каждого потока.
- `reduction / atomic / critical`: Не используются, так как потоки не модифицируют общие счетчики или аккумуляторы.
- `schedule`: Явно не указан, поэтому применяется стандартный `schedule(static)`. Это оптимальный выбор, так как
  вычислительная нагрузка (размер участков массива) распределена равномерно.
- **Барьеры:** В конце директив `omp parallel` и `omp parallel for` стоят **неявные барьеры**. Это критически важно:
  этап слияния не может начаться, пока все потоки не завершат локальную сортировку, а шаг слияния $2k$ не может
  начаться, пока не завершится шаг $k$.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

В реализации предусмотрена оптимизация: если размер массива слишком мал ($N < num\_threads \cdot 100$), алгоритм
переключается в однопоточный режим, чтобы избежать накладных расходов на создание потоков.

### Параллельная локальная сортировка

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel num_threads(num_threads) default(none) shared(num_threads, offsets, out)
{
  int tid = omp_get_thread_num();
  auto begin = out.begin() + offsets[tid];
  auto end = out.begin() + offsets[tid + 1];

  RadixSortLocal(begin, end);
}
```

_Пояснение:_ Здесь создается пул потоков. Каждый поток забирает свой уникальный идентификатор `tid` и с помощью
предвычисленного массива `offsets` определяет свои границы работы. Вызов `RadixSortLocal` безопасен, так как он работает
с независимым участком общего массива `out`.

### Оптимизация работы с памятью (буфер temp)

```cpp
// Внутри функции RadixSortLocal
std::vector<int> temp(n);
for (size_t i = 0; i < sizeof(int); i++) {
  if (i % 2 == 0) {
    CountingSortStep(begin, end, temp.begin(), i);
  } else {
    CountingSortStep(temp.begin(), temp.end(), begin, i);
  }
}
```

_Пояснение:_ В многопоточной среде частые вызовы аллокатора памяти внутри параллельной секции приводят к конкуренции
потоков за системный менеджер кучи. Чтобы этого избежать, временный буфер `temp` выделяется в `RadixSortLocal` ровно
один раз для каждого потока. Использование схемы четный/нечетный проход позволяет избежать лишних копирований: так как
`int` занимает 4 байта, итоговый результат гарантированно оказывается в исходном массиве `begin`.

### Древовидное слияние

```cpp
// File: omp/src/ops_omp.cpp
for (int step = 1; step < num_threads; step *= 2) {
#pragma omp parallel for num_threads(num_threads) default(none) shared(step, num_threads, offsets, out)
  for (int i = 0; i < num_threads; i += 2 * step) {
    if (i + step < num_threads) {
      // ... вычисление begin, middle, end на основе offsets ...
      std::inplace_merge(begin, middle, end);
    }
  }
}
```

_Пояснение:_ Внешний цикл `step` выполняется последовательно. Внутренний цикл распараллеливается: доступные потоки
распределяют между собой пары независимых отсортированных участков и сливают их. Неявный барьер в конце
`#pragma omp parallel for` гарантирует, что следующий шаг укрупнения кусков начнется только при консистентном состоянии
всего массива.

## 6. Проверка корректности

Сравнение результатов выполняется с эталонным `std::ranges::sort`. Тестовый каркас гарантирует детерминированность
выполнения. Алгоритм успешно проходит функциональные тесты на граничные случаи (пустые массивы, уже отсортированные
данные, наличие `INT_MAX/MIN`). Разделение памяти строго контролируется через массив `offsets`, что исключает гонки
данных на границах кусков.

_Примечание к результатам:_

1. **Сверхэффективность на 1 потоке (>100%):** Время на 1 потоке (81.14 мс) лучше базового SEQ (124.81 мс). Это связано
   с оптимизацией памяти в OMP-версии (однократное выделение буфера `temp`), в то время как в SEQ память динамически
   перевыделяется 4 раза внутри цикла.
2. **Масштабируемость:** Пик ускорения достигается на 2 потоках. Рост до 4 и 6 потоков (в пределах физических ядер)
   приводит к стагнации из-за того, что фаза древовидного слияния (`std::inplace_merge`) упирается в пропускную
   способность памяти. Переход к 12 потокам вызывает деградацию производительности, так как технология Hyper-Threading
   заставляет потоки конкурировать за ресурсы кэша L1/L2 на одних и тех же физических ядрах.

## 9. Выводы

OpenMP дал существенный выигрыш в скорости благодаря равномерному распределению независимых участков массива по
вычислительным ядрам. Оптимизация с однократным выделением `temp` в каждом потоке позволила минимизировать накладные
расходы на работу с кучей. Директивы `#pragma omp` позволили реализовать паттерн "разделяй и властвуй" с минимальным
вмешательством в базовый код. Однако выигрыш ограничен накладными расходами на слияние: чем больше потоков, тем больше
кусков нужно слить, и тем длиннее последовательная фаза на вершине дерева слияний.
