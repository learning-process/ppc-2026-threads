## 1. Контекст

В данном отчете рассматривается базовая последовательная реализация алгоритма поразрядной сортировки (`Radix Sort`). Эта
версия необходима для определения корректности математической логики алгоритма (в частности, работы с отрицательными
числами) и служит эталоном для измерения ускорения в последующих параллельных реализациях (`OMP`, `TBB`, `STL`, `ALL`).

## 2. Постановка задачи

- **Входные данные:** Неотсортированный массив целых чисел `std::vector<int>`.
- **Выходные данные:** Тот же массив, отсортированный по неубыванию.
- **Ограничения и крайние случаи:** Тип `int` может иметь различный размер на разных платформах (обычно 4 байта).
  Алгоритм должен корректно обрабатывать весь диапазон значений `int`, включая нули, положительные, отрицательные числа,
  а также граничные значения `INT_MAX` и `INT_MIN`.

## 3. Базовый алгоритм

Для сортировки выбран алгоритм LSD Radix Sort (сортировка от младшего разряда к старшему). Алгоритм работает побайтово.
Поскольку `sizeof(int)` обычно равен 4, выполняется 4 прохода устойчивой сортировки подсчетом (Counting Sort) по
основанию 256.

- **Асимптотика по времени:** $O(d \cdot (N + K))$, где $N$ — размер массива, $d$ — количество байт (разрядов, $d=4$),
  $K$ — размер корзины ($K=256$). Итоговая сложность строго линейная: $O(N)$.
- **Асимптотика по памяти:** $O(N + K)$ на каждом шаге для создания временного массива `output` и массива частот
  `count`.
- **Инварианты и критерий корректности:** После $i$-го прохода массив стабильно отсортирован по $i$ младшим байтам.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

- **`ValidationImpl()`**: Проверяет валидность входных данных. В данном случае всегда возвращает `true`, так как любой
  вектор (в том числе пустой) является допустимым для сортировки.
- **`PreProcessingImpl()`**: Выполняет начальную подготовку, копируя входные данные (`GetInput()`) в выходной буфер
  (`GetOutput()`), чтобы алгоритм мог выполнять сортировку "на месте" выходного массива.
- **`RunImpl()`**: Выполняет основной цикл побайтовой сортировки
  `for (size_t byte_index = 0; byte_index < sizeof(int); byte_index++)`. Для каждого байта вызывается функция устойчивой
  сортировки подсчетом `CountingSortByByte`.
- **`PostProcessingImpl()`**: Возвращает `true`, так как дополнительных действий по форматированию результата не
  требуется - отсортированный массив уже находится в `GetOutput()`.

### Обработка знаковых чисел

```cpp
// Выделение текущего байта
unsigned int byte_val = (raw_val >> shift) & 0xFF;

// Инверсия старшего бита для корректной сортировки знаковых чисел
if (byte_index == sizeof(int) - 1) {
  byte_val ^= 128;
}
count[byte_val]++;
```

_Пояснение:_ Прямое применение Radix Sort к знаковым целым числам приведет к тому, что отрицательные числа окажутся
больше положительных (из-за единицы в знаковом бите). Чтобы этого избежать, при обработке самого старшего байта
применяется операция XOR с маской 128 (`10000000` в двоичной системе). Это инвертирует знаковый бит, программно сдвигая
шкалу так, что отрицательные числа начинают восприниматься алгоритмом как меньшие по отношению к положительным.

Также для сохранения устойчивости алгоритма финальное распределение элементов по корзинам выполняется обратным циклом
`for (size_t i = n; i > 0; i--)`.

## 5. Проверка корректности

Корректность алгоритма проверяется сравнением с эталонным `std::ranges::sort`. Функциональные тесты покрывают следующие
граничные случаи:

- Пустой массив и массив из 1 элемента;
- Уже отсортированный и отсортированный в обратном порядке массивы;
- Массив, состоящий только из отрицательных чисел;
- Смешанные знаки, наличие нулей и дубликатов;
- Массивы, содержащие `INT_MAX` и `INT_MIN`.

**Характерные примеры входа и ожидаемого выхода:**

1. _Смешанные знаки и нули:_
    - Вход: `[-10, 50, -1, 0, 100, -200, 5]`
    - Выход: `[-200, -10, -1, 0, 5, 50, 100]`
2. _Наличие дубликатов:_
    - Вход: `[5, 1, 5, 2, 1, 5, 5]`
    - Выход: `[1, 1, 2, 5, 5, 5, 5]`
3. _Граничные значения (INT_MAX / INT_MIN):_
    - Вход: `[2147483647, -2147483648, 0, -1, 1]`
    - Выход: `[-2147483648, -1, 0, 1, 2147483647]`

Во всех случаях последовательная версия успешно проходит валидацию.

Разработанная последовательная версия корректно выполняет поразрядную сортировку за линейное время $O(N)$ и успешно

**Почему эта версия используется как эталон для ускорения:**

1. Данная версия представляет собой "чистую" реализацию алгоритма без каких-либо затрат на синхронизацию потоков,
    барьеры или межпроцессные обмены. Она показывает реальное время выполнения $O(N)$ логики на одном ядре.
2. Реализация прошла все функциональные тесты и доказала свою устойчивость. Сравнение результатов параллельных версий
    именно с этим эталоном позволяет гарантировать, что распараллеливание не внесло логических ошибок.
3. В параллельных версиях используется та же вычислительная логика. Использование SEQ-версии в качестве числителя для
    формулы $S = T_1 / T_p$ позволяет точно оценить накладные расходы конкретных технологий (OpenMP, TBB, MPI) и
    определить границы эффективности параллельного кода.
