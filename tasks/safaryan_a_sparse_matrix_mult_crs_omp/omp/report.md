# Подробное описание OpenMP-реализации

## 1. Назначение OMP-версии

OpenMP-реализация предназначена для параллельного умножения разреженных матриц
с элементами типа `double`, представленных в формате CCS. Она сохраняет ту же
модель данных, что и последовательная версия, но распределяет вычисление
столбцов результата между потоками.

Задача состоит в вычислении матрицы `C = A * B`. Входные и выходная матрицы
хранятся в CCS с использованием массивов `values`, `row_indices` и `col_ptrs`.

## 2. Формат CCS

В формате CCS ненулевые элементы группируются по столбцам. Для столбца `j`
диапазон его элементов задается парой границ:

```text
col_ptrs[j] .. col_ptrs[j + 1]
```

Массив `values` содержит сами ненулевые значения, а `row_indices` хранит номера
строк для этих значений. Такой формат удобен для данной задачи, потому что
результат также строится по столбцам.

## 3. Подготовка и проверки

Метод `ValidationImpl` проверяет корректность CCS-структуры обеих матриц:

- размер `col_ptrs` должен быть равен `cols + 1`;
- `values` и `row_indices` должны иметь одинаковую длину;
- первый элемент `col_ptrs` должен быть равен нулю;
- последний элемент `col_ptrs` должен совпадать с числом ненулевых элементов;
- значения `col_ptrs` должны быть неубывающими;
- индексы строк должны находиться в допустимом диапазоне.

Также проверяется основное условие умножения: `A.cols == B.rows`.

Метод `PreProcessingImpl` создает пустую выходную матрицу размера
`A.rows x B.cols`. Перед каждым запуском вычисления результат перестраивается
заново, поэтому старые данные не накапливаются между запусками.

## 4. Последовательная основа алгоритма

Для каждого столбца матрицы `B` вычисляется соответствующий столбец результата.
Временный плотный аккумулятор длины `A.rows` хранит значения текущего столбца
до переноса в CCS.

Если в столбце `j` матрицы `B` найден ненулевой элемент `B(k, j)`, то столбец
`k` матрицы `A` добавляет вклад:

```text
C(i, j) += A(i, k) * B(k, j)
```

После обработки столбца ненулевые значения аккумулятора записываются в
`values` и `row_indices`, а границы столбцов фиксируются в `col_ptrs`.

## 5. Схема OpenMP-параллелизации

Параллелизуется диапазон столбцов результата. Число рабочих частей выбирается
как минимум из количества потоков `PPC_NUM_THREADS` и количества столбцов
матрицы `B`. Диапазон `0 .. B.cols` разбивается на непересекающиеся интервалы.

Директива `#pragma omp parallel for` запускает обработку этих интервалов.
Каждый поток получает свой номер части и вычисляет только принадлежащий ему
диапазон столбцов. Внутри потока создается локальная частичная матрица CCS.

## 6. Shared и private данные

Матрицы `A` и `B`, массив границ частей и контейнер частичных результатов
являются shared-данными. Они доступны всем потокам, но входные матрицы только
читаются. Индекс части, временный аккумулятор и локальная матрица результата
являются private-данными для соответствующего потока.

Каждый поток записывает данные только в свой элемент массива частичных
результатов. Поэтому одновременной записи в одни и те же контейнеры не
происходит.

## 7. Отсутствие гонок и детерминированность

Гонки данных предотвращаются за счет независимых диапазонов столбцов. Потоки
не изменяют общие массивы `values`, `row_indices` и `col_ptrs` итоговой
матрицы во время параллельной секции. Они формируют только локальные частичные
матрицы.

После завершения параллельной секции частичные результаты объединяются одним
потоком в порядке возрастания номеров диапазонов. Благодаря этому порядок
столбцов и порядок элементов внутри CCS-представления остаются
детерминированными.

## 8. Управление числом потоков

В учебной инфраструктуре число потоков задается переменной `PPC_NUM_THREADS`.
Она используется через вспомогательную функцию проекта. При локальном запуске
также можно задать `OMP_NUM_THREADS`, но основным способом для тестов является
`PPC_NUM_THREADS`.

Пример:

```powershell
$env:PPC_NUM_THREADS="4"
```

## 9. Проверка корректности

Функциональные тесты сравнивают результат с заранее заданным CCS-представлением.
Проверяются размеры матрицы, массив `col_ptrs`, массив `row_indices` и значения
`values`. Для `double` используется сравнение с малой допустимой погрешностью.

Набор тестов покрывает квадратные и прямоугольные матрицы, диагональный случай,
умножение на единичную матрицу и результат без ненулевых элементов.

## 10. Производительные тесты

Производительные тесты генерируют случайные разреженные матрицы в формате CCS.
Корректность проверяется через преобразование результата в плотный вид и
сравнение с плотным эталонным умножением.

Тесты запускаются в режимах `pipeline` и `task_run`. Конкретные таблицы
измерений не приводятся, так как результат зависит от локального окружения,
сборки и выбранного числа потоков.

## 11. Команды запуска

Команды для локальной проверки OMP-реализации:

```powershell
cmake --build build --config Release --parallel
$env:PPC_NUM_THREADS="4"
.\build\bin\ppc_func_tests.exe --gtest_filter="*safaryan_a_sparse_matrix_mult_crs_omp*"
.\build\bin\ppc_perf_tests.exe --gtest_filter="*safaryan_a_sparse_matrix_mult_crs_omp*"
```

## 12. Выводы

OpenMP-версия сохраняет CCS-формат данных и распараллеливает вычисление по
независимым диапазонам столбцов. Локальные частичные результаты исключают
гонки данных, а последовательное объединение обеспечивает детерминированный
результат. Реализация пригодна для сравнения с SEQ, TBB, STL и ALL версиями.
