## 1. Контекст

Считаем определённый интеграл функции произвольной размерности по
прямоугольной области методом средних прямоугольников. Схема простая:
каждую ось разбиваем на равные отрезки, в центре каждой ячейки берём
значение функции, складываем все значения и в конце умножаем сумму на
объём ячейки.

Последовательная версия — эталон правильности. С её результатом потом
сверяю все параллельные реализации.

## 2. Постановка задачи

Что подаётся на вход (структура IntegralInput):

- limits — пары чисел (нижняя и верхняя граница) по каждой оси;
- n_steps — число разбиений по каждой оси;
- func — сама функция, которую интегрируем.

Что возвращаем: одно число double — приближённое значение интеграла.

Корректным считается результат, отличающийся от точного не более чем на eps.
В функциональных тестах eps = 1e-4, в перф-тестах 1e-3.

Что отсекает валидация:

- пустую функцию;
- несовпадение длин limits и n_steps, либо вообще пустой вход;
- нулевое или отрицательное число разбиений;
- перевёрнутые границы (нижняя >= верхней).

## 3. Базовый алгоритм

1 - Подготовка
  1.1 Сохраняем входные данные локально и обнуляем сумму
  1.2 Считаем шаг по каждой оси (длина оси делённая на число разбиений)
  1.3 Считаем объём одной ячейки (произведение шагов по всем осям)
  1.4 Заводим вектор индексов idx размером с число осей и заполняем нулями
2 - Обход всех ячеек
  2.1 По текущему набору индексов вычисляем координаты центра ячейки
  2.2 Вычисляем значение функции в этой точке
  2.3 Если значение не конечное (NaN или бесконечность) — останавливаемся и
      возвращаем false
  2.4 Прибавляем значение к общей сумме
  2.5 Двигаем индексы с переносом разряда. «Младшей» здесь зову последнюю
      ось в массиве (idx[dim-1]) — она меняется чаще всех; «старшей» — первую
      (idx[0]).
    2.5.1 Увеличиваем индекс младшей оси (idx[dim-1])
    2.5.2 Если он стал равен числу разбиений — сбрасываем в 0, переходим
          к более старшей (level--)
    2.5.3 Если уехали ниже idx[0] — все ячейки обошли, выходим
3 - Финальный расчёт
  3.1 Умножаем накопленную сумму на объём ячейки — это и есть приближение интеграла
  3.2 Записываем результат в выход задачи

Сложность по времени — пропорциональна общему числу ячеек (произведение всех n_steps).
Сложность по памяти — пропорциональна числу осей (хранится только текущий индекс
и координаты).

## 4. Детали реализации

Файлы:

- seq/include/ops_seq.hpp
- seq/src/ops_seq.cpp

ValidationImpl — отсекает некорректный вход (см. раздел 2). После прохождения
валидации вычисление никогда не падает из-за нелепых данных.

PreProcessingImpl — копирует вход во внутреннее поле и обнуляет аккумулятор.

RunImpl — основной цикл. Реализован без рекурсии: индексы по осям хранятся в
обычном векторе, инкремент с переносом разряда работает как «счётчик», а
размерность задачи может быть любой. Главное условие, ради которого добавлена
проверка isfinite — функция func приходит снаружи и теоретически может вернуть
NaN или бесконечность; такой результат немедленно прерывает работу с возвратом
false, чтобы не получить «мусорное» число. После выхода из цикла накопленная
сумма умножается на объём ячейки `cell_volume`, и итог сохраняется в
внутреннее поле `res_`.

PostProcessingImpl — копирует значение из `res_` в выход задачи через
`GetOutput()`. Никакой дополнительной обработки на этом шаге нет.

```cpp
// File: seq/src/ops_seq.cpp
while (true) {
  for (std::size_t i = 0; i < dim; ++i) {
    point[i] = local_input_.limits[i].first + ((static_cast<double>(idx[i]) + 0.5) * h[i]);
  }
  double f_val = local_input_.func(point);
  if (!std::isfinite(f_val)) return false;
  sum += f_val;

  int level = static_cast<int>(dim) - 1;
  while (level >= 0) {
    idx[level]++;
    if (idx[level] < local_input_.n_steps[level]) break;
    idx[level] = 0;
    level--;
  }
  if (level < 0) break;
}
```

В этом куске видна центральная часть алгоритма: пересчёт координат, вычисление функции,
проверка конечности, прибавление к сумме и перенос разряда между осями.

## 5. Проверка корректности

В tests/functional/main.cpp лежат 10 тестов с допуском eps = 1e-4: константы
по 1, 2 и 3 осям, линейные функции, квадратичная, sin, exp, узкий интервал
[0, 1e-3]. Каждый результат сверяется с заранее известным аналитическим
значением. Несколько характерных кейсов:

- интеграл единицы по [0, 1] равен 1.0 при n = 100;
- интеграл x² по [-1, 2] равен 3.0 при n = 200;
- интеграл sin(x) по [0, π] равен 2.0 при n = 200.

Оба режима показывают практически одно и то же время — это ожидаемо, потому
что в SEQ нет внутренней параллельной фазы, которую pipeline мог бы
скрыть за подготовкой/постобработкой.

Это время является знаменателем для ускорения в других отчётах:
speedup = T_seq / T_parallel, efficiency = speedup / число работников.
В остальных отчётах для удобства взято округлённое baseline = 0.0468 с.

Где тратится время. Почти всё уходит на вызовы `func`.
Индексная арифметика и пересчёт координат вклада практически не дают
(несколько умножений и одно сложение на ячейку).

## 8. Выводы

SEQ-версия идёт как baseline по одной причине: её результат напрямую следует
из определения квадратуры и не зависит от того, как поделена работа. Скрытых
ускорений внутри обхода тоже нет так как это обычный счётчик с переносом разряда,
никаких векторных трюков и спекулятивной оптимизации. Поэтому сравнение
с OMP/TBB/STL/ALL получается честным.
