## 1. Контекст

Здесь работа с потоками выполняется вручную: total_points делится на
блоки, создаются `std::thread`, программа дожидается их завершения
через `join` и складывает результаты. Контроля больше, чем в OMP и TBB,
но и шансов ошибиться больше. Основных источников ошибок два: гонка
на общем аккумуляторе и неправильный порядок create/join — если `join`
вызывать прямо в цикле создания, фактический параллелизм пропадает,
потому что каждый следующий поток ждёт завершения предыдущего.

## 2. Постановка задачи

Та же, что в seq/report.md. Результат сравнивается с SEQ.

## 3. Базовый алгоритм

1 - Подготовка
  1.1 Считаем шаги по осям и объём ячейки
  1.2 Считаем общее число точек total_points
  1.3 Определяем число потоков: берём PPC_NUM_THREADS, но ограничиваем сверху
      total_points (нет смысла создавать поток без работы, если итераций мало)
  1.4 Делим total_points на блоки: каждому потоку — base = total_points /
      thread_count итераций, плюс по одной дополнительной первым нескольким
      потокам, пока не разойдётся остаток
  1.5 Заводим вектор partial_sums размером thread_count, обнулённый
2 - Параллельная работа
  2.1 В цикле создаём все потоки сразу. Каждый поток получает свой диапазон
      [start, end), ссылки на h и вход задачи, плюс адрес своей ячейки в
      partial_sums
  2.2 Внутри потока пробегаем свой диапазон, восстанавливая многомерные
      индексы из линейных, и пишем итог в свою ячейку partial_sums[i]
  2.3 Когда все потоки запущены, в отдельном цикле делаем join на каждом
3 - Сборка результата
  3.1 Складываем все partial_sums в одну сумму
  3.2 Умножаем её на объём ячейки и записываем в выход

## 4. Схема распараллеливания

```cpp
// File: stl/src/ops_stl.cpp
for (int i = 0; i < thread_count; ++i) {
  std::size_t extra = std::cmp_less(static_cast<std::size_t>(i), remainder) ? 1 : 0;
  std::size_t end = start + chunk + extra;
  if (start >= end) {
    partial_sums[i] = 0.0;
    continue;
  }
  ChunkData data = {.start = start, .end = end, .h = &h,
                    .input = &local_input_, .output = &partial_sums[i]};
  threads.emplace_back(ComputeChunkSum, data);
  start = end;
}

for (auto &t : threads) {
  if (t.joinable()) {
    t.join();
  }
}
```

Сначала идёт цикл, который запускает все потоки. Только после того, как они
все запущены, идёт второй цикл, который ждёт их завершения через join.
Если бы join стоял прямо внутри первого цикла, после запуска каждого потока
программа немедленно ждала бы его завершения — то есть фактически потоки
шли бы по очереди, параллелизма не было бы вовсе.

Гонок здесь нет:

- каждый поток пишет в свою собственную ячейку partial_sums[i] — нет даже
  возможности «двое пишут в одно место»;
- координаты точки point хранятся как thread_local внутри рабочей функции —
  у каждого потока он свой и переиспользуется между итерациями;
- общие данные (h, входная структура) только читаются.

## 5. Детали реализации

Файлы:

- stl/include/ops_stl.hpp
- stl/src/ops_stl.cpp

Структура `ChunkData` просто пакует все аргументы рабочей функции в одно
значение, чтобы было удобно передавать его в std::thread.

Распределение нагрузки: первые remainder потоков получают по chunk + 1
итераций, остальные — по chunk. Сумма длин всех блоков равна total_points.

## 9. Выводы

Ручная реализация на std::thread показывает: когда работа в каждой итерации
одинаковая, даже самое простое разбиение «поровну с распределением остатка»
даёт хорошее ускорение. Главный источник накладных расходов здесь это само
создание потоков и их завершение через join. На больших задачах эти расходы
маленькие по сравнению с полезной работой, и кривая ускорения близка к OMP
и TBB.
