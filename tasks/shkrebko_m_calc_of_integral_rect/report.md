- Локальные отчёты: seq/report.md, omp/report.md, tbb/report.md, stl/report.md,

## 1. Введение

Задача — численно вычислить определённый интеграл функции произвольной
размерности по гиперпрямоугольной области с использованием метода средних
прямоугольников. В такой задаче все ячейки независимы: значение функции в
одной точке никак не зависит от значения в другой. Это значит, что её удобно
использовать как тестовое поле для сравнения разных моделей параллельного
программирования — работа делится на куски естественно, и видно, как ведёт
себя каждая технология на одной и той же нагрузке.

## 2. Единая постановка задачи

На вход подаётся структура IntegralInput: пары границ по каждой оси, число
разбиений по каждой оси и сама функция. На выходе — одно число double,
приближённое значение интеграла.

Корректным считается результат, отличающийся от точного не более чем на eps.
В функциональных тестах eps = 1e-4, в перф-тестах 1e-3.

Валидация отсекает пустую функцию, несогласованные размеры, нулевые или
отрицательные шаги и перевёрнутые границы.

## 3. Единая методика эксперимента

- Окружение: CPU AMD Ryzen 7 9700X (8 ядер / 16 потоков), RAM 32 ГБ,
  OS Windows 10 Pro (10.0.19045), компилятор MSVC (Visual Studio 17 2022),
  тип сборки Release.
- MPI: Microsoft MPI.
- Переменные окружения: PPC_NUM_THREADS — число потоков для OMP/TBB/STL/ALL,
  PPC_NUM_PROC (или ключ `mpiexec -n N`) — число процессов для MPI,
  OMP_NUM_THREADS выставляется runner-ом из PPC_NUM_THREADS.
- Данные: тестовые функции с известным точным ответом задаются прямо в
  tests/functional/main.cpp. Перф-тест единственный — 3D интеграл функции
  sin(x) + cos(y) + z по единичному кубу с 200 разбиениями на каждой оси
  (всего 8 миллионов точек).
- Формулы:
  - ускорение = время SEQ / время параллельной версии;
  - эффективность = ускорение / число работников;
  - для OMP/TBB/STL работник — это поток, для ALL — процессы умножить
    на потоки внутри процесса.
- Повторы: 5 прогонов на каждую конфигурацию, в таблице приведены медианы.
  Прогрев перф-каркаса встроен в каждый прогон (первая итерация внутри
  PerfRun отбрасывается самим каркасом).

## 4. Сводка корректности

Все пять реализаций прогоняются через общий каркас функциональных тестов
ppc::util::BaseRunFuncTests и общий список тестовых случаев (10 функций:
константы по 1, 2 и 3 осям; линейные функции; квадратичная; sin; exp;
узкий интервал). Допуск 1e-4 одинаков для всех. Расхождений с эталоном не
зафиксировано.

Перф-тест отдельно сверяет результат каждой реализации с аналитическим
значением 3D интеграла с допуском 1e-3.

Особенность: в ALL-версии итоговое значение есть только на rank = 0,
поэтому CheckTestOutputData проверяет результат только на нулевом процессе,
а на остальных возвращает true.

## 6. Сравнение технологий

- **SEQ.** Базовая версия, никакого параллелизма нет. Время 0.0468 с —
  точка отсчёта для всех остальных.
- **OMP.** На 1 потоке даёт 0.76× от SEQ — то есть в 1.3 раза медленнее.
  Причина — переход с SEQ-овского счётчика с переносом разряда на
  плоский 1D-цикл с `%` и `/`. С ростом числа потоков OMP отыгрывается,
  но на 8 потоках выходит только на 1.75×, и эффективность уже 22%.
- **TBB.** Единственная технология, которая показала монотонное
  ускорение во всём диапазоне 1-8 потоков. На 1 потоке практически
  совпадает с SEQ (0.99×), на 8 потоках — лучший результат среди всех:
  3.71×, эффективность 46%.
- **STL.** Самая многословная и низкоуровневая версия: ручное деление
  диапазона, ручное создание потоков, ручной join. На 1 потоке —
  0.77× от SEQ. На 8 потоках лучшее ускорение 2.23× (task_run), 
  но измерения шумные: на 2 потоках разброс между min и max доходил до 70%.
- **ALL.** Из-за обмена данными между процессами на задаче такого
  размера почти не масштабируется: с 1 работника до 8 ускорение
  выросло только с 1.05× до 1.25×. Хорошо видно, что добавление
  OpenMP-потоков внутри ранга (2×1 → 2×2) почти не даёт прироста —
  потокам не хватает работы. На больших задачах эта картина должна
  выровняться.


## 7. Заключение

На задаче многомерного интегрирования методом средних прямоугольников
TBB оказался явным лидером по масштабированию: 3.71× на 8 потоках при
эффективности 46%. OMP и STL дают близкое друг другу более скромное
ускорение (1.75× и 2.23× соответственно), причём STL заметно шумнее по
измерениям. Гибридная версия ALL на размере задачи 8 миллионов точек
не показывает выигрыша от добавления процессов и потоков — стоимость
коммуникации сравнима с полезной работой.
