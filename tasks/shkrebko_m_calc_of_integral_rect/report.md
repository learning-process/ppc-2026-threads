## 1. Введение

Считаю определённый интеграл функции произвольной размерности по
гиперпрямоугольной области методом средних прямоугольников. Ячейки сетки
независимы — значение функции в одной не зависит от другой — поэтому
задача хорошо подходит для сравнения параллельных моделей: работа
делится естественным образом, и поведение каждой технологии на одной и
той же нагрузке наблюдается напрямую.

## 2. Единая постановка задачи

На вход подаётся структура IntegralInput: пары границ по каждой оси, число
разбиений по каждой оси и сама функция. На выходе — одно число double,
приближённое значение интеграла.

Корректным считается результат, отличающийся от точного не более чем на eps.
В функциональных тестах eps = 1e-4, в перф-тестах 1e-3.

Валидация отсекает пустую функцию, несогласованные размеры, нулевые или
отрицательные шаги и перевёрнутые границы.

## 3. Единая методика эксперимента

- Окружение: CPU AMD Ryzen 7 9700X (8 ядер / 16 потоков), RAM 32 ГБ,
  OS Windows 10 Pro (10.0.19045), компилятор MSVC (Visual Studio 17 2022),
  тип сборки Release.
- MPI: Microsoft MPI.
- Переменные окружения: PPC_NUM_THREADS — число потоков для OMP/TBB/STL/ALL,
  PPC_NUM_PROC (или ключ `mpiexec -n N`) — число процессов для MPI,
  OMP_NUM_THREADS выставляется runner-ом из PPC_NUM_THREADS.
- Данные: тестовые функции с известным точным ответом задаются прямо в
  tests/functional/main.cpp. Перф-тест единственный — 3D интеграл функции
  sin(x) + cos(y) + z по единичному кубу с 200 разбиениями на каждой оси
  (всего 8 миллионов точек).
- Формулы:
  - ускорение = время SEQ / время параллельной версии;
  - эффективность = ускорение / число работников;
  - для OMP/TBB/STL работник — это поток, для ALL — процессы умножить
    на потоки внутри процесса.
- Повторы: на каждую конфигурацию — 5 запусков перф-теста, в таблице
  приведены медианы. Сам `Perf::TaskRun` / `Perf::PipelineRun` внутри
  выполняет задачу несколько раз и возвращает усреднённое время одной
  итерации.

## 4. Сводка корректности

Все пять реализаций прогоняются через общий каркас функциональных тестов
ppc::util::BaseRunFuncTests и общий список тестовых случаев (10 функций:
константы по 1, 2 и 3 осям; линейные функции; квадратичная; sin; exp;
узкий интервал). Допуск 1e-4 одинаков для всех. Расхождений с эталоном не
зафиксировано.

Перф-тест отдельно сверяет результат каждой реализации с аналитическим
значением 3D интеграла с допуском 1e-3.

Особенность: в ALL-версии итоговое значение есть только на rank = 0,
поэтому CheckTestOutputData проверяет результат только на нулевом процессе,
а на остальных возвращает true.

## 6. Сравнение технологий

**SEQ.** 0.0468 с. Baseline, точка отсчёта.

**OMP.** На 1 потоке OMP-версия медленнее SEQ (0.76×).
Дело не в OpenMP — для параллельного цикла пришлось переписать обход
индексного пространства в плоский 1D-цикл с восстановлением координат
через `%` и `/`. Это дороже SEQ-овского счётчика с переносом разряда.
С 2 потоков начинается выигрыш, но на 8 потоках выходим только на 1.75×
при эффективности 22%.

**TBB.** Единственная из четырёх параллельных реализаций, которая
масштабируется монотонно от 1 до 8 потоков. На 1 потоке — фактически
SEQ (0.99×), на 8 — 3.71× при эффективности 46%. Лучший результат во
всех замерах.

**STL.** Та же арифметика, что и в OMP, только потоки создаются и
синхронизируются вручную. На 1 потоке — 0.77× от SEQ (стоимость одного
`std::thread` и `join`). Дальше неравномерно: на 4 потоках pipeline даёт
1.73×, а task_run всего 1.25× в той же конфигурации. По 5 прогонам
разброс на 2 потоках доходил до 70% — заметно шумнее, чем у TBB.

**ALL.** На нашем размере почти не масштабируется: с 1
работника до 8 выходит от 1.05× до 1.25×. MPI-обмен и старт `mpiexec` едят
весь выигрыш. Видно, что переход 2×1 → 2×2 уже почти ничего не даёт —
потокам внутри ранга не хватает работы.

## 7. Заключение

По результатам замеров TBB здесь оказывается лучшим из четырёх
параллельных вариантов: 3.71× на 8 потоках, эффективность 46%,
монотонное масштабирование. OMP и STL дают более скромное и близкое
друг к другу ускорение (1.75× и 2.23×), причём измерения STL заметно
шумнее. ALL на размере задачи 8 млн точек не выигрывает у TBB на тех
же 8 потоках — обмен между процессами сводит на нет потенциальный
прирост.

На другом железе и для другой подынтегральной функции картина может
отличаться.
