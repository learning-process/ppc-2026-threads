## 1. Контекст

В библиотеке работа описывается как набор подзадач над общим диапазоном.
Сама библиотека потом распределит эти подзадачи между потоками — и,
если одному потоку достанется блок медленнее, другие могут забрать
у него часть работы. Для нашей задачи с одинаковой нагрузкой
на каждой итерации хорошо подходит готовый шаблон parallel_reduce:
он сам делит диапазон и сам собирает частичные суммы.

## 2. Постановка задачи

Та же, что в seq/report.md. Результат сравнивается с SEQ.

## 3. Базовый алгоритм

1 - Подготовка
  1.1 Считаем шаги по осям и объём ячейки
  1.2 Считаем общее число точек total_points (произведение n_steps)
  1.3 Через tbb::global_control ограничиваем число одновременно работающих
      потоков значением из ppc::util::GetNumThreads()
2 - Параллельный подсчёт суммы через parallel_reduce
  2.1 Задаём диапазон работы от 0 до total_points как blocked_range — это
      просто «отрезок, который можно делить на части»
  2.2 Для каждого блока TBB вызывает нашу лямбду, передавая начало и конец
      блока, а также текущую частичную сумму
  2.3 Внутри лямбды зовём ComputeBlockSum — она пробегает блок и возвращает
      сумму значений функции по этому блоку
  2.4 TBB сама собирает все локальные суммы в одну через std::plus<>()
3 - Финальный результат
  3.1 Умножаем итоговую сумму на объём ячейки
  3.2 Записываем в выход задачи

## 4. Схема распараллеливания

Главный вызов выглядит так:

```cpp
// File: tbb/src/ops_tbb.cpp
int num_threads = ppc::util::GetNumThreads();
tbb::global_control global_limit(tbb::global_control::max_allowed_parallelism, num_threads);

double total_sum = tbb::parallel_reduce(
    tbb::blocked_range<std::size_t>(0, total_points), 0.0,
    [&](const tbb::blocked_range<std::size_t> &range, double partial_sum) {
      return partial_sum + ComputeBlockSum(range.begin(), range.end(), h);
    },
    std::plus<>());
```

Что здесь происходит:

- `blocked_range(0, total_points)` задаёт диапазон от 0 до `total_points`,
  допускающий разбиение на части. Размер блоков явно не задаётся — TBB
  подбирает его автоматически.
- `parallel_reduce` — шаблон параллельного обхода с накоплением. Для
  каждого блока диапазона он вызывает лямбду, а в конце складывает все
  частичные суммы через `std::plus<>()`.
- `global_control` с `max_allowed_parallelism` — корректный способ
  ограничить число рабочих потоков в TBB. В отличие от прямого задания
  количества потоков, этот вариант указывает самой библиотеке верхнюю
  границу одновременно выполняющихся задач.

Внутри `ComputeBlockSum` координаты центра ячейки восстанавливаются один
раз по стартовому индексу блока, далее они смещаются переносом разряда.
Это убирает лишние операции деления внутри горячего цикла.

## 5. Детали реализации

Файлы:

- tbb/include/ops_tbb.hpp
- tbb/src/ops_tbb.cpp

В `RunImpl` готовятся вектор шагов `h`, скалярные `cell_volume` и
`total_points`. Дальше `global_control` ограничивает число потоков, и
запускается `parallel_reduce`. Грязная арифметика с восстановлением
индексов вынесена в `ComputeBlockSum`, чтобы основная функция оставалась
читаемой.

Гонок здесь нет по построению: лямбда не пишет в общую память. Все частичные
суммы возвращаются как значение из лямбды и складываются библиотекой
автоматически — блокировки и атомарные операции не нужны.

TBB здесь масштабируется монотонно от 1 до 8 потоков — у OMP и STL так
не получилось. На 1 потоке время практически совпадает с SEQ — накладные
расходы на запуск `parallel_reduce` минимальны. На 8 потоках получаем
3.71× при эффективности 46% — лучший результат среди всех замеров.
Размер блоков явно не задавался, `auto_partitioner` сам подобрал
подходящее значение.

## 9. Выводы

Для однородной независимой работы, TBB оказался
наиболее удобным из четырёх реализаций: хорошая
масштабируемость без ручной настройки параметров. На очень маленьких
задачах накладные расходы `parallel_reduce` могут превысить выигрыш от
параллелизма, но при размере 8 млн точек этого не происходит.
