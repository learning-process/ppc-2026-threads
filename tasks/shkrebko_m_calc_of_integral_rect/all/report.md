## 1. Контекст

В ALL-версии два уровня параллелизма: MPI между процессами и OpenMP
внутри каждого процесса. Эта схема имеет смысл прежде всего для запусков
на нескольких узлах или сокетах. Для данной задачи разрез выполняется по
первой оси: каждый MPI-ранг получает свою «полосу» шагов оси 0, а внутри
неё OpenMP параллелит цикл по этой же оси. По остальным осям внутри
каждого шага идёт последовательный обход.

## 3. Базовый алгоритм

1 - Узнаём свою роль в MPI
  1.1 Получаем номер процесса rank и общее число процессов size
  1.2 Если запуск не под mpirun — считаем, что rank = 0 и size = 1
2 - Рассылка общих данных
  2.1 Процесс с rank = 0 уже знает все параметры задачи
  2.2 Остальным процессам через MPI_Bcast передаём размерность задачи и
      параметры всех осей, кроме первой (число шагов и границы)
3 - Распределение работы по оси 0
  3.1 Считаем, сколько шагов оси 0 должно достаться каждому процессу:
      всего шагов поделить на число процессов; остаток — по одному
      дополнительному шагу первым нескольким процессам
  3.2 Считаем свои локальные границы по оси 0
4 - Локальный подсчёт через OpenMP
  4.1 Запускаем параллельный цикл по локальным шагам оси 0 с reduction
  4.2 Внутри каждой итерации идёт обычный последовательный обход по
      остальным осям, накапливающий сумму
  4.3 После выхода из параллельного цикла умножаем накопленную сумму на
      шаг по оси 0 и на произведение шагов по остальным осям — получаем
      локальный вклад процесса в интеграл (это и есть значение, которое
      ComputeSliceSum возвращает)
5 - Сборка результата
  5.1 Через MPI_Reduce собираем все локальные вклады на процессе с
      rank = 0, складывая их в итоговое значение интеграла
  5.2 Только rank = 0 пишет итог в выход задачи. Никакого дополнительного
      умножения на объём ячейки тут уже не нужно — он целиком зашит в
      локальные вклады на предыдущем шаге

## 4. Уровень между процессами (MPI)

```cpp
// File: all/src/ops_all.cpp
int rank = 0, size = 1;
const bool is_mpi = ppc::util::IsUnderMpirun();
if (is_mpi) {
  MPI_Comm_rank(MPI_COMM_WORLD, &rank);
  MPI_Comm_size(MPI_COMM_WORLD, &size);
}
BroadcastCommonData(rank);

double my_left = 0.0, my_right = 0.0;
int my_steps = 0, my_offset = 0;
AssignMpiSlice(rank, size, my_left, my_right, my_steps, my_offset);

const double local_slice =
    ComputeSliceSum(my_left, my_right, my_steps, h_other, limits_other, n_steps_other);

double global_integral = 0.0;
if (is_mpi) {
  MPI_Reduce(&local_slice, &global_integral, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
  if (rank == 0) res_ = global_integral;
} else {
  res_ = local_slice;
}
```

По шагам:

- MPI_Comm_rank / MPI_Comm_size — стандартный способ узнать свой номер и
  общее число процессов.
- BroadcastCommonData — рассылает всем процессам параметры осей кроме первой.
  Параметры первой оси у всех общие изначально, а свою долю каждый процесс
  считает сам.
- AssignMpiSlice — режет число шагов оси 0 на size частей с равномерным
  распределением остатка. Это даёт максимально равное число шагов даже
  когда делится неровно.
- MPI_Reduce — собирает локальные значения local_slice со всех процессов
  на rank = 0 через сложение. Отдельный барьер не нужен: сам Reduce уже
  ждёт, пока все процессы дойдут до него.

## 5. Уровень внутри процесса (OpenMP)

Внутри ComputeSliceSum параллелится внешний цикл по локальным шагам оси 0:

```cpp
// File: all/src/ops_all.cpp
#pragma omp parallel for default(none) \
    shared(left0, h0, steps0, h_other, limits_other, n_steps_other, other_dims) \
    reduction(+ : total) schedule(static)
for (int i = 0; i < steps0; ++i) {
  const double x0 = left0 + ((static_cast<double>(i) + 0.5) * h0);

  double local_sum = 0.0;
  std::size_t total_other_points = 1;
  for (std::size_t dim = 0; dim < other_dims; ++dim) {
    total_other_points *= n_steps_other[dim];
  }
  for (std::size_t idx = 0; idx < total_other_points; ++idx) {
    std::vector<double> point(other_dims + 1);
    point[0] = x0;
    local_sum += local_input_.func(point);
  }
  total += local_sum;
}
// после выхода из параллельной области:
return total * h0 * volume_other;
```

- default(none) — заставляет явно перечислить все используемые переменные.
- reduction(+ : total) — каждый поток ведёт свою копию total, в конце они
  складываются автоматически.
- schedule(static) — итерации делятся между потоками равными кусками. Это
  годится, потому что работа в каждой итерации одинаковая.

После закрытия параллельной области накопленное `total` ещё не является
вкладом в интеграл — это просто сумма значений функции по ячейкам данного
среза. Чтобы получить именно вклад в интеграл, перед `return` сумма
умножается на шаг по оси 0 и на произведение шагов по остальным осям
(`return total * h0 * volume_other;`). Это и есть значение, которое потом
уходит в `MPI_Reduce`.

В одномерном случае распараллеливание через OpenMP не
включается: работа линейна и обычно невелика, OpenMP добавил бы только
накладных расходов. В этой ветке умножение на `h0` тоже происходит перед
возвратом, просто без вектора `h_other`.

## 6. Детали реализации

Файлы:

- all/include/ops_all.hpp
- all/src/ops_all.cpp

RunImpl делает шесть шагов:
1 - Узнаёт rank и size
2 - Через BroadcastCommonData рассылает параметры осей с номерами 1 и выше
3 - Через AssignMpiSlice определяет свой кусок оси 0
4 - Заполняет вспомогательные векторы h_other, limits_other, n_steps_other
5 - Считает свой вклад в интеграл через ComputeSliceSum
6 - Через MPI_Reduce собирает результат на rank = 0

PostProcessingImpl пишет результат только на rank = 0 — на остальных
процессах выходное поле остаётся пустым.

Ускорение ALL почти не растёт: с 1 работника до 8 оно увеличивается

на 2×2: добавление OpenMP-потока внутри ранга почти не повлияло на
время. Потокам внутри полосы из 100 шагов оси 0 недостаточно работы,
чтобы окупить распараллеливание.

Чтобы гибридная схема показала прирост, нужна задача большего размера,
например, n = 400 по каждой оси (64 млн точек), либо более дорогая
подынтегральная функция.

## 10. Выводы

На текущем размере задачи ALL проигрывает чистым потоковым реализациям, но
MPI-обмен присутствует в любом случае, а полезной работы недостаточно,
чтобы его окупить. По существу гибридная схема оправдана в двух случаях:
когда требуется запуск на нескольких узлах кластера, либо когда
расчёт одной итерации заметно дороже, тогда стоимость `MPI_Reduce`
становится незначительной относительно самих вычислений.
