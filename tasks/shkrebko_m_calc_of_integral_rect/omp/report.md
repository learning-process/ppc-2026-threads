## 1. Контекст

Значения функции в разных ячейках сетки считаются независимо, итерации
никак не связаны между собой. Это удобный случай для OpenMP: достаточно
одной директивы `parallel for` с `reduction`, чтобы перейти от SEQ к
параллельной версии без atomic и без ручного ведения локальных сумм.

## 2. Постановка задачи

Та же, что в seq/report.md. Эталоном правильности служит SEQ.

## 3. Базовый алгоритм

1 - Подготовка
  1.1 Считываем границы и числа разбиений
  1.2 Считаем шаги по каждой оси и объём ячейки
  1.3 Считаем общее число точек total_points (произведение всех n_steps) —
      это длина одного плоского цикла
  1.4 Выставляем число потоков: omp_set_num_threads(GetNumThreads())
2 - Параллельный обход всех ячеек
  2.1 Заводим общий аккумулятор total_sum и говорим OpenMP считать его как
      reduction (каждый поток получит свою копию, в конце они сложатся вместе)
  2.2 Запускаем цикл по линейному индексу idx от 0 до total_points
  2.3 Внутри цикла восстанавливаем многомерные индексы из линейного через
      деление и остаток
  2.4 По индексам считаем координаты центра ячейки
  2.5 Прибавляем значение функции к своей копии total_sum
3 - Сборка результата
  3.1 По выходе из параллельной области OpenMP сам складывает все частичные
      суммы в одно число
  3.2 Умножаем итог на объём ячейки и записываем в выход

В отличие от SEQ обход индексного пространства переведён в плоский одномерный
цикл. Это сделано специально: счётчик с переносом разряда из SEQ — это
итерации, зависящие друг от друга (каждая следующая определяется через
предыдущую), и OpenMP такой цикл просто не сможет распараллелить. Линейный
индекс убирает зависимость, и каждая итерация становится самодостаточной.

## 4. Схема распараллеливания

```cpp
// File: omp/src/ops_omp.cpp
omp_set_num_threads(ppc::util::GetNumThreads());

#pragma omp parallel default(none) \
    shared(total_points, h, limits, n_steps, func, dims) \
    reduction(+ : total_sum)
{
  std::vector<double> point(dims);

#pragma omp for
  for (std::int64_t idx = 0; idx < total_points; ++idx) {
    std::int64_t tmp = idx;
    for (int dim = static_cast<int>(dims) - 1; dim >= 0; --dim) {
      const int coord_index = static_cast<int>(tmp % n_steps[dim]);
      tmp /= n_steps[dim];
      point[dim] = limits[dim].first + ((static_cast<double>(coord_index) + 0.5) * h[dim]);
    }
    total_sum += func(point);
  }
}
```

По частям директивы:

- `default(none)` — страховка от неявных правил. Если забыть перечислить
  переменную, компилятор выдаст ошибку, а не сделает её общей по
  умолчанию. На больших задачах без этого можно получить незаметную
  гонку данных.
- `shared(...)` — всё, что в списке, общее. Но переменные только читаются,
  гонок нет.
- `reduction(+ : total_sum)` — ключевая часть. Каждый поток получает
  собственную копию `total_sum`, накапливает её независимо, а в конце
  OpenMP сам складывает все копии. Без этого пришлось бы использовать
  либо atomic, либо вручную поддерживать локальные суммы с последующей
  редукцией.
- `point` объявлен внутри parallel-региона, поэтому он private у
  каждого потока, поэтому отдельно помечать его не требуется.

`schedule` явно не указан. По умолчанию используется реализационно
зависимое значение (обычно `static`). Для данной задачи
это подходит: работа в каждой итерации одинаковая, балансировать
нечего. Если бы итерации существенно различались по стоимости, имело
бы смысл попробовать `schedule(dynamic)`.

После закрытия parallel-региона OpenMP сам ждёт завершения всех потоков —
к этому моменту `total_sum` уже собран.

Runner курса экспортирует `PPC_NUM_THREADS` и дублирует его в
`OMP_NUM_THREADS`, так что конфигурация управляется одной переменной
окружения.

## 5. Детали реализации

Файлы:

- omp/include/ops_omp.hpp
- omp/src/ops_omp.cpp

ValidationImpl и PreProcessingImpl совпадают с SEQ. Изменения только в RunImpl —
там обход индексного пространства переписан в одномерной форме и обёрнут
в директиву.

Все возможные гонки устранены по построению: общие данные читаются, но не
пишутся; сумма копится через reduction; буфер координат — локальный для
каждого потока.

Комментарий по масштабируемости. На 1 потоке OMP-версия медленнее SEQ
примерно в 1.3 раза: причина — плоский 1D-цикл с восстановлением многомерных
индексов через `%` и `/`, который чуть дороже SEQ-овского счётчика с
переносом разряда. С ростом числа потоков версия начинает выигрывать, но
эффективность падает быстро — на 8 потоках уже около 21%.

## 9. Выводы

OpenMP даёт выигрыш, когда объём полезной работы существенно превышает
стоимость запуска параллельной области и сборки редукции. В данной
задаче (8 млн точек) выигрыш есть, но скромнее, чем у TBB: 1.75× против
3.71× на 8 потоках. Причина не в самом OpenMP — обоим backend-ам
требуется плоский 1D-цикл с восстановлением координат через `%` и `/`.
Разница объясняется стоимостью запуска parallel-региона на каждом
прогоне, которая у OMP оказывается заметнее.
