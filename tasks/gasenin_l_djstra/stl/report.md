## 1. Введение

Данный отчёт описывает реализацию алгоритма Дейкстры с использованием `std::thread`
из стандартной библиотеки C++. В отличие от OpenMP и TBB, `std::thread` не предоставляет
готовых примитивов параллельных циклов, поэтому потоки управляются вручную: создаётся
постоянный пул рабочих потоков, которые через `condition_variable` получают фазы
`kFindMin` и `kRelax`, выполняют работу и сигнализируют главному потоку о завершении.

## 2. Постановка задачи

Дан неориентированный полносвязный граф с $n$ вершинами.

- Вес ребра между вершинами $u$ и $v$ равен $|u - v|$.
- **Входные данные:** количество вершин $n$.
- **Выходные данные:** сумма длин кратчайших путей от вершины 0 до всех остальных.
- **Ограничения:** $n > 0$. Теоретическая сумма: $\frac{n(n-1)}{2}$.

## 3. Схема параллельного алгоритма

### 3.1 Архитектура пула потоков

Потоки создаются в `PreProcessingImpl` и живут до вызова `PostProcessingImpl`, чтобы
избежать накладных расходов на повторное создание. Синхронизация основана на трёх полях:

- `generation_` — монотонно растущий счётчик поколений; рабочий поток блокируется
  на `cv_start_`, пока `generation_` не превысит его собственный `my_gen`.
- `phase_` — текущая фаза (`kFindMin`, `kRelax`, `kStop`).
- `pending_` — число ещё не завершивших работу потоков.

Главный поток вызывает `Dispatch(phase)`, который атомарно обновляет `phase_`,
сбрасывает `pending_ = num_threads_`, увеличивает `generation_` и будит всех рабочих.
Затем он блокируется на `cv_done_` до `pending_ == 0`.

### 3.2 Распределение работы (карта поток → блок данных)

Каждый поток получает полосу индексов с шагом `num_threads_`:

```text
поток 0: 0, T, 2T, 3T, ...
поток 1: 1, T+1, 2T+1, ...
поток k: k, T+k, 2T+k, ...
```

Такая полосатая (striped) схема обеспечивает равномерное распределение нагрузки
без вычисления границ блоков. Локальные результаты каждого потока хранятся в
`local_min_[thread_id]` и `local_vert_[thread_id]` — отдельных элементах разделяемых
векторов, что исключает гонки при записи.

### 3.3 Ключевые фрагменты

Файл: `stl/src/ops_stl.cpp`.

**Запуск потоков и `join`:**

```cpp
// PreProcessingImpl — запуск всех рабочих потоков
workers_.resize(num_threads_);
for (int thread_id = 0; thread_id < num_threads_; ++thread_id) {
    workers_[thread_id] = std::thread(&GaseninLDjstraSTL::WorkerLoop, this, thread_id);
}

// PostProcessingImpl — ожидание завершения всех потоков
{
    std::scoped_lock<std::mutex> lock(mtx_);
    phase_ = Phase::kStop;
    ++generation_;
    cv_start_.notify_all();
}
for (auto &worker : workers_) {
    worker.join();  // join вызывается для каждого потока после kStop
}
workers_.clear();
```

`join` вызывается после отправки сигнала `kStop` всем рабочим; все потоки
корректно завершаются до уничтожения объекта задачи. Это гарантирует отсутствие
use-after-free при доступе к полям класса из рабочих потоков.

**Цикл рабочего потока:**

```cpp
void GaseninLDjstraSTL::WorkerLoop(int thread_id) {
    int my_gen = 0;
    while (true) {
        Phase current_phase{};
        {
            std::unique_lock<std::mutex> lock(mtx_);
            cv_start_.wait(lock, [&] { return generation_ > my_gen; });
            my_gen        = generation_;
            current_phase = phase_;
        }
        if (current_phase == Phase::kStop) return;
        if (current_phase == Phase::kFindMin)
            DoFindMin(thread_id);
        else
            DoRelax(thread_id);
        {
            std::scoped_lock<std::mutex> lock(mtx_);
            if (--pending_ == 0) cv_done_.notify_one();
        }
    }
}
```

## 4. Детали реализации

- **Заголовочный файл:** `stl/include/ops_stl.hpp`.
- **Файл реализации:** `stl/src/ops_stl.cpp`.
- **Число потоков:** `ppc::util::GetNumThreads()`.
- **Синхронизация:** один `std::mutex` + два `std::condition_variable`
  (`cv_start_`, `cv_done_`). Счётчики `pending_` и `generation_` защищены этим
  мьютексом. `atomic` не используется — все обновления происходят под lock.
- **`join`:** вызывается в `PostProcessingImpl` после отправки фазы `kStop`;
  гарантирует завершение всех потоков до выхода из области видимости объекта.
- **Отсутствие гонок:** фаза `kFindMin` — каждый поток пишет только в
  `local_min_[thread_id]` (уникальный индекс); фаза `kRelax` — каждый поток
  обновляет `dist_[vertex]` для уникального `vertex` (полосатое разбиение).
- **Память:** $O(n)$ для `dist_`, $O(n)$ для `visited_`, $O(T)$ для `local_min_`
  и `local_vert_`. Матрица смежности не создаётся.

## 6. Результаты

### 6.1 Корректность

STL-версия прошла все функциональные тесты для $n \in \{3, 5, 7\}$, результаты совпали
с SEQ-эталоном. Корректность схемы синхронизации дополнительно проверена запуском
с разным числом MPI-процессов (2, 4, 6): расхождений в выходных данных не наблюдалось.

### 6.2 Производительность

Определения столбцов таблицы:

- **Режим** — сценарий (`task_run` / `pipeline`).
- **MPI-процессов** — число MPI-рангов, каждый содержит свой пул `std::thread`.
- **Время, с** — время выполнения по фреймворку.
- **Ускорение** — $S = T_{\text{seq}} / T_{\text{stl}}$, baseline SEQ = 0.01629 с.
- **Эффективность** — не вычисляется, так как число потоков динамическое (—).

| Режим     | MPI-процессов | Время, с | Ускорение |
|-----------|---------------|----------|-----------|
| task\_run | 2             | 0.000409 | 39.8×     |
| task\_run | 4             | 0.000552 | 29.5×     |
| task\_run | 6             | 0.002608 | 6.24×     |
| pipeline  | 2             | 0.004962 | 3.28×     |
| pipeline  | 4             | 0.008029 | 2.03×     |
| pipeline  | 6             | 0.011995 | 1.36×     |

**Анализ.** STL-версия показывает наивысшее ускорение среди всех технологий при
двух MPI-процессах (39.8×). Постоянный пул потоков устраняет расходы на создание
и уничтожение потоков между итерациями, а полосатая раздача работы минимизирует
простой. При росте числа MPI-процессов ускорение снижается: несколько пулов конкурируют
за ядра CPU. Режим `pipeline` добавляет оверхед конвейерной обёртки фреймворка.

## 7. Выводы

`std::thread`-реализация является наиболее производительной при двух MPI-процессах
(39.8× в `task_run`). Постоянный пул с фазовой диспетчеризацией через
`condition_variable` эффективнее разовых вызовов `parallel_for` для задачи
с большим числом коротких итераций. Основное ограничение — чувствительность к
числу активных MPI-процессов на одной машине.
