## 1. Введение

В этой задаче реализовано численное интегрирование функции d переменных по прямоугольной области методом средней точки
(rectangle/midpoint rule). Написаны последовательная версия и четыре параллельные реализации: OpenMP, oneTBB,
`std::thread` и гибридная MPI + OpenMP версия в каталоге `all`.

Все реализации работают с одним типом входных данных и проверяются общими тестами из `tests/functional/main.cpp` и
`tests/performance/main.cpp`. Поэтому в отчёте сравниваются именно технологии распараллеливания, а не разные постановки
задачи.

## 2. Единая постановка задачи

**Вход:** функция `f : ℝᵈ → ℝ`, вектор пар границ `{(aᵢ, bᵢ)}`, целое `n > 0`.  
**Выход:** вещественное число — приближение интеграла.

**Корректность:** совпадение с последовательным эталоном (SEQ) на детерминированных тестах с допуском 1e-4.

**Ограничения:** `borders` непустой; все `aᵢ < bᵢ` конечны; `func` не null; `n > 0`.

## 3. Базовый алгоритм

По каждой оси строятся `n` равных ячеек шириной `hᵢ = (bᵢ − aᵢ) / n`. Объём ячейки: `V_cell = ∏ hᵢ`.
Центр ячейки с индексом `(i₀, …, iₐ₋₁)`: `xⱼ = aⱼ + (iⱼ + 0.5) × hⱼ`.

```text
I ≈ (Σ_{все ячейки} f(центр ячейки)) × V_cell
```

Линейный индекс ячейки: `linear_idx ∈ [0, nᵈ)`, декодируется через `% n` и `/ n` по каждой оси.

**Сложность:** O(nᵈ) вычислений `f` — объём d-мерной сетки.

## 4. Реализации

| Реализация | Файл | Технология | Ключевая особенность |
| ---------- | ---- | ---------- | --------------------- |
| SEQ | `seq/src/ops_seq.cpp` | Последовательная | Эталон; `NextIndex()` итератор; `x[]` вне цикла |
| OMP | `omp/src/ops_omp.cpp` | OpenMP | `parallel for` + `reduction`; `x[]` внутри цикла |
| TBB | `tbb/src/ops_tbb.cpp` | oneTBB | `parallel_reduce`; `x[]` вне внутреннего цикла |
| STL | `stl/src/ops_stl.cpp` | `std::thread` | Ручное блочное разбиение; join после всех spawn |
| ALL | `all/src/ops_all.cpp` | MPI + OpenMP | Двухуровневый: MPI-блоки + OMP внутри ранга |

## 5. Единая методика эксперимента

**Окружение:** CPU Apple M4 Pro, 12 физических ядер, 24 GB RAM, macOS 26.3.1, Apple clang 21.0.0, сборка Release.

**Переменные окружения:** `PPC_NUM_THREADS` задаёт число потоков для OMP и STL; TBB игнорирует эту переменную и
использует весь пул; для MPI — `mpirun -n N`.

**Данные:** 2D функция sin(x₀)·sin(x₁) на [0,π]², `n = 10 000` (total\_cells = 10⁸).

**Метрики:**

- Ускорение относительно SEQ: `S = T_seq / T_impl` (один и тот же размер задачи).
- Эффективность для потоковых версий: `E = S / T_раб`, где `T_раб` — число потоков.

**Режимы замера:**

- **task\_run** — время только `RunImpl()`;
- **pipeline** — полный цикл `Validation → PreProcessing → Run → PostProcessing`.

Для сравнения реализаций используется `task_run`.

## 6. Сводка корректности

Все включённые в `settings.json` реализации (seq, omp, tbb, stl, all) проверяются одним набором из 23 функциональных
тестов. В каждом тесте задаются функция, границы интегрирования, `n` и ожидаемое значение; проверка проходит с
точностью 1e-4.

| Группа | Что проверяется | Примеры |
| ------ | --------------- | ------- |
| 1D | точность на простых функциях | `1`, `x`, `x²`, `x³`, `sin(x)`, `exp(x)` |
| 2D | произведения и суммы переменных | `1`, `x·y`, `x²+y`, `exp(x+y)`, `sin(x)·cos(y)` |
| 3D | переход к трём измерениям | `1`, `x·y·z`, `x²+y²+z²`, `sin(x)·cos(y)·exp(z)` |
| 4D | высокая размерность | `1`, произведения координат |

По результатам прогона значения всех реализаций согласованы с ожидаемыми значениями.

## 8. Интерпретация различий

- **SEQ** — опорная линия по времени и эталон значения.
- **OMP** — `parallel for` со `schedule(static)` и `reduction`; `x[]` выделяется на каждой итерации — это поглощает
  выигрыш от параллелизма при T≤8.
- **TBB** — `parallel_reduce` управляет пулом сам; `x[]` выделяется один раз на поддиапазон; наилучший результат (4.30×).
- **STL** — ручное блочное разбиение с `std::thread`; корректный порядок spawn → work → join; `x[]` выделяется один раз
  на поток; при T=8 превышает SEQ на 6%.
- **ALL** — при 1 ранге идентичен OMP (per-iteration аллокация `x` доминирует); при 4 рангах × 4 потока даёт
  1.12× vs SEQ: каждый ранг обрабатывает 25% ячеек, снижая overhead аллокаций на ранг.

## 10. Заключение

Поставленная цель — параллельные реализации одной квадратуры и сравнение на общей методике — выполнена. TBB показал
наилучшее ускорение (4.30×) благодаря автоматическому управлению пулом потоков и отсутствию per-iteration аллокаций.
OMP-реализация ограничена per-iteration аллокацией вектора `x` — устранение этого недостатка позволило бы ожидать
ускорение, сопоставимое с TBB. STL при T=8 впервые превысила SEQ (1.06×), подтвердив корректность ручной схемы
распараллеливания.
