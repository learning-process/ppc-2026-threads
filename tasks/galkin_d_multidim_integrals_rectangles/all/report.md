## 1. Контекст

ALL-реализация объединяет два уровня параллелизма: MPI распределяет ячейки сетки по процессам
(межузловой уровень), OpenMP распараллеливает обработку ячеек внутри каждого процесса
(внутриузловой уровень). Число MPI-процессов задаётся при запуске (`mpirun -n N`), число
OMP-потоков — через `PPC_NUM_THREADS`.

## 2. Постановка задачи

Идентична SEQ: входные данные — функция `f`, вектор границ и `n > 0`; выход — значение
многомерного интеграла методом средней точки. В MPI-среде все процессы получают одинаковый
вход и записывают одинаковый результат (broadcast после редукции).

## 3. Описание алгоритма и схемы распараллеливания

### Уровень 1: MPI-разбиение

Диапазон `[0, total_cells)` делится по рангам блочным способом:

```text
chunk     = total_cells / world_size
remainder = total_cells % world_size

mpi_begin[rank] = rank * chunk + min(rank, remainder)
mpi_end[rank]   = mpi_begin + chunk + (rank < remainder ? 1 : 0)
```

### Уровень 2: OMP внутри ранга

```cpp
// File: all/src/ops_all.cpp
#pragma omp parallel for reduction(+ : local_sum) schedule(static)     \
    num_threads(std::max(1, ppc::util::GetNumThreads())) default(none) \
    shared(borders, h, dim, func, n, mpi_begin, mpi_end)
for (std::int64_t linear_idx = mpi_begin; linear_idx < mpi_end; ++linear_idx) {
    std::vector<double> x(dim);  // выделяется на каждой итерации
    // ... декодировать linear_idx, вычислить func(x), накопить local_sum
    local_sum += func(x);
}
```

Так же, как в OMP-реализации, `x` выделяется внутри цикла на каждой итерации. При увеличении
числа MPI-рангов каждый ранг получает меньше ячеек, что снижает абсолютное число аллокаций на
ранг и позволяет MPI-уровню дать реальное ускорение.

### Редукция и трансляция результата

```cpp
MPI_Reduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
MPI_Bcast(&global_sum, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
GetOutput() = global_sum * cell_v;
```

`MPI_Reduce` собирает сумму на ранге 0; `MPI_Bcast` рассылает глобальный результат всем рангам.
`MPI_Barrier` не используется — синхронизация обеспечивается семантикой `Reduce` и `Bcast`.

## 4. Схема коммуникаций

```text
Ранг 0        Ранг 1        ...  Ранг k
  │               │               │
  │ OMP parallel  │ OMP parallel  │ OMP parallel
  │ local_sum₀    │ local_sum₁    │ local_sumₖ
  └───────────────┴───────────────┘
           MPI_Reduce (SUM → ранг 0)
                  │
           MPI_Bcast → все ранги
```

## 8. Результаты

SEQ-эталон: task\_run = 0.516 с.

| Ранги | OMP-потоков | task\_run, с | Ускорение vs SEQ |
| ----- | ----------- | ------------ | ---------------- |
| 1     | 1           | 2.168        | 0.24×            |
| 1     | 4           | 1.067        | 0.48×            |
| 1     | 8           | 1.083        | 0.48×            |
| 2     | 4           | 0.715        | 0.72×            |
| 2     | 8           | 0.743        | 0.69×            |
| 4     | 4           | 0.462        | **1.12×**        |

При 1 ранге ALL ведёт себя идентично OMP-реализации — доминирует per-iteration аллокация `x`.
При 4 рангах × 4 потока ALL впервые превышает SEQ (1.12×): каждый ранг обрабатывает лишь 25%
ячеек, что пропорционально снижает overhead аллокаций. Коммуникационные расходы (`MPI_Reduce` +
`MPI_Bcast`) пренебрежимо малы на фоне 10⁸ вызовов `func`.

Рост числа OMP-потоков с 4 до 8 при фиксированном числе рангов не даёт прироста — тот же
эффект насыщения, что и в чистой OMP-реализации.

## 9. Выводы

ALL-реализация корректно строит двухуровневый параллелизм: MPI на уровне процессов + OpenMP
внутри процесса. MPI-уровень компенсирует основное узкое место (per-iteration аллокацию `x`)
за счёт уменьшения объёма работы на ранг: при 4 рангах ускорение составляет 1.12× vs SEQ.
Для дальнейшего роста производительности необходимо устранить аллокацию `x` внутри цикла
(перенос до тела параллельной секции с `firstprivate`).
