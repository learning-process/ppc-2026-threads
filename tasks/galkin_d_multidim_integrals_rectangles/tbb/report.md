## 1. Контекст

TBB-реализация использует `tbb::parallel_reduce` для распределения итераций по потокам пула TBB.
В отличие от OMP, число потоков не управляется через `PPC_NUM_THREADS` — TBB runtime самостоятельно
выбирает число рабочих потоков (по умолчанию — все логические ядра машины).

## 2. Постановка задачи

Идентична SEQ: входные данные — функция `f`, вектор границ и `n > 0`; выход — значение
многомерного интеграла методом средней точки.

## 3. Описание алгоритма и схемы распараллеливания

### parallel_reduce

```cpp
double sum = tbb::parallel_reduce(
    tbb::blocked_range<std::int64_t>(0, total_cells_i64),
    0.0,
    [&](const tbb::blocked_range<std::int64_t> &r, double local_sum) {
        std::vector<double> x(dim);
        for (std::int64_t linear_idx = r.begin(); linear_idx != r.end(); ++linear_idx) {
            auto tmp = static_cast<std::size_t>(linear_idx);
            for (std::size_t i = 0; i < dim; ++i) {
                const std::size_t idx_i = tmp % static_cast<std::size_t>(n);
                tmp /= static_cast<std::size_t>(n);
                x[i] = borders[i].first + ((static_cast<double>(idx_i) + 0.5) * h[i]);
            }
            local_sum += func(x);
        }
        return local_sum;
    },
    std::plus<>());
```

| Элемент | Описание |
| ------- | -------- |
| `blocked_range<int64_t>(0, total_cells)` | диапазон линейных индексов ячеек |
| grain size | не задаётся явно — TBB выбирает автоматически на основе объёма работы |
| лямбда-тело (split) | вызывается для каждого поддиапазона; `x[]` объявлен **внутри** лямбды, но **вне** внутреннего цикла — одна аллокация на поддиапазон |
| лямбда-объединение (`std::plus<>`) | суммирует частичные результаты из поддиапазонов |

### Почему TBB быстрее OMP

В OMP вектор `x` выделяется на каждой из 10⁸ итераций. В TBB `x` объявлен в теле лямбды-split —
один раз на каждый поддиапазон (поток). Типичный grain size для 10⁸ элементов на 12 ядрах
составляет ~10⁶–10⁷ итераций, то есть выполняется лишь ~12–100 аллокаций вместо 10⁸.

### Управление потоками

TBB runtime создаёт и управляет пулом потоков самостоятельно, используя все доступные физические ядра
(12 на Apple M4 Pro). `PPC_NUM_THREADS` игнорируется — для изменения числа потоков TBB требует
`tbb::global_control`.

## 4. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

- `ValidationImpl` — идентична SEQ: проверяет непустоту `borders`, конечность границ, `a < b`, ненулевой `n`;
- `PreProcessingImpl` — сбрасывает выход в 0;
- `RunImpl` — вычисляет шаги и объём ячейки; проверяет переполнение; вызывает `parallel_reduce`;
- `PostProcessingImpl` — проверяет `isfinite(result_)`.

Вектор `x` выделяется один раз на поддиапазон и переиспользуется в теле внутреннего цикла.

## 7. Результаты

SEQ-эталон: task\_run = 0.516 с.

TBB использует все 12 ядер независимо от `PPC_NUM_THREADS`, поэтому в таблице приведено
фактическое поведение при разных значениях переменной окружения.

| PPC\_NUM\_THREADS | task\_run, с | Ускорение vs SEQ |
| ----------------- | ------------ | ---------------- |
| 1                 | 0.120        | 4.30×            |
| 2                 | 0.121        | 4.26×            |
| 4                 | 0.119        | 4.34×            |
| 8                 | 0.120        | 4.30×            |

Ускорение стабильно на уровне 4.3× независимо от значения `PPC_NUM_THREADS`, поскольку TBB
всегда задействует весь пул из 12 потоков. Достигнутое ускорение ~4.3× объясняется двумя
факторами: (1) эффективное использование ~12 ядер и (2) значительная экономия на аллокациях
памяти по сравнению с OMP.

## 8. Выводы

TBB-реализация показывает наилучший результат среди однопроцессорных реализаций (4.3× vs SEQ).
Ключевые причины: TBB runtime использует все 12 физических ядер, а вектор `x` выделяется
один раз на поддиапазон, а не на каждую итерацию. Ограничение: нет возможности управлять
числом потоков через `PPC_NUM_THREADS` без явного задания `tbb::global_control`.
