## 1. Контекст

OpenMP-реализация распараллеливает перебор ячеек сетки через директиву `#pragma omp parallel for`.
Число потоков задаётся переменной окружения `PPC_NUM_THREADS`; при её отсутствии используется 1 поток.
Линейная нумерация ячеек 0..total−1 позволяет применить стандартный `parallel for` без модификации алгоритма.

## 2. Постановка задачи

Идентична SEQ: входные данные — функция `f`, вектор границ и `n > 0`; выход — значение
многомерного интеграла методом средней точки.

## 3. Описание алгоритма и схемы распараллеливания

### Линейная нумерация ячеек

В отличие от SEQ (где обход ведётся через `NextIndex`), OMP вводит линейный индекс
`linear_idx ∈ [0, total_cells)`, где `total_cells = n^d`. Преобразование:

```text
tmp = linear_idx
для i = 0..d-1:
    idx_i = tmp % n
    tmp   = tmp / n
    x[i]  = left[i] + (idx_i + 0.5) * h[i]
```

Такое преобразование работает потому, что делитель постоянен: операции `%` и `/` выполняются
за O(1) на любой итерации без зависимостей от соседних итераций.

### OpenMP-директива

```cpp
#pragma omp parallel for default(none)                              \
    shared(borders, h, dim, func, n, total_cells_i64, num_threads) \
    reduction(+ : sum) schedule(static) num_threads(num_threads)
for (std::int64_t linear_idx = 0; linear_idx < total_cells_i64; ++linear_idx) {
    std::vector<double> x(dim);
    // ...заполнить x[], накопить sum
    sum += func(x);
}
```

| Клауза | Смысл |
| ------ | ----- |
| `default(none)` | все переменные должны быть явно перечислены — защита от случайного разделяемого состояния |
| `shared(...)` | константы сетки и функция разделяются потоками (только чтение) |
| `reduction(+:sum)` | каждый поток ведёт локальный аккумулятор; результат суммируется в конце параллельной секции |
| `schedule(static)` | равномерное распределение итераций блоками; оптимально, так как стоимость каждой итерации одинакова |
| `num_threads(...)` | число потоков задаётся из `PPC_NUM_THREADS`, не превышает `total_cells` |

### Узкое место: выделение памяти внутри цикла

`std::vector<double> x(dim)` объявлен **внутри** тела параллельного цикла — на каждой из 10⁸ итераций
происходит аллокация и деаллокация heap-объекта. SEQ объявляет `x` **до** цикла и переиспользует его.
Это основная причина замедления относительно SEQ при малом числе потоков.

## 4. Детали реализации

**Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

- `ValidationImpl` — идентична SEQ: проверяет непустоту `borders`, конечность границ, `a < b`, ненулевой `n`;
- `PreProcessingImpl` — сбрасывает выход в 0;
- `RunImpl` — вычисляет шаги и объём ячейки; проверяет переполнение `total_cells`; запускает parallel for;
- `PostProcessingImpl` — проверяет `isfinite(result_)`.

Ключевой фрагмент:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none)                              \
    shared(borders, h, dim, func, n, total_cells_i64, num_threads) \
    reduction(+ : sum) schedule(static) num_threads(num_threads)
for (std::int64_t linear_idx = 0; linear_idx < total_cells_i64; ++linear_idx) {
    std::vector<double> x(dim);  // выделяется на каждой итерации
    auto tmp = static_cast<std::size_t>(linear_idx);
    for (std::size_t i = 0; i < dim; ++i) {
        const std::size_t idx_i = tmp % static_cast<std::size_t>(n);
        tmp /= static_cast<std::size_t>(n);
        x[i] = borders[i].first + ((static_cast<double>(idx_i) + 0.5) * h[i]);
    }
    sum += func(x);
}
GetOutput() = sum * cell_v;
```

## 7. Результаты

SEQ-эталон: task\_run = 0.516 с.

| Потоков | task\_run, с | Ускорение vs SEQ | Ускорение vs OMP T=1 |
| ------- | ------------ | ---------------- | -------------------- |
| 1 | 2.091 | 0.25× | 1.00× |
| 2 | 1.348 | 0.38× | 1.55× |
| 4 | 1.285 | 0.40× | 1.63× |
| 8 | 0.668 | 0.77× | 3.13× |

OMP при T=1 выполняется в 4× медленнее SEQ из-за 10⁸ heap-аллокаций вектора `x` внутри цикла.
С ростом числа потоков параллелизм частично компенсирует этот overhead: при T=8 время составляет
0.668 с — всё ещё медленнее SEQ, но в 3.13 раза быстрее OMP T=1.

## 8. Выводы

OMP-реализация демонстрирует линейное ускорение относительно себя же при увеличении числа потоков
(×3.13 при T=8), однако из-за per-iteration heap-аллокации вектора `x` при всех значениях T
она проигрывает последовательному эталону. Устранение аллокации (перенос `x` до цикла с объявлением
`firstprivate`) позволило бы добиться ожидаемого суперлинейного ускорения. Схема распараллеливания
через `schedule(static)` и `reduction` корректна и масштабируется.
