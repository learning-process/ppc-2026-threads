# Маркировка компонент на бинарном изображении (черные области соответствуют объектам, белые – фону) - TBB

## 1. Контекст

TBB-реализация использует задачно-ориентированный runtime Intel oneTBB. Цель — показать, как разделить изображение
на независимые подзадачи и как runtime управляет конкуренцией.

## 2. Постановка задачи

Та же задача, что и в `seq/report.md`: бинарное изображение, вывод меток компонент, проверка на наборе тестов
`1..14`.

## 3. Базовый алгоритм

Основная логика повторяет SEQ, но в TBB выбран горизонтальный штриховой способ распараллеливания:

- разделение изображения на `num_threads` полос по строкам,
- параллельная обработка каждой полосы,
- последовательная проверка границ между полосами,
- параллельное сжатие путей и нормализация меток.

## 4. Схема распараллеливания

Примитив: `tbb::parallel_for` по диапазону `0..num_threads`.

Диапазон работы:

- каждая итерация получает `thread_id`, `start_row`, `end_row`.
- внутри обрабатываются строки от `start_row` до `end_row`.

`grainsize` задаётся косвенно через размер блока строк `rows_per_thread = (height + num_threads - 1) / num_threads`.

Partitioner: по умолчанию, поскольку код делит работу на равные полосы и каждая полоса примерно одинаковая по
нагрузке.

Ограничение конкуренции: `num_threads = std::max(1, tbb::this_task_arena::max_concurrency())`.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

`PreProcessingImpl` инициализирует изображение, `labels_`, `parent_` и `current_label_`.

`FirstPass()` содержит:

- параллельный проход по полосам `parallel_for(0, num_threads, ...)`,
- `ProcessStripePixel` — локальная разметка каждой полосы,
- `MergeBoundariesTbb` — последовательное объединение соседних полос по вертикали.

`SecondPass()` выполняет:

- параллельное сжатие путей для всех пикселей,
- построение массива `root_mapping` для нормализации меток,
- параллельное переписывание меток.

Локальная структура:

- каждая полоса сама проставляет метки, присваивая при необходимости `idx + 1`,
- границы полос связываются отдельно,
- глобальный DSU `parent_` хранит равенства меток,
- нормализация происходит через массив `root_mapping`.

## 6. Проверка корректности

Сравнение с SEQ выполняется через общий функциональный тест `tests/functional/main.cpp`.

Особое внимание:

- метки на границе полос не должны образовывать разные классы,
- результаты должны совпадать для `PPC_NUM_THREADS=1` и `>1`.

## 7. Экспериментальная среда

Команды:

```bash
set PPC_NUM_THREADS=4
scripts/run_tests.py --running-type=threads --counts 1 2 4
```

Сборка та же, что и для SEQ и OMP.

- TBB показывает лучшую эффективность среди всех потоковых backend-ов на данной задаче.

## 9. Выводы

TBB удобен для этой задачи, потому что позволяет описать действительные подзадачи через параллельный проход по
полосам, не связывая ручную организацию потоков с деталями планировщика. Однако на малых изображениях накладные
расходы runtime и объединения границ могут снизить выигрыши.
