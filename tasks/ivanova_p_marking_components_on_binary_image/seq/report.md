# Маркировка компонент на бинарном изображении (черные области соответствуют объектам, белые – фону) - SEQ

## 1. Контекст

Последовательная версия является эталоном корректности и базой для сравнения всех остальных backend-ов.
Она реализует класическую разметку связных компонент на бинарном изображении и задаёт эталонное время `p=1`.

## 2. Постановка задачи

Вход: одно целое число `InType`. Оно кодирует тестовый сценарий и размер изображения.

- Для `1..10` изображение генерируется программно с фиксированным размером 100x100.
- Для `11..14` изображение загружается из файлов
  `tasks/ivanova_p_marking_components_on_binary_image/data/image*.txt`.

Данные изображения являются бинарными: `0` — фон, `1` — объект.

Выход: `OutType` = `std::vector<int>`, содержащий:

- ширину изображения,
- высоту изображения,
- количество обнаруженных компонент,
- вектор меток каждого пикселя в строковом порядке.

Ограничения:

- `width > 0`, `height > 0`,
- размер буфера данных должен быть `width * height`,
- фоновые пиксели должны оставаться `0`, объектные — получать метку от `1` до `num_components`.

## 3. Базовый алгоритм

Алгоритм работает в два прохода.

### Первый проход

Сканирование выполняется по строкам. Для каждого пикселя с объектом проверяются только левые и верхние соседи.
Возможны три случая:

- оба соседа фоновые → новая метка,
- один сосед объектный → копированная метка,
- оба соседа объектные и метки разные → сохраняется одна метка, а метки объединяются через DSU.

### Второй проход

После первого прохода выполняется сжатие путей в структуре `parent_` и нормализация меток в диапазон
`1..num_components`.

Асимптотика:

- Время: `O(N α(N))`, где `N = width * height`.
- Память: `O(N)` для массива меток `labels_` и структуры `parent_`.

Инварианты:

- после первого прохода все пиксели одной компоненты связаны через эквивалентность меток,
- после второго прохода каждому корню сопоставляется уникальная итоговая метка,
- количество непустых меток совпадает с числом компонент.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

`ValidationImpl` проверяет, что изображение загружено и имеет корректный размер. Если изображение не
проинициализировано, оно создаётся через `CreateTestImage` или загружается из файла.

`PreProcessingImpl` инициализирует `input_image_`, `width_`, `height_`, `labels_`, `parent_` и `current_label_`.

`RunImpl` вызывает `FirstPass()` и `SecondPass()`.

`PostProcessingImpl` собирает результат в `OutType` в формате `[width, height, component_count, labels...]`.

Критическое место: `ProcessPixel()`.

- если пиксель фон, он пропускается,
- если соседние метки различны, вызывается `UnionLabels()`,
- `UnionLabels()` выполняет поиск корней через `FindRoot()` и объединяет классы эквивалентности.

## 5. Проверка корректности

Корректность базируется на сравнении со спецификацией тестового набора.

Функциональные тесты `tests/functional/main.cpp` покрывают 14 случаев:

- `1..10` — программно сгенерированные шаблонные изображения,
- `11..14` — реальные изображения из файлов.

Проверяется:

- совпадение размеров `width` и `height`,
- ожидаемое число компонент для каждого теста,
- отсутствие меток на фоновых пикселях,
- присутствие всех меток от `1` до `num_components`.

Это делает SEQ эталоном для сравнения с OMP, TBB, STL и ALL.
