# Маркировка компонент на бинарном изображении (черные области соответствуют объектам, белые – фону) - OMP

- Студент: Иванова Полина Сергеевна
- Технология: OMP
- Вариант: 29

## 1. Контекст

OMP-версия переносит базовый алгоритм SEQ на модель разделяемой памяти. Цель — сохранить тот же результат и
ускорить поиск компонент за счёт распараллеливания двух основных фаз.

## 2. Постановка задачи

Задача та же, что в `seq/report.md`: бинарное изображение, метки компонент, проверка по тестам `1..14`.

## 3. Базовый алгоритм

OpenMP-версия использует ту же логику меток и DSU, но разделяет работу на параллельные проходы по пикселям.
Базовый алгоритм:

- параллельная инициализация меток объектов,
- параллельное объединение горизонтальных соседей,
- параллельное объединение вертикальных соседей,
- параллельная финализация корней,
- нормализация меток.

## 4. Схема распараллеливания

Области параллелятся через `#pragma omp parallel for default(none) ... num_threads(n_threads)`.

`shared`:

- `labels`, `input_image`, `parent_`, `width_`, `height_`, `total_pixels`.

`private`:

- локальные переменные цикла, `idx`, `xx`, `yy`, `cur_label`, `right_label`, `bottom_label`.

`UnionLabels` реализован с `#pragma omp critical(dsu_union)` из-за общего массива `parent_`. Это исключает гонку
при обновлении двух корней.

`schedule` не указан явно, поэтому OpenMP применяет расписание по умолчанию. Для этой задачи это допустимо, так как
работа над каждой строкой примерно равномерная.

Неявные барьеры:

- каждый `parallel for` содержит неявный барьер в конце, поэтому фазы выполняются последовательно по этапам.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

В `ValidationImpl` достаточно `GetInput() > 0`, так как `PreProcessingImpl` загружает или генерирует изображение.

`PreProcessingImpl`

- загружает файл для `11..14` или генерирует изображение,
- проверяет корректность размеров,
- выделяет `labels_`, инициализирует `parent_` размером `total_pixels + 1`.

`RunImpl` выполняет:

- `InitLabelsOmp` — параллельно проставляет `labels[i] = i + 1` для объектных пикселей,
- `MergeHorizontalPairsOmp` — параллельно объединяет соседей по горизонтали,
- `MergeVerticalPairsOmp` — параллельно объединяет соседей по вертикали,
- `FinalizeRootsOmp` — параллельно сжимает пути,
- `NormalizeLabelsOmp` — последовательно строит карту уникальных корней, затем параллельно переписывает метки.

Особенность: `TouchFrameworkOmp()` запускает пустую параллельную секцию с `std::atomic<int>`, чтобы гарантировать
инициализацию OpenMP-рантайма до основных измерений.

## 6. Проверка корректности

Корректность проверяется сравнением с SEQ через единый тестовый каркас `tests/functional/main.cpp`.

Проверяются все 14 случаев, включая файловые изображения. Важные проверки:

- сохранение размеров,
- совпадение числа компонент,
- нулевые метки на фоновых пикселях,
- отсутствие разбиения одной компоненты на несколько меток.

## 7. Экспериментальная среда

Команды:

```bash
set PPC_NUM_THREADS=4
set OMP_NUM_THREADS=4
scripts/run_tests.py --running-type=threads --counts 1 2 4
```

Сборка как в SEQ:

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

## 8. Результаты

Таблица замеров OMP.

| mode | threads | median time | speedup vs SEQ | efficiency | notes |
| --- | --- | --- | --- | --- | --- |
| task_run | 4 | 0.0347 | 0.71 | 18% | `PPC_NUM_THREADS=4` |
| pipeline | 4 | 0.0543 | 0.77 | 19% | `PPC_NUM_THREADS=4` |

Комментарий:

- при `PPC_NUM_THREADS=4` OMP не показал ускорения относительно SEQ, поскольку последовательная часть и
  синхронизация ограничивают масштабируемость,
- `MergeHorizontalPairsOmp` и `MergeVerticalPairsOmp` всё же выполняются параллельно,
- узким местом остаётся `critical` в `UnionLabels` и последовательная нормализация `is_root_used`.

## 9. Выводы

OpenMP выгоден для этой задачи при достаточном числе объектов и больших изображениях. Ограничение скорости даёт
синхронизация при объединении меток и финализация структуры DSU, поэтому на малых тестах масштабируемость будет
слабой.
