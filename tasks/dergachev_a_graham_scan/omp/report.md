## 1. Контекст

Перенос последовательной версии на OpenMP выполняется с целью проверки масштабируемости базовых этапов подготовки
данных и линейного поиска.

## 2. Постановка задачи

Задана генерация `n = 500 000` точек, среди которых необходимо построить выпуклую оболочку методом Грэхема.
Базовым эталоном производительности служит последовательная версия (0.1275 секунды).

## 3. Базовый алгоритм

Последовательная версия находит опорную точку за `O(n)`, сортирует массив точек по полярному углу
за `O(n log n)` и собирает оболочку стеком за `O(n)`.

## 4. Схема распараллеливания

В связи с тем, что фаза сортировки по углу (с вычислением предикатов) является проблемным местом
алгоритма Грэхема и сложно эффективно параллелится, было принято решение распараллелить две `O(n)` фазы:

1. **Генерация точек (PreProcessingImpl):** Распараллеливание инициализации с использованием директивы
   `#pragma omp parallel for`. Все переменные распределяются автоматически, так как вычисляются независимо.
2. **Поиск опорной точки (FindPivotIndex):** Поиск локального минимума внутри независимых блоков потоков с
   последующей критической секцией. Использовалась схема `shared(pts, size, pivot_idx)`. Внутри `omp parallel`
   каждый поток выполняет поиск локального `local_idx` (на основе предикатов `nowait`), а затем обновляет
   глобальный `pivot_idx` через `#pragma omp critical`. Барьер гарантирует завершение поиска.

Сортировка и построение стека оставлены последовательными, поскольку алгоритм требует строгого
порядка добавления.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Устранены риски гонок за счет применения локальных переменных для поиска опорной точки
и их слияния на единой критической секции.

```cpp
#pragma omp parallel default(none) shared(pts, size, pivot_idx)
{
    int local_idx = 0;
#pragma omp for nowait
    for (int i = 1; i < size; i++) {
        if (IsLowerLeft(pts[i], pts[local_idx])) local_idx = i;
    }
#pragma omp critical
    {
        if (IsLowerLeft(pts[local_idx], pts[pivot_idx])) pivot_idx = local_idx;
    }
}
```

Директива `default(none)` явно требует указать атрибуты переменных, защищая от случайных состояний гонки.
Блок `nowait` отключает неявный барьер в конце `for`, так как синхронизация все равно обеспечивается
`critical` и завершением блока `parallel`.

## 9. Выводы

В условиях, когда основная алгоритмическая сложность сосредоточена в нераспараллеленной сортировке, OpenMP
не даёт выигрыша и показывает замедление (Ускорение < 1.0). Версия с OMP оказалась неэффективна для данной задачи,
но доказывает корректность применения базовых примитивов.
