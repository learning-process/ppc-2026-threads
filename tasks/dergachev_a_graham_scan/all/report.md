# Построение выпуклой оболочки — проход Грэхема (ALL)

- Студент: Дергачёв Арсений Сергеевич

- Вариант: 22

## 1. Контекст

Гибридная реализация проверяет способность алгоритма масштабироваться за пределы одного узла с использованием MPI и
OpenMP. Задача выпуклой оболочки является сложной для распределенных систем из-за глобальной зависимости точек: опорная
вершина и порядок обхода определяются всем множеством.

## 2. Постановка задачи

Задана генерация `n = 500 000` точек, среди которых необходимо построить выпуклую оболочку методом Грэхема. Базовым
эталоном производительности служит версия `SEQ`.

## 3. Базовый алгоритм

Базовый алгоритм Грэхема: поиск опорной точки `O(n)`, сортировка по полярному углу `O(n log n)`, фильтрация стеком
`O(n)`. В распределённой модели точки делятся между MPI-процессами; каждый процесс выполняет только локальную
сортировку своего фрагмента, после чего отсортированные блоки сливаются на нулевом ранге и один раз проходит финальное
построение стека оболочки.

## 4. Межпроцессная схема

- **Распределение:** Нулевой ранг генерирует полный набор точек в `PreProcessingImpl`. В `RunImpl` размеры `original_size`
  и `padded_size` рассылаются через `MPI_Bcast`. Точки дополняются маркерами-заглушками до кратности числу процессов и
  распределяются функцией `MPI_Scatterv` (тип `MPI_Type_contiguous(2, MPI_DOUBLE)`).
- **Глобальная опорная точка:** Каждый процесс находит локальный минимум (`FindLocalPivotIndex`), локальные кандидаты
  собираются через `MPI_Gather`, глобальный pivot выбирается на rank 0 и рассылается `MPI_Bcast`. Владелец pivot
  удаляет эту точку из локального массива (`MPI_Allreduce` + `MPI_MAX` для определения владельца).
- **Сбор отсортированных блоков:** После локальной параллельной сортировки размеры блоков передаются `MPI_Gather`, сами
  точки — `MPI_Gatherv` на rank 0.
- **Финал на rank 0:** Итеративное слияние блоков (`MergeAllBlocks`) и одно построение оболочки (`BuildHullFromSorted`).
  Размер результата рассылается всем процессам через `MPI_Bcast` (`BcastHullSize`).

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса применяется OpenMP:

- параллельная генерация точек в `PreProcessingImpl` (`#pragma omp parallel for`);
- параллельный поиск локальной опорной точки (`FindLocalPivotIndex`);
- параллельная сортировка фрагмента: независимая сортировка чанков потоками и последовательное `std::inplace_merge`
  (`ParallelSortRange`).

Число потоков задаётся переменной окружения `PPC_NUM_THREADS` через `ppc::util::GetNumThreads()`. Компаратор
`AngleLessSeq` совпадает с SEQ-версией — это критично для корректности на `n = 500 000`.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Эволюция реализации:

1. **MPI + `std::thread`:** локальные чанки → частичные оболочки → слияние. Некорректная декомпозиция и высокие
   накладные расходы (~0.22 с на 1 процессе).
2. **MPI + OpenMP (локальная оболочка):** полный Грэхем на каждом ранге, затем слияние оболочек (~0.04 с).
3. **Текущая версия (sort-merge):** scatter → локальная сортировка → gather → merge → **один** проход Грэхема на rank 0.

Основной цикл:

```cpp
ScatterLocalPoints(bufs, rank, block_size, padded_size, original_size, mpi_point, points_);
const Pt global_pivot = FindGlobalPivot(bufs, rank, world_size, mpi_point, local_pivot);
RemoveGlobalPivot(bufs, rank, pivot_owner, global_pivot);
ParallelSortRange(bufs.local_data.begin(), bufs.local_data.end(), global_pivot);
GatherSortedAndBuildHull(bufs, rank, world_size, mpi_point, global_pivot, hull);
```

Буферы переиспользуются через `thread_local WorkBuffers`, что снижает число аллокаций между вызовами.

## 7. Проверка корректности

Сравнение с SEQ: логика проверена на малых и средних размерах (`mpiexec -n 1,2,3,4`, 13/13 функциональных тестов).
Слияние уже отсортированных по одному pivot блоков эквивалентно глобальной сортировке; финальный проход Грэхема даёт
точный размер оболочки. Обрабатываются тривиальные случаи (`n <= 1`, все точки совпадают) до распределённого пути.

## 10. Выводы

Текущая реализация MPI + OpenMP следует схеме «распределённая сортировка + одно построение оболочки», что
эквивалентно SEQ и существенно быстрее прежних вариантов с локальными оболочками. На одном процессе
накладные расходы MPI заметны, однако при **2 процессах × 2 потока** достигается ускорение **~1.6×**.
Конфигурация 8×8 страдает от переподписки.
