## 1. Контекст

Гибридная реализация проверяет способность алгоритма масштабироваться за пределы одного узла с использованием MPI и
`std::thread`. Задача выпуклой оболочки является сложной для распределенных систем из-за глобальной зависимости точек.

## 2. Постановка задачи

Задана генерация `n = 500 000` точек, среди которых необходимо построить выпуклую оболочку методом Грэхема. Базовым
эталоном производительности служит версия `SEQ`.

## 3. Базовый алгоритм

Базовый алгоритм Грэхема: поиск опорной точки `O(n)`, сортировка `O(n log n)`, фильтрация стеком `O(n)`. В
распределенной модели данные делятся на непересекающиеся множества, локально вычисляются выпуклые оболочки, а затем эти
фрагменты собираются в единую оболочку.

## 4. Межпроцессная схема

- **Распределение:** Нулевой ранг генерирует полный набор точек и использует `MPI_Bcast` для рассылки числа
  вершин `n`. Далее `MPI_Scatterv` распределяет точки по всем процессам. Данные сериализуются в одномерный
  массив для пересылки.
- **Сборка:** Локальные оболочки, рассчитанные каждым процессом, сериализуются обратно. Их размеры
  отправляются через `MPI_Gather`, а сами точки собираются на rank 0 с помощью `MPI_Gatherv`.
- **Широковещание результата:** Нулевой процесс строит итоговую оболочку и рассылает ее всем остальным процессам
  через `MPI_Bcast`, чтобы каждый процесс имел консистентное состояние и вернул идентичный ответ в PostProcessingImpl.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса применяется технология STL (`std::thread`).
Вызывается функция `ThreadedHull(local_pts, num_threads)`. Внутри ThreadedHull локальный набор local_pts разбивается на
num_threads чанков. Рабочие потоки выполняют построение частичной оболочки для каждого чанка независимо:

```cpp
workers.emplace_back(
    [&partial, &pts, off, len, ti]() { 
        partial[ti] = BuildHull({pts.begin() + off, pts.begin() + off + len}); 
    });
```

Затем `std::thread::join()` дожидается потоков, а главный поток ранга сливает их результаты.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Узкие места архитектуры лежат в коммуникациях. Алгоритм требует нескольких барьеров:

1. Ожидание рассылки (Scatterv).
2. Ожидание передачи локальных размеров (Gather).
3. Ожидание сбора точек (Gatherv).
4. Ожидание широковещания результата (Bcast x2).

Основной цикл межпроцессного обмена:

```cpp
// рассылка сырых данных
MPI_Scatterv(flat_all.data(), send_counts.data(), send_displs.data(), MPI_DOUBLE, 
             local_flat.data(), local_size, MPI_DOUBLE, 0, MPI_COMM_WORLD);
// внутрипроцессная STL-обработка
std::vector<Pt> local_hull = ThreadedHull(local_pts, num_threads);

// сбор размеров локальных оболочек
int local_hull_size = static_cast<int>(local_hull.size()) * 2;
MPI_Gather(&local_hull_size, 1, MPI_INT, recv_counts.data(), 1, MPI_INT, 0, MPI_COMM_WORLD);

// сбор всех точек на нулевом ранге
MPI_Gatherv(hull_flat.data(), local_hull_size, MPI_DOUBLE, gathered_flat.data(), 
            recv_counts.data(), recv_displs.data(), MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

## 7. Проверка корректности

Сравнение с SEQ: логика проверена на малых и средних размерах. Сборка оболочек из частичных оболочек является корректным
математическим приемом: любая точка, не входящая в локальную выпуклую оболочку, гарантированно не войдет в глобальную.
Сборка глобальной оболочки над всеми вершинами локальных оболочек дает точный ответ.

## 10. Выводы

Гибридная схема оправдана на гигантских наборах данных, не помещающихся в память одного узла. Однако для `n = 500 000`
накладные расходы на MPI-сети (пересылка массивов точек, упаковка/распаковка) убивают всякую выгоду от локального
многопоточного построения оболочек.
