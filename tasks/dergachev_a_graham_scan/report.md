## 1. Введение

Построение минимальной выпуклой оболочки является одной из множества алгоритмических задач. Алгоритм Грэхема решает её
за O(n log n). Хотя классический алгоритм в целом слабо поддается эффективному распараллеливанию из-за доминирующей
последовательной сортировки, эта особенность делает задачу отличным примером для исследования. Наличие как легко
параллелизуемых фаз (поиск минимума за `O(n)`), так и узких мест наглядно демонстрирует действие закона Амдала и влияние
накладных расходов различных систем параллельного программирования.

## 2. Постановка задачи

- **Вход:** Целое число `n >= 0` — количество точек.
- **Генерация:** `n` точек равномерно располагаются на окружности с добавлением `(0, 0)` при `n > 3`.
- **Выход:** Количество вершин выпуклой оболочки.
- **Ограничения:** `n >= 0`. При `n = 0` оболочка пустая, при `n = 1` содержит одну точку.
- **Критерий корректности:** Ровно `n` точек, центральная точка `(0, 0)` отсеивается.

## 3. Единая методика эксперимента

- **CPU:** AMD Ryzen 7 9800X3D @ 4.7–5.2 GHz (8 ядер / 16 потоков)
- **RAM:** 32 ГБ DDR5
- **OS:** Windows 11 Pro 10.0.26200
- **Компилятор:** MSVC 19.50 (Release сборка)
- **Размер задачи:** `n = 500 000` точек
- **Команды запуска:**
  - Обычный запуск: `.\ppc_perf_tests --gtest_filter="*dergachev_a_graham_scan*"`
  - Запуск с потоками: `$env:PPC_NUM_THREADS="<N>"; .\ppc_perf_tests --gtest_filter="*dergachev_a_graham_scan*"`
  - Запуск с MPI: `$env:PPC_NUM_THREADS="<N>"; mpiexec -n <N> .\ppc_perf_tests --gtest_filter="*dergachev_a_graham_scan*"`
- **Формулы:** Ускорение = `Time(seq) / Time(parallel)`. Эффективность = `Ускорение / Workers`.
- **Измерения:** Приводятся медианные показатели для режимов `task_run` и `task_pipeline`.

## 4. Сводка корректности

Все реализации (SEQ, OMP, TBB, STL, ALL) прошли тесты корректности фреймворка `ppc_func_tests`. На граничных случаях
(n=1, коллинеарные множества) алгоритм не зависает и выдает корректный размер. Отклонений в выходных результатах не
зафиксировано.

## 6. Интерпретация различий

- **SEQ:** Установил хороший базовый результат. Сортировка `O(n log n)` доминирует по времени выполнения.
- **OMP:** Несмотря на распараллеливание подготовки, накладные расходы OpenMP превышают выгоду. Сортировка не была
  распараллелена, в результате закон Амдала ограничил масштабируемость.
- **TBB:** Падение производительности. Скорее всего, использование внутренних структур `parallel_sort` вызвало огромное
  количество переключений контекста задач, где компаратор содержит дорогую математику, что сделало TBB версию медленной.
- **STL:** Единственная технология, обогнавшая SEQ. Ручное деление на крупные блоки данных без участия динамических
  планировщиков позволило ускорить фазу генерации и поиска с минимальными накладными расходами.
- **ALL:** Накладные расходы на распределение `500 000 * 2` чисел типа `double` по сети (даже локальной) и сборка
  результатов нивелируют пользу от нескольких рангов. Гибридная схема хороша лишь для гигантских массивов, не
  влезающих в один узел.

## 7. Заключение

Наилучшей версией для алгоритма Грэхема на одной машине признан `STL`. Однако, в целом, классический Грэхем слабо
поддается распараллеливанию.
