## 1. Введение

Построение выпуклой оболочки — одна из фундаментальных задач вычислительной геометрии,
применяемая в компьютерной графике, ГИС, робототехнике и задачах оптимизации.
Алгоритм Грэхема решает её за O(n log n) за счёт однократной угловой сортировки
и линейного прохода стека.
Цель работы — реализовать алгоритм в пяти вариантах (SEQ, OMP, TBB, STL, ALL)
и измерить масштабируемость по числу потоков на наборе из 500 000 точек.

---

## 2. Постановка задачи

**Вход:** целое число `n >= 0` — количество точек.

**Генерация данных:** `n` точек равномерно располагаются на единичной окружности
(`cos(2*pi*i/n), sin(2*pi*i/n)`), при `n > 3` добавляется центральная точка `(0, 0)`.

**Выход:** целое число — количество вершин выпуклой оболочки.

**Ожидаемый результат:** ровно `n` точек (центральная не попадает в оболочку).

**Ограничения:** `n >= 0`; при `n = 0` оболочка пустая, при `n = 1` содержит одну точку.

---

## 3. Описание последовательного алгоритма

Алгоритм Грэхема состоит из трёх фаз:

1. **Нахождение опорной точки (pivot):** точка с минимальной координатой `y`
   (при равенстве — с минимальной `x`) выбирается за O(n) линейным сканированием.

2. **Угловая сортировка:** все остальные точки сортируются по полярному углу
   относительно опорной точки. Порядок определяется знаком векторного произведения;
   при нулевом знаке — по расстоянию (ближняя первая). Сложность O(n log n).

3. **Построение оболочки (стек):** точки добавляются в стек последовательно.
   Если добавление создаёт правый поворот (векторное произведение <= 0),
   предыдущая точка извлекается из стека. Проход выполняется за O(n).

```text
pivot <- argmin_y(pts)
swap(pts[0], pts[pivot])
sort pts[1..n-1] by angle relative to pts[0]
hull <- []
for p in pts:
    while |hull| > 1 and cross(hull[-2], hull[-1], p) <= 0:
        hull.pop()
    hull.push(p)
return hull
```

## 4. Схема распараллеливания

### SEQ

Полностью последовательная реализация без использования параллельных примитивов.
Служит базой для сравнения.

### OMP (OpenMP)

Параллелизм применяется в двух местах:

- **Генерация точек** (`PreProcessingImpl`): вычисление `cos/sin` для каждой точки
  распараллелено директивой `#pragma omp parallel for`.
- **Поиск опорной точки** (`FindPivotIndex`): каждый поток находит локальный минимум
  в своём подмножестве (`#pragma omp for nowait`), затем глобальный минимум
  обновляется в `#pragma omp critical` секции.
- **Угловая сортировка и построение стека** остаются последовательными.

### TBB (Intel Threading Building Blocks)

Параллелизм строится на задачных примитивах TBB:

- **Генерация точек:** `tbb::parallel_for` по диапазону `[0, n)`.
- **Поиск опорной точки:** `tbb::parallel_reduce` — каждый поддиапазон возвращает
  локальный индекс минимума, редукция выбирает глобальный.
- **Угловая сортировка:** `tbb::parallel_sort` вместо `std::sort`.
- **Построение стека:** последовательно (зависимость данных).

### STL (C++ Standard Threads)

Параллелизм реализован вручную через `std::thread`:

- **Генерация точек** (`GeneratePointsParallel`): массив делится на `num_threads`
  равных блоков, каждый поток вычисляет свой блок.
- **Поиск опорной точки** (`FindPivotParallel`): каждый поток находит локальный
  минимум в своём чанке, главный поток выбирает глобальный минимум.
- **Угловая сортировка:** последовательный `std::sort`.
- **Построение стека:** последовательно.
- Количество потоков считывается из `ppc::util::GetNumThreads()`.

### ALL (MPI + STL Threads)

Гибридная реализация: MPI для запуска нескольких процессов,
STL-потоки внутри каждого процесса.

- **Декомпозиция:** набор точек делится на `num_threads` чанков по числу STL-потоков.
- Каждый поток строит локальную выпуклую оболочку своего чанка (`BuildHull`).
- Локальные оболочки объединяются, после чего вызывается финальный `BuildHull`.
- `MPI_Barrier` используется для синхронизации процессов.

**Тип точки:** в SEQ используется структура `Point {double x, y}`, в параллельных
реализациях — `std::pair<double, double>` для совместимости с алгоритмами STL и TBB.

**Граничные случаи:** при `n <= 1` или когда все точки совпадают — оболочка содержит
одну точку. STL и TBB дополнительно проверяют минимальный размер чанка и при
малом наборе данных используют последовательную версию.

**Потребление памяти:** O(n) — исходные точки, рабочая копия и стек оболочки.

### Анализ результатов

**OMP:** Overhead директив OpenMP при небольшом объёме параллельной работы превышает
выигрыш от распараллеливания. Только `FindPivotIndex` (O(n)) и генерация точек
параллелизованы, тогда как O(n log n) сортировка — последовательна.
В результате OMP незначительно медленнее SEQ на всех конфигурациях.

**TBB:** Время резко возрастает с увеличением числа потоков. Причина — использование
отладочной версии библиотеки TBB (`tbb12_debug.dll`), которая содержит дополнительные
проверки и синхронизацию. Это приводит к значительным накладным расходам при каждом
вызове `tbb::parallel_sort`, `tbb::parallel_reduce`, `tbb::parallel_for`.

**STL:** Показывает наилучшие результаты среди всех реализаций, опережая SEQ на
~26–29%. Параллельная генерация точек и поиск опорной точки дают небольшой выигрыш,
стабильный по числу потоков (сортировка остаётся узким местом).

**ALL:** Гибридная реализация показывает ускорение при 2 процессах (−16% к SEQ),
но затем ухудшается из-за накладных расходов MPI-барьера и дублирования работы
(каждый процесс обрабатывает полный набор данных). При 6 и 8 процессах
издержки синхронизации превышают выигрыш.

## 8. Выводы

- Алгоритм Грэхема плохо масштабируется «в лоб»: угловая сортировка O(n log n)
  сложно распараллеливаема, а построение стека строго последовательно.
- **OMP** даёт нейтральный результат: накладные расходы перекрывают выигрыш
  на небольших параллельных участках.
- **TBB** показал падение производительности из-за отладочных библиотек;
  в Release TBB результат был бы принципиально иным.
- **STL** демонстрирует стабильное небольшое ускорение (~26–29%) за счёт
  лёгкого распараллеливания подготовительных фаз.
- **ALL** в данной реализации работает медленнее SEQ при всех конфигурациях
  из-за отсутствия реального распределения данных и синхронизационного барьера.

## Приложение (выдержки из кода)

### Последовательный проход стека (общий для всех реализаций)

```cpp
for (const auto &p : pts) {
    while (hull_.size() > 1 &&
           CrossProduct(hull_[hull_.size() - 2], hull_.back(), p) <= 0.0) {
        hull_.pop_back();
    }
    hull_.push_back(p);
}
```

### OMP: параллельный поиск опорной точки

```cpp
#pragma omp parallel default(none) shared(pts, size, pivot_idx)
{
    int local_idx = 0;
#pragma omp for nowait
    for (int i = 1; i < size; i++) {
        if (IsLowerLeft(pts[i], pts[local_idx])) local_idx = i;
    }
#pragma omp critical
    {
        if (IsLowerLeft(pts[local_idx], pts[pivot_idx])) pivot_idx = local_idx;
    }
}
```

### TBB: parallel\_reduce для поиска опорной точки

```cpp
return tbb::parallel_reduce(
    tbb::blocked_range<int>(1, n), 0,
    [&pts](const tbb::blocked_range<int>& r, int best) {
        for (int i = r.begin(); i < r.end(); ++i)
            if (pts[i].second < pts[best].second) best = i;
        return best;
    },
    [&pts](int a, int b) {
        return pts[a].second < pts[b].second ? a : b;
    });
```

### ALL: двухуровневый алгоритм (STL-потоки внутри MPI-процесса)

```cpp
workers.emplace_back([&partial, &pts, off, len, ti]() {
    partial[ti] = BuildHull({pts.begin() + off, pts.begin() + off + len});
});
std::vector<Pt> merged;
for (const auto& h : partial)
    merged.insert(merged.end(), h.begin(), h.end());
return BuildHull(std::move(merged));
```
