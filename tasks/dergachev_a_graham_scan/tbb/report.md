## 1. Контекст

Версия с использованием библиотеки Intel Threading Building Blocks призвана перенести задачно-ориентированный подход на
алгоритм Грэхема. В отличие от OMP, TBB предоставляет готовые высокоуровневые алгоритмы (`parallel_sort`,
`parallel_reduce`), что позволяет распараллелить не только поиск, но и саму фазу сортировки.

## 2. Постановка задачи

Задано вычисление выпуклой оболочки на наборе из 500 000 генерируемых точек. Эталон `SEQ` отрабатывает за 0.1275 с.
Цель: оценить эффективность TBB примитивов при решении задачи.

## 3. Базовый алгоритм

Базовый процесс включает поиск опорной точки `O(n)`, сортировку по углу `O(n log n)` и сборку оболочки стеком `O(n)`.
Сборка остается строго последовательной.

## 4. Схема распараллеливания

В этой реализации применены задачные паттерны TBB:

1. **Генерация точек:** `tbb::parallel_for` обрабатывает точки в диапазоне `[0, n)`.
2. **Поиск опорной точки:** Использована функция `tbb::parallel_reduce`. Диапазон `tbb::blocked_range<int>`
   автоматически разбивается на чанки (`grainsize` управляется внутренним partitioner-ом по умолчанию
   `auto_partitioner`). Каждый чанк возвращает локальный индекс минимума, а лямбда-функция редукции
   осуществляет объединение этих индексов в глобальный оптимум.
3. **Угловая сортировка:** Применена функция `tbb::parallel_sort`. Это позволяет распараллелить самую
   ресурсоемкую часть `O(n log n)`, чего не было сделано в OMP-версии из-за сложности реализации.
4. Конкуренция контролируется переменной среды окружения, а TBB подстраивает число worker-потоков самостоятельно под
   системные ограничения.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

```cpp
return tbb::parallel_reduce(
    tbb::blocked_range<int>(1, n), 0,
    [&pts](const tbb::blocked_range<int>& r, int best) {
        for (int i = r.begin(); i < r.end(); ++i)
            if (pts[i].second < pts[best].second) best = i;
        return best;
    },
    [&pts](int a, int b) {
        return pts[a].second < pts[b].second ? a : b;
    });
```

Данный фрагмент иллюстрирует механизм `parallel_reduce`. Разделение данных позволяет избежать гонок (data races), так
как каждый чанк аккумулирует свой результат независимо, а редукция выполняется попарно.

## 9. Выводы

Несмотря на красивую и логичную декомпозицию `parallel_sort` и `parallel_reduce`, огромные накладные расходы `Intel TBB`
сделали данную реализацию самой медленной среди тестируемых. Накладные расходы масштабируются вместе с числом потоков,
что указывает на проблемы диспетчеризации задач в пуле TBB для слишком малого размера блока сложных вычислений.
