## 1. Введение

В работе исследуется задача умножения двух разреженных матриц комплексного типа,
хранящихся в формате CRS. Цель работы - реализовать алгоритм в пяти вариантах
и сравнить их по корректности и производительности.

---

## 2. Постановка задачи

**Вход:** две матрицы `A` и `B` в формате CRS.

**Выход:** матрица `C = A * B`, также в формате CRS.

**Условия корректности:**

- число столбцов матрицы `A` должно совпадать с числом строк матрицы `B`;
- структура CRS должна быть корректной;
- в результирующую матрицу записываются только точные ненулевые значения.

---

## 3. Последовательный алгоритм

Общая математическая логика во всех версиях одинакова.

Для строки `i` матрицы `A` просматриваются все ее ненулевые элементы `A[i,j]`.
Для каждого такого элемента перебираются ненулевые элементы строки `j`
матрицы `B`, а произведения `A[i,j] * B[j,k]` накапливаются по столбцам `k`
во временном аккумуляторе.

Используется маркерная схема:

- `marks[k] != row_index` означает, что столбец `k` в текущей строке еще не встречался;
- при первом обращении столбец добавляется в список `touched_cols`;
- при повторных обращениях значение просто суммируется.

После обработки строки список затронутых столбцов сортируется, а точные ненулевые
значения сохраняются в формат CRS.

---

## 4. Схема распараллеливания

### SEQ

Базовая последовательная версия без использования параллельных примитивов.

### OMP

Обработка строк матрицы `A` распараллеливается между потоками OpenMP.
Каждый поток использует локальные рабочие буферы, а затем результат
собирается в итоговый CRS.

### TBB

Распараллеливание выполняется через `oneapi::tbb::parallel_for`
по диапазонам строк. Размер блоков подбирается функцией `ResolveGrainSize`.

### STL

Используется `std::thread`. Для более равномерной загрузки потоков
предварительно оценивается стоимость строк, после чего строятся границы
диапазонов строк для каждого потока.

### ALL

Гибридная реализация `MPI + OMP`.

- MPI распределяет строки матрицы `A` между процессами;
- матрица `B` рассылается всем процессам;
- внутри каждого процесса локальные строки вычисляются с помощью OpenMP;
- локальные части результата собираются на процессе `0`.

## 7. Тестирование

### Функциональные тесты

Функциональные тесты покрывают:

- фиксированный пример `3x3`;
- случайные разреженные матрицы размеров `10`, `20`, `30`;
- сравнение результата с плотным эталоном, вычисленным функцией `DenseMul`.

Генерация входных данных сделана детерминированной.

### Производительные тесты

Производительный тест использует:

- размер матриц: `15000 x 15000`;
- плотность ненулевых элементов: `0.001`;
- детерминированную генерацию входа.

Для `ALL` локальные данные генерируются сразу в формате CRS, чтобы не создавать
слишком большие плотные матрицы в памяти.

### Анализ результатов

- **SEQ** задает базовую линию и показывает время `0.1683691002 с`.
- **OMP** по актуальным замерам ускоряется относительно SEQ на `2-16` потоках,
  а лучший результат достигается в области `4-8` потоков.
- **TBB** показала лучшие актуальные времена и наиболее стабильное ускорение;
  оптимальная область в текущих измерениях находится около `8` потоков.
- **STL** также дает уверенное ускорение и показывает лучший результат в области
  `8-12` потоков.
- **ALL** показывает лучший результат на `2x2`, после чего начинает проигрывать
  из-за стоимости MPI-коммуникаций и сборки результата.

## 9. Выводы

- Все реализации корректно решают задачу умножения `CRS x CRS -> CRS`.
- Логика обработки ненулевых элементов и оценки результата согласована между версиями.
- Лучшую актуальную производительность на исследуемой машине показала `TBB`.
- `OMP` дает умеренное ускорение, но масштабируется менее стабильно.
- `ALL` является рабочей гибридной реализацией, однако для данной задачи и данной
  конфигурации ее производительность ограничена накладными расходами MPI.
