## 1. Введение

В работе исследуется задача умножения двух разреженных матриц комплексного типа,
хранящихся в формате CRS. Эта задача хорошо подходит для сравнения разных моделей
параллелизма, потому что математическое ядро во всех версиях одинаково, а различаются
именно способы организации вычисления: последовательный baseline, OpenMP, oneTBB,
ручное управление потоками через `std::thread` и гибридная схема `MPI + OMP`.

## 2. Единая постановка задачи

Вход:

- две матрицы `A` и `B` в формате CRS;
- каждая матрица задается полями `rows`, `cols`, `values`, `col_index`, `row_ptr`.

Выход:

- матрица `C = A * B`, также в формате CRS.

Ограничения и критерий корректности:

- число столбцов `A` должно совпадать с числом строк `B`;
- структура CRS должна быть корректной:
  - `row_ptr.size() == rows + 1`;
  - `values.size() == col_index.size()`;
  - `row_ptr.front() == 0`;
  - `row_ptr.back() == nnz`;
- индексы столбцов должны лежать в диапазоне `[0, cols)`;
- корректным результатом считается CRS-матрица, совпадающая с плотным эталоном
  умножения по структуре строк, индексам столбцов и значениям комплексных элементов.

## 3. Единая методика эксперимента

Окружение:

- CPU: AMD Ryzen 5 1600
- RAM: 8 GB
- OS: Windows 11
- compiler: MSVC
- build type: `Release`
- MPI: MS-MPI 10.0

Переменные окружения:

- `OMP_NUM_THREADS` использовалась для чистой `OMP`-версии;
- `PPC_NUM_THREADS` использовалась для `TBB`, `STL` и для OpenMP-части гибридной
  `ALL`-версии;
- число MPI-процессов задавалось через `mpiexec -n`.

Источник данных:

- functional-тесты используют фиксированный пример `3x3` и детерминированно
  сгенерированные матрицы размеров `10x10`, `20x20`, `30x30`;
- performance-тест использует детерминированно сгенерированные CRS-матрицы
  размера `15000 x 15000` с плотностью `0.001`.

Сравниваемые размеры задач:

- для корректности: `3`, `10`, `20`, `30`;
- для производительности: `15000 x 15000`.

Метрики:

- `time` — измеренное время `task_run`;
- `speedup = T_seq / T_backend`;
- для `OMP`, `TBB`, `STL`: `efficiency = speedup / threads`;
- для `ALL`: `efficiency = speedup / total_workers`, где
  `total_workers = ranks * threads_per_rank`.

О повторностях:

- в таблицы вынесены representative run-значения `task_run`, полученные на одной машине
  в одинаковом окружении;
- отдельная серия повторов с медианой и прогревом в этом отчете не фиксировалась, поэтому
  результаты следует трактовать как практическое сравнительное измерение, а не как строгий
  статистический бенчмарк.

## 4. Сводка корректности

Все backend-ы сравнивались с последовательной версией `SEQ`, которая играет роль baseline
по корректности и времени. Проверка выполнялась общей тестовой инфраструктурой:

- functional-тесты из `tests/functional/main.cpp`;
- performance-тесты из `tests/performance/main.cpp`.

Использованные функциональные сценарии:

- фиксированный пример `SmallFixed` размером `3x3`;
- детерминированные разреженные матрицы `10x10`, `20x20`, `30x30`;
- сравнение с плотным эталоном `DenseMul`.

Ограничения применимости отдельных backend-ов:

- `SEQ` является эталоном по корректности;
- `OMP`, `TBB`, `STL` используют общий shared-memory сценарий и сравниваются по числу
  потоков;
- `ALL` требует согласованной конфигурации `ranks × threads` и в функциональном режиме
  дополнительно рассылает итоговый результат всем rank-ам, чтобы тест сравнивал именно
  общий CRS-результат, а не локальный фрагмент процесса `0`.

## 6. Интерпретация различий

`SEQ` показывает реальную цену базового алгоритма без накладных расходов на runtime,
синхронизацию и коммуникации. Именно поэтому его время используется знаменателем для
всех формул ускорения.

`OMP` дает умеренный выигрыш, но масштабируется слабее `TBB` и `STL`. Для этой задачи
узким местом остаются нерегулярный доступ к строкам `B`, память и последовательная
финализация итогового CRS после завершения parallel region. То, что лучшие результаты
`OMP` достигаются около `4–8` потоков, показывает: вычислительная часть еще ускоряется,
но дополнительный рост числа потоков уже не компенсирует overhead.

`TBB` оказался самым сильным backend-ом на данной задаче. Это не означает, что `TBB`
«вообще лучше OpenMP», а отражает конкретное сочетание факторов: `parallel_for`,
`blocked_range`, удачно подобранный `grain_size` и thread-local workspace хорошо легли
на построчную природу CRS-умножения. После `8` workers ускорение почти насыщается, что
похоже уже не на проблему runtime, а на ограничение алгоритма и памяти.

`STL` показал результат, близкий к `TBB`, но ценой более ручной инженерной работы.
Разбиение диапазонов, запуск потоков, управление `Workspace` и `join()` реализованы
вручную, поэтому эта версия методически важна как демонстрация прозрачной thread-based
схемы. Ее слабое место — накладные расходы на создание и синхронизацию потоков, которые
видны при `16` потоках.

`ALL` нельзя интерпретировать по тем же правилам, что и чисто shared-memory backend-ы.
Здесь к локальному вычислению добавляются `Broadcast`, `Scatterv`, `Gatherv`, упаковка
комплексных значений и финальная сборка результата на `rank 0`. Поэтому проигрыш больших
конфигураций — это не «провал MPI», а ожидаемое следствие коммуникационных и
синхронизационных издержек на машине данного масштаба. Конфигурация `2 × 2` показывает,
что умеренная гибридность еще окупается, но рост `ranks × threads` быстро делает схему
communication-bound.

## 7. Репродуцируемость

Команды сборки:

```powershell
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

Команды запуска functional-тестов:

```powershell
cd build\bin
.\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_seq_enabled*"
$env:OMP_NUM_THREADS='8'; .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_omp_enabled*"
$env:PPC_NUM_THREADS='8'; .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_tbb_enabled*"
$env:PPC_NUM_THREADS='8'; .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_stl_enabled*"
$env:PPC_NUM_THREADS='8'; mpiexec -n 8 .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_all_enabled*"
```

Команды получения основных `task_run`-замеров:

```powershell
.\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_seq_enabled"
$env:OMP_NUM_THREADS='8'; .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_omp_enabled"
$env:PPC_NUM_THREADS='8'; .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_tbb_enabled"
$env:PPC_NUM_THREADS='8'; .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_stl_enabled"
$env:PPC_NUM_THREADS='8'; mpiexec -n 8 .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_all_enabled"
```

Для гибридной версии конфигурация задается совместно:

- число rank-ов — через `mpiexec -n`;
- число потоков внутри rank-а — через `PPC_NUM_THREADS`.

## 8. Заключение

Все пять реализаций корректно решают задачу умножения `CRS × CRS -> CRS` и проходят
общий набор functional-тестов. На исследуемой машине лучшей по ускорению оказалась
версия `TBB`; близкий результат показала `STL`; `OMP` дал умеренное ускорение; гибридная
`ALL` оказалась оправданной только в небольшой конфигурации `2 × 2`.

Ограничения сравнения:

- измерения выполнены на одной машине и в одном окружении;
- в таблицы вынесены representative run-значения, а не медианы длинных серий;
- для `ALL` эффективность нормировалась по `total_workers = ranks * threads_per_rank`,
  что важно учитывать при сравнении с чисто потоковыми backend-ами.

Практическое направление дальнейшего улучшения:

- уменьшение стоимости финальной сборки CRS;
- снижение накладных расходов на коммуникации в `ALL`;
- более глубокая работа с локальностью данных и балансировкой нагрузки.
