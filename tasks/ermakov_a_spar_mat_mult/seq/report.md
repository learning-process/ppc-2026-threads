## 1. Контекст

Последовательная версия задает эталонный смысл всей задачи. Именно здесь
фиксируются вход, выход, ограничения, способ проверки корректности и baseline-
время, относительно которого затем считаются ускорения параллельных версий.
Если последовательный отчет описан слабо, сравнение `OMP`, `TBB`, `STL` и `ALL`
теряет методический смысл.

## 2. Постановка задачи

На вход подаются две разреженные матрицы `A` и `B` комплексного типа в формате
CRS. На выходе требуется получить матрицу `C = A * B`, также в формате CRS.

Корректным входом считаются матрицы, для которых:

- `A.cols == B.rows`;
- `rows >= 0`, `cols >= 0`;
- `row_ptr.size() == rows + 1`;
- `values.size() == col_index.size()`;
- `row_ptr.front() == 0`, `row_ptr.back() == nnz`;
- `row_ptr` монотонен;
- все индексы `col_index[k]` лежат в диапазоне `[0, cols)`.

Допустимый `InType` в этой задаче — пара CRS-матриц, удовлетворяющих условиям
выше. Крайние случаи, которые блокируются в `ValidationImpl`, — несовместимые
размеры матриц и некорректная внутренняя структура CRS.

Корректным результатом считается CRS-матрица, у которой:

- размер равен `rows(A) x cols(B)`;
- `row_ptr` корректно описывает границы строк;
- `values` и `col_index` согласованы по длине;
- каждая строка совпадает с эталоном, полученным независимым плотным умножением.

## 3. Базовый алгоритм

Главная структура данных задачи — формат CRS:

- `row_ptr` задает границы строк;
- `col_index` хранит номера столбцов ненулевых элементов;
- `values` хранит сами комплексные значения.

Алгоритм выполняется построчно.

1. Для строки `i` матрицы `A` просматриваются все ненулевые элементы `A[i, j]`.
2. Для каждого такого элемента берется строка `j` матрицы `B`.
3. Произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k` во временном
   аккумуляторе `row_vals`.
4. Массив `row_mark` позволяет различать первое обращение к столбцу и повторное
   накопление.
5. Список `used_cols` хранит только реально затронутые столбцы.
6. После завершения строки `used_cols` сортируется, а ненулевые значения
   переносятся в итоговые `c_.col_index` и `c_.values`.

Асимптотика по времени:
`O(sum_i sum_{(i,j) in A} nnz(B_j))`.

Асимптотика по памяти:
`O(cols(B))` дополнительной памяти на `row_vals` и `row_mark`, плюс `O(nnz(C))`
на сам результат.

Главный инвариант: после завершения обработки строки `i` массивы
`c_.row_ptr`, `c_.col_index`, `c_.values` содержат корректное CRS-представление
всех строк результата с индексами от `0` до `i`.

Именно это решение выбрано как baseline, потому что оно выражает вычислительную
логику без параллельных накладных расходов и без дополнительной синхронизации.

## 4. Детали реализации

Файлы:

- `seq/include/ops_seq.hpp`
- `seq/src/ops_seq.cpp`

Pipeline реализации:

- `ValidationImpl` проверяет совместимость размеров и корректность структуры CRS;
- `PreProcessingImpl` копирует `A` и `B` во внутренние поля `a_`, `b_` и
  подготавливает пустую матрицу `c_`;
- `RunImpl` создает рабочие буферы строки и последовательно вызывает `ProcessRow`
  для всех строк `A`;
- `PostProcessingImpl` передает сформированную `c_` наружу.

Исключительные и граничные случаи, которые обрабатываются:

- несовместимые размеры `A` и `B`;
- пустой или некорректный `row_ptr`;
- выход индексов столбцов за допустимый диапазон;
- точный нулевой комплексный результат, который не должен попадать в CRS.

## 5. Проверка корректности

Результат сравнивается с плотным эталоном `DenseMul`, который вычисляется в
`tests/functional/main.cpp` независимо от CRS-реализации.

Использованные функциональные тесты:

- `SmallFixed`, размер `3x3`;
- `VerySparse`, размер `10x10`, плотность `0.05`;
- `MediumSparse`, размер `20x20`, плотность `0.2`;
- `Dense`, размер `30x30`, плотность `0.7`.

Характерные примеры:

1. `SmallFixed (3x3)`:
   в тесте заполняются элементы `A[0,0] = 1`, `A[0,2] = 2`, `A[2,1] = 3`,
   `B[0,1] = 4`, `B[1,2] = 5`, `B[2,0] = 6`. Ожидаемый результат:
   `C[0,0] = 12`, `C[0,1] = 4`, `C[2,2] = 15`.
2. `VerySparse (10x10)`:
   проверяет корректность на матрицах с малым числом ненулевых элементов и
   большим числом пропусков.
3. `Dense (30x30)`:
   проверяет, что логика не зависит от редкости структуры и корректно работает
   при более насыщенном заполнении.

## 8. Выводы

Последовательная версия используется как эталон для ускорения, потому что она:

- реализует полную математическую логику задачи без параллельных накладных

### Приложение

Путь к листингу: `seq/src/ops_seq.cpp`, функция `ProcessRow`.

```cpp
void ErmakovASparMatMultSEQ::ProcessRow(
    int i, std::vector<std::complex<double>> &row_vals, std::vector<int> &row_mark,
    std::vector<int> &used_cols, int &nnz_so_far) {
  const int a_start = a_.row_ptr[i];
  const int a_end = a_.row_ptr[i + 1];

  for (int ak = a_start; ak < a_end; ++ak) {
    const int j = a_.col_index[ak];
    const auto a_ij = a_.values[ak];

    const int b_start = b_.row_ptr[j];
    const int b_end = b_.row_ptr[j + 1];

    for (int bk = b_start; bk < b_end; ++bk) {
      const int k = b_.col_index[bk];
      const auto b_jk = b_.values[bk];

      if (row_mark[k] != i) {
        row_mark[k] = i;
        row_vals[k] = a_ij * b_jk;
        used_cols.push_back(k);
      } else {
        row_vals[k] += a_ij * b_jk;
      }
    }
  }
}
```

Этот фрагмент показывает базовое вычислительное ядро: переход по строке `A`,
выбор соответствующей строки `B` и marker-схему накопления значений по столбцам.
В листинг не входит нижняя часть функции, где сортируется `used_cols` и строка
результата переводится в формат CRS; это уже шаг упаковки результата после
завершения самого накопления.
