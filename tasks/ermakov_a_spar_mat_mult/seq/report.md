## 1. Контекст

Последовательная версия задает эталонный смысл всей задачи. Именно здесь

## 2. Постановка задачи

На вход подаются две комплексные матрицы `A` и `B` в формате CRS. На выходе
требуется получить матрицу `C = A * B`, также в формате CRS.

Корректным входом считаются матрицы, для которых:

- `A.cols == B.rows`;
- `row_ptr.size() == rows + 1`;
- `values.size() == col_index.size()`;
- `row_ptr.front() == 0`, `row_ptr.back() == nnz`;
- все `col_index[k]` лежат в диапазоне `[0, cols)`.

Крайние случаи, блокируемые в `ValidationImpl`:

- несовместимые размеры `A` и `B`;
- некорректная CRS-структура;
- выход индексов столбцов за допустимый диапазон.

Корректным результатом считается CRS-матрица, совпадающая с плотным эталоном
умножения по структуре строк, индексам столбцов и комплексным значениям.

## 3. Базовый алгоритм

Главная структура данных задачи - CRS:

- `row_ptr` задает границы строк;
- `col_index` хранит столбцы ненулевых элементов;
- `values` хранит сами комплексные значения.

Пошаговая схема:

1. Для строки `i` матрицы `A` просматриваются все ее ненулевые элементы `A[i, j]`.
2. Для каждого такого элемента берется строка `j` матрицы `B`.
3. Произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`.
4. Затронутые столбцы сортируются.
5. Ненулевые значения строки переносятся в итоговый CRS.

Асимптотика по времени:
`O(sum_i sum_{(i,j) in A} nnz(B_j))`.

Асимптотика по памяти:
`O(cols(B))` дополнительной памяти на рабочие буферы строки,
плюс `O(nnz(C))` на результат.

Инвариант:
после завершения обработки строки `i` массивы `row_ptr`, `col_index`, `values`
содержат корректное CRS-представление всех строк результата от `0` до `i`.

Именно эта версия выбрана как baseline, потому что реализует полную
математическую логику без параллельных накладных расходов.

## 4. Детали реализации

Файлы:

- `seq/include/ops_seq.hpp`
- `seq/src/ops_seq.cpp`

Pipeline реализации:

- `ValidationImpl` проверяет совместимость размеров и корректность CRS;
- `PreProcessingImpl` копирует `A` и `B` во внутренние поля и подготавливает `C`;
- `RunImpl` последовательно вызывает `ProcessRow` для всех строк `A`;
- `PostProcessingImpl` передает готовую матрицу `C` наружу.

Обработанные граничные случаи:

- несовместимые размеры матриц;
- пустой или некорректный `row_ptr`;
- выход `col_index` за допустимый диапазон;
- точный нулевой комплексный результат, который не должен попадать в CRS.

### Приложение

Путь к листингу: `seq/src/ops_seq.cpp`, функция `ProcessRow`.

```cpp
void ErmakovASparMatMultSEQ::ProcessRow(
    int i, std::vector<std::complex<double>> &row_vals, std::vector<int> &row_mark,
    std::vector<int> &used_cols, int &nnz_so_far) {
  const int a_start = a_.row_ptr[i];
  const int a_end = a_.row_ptr[i + 1];

  for (int ak = a_start; ak < a_end; ++ak) {
    const int j = a_.col_index[ak];
    const auto a_ij = a_.values[ak];

    const int b_start = b_.row_ptr[j];
    const int b_end = b_.row_ptr[j + 1];

    for (int bk = b_start; bk < b_end; ++bk) {
      const int k = b_.col_index[bk];
      const auto b_jk = b_.values[bk];

      if (row_mark[k] != i) {
        row_mark[k] = i;
        row_vals[k] = a_ij * b_jk;
        used_cols.push_back(k);
      } else {
        row_vals[k] += a_ij * b_jk;
      }
    }
  }
}
```

Фрагмент показывает базовое вычислительное ядро: переход по строке `A`,
обращение к соответствующей строке `B` и marker-схему накопления по столбцам.
