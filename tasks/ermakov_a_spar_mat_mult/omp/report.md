## 1. Контекст

OpenMP-версия переносит baseline-алгоритм умножения CRS-матриц в модель общей
памяти. Цель этой реализации — распараллелить независимую обработку строк
результата, сохранив ту же математическую схему, что и в
`seq/report.md`, но уменьшив время выполнения на многопроцессорной машине.

## 2. Постановка задачи

Предметная постановка полностью совпадает с последовательной версией: на вход
подаются две комплексные CRS-матрицы `A` и `B`, на выходе требуется получить
матрицу `C = A * B` в формате CRS.

В качестве baseline используется последовательная реализация из
`seq/report.md`, поэтому:

- критерий корректности результата тот же;
- ограничения на входные данные те же;
- ускорение считается относительно времени SEQ.

## 3. Базовый алгоритм

Последовательная версия для каждой строки `i` матрицы `A` просматривает все ее
ненулевые элементы `A[i, j]`, затем проходит по строке `j` матрицы `B` и
накапливает произведения `A[i, j] * B[j, k]` по столбцам `k`.

OpenMP-версия не меняет эту математику. Она меняет только способ организации
исполнения: строки результата вычисляются параллельно, а итоговая CRS-структура
собирается после завершения параллельной области.

## 4. Схема распараллеливания

Параллелится центральный цикл по строкам матрицы `A` в `RunImpl`.

Фактический фрагмент директивы:

```cpp
#pragma omp parallel default(none) shared(m, p, row_values, row_cols)
{
  std::vector<std::complex<double>> row_vals(static_cast<std::size_t>(p),
                                             std::complex<double>(0.0, 0.0));
  std::vector<int> row_mark(static_cast<std::size_t>(p), -1);
  std::vector<int> used_cols;
  used_cols.reserve(256);

#pragma omp for
  for (int i = 0; i < m; ++i) {
    AccumulateRowProducts(i, row_vals, row_mark, used_cols);
    SortUsedCols(used_cols);
    CollectRowValues(row_vals, used_cols, row_cols[static_cast<std::size_t>(i)],
                     row_values[static_cast<std::size_t>(i)]);
  }
}
```

Расшифровка решений:

- `default(none)` выбран специально, чтобы все действительно используемые общие
  данные были перечислены явно. Это уменьшает риск случайно захватить лишнюю
  переменную из внешней области видимости.
- `shared(m, p, row_values, row_cols)` означает, что размеры задачи и массивы
  построчных результатов общие для всех потоков.
- `row_vals`, `row_mark`, `used_cols` создаются внутри parallel region, поэтому
  каждый поток получает собственные private-буферы накопления.
- `reduction` не нужен, потому что общего аккумулятора суммы нет: каждый поток
  пишет только в свою локальную память и затем в уникальную строку результата.
- `atomic` и `critical` не нужны по той же причине: потоки не делают
  конкурентную запись в один и тот же `row_values[i]` или `row_cols[i]`.
- `schedule(...)` явно не задан. Фактическая политика разбиения итераций
  определяется runtime OpenMP. Для данной реализации это допустимо.
  `schedule(static)` или `schedule(dynamic)` не
  были выбраны вручную.
- после завершения `#pragma omp parallel` существует неявный барьер. В этой
  задаче он безопасен и необходим, потому что только после него можно начинать
  последовательную сборку `c_.row_ptr`, `c_.values` и `c_.col_index`.

Отдельно отмечу число потоков. В текущей реализации оно не задается
внутри `num_threads(...)`, поэтому конфигурация исполнения определяется окружением
запуска, в частности `OMP_NUM_THREADS`.

## 5. Детали реализации

Файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Относительно SEQ были изменены следующие участки:

- вместо немедленной записи строки в итоговый CRS введены построчные контейнеры
  `row_values` и `row_cols`;
- функция `AccumulateRowProducts` выносит вычисление одной строки в отдельный
  helper, удобный для вызова из параллельной области;
- функция `CollectRowValues` отделяет фазу накопления от фазы упаковки строки;
- финальная сборка результата в `c_` вынесена за пределы parallel region.

Главный риск гонок здесь был связан не с арифметикой, а с памятью:

- если бы потоки делили `row_vals` или `row_mark`, возникла бы гонка на
  накопителе строки;
- если бы несколько потоков писали в один и тот же контейнер результата, нужна
  была бы дополнительная синхронизация.

Эти риски устранены тем, что:

- рабочие буферы строки private;
- каждая итерация `i` записывает данные только в `row_values[i]` и `row_cols[i]`,
  то есть в уникальную позицию.

## 6. Проверка корректности

Корректность OMP-версии проверялась сравнением с SEQ и с плотным эталоном из
общей тестовой инфраструктуры.

Использованные функциональные тесты:

- `SmallFixed`, размер `3x3`;
- `VerySparse`, размер `10x10`;
- `MediumSparse`, размер `20x20`;
- `Dense`, размер `30x30`.

Характерные случаи:

1. фиксированный пример `3x3`, где можно явно проверить несколько ненулевых
   элементов результата вручную;
2. очень разреженный случай, где важно не создавать ложные ненули;
3. более плотный случай, где возрастает число накоплений в уже встреченные
   столбцы.

Таким образом, OMP-версия проверялась не только на “все прошло”, но и на
совпадение с baseline по конкретным сценариям разной плотности.

Интерпретация:

- переход от `1` к `2` и `4` потокам дает заметный выигрыш;
- область `4–8` потоков оказывается лучшей по времени;
- дальнейший рост числа потоков уже не дает линейного ускорения.

Основные узкие места:

- задача memory-bound из-за нерегулярного доступа к строкам `B`;
- итоговая CRS-сборка остается последовательной;
- отсутствие явно подобранного `schedule` оставляет распределение строк на
  усмотрение runtime, что не всегда оптимально для неравномерной нагрузки.

## 9. Выводы

OpenMP дал выигрыш на этой задаче, потому что строки результата можно вычислять
независимо и достаточно дешево распараллелить через `omp for`. При этом выигрыш
ограничен не арифметикой как таковой, а двумя факторами:

- нерегулярным доступом к памяти в разреженном умножении;
- последовательной фазой сборки итогового CRS после завершения parallel region.

Методически OMP-версия получилась корректной: область данных и синхронизация
описаны явно, гонки на аккумуляторе устранены локальными буферами, а неявный
барьер используется осознанно как граница перед финальной сборкой результата.
