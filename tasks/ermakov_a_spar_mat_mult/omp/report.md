## 1. Контекст

OpenMP-версия переносит baseline-алгоритм CRS-умножения в модель общей памяти.
Цель этой реализации - распараллелить независимую обработку строк результата,
сохранив ту же математическую схему, что и в `seq/report.md`.

## 2. Постановка задачи

На вход подаются две комплексные CRS-матрицы `A` и `B`. На выходе требуется
получить `C = A * B` в формате CRS.

Baseline для корректности и для формулы ускорения - последовательная версия:

- критерий корректности результата тот же;
- ограничения на вход те же;
- `speedup = T_seq / T_omp`.

## 3. Базовый алгоритм

Последовательная версия для каждой строки `i` матрицы `A` просматривает все ее
ненулевые элементы `A[i, j]`, затем проходит по строке `j` матрицы `B` и
накапливает произведения `A[i, j] * B[j, k]` по столбцам `k`.

OpenMP-версия не меняет математику. Она меняет только организацию исполнения:
строки результата вычисляются параллельно, а итоговая CRS-структура собирается
после завершения parallel region.

## 4. Схема распараллеливания

Параллелится центральный цикл по строкам матрицы `A` в `RunImpl`.

```cpp
#pragma omp parallel default(none) shared(m, p, row_values, row_cols)
{
  std::vector<std::complex<double>> row_vals(static_cast<std::size_t>(p),
                                             std::complex<double>(0.0, 0.0));
  std::vector<int> row_mark(static_cast<std::size_t>(p), -1);
  std::vector<int> used_cols;
  used_cols.reserve(256);

#pragma omp for
  for (int i = 0; i < m; ++i) {
    AccumulateRowProducts(i, row_vals, row_mark, used_cols);
    SortUsedCols(used_cols);
    CollectRowValues(row_vals, used_cols, row_cols[static_cast<std::size_t>(i)],
                     row_values[static_cast<std::size_t>(i)]);
  }
}
```

Расшифровка решений:

- `default(none)` выбран специально, чтобы все общие данные были перечислены явно;
- `shared(m, p, row_values, row_cols)` означает, что размеры задачи и массивы
  построчных результатов общие для всех потоков;
- `row_vals`, `row_mark`, `used_cols` создаются внутри parallel region, поэтому
  каждый поток получает собственные private-буферы;
- `reduction` не нужен, потому что общего аккумулятора нет;
- `atomic` и `critical` не нужны, потому что потоки не пишут в одну и ту же строку;
- `schedule(...)` явно не задан, то есть разбиение итераций оставлено runtime OpenMP;
- после `#pragma omp parallel` существует неявный барьер, и здесь он нужен,
  потому что только после него можно безопасно собирать итоговый CRS.

Число потоков задается окружением запуска, прежде всего `OMP_NUM_THREADS`.

## 5. Детали реализации

Файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Относительно SEQ были изменены следующие участки:

- вместо немедленной записи в итоговый CRS введены построчные контейнеры
  `row_values` и `row_cols`;
- `AccumulateRowProducts` выносит вычисление одной строки в отдельный helper;
- `CollectRowValues` отделяет фазу накопления от фазы упаковки строки;
- финальная сборка `c_` вынесена за пределы parallel region.

Риски гонок были устранены так:

- рабочие буферы строки private;
- каждая итерация `i` записывает данные только в `row_values[i]` и `row_cols[i]`.

## 6. Проверка корректности

Корректность OMP-версии проверялась сравнением с SEQ и с плотным эталоном.

Использованные функциональные тесты:

- `SmallFixed`, размер `3x3`;
- `VerySparse`, размер `10x10`;
- `MediumSparse`, размер `20x20`;
- `Dense`, размер `30x30`.

Характерные случаи:

1. фиксированный пример `3x3`, где можно вручную проверить несколько ненулевых
   элементов результата;
2. очень разреженный случай, где важно не создавать ложные ненули;
3. более плотный случай, где возрастает число накоплений в уже встреченные столбцы.

```powershell
$env:OMP_NUM_THREADS='8'; .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_omp_enabled*"
$env:OMP_NUM_THREADS='8'; .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_omp_enabled"
```

Размер performance-задачи:

- `15000 x 15000`;
- плотность `0.001`;
- детерминированная генерация входа.

Основные узкие места:

- memory-bound характер задачи из-за нерегулярного доступа к строкам `B`;
- последовательная финальная сборка итогового CRS;

## 9. Выводы

OpenMP на этой задаче дает только умеренный выигрыш и лучше всего показывает
себя около `4` потоков. Дальше рост числа потоков уже не компенсирует
нерегулярный доступ к памяти и последовательную финализацию результата.
