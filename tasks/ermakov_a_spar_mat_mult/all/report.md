## 1. Контекст

Гибридная версия сочетает два уровня параллелизма:

- межпроцессный уровень на MPI;
- внутрипроцессный уровень на OpenMP.

Такая схема позволяет распределить строки матрицы `A` между
MPI-процессами, а локальное умножение внутри каждого rank-а
выполнять многопоточно.

## 2. Постановка задачи

На вход подаются две комплексные матрицы `A` и `B` в формате CRS.
На выходе требуется получить матрицу `C = A * B` в том же формате.

Базовая постановка, входные ограничения и критерий корректности
описаны в
`ppc-2026-threads/tasks/ermakov_a_spar_mat_mult/seq/report.md`.
В этом отчете фиксируется именно гибридная схема
`ranks x threads`, где:

- `ranks` - число MPI-процессов;
- `threads` - число OpenMP-потоков внутри каждого rank-а;
- `total_workers = ranks * threads`.

Метрики:

- `speedup = T_seq / T_all`;
- `efficiency = speedup / total_workers`.

## 3. Базовый алгоритм

Математическая логика совпадает с последовательной версией:

1. для строки `i` матрицы `A` перебираются ее ненулевые элементы
   `A[i, j]`;
2. по каждому такому элементу просматривается строка `j`
   матрицы `B`;
3. произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`;
4. затронутые столбцы сортируются;
5. строка переносится в результирующий CRS.

В гибридной версии эта схема организована так:

- матрица `B` рассылается всем rank-ам;
- строки `A` делятся между rank-ами;
- каждый rank считает свой локальный блок строк;
- внутри rank-а локальные строки обрабатываются через OpenMP;
- частичные CRS-результаты собираются на `rank 0`;
- в functional-режиме итоговая матрица дополнительно
  рассылается всем rank-ам.

## 4. Межпроцессная схема

Роли rank-ов:

- `rank 0` вычисляет схему разбиения строк, инициирует
  коллективные MPI-операции и собирает глобальный результат;
- остальные rank-ы получают локальный блок строк `A`,
  полную матрицу `B` и считают свой локальный фрагмент `C`.

Ключевые MPI-операции:

- `MPI_Comm_rank`, `MPI_Comm_size` - определяют текущий ранг
  и размер коммуникатора;
- `BroadcastMatrix(b_, rank)` - рассылает матрицу `B`
  всем rank-ам;
- `MPI_Bcast(row_bounds...)` и `MPI_Bcast(nnz_counts...)` -
  рассылают границы блоков строк и размеры участков CRS;
- `MPI_Scatterv(...)` внутри `ScatterRows(...)` -
  передает каждому процессу его блок строк матрицы `A`;
- `MPI_Gather(...)` и `MPI_Gatherv(...)` внутри
  `GatherMatrix(...)` - собирают локальные результаты
  на `rank 0`;
- `MPI_Barrier(...)` - завершает коллективную фазу
  перед выходом из `RunImpl()`.

Разбиение строк строится функцией
`BuildRowBounds(const MatrixCRS &a, const MatrixCRS &b, int proc_count)`.
Балансировка выполняется не по простому числу ненулевых элементов
строки `A`, а по оценке ее вычислительной стоимости: для строки
`A[i, *]` суммируются длины строк `B`, к которым она обращается.
Такое разбиение лучше соответствует реальной нагрузке
при `CRS x CRS`.

Комплексные значения передаются через собственный MPI-тип
`GetComplexBytesType()`, построенный на основе `MPI_BYTE`
и размера `std::complex<double>`. Благодаря этому значения
`values` рассылаются и собираются напрямую, без промежуточного
преобразования в массивы `double`.

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса локальный блок строк умножается
функцией `MultiplyLocalOMP(local_a, b_)`.

Как делится работа между потоками:

- `local_a` содержит только строки, назначенные текущему rank-у;
- для каждой строки создается локальная структура `LocalRowData`;
- цикл по строкам делится директивой `#pragma omp for`.

Как устроена локальная вычислительная схема:

- поток использует собственные буферы `row_vals`,
  `row_mark`, `used_cols`;
- `AccumulateRowProducts(...)` накапливает вклад строки результата;
- `CollectRowValues(...)` сортирует затронутые столбцы
  и переносит ненулевые значения в CRS-представление;
- после завершения OpenMP-области локальный `MatrixCRS`
  собирается последовательно из `rows_data`.

Число потоков внутри rank-а задается через `PPC_NUM_THREADS`,
но дополнительно ограничивается аппаратной конкуренцией
`std::thread::hardware_concurrency()` с учетом числа
MPI-процессов. Это уменьшает риск oversubscription
в конфигурациях с большим числом rank-ов.

## 6. Детали реализации

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

Что делают основные части:

- `ValidationImpl()` проверяет совместимость размеров
  и корректность CRS-структуры;
- `PreProcessingImpl()` копирует `A`, `B`
  и инициализирует `C`;
- `RunImpl()` выполняет весь распределенный сценарий:
  рассылку `B`, разбиение строк, `ScatterRows(...)`,
  локальное умножение и `GatherMatrix(...)`;
- `PostProcessingImpl()` переносит `c_` в `GetOutput()`.

Ключевые участки кода:

- `BuildRowBounds(a_, b_, size)` - строит границы строк
  по оценке вычислительной стоимости;
- `ScatterRows(...)` - формирует локальный CRS-блок строк `A`;
- `MultiplyLocalOMP(...)` - считает локальный результат
  внутри rank-а;
- `GatherMatrix(...)` - собирает глобальный CRS на `rank 0`.

Потенциальные узкие места:

- полная рассылка матрицы `B` всем rank-ам;
- финальная сборка результата на `rank 0`;
- чувствительность к конфигурации `ranks x threads`;
- неравномерная структура `A` и `B`, из-за которой даже
  балансировка по стоимости не гарантирует идеального
  распределения нагрузки.

## 7. Проверка корректности

Корректность подтверждалась сравнением с последовательной
версией и общим функциональным набором:

- `SmallFixed (3x3)`;
- `VerySparse (10x10)`;
- `MediumSparse (20x20)`;
- `Dense (30x30)`.

Functional-тесты сравнивают ответ с плотным эталоном
`DenseMul`, а затем проверяют совпадение структур
`row_ptr`, `col_index` и комплексных значений
с допуском по вещественной и мнимой части.

В functional-режиме после сборки результата на `rank 0`
итоговая матрица рассылается всем rank-ам через
`BroadcastMatrix(c_, rank)`, поэтому `PostProcessingImpl()`
на любом процессе получает согласованный результат.

## 10. Выводы

Гибридная схема `MPI + OMP` для этой задачи оправдана
в умеренных конфигурациях `ranks x threads`, где цена
межпроцессного обмена еще не доминирует над локальным
умножением. Наиболее удачным режимом в проведенных
замерах оказался `4 x 4`.

Ключевые свойства текущей реализации:

- разбиение строк учитывает ожидаемую стоимость
  локального умножения;
- комплексные значения передаются через собственный
  MPI-тип напрямую;
- число потоков внутри rank-а ограничивается
  с учетом аппаратной конкуренции.

При дальнейшем росте числа процессов и потоков начинают
преобладать расходы на рассылку `B`, синхронизацию и
сборку итогового CRS на `rank 0`, поэтому конфигурации
`12 x 12` и `16 x 16` теряют эффективность.
