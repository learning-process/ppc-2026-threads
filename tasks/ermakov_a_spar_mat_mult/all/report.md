## 1. Контекст

Гибридная версия совмещает два уровня параллелизма:

- межпроцессный уровень на MPI;
- внутрипроцессный уровень на OpenMP.

## 2. Постановка задачи

На вход подаются две комплексные матрицы `A` и `B` в формате CRS, на выходе
требуется получить `C = A * B` в том же формате.

Полная постановка, входные ограничения и baseline по корректности описаны в
`seq/report.md`, а общая сравнительная методика - в корневом `report.md`.

В этом отчете конфигурация задается парой `ranks x threads`. Например,
`4 x 4` означает:

- `4` MPI-процесса;
- `4` OpenMP-потока внутри каждого rank-а;
- всего `16` workers.

Ускорение считается так:

- `speedup = T_seq / T_all`;
- `efficiency = speedup / total_workers`,
  где `total_workers = ranks * threads_per_rank`.

## 3. Базовый алгоритм

Математическая логика совпадает с последовательной версией:

1. для строки `i` матрицы `A` перебираются ее ненулевые элементы `A[i, j]`;
2. по каждому такому элементу просматривается строка `j` матрицы `B`;
3. произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`;
4. затронутые столбцы сортируются;
5. строка переносится в результирующий CRS.

Гибридная реализация меняет только организацию исполнения:

- строки `A` сначала распределяются между MPI-процессами;
- затем локальные строки на каждом rank-е умножаются в OpenMP;
- после этого частичные CRS-результаты собираются обратно.

## 4. Межпроцессная схема

Роли rank-ов:

- `rank 0` строит разбиение строк `A`, готовит размеры фрагментов и собирает итог;
- остальные rank-и получают свои локальные блоки и участвуют в вычислении.

Распределение данных:

- `BuildRowBounds(a_, size)` строит границы строк для процессов;
- `BuildNNZCounts(a_, row_bounds)` вычисляет размеры фрагментов;
- `ScatterRows(...)` рассылает локальные блоки матрицы `A`;
- `BroadcastMatrix(b_, rank)` рассылает полную матрицу `B` всем rank-ам.

Какие MPI-вызовы используются и зачем:

- `MPI_Comm_rank` и `MPI_Comm_size` определяют ранг и размер коммуникатора;
- `MPI_Bcast` используется для `B`, `row_bounds`, `nnz_counts`;
- `MPI_Scatterv` передает разные по размеру фрагменты `A`;
- `MPI_Gather` и `MPI_Gatherv` собирают частичные CRS-блоки обратно на `rank 0`;
- `MPI_Barrier` фиксирует точку, в которой все процессы завершили коллективную фазу.

Комплексные значения перед обменами упаковываются в массивы `double` через
`PackComplexValues` и восстанавливаются через `UnpackComplexValues`.

## 5. Внутрипроцессная схема

Внутри каждого rank-а используется OpenMP-функция `MultiplyLocalOMP(local_a, b_)`.

Как делится работа между потоками:

- `local_a` уже содержит только строки, принадлежащие данному rank-у;
- внутри `MultiplyLocalOMP` создается `rows_data` для локального числа строк;
- число потоков выбирается как
  `thread_count = max(1, min(ppc::util::GetNumThreads(), a.rows))`;
- цикл по локальным строкам делится директивой `#pragma omp for`.

Как объединяются локальные результаты:

- каждый поток использует собственные буферы `row_vals`, `row_mark`, `used_cols`;
- для каждой локальной строки формируется `LocalRowData`;
- после завершения OpenMP-области локальный CRS-результат собирается
  последовательно в `result`.

Для этой реализации число потоков внутри rank-а задается через `PPC_NUM_THREADS`,
потому что код читает его через `ppc::util::GetNumThreads()`.

## 6. Детали реализации

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

Где стоят основные коллективные операции:

- `BroadcastMatrix(b_, rank)` - в начале `RunImpl`;
- `MPI_Bcast(row_bounds...)` и `MPI_Bcast(nnz_counts...)` - после подготовки схемы разбиения;
- `ScatterRows(...)` - перед локальным умножением;
- `GatherMatrix(...)` - после локального умножения;
- `BroadcastMatrix(c_, rank)` - в функциональном режиме после сборки результата;
- `MPI_Barrier(MPI_COMM_WORLD)` - в конце коллективной фазы.

Зачем здесь нужен `MPI_Barrier`:

- в performance-режиме он гарантирует, что все rank-и завершили вычисление и
  сборку до конца измеряемой коллективной работы;
- в functional-режиме он отделяет завершившуюся рассылку итоговой матрицы `C`
  от выхода из `RunImpl`.

Потенциальные узкие места:

- рассылка полной матрицы `B`;
- упаковка и распаковка комплексных значений;
- финальная сборка результата на `rank 0`;
- неравномерность числа ненулевых элементов по строкам.

## 7. Проверка корректности

Корректность гибридной версии подтверждалась тремя уровнями:

- сравнение с baseline `SEQ`;
- сравнение с плотным эталоном `DenseMul`;
- проверка согласованности результата между rank-ами.

Использованные тесты:

- `SmallFixed (3x3)`;
- детерминированные входы `10x10`, `20x20`, `30x30`;
- performance-вход `15000 x 15000` с плотностью `0.001`.

Согласованность между rank-ами обеспечивается тем, что в функциональном режиме
итоговая матрица после сборки на `rank 0` дополнительно рассылается всем через
`BroadcastMatrix(c_, rank)`.

## 8. Экспериментальная среда

- CPU: AMD Ryzen 5 1600
- RAM: 8 GB
- OS: Windows 11
- compiler: MSVC
- build type: `Release`

Конфигурация задается так:

- `mpiexec -n` определяет число rank-ов;
- `PPC_NUM_THREADS` определяет число OpenMP-потоков внутри rank-а;
- в таблицах ниже конфигурация записывается как `ranks x threads`.

Команды запуска:

```powershell
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake -S . -B build -D ENABLE_ADDRESS_SANITIZER=ON -D CMAKE_BUILD_TYPE=RelWithDebInfo
cmake --build build --config Release --parallel
cd build\bin
$env:PPC_NUM_THREADS='4'; mpiexec -n 4 .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_all_enabled*"
$env:PPC_NUM_THREADS='4'; mpiexec -n 4 .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_all_enabled"
```

Размер performance-задачи:

- `15000 x 15000`;
- плотность `0.001`;
- детерминированная генерация CRS-входа.

Комментарий о цене коммуникации и синхронизации:

- даже лучшая медианная конфигурация `4 x 4` остается медленнее `SEQ`;
- уже на `2 x 2` к локальному OpenMP-вычислению добавляется ощутимая цена
  `Broadcast`, `Scatterv`, `Gatherv`, упаковки значений и финальной сборки;
- при больших конфигурациях время определяется уже не локальным умножением,
  а коммуникациями и сборкой на `rank 0`;
- поэтому эффективность, нормированная по `total_workers`, быстро падает.

## 10. Выводы

Гибридная схема здесь методически корректна: она действительно совмещает
MPI-распределение строк и локальное OpenMP-умножение. Однако на данной машине
и для данной задачи она не дает реального ускорения относительно baseline `SEQ`.
Причиной становятся коммуникации, упаковка данных и централизованная финальная
сборка результата.
