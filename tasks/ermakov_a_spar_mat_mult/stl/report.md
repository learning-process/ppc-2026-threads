## 1. Контекст

Ручная `std::thread`-версия нужна для того, чтобы реализовать многопоточное
умножение CRS-матриц без OpenMP и без runtime oneTBB. Здесь программист сам
отвечает за разбиение работы, запуск потоков, локальные буферы и порядок
`create -> work -> join`.

## 2. Постановка задачи

На вход подаются две разреженные матрицы `A` и `B` в формате CRS, на выходе
требуется получить `C = A * B` в том же формате.

SEQ считается эталоном:

- `speedup = T_seq / T_stl`;
- `efficiency = speedup / threads`.

## 3. Базовый алгоритм

Последовательная логика не меняется:

1. для строки `i` матрицы `A` перебираются все `A[i, j]`;
2. для каждого такого элемента просматривается строка `j` матрицы `B`;
3. произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`;
4. затронутые столбцы сортируются;
5. строка переносится в итоговый CRS.

STL-версия меняет только организацию исполнения: строки результирующей матрицы
разбиваются на поддиапазоны и вручную распределяются между worker-потоками.

## 4. Схема распараллеливания

Разбиение делается не по фиксированному числу строк, а по оценке трудоемкости:

- `BuildRowCosts(a_, b_)` строит стоимость каждой строки;
- `ResolveThreadCount(a_.rows, total_work)` выбирает число потоков с учетом
  `PPC_NUM_THREADS`, числа строк и минимального объема полезной работы;
- `BuildThreadBoundaries(row_costs, thread_count)` превращает стоимости строк
  в набор границ `boundaries`.

Что делает каждый поток:

- создает собственный `Workspace`;
- обрабатывает только строки своего диапазона;
- для каждой строки вызывает `MultiplyRow(...)`;
- записывает результат только в `rows_data[row]`.

Где хранятся локальные результаты:

- внутри потока - в `Workspace`;
- после завершения строки - в `rows_data[row]`.

Синхронизация:

- `atomic`, `mutex`, `condition_variable` не используются;
- это безопасно, потому что каждый поток работает со своим `Workspace`
  и со своим диапазоном строк;
- единственная обязательная точка синхронизации - `join()` после запуска
  всех worker-потоков.

Критически важно, что `join()` вызывается после создания всех потоков,
а не внутри цикла создания.

## 5. Детали реализации

Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Карта `поток -> диапазон` строится через `boundaries`:

- поток `thread_id` получает диапазон
  `[boundaries[thread_id], boundaries[thread_id + 1])`;
- границы выбираются так, чтобы уравнять суммарную стоимость строк.

Схема объединения:

1. каждый поток независимо заполняет свои элементы `rows_data`;
2. после завершения всех потоков главный поток вызывает `FinalizeResult(rows_data)`;
3. `FinalizeResult` последовательно собирает итоговые `row_ptr`, `col_index`,
   `values`.

## 6. Проверка корректности

Корректность STL-версии проверялась сравнением с эталонным результатом:

- `SmallFixed (3x3)`;
- детерминированные матрицы размеров `10x10`, `20x20`, `30x30`;
- сравнение с плотным эталоном `DenseMul`;
- сравнение CRS-структуры и всех комплексных значений с SEQ.

Крайние случаи:

- несовместимые размеры блокируются в `ValidationImpl`;
- некорректная CRS-структура блокируется `ValidateMatrix`;
- пустой результат корректно дает пустой CRS без запуска многопоточной фазы.

Проверка на отсутствие гонок:

- каждый поток использует собственный `Workspace`;
- каждый поток пишет только в свой диапазон `rows_data`;
- общий итоговый CRS собирается только после всех `join()`.

  потоков здесь управляются вручную.

## 9. Выводы

Ручная реализация на `std::thread` показала, что задача хорошо распараллеливается
и без специализированного runtime, если аккуратно разделить строки между потоками
и вынести все промежуточные буферы в thread-local память. Ограничение подхода -
отсутствие автоматической балансировки нагрузки и заметная цена `create/join`.
