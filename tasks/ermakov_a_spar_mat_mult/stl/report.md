## 1. Контекст

Ручная `std::thread`-версия нужна для того, чтобы реализовать многопоточное умножение
CRS-матриц без OpenMP и без runtime oneTBB. В этой реализации программист сам отвечает за
разбиение работы, запуск потоков, организацию локальных буферов и порядок
`create -> work -> join`.

## 2. Постановка задачи

На вход подаются две разреженные матрицы `A` и `B` комплексного типа в формате CRS:

- `rows`, `cols` — размеры матрицы;
- `values` — ненулевые комплексные элементы;
- `col_index` — номера столбцов для каждого ненулевого элемента;
- `row_ptr` — указатели на начало строк.

Требуется вычислить произведение `C = A * B` и вернуть его в том же формате CRS.
Последовательная версия из `seq/report.md` считается эталоном по корректности результата
и baseline для вычисления ускорения:

- `speedup = T_seq / T_stl`;
- `efficiency = speedup / threads`.

## 3. Базовый алгоритм

Последовательная логика не меняется относительно baseline:

1. для строки `i` матрицы `A` перебираются все ненулевые элементы `A[i, j]`;
2. по каждому такому элементу просматривается строка `j` матрицы `B`;
3. произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`;
4. затронутые столбцы сортируются;
5. ненулевые значения строки переносятся в итоговую CRS-структуру.

STL-версия меняет только организацию исполнения: строки результирующей матрицы
разбиваются на поддиапазоны, которые вручную распределяются между worker-потоками.

## 4. Схема распараллеливания

Данные делятся не по фиксированному числу строк, а по оценке трудоемкости:

- `BuildRowCosts(a_, b_)` строит стоимость каждой строки как сумму длин строк `B`,
  которые будут затронуты при умножении;
- `ResolveThreadCount(a_.rows, total_work)` выбирает число потоков с учетом
  `PPC_NUM_THREADS`, числа строк и минимального объема полезной работы;
- `BuildThreadBoundaries(row_costs, thread_count)` преобразует массив стоимостей в
  набор границ `boundaries`, задающих полуинтервалы `[row_begin, row_end)`.

Что делает каждый поток:

- создает собственный объект `Workspace`;
- обрабатывает только строки своего диапазона;
- для каждой строки вызывает `MultiplyRow(...)`;
- записывает результат только в `rows_data[row]`, где `row` принадлежит его диапазону.

Локальные результаты хранятся в двух уровнях:

- внутри потока — в `Workspace`, где находятся `accum`, `marks`, `touched_cols`;
- после завершения строки — в `rows_data[row]`, где лежат готовые `cols` и `vals`.

Синхронизация:

- `atomic`, `mutex`, `condition_variable` не используются;
- это безопасно, потому что у потоков нет общей записи в один `Workspace`, а строки
  результата заранее разделены между потоками;
- единственная обязательная точка синхронизации — `join()` после запуска всех
  `worker`-потоков.

Критически важно, что `join()` вызывается не внутри цикла создания потоков, а после
того как все `std::thread` уже созданы. Поэтому реализация действительно выполняется
параллельно, а не сериализуется искусственно.

## 5. Детали реализации

Файлы реализации:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Карта `поток -> диапазон` строится через массив `boundaries`:

- поток с номером `thread_id` получает диапазон
  `[boundaries[thread_id], boundaries[thread_id + 1])`;
- границы выбираются так, чтобы уравнять суммарную стоимость строк, а не просто
  поделить `a_.rows` на равные куски.

Схема объединения локальных результатов:

1. каждый поток независимо заполняет свои элементы `rows_data`;
2. после завершения всех потоков главный поток вызывает `FinalizeResult(rows_data)`;
3. `FinalizeResult` последовательно рассчитывает `row_ptr`, общее число ненулевых
   элементов и копирует построчные `cols`/`vals` в финальные массивы `c_`.

С точки зрения потокобезопасности ключевое решение состоит в том, что каждая строка
результата формируется ровно одним потоком. Поэтому защита записи не нужна ни для
аккумулятора строки, ни для массива строковых результатов.

## 6. Проверка корректности

Корректность STL-версии проверялась сравнением с эталонным результатом:

- фиксированный пример `3x3` (`SmallFixed`);
- детерминированные разреженные матрицы размеров `10x10`, `20x20`, `30x30`;
- сравнение с плотным эталоном `DenseMul`, построенным в `tests/functional/main.cpp`;
- сравнение CRS-структуры и всех комплексных значений с baseline-версией.

Крайние случаи:

- несовместимые размеры матриц блокируются в `ValidationImpl`;
- некорректная CRS-структура блокируется `ValidateMatrix`;
- пустая матрица или нулевое число столбцов корректно приводят к пустому результату
  без запуска многопоточной фазы.

Аргумент против гонок:

- каждый поток использует собственный `Workspace`;
- каждый поток пишет только в свой диапазон `rows_data`;
- общий итоговый CRS собирается только после завершения всех `join()`.

| 16      | 0.0793984602 | 2.121   | 0.133      |

Комментарий о накладных расходах `create/join`:

- переход от `1` к `4` потокам дает заметный выигрыш, потому что стоимость строк
  достаточно велика, чтобы окупить создание потоков;
- диапазон `8–12` потоков дает максимальный эффект, после чего ускорение насыщается;
- при `16` потоках рост overhead на создание, планирование и синхронизацию начинает
  конкурировать с полезной работой;
- STL-версия все еще уступает TBB, потому что балансировка и управление жизненным
  циклом потоков здесь реализованы вручную и менее гибки, чем runtime oneTBB.

## 9. Выводы

Ручная реализация на `std::thread` показала, что задача хорошо распараллеливается и без
специализированного runtime, если аккуратно разделить строки между потоками и вынести
все промежуточные буферы в thread-local память. Сильная сторона этой версии —
прозрачность: в коде явно видны формула разбиения диапазонов, жизненный цикл потоков и
схема объединения результата. Ограничение подхода в том, что `std::thread` не дает
автоматической балансировки нагрузки, поэтому качество результата сильнее зависит от
ручного выбора числа потоков и от накладных расходов на `create/join`.
