## 1. Контекст

Для этой задачи TBB-версия интересна тем, что позволяет описать
распараллеливание не через явное управление потоками, а через задачно-
ориентированную декомпозицию диапазонов строк. Для разреженного умножения CRS это
удобно: вычисление по строкам естественно выражается через `parallel_for`, а
локальные буферы можно хранить в thread-local workspace.

## 2. Постановка задачи

Постановка задачи совпадает с baseline-версией из `seq/report.md`: на вход
подаются две комплексные CRS-матрицы `A` и `B`, на выходе требуется получить
матрицу `C = A * B` в том же формате.

Baseline-реализация остается эталоном:

- по корректности результата;
- по определению входных ограничений;
- по формуле `speedup = T_seq / T_tbb`.

## 3. Базовый алгоритм

Математическая логика не меняется относительно SEQ:

1. для строки `i` матрицы `A` просматриваются все ненулевые элементы `A[i, j]`;
2. для каждого такого элемента берется строка `j` матрицы `B`;
3. произведения `A[i, j] * B[j, k]` накапливаются по столбцам `k`;
4. после завершения строки затронутые столбцы сортируются;
5. ненулевые значения сохраняются в итоговую CRS-структуру.

TBB-версия меняет только организацию исполнения: строки распределяются по
worker-потокам через диапазоны `blocked_range`, а промежуточные буферы хранятся
в отдельном workspace на поток.

## 4. Схема распараллеливания

Выбранный примитив — `oneapi::tbb::parallel_for`.

Текущая схема выглядит так:

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, m, grain_size), [&](const tbb::blocked_range<int> &range) {
  auto &local = workspace.local();

  for (int i = range.begin(); i != range.end(); ++i) {
    AccumulateRowProducts(i, local.row_vals, local.row_mark, local.used_cols);
    SortUsedCols(local.used_cols);
    CollectRowValues(local.row_vals, local.used_cols, row_cols[static_cast<std::size_t>(i)],
                     row_values[static_cast<std::size_t>(i)]);
  }
});
```

Расшифровка принятых решений:

- диапазон работы задается как `blocked_range<int>(0, m, grain_size)`, то есть
  параллелизм строится по строкам результирующей матрицы;
- каждая задача получает не одну строку, а блок строк, что уменьшает overhead на
  слишком мелкое дробление;
- `grain_size` вычисляется функцией `ResolveGrainSize(m)` и равен примерно
  `m / 16`, но не меньше `1`;
- это означает, что реализация целенаправленно избегает слишком маленьких chunk-ов,
  так как они увеличивают стоимость планирования задач относительно полезной
  работы;
- `partitioner` явно не задан, поэтому используется поведение по умолчанию runtime
  oneTBB;
- `global_control::max_allowed_parallelism` в этой реализации не используется;
- фактическая конкуренция контролировалась на уровне инфраструктуры запуска через
  число workers, задаваемое средой курса (`PPC_NUM_THREADS`), а не явным
  ограничителем внутри кода TBB.

TBB просто запускается на N потоках, а в коде задается только декомпозиция работы,
детальная политика планирования остается за runtime oneTBB.

## 5. Детали реализации

Файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

Ключевые элементы реализации:

- `RowWorkspace` хранит локальные буферы `row_vals`, `row_mark`, `used_cols`;
- `tbb::enumerable_thread_specific<RowWorkspace>` гарантирует, что каждый
  worker-поток использует собственный аккумулятор строки;
- `AccumulateRowProducts` выполняет арифметическое накопление;
- `CollectRowValues` переносит итог строки в построчные контейнеры;
- после завершения `parallel_for` итоговый CRS собирается последовательно.

С точки зрения памяти это означает:

- гонки на аккумуляторах строки отсутствуют;
- ложное разделение кэша специально не оптимизировалось через
  `cache_aligned_allocator` или `scalable_allocator`;
- то есть реализация делает ставку на простую и безопасную thread-local память,
  а не на дополнительную memory-tuning оптимизацию.

## 6. Проверка корректности

Корректность TBB-версии проверялась в том же наборе тестов, что и остальные
backend-ы:

- фиксированный пример `3x3`;
- детерминированные разреженные матрицы размеров `10`, `20`, `30`;
- сравнение с плотным эталоном `DenseMul`;
- сопоставление с результатом baseline-версии.

Тесты таким образом проверяют и граничные размеры, и разные уровни плотности
входных матриц, и совпадение итоговой CRS-структуры с эталоном.

## 7. Экспериментальная среда

Окружение:

- CPU: AMD Ryzen 5 1600;
- RAM: 8 GB;
- OS: Windows 11;
- compiler: MSVC;
- build type: `Release`.

Способ ограничения конкуренции:

- в коде `global_control` не используется;
- фактическое число worker-ов задавалось через среду запуска курса,
  в частности `PPC_NUM_THREADS`.

Команды запуска:

```powershell
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
cd build\bin
$env:PPC_NUM_THREADS='8'; .\ppc_func_tests.exe --gtest_filter="*ermakov_a_spar_mat_mult_tbb_enabled*"
$env:PPC_NUM_THREADS='8'; .\ppc_perf_tests.exe --gtest_filter="*task_run_ermakov_a_spar_mat_mult_tbb_enabled"
```

Размер performance-задачи:

- `15000 x 15000`;
- плотность `0.001`;
- детерминированная генерация CRS-входа.

Комментарий по balancing и overhead:

- TBB показывает лучший результат среди всех shared-memory реализаций;
- диапазон `4–8` worker-ов дает наибольший рост производительности;
- после `8` workers выигрыш почти насыщается, что указывает на ограничение не
  только планировщиком задач, но и характером самой memory-bound нагрузки;
- выбранный `grain_size` оказался достаточно крупным, чтобы не утонуть в
  overhead на задачах, и достаточно мелким, чтобы runtime мог балансировать
  работу между worker-ами.

## 9. Выводы

TBB особенно удобна для этой задачи тогда, когда нужно компактно описать
распараллеливание по строкам и при этом не управлять потоками вручную. Сильная
сторона реализации — сочетание `parallel_for`, `blocked_range` и thread-local
workspace. Overhead начинает мешать тогда, когда число worker-ов растет быстрее,
чем полезная вычислительная интенсивность задачи, а также когда память и
нерегулярный доступ к строкам `B` начинают ограничивать масштабируемость
сильнее, чем сам runtime TBB.
