## 1. Контекст
Данная реализация использует механизмы управления потоками, предоставляемые стандартной библиотекой C++ (`std::thread`). Это требует ручного контроля над жизненным циклом потоков, декомпозицией данных и барьерами синхронизации, в отличие от OMP или TBB.

## 2. Постановка задачи
Многопоточная поразрядная сортировка массива 32-битных знаковых целых чисел по неубыванию. В качестве базового эталона корректности и времени выполнения выступает последовательная реализация `seq` ([см. seq/report.md](../seq/report.md#2-постановка-задачи)).

## 3. Базовый алгоритм
Базовый алгоритм полностью аналогичен `SEQ`.
Идея распараллеливания заключается в двух этапах:
1. **Локальная сортировка**: исходный массив нарезается на независимые непересекающиеся подмассивы (чанков), каждый из которых сортируется в отдельном `std::thread`.
2. **Параллельное иерархическое слияние**: отсортированные чанки попарно сливаются в виде бинарного дерева. На каждом уровне дерева слияние пар происходит параллельно, с барьером `join()` в конце каждого уровня.

## 4. Схема распараллеливания
Алгоритм распараллелен по принципу data-parallelism с ручным контролем потоков:
- **Разбиение данных**: Входной массив нарезается на `num_threads` чанков (где `num_threads` вычисляется через `ppc::util::GetNumThreads()`). Если массив не делится нацело, размер чанка вычисляется с округлением вверх: `chunk_size = (size + threads - 1) / threads`. Первые потоки получают `chunk_size` элементов, а последний поток обрабатывает оставшийся "хвост". 
- **Работа потока**: Главный поток (Main Thread) в цикле порождает `N` рабочих потоков (Workers). Каждый worker получает индекс `i` (по значению, чтобы избежать гонки) и извлекает свой диапазон из вектора конфигураций `ranges`. Поток выполняет LSD Radix Sort строго внутри отрезка `[range.begin, range.end)`.
- **Локальные результаты**: Временные результаты не требуют создания множества новых векторов. Два глобальных вектора (`output` и `buffer`) переиспользуются. Благодаря строгим границам чанков каждый поток читает и пишет только в свой выделенный участок памяти, что полностью устраняет False Sharing и состояние гонки данных (Data Race).
- **Синхронизация и `join`**: Для обеспечения безопасности используется модель Fork-Join.
  ```cpp
  // File: stl/src/ops_stl.cpp
  std::vector<std::thread> workers;
  // 1. Fork (создание потоков)
  for (int i = 0; i < active_ranges; ++i) {
    workers.emplace_back([&, i]() { /* локальная сортировка */ });
  }
  // 2. Join (барьер синхронизации)
  for (auto &worker : workers) {
    worker.join();
  }
  ```
  Метод `join()` вызывается в отдельном цикле после создания всех потоков. Это критически важное решение, так как вызов `join()` внутри цикла создания привел бы к последовательной работе потоков. 
- Из-за абсолютной независимости чанков отпала необходимость в примитивах синхронизации вроде `std::mutex`, `std::atomic` или `std::condition_variable`.

## 5. Детали реализации
Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Ключевым отличием является ручная реализация параллельного слияния в `MergeSortedRanges`.
На каждом уровне дерева слияния количество пар чанков равно `merged_count = (current_ranges.size() + 1) / 2`.
Главный поток порождает `merged_count` потоков, каждый из которых вызывает `ProcessMergePair`.
В конце каждой итерации уровня дерева (главного цикла `while`) происходит барьерная синхронизация через `join()` всех потоков-мерджеров. Это гарантирует, что следующий уровень дерева начнет работу только когда все пары предыдущего уровня полностью слиты.

## 6. Проверка корректности
- **Сравнение с SEQ**: Полное функциональное тестирование на совпадение результатов.
- **Крайние случаи**: Массивы малого размера (где число потоков превышает число элементов) обрабатываются корректно, так как лишние потоки просто не получают валидных диапазонов (`range.begin < range.end` возвращает `false`).
- **Гонки данных (Data Races)**: Отсутствуют. Захват переменной цикла `i` в лямбду осуществляется по значению `[&, i]`, что устраняет самую частую проблему захвата ссылки на меняющийся итератор в `std::thread`. Чанки памяти строго разделены.

## 9. Выводы
Иерархическое параллельное слияние на базе чистых потоков позволило достичь хорошего разгона: пик масштабируемости (2.57x) достигается уже на 4 воркерах при эффективности в 64%. Однако, при увеличении числа потоков до 6 (максимум физических ядер) и далее до 10-12, наблюдается падение ускорения и значительное снижение эффективности (вплоть до 20%). 

Главным **узким местом** STL-версии является системный overhead. На каждом из уровней бинарного дерева слияний алгоритм заново делает системные вызовы на порождение `std::thread` и их синхронизацию через `join()`. В отличие от пула потоков TBB, операционная система тратит время на аллокацию контекстов. При превышении физического предела в 6 ядер (запуски на 10 и 12) ситуацию усугубляет переключение контекста. В итоге ручное распараллеливание работает отлично для небольшого количества потоков (до 4), но не справляется с накладными расходами при дальнейшем расширении пула.
