## 1. Контекст
В данном разделе описывается распараллеливание поразрядной сортировки целых чисел с использованием технологии OpenMP. Базовый последовательный алгоритм сортировки не поддается тривиальному распараллеливанию "в лоб" из-за жесткой зависимости по данным на каждом проходе. Поэтому осуществляется декомпозиция задачи: входной массив разбивается на независимые блоки (чанки), которые сортируются параллельно, а затем выполняется слияние отсортированных блоков.

## 2. Постановка задачи
Многопоточная поразрядная сортировка массива 32-битных знаковых целых чисел по неубыванию. Подробная постановка, включая ограничения и краевые случаи, описана в базовом отчете ([см. seq/report.md](../seq/report.md#2-постановка-задачи)).

## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).
Основные этапы:
1. Поиск минимального элемента массива для вычисления смещения `offset`, чтобы сделать все числа неотрицательными.
2. Итеративный проход по каждому из 4-х байт числа (от младшего к старшему разряду).
3. Применение устойчивой сортировки подсчетом (Counting Sort) на каждом проходе.

**Идея распараллеливания:**
Поскольку поразрядная сортировка содержит жесткую зависимость по данным, алгоритм распараллелен через разбиение данных. Исходный массив нарезается на независимые подмассивы. Эти чанки сортируются локально и параллельно каждым доступным потоком, а затем результаты сливаются в один итоговый отсортированный массив.

## 4. Схема распараллеливания
Алгоритм распараллелен по принципу разбиения данных (data-parallelism):
1. Количество рабочих потоков `num_threads` явно вычисляется на основе конфигурации окружения (с помощью `ppc::util::GetNumThreads()`). Это значение ограничивается размером самого массива, чтобы избежать создания лишних пустых потоков.
2. Входной массив делится на `num_threads` поддиапазонов (чанков) с помощью функции `BuildInitialRanges`. Если размер массива нацело не делится на количество потоков, размер базового чанка округляется в большую сторону: `chunk_size = (size + threads - 1) / threads`. В таком случае первые потоки получают чанки строго заданного размера `chunk_size`, а последний задействованный поток обрабатывает оставшийся "хвост" меньшего размера (безопасность границ обеспечивает `std::min(begin + chunk_size, data_size)`). Это обеспечивает корректную работу без выхода за пределы памяти.
3. Запускается параллельный регион:
```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(active_ranges, buffer, output, ranges) schedule(static) \
    num_threads(num_threads)
  for (int range_index = 0; range_index < active_ranges; ++range_index) {
    const Range range = ranges[static_cast<std::size_t>(range_index)];
    if (range.begin < range.end) {
      RadixSortRange(output, buffer, range.begin, range.end);
    }
  }
```

**Расшифровка директивы и схемы:**
- `default(none)`: Выбран для явного контроля области видимости всех переменных, предотвращая скрытые гонки данных.
- `shared(active_ranges, buffer, output, ranges)`: Векторы `output`, `buffer` и массив конфигурации `ranges` доступны всем потокам. Разделение доступа безопасно, так как каждый поток сортирует (читает и пишет) строго свой, непересекающийся с другими потоками отрезок памяти (`[range.begin, range.end)`).
- `private`: Индексная переменная `range_index` (неявно приватная для цикла `for`) и локальная копия `range` внутри итерации приватны для каждого потока.
- `reduction`: Не применяется, так как независимые чанки сортируются "на месте" и не модифицируют глобальных аккумуляторов. Атомарные операции (`atomic`, `critical`) также не нужны.
- `schedule(static)`: Размер всех чанков практически идентичен, и сложность поразрядной сортировки константна для одинакового количества чисел. `static` планировщик является оптимальным, так как минимизирует накладные расходы OpenMP Runtime без потери балансировки нагрузки.
- **Барьеры**: В конце директивы `parallel for` присутствует **неявный барьер**. Он критически важен: главный поток должен дождаться окончания работы всех рабочих потоков, прежде чем переходить к этапу слияния (`MergeSortedRanges`), иначе слияние начнет обрабатывать частично неотсортированные данные.
- **Слияние**: Функция `MergeSortedRanges` выполняется последовательно одним (главным) потоком.

## 5. Детали реализации
Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

По сравнению с SEQ, в реализацию добавлены:
- Функция `BuildInitialRanges` для нарезки исходного массива на независимые блоки (Range).
- Изолированная функция локальной поразрядной сортировки `RadixSortRange`, принимающая границы отрезка.
- Функция итеративного слияния отсортированных блоков `MergeSortedRanges`, использующая стандартный подход "слияния двух указателей" во вспомогательный буфер.
Риски гонок данных в `MergeSortedRanges` устранены путем вынесения функции за пределы параллельной секции OpenMP.

## 9. Выводы
Анализ экспериментальных данных показывает, что технология OpenMP успешно распараллеливает стадию независимой сортировки. Ускорение линейно растет до 4 потоков (2.44x), достигая предела масштабируемости на 6 потоках с пиковым ускорением 2.55x. При дальнейшем увеличении количества воркеров (10 и 12) наблюдается деградация производительности и падение эффективности ниже 20%. 

Главное **узкое место** данной реализации — последовательное слияние блоков (`MergeSortedRanges`), которое выполняется строго в один поток. Кроме того, поскольку на тестовом стенде CPU располагает лишь 6 физическими ядрами, создание более 6 потоков приводит к накладным расходам на переключение контекста, усугубляя деградацию.
