# Поразрядная сортировка целых чисел с простым слиянием — ALL
- Student: Мельник Иван Вадимович
- Technology: ALL (MPI + STL Threads)
- Variant: 17

## 1. Контекст
Гибридная версия (ALL) необходима для эффективного масштабирования задачи на кластерных архитектурах с распределенной памятью, где узлы (Nodes) обмениваются сообщениями по сети, а внутри каждого узла используется разделяемая память (Shared Memory) многоядерных процессоров. Это позволяет преодолеть ограничения масштабируемости как чистого MPI (слишком много процессов порождают избыточный сетевой трафик), так и чистых потоков (ограничены одним физическим узлом).

## 2. Постановка задачи
Реализовать многопоточную и распределенную поразрядную сортировку 32-битных знаковых целых чисел по неубыванию. В качестве единого эталона производительности и корректности для всей иерархии используется базовая реализация [SEQ](../seq/report.md). Сводная таблица и нормировки описаны в [корневом отчете](../report.md).

## 3. Базовый алгоритм
Последовательная логика не изменилась: LSD Radix Sort со смещением отрицательных чисел (от младшего байта к старшему) и простое слияние отсортированных чанков.

## 4. Межпроцессная схема
Архитектура строится на паттерне Master-Worker:
- **Роли rank-ов**: Процесс с рангом `0` (Master) берет на себя управляющую роль. Он владеет исходным массивом, вычисляет границы декомпозиции для всех процессов, а в конце выполняет финальное объединение результатов. Остальные процессы (`rank > 0`) играют роль вычислителей (Workers).
- **Распределение данных**: 
  1. `rank 0` вычисляет смещения (`displacements`) и размеры чанков (`send_counts`) для каждого процесса. 
  2. Эти метаданные рассылаются всем процессам с помощью `MPI_Bcast`.
  3. Исходный массив физически распределяется по процессам с помощью `MPI_Scatterv`. Каждый процесс получает свой `local` массив.
- **Сборка данных**: 
  1. После локальной сортировки отсортированные куски собираются обратно на `rank 0` с помощью `MPI_Gatherv`.
  2. `rank 0` выполняет финальное глобальное слияние.
  3. Финальный отсортированный массив рассылается всем процессам через `MPI_Bcast` (так как по требованиям интерфейса все процессы должны завершить работу с валидным массивом).
- **Барьеры**: Явный `MPI_Barrier` в самом алгоритме отсутствует, так как коллективные операции (`Scatterv`, `Gatherv`, `Bcast`) содержат неявную синхронизацию внутри коммуникатора `MPI_COMM_WORLD`.

## 5. Внутрипроцессная схема
Внутри каждого ранга (как Master, так и Worker) для утилизации многоядерности используется `std::thread`.
- **Деление работы**: Локальный массив, полученный процессом из `MPI_Scatterv`, снова нарезается на `num_threads` независимых блоков. Каждый поток (`std::thread`) параллельно выполняет LSD Radix Sort на своем блоке.
- **Объединение результатов**: Отсортированные потоками блоки сливаются в один отсортированный массив уровня процесса с помощью параллельного иерархического слияния на базе потоков (`MergeSortedRangesParallel`), барьером для которого служат циклы с вызовом `join()`.

## 6. Детали реализации
Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Потенциальные узкие места гибридной схемы:
1. **Трафик**: `MPI_Scatterv` и `MPI_Gatherv` пересылают весь массив по сети. На сверхбольших массивах пропускная способность интерконнекта может стать "бутылочным горлышком".
2. **Асимметрия нагрузки на финише**: В то время как локальная сортировка выполняется параллельно всеми рангами, финальное дерево слияний чанков, собранных через `Gatherv`, выполняется **только на `rank 0`** (в функции `GlobalMergeChunks`). Несмотря на то, что это слияние внутри `rank 0` распараллелено на `std::thread`, остальные ранги в этот момент простаивают в ожидании финального `MPI_Bcast`.

## 7. Проверка корректности
- **Сравнение**: В функциональных тестах результаты `ALL` побитово сравниваются с `SEQ`.
- **Согласованность rank-ов**: Благодаря финальному `MPI_Bcast(output.data(), ...)` гарантируется, что после выхода из `RunImpl` все процессы обладают идентичной, отсортированной копией массива, что валидируется на каждом ранге.
- Тесты стабильно проходят как на средних размерах, так и на смешанных конфигурациях процессов и потоков.

## 10. Выводы
Гибридная схема (MPI + `std::thread`) оправдана при обработке колоссальных массивов данных, не помещающихся в оперативную память одного узла. В этом случае накладные расходы на сетевые пересылки (`Scatterv`, `Gatherv`) окупаются за счет привлечения вычислительных мощностей и памяти распределенного кластера. При запуске на одной машине (в рамках одного узла) гибридная версия ожидаемо может проигрывать чистым thread-based реализациям (TBB, OMP) из-за тяжеловесных межпроцессных коммуникаций и простоя рабочих рангов во время финального слияния на Master-узле.
