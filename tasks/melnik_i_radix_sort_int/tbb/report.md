## 1. Контекст
Данный раздел описывает адаптацию поразрядной сортировки (Radix Sort) с использованием библиотеки oneTBB (Intel Threading Building Blocks). В отличие от OpenMP, который использует жесткую привязку итераций к потокам, TBB реализует концепцию пула потоков и алгоритм work-stealing, динамически балансируя нагрузку между доступными потоками.

## 2. Постановка задачи
Требуется реализовать многопоточную сортировку массива целых чисел с использованием oneTBB. Эталоном корректности и базовой производительности выступает последовательная реализация `seq` ([см. seq/report.md](../seq/report.md#2-постановка-задачи)).

## 3. Базовый алгоритм
Базовый алгоритм совпадает с последовательной версией ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).
Идея распараллеливания заключается в двух этапах:
1. **Локальная сортировка**: массив нарезается на независимые чанки, каждый из которых сортируется параллельно.
2. **Иерархическое слияние**: отсортированные чанки попарно сливаются в виде дерева. В отличие от OMP-версии, слияние на каждом уровне дерева также выполняется параллельно.

## 4. Схема распараллеливания
- **Примитив**: Для обоих этапов (и сортировки, и слияния) используется `tbb::parallel_for`.
- **Диапазон работы (`blocked_range`)**: Вместо того чтобы передавать индексы массива напрямую в `parallel_for`, алгоритм сначала разбивает массив на `num_ranges` чанков (где `num_ranges` зависит от желаемого уровня параллелизма). В `tbb::parallel_for` передается `tbb::blocked_range<std::size_t>(0, ranges.size())`. То есть задачи TBB итерируются не по элементам массива, а по макро-блокам (чанкам).
- **Grainsize**: В конструкторе `blocked_range` grainsize не указан явно, поэтому используется значение по умолчанию (1). Поскольку каждый чанк представляет собой огромный объем вычислений (сортировка подмассива или слияние), накладные расходы  на планирование задач минимальны, и grainsize является адекватным.
- **Partitioner**: Явно не указан, поэтому применяется стандартный `tbb::auto_partitioner`. Он эвристически определяет оптимальное разделение диапазона, чтобы минимизировать overhead пула потоков и обеспечить оптимальную загрузку кэша.
- **Контроль конкуренции**: Внутри задачи `num_ranges` вычисляется через `ppc::util::GetNumThreads()`. Фактическая конкуренция (размер пула worker-потоков) контролируется на уровне тестового фреймворка с помощью объекта `tbb::global_control(tbb::global_control::max_allowed_parallelism, threads)`, что является правильным архитектурным паттерном для ограничения числа потоков в TBB.

## 5. Детали реализации
Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Ключевым отличием TBB-версии от OMP является распараллеленный процесс слияния.
В функции `MergeSortedRanges`:
```cpp
// File: tbb/src/ops_tbb.cpp
    tbb::parallel_for(tbb::blocked_range<std::size_t>(0, merged_count),
                      [&](const tbb::blocked_range<std::size_t> &pair_block) {
      for (std::size_t pair_index = pair_block.begin(); pair_index < pair_block.end(); ++pair_index) {
        // ... слияние левого и правого чанков пары ...
      }
    });
```
На каждом шаге иерархического слияния количество пар уменьшается вдвое. TBB позволяет независимо сливать пары параллельно. Это устраняет строгое "узкое место" однопоточного слияния, характерное для реализации OMP. 
Для работы с памятью используются стандартные аллокаторы `std::vector`, так как потоки работают со строгими, непересекающимися границами (False Sharing исключен крупнозернистой декомпозицией).

## 6. Проверка корректности
- **Сравнение с SEQ**: Корректность подтверждена функциональными тестами, проверяющими побитовое совпадение с результатами работы эталона (`seq`).
- **Специфические тесты**: Успешно проходятся тесты с нечетным количеством чанков и размерами массивов, не кратными количеству worker-потоков. В этих случаях логика TBB и функции распределения безопасно обрабатывают "неполные" пары слияний.

## 9. Выводы
Результаты замеров демонстрируют эффективность задачно-ориентированной архитектуры oneTBB. Ускорение уверенно растет вместе с добавлением воркеров, достигая своего пика в 2.84x на 6 потоках. Важно отметить высокую эффективность (92%) при переходе от 1 к 2 воркерам. Предел масштабируемости ожидаемо зафиксирован на 6 воркерах — это напрямую связано с аппаратными ограничениями стенда (процессор имеет ровно 6 физических ядер без гипертрейдинга). При запуске на 10 и 12 воркерах производительность упирается в потолок из-за расходов на переключение контекста, они неизбежны.

Внедрение `tbb::parallel_for` для иерархического слияния устранило **узкое место**, наблюдавшееся в OMP. Главным ограничителем производительности в данной задаче выступает пропускная способность памяти при работе с глобальными массивами, а также жесткий лимит физических ядер.
