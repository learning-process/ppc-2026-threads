## 1. Контекст

Библиотека oneTBB (Threading Building Blocks) реализует задачно-ориентированную
модель параллелизма, в отличие от потоково-ориентированной модели OpenMP. Для
задачи умножения разреженных матриц TBB подходит идеально: количество ненулевых
элементов в разных строках может сильно отличаться, что создает неравномерную
вычислительную нагрузку. Использование планировщика TBB с его алгоритмом "кражи
задач" позволяет динамически балансировать нагрузку между ядрами процессора
эффективнее, чем жесткое распределение итераций.

## 2. Постановка задачи

Постановка задачи полностью идентична базовой спецификации, описанной в отчете
`seq/report.md`.

- **Входные данные:** Две разреженные матрицы в формате CRS (тип элементов
  `double`).
- **Выходные данные:** Результирующая матрица в формате CRS.
- **Ограничения:** Матрицы должны быть согласованы по размерностям, элементы по
  модулю меньше 1e-15 отсекаются.

Последовательная реализация (SEQ) выступает в качестве базы (baseline) для
проверки математической корректности результатов и вычисления показателей
ускорения и эффективности.

## 3. Базовый алгоритм

Основная логика алгоритма:

1. Вторая матрица B предварительно транспонируется для обеспечения эффективного
   построчного доступа к ее элементам.
2. Инициируется двойной цикл: внешний обходит строки матрицы A, внутренний —
   строки транспонированной матрицы B.
3. Поиск совпадающих индексов столбцов и вычисление скалярного произведения
   выполняется методом двух указателей за один проход.
4. Вычисленные значения формируют результирующую структуру CRS.

## 4. Схема распараллеливания

Для распараллеливания самого вычислительно сложного участка кода — внешнего
цикла обхода строк — выбран базовый алгоритмический примитив
`tbb::parallel_for`. В TBB этот примитив реализует задачно-ориентированную
декомпозицию (task-based parallelism).

```cpp
  int num_threads = ppc::util::GetNumThreads();
  tbb::global_control global_limit(tbb::global_control::max_allowed_parallelism, num_threads);

  tbb::parallel_for(0, rows, [&](int i) {
    // ... локальные вычисления и запись ...
  });
```

**Разбор архитектурных решений:**

- **Контроль конкуренции:** Фреймворк TBB инициализирует пул рабочих потоков
  (workers) самостоятельно на уровне всей программы. Чтобы строго ограничить
  число используемых ядер (что необходимо для тестирования производительности),
  применяется объект `tbb::global_control` с параметром
  `max_allowed_parallelism`. Он ограничивает глобальную конкуренцию заданным
  числом потоков, считанным из окружения.
- **Диапазон работы (Range):** В коде используется упрощенный интерфейс
  `tbb::parallel_for(0, rows, lambda)`. На уровне библиотеки этот вызов
  автоматически разворачивается в работу с объектом `tbb::blocked_range<int>(0,
  rows)`. Лямбда-выражение захватывает внешний контекст по ссылке `[&]` и
  принимает индекс обрабатываемой строки `int i`.
- **Размер зерна (Grainsize) и Partitioner:** Так как `blocked_range` создается
  неявно без передачи явного параметра grainsize, библиотека oneTBB по умолчанию
  применяет `tbb::auto_partitioner`. Этот партиционер самостоятельно управляет
  глубиной рекурсивного дробления диапазона `[0, rows]`. Он гарантирует, что
  задачи будут достаточно крупными для минимизации накладных расходов (overhead)
  на создание task-объектов, но при этом достаточно мелкими для того, чтобы
  алгоритм "кражи задач" мог эффективно балансировать нагрузку между потоками.
  Для разреженных матриц, где количество ненулевых элементов в строке
  непредсказуемо, `auto_partitioner` является самым отказоустойчивым выбором.
- **Отсутствие `parallel_reduce`:** Поскольку строки результирующей матрицы не
  зависят друг от друга, нам не нужно схлопывать результат в одно скалярное
  значение. Вычислительная нагрузка сводится к независимому заполнению строк,
  поэтому обычного `parallel_for` достаточно.

## 5. Детали реализации

Файлы реализации: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Основная техническая проблема при распараллеливании CRS-формата заключается в
том, что размер каждой выходной строки становится известен только по завершении
ее вычисления. Прямое добавление элементов в общие массивы `values` и
`col_indices` из множества потоков неизбежно приводит к состоянию гонки данных.
Использование TBB-примитивов блокировки (например, `spin_mutex`) сделало бы
конкурентный доступ к массивам крайне медленным.

Поэтому реализована **two-pass (двухэтапная) схема управления памятью:**

1. **Изоляция потоков (Локальные аккумуляторы):** Перед запуском `parallel_for`
   выделяются глобальные массивы векторов:

   ```cpp
   std::vector<std::vector<double>> temp_values(rows);
   std::vector<std::vector<int>> temp_cols(rows);
   ```

Внутри каждый worker-поток работает только с локальными переменными `local_vals`
и `local_cols`. Вычислив все элементы строки `i`, поток без блокировок
перемещает (с помощью `std::move`) эти векторы в уникальную для этой строки
ячейку глобального массива.
2. **Последовательная глобальная сборка:** Функция `parallel_for` является
   блокирующей — она выступает неявным барьером. Когда управление возвращается в
   главный поток, гарантируется, что все локальные буферы заполнены. Главный
   поток в последовательном цикле рассчитывает правильные смещения префиксной
   суммы для `row_pointers` и копирует данные из двумерного буфера в итоговые
   плоские CRS-массивы.

## 6. Проверка корректности

Корректность алгоритма TBB подтверждается успешным прохождением
параметризованного набора функциональных тестов.

В рамках тестов проверяются все сценарии и крайние случаи, описанные ранее в
`seq/report.md` и `omp/report.md` (включая работу с пустыми матрицами,
векторами, сильно разреженными структурами и матрицами минимального размера
1x1). Успешное выполнение тестов доказывает, что асинхронное выполнение задач
пулом потоков oneTBB не приводит к потере элементов или нарушению структуры
массивов CRS.

### Комментарий о balancing и overhead

**Оптимизация памяти:** Как и в других реализациях с разделением буферов,
алгоритм на TBB при $p=1$ работает быстрее чистого SEQ (ускорение 1.14x, время
снизилось с 428 до 374 мс). Это связано с предварительным резервированием
памяти, которое устраняет затратные переаллокации (динамические `push_back`)
внутри основного цикла расчета.

**Масштабируемость:** Библиотека oneTBB демонстрирует практически идеальное
масштабирование на малом числе потоков. На 2 и 4 workers эффективность держится
на уровне 97%, что подтверждает отличную работу механизма work-stealing и
`auto_partitioner`, которые равномерно загружают ядра. При переходе к 8 и 12
потокам эффективность плавно снижается (до 71%). Это ожидаемый результат: время
параллельной фазы становится настолько малым (менее 50 мс), что значительную
долю общего времени начинает отнимать последовательный overhead на сборку
двумерных векторов в единые плоские массивы CRS, а также задержки при
конкурентном доступе к памяти (Memory Bandwidth).

## 9. Выводы

Реализация алгоритма умножения разреженных матриц с использованием oneTBB
признана полностью успешной. Технология позволила корректно и безопасно
распараллелить вычислительное ядро алгоритма без изменения его математической
сути.

Разработанный TBB-код показал высокую масштабируемость (ускорение в 8.5 раз на
12 ядрах) и доказал, что задачно-ориентированная модель отлично справляется с
алгоритмами линейной алгебры.

**Особенности применения технологии:**

- **Удобство балансировки:** Главное преимущество oneTBB для данной задачи
  заключается в абстракции над управлением потоками. Применение
  `tbb::parallel_for` в связке с `auto_partitioner` позволяет переложить
  ответственность за выбор `grainsize` (размера чанка) на библиотеку.
  Планировщик автоматически подстраивается под неравномерную плотность строк в
  разреженных матрицах, не допуская простоя ядер.
- **Влияние Overhead:** Эффективность подхода напрямую зависит от соотношения
  времени вычислений и времени на управление памятью. Overhead становится
  заметным препятствием на 8-12 ядрах, когда скорость расчетов превышает
  скорость шины памяти. Встроенный механизм `task_group` или `parallel_for`
  требует накладных расходов на постановку задач в очередь, а закон Амдала
  непреодолимо ограничивает ускорение из-за наличия последовательного этапа
  слияния памяти.
