## 1. Введение

Задача умножения разреженных матриц в формате CRS (Compressed Row Storage)
является вычислительно трудоемкой и критически зависит от эффективности работы с
памятью. Особенности формата CRS (непрямая адресация через массивы индексов)
делают эту задачу отличным полигоном для сравнения различных моделей
параллелизма. В ходе работы исследуются как классические модели с общей памятью
(OpenMP, TBB, STL), так и гибридная модель (MPI + OpenMP), позволяющая
масштабировать вычисления на распределенные системы.

## 2. Единая постановка задачи

Требуется реализовать алгоритм вычисления произведения двух разреженных матриц
$A$ и $B$ типа `double`, представленных в формате CRS.

- **Вход:** Две структуры `SparseMatrix` (число строк/столбцов, массивы
  `values`, `col_indices`, `row_pointers`).
- **Выход:** Результирующая структура `SparseMatrix`, представляющая собой $C =
  A \times B$.
- **Ограничения:** Согласованность размерностей (столбцы $A$ равны строкам $B$).
  Элементы, итоговое значение которых по модулю меньше $10^{-15}$, отсекаются
  для сохранения свойств разреженности.
- **Критерий корректности:** Поэлементное совпадение результата с плотным
  перемножением тех же матриц с погрешностью не более $10^{-9}$.

## 3. Единая методика эксперимента

**Окружение:**

- **Процессор (CPU):** Intel(R) Core(TM) Ultra 5 125H (1.20 GHz), 14 физических
  ядер (18 потоков).
- **Оперативная память (RAM):** 32 GB.
- **Операционная система:** Ubuntu 24.04.4 LTS (Docker).
- **Компилятор:** GCC 13.3.0
- **Тип сборки (CMake):** Release

**Методика:**

- **Генерация данных:** Для всех тестов производительности используются
  ленточные матрицы размером **3000 x 3000** с шириной полосы **30**. Это
  обеспечивает равномерную и высокую вычислительную нагрузку.
- **Метрики:**
  - Время выполнения ($T$) — медианное значение фазы `task_run`.
  - Ускорение ($S$) рассчитывается как $S = T_{SEQ} / T_{parallel}$.
  - Эффективность ($E$) рассчитывается как $E = S / \text{Total Workers}
      \times 100\%$.
- **Конфигурация:** Число потоков регулируется через переменную
  `PPC_NUM_THREADS`, число процессов — через параметры `mpirun`.

## 4. Сводка корректности

Все разработанные backend-реализации (OMP, TBB, STL, ALL) прошли полную
верификацию. Тестирование подтвердило, что введение параллелизма не нарушило
математическую логику алгоритма умножения разреженных матриц. Результаты всех
параллельных версий полностью совпадают с результатами последовательной версии
(SEQ).

| `SmallSparse`   | Матрицы 10x10, плотность 10%    | Базовая арифметика и обход CRS          |

| `SingleElement` | Матрицы 1x1                     | Минимальные граничные условия           |

**Ограничения применимости отдельных backend-ов:**

- **SEQ:** Ограничена производительностью одного ядра и неэффективным
  управлением памятью при больших объемах данных.
- **STL (`std::thread`):** Имеет высокий порог накладных расходов на
  создание/завершение потоков. На малых задачах системный overhead может
  превышать время самих вычислений.
- **ALL (Гибридная схема):** Имеет существенные ограничения по памяти. Так как
  технология требует полного дублирования транспонированной матрицы $B$ в каждом
  MPI-процессе, на задачах экстремально большого размера (сотни миллионов
  элементов) оперативная память узла может быть исчерпана быстрее, чем достигнут
  предел производительности. Также эффективность ALL сильно зависит от
  стабильности сетевого соединения (или шины данных) при выполнении
  `MPI_Gatherv`.

## 6. Интерпретация различий

- **SEQ:** Последовательная версия показала избыточные затраты времени на
  динамическое расширение векторов. Использование `push_back` в один общий
  массив приводило к частым переаллокациям ОЗУ, что сделало эту версию самой
  медленной.
- **OMP:** Сильная сторона — простота внедрения и эффективный динамический
  планировщик (`schedule(dynamic)`). За счет изоляции памяти потоков (локальные
  векторы для каждой строки) OMP показал значительное ускорение уже на одном
  потоке относительно SEQ. Слабость — постепенное снижение эффективности на
  большом числе потоков из-за конкуренции за шину памяти.
- **TBB:** Продемонстрировал отличную балансировку нагрузки благодаря алгоритму
  "work-stealing". Роль `grain size` была минимизирована использованием
  `auto_partitioner`, который самостоятельно подбирал размер задач под нагрузку.
  TBB оказался наравне с OMP, но с чуть большими накладными расходами
  runtime-библиотеки на 12 потоках.
- **STL:** Ручное управление потоками выявило "цену" создания `std::thread`.
  Запуск потоков на лету без использования пула — самое слабое место технологии,
  что видно по деградации времени на 1 потоке. Однако при статическом разбиении
  на крупные диапазоны на 4-12 потоках STL не уступает OMP по чистой скорости
  вычислений.
- **ALL:** Гибридная версия показала сложную зависимость от топологии.
  Конфигурации с большим числом рангов (например, 4x2) оказались эффективнее
  «многопоточных» (2x4) из-за лучшей локализации данных в кэш-памяти и снижения
  нагрузки на планировщик OpenMP. Главное ограничение — высокая цена
  коммуникаций: на 24 Workers (6x4) время сборки через `MPI_Gatherv` начало
  доминировать над временем расчетов.

## 7. Репродуцируемость

Для проведения экспериментов и верификации результатов в изолированной среде
(Docker) необходимо выполнить последовательность команд сборки и запуска.

**Сборка проекта:** Используется CMake для генерации Makefile с оптимизацией
`Release`:

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

**Запуск функциональных тестов:** Для всех технологий (кроме ALL) тесты
запускаются напрямую. Для ALL используется `mpirun`:

```bash
# Потоковые версии
./build/bin/ppc_func_tests --gtest_filter="*DolovV*"

# Гибридная версия
PPC_NUM_THREADS=2 mpirun --allow-run-as-root -n 2 ./build/bin/ppc_func_tests --gtest_filter="*DolovV*"
```

**Проведение замеров производительности:** Рекомендуется запускать каждую
конфигурацию 3-5 раз для получения медианного значения.

```bash
# Пример замера для OMP (например для 4 потоков)
PPC_NUM_THREADS=4 ./build/bin/ppc_perf_tests --gtest_filter="*DolovV*"

# Пример замера для ALL (например конфигурация 4 ранга по 4 потока)
PPC_NUM_THREADS=4 mpirun --allow-run-as-root -n 4 ./build/bin/ppc_perf_tests --gtest_filter="*DolovV*"
```

## 8. Заключение

Экспериментальное исследование показало, что задача умножения разреженных матриц
CRS обладает высоким потенциалом для параллелизма. Лучшими инструментами для
локальных вычислений на современном многоядерном процессоре являются **OpenMP**
и **oneTBB**, так как они автоматически решают вопросы балансировки нагрузки и
минимизируют накладные расходы на управление потоками.

Основные ограничения ускорения связаны не с вычислительной мощностью ядер, а с
архитектурными особенностями: последовательной сборкой финальных данных (Закон
Амдала) и пропускной способностью памяти при одновременном чтении
транспонированной матрицы множеством потоков. Гибридная технология (ALL)
наиболее полно раскрывается на распределенных системах, в то время как на одном
узле она избыточна из-за дублирования структур данных.
