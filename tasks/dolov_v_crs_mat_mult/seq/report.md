## 1. Контекст

Умножение разреженных матриц является ключевой и вычислительно сложной операцией
во многих задачах линейной алгебры и математического моделирования. Формат
хранения CRS (Compressed Row Storage) позволяет существенно экономить память, но
требует специфического подхода к обходу элементов. Данная последовательная
реализация является базовой. Она необходима для проверки и фиксации корректной
математической логики работы с CRS-структурами, а также выступает в качестве
отправной точки для расчета реального ускорения и эффективности в последующих
реализациях (`OMP`, `TBB`, `STL`, `ALL`).

## 2. Постановка задачи

- **Входные данные:** Вектор `InType`, содержащий ровно две разреженные матрицы
  типа `SparseMatrix`. Данные хранятся в формате CRS (три массива: `values` для
  значений типа `double`, `col_indices` для индексов столбцов и `row_pointers`
  для указателей на начало строк).
- **Выходные данные:** Результирующая матрица `OutType` (`SparseMatrix`),
  являющаяся произведением первой и второй входных матриц.
- **Ограничения и крайние случаи:**
  - Вектор входных данных должен содержать строго два элемента.
  - Матрицы должны быть согласованы для умножения: количество столбцов первой
    матрицы должно совпадать с количеством строк второй (`matrix_a.num_cols ==
    matrix_b.num_rows`).
  - Размерности матриц должны быть строго больше нуля.
  - Значения скалярных произведений, по модулю меньшие $10^{-15}$, считаются
    «машинным нулем», отбрасываются и не добавляются в результирующую матрицу
    для сохранения её разреженности.

## 3. Базовый алгоритм

Основная проблема умножения в формате CRS заключается в том, что доступ к
столбцам второй матрицы неэффективен. Поэтому алгоритм строится на
предварительном транспонировании:

1. **Транспонирование второй матрицы ($B^T$).** Создается новая CRS-матрица, где
   столбцы исходной матрицы становятся строками. Это позволяет вычислять
   итоговые элементы через быстрое скалярное произведение строк $A$ и строк
   $B^T$.
2. **Скалярное произведение.** Для каждой строки первой матрицы ($i$) и каждой
   строки транспонированной второй матрицы ($j$) запускается параллельный проход
   по их ненулевым элементам методом двух указателей.
3. Указатели сдвигаются в зависимости от совпадения индексов (если индексы равны
   — элементы перемножаются и суммируются, иначе сдвигается указатель с меньшим
   индексом).
4. Если итоговая сумма для позиции $(i, j)$ больше $10^{-15}$, она записывается
   в результирующую матрицу, а массив `row_pointers` обновляется.

- **Асимптотика по времени:** 
  - Транспонирование: $O(NNZ_B + rows_B)$, где $NNZ$ — количество ненулевых
    элементов.
  - Умножение: в худшем случае $O(rows_A \cdot cols_B \cdot \max(K_A, K_B))$,
    где $K$ — максимальное количество ненулевых элементов в строке.
- **Асимптотика по памяти:** $O(NNZ_{res} + rows_A)$ для хранения результата и
  $O(NNZ_B + cols_B)$ для хранения временно транспонированной матрицы $B^T$.
- **Инварианты и критерий корректности:** Размер массива `row_pointers` всегда
  равен `num_rows + 1`. Последний элемент массива `row_pointers` всегда точно
  совпадает с текущим размером векторов `values` и `col_indices`.

## 4. Детали реализации

Файлы реализации: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

Методы фреймворка решают следующие задачи:
- **`ValidationImpl()`**: Проверяет количество матриц во входных данных (должно
  быть 2), согласованность их размерностей и то, что их размеры больше нуля.
- **`PreProcessingImpl()`**: Настраивает выходную матрицу, задавая ей итоговые
  размеры (строки от $A$, столбцы от $B$). Выделяет память под массив
  `row_pointers` (размер `num_rows + 1`) и очищает векторы значений и индексов.
- **`RunImpl()`**: Содержит главную логику. Вызывает `TransposeMatrix` для
  матрицы $B$, после чего во вложенных циклах обходит строки матриц $A$ и $B^T$,
  вызывая `DotProduct`. Сразу же формирует CRS-структуру результата.
- **`PostProcessingImpl()`**: Возвращает `true`, так как все выходные данные уже
  корректно сформированы внутри `RunImpl`.

Дополнительные приватные методы класса `DolovVCrsMatMultSeq`:
- **`TransposeMatrix()`**: Эффективно преобразует одну CRS матрицу в другую
  (транспонированную), вычисляя смещения для массива `row_pointers` и
  перераспределяя элементы.
- **`DotProduct()`**: Изолирует логику перемножения двух строк (векторов) с
  использованием метода двух указателей.

**Ключевой фрагмент (вычисление в `RunImpl`):**

```cpp
bool DolovVCrsMatMultSeq::RunImpl() {
  const auto &input_data = GetInput();
  const auto &matrix_a = input_data[0];
  const auto &matrix_b = input_data[1];
  auto &result = GetOutput();

  SparseMatrix matrix_b_t = TransposeMatrix(matrix_b);

  for (int i = 0; i < matrix_a.num_rows; ++i) {
    for (int j = 0; j < matrix_b_t.num_rows; ++j) {
      double sum = DotProduct(matrix_a, i, matrix_b_t, j);

      // Отсечение машинного нуля для сохранения свойств разреженности
      if (std::abs(sum) > 1e-15) {
        result.values.push_back(sum);
        result.col_indices.push_back(j);
      }
    }
    result.row_pointers[i + 1] = static_cast<int>(result.values.size());
  }
  return true;
}
```

## 5. Проверка корректности

Корректность работы алгоритма подтверждается успешным прохождением набора
функциональных тестов. Сравнение результатов реализовано следующим образом:
полученная после работы программы разреженная матрица временно конвертируется в
плотный массив, а затем поэлементно сверяется с результатом классического
плотного умножения этих же самых матриц (функция `DenseMultiply`). Из-за
специфики машинного представления типа `double` сравнение элементов происходит
не на строгое равенство, а с допуском абсолютной погрешности 1e-9.

**Использованные функциональные тесты:** Тестовое покрытие реализовано через
параметризованный набор `CRS_Functional_Tests` и включает 7 сценариев:
1. `SmallSparse`: перемножение небольших матриц (10x10) с плотностью 10%.
   Проверяет базовую арифметику алгоритма.
2. `Rectangular`: умножение матриц разных, но согласованных размерностей (20x10
   и 10x15). Проверяет, что итоговая матрица получает правильные размеры
   (20x15).
3. `ZeroMatrix`: умножение на полностью нулевую матрицу, где хранятся только
   нули в массиве `row_pointers`. Гарантирует защиту от выхода за границы памяти
   при полном отсутствии элементов.
4. `DotProduct`: вектор-строка на вектор-столбец (1x100 и 100x1). Проверяет
   вырождение алгоритма в вычисление скалярного произведения и правильное
   накопление суммы.
5. `LargeSparse`: разреженные матрицы 100x100 с низкой плотностью (2%).
   Убеждается, что алгоритм эффективно пропускает длинные пустые участки строк.
6. `DenseRow`: умножение заранее заданной вручную матрицы 3x3 на случайно
   сгенерированную.
7. `SingleElement`: матрицы минимального размера (1x1). Проверяет работу
   алгоритма на предельно малых граничных значениях размерностей.

**Характерные примеры входа и ожидаемого выхода:**

1. **Тест `SingleElement`:**
   - **Вход:** Матрица A размером 1x1 с единственным значением 1.0. Матрица B
     размером 1x1 со значением 1.0.
   - **Ожидаемый выход:** Матрица C размером 1x1 с элементом 1.0. Вектор
     `row_pointers` содержит значения `[0, 1]`.

2. **Тест `ZeroMatrix`:**
   - **Вход:** Матрица A размером 5x5, заполненная случайными числами (плотность
     50%). Матрица B размером 5x5 без ненулевых элементов (массивы значений и
     индексов пусты).
   - **Ожидаемый выход:** Пустая матрица C размером 5x5. Массив `row_pointers`
     равен `[0, 0, 0, 0, 0, 0]`, векторы значений и столбцов пустые.

3. **Тест `DotProduct`:**
   - **Вход:** Вектор-строка (матрица A) размером 1x100 и вектор-столбец
     (матрица B) размером 100x1. Обе матрицы имеют плотность заполнения 50%.
   - **Ожидаемый выход:** Матрица C размером 1x1, содержащая единственное число
     — сумму попарных произведений элементов. Вектор `row_pointers` равен `[0,
     1]`.

## 8. Выводы

Реализованная последовательная версия полностью решает задачу перемножения
матриц в формате CRS и является фундаментом для дальнейшей работы над проектом.
Она играет ключевую роль по трем причинам:

1. **База для тестирования корректности:** Последовательный алгоритм не содержит
   состояний гонки и проблем с разделением памяти. Выдаваемый им результат
   является строго детерминированным. Во всех параллельных реализациях
   получаемые матрицы будут сверяться с результатами этой программы, чтобы
   гарантировать отсутствие ошибок при распараллеливании.
2. **Точка отсчета производительности (Baseline):** Полученное время выполнения
   ($T_1 \approx 428.11$ мс) станет знаменателем при расчете ускорения ($S = T_1
   / T_p$). Только опираясь на показатели чистого последовательного кода, можно
   объективно оценить накладные расходы различных параллельных технологий на
   создание потоков или пересылку сообщений.
3. **Обоснование стратегии распараллеливания:** Анализ работы алгоритма
   подтвердил, что узким местом является вложенный цикл вычисления скалярных
   произведений. При этом расчет каждой строки результирующей матрицы не зависит
   от остальных строк. Это прямо указывает на то, что в параллельных версиях
   наиболее эффективным подходом будет распределение итераций внешнего цикла (по
   строкам матрицы $A$) между доступными потоками или процессами.
