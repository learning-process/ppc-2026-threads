## 1. Контекст

Использование `std::thread` из стандартной библиотеки C++ переносит
ответственность за управление параллелизмом с уровня специализированных
программных сред (таких как OpenMP или oneTBB) на уровень разработчика. Данная
реализация необходима для демонстрации глубокого понимания низкоуровневых
механизмов многопоточности. В отличие от высокоуровневых решений, которые
самостоятельно распределяют задачи и управляют пулами потоков, здесь требуется
вручную разработать математику декомпозиции данных, явно контролировать
жизненный цикл каждого системного потока (от запуска до синхронизации) и на
архитектурном уровне гарантировать отсутствие гонок данных. Это делает
STL-версию наиболее строгой с инженерной точки зрения.

## 2. Постановка задачи

Постановка задачи идентична базовой спецификации, описанной в отчете
`seq/report.md`.
- **Входные данные:** Две разреженные матрицы в формате CRS (тип элементов
  `double`).
- **Выходные данные:** Результирующая матрица в формате CRS.
- **Ограничения:** Матрицы согласованы по размерностям, скалярные произведения

Последовательная реализация (SEQ) выступает базы. Именно с её временем работы
сравнивается производительность многопоточной версии.

## 3. Базовый алгоритм

Базовая логика умножения матриц в формате CRS:
1. Матрица B предварительно транспонируется.
2. Во вложенных циклах происходит обход строк матрицы A и матрицы B^T.
3. Скалярное произведение строк вычисляется методом двух указателей за один
   проход по массивам индексов столбцов.
4. Вычисленные значения добавляются в результирующую структуру с одновременным
   обновлением массива `row_pointers`.

## 4. Схема распараллеливания

Для распараллеливания применена статическая блочная декомпозиция. Итеративным
пространством выступают строки первой матрицы (матрицы $A$). 

- **Разбиение на диапазоны:** Пространство итераций делится на непрерывные
  непересекающиеся блоки. Главный поток заранее вычисляет, какой диапазон строк
  достанется каждому рабочему потоку. Для этого общее количество строк делится
  на число доступных потоков с вычислением остатка. Первые несколько потоков
  берут на себя на одну строку больше, чтобы компенсировать неделимый остаток.
- **Работа потока:** Каждый поток инициализируется с передачей ему границ
  диапазона `[start_row, end_row)`. Внутри потока выполняется локальный цикл,
  который обходит назначенные строки матрицы $A$, выполняет скалярное умножение
  на строки матрицы $B^T$ и собирает результаты.
- **Хранение локальных результатов:** Локальные суммы сначала накапливаются в
  динамических векторах, расположенных на стеке самого потока (`local_vals` и
  `local_cols`). После того как строка полностью вычислена, поток переносит эти
  векторы в глобальный массив.
- **Синхронизация и потокобезопасность:** В коде **полностью отсутствуют**
  примитивы синхронизации (`std::mutex`, `std::atomic` или
  `std::condition_variable`). Они намеренно исключены, так как любая блокировка
  при записи элементов матрицы сделала бы алгоритм неэффективным.
  Потокобезопасность достигается архитектурно: каждый поток пишет данные строго
  в свой независимый индекс предварительно выделенного глобального вектора
  `temp_values[i]`. Поскольку диапазоны строк не пересекаются, разные потоки
  никогда не обращаются к одной и той же ячейке памяти.
- **Вызов `join`:** Сборка потоков происходит строго после того, как все потоки
  были созданы и запущены. Главный поток в отдельном цикле вызывает `join()` для
  каждого объекта `std::thread`, дожидаясь полного завершения вычислений, прежде
  чем переходить к итоговому формированию матрицы.

## 5. Детали реализации

Файлы реализации: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

**Карта "поток -> диапазон" (распределение нагрузки):** Чтобы избежать
неравномерной нагрузки или потери строк при делении, реализован точный алгоритм
расчета границ. 
```cpp
  int chunk_size = rows / num_threads;
  int remainder = rows % num_threads;
  int current_start = 0;

  for (int thread_idx = 0; thread_idx < num_threads; ++thread_idx) {
    int current_end = current_start + chunk_size + (thread_idx < remainder ? 1 : 0);
    if (current_start < current_end) {
      threads.emplace_back(worker, current_start, current_end);
    }
    current_start = current_end;
  }
```
В этом цикле главный поток рассчитывает конец текущего блока (`current_end`),
прибавляя единицу, если индекс потока попадает в остаток от деления
(`remainder`). Защитное условие `if (current_start < current_end)` гарантирует,
что если число потоков превышает число строк (например, матрица 1x1 и 12
потоков), "лишние" объекты `std::thread` просто не будут созданы. Это экономит
системные ресурсы.

**Жизненный цикл потоков и барьерная синхронизация:** После того как все объекты
добавлены в `std::vector<std::thread>`, главный поток приостанавливает свое
выполнение до тех пор, пока все рабочие потоки не завершат свои задачи:
```cpp
  for (auto &thread : threads) {
    thread.join();
  }
  
  GetOutput() = AssembleResult(rows, matrix_b.num_cols, temp_values, temp_cols);
```
Разделение циклов `emplace_back` и `join` является критически важным: если бы
`join()` вызывался внутри первого цикла, многопоточность превратилась бы в
последовательное выполнение (сериализацию).

**Схема объединения локальных результатов:** Функция `AssembleResult`
выполняется строго в один поток после барьера `join`. Сначала она выделяет
память под массив `row_pointers` и заполняет его, вычисляя префиксные суммы
(накапливая размеры обработанных векторов из `temp_values[i]`). Определив точное
количество ненулевых элементов (`total_nz`), функция делает `reserve` для
результирующих массивов `values` и `col_indices`, предотвращая переаллокацию.
Финальным этапом является массовое копирование (через `insert`) локальных
двумерных буферов в итоговые одномерные массивы структуры CRS.

## 6. Проверка корректности

Как и в предыдущих реализациях, корректность работы алгоритма проверяется
набором `CRS_Functional_Tests`.

В ходе тестирования проверяется отсутствие гонок данных при многопоточном
выполнении. Особенное внимание уделяется проверке алгоритма разбиения:
- Крайний случай `SmallSparse` (размер 10x10, число потоков > строк): алгоритм
  блочного разбиения корректно оставляет `start == end` для "лишних" потоков, и
  проверка `if (current_start < current_end)` предотвращает создание холостых
  потоков `std::thread`, экономя системные ресурсы.
- Случаи с некратными размерами: проверка того, что ни одна строка не
  дублируется и не пропускается благодаря переменной `remainder`.

### Комментарий

Самым показательным результатом является время выполнения STL-версии на **1
потоке (487.22 мс)**. В отличие от OMP и TBB, где на 1 потоке мы получали
ускорение за счет оптимизации памяти, здесь алгоритм отработал медленнее базовой
последовательной реализации (эффективность 88%). Это наглядная демонстрация
системных накладных расходов: конструктор `std::thread` осуществляет
дорогостоящий системный вызов к ядру ОС для выделения стека и запуска реального
аппаратного потока. В OMP и TBB эта стоимость амортизируется за счет заранее
созданного пула потоков (Thread Pool), а в STL мы платим эту цену прямо в момент
выполнения алгоритма.

Тем не менее, при увеличении числа потоков (начиная с 2) выгода от
распараллеливания полностью перекрывает накладные расходы ОС, обеспечивая
отличное масштабирование (ускорение в 8.39 раз на 12 ядрах).

## 9. Выводы

Ручная реализация с использованием `std::thread` продемонстрировала высокую
гибкость и производительность, практически не уступающую высокоуровневым
фреймворкам на большом числе ядер.

**Основные инженерные итоги:**
1. **Ответственность за декомпозицию:** Алгоритмы статического блочного
   разбиения эффективны, но требуют тщательного написания шаблонного кода
   (вычисление шага и остатка). В отличие от `auto_partitioner` из TBB, STL не
   прощает математических ошибок на границах диапазонов.
2. **Бесшовная потокобезопасность:** Отказ от примитивов синхронизации
   (`std::mutex`) в пользу разнесения записи по независимым буферам памяти
   доказал свою эффективность. Алгоритм не тратит время на ожидание
   разблокировки ресурсов.
3. **Цена системных вызовов:** Реализация подтвердила, что порождение потоков на
   лету — тяжелая операция. Для задач малого объема или при необходимости
   многократного запуска `RunImpl` использование `std::thread` без пула потоков
   нецелесообразно из-за доминирующих накладных расходов на `create/join`.
