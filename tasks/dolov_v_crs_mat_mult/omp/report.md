## 1. Контекст

Последовательная версия алгоритма доказала корректность математической логики
обхода массивов в формате CRS и обработки данных. Однако вычисление каждой
строки результирующей матрицы $C$ при матричном умножении $C = A x B$ является
абсолютно независимой задачей. Это делает алгоритм идеальным кандидатом для
распараллеливания на уровне данных (Data Parallelism) с использованием парадигмы
общей памяти. Технология OpenMP внедряется для того, чтобы распределить
независимые вычисления строк между доступными ядрами процессора, тем самым
сократив общее время выполнения вычислительно плотного цикла.

## 2. Постановка задачи

Постановка задачи полностью идентична базовой спецификации, описанной в
`seq/report.md`.

- **Входные данные:** Две разреженные матрицы в формате CRS (тип элементов
  `double`).
- **Выходные данные:** Результирующая матрица в формате CRS.
- **Ограничения:** Матрицы должны быть согласованы по размерностям, элементы
меньше 1e-15 отсекаются. Последовательная реализация из `seq/report.md`
выступает в качестве базы (baseline) для проверки корректности результатов и
вычисления показателей ускорения.

## 3. Базовый алгоритм

Основная логика алгоритма:

1. Вторая матрица B предварительно транспонируется для быстрого построчного
   доступа к ее столбцам.
2. Происходит двойной цикл: итерация по строкам матрицы A и строкам
   транспонированной матрицы B.
3. Поиск совпадающих индексов столбцов выполняется за один проход методом двух
   указателей.
4. Вычисленные скалярные произведения накапливаются и формируют результирующую
   структуру CRS.

## 4. Схема распараллеливания

Распараллеливанию подвергся самый тяжелый участок кода — внешний цикл в методе
`RunImpl`, который итерируется по строкам первой матрицы (переменная `i`).

```cpp
#pragma omp parallel for schedule(dynamic, 10) default(none) shared(matrix_a, matrix_b_t, temp_values, temp_cols, rows)
```

**Разбор параметров директивы:**

- `default(none)`: Выбрано как строгая мера безопасности, требующая явного
  указания класса памяти (shared/private) для каждой переменной, используемой
  внутри параллельной секции. Это исключает случайное разделение итераторов или
  локальных буферов.
- **Shared переменные:** `matrix_a`, `matrix_b_t` (используются только для
  чтения), а также `temp_values`, `temp_cols` и `rows`. Важно отметить, что в
  `temp_values` и `temp_cols` каждый поток пишет данные строго по своему
  уникальному индексу `i`, поэтому гонки данных не возникает.
- **Private переменные:** Неявно приватными становятся счетчик внешнего цикла
  `i`, счетчик вложенного цикла `j`, переменная `sum`, а также явно объявленные
  внутри тела цикла локальные векторы `local_vals` и `local_cols`.
- **Отсутствие `reduction`, `atomic` или `critical`:** В базовой реализации
  запись шла в единый массив `result.values`. В многопоточной среде это привело
  бы к состоянию гонки. Алгоритм был перепроектирован: каждый поток накапливает
  значения в приватных векторах `local_vals` и `local_cols`, а затем перемещает
  (`std::move`) их в свой уникальный слот разделяемого массива `temp_values[i]`.
  Это позволяет избежать любых блокировок.
- `schedule(dynamic, 10)`: В разреженных матрицах строки имеют разное количество
  ненулевых элементов, из-за чего время вычисления разных строк сильно
  отличается. Статическое планирование (`static`) привело бы к дисбалансу
  нагрузки. Выбран `dynamic` с размером чанка 10: потоки динамически забирают
  блоки по 10 строк, что обеспечивает равномерную загрузку ядер.
- **Барьеры:** В конце директивы `#pragma omp parallel for` присутствует неявный
  барьер. Он необходим для того, чтобы основной поток дождался завершения
  вычислений всех строк перед тем, как переходить к последовательной сборке
  итоговых векторов CRS.

## 5. Детали реализации

Файлы реализации: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Относительно базовой последовательной реализации был полностью переработан
подход к управлению памятью внутри метода `RunImpl`. Главной проблемой при
распараллеливании формирования CRS-матрицы является то, что мы заранее не знаем,
сколько ненулевых элементов сгенерирует каждая строка. Прямой `push_back` в
единый результирующий вектор из разных потоков привел бы к состоянию гонки (data
race). Использование критических секций (`#pragma omp critical`) при каждом
добавлении элемента сделало бы многопоточную версию медленнее однопоточной.

Для устранения рисков гонок данных применена двухэтапная схема (локальное
накопление и глобальная сборка):

1. **Предварительное выделение памяти:** До начала параллельной секции создаются
   массивы векторов для независимого хранения результатов каждой строки.

```cpp
  int rows = matrix_a.num_rows;
  std::vector<std::vector<double>> temp_values(rows);
  std::vector<std::vector<int>> temp_cols(rows);
```

2. **Параллельное вычисление:** Внутри `omp parallel for` каждый поток создает
   локальные буферы `local_vals` и `local_cols`. После вычисления строки эти
   буферы без блокировок перемещаются (через `std::move`) в соответствующую
   ячейку глобального массива по уникальному индексу `i`.

```cpp
#pragma omp parallel for schedule(dynamic, 10) default(none) shared(matrix_a, matrix_b_t, temp_values, temp_cols, rows)
  for (int i = 0; i < rows; ++i) {
    std::vector<double> local_vals;
    std::vector<int> local_cols;

    for (int j = 0; j < matrix_b_t.num_rows; ++j) {
      double sum = DolovVCrsMatMultOmp::DotProduct(matrix_a, i, matrix_b_t, j);
      if (std::fabs(sum) > 1e-15) {
        local_vals.push_back(sum);
        local_cols.push_back(j);
      }
    }
    // Безопасная запись по независимому индексу
    temp_values[i] = std::move(local_vals);
    temp_cols[i] = std::move(local_cols);
  }
```

3. **Глобальная сборка (последовательная часть):** После неявного барьера по
   завершении `parallel region`, главный поток безопасно рассчитывает массив
   `row_pointers` и сливает все данные из двумерной структуры в итоговые плоские
   массивы `values` и `col_indices`.

## 6. Проверка корректности

Корректность алгоритма подтверждается прохождением функциональных тестов из
набора. Логика валидации строится на строгом сравнении: получаемая разреженная
матрица программно разворачивается в плоский массив и поэлементно сравнивается с
результатом классического умножения плоских матриц (`DenseMultiply`).
Дополнительно результаты OMP-версии косвенно сверяются с базовой
последовательной версией (SEQ), так как обе они проходят идентичный набор
проверок с одинаковым порогом абсолютной погрешности 1e-9. Это гарантирует, что
введение потоков не нарушило математический порядок вычислений и не создало
гонок данных.

**Использованные функциональные тесты и характерные случаи:** Тестовый набор
включает 7 сценариев, покрывающих различные формы и состояния разреженных
матриц:

1. `SmallSparse`: перемножение небольших матриц (10x10) с плотностью 10%.
   Проверка общей работоспособности распределения итераций.
2. `Rectangular`: умножение матриц разных размерностей (20x10 и 10x15).
   Убеждаемся, что алгоритм правильно строит границы новой матрицы и не выходит
   за пределы памяти при асимметричных задачах.
3. `ZeroMatrix` (характерный краевой случай): умножение на полностью пустую
   матрицу. Проверяется, что потоки OpenMP корректно отрабатывают пустые пулы
   задач, а итоговый массив `row_pointers` заполняется нулями без попыток
   обращения к отсутствующим значениям.
4. `DotProduct`: вектор-строка умножается на вектор-столбец (1x100 и 100x1).
   Характерный случай вырождения алгоритма, когда внешнему циклу отдается всего
   одна итерация.
5. `LargeSparse`: разреженные матрицы 100x100 с экстремально низкой плотностью
   (2%). Тестирует эффективность метода двух указателей на длинных «пустых»
   промежутках.
6. `DenseRow`: комбинация заранее заданной вручную матрицы 3x3 на случайно
   сгенерированную матрицу.
7. `SingleElement` (характерный краевой случай): матрицы минимального размера
   (1x1). Проверка того, что динамический планировщик (`schedule(dynamic, 10)`)
   не ломается, когда общее количество задач (1 строка) строго меньше заданного
   размера чанка (10).

### Комментарий

Анализ результатов демонстрирует отличное параллельное масштабирование
алгоритма. Интересным артефактом является эффективность более 100% на 1 и 2
потоках. Это объясняется тем, что в процессе подготовки кода к OpenMP была
изменена структура выделения памяти (переход от постоянного `push_back` в единый
массив к независимым буферам `temp_values` с предварительным резервированием).
Эта алгоритмическая оптимизация снизила накладные расходы на реаллокацию памяти,
сделав 1-поточную версию OMP немного быстрее базовой SEQ-версии.

При дальнейшем увеличении числа потоков (до 4 и 8) алгоритм показывает почти
линейное ускорение, так как строки матрицы независимы, а `schedule(dynamic)`
успешно балансирует нагрузку. Однако на 12 потоках эффективность падает до 73%.
Основным узким местом становится пропускная способность оперативной памяти и
последовательный этап финальной сборки массивов, который выполняется одним
потоком в соответствии с законом Амдала.

## 9. Выводы

Реализация OpenMP успешно справилась со своей задачей: обеспечена корректная
многопоточная обработка разреженных матриц без состояний гонки и необходимости
использовать медленные блокировки.

**Выводы по применению технологии:**

1. Выигрыш достигнут за счет грамотной реструктуризации памяти (переход от
   единого вектора к массиву независимых буферов), что является необходимым
   условием для любого распараллеливания Data Parallelism.
2. Использование `schedule(dynamic, 10)` доказало свою пригодность для задач с
   неравномерной нагрузкой (разное количество элементов в строках CRS-матриц).
3. Технология OpenMP демонстрирует свои ограничения на небольших или слишком
   быстро решаемых задачах. Когда время полезных математических вычислений
   падает ниже 50 миллисекунд, накладные расходы на управление потоками начинают
   нивелировать выгоду от добавления новых вычислительных ядер. Для получения
   линейного масштабирования размер матриц или плотность их заполнения
   необходимо существенно увеличить.
