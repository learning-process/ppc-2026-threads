## 1. Контекст

OpenMP-версия реализована как первый шаг распараллеливания алгоритма Джарвиса. Основная цель - ускорить поиск следующей
точки оболочки за счёт распределения внутреннего цикла по потокам, сохранив при этом детерминированную
последовательность построения оболочки (внешний цикл остаётся последовательным). OpenMP выбран благодаря простоте
директивного подхода и возможности инкрементального распараллеливания критически важных участков.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

- Вход: вектор точек `std::vector<std::tuple<double, double>>`.
- Выход: выпуклая оболочка в порядке обхода против часовой стрелки, начиная с самой левой-нижней точки.
- Ограничения: если точек < 3, оболочка = вход; пустой вход → пустой выход.

## 3. Базовый алгоритм (из SEQ)

```cpp
// Последовательное ядро (внешний цикл по точкам оболочки, внутренний по всем точкам)
while(true) {
    ...
    for (size_t i = 0; i < GetInput().size(); ++i) {
        if (i == current) continue;
        double i_x = std::get<0>(GetInput()[i]);
        double i_y = std::get<1>(GetInput()[i]);
        double cross = CrossProduct(current_x, current_y, next_x, next_y, i_x, i_y);
        if (IsBetterPoint(cross, epsilon, current_x, current_y, i_x, i_y, next_x, next_y)) {
            next = i;
            next_x = i_x;
            next_y = i_y;
        }
    }
    ...
    if (current == leftmost) {
      break;
    }
}
```

Внешний `while` перебирает вершины оболочки, а внутренний `for` ищет лучшую следующую точку. Именно этот внутренний цикл
и распараллелен в OMP-версии.

## 4. Схема распараллеливания

### Выбранная область параллелизации

Параллелятся два участка:

- **Поиск самой левой точки** (`FindLeftmostPoint`) - однократный проход по всем точкам.
- **Поиск следующей точки оболочки** (внутренний цикл в `while`) - каждый шаг построения оболочки.

В обоих случаях используется один и тот же шаблон: каждый поток просматривает свой блок точек, находит локального
кандидата, а затем в критической секции выбирается глобально лучший.

### Переменные: shared / private / reduction

В директивах `#pragma omp parallel` используется `default(none)` и явно перечисляются все используемые переменные.

**Поиск левой точки:**

```cpp
#pragma omp parallel default(none) shared(input, leftmost)
```

- `input` - `shared` (только чтение, гонок нет).
- `leftmost` - `shared` (изменяется только внутри `critical`).
- `local_leftmost`, `local_leftmost_x/y` - локальные для каждого потока (неявно `private`).
- Переменные цикла `i`, `x`, `y` - `private`.

**Поиск следующей точки оболочки:**

```cpp
#pragma omp parallel default(none) shared(current, current_x, current_y, next, next_x, next_y, epsilon)
```

- `current`, `current_x/y`, `epsilon` - `shared` (только чтение).
- `next`, `next_x/y` - `shared` (изменяются в `critical`).
- `local_next`, `local_next_x/y` - `private` (локальные кандидаты потока).

`default(none)` используется для явного контроля: все переменные, которые поток видит, должны быть перечислены, иначе
ошибка компиляции.

### Почему не `reduction`?

Редукция неприменима: операция выбора «лучшей» точки неассоциативна (зависит от текущей точки `current` и порядка
сравнения). Поэтому используется паттерн «локальный кандидат + глобальное слияние в критической секции».

### Выбор `schedule` и `nowait`

В директивах `#pragma omp for` не указан `schedule` - значит используется `static` по умолчанию. Это оправдано: итерации
имеют примерно одинаковую вычислительную сложность, статическое распределение минимизирует накладные расходы.

Применена клауза `nowait`:

- В поиске левой точки - чтобы потоки не ждали друг друга у неявного барьера в конце `for`, а сразу переходили к
  критической секции.
- В поиске следующей точки - аналогично, позволяет ускорить переход к сравнению локальных кандидатов.

Это безопасно, так как после `for` следует только чтение `local_*` переменных (у каждого потока свои) и вход в
`critical`.

### Критические секции и барьеры

```cpp
#pragma omp critical
{
  // сравнение local_next с глобальным next и обновление
}
```

Критическая секция гарантирует, что только один поток одновременно обновляет глобального кандидата. Конфликт возможен,
но он корректно разрешается последовательным сравнением: если два потока одновременно пытаются войти в `critical`, один
ждёт; внутри секции проводится детерминированное сравнение с текущим глобальным лучшим.

Неявный барьер в конце `parallel`-региона (после `critical`) гарантирует, что все потоки завершили обновление `next`,
прежде чем основной поток продолжит внешний цикл.

### Число потоков

Управляется переменной `OMP_NUM_THREADS`, которая в инфраструктуре курса задаётся через `PPC_NUM_THREADS`. В коде не
вызывается `omp_set_num_threads()`, значит используется значение по умолчанию (или заданное окружением).

## 5. Детали реализации

**Файлы:**

- `omp/include/ops_omp.hpp` - объявление класса `KutuzovITestConvexHullOMP`.
- `omp/src/ops_omp.cpp` - реализация с OpenMP-директивами.

**Структура пайплайна** полностью повторяет SEQ:

- `ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` - без изменений.
- `RunImpl` - вызывает те же методы, но `FindLeftmostPoint` и внутренний цикл поиска `next` распараллелены.

**Ключевой фрагмент - параллельный поиск следующей точки:**

```cpp
#pragma omp parallel default(none) shared(current, current_x, current_y, next, next_x, next_y, epsilon)
{
  size_t local_next = (current + 1) % GetInput().size();
  double local_next_x = std::get<0>(GetInput()[local_next]);
  double local_next_y = std::get<1>(GetInput()[local_next]);

  #pragma omp for nowait
  for (size_t i = 0; i < GetInput().size(); ++i) {
    if (i == current) continue;
    // ... вычисление cross ...
    if (IsBetterPoint(...)) {
      local_next = i;
      local_next_x = i_x; local_next_y = i_y;
    }
  }

  #pragma omp critical
  {
    // сравнение local_next с глобальным next
    double cross = CrossProduct(current_x, current_y, next_x, next_y, local_next_x, local_next_y);
    if (IsBetterPoint(cross, epsilon, current_x, current_y, local_next_x, local_next_y, next_x, next_y)) {
      next = local_next;
      next_x = local_next_x; next_y = local_next_y;
    }
  }
}
```

Каждый поток инициализирует своего локального кандидата (`local_next`) первой доступной точкой (индекс `(current+1)%N`),
затем в цикле `for` перебирает свои итерации и обновляет `local_next`. После цикла (`nowait` - без ожидания других
потоков) поток входит в `critical`, где сравнивает своего кандидата с текущим глобальным `next`. При необходимости
глобальный `next` обновляется.

**Параллельный поиск левой точки** (`FindLeftmostPoint`) устроен аналогично:

```cpp
#pragma omp parallel default(none) shared(input, leftmost)
{
  size_t local_leftmost = 0;
  double local_leftmost_x = std::get<0>(input[0]);
  double local_leftmost_y = std::get<1>(input[0]);

  #pragma omp for nowait
  for (size_t i = 0; i < input.size(); ++i) {
    // обновление local_leftmost при нахождении точки левее/ниже
  }

  #pragma omp critical
  {
    // сравнение local_leftmost с глобальным leftmost
    if (local_leftmost_x < leftmost_x || (...)) {
      leftmost = local_leftmost;
    }
  }
}
```

**Синхронизация и корректность:**

- Чтение `input`, `current`, `epsilon` - безопасно, так как данные не изменяются.
- Запись в `local_*` - каждый поток работает со своими переменными.
- Критическая секция защищает обновление глобального `next`/`leftmost` - в один момент времени только один поток может
  изменить их, при этом он использует актуальное значение (последовательное согласование).

## 6. Проверка корректности

Используются те же **три функциональных теста**, что и для SEQ. Проверка - побитовое совпадение выходного вектора с
эталонным.

**Детерминизм:** В отличие от SEQ, OMP-версия может давать разный порядок обновления глобального `next` из-за гонок на
входе в `critical`. Однако финальное значение `next` детерминировано для данного набора точек и заданного числа потоков
при одинаковом разбиении итераций. Практически результаты совпадают с эталоном для всех тестов при
`num_threads = 1, 2, 4`.

**Пройденные тесты:**

- Квадрат с внутренними точками
- Все точки на одной прямой
- Треугольник

Все тесты пройдены успешно.

## 9. Анализ и выводы

**Суперлинейное ускорение на 2 потоках (2.57x, эффективность 128%)**\
На двух потоках зафиксировано ускорение выше идеального линейного. Это объясняется эффектами кэш-памяти: два потока
обрабатывают половины массива, которые лучше помещаются в L2/L3 кэш каждого физического ядра, снижая количество
кэш-промахов по сравнению с последовательным проходом по всему массиву. Дополнительный вклад может давать динамическое
повышение частоты (Turbo Boost) при неполной загрузке ядер. Такое поведение не является ошибкой, но должно учитываться
при интерпретации масштабируемости.

**Хорошее ускорение на 4 потоках (3.72x, эффективность 93%)**\
При четырёх потоках достигнуто почти четырёхкратное ускорение, эффективность остаётся высокой (93%). Это говорит о том,
что внутренний цикл поиска следующей точки доминирует над последовательной частью и накладными расходами синхронизации.

**Насыщение при 8 потоках (3.71x, эффективность 46%)**\
Переход с 4 на 8 потоков не дал прироста производительности - время практически не изменилось (0.0697 с). Причина в том,
что дальнейшее распараллеливание упирается в:

- Последовательный внешний цикл (обход вершин оболочки), доля которого становится заметной при малом абсолютном времени
  параллельной части.
- Конкуренцию за критическую секцию при слиянии локальных кандидатов - восемь потоков ждут друг друга, нивелируя выигрыш
  от распараллеливания.

Таким образом, оптимальная конфигурация для данной задачи на данном оборудовании - 4 потока. Использование 8 потоков
нецелесообразно из-за падения эффективности до 46% при том же абсолютном времени.

**Когда OpenMP эффективен:**

- При большом числе точек N (миллионы), когда доля последовательного кода мала.
- При числе потоков, не превышающем количество физических ядер (до 4–6), чтобы избежать пробуксовки на синхронизации.

**Ограничения:**

- Алгоритм Джарвиса принципиально ограничен последовательным обходом оболочки.
- При малом размере оболочки (H мало) ускорение будет минимальным, так как последовательная часть (поиск левой точки и
  инициализация) становится доминирующей.
