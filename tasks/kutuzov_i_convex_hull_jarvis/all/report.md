## 1. Контекст

Гибридная версия реализована для оценки иерархического параллелизма: MPI распределяет работу между процессами, а OpenMP
внутри каждого процесса ускоряет обработку локального блока. Такой подход позволяет задействовать несколько процессов
(потенциально на разных узлах) и все ядра внутри каждого процесса. В рамках работы тестирование проводилось на одной
машине с несколькими MPI-процессами (каждый с двумя потоками OpenMP).

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Вход: вектор точек `std::vector<std::tuple<double, double>>`, распределяемый между MPI-процессами. Выход: выпуклая
оболочка в порядке обхода против часовой стрелки, начиная с самой левой-нижней точки. При числе точек < 3 оболочка
совпадает с входным множеством.

## 3. Базовый алгоритм (из SEQ)

Алгоритм Джарвиса: внешний `while` перебирает вершины оболочки, внутренний цикл ищет следующую точку, максимизирующую
правый поворот. В гибридной версии каждый процесс обрабатывает свою порцию точек, а глобальный выбор лучшего кандидата
осуществляется коллективной операцией `MPI_Allreduce` с пользовательской функцией редукции.

## 4. Межпроцессная схема (MPI)

### Распределение данных

В `PreProcessingImpl` процесс 0 широковещательно рассылает количество точек `n`, а затем координаты всех точек
(упакованные в массив `double`). Все процессы получают полную копию входных данных, но при вычислениях каждый процесс
работает только со своим непрерывным блоком индексов `[start, end)`, определяемым по рангу:

```cpp
start = (n * rank) / size;
end   = (n * (rank + 1)) / size;
```

### Пользовательские MPI-операции редукции

Для глобального выбора лучшей точки реализованы две пользовательские функции редукции (`LeftmostReduce` и `NextReduce`)
и соответствующие `MPI_Op`. Они оперируют тройками `(x, y, index)` (тип `MPI_DOUBLE` × 3). Функции сравнивают две тройки
по геометрическому критерию (левее/ниже для стартовой точки; правее/дальше для следующей вершины) и оставляют лучшую.

### Поиск левой точки и следующей вершины

- `FindLeftmostPoint`: каждый процесс (с помощью OpenMP) ищет самую левую-нижнюю точку в своём блоке, затем
  `MPI_Allreduce` с `op_leftmost_` выбирает глобально левую точку. Результат доступен всем процессам.
- `FindNextPoint`: аналогично, каждый процесс ищет в своём блоке лучшего кандидата для следующей вершины оболочки
  (относительно текущей `current`), после чего `MPI_Allreduce` с `op_next_` находит глобально оптимальную точку. Перед
  вызовом `Allreduce` статические переменные `s_curr_x, s_curr_y, s_epsilon` обновляются для использования внутри
  `NextReduce`.

Таким образом, каждый шаг построения оболочки требует одной коллективной операции `MPI_Allreduce`, но при этом все
процессы выполняют симметричные вычисления, и внешний цикл одинаков для всех.

### Синхронизация и коммуникация

- `MPI_Bcast` используется при инициализации для рассылки данных.
- `MPI_Allreduce` на каждом шаге внешнего цикла синхронизирует процессы и выбирает глобального кандидата.
- Явные `MPI_Barrier` не используются; синхронизация обеспечивается коллективными операциями.

## 5. Внутрипроцессная схема (OpenMP)

Внутри каждого MPI-процесса распараллеливание повторяет OMP-версию, но на меньшем диапазоне (только локальный блок
процесса):

- `#pragma omp parallel` с `default(none)` и явным перечислением shared/private переменных.
- `#pragma omp for nowait` для распределения итераций между потоками.
- Каждый поток находит своего локального кандидата, затем в критической секции (`#pragma omp critical`) выбирается
  лучший кандидат процесса.
- После завершения параллельной области процесса результат участвует в `MPI_Allreduce`.

Число потоков задаётся через `OMP_NUM_THREADS` (обычно 2, как в проведённых тестах).

## 6. Детали реализации

**Файлы:**

- `all/include/ops_all.hpp` - объявление класса `KutuzovITestConvexHullALL`.
- `all/src/ops_all.cpp` - реализация с MPI и OpenMP.

**Инициализация (`PreProcessingImpl`):**

- Проверяется флаг инициализации MPI; при необходимости выполняется `MPI_Init_thread` с уровнем `MPI_THREAD_FUNNELED`.
- Определяются ранг и размер коммуникатора.
- Создаются пользовательские операции редукции (`LeftmostReduce`, `NextReduce`) и MPI-тип (тройка double).
- Процесс 0 рассылает размер данных и координаты точек через `MPI_Bcast`; все процессы формируют локальную копию
  входного вектора.

**Основной цикл (`RunImpl`):**

```cpp
while (true) {
    output.push_back(input[current]);
    next = FindNextPoint(...);
    if (next == leftmost) break;
    current = next;
    ...
}
```

- На каждом шаге вызывается `FindNextPoint`, который внутри:
  1. OpenMP-потоки ищут кандидата в блоке процесса.
  1. Лучший кандидат процесса сохраняется в `local_cand`.
  1. `MPI_Allreduce(local_cand, global_cand, 1, type_next_, op_next_, MPI_COMM_WORLD)`.
  1. Возвращается индекс глобально лучшей точки.
- После завершения цикла `PostProcessingImpl` освобождает MPI-ресурсы (операции и тип).

## 7. Проверка корректности

Гибридная версия проверена на тех же трёх функциональных тестах (квадрат, прямая, треугольник). Результаты побитово
совпадают с эталоном SEQ для конфигураций с 2, 4 и 8 MPI-процессами при 2 потоках OpenMP на каждом.

## 8. Экспериментальная среда

```text
CPU:       i5 14600KF, 6P+8E ядер, 20 потоков
RAM:       32GB DDR5
Build:     Release
Конфигурации:
  PPC_NUM_PROC: 2, 4, 8
  PPC_NUM_THREADS (OMP): 2
  Всего: процессы * потоки.
```

Размер задачи: 5 000 000 случайных точек в квадрате [-1000, 1000].

## 10. Анализ и выводы

**Ускорение и масштабируемость:**

- Конфигурация 2 процесса × 2 потока (4 workers) дала ускорение 3.59x, что близко к OMP 4 потока (3.72x), но немного
  ниже из-за дополнительных коммуникаций MPI.
- При 4 процессах × 2 потока (8 workers) ускорение 4.26x - выше, чем у OMP 8 потоков (3.71x), и значительно уступает TBB
  4 потока (14.27x). Это объясняется тем, что каждый MPI-процесс обрабатывает меньший объём данных, и внутрипроцессная
  конкуренция за критическую секцию снижена (всего 2 потока на процесс). Коллективная редукция `MPI_Allreduce` вносит
  свой overhead, но она выполняется эффективнее, чем сериализация через `critical` в 8-поточном OMP.
- Дальнейшее увеличение до 8 процессов × 2 потока (16 workers) даёт ускорение 5.45x, но эффективность падает до 34%.
  Прирост замедляется из-за растущих накладных расходов на коммуникацию (Allreduce на каждом шаге) и дисбаланса (внешний
  цикл по-прежнему последователен).

**Сравнение с другими технологиями:**

- **ALL vs OMP:** на 4 workers результаты сопоставимы. На 8 workers ALL превосходит OMP (4.26x vs 3.71x), так как
  разделение на процессы снижает нагрузку на критическую секцию.
- **ALL vs TBB:** TBB значительно быстрее (14.27x на 4 потока), поскольку использует эффективную редукцию `combinable`
  без межпроцессных обменов. ALL проигрывает из-за неизбежных MPI-коммуникаций.
- **ALL vs STL:** ALL (4.26x на 8 workers) быстрее STL (4.11x на 4 потока, 3.28x на 8). Однако STL на 8 потоках показал
  падение производительности, а ALL продолжает масштабироваться, пусть и с уменьшающейся отдачей.

**Накладные расходы гибридной версии:**

- `MPI_Allreduce` на каждом шаге оболочки - главный источник накладных расходов. При большом количестве вершин оболочки
  (H) эти затраты растут линейно.
- Дополнительное копирование данных (рассылка координат, создание локальных копий) также вносит вклад, но для 5 млн
  точек оно было однократным и не критичным.
- Внутрипроцессная синхронизация (OpenMP `critical`) с двумя потоками остаётся лёгкой.

**Когда гибридный подход оправдан:**

- При очень больших наборах данных, которые не помещаются в память одного узла - MPI позволяет распределить данные.
- При использовании многопроцессорных узлов кластера, где чисто потоковые технологии не могут выйти за пределы одного
  узла.
- Когда внутрипроцессное распараллеливание уже не даёт прироста из-за конкуренции, а добавление процессов позволяет
  задействовать дополнительные ядра с меньшей синхронизацией на каждом.

**Ограничения:**

- Внешний цикл алгоритма Джарвиса последователен, что ограничивает общее ускорение (закон Амдала).
- При малом размере оболочки (H) накладные расходы на коллективные операции становятся доминирующими.

**Итог:** Гибридная версия демонстрирует приемлемую масштабируемость и превосходит некоторые чисто потоковые реализации
при большом числе workers, однако уступает эффективной TBB-редукции. Она полезна как демонстрация иерархического
параллелизма и может быть предпочтительна в распределённых системах.
