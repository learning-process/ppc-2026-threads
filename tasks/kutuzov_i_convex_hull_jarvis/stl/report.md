## 1. Контекст

STL-версия реализует распараллеливание алгоритма Джарвиса с использованием низкоуровневого управления потоками
(`std::thread`). Цель - оценить накладные расходы ручного создания и объединения потоков, а также сравнить эффективность
с высокоуровневыми технологиями (OpenMP, TBB). В отличие от них, здесь каждый шаг построения оболочки порождает новый
набор потоков, что делает реализацию наглядной, но потенциально менее производительной при большом количестве итераций.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Вход: вектор точек `std::vector<std::tuple<double, double>>`, выход: выпуклая оболочка в порядке обхода против часовой
стрелки, начиная с самой левой-нижней точки. При числе точек < 3 оболочка совпадает с входным множеством.

## 3. Базовый алгоритм (из SEQ)

Внешний цикл `while` перебирает вершины оболочки. На каждом шаге внутренний цикл перебирает все точки, чтобы найти
следующую вершину, используя векторное произведение и сравнение расстояний. Именно внутренний цикл распараллелен в
STL-версии, так же как в OMP и TBB.

## 4. Схема распараллеливания

### Ручное разбиение на диапазоны

Массив точек делится на `num_threads` непрерывных блоков равного (или почти равного) размера. Формула разбиения:

```cpp
size_t start = (tid * n) / num_threads;
size_t end = ((tid + 1) * n) / num_threads;
```

Каждый поток обрабатывает свой диапазон независимо.

### Локальные кандидаты и объединение

Поток находит **локально лучшую** точку-кандидата в своём диапазоне с помощью `FindBestCandidateInRange` (или
`FindLeftmostInRange` для стартовой точки). Результат сохраняется в общий вектор `locals[tid]`. После завершения всех
потоков (`join`) главный поток вызывает `FindGlobalBest`, который последовательно сравнивает всех локальных кандидатов и
выбирает глобально лучшего. Синхронизация не требуется, так как каждый поток пишет в свою ячейку вектора, а чтение
общего входного массива безопасно (только чтение).

### Потоки создаются на каждом шаге

В отличие от OpenMP и TBB, где пул потоков переиспользуется, здесь на каждой итерации внешнего цикла (построения одной
вершины оболочки) заново создаются `num_threads` потоков. Это увеличивает накладные расходы, особенно при большом числе
потоков и большом размере оболочки.

### Число потоков

Используется `std::thread::hardware_concurrency()` для определения максимального числа потоков. В экспериментах число
потоков задавалось через переменную `PPC_NUM_THREADS` (2, 4, 8), однако в предоставленном коде `GetNumThreads()`
возвращает только аппаратное значение и не учитывает `PPC_NUM_THREADS`. Вероятно, для тестов использовалась
модифицированная версия или управление осуществлялось иным способом. В анализе мы ориентируемся на фактические замеры с
указанными конфигурациями.

## 5. Детали реализации

**Файлы:**

- `stl/include/ops_stl.hpp` – объявление класса `KutuzovITestConvexHullSTL`.
- `stl/src/ops_stl.cpp` – реализация с `std::thread`.

**Ключевые вспомогательные функции (анонимное пространство имён):**

- `FindLeftmostInRange` – линейный поиск самой левой (нижней) точки в диапазоне.
- `FindBestCandidateInRange` – поиск лучшей следующей точки для заданного `current` в диапазоне, используя критерий
  Джарвиса.
- `FindGlobalBest` – выбор глобально лучшей точки из вектора локальных кандидатов.
- `GetNumThreads` – возвращает `std::thread::hardware_concurrency()` (если 0, то 1).

**Основной цикл RunImpl (псевдокод):**

```cpp
while (true) {
    output.push_back(points[current]);

    // Создаём потоки для поиска следующей точки
    for (unsigned tid = 0; tid < num_threads; ++tid) {
        threads.emplace_back([&, tid, start, end]() {
            locals[tid] = FindBestCandidateInRange(points, start, end, current, ...);
        });
    }
    // Ожидаем завершения всех потоков
    for (auto& t : threads) t.join();

    // Выбираем глобального кандидата
    BestCandidate global = FindGlobalBest(locals, ...);
    current = global.index;
    if (current == leftmost) break;
}
```

**Поиск левой точки** выполняется аналогично, но однократно перед циклом.

**Синхронизация:** отсутствует, так как потоки только читают входной массив и пишут в выделенные им ячейки. Объединение
результатов - последовательное.

## 6. Проверка корректности

Используются те же три функциональных теста, что и для SEQ: квадрат с внутренними точками, все точки на прямой,
треугольник. Результат полностью совпадает с эталоном для конфигураций с 2, 4, 8 потоками. Детерминизм обеспечен
последовательным объединением локальных кандидатов и фиксированным порядком сравнения в `FindGlobalBest`.

## 9. Анализ и выводы

**Суперлинейное ускорение на 2 потоках (3.59x, эффективность 180%)**\
На двух потоках, как и в OMP/TBB, наблюдается суперлинейность благодаря лучшему использованию кэш-памяти: половина
данных помещается в кэш каждого ядра, снижая промахи.

**Умеренный рост на 4 потоках (4.11x, эффективность 103%)**\
Ускорение продолжается, но эффективность падает почти до 100% - накладные расходы на создание/join потоков начинают
сказываться.

**Падение производительности на 8 потоках (3.28x, эффективность 41%)**\
При 8 потоках время увеличивается по сравнению с 4, а ускорение становится ниже, чем на 2 потоках. Причина -
значительные накладные расходы на создание и уничтожение потоков на каждом шаге оболочки, а также возросшая конкуренция
за общие ресурсы (кэш, контроллер памяти). В отличие от OMP и TBB, где потоки переиспользуются, STL-версия создаёт их
заново для каждой вершины оболочки, что при 8 потоках приводит к доминированию накладных расходов над полезной работой.

**Сравнение с OMP и TBB:**

- STL (4 потока) чуть быстрее OMP (0.063 c против 0.070 c), но значительно медленнее TBB (0.018 c).
- На 8 потоках STL медленнее и OMP, и TBB, и даже своего варианта с 4 потоками. Это наглядно демонстрирует цену ручного
  управления потоками без переиспользования.

**Когда STL-подход оправдан:**

- Если число вершин оболочки мало (H мало), накладные расходы на создание потоков не успевают проявиться.
- При небольшом фиксированном числе потоков (2-4), где overhead ещё невелик.
- В образовательных целях для демонстрации низкоуровневого управления параллелизмом.

**Ограничения:**

- Потоки не переиспользуются - критично при большом числе итераций внешнего цикла.
- Отсутствие учёта `PPC_NUM_THREADS` в текущей реализации снижает гибкость, но для тестов, видимо, использовалась
  адаптированная версия.

**Вывод:** STL-версия демонстрирует приемлемое ускорение на 2-4 потоках, однако резкое падение эффективности на 8
потоках подчёркивает важность переиспользования потоков и баланса между накладными расходами и параллелизмом. Для задач
с длительным внешним циклом рекомендуется использовать технологии с пулом потоков (TBB, OpenMP).
