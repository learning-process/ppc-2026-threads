## 1. Контекст

TBB-версия продолжает распараллеливание алгоритма Джарвиса с использованием Intel oneTBB. В отличие от OpenMP, здесь
применяется задачно-ориентированный подход с высокоуровневыми примитивами `parallel_for` и `combinable` для
потокобезопасного сбора локальных результатов. Это позволяет избавиться от критических секций и получить более
эффективную редукцию, сохраняя последовательный внешний цикл.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Вход: вектор точек `std::vector<std::tuple<double, double>>`, выход: выпуклая оболочка в порядке обхода против часовой
стрелки, начиная с самой левой-нижней точки.

## 3. Базовый алгоритм (из SEQ)

Внешний цикл `while` по вершинам оболочки, внутренний `for` по всем точкам для поиска следующей вершины с помощью
cross-произведения и сравнения расстояний. Сложность O(N·H). В TBB-версии внутренний цикл заменён на
`FindNextPointHelper`, который выполняет параллельный поиск.

## 4. Схема распараллеливания

### Выбранный примитив: `tbb::parallel_for` + `tbb::combinable`

Распараллелены два участка:

- Поиск самой левой точки (один раз перед основным циклом) - `FindLeftmostPointHelper`.
- Поиск следующей точки оболочки на каждом шаге - `FindNextPointHelper`.

Оба используют один и тот же паттерн:

- `tbb::combinable<BestPoint>` создаёт независимый экземпляр структуры-кандидата для каждого потока.
- `tbb::parallel_for` обрабатывает диапазон индексов `[0, N)`.
- Внутри каждый поток обновляет свой локальный `BestPoint`.
- После цикла `combine()` объединяет все локальные результаты в глобальный путём попарного применения ассоциативной
  операции (выбор лучшей точки).

### Детали диапазона и разбиения

```cpp
tbb::parallel_for(tbb::blocked_range<size_t>(0, input.size()), ...,
                  tbb::static_partitioner{});
```

- Диапазон `0 .. input.size()`.
- `blocked_range` без явного grainsize - TBB автоматически выбирает размер блока.
- `static_partitioner` - принудительное статическое распределение итераций, аналогичное `schedule(static)` в OpenMP,
  минимизирует накладные расходы и даёт предсказуемую балансировку при равномерной нагрузке.

### Локальное хранилище: `tbb::combinable`

```cpp
tbb::combinable<BestPoint> local_best([&]() { return init; });
```

- Каждый поток получает копию `init` (начальная точка - первая доступная после `current` или левая нижняя для поиска
  стартовой).
- Доступ к локальному экземпляру через `local_best.local()` - без гонок.
- Финальное объединение: `local_best.combine(лямбда)`, которая из двух кандидатов выбирает лучшего, используя тот же
  критерий `IsBetterPoint` (через cross-произведение). Операция ассоциативна, поэтому результат детерминирован и не
  зависит от порядка потоков.

### Почему это эффективнее критических секций?

В OMP-версии каждый поток после обработки порции входил в `critical` для обновления глобального `next`. При большом
числе потоков `critical` сериализует этот этап и вызывает ожидание. TBB-версия полностью избегает критических секций:
`combine` вызывает редукционную лямбду в однопоточном режиме после завершения всех потоков, что быстрее и не создаёт
конкуренции.

### Число потоков

Используется значение `tbb::global_control` или настройка `PPC_NUM_THREADS` окружением (как и в OMP). В коде не задаётся
явно.

**Файлы:**

- `tbb/src/ops_tbb.cpp` - реализация с TBB.

- `RunImpl` использует `FindLeftmostPointHelper` и `FindNextPointHelper`.

**Ключевые фрагменты:**

*Поиск левой точки:*

```cpp
tbb::combinable<BestPoint> local_best([&]() { return init; });
tbb::parallel_for(tbb::blocked_range<size_t>(0, input.size()),
  [&](const tbb::blocked_range<size_t> &r) {
    BestPoint &best = local_best.local();
    for (size_t i = r.begin(); i != r.end(); ++i) {
      // обновление best
    }
  }, tbb::static_partitioner{});
BestPoint global = local_best.combine([](const BestPoint &a, const BestPoint &b) {
  return (b.x < a.x) || (b.x == a.x && b.y < a.y) ? b : a;
});
```

*Поиск следующей точки оболочки:*

```cpp
tbb::combinable<BestPoint> local_next([&]() { return init; });
tbb::parallel_for(tbb::blocked_range<size_t>(0, input.size()),
  [&](const tbb::blocked_range<size_t> &r) {
    BestPoint &best = local_next.local();
    for (size_t i = r.begin(); i != r.end(); ++i) {
      if (i == current) continue;
      double cross = CrossProduct(current_x, current_y, best.x, best.y, i_x, i_y);
      if (IsBetterPoint(cross, epsilon, ..., best.x, best.y)) {
        best = {i, i_x, i_y};
      }
    }
  }, tbb::static_partitioner{});
size_t next = local_next.combine([&](const BestPoint &a, const BestPoint &b) {
  double cross = CrossProduct(current_x, current_y, a.x, a.y, b.x, b.y);
  return IsBetterPoint(cross, epsilon, ..., b.x, b.y, a.x, a.y) ? b : a;
}).idx;
```

**Синхронизация:** отсутствует явная синхронизация, всё обеспечивается `combinable` и редукцией после `parallel_for`.
Потоки не конкурируют за общий ресурс.

## 9. Анализ и выводы

**Значительное суперлинейное ускорение (6.77x на 2, 14.27x на 4, 22.1x на 8 потоках)**\
TBB-версия демонстрирует существенно более высокую производительность по сравнению с OpenMP (на 4 потоках OMP - 3.72x,
TBB - 14.27x). Это объясняется комбинацией факторов:

- **Отсутствие критических секций:** редукция через `combinable.combine()` выполняется один раз после цикла, не создавая
  конкуренции.
- **Эффективное использование кэша:** при разбиении на потоки каждый поток обрабатывает компактный блок данных, что
  улучшает локальность и снижает кэш-промахи. Суперлинейный эффект особенно заметен на 2 и 4 потоках.
- **Статическое разбиение** без динамического перераспределения устраняет overhead планирования.

**Эффективность >100% на 2 и 4 потоках** - признак суперлинейности, обусловленной архитектурными особенностями (кэш,
частота). На 8 потоках эффективность снижается до 276%, но ускорение продолжает расти (22x), что говорит о хорошей
масштабируемости даже при гипертрединге.

**Сравнение с OMP:** TBB быстрее OMP на одинаковом числе потоков примерно в 3-4 раза. Это достигается за счёт более
эффективного механизма редукции и отсутствия ожидания в критической секции. OMP-версия страдает от сериализации при
`critical`, тогда как TBB использует полностью параллельную редукцию.

**Ограничения:**

- Внешний цикл остаётся последовательным, но при очень большом N его доля пренебрежимо мала.
- Суперлинейность не гарантируется на других платформах или меньших наборах данных.

**Оптимальная конфигурация:** на данной машине TBB эффективно использует все доступные потоки, и увеличение их числа до
8 даёт прирост (22x), хотя эффективность падает. Приоритетной остаётся конфигурация с 4 физическими ядрами (14x), как
баланс между производительностью и энергопотреблением.

Таким образом, TBB является наиболее производительным вариантом распараллеливания для данного алгоритма на данном
оборудовании.
