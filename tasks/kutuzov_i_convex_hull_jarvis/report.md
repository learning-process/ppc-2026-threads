## 1. Введение

Задача построения выпуклой оболочки множества точек на плоскости методом Джарвиса выбрана для сравнения различных
моделей параллельного программирования: директивной (OpenMP), задачно-ориентированной (oneTBB), низкоуровневой ручной
(std::thread) и гибридной (MPI + OpenMP). Задача интересна тем, что внутренний цикл сканирования точек хорошо
параллелится, а внешний цикл остаётся последовательным, что позволяет оценить влияние последовательной доли по закону
Амдала, а также накладные расходы на синхронизацию и коммуникацию.

## 2. Единая постановка задачи

**Входные данные:** вектор точек на плоскости `std::vector<std::tuple<double, double>>` (координаты x, y).

**Выходные данные:** вектор точек, образующих выпуклую оболочку в порядке обхода против часовой стрелки, начиная с самой
левой-нижней точки. Точки на рёбрах оболочки включаются (алгоритм оставляет коллинеарные точки с максимальным
расстоянием).

**Ограничения:**

- Если количество входных точек меньше трёх, оболочка совпадает с исходным множеством.
- Допускаются дубликаты точек, пустой вход (пустой выход).
- Все вычисления в double, сравнение коллинеарности с эпсилоном `1e-9`.

**Критерий корректности:** результат каждой параллельной реализации должен побитово совпадать с результатом
последовательной версии (SEQ).

## 3. Единая методика эксперимента

### Окружение

```text
CPU:       Intel Core i5 14600KF (6P+8E ядер, 20 потоков)
RAM:       32 GB DDR5
OS:        Ubuntu 24.04 (предположительно)
Compiler:  g++ 13.3.0
CMake:     версия 3.28
Build:     Release
```

### Переменные окружения

| Переменная      | Назначение                                                                                               |
| --------------- | -------------------------------------------------------------------------------------------------------- |
| PPC_NUM_THREADS | Число потоков для OMP, TBB, STL и внутрипроцессного уровня ALL (использовались 2, 4, 8)                  |
| PPC_NUM_PROC    | Число MPI-процессов для гибридной версии (использовались 2, 4, 8) при фиксированных 2 потоках на процесс |

### Генерация данных

Функциональные тесты: три детерминированных набора точек (квадрат с внутренними точками, все точки на прямой,
треугольник). Тесты производительности: 5 000 000 случайных точек, равномерно распределённых в квадрате [-1000, 1000] с
помощью `std::mt19937` и `std::uniform_real_distribution<double>`.

### Метрики

**Ускорение (Speedup):** `Speedup = T_seq / T_parallel`, где `T_seq = 0.2588773096 c` (медиана SEQ).

**Эффективность (Efficiency):** `Efficiency = Speedup / total_workers`, где `total_workers` - число потоков (OMP, TBB,
STL) или `процессы × потоки` (ALL). Эффективность >100% означает суперлинейное ускорение.

### Процедура измерений

- 10 повторов для каждой конфигурации.
- Фиксируется медиана времени.
- Использовался режим `--running-type=performance`.

## 4. Сводка корректности

Все backend-ы проверены на трёх функциональных тестах. Результаты побитово совпадают с эталоном SEQ.

| Backend | Тесты пройдены | Примечание                                           |
| ------- | -------------- | ---------------------------------------------------- |
| SEQ     | Да             | Эталон                                               |
| OMP     | Да             | `schedule(static)`, локализация через `critical`     |
| TBB     | Да             | `combinable` + `parallel_for`, `static_partitioner`  |
| STL     | Да             | Ручное разбиение, создание потоков на каждом шаге    |
| ALL     | Да             | MPI+OpenMP, `Allreduce` с пользовательской операцией |

**Ограничения применимости:**

- При числе точек < 3 все версии возвращают исходный набор.
- Пустой вход → пустой выход.
- При числе потоков/процессов, превышающем количество точек, некоторые workers могут простаивать.

## 6. Интерпретация различий

### SEQ (базовый эталон)

Последовательный алгоритм Джарвиса выполняет обход всех точек для каждой вершины оболочки. Время 0.2589 с на 5 млн
случайных точек. Это baseline для всех расчётов ускорения.

### OpenMP (до 3.72x на 4 потоках, насыщение на 8)

OpenMP показал суперлинейное ускорение на 2 потоках (2.57x) благодаря лучшему использованию кэша, но при 4 потоках
ускорение 3.72x с эффективностью 93%. Переход на 8 потоков не дал прироста (3.71x, 46%) из-за конкуренции за критическую
секцию и последовательной доли внешнего цикла.

### TBB (наилучшее ускорение, до 22.1x на 8 потоках)

TBB продемонстрировал впечатляющие результаты: 6.77x на 2, 14.27x на 4 и 22.1x на 8 потоках. Суперлинейность обусловлена
отсутствием критических секций (используется `combinable` с эффективной редукцией) и отличной локальностью данных. Это
самый быстрый backend на данной задаче.

### STL (пик на 4 потоках, деградация на 8)

STL показал ускорение 3.59x (2 потока) и 4.11x (4 потока), что сравнимо с OMP. Однако при 8 потоках время возросло до
0.0788 с (3.28x) - накладные расходы на создание и join потоков на каждом шаге оболочки стали доминировать. STL не
использует пул потоков, что критично при большом количестве итераций.

### ALL (MPI+OpenMP, умеренное ускорение)

Гибридная версия с 2 процессами × 2 потока (4 workers) дала ускорение 3.59x, с 4×2 (8 workers) - 4.26x, с 8×2 (16
workers) - 5.45x. Рост есть, но с падающей эффективностью. Коллективная операция `MPI_Allreduce` на каждом шаге и
дублирование данных в каждом процессе ограничивают масштабируемость. ALL превосходит OMP на 8 workers (4.26x vs 3.71x),
но значительно уступает TBB.

**Общие закономерности:**

- Во всех реализациях суперлинейное ускорение на малом числе потоков вызвано эффектами кэш-памяти.
- Технологии с низкими накладными расходами на синхронизацию (TBB) показывают наилучшие результаты.
- Последовательная часть алгоритма (внешний цикл) ограничивает максимальное ускорение при росте числа workers.

## 8. Заключение

### Какая версия лучше и при каких условиях

| Условия                                   | Рекомендуемая версия | Ускорение (4 workers) |
| ----------------------------------------- | -------------------- | --------------------- |
| Максимальная производительность           | TBB                  | 14.27x                |
| Хорошая производительность без TBB        | STL (до 4 потоков)   | 4.11x                 |
| Быстрая разработка и кроссплатформенность | OpenMP               | 3.72x                 |
| Распределённые вычисления                 | ALL (MPI+OpenMP)     | 3.59x (4 workers)     |

### Ограничения сравнения

- Размер задачи фиксирован (5 млн точек), распределение равномерное.
- Оборудование: один узел с процессором i5-14600KF (гибридная архитектура P/E ядер).
- ALL тестировалась на одной машине, без реальной сети.
- Число потоков/процессов подбиралось ограниченно; поведение на многих узлах может отличаться.

### Что можно улучшить

1. Использовать динамическую балансировку, если распределение точек неравномерное.
1. Оптимизировать редукцию в OMP/ALL (например, редукция массивов фиксированного размера вместо `critical`).
1. Переиспользовать потоки в STL (пул потоков) для уменьшения overhead на создание.
1. В ALL рассмотреть асинхронные MPI-операции для перекрытия коммуникаций и вычислений.

### Итоговая оценка

На данной задаче TBB показал выдающуюся производительность (14.27x на 4 потоках), значительно опередив остальные
технологии благодаря эффективной редукции и отличному использованию кэша. OpenMP и STL достигли скромного ускорения
(3.7–4.1x), ограниченного накладными расходами на синхронизацию. Гибридный MPI+OpenMP уступает TBB, но может быть
полезен в распределённой среде. Основное узкое место - последовательный внешний цикл алгоритма Джарвиса; для дальнейшего
роста ускорения требуется изменение самого алгоритма (например, переход к алгоритму Грэхема с параллельной сортировкой).
