## 1. Контекст

В данной работе представлена реализация на библиотеке **oneTBB (Threading Building Blocks)
*. TBB выбран по следующим причинам:

- Высокоуровневый задачно-ориентированный подход, скрывающий управление потоками.
- Автоматическая балансировка нагрузки и адаптивное разбиение диапазонов.
- Наличие примитива `parallel_reduce`, который естественно выражает операцию свёртки
(выбор лучшей точки) без явного массива локальных результатов.

Цель – оценить эффективность TBB по сравнению с последовательной версией.

## 2. Постановка задачи

Полностью совпадает с последовательной версией (см. `seq/report.md`). Вход – множество
точек, выход – вершины выпуклой оболочки в порядке обхода против часовой стрелки.
Обработка крайних случаев (1, 2 точки, дубликаты, коллинеарность) аналогична SEQ.

## 3. Базовый алгоритм

Строим выпуклую оболочку множества точек, последовательно находя следующую вершину с
максимальным левым поворотом относительно текущей, начиная с самой левой нижней точки.
Подробности – в отчёте SEQ.

## 4. Схема распараллеливания

Вместо явных потоков или директив OpenMP используется **tbb::parallel_reduce** –
высокоуровневый примитив для параллельного выполнения свёртки (редукции) над диапазоном.

**Ключевые решения**:

- **Примитив**: `tbb::parallel_reduce` – подходит для неассоциативной, но коммутативной
операции выбора «лучшей» точки (на основе левого поворота).
- **Диапазон**: `tbb::blocked_range<size_t>(0, n)` – разбивает массив точек на
поддиапазоны, которые распределяются между потоками.
- **Grainsize**: не указан явно; TBB выбирает размер зёрен автоматически. Для задачи с
2000 точек и лёгкой вычислительной нагрузкой на итерацию это может привести к большому
числу мелких зёрен и повышенному оверхеду.
- **Partitioner**: по умолчанию `auto_partitioner` (адаптивный). В текущей реализации не
переопределён.
- **Контроль конкуренции**: число потоков управляется через
`tbb::global_control::max_allowed_parallelism` (не используется в коде) или через
переменные окружения TBB (например, `TBB_NUM_THREADS`). В инфраструктуре курса обычно
задаётся `PPC_NUM_THREADS`, которое также влияет на TBB.

**Схема работы**:

1. Инициализируется начальное состояние `init` – первая точка, отличная от `current`. Если
такая не найдена, возвращается `current`.
2. Вызывается `tbb::parallel_reduce` с:
   - диапазоном `[0, n)`
   - начальным значением `init`
   - телом `body`, которое для каждого поддиапазона обновляет локальный `BestState`
   - функцией редукции `reduce`, объединяющей два `BestState` в один.
3. Тело `body` выполняется параллельно для разных поддиапазонов. Внутри него происходит
перебор точек поддиапазона и обновление локального лучшего кандидата.
4. Результат редукции – глобальная лучшая точка.

**Почему parallel_reduce, а не parallel_for?**
Операция выбора «лучшей» точки сводит множество значений к одному. `parallel_reduce`
естественно выражает эту схему, избегая явного массива локальных результатов и
последующего объединения (это делает библиотека).

## 5. Детали реализации

```cpp
Point OrehovNJarvisPassTBB::FindNext(Point current) const {
  const size_t n = input_.size();

  BestState init = FindInitialBestState(input_, current);
  if (!init.valid) {
    return current;
  }

  auto body = [&](const tbb::blocked_range<size_t> &r, BestState local) -> BestState {
    for (size_t i = r.begin(); i != r.end(); ++i) {
      const Point &p = input_[i];
      if (p == current) {
        continue;
      }
      if (!local.valid || IsBetterPoint(current, p, local.point)) {
        local.point = p;
        local.valid = true;
      }
    }
    return local;
  };

  auto reduce = [&](const BestState &a, const BestState &b) -> BestState { return
  ReduceBestStates(a, b, current); };

  BestState result = tbb::parallel_reduce(tbb::blocked_range<size_t>(0, n), init, body,
  reduce);
  return result.point;
}
```

## 6. Проверка корректности

- **Сравнение результатов**: выходной вектор `res_` сравнивается с эталонным вектором,
заданным в функциональных тестах. Порядок точек важен: оболочка должна быть перечислена,
начиная с самой левой нижней точки, против часовой стрелки.

- **Функциональные тесты**:
  - Тест 1: 7 точек, образующих выпуклый многоугольник. Ожидается 5 вершин: (0,0), (0,3),
  (2,4), (4,2), (3,0).
  - Тест 2: одна точка (0,0) → оболочка {(0,0)}.
  - Тест 3: две точки (0,0) и (1,0) → оболочка {(0,0), (1,0)}.

## 8. Результаты

Базовое время последовательной версии T_seq = 0.18437 с. Для TBB-версии использовался
режим task_run. Приведены результаты однократного запуска для каждого числа потоков;
повторные запуски показали стабильность (разброс менее 2%).

Потоков | Время, с | Speedup (T_seq / T_tbb) | Efficiency (Speedup / threads) |
1       | 0.57135  |          0.323           |             0.323              |
2       | 0.58905  |          0.313           |             0.156              |
4       | 0.57897  |          0.318           |             0.080              |

Отсутствие ускорения: TBB-версия работает медленнее последовательной в ~3 раза при любом
числе потоков. Это указывает на наличие значительных накладных расходов, которые не
компенсируются параллелизацией.

Нет масштабирования: увеличение числа потоков с 1 до 4 не изменяет время выполнения (все
значения лежат в пределах 0.57–0.59 с). Это говорит о том, что узкое место находится не в
вычислительной части, а в оверхеде.

Возможные причины:

Создание parallel_reduce на каждом шаге внешнего цикла – алгоритм Джарвиса выполняет около
2000 итераций. На каждой итерации вызывается parallel_reduce, который создаёт задачи,
распределяет диапазоны и выполняет редукцию. При малом объёме работы на итерацию (2000
лёгких операций) эти накладные расходы доминируют.

## 9. Выводы

TBB-реализация алгоритма Джарвиса не даёт выигрыша в производительности на данной задаче
(2000 точек на окружности). Напротив, она работает существенно медленнее последовательной
версии и не масштабируется с ростом числа потоков.

Когда TBB мог бы быть полезен:

- При гораздо большем размере задачи, когда объём работы на итерацию окупает оверхед.
