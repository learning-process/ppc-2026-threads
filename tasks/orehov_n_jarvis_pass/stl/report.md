## 1. Контекст
Алгоритм Джарвиса (Gift wrapping) – классический метод построения выпуклой оболочки множества точек на плоскости. Его вычислительная сложность составляет O(n·h), где n – количество точек, h – размер оболочки. В худшем случае (все точки лежат на оболочке) сложность становится квадратичной O(n²).

Основная вычислительная нагрузка приходится на внутренний цикл, который для каждой текущей вершины оболочки перебирает все n точек, вычисляя ориентацию (левый поворот) и выбирая следующую вершину. Этот цикл является идеальным кандидатом для параллелизации, так как итерации независимы.

В данной работе представлена реализация на **std::thread** – низкоуровневом примитиве библиотеки C++ для ручного управления потоками. Цель – оценить эффективность явного создания потоков по сравнению с высокоуровневыми подходами (OpenMP, TBB) и выявить накладные расходы, связанные с созданием, синхронизацией и объединением результатов.

## 2. Постановка задачи
Полностью совпадает с последовательной версией (см. `seq/report.md`). Вход – множество точек, выход – вершины выпуклой оболочки в порядке обхода против часовой стрелки. Обработка крайних случаев (1, 2 точки, дубликаты, коллинеарность) аналогична SEQ.

## 3. Базовый алгоритм
Строим выпуклую оболочку множества точек, последовательно находя следующую вершину с максимальным левым поворотом относительно текущей, начиная с самой левой нижней точки.
Подробности – в отчёте SEQ.

## 4. Схема распараллеливания
В реализации на `std::thread` распараллелен поиск следующей вершины в функции `FindNext`. Для каждой текущей вершины `current` необходимо найти точку `next` с максимальным левым поворотом. Этот поиск разбивается на независимые диапазоны, каждый из которых обрабатывается отдельным потоком.

**Ключевые решения**:
- **Разбиение данных**: массив точек `input_` делится на `num_threads` непрерывных диапазонов (chunk + remainder). Каждый поток получает свой поддиапазон `[start, end)`.
- **Локальные результаты**: каждый поток вычисляет лучшую точку в своём диапазоне и сохраняет её в массив `results` (структура `BestState`).
- **Синхронизация**: отсутствует внутри потоков, так как каждый поток пишет только в свою ячейку массива. После завершения всех потоков главный поток последовательно объединяет локальные результаты, применяя критерий `IsBetterPoint`.
- **Управление потоками**: потоки создаются в цикле, каждый запускается с лямбда-функцией, захватывающей необходимые данные по ссылке. После создания всех потоков вызывается `join()` для каждого.

**Почему не используются мьютексы или атомики?**
Поскольку каждый поток работает с независимой частью данных и записывает результат в отдельный элемент массива, гонки данных отсутствуют. Объединение выполняется последовательно после `join`, поэтому дополнительные средства синхронизации не требуются.

**Потенциальные проблемы**:
- Создание и уничтожение потоков на каждом шаге внешнего цикла (около 2000 раз) приводит к огромным накладным расходам.
- При малом размере задачи накладные расходы на создание потоков могут превысить выигрыш от параллелизации.

## 5. Детали реализации
'''cpp
Point OrehovNJarvisPassSTL::FindNext(Point current) const {
  const size_t n = input_.size();
  if (n == 0) return current;

  unsigned int num_threads = std::thread::hardware_concurrency();
  if (num_threads == 0) num_threads = 1;
  if (num_threads > n) num_threads = static_cast<unsigned int>(n);

  size_t chunk = n / num_threads;
  size_t remainder = n % num_threads;
  size_t start = 0;

  std::vector<std::thread> threads;
  std::vector<BestState> results(num_threads);

  for (unsigned int i = 0; i < num_threads; ++i) {
    size_t end = start + chunk + (i < remainder ? 1 : 0);
    threads.emplace_back([this, &results, i, start, end, current]() {
      results[i] = FindBestInRange(input_, current, start, end);
    });
    start = end;
  }

  for (auto &t : threads) t.join();

  BestState final_best;
  for (const auto &best : results) {
    final_best = CombineBestStates(final_best, best, current);
  }
  return final_best.valid ? final_best.point : current;
}
'''

 - FindBestInRange – перебирает точки в диапазоне [start, end), возвращает лучшую точку (структура BestState).

 - CombineBestStates – объединяет два состояния, выбирая лучшее на основе IsBetterPoint.

 - hardware_concurrency() – определяет число аппаратных потоков (на i5-9300H возвращает 8).

Комментарий о масштабируемости и узких местах:

Катастрофическое замедление: STL-версия работает медленнее последовательной примерно в 6 раз (1.08 с против 0.18 с) при любом числе потоков.

Отсутствие масштабирования: увеличение числа потоков с 1 до 4 практически не меняет время выполнения (1.08–1.11 с). Более того, время даже немного растёт.

Основные причины:

 - Создание потоков на каждом шаге внешнего цикла. Алгоритм Джарвиса выполняет около 2000 итераций (h ≈ n). На каждой итерации создаётся num_threads потоков, каждый из которых запускается и затем присоединяется (join). Создание и уничтожение потока – дорогостоящие операции (выделение стеков, системные вызовы). При 2000×4 = 8000 созданий потоков накладные расходы колоссальны.
 - Малый объём работы на поток. Каждый поток обрабатывает всего около 500 точек (при 4 потоках) на одной итерации. Время вычислений ничтожно мало по сравнению с затратами на создание потока.
 - Последовательное объединение результатов – требует дополнительного времени, но не является доминирующим фактором.

## 9. Выводы
STL-реализация на std::thread непригодна для данной задачи в её текущем виде. Она работает в 6 раз медленнее последовательной версии и не масштабируется.

Основные ограничения:

Многократное создание потоков – самое узкое место. Потоки должны создаваться один раз и переиспользоваться для всех шагов внешнего цикла. В текущей реализации потоки пересоздаются на каждом шаге, что ведёт к огромным накладным расходам.

Недостаточная вычислительная нагрузка на поток – даже если бы потоки создавались один раз, объём работы на итерацию мал (500–2000 точек). Для окупаемости оверхеда нужно, чтобы каждый поток обрабатывал значительно больше данных (например, десятки тысяч точек).

Алгоритмическая ограниченность – последовательная природа внешнего цикла Джарвиса ограничивает максимальное ускорение, но не объясняет замедление в 6 раз. Основная проблема – архитектурная ошибка в распараллеливании.
