## 1. Контекст

Данная реализация использует стандартные средства библиотеки C++
для организации параллельных вычислений.

В отличие от OpenMP и TBB, управление workers выполняется
через `std::async`, что делает реализацию более переносимой,
но увеличивает runtime-overhead.

---

## 2. Постановка задачи

Необходимо вычислить двумерный интеграл методом трапеций
на прямоугольной области.

Вычисления выполняются на равномерной сетке `5000 × 5000`.

В качестве baseline используется последовательная SEQ-реализация.

---

## 3. Базовый алгоритм

Алгоритм вычисляет интегральную сумму
по всем точкам двумерной сетки.

Каждая точка:

- вычисляет координаты `(x, y)`;
- получает весовые коэффициенты;
- вычисляет значение функции;
- добавляет вклад в интеграл.

Асимптотика:

```math
O(n_x \cdot n_y)
```

---

## 4. Схема распараллеливания

В STL-версии используется декомпозиция данных по строкам сетки.

Алгоритм работы:

1. Общее количество строк делится между workers.
2. Для каждого диапазона создаётся асинхронная задача.
3. Каждый worker вычисляет локальную сумму.
4. Главный поток объединяет результаты через `future.get()`.

Распараллеливание выполняется с помощью:

```cpp
std::async(std::launch::async, ...)
```

Каждый worker работает только со своим диапазоном строк,
поэтому состояния гонки отсутствуют.

---

## 5. Детали реализации

```cpp
futures.emplace_back(
    std::async(std::launch::async,
               ComputeChunk,
               std::cref(input_data),
               std::cref(steps),
               start_index,
               end_index));
```

*Пояснение к реализации:*  
Каждый worker вычисляет локальный partial sum,
после чего возвращает результат через `future`.

Общий accumulator во время вычислений отсутствует,
поэтому mutex и atomic-операции не требуются.

Количество workers фиксировано:

```cpp
int num_threads = 4;
```

Это ограничивает масштабируемость реализации
при запуске с большим числом потоков.

## 6. Проверка корректности

Корректность STL-реализации проверялась сравнением
с результатами SEQ-версии.

Использовались:

- функциональные тесты;
- performance-тесты;
- проверки при различных конфигурациях запуска.

Во всех тестах:

- результаты совпадали;
- детерминированность сохранялась;
- состояния гонки не наблюдались.

## 9. Выводы

STL-реализация обеспечила заметное ускорение
относительно последовательной версии.

Однако по сравнению с OpenMP и TBB наблюдаются
более высокие runtime-overhead.

Главным ограничением реализации является фиксированное
количество workers и более тяжёлый механизм управления задачами
через `std::async`.
