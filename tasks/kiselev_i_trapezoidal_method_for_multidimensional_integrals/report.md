## 1. Введение

В работе рассматривается задача численного вычисления двумерного определённого интеграла методом трапеций.

Проект содержит несколько реализаций алгоритма:

- последовательную реализацию (SEQ);
- реализацию на OpenMP (OMP);
- реализацию на oneTBB (TBB);
- реализацию на основе STL (`std::async`) (STL);
- гибридную MPI + OpenMP реализацию (ALL).

Цель работы — сравнение различных моделей параллелизма, способов декомпозиции вычислений
и характеристик масштабируемости на одинаковой вычислительной нагрузке.

В performance-тестах использовалась фиксированная сетка размера `1000 × 1000` на области `[0,1] × [0,1]`.

Интегрируемая функция:

```math
f(x, y) = x^2 + y^2
```

## 2. Постановка задачи

Требуется вычислить двумерный определённый интеграл методом трапеций.

Входные данные:

- левая граница интегрирования;
- правая граница интегрирования;
- размеры сетки дискретизации;
- тип функции;
- параметр `epsilon`.

Выходные данные:

- приближённое значение интеграла.

Вычислительная схема использует классические весовые коэффициенты метода трапеций.

## 6. Сравнительный анализ реализаций

### SEQ

Последовательная реализация используется как эталон для сравнения производительности остальных backend-ов.

Реализация не содержит накладных расходов синхронизации и демонстрирует полностью детерминированное выполнение.

### OMP

OpenMP-реализация показала хорошую масштабируемость за счёт эффективного
распараллеливания внешнего цикла и использования reduction.

Основная вычислительная область распараллеливается с помощью:

```cpp
#pragma omp parallel for reduction(+ : result)
```

Данная схема позволяет избежать гонок данных без использования mutex или critical section.

### TBB

Реализация на oneTBB использует `parallel_reduce` совместно с `blocked_range2d`.

Двумерная декомпозиция обеспечивает более равномерное распределение вычислительной нагрузки.

Используемый размер блока:

```cpp
32 × 32
```

позволяет уменьшить scheduler overhead при сохранении достаточного уровня параллелизма.

### STL

STL-реализация использует асинхронные задачи на основе `std::async`.

Каждый worker вычисляет независимую часть области интегрирования и возвращает локальную сумму.

По сравнению с OMP и TBB данная версия показывает более слабую масштабируемость из-за:

- большего runtime-overhead;
- менее оптимизированного scheduler-а;
- отсутствия work-stealing механизмов.

Также количество workers фиксировано внутри реализации:

```cpp
int num_threads = 4;
```

что ограничивает дальнейшее ускорение при увеличении числа потоков.

### ALL

Гибридная реализация сочетает:

- MPI-декомпозицию между процессами;
- OpenMP-параллелизм внутри процесса.

Строки сетки распределяются между MPI rank-ами, после чего OpenMP распараллеливает локальные вычисления.

Глобальное объединение результатов выполняется через:

```cpp
MPI_Allreduce(...)
```

Гибридная схема показала наилучшую производительность среди всех backend-ов.

## 8. Заключение

Проведённые эксперименты показывают, что метод трапеций хорошо подходит для data-parallel распараллеливания.

OpenMP-реализация обеспечила высокое ускорение при сравнительно небольшой сложности реализации.

TBB-версия продемонстрировала стабильную балансировку нагрузки и эффективную работу scheduler-а.

STL-реализация осталась работоспособной и переносимой, однако показала более высокие накладные расходы.

Гибридная MPI + OpenMP реализация продемонстрировала лучшую производительность
благодаря сочетанию межпроцессного и внутрипроцессного параллелизма.
