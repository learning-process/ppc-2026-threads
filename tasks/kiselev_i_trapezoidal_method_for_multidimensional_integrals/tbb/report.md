## 1. Контекст

Данная реализация использует task-based модель параллелизма библиотеки oneTBB.

Основная идея состоит в том,
чтобы передать runtime-системе задачу балансировки нагрузки между workers.

---

## 2. Постановка задачи

Требуется вычислить двумерный интеграл методом трапеций на прямоугольной сетке.

В качестве baseline используется SEQ-реализация.

Размер сетки performance-тестов:

```cpp
5000 × 5000
```

---

## 3. Базовый алгоритм

Вычисление интеграла выполняется методом трапеций по двум координатам.

Для каждой точки сетки:

- вычисляются координаты;
- вычисляются весовые коэффициенты;
- вычисляется значение функции;
- результат добавляется к локальной сумме.

Асимптотика:

```math
O(n_x \cdot n_y)
```

---

## 4. Схема распараллеливания

В TBB-реализации используется двумерная декомпозиция через `blocked_range2d`.

Основной вычислительный region:

```cpp
tbb::parallel_reduce(...)
```

Runtime oneTBB автоматически распределяет блоки между workers.

Используемый grain size:

```cpp
32 × 32
```

уменьшает scheduler-overhead и сохраняет достаточную степень параллелизма.

---

## 5. Детали реализации

```cpp
double result = tbb::parallel_reduce(
    Range2d(0, steps[0] + 1, 32,
            0, steps[1] + 1, 32),
    0.0,
    [&](const Range2d& r, double local_sum) {
      ...
      return local_sum;
    },
    std::plus<>());
```

*Пояснение к реализации:*  
Каждый worker получает собственный двумерный диапазон,
вычисляет локальную сумму и участвует в reduction.

Состояния гонки отсутствуют,
поскольку accumulation выполняется через локальные partial sums.

Балансировка нагрузки полностью делегируется scheduler-у oneTBB.

---

## 6. Проверка корректности

Корректность проверялась сравнением с последовательной реализацией.

Использовались:

- функциональные тесты Google Test;
- performance-тесты;
- проверки на различных размерах сетки.

Во всех случаях результаты совпадали.

## 7. Экспериментальная среда

- **Процессор:** Intel Core i5
- **Операционная система:** Linux / WSL2
- **Тип сборки:** Release
- **Компилятор:** GCC
- **Размер сетки:** 5000 × 5000
- **Количество workers:** 4 и 8

## 9. Выводы

TBB-реализация показала стабильную масштабируемость и хорошую балансировку вычислительной нагрузки.

Task-based модель оказалась эффективной для независимых вычислений на регулярной сетке.

Снижение эффективности при 8 workers связано с ростом scheduler-overhead и ограничениями memory subsystem.
