## 1. Контекст

ALL-реализация представляет собой гибридную схему,
сочетающую межпроцессное распараллеливание на основе MPI
и многопоточное распараллеливание с использованием OpenMP.

---

## 2. Постановка задачи

Необходимо вычислить двумерный интеграл методом трапеций на прямоугольной области.

В качестве baseline используется последовательная SEQ-реализация.

Размер сетки performance-тестов:

```cpp
5000 × 5000
```

---

## 3. Базовый алгоритм

Последовательная версия вычисляет интеграл по всем точкам двумерной сетки.

Гибридная версия сохраняет ту же вычислительную схему,
но распределяет вычисления между MPI rank-ами и OpenMP-потоками.

Асимптотика вычислительного ядра:

```math
O(n_x \cdot n_y)
```

---

## 4. Схема распараллеливания

Распараллеливание состоит из двух уровней.

### MPI-уровень

Строки сетки распределяются между MPI rank-ами:

```cpp
rows_per_proc = total_rows / size
```

Оставшиеся строки распределяются между первыми процессами.

Каждый rank вычисляет собственную локальную сумму.

### OpenMP-уровень

Внутри rank-а используется OpenMP:

```cpp
#pragma omp parallel for reduction(+ : local_sum)
```

Каждый поток обрабатывает собственный диапазон строк.

### Глобальное объединение

После завершения локальных вычислений результаты объединяются через:

```cpp
MPI_Allreduce(...)
```

---

## 5. Детали реализации

```cpp
#pragma omp parallel for reduction(+ : local_sum) schedule(static)
for (int i = begin; i < end; ++i) {
  ...
}
```

*Пояснение к реализации:*  
MPI отвечает за межпроцессное распределение строк сетки,
а OpenMP — за внутрипроцессный параллелизм.

Использование reduction устраняет состояния гонки
при накоплении локального результата.

Финальное объединение partial sums выполняется через
коллективную MPI-операцию `MPI_Allreduce`.

## 9. Выводы

Гибридная MPI + OpenMP реализация показала лучшую производительность среди всех реализаций.

Основной выигрыш достигается за счёт:

- распределения вычислений между MPI rank-ами;
- эффективного OpenMP reduction внутри процесса;
- высокой степени независимости вычислений.

Снижение эффективности при увеличении числа workers связано с overhead коллективных MPI-операций,
ограничениями memory subsystem.
