## 1. Контекст

Данная реализация переносит последовательный алгоритм вычисления двумерного интеграла
методом трапеций в многопоточную shared-memory среду с использованием OpenMP.

Основная цель распараллеливания заключается в ускорении обработки двумерной сетки
за счет независимого вычисления строк интегральной области различными потоками.

В качестве baseline используется SEQ-реализация.

---

## 2. Постановка задачи

Требуется вычислить двумерный определённый интеграл на прямоугольной области
методом трапеций.

Входные параметры:

- границы интегрирования;
- количество шагов по каждой координате;
- тип вычисляемой функции;
- параметр `epsilon`.

В performance-тестах использовалась фиксированная сетка:

```cpp
step_n_size = {5000, 5000}
```

Интегрируемая функция:

```math
f(x, y) = x^2 + y^2
```

---

## 3. Базовый алгоритм

SEQ-версия использует классическую двумерную схему метода трапеций.

Для каждой точки сетки:

1. вычисляются координаты `(x, y)`;
2. определяются весовые коэффициенты;
3. вычисляется значение функции;
4. вклад добавляется к общей сумме.

Итоговое значение интеграла вычисляется как:

```math
I \approx h_x h_y \sum_{i=0}^{n_x}\sum_{j=0}^{n_y} w_i w_j f(x_i, y_j)
```

Асимптотика алгоритма:

```math
O(n_x \cdot n_y)
```

---

## 4. Схема распараллеливания

В OMP-реализации применяется loop-level декомпозиция вычислений.

Внешний цикл по координате `x` распараллеливается между потоками:

```cpp
#pragma omp parallel for reduction(+ : result)
```

Каждый поток:

- получает собственный диапазон строк;
- независимо вычисляет локальный вклад;
- участвует в reduction-объединении результата.

Использование reduction позволяет избежать состояний гонки при накоплении
общей суммы интеграла.

Параллельная область завершается неявным барьером OpenMP.

---

## 5. Детали реализации

```cpp
#pragma omp parallel for reduction(+ : result) default(none) \
shared(nx, ny, hx, hy, x0, y0, func_type)
for (int i = 0; i <= nx; i++) {
  for (int j = 0; j <= ny; j++) {
    result += wx * wy * FunctionTypeChoose(func_type, x, y);
  }
}
```

*Пояснение к реализации:*  
Переменные геометрии сетки объявлены как `shared`,
а накопление результата выполняется через `reduction(+ : result)`.

Гонки данных отсутствуют, поскольку каждый поток вычисляет собственный
локальный accumulator.

Явные mutex, critical section и atomic-операции не используются.

## 6. Проверка корректности

Проверка корректности выполнялась сравнением результатов OMP-версии
с последовательной реализацией.

Использовались:

- функциональные тесты Google Test;
- performance-тесты инфраструктуры проекта;
- проверки при различных конфигурациях потоков.

Для всех тестов:

- расхождения результатов не обнаружены;
- вычисления оставались детерминированными;
- выходные данные были конечными (`std::isfinite`).

### Режим pipeline

| Потоки (p) | Время (сек) | Ускорение (S) |

*Примечание:*  
Высокое ускорение в `task_run` связано с тем,
что вычислительная нагрузка практически полностью состоит
из независимых арифметических операций.

## 9. Выводы

OpenMP-реализация продемонстрировала хорошую масштабируемость
и низкие накладные расходы синхронизации.

Использование reduction оказалось эффективным решением
для вычисления интегральной суммы.

При увеличении числа потоков до 8 ускорение продолжает расти,
однако эффективность постепенно снижается из-за ограничений
подсистемы памяти и накладных расходов runtime OpenMP.
