## 1. Введение

Финальный этап лабораторной работы — создание гибридной (ALL) реализации, ориентированной на кластерные
системы. Она сочетает два уровня параллелизма:

1. **Распределенная память (MPI):** передача данных по сети между независимыми процессами-узлами.
2. **Общая память (OpenMP):** утилизация всех ядер внутри одного физического узла для ускорения локальной
   сортировки.

Такая архитектура позволяет преодолеть физические ограничения одной материнской платы и масштабировать
сортировку на вычислительные кластеры.

## 2. Постановка задачи

- Вход: Неотсортированный массив `std::vector<int>` на главном узле (Rank 0).
- Выход: Отсортированный массив на главном узле.
- Ограничения: Минимизация дорогостоящих сетевых обменов. Классический алгоритм Бэтчера, где узлы обмениваются
  одиночными элементами, убьет производительность сети. Требуется переход к крупноблочному (block-wise)
  слиянию.

## 3. Схема распараллеливания и распределения данных

Алгоритм разбит на три фундаментальных этапа:

1. **Scatter (Распределение):** Главный процесс вычисляет размер блока `chunk_size` так, чтобы данные делились
   на `world_size` частей. Входной массив расширяется `INT_MAX` элементами до размера `chunk_size *
   world_size` и рассылается всем узлам через `MPI_Scatter`.
2. **Гибридная локальная сортировка:** Каждый процесс внутри себя вызывает `OMPLocalSort`. Этот метод
   разбивает локальный блок на подблоки по количеству доступных OpenMP-потоков, параллельно сортирует их (Data
   Parallelism), а затем делает финальную досортировку локального куска.
3. **Глобальная сеть Бэтчера (Compare-Split):** Сеть слияния выполняется не над элементами, а над *процессами
   (MPI Ranks)*. На каждом шаге Бэтчера два процесса обмениваются своими отсортированными блоками целиком,
   локально делают `std::ranges::merge` сливая их в двойной массив. Затем процесс с меньшим рангом оставляет
   себе "нижнюю" половину, а с большим — "верхнюю".

## 4. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.

Главная гордость этой реализации — функция `MpiCompareAndSwap`, которая реализует операцию слияния-разделения
(Compare-Split):

```cpp
void MpiCompareAndSwap(std::vector<int> &local_arr, int neighbor, bool keep_low_half) {
  int size = static_cast<int>(local_arr.size());
  std::vector<int> neighbor_arr(size);

  // Синхронный обмен блоками между узлами сети
  MPI_Sendrecv(local_arr.data(), size, MPI_INT, neighbor, 0, 
               neighbor_arr.data(), size, MPI_INT, neighbor, 0,
               MPI_COMM_WORLD, MPI_STATUS_IGNORE);

  // Слияние двух отсортированных массивов за O(N)
  std::vector<int> merged_arr(size * 2);
  std::ranges::merge(local_arr, neighbor_arr, merged_arr.begin());

  // Разделение результата
  if (keep_low_half) {
    std::copy(merged_arr.begin(), merged_arr.begin() + size, local_arr.begin());
  } else {
    std::copy(merged_arr.begin() + size, merged_arr.end(), local_arr.begin());
  }
}
```

Внутри `RunImpl` после завершения глобальной сортировки вызывается `MPI_Gather`, который собирает блоки со
всех узлов обратно в `gather_buffer` на Rank 0, где происходит усечение до исходного `total_n`.

## 5. Проверка корректности

Все тесты функционального каркаса пройдены (`std::ranges::is_sorted`). Код надежно обрабатывает краевые
случаи:

- Если `total_n <= 1`, происходит ранний возврат без вызова сетевых примитивов.
- Механизм padding'а гарантирует, что `MPI_Scatter` и `MPI_Gather` работают со строго одинаковыми размерами
  буферов на всех узлах, предотвращая дедлоки (deadlocks) и ошибки сегментации в MPI.

### Научный анализ архитектуры и узких мест

Гибридная версия показала выдающееся стартовое ускорение (~5x), но график выявил ярко выраженную деградацию
производительности при переходе с 2 на 4 и 8 потоков. Это явление прекрасно объясняется двумя факторами
архитектуры кода:

1. **Сила алгоритмической локальности (Блочность):** Базовое ускорение в 5 раз достигнуто не столько за счет
   многопоточности, сколько за счет перехода к блочному слиянию (`std::ranges::merge` в `MpiCompareAndSwap`).
   Это максимально "дружелюбно" к кэшу (Cache-friendly) по сравнению с базовой версией.
2. **Закон Амдала и бутылочное горлышко в `OMPLocalSort`:** Рассмотрим реализацию гибридной локальной
   сортировки:

   ```cpp
   #pragma omp parallel for ...
   for (int i = 0; i < num_threads; ++i) { /* сортировка подблоков */ }
   
   // Финальная сборка локального блока
   OptimizedHoareSort(arr, 0, size - 1); 
   ```

   Разбив массив на независимые куски и отсортировав их в OMP, код в конце запускает *последовательный*
   алгоритм Хоара над всем массивом целиком, чтобы досортировать его. Алгоритм Хоара крайне неэффективен на
   "кусочно-отсортированных" данных. Чем больше потоков (и кусков), тем сильнее нарушается глобальный порядок,
   и тем больше работы делает финальный (и строго однопоточный) `OptimizedHoareSort`. На 4 и 8 потоках эта
   последовательная часть начинает доминировать во времени выполнения, нивелируя пользу от параллелизма OpenMP
   и накладываясь на сетевой overhead MPI.

## 8. Выводы

Гибридная ALL-версия успешно доказала принципиальную возможность распределения сортировки Бэтчера на
кластерные архитектуры. Переход от обмена элементами к блочной операции `Compare-Split` (передача массивов
целиком) является единственно верным подходом для MPI-систем, защищающим сеть от перегрузки.

Пиковое время работы на 2 потоках составило **~0.0498 с** (ускорение более чем в 5 раз). Однако для достижения
идеального масштабирования на большом числе ядер (8 и более) алгоритм требует доработки: финальный шаг в
`OMPLocalSort` следует заменить с последовательной быстрой сортировки на классическое параллельное k-путевое
слияние (k-way merge) или дерево слияний, что окончательно устранит последовательное узкое место.
