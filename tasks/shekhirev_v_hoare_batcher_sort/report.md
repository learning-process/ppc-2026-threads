## 1. Введение

Быстрая сортировка с последующим слиянием Бэтчера — это отличный полигон для изучения параллельных архитектур.
Задача состоит из двух фундаментально разных фаз. Первая фаза (локальная сортировка блоков) представляет собой
идеальный Data Parallelism: независимые вычисления, которые превосходно масштабируются. Вторая фаза (сеть
слияния Бэтчера) обладает жесткой математической топологией с частыми барьерами синхронизации (Fine-grained
parallelism).

В данной работе алгоритм реализован с применением пяти различных технологий (SEQ, OMP, TBB, STL, ALL). Главная
цель исследования — оценить, как легковесные внутриузловые потоки и гибридные распределенные модели
справляются с накладными расходами на синхронизацию мелкогранулярных задач.

## 2. Единая постановка задачи

- **Вход:** Неотсортированный массив `std::vector<int>`.
- **Выход:** Тот же массив, отсортированный по неубыванию.
- **Ограничения:** Алгоритм слияния требует размерностей, кратных степеням двойки. Для произвольных размеров
  реализован механизм выравнивания (padding) фиктивными элементами `INT_MAX`. Использование рекурсии запрещено
  для исключения `Stack Overflow`.
- **Критерий корректности:** Строгое выполнение предиката `std::ranges::is_sorted` для выходного массива.
  Результат каждой параллельной реализации должен размерно и покомпонентно совпадать с результатами работы
  последовательного (SEQ) эталона.

## 3. Единая методика эксперимента

Все замеры производились в единой изолированной среде с использованием функционального и нагрузочного (Perf)
каркаса курса.

- **Hardware/OS:** Контейнер `ghcr.io/learning-process/ppc-ubuntu:1.1` (Ubuntu 24.04 LTS).
- **Toolchain:** GCC 13.3.0, CMake 3.28.3, OpenMPI, TBB. Сборка в профиле `Release`.
- **Данные:** Вектор из **1000000 элементов**.
- **Паттерн:** Обратно отсортированный массив (классический стресс-кейс для сортировки Хоара, генерирующий
  максимум перестановок).
- **Управление параллелизмом:** Осуществлялось через переменные окружения `PPC_NUM_THREADS` и
  `OMP_NUM_THREADS`. Для гибридной версии применялась настройка `mpirun`.
- **Агрегация метрик:** Фреймворк `ppc_perf_tests` автоматически выполнял многократные прогоны. В таблицу
  результатов вынесено чистое алгоритмическое время фазы вычислений (`task_run`).
- **Формулы:**
  - $Speedup$ = $T_{seq} / T_{backend}$
  - $Efficiency$ = $Speedup / W$ (где $W$ — заданное число потоков).

## 6. Анализ результатов и интерпретация аномалий

Результаты замеров наглядно иллюстрируют фундаментальное правило: **структура данных и локальность кэша влияют
на скорость сильнее, чем сырое добавление ядер.**

1. **Алгоритмический скачок (OMP и TBB):** На одном потоке обе технологии показали аномальное ускорение ~2.8x
   относительно SEQ. Это произошло не из-за магии потоков, а из-за изменения базового алгоритма: в SEQ сеть
   Бэтчера использовала `std::vector` как стек состояний (нагружая аллокатор кучи), а в OMP/TBB слияние было
   переписано на "плоские" вложенные циклы `for`. Отсутствие работы с динамической памятью дало колоссальный
   прирост.
2. **Проблема мелкогранулярности:** При масштабировании (2, 4, 8 потоков) OMP и TBB показали слабый рост
   (ускорение остановилось на ~2.9x). Внутренний цикл Бэтчера — это буквально одно сравнение `if (a > b)
   std::swap`. Накладные расходы рантайма на создание/кражу задач и барьеры синхронизации между шагами сети
   съедают профит распараллеливания.
3. **Победа Cache-Friendly архитектуры (STL):** Версия `std::thread` стала абсолютным рекордсменом (ускорение
   5.5x). Секрет кроется в переходе от поэлементного слияния к *блочному* (Block-wise Merge). Вместо того
   чтобы прыгать по всему массиву, ломая кэш процессора, STL-версия сливает непрерывные куски памяти.
   *Примечание: STL-реализация жестко запрашивает `hardware_concurrency` и игнорирует тестовые лимиты
   `PPC_NUM_THREADS`, поэтому она стабильно отрабатывает за ~0.046 с во всех запусках.*
4. **Гибридная модель (ALL):** Показала мощный старт на 2 процессах (5.15x) благодаря архитектуре
   `Compare-Split` (MPI-узлы обмениваются сразу целыми массивами, а не элементами). Однако при масштабировании
   до 8 воркеров произошло падение до 3.89x. Причина — классическое бутылочное горлышко (Закон Амдала). В
   конце гибридной сортировки главный узел вызывает последовательный `OptimizedHoareSort` для досортировки
   собранных "кусков", что на больших мощностях сводит на нет все параллельные усилия.

## 7. Репродуцируемость

Сборка проекта:

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

Замер масштабирования потоковых реализаций (на примере TBB):

```bash
for t in 1 2 4 8; do
  PPC_NUM_THREADS=$t OMP_NUM_THREADS=$t ./build/bin/ppc_perf_tests \
    --gtest_filter="*Shekhirev*tbb_enabled*"
done
```

Замер гибридной MPI-реализации:

```bash
PPC_NUM_THREADS=2 OMP_NUM_THREADS=2 OMPI_ALLOW_RUN_AS_ROOT=1 OMPI_ALLOW_RUN_AS_ROOT_CONFIRM=1 \
  mpirun --oversubscribe -np 2 ./build/bin/ppc_perf_tests \
  --gtest_filter="*Shekhirev*all_enabled*"
```

## 8. Заключение

Выполненная лабораторная работа позволила провести глубокое кросс-технологическое исследование. Главный
инженерный вывод: **для сети сортировки Бэтчера поэлементное распараллеливание является тупиковым путем.**

Высокоуровневые фреймворки (OMP, TBB) не могут эффективно распараллелить миллиарды микро-задач (свопов) из-за
накладных расходов планировщика. Реальное масштабирование и экстремальную производительность (как в STL и ALL
версиях) дает только укрупнение задач: переход к слиянию непрерывных блоков данных (chunking) и эффективная
утилизация кэша L1/L2 процессора.

Для вывода гибридной ALL-версии на идеальное линейное масштабирование в будущем необходимо заменить финальный
последовательный проход Хоара на параллельное K-путевое слияние (K-way tree merge).
