## 1. Введение

В отличие от OpenMP, который управляет потоками через директивы компилятора, Intel Threading Building Blocks
(TBB) предоставляет высокоуровневые C++ абстракции и опирается на концепцию задач (tasks). За балансировку
нагрузки отвечает встроенный планировщик TBB, использующий алгоритм "кражи задач" (task-stealing). Это
позволяет более гибко утилизировать ядра процессора, перераспределяя работу, если один из потоков освободился
раньше других.

## 2. Постановка задачи

- Вход: Неотсортированный массив `std::vector<int>`.
- Выход: Тот же массив, отсортированный по неубыванию.
- Ограничения: Алгоритм слияния Бэтчера требует, чтобы число сортируемых элементов (и блоков, на которые они
  разбиваются) было строгой степенью двойки. Входные данные произвольного размера требуют выравнивания
  фиктивными элементами `INT_MAX`.

## 3. Схема распараллеливания

TBB-реализация сохраняет двухфазную структуру, но использует примитив `tbb::parallel_for` для
распараллеливания циклов:

1. **Фаза локальной сортировки (Разделяй и властвуй):** Массив логически делится на `p_threads` блоков (где
   `p_threads` — ближайшая степень двойки, не превышающая лимит потоков). Запуск локальной итеративной
   сортировки Хоара для каждого блока обернут в `tbb::parallel_for`. Планировщик сам распределяет эти крупные
   куски работы между доступными воркерами (workers).

2. **Фаза слияния Бэтчера:** Внешние шаги алгоритма (`step_p`, `step_k`) жестко зависят друг от друга, поэтому
   они остаются последовательными. Параллелизм извлекается из внутреннего цикла (сравнение независимых пар). В
   TBB-версии этот этап вынесен в отдельную функцию `ProcessBatcherStep`, внутри которой `tbb::parallel_for`
   итерируется по пространству `blocked_range`.

## 4. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Один из важнейших аспектов вашей реализации — это явное управление пулом потоков TBB. В отличие от старых
версий библиотеки, здесь используется современный механизм `tbb::global_control`:

```cpp
// File: tbb/src/ops_tbb.cpp
int num_threads = ppc::util::GetNumThreads();
tbb::global_control global_limit(tbb::global_control::max_allowed_parallelism, num_threads);
```

Этот код гарантирует, что планировщик TBB не породит лишних аппаратных потоков и честно отработает на заданном
лимите (1, 2, 4 или 8).

**Распараллеливание локальной сортировки:** Вместо прагм используется лямбда-выражение, принимающее на вход
`tbb::blocked_range`:

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, p_threads), [&](const tbb::blocked_range<int> &r) {
  for (int i = r.begin(); i != r.end(); ++i) {
    HoareSort(a, i * chunk_size, ((i + 1) * chunk_size) - 1);
  }
});
```

Размер чанка для TBB здесь не задан явно (grainsize), поэтому используется `auto_partitioner`, который сам
решает, как глубоко дробить задачу, опираясь на текущую загрузку пула.

## 5. Проверка корректности

Все тесты из функционального набора успешно пройдены (проверка результата через `std::ranges::is_sorted`).
Алгоритм корректно обрабатывает пустые массивы, единичные элементы и произвольные размеры (путем дополнения
`INT_MAX`). Внутренний цикл Бэтчера, вынесенный в лямбду `ProcessBatcherStep`, не порождает состояний гонки
(data races), так как математика сети гарантирует уникальность индексов `idx1` и `idx2` для каждого
параллельного шага.

## 7. Результаты

Ниже приведена таблица агрегированного времени работы для TBB-реализации (режим `task_run`) на массиве в 1
миллион элементов.

| workers | time, s | speedup | efficiency |
| :--- | :--- | :--- | :--- |
| 1 | 0.0920 | 2.79 | > 100% |

Высокая эффективность на 1 и 2 потоках относительно SEQ объясняется изменением алгоритмической базы слияния:
отказом от вектора-стека в пользу плоских математических циклов.

### Анализ масштабирования и поведения планировщика

На массиве в 1000000 элементов TBB демонстрирует весьма специфичный профиль производительности:

1. **Алгоритмический "прыжок" на 1 потоке:** Как и в случае с OMP, отказ от динамических аллокаций памяти
   (работа со стеком `std::vector`) в фазе слияния Бэтчера дает огромное базовое ускорение. TBB-код на одном
   ядре работает в 2.8 раза быстрее исходного SEQ.
2. **Накладные расходы балансировщика (1 -> 2 потока):** При переходе на 2 потока время ухудшается (0.0920 ->
   0.1049 с). Планировщик TBB (`auto_partitioner`) тратит ресурсы на разбиение пространства задач
   (`blocked_range`) и синхронизацию воркеров. Поскольку внутренние циклы Бэтчера очень мелкие (буквально одно
   сравнение и `swap`), накладные расходы task-stealing механизма оказываются дороже, чем сама полезная
   работа.
3. **Восстановление баланса (4 и 8 потоков):** При дальнейшем увеличении числа воркеров параллелизм начинает
   побеждать накладные расходы. Время планомерно улучшается (0.1049 -> 0.0925 -> 0.0913), и на 8 ядрах
   многопоточность окончательно компенсирует все системные задержки, слегка обходя оптимизированный
   однопоточный прогон.

## 8. Выводы

Реализация на базе Threading Building Blocks позволила создать красивый и декларативный код без использования
C-style макросов. Использование `tbb::global_control` обеспечило корректную настройку пула потоков в тестовом
каркасе.

Алгоритм показал превосходную пиковую производительность (~0.0913 с на миллионе элементов, суммарно в 2.8 раза
быстрее эталонного SEQ). Однако результаты замеров подчеркивают слабое место сети Бэтчера — экстремально
мелкую гранулярность (fine-grained parallelism). Частые неявные барьеры между шагами сети мешают планировщику
TBB продемонстрировать идеальное линейное масштабирование. В реальных production-системах для подобных задач
рекомендуется вручную задавать размер чанка (`grainsize`) в `blocked_range`, чтобы искусственно укрупнить
задачи и снизить нагрузку на балансировщик.
