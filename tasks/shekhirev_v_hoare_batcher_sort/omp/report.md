## 1. Введение

Последовательная версия (SEQ) стала эталоном алгоритмической корректности. Цель данного этапа — распараллелить
алгоритм с помощью директив стандарта OpenMP. OpenMP позволяет распределять итерации циклов по потокам в
системах с общей памятью, минимально меняя основную бизнес-логику кода. В этом отчете исследуется, как
алгоритм Бэтчера ложится на потоковую модель и как размер данных влияет на эффективность параллелизма.

## 2. Постановка задачи

- Вход: Неотсортированный массив `std::vector<int>`.
- Выход: Тот же массив, отсортированный по неубыванию.
- Главная алгоритмическая сложность: Сеть сортировки Бэтчера исторически требует, чтобы размер массива был
  степенью двойки. При распараллеливании количество блоков (чанков), на которые бьется массив, также должно
  быть степенью двойки, иначе математика слияния нарушится.

## 3. Схема распараллеливания

Алгоритм логически разделен на две фазы, к каждой из которых применен свой подход:

1. **Независимая локальная сортировка (Data Parallelism):** Расширенный до степени двойки массив логически
   делится на `P` равных частей, где `P` — количество рабочих потоков (которое принудительно округляется вниз
   до ближайшей степени двойки). Каждый поток берет свой "чанк" и сортирует его локально алгоритмом Хоара.
   Потоки работают независимо, без разделения данных и риска состояния гонки (race conditions).

2. **Глобальное слияние (Loop Parallelism в сети Бэтчера):** Классическая сеть Бэтчера переписана в виде
   вложенных итеративных циклов. Внешние циклы (`step_p` и `step_k`) формируют шаги сети и зависят друг от
   друга — они остаются последовательными. Внутренний цикл перебирает независимые пары для сравнения (`idx1`,
   `idx2`) — именно он распараллеливается между потоками.

## 4. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Важнейший момент реализации — безопасный расчет количества потоков `p_threads`. Мы запрашиваем максимум
потоков у OMP, но для корректного слияния Бэтчера оставляем только степень двойки:

```cpp
// File: omp/src/ops_omp.cpp
int num_threads = omp_get_max_threads();
if (num_threads <= 0) num_threads = 1;

int p_threads = 1;
while ((p_threads * 2) <= num_threads && (p_threads * 2) <= n_pow2) {
  p_threads *= 2;
}
int chunk_size = n_pow2 / p_threads;
```

**Распараллеливание локальной сортировки:** Цикл раскидывает независимые вызовы функции по потокам.

```cpp
#pragma omp parallel for default(none) shared(a, p_threads, chunk_size)
for (int i = 0; i < p_threads; i++) {
  HoareSort(a, i * chunk_size, ((i + 1) * chunk_size) - 1);
}
```

**Распараллеливание слияния:** Директива применяется ко внутреннему циклу `for (int step = 0; step <
total_pairs; ++step)`.

```cpp
#pragma omp parallel for default(none) shared(a, step_p, step_k, start, total_pairs)
for (int step = 0; step < total_pairs; ++step) {
  // ... вычисление индексов idx1 и idx2 ...
  if ((idx1 / (step_p * 2)) == (idx2 / (step_p * 2)) && a[idx1] > a[idx2]) {
    std::swap(a[idx1], a[idx2]);
  }
}
```

Ключевое отличие от SEQ-версии: в OMP мы отказались от эмуляции стека с помощью `std::vector` в фазе слияния и
перешли к "плоским" итеративным циклам, вычисляя индексы аналитически. Это кардинально снижает нагрузку на
менеджер памяти.

**Анализ параметров OpenMP (`shared/private/reduction/schedule`):**

- **`shared`**: Массив данных `a`, конфигурация пула `p_threads` и параметры сети Бэтчера (`step_p`, `step_k`)
  объявляются общими. Разделение доступа к памяти гарантируется математикой алгоритма (каждый поток трогает
  строго только свои непересекающиеся индексы).
- **`private`**: Переменные итераторов циклов (`i`, `step`) приватизируются потоками автоматически. Явное
  указание `default(none)` заставило компилятор строго проконтролировать все внешние зависимости.
- **`reduction`**: Не используется. В отличие от задач агрегации (поиск минимума или суммы), наша сортировка
  производит операции обмена (`std::swap`) "на месте" в независимых ячейках, поэтому редукция глобальных
  состояний не требуется.
- **`schedule`**: Явно не задан, поэтому используется `schedule(static)` по умолчанию. Для сети Бэтчера это
  оптимальный выбор: каждая итерация внутреннего цикла содержит строго одинаковый микро-объем работы
  (вычисление индексов и один `if` со `swap`). Использование `dynamic` или `guided` здесь добавило бы
  фатальный оверхед на балансировку рантайма.

## 5. Проверка корректности

Все тесты из функционального набора пройдены успешно (проверка через `std::ranges::is_sorted`). Логика успешно
справляется как с "идеальными" размерами, так и с нечетными. Это подтверждает, что фиктивные элементы
`INT_MAX` не ломают конкурентные проходы, а неявные барьеры OpenMP в конце `#pragma omp parallel for`
корректно синхронизируют потоки между шагами `step_k`.

### Научный анализ результатов

Переход на массив в 1 миллион элементов ярко подсветил особенности работы OpenMP с мелкогранулярными задачами:

1. **Алгоритмическое превосходство (1 поток):** OMP-версия даже на одном потоке работает почти в 3 раза
   быстрее эталонной SEQ (0.0889 с против 0.2563 с). Это связано с отказом от использования `std::vector` в
   качестве стека во время слияния Бэтчера. "Плоские" циклы с аналитическим расчетом индексов не требуют
   обращений к куче (heap) и отлично кэшируются процессором.

2. **"Провал" на 2 потоках и восстановление масштабируемости (2 -> 8 потоков):** При переходе от 1 к 2 потокам
   время ухудшается (0.0889 с -> 0.1426 с). Это классический штраф за создание (fork) и синхронизацию (join)
   потоков на каждой итерации сети Бэтчера. Однако, в отличие от меньших массивов, на 1 000 000 элементов
   полезная вычислительная нагрузка в циклах становится достаточно большой. Поэтому при дальнейшем добавлении
   ядер (4 и 8) алгоритм начинает уверенно масштабироваться: время планомерно падает (0.1426 -> 0.1222 ->
   0.0883). На 8 потоках многопоточность окончательно компенсирует накладные расходы рантайма и немного
   обходит супер-оптимизированный однопоточный вариант.

## 8. Выводы

Внедрение OpenMP позволило параллелизовать алгоритм, при этом главной победой стало сопутствующее упрощение
логики слияния до "плоских" итеративных циклов, что само по себе дало ускорение в 2.8 раза.

Тесты на массиве из 1000000 элементов показали, что алгоритм Бэтчера очень чувствителен к накладным расходам
синхронизации. Тем не менее, при достаточном объеме данных (от 1 млн) мы видим явную тенденцию к физическому
ускорению при увеличении числа ядер (от 2 до 8). Для дальнейшей оптимизации OMP-версии имеет смысл внедрить
ручное укрупнение шагов (chunking) во внутреннем цикле Бэтчера, чтобы снизить частоту неявных барьеров
синхронизации потоков.
