## 1. Контекст

OpenMP-версия параллелизует единственный внешний цикл по узлам решётки с помощью одной
прагмы-директивы. Подход идеален для многоядерных однопроцессорных машин: минимальные
изменения кода при значительном ускорении. Последовательная версия описана в `seq/report.md`,
альтернативные параллельные подходы — в `stl/report.md`, `tbb/report.md`, `all/report.md`.

## 2. Постановка задачи

### Входные данные

Структура `SimpsonInput` из `common/include/common.hpp`:

- `lower_bounds` — вектор нижних пределов интегрирования;
- `upper_bounds` — вектор верхних пределов интегрирования;
- `n_steps` — вектор числа шагов (чётные, положительные);
- `func` — подынтегральная функция `std::function<double(const std::vector<double>&)>`.

### Выходные данные

Значение интеграла типа `double`.

### Ограничения

- Размеры всех трёх векторов должны совпадать и быть ненулевыми;
- `lower_bounds[i] <= upper_bounds[i]`;
- `n_steps[i] > 0` и `n_steps[i] % 2 == 0`;
- `func` должна быть непустой.

## 3. Базовый алгоритм

Математика идентична SEQ-версии: составной многомерный метод Симпсона на равномерной
декартовой сетке с тензорным произведением весов.

`total_points = ∏(n_steps[dim] + 1)`, узлы нумеруются линейно. Многомерный индекс
восстанавливается через смешанную систему счисления. Финальный масштабирующий фактор:
`∏(h[dim] / 3.0)`.

## 4. Схема параллелизма

### Директива

Единственный внешний цикл из `ComputeSimpsonMultiDim()` помечается прагмой:

```cpp
#pragma omp parallel for default(none) shared(input, total_points, num_dims, h) reduction(+ : sum)
for (size_t point_idx = 0; point_idx < total_points; ++point_idx) {
    // декодирование point_idx -> indices -> point
    // вычисление тензорного веса
    // sum += weight * input.func(point);
}
```

### Ключевые параметры

- `default(none)` — запрет неявного захвата переменных, все переменные указаны явно;
- `shared(...)` — `input`, `total_points`, `num_dims`, `h` разделяются между потоками (только чтение);
- `reduction(+ : sum)` — каждый поток накапливает локальную сумму, итоговое слияние выполняет
  OpenMP автоматически без критических секций;
- Планирование: неявное `schedule(static)` — равномерное распределение итераций по потокам.

### Размещение переменных

`std::vector<int> indices(num_dims)` и `std::vector<double> point(num_dims)` объявлены внутри
тела цикла — каждая итерация выделяет и освобождает их самостоятельно. Это потокобезопасно, но
создаёт накладные расходы на аллокацию в горячем пути. Альтернативы без per-iteration аллокации
реализованы в STL и TBB версиях.

### Методы класса `ZyuzinNSimpsonOMP`

`ValidationImpl()` — идентична SEQ: проверка размеров, пределов, чётности шагов, непустоты `func`.

## 6. Проверка корректности

Те же 12 функциональных тестов с допуском `1e-3`, что и в SEQ. Корректность гарантируется
механизмом `reduction`: OpenMP выполняет финальное суммирование частичных результатов
детерминированно для данного набора данных.

## 9. Выводы

OpenMP-реализация — минимально отличающийся от SEQ вариант: добавлена одна директива
`#pragma omp parallel for`. Корректна на всех 12 функциональных тестах.

Сильные стороны:

- Минимальный объём изменений кода относительно SEQ;
- `reduction(+: sum)` обеспечивает корректность без явной синхронизации;
- Статическое планирование эффективно для равномерной нагрузки (все итерации идентичны по весу).

Слабые стороны:

- Per-iteration выделение двух `std::vector` создаёт нагрузку на аллокатор при больших `total_points`;
- Статическое планирование не адаптируется к нагрузке (хотя для данной задачи она равномерна).
