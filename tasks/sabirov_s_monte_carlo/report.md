## 1. Введение

Метод Монте-Карло — стохастический способ оценки многомерных интегралов, в котором значение
интеграла приближается выборочной средней значений подынтегральной функции по точкам,
независимо и равномерно распределённым в области интегрирования. Сходимость метода имеет порядок
`O(1 / sqrt(N))` и не зависит от размерности задачи `d`, что делает его особенно удобным для
областей высокой размерности, где классические квадратурные формулы становятся
вычислительно неподъёмными.

В данной лабораторной работе реализованы пять вариантов одного и того же алгоритма (SEQ, OMP, TBB,
STL и ALL = MPI + OpenMP). Цель — сопоставить, как разные модели параллелизма ведут себя на
задаче, в которой выборки полностью независимы, а коммуникация сводится к ассоциативной редукции
одного числа.

## 2. Постановка задачи

- **Вход:** структура `MCInput`:
  - `lower`, `upper` — границы интегрирования по осям (`lower[i] < upper[i]`),
  - `num_samples` — число выборок `N > 0`,
  - `func_type` — тип подынтегральной функции (`FuncType`).
- **Выход:** действительное число — оценка интеграла.
- **Поддерживаемые функции (`FuncType`):**
  `kLinear`, `kSumCubes`, `kCosProduct`, `kExpNeg`, `kMixedPoly`, `kSinSum`, `kSqrtSum`,
  `kQuarticSum` — суммы степеней, тригонометрия, экспонента, корни.
- **Ограничения и крайние случаи:** размерность `1 ≤ dim ≤ 10`,
  `lower.size() == upper.size()`, для всех `i` `lower[i] < upper[i]`, `num_samples > 0`,
  `func_type` лежит в указанном диапазоне.
- **Критерий корректности:** результат должен попадать в интервал
  `|I_estimate − I_exact| ≤ 5.0 / sqrt(N)`. Точные значения интегралов для каждого `FuncType`
  собраны в общем хелпере `common/include/common.hpp` и используются функциональными тестами.

Математическая модель оценки:

> `I ≈ V · (1 / N) · Σ_{k=1..N} f(x^(k))`,
> где `V = Π (b_i − a_i)` — объём гиперпрямоугольника `Ω`, а `x^(k)` —
> независимые точки, равномерно распределённые в `Ω`.

## 3. Единая методика эксперимента

- **CPU:** AMD Ryzen 9 9950X3D (16 ядер / 32 потоков)
- **RAM:** 96 ГБ DDR5
- **OS:** Windows 11 Pro 26200.8457
- **Компилятор:** MSVC 19.50
- **Размер задачи:** `kLinear` на `[0, 1]^3`, `N = 10 000 000`
- **Команды сборки (из корня репозитория):**
  - Конфигурация: `cmake -S . -B build`
  - Перф-тесты: `cmake --build build --target ppc_perf_tests --config Release --parallel`
  - Функциональные тесты: `cmake --build build --target ppc_func_tests --config Release --parallel`
- **Команды запуска (из `build/bin/`):**
  - Обычный запуск: `.\ppc_perf_tests --gtest_filter="*sabirov_s_monte_carlo*"`
  - С потоками: `$env:PPC_NUM_THREADS="4"; .\ppc_perf_tests --gtest_filter="*sabirov_s_monte_carlo*"`
  - С MPI (для ALL): `$env:PPC_NUM_THREADS="4"; mpiexec -n 4 .\ppc_perf_tests --gtest_filter="*sabirov_s_monte_carlo*all*"`
  - Во всех командах подставляется конкретное число потоков или процессов (вместо `4`).
    Угловые скобки в значении `PPC_NUM_THREADS` не ставятся: иначе `libenvpp` не распарсит
    его в `int`, и `GetNumThreads()` вернёт `1`.
- **Формулы:** Ускорение `S = T_seq / T_par`. Эффективность `E = S / W`, где `W` — число рабочих
  (потоков для OMP/TBB/STL, `P · T` для ALL).
- **Измерения:** Приводятся одиночные показатели для режимов `task_run` и `task_pipeline`
  (5 итераций внутри каждого, усредняется фреймворком).

## 4. Сводка корректности

Все реализации (SEQ, OMP, TBB, STL, ALL) проходят 13 сценариев функционального теста с покрытием
всех восьми подынтегральных функций в 1D/2D/3D и допуском `5.0 / sqrt(N)`. Дополнительно
проверяются граничные случаи валидации входа: вырожденные интервалы, несовпадение размерностей,
непозитивный `num_samples`, недопустимый `func_type`, превышение лимита по размерности.

## 5. Краткое описание вариантов

| Вариант | Идея распределения работы | Где задаётся число рабочих |
| --- | --- | --- |
| SEQ | Один поток исполняет цикл по `N` выборкам и итоговое масштабирование `V / N`. | Не применимо |
| OMP | `parallel + for + reduction(+ : sum)` с `schedule(static)`; локальные ГСЧ у потоков. | `omp_set_num_threads(GetNumThreads())` в начале `RunImpl` |
| TBB | `tbb::parallel_reduce` по `blocked_range<int>(0, N)` с локальной выборочной суммой. | `oneapi::tbb::global_control(max_allowed_parallelism, GetNumThreads())` |
| STL | Диапазон `[0, N)` режется на `num_threads` равных блоков, каждому — свой `std::thread`. | `GetNumThreads()` |
| ALL | `MPI_Scatterv` распределяет число выборок по процессам; локально — OMP-редукция; `MPI_Allreduce(MPI_SUM)` собирает сумму. | `mpiexec -n P` (P — число процессов) и `omp_set_num_threads(GetNumThreads())` |

## 7. Интерпретация различий

- **SEQ.** Однопоточное накопление выборочной суммы. Задаёт корректный референс `~0.12 с` для базы
  сравнения.
- **OMP.** Цикл по выборкам распределяется директивой `for` с редукцией; у каждого потока — собственный
  `std::mt19937`. На 4 потоках ускорение 3.7× (E ≈ 92%, task_run), на 8 — 6.7× (E ≈ 84%). Отдельные
  «зубцы» на 2 потоках и в pipeline-режиме при 4 потоках связаны с одиночными прогонами на лёгком
  теле итерации и попадают в шум в несколько мс.
- **TBB.** `parallel_reduce` со связанным `oneapi::tbb::global_control(max_allowed_parallelism, ...)`.
  Масштабирование плавное: 1.76× при 2 потоках, 6.2× при 8. Лёгкое «налогообложение» на 1 потоке
  (`T(1)/T_seq ≈ 1.11`) — это стоимость арены и постановки задач, которой нет у SEQ.
- **STL.** Ручное разбиение `[0, N)` на равные блоки и собственный массив частичных сумм без
  атомиков. До 4 потоков масштабирование близко к линейному; с 6–8 потоков на лёгком теле
  проявляется плата за `std::thread` и насыщение пропускной способности памяти, эффективность
  опускается в коридор 55–65%.
- **ALL.** Распределение `N` между `P` MPI-процессами и внутрипроцессный OMP. На `2 × 2 = 4`
  рабочих эффективность 84–94%; при `4 × 4 = 16`, `6 × 6 = 36`, `8 × 8 = 64` рабочих наступает
  oversubscribe относительно числа физических ядер — абсолютное время продолжает падать, но
  эффективность по `W` падает быстрее. Это ожидаемо: коммуникация дешёвая
  (раздача параметров и `Allreduce` одного `double`), поэтому потолок задаётся именно числом ядер.

## 8. Заключение

Метод Монте-Карло относится к классу embarrassingly parallel задач: тело алгоритма — независимые
выборки и ассоциативная редукция. Все рассмотренные варианты дают ожидаемое ускорение относительно
SEQ; различия между OMP, TBB и STL объясняются стоимостью управляющего слоя соответствующей модели,
а ALL добавляет к локальному OpenMP межпроцессное распределение работы. На одной машине достаточно
любого из чисто потоковых вариантов; гибрид MPI+OpenMP оправдан, когда требуется выйти за пределы
одного узла или сознательно нарастить `W` за счёт нескольких процессов.
