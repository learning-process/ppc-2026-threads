## 1. Контекст

Метод Монте-Карло по своей природе хорошо подходит для распараллеливания на общей памяти: вклад каждой
выборки в сумму вычисляется независимо. OpenMP позволяет реализовать это идиоматично — через директивы
`parallel`, `for` и `reduction`, без явного управления потоками.

## 2. Постановка задачи

Та же задача, что в SEQ-варианте: численная оценка интеграла

> `I ≈ V · (1 / N) · Σ f(x^(k))`,

где `Ω` — гиперпрямоугольник `[a_i, b_i]`, `V = Π (b_i − a_i)`, поддерживаются восемь типов функций
из `FuncType`. Базовый эталон — SEQ-реализация на том же входе.

## 3. Базовый алгоритм

Сохраняется в неизменном виде: цикл по `N` выборкам, генерация точки в `d`-мерном кубе через
`std::uniform_real_distribution`, накопление выборочной суммы значений `f(x^(k))` и масштабирование
итога на `V / N`.

## 4. Схема распараллеливания

Цикл по выборкам распараллелен по итерациям с редукцией суммы:

1. **Подготовка вне параллельной области:** считаются объём `V` и размерность `dims` — обе величины
   используются только на чтение, что упрощает атрибуты данных и снижает риск ошибок.
2. **Локальные ресурсы потока (атрибуты данных):** внутри `#pragma omp parallel` каждый поток
   создаёт собственный `std::random_device`, `std::mt19937`, вектор
   `uniform_real_distribution<double>` по осям и буфер `point`. Эти объекты `private` по построению —
   они объявлены внутри параллельной области, поэтому отдельная клауза `private(...)` не нужна.
   В `shared(...)` перечислены только разделяемые на чтение величины (`plower`, `pupper`,
   `n_samples`, `ftype`, `dims`, `volume`) и аккумулятор `sum`, для которого редукция объявлена
   ниже. Сам цикл-распределитель добавляет `reduction(+ : sum)` и `schedule(static)` — это покрывает
   всю четвёрку `shared / private / reduction / schedule`.
3. **Раздача итераций:** на основном цикле стоит `#pragma omp for reduction(+ : sum) schedule(static)`.
   `schedule(static)` подходит для однородной нагрузки на итерацию — каждый «бросок» точки стоит
   одинаково, а статическое распределение исключает накладные расходы динамического диспетчера.
4. **Выравнивание числа потоков с курсом:** в начале `RunImpl` вызывается
   `omp_set_num_threads(max(1, ppc::util::GetNumThreads()))`. Так число воркеров OpenMP всегда совпадает с
   единой точкой управления `ppc::util::GetNumThreads()`, используемой и в других реализациях курса.

## 5. Детали реализации

Файлы: [omp/include/ops_omp.hpp](omp/include/ops_omp.hpp), [omp/src/ops_omp.cpp](omp/src/ops_omp.cpp).

Параметры задачи копируются в локальные переменные перед параллельной областью и подаются в `shared(...)`
вместе с `default(none)` — это явный контракт компилятору о видимости каждой переменной и защита от
случайных гонок.

```cpp
omp_set_num_threads(std::max(1, ppc::util::GetNumThreads()));
...
#pragma omp parallel default(none) shared(plower, pupper, n_samples, ftype, dims, volume, sum)
{
  std::random_device rd;
  std::mt19937 gen(rd());
  std::vector<std::uniform_real_distribution<double>> dists;
  ...
#pragma omp for reduction(+ : sum) schedule(static)
  for (int i = 0; i < n_samples; ++i) {
    for (int j = 0; j < dims; ++j) {
      point[j] = dists[j](gen);
    }
    sum += EvaluateAt(ftype, point);
  }
}
```

Вспомогательные функции `EvalLinear`, `EvalSumCubes`, … и диспетчер `EvaluateAt` совпадают с SEQ-версией
и расположены в анонимном `namespace`, что снижает когнитивную сложность `RunImpl` ниже порога 15.

## 6. Проверка корректности

OMP-вариант участвует в одном и том же наборе функциональных тестов с SEQ (`ppc_func_tests`): 13 сценариев,
покрывающих все восемь подынтегральных функций в 1D/2D/3D, проходят с допуском `5.0 / sqrt(N)`. Поскольку
у каждого потока собственный генератор, фактическая последовательность точек у OMP отличается от SEQ —
но оценка интеграла попадает в тот же доверительный интервал, что и подтверждают тесты.

## 7. Экспериментальная среда

- **RAM:** 96 ГБ DDR5
- **OS:** Windows 11 Pro 26200.8457
- **Компилятор:** MSVC 19.50
- **Размер задачи:** `kLinear` на `[0, 1]^3`, `N = 10 000 000`
- **Команды сборки (из корня репозитория):**
  - Конфигурация: `cmake -S . -B build`
  - Перф-тесты: `cmake --build build --target ppc_perf_tests --config Release --parallel`
  - Функциональные тесты: `cmake --build build --target ppc_func_tests --config Release --parallel`
- **Команды запуска (из `build/bin/`):**
  - Обычный запуск: `.\ppc_perf_tests --gtest_filter="*sabirov_s_monte_carlo*omp*"`
  - С потоками:
    `$env:PPC_NUM_THREADS="4"; .\ppc_perf_tests --gtest_filter="*sabirov_s_monte_carlo*omp*"`.
    Вместо `4` — желаемое число потоков. Угловые скобки в значении не ставятся: иначе
    `libenvpp` не распарсит его в `int`, и `GetNumThreads()` вернёт `1`.

## 8. Результаты

База — SEQ: `task_run = 0.1197 с`, `pipeline = 0.1223 с`. Ускорение `S = T_seq / T_par`,
эффективность `E = S / W`, где `W` — число рабочих (для OMP-варианта `W = PPC_NUM_THREADS`).

| 2 | 0.0859 | 1.39 | 69.7% |

### Таблица 2. Режим task_pipeline

| Потоки | Время, с | Ускорение | Эффективность |

| 1 | 0.1204 | 1.02 | 101.6% |
| 2 | 0.0601 | 2.04 | 101.8% |
| 4 | 0.0477 | 2.56 | 64.1% |
| 6 | 0.0256 | 4.78 | 79.6% |
| 8 | 0.0171 | 7.15 | 89.4% |

Основная часть времени — цикл по выборкам, и он эффективно распределяется между потоками: на 1 потоке
поведение совпадает с SEQ, при `N = 4..8` ускорение выходит на 3.7..6.7× (task_run) и 2.6..7.2×
(pipeline). Отдельные «зубцы» (например, `N = 2` task_run хуже ожидаемого, `N = 4` pipeline просел)
— следствие одиночных замеров на лёгком теле итерации: эффекты thread-affinity, частотный
буст и состояние L1/L2 кэша на конкретном прогоне создают разброс в несколько мс. Разница
`pipeline − task_run` остаётся в этом же шумовом диапазоне, потому что `PreProcessing/PostProcessing`
в Monte Carlo тривиальны.

## 9. Выводы

OpenMP-реализация даёт значительное ускорение относительно SEQ за счёт независимости выборок и
эффективной редукции. Выравнивание числа потоков через `ppc::util::GetNumThreads()` делает замеры
сопоставимыми с другими реализациями курса. Структура `parallel + for + reduction` минимизирует
накладные расходы и сохраняет когнитивную сложность `RunImpl` на низком уровне.
