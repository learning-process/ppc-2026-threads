## 1. Контекст

ALL-версия реализует двухуровневый параллелизм: на верхнем уровне данные распределяются между MPI-процессами,
на нижнем - каждый процесс сортирует свою долю параллельно через TBB. Это гибридная модель SPMD
(Single Program, Multiple Data): все процессы исполняют один и тот же код, но работают с разными фрагментами данных.
Процесс с `rank == 0` является корневым - он владеет исходными данными, рассылает их через `MPI_Scatterv` и собирает
результаты через `MPI_Gatherv`, после чего выполняет финальное слияние всех отсортированных фрагментов.

## 2. Постановка задачи

Входные данные, выходные данные, ограничения и критерии корректности идентичны SEQ-версии (см. `seq/report.md`).
Кратко: вход - непустой `std::vector<int>`, выход - тот же набор элементов, отсортированный по неубыванию.
Пустой вектор отклоняется в `ValidationImpl`. В ALL-версии `ValidationImpl` выполняется на всех процессах,
но значимым является только результат на процессе 0, поскольку только он владеет полными входными данными.

## 3. Базовый алгоритм

Локальная логика каждого процесса идентична TBB-версии: разбить свой фрагмент на TBB-чанки, отсортировать
каждый чанк алгоритмом Шелла через `tbb::parallel_for`, затем итеративно слить чанки через `CycleMerge`.
Асимптотика одного процесса: O((n/p) · log²(n/p)) для сортировки и O((n/p) · log(threads)) для локального
слияния, где `p` - число MPI-процессов. Финальное слияние на процессе 0: O(n · log(p)).

## 4. Межпроцессная схема

### Роли rank-ов

Процесс `rank == 0` - корневой: инициализирует разбиение, рассылает данные, собирает результаты и выполняет
финальное слияние. Процессы `rank != 0` - рабочие: получают фрагмент данных, сортируют его и возвращают результат.
После финального слияния процесс 0 рассылает отсортированный вектор всем процессам через `MPI_Bcast`, поскольку
`GetOutput()` проверяется на каждом процессе.

### Распределение данных

```cpp
// File: all/src/ops_all.cpp
if (rank == 0) {
  CalcCountsAndDispls(n, process_count, counts, displs);
}
MPI_Bcast(counts.data(), process_count, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(displs.data(), process_count, MPI_INT, 0, MPI_COMM_WORLD);

std::vector<ValType> local_data(static_cast<size_t>(counts[rank]));
MPI_Scatterv(vec.data(), counts.data(), displs.data(), MPI_INT,
             local_data.data(), counts[rank], MPI_INT, 0, MPI_COMM_WORLD);
```

`CalcCountsAndDispls` вычисляет размеры и смещения фрагментов с равномерным распределением остатка: первые `n % p`
процессов получают `n/p + 1` элементов, остальные - `n/p`. `MPI_Bcast` двух массивов `counts` и `displs` выполняется
до `MPI_Scatterv`, чтобы все процессы знали размеры фрагментов - это необходимо для корректного финального слияния
на процессе 0. `MPI_Scatterv` рассылает фрагменты разного размера в `local_data` каждого процесса.

### Сбор результатов и финальное слияние

```cpp
// File: all/src/ops_all.cpp
MPI_Gatherv(local_data.data(), counts[rank], MPI_INT,
            rank == 0 ? GetOutput().data() : nullptr,
            counts.data(), displs.data(), MPI_INT, 0, MPI_COMM_WORLD);

if (rank == 0) {
  auto root_bounds = BoundsFromCounts(counts);
  size_t root_chunk_count = root_bounds.size() - 1;
  std::vector<ValType> root_buffer(static_cast<size_t>(n));
  for (size_t size = 1; size < root_chunk_count; size *= 2) {
    CycleMerge(GetOutput(), root_buffer, root_bounds, size);
    GetOutput().swap(root_buffer);
  }
}

if (rank != 0) {
  GetOutput().resize(static_cast<size_t>(n));
}
MPI_Bcast(GetOutput().data(), n, MPI_INT, 0, MPI_COMM_WORLD);
```

`MPI_Gatherv` собирает отсортированные фрагменты в `GetOutput()` процесса 0 с учётом смещений `displs`.
`BoundsFromCounts` строит вектор границ из `counts` путём кумулятивного суммирования - каждый фрагмент
становится одним «чанком» для итеративного слияния. Финальный `MPI_Bcast` рассылает результат всем процессам.
В реализации нет `MPI_Barrier` - неявная синхронизация обеспечивается семантикой коллективных операций: каждая
из них (`Bcast`, `Scatterv`, `Gatherv`) блокирует вызвавший процесс до завершения операции на всех участниках
коммуникатора.

### Полная схема выполнения

```text
Процесс 0                  Процесс 1        ...   Процесс p-1
──────────────────────────────────────────────────────────────
CalcCountsAndDispls
MPI_Bcast(counts, displs) ───────────────────────────────────>
MPI_Scatterv ─────────────> local_data[1]   ...   local_data[p-1]
TBB sort+merge (local)     TBB sort+merge         TBB sort+merge
MPI_Gatherv <────────────── local_data[1]   ...   local_data[p-1]
CycleMerge финал (TBB)
MPI_Bcast(result) ───────────────────────────────────────────>
```

## 5. Внутрипроцессная схема

Внутри каждого процесса используется TBB - та же схема, что и в TBB-версии. После получения `local_data` каждый
процесс независимо выполняет:

1. `ChunkBoundaries(counts[rank], threads)` - делит локальный фрагмент на `threads` чанков.
2. `ShellSort` через `tbb::parallel_for(blocked_range<size_t>(0, chunk_count), ...)` - параллельная сортировка
чанков.
3. Цикл `CycleMerge` через `tbb::parallel_for(blocked_range<size_t>(0, merge_count), ...)` - параллельное
итеративное слияние.

Число TBB-потоков задаётся `ppc::util::GetNumThreads()` (читает `PPC_NUM_THREADS`) и одинаково на всех процессах.
Partitioner - `auto_partitioner` по умолчанию. Захват `[&]` в лямбдах безопасен: `local_data` и `buffer` принадлежат
текущему процессу и не разделяются между процессами. Внутри процесса записи ведутся в непересекающиеся диапазоны,
гонок нет.

Локальные результаты не хранятся отдельно - каждый TBB-поток пишет напрямую в `local_data` (фаза сортировки) или в
`buffer` (фаза слияния), после чего выполняется `local_data.swap(buffer)`. После завершения всех TBB-задач данные
готовы к передаче в `MPI_Gatherv`.

## 6. Детали реализации

**Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

`ValidationImpl` и `PreProcessingImpl` идентичны остальным версиям. `PostProcessingImpl` возвращает `true`
без дополнительных действий.

`ShellSort` и `CycleMerge` идентичны TBB-версии. `BatcherMerge`, `SplitEvenOdd` и `Merge` идентичны всем остальным
версиям.

Вспомогательная функция `BoundsFromCounts` строит вектор границ из массива `counts` - это позволяет переиспользовать
`CycleMerge` для финального слияния на процессе 0 без дополнительных преобразований данных.

**Потенциальные узкие места:**

- Финальное слияние на процессе 0 является частично последовательным: внешний цикл `size *= 2` не параллелится между
итерациями.
- `MPI_Gatherv` и `MPI_Bcast` на одной машине работают через shared memory, но всё равно добавляют ненулевой overhead
на синхронизацию.
- Аллокация локальных векторов внутри `BatcherMerge` при большом числе TBB-потоков создаёт давление на аллокатор.

## 7. Проверка корректности

Корректность ALL-версии проверялась для всех 6 функциональных тестов в 9 комбинациях процессов и потоков
(p: {1, 2, 4}, t: {1, 2, 4}). Результат во всех конфигурациях совпадает с SEQ-эталоном. Согласованность результата
между rank-ами гарантируется финальным `MPI_Bcast`: после его завершения `GetOutput()` идентичен на всех процессах.
Корректность финального слияния подтверждается тем, что `BoundsFromCounts` строит границы точно по тем же смещениям,
что использовались при `MPI_Scatterv` и `MPI_Gatherv`.

## 9. Результаты

Baseline SEQ: `T_seq = 0.9250 с` (task_run, из `seq/report.md`). Размер задачи: N = 10 000 000 элементов.

Ускорение рассчитано по формуле `S = T_seq / T(p, t)`. Эффективность нормируется только по числу MPI-процессов
`p` (`E = S / p`), а не по `p × t`: на одной машине TBB-потоки работают в рамках тех же физических ядер, что и
сами процессы, поэтому нормировка по `p × t` давала бы нефизично низкие значения и не отражала бы реальную
утилизацию MPI-уровня параллелизма.

| 1 | 1 | 1 | 0.5053 | 1.83 | 183% |

**Комментарий о масштабируемости:** Наилучший результат в `task_run` - конфигурация 4×1 (0.4039 с, S ≈ 2.29), в
`pipeline` - 4×4 (0.4005 с, S ≈ 2.31). MPI-параллелизм по процессам на данном оборудовании масштабируется лучше,
чем TBB внутри процесса: при p=1 добавление TBB-потоков даёт отрицательный эффект (0.5053 -> 0.8800 с при t=4),
тогда как увеличение числа процессов с 1 до 4 при t=1 снижает время с 0.5053 до 0.4039 с. ALL-версия уступает чистой
TBB-версии (лучший TBB: 0.3080 с), что является ожидаемым результатом на одной машине: MPI-коммуникации через shared
memory добавляют overhead, не компенсируемый на данных размерах. Преимущество ALL реализуется на многоузловом кластере
с распределённой памятью.

## 10. Выводы

Гибридная реализация MPI + TBB корректно работает во всех 9 протестированных конфигурациях. Двухуровневая схема
параллелизма оправдана: MPI-уровень обеспечивает крупное разбиение данных без синхронизации внутри параллельных
секций, TBB-уровень эффективно использует ядра внутри процесса. Финальное слияние на процессе 0 переиспользует
`CycleMerge` с TBB.

На одной машине ALL уступает TBB (S=2.29 против S=3.00): коммуникационный overhead перевешивает выигрыш от второго
уровня параллелизма. Гибридная схема оправдана на кластере с несколькими физическими узлами, где TBB обеспечивает
внутриузловой параллелизм, а MPI - межузловой. На одной машине предпочтительна чистая TBB-версия.
