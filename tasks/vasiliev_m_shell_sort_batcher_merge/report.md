## 1. Введение

Задача сортировки массива чисел методом, объединяющим алгоритм Шелла и чётно-нечётное слияние Бэтчера,
хорошо подходит для сравнения разных моделей параллелизма по нескольким причинам. Во-первых, алгоритм
имеет две естественные точки параллельного разбиения: независимую сортировку чанков и независимое попарное
слияние чанков на каждой итерации. Это позволяет распараллелить как фазу сортировки, так и фазу слияния.
Во-вторых, алгоритм содержит последовательный внешний цикл слияния O(log(chunks)), который ограничивает
масштабируемость - это позволяет обнаружить максимум каждой реализации. В-третьих, задача работает с
динамически выделяемой памятью внутри параллельных секций (локальные векторы в `BatcherMerge`), что
нагружает аллокатор и делает измерения чувствительными к особенностям runtime каждой технологии.

## 2. Единая постановка задачи

**Входные данные (`InType`):** `std::vector<ValType>` - непустой вектор чисел произвольного знака.
Пустой вектор является недопустимым входом и отклоняется в `ValidationImpl` всех реализаций.

**Выходные данные (`OutType`):** `std::vector<ValType>` - перестановка входного вектора, упорядоченная
по неубыванию.

**Критерий корректности:** выходной вектор является перестановкой входного и для всех соседних пар
выполняется `result[i] <= result[i+1]`.

**Общие типы** определены в `common/include/common.hpp`:

```cpp
using ValType  = int;
using InType   = std::vector<ValType>;
using OutType  = std::vector<ValType>;
using BaseTask = ppc::task::Task<InType, OutType>;
```

Все пять реализаций наследуют `BaseTask` и переопределяют `ValidationImpl`, `PreProcessingImpl`, `RunImpl`,
`PostProcessingImpl`. Это обеспечивает единообразие интерфейса и позволяет использовать общую тестовую
инфраструктуру без изменения тестового кода.

**Алгоритм** состоит из двух этапов:

1. Входной вектор делится на части. Каждая часть независимо сортируется алгоритмом Шелла - убывающая
последовательность шагов `gap = n/2, n/4, …, 1`, на каждом шаге - сортировка вставками. Асимптотика одной
части размера `k`: O(k · log²k).
2. Отсортированные части объединяются итеративным чётно-нечётным слиянием Бэтчера (`BatcherMerge`): разбиение
на чётные/нечётные подпоследовательности, двухпутевое слияние, чередование, финальный проход попарных сравнений.
Число итераций внешнего цикла - O(log(число частей)).

**Итоговая асимптотика:** O(n · log²n) по времени, O(n) по памяти.

этих значений. Дополнительные меры стабилизации не применялись.

## 6. Интерпретация различий

**SEQ - что показывает baseline.** Время 0.9250 с - это стоимость двукратного вызова сортировки Шелла
на половинах вектора и однократного вызова `BatcherMerge`. Декомпозиция на две половины является узким местом:
при переходе к многочанковой схеме (OMP/TBB/STL при p=1) время немедленно падает в 1.98–2.88 раза, не добавляя
ни одного реального потока. Это означает, что SEQ-реализация намеренно проста, а не оптимальна - что корректно
для роли baseline.

**OMP - сильные и слабые стороны на этой задаче.** Сильная сторона: `schedule(static)` при равномерных чанках даёт
минимальный overhead планировщика - при p=1 OMP (0.3775 с) заметно быстрее STL (0.4681 с) именно потому, что OpenMP
runtime не создаёт системных потоков при одном потоке. Слабая сторона: статическое распределение не адаптируется
к дисбалансу, поэтому при 4+ потоках TBB обгоняет OMP. Точка оптимума - 2 потока - нетипично мала; это следствие того,
что уже многочанковая декомпозиция при p=1 даёт большую часть прироста, а собственно параллелизм добавляет лишь
дополнительные 9% сверху (0.3775 -> 0.3447 с).

**TBB - роль grain size и runtime.** Лучший абсолютный результат (0.3080 с, S=3.00) достигается при 4 потоках благодаря
`auto_partitioner`: он адаптивно укрупняет подзадачи при малой нагрузке и дробит их при дисбалансе, что особенно важно
на фазе слияния, где размеры сливаемых пар чанков различаются на разных итерациях цикла. Регрессия при 2 потоках
(0.3349 с хуже, чем 0.3209 с при 1 потоке) - типичный для TBB overhead инициализации второго worker-потока; при 4
потоках объём работы уже достаточен для его покрытия.

**STL - цена ручного управления потоками.** При p=1 STL (0.4681 с) является самым медленным однопроцессным backend-ом:
создание `std::thread` происходит при каждом вызове `ShellSort` и каждой итерации цикла слияния, что суммарно даёт
значимый системный overhead. При 4 потоках этот overhead перекрывается параллелизмом, и STL выходит на результат
(0.3284 с), сопоставимый с OMP. Преимущество STL-реализации - отсутствие зависимости от внешнего runtime и полный
контроль над жизненным циклом потоков; недостаток - повторное создание потоков вместо использования пула.

**ALL - цена коммуникации и выигрыш гибридности.** На одной машине ALL уступает всем однопроцессным реализациям:
MPI-коммуникации (`Scatterv`, `Gatherv`, `Bcast`) добавляют ненулевой overhead даже в shared-memory среде, а
финальное слияние на процессе 0 является частично последовательным. Увеличение числа TBB-потоков при фиксированном
числе MPI-процессов не улучшает `task_run` - данных на поток становится меньше, а overhead TBB runtime остаётся.
Конфигурация 4×1 оказывается лучшей: MPI даёт нужный параллелизм без накладных расходов второго уровня. Реальный
выигрыш ALL достижим только на многоузловом кластере.

## 8. Заключение

**Лучшая версия и условия её применимости:** На данном оборудовании лучший результат даёт **TBB** при 4 потоках
(task_run = 0.3080 с, S = 3.00). Она оптимальна для однопроцессных задач на многоядерном CPU: адаптивный планировщик
сам находит правильный grain size, не требуя ручной настройки. **OMP** при 2 потоках занимает второе место по
ускорению (S = 2.68) при минимальном коде изменений - предпочтителен когда важна простота и предсказуемость.
**STL** при 4 потоках (S = 2.82) сопоставима с TBB и OMP, но требует наибольшего объёма кода - оправдана когда
нет доступа к TBB/OMP или нужен полный контроль над потоками. **ALL** уступает на одной машине, но является
единственной версией, масштабируемой на кластер с распределённой памятью.

**Ограничения сравнения:** Замеры выполнялись на виртуальной машине без дополнительной стабилизации: VirtualBox
вносит непредсказуемый jitter, частотный скейлинг не отключался, процесс не закреплялся за ядром. Качественная
иерархия TBB > STL ≈ OMP > ALL на одной машине устойчива, однако абсолютные значения времени и точные значения
ускорения могут меняться при повторных запусках.

**Что можно было бы улучшить:** Основное узкое место всех реализаций - аллокация локальных векторов в `BatcherMerge`
внутри параллельных секций. Применение `tbb::scalable_allocator` или предварительного выделения буферов снизило бы
давление на аллокатор и потенциально улучшило масштабируемость при 6–8 потоках. Для STL-версии переход на пул потоков
устранил бы overhead пересоздания потоков на каждой итерации цикла слияния.

### Краткая схема алгоритма

```text
Входной вектор [0 .. n-1]
        │
        ▼
┌───────────────────────────────┐
│  ChunkBoundaries(n, threads)  │  -> bounds[]
└───────────────────────────────┘
        │
        ▼
┌───────────────────────────────┐
│  ShellSort (параллельно)      │  каждый поток сортирует свой чанк
│  chunk_0 | chunk_1 | ... | k  │
└───────────────────────────────┘
        │
        ▼
┌───────────────────────────────┐
│  for size = 1, 2, 4, ...      │  последовательный внешний цикл
│    CycleMerge (параллельно)   │  пары чанков сливаются независимо
│    vec.swap(buffer)           │
└───────────────────────────────┘
        │
        ▼
  Отсортированный вектор
```

Для ALL между `ShellSort` и финальным `CycleMerge` вставлены `MPI_Scatterv` (раздача чанков процессам) и
`MPI_Gatherv` (сбор результатов на процессе 0).
