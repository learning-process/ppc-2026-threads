## 1. Контекст

Задача состоит в сортировке массива чисел методом, объединяющим сортировку Шелла и чётно-нечётное слияние Бэтчера.
Последовательная версия реализует полный алгоритм без какого-либо параллелизма и служит эталоном корректности и
производительности для всех последующих параллельных реализаций (OMP, TBB, STL, ALL). Именно время выполнения
SEQ-версии будет знаменателем при расчёте ускорения в остальных отчётах.

## 2. Постановка задачи

**Входные данные (`InType`):** `std::vector<ValType>` - непустой вектор чисел произвольного знака и величины.

**Выходные данные (`OutType`):** `std::vector<ValType>` - тот же набор элементов, отсортированный по неубыванию.

**Ограничения и крайние случаи:**

- Пустой вектор считается недопустимым входом и отклоняется на этапе валидации.
- Вектор из одного элемента является допустимым и возвращается как уже отсортированный.
- Вектор с уже отсортированными элементами (в том числе в обратном порядке) должен быть корректно обработан.
- Элементы могут быть отрицательными, нулевыми и повторяющимися.
- Формальный тип элементов - `ValType = int`, определён в `common/include/common.hpp`.

**Критерий корректности:** выходной вектор является перестановкой входного и для всех соседних пар выполняется
`result[i] <= result[i+1]`.

## 3. Базовый алгоритм

Алгоритм состоит из двух логических этапов.

**Этап 1 - сортировка Шелла двух половин.**
Входной вектор делится на левую и правую половины. Каждая половина независимо сортируется алгоритмом Шелла с убывающей
последовательностью шагов: начальный шаг `gap = n/2`, затем `gap /= 2` до `gap == 0`. На каждом шаге выполняется
сортировка вставками для элементов, разделённых этим шагом.

**Этап 2 - чётно-нечётное слияние Бэтчера.**
Две отсортированные половины объединяются методом `BatcherMerge`:

1. Каждая из половин разбивается на подпоследовательности чётных и нечётных позиций (`SplitEvenOdd`).
2. Чётные подпоследовательности обеих половин сливаются стандартным двухпутевым слиянием (`Merge`), то же -
для нечётных.
3. Результаты чередуются в выходной вектор: сначала элемент из чётной последовательности, затем из нечётной,
и так далее.
4. Выполняется финальный проход: соседние пары на нечётных позициях (индексы 1,2; 3,4; ...) сравниваются и при
необходимости переставляются (`swap`).

**Асимптотика по времени:**

- Сортировка Шелла одной половины размера `n/2`: в среднем O((n/2) · log²(n/2)) ≈ O(n log²n).
- Обе половины вместе: O(n log²n).
- Слияние Бэтчера: O(n log n).
- Итоговая асимптотика: **O(n log²n)**.

**Асимптотика по памяти:** O(n) - вспомогательные векторы суммарного размера не превышают O(n).

**Инварианты:**

- После `ShellSort` каждая из половин является строго отсортированной последовательностью.
- После `BatcherMerge` результирующий вектор содержит ровно `l.size() + r.size()` элементов и отсортирован по
неубыванию.

## 4. Детали реализации

**Файлы:** `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

**`ValidationImpl`** - проверяет, что входной вектор непуст. Если вектор пуст, возвращает `false`, и задача не
исполняется.

```cpp
// File: seq/src/ops_seq.cpp
bool VasilievMShellSortBatcherMergeSEQ::ValidationImpl() {
  return !GetInput().empty();
}
```

**`PreProcessingImpl`** - очищает выходной вектор перед запуском, чтобы исключить накопление результатов при
повторных запусках.

```cpp
// File: seq/src/ops_seq.cpp
bool VasilievMShellSortBatcherMergeSEQ::PreProcessingImpl() {
  GetOutput().clear();
  return true;
}
```

**`RunImpl`** - основная логика: делит входной вектор на две половины, сортирует каждую из них функцией `ShellSort`,
затем объединяет через `BatcherMerge`. Результат записывается в выходной вектор.

```cpp
// File: seq/src/ops_seq.cpp
bool VasilievMShellSortBatcherMergeSEQ::RunImpl() {
  auto &vec = GetInput();
  const size_t n = vec.size();
  if (vec.empty()) return false;

  size_t mid = n / 2;
  std::vector<ValType> l(vec.begin(), vec.begin() + static_cast<std::ptrdiff_t>(mid));
  std::vector<ValType> r(vec.begin() + static_cast<std::ptrdiff_t>(mid), vec.end());

  ShellSort(l);
  ShellSort(r);

  GetOutput() = BatcherMerge(l, r);
  return true;
}
```

Здесь `ShellSort` вызывается дважды независимо - это точка, в которой параллельные версии смогут распараллелить
работу.
После сортировки обеих половин `BatcherMerge` выполняет детерминированное объединение по схеме Бэтчера.

**`PostProcessingImpl`** - в данной реализации не выполняет дополнительных действий, результат уже записан в
выходной вектор в `RunImpl`. Возвращает `true`.

**Граничные случаи:** при нечётном размере входного вектора левая половина оказывается на один элемент короче
правой (деление нацело). Это корректно обрабатывается: `SplitEvenOdd` и `Merge` не предполагают равенства
размеров аргументов.

## 7. Результаты

Тесты производительности запускались в двух режимах: `task_run` и `pipeline`.

| Режим | N | Workers | Время (с) | Примечание |
| --- | --- | --- | --- | --- |
| `task_run` | 10 000 000 | 1 | 0.9250 | baseline для ускорения |
| `pipeline` | 10 000 000 | 1 | 0.9284 | baseline для pipeline-ускорения |

Время в режиме `task_run` (0.9250 с) будет использоваться как знаменатель при расчёте ускорения во всех параллельных
реализациях. Время `pipeline` (0.9284 с) незначительно отличается от `task_run`, что говорит о минимальных накладных
расходах на валидацию и пре/постобработку - вычислительная работа сосредоточена целиком в `RunImpl`.

Наиболее дорогой фрагмент - двойной вызов `ShellSort` и последующий `BatcherMerge` внутри `RunImpl`. Именно эти три
вызова являются естественными точками параллельного разбиения в последующих реализациях.

## 8. Выводы

Последовательная реализация корректно решает задачу для всех проверенных входных данных, включая граничные случаи
(обратный порядок, отрицательные числа, нечётная длина). Алгоритм детерминирован и не зависит от порядка обхода,
что делает его надёжным эталоном для сравнения с параллельными версиями.

Время `task_run = 0.9250 с` фиксируется как официальный baseline: все значения ускорения S и эффективности E в
отчётах OMP, TBB, STL и ALL будут вычисляться по формулам:

```text
S(p) = T_seq / T(p)
E(p) = (S(p) / p) * 100%
```

где `T_seq = 0.9250 с`, `p` - число рабочих единиц (потоков или процессов) в соответствующей реализации.
