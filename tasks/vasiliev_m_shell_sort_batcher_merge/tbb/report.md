## 1. Контекст

TBB-версия реализует ту же схему параллельной сортировки, что и OMP-версия, но средствами библиотеки
oneTBB: задачно-ориентированная модель с динамическим планировщиком вместо явных директив компилятора.
Число рабочих потоков задаётся через `ppc::util::GetNumThreads()`, которая читает переменную
окружения `PPC_NUM_THREADS`. Параллельное разбиение реализовано через `tbb::parallel_for` с
`tbb::blocked_range`, что позволяет runtime самостоятельно управлять распределением работы между потоками.

## 2. Постановка задачи

Входные данные, выходные данные, ограничения и критерии корректности идентичны SEQ-версии (см. `seq/report.md`).
Кратко: вход - непустой `std::vector<int>`, выход - тот же набор элементов, отсортированный по неубыванию.
Пустой вектор отклоняется в `ValidationImpl`.

## 3. Базовый алгоритм

Последовательная логика: разбить вектор на части, каждую отсортировать алгоритмом Шелла, затем итеративно
объединить отсортированные части через чётно-нечётное слияние Бэтчера. Асимптотика и инварианты описаны в
`seq/report.md`. TBB-версия не меняет алгоритм - она заменяет OpenMP-директивы на вызовы `tbb::parallel_for`.

## 4. Схема распараллеливания

### Разбиение на чанки

Функция `ChunkBoundaries(n, threads)` делит вектор размером `n` на `min(threads, n)` чанков с равномерным
распределением остатка. Число чанков определяется значением `ppc::util::GetNumThreads()`.

### Примитив параллелизма

Оба параллельных этапа используют `tbb::parallel_for` с `tbb::blocked_range<size_t>`. Partitioner не задаётся
явно - используется partitioner по умолчанию (`auto_partitioner`), который позволяет TBB runtime адаптивно
разбивать диапазон на подзадачи в зависимости от нагрузки. Grain size также не задаётся явно: `auto_partitioner`
выбирает его самостоятельно исходя из размера диапазона и числа доступных потоков.

### Фаза 1 - параллельная сортировка Шелла

```cpp
// File: tbb/src/ops_tbb.cpp
tbb::parallel_for(tbb::blocked_range<size_t>(0, chunk_count),
  [&](const tbb::blocked_range<size_t> &range) {
    for (size_t chunk = range.begin(); chunk < range.end(); ++chunk) {
      size_t first = bounds[chunk];
      size_t last  = bounds[chunk + 1];
      size_t n     = last - first;

      for (size_t gap = n / 2; gap > 0; gap /= 2) {
        for (size_t i = first + gap; i < last; i++) {
          ValType tmp = vec[i];
          size_t j = i;
          while (j >= first + gap && vec[j - gap] > tmp) {
            vec[j] = vec[j - gap];
            j -= gap;
          }
          vec[j] = tmp;
        }
      }
    }
  });
```

Диапазон `[0, chunk_count)` разбивается на поддиапазоны, каждый из которых обрабатывается одним вызовом лямбды.
Внутри лямбды каждый чанк сортируется независимо в своём диапазоне `[bounds[chunk], bounds[chunk+1])`. Захват
`[&]` безопасен: `vec` читается и пишется в непересекающиеся диапазоны разными потоками, `bounds` используется
только на чтение. Гонок на данных нет. После завершения `parallel_for` все потоки синхронизированы - неявный барьер
гарантируется семантикой `parallel_for` в oneTBB.

### Фаза 2 - параллельное итеративное слияние

```cpp
// File: tbb/src/ops_tbb.cpp
const size_t merge_count = (chunk_count + (2 * size) - 1) / (2 * size);

tbb::parallel_for(tbb::blocked_range<size_t>(0, merge_count),
  [&](const tbb::blocked_range<size_t> &range) {
    for (size_t idx = range.begin(); idx < range.end(); ++idx) {
      const size_t l   = idx * 2 * size;
      const size_t mid = std::min(l + size, chunk_count);
      const size_t r   = std::min(l + (2 * size), chunk_count);
      // ...вычисление границ, вызов BatcherMerge, запись в buffer...
    }
  });
```

`merge_count` - число независимых пар групп чанков на текущей итерации внешнего цикла. Каждая пара сливается
независимо и записывает результат в свой непересекающийся участок `buffer`. После `parallel_for` выполняется
`vec.swap(buffer)` - последовательная операция вне параллельной области. Внешний цикл
`for (size_t size = 1; size < chunk_count; size *= 2)` последователен: каждая следующая итерация начинается
только после полного завершения предыдущего `parallel_for`.

### Контроль конкуренции

Число worker-потоков TBB runtime ограничивается через `tbb::task_arena`, инициализируемый значением
`ppc::util::GetNumThreads()`. Таким образом, `task_arena` задаёт верхнюю границу параллелизма.

## 5. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

`ValidationImpl`, `PreProcessingImpl` и `PostProcessingImpl` идентичны SEQ и OMP версиям. `RunImpl`
отличается от OMP-версии только в двух местах: вместо `omp_get_max_threads()` используется
`ppc::util::GetNumThreads()`, а вместо `#pragma omp parallel for` - `tbb::parallel_for` (см. `omp/report.md`).

`BatcherMerge`, `SplitEvenOdd` и `Merge` полностью идентичны SEQ-версии. Каждый поток вызывает их со своими
локальными векторами, глобального состояния эти функции не изменяют, что исключает гонки.

Потенциальным узким местом является выделение локальных векторов `l_vect`, `r_vect`, `merged` внутри лямбды
`CycleMerge` - каждый поток выполняет несколько аллокаций на итерацию. При большом числе потоков это создаёт
нагрузку на аллокатор. Для снижения этого эффекта можно было бы использовать `tbb::scalable_allocator`, однако
в данной реализации это не применялось, так как по результатам теста производительности, дополнительная нагрузка
была незначительна.

TBB runtime при инициализации второго worker-потока на задаче, которая уже хорошо решается одним потоком. При 4
потоках
runtime находит оптимальный баланс. При 6 и 8 потоках производительность деградирует: конкуренция за память
при аллокации
локальных векторов в `BatcherMerge` и насыщение пропускной способности памяти.

## 9. Выводы

TBB-версия демонстрирует лучший абсолютный результат среди однопроцессных реализаций: `task_run = 0.3080 с`
при 4 потоках
(S ≈ 3.00). При этом, адаптивный `auto_partitioner` обеспечивает более эффективное разбиение диапазона по сравнению
со статическим распределением OpenMP. Отсутствие явного задания grain size оправдано однородностью работы в
чанках - runtime самостоятельно определяет оптимальный размер подзадачи. Ограничением масштабируемости остаётся
аллокация временных векторов внутри `BatcherMerge`; применение `tbb::scalable_allocator` могло бы улучшить результаты
при большом числе потоков.
