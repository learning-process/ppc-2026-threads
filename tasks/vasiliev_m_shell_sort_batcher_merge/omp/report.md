## 1. Контекст

OMP-версия переносит задачу сортировки на многопоточную модель OpenMP. По сравнению с SEQ-реализацией, которая
делит вектор ровно на две половины и сортирует их последовательно, OMP-версия делит вектор на столько чанков,
сколько доступно потоков, сортирует все чанки параллельно, а затем выполняет итеративное параллельное слияние
по схеме Бэтчера. Это позволяет задействовать все ядра процессора как на фазе сортировки, так и на фазе слияния.

## 2. Постановка задачи

Входные данные, выходные данные, ограничения и критерии корректности идентичны SEQ-версии (см. `seq/report.md`).
Кратко: вход - непустой `std::vector<ValType>`, выход - тот же набор элементов, отсортированный по неубыванию.
Пустой вектор отклоняется в `ValidationImpl`.

## 3. Базовый алгоритм

Последовательная логика: разбить вектор на части, каждую отсортировать алгоритмом Шелла, затем объединить
отсортированные части через чётно-нечётное слияние Бэтчера. Асимптотика и инварианты сортировки и слияния
описаны в `seq/report.md`. OMP-версия не меняет алгоритм - она распараллеливает оба этапа.

## 4. Схема распараллеливания

### Разбиение на чанки

Функция `ChunkBoundaries(n, threads)` делит вектор размером `n` на `min(threads, n)` чанков. Первые `n % threads`
чанков получают на один элемент больше (равномерное распределение остатка). Результат - вектор границ `bounds`,
где чанк `i` занимает диапазон `[bounds[i], bounds[i+1])`.

### Фаза 1 - параллельная сортировка Шелла

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(vec, bounds, chunk_count) schedule(static)
for (size_t chunk = 0; chunk < chunk_count; chunk++) {
  size_t first = bounds[chunk];
  size_t last = bounds[chunk + 1];
  size_t n = last - first;

  for (size_t gap = n / 2; gap > 0; gap /= 2) {
    for (size_t i = first + gap; i < last; i++) {
      ValType tmp = vec[i];
      size_t j = i;
      while (j >= first + gap && vec[j - gap] > tmp) {
        vec[j] = vec[j - gap];
        j -= gap;
      }
      vec[j] = tmp;
    }
  }
}
```

Каждый поток обрабатывает свой чанк независимо, без пересечения диапазонов. Переменные `chunk`, `first`, `last`,
`n`, `tmp`, `j` - локальные для каждой итерации, поэтому явно объявлять их `private` не требуется - OpenMP
автоматически делает переменные, объявленные внутри параллельной области, приватными. Переменные `vec`, `bounds`,
`chunk_count` разделяются между потоками только на чтение (запись в `vec` ведётся в непересекающиеся диапазоны),
поэтому они безопасно помечены как `shared`. Гонок на данных нет. `schedule(static)` выбран, поскольку чанки имеют
почти одинаковый размер и равномерное статическое распределение минимизирует накладные расходы планировщика. В конце
`parallel for` существует неявный барьер: все потоки завершают сортировку своих чанков прежде, чем начнётся фаза
слияния.

### Фаза 2 - параллельное итеративное слияние

```cpp
// File: omp/src/ops_omp.cpp
std::vector<ValType> buffer(n);
for (size_t size = 1; size < chunk_count; size *= 2) {
  CycleMerge(vec, buffer, bounds, size);
  vec.swap(buffer);
}
```

На каждой итерации внешнего цикла пары соседних групп чанков сливаются через `BatcherMerge`. Внутри `CycleMerge`
пары групп независимы друг от друга, поэтому слияние каждой пары выполняется параллельно:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) \
  shared(vec, buffer, bounds, size, chunk_count) schedule(static)
for (size_t l_chunk = 0; l_chunk < chunk_count; l_chunk += (2 * size)) {
  // ...вычисление границ пары групп...
  std::vector<ValType> merged = BatcherMerge(l_vect, r_vect);
  for (size_t i = 0; i < merged.size(); i++) {
    buffer[start + i] = merged[i];
  }
}
```

Каждый поток записывает результат слияния в свой непересекающийся участок `buffer`, поэтому синхронизация записи
не нужна. После завершения `CycleMerge` выполняется `vec.swap(buffer)` - это последовательная операция вне
параллельной области, гонок нет. Неявный барьер в конце `parallel for` гарантирует, что все слияния завершены до
`swap`.

### Атрибуты переменных - сводка

| Переменная | Атрибут | Обоснование |
| --- | --- | --- |
| `vec`, `buffer` | `shared` | запись в непересекающиеся диапазоны |
| `bounds`, `size`, `chunk_count` | `shared` | только чтение |
| `chunk`, `first`, `last`, `n`, `tmp`, `j` | неявно private | объявлены внутри тела цикла |

### Число потоков

Число потоков определяется через `omp_get_max_threads()` в `RunImpl` при вычислении границ чанков. Это значение
управляется переменной окружения `OMP_NUM_THREADS`.

## 5. Детали реализации

**Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

`ValidationImpl` и `PreProcessingImpl` идентичны SEQ-версии. `PostProcessingImpl` возвращает `true` без
дополнительных действий.

Ключевое отличие от SEQ - в `RunImpl` вектор делится не на две, а на `chunk_count` частей, где
`chunk_count = omp_get_max_threads()` (но не более `n`). После параллельной сортировки фаза слияния выполняется
в цикле удвоения: на первом шаге сливаются пары соседних чанков, на втором - пары групп по два чанка, и так далее,
пока не останется один отсортированный вектор. Число итераций внешнего цикла - O(log(chunk_count)).

`BatcherMerge`, `SplitEvenOdd` и `Merge` полностью идентичны SEQ-версии и вызываются из тел параллельных
секций - каждый поток работает со своими локальными векторами, глобального состояния эти функции не изменяют.

| 8 | 10M | 0.3496 | 2.65 | 33% |

**Комментарий о масштабируемости:** Уже при 1 потоке OMP-версия заметно быстрее SEQ (≈2.45×), что объясняется
другой декомпозицией задачи: вместо сортировки двух половин сортируется `chunk_count` меньших фрагментов, каждый
из которых быстрее обрабатывается алгоритмом Шелла. Наилучший результат в режиме `task_run` достигается при 2 потоках
(0.3447 с, S ≈ 2.68); при 4 потоках время практически не меняется (0.3415 с), что свидетельствует о выходе на плато
масштабируемости. При 6 и 8 потоках время незначительно растёт: накладные расходы на порождение потоков и конкуренция
за память при выделении локальных векторов внутри `BatcherMerge` начинают перевешивать выигрыш от параллелизма. При 8
потоках (oversubscription - потоков больше, чем физических ядер) ускорение не ухудшается катастрофически (S ≈ 2.49),
однако эффективность падает до 31%, что типично для режима перегрузки ядер. Эффективность монотонно убывает с ростом
числа потоков, что отражает ограничения алгоритма: фаза слияния содержит последовательный цикл удвоения (`size *= 2`),
ограничивающий параллелизм.

## 9. Выводы

OpenMP позволил распараллелить оба ключевых этапа алгоритма - сортировку чанков и итеративное слияние - с минимальными
изменениями относительно последовательной логики. Наилучший результат получен при 2 потоках
(`task_run = 0.3447 с`, S ≈ 2.68); при 4 потоках результат практически идентичен, что говорит о плато
масштабируемости.
Дальнейший рост числа потоков до 6 и 8 не даёт выигрыша из-за накладных расходов на создание локальных векторов в
`BatcherMerge` и насыщения пропускной способности памяти. Режим oversubscription (8 потоков при 6 ядрах) не приводит
к деградации производительности, однако и прироста не даёт. Выбор `schedule(static)` оправдан равномерностью размеров
чанков; при сильно неравномерных данных имело бы смысл рассмотреть `schedule(dynamic)`. Отсутствие `reduction` и
`atomic` стало возможным благодаря строгому разбиению данных на непересекающиеся диапазоны с записью в отдельный
буфер.
