## 1. Контекст

STL-версия реализует параллельную сортировку средствами стандартной библиотеки C++ - через `std::thread`
без каких-либо сторонних runtime-библиотек. В отличие от OMP и TBB, где планировщик управляет потоками
автоматически, здесь всё управление потоками явное: создание, распределение работы, синхронизация через
`join`. Это делает STL-версию наиболее «инженерной» из всех реализаций и наглядно показывает стоимость
ручного управления потоками.

## 2. Постановка задачи

Входные данные, выходные данные, ограничения и критерии корректности идентичны SEQ-версии (см. `seq/report.md`).
Кратко: вход - непустой `std::vector<int>`, выход - тот же набор элементов, отсортированный по неубыванию.
Пустой вектор отклоняется в `ValidationImpl`.

## 3. Базовый алгоритм

Последовательная логика: разбить вектор на части, каждую отсортировать алгоритмом Шелла, затем итеративно
объединить отсортированные части через чётно-нечётное слияние Бэтчера. Асимптотика и инварианты описаны в
`seq/report.md`. STL-версия не меняет алгоритм - она заменяет последовательные циклы на явно создаваемые
`std::thread`.

## 4. Схема распараллеливания

### Разбиение данных по потокам

Функция `ChunkBoundaries(n, threads)` идентична OMP и TBB версиям: делит вектор на `min(threads, n)` чанков
с равномерным распределением остатка. Число чанков определяется `ppc::util::GetNumThreads()`.

Дополнительно в `ShellSort` и `CycleMerge` вводится второй уровень разбиения: чанки (или пары групп чанков)
распределяются между `worker_count = min(chunk_count, num_threads)` потоками. Каждый поток получает диапазон
индексов `[begin, end)` чанков для обработки:

```cpp
// File: stl/src/ops_stl.cpp
const size_t base = chunk_count / worker_count;
const size_t rem  = chunk_count % worker_count;
size_t current = 0;

for (size_t thread_id = 0; thread_id < worker_count; thread_id++) {
  const size_t count = base + (thread_id < rem ? 1 : 0);
  const size_t begin = current;
  const size_t end   = current + count;
  current = end;

  threads.emplace_back([&, begin, end]() {
    for (size_t chunk = begin; chunk < end; chunk++) {
      ShellSortChunk(vec, bounds[chunk], bounds[chunk + 1]);
    }
  });
}
```

Первые `rem` потоков получают на один чанк больше - это равномерное ручное распределение остатка, аналог того,
что `schedule(static)` делает в OMP автоматически. Переменные `begin` и `end` захватываются по значению
`[&, begin, end]`, что критически важно: если бы они захватывались по ссылке, к моменту запуска потока значение
`current` уже изменилось бы следующей итерацией цикла. Захват `vec` и `bounds` - по ссылке, что безопасно, так
как эти объекты живут на протяжении всего времени работы потоков и доступ к ним ведётся в непересекающиеся диапазоны.

### Фаза 1 - параллельная сортировка Шелла

Каждый поток вызывает `ShellSortChunk` для своих чанков. Функция работает в диапазоне `[first, last)` и
не обращается к элементам вне него:

```cpp
// File: stl/src/ops_stl.cpp
void VasilievMShellSortBatcherMergeSTL::ShellSortChunk(
    std::vector<ValType> &vec, size_t first, size_t last) {
  size_t n = last - first;
  for (size_t gap = n / 2; gap > 0; gap /= 2) {
    for (size_t i = first + gap; i < last; i++) {
      ValType tmp = vec[i];
      size_t j = i;
      while (j >= first + gap && vec[j - gap] > tmp) {
        vec[j] = vec[j - gap];
        j -= gap;
      }
      vec[j] = tmp;
    }
  }
}
```

После запуска всех потоков выполняется `join` в отдельном цикле - это принципиально важно: `join` вызывается только
после того, как **все** потоки созданы и запущены, что обеспечивает реальный параллелизм.

```cpp
// File: stl/src/ops_stl.cpp
for (auto &th : threads) {
  if (th.joinable()) {
    th.join();
  }
}
```

Если бы `join` вызывался внутри цикла создания потоков - это сериализовало бы выполнение. В данной реализации этой
ошибки нет.

### Фаза 2 - параллельное итеративное слияние

`CycleMerge` использует ту же схему: вычисляется `merge_count` - число независимых пар групп на текущей итерации,
они распределяются между потоками, каждый поток вызывает `MergeChunks` для своего диапазона пар. `MergeChunks`
записывает результат в непересекающийся участок `buffer`:

```cpp
// File: stl/src/ops_stl.cpp
void VasilievMShellSortBatcherMergeSTL::MergeChunks(
    std::vector<ValType> &vec, std::vector<ValType> &buffer,
    size_t start, size_t middle, size_t end_pos) {
  if (middle == end_pos) {
    std::copy(vec.begin() + start, vec.begin() + end_pos,
              buffer.begin() + start);
    return;
  }
  std::vector<ValType> l_vect(vec.begin() + start, vec.begin() + middle);
  std::vector<ValType> r_vect(vec.begin() + middle, vec.begin() + end_pos);
  std::vector<ValType> merged = BatcherMerge(l_vect, r_vect);
  for (size_t i = 0; i < merged.size(); i++) {
    buffer[start + i] = merged[i];
  }
}
```

После `join` всех потоков выполняется `vec.swap(buffer)` - последовательная операция вне параллельной области.

### Карта поток -> диапазон данных

| Поток | Диапазон чанков (фаза 1) | Диапазон пар групп (фаза 2) |
| :---: | :---: | :---: |
| 0 | `[0, base + (0 < rem ? 1 : 0))` | аналогично |
| 1 | следующий диапазон | аналогично |
| … | … | … |
| p-1 | последний диапазон | аналогично |

Локальные результаты не хранятся отдельно - каждый поток пишет напрямую в `vec` (фаза 1) или в `buffer` (фаза 2) в
свой непересекающийся участок. Синхронизация через `atomic` или `mutex` не требуется.

### Синхронизация - сводка

| Механизм | Где используется |
| --- | --- |
| `join` после создания всех потоков | конец `ShellSort`, конец `CycleMerge` |
| захват `begin`, `end` по значению | лямбды потоков - защита от гонки на переменной цикла |
| непересекающиеся диапазоны записи | исключает гонки на `vec` и `buffer` |
| `mutex` / `atomic` / `condition_variable` | не используются |

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

`ValidationImpl`, `PreProcessingImpl` и `PostProcessingImpl` идентичны остальным версиям. `RunImpl` по структуре
совпадает с OMP и TBB: `ChunkBoundaries` -> `ShellSort` -> цикл `CycleMerge` -> `swap` -> запись в `GetOutput()`.

Отличительная особенность STL-версии - вынесение логики сортировки чанка в отдельную статическую функцию
`ShellSortChunk` и логики слияния - в `MergeChunks`. Это обеспечивает чистоту лямбд и упрощает отладку:
каждая функция имеет строго определённые входные и выходные данные без захвата лишнего контекста.

По сравнению с OMP и TBB, STL-версия несёт дополнительные накладные расходы на создание и уничтожение объектов
`std::thread` при каждом вызове `ShellSort` и `CycleMerge`. При большом числе итераций внешнего цикла слияния
(O(log(chunk_count))) это создаёт заметный overhead, особенно при малом числе потоков, когда стоимость создания
потока сопоставима с объёмом работы.

## 9. Выводы

STL-версия демонстрирует корректную работу и реальный параллелизм при правильной схеме `create -> work -> join`:
все потоки создаются до первого `join`, что исключает скрытую сериализацию. Наилучший результат при 4 потоках
(S ≈ 2.82) сопоставим с OMP при 4 потоках (S ≈ 2.71) и незначительно уступает TBB при 4 потоках (S ≈ 3.00). Главным
ограничением является стоимость создания и уничтожения `std::thread` при каждом вызове `ShellSort` и `CycleMerge`:
в отличие от OMP и TBB, где пул потоков создаётся один раз, STL-версия создаёт потоки заново на каждом шаге.
Это особенно заметно при малом числе потоков, где overhead пересоздания потоков составляет значительную долю
общего времени. Использование `std::thread` оправдано как демонстрация ручного управления параллелизмом, однако
для производительных задач предпочтительнее TBB или OMP.
