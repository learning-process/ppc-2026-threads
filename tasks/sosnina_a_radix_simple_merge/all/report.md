## 1. Контекст

Нужна **гибридная** реализация: **MPI-процессы** распределяют глобальный массив; **внутри каждого ранга** локальный кусок сортируется **той же схемой**, что и STL-ветка (`std::thread`, `ParallelForRange`, radix + дерево merge). При **`mpi_size > 1`** выполняется **межпроцессное** слияние по **гиперкубу**; результат доступен **на всех рангах** через **`MPI_Bcast`**.

## 2. Постановка задачи

**Цель.** Дан непустой одномерный массив целых чисел. Требуется построить **перестановку** тех же элементов, упорядоченную по **неубыванию**.

**Типы.** В `common/include/common.hpp`: **`InType` = `OutType` = `std::vector<int>`**, **`BaseTask` = `ppc::task::Task<InType, OutType>`**.

**Вход.** Вектор `data` типа `InType`, **`|data| ≥ 1`**. Пустой вход отклоняется в **`ValidationImpl()`**.

**Выход.** Вектор `OutType` той же длины, **отсортированный по неубыванию**, с совпадающим с входом **мультимножеством** значений.

**Проверка корректности.** Общие функциональные тесты в `tests/functional/main.cpp` (**48** кейсов), сравнение с эталоном **`std::sort`**; в **`RunImpl()`** дополнительно **`std::ranges::is_sorted`**.

**Особые случаи.** При **`|data| ≤ 1`** сортировка тривиальна.

**Особенность ветки ALL.** Глобальный вход изначально хранится на ранге **0**; через **`MPI_Scatterv`** каждый MPI-процесс получает свой непрерывный фрагмент. После локальной сортировки и межпроцессного слияния **на всех рангах** должен быть восстановлен **тот же** глобальный отсортированный массив (в коде — **`MPI_Bcast`** полного результата), чтобы контракт «выход = перестановка входа по неубыванию» выполнялся **на каждом ранге** так же, как в потоковых ветках на одном процессе.

**Запуск тестов.** Функциональные и perf-тесты для типа задачи **`all`** выполняются под **`mpirun`** (см. `scripts/run_tests.py`, режим `processes`); без MPI соответствующие кейсы в CI могут пропускаться — это ограничение **среды**, а не постановки.

## 3. Базовый алгоритм

На каждом ранге после получения `local_data` — LSD radix по частям и итеративные слияния (как STL). Глобально при нескольких рангах — последовательность шагов `step = 1,2,4,…`: партнёр `rank ^ step`, обмен размерами и массивами `MPI_Sendrecv`, merge двух отсортированных последовательностей.

## 4. Межпроцессная схема

1. **`MPI_Comm_rank` / `MPI_Comm_size`**.
2. **`ComputeChunkParams`** — размеры кусков и смещения; остаток `n % P` распределяется по первым рангам.
3. **`MPI_Scatterv`** — корень рассылает фрагменты (`ScatterData`).
4. **`ParallelHypercubeMerge`** — обмены и merge с партнёром.
5. На ранге 0 в `data` записывается полный результат; на остальных `data` очищается, затем **`BcastSortedVector`** (размер `size_t`, затем элементы `MPI_INT` с ранга 0).
6. **`MPI_Barrier`** перед финальной проверкой.

## 5. Внутрипроцессная схема

Функция **`SortLocalStlParallel`** в `all/src/ops_all.cpp` совпадает по структуре со **STL**: `num_parts`, `ParallelForRange` для radix и для merge, `std::ranges::merge` в `SimpleMerge`.


## 6. Детали реализации

- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`, класс `SosninaATestTaskALL`.

Фрагмент гиперкуба:

```229:240:tasks/sosnina_a_radix_simple_merge/all/src/ops_all.cpp
void ParallelHypercubeMerge(std::vector<int> &merged_data, int mpi_rank, int mpi_size) {
  int step = 1;

  while (step < mpi_size) {
    const int partner = mpi_rank ^ step;

    if (partner < mpi_size) {
      ExchangeAndMerge(partner, merged_data);
    }

    step <<= 1;
  }
}
```

## 9. Результаты и метрики

Для **`P = 1`** ускорение относительно SEQ: **`S = T_seq / T_all(N)`**, эффективность **`Eff = S / N`**. Для **`P > 1`** в скобках — **`S / (P · N)`**. Ниже — **два** режима каркаса perf на одном входе **20 млн** элементов (типичные значения на Apple M2, Release).

### 9.1. Режим **`task_run`**

| P | N | S vs SEQ | Eff |
| - | - | -------- | --- |
| 1 | 2 | 1,88 | 94% |
| 1 | 4 | 2,28 | 57% |
| 1 | 8 | 3,04 | 38% |
| 4 | 2 | 2,64 | 33% (S/(P·N)) |
| 4 | 4 | 2,56 | 16% (S/(P·N)) |

### 9.2. Режим **`pipeline`**

| P | N | S vs SEQ | Eff |
| - | - | -------- | --- |
| 1 | 2 | 1,74 | 87% |
| 1 | 4 | 2,10 | 52,5% |
| 1 | 8 | 2,80 | 35% |
| 4 | 2 | 2,48 | 31% (S/(P·N)) |
| 4 | 4 | 2,32 | 14,5% (S/(P·N)) |

Агрегированное сравнение с OMP/TBB/STL и пояснения по **`task_run` / `pipeline`** — в корневом **`report.md`**, §5.

## 10. Выводы

**Соответствие постановке.** Ветка **ALL** решает ту же задачу сортировки, что и остальные backend-ы: на каждом ранге после **`MPI_Bcast`** проверяется **`std::ranges::is_sorted`**, а функциональные тесты гарантируют совпадение с эталоном **`std::sort`**. Гибридность проявляется только в **распределении данных** (`Scatterv`), **локальной** многопоточной сортировке (**`std::thread`**) и **межпроцессном** merge гиперкуба.

**Производительность при `P = 1`.** При одном MPI-процессе измеряется в основном та же схема radix + merge, что и в **STL**, плюс неизбежные накладные расходы **`Scatterv`** и рассылки результата **`Bcast`**. В таблице §9.1 видно, что **`Eff`** при **`N = 2`** остаётся **высокой** (около **94%**), а при росте **`N`** падает из‑за барьеров дерева merge и конкуренции за память — картина согласована с потоковыми ветками.

**Режим `pipeline`.** В §9.2 **`S`** и **`Eff`** ниже, чем в **`task_run`**: в замер попадают **`Validation`**, **`PreProcessing`**, **`PostProcessing`**, поэтому относительно SEQ «цена» полного вызова задачи выше. Для отчёта важно хранить **обе** строки: по **`task_run`** удобнее сравнивать чистую сортировку, по **`pipeline`** — поведение в CI.

**Конфигурация `P > 1` на одном узле.** При **`P = 4`** и **`N = 2` или 4** эффективность по **`S/(P·N)`** остаётся **умеренной** (ориентир **~31–33%** при **`N = 2`** и **~14,5–16%** при **`N = 4`** в таблицах §9.1–9.2): процессы и потоки делят одни и те же ядра, а **`ParallelHypercubeMerge`** добавляет обмены. Это не аргумент против MPI как технологии, а следствие выбранной **учебной** конфигурации на одной машине.

**Итог.** ALL оправдан в сценариях, где данные **изначально** распределены по MPI-процессам или узлам; для **одноузловой** сортировки разумнее опираться на **TBB/OpenMP/STL**, а ALL использовать как связку «процессы + потоки» при осмысленном **`P`** и достаточном объёме данных, чтобы амортизировать коммуникацию.
