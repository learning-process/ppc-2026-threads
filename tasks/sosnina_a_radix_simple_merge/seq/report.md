## 1. Контекст

Нужна **последовательная** реализация сортировки массива целых по неубыванию как **эталон корректности** и **базовое
время** для сравнения с параллельными ветками (OpenMP, TBB, STL, ALL). Задача из раздела поразрядной сортировки с
последующим слиянием отсортированных частей.

## 2. Постановка задачи

**Цель.** Дан непустой одномерный массив целых чисел. Требуется построить **перестановку** тех же элементов,
упорядоченную по **неубыванию**.

**Типы.** В `common/include/common.hpp`: **`InType` = `OutType` = `std::vector<int>`**, **`BaseTask` =
`ppc::task::Task<InType, OutType>`**.

**Вход.** Вектор `data` типа `InType`, **`|data| ≥ 1`**. Пустой вектор недопустим: в `ValidationImpl()` возвращается
`false`, если `GetInput().empty()`.

**Выход.** Вектор `OutType` длины **`|data|`**, отсортированный по неубыванию, с тем же **мультимножеством** значений,
что и у входа.

**Проверка корректности.**

- Функциональные тесты в `tests/functional/main.cpp` (**48** кейсов `kStaticTestCases`): сравнение с эталоном после
  **`std::sort`**.
- В конце **`RunImpl()`** — **`std::ranges::is_sorted(data)`** для выходного вектора.

**Особые случаи.** При **`|data| ≤ 1`** массив уже упорядован; полная сортировка не требуется, реализация может
завершиться раньше.

**Предметная связь с реализацией SEQ.** В этой ветке массив делится **пополам**, каждая половина сортируется **LSD
radix** (8 бит × 4 прохода, XOR для знака), затем половины сливаются **устойчивым** `SimpleMerge`. Постановка **не
требует** именно двух частей — это выбранная последовательная декомпозиция; параллельные ветки используют иное число
частей, но **тот же** контракт входа/выхода.

## 3. Базовый алгоритм

1. Массив делится **пополам** (`mid = n/2`): `left_part`, `right_part`.
2. Каждая половина сортируется **LSD radix sort** по 8 бит за проход, 4 прохода для `int`; для знаковых ключей — XOR с
   `0x80000000U` до и после проходов.
3. Две отсортированные половины сливаются **устойчивым** слиянием `SimpleMerge` в исходный вектор `data`.

**Время:** при фиксированной разрядности `O(n)` по числу элементов `n`.  
**Память:** линейная под копии половин и буферы radix.

**Инвариант:** после radix каждая половина отсортирована по ключу; merge сохраняет порядок равных элементов из левой
половины раньше правой.

  size_t idx = 0;
  for (int elem : data) {
    buffer[idx++] = static_cast<int>(static_cast<uint32_t>(elem) ^ kSignFlip);
  }
  std::swap(data, buffer);

  for (int pass = 0; pass < kNumPasses; ++pass) {

Для **SEQ** каркас perf также строит кейс **`pipeline`**: в замер входит полный конвейер

`ppc_perf_tests` (фильтр `*pipeline*…*seq*`); в корневом отчёте §5.3 для **seq** указана ориентировочная доля.

## 8. Выводы

**Роль SEQ.** Последовательная реализация — не «вспомогательный файл», а **эталон смысла** задачи и **единственный
корректный знаменатель** для **`S`** у всех параллельных веток. Любая ошибка в SEQ делает бессмысленными таблицы
ускорения; поэтому постановка, инварианты radix/merge и прохождение всех **48** функциональных тестов здесь критичны.

**Сложность и узкое место.** При фиксированной разрядности ключа время **`RunImpl`** масштабируется как **`O(n)`** по
числу элементов; наибольшую долю времени обычно составляют **четыре прохода LSD** по половинам и **слияние** двух
отсортированных половин. На больших `n` важнее всего **пропускная способность памяти** и работа с буферами radix, а не
мелкая микрооптимизация ветвлений.

**Связь с perf.** На входе **20 млн** элементов **`T_seq`** из **`task_run`** задаёт масштаб для §5 корневого отчёта;
режим **`pipeline`** для SEQ показывает полную цену вызова задачи в инфраструктуре курса и **не должен** подменять
собой **`T_seq`** в формуле **`S`** для параллельных веток без явной перенормировки.

**Итог.** SEQ-версия выполняет поставленную задачу, задаёт **baseline** и подтверждает, что алгоритм radix + merge
**корректен**; дальнейший выбор технологии (OMP/TBB/STL/ALL) — это уже вопрос **масштабирования** при том же
входе/выходе, см. корневой **`report.md`**.
