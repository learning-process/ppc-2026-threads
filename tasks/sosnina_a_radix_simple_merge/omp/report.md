## 1. Контекст

Реализация переносит схему **radix по частям + дерево слияний** на **OpenMP**: независимая поразрядная сортировка частей в параллельном цикле и параллельное слияние пар на каждом уровне. Базовая логика radix/merge совпадает по смыслу с SEQ, декомпозиция — на **до `num_parts`** частей.

## 2. Постановка задачи

**Цель.** Дан непустой одномерный массив целых чисел. Требуется построить **перестановку** тех же элементов, упорядоченную по **неубыванию**.

**Типы.** В `common/include/common.hpp`: **`InType` = `OutType` = `std::vector<int>`**, задача наследует **`BaseTask` = `ppc::task::Task<InType, OutType>`**.

**Вход.** Вектор `data` типа `InType`, **`|data| ≥ 1`**. Пустой вход отклоняется в **`ValidationImpl()`**.

**Выход.** Вектор `OutType` той же длины, **отсортированный по неубыванию**, с совпадающим с входом **мультимножеством** значений.

**Проверка корректности.** Общие функциональные тесты в `tests/functional/main.cpp` (**48** кейсов), сравнение с эталоном **`std::sort`**; в **`RunImpl()`** дополнительно **`std::ranges::is_sorted`**.

**Особые случаи.** При **`|data| ≤ 1`** сортировка тривиальна.

Постановка **полностью совпадает** с корневым `report.md` и с веткой **SEQ**; в ветке **OMP** меняется только способ распараллеливания внутри **`RunImpl`** (и при необходимости детали `PreProcessingImpl` / `PostProcessingImpl` — без изменения контракта входа/выхода).

## 3. Базовый алгоритм

Последовательная суть: LSD radix по 8 бит, 4 прохода, XOR для знака; затем merge отсортированных последовательностей. В OMP массив режется на части, каждая часть проходит тот же `RadixSortLSD`, затем итеративно попарные `SimpleMerge`.

## 4. Схема распараллеливания

1. **`num_parts = min(GetNumThreads(), |data|)`** — число частей и число потоков на фазе radix совпадают (верхняя граница по размеру).
2. **Фаза radix:** `#pragma omp parallel for num_threads(num_parts) default(none) shared(parts, num_parts)` — каждая итерация `i` обрабатывает `parts[i]` с **локальным** буфером в теле цикла (нет гонок по данным между итерациями).
3. **Фаза merge:** цикл по уровням дерева; на каждом уровне `#pragma omp parallel for default(none) shared(current, next) schedule(static)` по парам индексов `i`, внутри — выделение `next[i]` и `SimpleMerge` двух соседних векторов.
4. **`schedule(static)`** — статическое разбиение итераций по потокам; при примерно равных размерах пар на уровне даёт предсказуемое распределение. Неявный барьер в конце каждой `parallel for` области синхронизирует уровень перед следующей итерацией `while`.

**Замечание:** в отличие от STL/TBB, здесь **нет** дополнительных порогов минимального размера части — при большом `GetNumThreads()` части могут быть мельче, растёт число уровней merge.

## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`, класс `SosninaATestTaskOMP`.
- **`default(none)`** — все используемые в регионе переменные явно перечислены в `shared`, что соответствует рекомендациям OpenMP к явности зависимостей.

Фрагмент параллельного radix:

```119:123:tasks/sosnina_a_radix_simple_merge/omp/src/ops_omp.cpp
#pragma omp parallel for num_threads(num_parts) default(none) shared(parts, num_parts)
for (int i = 0; i < num_parts; ++i) {
  std::vector<int> buffer(parts[i].size());
  RadixSortLSD(parts[i], buffer);
}
```

## 9. Выводы

**Схема распараллеливания.** OpenMP переносит radix по частям и merge по уровням в **`parallel for`** с **`default(none)`**, явным **`shared`** и **`schedule(static)`** на merge; число потоков на фазе radix совпадает с **`num_parts = min(GetNumThreads(), n)`**. Неявные барьеры в конце каждого `parallel for` синхронизируют уровни дерева merge — это **упрощает рассуждения о корректности**, но ограничивает масштабирование при большом **`N`**.

**Сравнение с TBB/STL.** Отсутствие отдельной «зернистости» частей (как в TBB/STL) при большом числе потоков может дать **более мелкие** части radix и **глубже** дерево merge; в §8.1 видно, что **`S`** чуть ниже, чем у TBB, и сопоставимо с STL — это согласуется с моделью накладных расходов, а не с ошибкой постановки.

**`task_run` vs `pipeline`.** В §8.2 **`S`** и **`Eff`** ниже: в **`pipeline`** время включает шаги конвейера вне **`RunImpl`**. Для сдачи отчёта оба режима должны **проходить** в CI; в тексте нельзя подменять одно другим.

**Итог.** OpenMP даёт **компактную** и **читаемую** параллельную реализацию с предсказуемым **`schedule(static)`**; для данной задачи на M2 она **уступает** TBB по **`S`**, но остаётся **практичным** выбором при уже используемой в проекте OpenMP-инфраструктуре. Детальные выводы по всем backend-ам — в корневом **`report.md`**.
