## 1. Контекст

В данной работе последовательная реализация алгоритма поразрядной сортировки (Radix Sort) вещественных чисел типа `double` переносится на низкоуровневую модель параллелизма `std::thread` стандартной библиотеки C++.

Поскольку стандартная библиотека не предоставляет встроенного автоматического планировщика (scheduler) или готовых директив параллелизма (как OpenMP или TBB), вся логика организации многопоточности реализована вручную:

- статическое разбиение индексов массива на блоки;
- явное создание и управление жизненным циклом потоков;
- барьерная синхронизация через ожидание завершения работы потоков (`join`);
- использование атомарных операций для предотвращения состояния гонки.

Такая реализация позволяет напрямую контролировать накладные расходы thread-based модели на уровне операционной системы и оценить стоимость частого создания/уничтожения потоков.

Последовательная версия используется как baseline для проверки корректности и вычисления ускорения.

---

## 2. Постановка задачи

Требуется реализовать параллельную поразрядную сортировку массива вещественных чисел двойной точности с простым слиянием с использованием ручного управления потоками через `std::thread`.

Входные данные:

$$A \in \mathbb{R}^{N}  $$

Передаются в виде структуры `std::vector<double>`.

Результирующий массив:

$$B \in \mathbb{R}^{N}  $$

должен содержать упорядоченные по возрастанию элементы ($B[i] \le B[j]$ для всех $i < j$).

Основная цель STL-версии — исследовать производительность ручного распределения вычислительных блоков между потоками ОС и оценить накладные расходы на создание потоков во внутренних циклах алгоритма.

---

## 3. Базовый алгоритм

Алгоритм выполняет 8 последовательных проходов (побайтово) по битовым представлениям чисел `double`. Распараллеливание применяется к этапу сбора гистограммы (подсчета частот) на каждом проходе:

1. Функция `ComputeHistogram` распределяет элементы массива между потоками. Каждый поток обрабатывает свой участок и атомарно обновляет общую таблицу частот байт (256 корзин).
2. Вычисляется префиксная сумма для определения позиций элементов в промежуточном массиве.
3. Элементы перемещаются во временный буфер согласно вычисленным смещениям.
4. По завершении 8 проходов последовательно выполняется финальная коррекция знаков (инверсия порядка отрицательных чисел и слияние их с положительными).

## 4. Схема распараллеливания

### Разбиение диапазонов

Размер порции данных (`chunk_size`) для каждого потока рассчитывается статически перед запуском:

$$chunk\_size = \left\lceil \frac{N}{num\_threads} \right\rceil$$

Каждому потоку с идентификатором `tid` сопоставляется непрерывный непересекающийся диапазон индексов исходного массива:

```cpp
std::size_t start = tid * chunk_size;
std::size_t end = std::min(start + chunk_size, n);

```

### Работа потока

Каждый поток внутри лямбда-выражения обрабатывает элементы из своего диапазона `[start, end)`. Он извлекает байт, соответствующий текущему разряду (пасу), и инкрементирует счетчик в общем массиве атомиков:

```cpp
for (std::size_t i = start; i < end; ++i) {
    auto bits = std::bit_cast<std::uint64_t>(data[i]);
    int byte = static_cast<int>((bits >> (pass * kNumBits)) & 0xFF);
    count[byte].fetch_add(1, std::memory_order_relaxed);
}

```

### Create - Work - Join (Жизненный цикл потоков)

Потоки создаются динамически **на каждом из 8 проходов сортировки** внутри функции `ComputeHistogram`:

```cpp
std::vector<std::thread> threads;
for (unsigned int tid = 0; tid < num_threads; ++tid) {
    threads.emplace_back([&, start, end]() { ... });
}

```

После создания главного пула вычислений, основной поток сразу же ожидает завершения работы всех worker-потоков:

```cpp
for (auto &th : threads) {
    th.join();
}

```

Этот барьер гарантирует, что к моменту расчета префиксных сумм вся гистограмма текущего разряда полностью сформирована.

### Синхронизация и Потокобезопасность

Поскольку потоки могут параллельно увеличивать значения одних и тех же корзин (при совпадении байт в числах), используется массив атомарных переменных:

```cpp
std::array<std::atomic<int>, kRadix> count{};

```

Для минимизации накладных расходов на когерентность кэшей процессора применяется расслабленный порядок доступа к памяти `std::memory_order_relaxed`, который обеспечивает атомарность операции без выставления тяжеловесных барьеров памяти на уровне CPU.

## 5. Детали реализации

### Структура файлов

- `stl/include/ops_stl.hpp` — объявление класса `FrolovaSRadixSortDoubleSTL`;
- `stl/src/ops_stl.cpp` — реализация поразрядной сортировки с использованием технологии STL.

### Основные отличия от SEQ и особенности

В отличие от высокоуровневых абстракций (OpenMP/TBB), данная реализация выполняет явное пересоздание тяжеловесных системных объектов `std::thread` ровно $8$ раз за время работы алгоритма (по разу на каждый байт структуры `double`). Количество потоков автоматически ограничивается сверху размером входного массива данных, чтобы исключить генерацию пустых потоков.

### Работа с памятью

- Промежуточный рабочий вектор `temp` выделяется один раз в начале сортировки.
- Внутри функции `ComputeHistogram` на каждом проходе локально выделяется память под вектор `threads`, что приводит к регулярным обращениям к системному аллокатору памяти.

- Test 2: Проверяет работу с массивом (10 элементов), отсортированным в обратном порядке. Содержит только положительные дробные числа (от 9.5 до 0.5). Это худший случай для алгоритмов сортировки, проверяющий корректность полного перестроения массива.

- Test 3: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из положительных вещественных чисел.

- Test 4: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из отрицательных вещественных чисел. Он показывает, как алгоритм обрабатывает специфику инверсии порядка отрицательных чисел в стандарте IEEE 754.

- Test 5: Проверяет сортировку небольшого смешанного массива (10 элементов). Включает положительные и отрицательные числа, а также нуль, расположенные в случайном порядке.

- Test 6: Проверяет работу алгоритма на массиве (10 элементов) с большим количеством дубликатов. Массив содержит повторяющиеся значения, что тестирует стабильность алгоритма при обработке одинаковых элементов.

- Test 7: Проверяет работу на среднем объеме данных (100 элементов). Содержит случайный разброс положительных и отрицательных чисел разной разрядности.

- Test 8: Стресс-тест на граничные случаи (1000 элементов). Включает числа в экспоненциальной записи (`1e-10`, `3e8`), экстремально большие и малые значения. Этот тест проверяет, насколько корректно побитовые операции работают с границами точности формата `double`.

- Test 9: Объемный тест (1000 элементов), сфокусированный на обработке массовых повторений. Состоит из большого количества дублирующихся значений (как положительных, так и отрицательных), идущих парами (например, `0.5 0.5`, `1.5 1.5`).

- Test 10: Тест на 2000 элементов, проверяющий алгоритм на чередовании знаков и высокой точности. Включает попеременные положительные и отрицательные числа (`-1234.56 789.01...`) , а также длинные последовательности очень малых дробных чисел (например, `0.001 -0.002 0.003...`). Это гарантирует, что финальное разделение и слияние положительной и отрицательной частей массива работает безупречно на больших объемах данных.

## 8. Результаты

Для проведения замеров использовался массив случайных вещественных чисел размером $N = 10^7$ элементов.

| Threads          | Time (s) | Speedup | Efficiency |
| ---------------- | -------- | ------- | ---------- |
| 1 (SEQ baseline) | 0.4820   | 1.00    | 100.0%     |
| 2                | 0.3420   | 1.41    | 70.5%      |
| 4                | 0.2850   | 1.69    | 42.2%      |
| 8                | 0.2990   | 1.61    | 20.1%      |

### Анализ масштабируемости

STL-версия демонстрирует умеренный прирост производительности на двух потоках, однако при увеличении числа потоков до 4 и 8 эффективность распараллеливания резко падает. Основными факторами, сдерживающими масштабируемость, являются:

1. Накладные расходы на `create/join`: Создание потоков ОС — дорогостоящая операция. Поскольку функция `ComputeHistogram` вызывает этот процесс 8 раз подряд, затраты времени на системные вызовы начинают перекрывать выигрыш от параллельных вычислений.
2. Конкуренция за кэш (Cache Bouncing): Атомарные инкременты в массив из всего 256 ячеек при большом числе потоков вызывают постоянные инвалидации кэш-линий процессора, так как ядра непрерывно борются за монопольный доступ к одним и те же адресам памяти.

## 9. Выводы

Низкоуровневая реализация на базе `std::thread` предоставила полный контроль над распределением данных и механизмами синхронизации без скрытого оверхеда тяжелых сторонних runtime-систем.

Однако для мелкозернистых задач, таких как покопийный проход Radix Sort, стратегия создания потоков "на лету" (без пула потоков / thread pool) оказывается неоптимальной. Накладные расходы на операционную систему при вызовах `std::thread` и барьеры `join` на каждом разряде сильно ограничивают потенциальное ускорение алгоритма.

Для достижения максимальной эффективности в таких Memory Bound задачах рекомендуется использовать структуры с сохранением потоков (thread pool) или переходить на блочное распараллеливание (как в OpenMP), где параллельная область открывается единожды на весь процесс сортировки.
