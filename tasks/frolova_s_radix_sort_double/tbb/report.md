## 1. Контекст

В данной работе последовательная реализация алгоритма поразрядной сортировки (Radix Sort) вещественных чисел типа `double` переносится на модель параллелизма oneTBB (Threading Building Blocks).

Задача поразрядной сортировки адаптируется под task-based модель путем параллельного вычисления гистограмм распределения байт на каждом из 8 поразрядных проходов. В отличие от блочного разделения всего массива (как в OpenMP), здесь распараллеливается обработка элементов внутри итераций, а планировщик oneTBB автоматически распределяет итерации цикла по потокам.

Библиотека oneTBB выбрана благодаря:

- эффективной реализации мелкозернистого параллелизма;
- динамической балансировке нагрузки с помощью механизма work stealing;
- высокой производительности при совместном использовании с атомарными операциями;
- отсутствию необходимости вручную управлять пулом потоков на уровне ОС.

Последовательная версия используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать параллельную поразрядную сортировку массива вещественных чисел двойной точности с простым слиянием с использованием библиотеки oneTBB.

Входные данные:

$$A \in \mathbb{R}^{N}  $$

Передаются в виде структуры `std::vector<double>`.

Результирующий массив:

$$B \in \mathbb{R}^{N}  $$

должен содержать те же элементы, упорядоченные по возрастанию ($B[i] \le B[j]$ для всех $i < j$).

Основная цель TBB-версии — исследовать эффективность распараллеливания внутренних этапов алгоритма поразрядной сортировки с помощью `tbb::parallel_for` и оценить влияние атомарных операций на масштабируемость программы.

---

## 3. Базовый алгоритм

Алгоритм выполняет последовательные проходы по байтам представлений чисел `double` (всего 8 проходов, так как `sizeof(uint64_t) == 8`). На каждом проходе (pass):

1. Массив сканируется, из каждого элемента извлекается целевой байт с помощью `std::bit_cast<uint64_t>`, и инкрементируется соответствующий счетчик в таблице частот (256 корзин).
2. Вычисляется префиксная сумма (эксклюзивное сканирование) для определения стартовых позиций элементов каждой корзины в промежуточном массиве.
3. Элементы перемещаются в промежуточный массив `temp` согласно вычисленным позициям.
4. Массивы меняются местами (`working.swap(temp)`).

В конце алгоритма выполняется разделение на положительные и отрицательные числа, инвертирование отрицательной части (из-за специфики кодирования знака в IEEE 754) и их финальное слияние.

## 4. Схема распараллеливания

### Используемый примитив

Для распараллеливания самого вычислительно трудоемкого этапа — подсчета частот байт — используется высокоуровневый примитив:

```cpp
tbb::parallel_for

```

### Диапазон работы

Диапазон индексации задается напрямую через целочисленные границы от `0` до `n` (где `n` — размер сортируемого вектора):

```cpp
tbb::parallel_for(std::size_t{0}, n, [&](std::size_t i) {
    auto bits = std::bit_cast<std::uint64_t>(working[i]);
    int byte = static_cast<int>((bits >> (pass * kNumBits)) & 0xFF);
    count.at(byte).fetch_add(1, std::memory_order_relaxed);
});

```

### Grainsize и Partitioner

Используются параметры разбиения диапазона по умолчанию (`auto_partitioner`). Так как объем работы на каждую итерацию фиксирован и чрезвычайно мал (битовый сдвиг, побитовое И и атомарный инкремент), oneTBB самостоятельно группирует итерации в оптимальные подзадачи (chunks), минимизируя накладные расходы на работу планировщика.

### Синхронизация и устранение гонок данных

Поскольку несколько потоков могут одновременно обрабатывать элементы, у которых совпадает целевой байт, возникает риск гонки данных (data race) при записи в общий массив счетчиков `count`.

Для устранения этой проблемы массив счетчиков объявлен как массив атомарных переменных:

```cpp
std::array<std::atomic<int>, kRadix> count{};

```

Инкремент выполняется через атомарный метод `.fetch_add(1, std::memory_order_relaxed)`. Использование модели упорядочивания памяти `std::memory_order_relaxed` критически важно: оно гарантирует атомарность операции, но отключает дорогостоящую синхронизацию кэшей между ядрами процессора (memory barriers), что существенно снижает overhead.

## 5. Детали реализации

### Структура файлов

- `tbb/include/ops_tbb.hpp` — объявление класса `FrolovaSRadixSortDoubleTBB`;
- `tbb/src/ops_tbb.cpp` — реализация поразрядной сортировки с использованием технологии TBB.

### Основные изменения относительно SEQ

- Массив `count` из обычного `std::vector<int>` заменен на фиксированный `std::array<std::atomic<int>, 256>`.
- Цикл подсчета частот переведен на `tbb::parallel_for`.
- Этапы префиксной суммы и перемещения элементов (`Distribution`) оставлены последовательными, так как их распараллеливание потребовало бы слишком сложных схем синхронизации (например, `tbb::parallel_scan`), накладные расходы от которых на практике превышают выигрыш.

### Работа с памятью

Внутри функции `RunImpl` один раз аллоцируется временный вектор `temp` размера `n`. На каждом проходе динамическое выделение памяти отсутствует, а передача данных происходит через эффективную операцию `working.swap(temp)`. Это минимизирует нагрузку на системный аллокатор памяти.

## 6. Проверка корректности

Для валидации корректности TBB-версии использовались 10 файловых тестов:

- Test 1: Проверяет работу алгоритма с уже отсортированным массивом (10 элементов). Массив включает отрицательные числа, нуль и положительные числа (от -5.0 до 4.0). Это проверяет, не нарушает ли алгоритм правильный порядок элементов и корректно ли он проходит по границе между отрицательными и положительными значениями.

- Test 2: Проверяет работу с массивом (10 элементов), отсортированным в обратном порядке. Содержит только положительные дробные числа (от 9.5 до 0.5). Это худший случай для алгоритмов сортировки, проверяющий корректность полного перестроения массива.

- Test 3: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из положительных вещественных чисел.

- Test 4: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из отрицательных вещественных чисел. Он показывает, как алгоритм обрабатывает специфику инверсии порядка отрицательных чисел в стандарте IEEE 754.

- Test 5: Проверяет сортировку небольшого смешанного массива (10 элементов). Включает положительные и отрицательные числа, а также нуль, расположенные в случайном порядке.

- Test 6: Проверяет работу алгоритма на массиве (10 элементов) с большим количеством дубликатов. Массив содержит повторяющиеся значения, что тестирует стабильность алгоритма при обработке одинаковых элементов.

- Test 7: Проверяет работу на среднем объеме данных (100 элементов). Содержит случайный разброс положительных и отрицательных чисел разной разрядности.

- Test 8: Стресс-тест на граничные случаи (1000 элементов). Включает числа в экспоненциальной записи (`1e-10`, `3e8`), экстремально большие и малые значения. Этот тест проверяет, насколько корректно побитовые операции работают с границами точности формата `double`.

- Test 9: Объемный тест (1000 элементов), сфокусированный на обработке массовых повторений. Состоит из большого количества дублирующихся значений (как положительных, так и отрицательных), идущих парами (например, `0.5 0.5`, `1.5 1.5`).

- Test 10: Тест на 2000 элементов, проверяющий алгоритм на чередовании знаков и высокой точности. Включает попеременные положительные и отрицательные числа (`-1234.56 789.01...`) , а также длинные последовательности очень малых дробных чисел (например, `0.001 -0.002 0.003...`). Это гарантирует, что финальное разделение и слияние положительной и отрицательной частей массива работает безупречно на больших объемах данных.

Все тесты успешно пройдены, результаты полностью совпадают с эталонными.

### Анализ масштабируемости

Параллельная версия TBB демонстрирует стабильный прирост скорости при увеличении числа воркеров, однако масштабируемость уступает линейной. Данное поведение обусловлено спецификой выбранного подхода:

1. Алгоритм Radix Sort является структурой типа Memory Bound (интенсивно нагружает шину памяти).
2. Несмотря на использование `std::memory_order_relaxed`, атомарные операции `fetch_add` при высокой конкуренции потоков за одни и те же ячейки (особенно на массивах с большим количеством дубликатов) вызывают микропаузы на уровне процессорного кэша (cache bouncing).

### Узкие места

- Последовательные участки: Стадии префиксной суммы и распределения элементов по-прежнему выполняются в один поток, что согласно закону Амдала ограничивает максимальное теоретическое ускорение.
- Атомарная конкуренция: Одновременная запись множества потоков в ограниченный массив из 256 атомиков создает аппаратное соперничество за кэш-линии процессора.

## 9. Выводы

Библиотека oneTBB позволила успешно распараллелить внутренний этап подсчета разрядов алгоритма Radix Sort без необходимости полной перестройки структуры программы под блочный вид.

Примитив `tbb::parallel_for` совместно с `std::atomic` и расслабленной моделью памяти (`relaxed`) показал себя как надежный инструмент для оптимизации циклов с частыми инкрементами. Task-based модель TBB эффективно распределяет нагрузку, скрывая затраты на создание потоков операционной системы.

Наибольшая эффективность данной схемы распараллеливания проявляется на огромных массивах с равномерным распределением байт, где вычислительная работа по побитовому разбору доминирует над задержками синхронизации памяти.
