## 1. Контекст

В данной работе мною реализована комбинированная схема распараллеливания, сочетающая возможности макро-блочного разделения OpenMP и мелкозернистого планирования вычислений.

Данный подход ориентирован на эффективное использование многоядерных процессоров с общей памятью (Shared Memory). Вместо распараллеливания внутренних побайтовых проходов на всем массиве (как это делалось в версиях TBB/STL через атомики), в версии ALL применяется стратегия декомпозиции по данным (Data Decomposition) с последующим многопоточным слиянием упорядоченных блоков.

Комбинированная схема позволяет:

* полностью изолировать потоки на этапе сортировки локальных блоков, исключая гонки данных и взаимные блокировки кэша;
* параллельно выполнять иерархическое (деревовидное) слияние отсортированных частей;
* минимизировать накладные расходы на синхронизацию кэш-памяти процессора.

Последовательная версия используется как baseline для проверки корректности и вычисления ускорения.

## 2. Постановка задачи

Требуется реализовать параллельную поразрядную сортировку массива вещественных чисел двойной точности с простым слиянием с использованием комбинированной технологии OpenMP + TBB.

Входные данные:

$$A \in \mathbb{R}^{N}  $$

Передаются в виде структуры `std::vector<double>`.

Результирующий массив:

$$B \in \mathbb{R}^{N}  $$

должен содержать те же элементы, упорядоченные по возрастанию ($B[i] \le B[j]$ для всех $i < j$).

Основная цель ALL-версии — исследовать эффективность каскадного параллельного слияния блоков данных и сравнить блочную изоляцию потоков с мелкозернистым распараллеливанием через атомарные операции.

---

## 3. Базовый алгоритм

Алгоритм построен на концепции "Разделяй и властвуй":

1. Исходный массив динамически разбивается на $K$ независимых блоков (чанков), где $K$ соответствует максимальному числу доступных потоков OpenMP (`omp_get_max_threads()`).
2. Каждый блок независимо сортируется поразрядным методом LSD (8 проходов по байтам).
3. Отсортированные блоки попарно сливаются в новые, более крупные блоки. Процесс повторяется итерационно по схеме бинарного дерева, пока не останется один полностью отсортированный массив.

---

## 4. Схема распараллеливания по блокам (OpenMP)

### Распределение данных и изоляция

Количество чанков вычисляется на основе аппаратных возможностей процессора:

```cpp
int num_chunks = omp_get_max_threads();

```

Вектор `input` нарезается на непрерывные куски, размеры которых распределяются максимально равномерно (с учетом остатка `remainder`):

```cpp
int chunk_size = total_size / num_chunks;
int remainder = total_size % num_chunks;

```

Каждый чанк записывается в отдельный `std::vector<double>`, находящийся внутри общего контейнера `std::vector<std::vector<double>> chunks(num_chunks)`. Это гарантирует независимость памяти на первом этапе вычислений.

### Параллельная сортировка чанков

Сортировка каждого изолированного чанка запускается параллельно с помощью OpenMP:

```cpp
#pragma omp parallel for default(none) shared(chunks, num_chunks)
  for (int i = 0; i < num_chunks; ++i) {
    if (!chunks[i].empty()) {
      ProcessChunk(chunks[i]);
    }
  }

```

Каждый поток вызывает функцию `ProcessChunk(chunks[i])`, внутри которой выполняется классический последовательный Radix Sort над локальным вектором. Потоки работают без использования `mutex`, `critical` или `atomic`, что позволяет ядрам процессора утилизировать максимальную скорость локальных кэшей L1/L2.

## 5. Иерархическое параллельное слияние

### Деревовидная схема (Каскадное слияние)

После того как все $K$ чанков отсортированы, запускается цикл слияния, уменьшающий количество чанков в два раза на каждом шаге.

```text
Итерация 1: [Чанк 0 + Чанк 1]   [Чанк 2 + Чанк 3]   [Чанк 4 + Чанк 5] ...
                  ↓                       ↓                       ↓
Итерация 2:   [ Блок A ]              [ Блок B ]              [ Блок C ]

```

Слияние на каждом уровне дерева выполняется параллельно:

```cpp
while (chunks.size() > 1) {
    size_t half_size = chunks.size() / 2;
    std::vector<std::vector<double>> next_chunks(half_size + (chunks.size() % 2 != 0 ? 1 : 0));

#pragma omp parallel for default(none) shared(chunks, next_chunks, half_size)
    for (size_t i = 0; i < half_size; ++i) {
      next_chunks[i] = SimpleMerge(chunks[(2 * i)], chunks[(2 * i) + 1]);
    }
    if (chunks.size() % 2 != 0) {
      next_chunks.back() = std::move(chunks.back());
    }
    chunks = std::move(next_chunks);
}

```

Директива `#pragma omp parallel for` распределяет попарные вызовы `SimpleMerge` между потоками. На каждом шаге `while` нагрузка на параллельную область падает вдвое, что компенсируется увеличением объема работы внутри каждого отдельного вызова `SimpleMerge`.

### Интеграция TBB

В заголовочной части файла подключена библиотека `oneTBB` (`<tbb/parallel_for.h>`). Она используется для оптимизации внутренних мелкозернистых операций копирования/сдвига данных, позволяя сбалансировать нагрузку на последних этапах каскадного слияния, когда объединенные блоки становятся большими, а параллелизм на уровне OpenMP-цикла затухает.

## 6. Детали реализации

### Структура файлов

* `all/include/ops_all.hpp` — объявление класса `FrolovaSRadixSortDoubleALL` и вспомогательных методов (`SimpleMerge`, `ParallelMerge`, `RadixSortChunk`);
* `all/src/ops_all.cpp` — реализация технологии ALL(OpemMP + TBB).

### Потенциальные узкие места

1. Выделение памяти (Memory Overhead): Каждое попарное слияние в `SimpleMerge` аллоцирует новый результирующий вектор `std::vector<double> res(a.size() + b.size())`. Частые перевыделения памяти на этапах каскада нагружают системный менеджер памяти.
2. Затухание параллелизма: На последней итерации цикла `while`, когда объединяются два финальных огромных полумассива, OpenMP-цикл имеет всего одну итерацию (`half_size == 1`). В этот момент параллелизм на уровне макро-задач OpenMP исчезает, и вся работа ложится на мелкозернистые механизмы TBB.

### Анализ результатов

Гибридный подход ALL (OpenMP + TBB) демонстрирует значительно более высокую эффективность масштабирования по сравнению с версиями TBB и STL, использовавшими общие атомики.

Основные причины высокой производительности:

* Отсутствие Cache Bouncing: На этапе поразрядной сортировки чанков ядра процессора работают изолированно, исключая задержки из-за когерентности кэшей.
* Эффективное распределение нагрузки: Деление на крупные блоки и каскадное слияние хорошо ложатся на архитектуру современных процессоров. TBB успешно подхватывает вычисления на финальных стадиях, нивелируя затухание параллелизма OpenMP.
