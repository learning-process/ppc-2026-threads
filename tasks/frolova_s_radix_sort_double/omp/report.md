## 1. Контекст

В данной работе мною последовательная реализация алгоритма поразрядной сортировки (Radix Sort) вещественных чисел типа `double` переносится на параллельную модель OpenMP.

Идея распараллеливания заключается в блочном разделении исходного массива данных на независимые части (chunks). Каждый поток автономно выполняет поразрядную сортировку своего блока, осуществляет коррекцию знаков по стандарту IEEE 754 и формирует локально отсортированный подмассив. После этого подмассивы последовательно сливаются в единый результирующий массив.

OpenMP выбран как средство распараллеливания благодаря:

* возможности эффективного разделения данных в рамках общей памяти (shared-memory);
* встроенным механизмам управления пулом потоков;
* низким накладным расходам на создание параллельных областей;
* хорошей переносимости между операционными системами.

Последовательная версия (SEQ) используется как baseline для проверки корректности результатов параллельных потоков и вычисления итогового ускорения.

---

## 2. Постановка задачи

Требуется реализовать параллельную поразрядную сортировку массива вещественных чисел двойной точности с простым слиянием с использованием OpenMP.

Входные данные:

$$A \in \mathbb{R}^{N}  $$

Передаются в виде структуры `std::vector<double>`.

Результат:

$$B \in \mathbb{R}^{N}  $$

где элементы упорядочены по возрастанию ($B[i] \le B[j]$ для всех $i < j$).

Подробное описание специфики битового представления `double` и поразрядных проходов приведено в описании SEQ-версии. Основная цель OMP-версии — ускорение процесса сортировки за счет одновременной обработки сегментов массива на разных вычислительных ядрах CPU.

---

## 3. Базовый алгоритм

Каждый поток выполняет изолированную обработку выделенного ему участка данных:

1. Выполняются 8 поразрядных проходов (по байтам) с использованием `std::bit_cast<uint64_t>` для распределения элементов по 256 корзинам.
2. Проводится коррекция порядка отрицательных чисел: элементы разделяются на группы `negative` и `positive`.
3. Локальный массив отрицательных чисел инвертируется через `std::ranges::reverse`.
4. Отрицательные и положительные элементы сливаются обратно, формируя полностью отсортированный кусок (chunk).

После завершения параллельного этапа куски объединяются с использованием стандартного алгоритма слияния двух упорядоченных последовательностей.

## 4. Схема распараллеливания

### Параллельная область

Параллельная область организуется явным вызовом директивы `#pragma omp parallel` с ручным распределением задач по идентификатору потока (`tid`), вместо стандартного цикла `for`:

```cpp
#pragma omp parallel num_threads(num_threads_to_use) default(none) \
    shared(working, chunk_sizes, chunk_offsets, num_threads_to_use)
{
    int tid = omp_get_thread_num();
    if (tid < num_threads_to_use) {
        size_t offset = chunk_offsets[tid];
        size_t size = chunk_sizes[tid];
    }
}

```

Каждому потоку на основе его `tid` сопоставляется уникальный диапазон индексов `offset` и размер блока `size`.

### Shared-переменные

Общими для всех потоков являются:

* `working` — исходный рабочий вектор с данными;
* `chunk_sizes` — массив размеров блоков для каждого потока;
* `chunk_offsets` — массив стартовых смещений блоков в общем векторе;
* `num_threads_to_use` — количество задействованных потоков.

Переменные считываются всеми потоками, а запись производится строго в свои изолированные диапазоны индексов.

### Private-переменные

Локальными для каждого потока (объявленными внутри параллельного блока) являются:

* `tid` — логический номер потока;
* `offset`, `size` — границы зоны ответственности потока;
* `chunk` — копия выделенного сегмента данных, над которой проводятся операции сортировки и знаковой коррекции.

### Синхронизация

Гонки данных полностью исключены на этапе сортировки, так как потоки оперируют над локальными векторами `chunk`. Обратная запись результатов в общий вектор `working` производится по строго разграниченным смещениям:

```cpp
working[offset + i] = chunk[i];

```

Поскольку зоны `[offset, offset + size]` для разных потоков не пересекаются, синхронизация через `critical` или `atomic` внутри параллельной области не требуется.

### Schedule (Распределение нагрузки)

Распределение элементов выполняется по блочному принципу (аналог статического планирования): массив делится на $P$ максимально равных частей до входа в параллельную область. Это обусловлено тем, что вычислительная сложность поразрядной сортировки зависит исключительно от объема данных ($O(N)$) и одинакова для любых наборов вещественных чисел.

### Барьеры

В конце параллельного блока OpenMP выставляет неявный барьер синхронизации. Основной поток (master) продолжает выполнение кода только после того, как все рабочие потоки запишут свои отсортированные куски обратно в вектор `working`. После барьера запускается финальное последовательное слияние.

## 5. Детали реализации

### Структура файлов

* `omp/include/ops_omp.hpp` — объявление OpenMP backend класса задач;
* `omp/src/ops_omp.cpp` — параллельная реализация алгоритма.

### Основные изменения относительно SEQ

* Массив разбивается на порции. Число потоков динамически ограничивается сверху размером массива (чтобы избежать пустых потоков при малых $N$).
* Логика сортировки вынесена в изолированную функцию `ProcessChunk(std::vector<double> &chunk)`.
* Добавлен этап последовательного каскадного слияния результатов работы потоков:

```cpp
for (int i = 1; i < num_threads_to_use; i++) {
    std::vector<double> merged(merged_result.size() + chunk_sizes[i]);
    std::ranges::merge(merged_result, next_chunk, merged.begin());
    merged_result = std::move(merged);
}

```

### Устранённые риски гонок

Риски гонок данных при распределении элементов по корзинам (которые присутствовали бы при попытке распараллелить внутренние циклы Radix Sort) устранены за счет того, что у каждого потока есть свой независимый вектор счетчиков `count` и временный массив `temp` внутри функции `RadixSortChunk`.

## 6. Проверка корректности

Для валидации параллельной версии использовались 10 файловых тестов:

* Test 1: Проверяет работу алгоритма с уже отсортированным массивом (10 элементов). Массив включает отрицательные числа, нуль и положительные числа (от -5.0 до 4.0). Это проверяет, не нарушает ли алгоритм правильный порядок элементов и корректно ли он проходит по границе между отрицательными и положительными значениями.

* Test 2: Проверяет работу с массивом (10 элементов), отсортированным в обратном порядке. Содержит только положительные дробные числа (от 9.5 до 0.5). Это худший случай для алгоритмов сортировки, проверяющий корректность полного перестроения массива.

* Test 3: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из положительных вещественных чисел.

* Test 4: Проверяет базовую сортировку неупорядоченного массива (10 элементов), состоящего только из отрицательных вещественных чисел. Он показывает, как алгоритм обрабатывает специфику инверсии порядка отрицательных чисел в стандарте IEEE 754.

* Test 5: Проверяет сортировку небольшого смешанного массива (10 элементов). Включает положительные и отрицательные числа, а также нуль, расположенные в случайном порядке.

* Test 6: Проверяет работу алгоритма на массиве (10 элементов) с большим количеством дубликатов. Массив содержит повторяющиеся значения, что тестирует стабильность алгоритма при обработке одинаковых элементов.

* Test 7: Проверяет работу на среднем объеме данных (100 элементов). Содержит случайный разброс положительных и отрицательных чисел разной разрядности.

* Test 8: Стресс-тест на граничные случаи (1000 элементов). Включает числа в экспоненциальной записи (`1e-10`, `3e8`), экстремально большие и малые значения. Этот тест проверяет, насколько корректно побитовые операции работают с границами точности формата `double`.

* Test 9: Объемный тест (1000 элементов), сфокусированный на обработке массовых повторений. Состоит из большого количества дублирующихся значений (как положительных, так и отрицательных), идущих парами (например, `0.5 0.5`, `1.5 1.5`).

* Test 10: Тест на 2000 элементов, проверяющий алгоритм на чередовании знаков и высокой точности. Включает попеременные положительные и отрицательные числа (`-1234.56 789.01...`) , а также длинные последовательности очень малых дробных чисел (например, `0.001 -0.002 0.003...`). Это гарантирует, что финальное разделение и слияние положительной и отрицательной частей массива работает безупречно на больших объемах данных.

Параллельный алгоритм успешно прошел все тесты, подтвердив корректность сборки данных из независимых потоков.

### Анализ масштабируемости

Параллельная версия OpenMP демонстрирует прирост производительности при увеличении числа ядер, однако масштабируемость не является строго линейной. Это обусловлено характером задачи: алгоритм Radix Sort сильно ограничен пропускной способностью памяти (Memory Bound), и при одновременном обращении нескольких потоков к RAM возникает конкуренция за шину данных.

### Узкие места

1. Каскадное слияние (Merge): Выполняется последовательно на одном потоке после выхода из параллельной области. Сложность слияния составляет $O(N \cdot P)$, где $P$ — число потоков. При росте числа потоков время, затрачиваемое на сборку массива, начинает отнимать существенную долю от общего времени работы программы.
2. Аллокации памяти: Каждый поток динамически выделяет векторы под локальные куски и массивы коррекции знаков, что создает накладные расходы на уровне аллокатора ОС.
