## 1. Контекст

Ручная реализация через `std::thread` / `std::async` позволяет оценить чистые накладные расходы
операционной системы на управление потоками без применения интеллектуальных пулов (пример - `TBB`).

## 2. Схема распараллеливания

Алгоритм все так же вычисляет количество потоков `limit` (степень двойки) и нарезает массив на чанки.
Для фазы `RadixSort(...)` в цикле создаются задачи через `std::async(std::launch::async, ...)`.
Результирующие `std::future` сохраняются в вектор, после чего основной поток вызывает
барьерную синхронизацию через `get()` для каждого объекта.
Та же логика повторяется во вложенных циклах `OddEvenMerge(...)`.

## 3. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Явный запуск через `std::launch::async` гарантирует создание нового системного потока
(в отличие от `std::launch::deferred`). Мьютексы не используются, поскольку потоки
модифицируют непересекающиеся области `array_` (Data Parallelism).

```c++
  std::vector<std::future<void>> sort;
  for (size_t i = 0; i < limit; i += 1) {
    size_t left = i * chunk_size;
    size_t right = left + chunk_size - 1;

    sort.push_back(std::async(std::launch::async, [this, left, right]() {
      ShemetovDRadixOddEvenMergeSortSTL::RadixSort(this->array_, left, right);
    }));
  }
  for (auto &sorted : sort) {
    sorted.get();
  }
```

## 4. Проверка корректности

Реализация строго соблюдает порядок слияния. Использование барьеров `get()` на каждом шаге `segment`
не дает алгоритму перейти на следующую итерацию сети Бэтчера до полного завершения текущей.

## 7. Выводы

Наблюдается сильное замедление. Это классический антипаттерн в архитектуре высокопроизводительного кода: вложенные
циклы сети Бэтчера выполняют очень мало полезной работы на итерацию, однако STL-версия на каждой итерации динамически
создает и уничтожает системные потоки (`std::async`). Оверхед на `spawn/join`
системных потоков в сотни раз превысил полезное время вычислений.
