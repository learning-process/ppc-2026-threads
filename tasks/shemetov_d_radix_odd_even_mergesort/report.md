# Поразрядная сортировка для целых чисел с четно-нечетным слиянием Бэтчера

- Студент: Шеметов Даниил Олегович
- Группа: 3823Б1ПР3
- Вариант: 18
- Локальные отчеты: `seq/report.md`, `omp/report.md`, `tbb/report.md`, `stl/report.md`, `all/report.md`

## 1. Введение

Проект исследует эф-сть параллелизации поразрядной сортировки со слиянием Бэтчера,
используя технологии `OMP, TBB, STL, MPI + OMP`. Алгоритм имеет жесткую структуру
синхронизации, что помогает выявить накладные расходы в планировщиках потоков.

## 2. Единая постановка задачи

- **Вход:** Размер массива и `std::vector<int>`.
- **Ограничения:** Алгоритм Бэтчера требует массива размера `2^n`.
Логика `padding'а` (дополнение значением `INT_MAX`) реализована на уровне `PreProcessingImpl(...)` базового класса.
- **Выход:** Отсортированный вектор. Корректность подтверждается `std::ranges::is_sorted`.

## 3. Единая методика эксперимента

- CPU: AMD Ryzen 7 5800x
- OS: Windows 11
- Compiler: MSVC
- Build Type: Release

Задача генерируется динамически размером $N = 2^17$ с элементами из равномерного распределения $[-1000000, 1000000]$.

- Сборка и запуск: Сборка и тестирование проводилось на Windows 11

```Bash
cmake -S . -B build
cmake --build build --parallel --config Release

$env:PPC_NUM_THREADS="4"
$env:OMP_NUM_THREADS="4"
$env:PPC_NUM_PROC="2"

.\build\bin\ppc_perf_tests.exe --gtest_filter="*ShemetovDRunPerfTestsThreads*"

mpiexec -n 4 .\build\bin\ppc_perf_tests.exe --gtest_filter="*ShemetovDRunPerfTestsThreads*"
```

- Измерения: Рассматриваются медианные значения для режимов `task_run` и `pipeline`.
Ускорение = `Time(SEQ) / Time(Target)`. Эффективность = Ускорение / Кол-во потоков в работе

## 4. Сводка корректности

Все реализации успешно прошли функциональные тесты. Для предотвращения гонок данных во всех
параллельных версиях поразрядная сортировка выполнялась на независимых индексах с использованием локальных преаллоцированных
буферов, а фаза слияния Бэтчера распараллеливалась исключительно по непересекающимся блокам.

## 6. Интерпретация различий

- **TBB** показал лучший результат. Планировщик TBB `work-stealing` отлично справился с динамическим разбиением
итераций сети Бэтчера через `blocked_range` без необходимости пересоздавать системные потоки на каждом шаге.
- **OMP** показал неоднозначные результаты. Для массива ~131к элементов барьерная синхронизация `#pragma omp barrier`
и неявные барьеры директивы `for` начали потреблять время, сопоставимое с самими вычислениями.
- **STL** показал самое сильное замедлением. Поскольку структура слияния Бэтчера требует множественных вложенных проходов,
создание `std::async` внутри этих циклов привело к катастрофическим накладным расходам на уровень ОС.
- **ALL** показал умеренное ускорение. Задержки на MPI-коммуникации (передача массива)
лишь немного перекрываются локальной внутриузловой OpenMP оптимизацией.

## 7. Заключение

Наилучшей технологией для поразрядной сортировки со слиянием Бэтчера в рамках одного узла является `TBB`,
архитектура которого идеально ложится на мелкие блочные вычисления сети слияния. Версия со стандартными потоками `STL` доказала,
что ручное создание потоков в коротких циклах вредит производительности.
