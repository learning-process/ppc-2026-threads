## 1. Контекст

Распараллелим базовый алгоритм на системах с общей памятью, распределив сортировку чанков и итерации слияния между потоками.

## 2. Постановка задачи

Необходимо адаптировать `SEQ`-версию, разбив массив не просто на две половины,
а предусмортрев возмождность создания множества блоков, но, что важно - количество
которых кратно степени двойки и не превышает лимит доступных аппаратных потоков.

## 3. Схема распараллеливания

1. **Декомпозиция сортировки:** Вычисляется `limit` потоков (ближайшая снизу степень двойки).
Массив делится на `limit` чанков.
2. **Параллельная секция:** Открывается `#pragma omp parallel num_threads(limit)`. Каждый поток вычисляет свои границы
`left` и `right`, аллоцирует локальные буферы и вызывает `RadixSort(...)`.
3. **Барьерная синхронизация:** `#pragma omp barrier` гарантирует, что все чанки отсортированы перед началом слияния.
4. **Слияние Бэтчера:** Внешний цикл увеличивает размер сегмента. Внутренний цикл по смещениям распараллелен через
`#pragma omp for`, который автоматически распределяет независимые операции слияния между потоками.

## 4. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

Используется `default(none)` для строгого контроля области видимости.
Массивы позиций и буферов для `RadixSort(...)` инициализируются внутри параллельной секции, оставаясь `private`,
что исключает состояния гонки при подсчете элементов в корзинах.

```c++
#pragma omp parallel num_threads(limit) default(none) shared(ref_array, ref_power, chunk_size, limit)
  {
    size_t thread_num = omp_get_thread_num();
    size_t left = thread_num * chunk_size;
    size_t right = left + chunk_size - 1;

    std::vector<int> buffer;
    std::vector<int> position;

    RadixSort(ref_array, left, right, buffer, position);
#pragma omp barrier
    for (size_t segment = chunk_size * 2; segment <= ref_power; segment *= 2) {
#pragma omp for
      for (size_t i = 0; i < ref_power; i += segment) {
        OddEvenMerge(ref_array, i, segment);
      }
#pragma omp barrier
    }
  }
```

Для цикла `#pragma omp for` при слиянии Бэтчера планировщик задач не указывается явно, поэтому используется
*распределение по умолчанию* — `schedule(static)`. Это оптимальный выбор для данной задачи, так как итерации
имеют одинаковую вычислительную сложность - статическое распределение блоков уменьшает накладные расходы планировщика.
Редукция `reduction` в данном алгоритме не требуется, так как потоки модифицируют непересекающиеся индексы массива.

## 5. Проверка корректности

Логика проверена на функциональных тестах. Строгое разделение памяти при поразрядной
сортировке и неявные барьеры в конце `omp for` защищают массив от коллизий.

## 7. Результаты

Стоить отметить сейчас и взять на заметку в дальнейшем - как уже сказано ранее, при количестве потоков отличных
от степени двойки (`$env:PPC_NUM_THREADS="1/3/5/6/..."`), это значение автоматически окургляется до степени двойуи
СНИЗУ, что означает, что алгоритм будет вести сея относительно одинакого, например для 2 и 3 потоков, для 4 и 5 и т.д.
*Относитетльно* - потому что мы не мождем однозначно установить поведение системы в конкретный момент.
