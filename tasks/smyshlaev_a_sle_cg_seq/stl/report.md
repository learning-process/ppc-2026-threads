## 1. Контекст

Данная реализация использует стандартную библиотеку потоков C++ (`std::thread`) для низкоуровневого управления
параллелизмом. Цель — продемонстрировать ручное разделение данных и управление жизненным циклом потоков без
использования высокоуровневых планировщиков (как в OpenMP или TBB).

## 2. Постановка задачи

Краткая постановка: итерационное решение СЛАУ $Ax = b$. Последовательная версия из `seq/report.md` принята за эталон с

## 3. Базовый алгоритм

Алгоритм полностью идентичен последовательному эталону. Основные вычислительные блоки (умножение матрицы на вектор,
скалярное произведение) вынесены в отдельные функции, которые распараллеливаются вручную.

## 4. Схема распараллеливания

Использована стратегия статического разбиения диапазонов (static range partitioning):

1. **Разбиение данных**: Функция `GetThreadBounds` вычисляет для каждого потока индексы `start` и `end`. Если общее
   количество элементов $N$ не делится нацело на число потоков, остаток равномерно распределяется между первыми потоками
   (по принципу $+1$ к размеру чанка).
2. **Создание потоков**: Для каждой операции запускается массив `std::vector<std::thread>`.
3. **Хранение результатов**: Для операций редукции (например, скалярное произведение) используются локальные буферы
   `std::vector<double> partial_results`, куда каждый поток пишет свою часть суммы, чтобы избежать состояния гонки (data
   race).
4. **Синхронизация**: Выполняется через явный вызов `join()` для каждого объекта потока перед агрегацией финального
   результата.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`. В коде реализован классический паттерн "Fork-Join".

**Листинг ключевого фрагмента (Ручное разделение и запуск):**

```cpp
// File: stl/src/ops_stl.cpp
for (int i = 0; i < num_threads; ++i) {
  threads.emplace_back([&, i]() {
    int start; int end;
    GetThreadBounds(n, num_threads, i, start, end);
    double local_sum = 0.0;
    for (int j = start; j < end; ++j) {
      local_sum += v1[j] * v2[j];
    }
    partial_results[i] = local_sum;
  });
}
for (auto &t : threads) t.join();
```

_Пояснение:_ В данном фрагменте продемонстрирован расчет границ для каждого `tid`. Использование `partial_results[i]`
гарантирует отсутствие ложного разделения кэша на уровне записи, так как каждый поток пишет в свой слот. Однако создание
и уничтожение потоков происходит при каждом вызове функции.

## 9. Выводы

Реализация на `std::thread` показала отрицательную динамику масштабируемости (замедление при росте числа потоков).
**Анализ причин:**

1. **Overhead на создание потоков:** В методе сопряженных градиентов на каждой итерации (до $2N = 6000$ раз) вызывается
   несколько функций. В текущей реализации потоки создаются и уничтожаются внутри каждой такой функции. Суммарные
   затраты операционной системы на системные вызовы `create_thread` и переключение контекста превышают выигрыш от
   параллельных вычислений.
2. **Отсутствие пула потоков:** В отличие от OMP и TBB, здесь не используется Thread Pool.
3. **Рекомендация:** Для данной задачи ручное управление потоками через `std::thread` без использования пула потоков
   является неэффективным.
