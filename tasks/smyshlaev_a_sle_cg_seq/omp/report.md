## 1. Контекст

Данная работа посвящена распараллеливанию метода сопряженных градиентов с использованием технологии OpenMP. Поскольку
алгоритм CG состоит из векторно-матричных операций, он хорошо поддается распараллеливанию по данным, что позволяет
сократить время решения СЛАУ большой размерности на многопроцессорных системах с общей памятью.

## 2. Постановка задачи

Краткая постановка задачи: решение СЛАУ $Ax = b$ для симметричных положительно определенных матриц. Последовательная
версия из `seq/report.md` принимается за эталон (baseline) с временем выполнения $T_{seq} \approx 0.128$ с для задачи
$3000 \times 3000$.

## 3. Базовый алгоритм

Алгоритм повторяет последовательную логику: итерационное уточнение вектора решения $x$ через вычисление невязки $r$ и
направления $p$. Ключевые вычислительные узлы: умножение матрицы на вектор ($Ap$), скалярное произведение векторов
($(v_1, v_2)$) и линейные комбинации векторов ($v = v + \alpha p$).

## 4. Схема распараллеливания

Для распараллеливания выбрана стратегия декомпозиции по итерациям циклов в вычислительных функциях. Использована
директива `#pragma omp parallel for` со следующими параметрами:

- `default(none)`: Для исключения неявного разделения данных и минимизации ошибок.
- `shared(...)`: Передача векторов и констант всем потокам.
- `reduction(+:...)`: Для агрегации результатов скалярного произведения и суммы квадратов невязки.
- `num_threads(num_threads)`: Явное ограничение числа потоков из переменной окружения `PPC_NUM_THREADS`.
- `schedule(static)` (по умолчанию): Обеспечивает минимальный overhead при распределении равномерной нагрузки (все
  строки матрицы обрабатываются за одинаковое время).

**Схема обработки данных:** Каждый поток обрабатывает свой диапазон строк матрицы и векторов, выполняя независимые
вычисления, после чего результаты синхронизируются через редукцию или неявный барьер в конце параллельной области.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`. В коде реализовано ветвление: если размер задачи слишком мал
($N < threads \times 100$), вызывается последовательный метод `RunSequential()`, чтобы избежать потерь на создание
параллельных областей.

**Листинг ключевого фрагмента (Умножение матрицы на вектор):**

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(n, matrix, p, ap) num_threads(num_threads)
for (int i = 0; i < n; ++i) {
  double sum = 0.0;
  for (int j = 0; j < n; ++j) {
    sum += matrix[(i * n) + j] * p[j];
  }
  ap[i] = sum;
}
```

_Пояснение:_ Внешний цикл по строкам $i$ распределяется между потоками. Внутренний цикл по $j$ выполняется каждым
потоком локально. Благодаря использованию `flat_A_` доступ к `matrix[...]` идет линейно, что эффективно использует
кэш-линии процессора.
