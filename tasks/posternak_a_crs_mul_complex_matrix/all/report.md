## 1. Контекст

Гибридная версия объединяет два уровня параллелизма для эффективного использования как распределённой, так и разделяемой
памяти:

- Межпроцессный уровень (MPI) — распределение строк результирующей матрицы между независимыми процессами
- Внутрипроцессный уровень (OpenMP) — распараллеливание обработки строк внутри каждого процесса

Такой подход позволяет масштабировать вычисления за пределы одного узла, сохраняя при этом преимущества shared-memory
параллелизма внутри узла. Последовательная версия (`seq/report.md`) служит эталоном корректности, а чистые потоковые
версии (`omp/report.md`, `tbb/report.md`, `stl/report.md`) — для сравнения с одноуровневыми подходами.

## 2. Постановка задачи

Краткая постановка идентична описанной в seq/report.md:

- Вход: две разреженные матрицы `A` и `B` типа `std::complex<double>` в формате CRS
- Выход: матрица `C = A × B` в том же формате
- Ограничения: `A.cols == B.rows`, фильтрация элементов с модулем ≤ `1e-12`

Детали алгоритма, инварианты и критерии корректности приведены в последовательном отчёте и не дублируются здесь.

## 3. Базовый алгоритм

Кратко: для каждой строки `i` результата вычисляется скалярное произведение строки `A[i]` на все столбцы `B` через
накопление в `unordered_map`, последующую сортировку по номерам столбцов и запись в результирующий CRS-массив.

Полное описание — в `seq/report.md`, раздел «Базовый алгоритм».

## 4. Межпроцессная схема (MPI)

### Распределение данных

Строки результирующей матрицы распределяются между процессами циклически с учётом остатка:

```cpp
// Файл: all/src/ops_all.cpp — вычисление локального диапазона строк
int rows_per_proc = res.rows / size;
int rem = res.rows % size;
int local_start = (rank * rows_per_proc) + std::min(rank, rem);
int local_end = local_start + rows_per_proc + (rank < rem ? 1 : 0);
int local_count = local_end - local_start;
```

Параметры:

- `rank` — номер текущего процесса в коммуникаторе `MPI_COMM_WORLD`
- `size` — общее число процессов
- `rows_per_proc` — базовое число строк на процесс
- `rem` — остаток, распределяемый по одному дополнительному ряду первым `rem` процессам
- `local_start`, `local_end` — полуоткрытый интервал строк, обрабатываемых данным процессом

Такая схема гарантирует:

- Балансировку нагрузки: разница в числе строк между любыми двумя процессами не превышает 1
- Детерминизм: один и тот же `rank` всегда получает один и тот же диапазон при фиксированных `size` и `res.rows`

### MPI-вызовы и их назначение

| Вызов                                      | Назначение                                                       | Точка синхронизации                                 |
| ------------------------------------------ | ---------------------------------------------------------------- | --------------------------------------------------- |
| `MPI_Comm_rank`                            | Получение номера текущего процесса                               | Нет                                                 |
| `MPI_Comm_size`                            | Получение общего числа процессов                                 | Нет                                                 |
| `MPI_Gatherv` (сбор `local_counts`)        | Сбор локальных подсчётов ненулевых элементов на корневой процесс | Неявный барьер в конце операции                     |
| `MPI_Bcast` (рассылка `index_row`)         | Рассылка структуры результата всем процессам после инициализации | Явный барьер: все процессы ждут завершения рассылки |
| `MPI_Gatherv` (сбор `values`, `index_col`) | Сбор вычисленных данных результата на корневой процесс           | Неявный барьер в конце операции                     |
| `MPI_Bcast` (рассылка результата)          | Рассылка финального результата всем процессам для валидации      | Явный барьер: гарантия согласованности              |

### Обоснование выбора коллективных операций

- `MPI_Gatherv` вместо `MPI_Gather`: количество собираемых элементов (`local_nnz`) различается между процессами из-за
  неравномерной разреженности строк
- `MPI_Bcast` для `index_row` и финального результата: все процессы должны иметь полную структуру матрицы для корректной
  валидации и совместимости с каркасом курса
- Отсутствие `MPI_Barrier` в явном виде: неявные барьеры коллективных операций достаточны для синхронизации; явные
  барьеры добавили бы избыточные задержки

## 5. Внутрипроцессная схема (OpenMP)

### Примитив параллелизма

Внутри каждого процесса используется `#pragma omp parallel for` для параллельной обработки локального диапазона строк:

```cpp
// Файл: all/src/ops_all.cpp — параллельное вычисление локальных строк
#pragma omp parallel for schedule(dynamic) default(none) shared(a, b, res, local_start, local_count, threshold)
for (int i = 0; i < local_count; ++i) {
  ComputeAndWriteRow(a, b, res, local_start + i, threshold);
}
```

### Атрибуты переменных

| Переменная                                                | Атрибут                         | Обоснование                                                           |
| --------------------------------------------------------- | ------------------------------- | --------------------------------------------------------------------- |
| `a`, `b`                                                  | `shared`                        | Только чтение, данные не модифицируются                               |
| `res`                                                     | `shared`                        | Запись в непересекающиеся диапазоны по разным строкам — безопасно     |
| `local_counts`, `local_start`, `local_count`, `threshold` | `shared`                        | Константные или предварительно вычисленные значения                   |
| `i`, `idx_a`, `idx_b`, `col_a`, `col_b`                   | `private` (неявно)              | Локальные счётчики цикла                                              |
| `row_sum`, `sorted`, `pos`                                | `private` (локальные в функции) | Создаются внутри `ComputeAndWriteRow` — каждый поток имеет свою копию |

### Планирование: `schedule(dynamic)`

Выбран динамический планировщик по тем же причинам, что и в omp/report.md:

- Неравномерная вычислительная нагрузка по строкам из-за вариаций в разреженности
- Балансировка между потоками внутри процесса через work-stealing
- Минимизация простоя быстрых потоков

### Отсутствие `reduction`

Операция накопления `row_sum[col_b] += val_a * val_b` выполняется внутри локального `unordered_map`, приватного для
каждой итерации. Конфликтов при записи в `res` нет благодаря непересекающимся диапазонам строк, назначенным разным
потокам.

## 6. Детали реализации

### Файлы

- `all/include/ops_all.hpp` — объявление класса `PosternakACRSMulComplexMatrixALL`
- `all/src/ops_all.cpp` — реализация методов

### Многоуровневая схема вычислений

1. **Фаза 0: Инициализация MPI**
    - Получение `rank` и `size` через `MPI_Comm_rank` / `MPI_Comm_size`
    - Вычисление локального диапазона строк `[local_start, local_end)`

2. **Фаза 1: Локальный подсчёт ненулевых элементов (OpenMP)**
    - Каждый процесс параллельно вычисляет `local_counts` для своих строк
    - Используется `#pragma omp parallel for schedule(dynamic)`

3. **Фаза 2: Сбор глобальной структуры (MPI)**
    - `MPI_Gatherv` собирает `local_counts` на корневой процесс
    - `BuildResultStructure` инициализирует `res.index_row` на корне
    - `MPI_Bcast` рассылает `index_row` всем процессам

4. **Фаза 3: Локальное вычисление значений (OpenMP)**
    - Каждый процесс параллельно вычисляет и записывает значения в `res.values` и `res.index_col`
    - Используется тот же `#pragma omp parallel for`

5. **Фаза 4: Сбор результата (MPI)**
    - `MPI_Gatherv` собирает `values` и `index_col` на корневой процесс
    - `MPI_Bcast` рассылает финальный результат всем процессам для валидации

### Ключевой фрагмент: гибридный параллелизм

```cpp
// Файл: all/src/ops_all.cpp — ядро гибридного выполнения
int rank = 0, size = 1;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

// Вычисление локального диапазона строк
int rows_per_proc = res.rows / size;
int rem = res.rows % size;
int local_start = (rank * rows_per_proc) + std::min(rank, rem);
int local_count = rows_per_proc + (rank < rem ? 1 : 0);

// Фаза 1: локальный подсчёт (OpenMP)
std::vector<size_t> local_counts(local_count);
ComputeLocalCounts(a, b, local_counts, local_start, local_count, kThreshold);

// Фаза 2: сбор структуры (MPI)
GatherCountsToRoot(local_counts, local_count, global_counts, recv_counts, displs, rank);
BuildAndBroadcastStructure(res, global_counts_copy, rank);

// Фаза 3: локальное вычисление (OpenMP)
ComputeLocalRows(a, b, res, local_start, local_count, kThreshold);

// Фаза 4: сбор результата (MPI)
GatherResultData(res, local_start, local_nnz, g_counts, g_displs, rank);
BroadcastResult(res, total_nnz);
```

Пояснение:

- Каждый процесс работает только со своим поддиапазоном строк — нет межпроцессных гонок
- OpenMP-параллелизм внутри процесса ускоряет обработку локального блока
- Коллективные MPI-операции синхронизируют процессы в ключевых точках без явных барьеров
- Финальный `MPI_Bcast` гарантирует, что все процессы имеют идентичный результат для валидации

### Конфигурация теста производительности

- Размер матриц: `10000 × 10000`
- Полуширина ленты: `40` → ~81 ненулевой элемент на строку
- Режимы измерения: `pipeline`, `task_run` (инфраструктура курса)
- Общее число рабочих единиц фиксировано: `ranks × threads = 16`

### Замеры времени (ALL, режим `task_run`)

| Конфигурация  | Всего потоков | Время (с) | `speedup` | `efficiency`, % |
| ------------- | ------------- | --------- | --------- | --------------- |
| MPI=1, OMP=16 | 16            | 1.151     | 1.00×     | 100.0           |
| MPI=2, OMP=8  | 16            | 0.663     | 1.74×     | 87.0            |
| MPI=4, OMP=4  | 16            | 0.344     | 3.35×     | 83.8            |
| MPI=8, OMP=2  | 16            | 0.230     | 5.00×     | 62.5            |

Формулы:

- `speedup = time_baseline / time_all`, где `time_baseline = 1.151 с` (конфигурация 1×16)
- `efficiency = speedup / total_workers × 100%`, где `total_workers = ranks × threads`

### Интерпретация результатов

1. Линейный рост ускорения при увеличении числа рангов (до 4) свидетельствует о хорошей масштабируемости межпроцессного
   уровня: распределение строк между процессами эффективно снижает нагрузку на каждый узел.

2. Падение эффективности при переходе к 8 рангам (62.5%) объясняется:
    - Накладными расходами на коллективные операции `MPI_Gatherv` и `MPI_Bcast`
    - Увеличением времени синхронизации при росте числа участников коммуникатора
    - Ограничениями пропускной способности памяти внутри каждого процесса при уменьшении числа потоков OpenMP

3. Конфигурация `MPI=4, OMP=4` демонстрирует оптимальный баланс: ускорение 3.35× при эффективности 84% — наилучшее
   соотношение производительности и накладных расходов для данной задачи на одном узле.

4. Сравнение с чистыми backend-ами (при 16 потоках):
    - OMP: 4.68×, TBB: 4.67×, STL: 3.14×, ALL: 5.00×
    - ALL показывает наилучшее абсолютное ускорение благодаря комбинации распределения данных (MPI) и эффективного
      внутрипроцессного параллелизма (OpenMP)

## 10. Выводы

1. Гибридная схема (MPI + OpenMP) позволила достичь максимального ускорения ~5.0× при 16 рабочих единицах, превзойдя
   чистые потоковые реализации.

2. Ключевые решения, обеспечившие корректность и производительность:
    - Циклическое распределение строк с учётом остатка для балансировки нагрузки между рангами
    - Использование `MPI_Gatherv` для сбора переменного количества данных от каждого процесса
    - Применение `schedule(dynamic)` в OpenMP для балансировки внутрипроцессной нагрузки
    - Отсутствие явных `MPI_Barrier`: неявные барьеры коллективных операций достаточны и минимизируют простои

3. Ограничения масштабируемости:
    - Накладные расходы на коллективные коммуникации растут с увеличением числа рангов
    - При малом числе потоков OpenMP (2) снижается эффективность внутрипроцессного параллелизма
    - Тестирование на одном узле (loopback) не раскрывает потенциал MPI для распределённых кластеров

4. Рекомендации для реальных кластерных развёртываний:
    - Использовать конфигурацию `ranks = число узлов`, `threads = число ядер на узел`
    - Применять асинхронные коммуникации (`MPI_Isend`/`MPI_Irecv`) для перекрытия вычислений и передачи данных
    - Рассмотреть оптимизацию формата данных для уменьшения объёма передаваемой информации

5. Версия готова к использованию как часть сравнительного анализа в корневом отчёте; метрики `speedup` и `efficiency`
   рассчитаны относительно базовой гибридной конфигурации `MPI=1, OMP=16` с `task_run = 1.151 с`.
