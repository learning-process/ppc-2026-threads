## 1. Введение

Задача умножения разреженных матриц является одной из фундаментальных операций вычислительной линейной алгебры, широко
применяемой в научных расчётах, машинном обучении, компьютерной графике и физическом моделировании. При работе с
матрицами, где большинство элементов равно нулю, использование плотного представления приводит к неэффективному расходу
памяти и вычислительных ресурсов.

Формат `CRS` (`Compressed Row Storage`, строковый формат) позволяет хранить только ненулевые элементы, обеспечивая
существенную экономию памяти. Однако алгоритмы работы с разреженными структурами часто имеют нерегулярный доступ к
памяти и переменную вычислительную нагрузку, что делает их интересным объектом для сравнения различных моделей
параллелизма.

Данный отчёт представляет сравнительный анализ пяти реализаций одной и той же задачи:

- `SEQ`: последовательный эталон
- `OMP`: параллелизм на базе OpenMP
- `TBB`: параллелизм на базе oneTBB
- `STL`: ручное управление потоками через `std::thread`
- `ALL`: гибридная схема `MPI + OpenMP`

Цель работы — не только продемонстрировать корректность каждой реализации, но и честно оценить границы применимости
каждой технологии для данной конкретной задачи.

## 2. Единая постановка задачи

### Входные данные

Две разреженные матрицы `A` и `B` с элементами типа `std::complex<double>`, представленные в формате `CRS`:

```cpp
struct CRSMatrix {
    int rows, cols;
    std::vector<std::complex<double>> values;
    std::vector<int> index_col;
    std::vector<int> index_row;
};
```

Инварианты формата:

- `index_row.size() == rows + 1`
- `index_row[0] == 0`, `index_row[i] <= index_row[i+1]`
- Внутри каждой строки элементы в `index_col` строго возрастают
- Все сохранённые значения удовлетворяют `|value| > 1e-12`

### Выходные данные

Матрица `C = A × B` в том же формате `CRS`, удовлетворяющая тем же инвариантам.

### Ограничения

- `A.cols == B.rows` (согласованность размерностей)
- Элементы с модулем `<= 1e-12` считаются нулевыми и отбрасываются

### Критерий корректности

Результат считается корректным, если для всех `i, j`:

- Если эталонное значение `C[i][j]` имеет модуль `<= 1e-12`, то в `CRS`-представлении этот элемент отсутствует
- Иначе в `CRS`-представлении есть элемент `(i, j)` со значением, отличающимся от эталонного не более чем на `1e-9` по
  модулю

Подробное описание алгоритма, инвариантов и деталей реализации приведено в `seq/report.md` и не дублируется в данном
отчёте.

## 6. Интерпретация различий

### Почему OMP и TBB показывают близкие результаты?

Обе технологии используют схожую схему распараллеливания по строкам и адаптивное планирование (`schedule(dynamic)` в
OpenMP, `auto_partitioner` в TBB). Различия во времени (< 2%) находятся в пределах погрешности измерений и объясняются:

- Разными стратегиями work-stealing
- Особенностями аллокаторов памяти в runtime'ах
- Минорными различиями в коде генерации компилятором

### Почему STL показывает меньшее ускорение?

- Отсутствие пула потоков: каждый вызов `RunImpl` создаёт и уничтожает `std::thread` заново
- Нет адаптивной балансировки: разбиение диапазона фиксировано до начала выполнения
- Более высокий оверхед на `create`/`join` по сравнению с runtime'ами OpenMP/TBB

### Почему ALL достигает максимального абсолютного ускорения?

- Гибридная схема позволяет одновременно использовать распределение данных между процессами (MPI) и эффективный
  shared-memory параллелизм внутри процесса (OpenMP)
- Конфигурация `8×2` минимизирует конкуренцию за память внутри каждого процесса, сохраняя при этом выгоду от
  распределения нагрузки между рангами
- Коллективные операции `MPI_Gatherv`/`MPI_Bcast` имеют приемлемую стоимость на одном узле с коммуникациями через
  loopback

### Почему эффективность падает при росте числа рабочих единиц?

Для всех бэкендов наблюдается снижение `efficiency` при переходе к 16 потокам:

- **Гиперпоточность**: логические потоки делят физические исполнительные ресурсы и кэши
- **Ограничения памяти**: задача становится memory-bound, а не compute-bound
- **Накладные расходы синхронизации**: неявные барьеры, атомарные операции, планировщик

Это типичное поведение для задач с нерегулярным доступом к памяти и не является дефектом реализации.

### Ограничения сравнения

- Все замеры выполнены на одном физическом узле; результаты для `ALL` в распределённом кластере могут отличаться из-за
  сетевых задержек
- Тестовые матрицы имеют регулярную ленточную структуру; для нерегулярных разреженных паттернов балансировка нагрузки
  может измениться
- Не проводилось отключение частотного скейлинга и закрепление за ядрами; это может вносить вариативность в замеры

## 8. Заключение

1. **Лучшая абсолютная производительность**: гибридная версия `ALL` с конфигурацией `8×2` показала максимальное
   ускорение `5.00×` относительно последовательного baseline.

2. **Лучшая эффективность**: конфигурация `ALL: 4×4` демонстрирует оптимальный баланс между ускорением (`3.35×`) и
   эффективностью (`83.8%`), что делает её предпочтительной для задач с умеренными требованиями к времени выполнения.

3. **Простота и переносимость**: версии `OMP` и `TBB` обеспечивают близкие результаты (`~4.7×` ускорения при 16 потоках)
   при минимальных изменениях кода относительно `SEQ`. Выбор между ними может определяться предпочтениями команды или
   доступностью runtime в целевой среде.

4. **Учебная ценность `STL`**: несмотря на меньшее ускорение, реализация на `std::thread` наглядно демонстрирует
   принципы ручного управления потоками, что полезно для понимания цены абстракций более высокого уровня.

5. **Методические выводы**:
    - Честное сравнение технологий требует единой методики замеров, согласованных определений метрик и явного описания
      конфигурации исполнения
    - «Оптимальность» реализации всегда относительна и зависит от целевой платформы, размера задачи и требований к
      эффективности
    - Гибридные схемы оправданы при наличии как распределённой, так и разделяемой памяти; на одном узле выигрыш может
      быть неочевидным из-за накладных расходов коммуникаций

## 10. Приложение

### Ключевые фрагменты кода

```cpp
// Файл: all/src/ops_all.cpp — гибридный параллелизм
int rank = 0, size = 1;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);
// Распределение строк между рангами
int rows_per_proc = res.rows / size;
int rem = res.rows % size;
int local_start = (rank * rows_per_proc) + std::min(rank, rem);
int local_count = rows_per_proc + (rank < rem ? 1 : 0);
// Внутрипроцессный параллелизм (OpenMP)
#pragma omp parallel for schedule(dynamic)
for (int i = 0; i < local_count; ++i) {
  ComputeAndWriteRow(a, b, res, local_start + i, kThreshold);
}
// Сбор результата (MPI)
MPI_Gatherv(...);
MPI_Bcast(...);
```
