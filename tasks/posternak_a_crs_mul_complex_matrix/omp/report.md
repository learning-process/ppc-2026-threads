## 1. Контекст

Версия на базе OpenMP реализует распараллеливание последовательного алгоритма умножения разреженных матриц в формате CRS
с использованием директив компилятора. Выбор OpenMP обусловлен его простотой интеграции в существующий код, минимальными
изменениями исходного текста и эффективной поддержкой циклических конструкций, которые составляют основу вычислений в
данной задаче.

Последовательная версия (`seq/report.md`) служит эталоном корректности и базой для расчёта метрик ускорения (`speedup`)
и эффективности (`efficiency`). В данном отчёте фиксируются все решения по распараллеливанию: атрибуты переменных,
стратегия планирования, точки синхронизации и обоснование отсутствия гонок данных.

## 2. Постановка задачи

Краткая постановка идентична описанной в `seq/report.md`:

- Вход: две разреженные матрицы `A` и `B` типа `std::complex<double>` в формате CRS
- Выход: матрица `C = A × B` в том же формате
- Ограничения: `A.cols == B.rows`, фильтрация элементов с модулем ≤ `1e-12`

Детали алгоритма, инварианты и критерии корректности приведены в последовательном отчёте и не дублируются здесь.

## 3. Базовый алгоритм

Кратко: для каждой строки `i` результата вычисляется скалярное произведение строки `A[i]` на все столбцы `B` через
накопление в `unordered_map`, последующую сортировку по номерам столбцов и запись в результирующий CRS-массив.

Полное описание — в `seq/report.md`, раздел «Базовый алгоритм».

## 4. Схема распараллеливания

### Область параллелизма

Распараллеливание выполнено на уровне строк результирующей матрицы: каждая строка `C[row]` вычисляется независимо от
других, так как не требует данных из соседних строк результата. Это позволяет применить `#pragma omp parallel for` к
внешнему циклу по `row`.

### Атрибуты переменных

Директива параллелизма:

```cpp
// Файл: omp/src/ops_omp.cpp
#pragma omp parallel for default(none) shared(a, b, res, no_zero_rows) schedule(dynamic)
for (int row = 0; row < res.rows; ++row) {
  no_zero_rows[row] = ComputeRowNoZeroCount(a, b, row, kThreshold);
}
```

| Переменная                                | Атрибут                         | Обоснование                                                                                     |
| ----------------------------------------- | ------------------------------- | ----------------------------------------------------------------------------------------------- |
| `a`, `b`                                  | `shared`                        | Только чтение, данные не модифицируются                                                         |
| `res`                                     | `shared`                        | Запись в разные диапазоны `index_row`, `values`, `index_col` по разным `row` — безопасно        |
| `no_zero_rows`                            | `shared`                        | Каждый поток пишет в свой элемент `no_zero_rows[row]` — нет пересечений                         |
| `row`, `idx_a`, `idx_b`, `col_a`, `col_b` | `private` (неявно)              | Локальные счётчики цикла, автоматически приватизируются                                         |
| `row_sum`, `sorted`, `pos`                | `private` (локальные в функции) | Создаются внутри `ComputeRowNoZeroCount` / `ComputeAndWriteRow` — каждый поток имеет свою копию |

### Отсутствие `reduction`

Операция накопления `row_sum[col_b] += val_a * val_b` выполняется внутри локального `unordered_map`, который является
приватным для каждой итерации внешнего цикла. Поэтому `reduction` не требуется: каждая строка результата вычисляется в
изолированном контексте, а конфликтов при записи в `res` нет благодаря предварительному выделению памяти и
непересекающимся диапазонам записи.

### Планирование: `schedule(dynamic)`

Выбран динамический планировщик `schedule(dynamic)` по следующим причинам:

- Количество ненулевых элементов в строках результирующей матрицы может существенно различаться даже для ленточных
  входных данных
- Динамическое распределение итераций позволяет сбалансировать нагрузку между потоками, избегая простоя быстрых потоков
- Размер чанка по умолчанию (1) обеспечивает достаточную гранулярность для данной задачи

Альтернативы:

- `schedule(static)`: мог бы дать меньшие накладные расходы, но рисковал дисбалансом при неравномерной разреженности
- `schedule(guided)`: избыточен для задачи с предсказуемой структурой данных

### Барьеры и синхронизация

- Неявный барьер в конце `#pragma omp parallel for` гарантирует завершение всех вычислений перед выходом из области
  параллелизма
- Конструкция `#pragma omp single` используется для однократного выполнения `BuildResultStructure`, чтобы избежать гонки
  при инициализации структуры результата:

```cpp
// Файл: omp/src/ops_omp.cpp
#pragma omp single
{
  BuildResultStructure(res, no_zero_rows);
}
```

Этот блок выполняется ровно одним потоком; остальные ожидают его завершения на неявном барьере в конце
`single`-конструкции.

### Число потоков

Число потоков ограничивается через переменную окружения `PPC_NUM_THREADS`. Было использовано 2, 4, 8 и 16 потоков.

## 5. Детали реализации

### Файлы

- `omp/include/ops_omp.hpp` — объявление класса `PosternakACRSMulComplexMatrixOMP`
- `omp/src/ops_omp.cpp` — реализация методов

### Изменения относительно SEQ

1. Двухфазная схема вычислений:
    - Фаза 1: параллельный подсчёт количества ненулевых элементов в каждой строке результата
    - Фаза 2: однопоточная инициализация структуры `res` через `#pragma omp single`
    - Фаза 3: параллельная запись вычисленных значений в заранее выделенные диапазоны

2. Выделение вспомогательных функций:
    - `ComputeRowNoZeroCount`: подсчёт ненулевых элементов для строки (используется в фазе 1)
    - `BuildResultStructure`: инициализация `index_row`, `values`, `index_col` (выполняется одним потоком)
    - `ComputeAndWriteRow`: вычисление и запись строки результата (используется в фазе 3)

3. Предварительное выделение памяти:
    - Массивы `res.values` и `res.index_col` резервируются до параллельной записи, что исключает аллокации в критической
      секции и предотвращает гонки

### Ключевой фрагмент: параллельный цикл

```cpp
// Файл: omp/src/ops_omp.cpp — параллельное вычисление строк
#pragma omp parallel for default(none) shared(a, b, res, no_zero_rows) schedule(dynamic)
for (int row = 0; row < res.rows; ++row) {
  no_zero_rows[row] = ComputeRowNoZeroCount(a, b, row, kThreshold);
}

#pragma omp single
{
  BuildResultStructure(res, no_zero_rows);
}

#pragma omp parallel for default(none) shared(a, b, res) schedule(dynamic)
for (int row = 0; row < res.rows; ++row) {
  ComputeAndWriteRow(a, b, res, row, kThreshold);
}
```

Пояснение:

- Первый цикл параллельно оценивает «плотность» каждой строки результата
- Блок `single` гарантирует, что структура `res` инициализируется ровно один раз, без гонок
- Второй цикл параллельно заполняет `res.values` и `res.index_col`, используя заранее вычисленные смещения из
  `res.index_row`
- Все потоки работают с непересекающимися участками памяти, что обеспечивает потокобезопасность без дополнительных
  примитивов синхронизации

### Интерпретация результатов

1. Ускорение ~4× указывает на существенный параллельный потенциал задачи, но эффективность ~31% отражает наличие
   накладных расходов:
    - Синхронизация на неявных барьерах
    - Накладные расходы динамического планировщика
    - Возможный эффект false sharing при записи в соседние элементы `res.values`

2. Режим `pipeline` показывает большее время (0.976 с) из-за дополнительных измерений инфраструктуры; для сравнения
   технологий используется режим `task_run`.

3. Дальнейший рост числа потоков сверх 8 нецелесообразен в связи с гиперпоточностью

## 9. Выводы

1. OpenMP позволил достичь ускорения ~4× на 8(16) потоках при сохранении полной корректности результата.

2. Ключевые решения, обеспечившие корректность:
    - Использование `default(none)` для явного контроля атрибутов переменных
    - Применение `#pragma omp single` для однократной инициализации структуры результата
    - Отсутствие `reduction` благодаря изоляции вычислений по строкам

3. Выбор `schedule(dynamic)` оправдан неравномерной вычислительной нагрузкой по строкам и обеспечил лучшую балансировку
   по сравнению со статическим планированием.

4. Ограничения масштабируемости:
    - Накладные расходы на создание/синхронизацию потоков
    - Потенциальный false sharing при записи в смежные элементы массива
    - Ограничения пропускной способности памяти при работе с большими разреженными матрицами

5. Версия готова к использованию как часть сравнительного анализа в корневом отчёте; метрики `speedup` и `efficiency`
   рассчитаны относительно эталонного SEQ `task_run` = 0.786 с.
