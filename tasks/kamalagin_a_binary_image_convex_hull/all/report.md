## 1. Контекст

ALL-версия демонстрирует совмещение нескольких thread-технологий в одной реализации. В данном проекте это не MPI-версия:
задача имеет `tasks_type: threads`, а в `all/src/ops_all.cpp` используются OpenMP, `std::thread` и TBB внутри одного
процесса.

## 2. Постановка задачи

Постановка совпадает с root report и SEQ baseline: для каждой 4-связной foreground-компоненты бинарного изображения
требуется построить выпуклую оболочку.

Результат ALL сравнивается с теми же ожидаемыми ответами, что и остальные backend-и.

## 3. Базовый алгоритм

Базовая логика остаётся такой же:

1. Проверить входное изображение.
2. Последовательно найти все компоненты функцией `CollectComponents`.
3. Разделить массив компонент на три диапазона.
4. Первый диапазон обработать OpenMP.
5. Второй диапазон обработать `std::thread`.
6. Третий диапазон обработать TBB.
7. Вернуть общий `HullList`.

## 4. Межпроцессная схема

Межпроцессная схема в этой реализации отсутствует. MPI-вызовы, ранги, `MPI_Barrier`, `Broadcast`, `Scatter`, `Gather` и
`Reduce` не используются.

Это важно зафиксировать отдельно: ALL здесь означает совмещение нескольких thread-backend-ов, а не распределение работы
между MPI-процессами.

## 5. Внутрипроцессная схема

Компоненты делятся на три непрерывных диапазона:

```cpp
const size_t b1 = count / 3;
const size_t b2 = (2 * count) / 3;
RunHullRangeOmp(components, hulls, 0, b1);
RunHullRangeStl(components, hulls, b1, b2);
RunHullRangeTbb(components, hulls, b2, count);
```

Внутри первого диапазона используется `#pragma omp parallel for schedule(dynamic)`. Во втором диапазоне создаются потоки
`std::thread`, которые получают непересекающиеся поддиапазоны. В третьем диапазоне используется `tbb::parallel_for` с
`blocked_range<size_t>`.

Этапы OMP → STL → TBB запускаются последовательно, поэтому эта версия не создаёт три runtime одновременно.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.

Потенциальные узкие места:

- `CollectComponents` остаётся последовательным;
- три технологических этапа выполняются последовательно;
- каждый этап имеет собственные накладные расходы runtime;
- при малой стоимости одной компоненты overhead может быть больше полезного выигрыша.

Гонок записи нет, потому что диапазоны `[0, b1)`, `[b1, b2)` и `[b2, count)` не пересекаются, а внутри каждого диапазона
используется запись в уникальный элемент `hulls[i]`.

## 7. Проверка корректности

ALL проверяется на тех же functional-тестах, что и SEQ, OMP, TBB и STL: пустой вход, одна точка, изолированные точки,
линии, прямоугольник и две компоненты.

Согласованность результата обеспечивается тем, что все технологии записывают оболочки в общий `hulls` по исходному
индексу компоненты. Перед сравнением список оболочек нормализуется.

## 8. Экспериментальная среда

- CPU: Intel Core i5-12400F
- OS: Windows
- compiler: MSVC
- build type: `Release`
- performance input: `5000 × 5000`
- `PPC_NUM_THREADS`: `1, 2, 4, 8`
- `PPC_NUM_PROC`: `1`

Конфигурация `ranks × threads` для этой реализации не применяется, потому что MPI отсутствует. Для таблицы используется
только число thread-workers.

Команды запуска:

```powershell
$env:PPC_NUM_THREADS='4'
$env:PPC_NUM_PROC='1'
.\build\bin\ppc_func_tests.exe --gtest_filter="*kamalagin_a_binary_image_convex_hull_all*"
.\build\bin\ppc_perf_tests.exe --gtest_filter="*kamalagin_a_binary_image_convex_hull_all*"
```

## 9. Результаты

| Threads | Time, s | Speedup | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.0386 | 0.96 | 96.1% |
| 2 | 0.0386 | 0.96 | 48.1% |
| 4 | 0.0386 | 0.96 | 24.0% |

ALL не показывает ускорения относительно SEQ. Это ожидаемо для данной структуры: поиск компонент последовательный, а три
parallel-runtime запускаются на разных диапазонах один за другим.

## 10. Выводы

ALL-версия корректно демонстрирует совмещение OpenMP, `std::thread` и TBB внутри одной реализации. Однако для этой
задачи она полезнее как учебная демонстрация, чем как оптимальная схема ускорения. Для реального выигрыша гибридную
версию нужно перестраивать так, чтобы параллелился не только этап `GrahamHull`, но и выделение компонент изображения.
