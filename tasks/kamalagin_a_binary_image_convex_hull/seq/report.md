## 1. Контекст

SEQ-версия является базовой реализацией задачи. Она нужна как эталон корректности для всех параллельных backend-ов и как
baseline для расчёта ускорения.

В этой версии весь pipeline выполняется одним потоком: проверка входа, поиск компонент бинарного изображения и
построение выпуклой оболочки для каждой найденной компоненты.

## 2. Постановка задачи

Входные данные: `BinaryImage` с полями `rows`, `cols`, `data`. Массив `data` хранит пиксели в построчном порядке.
Пиксель со значением `0` считается фоном, любое ненулевое значение считается foreground-пикселем.

Выходные данные: `HullList`, то есть список выпуклых оболочек. Каждая оболочка — вектор точек `Point {x, y}`.

Ограничения:

- `rows >= 0`, `cols >= 0`;
- если `rows == 0` или `cols == 0`, массив `data` должен быть пустым;
- если изображение непустое, должно выполняться `data.size() == rows * cols`;
- максимальный допустимый размер — `8192 × 8192`.

Граничные случаи: пустое изображение, одна точка, две изолированные точки, горизонтальная линия, вертикальная линия,
заполненный прямоугольник, несколько компонент.

## 3. Базовый алгоритм

1. Проверяется корректность размеров изображения и длины массива данных.
2. Создаётся массив меток `label` размера `rows * cols`.
3. Изображение просматривается построчно.
4. Для каждого ещё не посещённого foreground-пикселя запускается итеративный flood fill по 4-соседям.
5. Найденные точки компоненты передаются в алгоритм Грэхема.
6. Полученная оболочка добавляется в `HullList`.

Асимптотика по времени: `O(rows * cols + sum(k_i log k_i))`, где `k_i` — число точек в i-й компоненте.

Асимптотика по памяти: `O(rows * cols)` для массива меток и стека обхода.

Инварианты корректности: каждый foreground-пиксель помечается не более одного раза; каждая найденная компонента
обрабатывается отдельно; алгоритм Грэхема строит оболочку только по точкам одной компоненты.

## 4. Детали реализации

Файлы: `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`.

`ValidationImpl` проверяет размеры изображения, пустой случай, верхнее ограничение `8192 × 8192` и совпадение длины
`data` с произведением размеров.

`PreProcessingImpl` очищает выходной контейнер.

`RunImpl` вызывает последовательную функцию обработки изображения.

`PostProcessingImpl` не выполняет дополнительных преобразований и возвращает `true`.

Ключевой фрагмент:

```cpp
for (int row = 0; row < rows; ++row) {
  for (int col = 0; col < cols; ++col) {
    const size_t idx = img.Index(row, col);
    if (img.data[idx] == 0 || label[idx] != 0) {
      continue;
    }
    FloodFillComponent(img, row, col, label, component_pts);
    Hull hull;
    GrahamHull(component_pts, hull);
    hulls.push_back(std::move(hull));
  }
}
```

Этот фрагмент показывает полный последовательный цикл: обнаружение новой компоненты, сбор её точек и построение
оболочки.

## 5. Проверка корректности

Результат SEQ сравнивается с заранее заданными ожидаемыми ответами в functional-тестах. Перед сравнением список оболочек
нормализуется: точки внутри оболочки сортируются, затем сортируется список оболочек.

Использованные тесты:

- `empty` - ожидается пустой список оболочек;
- `single_pixel` - ожидается оболочка из одной точки `(0, 0)`;
- `two_isolated` - ожидаются две одноточечные оболочки;
- `vertical_line` и `horizontal_line` - ожидаются крайние точки отрезка;
- `filled_rect` - ожидаются четыре угла прямоугольника;
- `two_components` - ожидаются две независимые компоненты.

SEQ-версия задаёт эталонную логику для проверки OMP, TBB, STL и ALL.
