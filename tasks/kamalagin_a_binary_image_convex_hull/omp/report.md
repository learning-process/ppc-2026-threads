- Вариант: 30

## 1. Контекст

OMP-версия переносит в OpenMP ту часть SEQ-алгоритма, которую можно выполнять независимо: построение выпуклых оболочек
для уже найденных компонент.

Поиск компонент остаётся последовательным, потому что он использует общий массив меток и flood fill по изображению.

## 2. Постановка задачи

Постановка совпадает с `seq/report.md`: на вход подаётся бинарное изображение, на выходе требуется список выпуклых
оболочек для всех 4-связных foreground-компонент.

SEQ считается baseline для корректности и расчёта ускорения.

## 3. Базовый алгоритм

Сначала выполняется последовательный сбор компонент функцией `CollectComponents`. После этого каждая компонента
обрабатывается независимо: для её набора точек вызывается `GrahamHull`.

Именно второй этап распараллеливается в OMP-версии.

## 4. Схема распараллеливания

Параллелится цикл по массиву `components`.

Ключевой фрагмент:

```cpp
#pragma omp parallel for default(none) shared(components, hulls, count) schedule(dynamic)
for (int i = 0; i < count; ++i) {
  const size_t idx = static_cast<size_t>(i);
  GrahamHull(components[idx], hulls[idx]);
}
```

Переменные `components`, `hulls` и `count` объявлены как `shared`, потому что они доступны всем потокам. Индекс цикла
является приватным для каждой итерации. `reduction` не требуется, так как каждый поток записывает результат только в
свой элемент `hulls[idx]`.

Используется `schedule(dynamic)`, потому что компоненты могут иметь разный размер, а значит стоимость построения
оболочки для разных компонент может отличаться. Неявный барьер в конце `parallel for` безопасен: после него все элементы
`hulls` уже заполнены.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Относительно SEQ изменён только этап обработки списка компонент. Поиск компонент и алгоритм Грэхема сохраняют ту же
логику.

Гонки устранены за счёт того, что:

- `components` после построения только читается;
- `hulls` заранее изменяет размер через `resize`;
- каждая итерация записывает в уникальную позицию `hulls[idx]`;
- общих аккумуляторов нет.

## 6. Проверка корректности

OMP-версия проверяется тем же набором functional-тестов, что и SEQ: `empty`, `single_pixel`, `two_isolated`,
`vertical_line`, `horizontal_line`, `filled_rect`, `two_components`.

Результат нормализуется перед сравнением, поэтому возможный другой порядок выполнения потоков не влияет на проверку.

## 8. Результаты

| Threads | Time, s | Speedup | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.0395 | 0.94 | 93.9% |
| 2 | 0.0374 | 0.99 | 49.6% |
| 4 | 0.0369 | 1.01 | 25.1% |
| 8 | 0.0382 | 0.97 | 12.1% |

OMP даёт лучший результат при `4` потоках, но ускорение относительно SEQ@1 близко к единице. Основная причина —
последовательный поиск компонент и малая стоимость обработки каждой отдельной компоненты на performance-входе.

## 9. Выводы

OpenMP-версия корректно распараллеливает независимое построение оболочек. Для данной реализации выигрыш ограничен тем,
что самая общая часть pipeline — обход изображения и выделение компонент — остаётся последовательной, а overhead OpenMP
становится заметным на большом числе маленьких компонент.
