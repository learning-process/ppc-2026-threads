## 1. Контекст

TBB-версия использует задачно-ориентированный подход oneTBB для обработки независимых компонент изображения. Как и в
OMP/STL, параллелится только построение выпуклых оболочек после последовательного выделения компонент.

## 2. Постановка задачи

Постановка совпадает с SEQ: для каждой 4-связной foreground-компоненты бинарного изображения нужно построить выпуклую
оболочку.

SEQ используется как baseline для корректности и ускорения.

## 3. Базовый алгоритм

Сначала `CollectComponents` формирует список компонент. Затем для каждой компоненты вызывается `GrahamHull`. В
TBB-версии цикл по компонентам заменён на `tbb::parallel_for`.

## 4. Схема распараллеливания

Выбран примитив `tbb::parallel_for`, потому что обработка каждой компоненты независима и не требует редукции.

Ключевой фрагмент:

```cpp
tbb::parallel_for(tbb::blocked_range<size_t>(0, count), [&](const tbb::blocked_range<size_t> &range) {
  for (size_t i = range.begin(); i != range.end(); ++i) {
    GrahamHull(components[i], hulls[i]);
  }
});
```

Диапазон работы — индексы компонент от `0` до `count`. Используется `blocked_range<size_t>`. Явный `grainsize` не задан,
поэтому разбиение оставлено runtime oneTBB. Это приемлемо для данной версии, потому что количество компонент на
performance-входе большое.

Конкуренция контролируется настройками окружения курса через `PPC_NUM_THREADS`.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Перед запуском `parallel_for` вектор `hulls` изменяет размер до числа компонент. Внутри диапазона каждый worker
записывает только в `hulls[i]`, поэтому mutex или atomic не требуются.

`components` читается параллельно, а каждая компонента передаётся в `GrahamHull` по своему индексу. Общего аккумулятора
нет.

## 6. Проверка корректности

TBB-версия сравнивается с теми же ожидаемыми ответами, что и SEQ. Тесты покрывают пустой вход, одну точку, две
компоненты, коллинеарные случаи и заполненный прямоугольник.

Нормализация результата делает сравнение независимым от порядка выполнения задач.

- CPU: Intel Core i5-12400F

## 8. Результаты

| Workers | Time, s | Speedup | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.0379 | 0.98 | 97.9% |
| 2 | 0.0369 | 1.01 | 50.3% |
| 4 | 0.0372 | 1.00 | 24.9% |
| 8 | 0.0375 | 0.99 | 12.4% |

Лучший результат получен при `2` worker-ах. Дальнейшее увеличение числа потоков почти не улучшает время, потому что
последовательное выделение компонент и overhead планировщика ограничивают масштабирование.

## 9. Выводы

TBB оказался наиболее удачным backend-ом на внесённых замерах, но ускорение небольшое. Для этой задачи TBB удобен тем,
что естественно выражает независимую обработку массива компонент через `parallel_for`, однако общий bottleneck остаётся
в последовательном поиске компонент.
