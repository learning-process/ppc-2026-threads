## 1. Контекст

STL-версия использует ручное управление потоками через `std::thread`. Она нужна для сравнения готовых runtime-подходов
OpenMP/TBB с вариантом, где разбиение диапазона, создание потоков и `join` выполняются явно.

## 2. Постановка задачи

Постановка совпадает с SEQ: требуется построить выпуклую оболочку для каждой 4-связной компоненты бинарного изображения.
SEQ считается эталоном результата.

## 3. Базовый алгоритм

Сначала последовательно выполняется `CollectComponents`. После этого массив компонент делится на непрерывные диапазоны,
и каждый поток строит оболочки для своего диапазона.

## 4. Схема распараллеливания

Данные делятся по индексам массива `components`. Число потоков берётся из настроек курса и ограничивается числом
компонент.

Ключевой фрагмент:

```cpp
std::vector<std::thread> threads;
threads.reserve(num_threads);

for (size_t ti = 0; ti < num_threads; ++ti) {
  const size_t start = offset;
  const size_t end = start + chunk + (ti < remainder ? 1 : 0);
  offset = end;
  threads.emplace_back([&components, &hulls, start, end]() {
    for (size_t i = start; i < end; ++i) {
      GrahamHull(components[i], hulls[i]);
    }
  });
}

for (auto &th : threads) {
  th.join();
}
```

Каждый поток получает диапазон `[start, end)`. Локальные результаты записываются сразу в общий `hulls`, но каждый поток
работает со своими индексами, поэтому гонки записи нет. `join` вызывается после запуска всех потоков, поэтому работа
действительно выполняется параллельно, а не сериализуется.

Синхронизация через `mutex` или `atomic` не используется, потому что общих счётчиков и общих позиций записи нет.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

Карта поток → диапазон задаётся формулой равномерного деления:

- `chunk = count / num_threads`;
- `remainder = count % num_threads`;
- первые `remainder` потоков получают на одну компоненту больше.

Перед запуском потоков `hulls.resize(count)` резервирует место под все результаты.

## 6. Проверка корректности

STL-версия проверяется на общем наборе functional-тестов и сравнивается с теми же ожидаемыми ответами, что и SEQ.

Отсутствие гонок обеспечивается структурой записи: поток с диапазоном `[start, end)` изменяет только элементы
`hulls[start] ... hulls[end - 1]`.

## 8. Результаты

| Threads | Time, s | Speedup | Efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.0387 | 0.96 | 95.9% |
| 2 | 0.0381 | 0.97 | 48.7% |
| 4 | 0.0381 | 0.97 | 24.3% |
| 8 | 0.0369 | 1.01 | 12.6% |

STL показывает лучший результат при `8` потоках, но прирост относительно SEQ минимален. Основная причина — overhead
создания и завершения потоков при небольшой стоимости одной компоненты.

## 9. Выводы

Ручная реализация на `std::thread` корректно делит работу между потоками и не содержит общей записи в один элемент
результата. Однако для данной задачи ручное управление потоками не даёт заметного преимущества перед SEQ, потому что
полезная работа на одну компоненту мала, а выделение компонент остаётся последовательным.
