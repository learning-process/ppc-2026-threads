## 1. Введение

В работе решается задача построения выпуклой оболочки для каждой компоненты бинарного изображения. Foreground-пикселем
считается пиксель со значением, отличным от нуля. Компоненты выделяются по 4-связности, затем для множества точек каждой
компоненты строится выпуклая оболочка алгоритмом Грэхема.

Задача подходит для сравнения разных моделей параллелизма, потому что после выделения компонент их оболочки можно
строить независимо. Последовательная версия используется как эталон корректности и baseline для расчёта ускорения.
Версии OMP, TBB и STL распараллеливают обработку найденных компонент внутри одного процесса, а версия ALL использует
гибрид MPI + OpenMP: компоненты распределяются между MPI-процессами, оболочки строятся на каждом rank с OpenMP.

## 2. Единая постановка задачи

Вход: структура `BinaryImage`, содержащая размеры изображения `rows`, `cols` и одномерный массив `data` в построчном
порядке.

Выход: `HullList`, то есть список выпуклых оболочек. Каждая оболочка хранится как набор точек `Point {x, y}`.

Ограничения:

- `rows >= 0`, `cols >= 0`;
- для пустого изображения массив `data` должен быть пустым;
- для непустого изображения должно выполняться `data.size() == rows * cols`;
- размеры изображения не должны превышать `8192 × 8192`;
- пиксели со значением `0` считаются фоном, остальные пиксели — частью компоненты.

Критерий корректности:

- все foreground-пиксели разбиты на 4-связные компоненты;
- каждый foreground-пиксель относится ровно к одной компоненте;
- для каждой компоненты построена выпуклая оболочка;
- для одноточечной компоненты оболочка состоит из одной точки;
- для коллинеарной компоненты оболочка содержит крайние точки;
- при сравнении результатов порядок компонент и порядок точек нормализуются.

## 3. Единая методика эксперимента

1. Окружение

- CPU: Intel Core i5-12400F
- RAM: 16 ГБ
- OS: Windows
- compiler: MSVC
- CMake build type: `Release`

2. Переменные окружения

- `PPC_NUM_THREADS` использовалась для OMP, TBB, STL и для OpenMP внутри ALL;
- `PPC_NUM_PROC = 1` для SEQ, OMP, TBB, STL;
- для ALL functional- и performance-замеров использовалось `PPC_NUM_PROC = 2` и запуск через `mpiexec -n 2`.

3. Как генерируются или читаются данные

- функциональные тесты используют фиксированные синтетические изображения малого размера;
- performance-тест создаёт детерминированное разреженное бинарное изображение размера `5000 × 5000`;
- в каждой строке performance-изображения выставляются две foreground-точки, поэтому вход содержит около `10000` малых
  компонент;
- одинаковый вход используется для всех backend-ов, что позволяет сравнивать времена выполнения напрямую.

4. Размеры задач

- основной сравниваемый размер performance-входа: `5000 × 5000`;
- число рабочих потоков (`PPC_NUM_THREADS`) для OMP, TBB, STL и ALL: `1`, `2`, `4`, `8`;
- для ALL дополнительно фиксировалось `PPC_NUM_PROC = 2` (число MPI-процессов); в таблицах §5 столбцы `p` для ALL — это
  потоки OpenMP при двух MPI-процессах.

5. Метрики

- `time` — агрегированное время выполнения режима `task_run` по внесённым performance-замерам;
- `speedup = T_seq / T_backend`, где `T_seq = 0.0371 s` для SEQ;
- `efficiency = speedup / workers × 100%`, где `workers` — `PPC_NUM_THREADS` для OMP, TBB, STL и для OpenMP в ALL.

6. Серия замеров

- для сравнения использовались конфигурации `PPC_NUM_THREADS = 1, 2, 4, 8`;
- в таблицы занесены агрегированные значения по выполненным performance-замерам;
- дополнительные прогревочные запуски отдельно не фиксировались, поэтому результаты следует рассматривать как
  практическое сравнение на одной машине, а не как benchmark.

## 4. Сводка корректности

Все реализации проверяются одним набором functional-тестов через общий класс `KamalaginRunFuncTests`. Это означает, что
SEQ, OMP, TBB, STL и ALL получают одинаковые входы и сравниваются с одинаковыми ожидаемыми ответами.

1. Проверка корректности

- проверяется корректная обработка пустого изображения;
- проверяется построение оболочки для одной точки;
- проверяется разделение нескольких несвязанных компонент;
- проверяется обработка горизонтальной и вертикальной коллинеарности;
- проверяется оболочка заполненного прямоугольника;
- перед сравнением результат нормализуется функцией `NormalizeHullList`.

2. Функциональные тесты

- `empty` — пустое изображение;
- `single_pixel` — одна точка;
- `two_isolated` — две изолированные точки;
- `vertical_line` — вертикальный отрезок;
- `horizontal_line` — горизонтальный отрезок;
- `filled_rect` — заполненный прямоугольник;
- `two_components` — две компоненты в одной строке.

3. Ограничения применимости

- поиск компонент выполняется последовательно во всех версиях;
- параллельная часть начинается только после получения списка компонент;
- ускорение зависит от числа компонент и стоимости построения каждой оболочки.

## 6. Интерпретация различий

1. SEQ

Последовательная реализация выполняет полный цикл: проверка входа, поиск 4-связных компонент, построение оболочки
Грэхема для каждой компоненты. Значение `0.0371 s` используется как baseline.

2. OMP

OMP-версия использует `#pragma omp parallel for schedule(dynamic)` для обработки массива компонент. На выбранном входе
выигрыш почти полностью компенсируется накладными расходами OpenMP, потому что компоненты маленькие, а поиск компонент
остаётся последовательным.

3. TBB

TBB-версия использует `tbb::parallel_for` и `tbb::blocked_range`. Лучший результат получен при `2` worker-ах: `0.0369
s`, `speedup ≈ 1.01`. Это самый устойчивый thread-runtime для данной структуры независимых задач, но прирост всё равно
мал из-за последовательного bottleneck.

4. STL

STL-версия вручную делит диапазон компонент между потоками и вызывает `join` после создания всех потоков. Реализация
корректна, но ручное создание потоков даёт заметный overhead на мелких компонентах.

5. ALL

ALL-версия использует MPI + OpenMP. Процесс с `rank = 0` выделяет компоненты и распределяет их между
MPI-процессами через `MPI_Scatter` и `MPI_Scatterv`. Каждый процесс строит локальные оболочки с помощью OpenMP,
после чего результаты собираются через `MPI_Gatherv` и рассылаются через `MPI_Bcast`.

Такой вариант лучше соответствует гибридной модели, но его эффективность зависит от баланса между ценой MPI-обменов и
полезной работой построения оболочек. На входе с маленькими компонентами коммуникационные расходы могут заметно
ограничивать ускорение.

## 7. Репродуцируемость

1. Команды сборки:

```powershell
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON
cmake --build build --config Release --parallel
```

2. Functional-тесты для SEQ, OMP, TBB, STL:

```powershell
$env:PPC_NUM_THREADS='2'
$env:PPC_NUM_PROC='1'
.\build\bin\ppc_func_tests.exe `
  --gtest_filter="*kamalagin_a_binary_image_convex_hull_*:-*kamalagin_a_binary_image_convex_hull_all*"
```

3. Functional-тесты для ALL (обязателен `mpiexec`):

```powershell
$env:PPC_NUM_THREADS='2'
$env:PPC_NUM_PROC='2'
mpiexec -n 2 -env PPC_NUM_THREADS 2 -env OMP_NUM_THREADS 2 `
  .\build\bin\ppc_func_tests.exe --gtest_filter="*kamalagin_a_binary_image_convex_hull_all*"
```

4. Performance `task_run` для SEQ, OMP, TBB, STL:

```powershell
$env:PPC_NUM_THREADS='2'
$env:PPC_NUM_PROC='1'
.\build\bin\ppc_perf_tests.exe `
  --gtest_filter="*kamalagin_a_binary_image_convex_hull_*task_run*:-*kamalagin_a_binary_image_convex_hull_all*task_run*"
```

5. Performance `task_run` для ALL:

```powershell
$env:PPC_NUM_THREADS='4'
$env:PPC_NUM_PROC='2'
mpiexec -n 2 -env PPC_NUM_THREADS 4 -env OMP_NUM_THREADS 4 `
  .\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_kamalagin_a_binary_image_convex_hull_all_enabled*"
```

Пример запуска через runner курса из корня репозитория:

```bash

scripts/run_tests.py --running-type=threads --counts 1 2 4 8

scripts/run_tests.py --running-type=performance
```

## 8. Заключение

1. Лучший результат

На внесённых замерах лучший результат показывает TBB при `2` worker-ах: `0.0369 s`, `speedup ≈ 1.01`.

2. Ограничения сравнения

Главное ограничение — последовательный поиск компонент. Даже если построение оболочек выполняется параллельно,
значительная часть работы остаётся общей для всех backend-ов. Кроме того, performance-вход состоит из большого числа
маленьких компонент, поэтому накладные расходы runtime становятся сравнимы с полезной работой.

3. Возможные улучшения

Дальнейшее ускорение возможно за счёт параллельного алгоритма маркировки компонент, уменьшения количества копирований
вектора точек и подбора входов с более тяжёлыми компонентами, где построение выпуклых оболочек занимает большую долю
времени.
