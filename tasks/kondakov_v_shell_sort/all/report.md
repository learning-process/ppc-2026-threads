## 1. Контекст
Гибридная версия нужна для того, чтобы программу можно было запускать не только на одном компьютере, но и на кластере. MPI берет на себя общение между компьютерами (узлами), а внутри каждого узла потоки задействуют все доступные процессорные ядра.

## 2. Постановка задачи
Задача состоит в сортировке одномерного вектора целых чисел (`std::vector<int>`) по возрастанию в распределенной вычислительной среде. Ограничения: использование $O(N)$ дополнительной памяти. Алгоритм должен корректно обрабатывать крайние случаи. Последовательная версия (SEQ) служит базовым эталоном для сравнения результатов гибридной реализации.

## 3. Базовый алгоритм
Последовательная логика разделения и слияния адаптируется: данные сначала рассылаются по узлам, на каждом узле сортируются параллельно потоками, а затем собираются обратно на главном узле для финального слияния.

## 4. Межпроцессная схема
В коммуникаторе MPI процесс с рангом 0 выполняет роль "мастера". Он хранит исходный массив и в конце собирает ответ. Остальные ранги работают "воркерами".
Исходные данные распределяются с помощью функции `MPI_Scatterv`, так как она умеет делить массив на неравные части (полезно, когда размер массива не делится нацело на число процессов). `MPI_Bcast` используется для рассылки всем процессов глобального размера массива и в конце - готового отсортированного результата. Для сбора частей обратно на мастере применяется `MPI_Gatherv`.

## 5. Внутрипроцессная схема
Внутри каждого процесса (ранга) данные сортируются с помощью пула потоков на основе `std::thread` (STL).
Массив делится на сегменты, индексы которых раздаются потокам. Каждый поток сортирует свой кусок. После этого главный поток процесса сливает эти локальные куски в один отсортированный локальный массив, который затем отдается в MPI.

## 6. Детали реализации
Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
В функции `RunImpl` вызовы расставлены в следующем порядке:
Сначала стоит `MPI_Bcast` для метаданных. Затем `MPI_Scatterv` раздает данные. После отработки внутренних потоков вызывается `MPI_Gatherv`. Наконец, финальный `MPI_Bcast` рассылает готовый ответ. Явных вызовов `MPI_Barrier` в коде нет, так как коллективные функции `Scatterv` и `Gatherv` сами по себе являются точками синхронизации и заставляют процессы ждать друг друга. 
Потенциальные узкие места находятся в функции `MergeManySortedChunks` на ранге 0, которая последовательно сливает все присланные массивы.

## 7. Проверка корректности
Сравнение с SEQ и другими бэкендами показывает совпадение результатов. Тесты на малых и средних данных успешно проходят. Согласованность данных подтверждается тем, что в конце работы функции все ранги получают идентичную копию финального отсортированного массива через Broadcast.

## 8. Экспериментальная среда
Переменные окружения: `PPC_NUM_PROC=2`, `PPC_NUM_THREADS=4` (задает конфигурацию ranks × threads). Сборка Release, компилятор GCC 14.2.0. Команда запуска: `scripts/run_tests.py --running-type=processes`.

## 10. Выводы
Гибридная схема полностью оправдывает себя, когда массив физически не влезает в оперативную память одного компьютера. На малых задачах она не имеет смысла из-за высокой цены коммуникаций.
