## 1. Контекст
Версия с использованием Intel oneTBB выбрана, чтобы показать работу задачно-ориентированного подхода. TBB умеет сам управлять пулом потоков и балансировать нагрузку (work-stealing), что может быть полезно на сложных этапах, таких как иерархическое слияние массивов.

## 2. Постановка задачи
Требуется упорядочить вектор целых чисел, используя библиотеку TBB. Базовым эталоном производительности выступает последовательная версия (SEQ), выдающая результат 0.01425с на 100 тысячах элементов.

## 3. Базовый алгоритм
Последовательная версия делит данные пополам, отдельно сортирует и объединяет. В TBB этот алгоритм расширяется: данные разбиваются на множество блоков, каждый сортируется параллельно, а затем эти блоки попарно сливаются в виде дерева (иерархически).

## 4. Схема распараллеливания
Для выполнения независимых задач используется примитив `oneapi::tbb::parallel_for`.
Диапазон работы задается через `tbb::blocked_range`, который описывает индексы обрабатываемых прогонов. 
Размер grainsize оставлен по умолчанию. TBB сам оптимально делит итерации между доступными воркерами с помощью встроенного балансировщика, который балансирует нагрузку. Конкуренция и общее число потоков ограничиваются с помощью объекта `tbb::global_control`, чтобы не плодить лишние потоки сверх запрошенных через `PPC_NUM_THREADS`.

## 5. Детали реализации
Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`
Для работы с памятью используются обычные векторы `std::vector`. Локальные аккумуляторы здесь не применяются. Вместо этого функция `MakeSortedRuns` создает массив отсортированных кусков, а функция `MergeRunsByLevels` попарно сливает их. Чтобы не копировать память лишний раз, результаты слияния передаются на следующий уровень через `std::move`. 

## 6. Проверка корректности
Сравнение с SEQ показывает полную идентичность ответов. Тесты успешно отрабатывают на границах: массивах с четным и нечетным количеством элементов, что особенно важно для корректной работы алгоритма иерархического попарного слияния.

## 7. Экспериментальная среда
Сборка: Release, GCC 14.2.0. Способ ограничения потоков: `tbb::global_control`. Размер задач: 100,000 элементов. Команда запуска: `python3 scripts/run_tests.py --running-type=performance`.

## 8. Результаты
| workers | time (s) | speedup | efficiency |
|---------|----------|---------|------------|
| 1       | 0.01361  | 0.96    | 96.0%      |
| 2       | 0.00710  | 1.85    | 92.5%      |
| 4       | 0.00471  | 2.80    | 70.0%      |
| 8       | 0.00372  | 3.54    | 44.2%      |

На 4 воркерах ускорение составляет 2.80, что немного уступает OMP из-за балансировщика задач TBB. Однако на 8 потоках TBB показывает лучшую масштабируемость (ускорение 3.54), так как механизм work-stealing эффективно справляется с неравномерной нагрузкой при иерархическом слиянии большего числа фрагментов.

## 9. Выводы
TBB гораздо удобнее для описания сложных параллельных схем (например, попарного слияния дерева). Хотя накладные расходы на управление задачами могут уступать OMP на малых конфигурациях потоков, при увеличении числа воркеров динамическая балансировка TBB показывает себя с лучшей стороны.
