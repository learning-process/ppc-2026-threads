## 1. Введение

Данный проект исследует ускорение алгоритма сортировки Шелла. Эта задача отлично подходит для сравнения параллелизма,
так как алгоритм имеет нелокальный доступ к памяти на ранних стадиях, что обнажает особенности работы с кэшем и
планировщиками потоков в разных технологиях (OpenMP, TBB, STL, MPI).

## 2. Единая постановка задачи

Вход: вектор `std::vector<int>` случайных чисел.
Выход: тот же вектор, отсортированный по возрастанию.
Критерий корректности: побитовое совпадение с результатом функции `std::sort`.

## 3. Единая методика эксперимента

Окружение: CPU AMD Ryzen 5 5625U, RAM 16 GB, OS Ubuntu 22.04.5 LTS (Docker), compiler GCC 14.2.0, сборка Release (-O3).
Управление потоками и процессами шло через переменные `PPC_NUM_THREADS` и `PPC_NUM_PROC`. Данные генерировались случайно
фреймворком `ppc_perf_tests`.
Сравнивались массивы размера 100,000 элементов.
Ускорение (speedup) считалось как $T_{seq} / T_{par}$.
Эффективность (efficiency) считалась как $Speedup / Workers$.
Выполнено по 5 повторов в режиме pipeline, из которых бралось медианное время.

## 4. Сводка корректности

Все бэкенды успешно прошли 12 функциональных тестов, показав 100% совпадение с эталоном SEQ. Ограничений по применению
на данном алгоритме у бэкендов не выявлено, все они корректно обрабатывают пустые и маленькие массивы.

## 6. Интерпретация различий

- **SEQ** показывает базовое время работы одного ядра на алгоритме Шелла. Накладные расходы отсутствуют.
- **OMP** показал лучший результат на 4 потоках (ускорение в 3 раза), но на 8 потоках его производительность
деградировала. Директива `parallel for` с расписанием `static` имеет низкие накладные расходы, но не справляется с
избыточной фрагментацией.
- **TBB** проиграл OMP на 4 потоках, но на 8 потоках (ускорение 3.20) показал лучший результат. Встроенный алгоритм
кражи задач (work-stealing) и иерархическое слияние компенсируют накладные расходы при большом количестве воркеров.
- **STL** (ручные потоки) парадоксальным образом хорошо масштабировался до 8 потоков (ускорение 2.96).
- **ALL** получил самое большое падение эффективности из-за цены упаковки данных и коммуникации через MPI. Выигрыш от
гибридности здесь перекрыт накладными расходами межпроцессного взаимодействия.

## 10. Приложение

Короткий фрагмент последовательного ядра слияния, который используется всеми технологиями:

```cpp
void SimpleMerge(const std::vector<int> &left, const std::vector<int> &right, std::vector<int> &result) {
  size_t i = 0, j = 0, k = 0;
  while (i < left.size() && j < right.size()) {
    if (left[i] <= right[j]) result[k++] = left[i++];
    else result[k++] = right[j++];
  }
}
```

Фрагмент распределения нагрузки в OpenMP с использованием статического планировщика:

```cpp
#pragma omp parallel for default(none) shared(data, segments, parts) num_threads(num_threads) schedule(static)
  for (size_t part = 0; part < parts; ++part) {
    const size_t begin = (part * data.size()) / parts;
    const size_t end = ((part + 1) * data.size()) / parts;
    segments[part] = std::vector<int>(data.begin() + static_cast<std::ptrdiff_t>(begin),
                                      data.begin() + static_cast<std::ptrdiff_t>(end));
    ShellSort(segments[part]);
  }
```
