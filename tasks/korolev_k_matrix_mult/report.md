# Отчёт по лабораторным работам

**Дисциплина:** параллельное программирование (модуль «threads»).  
**Задача:** умножение плотных матриц, алгоритм Штрассена, элементы типа `double`.  
**Вариант:** 1.  
**Исполнитель:** Королёв Кузьма Никитич, группа 3823Б1ПР3, номер задания по списку: 3.

---

## 1. Цель работы

Изучить и реализовать параллельные варианты умнождения квадратных матриц на основе алгоритма Штрассена; сравнить последовательную реализацию с версиями на OpenMP tasks, потоках C++ STL и Intel oneTBB (`parallel_invoke`); обеспечить корректность результатов автоматическими тестами.

---

## 2. Постановка задачи и исходные данные

- **Вход:** размер \(n\), матрицы \(A\) и \(B\) в **row-major**, по \(n \times n\) элементов типа `double`.  
- **Выход:** матрица \(C = AB\) тех же размеров.  
- **Особенности реализации:**  
  - рекурсивное разбиение на подматрицы; на каждом уровне **семь** рекурсивных умножений \(M_1,\ldots,M_7\) вместо восьми в наивном блочном разложении;  
  - если \(n\) не является степенью двойки, матрицы **дополняются нулями** до размера \(n' = \lceil \text{nearest power of } 2\rceil\) (реализовано через `NextPowerOf2`), после умножения берётся левый верхний блок \(n \times n\) результата;  
  - **база рекурсии:** при \(n \le 64\) (`kStrassenThreshold`) выполняется наивное умножение сложности \(O(n^3)\) для снижения накладных расходов на мелких блоках.

Общая логика алгоритма (сложение/вычитание блоков, рекурсия, сборка результата) сосредоточена в `common/include/strassen_impl.hpp`; варианты параллелизма задаются через обёртку `parallel_run` над семью лямбдами задач одного уровня рекурсии.

---

## 3. Краткие теоретические сведения

Классическое умножение двух матриц \(n \times n\) требует порядка \(O(n^3)\) операций. Алгоритм Штрассена одним уровнем рекурсии заменяет восемь подзадач умножения размера \(n/2\) на **семь**, за счёт дополнительных сложений и вычитаний блоков. Асимптотика при больших \(n\) улучшается до \(O(n^{\log_2 7}) \approx O(n^{2.807})\). На практике на малых размерах выгодно переключаться на наивный тройной цикл (как в данной работе при \(n \le 64\)).

Параллелизм естественно вводится на **уровне семи независимых рекурсивных вызовов** внутри одного шага Штрассена: их можно выполнять одновременно на разных потоках, если среда исполнения это поддерживает.

---

## 4. Реализованные варианты

| Обозначение | Технология | Суть параллельного выполнения семи задач |
|-------------|------------|------------------------------------------|
| **SEQ** | последовательно | Цикл по вектору `std::function`: задачи выполняются строго по очереди. |
| **OMP** | OpenMP | Область `#pragma omp parallel` с `#pragma omp single`; для каждой из семи задач создаётся `#pragma omp task`; синхронизация уровня — `#pragma omp taskwait`. Число потоков берётся из `ppc::util::GetNumThreads()`. |
| **STL** | `std::thread` | На каждый из семь лямбд запускается отдельный поток с последующим `join()`. |
| **TBB** | oneTBB | `tbb::parallel_invoke` над семью задачами уровня. |
| **ALL** | MPI | В репозитории подготовлена MPI-ориентированная заготовка для гибридного сценария (умножение на ранге с OpenMP по описанию в `all/PR.md`); в **тестах производительности** для модуля threads подключены варианты SEQ, OMP, TBB и STL. |

Структуры ввода/вывода и контракт задачи описаны в `common/include/common.hpp`. Валидация: положительный \(n\), размеры векторов \(n^2\), выход до запуска пустой; после выполнения — непустой результат.

---

## 5. Тестирование

1. **Функциональные тесты** (`tests/functional/main.cpp`): для размеров \(n \in \{4, 8, 16\}\) результат сравнивается с **эталоном** — наивным умножением в порядке циклов `i–k–j` (удобство для кэша). Допуск по элементам \(10^{-9}\). Запускаются реализации OMP, STL, TBB и SEQ.

2. **Тесты производительности** (`tests/performance/main.cpp`): матрицы **\(256 \times 256\)**, генерация элементов по детерминированным формулам; проверка — снова наивное умножение с тем же допуском. Режимы прогона подставляются через общую инфраструктуру `ppc::util` и настройки `PPC_SETTINGS_korolev_k_matrix_mult`.

Таким образом, корректность параллельных веток подтверждается согласованием с последовательным эталоном, а не только между собой.

---

## 6. Результаты и обсуждение

Абсолютное время выполнения и ускорение \(S = T_{\text{SEQ}} / T_{\text{par}}\) **зависят от процессора, числа потоков, компилятора и нагрузки ОС**. В рамках курса их фиксируют по выводу инфраструктуры прогона производительности на своей машине или из артефактов CI.

Качественно ожидается:

- У **SEQ** минимальные накладные расходы на оркестрацию, но нет параллелизма между семью подзадачами уровня.  
- У **OMP** и **TBB** использование пула потоков / задач OpenMP даёт более предсказуемое поведение под нагрузкой, чем «сырые» `std::thread` на **каждом** рекурсивном уровне (**STL** может создавать большое число короткоживущих потоков из-за рекурсии Штрассена).  
- Порог `kStrassenThreshold = 64` ограничивает глубину дерева задач и снижает overhead на малых блоках.

Для включения в отчёт таблицы с временем достаточно один раз выполнить тесты производительности в среде курса и перенести числа из лога в подраздел «Таблица результатов».

## 7. Выводы

1. Реализовано умножение матриц алгоритмом Штрассена с общим ядром в `strassen_impl.hpp` и четырьмя вариантами параллельного выполнения уровней рекурсии (SEQ, OpenMP tasks, STL threads, TBB `parallel_invoke`), плюс MPI-ориентированный вариант в каталоге `all/` для смешанной технологии.  
2. Дополнение нулями до степени двойки и база наивного умножения при \(n \le 64\) обеспечивают корректность для произвольного \(n\) и практическую эффективность.  
3. Функциональные и производительные тесты верифицируют совпадение с наивным \(O(n^3)\)-эталогом на выбранных размерах.  
4. Для количественного сравнения ускорений необходимо зафиксировать замеры времени на целевом оборудовании при заданном `GetNumThreads()`.
