# Отчёт по лабораторным работам

**Дисциплина:** параллельное программирование (модуль «threads»).  
**Задача:** умножение плотных матриц, алгоритм Штрассена, элементы типа
`double`.  
**Вариант:** 1.  
**Исполнитель:** Королёв Кузьма Никитич, группа 3823Б1ПР3, номер задания по
списку: 3.

---

## 1. Цель работы

Изучить и реализовать параллельные варианты умнождения квадратных матриц на
основе алгоритма Штрассена; сравнить последовательную реализацию с версиями на
OpenMP tasks, потоках C++ STL и Intel oneTBB (`parallel_invoke`); обеспечить
корректность результатов автоматическими тестами.

---

## 2. Постановка задачи и исходные данные

- **Вход:** размер \(n\), матрицы \(A\) и \(B\) в **row-major**, по
  \(n \times n\) элементов типа `double`.
- **Выход:** матрица \(C = AB\) тех же размеров.
- **Особенности реализации:**
  - рекурсивное разбиение на подматрицы; на каждом уровне **семь** рекурсивных
    умножений \(M_1,\ldots,M_7\) вместо восьми в наивном блочном разложении;
  - если \(n\) не является степенью двойки, матрицы **дополняются нулями** до
    размера \(n' = \lceil \text{nearest power of } 2\rceil\) (через
    `NextPowerOf2`), после умножения берётся левый верхний блок \(n \times n\)
    результата;
  - **база рекурсии:** при \(n \le 64\) (`kStrassenThreshold`) выполняется
    наивное умножение сложности \(O(n^3)\) для снижения накладных расходов на
    мелких блоках.

Общая логика алгоритма (сложение/вычитание блоков, рекурсия, сборка результата)
сосредоточена в `common/include/strassen_impl.hpp`; варианты параллелизма
задаются через обёртку `parallel_run` над семью лямбдами задач одного уровня
рекурсии.

---

## 3. Краткие теоретические сведения

Классическое умножение двух матриц \(n \times n\) требует порядка \(O(n^3)\)
операций. Алгоритм Штрассена одним уровнем рекурсии заменяет восемь подзадач
умножения размера \(n/2\) на **семь**, за счёт дополнительных сложений и
вычитаний блоков. Асимптотика при больших \(n\) улучшается до
\(O(n^{\log_2 7}) \approx O(n^{2.807})\). На практике на малых размерах выгодно
переключаться на наивный тройной цикл (как в данной работе при \(n \le 64\)).

Параллелизм естественно вводится на **уровне семи независимых рекурсивных
вызовов** внутри одного шага Штрассена: их можно выполнять одновременно на
разных потоках, если среда исполнения это поддерживает.

## 4. Реализованные варианты

| Обозначение | Технология | Кратко |
| --- | --- | --- |
| **SEQ** | последовательно | Цикл по `std::function`, задачи по очереди. |
| **OMP** | OpenMP | `parallel` + `single`, семь `task`, `taskwait`. |
| **STL** | `std::thread` | Семь лямбд — семь потоков и `join()`. |
| **TBB** | oneTBB | `tbb::parallel_invoke` на уровень. |
| **ALL** | MPI | Гибрид на ранге с OpenMP; в perf-тестах threads — SEQ…STL. |

Строка **OMP:** потоки из `ppc::util::GetNumThreads()`. Область
`#pragma omp parallel`, внутри `#pragma omp single`, на каждую подзадачу
`#pragma omp task`, синхронизация — `#pragma omp taskwait`.

Структуры ввода/вывода и контракт задачи — в `common/include/common.hpp`.
Валидация: положительный \(n\), размеры векторов \(n^2\), выход до запуска
пустой; после выполнения — непустой результат.

## 5. Тестирование

1. **Функциональные тесты** (`tests/functional/main.cpp`): для
   \(n \in \{4, 8, 16\}\) сравнение с **эталоном** — наивным умножением в порядке
   `i–k–j`. Допуск \(10^{-9}\). Охвачены OMP, STL, TBB и SEQ.

2. **Тесты производительности** (`tests/performance/main.cpp`): матрицы
   \(256 \times 256\), детерминированная генерация; проверка тем же эталоном.
   Режимы — через `ppc::util` и `PPC_SETTINGS_korolev_k_matrix_mult`.

Корректность параллельных веток подтверждается согласованием с последовательным
эталоном.

---

## 6. Результаты и обсуждение

Абсолютное время и ускорение \(S = T_{\text{SEQ}} / T_{\text{par}}\) зависят от
CPU, числа потоков, компилятора и нагрузки ОС. В курсе их фиксируют по логу
прогона производительности или артефактам CI.

Качественно ожидается:

- У **SEQ** малые накладные расходы, но нет параллелизма между семью подзадачами
  уровня.
- У **OMP** и **TBB** пул потоков/задач предсказуемее, чем «сырые»
  `std::thread` на каждом уровне рекурсии (**STL** даёт много коротких потоков
  из-за рекурсии Штрассена).
- Порог `kStrassenThreshold = 64` ограничивает глубину дерева задач и накладные
  расходы на малых блоках.

Таблицу времён можно добавить после одного прогона perf-тестов и переноса чисел
из лога.

## 7. Выводы

1. Реализовано умножение Штрассена с ядром в `strassen_impl.hpp` и вариантами
   SEQ, OpenMP tasks, STL threads, TBB `parallel_invoke`; отдельно
   MPI-ориентированный сценарий в `all/`.
2. Дополнение до степени двойки и база \(n \le 64\) дают корректность для
   произвольного \(n\) и запас по эффективности.
3. Функциональные и производительные тесты сверяют результат с наивным
   \(O(n^3)\)-эталогом на выбранных размерах.
4. Для численного сравнения ускорений нужны замеры на целевом железе при
   заданном `GetNumThreads()`.
