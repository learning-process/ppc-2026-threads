## 1. Введение

Задача курса — реализовать численное интегрирование функции d переменных по прямоугольной области составной формулой
Симпсона и сравнить несколько моделей параллелизма (SEQ, OpenMP, oneTBB, `std::async`, MPI + локальная
многопоточность в all) на одной и той же постановке и общих тестах (`tests/functional/main.cpp`,
`tests/performance/main.cpp`).

## 2. Единая постановка задачи

Вход: границы a, b, вектор чётных n_i, функция f. Выход: скаляр — приближение интеграла.

Корректность: совпадение с последовательным эталоном (SEQ) на детерминированных тестах с допусками курса.

Ограничения: согласованность размерностей; a_i < b_i; n_i положительные чётные; f определена во всех узлах
сетки ∏(n_i+1).

## 3. Единая методика эксперимента

Окружение: CPU AMD Ryzen 7 7840HS; ОС Windows 11; g++ 13.3.0; сборка Release.

Переменные окружения: `PPC_NUM_THREADS` (пробрасывается в `OMP_NUM_THREADS` для OpenMP); для MPI —
`PPC_NUM_PROC`; лимиты времени — `PPC_TASK_MAX_TIME`, `PPC_PERF_MAX_TIME`.

Данные: в функциональных тестах параметры задаются в коде; для производительности использовался сценарий курса с
трёхмерной функцией Гаусса и сеткой 100^3 (см. тесты производительности задачи).

Метрики:

- Ускорение относительно SEQ: S = T_seq / T_realization (один и тот же размер задачи).
- Эффективность для потоковых версий: E = S / T_раб, где T_раб — число потоков OpenMP/TBB или число
  параллельных задач `std::async` на замере; для ALL отдельно фиксируются ranks и локальные потоки.

Повторяемость: несколько прогонов тестов производительности; в таблице ниже — одно из полученных значений.

## 4. Сводка корректности

Все включённые в `settings.json` реализации (seq, omp, tbb, stl, all) проверяются общим набором
функциональных тестов из `tests/functional/main.cpp` (1D/2D/3D, разные функции f).
По результатам функционального прогона значения всех реализаций численно согласованы с SEQ в пределах тестовых допусков.

## 5. Производительность

Сценарий: 3D Гаусс, сетка 100^3 (см. локальные отчёты).

| реализация | режим (потоки / процессы) | размер (узлы/параметр) | время, с | ускорение S относительно SEQ |
| ------- | ------------------------ | --------------------- | ------- | -------------------- |
| seq | 1 | 3D 100^3 | 0.108102 | 1.00 |
| omp | OpenMP | 3D 100^3 | 0.064491 | 1.68 |
| tbb | `parallel_reduce` | 3D 100^3 | 0.066552 | 1.62 |
| stl | `std::async` | 3D 100^3 | 0.054318 | 1.99 |
| all | MPI + async (P=1) | 3D 100^3 | 0.05765 | 1.88 |

Интерпретация: на этой конфигурации STL дала наилучшее время; OMP и TBB близки. Разница не означает
«оптимальность» в общем случае — влияют планировщик ОС, кэш и фактическое T.

## 6. Интерпретация различий

- SEQ — опорная линия по времени и эталон значения.
- OMP — предсказуемое статическое разбиение idx; reduction убирает гонки.
- TBB — снижение ручного кода за счёт parallel_reduce; накладные расходы рантайма сопоставимы с OpenMP на данной
  задаче.
- STL — явное нарезание диапазонов и async; на замере ускорение выше, но оценка должна подкрепляться серией
  повторов.
- ALL — выигрыш ожидается при P>1; на одном процессе доминирует та же арифметика, плюс возможные накладные расходы
MPI.

## 7. Репродуцируемость

Команды в духе документации курса (из корня репозитория, пути уточнить по своему `CMake`):

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
# Пример: потоки
export PPC_NUM_THREADS=4
python scripts/run_tests.py --running-type=threads --counts 1 2 4
# Пример: MPI / процессы
export PPC_NUM_PROC=2
python scripts/run_tests.py --running-type=processes --counts 2 4
# Производительность
python scripts/run_tests.py --running-type=performance
```

Точные цели бинарников и фильтры gtest — по актуальной структуре репозитория `ppc-2026-threads`.

## 8. Заключение

Поставленная цель — параллельные реализации одной квадратуры и сравнение реализаций на общей методике — выполнена.
Наиболее быстрой на одном наборе замеров оказалась STL-версия; OMP и TBB дают сходное ускорение около 1.6x.
Уточнение ускорения и эффективности требует фиксации `PPC_NUM_THREADS` и серии повторов.

Ограничения: результаты зависят от железа и фона ОС; ALL нужно оценивать отдельно при выбранных P и нагрузке сети.
