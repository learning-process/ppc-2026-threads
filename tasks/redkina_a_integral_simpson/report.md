## 1. Введение

В этой задаче реализовано численное интегрирование функции d переменных по прямоугольной области составной формулой
Симпсона. Для сравнения написаны последовательная версия и четыре параллельные реализации: OpenMP, oneTBB,
`std::async` и гибридная MPI + `std::async` версия в каталоге `all`.

Все реализации работают с одним типом входных данных и проверяются общими тестами из `tests/functional/main.cpp` и
`tests/performance/main.cpp`. Поэтому в отчёте сравниваются именно технологии распараллеливания, а не разные постановки
задачи.

## 2. Единая постановка задачи

Вход: границы a, b, вектор чётных n_i, функция f. Выход: скаляр — приближение интеграла.

Корректность: совпадение с последовательным эталоном (SEQ) на детерминированных тестах с допусками курса.

Ограничения: согласованность размерностей; a_i < b_i; n_i положительные чётные; f определена во всех узлах
сетки ∏(n_i+1).

## 3. Единая методика эксперимента

Окружение: CPU AMD Ryzen 7 7840HS; ОС Windows 11; g++ 13.3.0; сборка Release.

Переменные окружения: `PPC_NUM_THREADS` (пробрасывается в `OMP_NUM_THREADS` для OpenMP); для MPI —
`PPC_NUM_PROC`; лимиты времени — `PPC_TASK_MAX_TIME`, `PPC_PERF_MAX_TIME`.

Данные: в функциональных тестах параметры задаются прямо в коде. Для производительности используется трёхмерная функция
Гаусса на области [-1, 1]^3 с разбиением `n = {200, 200, 200}`.

Метрики:

- Ускорение относительно SEQ: S = T_seq / T_realization (один и тот же размер задачи).
- Эффективность для потоковых версий: E = S / T_раб, где T_раб — число потоков OpenMP/TBB или число
параллельных задач `std::async` на замере; для ALL отдельно фиксируются ranks и локальные потоки.

Повторяемость: несколько прогонов тестов производительности; в таблице ниже — одно из полученных значений.

## 4. Сводка корректности

Все включённые в `settings.json` реализации (seq, omp, tbb, stl, all) проверяются одним набором из 20 функциональных
тестов. В каждом тесте задаются функция, границы интегрирования, число разбиений и ожидаемое значение; проверка проходит
с точностью `1e-6`.

Покрытие функциональных тестов:

| Группа | Что проверяется | Примеры |
| ------ | --------------- | ------- |
| 1D | точность на простых функциях | 1, x, x^2, x^3, x^4, sin(x), exp(x) |
| 2D | произведения и суммы переменных | 1, x*y, x^2+y, x*y^2, exp(x+y), sin(x+y) |
| 3D | переход к трём измерениям | 1, x*y*z, x^2+y^2+z^2, sin(x)cos(y)exp(z) |

Отдельно проверяются разные области интегрирования: [0, 1], [0, pi], [-1, 1]. Для полиномов низкой степени метод
Симпсона даёт точный результат, а для `sin` и `exp` используются достаточно мелкие сетки (`n = 40` или `n = 200`).
По результатам прогона значения всех реализаций согласованы с ожидаемыми значениями.

## 5. Производительность

Сценарий: 3D Гаусс, область [-1, 1]^3, `n = {200, 200, 200}` (см. `tests/performance/main.cpp`).

| реализация | режим (потоки / процессы) | размер (узлы/параметр) | время, с | ускорение S относительно SEQ |
| ------- | ------------------------ | --------------------- | ------- | -------------------- |
| seq | 1 | 3D, n=200 | 0.108102 | 1.00 |
| omp | OpenMP | 3D, n=200 | 0.064491 | 1.68 |
| tbb | `parallel_reduce` | 3D, n=200 | 0.066552 | 1.62 |
| stl | `std::async` | 3D, n=200 | 0.054318 | 1.99 |
| all | MPI + async (P=1) | 3D, n=200 | 0.057650 | 1.88 |

Интерпретация: на этой конфигурации STL дала наилучшее время; OMP и TBB близки. Разницу не стоит считать универсальной:
на неё влияют планировщик ОС, состояние кэша и фактическое число рабочих потоков.

## 6. Интерпретация различий

- SEQ — опорная линия по времени и эталон значения.
- OMP — предсказуемое статическое разбиение idx; reduction убирает гонки.
- TBB — снижение ручного кода за счёт parallel_reduce; накладные расходы рантайма сопоставимы с OpenMP на данной
  задаче.
- STL — явное нарезание диапазонов и async; на замере ускорение выше, но оценка должна подкрепляться серией
  повторов.
- ALL — при P=1 работает почти как локальная STL-схема, но с MPI-обвязкой; при P>1 уже появляется цена редукции между
  процессами.

## 7. Репродуцируемость

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DUSE_FUNC_TESTS=ON -DUSE_PERF_TESTS=ON -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
# Пример: потоки
export PPC_NUM_THREADS=4
python scripts/run_tests.py --running-type=threads --counts 1 2 4
# Пример: MPI / процессы
export PPC_NUM_PROC=2
python scripts/run_tests.py --running-type=processes --counts 2 4
# Производительность
python scripts/run_tests.py --running-type=performance
```

Точные цели бинарников и фильтры gtest — по актуальной структуре репозитория `ppc-2026-threads`.
