## 1. Контекст

Для той же суммы по узлам сетки выбран примитив **`parallel_reduce`** из Intel oneTBB: библиотека сама дробит диапазон
линейных индексов и объединяет частичные суммы. Это альтернатива явному циклу OpenMP с **reduction**.

## 2. Постановка задачи

Входные данные: границы a и b, чётные n_i по каждой оси, функция f.
Выход: численное значение интеграла.
SEQ используется как эталон результата и базовое время.

## 3. Базовый алгоритм

Идентичен **SEQ** по математике; меняется только способ параллельного накопления суммы по **idx**.

## 4. Схема распараллеливания

- **Примитив:** **`tbb::parallel_reduce`** (в коде через **`oneapi/tbb.h`**, пространство имён **`tbb::`**).
- **Диапазон:** **`tbb::blocked_range<size_t>(0, total_points)`** — линейный индекс узла; **`total_points`** совпадает с
  числом узлов в **OMP**-версии (**strides[0]·(n[0]+1)**).
- **Grain size:** явно не задаётся; используется разбиение по умолчанию для **`blocked_range`** (автоматический выбор
  поддиапазонов и балансировка на стороне рантайма).
- **Объединение:** бинарная операция **`[](double x, double y) { return x + y; }`** для частичных сумм.
- **Конкуренция:** ограничивается политикой oneTBB (worker-потоки); при необходимости в других задачах используют
  **`global_control`** — в данной реализации не выставляется явно.

В теле лямбды для поддиапазона восстанавливаются координаты узла через **strides**, считается вклад **w_prod·f(x)** —
отдельной функции «узлового вклада» в проекте нет, логика inlined в лямбду.

## 5. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Используются те же `PrecomputeWeights` и `ComputeStrides`, что и в OMP/STL, чтобы сравнение реализаций было корректным.

```cpp
// File: tbb/src/ops_tbb.cpp (фрагмент)
double sum = tbb::parallel_reduce(
    tbb::blocked_range<size_t>(0, total_points), 0.0,
    [&](const tbb::blocked_range<size_t> &range, double local_sum) { /* ... */ },
    [](double x, double y) -> double { return x + y; });
```

## 6. Проверка корректности

TBB-версия проверяется теми же 20 тестами из `tests/functional/main.cpp`. Набор специально не зависит от backend-а:
каждой реализации передаётся одинаковый `InType`, а ожидаемое значение интеграла задано в самом тесте.

Тесты покрывают:

- простые 1D функции, где легко проверить веса Симпсона;
- 2D функции с произведениями и суммами переменных;
- 3D случаи, чтобы проверить восстановление координат через `strides` для нескольких измерений.

Для TBB отдельно важно, что `parallel_reduce` корректно объединяет частичные суммы. Поэтому сравнение идёт не с внутренним
порядком обхода, а с итоговым числом: результат должен отличаться от ожидаемого меньше чем на `1e-6`.

## 7. Экспериментальная среда

Release, компилятор с поддержкой oneTBB. Для сопоставимости с другими реализациями полезно зафиксировать
`PPC_NUM_THREADS` (если runner/рантайм учитывает ограничение параллелизма) и повторить замеры несколько раз.

## 8. Результаты

| Потоки (пул TBB) | Размер | Время, с | Ускорение относительно SEQ |
| ---------------- | ------ | -------- | -------------------------- |
| по умолчанию | 3D Гаусс, `n = {200, 200, 200}` | 0.066552 | 1.62 |

Накладные расходы **parallel_reduce** и выбор поддиапазонов могут чуть отличаться от OpenMP на том же **CPU**.

## 9. Выводы

**parallel_reduce** хорошо подходит для ассоциативной суммы по большому **blocked_range**. На тестовой машине время
близко к **OMP**; выбор между ними для этой задачи часто определяется инфраструктурой сборки и предпочтениями по рантайму,
а не только абсолютным временем в одном замере.
