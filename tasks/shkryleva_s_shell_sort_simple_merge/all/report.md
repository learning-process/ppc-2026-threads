## 1. Контекст
Гибридная версия (MPI + OMP) предназначена для распределённых вычислительных
систем, где данные не помещаются в память одного узла. MPI отвечает за
межпроцессное взаимодействие, а OpenMP – за эффективное использование
многоядерности внутри каждого узла.

## 2. Постановка задачи
Сортировка массива `int` размера $10^6$. Базовая линия – SEQ (250 мс).
MPI-процессы работают на нескольких узлах (или на одном узле для тестирования).

## 3. Базовый алгоритм
Каждый MPI-процесс получает свою часть данных, сортирует её с помощью
параллельной сортировки Шелла (OpenMP), затем процесс 0 собирает все части и
выполняет простое последовательное слияние.

## 4. Межпроцессная схема (MPI)
- **Роли:** Rank 0 – координатор: раздаёт данные через `MPI_Scatterv`, принимает
  отсортированные части от остальных процессов, выполняет финальное слияние и
  рассылает результат через `MPI_Bcast`.
- **Распределение данных:** используется `MPI_Scatterv` для неравномерного
  распределения, если размер не кратен числу процессов.
- **Передача данных:** процессы с rank > 0 отправляют свои отсортированные
  данные процессу 0 через `MPI_Send`.

## 5. Внутрипроцессная схема (OpenMP)
- **Параллелизация:** внутри каждого MPI-процесса вызывается
  `ParallelShellSort`, которая разбивает локальный массив на блоки по числу
  потоков OMP, сортирует каждый блок, затем выполняет иерархическое слияние
  (параллельное, как в STL-версии).
- **Детали:** используется `#pragma omp parallel for` для сортировки блоков и
  для слияния на каждом уровне.

## 6. Детали реализации
- **Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.
- **Функции:** `ShellSort`, `Merge`, `ParallelShellSort`, `SimpleMerge`
  (последовательное слияние на rank 0).
- **Синхронизация:** `MPI_Bcast` в конце обеспечивает идентичность выходных
  данных на всех процессах.

## 7. Проверка корректности
Тесты пройдены на 8 функциональных наборах. Дополнительно проверено, что после
`MPI_Bcast` все процессы возвращают одинаковый отсортированный массив.

## 8. Экспериментальная среда
- **Конфигурация:** 4 MPI-процесса (эмулируется на одном узле, так как кластер
  отсутствует).
- **Потоки на процесс:** 4, 8, 12.
- **Команда запуска:** `mpiexec -n 4 ./ppc_perf_tests.exe --gtest_filter=*all*`

## 10. Выводы
Гибридная версия демонстрирует замедление относительно последовательной
(ускорение < 1). Причины:
- Накладные расходы на MPI-коммуникации (`Scatterv`, `Send`/`Recv`) на малом
  объёме данных (1 млн элементов).
- Последовательное слияние на процессе 0, которое становится узким местом и не
  масштабируется.
- Для небольших массивов, помещающихся в память одного узла, MPI не даёт
  выигрыша.

Рекомендации: использовать MPI только при объёмах данных, превышающих память
одного узла, и заменить простое слияние на каскадное (параллельное) или
использовать `MPI_Gatherv` с последующей сортировкой слиянием на нескольких
узлах.
