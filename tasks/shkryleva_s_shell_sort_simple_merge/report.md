## 1. Введение
В работе реализованы и проанализированы параллельные версии алгоритма сортировки
Шелла с простым слиянием для целочисленных массивов. Использованы технологии:
OpenMP, TBB, STL (std::thread) и гибрид MPI+OpenMP. Цель – сравнить
эффективность различных моделей параллелизма для задачи, имеющей как
вычислительно насыщенные участки (сортировка), так и требующие синхронизации
(слияние).

## 2. Единая постановка задачи
- **Вход:** вектор целых чисел `int` размера $N = 10^6$.
- **Выход:** отсортированный по возрастанию вектор.
- **Критерий корректности:** для любых $i<j$ выполняется $result_i \le
  result_j$.

## 3. Единая методика эксперимента
### Окружение
- **CPU:** 11th Gen Intel(R) Core(TM) i5-11400H @ 2.70GHz
- **RAM:** 16 ГБ
- **OS:** Windows 10 + WSL, Ubuntu 24.04.1 LTS
- **Compiler:** g++ 13.3.0
- **Build type:** Release
- **Переменные окружения:** `PPC_NUM_THREADS` (для OMP, TBB, STL),
  `PPC_NUM_PROC` (для MPI).

### Генерация данных
Вектор случайных целых чисел в диапазоне $[-100, 100]$ с использованием
`std::mt19937`. Объём данных – $10^6$ элементов.

Время каждого бэкенда измерялось как минимальное из трёх запусков.  
Ускорение: $S = T_{seq} / T_{p}$, эффективность: $E = S / \text{workers}$.

## 4. Сводка корректности

## 6. Интерпретация различий
- **SEQ:** базовая линия. Время 250 мс на 1 млн элементов отражает сложность
  $O(N \log^2 N)$ сортировки Шелла.
- **OMP:** ускорение до 1.92x. Эффективность падает с ростом потоков из-за
  динамического планирования и накладных расходов на слияние.
- **TBB:** ускорение до 1.67x. Основной недостаток – последовательное слияние;
  TBB не даёт преимущества перед OMP в данной реализации.
- **STL:** лучшее ускорение (2.63x). Полностью параллельное слияние позволяет
  эффективно использовать все ядра, хотя создание потоков вручную требует
  аккуратности.
- **ALL:** замедление (ускорение < 1). Межпроцессное взаимодействие на малом
  объёме данных и последовательное слияние на мастере перекрывают выигрыш от
  параллельной сортировки внутри узлов.

## 8. Заключение

Наилучшую производительность показала реализация на STL (ускорение 2.63x),
благодаря полному параллелизму на всех этапах. OpenMP и TBB уступают из-за менее
эффективного слияния (последовательное или с динамическим планированием).
Гибридная MPI+OMP версия неэффективна на одном узле и требует больших объёмов
данных для оправдания накладных расходов на коммуникацию. Основные ограничения:

Сортировка Шелла имеет неоптимальную теоретическую сложность; для больших
массивов лучше использовать алгоритмы O(NlogN).

Последовательное слияние в TBB и ALL является узким местом.

Направления улучшения:

Заменить сортировку Шелла на быструю сортировку или поразрядную.

Реализовать параллельное слияние в TBB-версии (например, с использованием
tbb::parallel_invoke).

Для MPI использовать каскадное слияние (например, попарное слияние на разных
процессах).
