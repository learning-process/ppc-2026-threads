## 1. Контекст
OpenMP используется для распараллеливания на многоядерных CPU. Алгоритм
разбивается на независимые блоки, которые сортируются параллельно, затем
производится иерархическое слияние. Это позволяет сократить общее время
выполнения за счёт загрузки всех ядер.

## 2. Постановка задачи
Задача – отсортировать массив из $10^6$ целых чисел типа `int`. Базовая линия –
последовательная версия (SEQ) со временем 250 мс.

## 3. Базовый алгоритм
Используется тот же принцип, что и в SEQ: сортировка Шелла + слияние. Однако
вместо деления на две половины массив разбивается на блоки по числу потоков.

## 4. Схема распараллеливания
- **Декомпозиция данных:** массив делится на `num_threads` блоков примерно
  равного размера.
- **Параллельная сортировка:** каждый поток сортирует свой блок сортировкой
  Шелла (директива `#pragma omp for schedule(dynamic)`).
- **Иерархическое слияние:** в цикле размер блока удваивается, и на каждом шаге
  пары соседних блоков сливаются параллельно с помощью `#pragma omp for
  schedule(dynamic)`. Слияние выполняется во временный буфер (индивидуальный для
  каждого потока, чтобы избежать гонок).
- **Синхронизация:** неявные барьеры в конце каждой директивы `parallel for`.

## 5. Детали реализации
- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.
- **Функции:** `ShellSort(left, right, arr)` – сортировка диапазона;
  `Merge(left, mid, right, arr, buffer)` – слияние двух упорядоченных частей.
- **Управление потоками:** используется `omp_get_max_threads()`, число блоков
  равно числу потоков.
- **Особенности:** буфер для слияния создаётся внутри каждого потока и
  переиспользуется. Планирование `dynamic` выбрано для балансировки нагрузки.

## 6. Проверка корректности
Корректность подтверждена теми же 8 функциональными тестами, что и для SEQ.
Результат OMP-версии сравнивался с `std::sort` и с результатом SEQ – все тесты
пройдены.

## 7. Экспериментальная среда
Аналогично SEQ (см. `seq/report.md`). Число потоков задаётся переменной
окружения `PPC_NUM_THREADS`.

## 8. Результаты
Эксперименты на **1 000 000** элементов типа `int`.

| Threads | Время (ms) | Ускорение ($S$) | Эффективность ($E$) |
| :-----: | :--------: | :-------------: | :-----------------: |
| 1 (SEQ) |    250     |      1.00       |        1.00         |
|    4    |    140     |      1.79       |        0.45         |
|    8    |    130     |      1.92       |        0.24         |
|   12    |    135     |      1.85       |        0.15         |

## 9. Выводы
OpenMP-версия даёт ускорение до 1.92x на 8 потоках. Дальнейший рост числа
потоков не улучшает результат из-за накладных расходов на синхронизацию при
слиянии и ограничений пропускной способности памяти. Для лучшей масштабируемости
требуется уменьшение числа последовательных слияний (например, использование
параллельной сортировки слиянием).
