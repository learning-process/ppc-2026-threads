## 1. Введение

Задача состоит в численном вычислении двойного интеграла от заданной функции на
прямоугольной области методом прямоугольников (средних точек).
Вычисления в каждой точке сетки независимы, поэтому задача хорошо подходит для изучения параллельных технологий.
В работе сравниваются последовательная реализация и четыре параллельных подхода: OMP, TBB, STL и
гибридная MPI+STL (ALL).

## 2. Единая постановка задачи

**Входные данные** (InType): `(steps, func_id)`, где `steps` – количество разбиений по каждому
измерению (общее число точек сетки = steps²), `func_id` – номер интегрируемой функции (1–4).

**Выходные данные** (OutType): приближённое значение двойного интеграла.

**Интегрируемые функции и области:**

| func_id | Функция       | Область [xmin,xmax]×[ymin,ymax] | Точное значение (приближённое) |
|---------|---------------|---------------------------------|--------------------------------|
| 1       | x² + y²       | [0,1]×[0,1]                     | 2/3 ≈ 0.6666667                |
| 2       | sin(x)·cos(y) | [0,π]×[0,π/2]                   | 2.0                            |
| 3       | exp(-(x²+y²)) | [-1,1]×[-1,1]                   | ≈ 2.230                        |
| 4       | sin(x+y)      | [0,π]×[0,π]                     | 0.0                            |

**Метод прямоугольников (средних точек):**

Для области [a,b]×[c,d] с шагами `hx = (b-a)/steps`, `hy = (d-c)/steps` значение интеграла:

`I ≈ hx·hy · Σ_{i=0}^{steps-1} Σ_{j=0}^{steps-1} f(a + (i+0.5)·hx, c + (j+0.5)·hy)`

**Критерий корректности:** абсолютная погрешность относительно точного значения не более 0.05 для `steps = 20, 40`.

## 3. Единая методика эксперимента

**Окружение:**

- CPU: AMD Ryzen 7 PRO 3700 (8 ядер, 16 логических потоков)
- RAM: 16 ГБ
- OS: Windows
- Компилятор: GCC / Clang с поддержкой C++20, флаги `-O3 -march=native`

**Переменные окружения:**

- `PPC_NUM_THREADS` – число потоков для OpenMP, TBB, STL
- `PPC_NUM_PROC` – число MPI-процессов для ALL версии
- `OMP_NUM_THREADS` – устанавливается равным `PPC_NUM_THREADS`

**Размер задачи:** `steps = 10000`  – достаточен для устойчивых замеров.

**Измеряемые величины:**

- Время выполнения `RunImpl()`
- Ускорение `S(p) = T_seq / T_par(p)`
- Эффективность `E(p) = S(p) / p` (p – число потоков для OMP/TBB/STL; для ALL – произведение рангов на потоки)

**Порядок измерений:** прогрев (1 запуск без замера), затем 5 замеров, медиана.

## 6. Интерпретация различий

- **SEQ** – эталон, время ~0.484 с.
- **OMP** – лучшее ускорение среди всех реализаций (до 4.56 на 8 потоках).
Эффективность высокая на малом числе потоков, снижается при 8 потоках из-за накладных
расходов на редукцию и конкуренции за кэш.
- **TBB** – показывает низкое ускорение (менее 1 на 2 потоках). Причина – вероятно, некорректная настройка
 grain size или высокие накладные расходы `parallel_reduce` для лёгкой вычислительной нагрузки.
 В данной ситуации TBB неэффективна.
- **STL** – ускорение выше, чем у TBB, но всё ещё ниже OMP. Накладные расходы на создание потоков через `std::async`
 и отсутствие адаптивного планирования снижают эффективность.
- **ALL** (гибрид MPI + потоки) – даёт хорошее ускорение (до 6.36 при 16 workers), но эффективность падает с ростом
 числа процессов из-за коммуникационных затрат (MPI_Reduce, Bcast, барьеры).
 Для одного узла чисто потоковые решения (OMP) предпочтительнее.

## 8. Заключение

OpenMP демонстрирует наилучшую эффективность для данной задачи благодаря простоте распараллеливания внешнего
цикла и низким накладным расходам. TBB и STL показывают значительно худшие результаты из-за особенностей реализации
и дополнительных накладных расходов. Гибридная версия ALL даёт приемлемое ускорение, но уступает OpenMP при работе
на одном узле. Рекомендуется использовать OpenMP для подобных регулярных вычислительных задач. Гибридную версию
лучше сиспользовать в более сложных задачах.
