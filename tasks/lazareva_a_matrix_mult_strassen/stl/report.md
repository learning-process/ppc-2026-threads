## 1. Контекст

Ручная thread-based версия с использованием `std::thread` и современных STL алгоритмов демонстрирует низкоуровневый контроль над параллелизмом без зависимости от внешних библиотек. В отличие от OpenMP и TBB, данная реализация обеспечивает полную прозрачность управления потоками и явное разбиение данных, что важно для понимания накладных расходов параллелизации и оптимизации memory layout. Дополнительно используются современные STL алгоритмы для оптимизации операций с векторами.

## 2. Постановка задачи

Входные и выходные данные полностью идентичны последовательной версии. Подробная постановка задачи документирована в `seq/report.md`. Цель STL-версии — продемонстрировать эффективность ручного управления потоками с использованием стандартной библиотеки C++20/23 и современных алгоритмов.

## 3. Базовый алгоритм

Алгоритм Штрассена O(n^2.807) с итеративной реализацией, полностью сохраняющий структуру последовательной версии для корректного сравнения производительности. Добавлены STL алгоритмы для оптимизации вспомогательных операций.

## 4. Схема распараллеливания

### Разбиение данных на диапазоны

**Стратегия 1: Round-robin распределение строк (NaiveMult)**

```cpp
auto worker = [&](int thread_id) {
    for (int i = thread_id; i < n; i += num_threads) {
        // Обработка строки i
        for (int k = 0; k < n; ++k) {
            const double aik = a[i*n + k];
            for (int j = 0; j < n; ++j) {
                c[i*n + j] += aik * b[k*n + j];
            }
        }
    }
};
```

**Формула разбиения:** Поток thread_id обрабатывает строки {thread_id, thread_id + num_threads, thread_id + 2*num_threads, ...}

**Преимущества round-robin:**
- Равномерное распределение нагрузки при переменной сложности строк
- Хорошая cache locality для матрицы A (последовательный доступ по строкам)
- Естественная балансировка при n не кратном num_threads

### Что делает каждый поток

**В наивном умножении:**
- Каждый поток получает уникальный thread_id от 0 до num_threads-1
- Выполняет полное умножение для своих строк без промежуточных синхронизаций
- Записывает результат напрямую в общий вектор c без локальных буферов

**При вычислении 7 произведений Штрассена:**

```cpp
std::array<std::future<std::vector<double>>, 7> futures;
for (size_t k = 0; k < 7; ++k) {
    futures.at(k) = std::async(std::launch::async, [&lhs, &rhs, k, half]() {
        // Полное наивное умножение для k-того произведения
        return c;  // Локальный результат
    });
}
```
### Локальные результаты и объединение

**В наивном умножении:**
- Отсутствие локальных буферов: каждый поток записывает в уникальные элементы c[i*n+j] где i принадлежит только данному потоку
- Прямая запись: результат накапливается непосредственно в финальном векторе без дополнительных операций объединения

**При асинхронных вычислениях:**
- Локальные векторы: каждая async-задача создает собственный вектор результата
- Объединение через futures: std::ranges::transform(futures, m.begin(), [](auto& f) { return f.get(); })

### Порядок create → work → join

```cpp
// CREATE: создание потоков
std::vector<std::thread> threads;
threads.reserve(num_threads);
for (int thread_idx = 0; thread_idx < num_threads; ++thread_idx) {
    threads.emplace_back(worker, thread_idx);  // Немедленный запуск
}

// WORK: потоки выполняются параллельно

// JOIN: ожидание завершения всех потоков
std::ranges::for_each(threads, [](std::thread &thr) { thr.join(); });
```
### Синхронизация

**Полное отсутствие явной синхронизации:**
- Каждый поток работает с непересекающимися диапазонами данных
- Гонки исключены архитектурно: thread_id гарантирует уникальность обрабатываемых строк
- join() обеспечивает барьер перед возвратом результата
- Использование std::launch::async гарантирует асинхронное выполнение

## 6. Проверка корректности

**Сравнение с SEQ:**
Результаты STL-версии идентичны последовательной реализации благодаря:
- Детерминированному порядку операций внутри каждой строки
- Отсутствию гонок при записи (уникальные индексы)
- Использованию тех же floating-point операций

**Крайние случаи**
- n < num_threads: некоторые потоки не получают работы (корректно обрабатывается round-robin)
- n = 64: граница между рекурсией и наивным умножением
- Нечетные размеры: требующие паддинга до степени двойки

### Проверка на отсутствие гонок

**Анализ доступа к памяти:**
-Чтение: все потоки безопасно читают a и b (только чтение)
- Запись: каждый поток пишет в c[i*n + j] где i уникален для потока
- Формальное доказательство: для потоков t1 ≠ t2 и строк i1 = tid1 + k1*num_threads, i2 = tid2 + k2*num_threads выполняется i1 ≠ i2 при tid1 ≠ tid2

**Комментарий о накладных расходах create/join**

Overhead анализ:
- Создание потоков: ~10-50 мкс на поток, незначительно для больших задач
- Контекстные переключения: минимальны благодаря крупной грануляции (строки матрицы)
- Join synchronization: ~1-5 мкс на поток, пренебрежимо мало
- Memory allocation: STL алгоритмы оптимизированы компилятором

**Сравнение с другими технологиями:**
- Быстрее OpenMP на малых задачах (отсутствие runtime overhead)
- Медленнее TBB на больших задачах (менее эффективная балансировка)
- Предсказуемое поведение без скрытых оптимизаций

**Узкие места:**
- Статическое распределение нагрузки может быть неоптимальным
- Отсутствие work-stealing при неравномерной нагрузке
- Ограниченная масштабируемость при большом количестве потоков

## 9. Выводы

### Преимущества STL подхода:

**Полный контроль:** Ручное управление потоками обеспечивает максимальную прозрачность и возможность точной оптимизации под конкретную задачу без скрытых накладных расходов библиотек.

**Переносимость:** Использует только стандартную библиотеку C++, обеспечивая совместимость без дополнительных зависимостей.

**Современные STL алгоритмы:** Демонстрируют, как стандартные алгоритмы могут оптимизировать код даже в параллельном контексте.

### Ограничения ручного подхода

**Статическая балансировка:** Round-robin распределение не адаптируется к изменениям нагрузки, что может приводить к неравномерному использованию ресурсов.

**Сложность масштабирования:** При увеличении числа потоков требуется ручная оптимизация стратегии разбиения данных.

**Отсутствие продвинутых оптимизаций:** Нет автоматического NUMA-aware распределения и work-stealing, доступных в специализированных библиотеках.


Ручная реализация успешно демонстрирует основы многопоточного программирования и может служить baseline для оценки эффективности высокоуровневых библиотек параллелизации.
