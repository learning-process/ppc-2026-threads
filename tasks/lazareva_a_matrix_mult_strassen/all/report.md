## 1. Контекст

Гибридная версия объединяет преимущества distributed-memory параллелизма (MPI) и shared-memory 
параллелизма (TBB) для эффективного использования современных многоузловых кластеров. В алгоритме 
Штрассена естественным образом возникает двухуровневая структура: межпроцессное распределение 7 
независимых произведений между узлами и внутрипроцессное параллельное вычисление каждого произведения 
с использованием всех ядер узла. Это позволяет масштабировать алгоритм на кластерных системах при 
сохранении высокой утилизации отдельных узлов.
## 2. Постановка задачи

Входные и выходные данные идентичны последовательной версии. Подробная постановка документирована 
в `seq/report.md`. Цель ALL-версии — продемонстрировать эффективность иерархического параллелизма 
на системах с несколькими узлами, где каждый узел содержит multiple cores.

## 3. Базовый алгоритм

Алгоритм Штрассена O(n^2.807) с итеративной реализацией и порогом n ≤ 64. Основная структура 
алгоритма сохраняется для корректного сравнения, но добавляется двухуровневая параллелизация 
на критическом этапе вычисления 7 произведений.

## 4. Межпроцессная схема

### Роли rank-ов

**Master-процесс (rank = 0):**
- Получает и обрабатывает входные данные
- Выполняет паддинг и разбиение на подматрицы
- Распределяет 7 произведений между всеми процессами (включая себя)
- Собирает результаты от worker-процессов
- Выполняет финальную сборку результирующей матрицы

**Worker-процессы (rank = 1, 2, ..., size-1):**
- Получают подматрицы для назначенных произведений
- Выполняют локальные вычисления с использованием TBB
- Отправляют результаты обратно master-процессу

### Распределение данных

**Стратегия распределения 7 произведений:**
```cpp
for (int k = 0; k < 7; ++k) {
    const int dest = k % size;  // Round-robin распределение
    // k=0 → rank 0, k=1 → rank 1, k=2 → rank 2, ...
}
```

**Карта распределения для size=4:**

Произведение	M0	M1	M2	M3	M4	M5	M6
Rank	        0	1	2	3	0	1	2

### MPI-вызовы и их назначение

**Инициализация и broadcast параметров:**
```cpp
MPI_Bcast(&n_, 1, MPI_INT, 0, MPI_COMM_WORLD);        // Размер матрицы
MPI_Bcast(&padded_n_, 1, MPI_INT, 0, MPI_COMM_WORLD); // Размер с паддингом
```

**Неблокирующая отправка подматриц (master → workers):**
```cpp
MPI_Isend(lhs[k].data(), h_sz, MPI_DOUBLE, dest, k*2, MPI_COMM_WORLD, &send_requests[req_idx]);
MPI_Isend(rhs[k].data(), h_sz, MPI_DOUBLE, dest, k*2+1, MPI_COMM_WORLD, &send_requests[req_idx+1]);
```

**Получение подматриц (workers):**
```cpp
MPI_Recv(l.data(), h_sz, MPI_DOUBLE, 0, k*2, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
MPI_Recv(r.data(), h_sz, MPI_DOUBLE, 0, k*2+1, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
```

**Сбор результатов (workers → master):**
```cpp
MPI_Send(res.data(), h_sz, MPI_DOUBLE, 0, k+100, MPI_COMM_WORLD); 
MPI_Recv(m[k].data(), h_sz, MPI_DOUBLE, src, k+100, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
```
**Финальный broadcast результата:**
```cpp
MPI_Bcast(GetOutput().data(), final_sz, MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

## Внутрипроцессная схема

### Thread-technology: Intel oneTBB

Внутри каждого MPI-процесса используется TBB для эффективного использования всех ядер узла:
```cpp
oneapi::tbb::parallel_invoke(
    [&]() { m0 = StrassenTBB(Add(a11, a22, h), Add(b11, b22, h), h); },
    [&]() { m1 = StrassenTBB(Add(a21, a22, h), b11, h); },
    [&]() { m2 = StrassenTBB(a11, Sub(b12, b22, h), h); },
    [&]() { m3 = StrassenTBB(a22, Sub(b21, b11, h), h); },
    [&]() { m4 = StrassenTBB(Add(a11, a12, h), b22, h); },
    [&]() { m5 = StrassenTBB(Sub(a21, a11, h), Add(b11, b12, h), h); },
    [&]() { m6 = StrassenTBB(Sub(a12, a22, h), Add(b21, b22, h), h); }
);
```
### Разделение работы между потоками

**TBB parallel_invoke:**
- Автоматически распределяет 7 задач между доступными worker threads
- Каждая задача представляет рекурсивный вызов StrassenTBB для подматрицы размером h×h
- Work-stealing scheduler обеспечивает динамическую балансировку нагрузки

**Рекурсивная структура:**
- При n > 64: используется parallel_invoke для 7 произведений
- При n ≤ 64: переход к наивному умножению без дополнительного параллелизма

### Объединение локальных результатов

Локальные результаты объединяются внутри каждого процесса через TBB task scheduler без 
дополнительной синхронизации, поскольку каждая задача записывает в 
уникальную переменную (m0, m1, ..., m6).

### Критические точки синхронизации

**MPI_Bcast в ValidationImpl:**
- Все процессы синхронизируются на результате валидации от master-процесса
- Гарантирует корректную инициализацию всех процессов или единообразное завершение

**MPI_Waitall в StrassenMaster:**
- Master ждет завершения всех неблокирующих отправок перед освобождением буферов
- Предотвращает race condition между освобождением памяти и передачей данных

**MPI_Bcast в PostProcessingImpl:**
- Финальная синхронизация результата между всеми процессами
- Обеспечивает, что все процессы имеют одинаковый итоговый результат

### Потенциальные узкие места

**Коммуникационный overhead:**
- Передача матриц размером h²×8 байт (для каждого из 7 произведений)
- Network bandwidth может стать ограничением при больших матрицах
- Latency между MPI_Send/Recv создает ожидания

**Неравномерная балансировка:**
- При size > 7 некоторые процессы остаются без работы
- Round-robin может создавать дисбаланс при переменной сложности подзадач

**Memory pressure:**
- Каждый процесс хранит полные копии назначенных подматриц
- Пиковое потребление памяти на master-процессе

**Broadcast валидация:**
- Все процессы получают одинаковые параметры (n_, padded_n_)

**Результирующий broadcast:**
- Финальный результат транслируется всем процессам
- Гарантирует единообразный выход независимо от rank

### Комментарий о цене коммуникации и синхронизации

**Оптимальная конфигурация:**
- **3 процесса × 2 TBB threads** показывает **максимальное ускорение 4.30x** при высокой 
эффективности 0.72
- Соответствует оптимальному распределению 7 произведений: процессы получают 2-3-2 задачи 
соответственно  
- Обеспечивает лучший баланс между коммуникационными расходами и вычислительным параллелизмом

**Коммуникационные расходы:**
- Передача подматриц: ~512 KB на произведение (256×256×8 байт)
- MPI latency в WSL environment: ~10-50 μs per message
- Общий объем межпроцессных данных: ~3.6 MB для всех 7 произведений
- Overhead составляет ~15-20% от общего времени выполнения

**Анализ влияния конфигурации:**
- **1×6 (чистая TBB):** хорошее ускорение 2.38x, но ограниченная масштабируемость
- **2×3:** улучшение до 3.33x за счет распределения нагрузки, эффективность 0.56
- **3×2:** оптимальная точка - 4.30x ускорение при 0.72 эффективности
- **4×2:** снижение до 3.43x из-за неравномерного распределения 7 задач между 4 процессами
- **6×1:** деградация эффективности (0.58) при избыточном количестве процессов


## 10. Выводы

### Когда гибридная схема оправдана

**Превосходящая производительность:**
Гибридная версия демонстрирует **выдающееся ускорение 4.30x** на оптимальной конфигурации, 
что на 81% превышает результат чистой TBB версии (2.38x) и показывает высокую 
эффективность 0.72 на 6 workers.

**Естественная декомпозиция:**
Алгоритм Штрассена идеально подходит для гибридного подхода: 7 независимых произведений 
обеспечивают естественное распределение между процессами, а внутренние матричные операции 
эффективно параллелятся через TBB.

**Эффективное использование ресурсов:**
Конфигурация 3×2 достигает эффективности 0.72, что значительно лучше чем простое 
увеличение TBB потоков (0.40) и демонстрирует преимущества иерархического параллелизма.

### Когда гибридная схема неэффективна

**Избыточная декомпозиция:**
При 6 процессах эффективность снижается до 0.58 из-за коммуникационного overhead'а, 
превышающего выигрыш от дальнейшего распределения работы.

**Неоптимальное распределение задач:**
4 процесса показывают низкую эффективность 0.43 из-за неравномерного распределения 7 
произведений (1-2-2-2), приводящего к load imbalance.

**Ограничения локального окружения:**
В WSL environment коммуникационные расходы становятся заметными уже при 4+ процессах, 
что ограничивает масштабируемость на одном узле.

Гибридный подход успешно демонстрирует **4.30x ускорение при 0.72 эффективности**, 
что доказывает эффективность иерархического параллелизма для алгоритма Штрассена и закладывает основу
 для масштабирования решения на многоузловые вычислительные кластеры.
