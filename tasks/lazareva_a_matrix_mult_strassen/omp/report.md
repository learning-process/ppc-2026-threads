## 1. Контекст

Данная реализация переносит алгоритм Штрассена из последовательной версии (SEQ) в параллельную с использованием OpenMP. Основная цель — распараллелить наиболее вычислительно затратные участки: вычисление 7 произведений матриц M1-M7 на каждом уровне рекурсии и наивное умножение для базовых случаев. OpenMP выбран как стандартная технология shared-memory параллелизма с простой интеграцией в существующий код через директивы компилятора.

## 2. Постановка задачи

Входные и выходные данные полностью идентичны последовательной версии: две плотные квадратные матрицы A и B размером n×n, результат C = A×B. Подробная постановка задачи, ограничения и корректность результата документированы в `seq/report.md`. Цель OMP-версии — ускорить вычисления за счет использования нескольких потоков на многоядерных системах.

## 3. Базовый алгоритм

Алгоритм Штрассена с асимптотикой O(n^2.807) разбивает матрицу на 4 подматрицы и вычисляет 7 произведений вместо 8, используя рекурсию до достижения порога n ≤ 64, после чего применяется наивное умножение O(n³). Итеративная реализация с явным стеком предотвращает переполнение стека вызовов.

## 4. Схема распараллеливания

### Область параллелизации 1: Наивное умножение (базовый случай)

```cpp
#pragma omp parallel for schedule(static) default(none) shared(a, b, c, n)
for (int i = 0; i < n; ++i) {
    for (int k = 0; k < n; ++k) {
        const double aik = a[static_cast<size_t>((static_cast<ptrdiff_t>(i) * n) + k)];
        for (int j = 0; j < n; ++j) {
            c[static_cast<size_t>((static_cast<ptrdiff_t>(i) * n) + j)] += 
                aik * b[static_cast<size_t>((static_cast<ptrdiff_t>(k) * n) + j)];
        }
    }
}
```

**Переменные и область данных:**
- shared(a, b, c, n): входные матрицы доступны всем потокам только для чтения, результирующая матрица c — для записи
- private: переменные i, k, j, aik автоматически приватные для каждого потока
- Отсутствие гонок: каждый поток обрабатывает уникальные строки матрицы (i уникален), поэтому элементы c[i*n+j] обновляются без конфликтов

### Область параллелизации 2: Вычисление 7 произведений Штрассена

```cpp
#pragma omp parallel for schedule(static) default(none) shared(m, lhs, rhs, half)
for (int k = 0; k < 7; ++k) {
    const auto uk = static_cast<size_t>(k);
    // Локальное наивное умножение lhs[k] × rhs[k]
    m.at(uk) = std::move(c);
}
```
**Переменные и область данных:**
- shared(m, lhs, rhs, half): массивы операндов и результатов доступны всем потокам
- private: все локальные переменные цикла (k, uk, c, i, ki, j, aik) приватные
- Отсутствие гонок: каждый поток записывает в уникальный элемент m[k]

**Выбор планирования:**
- schedule(static): итерации равномерно распределяются между потоками статически
- Подходит для данной нагрузки, поскольку все 7 произведений имеют одинаковую вычислительную сложность O(n³/8)
- Отсутствует overhead динамического распределения задач

**Барьеры и синхронизация:**
- Неявный барьер в конце каждой parallel for области обеспечивает завершение всех потоков перед продолжением
- Барьер необходим для корректности: последующие операции (Add/Sub для сборки C11, C12, C21, C22) требуют готовности всех M1-M7
- Барьер не влияет на масштабируемость, так как все потоки выполняют одинаковый объем работы

## 4. Детали реализации

**Файлы:** 
- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

**Изменения относительно SEQ:**
1. В функции NaiveMult: добавлена директива #pragma omp parallel for для распараллеливания внешнего цикла по строкам матрицы
2. В функции Strassen: добавлена директива для параллельного вычисления 7 произведений, где каждый поток выполняет полное наивное умножение для одной пары подматриц

**Устранение рисков гонок:**
- В NaiveMult: каждый поток работает с уникальным индексом строки i, исключая конфликты записи в c[i*n+j]
- В Strassen: каждый поток записывает результат в уникальный элемент массива m[k]
- Использование default(none): принудительное объявление области видимости всех переменных предотвращает случайное совместное использование

**Ограничения текущей реализации:**
- Параллелизм применяется только на верхнем уровне рекурсии Штрассена
- Глубокие уровни рекурсии остаются последовательными
- Эффективно только при достаточно больших матрицах (n ≥ 128)

**Комментарий о масштабируемости:**

Наблюдается хорошее ускорение до 4 потоков с эффективностью 0.81, что соответствует ожиданиям для алгоритма с частично последовательными участками. Снижение эффективности при 6 потоках объясняется несколькими факторами:
- Ограниченный параллелизм: только 7 произведений могут выполняться параллельно на верхнем уровне
- Последовательные участки: операции Split, Add, Sub, Merge остаются последовательными
- Overhead синхронизации: барьеры и управление потоками создают дополнительные затраты
- Неравномерная загрузка: при 6 потоках распределение 7 задач неоптимально

**Узкие места:**
- Операции подготовки подматриц (Split) — O(n²) последовательно
- Финальная сборка результата (Merge) — O(n²) последовательно
- Глубокие уровни рекурсии не параллелятся

## 8. Выводы

**Условия эффективности OpenMP:** OpenMP демонстрирует значительный выигрыш (ускорение 4.18x на 6 ядрах) для больших матриц (n ≥ 256), где доминирующими становятся параллелизуемые вычисления 7 произведений. Наивное умножение для базовых случаев также хорошо масштабируется благодаря регулярному доступу к памяти и отсутствию зависимостей по данным.

**Ограничения ускорения:**
-Закон Амдала: последовательные участки (Split/Merge) ограничивают теоретическое ускорение
-Архитектурные ограничения: на верхнем уровне только 7 независимых задач доступны для параллелизации
-Memory bandwidth: при большом количестве потоков возникает конкуренция за пропускную способность памяти

**Рекомендации по применению:** 
- Оптимально для 2-4 потоков на типичных многоядерных системах
- Эффективно для матриц размером n ≥ 128
- Требует дополнительной оптимизации для полного использования систем с большим количеством ядер

OpenMP версия успешно ускоряет алгоритм Штрассена при сохранении корректности и детерминированности результатов, обеспечивая хороший баланс между простотой реализации и производительностью.
