## 1. Введение

Алгоритм Штрассена для умножения матриц представляет собой идеальный тестовый случай для сравнения различных моделей параллелизма благодаря своей многоуровневой структуре. Алгоритм естественным образом декомпозируется на 7 независимых произведений подматриц на каждом уровне рекурсии, что позволяет эффективно применять как shared-memory параллелизм (потоки), так и distributed-memory подходы (процессы). Базовые случаи используют наивное матричное умножение с регулярными циклами, оптимальными для векторизации и кэш-оптимизации.

Асимптотическая сложность O(n^2.807) делает алгоритм вычислительно интенсивным для больших матриц, что позволяет амортизировать накладные расходы параллелизации и получить измеримое ускорение на современных многоядерных системах.

## 2. Единая постановка задачи

**Входные данные:** Две плотные квадратные матрицы A и B размером n×n, представленные как векторы `std::vector<double>` размером n² в row-major порядке.

**Выходные данные:** Матрица C = A×B размером n×n.

**Ограничения:** n > 0, размеры векторов строго равны n².

**Критерий корректности:** Результат должен совпадать с классическим матричным умножением с точностью ε = 1e-5 для компенсации погрешностей floating-point арифметики.

## 6. Интерпретация различий

**SEQ - базовая линия:** Итеративная реализация Штрассена с порогом 64 для наивного умножения обеспечивает стабильную производительность 0.0610 с и служит эталоном для всех сравнений.

**OMP - директивная модель:** Показывает хорошее линейное ускорение до 4 потоков (3.14x) благодаря простой интеграции `#pragma omp parallel for`. Статическое планирование эффективно для регулярных матричных операций. Снижение эффективности при 6 потоках (0.66) связано с ограниченным количеством независимых задач (7 произведений).

**TBB - задачно-ориентированная модель:** Демонстрирует оптимальную производительность на 3 workers (2.72x ускорение, 0.91 эффективность), что указывает на эффективную работу автоматической балансировки нагрузки. Отсутствие линейного масштабирования beyond 3 потоков связано с архитектурными ограничениями алгоритма и memory bandwidth saturation.

**STL - ручное управление потоками:** Показывает предсказуемое поведение с умеренным ускорением (3.77x на 6 потоках). Round-robin распределение обеспечивает равномерную нагрузку, но статическая балансировка уступает динамическим планировщикам TBB и OpenMP. Полная прозрачность управления потоками ценой несколько худшей производительности.

**ALL - гибридная модель:** **Достигает лучшего результата** (4.30x ускорение) на конфигурации 3×2, демонстрируя преимущества иерархического параллелизма. Распределение 7 произведений между 3 процессами (2-3-2) минимизирует коммуникационные расходы при максимальной утилизации ресурсов. Эффективность 0.72 превосходит возможности pure shared-memory подходов.
