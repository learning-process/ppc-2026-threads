## 1. Введение

Алгоритм Штрассена для умножения матриц представляет собой идеальный тестовый случай для сравнения различных моделей параллелизма благодаря своей многоуровневой структуре. Алгоритм естественным образом декомпозируется на 7 независимых произведений подматриц на каждом уровне рекурсии, что позволяет эффективно применять как shared-memory параллелизм (потоки), так и distributed-memory подходы (процессы). Базовые случаи используют наивное матричное умножение с регулярными циклами, оптимальными для векторизации и кэш-оптимизации.

Асимптотическая сложность O(n^2.807) делает алгоритм вычислительно интенсивным для больших матриц, что позволяет амортизировать накладные расходы параллелизации и получить измеримое ускорение на современных многоядерных системах.

## 2. Единая постановка задачи

**Входные данные:** Две плотные квадратные матрицы A и B размером n×n, представленные как векторы `std::vector<double>` размером n² в row-major порядке.

**Выходные данные:** Матрица C = A×B размером n×n.

**Ограничения:** n > 0, размеры векторов строго равны n².

**Критерий корректности:** Результат должен совпадать с классическим матричным умножением с точностью ε = 1e-5 для компенсации погрешностей floating-point арифметики.

## 6. Интерпретация различий

**SEQ — что показывает baseline:**
Время выполнения 61.0 мс демонстрирует производительность итеративной реализации алгоритма Штрассена без параллелизации. Baseline показывает эффективность порога переключения на наивное умножение (n ≤ 64) и оптимизацию компилятора для последовательного кода. Служит эталоном для измерения реального ускорения параллельных версий и выявления overhead различных технологий параллелизации.

**OMP — сильные и слабые стороны именно на этой задаче:**
*Сильные стороны:* Показывает отличное линейное масштабирование (2.95x → 3.67x → 4.04x) благодаря простой интеграции `#pragma omp parallel for` в существующую структуру циклов. Статическое планирование идеально подходит для регулярных матричных операций алгоритма Штрассена. Директивная модель требует минимальных изменений кода при получении значительного ускорения.

*Слабые стороны:* Относительно высокий overhead OpenMP runtime (15.1 мс vs 13.5 мс у STL на оптимальных конфигурациях). Эффективность > 1.0 на меньшем количестве потоков указывает на неоптимальность baseline или measurement artifacts. Ограниченная масштабируемость beyond 4 потоков из-за структурных ограничений алгоритма (7 произведений).

**TBB — роль grain size и runtime:**
Work-stealing планировщик с auto_partitioner показывает время 22.4 мс (0.91 эффективность), демонстрируя роль автоматической балансировки нагрузки. Grain size автоматически адаптируется для 7 произведений Штрассена, но task creation overhead становится заметным по сравнению с прямым управлением потоками. Runtime система TBB обеспечивает хорошую утилизацию ресурсов (0.91 эффективность), но ценой дополнительных вычислительных затрат на управление задачами. Parallel_invoke эффективно распределяет неоднородные подзадачи, но не может преодолеть фундаментальные ограничения алгоритма.

**STL — цена ручного управления потоками:**
Демонстрирует лучшую абсолютную производительность (13.5 мс), но за счет полной ответственности за управление потоками. Round-robin распределение строк требует тщательного проектирования для предотвращения load imbalance и гонок данных. Отсутствие runtime overhead библиотек дает преимущество в производительности, но увеличивает сложность разработки. Необходимость явного управления join() и синхронизацией создает риски ошибок программирования. Статическая балансировка уступает динамическим планировщикам при неоднородной нагрузке, но показывает превосходные результаты для регулярных задач.

**ALL — цена коммуникации и выигрыш гибридности:**
Конфигурация 3×2 (14.2 мс, 4.30x ускорение) демонстрирует успешную амортизацию MPI overhead через достаточно крупные подзадачи. Коммуникационные расходы (~3.6 MB для передачи подматриц) составляют минимальную долю от времени вычислений благодаря высокой computational intensity алгоритма Штрассена. 

Выигрыш гибридности проявляется в оптимальном распределении 7 произведений между 3 процессами (2-3-2), что минимизирует idle time и максимизирует утилизацию ресурсов. Иерархический параллелизм позволяет преодолеть ограничения pure shared-memory подходов, показывая конкурентоспособность с лучшими STL результатами. MPI+TBB комбинация обеспечивает масштабируемость beyond одного узла при сохранении эффективности локальных вычислений через work-stealing планировщик.

## 8. Заключение

**Лучшая версия:** ALL (3×2 конфигурация) с ускорением 4.30x и эффективностью 0.72 представляет собой оптимальное решение для данной задачи, демонстрируя превосходство иерархического параллелизма над чистыми shared-memory подходами.

### Условия применимости:
- **TBB (3 workers)** - для задач, требующих автоматической балансировки с минимальными изменениями кода
- **OpenMP (4 threads)** - для регулярных вычислений с хорошим линейным масштабированием
- **STL (4-6 threads)** - когда требуется полный контроль над управлением потоками
- **ALL (3×2)** - для scaling beyond одного узла и максимальной производительности

**Ограничения сравнения:** Замеры выполнялись в WSL environment с ограниченными ресурсами (6 ядер, 8GB RAM), что может не отражать поведение на production кластерах с быстрой межузловой связью. Алгоритм Штрассена показывает ограниченную масштабируемость beyond 6-8 workers из-за структурных особенностей (7 произведений на уровень).

### Возможные улучшения:
- Гибридная OpenMP+MPI версия для сравнения с TBB+MPI
- Адаптивный порог переключения на наивное умножение в зависимости от количества потоков
- NUMA-aware распределение памяти для больших матриц
- Векторизация базовых операций для дополнительного ускорения
