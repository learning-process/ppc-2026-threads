## 1. Контекст

Для данной задачи выбрана oneTBB версия как представитель задачно-ориентированной модели параллелизма. В отличие от OpenMP с явным управлением потоками, TBB предоставляет высокоуровневые конструкции для декомпозиции задач с автоматическим балансированием нагрузки и work-stealing планировщиком. Алгоритм Штрассена хорошо подходит для TBB благодаря структурированным циклам в наивном умножении и естественному разбиению на независимые подзадачи при вычислении 7 произведений.

## 2. Постановка задачи

Входные и выходные данные идентичны последовательной версии: умножение двух плотных квадратных матриц A и B размером n×n с получением результата C = A×B. Подробная постановка задачи, ограничения и критерии корректности документированы в `seq/report.md`. Цель TBB-версии — продемонстрировать эффективность задачно-ориентированного параллелизма с автоматическим управлением ресурсами.

## 3. Базовый алгоритм

Алгоритм Штрассена O(n^2.807) с итеративной реализацией, паддингом до степеней двойки и порогом переключения на наивное умножение при n ≤ 64. Структура алгоритма полностью сохраняется из последовательной версии для корректного сравнения производительности.

## 4. Схема распараллеливания

### Выбранный примитив: `parallel_for`

Используется `oneapi::tbb::parallel_for` как наиболее подходящий примитив для параллелизации структурированных циклов с независимыми итерациями.

### Область распараллеливания 1: Наивное умножение

```cpp
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<int>(0, n), 
    [&](const oneapi::tbb::blocked_range<int> &range) {
        for (int i = range.begin(); i < range.end(); ++i) {
            for (int k = 0; k < n; ++k) {
                const double aik = a[static_cast<size_t>((static_cast<ptrdiff_t>(i) * n) + k)];
                for (int j = 0; j < n; ++j) {
                    c[static_cast<size_t>((static_cast<ptrdiff_t>(i) * n) + j)] +=
                        aik * b[static_cast<size_t>((static_cast<ptrdiff_t>(k) * n) + j)];
                }
            }
        }
    });
```

**Диапазон работы и grainsize:**
- Диапазон: blocked_range<int>(0, n) — разбиение по строкам матрицы от 0 до n-1
- Grainsize: используется значение по умолчанию (неявно 1), что позволяет TBB runtime автоматически выбирать оптимальный размер блока на основе системных характеристик
- Обоснование: для матриц 512×512 каждая строка содержит достаточно работы (512² операций), поэтому мелкая грануляция не создает критического overhead

### Область распараллеливания 2: Вычисление 7 произведений Штрассена

```cpp
oneapi::tbb::parallel_for(
    oneapi::tbb::blocked_range<int>(0, 7), 
    [&](const oneapi::tbb::blocked_range<int> &range) {
        for (int k = range.begin(); k < range.end(); ++k) {
            // Полное наивное умножение для k-того произведения
            m.at(uk) = std::move(c);
        }
    });
```
**Диапазон работы и grainsize:**
- Диапазон: blocked_range<int>(0, 7) — распределение 7 независимых произведений
- Grainsize: по умолчанию 1, каждая задача выполняет полное умножение матриц размером n/2 × n/2
- Обоснование: каждое произведение является крупной вычислительной задачей O((n/2)³), обеспечивая достаточную грануляцию

**Partitioner:**
Используется auto_partitioner по умолчанию, который:
- Автоматически адаптирует размер блоков в зависимости от доступных worker-потоков
- Применяет work-stealing для балансировки нагрузки
- Минимизирует overhead от создания задач при сохранении параллелизма

**Контроль конкуренции:**
TBB автоматически ограничивает количество worker-потоков через:
- global_control::max_allowed_parallelism (неявно установлено в количество логических ядер)
- Task scheduler автоматически создает оптимальное число worker threads
- Work-stealing обеспечивает равномерное распределение нагрузки между потоками

### Локальные аккумуляторы и объединение результатов:
**В наивном умножении:**
- Каждый worker-поток обрабатывает диапазон строк [range.begin(), range.end())
- Результат записывается напрямую в общий вектор c без промежуточных аккумуляторов
- Отсутствие гонок гарантировано: каждый элемент c[i*n+j] модифицируется только потоком, ответственным за строку i

**При вычислении 7 произведений:**
- Каждый worker выполняет полное наивное умножение и записывает результат в уникальный элемент m[k]
- Используются локальные векторы c внутри lambda для промежуточных результатов
- std::move оптимизирует передачу результата в итоговый массив

**Работа с памятью:**
- Все аллокации используют стандартный std::allocator
- Нет специальной оптимизации для false sharing, поскольку доступ к памяти структурирован по строкам
- TBB runtime обеспечивает NUMA-aware распределение задач автоматически

**Комментарий о балансировке и overhead:**

Преимущества TBB балансировки:
- Work-stealing scheduler автоматически перераспределяет задачи между worker threads при неравномерной нагрузке
- Auto_partitioner адаптивно изменяет размер блоков в зависимости от доступных ресурсов

**Overhead и ограничения:**
- Создание и управление задачами добавляет небольшой overhead по сравнению с прямым управлением потоками
- При малом количестве задач (7 произведений) преимущества work-stealing менее заметны
- Task-oriented подход показывает лучшие результаты при неравномерном распределении вычислительной нагрузки

## 9. Выводы

### Когда TBB удобнее:

**Автоматическое управление ресурсами:** TBB демонстрирует лучшую эффективность (4.37x ускорение на 6 workers) благодаря адаптивному планировщику, который автоматически оптимизирует распределение задач без необходимости ручной настройки.

**Масштабируемость:** Work-stealing планировщик обеспечивает лучшую балансировку нагрузки при неоднородных задачах, что особенно важно для алгоритма Штрассена с переменным размером подзадач на разных уровнях рекурсии.

**Простота интеграции:** Замена последовательных циклов на parallel_for требует минимальных изменений кода при сохранении читаемости и корректности.

### Когда overhead мешает:

**Малые задачи:** Для матриц n < 128 overhead от создания задач может превышать выигрыш от параллелизации

**Простые вычисления:** При операциях с низкой вычислительной интенсивностью (например, простое сложение векторов) overhead task scheduler становится заметным

**Детерминированные требования:** Если требуется строгий контроль над распределением потоков и планированием, явное управление через OpenMP может быть предпочтительнее

TBB версия успешно демонстрирует преимущества задачно-ориентированного параллелизма для структурированных вычислительных алгоритмов, обеспечивая отличное соотношение производительности и простоты разработки.
