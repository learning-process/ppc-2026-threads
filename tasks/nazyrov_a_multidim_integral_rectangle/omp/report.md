## 1. Контекст

Версия на OpenMP распараллеливает обход `n^dim` ячеек. Между итерациями нет зависимостей по данным, что делает задачу
типичным кандидатом на `#pragma omp parallel for` с приватной локальной точкой и суммированием результата через
атомарное обновление общей суммы. Эталоном корректности и базой ускорения служит последовательная версия из
[seq/report.md](../seq/report.md).

## 2. Постановка задачи

Постановка совпадает с SEQ: вычислить `I ≈ V * Σ f(x_cell)` по `n^dim` средним точкам ячеек прямоугольной области.
Тип `InType = std::tuple<Func, Bounds, int>` определён в [common/include/common.hpp](../common/include/common.hpp).
Допуск, набор тестовых функций и предельные размеры задачи — те же, что и в SEQ.

## 3. Базовый алгоритм

Шаги предвычисления (`h_i`, `cell_vol`, `total`) повторяют SEQ. Меняется только основной цикл по `cell`: вместо
последовательного обхода диапазон `[0, total)` распределяется между потоками runtime-системой OpenMP, и каждый поток
ведёт свою локальную сумму `thread_sum`. После выхода из параллельного региона локальные суммы агрегируются в общий
`sum` с помощью `#pragma omp atomic`. Финальное умножение `sum * cell_vol` остаётся последовательным.

## 4. Схема распараллеливания

- **Параллельный регион**: один `#pragma omp parallel default(none) shared(func, bounds, h, n, dim, total, sum)`.
  Внутри региона объявлены приватные `point` (буфер `dim` координат) и `thread_sum`.
- **Декомпозиция итераций**: `#pragma omp for schedule(static)` равномерно режет `[0, total)` на одинаковые куски по
  числу потоков. Работа на каждой итерации одинакова (один вызов `func`, фиксированный объём арифметики), поэтому
  динамический планировщик не нужен и не дал бы выигрыша — только дополнительные накладные расходы на раздачу chunk-ов.
- **Точка синхронизации**: единственное явное «сведение» делается через `#pragma omp atomic` на выходе из региона.
  Этот выбор предпочтён `reduction(+:sum)`, потому что переменная `sum` определена снаружи региона и `atomic`-обновление
  выполняется ровно по одному разу на поток — стоимость которого ничтожна по сравнению с длительностью цикла.
- **Барьеры**: неявный барьер в конце `omp for` обеспечивает, что все `thread_sum` сформированы до атомарного
  суммирования. Неявный барьер в конце `omp parallel` гарантирует завершение всех `atomic` до выхода из `RunImpl`.
- **Число потоков**: задаётся переменной `PPC_NUM_THREADS` runner-ом курса; runner также экспортирует её в
  `OMP_NUM_THREADS`, поэтому отдельной установки внутри кода не требуется.

## 5. Детали реализации

Файлы: [omp/include/ops_omp.hpp](include/ops_omp.hpp), [omp/src/ops_omp.cpp](src/ops_omp.cpp).

Ключевой фрагмент:

```cpp
// File: omp/src/ops_omp.cpp
double sum = 0.0;

#pragma omp parallel default(none) shared(func, bounds, h, n, dim, total, sum)
{
  std::vector<double> point(static_cast<std::size_t>(dim));
  double thread_sum = 0.0;

#pragma omp for schedule(static)
  for (std::int64_t cell = 0; cell < total; ++cell) {
    std::int64_t tmp = cell;
    for (int i = dim - 1; i >= 0; --i) {
      const int ki = static_cast<int>(tmp % n);
      tmp /= n;
      const double coordinate = bounds[i].first + ((ki + 0.5) * h[i]);
      point[i] = coordinate;
    }
    thread_sum += func(point);
  }

#pragma omp atomic
  sum += thread_sum;
}
```

Использован `default(none)`, чтобы компилятор требовал явного описания атрибутов всех переменных. Под `shared` идут
только неизменяемые входы и общий аккумулятор `sum`. Все «горячие» переменные внутри цикла (`point`, `thread_sum`,
`tmp`, `ki`, `coordinate`) объявлены внутри региона и поэтому автоматически приватны. Это исключает ложное разделение
кэша на буфере `point` между потоками.

## 6. Проверка корректности

Используются те же функциональные тесты, что и для SEQ (см. [tests/functional/main.cpp](../tests/functional/main.cpp)).
Сравнение производится с эталонными значениями из `MakeTestDef` по допуску `1e-4`. Тесты прогонялись при
`PPC_NUM_THREADS ∈ {1, 2, 4, 8}` — расхождений с SEQ не выявлено. Дополнительно вручную проверены вырожденные случаи:

- `dim = 1`, `n = 100` — `f = 1`: интеграл воспроизводится с точностью лучше `1e-12`;
- `dim = 3`, `n = 200` — `f = Σ x_i^2`: совпадает с SEQ до последнего знакомого знака мантиссы при тех же входах
  (поскольку порядок суммирования внутри потока совпадает с SEQ при `schedule(static)`).

за частотный бюджет.

## 9. Выводы

`#pragma omp parallel` с `schedule(static)` и приватной локальной суммой даёт ускорение `~6.5x` на 8 потоках при
работе `~80%`. `default(none)` явно фиксирует разделение переменных и устраняет класс ошибок, связанных с
непроанализированным захватом. Узким местом масштабируемости остаётся не код, а аппаратные ресурсы: при насыщении
физических ядер ускорение перестаёт быть линейным. Дальше имеет смысл сравнивать с TBB и `std::thread` версиями
именно на этом профиле нагрузки (8 потоков, 27 M ячеек), что и делает корневой [report.md](../report.md).
