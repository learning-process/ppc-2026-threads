## 1. Контекст

Гибридная версия добавляет к OpenMP-параллелизму внутри процесса распараллеливание между процессами через MPI.
На небольшой одно-узловой машине это не даёт принципиального ускорения по сравнению с чистым OMP, но именно эта
реализация демонстрирует разделение задачи на двух уровнях: между рангами (статическое блочное распределение
диапазона ячеек) и внутри ранга (`#pragma omp parallel for` с атомарной редукцией). Эталоном корректности и базой
ускорения остаётся последовательная версия ([seq/report.md](../seq/report.md)).

## 2. Постановка задачи

Та же, что в SEQ: вычислить `I ≈ V * Σ f(x_cell)`. Существенное отличие — в гибридной модели результат должен
агрегироваться только на ранге `0` (на нём же выполняется валидация входа), а остальные ранги после `MPI_Reduce`
не пишут в `GetOutput()`.

## 3. Базовый алгоритм

1. Все ранги одинаково знают вход (тесты курса подают `InType` в каждый ранг через тест-инфраструктуру).
2. Каждый ранг вычисляет общие `h_i`, `cell_vol`, `total`.
3. Глобальный диапазон `[0, total)` делится между `world_size` рангами по блочной схеме с учётом остатка:
   `cells_per_rank = total / world_size`, `extra = total % world_size`, ранг `r` получает
   `[r*cells_per_rank + min(r, extra), ...)` длиной `cells_per_rank + (r < extra ? 1 : 0)`.
4. Внутри ранга свой кусок параллелится OpenMP-блоком с приватной точкой и `atomic`-редукцией в `local_sum`.
5. `MPI_Reduce(MPI_SUM)` собирает `local_sum` со всех рангов на ранг `0`.
6. На ранге `0` записывается `GetOutput() = global_sum * cell_vol`.

## 4. Межпроцессная схема

- **Коммуникатор**: `MPI_COMM_WORLD` — единственный коммуникатор, используемый в задаче.
- **Распределение работы**: фиксированное блочное по `cell`-индексу. Каждый ранг знает свой `[begin, end)` без
  обмена: распределение чисто арифметическое.
- **Коммуникации**: ровно один обмен на запуск — `MPI_Reduce` с операцией `MPI_SUM` над одним `MPI_DOUBLE`.
  Это даёт минимальные накладные расходы; рост числа рангов влияет на масштабируемость только через
  доступные физические ядра, а не через объём пересылок.
- **Валидация и постобработка**: `ValidationImpl` на ранге `0` проверяет вход, остальные ранги возвращают `true`
  без чтения входа (на них валидация не нужна и она ничего не меняет в логике распределения).
  `PostProcessingImpl` тривиальна, ответ уже лежит в `GetOutput()` ранга `0`.
- **Барьеры**: явный `MPI_Barrier` не нужен, потому что `MPI_Reduce` уже является блокирующей коллективной
  операцией — она сама выступает точкой синхронизации.
- **Запрет двойной записи в `GetOutput()`**: финальная присваивающая запись делается под условием `rank == 0`.
  Это устраняет неоднозначность при чтении результата тестом из MPI-окружения.

## 5. Внутрипроцессная схема

- **Технология внутри ранга**: OpenMP. Использован тот же шаблон, что и в [omp/report.md](../omp/report.md):
  `#pragma omp parallel default(none) shared(...)` плюс `#pragma omp for schedule(static)`.
- **Локальный аккумулятор**: каждый поток ведёт `thread_sum`, в конце региона делает `#pragma omp atomic +=` в
  `local_sum`. `local_sum` затем уходит в `MPI_Reduce`.
- **Декомпозиция**: `schedule(static)` по `[begin, end)` ранга равномерно распределяет работу между OMP-потоками
  ранга. Внутрипроцессного дисбаланса нет — стоимость итерации одинакова.
- **Число потоков на ранг**: задаётся `PPC_NUM_THREADS` (экспортируется runner-ом в `OMP_NUM_THREADS`).
  Число рангов — `PPC_NUM_PROC`. В таблицах ниже конфигурация обозначена парой `ranks × threads_per_rank`.

## 6. Детали реализации

Файлы: [all/include/ops_all.hpp](include/ops_all.hpp), [all/src/ops_all.cpp](src/ops_all.cpp).

Распределение диапазона и гибридное вычисление:

```cpp
// File: all/src/ops_all.cpp
const auto rk = static_cast<std::int64_t>(rank);
const auto ws = static_cast<std::int64_t>(world_size);
const std::int64_t cells_per_rank = total / ws;
const std::int64_t extra = total % ws;
const std::int64_t begin = (rk * cells_per_rank) + std::min(rk, extra);
const std::int64_t end = begin + cells_per_rank + (rk < extra ? 1LL : 0LL);

double local_sum = 0.0;

#pragma omp parallel default(none) shared(func, bounds, h, n, dim, begin, end, local_sum)
{
  std::vector<double> point(static_cast<std::size_t>(dim));
  double thread_sum = 0.0;

#pragma omp for schedule(static)
  for (std::int64_t cell = begin; cell < end; ++cell) {
    // декодирование индекса и thread_sum += func(point);
  }

#pragma omp atomic
  local_sum += thread_sum;
}

double global_sum = 0.0;
MPI_Reduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);

if (rank == 0) {
  GetOutput() = global_sum * cell_vol;
}
```

Карта «ранг → диапазон» (для `total = 27 000 000`, `world_size = 4`, `extra = 0`):

| `rank` |   `begin`   |    `end`    |  `end - begin` |
|:------:|------------:|------------:|---------------:|
|   0    |          0  |   6 750 000 |     6 750 000  |
|   1    |   6 750 000 |  13 500 000 |     6 750 000  |
|   2    |  13 500 000 |  20 250 000 |     6 750 000  |
|   3    |  20 250 000 |  27 000 000 |     6 750 000  |

Если `total % world_size != 0`, остаток `extra` раздаётся первым `extra` рангам по одной ячейке через
`(rk < extra ? 1 : 0)` — та же схема, что в STL-версии для потоков.

## 10. Выводы

Гибридная версия не даёт выигрыша на единственном узле сверх лучшего «потокового» бэкэнда: при `total_workers ≤ 8`
все конфигурации `ranks × threads` укладываются в окрестность `6.0–6.3x` ускорения (то же значение, что и у OMP и
STL). MPI здесь добавляет очень дешёвую коммуникацию (`MPI_Reduce` на одном double), но не открывает
дополнительных физических ресурсов. Эта версия становится оправданной только при выходе за пределы одного узла,
когда несколько физических машин могут одновременно обрабатывать свои поддиапазоны. Внутри узла оптимальная
конфигурация — это либо «1 ранг × N потоков», либо «N рангов × 1 поток», но не их произведение, превышающее
число физических ядер.
