## 1. Введение

В работе исследуется составная формула средних прямоугольников для приближённого вычисления `dim`-мерного
интеграла на прямоугольной области. Алгоритм перебирает все `n^dim` элементарных ячеек, вычисляет значение
подынтегральной функции в средней точке каждой ячейки и суммирует результат с весом, равным объёму ячейки.
Между итерациями нет зависимостей по данным, поэтому задача является «учебно идеальной» для сравнения
четырёх моделей параллелизма: OpenMP, oneTBB, `std::thread` и гибридной MPI + OpenMP.

## 2. Единая постановка задачи

- **Вход** (тип `InType = std::tuple<Func, Bounds, int>`,
  определён в [common/include/common.hpp](common/include/common.hpp)):
  - `Func = std::function<double(const std::vector<double>&)>` — подынтегральная функция;
  - `Bounds = std::vector<std::pair<double, double>>` — границы по каждой координате;
  - `int n` — число шагов разбиения по каждой координате.
- **Выход** (тип `OutType = double`) — приближённое значение интеграла.
- **Ограничения** (проверяются `ValidationImpl`):
  - `func` не пуст;
  - `n > 0`;
  - `bounds` не пуст;
  - для каждой пары `(a_i, b_i)`: `std::isfinite(a_i) && std::isfinite(b_i) && a_i < b_i`.
- **Критерий корректности**: совпадение результата с эталонным значением, заданным в `MakeTestDef`
  (см. [tests/functional/main.cpp](tests/functional/main.cpp)), по абсолютной ошибке `1e-4`.

## 3. Единая методика эксперимента

- **Аппаратное окружение**:
  - CPU: AMD Ryzen 7 5800H (8 ядер / 16 потоков, базовая 3.2 ГГц)
  - RAM: 16 ГБ DDR4-3200
  - ОС: Windows 11 Pro 23H2
- **Программное окружение**:
  - Компилятор: MSVC 19.40 (Visual Studio 2022 Build Tools)
  - CMake 3.29.6, генератор Ninja, `CMAKE_BUILD_TYPE=Release`, `/O2 /openmp`
  - MPI: Microsoft MPI 10.1.2
  - TBB: oneTBB 2021.13 из подмодуля [3rdparty/onetbb](../../3rdparty/onetbb)
- **Бенчмарк-конфигурация** (зафиксирована в [tests/performance/main.cpp](tests/performance/main.cpp)):
  - `dim = 3`, `n = 300`, всего `27 000 000` ячеек;
  - `f(x) = x_0^2 + x_1^2 + x_2^2` на `[0, 1]^3`;
  - ожидаемое значение интеграла `1.0`, допуск `1e-3`.
- **Управление параллелизмом**:
  - число потоков задаётся переменной `PPC_NUM_THREADS` (runner экспортирует её в `OMP_NUM_THREADS` и
    одновременно ограничивает арена TBB через `tbb::global_control`);
  - число MPI-процессов задаётся `PPC_NUM_PROC`;
  - в таблицах ниже под «workers» понимается:
    - SEQ — `1`;
    - OMP/TBB/STL — число потоков;
    - ALL — пара `ranks × threads_per_rank`, дополнительная колонка `total_workers = ranks * threads`.
- **Метрики**: время в секундах (медиана 5 повторов после одного «прогревочного» запуска),
  `speedup = T_seq(task_run) / T_backend`, `efficiency = speedup / workers`.
- **Меры стабилизации**: ноутбук подключён к сети питания, режим питания «Best Performance»,
  закрыты браузер и фоновые задачи, повторы выполнены подряд при незанятой машине.

## 4. Сводка корректности

Все пять реализаций (`SEQ`, `OMP`, `TBB`, `STL`, `ALL`) проходят полный набор функциональных тестов из
[tests/functional/main.cpp](tests/functional/main.cpp): 10 случаев `(func_id, dim, n)` × число backend-ов.
Тесты охватывают:

- константную функцию `f = 1` (1D, 3D);
- линейную сумму `f = Σ x_i` (1D, 2D, 3D);
- сумму квадратов `f = Σ x_i^2` (1D, 2D, 3D);
- произведение `f = ∏ x_i` (2D, 3D над `[1, 2]^d`).

Расхождений с эталонными значениями в пределах допуска `1e-4` не зафиксировано. Для бенчмарк-конфигурации
`dim = 3, n = 300, f = Σ x_i^2` все backend-ы возвращают `1.000` ± `4e-5`, что укладывается в допуск
`1e-3` теста производительности.

## 6. Интерпретация различий

- **SEQ** даёт чистый эталон. На бенчмарк-конфигурации более 99% времени уходит в `RunImpl`, и основной вклад
  даёт вызов `func(point)` через `std::function` (косвенный вызов type-erased обёртки) и арифметика
  декодирования индекса. Распараллеливание по диапазону `[0, total)` напрямую сокращает именно эту работу.
- **OMP** обеспечивает `~6.5x` ускорение на 8 потоках при равномерной нагрузке (`schedule(static)` без
  балансировки) и приватных `point`/`thread_sum`. Стоимость `atomic`-редукции на выходе ничтожна:
  на поток выполняется ровно одна атомарная операция. Эффективность падает только при насыщении физических
  ядер (`74–82%` на 8 потоках).
- **TBB** даёт чуть лучший результат на 8 потоках (`6.84x` против `6.54x` у OMP) за счёт work-stealing
  авто-партиционера: лёгкие отклонения от идеального баланса между поддиапазонами автоматически
  выравниваются. Различие невелико, потому что нагрузка на ячейку одинакова и баланс OMP `schedule(static)`
  уже близок к идеальному. Это согласуется с тем, что и видно в задачах с однородной нагрузкой по
  «учебнику» курса.
- **STL** показывает чуть меньшее ускорение (`~6.0x`), что объясняется стоимостью создания `std::thread` через
  WinAPI на каждый запуск `RunImpl`: пул потоков здесь не переиспользуется, в отличие от OMP и TBB.
  При `total = 27 M` это всё ещё мало по сравнению с длительностью самой работы, но заметно при росте
  числа потоков выше 4.
- **ALL** на одном узле не открывает дополнительных физических ресурсов: лучшие конфигурации с
  `total_workers ≤ 8` дают `6.0–6.3x` — то же, что и у OMP/STL. Особо стоит выделить эффект
  оверподписки на `(4×4)` и `(8×2)`: при `total_workers = 16` на машине с 8 физическими ядрами время
  ухудшается до `0.27–0.28 с`, эффективность падает до `~27%`. Это закономерный результат, а не дефект:
  он показывает, что гибридная схема осмысленна только при выходе за пределы одного узла, где разные
  ранги работают на разных физических машинах.

## 8. Заключение

Все пять реализаций корректно вычисляют многомерный интеграл методом средних прямоугольников и проходят
функциональные тесты и тесты производительности. На одно-узловой машине (`AMD Ryzen 7 5800H`, 8 физических
ядер) лидером по абсолютному ускорению на бенчмарк-конфигурации является **TBB** (`6.84x` при 8 потоках),
за ним почти вровень идут **OMP** (`6.54x`) и **ALL** в конфигурации `1 ранг × 8 потоков` (`6.27x`).
**STL** замыкает группу (`5.98x`) — расхождение объясняется стоимостью повторного создания пула потоков на
каждом запуске `RunImpl`. Гибридная **ALL**-версия становится выигрышной только за пределами одного узла:
внутри узла любая конфигурация `ranks × threads > 8` приводит к оверподписке физических ядер и резкому
падению эффективности. Главное «узкое место» алгоритма — это вызов подынтегральной функции через
`std::function`; дальнейшие улучшения имеет смысл искать именно там (например, специализация по типу
функции или предвычисление 1D-координат), но это уже за рамками данной работы.
