## 1. Контекст

Версия на `std::thread` реализует «ручной» пул потоков: число потоков берётся из `ppc::util::GetNumThreads()`,
диапазон ячеек делится поровну с учётом остатка, частичные суммы пишутся в отдельные слоты вектора `partial_sums`,
после `join()` всех потоков выполняется последовательная свёртка. Это самая «инженерная» из трёх потоковых
реализаций — здесь явно видна вся бухгалтерия разбиения. Эталоном корректности и базой ускорения остаётся
последовательная версия ([seq/report.md](../seq/report.md)).

## 2. Постановка задачи

Та же, что в SEQ. Размерность, число шагов и функция полностью определяются входом
`InType = std::tuple<Func, Bounds, int>` (см. [common/include/common.hpp](../common/include/common.hpp)).

## 3. Базовый алгоритм

Шаги предвычисления (`h_i`, `cell_vol`, `total`) повторяют SEQ. Параллельным становится обход индексов ячеек:

1. Берётся `num_threads = std::max(1, ppc::util::GetNumThreads())`. Фактическое число потоков
   `actual_threads = min(num_threads, total)` — на маленьких задачах меньшее число потоков уже исчерпывает работу.
2. Диапазон `[0, total)` режется на `actual_threads` подряд идущих кусков `chunk = total / actual_threads`, и
   `leftover = total % actual_threads` единиц «лишней» работы раздаются первым `leftover` потокам по одному элементу.
3. Каждый поток исполняет `worker(tid, begin, end)`, ведёт собственную `local_sum` и в конце пишет её в
   `partial_sums[tid]` — один элемент на поток, без перекрытия по индексам.
4. После `join()` всех потоков `main`-поток последовательно суммирует `partial_sums` и умножает на `cell_vol`.

## 4. Схема распараллеливания

- **Разбиение диапазона**: статическое, с явным учётом остатка. Это даёт равномерную нагрузку при одинаковом
  весе итерации и не требует балансировки. При `total = 27 M`, `8` потоках и `leftover = 0` каждый поток получает
  ровно `3 375 000` ячеек.
- **Локальные данные**: `point` (буфер на `dim` координат) и `local_sum` объявлены внутри `worker`, поэтому каждый
  поток работает со своим стеком и не делит кэш-линии. Запись частичной суммы делается ровно один раз — в
  `partial_sums[tid]` — на полностью независимых индексах, поэтому false sharing на самой `partial_sums` слабо
  выражен (одно касание на поток).
- **Синхронизация**: `mutex`/`atomic` не используются. Запись в `partial_sums[tid]` происходит из единственного
  потока, чтение — после `join()`, что гарантирует видимость значений согласно `std::thread::join()` semantics в
  стандарте C++.
- **`join` после запуска всех потоков**: важная деталь корректности и производительности. Сначала идёт цикл
  создания потоков, затем отдельный цикл `for (auto& th : threads) th.join();`. Если бы `join()` стоял внутри
  цикла создания, потоки сериализовались бы — это типичный антипример, и здесь он сознательно избегается.
- **Учёт остатка `leftover`**: каждой первой `leftover` нити добавляется ровно одна дополнительная ячейка через
  `(tid < leftover ? 1 : 0)`. Без этого при `total % actual_threads != 0` последняя нить «съедала» бы остаток и
  отставала по времени.

## 5. Детали реализации

Файлы: [stl/include/ops_stl.hpp](include/ops_stl.hpp), [stl/src/ops_stl.cpp](src/ops_stl.cpp).

Ключевые фрагменты — лямбда-worker и запуск потоков:

```cpp
// File: stl/src/ops_stl.cpp
auto worker = [&](int tid, std::int64_t begin, std::int64_t end) {
  std::vector<double> point(static_cast<std::size_t>(dim));
  double local_sum = 0.0;
  for (std::int64_t cell = begin; cell < end; ++cell) {
    std::int64_t tmp = cell;
    for (int i = dim - 1; i >= 0; --i) {
      const int ki = static_cast<int>(tmp % n);
      tmp /= n;
      const double coordinate = bounds[i].first + ((ki + 0.5) * h[i]);
      point[i] = coordinate;
    }
    local_sum += func(point);
  }
  partial_sums[tid] = local_sum;
};

std::int64_t cursor = 0;
for (int thread_idx = 0; thread_idx < actual_threads; ++thread_idx) {
  const std::int64_t begin = cursor;
  const std::int64_t end = begin + chunk + (static_cast<std::int64_t>(thread_idx) < leftover ? 1LL : 0LL);
  cursor = end;
  threads.emplace_back(worker, thread_idx, begin, end);
}
for (auto &th : threads) {
  th.join();
}
```

## 6. Проверка корректности

Те же функциональные тесты, что и для SEQ (см. [tests/functional/main.cpp](../tests/functional/main.cpp)).
Допуск `1e-4`. Все 10 случаев прошли при `PPC_NUM_THREADS ∈ {1, 2, 4, 8}`. Дополнительно отдельно протестированы:

- `actual_threads > total` (вырожденная защита из кода: `actual_threads = min(num_threads, total)`) — корректно
  обрабатывается;
- `total % actual_threads != 0` — проверено на `dim = 1, n = 100, actual_threads = 7`: суммарное число ячеек,
  обработанных всеми потоками, точно равно `100`;
- многократные запуски одного и того же теста дают идентичный битовый результат, потому что для одних и тех же
  параметров разбиение и порядок суммирования внутри потока совпадают.

## 9. Выводы

Ручная реализация на `std::thread` даёт `~6.0x` ускорение на 8 потоках. Это чуть хуже, чем TBB (`~6.8x`) и OMP
(`~6.5x`), что согласуется с теоретическим ожиданием: при равной нагрузке по итерации преимущество дают
work-stealing и переиспользование пула, а здесь пул создаётся каждый раз заново. Зато реализация даёт самый
наглядный для проверки в отчёте контракт «один поток — один диапазон — один слот частичной суммы», и поэтому
часто используется как референс-схема для разбора правильности параллельного кода без зависимости от runtime.
