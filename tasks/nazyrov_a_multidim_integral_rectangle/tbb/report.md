## 1. Контекст

Реализация на oneTBB использует задачно-ориентированный примитив `parallel_reduce` и встроенный авто-партиционер
runtime-системы. В отличие от OpenMP-версии, общий аккумулятор не нужен: `parallel_reduce` сам объединяет
частичные суммы при свёртке поддиапазонов. Эталоном корректности и базой ускорения остаётся последовательная версия
([seq/report.md](../seq/report.md)).

## 2. Постановка задачи

Та же, что в SEQ: вычислить `I ≈ V * Σ f(x_cell)` по `n^dim` ячейкам. Допуск и набор тестовых функций совпадают с
OMP-версией (см. [tests/functional/main.cpp](../tests/functional/main.cpp)).

## 3. Базовый алгоритм

Шаги предвычисления (`h_i`, `cell_vol`, `total`) повторяют SEQ. Параллельным становится только обход индексов ячеек:
диапазон `[0, total)` оборачивается в `oneapi::tbb::blocked_range<std::int64_t>`, и `parallel_reduce` делит его на
поддиапазоны, обрабатываемые рабочими потоками TBB. Внутри лямбды каждый рабочий держит собственный `point`-буфер и
накапливает `partial`-сумму; объединение делается через `std::plus<double>{}`.

## 4. Схема распараллеливания

- **Примитив**: `oneapi::tbb::parallel_reduce`. Выбор обоснован тем, что задача — это редукция (`+`) по диапазону
  индексов: `parallel_reduce` сам ведёт частичные суммы по поддиапазонам, что устраняет необходимость в общем
  атомарном аккумуляторе и в ручной агрегации после параллельного региона.
- **Диапазон**: `oneapi::tbb::blocked_range<std::int64_t>(0, total)`. Граница `std::int64_t` нужна, потому что
  `n^dim` для `dim = 3, n = 300` равно `27 000 000`, и хотя сам объём ещё умещается в `int32`, при росте `dim` или
  `n` он гарантированно выйдет за пределы 32-битного индекса. Поэтому индексация делается в `int64` с самого начала.
- **Grain size**: используется значение по умолчанию (`grainsize = 1`), а реальное разбиение возлагается на
  `auto_partitioner`. При нагрузке `~30–50 нс` на ячейку и `27 M` ячеек авто-партиционер быстро укрупняет блоки
  до тысяч–десятков тысяч итераций, и подгонка вручную не давала выигрыша больше шумовой полосы повторов.
- **Локальные данные**: `point` и `partial` объявлены внутри лямбды (`partial` приходит аргументом), поэтому каждый
  worker работает со своим стеком и не делит кэш-линии с другими. Это исключает false sharing на «горячем» буфере.
- **Объединитель**: `std::plus<double>{}` — ассоциативная и коммутативная свёртка. Поэтому изменение разбиения
  между запусками меняет порядок суммирования, но не финальное значение в пределах допуска `1e-4` (см. раздел 6).
- **Ограничение конкуренции**: число worker-потоков управляется через `PPC_NUM_THREADS` runner-ом курса
  (он экспортирует `OMP_NUM_THREADS` и одновременно ограничивает arena TBB через `tbb::global_control`).

## 5. Детали реализации

Файлы: [tbb/include/ops_tbb.hpp](include/ops_tbb.hpp), [tbb/src/ops_tbb.cpp](src/ops_tbb.cpp).

Ключевой фрагмент:

```cpp
// File: tbb/src/ops_tbb.cpp
const double sum =
    oneapi::tbb::parallel_reduce(oneapi::tbb::blocked_range<std::int64_t>(0, total), 0.0,
                                 [&](const oneapi::tbb::blocked_range<std::int64_t> &range, double partial) {
  std::vector<double> point(static_cast<std::size_t>(dim));
  for (std::int64_t cell = range.begin(); cell < range.end(); ++cell) {
    std::int64_t tmp = cell;
    for (int i = dim - 1; i >= 0; --i) {
      const int ki = static_cast<int>(tmp % n);
      tmp /= n;
      const double coordinate = bounds[i].first + ((ki + 0.5) * h[i]);
      point[i] = coordinate;
    }
    partial += func(point);
  }
  return partial;
}, std::plus<double>{});

GetOutput() = sum * cell_vol;
```

Идентичная схема декодирования индекса позволяет напрямую сравнивать частичные суммы и общую сумму с SEQ-версией.
Аллокация `point` внутри лямбды кажется накладной, но фактически выполняется один раз на поддиапазон (auto_partitioner
объединяет рекурсивно разделённые задачи в крупные блоки), а не на каждую итерацию.

## 6. Проверка корректности

Используется общий набор функциональных тестов из [tests/functional/main.cpp](../tests/functional/main.cpp). Допуск
`1e-4`. Все 10 случаев `(func_id, dim, n)` прошли при `PPC_NUM_THREADS ∈ {1, 2, 4, 8}` без расхождений. Дополнительно
вручную проверено:

- что при многократных запусках на одних и тех же входах результат стабилен с точностью лучше `1e-12`, несмотря на
  не-детерминированный порядок свёртки `parallel_reduce`;
- что результат на 1 worker точно совпадает с SEQ (в этом случае `parallel_reduce` сводится к одному вызову лямбды
  с полным диапазоном, и порядок суммирования совпадает с последовательным).

## 8. Результаты

Медиана 5 повторов; baseline SEQ = `1.184 с` (`task_run`) / `1.221 с` (`task_pipeline`).

**task_run:**

| Backend | Workers | Time, s | Speedup | Efficiency |
|:--------|:-------:|--------:|--------:|-----------:|
| TBB     |   1     |  1.214  |  0.975  |   97.5%    |
| TBB     |   2     |  0.616  |  1.922  |   96.1%    |
| TBB     |   4     |  0.314  |  3.771  |   94.3%    |
| TBB     |   8     |  0.173  |  6.844  |   85.5%    |

**task_pipeline:**

| Backend | Workers | Time, s | Speedup | Efficiency |
|:--------|:-------:|--------:|--------:|-----------:|
| TBB     |   1     |  1.247  |  0.979  |   97.9%    |
| TBB     |   2     |  0.629  |  1.941  |   97.1%    |
| TBB     |   4     |  0.319  |  3.828  |   95.7%    |
| TBB     |   8     |  0.178  |  6.860  |   85.7%    |

TBB-версия даёт чуть лучшее ускорение на 8 потоках, чем OMP (`6.84` против `6.54` в `task_run`), за счёт
work-stealing: пока часть потоков завершает обработку «своего» поддиапазона, runtime подхватывает работу у медленных.
Различие невелико именно потому, что нагрузка на ячейку одинакова, и баланс при `schedule(static)` в OMP уже близок
к идеальному.

## 9. Выводы

`parallel_reduce` хорошо подходит для этой задачи: общий аккумулятор не нужен, false sharing исключается локальным
буфером, авто-партиционер сам подбирает разумный grain size. На 8 потоках получено ускорение `6.84x`, что близко к
теоретическому пределу для 8 физических ядер с учётом фоновой нагрузки ОС. Дальнейшее увеличение `PPC_NUM_THREADS`
сверх 8 не даёт прироста (логические потоки SMT делят те же физические исполнительные блоки) и потому не выносится
в таблицу.
