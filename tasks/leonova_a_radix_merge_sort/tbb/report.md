## 1. Введение

Версия с использованием Intel oneTBB (Threading Building Blocks) реализует задачно-ориентированное распараллеливание
 поразрядной сортировки. В отличие от OpenMP, где используются директивы компилятора, TBB предоставляет библиотеку
  высокоуровневых примитивов с планировщиком задач и механизмом work-stealing для динамической балансировки нагрузки.

## 2. Постановка задачи

- **Вход**: Массив `std::vector<int64_t>` целых чисел со знаком (64-бит).
- **Выход**: Отсортированный по возрастанию массив той же длины.
- **Ограничения**:
  - Размер массива `N ≥ 1`.
  - Корректная обработка отрицательных чисел.
  - Результат должен совпадать с отсортированной версией.

## 3. Базовый алгоритм

Последовательная версия реализует гибридную схему:

- Для подмассивов размером **≤ 131072** элементов применяется **поразрядная сортировка**.
- Для подмассивов больше порога — **сортировка слиянием**.

**Поразрядная сортировка** (ключевой этап для распараллеливания):

1. Преобразование знаковых чисел в беззнаковые ключи с инвертированием знакового бита.
2. Для каждого из 8 байтов (от младшего к старшему):
   - Подсчёт гистограммы частот (256 корзин).
   - Вычисление префиксных сумм (позиций записи).
   - Перераспределение элементов во временный массив.
3. Копирование отсортированных элементов обратно.

## 4. Схема распараллеливания

### 4.1 Задачно-ориентированный подход

TBB использует не потоки напрямую, а **задачи** (tasks), которые планировщик динамически распределяет между рабочими
потоками. Это позволяет:

- Автоматически балансировать нагрузку через work-stealing.
- Эффективно использовать кэш-память.
- Избегать ручного управления числом потоков.

### 4.2 Область параллелизации

Распараллелены следующие этапы поразрядной сортировки:

| Этап               | Примитив TBB        | Описание                                     |
|--------------------|---------------------|----------------------------------------------|
| Заполнение ключей  | `tbb::parallel_for` | Преобразование элементов в беззнаковые ключи |
| Подсчёт гистограмм | `tbb::parallel_for` | Каждый поток обрабатывает свой блок данных   |
| Слияние гистограмм | последовательно     | `ReduceCounts` + `BuildOffsets`              |
| Перераспределение  | `tbb::parallel_for` | Запись элементов во временный массив         |

### 4.3 Планирование и гранулярность

```cpp
static constexpr size_t kMinParallelSize = 10000;
```

Для массивов размером `< 10 000` элементов используется последовательная версия (`SequentialRadixSort`).

Порог выбран для минимизации накладных расходов на создание задач.

### 4.4 Разбиение данных

```cpp
std::pair<size_t, size_t> GetChunk(size_t tid, size_t num_threads, size_t size) {
    const size_t chunk = (size + num_threads - 1) / num_threads;
    const size_t begin = tid * chunk;
    const size_t end = std::min(begin + chunk, size);
    return {begin, end};
}
```

Каждый поток получает непрерывный блок данных для обработки (статическое разбиение с округлением вверх).

### 4.5 Управление конкуренцией

```cpp
tbb::task_arena &GetTbbArena() {
    static int num_threads = std::max(1, ppc::util::GetNumThreads());
    static tbb::task_arena arena(num_threads);
    return arena;
}
```

Используется `tbb::task_arena` для ограничения числа рабочих потоков.

Количество потоков задаётся через `PPC_NUM_THREADS`.

### 4.6 Синхронизация

В отличие от OpenMP, TBB не требует явных барьеров:

- `parallel_for` неявно синхронизирует завершение всех итераций.
- Отсутствие гонок данных обеспечивается за счёт:
  - записи в локальные массивы (`local_counts[tid]`);
  - последовательного слияния (`ReduceCounts`, `BuildOffsets`) после завершения параллельных секций.

## 5. Детали реализации

### 5.1 Логика параллельной поразрядной сортировки

```cpp
void RadixSort(std::vector<int64_t>& arr, size_t left, size_t right) {
    if (size < kMinParallelSize) {
        SequentialRadixSort(arr, left, right);
        return;
    }
    
    arena.execute([&] {
        FillUnsignedKeys(arr, left, size, keys, num_threads);
        
        for (int byte_pos = 0; byte_pos < kNumBytes; ++byte_pos) {
            // Обнуление локальных счётчиков
            for (auto& row : local_counts) std::ranges::fill(row, 0);
            
            CountBytesParallel(keys, size, shift, local_counts, num_threads);
            ReduceCounts(local_counts, global_counts);
            BuildOffsets(local_counts, local_offsets, global_counts, num_threads);
            ScatterParallel(keys, arr, left, size, shift,
                           local_offsets, temp_arr, temp_keys, num_threads);
            
            std::ranges::copy(temp_arr, arr.begin() + left);
            keys.swap(temp_keys);
        }
    });
}
```

### 5.2 Гибридная стратегия (Radix + Merge)

- Порог переключения на `Radix Sort`: `131072` элементов.
- Порог переключения на параллельную версию `Radix Sort`: `10 000` элементов.
- Для подмассивов между `10 000` и `131 072` применяется параллельная `Radix Sort`.
- Для подмассивов больше `131 072` — рекурсивное разделение и слияние.

### 5.3 Граничные случаи

- Пустой массив — отклоняется в `ValidationImpl()`.
- Размер `< 10 000` — последовательная `Radix Sort` (избегаем накладных расходов).
- Массив `≤ 131 072` — только поразрядная сортировка.
- Массив `> 131 072` — рекурсивное разделение и слияние.

### 8.2 Анализ масштабируемости

На `1–6` потоках наблюдается значительное ускорение:

- пиковое ускорение `task_run` на `6` потоках: `9.28×`;
- эффективность на `2` потоках достигает `305%` (сверхлинейное ускорение);
- на `6` потоках эффективность остаётся высокой (`155%`).

На `8` потоках производительность резко падает:

- ускорение `pipeline` падает до `0.56×` (медленнее SEQ);
- ускорение `task_run` снижается до `1.28×`.

Причины

- гиперпоточность (8 потоков на 4 физических ядра) неэффективна для данного алгоритма;
- конфликты кэш-памяти и ложное разделение (`false sharing`);
- накладные расходы на синхронизацию при слиянии гистограмм.

### 8.3 Сравнение TBB с OpenMP

| Число потоков | TBB (`task_run`) | OpenMP (`task_run`) |
|---------------|------------------|---------------------|
| 1             | 5.78×            | 8.15×               |
| 2             | 6.09×            | 4.82×               |
| 4             | 7.74×            | 6.65×               |
| 6             | 9.28×            | 4.54×               |
| 8             | 1.28×            | 0.69×               |

Наблюдения

- На `1–4` потоках OpenMP показывает более высокое ускорение.
- На `6` потоках TBB значительно обгоняет OpenMP (`9.28×` против `4.54×`).
- На `8` потоках обе технологии теряют эффективность, но TBB держится лучше.

### 8.4 Преимущества TBB

- `work-stealing` позволяет эффективно использовать `6` потоков на `4`-ядерном процессоре.
- Меньшие накладные расходы на синхронизацию по сравнению с барьерами OpenMP.
- Автоматическая балансировка нагрузки при неравномерном распределении данных.

## 9. Выводы

### 9.1 Что сработало

- Эффективное распараллеливание подсчёта гистограмм и перераспределения через `tbb::parallel_for`.
- Ускорение до `9.28×` на `6` потоках (сверхлинейное).
- Корректная работа с отрицательными и граничными значениями.
- Механизм `work-stealing` обеспечивает хорошую масштабируемость до `6` потоков.
- Порог `kMinParallelSize = 10000` эффективно отсекает накладные расходы на малых данных.

### 9.2 Ограничения

- Гиперпоточность неэффективна (на `8` потоках ускорение падает до `1.28×`).
- Последовательное слияние гистограмм (`ReduceCounts`, `BuildOffsets`) становится узким местом.
- Для массивов `< 10 000` элементов последовательная версия предпочтительнее.

### 9.3 Рекомендации

- Оптимальное число потоков для данной задачи на Intel Core i5-1135G7: `4–6`.
- Для массивов менее `10 000` элементов использовать последовательную версию.
- При возможности задействовать до `6` потоков (если позволяет ОС и планировщик).
