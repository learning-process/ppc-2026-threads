## 1. Введение

Гибридная версия сочетает **межпроцессное взаимодействие через MPI** и **внутрипроцессное распараллеливание через STL
 (`std::async`)**. Такой подход позволяет распределять данные между несколькими процессами MPI, каждый из которых
 дополнительно использует все доступные ядра для локальной сортировки, а затем выполняет иерархическое слияние
 результатов. Это даёт возможность эффективно использовать ресурсы как многоядерного узла, так и кластера.

## 2. Постановка задачи

- **Вход**: Массив `std::vector<int64_t>` целых чисел со знаком (64-бит).
- **Выход**: Отсортированный по возрастанию массив той же длины.
- **Ограничения**:
  - Размер массива `N ≥ 1`.
  - Корректная обработка отрицательных чисел.
  - Результат должен совпадать с отсортированной версией.
  - Поддержка произвольного числа MPI-процессов.

## 3. Базовый алгоритм

Гибридный алгоритм реализует **трёхуровневую схему**:

1. **Распределение данных** (MPI): входной массив разбивается на блоки и рассылается процессам.
2. **Локальная сортировка** (STL): каждый процесс сортирует свой блок с использованием `std::async`.
3. **Иерархическое слияние** (MPI): процессы попарно сливают отсортированные блоки (древовидная схема).
4. **Широковещательная рассылка** (MPI): итоговый массив передаётся всем процессам.

## 4. Схема распараллеливания

### 4.1 Межпроцессный уровень (MPI)

Роли процессов

|  Rank  | Роль                                                                               |
|--------|------------------------------------------------------------------------------------|
| 0      | Координатор: распределяет входные данные, участвует в слиянии, рассылает результат |
| 1..N-1 | Вычислители: получают блок, сортируют, участвуют в слиянии                         |

Распределение данных

```cpp
// Root-процесс вычисляет размеры блоков
const int base = total / world_size;
const int remainder = total % world_size;
for (int pndex = 0; pndex < world_size; ++pndex) {
    send_counts[pndex] = base + (pndex < remainder ? 1 : 0);
}

// Рассылка всем процессам
MPI_Scatterv(sendbuf, send_counts.data(), send_displs.data(), 
             MPI_INT64_T, recvbuf, recv_count, MPI_INT64_T, 0, MPI_COMM_WORLD);
```

- Блоки распределяются максимально равномерно.
- Разница в размере блоков между процессами не более `1` элемента.
- Все процессы получают информацию о распределении через `MPI_Bcast`.

### 4.2 Внутрипроцессный уровень (STL)

Каждый MPI-процесс сортирует свой блок с помощью `ParallelRadixSort`:

```cpp
void ParallelRadixSort(std::vector<int64_t> &arr, size_t left, size_t right) {
    const size_t size = right - left;

    const auto hw = static_cast<size_t>(std::max(1, ppc::util::GetNumThreads()));
    const size_t num_threads = std::min(hw, size);

    // Разбиение блока на подблоки
    std::vector<size_t> boundaries(num_threads + 1);
    const size_t chunk = (size + num_threads - 1) / num_threads;

    // Параллельная сортировка подблоков через std::async
    std::vector<std::future<void>> futures;

    for (size_t tndex = 0; tndex < num_threads; ++tndex) {
        futures.push_back(std::async(std::launch::async, sort_chunk, ...));
    }

    // Слияние отсортированных подблоков
    MergeChunks(arr, boundaries);
}
```

- Каждый MPI-процесс дополнительно распараллеливает сортировку.
- Используется та же схема, что и в STL-версии: разбиение → `async` → слияние.

### 4.3 Иерархическое слияние (древовидная схема)

```cpp
void HierarchicalMerge(std::vector<int64_t> &local_data,
                       int rank,
                       int world_size) {
    int step = 1;

    while (step < world_size) {
        const int mask = step * 2;

        if ((rank & (mask - 1)) == 0) {           // Принимающий процесс
            const int partner = rank + step;

            if (partner < world_size) {
                // Получение данных от партнёра
                MPI_Recv(&incoming_size, 1, MPI_INT, partner, 0, ...);
                MpiRecvInt64(incoming, incoming_size, partner, 1);

                MergeSortedVectors(local_data, incoming);  // Слияние
            }

        } else if ((rank & (mask - 1)) == step) { // Отправляющий процесс
            const int partner = rank - step;

            // Отправка данных партнёру
            MPI_Send(&send_size, 1, MPI_INT, partner, 0, ...);
            MpiSendInt64(local_data, send_size, partner, 1);

            local_data.clear();                   // Очистка (данные отправлены)
        }

        MPI_Barrier(MPI_COMM_WORLD);
        step *= 2;
    }
}
```

Схема работы (пример для 4 процессов)

Шаг 1 (step=1):
  Rank0 ← Rank1 (Rank0 сливает данные Rank1)
  Rank2 ← Rank3 (Rank2 сливает данные Rank3)
  Rank1, Rank3 очищают данные

Шаг 2 (step=2):
  Rank0 ← Rank2 (Rank0 сливает данные Rank2)
  Rank2 очищает данные

Результат: все данные у Rank0

Особенности

- Древовидная схема требует `O(log P)` шагов (`P` — число процессов).
- На каждом шаге пары процессов обмениваются данными.
- Данные концентрируются в процессе с рангом `0`.

### 4.4 Широковещательная рассылка результата

```cpp
void BroadcastResult(std::vector<int64_t> &local_data,
                     int rank,
                     int total) {
    int result_size = (rank == 0)
        ? static_cast<int>(local_data.size())
        : 0;

    MPI_Bcast(&result_size, 1, MPI_INT, 0, MPI_COMM_WORLD);

    if (rank != 0) {
        local_data.resize(total);
    }

    MpiBcastInt64(local_data, result_size);
}
```

- Процесс `0` рассылает отсортированный массив всем процессам.
- Гарантирует, что после завершения все процессы имеют одинаковый результат.

### 4.5 Синхронизация

| Примитив              | Назначение                                           |
|-----------------------|------------------------------------------------------|
| `MPI_Barrier`         | Барьерная синхронизация после каждого уровня слияния |
| `MPI_Send / MPI_Recv` | Обмен данными между партнёрами                       |
| `MPI_Bcast`           | Рассылка информации о распределении и результата     |
| `MPI_Scatterv`        | Распределение входных данных (только root)           |
| `std::future::get()`  | Ожидание завершения локальных потоков                |

### 4.6 Управление числом ресурсов

```cpp
// Число MPI-процессов (PPC_NUM_PROC)
int world_size;
MPI_Comm_size(MPI_COMM_WORLD, &world_size);

// Число потоков внутри процесса (PPC_NUM_THREADS)
const auto hw = static_cast<size_t>(std::max(1, ppc::util::GetNumThreads()));
```

- Общее число работников = `MPI_процессы × потоки_в_процессе`.
- Для эффективного использования необходимо балансировать уровни.

## 5. Детали реализации

Локальная сортировка (`ParallelRadixSort`) повторяет схему STL-версии:

- разбиение блока на подблоки по числу потоков;
- `std::async` для параллельной сортировки каждого подблока;
- `MergeChunks` для последовательного слияния подблоков.

Отличие от чистой STL-версии

В чистом STL сортировка выполнялась на всём массиве один раз.  
В ALL-версии сортировка выполняется на каждом процессе независимо над его блоком.

### 5.1 Граничные случаи

- Пустой массив — отклоняется в `ValidationImpl()` через `MPI_Bcast`.
- Один процесс (`world_size == 1`) — гибридная версия работает как чистая STL.
- Блок меньше `2` элементов — сортировка не требуется.
- Неравномерное распределение — корректно обрабатывается через `MPI_Scatterv`.

### 8.2 Анализ масштабируемости

На `1–4` процессах наблюдается нестабильная картина:

- `task_run` на `2` процессах: `3.51×` (приемлемо);
- `pipeline` на `2` процессах: `0.27×` (аномально низко).

На `4` процессах производительность близка к `1` процессу.

На `6–8` процессах производительность катастрофически падает:

- ускорение `task_run` на `6` процессах: `0.006×` (почти в `200` раз медленнее SEQ).

Причины

- гигантские накладные расходы на MPI-коммуникации при малом объёме данных;
- каждый процесс получает блок размером `≈ 10 000 000 / 8 = 1.25M` элементов;
- древовидное слияние требует многократной передачи данных через MPI;
- `MPI_Barrier` после каждого уровня слияния создаёт серьёзные задержки.

- Для эффективного использования MPI требуется значительно больший объём данных (миллиарды элементов).

### 8.4 Преимущества и недостатки ALL-версии

Преимущества

- Потенциальная масштабируемость на кластере с распределённой памятью.
- Возможность обработки данных, не помещающихся в память одного узла.
- Чёткое разделение уровней параллелизма.

Недостатки

- Огромные накладные расходы на MPI-коммуникации для малых и средних объёмов данных.
- Древовидное слияние требует `O(P log P)` передач данных.
- `MPI_Barrier` создаёт серьёзные задержки синхронизации.
- Сложность отладки и профилирования гибридных приложений.

## 9. Выводы

### 9.1 Что сработало

- Корректная работа гибридной схемы на всех конфигурациях.
- Эффективная локальная сортировка через STL внутри каждого процесса.
- Равномерное распределение данных через `MPI_Scatterv`.
- Древовидная схема слияния корректно работает для любого числа процессов.

### 9.2 Ограничения

- MPI неэффективен для малых и средних объёмов данных (до `10` млн элементов на `4–8` узлах).
- Накладные расходы на коммуникации превышают выгоду от распределения.

Для эффективного использования MPI требуется:

- объём данных: `≥ 1` млрд элементов;
- число элементов на процесс: `≥ 10–50` млн.

`MPI_Barrier` после каждого уровня слияния создаёт избыточные задержки.

### 9.3 Рекомендации

- Для одного узла (`≤ 10` млн элементов): использовать TBB (лучшее ускорение) или OpenMP.
- Для кластера с большими данными (`≥ 1` млрд элементов): ALL-версия может дать выигрыш.

Оптимизации для ALL

- использовать неблокирующие MPI-вызовы (`MPI_Isend` / `MPI_Irecv`);
- заменить `MPI_Barrier` на более лёгкие механизмы синхронизации;
- рассмотреть `MPI_Allgather` вместо древовидного слияния для финального шага.
