## 1. Введение

Версия с использованием стандартной библиотеки потоков C++ реализует ручное управление параллелизмом
через `std::async` и `std::future`. В отличие от OpenMP и TBB, где параллелизм управляется автоматически
или через директивы, STL-версия требует явного разделения данных, создания задач и ожидания их завершения.
Это даёт полный контроль над распределением работы, но накладывает ответственность за корректную
синхронизацию и балансировку нагрузки.

## 2. Постановка задачи

- **Вход**: Массив `std::vector<int64_t>` целых чисел со знаком (64-бит).
- **Выход**: Отсортированный по возрастанию массив той же длины.
- **Ограничения**:
  - Размер массива `N ≥ 1`.
  - Корректная обработка отрицательных чисел.
  - Результат должен совпадать с отсортированной версией.

## 3. Базовый алгоритм

Последовательная версия реализует гибридную схему:

- Для подмассивов размером **≤ 131072** элементов применяется **поразрядная сортировка**.
- Для подмассивов больше порога — **сортировка слиянием**.

**Поразрядная сортировка** (ключевой этап для распараллеливания):

1. Преобразование знаковых чисел в беззнаковые ключи с инвертированием знакового бита.
2. Для каждого из 8 байтов (от младшего к старшему):
   - Подсчёт гистограммы частот (256 корзин).
   - Вычисление префиксных сумм (позиций записи).
   - Перераспределение элементов во временный массив.
3. Копирование отсортированных элементов обратно.

## 4. Схема распараллеливания

### 4.1 Подход на основе задач (task-based)

STL-версия использует **двухуровневую схему** распараллеливания:

1. **Уровень 1 — параллельная сортировка блоков**:
   - Массив делится на `num_threads` примерно равных блоков.
   - Каждый блок сортируется независимо с помощью `std::async`.

2. **Уровень 2 — последовательное слияние**:
   - Отсортированные блоки сливаются попарно (как в сортировке слиянием).

### 4.2 Разбиение данных

```cpp
const size_t chunk = (size + num_threads - 1) / num_threads;
for (size_t tndex = 0; tndex <= num_threads; ++tndex) {
    boundaries[tndex] = left + std::min(tndex * chunk, size);
}
```

Используется статическое разбиение с округлением вверх.

- Последний блок может быть меньше остальных.
- Каждый поток получает непрерывный диапазон индексов.

### 4.3 Параллельная сортировка блоков

```cpp
std::vector<std::future<void>> futures;

for (size_t tndex = 0; tndex < num_threads; ++tndex) {
    futures.push_back(std::async(std::launch::async, [&arr, l, r] {
        RadixSort(arr, l, r);
    }));
}

for (auto &f : futures) {
    f.get();  // ожидание завершения всех потоков
}
```

- Каждый блок сортируется независимо с помощью `RadixSort`.
- `std::async` с политикой `std::launch::async` гарантирует создание отдельного потока.
- `std::future::get()` блокирует выполнение до завершения задачи.

### 4.4 Последовательное слияние

После сортировки всех блоков выполняется итеративное слияние:

```cpp
size_t step = chunk;

while (step < size) {
    size_t double_step = step * 2;

    for (size_t pos = left; pos < right; pos += double_step) {
        size_t mid = std::min(pos + step, right);
        size_t end = std::min(pos + double_step, right);

        if (mid < end) {
            SimpleMerge(arr, pos, mid, end);
        }
    }

    step = double_step;
}
```

- Слияние выполняется последовательно (не распараллеливается).
- Это узкое место алгоритма, так как накладные расходы на параллельное слияние для данной задачи не оправданы.

### 4.5 Управление числом потоков

```cpp
const auto hw = static_cast<size_t>(std::max(1, ppc::util::GetNumThreads()));
const size_t num_threads = std::min(hw, size);
```

- Количество потоков ограничено размером массива (бессмысленно создавать больше потоков, чем элементов).
- Число потоков задаётся через `PPC_NUM_THREADS`.

### 4.6 Синхронизация

- Явная синхронизация через `std::future::get()` после каждого уровня параллелизма.
- Нет гонок данных, так как каждый поток работает с непересекающимся диапазоном массива.
- Отсутствие блокировок (`mutex`, атомарных операций) благодаря независимой обработке блоков.

## 5. Детали реализации

Последовательная поразрядная сортировка (в отличие от параллельной в OMP/TBB):

```cpp
void RadixSort(std::vector<int64_t> &arr, size_t left, size_t right) {
    // Полностью последовательная реализация
    // Используется для сортировки отдельных блоков
}
```

Почему `RadixSort` не распараллелен внутри блока?

- Блоки имеют размер `≈ N / num_threads`.
- Для типичных размеров (`10 000–1 000 000`) блоки достаточно малы.
- Внутреннее распараллеливание добавило бы накладные расходы, не давая выигрыша.

### 5.1 Граничные случаи

- Пустой массив — отклоняется в `ValidationImpl()`.
- Размер `< числа потоков` — число потоков ограничивается размером.
- Массив `≤ 131072` — однопоточная поразрядная сортировка (слияние не требуется).
- Массив `> 131072` — параллельная сортировка блоков + слияние.

### 8.2 Анализ масштабируемости

На `1–4` потоках наблюдается хорошее ускорение:

- пиковое ускорение `task_run` на `2` потоках: `4.79×`;
- эффективность на `2` потоках достигает `240%` (сверхлинейное ускорение);
- на `4` потоках эффективность падает до `95%` (всё ещё приемлемо).

На `6–8` потоках производительность резко падает:

- ускорение `pipeline` падает до `0.66×` на `6` потоках;
- ускорение `task_run` на `8` потоках снижается до `0.68×` (медленнее SEQ).

### Причины

- последовательное слияние становится узким местом при большом числе блоков;
- накладные расходы на создание потоков через `std::async`;
- гиперпоточность неэффективна для данного алгоритма.

- На `6+` потоках STL сильно проигрывает из-за последовательного слияния.

### 8.4 Преимущества и недостатки STL-версии

Преимущества

- Полный контроль над распределением работы.
- Отсутствие зависимостей от внешних библиотек (только стандартная библиотека).
- Простая модель синхронизации (непересекающиеся диапазоны).

Недостатки

- Последовательное слияние — основное узкое место при масштабировании.
- Накладные расходы на создание потоков через `std::async`.
- Отсутствие балансировки нагрузки (при неравномерном распределении ключей).
- Невозможность динамического перераспределения задач (в отличие от TBB).

### 9.2 Ограничения

- Последовательное слияние ограничивает масштабируемость при числе потоков `> 4`.
- Накладные расходы на `std::async` заметны при большом числе потоков.
- Гиперпоточность неэффективна (на `8` потоках ускорение `< 1×`).
- Статическое разбиение может привести к дисбалансу при неравномерном распределении данных.

### 9.3 Рекомендации

- Оптимальное число потоков для данной задачи: `2–4`.
- Для массивов менее `10 000` элементов последовательная версия предпочтительнее.

При необходимости масштабирования на большее число потоков следует:

- распараллелить этап слияния (например, через `std::async` для пар блоков);
- использовать TBB для автоматической балансировки нагрузки.
