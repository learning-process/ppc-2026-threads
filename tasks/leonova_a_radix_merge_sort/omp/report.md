## 1. Введение

Версия с использованием OpenMP реализует многопоточное выполнение поразрядной сортировки с общей памятью.
Основная цель — распараллелить наиболее трудоёмкие этапы: подсчёт гистограмм частот байтов и
перераспределение элементов во временный массив. Это позволяет достичь ускорения на больших объёмах
данных за счёт эффективного использования всех доступных ядер процессора.

## 2. Постановка задачи

- **Вход**: Массив `std::vector<int64_t>` целых чисел со знаком (64-бит).
- **Выход**: Отсортированный по возрастанию массив той же длины.
- **Ограничения**:
  - Размер массива `N ≥ 1`.
  - Корректная обработка отрицательных чисел.
  - Результат должен совпадать с отсортированной версией.

## 3. Базовый алгоритм

Последовательная версия реализует гибридную схему:

- Для подмассивов размером **≤ 131072** элементов применяется **поразрядная сортировка**.
- Для подмассивов больше порога — **сортировка слиянием**.

**Поразрядная сортировка** (ключевой этап для распараллеливания):

1. Преобразование знаковых чисел в беззнаковые ключи с инвертированием знакового бита.
2. Для каждого из 8 байтов (от младшего к старшему):
   - Подсчёт гистограммы частот (256 корзин).
   - Вычисление префиксных сумм (позиций записи).
   - Перераспределение элементов во временный массив.
3. Копирование отсортированных элементов обратно.

## 4. Схема распараллеливания

### 4.1 Область параллелизации

Распараллелены следующие этапы поразрядной сортировки:

- **Заполнение массива беззнаковых ключей** — независимая операция для каждого элемента.
- **Подсчёт гистограммы частот** — каждый поток обрабатывает свой блок данных.
- **Перераспределение элементов** — каждый поток записывает элементы в непересекающиеся области.
- **Слияние гистограмм** — выполняется последовательно (одним потоком).

### 4.2 Управление данными

| Переменная      | Тип доступа            | Описание                          |
|-----------------|------------------------|-----------------------------------|
| `arr`           | shared                 | Входной/выходной массив           |
| `keys`          | shared                 | Беззнаковые ключи                 |
| `temp_arr`      | shared                 | Временный массив для перестановки |
| `temp_keys`     | shared                 | Временные ключи                   |
| `local_counts`  | shared (разные строки) | Локальные гистограммы потоков     |
| `local_offsets` | shared (разные строки) | Локальные смещения потоков        |

### 4.3 Синхронизация

```cpp
#pragma omp barrier  // после каждой стадии
#pragma omp single    // для последовательных операций (слияние гистограмм)
```

### 4.4 Планирование

Используется `schedule(static)` для равномерного распределения итераций между потоками:

- `FillUnsignedKeys` — заполнение ключей.
- `CountByteValues` — подсчёт частот.
- `ScatterByte` — перераспределение.

## 5. Детали реализации

Гибридная стратегия (Radix + Merge)

Порог переключения на Radix Sort увеличен до **131072 элементов** (против 32 в SEQ).  
Это позволяет эффективнее использовать параллельную поразрядную сортировку на больших подмассивах.

Граничные случаи

- **Пустой массив** — отклоняется в `ValidationImpl()`.
- **Размер < числа потоков** — число потоков ограничивается размером массива.
- **Массив ≤ порога** — применяется только поразрядная сортировка.
- **Массив > порога** — рекурсивное разделение и слияние (без распараллеливания на этом уровне).

## 8. Результаты и обсуждение

### 8.1 Производительность

**Методика:**  

- Замеры выполняются в Google Test (`pipeline` и `task_run` режимы).
- Для каждого числа потоков (1, 2, 4, 6, 8) выполняется отдельный запуск.
- Время указано в секундах.
- Ускорение рассчитано относительно среднего времени последовательной версии (SEQ) по всем запускам.

Базовое среднее время SEQ (эталон)

|----------|---------------------------------------------------------------------------|
| pipeline | (0.001316 + 0.001488 + 0.001451 + 0.003818 + 0.004345) / 5 = **0.002484** |
| task_run | (0.001328 + 0.001946 + 0.002839 + 0.003706 + 0.005307) / 5 = **0.003025** |

### 8.2 Анализ масштабируемости

**На 1–4 потоках** наблюдается значительное ускорение относительно среднего SEQ:

- Пиковое ускорение `task_run` на 4 потоках: **6.65×**.
- Эффективность на 2 потоках достигает **241%** (сверхлинейное ускорение).

**На 6–8 потоках** производительность падает:

- На 8 потоках ускорение становится **< 1×** (медленнее среднего SEQ).
- Причины:
  - Накладные расходы на синхронизацию (барьеры, `single`).
  - Число физических ядер = 4, гиперпоточность не даёт выигрыша.
  - Увеличение числа потоков приводит к перераспределению кэша и ложному разделению.

### 8.3 Сравнение с базовой линией

| Метрика              | 1 поток | 2 потока | 4 потока | 6 потоков | 8 потоков |
|----------------------|---------|----------|----------|-----------|-----------|
| Ускорение (pipeline) | 4.48×   | 3.55×    | 5.34×    | 3.18×     | 1.15×     |
| Ускорение (task_run) | 8.15×   | 4.82×    | 6.65×    | 4.54×     | 0.69×     |

### Причины

- Накладные расходы на синхронизацию (барьеры, `single`)
- 4 физических ядра → гиперпоточность неэффективна
- Кэш-конфликты и false sharing

### 9.3 Рекомендации

- Оптимальное число потоков: **4**
