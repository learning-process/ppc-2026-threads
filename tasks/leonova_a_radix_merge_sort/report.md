## 1. Введение

В данной работе проводится сравнительный анализ эффективности различных технологий параллельного программирования
 для реализации поразрядной сортировки (Radix Sort) целых чисел. Рассматриваются следующие подходы:

- **SEQ** – последовательный эталонный алгоритм.
- **OpenMP** – распараллеливание с использованием директив компилятора для общей памяти.
- **Intel TBB** – библиотека задач с динамическим планированием (work-stealing).
- **STL** – ручное управление потоками на уровне стандартной библиотеки C++.
- **ALL (MPI + STL)** – гибридная версия с межпроцессным взаимодействием (MPI) и внутрипроцессной многопоточностью.

Цель работы — определить наиболее эффективную технологию для сортировки массивов целых чисел на одном узле и оценить
применимость гибридного подхода.

## 2. Единая постановка задачи

- **Вход:** Массив `std::vector<int64_t>` целых чисел со знаком (64 бита).
- **Выход:** Отсортированный по возрастанию массив той же длины.
- **Ограничения:**
  - Размер массива `N ≥ 1`.
  - Корректная обработка отрицательных чисел (преобразование знакового бита).
  - Результат должен совпадать с эталонной последовательной сортировкой.

### 3.2 Данные для тестов

- **Размеры массивов:** 10 000, 100 000, 1 000 000, 10 000 000 элементов.
- **Диапазон значений:** `[-1e6, 1e6]` (равномерное распределение).
- **Функциональные тесты:** 32 набора данных, включая граничные случаи (пустые массивы, один элемент, обратный
 порядок, `INT64_MIN` / `INT64_MAX`).

### 3.3 Методика измерений

- **Инструмент:** Google Test Framework (режимы `pipeline` и `task_run`).
- **Базовое время (SEQ):** Среднее арифметическое по 5 запускам.
- **Ускорение (Speedup):** $S = \frac{T_{seq}}{T_{parallel}}$, где $T_{seq}$ – среднее время последовательной версии,
 $T_{parallel}$ – время параллельной версии.
- **Эффективность:** $E = \frac{S}{P} \cdot 100\%$, где $P$ – количество потоков / процессов.

## 6. Интерпретация различий

### 6.1 OpenMP

- **Сильные стороны:** Простота реализации. Показывает отличное ускорение на 4 потоках (до 6.65×), что соответствует
количеству физических ядер.
- **Слабые стороны:** При переходе к гиперпоточности (6-8 потоков) производительность падает из-за барьерной
синхронизации и конкуренции за кэш.

### 6.2 Intel TBB

- **Сильные стороны:** **Абсолютный лидер по ускорению** (до 9.28× на 6 потоках). Механизм work-stealing позволяет
эффективно использовать ресурсы даже при дисбалансе. Превосходит OpenMP на 6 потоках.
- **Слабые стороны:** На 8 потоках также наблюдается деградация, но менее выраженная, чем у OpenMP.

### 6.3 STL

- **Сильные стороны:** Полный контроль над потоками. Хорошее ускорение на 2-4 потоках.
- **Слабые стороны:** Последовательное слияние блоков становится узким местом. Накладные расходы на создание потоков
 выше, чем у TBB/OpenMP.

### 6.4 ALL (MPI + STL)

- **Сильные стороны:** При малом числе процессов (1-2) показывает результаты, сравнимые с чистыми потоковыми версиями.
- **Слабые стороны:** **Катастрофическая деградация** при 6-8 процессах. Накладные расходы на MPI-коммуникации
(даже на одном узле) превышают выгоду от распараллеливания. **Неприменимо** для данного объёма данных на одной машине.

## 7. Выводы

1. **Лучшая технология для одного узла:** **Intel TBB**. Обеспечивает максимальное ускорение (до 9.28×) и лучшую
масштабируемость благодаря интеллектуальному планировщику задач.
2. **OpenMP** – отличный выбор для "быстрого старта", но уступает TBB при попытке использовать больше потоков,
чем физических ядер.
3. **Ручное управление потоками (STL)** оправдано только для специфических задач с простой декомпозицией.
В данном случае оно проигрывает TBB из-за последовательной стадии слияния.
4. **Гибридная версия (ALL) неэффективна** на одной вычислительной ноде. Её применение целесообразно только в
кластерных средах с огромными объёмами данных (миллиарды элементов), где время коммуникации окупается распределением
памяти.

## 8. Репродуцируемость

### 8.1 Сборка проекта

```bash
git submodule update --init --recursive --depth=1
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

### 8.2 Запуск тестов производительности

#### Для потоковых версий (OpenMP, TBB, STL)

```bash
export PPC_NUM_THREADS=4

./build/bin/ppc_perf_tests --gtest_filter="*leonova*"
```

#### Для гибридной версии (ALL)

```bash
export PPC_NUM_PROC=2

mpiexec -n 2 ./build/bin/ppc_perf_tests --gtest_filter="*leonova_all*"
```
