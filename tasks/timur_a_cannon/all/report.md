## 1. Контекст

Сочетаются **два уровня** параллелизма:

1. **MPI** — несколько процессов, каждый считает свою часть строк блоков сетки.
2. **OpenMP** — внутри процесса потоки обрабатывают свои строки.

Запуск обязателен через **`mpiexec`**, иначе тесты `all` пропускаются.

## 2. Постановка задачи

- **Вход:** кортеж `(b_size, A, B)` — размер блока `b_size` и две квадратные матрицы `A`, `B` размера `n×n` (`common/include/common.hpp`).
- **Выход:** матрица `C = A·B` размера `n×n`.
- **Ограничения:** `b_size > 0`; `n` кратно `b_size`; матрицы квадратные (проверяется в `ValidationImpl`).
- **Эталон для сравнения:** SEQ, режим `task_run`, один поток: **\(T_{seq} = 0.0499\) с** (матрицы `512×512`, блок `32×32`).

## 3. Базовый алгоритм

**Блочный алгоритм Кэннона** с делением работы между процессами:

1. `grid_sz = n / b_size`.
2. Матрицы `A` и `B` рассылаются на все процессы (`MPI_Bcast`), чтобы у каждого был полный вход.
3. Каждый MPI-процесс считает **свои строки блоков** сетки (диапазон `global_i`).
4. Для каждой локальной строки блока и каждого столбца `j` выполняются все шаги Кэннона: копирование нужных блоков `A`, `B`, умножение `BlockMultiplyAccumulate`, накопление в локальный фрагмент результата.
5. Локальные части матрицы `C` собираются в полный результат на всех процессах (`MPI_Allgatherv`).

Внутри процесса цикл по строкам блоков распараллен OpenMP. Сложность по времени: \(O(n^3)\). По памяти: \(O(n^2)\) на процесс (плюс обмены MPI).

## 4. Схема распараллеливания

**Между процессами (MPI):**

- процесс `rank` получает свой диапазон строк блоков;
- входные данные разосланы `MPI_Bcast`;
- результат собран `MPI_Allgatherv`.

**Внутри процесса (OpenMP):**

- цикл по локальным строкам блоков — `omp parallel for`.

```cpp
// File: all/src/ops_all.cpp
#pragma omp parallel for default(none) shared(...)
for (int local_i = 0; local_i < local_block_rows; ++local_i) {
  for (int j = 0; j < grid_sz; ++j) {
  // шаги Кэннона для одного блока результата
  }
}
```

_Комментарий:_ на одном процессе ALL медленнее SEQ — есть расходы MPI даже при `n=1`. При росте $p$ время уменьшается; на **8 работников** ускорение почти **2×**. Часть времени уходит на обмен данными (`Bcast`, `Allgatherv`), поэтому эффективность ниже, чем у чистого OMP/TBB.
