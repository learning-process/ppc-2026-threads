## 1. Контекст

STL-версия — ручная реализация пула потоков на `std::thread`. В отличие от
OMP и TBB, здесь нет внешнего runtime, который сам решает, как делить
работу: весь bookkeeping (расчёт диапазонов, владение потоками, join)
написан в коде. Цель отчёта — показать, что разбиение строк по потокам
сделано корректно (нет пересечений диапазонов и нет потерянных строк), а
гонки исключены за счёт работы каждого потока только с собственными
строками результата.

## 2. Постановка задачи

См. [seq/report.md](../seq/report.md).

## 3. Базовый алгоритм

Тот же Густавсон row-wise со SPA. Каждый поток обрабатывает непрерывный
поддиапазон строк `[begin, end)` и пишет в `per_row[i]` только для своих
`i`.

## 4. Схема распараллеливания

- Число потоков: `worker_count = max(1, min(GetNumThreads(), total_rows))`.
  Ограничение сверху `total_rows` нужно, чтобы при очень маленьких задачах
  не плодить пустых потоков.
- Разбиение строк: `chunk = total_rows / worker_count`,
  `leftover = total_rows % worker_count`. Первые `leftover` потоков
  получают `chunk + 1` строк, остальные — `chunk`. Это гарантирует разницу
  не более 1 строки между диапазонами.
- Локальные буферы: каждый поток внутри лямбды объявляет собственные
  `spa`, `touched_by_row`, `touched_cols` — они живут в стеке потока,
  никак не пересекаются с другими.
- Объединение результатов: всё пишется напрямую в общий `per_row[i]`, но
  поскольку `i` каждого потока не пересекается с `i` другого, записи
  независимы. Никаких `mutex`/`atomic` не требуется.
- `join`: все потоки запускаются в первом цикле, второй цикл вызывает
  `join` у каждого. Это **критически важно**: если `join` поставить в один
  цикл с `emplace_back`, потоки исполнятся последовательно (антипаттерн
  из методички). Здесь циклы разделены, поэтому работа реально идёт
  параллельно.
- Финальная сборка через `Assemble` — та же, что в OMP/TBB.

## 5. Детали реализации

Файлы: [stl/include/ops_stl.hpp](include/ops_stl.hpp),
[stl/src/ops_stl.cpp](src/ops_stl.cpp).

Лямбда-worker:

```cpp
// File: stl/src/ops_stl.cpp
auto worker = [&](int begin, int end) {
  std::vector<Cplx> spa(static_cast<std::size_t>(rhs.n_cols), Cplx(0.0, 0.0));
  std::vector<int> touched_by_row(static_cast<std::size_t>(rhs.n_cols), -1);
  std::vector<int> touched_cols;
  touched_cols.reserve(static_cast<std::size_t>(rhs.n_cols));

  for (int i = begin; i < end; ++i) {
    GustavsonRow(lhs, rhs, i, spa, touched_by_row, touched_cols, per_row[i]);
  }
};
```

`spa` и сопутствующие буферы аллоцируются на стеке worker-функции — это и
есть thread-local буфер, гарантированно приватный для каждого потока.
`per_row[i]` пишется только для тех `i`, которые попали в этот диапазон,
поэтому записи не пересекаются с другими потоками.

Запуск и join:

```cpp
// File: stl/src/ops_stl.cpp
int cursor = 0;
for (int worker_index = 0; worker_index < worker_count; ++worker_index) {
  const int begin = cursor;
  const int end = begin + chunk + (worker_index < leftover ? 1 : 0);
  cursor = end;
  if (begin < end) {
    pool.emplace_back(worker, begin, end);
  }
}
for (auto &th : pool) {
  th.join();
}
```

Циклы разделены: сначала все потоки стартуют, потом главный поток ждёт.
Условие `if (begin < end)` страхует от пустых диапазонов (актуально, когда
`worker_count > total_rows` — хотя выше мы это уже ограничили, проверка
оставлена как защитная). Имя переменной цикла `worker_index` намеренно
длиннее обычного `t` — чтобы не пересекаться по имени с лямбдой `worker` и
не путать читателя.

Карта «поток → диапазон строк» получается такой:

- worker 0: `[0, chunk + (0 < leftover ? 1 : 0))`
- worker 1: `[end_0, end_0 + chunk + (1 < leftover ? 1 : 0))`
- ... и так далее, диапазоны идут подряд без пробелов и пересечений.

## 6. Проверка корректности

Functional-тест [tests/functional/main.cpp](../tests/functional/main.cpp)
прогоняет STL-версию через
`AddFuncTask<KlimovichVCrsComplexMatMulStl, InType>` на тех же 10 наборах
размеров. Все наборы проходят с допуском `1e-9`. Результат побайтно
совпадает с SEQ/OMP/TBB на одинаковых входных данных при любом
`PPC_NUM_THREADS`.

Гонок нет: SPA — на стеке каждого потока, `per_row[i]` пишется не более
одного раза и только своим потоком. Покрытие диапазонов проверяется
арифметически (сумма размеров равна `total_rows`).

## 8. Результаты

Размер задачи `1500×1500`, `T_seq(task) = 0.152 s`,
`T_seq(pipeline) = 1.498 s`. Медианы по 10 повторам.

| threads | mode     | median time, s | speedup vs seq | efficiency, % |
| ------: | -------- | -------------: | -------------: | ------------: |
|       1 | task     |          0.158 |           0.96 |            96 |
|       2 | task     |          0.085 |           1.79 |            89 |
|       4 | task     |          0.049 |           3.10 |            78 |
|       8 | task     |          0.034 |           4.47 |            56 |
|       1 | pipeline |          1.534 |           0.98 |            98 |
|       2 | pipeline |          0.842 |           1.78 |            89 |
|       4 | pipeline |          0.467 |           3.21 |            80 |
|       8 | pipeline |          0.318 |           4.71 |            59 |

Накладные расходы `std::thread::create/join` единоразовые (по одной паре
на поток), поэтому на матрице `1500×1500` они теряются в стоимости
вычисления. По сравнению с OMP/TBB, STL стабильно медленнее на 5–15% при
тех же числах потоков. Причина — статическое разбиение: если строки
сильно неравномерны по стоимости, поток, которому достались «тяжёлые»
строки, становится узким местом, а работающий стилинг (как в TBB) или
динамический schedule (как в OMP) этого избегают. Особенно это видно на
8 потоках: эффективность падает до 56–59% против 66–75% у OMP/TBB.

## 9. Выводы

`std::thread` даёт полностью контролируемое разбиение работы, что хорошо
для учебного отчёта (видно, кто и что делает) и плохо для нагрузок с
сильной неоднородностью строк (нет work stealing, потоки нельзя
«перебалансировать» на лету). На 4 потоках STL достигает ускорения
≈3.1–3.2×, на 8 потоках — ≈4.5–4.7×. Это меньше, чем у OMP/TBB на той же
конфигурации, но цена тоже видна — реализация полностью прозрачна и не
зависит от внешнего runtime. Корректность строится на двух базовых
инвариантах: непересекающиеся диапазоны строк и thread-local SPA на стеке
worker-функции. `join` намеренно вынесен в отдельный цикл — это устраняет
распространённый антипаттерн скрытой сериализации.
