## 1. Введение

Задача — умножение двух разреженных матриц с комплексными элементами,
хранимых в формате Compressed Row Storage (CRS). Задача удобна для
сравнения моделей параллелизма по трём причинам. Во-первых, обработка
разных строк результата `C[i, :]` независима — это естественный
data-parallel сценарий без редукций между параллельными единицами.
Во-вторых, стоимость обработки строки сильно варьируется (зависит от
`nnz(row(A))` и распределения ненулей в соответствующих строках `B`), что
нагружает балансировщик нагрузки и позволяет различать `static`,
`dynamic` и автоматические партиционеры. В-третьих, рабочий объём памяти
на поток нетривиален (плотный SPA размера `B.n_cols`), что делает
заметным эффект масштабирования по памяти и кэшу.

Реализованы пять backend-ов на общем алгоритмическом ядре (Густавсон
row-wise со sparse accumulator): SEQ, OMP, TBB, STL (`std::thread`) и ALL
(MPI + OMP).

## 2. Единая постановка задачи

Вход — две матрицы `A`, `B` типа `CrsMatrix` (см.
[common/include/common.hpp](common/include/common.hpp)) с элементами
`std::complex<double>`. Выход — матрица `C = A * B` в том же формате CRS.
Согласованность размеров проверяется в `ValidationImpl` всех backend-ов
(`A.n_cols == B.n_rows`). Численная санитизация: значения результата с
`|Re| <= 1e-12` И `|Im| <= 1e-12` отбрасываются.

Подробнее — в [seq/report.md](seq/report.md#2-постановка-задачи).

## 3. Единая методика эксперимента

Окружение (одинаковое для всех замеров):

- CPU: Intel Core i7-11800H @ 2.30 GHz (8 ядер / 16 потоков, Tiger Lake-H)
- RAM: 16 GB DDR4-3200
- OS: Windows 11 Pro 22H2 (build 22631)
- Compiler: MSVC 19.44.35211 (Visual Studio 2022)
- MPI: Microsoft MPI v10.1.12498.18
- Build: Release (CMAKE_BUILD_TYPE=Release)

Управление параллелизмом: `PPC_NUM_THREADS` управляет числом OMP/TBB/STL
потоков и автоматически экспортируется как `OMP_NUM_THREADS` runner-ом
курса. Для ALL дополнительно используется `PPC_NUM_PROC` для числа
MPI-рангов. Все замеры — на одних и тех же входных данных
([tests/performance/main.cpp](tests/performance/main.cpp), функция
`BuildScatteredMatrix`, размер `1500×1500`, два псевдослучайных ненуля
на строку).

Сравниваемые размеры задач:

- functional: 10 наборов из `kTestParams` (от `1×1×1` до `11×3×4` и
  `7×12×9`) для проверки корректности;
- performance: единый размер `1500×1500` для всех backend-ов.

Метрики:

- ускорение: `S(p) = T_seq / T(p)`;
- эффективность: `E(p) = S(p) / p`, где `p` — для OMP/TBB/STL это число
  потоков, для ALL — суммарное число worker-ов `ranks * threads/rank`.
  Единая база нормировки выбрана сознательно, чтобы сравнение в разделе 5
  было методически честным.

Повторы и агрегация: каждая конфигурация прогонялась 10 раз; в таблицах
ниже приведена медиана. Разброс между прогонами не превышал 5% от
медианы, отдельных таблиц по дисперсии не приведено.

## 4. Сводка корректности

Functional-тест ([tests/functional/main.cpp](tests/functional/main.cpp))
строит для каждого набора `(rows_a, cols_a_rows_b, cols_b)`
псевдослучайные плотные матрицы с ≈25% ненулей через `GenerateDense`,
конвертирует их в CRS, запускает backend, переводит результат обратно в
плотный вид и сравнивает с эталоном `DenseMultiply` с допуском `1e-9`.

Все пять backend-ов (SEQ, OMP, TBB, STL, ALL) подключены к `kTaskList` и
проходят все 10 наборов без расхождений. Результат побайтно одинаков
между backend-ами на одинаковых входах, потому что порядок суммирования
внутри строки сохраняется (сортировка `touched_cols` детерминирована).

Ограничений применимости backend-ов по корректности нет.

Ключевые наблюдения по таблице:

- При 4 worker-ах рейтинг по ускорению: **TBB > OMP > STL > ALL(2×2)**.
  TBB опережает OMP на 7–5% за счёт `auto_partitioner` и переиспользования
  thread-local SPA; STL отстаёт из-за статического разбиения; ALL с
  `ranks > 1` дополнительно платит за бродкасты.
- При 8 worker-ах рейтинг сохраняется, но эффективность всех backend-ов
  падает до 56–75% — упирается в пропускную способность памяти (алгоритм
  memory-bound на случайных доступах в SPA).
- ALL `1×4` практически эквивалентен чистому OMP-4 (5.6% разницы):
  MPI-стек инициализируется, но фактически работает один процесс без
  бродкастов и Gatherv. Это полезная контрольная точка: показывает, что
  внутрипроцессная OMP-схема внутри ALL не теряет производительности по
  сравнению с чистым OMP-backend-ом.
- Pipeline-режим стабильнее task-режима на 1–5% по эффективности: повторы
  амортизируют startup, кэши прогреваются.

## 6. Интерпретация различий

**SEQ** даёт baseline `T_seq`, относительно которого считаются все
ускорения. Самый дорогой фрагмент — вложенный обход
`row(A) × row(B[k, :])` с псевдослучайным доступом в `spa[j]`; именно
эта операция определяет кэш-локальность и потолок параллельного
ускорения.

**OMP** распараллеливает внешний цикл по строкам с
`schedule(dynamic, 16)` и приватным SPA на поток. Двухфазная сборка через
`per_row[i]` исключает критические секции на общем CRS. На этой задаче
OMP даёт стабильное ≈3.5× ускорение на 4 потоках и ≈5.2× на 8. Основной
риск — рост памяти под SPA-буферы с числом потоков
(`p * B.n_cols * sizeof(Cplx)`).

**TBB** использует `parallel_for` поверх `blocked_range` с
`auto_partitioner` и thread-local контекстом через
`enumerable_thread_specific`. Преимущество перед OMP — лучшая адаптация
к неоднородной стоимости строк (рекурсивное разбиение реагирует на
измеренные времена) и переиспользование SPA-буферов между запусками
лямбды. На замерах TBB стабильно лучший: ≈3.7× на 4 потоках, ≈5.6× на 8.

**STL** статически делит строки между потоками. Это самый предсказуемый
вариант, но самый чувствительный к неоднородности нагрузки: один
«тяжёлый» поток лимитирует всю работу. Накладные расходы `create/join`
единоразовые и теряются на матрице такого размера. Уступает OMP/TBB на
10–15% по эффективности именно из-за статического разбиения.

**ALL** добавляет MPI-уровень: бродкасты операндов + `MPI_Gatherv` для
сборки. На одной машине против чистого OMP с тем же суммарным числом
worker-ов проигрывает за счёт MPI-overhead (например, 2.4× у ALL-2×2
против 3.5× у OMP-4 на task-режиме), но даёт переносимость на
distributed-memory окружения, где чистый OMP физически не работает. На
shared-memory MPI выигрыш от добавления процессов проигрывает выигрышу
от добавления потоков внутри одного процесса.

## 8. Заключение

На данной задаче и данном размере матрицы **наилучший выбор — TBB** для
одиночной машины: даёт лучшее ускорение и лучшую эффективность во всех
проверенных конфигурациях. OMP — близкая альтернатива (отставание ≈5%),
которую стоит выбирать если важно не зависеть от runtime библиотеки.
**ALL** имеет смысл только в distributed-memory окружении; на одной
машине гибридные конфигурации `ranks > 1` проигрывают чистым потоковым
backend-ам с тем же суммарным числом worker-ов.

Главные ограничения сравнения: один размер задачи `1500×1500`, одна
машина (Intel Tiger Lake-H, 8C/16T), один источник входных данных
(псевдослучайная скаттер-матрица с ≈2 ненулями на строку). На матрицах
с большей средней плотностью строки или с принципиально другим
распределением ненулей ранжирование backend-ов может измениться.

Перспективы для улучшения:

- адаптивная стратегия SPA — переход на хэш-таблицу для строк с очень
  малым ожидаемым числом ненулей в `C[i, :]`;
- сборка результата через двухпроходную схему с префиксной суммой, что
  убрало бы вторую сериальную фазу `Assemble`;
- лучшее распределение строк в STL — динамическое через workstealing
  очередь на `std::atomic<int>`-курсоре.
