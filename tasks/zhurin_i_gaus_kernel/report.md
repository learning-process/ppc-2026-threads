## 1. Введение

Целью лабораторной работы является реализация фильтрации изображений с использованием ядра Гаусса и сравнение
различных моделей параллельного программирования:

- SEQ;
- OpenMP;
- Intel oneTBB;ы
- `std::thread`;
- MPI + OpenMP.

Задача хорошо подходит для распараллеливания, поскольку вычисление каждого пикселя зависит только от соседних
элементов изображения и может выполняться независимо для различных частей изображения.

В работе были реализованы:

- последовательная версия (SEQ);
- OpenMP-реализация (OMP);
- реализация Intel oneTBB (TBB);
- реализация на `std::thread` (STL);
- гибридная MPI + OpenMP реализация (ALL).

Все реализации используют единый набор функциональных и производительных тестов.

## 2. Единая постановка задачи

Входные данные:

- ширина изображения `width`;
- высота изображения `height`;
- число частей/потоков `parts`;
- двумерная матрица пикселей изображения.

Выходные данные:

- изображение после применения фильтра Гаусса.

Используется ядро Гаусса размера `3×3`:

```cpp
static constexpr std::array<std::array<int, 3>, 3> kKernel = {{
    {1, 2, 1},
    {2, 4, 2},
    {1, 2, 1}
}};
```

Результат вычисляется как взвешенная сумма соседних пикселей с последующей нормализацией делением на 16
(реализовано через побитовый сдвиг вправо).

Для обработки границ изображения используется нулевое дополнение (zero padding).

Корректность результата определяется совпадением всех параллельных реализаций с последовательной версией SEQ.

Ограничения:

- `width > 0`;
- `height > 0`;
- `parts > 0`;
- `parts <= width`;
- размеры матрицы изображения должны соответствовать переданным параметрам.

## 3. Единая методика эксперимента

Экспериментальная среда:

- ОС: Windows 11;
- компилятор: g++ 13.3.0;
- тип сборки: Release.

Используемые переменные окружения:

- `PPC_NUM_THREADS` — число потоков;
- `OMP_NUM_THREADS` — число потоков OpenMP;
- `PPC_NUM_PROC` — число MPI-процессов;
- `PPC_TASK_MAX_TIME` и `PPC_PERF_MAX_TIME` — ограничения времени тестирования.

Методика измерений:

- все реализации используют одинаковые входные данные;
- измеряется время выполнения метода `RunImpl`;
- вычисляется ускорение относительно последовательной версии:

S = T_seq / T_parallel

- эффективность вычисляется как:

E = S / P

где `P` — число потоков или процессов.

Во всех backend-ах используется одинаковое гауссово ядро и единая схема обработки границ.

## 4. Сводка корректности

Корректность всех реализаций проверяется общим набором функциональных тестов.

Проверяются:

- минимальные размеры изображений;
- квадратные изображения;
- прямоугольные изображения;
- случайные изображения;
- разные числа потоков и процессов;
- корректность обработки границ.

Все реализации (`seq`, `omp`, `tbb`, `stl`, `all`) выдают результаты,
совпадающие с SEQ.

## 5. Производительность

Сценарий тестирования:

- изображения большого размера;
- одинаковые входные данные для всех реализаций;
- несколько повторных запусков.

| реализация | режим | конфигурация | ускорение относительно SEQ |
| ----------- | ------ | ------------ | --------------------------- |
| seq | последовательная | 1 поток | 1.00 |
| omp | OpenMP | несколько потоков | наблюдается ускорение за счёт parallel for |
| tbb | oneTBB | task-based parallelism | динамическое распределение нагрузки |
| stl | std::thread | ручное управление потоками | ускорение зависит от числа потоков |
| all | MPI + OpenMP | ranks × threads | ускорение при распределении строк между процессами |

Основные наблюдения:

- OpenMP обеспечивает хорошее ускорение при минимальном количестве изменений кода;
- TBB позволяет гибко балансировать нагрузку между задачами;
- STL предоставляет полный контроль над потоками;
- гибридная ALL-реализация показывает преимущества на больших размерах изображений.

## 6. Интерпретация различий

### SEQ

Последовательная версия используется как эталон корректности и baseline
для вычисления ускорения остальных backend-ов.

### OMP

OpenMP использует директиву `#pragma omp parallel for`
для распараллеливания циклов обработки изображения.

Преимущества:

- минимальные изменения кода;
- автоматическое управление потоками;
- хорошая масштабируемость.

Недостатки:

- зависимость от runtime OpenMP;
- ограниченный контроль над планировщиком.

### TBB

Версия Intel oneTBB использует `parallel_for`
и `blocked_range` для распределения диапазонов.

Преимущества:

- автоматическая балансировка нагрузки;
- task-based модель вычислений;
- гибкое управление grain size.

Недостатки:

- дополнительные накладные расходы runtime;
- более сложная инфраструктура.

### STL

STL-реализация использует `std::thread`.

Преимущества:

- полный контроль над потоками;
- ручное распределение диапазонов.

Недостатки:

- необходимость ручной синхронизации;
- накладные расходы create/join;
- более высокий риск ошибок многопоточности.

### ALL

Гибридная реализация объединяет MPI и OpenMP.

Особенности реализации:

- изображение распределяется по строкам между MPI-процессами;
- внутри процесса используется OpenMP;
- сбор результатов выполняется через `MPI_Allgatherv`.

Преимущества:

- масштабируемость на нескольких процессах;
- комбинирование межпроцессного и внутрипроцессного параллелизма.

Недостатки:

- стоимость MPI-коммуникаций;
- сложность синхронизации;
- увеличение накладных расходов на малых задачах.

## 7. Заключение

В ходе лабораторной работы были реализованы и исследованы пять версий фильтрации изображения
с использованием гауссова ядра.

Основные результаты:

- подтверждена корректность всех реализаций;
- OpenMP показал хорошее соотношение простоты и производительности;
- TBB обеспечил гибкое распределение задач;
- STL позволил реализовать ручное управление потоками;
- гибридная ALL-версия продемонстрировала возможность объединения MPI и OpenMP.

Наибольший эффект параллелизма достигается на изображениях большого размера,
где вычислительная нагрузка превышает накладные расходы на синхронизацию и создание потоков.
