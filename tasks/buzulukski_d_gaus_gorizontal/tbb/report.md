## 1. Контекст
Параллельная версия с использованием библиотеки Intel oneAPI Threading Building Blocks (oneTBB) разработана для ускорения базового алгоритма линейной фильтрации за счет применения продвинутого планировщика задач (Task Scheduler) и концепции абстракции потоков. В отличие от сырых потоков или жесткого секционирования циклов, TBB оперирует понятием диапазонов и задач, что позволяет эффективно балансировать нагрузку (Work Stealing) на многоядерных процессорах со сложной кэш-архитектурой.

## 2. Постановка задачи
Краткая постановка полностью совпадает с seq/report.md. Выходные данные TBB-версии проходят обязательное попиксельное сравнение с последовательным baseline-ядром (SEQ, T_1 = 2.660470 сек), что гарантирует абсолютную корректность и детерминированность результатов фильтрации.

## 3. Базовый алгоритм
Последовательная версия обходит матрицу пикселей во вложенных циклах, вычисляет значение свёртки матрицы 3x3 по всем трем каналам RGB с clamp-обработкой крайних пикселей окрестности и записывает результат в выходной буфер кадра.

## 4. Схема распараллеливания
Распараллеливание вычислительного ядра реализовано с помощью алгоритма tbb::parallel_for.

- Примитив: Выбран tbb::parallel_for, так как итерации по строкам изображения полностью независимы друг от друга и не содержат перекрестных зависимостей по данным.
- Диапазон: Используется класс tbb::blocked_range<int>(0, height_), который адаптивно разбивает вертикальный размер кадра на независимые поддиапазоны.
- Размер чанка (grainsize): Специально оставлен по умолчанию. Это указывает рантэйму TBB применять рекурсивное деление диапазона до тех пор, пока накладные расходы на диспетчеризацию задач не начнут превышать полезную вычислительную работу. Такой подход защищает систему от избыточного overhead на мелких задачах.
- Разбиение (Partitioner): Применяется дефолтный tbb::auto_partitioner, который автоматически подбирает оптимальный размер поддиапазонов строк в зависимости от текущей аппаратной загрузки CPU воркерами.
- Конкуренция: Количество активных рабочих потоков динамически регулируется фреймворком курса через глобальные настройки рантэйма, контролирующие выполнение пула задач внутри фабрики MakeAllPerfTasks посредством механизмов tbb::global_control.

### Листинг ключевого фрагмента (ядро распараллеливания)
```cpp
bool BuzulukskiDGausGorizontalTBB::RunImpl() {
  const int h = height_;
  const int w = width_;
  const uint8_t *in_ptr = input_image_.data();
  uint8_t *out_ptr = output_image_.data();

  tbb::parallel_for(tbb::blocked_range<int>(0, h), [&](const tbb::blocked_range<int> &r) {
    for (int py = r.begin(); py < r.end(); ++py) {
      for (int px = 0; px < w; ++px) {
        for (int ch = 0; ch < kChannels; ++ch) {
          size_t out_idx = (((static_cast<size_t>(py) * static_cast<size_t>(w)) + static_cast<size_t>(px)) * kChannels) + static_cast<size_t>(ch);
          out_ptr[out_idx] = CalculatePixelTBB(in_ptr, py, px, w, h, ch);
        }
      }
    }
  });
  return true;
}
```

## 5. Детали реализации
Файлы: tbb/include/ops_tbb.hpp, tbb/src/ops_tbb.cpp

В реализации строго разграничена логика инфраструктурных этапов фреймворка. Метод RunImpl содержит параллельное ядро TBB. Использование захвата по ссылке `[&]` внутри лямбда-функции в `tbb::parallel_for` обеспечивает разделяемый доступ потоков-воркеров к общим неизменяемым структурам и указателям данных (`in_ptr`, `out_ptr`, `w`). При этом все локальные переменные, объявляемые непосредственно внутри тела вложенных циклов (`py`, `px`, `ch`, `out_idx`), автоматически размещаются в индивидуальном стеке каждого конкретного потока-воркера. Это гарантирует полную потокобезопасность (thread-safety) без использования ресурсоёмких блокировок или атомарных операций. Потоки работают со своими изолированными поддиапазонами строк общего массива `output_image_` (благодаря интервалам `r.begin()` и `r.end()`), что минимизирует риски ложного разделения кэша (false sharing) на стыках обрабатываемых блоков. В работе используется автоматический планировщик `tbb::auto_partitioner` и дефолтный расчет `grainsize`, что позволяет рантайму TBB самостоятельно адаптировать размер подблоков строк под текущую нагрузку процессора.
