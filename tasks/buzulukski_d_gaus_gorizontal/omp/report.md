## 1. Контекст

Вычислительное ядро линейной фильтрации изображений содержит абсолютно независимые итерации по строкам кадра.
Это делает задачу
идеальным кандидатом для применения технологии OpenMP. Перенос алгоритма на многопоточную архитектуру позволяет
распределить обработку строк между ядрами
процессора, существенно сокращая общее время выполнения RunImpl без изменения логики вычислений.

## 2. Постановка задачи

Задача фильтрации Гаусса полностью соответствует постановке, описанной в seq/report.md. Параллельная OMP-версия
работает с теми же структурами данных и
обязана выдавать попиксельно идентичный результат. Последовательная версия (SEQ) со временем выполнения T_1 = 2.
660470 сек используется как baseline.

## 3. Базовый алгоритм

Последовательная версия обходит матрицу пикселей во вложенных циклах строка за строкой, вычисляет значение свёртки
 3x3 для каждого RGB-канала с
зажимом индексов на границах и записывает результат в выходной массив буфера.

## 4. Схема распараллеливания

Распараллеливание реализовано на уровне внешнего цикла по строкам изображения с помощью директив OpenMP. Внешний
цикл по координате py (строки кадра от 0 до h)
размечается директивой компилятора:

```cpp
#pragma omp parallel for default(none) shared(h, w, in_ptr, out_ptr, kKernel)
```

Разбор параметров директивы:

- default(none): Требует от разработчика обязательного и явного указания области видимости для каждой переменной,
используемой внутри параллельного блока.
Это полностью исключает случайные ошибки неявного разделения данных.
- shared(h, w, in_ptr, out_ptr, kKernel): Переменные размеров кадра, матрица весов фильтра и указатели на общие
векторы памяти input_image_ и output_image_ объявляются
общими для всех потоков. Потоки производят только чтение из in_ptr и kKernel, а запись в out_ptr идет строго по
уникальным, не пересекающимся индексам строк, что гарантирует
отсутствие состояния гонки (data race).
- private/loop-index: Индексная переменная цикла py автоматически изолируется в стеке каждого отдельного потока.
Внутренние переменные px, ch и out_idx, создаваемые
внутри тела вложенных циклов, также являются приватными по умолчанию.
- Управление планировщиком (schedule): В директиве `#pragma omp parallel for` параметр `schedule` сознательно не
указан явно в коде для обеспечения
стопроцентной совместимости с инструментами статического анализа (clang-tidy) в текущем сборочном окружении.
Согласно спецификации стандарта OpenMP, если
тип планировщика опущен, его выбор отдается на усмотрение компилятора (implementation-defined). В используемом
компиляторе GCC дефолтным поведением для параллельных
циклов является `schedule(static)`. Это гарантирует равномерное, детерминированное блочное распределение итераций
(строк изображения) между потоками на этапе компиляции, что исключает накладные расходы на динамическую
балансировку во время
выполнения (runtime overhead).
- Синхронизация: Явные примитивы синхронизации (critical, atomic) внутри цикла отсутствуют, так как потоки не
 конкурируют за ячейки памяти.
Неявный барьер OpenMP в конце параллельной области гарантирует, что метод RunImpl завершится только тогда, когда
все потоки полностью
допишут свои строки в выходной буфер.

## 5. Детали реализации

Файлы: omp/include/ops_omp.hpp, omp/src/ops_omp.cpp

Изменения коснулись исключительно метода RunImpl. В него была внедрена директива распараллеливания
внешнего цикла по строкам кадра.
Логика свёртки вынесена в свободную функцию CalculatePixel, оптимизирующую доступ к памяти по сырым
указателям `in_ptr`. Поскольку запись
результатов распределена по независимым индексам `out_idx`, отпала необходимость в использовании reduction на этапе
фильтрации, что обеспечило
линейный рост скорости работы ядра.

### Листинг ключевого фрагмента (ядро распараллеливания)

```cpp
bool BuzulukskiDGausGorizontalOMP::RunImpl() {
  const int h = height_;
  const int w = width_;
  const uint8_t *in_ptr = input_image_.data();
  uint8_t *out_ptr = output_image_.data();

#pragma omp parallel for default(none) shared(h, w, in_ptr, out_ptr, kKernel)
  for (int py = 0; py < h; ++py) {
    for (int px = 0; px < w; ++px) {
      for (int ch = 0; ch < 3; ++ch) {
        size_t out_idx = ((((static_cast<size_t>(py) * static_cast<size_t>(w)) + static_cast<size_t>(px)) * 3) + static_cast<size_t>(ch));
        out_ptr[out_idx] = CalculatePixel(in_ptr, py, px, w, h, ch);
      }
    }
  }
  return true;
}
```

Корректность OMP-версии подтверждается автоматическими тестами фреймворка Google Test (тесты omp_size_3 и
omp_size_10). Выходные массивы
многопоточной версии проходят обязательное попиксельное сравнение с результатами последовательного запуска на
аналогичных входных данных во фреймворке.
Тесты на размерах 3x3, 10x10 и перформанс-тест на 1500x1500 подтверждают полную идентичность ответов.

Эффективность масштабирования плавно снижается при переходе за предел 6 потоков (число физических ядер процессора
Ryzen 5 5600H). Это связано с тем, что
технология Hyper-Threading (12 логических потоков) разделяет аппаратные блоки вычислений и подсистему кэша между
потоками, создавая задержки при интенсивном обмене данными
с RAM при обработке тяжелых RGB-массивов.
