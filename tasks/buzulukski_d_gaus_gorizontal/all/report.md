## 1. Context
Разработка гибридной версии (MPI + OpenMP) обусловлена необходимостью эффективной 
утилизации вычислительных мощностей современных высокопроизводительных кластерных систем (HPC).
Архитектура таких систем является иерархической: они состоят из множества независимых 
узлов (нод), объединенных сетью, где каждый узел представляет собой многоядерную систему с общей памятью. 
Одиночные технологии параллелизма здесь неэффективны: чистый OpenMP не способен выйти за пределы одной 
ноды, а чистый MPI на одной ноде плодит избыточные процессы, порождая оверхед на межпроцессное 
взаимодействие 
(IPC) и дублирование данных. Гибридный подход объединяет модель распределенной памяти (MPI) 
для межконтекстного обмена и модель общей памяти (OpenMP) для эффективного внутрипроцессного распараллеливания, 
минимизируя накладные расходы.

## 2. Постановка задачи
Задача заключается в реализации двумерной линейной фильтрации квадратного RGB-изображения 
(3 байта на пиксель, kChannels = 3) размерностью 1500х1500 пикселей дискретным ядром 
Гаусса 3x3. Гибридная версия обязана обеспечивать стопроцентную попиксельную идентичность 
результирующего массива данных эталонному последовательному коду. В качестве базового baseline 
для расчета метрик ускорения и эффективности используется однопоточный запуск последовательной 
версии (SEQ) со временем выполнения T_1 = 2.660470 сек (подробное описание математической 
модели приведено в корневом отчете root report).

## 3. Базовый алгоритм
Последовательное вычислительное ядро осуществляет построчный обход матрицы пикселей кадра. 
Для каждой цветовой компоненты (RGB) текущего пикселя вычисляется операция дискретной 
свёртки — взвешенная сумма интенсивностей пикселей в радиусе локальной апертуры 3x3 с коэффициентами 
матрицы Гаусса, деленная на нормировочный делитель (16). Обработка крайних пикселей на 
границах изображения во избежание выхода за пределы динамического массива (Out of Bounds) реализована 
по защитной clamp-стратегии: граничные индексы принудительно проецируются на крайние валидные 
координаты кадра.

## 4. Межпроцессная схема
- Роли рангов: В архитектуре выделяется выделенный корневой процесс (Master, rank == 0) и пул рабочих 
процессов (Workers, rank > 0). Корневой процесс отвечает за инициализацию 
исходных данных, распределение параметров задачи, запуск локального вычислительного ядра, а также за последующий 
сбор, валидацию и агрегацию результатов. Рабочие процессы (rank > 0) 
выполняют исключительно вычисление свёртки Гаусса на выделенных им поддиапазонах данных и передачу локальных буферов 
обратно на Master.
- Распределение данных: Глобальное пространство строк изображения высотой h декомпозируется вдоль вертикальной оси 
между всеми запущенными MPI-процессами в коммуникаторе MPI_COMM_WORLD. Каждый процесс на основе своего
уникального rank и общего числа процессов size рассчитывает локальную зону ответственности по формуле:
  rows_per_proc = h / size
  start_row = rank * rows_per_proc
  end_row = (rank == size - 1) ? h : (i + 1) * rows_per_proc
  Тернарный оператор гарантирует, что последний ранг заберет остаток строк кадра при неделимом остатке, 
  исключая потерю данных.
- MPI-вызовы:
  * MPI_Comm_rank — запрашивает у рантэйма уникальный идентификатор (ранг) текущего процесса в глобальном 
  коммуникаторе для выполнения 
  условного ветвления логики (Master/Worker).
  * MPI_Comm_size — определяет общее число доступных процессов для расчета корректных шагов геометрической 
  декомпозиции строк.
  * MPI_Gatherv — коллективная операция сбора данных с переменным размером блоков. Используется в конце 
  вычислительного шага для агрегации 
  локальных буферов со всех рангов в единый результирующий вектор на корневом процессе (rank == 0).

### Иллюстративный листинг распределения и сборки данных на уровне MPI:
```cpp
  int rank = 0;
  int size = 0;
  MPI_Comm_rank(MPI_COMM_WORLD, &rank);
  MPI_Comm_size(MPI_COMM_WORLD, &size);

  int rows_per_proc = h / size;
  int start_row = rank * rows_per_proc;
  int end_row = (rank == size - 1) ? h : (rank + 1) * rows_per_proc;

  // Выделение локального буфера для записи результатов процесса
  std::vector<uint8_t> local_output((static_cast<size_t>(end_row) - start_row) * w * kChannels);
  uint8_t *local_ptr = local_output.data();

  // [... Здесь выполняется внутрипроцессный расчет OpenMP ...]

  // Сборка локальных результатов со всех рангов на корневом процессе
  MPI_Gatherv(local_ptr, static_cast<int>(local_output.size()), MPI_UNSIGNED_CHAR, output_image_.data(),
              recv_counts.data(), displs.data(), MPI_UNSIGNED_CHAR, 0, MPI_COMM_WORLD);
```

## 5. Внутрипроцессная схема
- Внутрипроцессная технология: На уровне отдельного MPI-процесса (внутри его изолированного адресного пространства) 
для 
многопоточного распараллеливания вычислений применяется технология OpenMP.
- Разделение работы между потоками: На вложенные циклы обхода локальной матрицы строк (start_row до end_row) и 
столбцов (0 до w) накладывается 
директива #pragma omp parallel for collapse(2). Использование collapse(2) объединяет пространства итераций двух 
внешних циклов в единый плотный линейный диапазон, который затем планировщик OpenMP 
нарезает на равные порции между доступными потоками внутри текущего процесса. Это позволяет максимизировать 
загрузку ядер и минимизировать дисбаланс нагрузки при малом 
количестве локальных строк.
- Объединение результатов: Поскольку потоки OpenMP работают в общей памяти процесса, они осуществляют запись 
вычисленных пикселей напрямую в локальный вектор 
local_output по уникальным, математически изолированным индексам local_idx. Дополнительная потоковая синхронизация 
(мьютексы, критические секции) не требуется, 
так как потоки физически никогда не пишут в одни и те же ячейки памяти. По завершении параллельного региона OpenMP 
происходит неявный барьер синхронизации потоков, 
гарантирующий полную готовность локального буфера процесса перед вызовом MPI-обмена.

### Иллюстративный листинг внутрипроцессного параллельного региона OpenMP:
```cpp

  #pragma omp parallel for collapse(2) default(none) shared(start_row, end_row, w, h, input_ptr, local_ptr)
  for (int py = start_row; py < end_row; ++py) {
    for (int px = 0; px < w; ++px) {
      for (int ch = 0; ch < kChannels; ++ch) {
        int local_py = py - start_row;
        size_t local_idx = ((static_cast<size_t>(local_py) * w + px) * kChannels) + ch;
        
        // Потокобезопасная прямая запись в изолированные индексы общего локального буфера
        local_ptr[local_idx] = CalculatePixelALL(input_ptr, py, px, w, h, ch);
      }
    }
  }
```

## 6. Детали реализации
- Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp
- Класс: BuzulukskiDGausGorizontalALL
- Точки синхронизации: Точкой глобальной межпроцессной синхронизации выступает вызов MPI_Gatherv. Он неявно 
заставляет процессы ожидать 
готовности данных от самого медленного участника. Явные вызовы MPI_Barrier в логику RunImpl сознательно не 
включались, так как коллективный 
сборщик Gatherv сам по себе гарантирует необходимый и достаточный барьерный барьер перед переходом к этапу 
постобработки. На этапе PostProcessingImpl() 
вычисления (подсчет среднего значения) локализованы строго внутри блока if (rank == 0).
- Потенциальные узкие места:
  1. Сериализация при сборке: На корневом ранге операция MPI_Gatherv производит последовательное копирование 
  данных из сетевых буферов в общую память, что 
  при большом числе рангов создает затор (Bottleneck).
  2. Memory Bound ограничение: Приложение активно считывает входную матрицу и пишет в выходную. При высокой 
  плотности потоков OpenMP на один процесс пропускная 
  способность шины памяти процессора становится узким местом.

## 7. Проверка корректности
- Сравнение с SEQ: Корректность работы гибридного ядра проверяется модульными тестами во фреймворке 
Google Test (тесты all_size_3 и all_size_10). Результат фильтрации 
на корневом процессе попиксельно сравнивается с массивом, полученным в ходе последовательного выполнения. 
Тесты проходят успешно во всех конфигурациях.
- Согласованность между рангами: Поскольку расчет границ (start_row/end_row) и смещений (displs) на Master-процессе 
и на Worker-процессах выполняется по абсолютно идентичным математическим 
формулам, гарантируется строгая непрерывность и непересекаемость собираемых сегментов изображения. Тестирование 
на вырожденных размерах (когда число рангов превышает высоту кадра) подтвердило 
стабильность: «избыточные» ранги вычисляют нулевые диапазоны и корректно передают 0 байт через MPI_Gatherv, не 
вызывая аварийного завершения.

## 8. Экспериментальная среда
- Управление конфигурацией: Параметры запуска гибридной среды задаются через внешние переменные окружения
  и флаги утилиты запуска. Количество MPI-процессов определяется флагом -n утилиты mpiexec.
  Количество OpenMP-потоков внутри каждого процесса контролируется переменной окружения `$env:OMP_NUM_THREADS`
  (или системной `PPC_NUM_THREADS` в зависимости от конфигурации каркаса).
- Конфигурация запуска `(ranks * threads)`: Тестирование проводилось на процессоре AMD Ryzen 5 5600H
  (6 физических ядер с Hyper-Threading, дающих 12 логических потоков) в трех гибридных конфигурациях,
  где суммарное число аппаратных работников `(total_workers = ranks * threads_per_rank)`
  жестко лимитировалось вычислительным пределом процессора.
- Сборка: Release сборка с флагом оптимизации -O3, компилятор GCC 13.2.0.
- Команда запуска:

  ```bash
  mpiexec -n <ranks> ./build/bin/ppc_perf_tests.exe --gtest_filter="*Buzulukski*all*"
  ```

*Нормировка метрик:* Расчет коэффициента ускорения и эффективности производится относительно
общего числа задействованных аппаратных работников (произведения числа процессов на количество потоков),
что обеспечивает методическую корректность сравнения гибридной версии с чистыми однородными backend-ами
(чистым STL или чистым OpenMP на 12 потоков).

*Комментарий о цене коммуникации:*
Данные таблицы наглядно демонстрируют цену межпроцессного взаимодействия (MPI communication overhead).
При неизменном общем числе вычислителей (12 ядер), наилучший результат достигается в конфигурации
двух рангов на шесть потоков (0.42 сек). Здесь OpenMP эффективно утилизирует общую память внутри сокета процессора,
а MPI выполняет всего одну быструю склейку двух больших кусков данных.

При сдвиге конфигурации в сторону чистых рангов (12 рангов на 1 поток) время ухудшается до 0.64 сек.
Это доказывает, что оверхед на копирование множества мелких фрагментов строк в сети или IPC
через функцию `MPI_Gatherv` начинает доминировать над математической свёрткой пикселей.

## 10. Выводы
Гибридная схема MPI + OpenMP полностью оправдана в сценариях крупноблочного параллелизма на реальных кластерах, 
когда объемы вычислений внутри одного ранга настолько велики, 
что накладные расходы на сетевой обмен становятся незаметными. На локальных многоядерных процессорах (в пределах 
одной ноды) гибридный параллелизм избыточен: наращивание числа MPI-рангов 
приводит к деградации производительности из-за накладных расходов на коммуникацию и синхронизацию. Для локальных 
систем более эффективным выбором остаются чистые технологии общей памяти 
(OpenMP, TBB, std::thread), минимизирующие оверхед на передачу сообщений.
