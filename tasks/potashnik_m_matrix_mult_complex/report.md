## 1. Введение

Задача умножения разреженных матриц с комплексными элементами
в формате CCS выбрана для сравнения различных моделей
параллельного программирования: директивной (OpenMP),
задачно-ориентированной (oneTBB), низкоуровневой ручной
(std::thread) и гибридной (MPI + STL). Задача интересна тем,
что разреженная структура данных создаёт нерегулярный доступ к
памяти, а комплексная арифметика добавляет вычислительную
нагрузку. Сравнение позволяет оценить, насколько каждая
технология справляется с нерегулярностью данных и каковы
накладные расходы на управление параллелизмом.

## 2. Единая постановка задачи

**Входные данные:** две разреженные матрицы A (height × width)
и B (height' × width') в формате CCS (три вектора: значения
`val`, индексы строк `row_ind`, индексы столбцов `col_ptr`).

**Выходные данные:** разреженная матрица C = A × B в формате CCS.

**Ограничения:**

- Умножение определено только если A.width == B.height.
- Нулевые элементы не хранятся.
- Комплексные числа умножаются по правилу (a+bi)(c+di) = (ac-bd) + (ad+bc)i.

**Критерий корректности:** результат каждой параллельной реализации должен
побитово совпадать с результатом последовательной версии.

### Генерация данных

Для функциональных тестов используются детерминированно
сгенерированные матрицы 10 различных размеров (от 1×1 до
9×3). Для тестов производительности — матрицы размером 11000×11000
с ровно 2 ненулевыми элементами на строку (nnz = 22000 для каждой).

### Размеры задач для производительности

- Матрицы: 11000 × 11000
- nnz(A) = 22000, nnz(B) = 22000
- Доля заполнения: ~0.018%

### Метрики

**Ускорение (Speedup):** Speedup = T_seq / T_parallel, где T_seq = 149 мс (медиана SEQ)

**Эффективность (Efficiency):** Efficiency = Speedup / total_workers,
где total_workers — число потоков (для OMP, TBB, STL) или ranks × threads (для ALL)

### Процедура измерений

- 10 повторов для каждой конфигурации
- Фиксируется медиана, минимум и максимум
- Измеряется время выполнения task_run (один вызов оператора умножения)

## 4. Сводка корректности

Все backend-ы сравнивались с последовательной версией через 10 функциональных тестов.

| Backend | Тесты пройдены | Примечание                                       |
|---------|----------------|--------------------------------------------------|
| SEQ     | Да             | Эталон                                           |
| OMP     | Да             | schedule(static), локализация через вектор карт  |
| TBB     | Да             | enumerable_thread_specific, blocked_range2d      |
| STL     | Да             | Ручное разбиение, все потоки запущены до join    |
| ALL     | Да             | MPI + STL, конфигурация 2×2                      |

**Ограничения применимости:**

- Все версии требуют совместимости размеров матриц.
- При nnz(A) = 0 все версии корректно возвращают пустую матрицу.
- При числе потоков/процессов, превышающем nnz(A), некоторые работники не получают работы.

## 6. Интерпретация различий

### SEQ (базовый эталон)

Последовательная версия выполняет двойной вложенный цикл за
149 мс. Эталон стабилен и детерминирован. Основная
вычислительная нагрузка приходится на проверку `col_left == row_right`
и вставку в `std::map`.

### OpenMP (2.81× на 4 потоках, эффективность 70.3%)

OpenMP показал хорошее ускорение на 2 потоках (1.84×,
эффективность 92.1%), но при переходе на 4 потока
эффективность упала до 70.3%. Причина — слияние 4 локальных
`std::map` выполняется однопоточно, и накладные расходы
на объединение растут с числом потоков.

### TBB (3.17× на 4 потоках, эффективность 79.3%)

TBB оказался эффективнее OpenMP благодаря двумерной
декомпозиции `blocked_range2d` и `enumerable_thread_specific`,
который более эффективно управляет поток-локальными данными.
Work stealing динамически балансирует нагрузку.

### STL (3.39× на 4 потоках, эффективность 84.8%)

STL показал наилучшее ускорение среди однопроцессных
реализаций. Ручное управление потоками позволило оптимально
разбить диапазоны и минимизировать накладные расходы. Цена —
бо́льшая многословность кода по сравнению с OMP и TBB.

### ALL (2.57× на 2×2, эффективность 64.3%)

Гибридная версия показала наименьшее ускорение. Причины:
коммуникационные overhead'ы MPI (Bcast, Send/Recv, Gatherv),
дублирование памяти (каждый процесс хранит копию B),
сериализация на процессе 0. Гибридный подход оправдан при
очень больших размерах задачи или на нескольких узлах кластера.

## 8. Заключение

### Какая версия лучше и при каких условиях

<!-- markdownlint-disable MD013 -->

| Условия                               | Рекомендуемая версия | Ускорение         |
|---------------------------------------|----------------------|-------------------|
| Максимальная производительность       | STL                  | 3.39×             |
| Баланс производительности и простоты  | TBB                  | 3.17×             |
| Быстрая разработка                    | OpenMP               | 2.81×             |
| Распределение на несколько узлов      | ALL (MPI + STL)      | 2.57× (на 1 узле) |

<!-- markdownlint-enable MD013 -->

### Ограничения сравнения

1. Разреженность фиксирована (2 элемента на строку).
2. Размер задачи фиксирован (11000 × 11000).
3. ALL тестировалась на одном узле.

### Что можно улучшить

1. Заменить `std::map` на `std::unordered_map` для уменьшения логарифмического фактора.
2. Использовать `std::vector` + sort + merge для промежуточных результатов.
3. Динамическое планирование для неравномерной нагрузки.
4. Асинхронные MPI-операции для наложения коммуникации на вычисления.

### Итоговая оценка

На данной задаче STL показал наилучшую производительность
(3.39×), TBB — лучший баланс между производительностью и
простотой (3.17×), OpenMP — самый простой способ
распараллеливания (2.81×). Основным узким местом всех
реализаций является не модель распараллеливания, а внутреннее
устройство `std::map`.
