## 1. Контекст

TBB-версия реализована для оценки задачно-ориентированной
модели параллелизма. В отличие от OpenMP, TBB использует пул
потоков с динамической балансировкой (work stealing) и
предоставляет высокоуровневые примитивы — `parallel_for`,
`blocked_range2d`, `enumerable_thread_specific`. Двумерная
декомпозиция пространства итераций (`blocked_range2d`)
позволяет тоньше контролировать зернистость задач.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md`. Напоминание:

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

## 3. Базовый алгоритм (из SEQ)

Двойной вложенный цикл является вычислительным ядром.
В TBB-версии оба цикла параллелизуются через `blocked_range2d`.

## 4. Схема распараллеливания

### Двумерная декомпозиция

```cpp
const size_t grain_i = 64;
const size_t grain_j = 256;

tbb::parallel_for(
  tbb::blocked_range2d<size_t>(0, left_count, grain_i, 0, right_count, grain_j),
  [&](const tbb::blocked_range2d<size_t> &r) {
    auto &local_buffer = local_buffers.local();
    for (size_t i = r.rows().begin(); i != r.rows().end(); ++i) {
      // ...
      for (size_t j = r.cols().begin(); j != r.cols().end(); ++j) {
        if (col_left == row_ind_right[j]) {
          local_buffer[{row_left, col_ptr_right[j]}] += left_val * val_right[j];
        }
      }
    }
  });
```

- **`blocked_range2d`** разбивает пространство `nnz_A × nnz_B` на двумерные блоки.
- **`grain_i = 64`** и **`grain_j = 256`** — размеры блоков,
подобранные для баланса между накладными расходами и полезной работой.
- **Work stealing:** свободные потоки захватывают блоки у занятых, обеспечивая динамическую балансировку.

### Преимущество двумерного разбиения перед одномерным

OMP и STL делят только внешний цикл (по `i`). TBB делит оба
цикла, что позволяет лучше балансировать нагрузку при
неравномерном распределении ненулевых элементов.

### Поток-локальные данные

```cpp
tbb::enumerable_thread_specific<LocalMap> local_buffers;
```

`enumerable_thread_specific` автоматически создаёт экземпляр
`LocalMap` для каждого потока при первом обращении через
`.local()`. Это эффективнее вектора по индексу (`local_buffers[tid]`),
так как исключает ложное разделение кэша и снижает contention.

### Слияние результатов

```cpp
std::map<Key, Complex> buffer;
for (const auto &local : local_buffers) {
  for (const auto &[key, value] : local) {
    buffer[key] += value;
  }
}
```

Слияние выполняется однопоточно после завершения `parallel_for`.

## 5. Детали реализации

**Файлы:**

- `tbb/include/ops_tbb.hpp` — объявление класса.
- `tbb/src/ops_tbb.cpp` — реализация с oneTBB.

**Структура пайплайна** полностью повторяет `seq`:

- `ValidationImpl` — проверка совместимости.
- `PreProcessingImpl` / `PostProcessingImpl` — без действий.
- `RunImpl` — `parallel_for`, слияние, формирование результата с `reserve`.

**Оптимизация финальной сборки:**

В TBB-версии используется `reserve(buffer.size())` для `val`, `row_ind`, `col_ptr`,
что избегает лишних аллокаций при построении результирующей матрицы.

## 8. Результаты

### Базовые замеры (task_run)

| Потоков | Время (мс), медиана (min-max) | Ускорение | Эффективность |
|---------|-------------------------------|-----------|---------------|
| 1 (SEQ) | 149 (147-151)                 | 1.00      | 100%          |

| 4       | 47 (45-49)                    | 3.17      | 79.3%         |

*Измерения: 10 повторов для каждого количества потоков.*

### Анализ масштабируемости

- `1 → 2` потока: ускорение 1.89× (эффективность 94.5%) — лучше, чем у OMP на тех же данных.
- `2 → 4` потока: ускорение 1.68× (с 79 до 47 мс), эффективность 79.3%.

TBB превосходит OMP за счёт двумерного разбиения и
`enumerable_thread_specific`, которые снижают contention и
улучшают балансировку нагрузки.

## 9. Выводы

1. **Лучшее ускорение среди однопроцессных технологий** (3.17× на 4 потоках).
2. **Двумерная декомпозиция** `blocked_range2d` обеспечивает более тонкую балансировку.
3. **`enumerable_thread_specific`** эффективнее вектора по индексу — меньше contention.
4. **Work stealing** динамически перераспределяет задачи, что важно при неравномерной нагрузке.
5. **Простота кода** сопоставима с OMP при более высокой производительности.

Основным узким местом остаётся `std::map` — логарифмические вставки и аллокации.
