## 1. Контекст

OMP-версия реализована для оценки директивной модели
параллелизма при умножении разреженных матриц. OpenMP
позволяет распараллелить внешний цикл минимальными изменениями
кода по сравнению с SEQ. Ключевой вопрос — насколько хорошо
директивная модель справляется с нерегулярным доступом к
памяти, характерным для разреженных матриц.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md`. Напоминание:

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

## 3. Базовый алгоритм (из SEQ)

Двойной вложенный цикл является вычислительным ядром.
В OMP-версии внешний цикл по элементам матрицы `A` распределяется между потоками.

## 4. Схема распараллеливания

### Распределение итераций

Итерации внешнего цикла по `nnz(A)` распределяются между потоками через `#pragma omp for schedule(static)`.

```cpp
#pragma omp parallel num_threads(threads) default(none) \
    shared(local_buffers, matrix_left, matrix_right, ...)
{
  int tid = omp_get_thread_num();
  auto &local_buffer = local_buffers[tid];
  size_t left_count = matrix_left.Count();

#pragma omp for schedule(static)
  for (size_t i = 0; i < left_count; i++) {
    // вычисления и накопление в local_buffer
  }
}
```

### Атрибуты переменных

| Переменная      | Атрибут  | Обоснование                                         |
|-----------------|----------|-----------------------------------------------------|
| `local_buffers` | shared   | Каждый поток обращается только к своему элементу    |
| `matrix_left`   | shared   | Только чтение                                       |
| `matrix_right`  | shared   | Только чтение                                       |
| `tid`           | private  | Каждый поток имеет свой идентификатор               |
| `local_buffer`  | private  | Ссылка на свой элемент вектора                      |

### Выбор schedule

`schedule(static)` выбран потому, что нагрузка на итерации
равномерна (каждая итерация внешнего цикла выполняет одинаковое
число операций по внутреннему циклу). Статическое планирование
минимизирует накладные расходы по сравнению с динамическим.

### Где хранятся локальные результаты

```cpp
int threads = omp_get_max_threads();
std::vector<std::map<std::pair<size_t, size_t>, Complex>> local_buffers(threads);
```

Каждый поток пишет в `local_buffers[tid]` — гонок нет, так как разные потоки пишут в разные элементы вектора.

### Слияние результатов

```cpp
std::map<std::pair<size_t, size_t>, Complex> buffer;
for (const auto &local : local_buffers) {
  for (const auto &[key, value] : local) {
    buffer[key] += value;
  }
}
```

Слияние выполняется однопоточно после завершения параллельной области.

## 6. Проверка корректности

Корректность проверяется теми же **10 функциональными тестами**, что и для SEQ.

Для всех размеров матриц и для `num_threads = 1, 2, 4` результаты побитово совпадают с SEQ.
Это подтверждает корректность разбиения и отсутствие гонок.

## 8. Результаты

### Базовые замеры (task_run)

| Потоков | Время (мс), медиана (min-max) | Ускорение | Эффективность |
|---------|-------------------------------|-----------|---------------|
| 1 (SEQ) | 149 (147-151)                 | 1.00      | 100%          |
| 2       | 81 (79-83)                    | 1.84      | 92.1%         |
| 4       | 53 (51-55)                    | 2.81      | 70.3%         |

*Измерения: 10 повторов для каждого количества потоков.*

### Анализ масштабируемости

- `1 → 2` потока: ускорение 1.84× (эффективность 92.1%) — почти линейное.
- `2 → 4` потока: ускорение 1.53× (с 81 до 53 мс), эффективность снижается до 70.3%.

Снижение эффективности при 4 потоках обусловлено накладными
расходами на слияние локальных `std::map` и конкуренцией за
аллокатор памяти.

## 9. Выводы

1. **Хорошее ускорение на 2 потоках:** 1.84× (эффективность 92.1%).
2. **Снижение эффективности на 4 потоках** до 70.3% — следствие однопоточного слияния локальных карт.
3. **Простота реализации:** директивный подход требует минимальных изменений относительно SEQ.
4. **Отсутствие гонок** обеспечено изоляцией данных через `local_buffers[tid]`.

Основным узким местом является не модель распараллеливания, а внутреннее устройство `std::map`.
