## 1. Контекст

Гибридная версия (MPI + STL) реализована для оценки
эффективности иерархического параллелизма при умножении
разреженных матриц. MPI обеспечивает межпроцессное
распределение данных (элементов матрицы `A` между
процессами), а `std::thread` внутри каждого процесса
распараллеливает обработку полученного блока. Такой подход
потенциально позволяет задействовать несколько узлов кластера
(через MPI) и все ядра каждого узла (через STL threads). В
рамках данной работы гибридная версия выполняется на одном
узле, но структура кода готова к масштабированию.

**Базовый эталон:** `seq/report.md`.

## 2. Постановка задачи

Полностью соответствует `seq/report.md`. Напоминание:

- Вход: две разреженные матрицы `A` (height × width) и `B` (height' × width'), где `width == height'`.
- Выход: разреженная матрица `C = A × B` в формате CCS.
- Алгоритм: перебор всех `nnz(A)` и `nnz(B)` с накоплением результатов в `std::map<(row, col), Complex>`.

В гибридной версии элементы `A` распределяются между MPI-процессами, матрица `B` рассылается всем процессам (broadcast).
Каждый процесс обрабатывает свою порцию с помощью `std::thread`, а результаты собираются на процессе 0.

## 3. Базовый алгоритм (из SEQ)

Двойной вложенный цикл является вычислительным ядром. В гибридной версии:

- **MPI** распределяет итерации внешнего цикла между процессами.
- **`std::thread`** внутри каждого процесса распределяет свою порцию итераций между потоками.

## 4. Межпроцессная схема (MPI)

### Роли rank-ов

- **Ранг 0 (главный процесс):**
  - Владеет исходными матрицами `A` и `B`.
  - Выполняет `Scatter` матрицы `A`: разбивает `nnz(A)` на блоки и рассылает их процессам через `MPI_Send`.
  - Выполняет `Broadcast` матрицы `B` через `MPI_Bcast`.
  - После вычислений собирает локальные результаты через `MPI_Gatherv`.

- **Ранги 1..N-1 (рабочие процессы):**
  - Получают свою порцию элементов `A` через `MPI_Recv`.
  - Получают копию матрицы `B` через `MPI_Bcast`.
  - Выполняют локальное умножение с внутрипроцессным распараллеливанием через `std::thread`.
  - Отправляют локальные результаты на процесс 0 через `MPI_Gatherv`.

### Распределение данных

Матрица `A` распределяется по итерациям внешнего цикла:

```cpp
int blocksize = static_cast<int>(total) / world_size;
int leftover = static_cast<int>(total) % world_size;
for (int proc = 0; proc < world_size; ++proc) {
  sendcounts[proc] = blocksize + (proc < leftover ? 1 : 0);
}
```

Каждый процесс получает непрерывный блок элементов `A`. Размеры блоков отличаются не более чем на 1.

### MPI-вызовы и их назначение

| Вызов                        | Назначение                                                                        |
|------------------------------|-----------------------------------------------------------------------------------|
| `MPI_Comm_rank`              | Получение ранга текущего процесса                                                 |
| `MPI_Comm_size`              | Получение общего числа процессов                                                  |
| `MPI_Bcast`                  | Рассылка матрицы `B` и метаданных матрицы `A` от процесса 0 всем процессам        |
| `MPI_Send` / `MPI_Recv`      | Пересылка блоков матрицы `A` от процесса 0 рабочим процессам                      |
| `MPI_Gather`                 | Сбор количества результатов от каждого процесса                                   |
| `MPI_Gatherv`                | Сбор данных результатов со всех процессов (разные объёмы у разных процессов)      |

**Почему `MPI_Send/Recv` вместо `MPI_Scatterv`:**
Ручная рассылка даёт больше контроля над типами данных при передаче
структур с несколькими полями (`row_ind`, `col_ptr`, `re`, `im`).

## 5. Внутрипроцессная схема (STL)

### Выбранная thread-технология: std::thread

Внутри каждого MPI-процесса используется `std::thread` для
распараллеливания обработки полученного блока. Число потоков
определяется через `std::thread::hardware_concurrency()`.

### Как делится работа между потоками

```cpp
LocalMap ComputeLocalChunk(const CCSMatrix &matrix_left, const CCSMatrix &matrix_right) {
  size_t num_threads = std::thread::hardware_concurrency();
  std::vector<LocalMap> local_buffers(num_threads);
  std::vector<std::thread> threads(num_threads);
  size_t chunk = (left_count + num_threads - 1) / num_threads;

  for (size_t thread_idx = 0; thread_idx < num_threads; ++thread_idx) {
    size_t begin = thread_idx * chunk;
    size_t end = std::min(begin + chunk, left_count);
    threads[thread_idx] = std::thread([&, thread_idx, begin, end]() {
      ProcessChunk(begin, end, matrix_right, val_left, row_ind_left,
                   col_ptr_left, local_buffers[thread_idx]);
    });
  }

  for (auto &th : threads) th.join();

  LocalMap result;
  for (const auto &local : local_buffers) {
    for (const auto &[key, value] : local) result[key] += value;
  }
  return result;
}
```

### Иерархия параллелизма

```text
MPI ranks = P (число MPI-процессов)
std::thread::hardware_concurrency() = T (число потоков на процесс)

Общее число работников: P × T
```

## 6. Детали реализации

**Файлы:**

- `all/include/ops_all.hpp` — объявление класса.
- `all/src/ops_all.cpp` — реализация гибридного умножения.

**Структура пайплайна:**

- `ValidationImpl` — проверка совместимости размеров (только на процессе 0).
- `PreProcessingImpl` / `PostProcessingImpl` — без действий.
- `RunImpl` — broadcast B, scatter A, локальное умножение, gather результатов.

**Ключевые функции:**

- `BroadcastMatrix` — рассылка матрицы `B` через 5 `MPI_Bcast` (метаданные, строки, столбцы, re, im).
- `ScatterMatrixLeft` — распределение элементов `A` через `MPI_Send`/`MPI_Recv`.
- `ComputeLocalChunk` — локальное умножение через `std::thread` с `ProcessChunk`.
- `GatherResult` — сбор результатов через `MPI_Gather` + `MPI_Gatherv`, слияние на процессе 0.

### Потенциальные узкие места

1. **Коммуникационные накладные расходы:** `MPI_Bcast` и scatter/gather при 22000 элементов относительно легковесны.
2. **Сериализация на процессе 0:** распределение и сбор данных выполняются только процессом 0.
3. **Дублирование памяти:** каждый процесс хранит полную копию матрицы `B`.
4. **Накладные расходы STL:** создание/join потоков при каждом вызове `RunImpl`.

## 10. Выводы

1. **Ускорение 2.57×** при конфигурации 2×2 — умеренный результат из-за MPI overhead.
2. **Коммуникационные накладные расходы** (scatter, gather, broadcast) существенны при `nnz = 22000`.
3. **Иерархический подход** (MPI + STL) потенциально полезен для крупных задач или многоузловых систем.

**Когда гибридная схема оправдана:**

- `nnz(A)` в миллионы — накладные расходы MPI нивелируются объёмом вычислений.
- Запуск на нескольких узлах кластера — MPI обеспечивает межузловую коммуникацию, `std::thread` — внутриузловую.
- Задача не помещается в память одного узла.

**Когда менее эффективна:**

- Запуск на одном узле с малым объёмом данных — коммуникационные расходы доминируют.
- Простой алгоритм с равномерной нагрузкой: однопроцессный STL быстрее.
