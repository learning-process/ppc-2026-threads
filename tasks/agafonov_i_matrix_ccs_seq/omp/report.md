## 1. Контекст

В данном модуле осуществляется перенос последовательного алгоритма умножения разреженных матриц
(формат CCS) на многопоточную архитектуру с использованием технологии OpenMP. Перенос необходим
для ускорения обработки больших разреженных структур за счет распределения независимых итераций
цикла по доступным вычислительным ядрам процессора.

## 2. Постановка задачи

Необходимо параллельно вычислить произведение двух разреженных матриц A и B, находящихся в формате CCS,
с получением результирующей матрицы C = A * B. Последовательная версия из отчета `seq/report.md`
принимается за базовую линию (baseline) производительности. Главная задача — сократить время
выполнения основного вычислительного ядра, сохранив строгую корректность и детерминированность данных.

## 3. Базовый алгоритм

Последовательный алгоритм выполняет умножение на основе линейной комбинации столбцов. Он итерируется по
столбцам матрицы B, извлекает ненулевые элементы, перемножает их на элементы соответствующих столбцов
матрицы A и аккумулирует промежуточные суммы. В OpenMP-версии сохраняется эта же математическая логика,
однако итерации внешнего цикла по столбцам матрицы B распределяются между несколькими потоками.

## 4. Схема распараллеливания

Вычислительное ядро распараллелено в методе `RunImpl`. Областью параллелизации является внешний
цикл `for`, проходящий по индексам столбцов `j` правой матрицы B (от `0` до `b.cols_num`).

Управление потоками и область данных

Параллельный регион создается с помощью директивы:
`#pragma omp parallel num_threads(ppc::util::GetNumThreads()) default(none) ...`

Число создаваемых потоков динамически запрашивается из инфраструктуры каркаса с помощью вызова
`ppc::util::GetNumThreads()`, что обеспечивает гибкую переносимость кода.

Явное указание `default(none)` обязывает разработчика вручную определить атрибуты доступа для каждой
переменной, исключая появление неявных ошибок разделения памяти.

Shared-переменные: Матрицы A и B, а также контейнеры `local_vals` и `local_rows` объявлены как
`shared`, так как они совместно используются всеми потоками. Функция `std::ranges::sort` также передана
в список общих ресурсов для обеспечения видимости внутри региона.
Private-переменные (локальные потоковые буферы): Контейнеры `accumulator`, `active_rows` и `row_mask`
объявлены и аллоцированы непосредственно внутри параллельного блока. Это автоматически делает их
приватными (`private`) для каждого потока. Каждый рабочий поток получает свой независимый рабочий
аккумулятор размера строк матрицы A, что гарантирует полную потокобезопасность вычислений.

Синхронизация и планирование

Разделение работы: Директива `#pragma omp for` распределяет итерации цикла по столбцам между потоками.
Отсутствие гонок (Race Conditions): В коде не используются директивы `reduction`, `atomic` или
`critical` для сборки элементов. Потокобезопасность записи результатов достигнута за счет бесконфликтной
схемы: поток, обрабатывающий столбец `j`, записывает вычисленные значения в строго выделенные ячейки
`local_vals[j]` и `local_rows[j]`. Так как индексы итераций `j` уникальны и не пересекаются, потоки никогда
не пишут в одни и те же области памяти одновременно.
Выбранный Schedule: Использован планировщик по умолчанию (`static`), который делит итерации на
равные блоки. Для матриц с регулярной структурой это минимизирует накладные расходы на диспетчеризацию.
Барьеры синхронизации: В конце параллельного региона существует неявный (implicit) барьер. Он
гарантирует, что ни один поток не выйдет из блока, пока все столбцы не будут полностью вычислены. Это
необходимо для безопасного перехода к последующему последовательному слиянию локальных буферов в общую
структуру CCS-матрицы C.

## 5. Детали реализации

Исходный код расположен в файлах `omp/include/ops_omp.hpp` и `omp/src/ops_omp.cpp`. Изменения относительно
версии `SEQ` затронули метод `RunImpl`:

1. Введена промежуточная структура хранения: `local_vals` и `local_rows` представляют собой векторы
векторов. Это позволило потокам работать полностью независимо без блокировок.
2. Логика обработки одного столбца вынесена в изолированный метод `ProcessColumn` для повышения
читаемости кода и упрощения локализации приватных переменных потока.
3. Добавлен финальный последовательный шаг (после барьера), который собирает размеры векторов,
резервирует память (`reserve`) и копирует данные в итоговые массивы `c.vals` и `c.row_inds`.

## 6. Проверка корректности

Корректность параллельной логики полностью подтверждена прохождением набора функциональных тестов
`AgafonovIFuncTests` с фильтрацией по имени студента. Результаты вычислений OpenMP-версии побитово
совпадают с последовательным эталоном на всех граничных сценариях: умножение плотных матриц, умножение
на единичную матрицу (`Identity_Check`), матрицы с нулевым результатом (`Zero_Result`) и прямоугольные структуры.
