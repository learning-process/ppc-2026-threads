## 1. Контекст

Данная задача умножения разреженных матриц характеризуется неравномерным распределением ненулевых элементов по
столбцам. Технология Intel oneTBB выбрана из-за её задачно-ориентированной модели (Task-based parallelism). В отличие
от классических потоков, oneTBB абстрагирует вычисления в виде легких задач, управляемых встроенным планировщиком.
Это позволяет эффективно балансировать нагрузку (используя алгоритм Work Stealing) при обработке столбцов с разной
плотностью заполнения, минимизируя простои ядер процессора.

## 2. Постановка задачи

Необходимо реализовать эффективный параллельный алгоритм умножения двух разреженных матриц A и B, представленных в
формате столбцового хранения CCS (Compressed Column Storage). Результат операции должен быть записан в матрицу C,
также в формате CCS. В качестве эталона для проверки корректности и оценки метрик эффективности используется
последовательная версия алгоритма.

## 3. Базовый алгоритм

Последовательный алгоритм выполняет умножение по столбцам. Для каждого столбца j матрицы B определяются ненулевые
элементы. Для каждого такого элемента со значением v_b в строке k извлекается соответствующий столбец k матрицы A.
Его элементы перемножаются на v_b, и результаты аккумулируются в плотном векторе. После обработки всех ненулевых
элементов текущего столбца j, данные из аккумулятора переносятся в итоговые структуры CCS-матрицы C.

## 4. Схема распараллеливания

В качестве основного параллельного примитива выбран `tbb::parallel_for`. Он осуществляет итерации по индексам
столбцов результирующей матрицы в диапазоне от 0 до `b.cols_num`.

Диапазон работы определяется типом `tbb::blocked_range<size_t>`, который описывает одномерное итерационное
пространство. Планировщик oneTBB автоматически рекурсивно расщепляет этот диапазон на поддиапазоны (chunks).

Размер базового поддиапазона (grainsize) оставлен по умолчанию. Планировщик oneTBB автоматически определяет
оптимальный размер чанка на основе доступных вычислительных ресурсов и эвристик внутреннего балансировщика, чтобы
избежать избыточных накладных расходов при делении.

Использован автоматический разбиватель диапазона по умолчанию (`tbb::auto_partitioner`). Он динамически оценивает
загрузку потоков и прекращает разбиение диапазона, если все рабочие ядра заняты, снижая overhead.

Фактическая конкуренция контролировалась на уровне среды выполнения через задание переменной окружения
`PPC_NUM_THREADS=4`. На уровне планировщика oneTBB это значение ограничивает максимальный параллелизм
(max_allowed_parallelism) пула рабочих потоков-воркеров, распределяющих задачи из общей очереди.

## 5. Детали реализации

Файлы: tbb/include/ops_tbb.hpp, tbb/src/ops_tbb.cpp

Параллельная обработка организована внутри метода `RunImpl`. Главной особенностью реализации является полная изоляция
вычислительного контекста для предотвращения состояния гонки (data races) без использования блокирующих примитивов
синхронизации (mutex, atomic). Плотный рабочий вектор `accumulator`, вспомогательный массив индексов `active_rows` и
логическая маска `row_mask` аллоцируются локально внутри тела лямбда-выражения для каждого выделенного поддиапазона

Такой подход гарантирует, что каждый поток работает со своим собственным независимым набором буферов памяти, полностью
исключая разрушительный эффект ложного разделения кэш-линий (False Sharing) на уровне ядер процессора. Локальные
аккумуляторы сбрасываются в исходное состояние и повторно используются внутри цикла по поддиапазону, что минимизирует
количество повторных тяжелых системных вызовов выделения динамической памяти

Поскольку точный размер итогового столбца заранее неизвестен из-за специфики разреженных вычислений, промежуточные
результаты обработки каждого отдельного столбца j сохраняются во временные векторы `local_vals[j]` и `local_rows[j]`.
Сборка финальной CCS-структуры результирующей матрицы C выполняется последовательно на одном потоке после полного
завершения работы параллельного цикла `tbb::parallel_for`. Она заключается в последовательном расчете префиксных сумм
для вектора указателей столбцов `col_ptrs` и копировании данных из локальных потоковых буферов в плотные результирующие
векторы значений `vals` и индексов строк `row_inds`

## 6. Проверка корректности

Корректность работы TBB-модуля проверяется путем покомпонентного сравнения полученных результатов (значений,
индексов строк и указателей столбцов) с результатами, сгенерированными последовательным baseline-модулем SEQ.
Тестирование включает валидацию на случайных разреженных матрицах различных размеров, а также проверку граничных
случаев (пустые матрицы, полностью заполненные столбцы, несовпадающие размерности).

## 8. Результаты

Замеры производительности:
| Mode       | Workers (p) | Время работы (сек) | Ускорение | Эффективность|
| `pipeline` | 4           | 0.0014866500       | 1.4830    | 0.3707       |
| `task_run` | 4           | 0.0013139312       | 2.2847    | 0.5711       |

В режиме `task_run` параллельная версия алгоритма TBB показала устойчивое ускорение в 2.28 раза по сравнению с
последовательным baseline. Использование динамического планировщика и изоляция локальных аккумуляторов внутри
поддиапазонов позволили снизить накладные расходы на синхронизацию. Некоторое снижение эффективности в режиме
`pipeline` обусловлено затратами времени на создание инфраструктуры задач oneTBB на относительно малом объеме данных

Снижение эффективности в режиме pipeline (ускорение в 1.48 раза относительно baseline
0.0022047194сек) обусловлено издержками на развертывание инфраструктуры TBB.
На относительно малом объеме вычислиений время, затрачиваемое на создание внутренних структур данных
автоматического разбивателя диапазонов, вносит заметный вклад в общее время выполнения теста

## 9. Выводы

Технология Intel TBB демонстрирует высокую эффективность при работе со структурами данных переменной и сильно
разреженной плотности, какими и являются матрицы в формате CCS.
Задачный подход позволяет нивелировать дисбаланс вычислений, возникающий из-за разного количества ненулевых элементов
в столбцах. Однако при работе на структурах малого объема накладные
расходы runtime-компонентов TBB могут накладывать ограничения на максимальную
эффективность параллелизации, что делает технологию применимой более для масштабных вычислений
