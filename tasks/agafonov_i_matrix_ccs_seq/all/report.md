## 1. Контекст

При масштабировании вычислений на кластерные архитектуры или многопроцессорные узлы возможностей технологий с общей
памятью (OpenMP, oneTBB) становится недостаточно. Гибридная версия (ALL), объединяющая MPI и oneTBB, необходима для
реализации двухуровневого параллелизма. На макроуровне MPI обеспечивает распределение
крупных блоков данных между независимыми узлами сети. На микроуровне Intel oneTBB эффективно
утилизирует вычислительные ядра каждого отдельного процессора, балансируя локальные задачи

## 2. Постановка задачи

Требуется реализовать параллельный гибридный алгоритм умножения двух разреженных матриц A и B в формате столбцового
хранения CCS. Алгоритм должен сочетать межпроцессное взаимодействие MPI и внутрипроцессное распараллеливание oneTBB.
Итоговый результат операции умножения формируется в матрице C. В качестве baseline для оценки полученного ускорения
и эффективности используется базовая последовательная версия алгоритма (модуль SEQ)

## 3. Базовый алгоритм

Последовательная логика выполняет поэлементное умножение по столбцам. Для каждого столбца j матрицы B находятся все
активные ненулевые элементы. Значение каждого такого элемента умножается на соответствующий столбец матрицы A. Полученные
частичные произведения аккумулируются в плотном векторе-аккумуляторе фиксированной длины. После завершения сканирования
текущего столбца j ненулевые значения из аккумулятора переносятся в динамические векторы итоговой CCS-матрицы

## 4. Межпроцессная схема

На межпроцессном уровне используется топология SPMD. Все запущенные процессы определяют свой уникальный номер в рамках
глобального коммуникатора `MPI_COMM_WORLD` с помощью вызова `MPI_Comm_rank`, а общее число процессов — через `MPI_Comm_size`.

Распределение данных организовано по столбцовому принципу. Общее количество столбцов результирующей матрицы `b.cols_num`
делится на число доступных MPI-процессов. Каждый процесс вычисляет свои локальные индексы начала `col_start` и окончания
`col_end` обрабатываемого блока столбцов с учетом возможного остатка `rem` от деления.

Процесс с рангом 0 выполняет координирующую роль: собирает объемы вычисленных данных через `MPI_Gather`, рассчитывает
смещения `displs` в глобальных массивах и осуществляет итоговую сборку CCS-структуры в функции `AssembleFinalMatrix`.
Остальные процессы выступают в роли вычислительных узлов, обрабатывающих свои независимые полосы столбцов.

## 5. Внутрипроцессная схема

Внутри каждого отдельного MPI-процесса для распараллеливания вычислений по локальному подмножеству столбцов используется
технология Intel oneTBB. Внутри метода `RunImpl` вызывается параллельный примитив `tbb::parallel_for`

Диапазон итераций `tbb::blocked_range<size_t>` настраивается на границы локального блока процесса от `col_start` до
`col_end`. Планировщик TBB динамически разбивает этот диапазон на мелкие задачи и распределяет их по доступным ядрам

Объединение результатов внутри процесса происходит без гонки данных за счет изоляции памяти. Векторы `accumulator`,
`active_rows` и `row_mask` аллоцируются строго внутри лямбда-функции TBB-потока. Промежуточные результаты каждого
столбца j записываются в независимые ячейки общих массивов векторов `local_vals[j]` и `local_rows[j]`

## 6. Детали реализации

Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp

Для синхронизации этапов и сбора данных используются следующие вызовы стандарта MPI:

- `MPI_Reduce` с операцией `MPI_SUM` собирает на корневом ранге информацию о количестве ненулевых элементов в каждом
столбце (`all_col_nnz`), суммируя локальные маски процессов
- `MPI_Gather` собирает на нулевом ранге размеры локальных векторов значений `my_count` от всех процессов для расчета
массива смещений `displs`
- `MPI_Gatherv` осуществляет непосредственный сбор разнородных по размеру локальных массивов значений `my_vals` и
индексов строк `my_rind` в единые плотные векторы нулевого процесса `all_vals` и `all_rind`
- Серия вызовов `MPI_Bcast` транслирует собранную и упакованную на нулевом ранге результирующую матрицу C обратно всем
остальным процессам для обеспечения идентичности состояния объектов на выходе из метода `RunImpl`

Узким местом реализации являются последовательные вызовы `MPI_Gatherv` и `MPI_Bcast`, скорость выполнения которых
напрямую ограничена пропускной способностью сети или внутрисистемной шины передачи данных при межпроцессном обмене

## 7. Проверка корректности

Корректность гибридного алгоритма проверяется автоматизированными тестами. Результаты вычислений (массивы значений,
индексов и указателей) покомпонентно сравниваются с результатами последовательного baseline модуля SEQ. Тестирование
проводится на случайных разреженных матрицах малых и средних размеров. Важной частью контроля является проверка на
согласованность данных — валидация подтверждает, что после финальных широковещательных рассылок все процессы имеют
абсолютно идентичные копии результирующей матрицы C.

## 10. Выводы

Гибридная схема параллелизма MPI+TBB полностью оправдана при обработке сверхбольших разреженных матриц, объем данных
которых превышает емкость оперативной памяти одного вычислительного узла. Иерархический подход позволяет масштабировать
задачу на кластерные системы. Однако на задачах малого и среднего объема (матрицы 1000x1000) накладные
расходы на сетевые обмены MPI_Gatherv, MPI_Bcast и синхронизацию процессов могут перекрывать выигрыш от
распараллеливания, делая гибридную схему менее эффективной, чем чистые внутрипроцессные решения
