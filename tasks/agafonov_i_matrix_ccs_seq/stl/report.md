## 1. Контекст

Использование стандартной библиотеки потоков C++ (`std::thread`) представляет собой наиболее низкоуровневый подход к
внутрипроцессному распараллеливанию. В отличие от высокоуровневых абстракций (OpenMP, TBB), скрывающих механизмы
планирования, ручная реализация требует от разработчика самостоятельного проектирования стратегии разделения данных,
управления жизненным циклом потоков и обеспечения потокобезопасности. Данная версия необходима для понимания базовых
инженерных принципов конкурентного программирования, накладных расходов ОС и причин возникновения дисбаланса нагрузки.

## 2. Постановка задачи

Требуется реализовать параллельный алгоритм умножения разреженных матриц A и B (формат CCS) с использованием
исключительно средств `<thread>` стандарта C++11 и выше. Результат помещается в матрицу C (CCS). Эффективность,
выраженная через метрики ускорения и масштабируемости, должна оцениваться путем сравнения с последовательным модулем
SEQ, выступающим в роли эталонного baseline-решения.

## 3. Базовый алгоритм

Последовательная версия алгоритма итеративно обрабатывает столбцы матрицы B. Для ненулевых элементов каждого столбца
j извлекаются соответствующие столбцы матрицы A, умножаются на значение элемента из B, а частичные суммы накапливаются
в плотном векторе. По завершении обхода столбца j, накопленные данные переносятся в структуру результирующей матрицы.

## 4. Схема распараллеливания

Распараллеливание построено на ручном разбиении одномерного итерационного пространства (индексов столбцов). Алгоритм
запрашивает количество доступных потоков через `ppc::util::GetNumThreads()`, с фоллбэком на аппаратный параллелизм
`std::thread::hardware_concurrency()`. Общее число столбцов матрицы B делится на количество потоков, формируя базовый
размер блока (`chunk_size`) и остаток (`remainder`), который равномерно распределяется среди первых потоков

Каждый поток создается через вызов `threads.emplace_back()`, куда передается лямбда-выражение, захватывающее по ссылке
общие структуры, а по значению — предварительно вычисленные границы локального диапазона `[current_start, current_end)`.
Запуск потока операционной системой происходит немедленно в момент вызова конструктора `std::thread`

Для обеспечения полной потокобезопасности (Thread Safety) без использования блокировок реализован Lock-Free подход.
Отсутствуют такие примитивы синхронизации как `std::mutex`, `std::atomic` или `std::condition_variable`. Вместо этого
используется жесткая изоляция записи: предварительно выделяются общие векторы векторов `local_vals` и `local_rows`
размером `b.cols_num`. Каждый поток гарантированно пишет данные только в те индексы внешнего вектора `j`, которые
строго принадлежат его локальному диапазону

## 5. Детали реализации

Файлы: tasks/agafonov_i_matrix_ccs_seq/stl/include/ops_stl.hpp, tasks/agafonov_i_matrix_ccs_seq/stl/src/ops_stl.cpp

Критически важным аспектом реализации является разделение фаз создания и ожидания. В коде реализован цикл создания
потоков `for (int i = 0; i < num_threads; ++i)`, внутри которого вычисляются границы и происходит `emplace_back`.
Только после того, как все потоки созданы и добавлены в вектор `std::vector<std::thread>`, запускается второй,
абсолютно независимый цикл `for (auto &t : threads)`, в котором для каждого объекта потока вызывается блокирующий
метод `t.join()`. Данная архитектура исключает фатальную ошибку сериализации вычислений, при которой главный поток
дожидался бы завершения созданного воркера перед созданием следующего

Внутри потока (в лямбда-выражении) каждый воркер локально аллоцирует тяжелые временные буферы (`accumulator`,
`active_rows`, `row_mask`). Это предотвращает состояние гонки по данным кэш-линий (False Sharing). Финальная сборка
CCS-матрицы происходит в главном потоке строго после барьера синхронизации — завершения всех вызовов `join()`

## 6. Проверка корректности

Корректность алгоритма STL верифицируется через функциональные тесты Google Test путем полного покомпонентного сравнения
результатов с baseline SEQ-модулем. Учитываются крайние случаи: обработка пустых матриц, матриц с нулевыми строками,
а также ситуации, когда запрошенное количество потоков превышает общее число столбцов результирующей матрицы (в этом
случае количество создаваемых потоков принудительно обрезается до размера задачи во избежание создания пустых воркеров)

## 9. Выводы

Реализация на базе std::thread наглядно демонстрирует слабость статического распределения  при
неравномерной вычислительной нагрузке. Так как ненулевые элементы распределены по столбцам хаотично, одни потоки
завершают свои current_start, current_end диапазоны мгновенно, а другие продолжают работать с плотными столбцами

Из-за отсутствия встроенного механизма кражи работы (Work Stealing), который присутствует в TBB, быстрые потоки
блокируются на этапе join(), простаивая в ожидании самых медленных воркеров. Кроме того, на малых размерах задач
ощущается высокий системный оверхед на системные вызовы создания (clone()) и уничтожения нативных потоков ОС
