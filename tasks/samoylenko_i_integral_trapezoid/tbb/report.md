## 1. Контекст

В этом отчёте описана параллельная версия алгоритма на oneTBB. Из последовательной версии в oneTBB переносится основной
цикл по точкам сетки: его итерации распределяются между потоками. Все остальные части алгоритма (валидация, выбор
функции, итоговая формула) не меняются.

## 2. Постановка задачи

Постановка задачи, входные и выходные данные, ограничения - те же, что в последовательной версии.

## 3. Базовый алгоритм

Область интегрирования разбивается на сетку с шагом $h_i$ по каждому измерению; все точки сетки нумеруются одним числом
и перебираются одним циклом. В каждой точке вычисляется значение функции с весом 1 или 2 по каждому измерению, затем
взвешенная сумма умножается на общий множитель $h_1 \cdots h_d / 2^d$.

## 4. Схема распараллеливания

Используется `tbb::parallel_reduce` - он одновременно делит работу между потоками и выполняет редукцию результатов.

Основной фрагмент `RunImpl` (`tbb/src/ops_tbb.cpp`):

```cpp
double sum = tbb::parallel_reduce(
    tbb::blocked_range<int64_t>(0, points), 0.0,
    [&h, &dimensions, &dim_sizes, &in, &integral_function]
    (const tbb::blocked_range<int64_t> &range, double local_sum) -> double {
      std::vector<double> current_point(dimensions);
      for (int64_t pnt = range.begin(); pnt != range.end(); ++pnt) {
        int64_t rem_index = pnt;
        int weight = 1;
        for (int dim = 0; dim < dimensions; dim++) {
          int dim_coord = static_cast<int>(rem_index % dim_sizes[dim]);
          rem_index /= dim_sizes[dim];
          current_point[dim] = in.a[dim] + (dim_coord * h[dim]);
          if (dim_coord > 0 && dim_coord < in.n[dim]) {
            weight *= 2;
          }
        }
        local_sum += integral_function(current_point) * weight;
      }
      return local_sum;
    },
    [](double a, double b) { return a + b; });
```

Так как сигнатура вызываемого объекта в parallel_reduce фиксирована, в этой реализации используются лямбда-функции с
захватами для упрощения передачи данных внутрь контекста.

- `tbb::blocked_range<int64_t>(0, points)` - диапазон всех точек сетки. TBB сам делит его на блоки и раздаёт блоки
  потокам.
- Первая лямбда - тело вычисления: она получает блок `range` и стартовое значение `local_sum`, проходит точки блока и
  возвращает локальную сумму.
- Вторая лямбда - функция редукции: складывает локальные суммы двух блоков.
- Число рабочих потоков TBB задаётся через переменную `PPC_NUM_THREADS`.
- Вектор координат `current_point` инициализируется один раз для каждого диапазона.
- Размер блока (`grainsize`) у `blocked_range` явно не задан, поэтому он равен 1, а фактический размер блоков подбирает
  `auto_partitioner`. Он меняет размер блоков так, чтобы накладные расходы
  на планирование оставались малыми, и при этом сохранялось достаточно блоков для балансировки нагрузки между потоками.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Относительно последовательной версии изменён только метод `RunImpl`: цикл по точкам заменён вызовом `parallel_reduce`.
Методы `ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` и функция `GetIntegrationFunction` совпадают с
версией SEQ.

`local_sum` - это аргумент первой лямбды, у каждого блока он свой. Объединение результатов выполняет функция редукции.
Общих изменяемых данных нет, поэтому гонок нет и синхронизация не требуется.

## 6. Проверка корректности

Корректность версии TBB проверялась сравнением с SEQ на всех функциональных тестах. Расхождений в результатах не
наблюдалось. Дополнительно проверена устойчивость результата при числе потоков от 1 до 6. Итог не зависит от числа
потоков.

## 7. Экспериментальная среда

Аппаратное обеспечение и ОС:

- CPU: AMD Ryzen 5 2600X
- Ядра и потоки: 3 физических ядра, 6 логических ядер
- OS: Ubuntu 24.04.2 LTS

Набор инструментов:

- Компилятор: g++ 13.3.0
- Тип сборки: Release
- CMake 3.28.3

Число потоков задаётся переменной окружения `PPC_NUM_THREADS`

Команды сборки и запуска:

```bash
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build -j1
PPC_NUM_THREADS=4 ./build/bin/ppc_func_tests --gtest_filter='*_tbb_*'
PPC_NUM_THREADS=4 ./build/bin/ppc_perf_tests --gtest_filter='*_tbb_*'
```

Данные для теста производительности: трёхмерный интеграл от суммы квадратов на области $[0,1]$ в трёх измерениях с
разбиением 421 по каждому измерению (~75 000 000 точек).

## 8. Результаты

| Потоки | Время, с | Ускорение | Эффективность |
| --- | --- | --- | --- |
| 1 | 1.058 | 1.00 | 100% |
| 2 | 0.572 | 1.84 | 92% |
| 3 | 0.407 | 2.59 | 86% |
| 4 | 0.319 | 3.30 | 83% |
| 5 | 0.293 | 3.59 | 72% |
| 6 | 0.247 | 4.26 | 71% |

Ускорение = Время SEQ / Время TBB, где время SEQ = 1.053 с.
Эффективность = Ускорение / Потоки.

На одном потоке версия TBB практически совпадает с SEQ. С ростом числа потоков время монотонно уменьшается.
Эффективность снижается так же, как у OMP и STL (и по тем же причинам), но TBB удержал её чуть выше: на 6 потоках TBB
дал лучшее время среди других версий.

## 9. Выводы

oneTBB даёт ускорение того же порядка, что и OpenMP за счёт `parallel_reduce`, который выполняет деление работы,
суммирование и подбор размера блоков. Дальнейший рост числа потоков ограничен числом ядер машины.
