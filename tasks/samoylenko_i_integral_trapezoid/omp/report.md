## 1. Контекст

В этом отчёте описана параллельная версия алгоритма на OpenMP. Из последовательной версии в OpenMP переносится основной
цикл по точкам сетки: его итерации распределяются между потоками. Все остальные части алгоритма (валидация, выбор
функции, итоговая формула) не меняются.

## 2. Постановка задачи

Постановка задачи, входные и выходные данные, ограничения - те же, что в последовательной версии.

## 3. Базовый алгоритм

Область интегрирования разбивается на сетку с шагом $h_i$ по каждому измерению; все точки сетки нумеруются одним числом
и перебираются одним циклом. В каждой точке вычисляется значение функции с весом 1 или 2 по каждому измерению, затем
взвешенная сумма умножается на общий множитель $h_1 \cdots h_d / 2^d$.

## 4. Схема распараллеливания

Распараллеливается главный цикл по точкам сетки. Все итерации цикла независимы (так как каждая точка обрабатывается
отдельно), поэтому цикл легко делится между потоками с помощью `#pragma omp parallel for`.

Основной фрагмент `RunImpl` (`omp/src/ops_omp.cpp`):

```cpp
#pragma omp parallel default(none) \
    shared(dimensions, points, dim_sizes, h, in, integral_function) \
    reduction(+ : sum) num_threads(ppc::util::GetNumThreads())
{
  std::vector<double> current_point_local(dimensions);

#pragma omp for
  for (int64_t pnt = 0; pnt < points; pnt++) {
    int64_t rem_index = pnt;
    int weight = 1;

    for (int dim = 0; dim < dimensions; dim++) {
      int dim_coord = static_cast<int>(rem_index % dim_sizes[dim]);
      rem_index /= dim_sizes[dim];

      current_point_local[dim] = in.a[dim] + (dim_coord * h[dim]);

      if (dim_coord > 0 && dim_coord < in.n[dim]) {
        weight *= 2;
      }
    }

    sum += integral_function(current_point_local) * weight;
  }
}
```

- `default(none)` - явно указывает класс каждой переменной.
- `shared(...)` - переменные `dimensions`, `points`, `dim_sizes`, `h`, `in`, `integral_function` общие для всех потоков.
  Используются только для чтения.
- `reduction(+ : sum)` - у каждого потока есть своя локальная копия `sum`, в которую он складывает свою часть точек. В
  конце параллельной области все локальные копии складываются в одну.
- `num_threads(ppc::util::GetNumThreads())` - число потоков берётся из переменной окружения `PPC_NUM_THREADS`.
- Вектор `current_point_local` объявлен внутри параллельной области, поэтому у каждого потока он свой (для итераций по
  своему диапазону точек). Он создаётся один раз вне цикла по точкам, чтобы не выделять память на каждой итерации.
- `schedule` явно не указан, поэтому `#pragma omp for` использует планирование
  по умолчанию (итерации делятся на равные блоки между потоками). Для этой задачи такой выбор подходит, так как все итерации
  примерно одинаковы по стоимости.
- В конце `#pragma omp for` и в конце параллельной области есть неявный барьер: потоки дожидаются друг друга, поэтому
  редукция `sum` гарантированно завершается до выхода из области.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Относительно последовательной версии изменён только метод `RunImpl`: основной цикл обёрнут в параллельную область.
Методы `ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` и функция `GetIntegrationFunction` совпадают с
версией SEQ.

Единственная переменная, в которую пишут все потоки, - `sum`. Редукция устраняет гонку записи. Вектор
`current_point_local` у каждого потока свой. Других общих изменяемых данных нет.

## 6. Проверка корректности

Корректность версии OMP проверялась сравнением с SEQ на всех функциональных тестах. Расхождений в результатах не
наблюдалось. Дополнительно проверена устойчивость результата при числе потоков от 1 до 6. Итог не зависит от числа
потоков.

## 7. Экспериментальная среда

Аппаратное обеспечение и ОС:

- CPU: AMD Ryzen 5 2600X
- Ядра и потоки: 3 физических ядра, 6 логических ядер
- OS: Ubuntu 24.04.2 LTS

Набор инструментов:

- Компилятор: g++ 13.3.0
- Тип сборки: Release
- CMake 3.28.3

Число потоков задаётся переменной окружения `PPC_NUM_THREADS`

Команды сборки и запуска:

```bash
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build -j1
PPC_NUM_THREADS=4 ./build/bin/ppc_func_tests --gtest_filter='*_omp_*'
PPC_NUM_THREADS=4 ./build/bin/ppc_perf_tests --gtest_filter='*_omp_*'
```

Данные для теста производительности: трёхмерный интеграл от суммы квадратов на области $[0,1]$ в трёх измерениях с
разбиением 421 по каждому измерению (~75 000 000 точек).

## 8. Результаты

| Потоки | Время, с | Ускорение | Эффективность |
| --- | --- | --- | --- |
| 1 | 1.105 | 0.95 | 95% |
| 2 | 0.552 | 1.91 | 95% |
| 3 | 0.392 | 2.69 | 90% |
| 4 | 0.340 | 3.10 | 77% |
| 5 | 0.277 | 3.80 | 76% |
| 6 | 0.284 | 3.71 | 62% |

Ускорение = Время SEQ / Время OMP, где время SEQ = 1.053 с.
Эффективность = Ускорение / Потоки.

На одном потоке версия OMP практически совпадает с SEQ. До 2-3 потоков ускорение близко к линейному. С 4 потоков
эффективность заметно падает из-за лимита ядер. На 6 потоках время даже чуть хуже, чем на 5.

## 9. Выводы

OpenMP даёт почти линейное ускорение, пока число потоков не превышает число физических ядер. Реализация очень
компактная: одна параллельная область с редукцией поверх последовательного кода. Дальнейший рост числа потоков ограничен
числом ядер машины.
