## 1. Контекст

В этом отчёте описана гибридная версия алгоритма на MPI + OpenMP. MPI распределяет работу между процессами, а OpenMP -
между потоками внутри каждого процесса.

## 2. Постановка задачи

Постановка задачи, входные и выходные данные, ограничения - те же, что в последовательной версии.

## 3. Базовый алгоритм

Область интегрирования разбивается на сетку с шагом $h_i$ по каждому измерению; все точки сетки нумеруются одним числом
и перебираются одним циклом. В каждой точке вычисляется значение функции с весом 1 или 2 по каждому измерению, затем
взвешенная сумма умножается на общий множитель $h_1 \cdots h_d / 2^d$.

## 4. Межпроцессная схема

Список всех точек сетки делится между процессами MPI. Каждый процесс знает свой ранг и общее число процессов через
`MPI_Comm_rank` и `MPI_Comm_size`, после чего по формуле вычисляет свой диапазон точек.

Отдельной рассылки массива данных между процессами нет. Все процессы получают одинаковые входные данные и независимо
строят у себя одни и те же шаги $h_i$ и число точек. Различается только диапазон, который каждый процесс считает сам из
своего ранга. Поэтому коммуникация нужна только в конце.

Когда каждый процесс считает сумму своего диапазона, частичные суммы всех процессов объединяются в общую сумму вызовом
`MPI_Allreduce` с операцией `MPI_SUM`.

## 5. Внутрипроцессная схема

Диапазон точек процесса дальше делится между потоками OpenMP - точно так же, как процессы делили общий список. Каждый
поток получает свой поддиапазон точек.

Основной фрагмент `RunImpl` (`all/src/ops_all.cpp`):

```cpp
int rank = 0;
int size = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

int64_t start = (points * rank) / size;
int64_t end = (points * (rank + 1)) / size;

const int num_threads = ppc::util::GetNumThreads();
std::vector<double> local_sums(num_threads, 0.0);

#pragma omp parallel for num_threads(num_threads) default(none) \
    shared(num_threads, start, end, dimensions, dim_sizes, h, in, integral_function, local_sums)
for (int thr = 0; thr < num_threads; thr++) {
  int64_t local_start = start + (((end - start) * thr) / num_threads);
  int64_t local_end = start + (((end - start) * (thr + 1)) / num_threads);
  local_sums[thr] = GetLocalSum(local_start, local_end, dimensions, dim_sizes, h, in, integral_function);
}

double sum = 0.0;
for (int thr = 0; thr < num_threads; thr++) {
  sum += local_sums[thr];
}

double total_sum = 0.0;
MPI_Allreduce(&sum, &total_sum, 1, MPI_DOUBLE, MPI_SUM, MPI_COMM_WORLD);
```

Распределение поддиапазонов по потокам выполняется через `#pragma omp parallel for`, число потоков задаётся через
`num_threads(ppc::util::GetNumThreads())`. Каждый поток считает сумму своего поддиапазона функцией `GetLocalSum` и
записывает её в массив `local_sums` по своему индексу. После параллельной области суммы потоков складываются в сумму
процесса, и только потом выполняется `MPI_Allreduce`.

В итоге получается двухуровневое разбиение: общий список $[0, points)$ делится между процессами на диапазоны
$[start, end)$, а каждый такой диапазон делится между потоками на поддиапазоны.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`.

Важное отличие этой версии - оптимизация вычисления координат.

Это вынесено в функцию `GetLocalSum` (`all/src/ops_all.cpp`):

```cpp
int64_t rem_index = start;
for (int dim = 0; dim < dimensions; dim++) {
  coords[dim] = static_cast<int>(rem_index % dim_sizes[dim]);
  rem_index /= dim_sizes[dim];
}

for (int64_t pnt = start; pnt < end; pnt++) {
  int weight = 1;
  for (int dim = 0; dim < dimensions; dim++) {
    current_point[dim] = in.a[dim] + (coords[dim] * h[dim]);
    if (coords[dim] > 0 && coords[dim] < in.n[dim]) {
      weight *= 2;
    }
  }
  local_sum += integral_function(current_point) * weight;

  for (int dim = 0; dim < dimensions; dim++) {
    coords[dim]++;
    if (coords[dim] < dim_sizes[dim]) {
      break;
    }
    coords[dim] = 0;
  }
}
```

Сначала по формуле деления один раз вычисляются координаты `coords` первой точки блока. Внутри цикла по точкам по
`coords` строится точка и её вес, а в конце итерации координаты сдвигаются на следующую точку: координата первого
измерения увеличивается на 1. Если мы достигли конца оси - обнуляем эту координату и увеличиваем на 1 следующую ось. Это
работает с любым количеством измерений.

Смысл в том, что деление и взятие остатка для ~75 млн точек сказывается на общем времени работы алгоритма. В ALL процесс
конвертации порядкового числа в координаты выполняется только для начальной точки диапазона. Этот подход сильно упрощает
вычислительную нагрузку, что можно увидеть в таблице.

Главный вычислительный алгоритм вынесен в отдельную функцию `GetLocalSum` - это снижает вложенность циклов. Методы
`ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` и функция `GetIntegrationFunction` совпадают с версией SEQ.

Из операций MPI используется только `MPI_Allreduce` - один раз в конце расчёта. `MPI_Barrier`, `MPI_Bcast`,
`MPI_Scatter` и `MPI_Gather` не нужны, так как все процессы получают одинаковые входные данные и считают свой диапазон
независимо. В этом решении межпроцессная коммуникация минимальна (то есть буквально одно число).

## 7. Проверка корректности

Корректность версии ALL проверялась сравнением с SEQ на всех функциональных тестах. Расхождений в результатах не
наблюдалось. Тесты выполнялись при 1 и 2-х процессах и 1-6 потоках. Итог не зависит от числа потоков и процессов (и любой их комбинации).

Команды сборки и запуска:

```bash

cmake --build build -j1
PPC_NUM_THREADS=2 mpirun --oversubscribe --bind-to none -n 2 ./build/bin/ppc_func_tests --gtest_filter='*_all_*'
PPC_NUM_THREADS=2 mpirun --oversubscribe --bind-to none -n 2 ./build/bin/ppc_perf_tests --gtest_filter='*_all_*'
```

Данные для теста производительности: трёхмерный интеграл от суммы квадратов на области $[0,1]$ в трёх измерениях с
разбиением 421 по каждому измерению (~75 000 000 точек).

## 10. Выводы

Получившаяся схема MPI + OpenMP позволяет занять работой все доступные ядра, комбинируя процессы и потоки. Стоимость
коммуникации мала - за весь расчёт выполняется один `MPI_Allreduce` на одно число. Дополнительный выигрыш даёт
оптимизация координат: она убирает деление из основного цикла и делает алгоритм быстрее во всех конфигурациях. Это даёт
лучшие показатели среди всех реализаций.
