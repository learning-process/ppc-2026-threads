## 1. Контекст

В этом отчёте описана параллельная версия алгоритма на `std::thread`. В отличие от OMP и TBB, здесь разбиение работы
между потоками и сбор результата выполнены вручную, без готовых примитивов.

## 2. Постановка задачи

Постановка задачи, входные и выходные данные, ограничения - те же, что в последовательной версии.

## 3. Базовый алгоритм

Область интегрирования разбивается на сетку с шагом $h_i$ по каждому измерению; все точки сетки нумеруются одним числом
и перебираются одним циклом. В каждой точке вычисляется значение функции с весом 1 или 2 по каждому измерению, затем
взвешенная сумма умножается на общий множитель $h_1 \cdots h_d / 2^d$.

## 4. Схема распараллеливания

Список точек вручную делится на блоки по числу потоков. Поток с номером `thr` получает блок
$[points \cdot thr / N,\; points \cdot (thr+1) / N)$, где $N$ - число потоков. Такая формула покрывает весь список без
пропусков и пересечений.

Основной фрагмент `RunImpl` (`stl/src/ops_stl.cpp`):

```cpp
const int num_threads = ppc::util::GetNumThreads();
std::vector<double> local_sums(num_threads, 0.0);
std::vector<std::thread> threads(num_threads);

for (int thr = 0; thr < num_threads; thr++) {
  int64_t start = (points * thr) / num_threads;
  int64_t end = (points * (thr + 1)) / num_threads;

  threads[thr] = std::thread([&h, &dimensions, &dim_sizes, &in, &integral_function, thr, start, end, &local_sums]() {
    local_sums[thr] = GetLocalSum(start, end, dimensions, dim_sizes, h, in, integral_function);
  });
}

for (auto &thread : threads) {
  thread.join();
}

double sum = 0.0;
for (int thr = 0; thr < num_threads; thr++) {
  sum += local_sums[thr];
}
```

- Число потоков берётся из переменной окружения `PPC_NUM_THREADS` через `ppc::util::GetNumThreads()`.
- Каждый поток считает сумму своего блока функцией `GetLocalSum` и записывает её в массив `local_sums` по своему индексу
  `thr`.
- Данные передаются в поток через захваты лямбда-функции: общие данные захватываются по ссылке, а номер потока и границы
  блока - по значению (у каждого потока они свои).
- `join` вызывается во втором цикле - после того как все потоки уже созданы в первом цикле. Если бы `join` стоял внутри
  цикла создания, потоки выполнялись бы по очереди (что буквально противоречит заданию).
- После того как все потоки завершены, главный поток складывает значения `local_sums` в итоговую сумму.

Главный вычислительный алгоритм вынесен в отдельную функцию `GetLocalSum` - это снижает вложенность циклов и упрощает
структуру решения.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

Относительно последовательной версии изменён только метод `RunImpl`: цикл по точкам заменён ручным разбиением на блоки и
запуском `std::thread`. Методы `ValidationImpl`, `PreProcessingImpl`, `PostProcessingImpl` и функция
`GetIntegrationFunction` совпадают с версией SEQ.

Каждый поток пишет только в свою ячейку `local_sums[thr]`, а читает только свой блок точек. Объединение результатов
выполняет главный поток уже после `join`. Общих изменяемых данных нет, поэтому гонок нет и синхронизация не требуется.

## 6. Проверка корректности

Корректность версии STL проверялась сравнением с SEQ на всех функциональных тестах. Расхождений в результатах не
наблюдалось. Дополнительно проверена устойчивость результата при числе потоков от 1 до 6. Итог не зависит от числа
потоков.

Эффективность = Ускорение / Потоки.

На одном потоке версия STL практически совпадает с SEQ. До 2-3 потоков ускорение близко к линейному, с 4 потоков
эффективность падает (та же причина, что и у других реализаций). Накладные расходы на создание и `join` потоков малы -
потоки создаются один раз за запуск.

## 9. Выводы

Ручная реализация на `std::thread` даёт ускорение того же порядка, что OMP и TBB. Так как блоки точек не пересекаются,
синхронизация между потоками не нужна - достаточно записать результат каждого потока в свою ячейку и сложить их
после `join`. Цена ручного подхода, как ни странно, в том, что он ручной. Нужно самому считать границы блоков, следить
за правильностью слияния потоков и конструировать функцию редукции.
