## 1. Контекст

Ручная STL-threading версия реализована для контроля над параллелизмом без использования OMP или TBB.

Цель - продемонстрировать управление потоками вручную;
Особенность:

- используется явное создание потоков;
- балансировка задач реализована через atomic counter.

## 2. Постановка задачи

Необходимо:

- бинаризовать изображение;
- найти компоненты связности;
- построить выпуклую оболочку для каждой компоненты.

В качестве baseline используется последовательная реализация (SEQ) - 0.0645536977 с.

## 3. Базовый алгоритм

Последовательная версия включает:

- Thresholding - бинаризация изображения;
- DFS/BFS (stack-based) - поиск компонент связности;
- Convex Hull - построение оболочки.

Особенности:

- каждая компонента независима;
- BFS остаётся последовательным bottleneck;
- convex hull хорошо параллелится.

## 4. Схема распараллеливания

1. Подход
Используется модель thread pool без пула:

- создаются N потоков вручную;
- задачи распределяются динамически через `atomic<size_t>`.

2. Разделение данных

- `components` - общий массив входных задач;
- каждый поток получает индекс компоненты через: `current_index.fetch_add(1)`.

3. Каждый поток:

- берёт индекс компоненты;
- строит convex hull;
- записывает результат в `hulls[idx]`.

4. Локальные данные

- локальные переменные внутри потоков;
- нет shared mutable state кроме: `atomic current_index`
- `hulls[]` (разные индексы - без гонок).

5. Join-синхронизация

- используется thread.join() после запуска всех потоков;
- это обеспечивает глобальный барьер завершения.

6. Синхронизация

- `std::atomic<size_t>` с `memory_order_relaxed`;
- отсутствуют `mutex` или `critical sections`;
- гонок нет из-за уникальных индексов записи.

## 5. Детали реализации

1. Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

2. Карта поток → диапазон
В данной реализации отсутствует статическое разбиение данных на фиксированные диапазоны.

Вместо этого используется динамическое распределение задач:

- Все компоненты связности хранятся в общем массиве components
- Потоки создаются через `std::thread`
- Каждому потоку назначается единый общий счётчик задач - `std::atomic<size_t> current_index{0}`;
- Каждый поток выполняет следующий цикл - `size_t idx = current_index.fetch_add(1)`;

Таким образом:

- каждый поток получает следующую доступную компоненту
- диапазоны формируются динамически и заранее не фиксируются
- балансировка нагрузки происходит автоматически

3. Схема объединения локальных результатов

- Потоки стартуют одновременно
- Берут компоненту через `atomic fetch_add`
- Считают выпуклую оболочку локально
- Записывают результат прямо в `hulls[idx]`
- Завершаются после исчерпания индексов
- `join()` только для синхронизации завершения потоков

## 6. Проверка корректности

Проверено:

- совпадение результатов с SEQ;
- корректная работа на:

1) пустых изображениях;
2) одиночных компонентах;
3) больших изображениях;

- отсутствие race conditions:

1) проверено логически (unique index writing);
2) atomic защищает только распределение задач.

## 8. Результаты

baseline - 0.0645536977.

Таблица STL-замеров:

| Threads | Time, s      | Speedup | Efficiency |
| ------: | -----------: | ------: | ---------: |
| 1       | 0.0723875600 | 0.89    | 0.892      |
| 2       | 0.0646669800 | 0.99    | 0.499      |
| 4       | 0.0578025600 | 1.11    | 0.279      |
| 8       | 0.0739120000 | 0.87    | 0.109      |
| 12      | 0.0811234000 | 0.79    | 0.066      |

- лучший результат: 4 потока (~1.12× speedup)
- при 1 потоке уже есть overhead std::thread (хуже SEQ)
- при >4 потоках деградация из-за:

1) cost создания/уничтожения потоков;
2) contention на atomic counter;
3) cache contention при доступе к shared arrays;

- STL показывает худшую масштабируемость по сравнению с TBB.

## 9. Выводы

Ручная STL-threading реализация показывает:

1) Плюсы:

- полный контроль над потоками;
- простая модель распределения задач;
- отсутствие внешних зависимостей.

2) Минусы:

- высокий overhead создания потоков;
- нет эффективного scheduling (как в TBB);
- плохая масштабируемость на большом числе потоков.
