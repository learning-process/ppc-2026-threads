## 1. Контекст

OpenMP-версия направлена:
на распараллеливание независимых этапов обработки - построения выпуклых оболочек для компонент.
Поиск компонент связности остаётся последовательным из-за глобального состояния `visited`.

## 2. Постановка задачи

На вход подается бинарное изображение `(BinaryImage)`, содержащее:

- размеры изображения (width, height);
- массив пикселей (pixels);
- выходные структуры (components, convex_hulls).

Требуется:

- Найти компоненты связности (4-связность);
- Для каждой компоненты построить выпуклую оболочку;
- Сохранить результат в convex_hulls.

Ограничения на вход:

`width > 0, height > 0`;
`pixels.size() == width * height`.

Корректный результат:

- компоненты найдены корректно;
- оболочки построены корректно;
- результат совпадает с SEQ-версией.

## 3. Базовый алгоритм

Алгоритм совпадает с последовательной версией:

- Бинаризация изображения (threshold = 128);
- Поиск компонент связности (BFS);
- Построение выпуклой оболочки (алгоритм Эндрю).

Построение оболочек для разных компонент - независимые задачи, что позволяет распараллелить данный этап.
Однако на практике эффективность ограничена тем,
что поиск компонент связности (BFS) остаётся последовательным и занимает значительную долю времени.

## 4. Схема распараллеливания

Распараллеливается этап построения выпуклых оболочек:

```cpp
#pragma omp parallel for default(none) shared(components, convex_hulls) \
    num_threads(ppc::util::GetNumThreads())
for (std::size_t i = 0; i < components.size(); ++i)
```

Параллелизация по компонентам:

- каждая итерация обрабатывает одну компоненту;
- отсутствуют зависимости между итерациями;
- число итераций равно числу компонент.

`default(none)` - повышает безопасность;
`components` - shared (только чтение);
`onvex_hulls` - shared, но каждая итерация пишет в уникальный индекс;
`num_threads(...)` - управление числом потоков через инфраструктуру.

## 5. Детали реализации

Файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Pipeline реализации:

- `ValidationImpl` проверяет корректность размеров изображения и массива пикселей;
- `PreProcessingImpl` копирует входное изображение и выполняет бинаризацию;
- `RunImpl` последовательно находит компоненты связности и строит их оболочки;
- `PostProcessingImpl` передает готовое изображение наружу.

BFS:

Поиск компонент реализован последовательно, так как:

- требует общей структуры `visited`;
- содержит зависимые операции;
- параллелизация усложнила бы код и могла привести к гонкам.

## 6. Проверка корректности

Корректность проверяется сравнением с ожидаемыми результатами и SEQ-версией.

Используются те же 5 функциональных тестов:

1. одиночная точка;
2. несколько компонент;
3. линия;
4. прямоугольник;
5. ромб.

Сравнение выполняется через нормализацию (NormalizeAll):

1. сортировка точек;
2. удаление дубликатов;
3. сравнение множеств оболочек.

Результат:

- OMP-версия полностью совпадает с SEQ по всем тестам.

baseline - 0.0645536977.

Наблюдается замедление относительно SEQ.
Это связано с тем, что распараллеливаемая часть (построение выпуклых оболочек)
имеет меньшую вычислительную сложность по сравнению с последовательным поиском компонент связности (BFS).
В результате накладные расходы OMP не компенсируются ускорением вычислений.

## 9. Выводы

В данной задаче OpenMP не даёт ускорения, а приводит к замедлению.
Основная причина - последовательный BFS, который доминирует по времени выполнения,
но при попытки распараллелить BFS, время только увеличивается и увеличиваются накладные расходы.
