## 1. Контекст

Гибридная версия ALL используется для масштабирования задачи на два уровня параллелизма:

- межпроцессный уровень (MPI) - распределение вычислений между процессами
- внутрипроцессный уровень (TBB) - ускорение обработки внутри одного процесса

Такой подход позволяет:

- уменьшить время обработки на больших изображениях
- разделить нагрузку между узлами/процессами
- использовать многопоточность внутри каждого MPI rank

## 2. Постановка задачи

Требуется:

- Бинаризовать изображение
- Найти компоненты связности
- Построить выпуклую оболочку для каждой компоненты

В качестве baseline используется последовательная реализация (SEQ) - 0.0645536977 с.

## 3. Базовый алгоритм

Последовательная версия включает:

- Thresholding (перевод пикселей в 0/255)
- BFS для поиска компонент связности
- Построение выпуклой оболочки (алгоритм Эндрю)

## 4. Межпроцессная схема

1. Роли процессов
В реализации используется классическая схема:

- rank 0 (master):

1) выполняет весь pipeline (component extraction + hull computation)
2) сериализует результат
3) распространяет результат другим процессам

- rank > 0 (workers):

1) не выполняют вычисления
2) получают готовый результат через broadcast

2. Обмен данными
Используются MPI операции:

- `MPI_Bcast` — распространение результата от rank 0 ко всем процессам
- `MPI_Barrier` — синхронизация завершения

3. Логика распределения
Фактически MPI слой в данной реализации:

1) не делит вычисления
2) выполняет роль координации и синхронизации

## 5. Внутрипроцессная схема

1. Внутри каждого MPI rank для параллелизации вычислений используется библиотека TBB,
обеспечивающая многопоточную обработку независимых задач внутри процесса.
Основная идея внутрипроцессного параллелизма заключается в разделении работы на два этапа:
последовательное выделение компонент связности и параллельное построение выпуклых оболочек для каждой компоненты.

2. Распределение работы между потоками
После завершения этапа поиска компонент связности (BFS выполняется последовательно)
массив компонент components используется как набор независимых задач.

Каждая компонента обрабатывается отдельно и независимо от других. Для распараллеливания применяется:

```cpp
tbb::parallel_for(static_cast<size_t>(0), components.size(), [&](size_t idx) {
  const auto &component = components[idx];
  hulls[idx] = (component.size() <= 2) ? component : BuildHull(component);
});
```

3. Объединение локальных результатов
Каждый поток вычисляет выпуклую оболочку для своей компоненты
и записывает результат напрямую в общий массив convex_hulls по уникальному индексу:

1) `hulls[idx]` соответствует результату обработки `components[idx]`;
2) индексы не пересекаются между потоками;
3) синхронизация через mutex или critical section не требуется.

TBB гарантирует корректное завершение всех задач перед выходом из parallel_for,
обеспечивая неявный барьер синхронизации.

## 6. Детали реализации

1. Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

2. Ключевые особенности

- MPI используется только для синхронизации результата
- TBB используется только внутри rank 0
- отсутствует распределение компонент между процессами

3. Потенциальные узкие места

- BFS выполняется последовательно (нет MPI/TBB параллелизации)
- rank 0 становится bottleneck (вся работа там)
- сериализация результатов `vector<int>` добавляет overhead

## 7. Проверка корректности

Проверка выполнялась через:

1) сравнение с SEQ
2) тестирование на разных изображениях
3) проверка совпадения компонент и hulls

Результаты совпадают полностью.

## 10. Выводы

Гибридная схема ALL эффективна при умеренном числе потоков (1–4),
где основной выигрыш обеспечивает TBB-распараллеливание построения выпуклых оболочек внутри rank 0.

MPI-слой в данной реализации не ускоряет вычисления напрямую,
а выполняет функции синхронизации и распространения результата (broadcast),
поэтому не влияет на вычислительную часть и масштабирование.

При увеличении числа ranks и потоков наблюдается снижение эффективности,
что связано с отсутствием реального распределения вычислительной нагрузки между процессами,
ростом накладных расходов на управление потоками и синхронизацию, а также эффектом oversubscription.
