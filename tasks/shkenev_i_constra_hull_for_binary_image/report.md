## 1. Введение

В работе исследуется задача поиска связных компонент бинарного изображения
и построения выпуклой оболочки для каждой компоненты.
Задача характеризуется высокой вычислительной стоимостью на этапе обработки компонент,
что делает её подходящей для анализа различных моделей параллелизма.

Цель - сравнить эффективность различных подходов: последовательного (SEQ), OpenMP, TBB, STL (std::thread)
и гибридного MPI+TBB (ALL).

## 2. Единая постановка задачи

Вход: бинарное изображение размера до 8000 × 8000.
В эксперименте использовался размер 6000 × 6000 из-за ограничений аппаратных ресурсов ноутбука.

Выход: набор выпуклых оболочек для каждой связной компоненты изображения.

Ограничения:

- `width > 0` и `height > 0`;
- `pixels.size() == width * height`;
- значения пикселей находятся в диапазоне [0, 255].

Критерий корректности:

- все компоненты связности (4-связность) выделены корректно;
- каждая точка принадлежит ровно одной компоненте;
- для каждой компоненты построена выпуклая оболочка;
- оболочка минимальна по включению;
- вершины оболочки упорядочены против часовой стрелки.

SEQ-версия используется как baseline для расчёта ускорения и эффективности остальных реализаций.

## 3. Единая методика эксперимента

1. Окружение

- CPU: Intel(R) Core(TM) 5 220H (2.70 GHz)
- RAM: 8 GB
- OS: Windows 11 Домашняя для одного языка 25H2
- compiler: MSVC
- CMake build type: `Release`

2. Переменные окружения

- `OMP_NUM_THREADS` использовалась для OMP-версии;
- `PPC_NUM_THREADS` использовалась для TBB, STL и OpenMP-части ALL;
- число MPI-процессов для ALL задавалось через `mpiexec -n`.

3. Как генерируются или читаются данные

- функциональные тесты используют фиксированные синтетические изображения малого размера (набор тестовых кейсов);
- performance-тест использует детерминированное разреженное бинарное изображение размера `8000 × 8000`,
но все замеры выполнялись на `6000 × 6000` из-за не хватки ресурсов ноутбука.

4. Метрики

- `time` — медиана времени выполнения `task_run`;
- `speedup = T_seq / T_backend`;
- для `OMP`, `TBB`, `STL` - `efficiency = speedup / threads`;
- для `ALL` - `efficiency = speedup / total_workers`, где `total_workers = ranks × threads_per_rank`.

5. Серия замеров

- для каждой конфигурации выполнялся 1 прогревочный запуск и 3 ещё дополнительных;
- в таблицы заносится медиана этих 4 значений `task_run`.

## 4. Сводка корректности

Все реализации (SEQ, OMP, TBB, STL, ALL) проверены на функциональных тестах и сравнении с SEQ.

1. Проверка корректности

- корректность бинаризации входного изображения;
- совпадение найденных компонент связности с SEQ;
- совпадение выпуклых оболочек (с учётом нормализации порядка точек).

2. Функциональные тесты
Использовался набор синтетических тестов:

- малые изображения 3×3 и 5×5 (простые случаи);
- изображения с несколькими компонентами;
- изображения с геометрическими фигурами (линии, прямоугольники, ромбы).

3. Ограничения применимости

- этап поиска компонент связности (BFS) выполняется последовательно во всех версиях;
- параллелизм применяется только на этапе обработки независимых компонент;
- эффективность зависит от количества компонент и их размера.

## 6. Интерпретация различий

1. SEQ
Последовательная реализация используется как baseline для всех измерений.  
Она выполняет полный цикл: бинаризация, поиск компонент связности и построение выпуклых оболочек без параллелизма.

2. OMP
Наблюдается деградация или слабое ускорение из-за того,
что основная стоимость приходится на BFS-поиск компонент, который плохо распараллеливается.  
Дополнительный overhead возникает из-за управления потоками и планирования задач runtime.

3. TBB
Наиболее стабильная реализация среди thread-based подходов.  
Эффективность достигается за счёт автоматического балансирования задач построения выпуклых оболочек между потоками.  
При этом BFS остаётся последовательным, поэтому масштабирование ограничено числом компонент.

4. STL
Ручное управление потоками приводит к нестабильному времени выполнения.  
Основные потери связаны с overhead создания потоков и отсутствием эффективного task scheduling, как в TBB.

5. ALL
Гибридная схема демонстрирует ускорение, обусловленное внутрипроцессным TBB-параллелизмом.
MPI-слой выполняет роль межпроцессной координации: используется для запуска независимых rank-ов и обмена результатами.
Основная вычислительная нагрузка остаётся внутри одного процесса и обрабатывается средствами TBB.
MPI в данной реализации не участвует в распределении вычислительной нагрузки между процессами.

## 8. Заключение

1. Наилучшую производительность демонстрирует TBB-реализация.  
Это объясняется эффективным динамическим планированием задач построения выпуклых оболочек
и хорошим балансированием нагрузки между потоками.
При этом runtime минимизирует накладные расходы по сравнению с ручным управлением потоками.

ALL показывает наибольшую производительность в конфигурациях с малым числом workers
за счёт внутрипроцессного TBB-параллелизма.
Однако при увеличении числа процессов наблюдается снижение эффективности,
связанное с ростом стоимости MPI-синхронизации.

OMP и STL демонстрируют более низкую эффективность.
Основные ограничения связаны с overhead управления потоками и тем,
что ключевая часть алгоритма (поиск компонент связности BFS) остаётся последовательной и ограничивает масштабирование.

2. Ограничения сравнения
Сравнение ограничено спецификой задачи:
значительная часть вычислений является memory-bound и включает последовательный BFS,
что снижает потенциальный параллелизм.  
Кроме того, все backend-ы тестировались на фиксированном размере входа (`6000 × 6000`),
поэтому результаты отражают поведение только в данной точке нагрузки.

3. Возможные улучшения
Дальнейшее улучшение производительности возможно за счёт:

- параллельного BFS;
- распределения компонент между MPI-процессами;
- уменьшения копирований при построении компонент и оболочек.
