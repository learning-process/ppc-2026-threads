## 1. Контекст

TBB обеспечивает:

- автоматическое планирование задач;
- балансировку нагрузки между потоками;
- снижение необходимости ручного управления потоками (в отличие от OpenMP).

## 2. Постановка задачи

Необходимо для бинарного изображения:

- выполнить пороговую фильтрацию;
- найти компоненты связности;
- для каждой компоненты построить выпуклую оболочку.

В качестве baseline используется последовательная реализация (SEQ) - 0.0645536977 с.

## 3. Базовый алгоритм

Последовательная версия включает три этапа:

- Thresholding - преобразование изображения в бинарное (0/255).
- Поиск компонент связности - BFS по 4-соседям с массивом visited.
- Построение выпуклой оболочки - алгоритм Эндрю (monotonic chain).

Особенность:

- компоненты независимы;
- BFS остаётся последовательным узким местом;
- построение оболочек хорошо параллелится.

## 4. Схема распараллеливания

1. Используемый примитив - `tbb::parallel_for`

Применяется в двух местах:

- бинаризация изображения;
- построение выпуклых оболочек.

2. Диапазон работы

1) Thresholding
`parallel_for(range(0, pixels.size()))`
Каждый поток обрабатывает независимый диапазон пикселей.

2) Convex hulls
`parallel_for(range(0, components.size()))`
Каждая итерация = отдельная компонента.

3. `grainsize` - явно не задан, так как используется `partitioning`.

Причина:

- размер компонент заранее неизвестен;
- TBB сам балансирует нагрузку;
- попытка фиксировать grainsize ухудшала баланс.

4. `partitioner` - используется стандартный `auto_partitioner`

5. Ограничение конкуренции

- отсутствуют shared write конфликты;
- каждый поток пишет в уникальный `hulls[i]`.

## 5. Детали реализации

1. Файлы:

`tbb/include/ops_tbb.hpp`
`tbb/src/ops_tbb.cpp`

2. Особенности реализации:

1) Локальность данных

- components - read-only на этапе hulls
- hulls - запись по уникальному индексу

2) BFS (ExtractComponent)

- выполнен последовательно из-за общей структуры visited
- используется `std::queue`

3) Convex hull

- алгоритм Эндрю (O(n log n))
- сортировка + два прохода (lower/upper hull)

4) Память

- visited локален для компоненты поиска
- отсутствуют атомики и критические секции

## 6. Проверка корректности

Корректность подтверждена:

- совпадением результатов с SEQ;
- тестами на:

1) пустые изображения;
2) одиночные пиксели;
3) большие плотные области;

- проверкой инварианта: каждая точка принадлежит исходной компоненте.

## 9. Выводы

TBB эффективен, когда:

- задачи независимы (как построение оболочек);
- нагрузка вариативна (разные компоненты);
- важен автоматический баланс нагрузки.

TBB теряет эффективность, когда:

- задача мелкозернистая;
- есть последовательный ограничивающий компонент (BFS);
- overhead планировщика сравним с полезной работой.
