## 1. Контекст
Ручное управление потоками через `std::thread` (или `std::jthread`) даёт максимальный контроль над разбиением данных, созданием потоков и синхронизацией. В отличие от OpenMP или TBB, здесь нет автоматического планировщика, и разработчик сам определяет, как именно делить диапазоны и объединять результаты. 

Эта версия создаётся для демонстрации низкоуровневого подхода к распараллеливанию и сравнения его эффективности с более высокоуровневыми решениями.


## 2. Постановка задачи
Полная постановка задачи приведена в [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- Вход: `std::vector<uint8_t>` – пиксели полутонового изображения.
- Выход: `std::vector<uint8_t>` той же длины – контрастированное изображение.
- Алгоритм:
  1. Найти `min` и `max` во входном векторе.
  2. Если `min == max` – скопировать вход в выход.
  3. Иначе для каждого пикселя `p`:
     - $output[i]=\frac{(input[i] - min_I) * (255 - 0)}{max_I-min_I}$
     - Привести к `[0,255]`.
- STL-версия должна давать результат, идентичный SEQ (проверяется через инварианты `CheckRange`), и обеспечивать ускорение за счёт ручной многопоточности.


## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).

Основные этапы:
1. Поиск глобальных `min` и `max` во входном векторе.
2. Если `min == max` – копирование входного вектора в выходной.
3. Иначе – преобразование каждого пикселя по формуле.

## 4. Схема распараллеливания
### 4.1. Разбиение данных на диапазоны
Исходный диапазон индексов `[0, size)` разбивается вручную на num_threads непересекающихся поддиапазонов.
```cpp
const size_t num_threads = std::min<size_t>(ppc::util::GetNumThreads(), size);
const size_t block = size / num_threads;
for (size_t tid = 0; tid < num_threads; ++tid) {
    size_t begin = tid * block;
    size_t end = (tid == num_threads - 1) ? size : begin + block;
    // поток обрабатывает [begin, end)
}
```
- Статическое (фиксированное) разбиение: каждый поток получает непрерывный блок примерно равного размера.
- Последний поток получает оставшиеся элементы, что корректно работает даже когда `size` не делится нацело на `num_threads`.
- Такой подход оптимален для равномерной нагрузки (каждый пиксель обрабатывается за константное время).

#### Пример разбиения для size=115, num_threads=4 (block=28):
| tid | begin | end |
| --- | ----- | --- |
| 0   | 0     | 27  |
| 1   | 28    | 55  |
| 2   | 56    | 84  |
| 3   | 85    | 114 |



### 4.2. Основные функции потоков
- **Поиск min/max:** каждый поток вызывает `std::ranges::minmax_element` на своём поддиапазоне и сохраняет локальную структуру `MinMax` в массив `local[tid]`.
- **Копирование:** каждый поток копирует элементы своего поддиапазона из `input` в `output`.
- **Преобразование:** каждый поток применяет формулу растяжения к своему поддиапазону и записывает результат в `output`.

### 4.3. Локальные результаты и их объединение (Поиск min/max)
- Для редукции при поиске min/max используется массив `std::vector<MinMax> local(num_threads)`. Каждый поток записывает свой локальный результат в отдельный элемент массива (разные индексы – нет гонок).
- После завершения всех потоков (выход из области видимости вектора `threads`) вызывается **последовательное объединение**:
    ```cpp
    MinMax result{.min = 255, .max = 0};
    for (size_t tid = 0; tid < num_threads; ++tid) {
        result.min = std::min(result.min, local[tid].min);
        result.max = std::max(result.max, local[tid].max);
    }
    ```
    - Никакой синхронизации (мьютексы, атомики) не требуется, так как запись в локальные массивы не пересекается, а чтение происходит после завершения всех потоков.

### 4.4. Управление потоками и автоматический join
Используется `std::jthread`, который вызывает join() в своём деструкторе.

- В поиске min/max (метод `ComputeMinMax`) потоки создаются внутри отдельного блока `{ ... }`:
    ```cpp
    {
    std::vector<std::jthread> threads;
        // ... создание потоков
    } // здесь деструкторы всех jthread вызывают join()
    ```
    Это гарантирует, что все потоки завершатся до того, как начнётся объединение локальных результатов.
- В копировании (`CopyInput`) и преобразовании (`LinearStretch`) отдельный блок не нужен, потому что вектор `threads` уничтожается при выходе из функции (сразу после цикла создания). Это также дожидается завершения всех потоков перед возвратом управления.

### 4.5. Синхронизация и потокобезопасность
- **Отсутствие гонок:**
    - При поиске min/max потоки пишут в разные ячейки `local[tid]` (нет конфликта).
    - При копировании и преобразовании каждый поток пишет только в свою часть output (индексы `[begin, end)` не пересекаются).
- **Нет мьютексов, атомиков, условных переменных** – это ускоряет выполнение, так как синхронизация не требуется.
- **Чтение** `input` – все потоки только читают, что безопасно.

### 4.6. Порог для параллельного копирования
```cpp
if (size > threshold_size) {
  CopyInput(input, output);
} else {
  for (size_t i = 0; i < size; ++i) {
    output[i] = input[i];
  }
}
```
- Для маленьких изображений накладные расходы на создание потоков могут превысить выигрыш. Порог выбран эмпирически.

## 5. Детали реализации
### 5.1. Файлы
- `stl/include/ops_stl.hpp` – объявление класса OtcheskovSContrastLinStretchSTL.
- `stl/src/ops_stl.cpp` – реализация методов.

### 5.2. Структура класса
```cpp
class OtcheskovSContrastLinStretchSTL : public BaseTask {
 public:
  static constexpr ppc::task::TypeOfTask GetStaticTypeOfTask() {
    return ppc::task::TypeOfTask::kSTL;
  }
  explicit OtcheskovSContrastLinStretchSTL(const InType &in);

 private:
  bool ValidationImpl() override;
  bool PreProcessingImpl() override;
  bool RunImpl() override;
  bool PostProcessingImpl() override;

  struct MinMax {
    uint8_t min{255};
    uint8_t max{0};
  };

  static MinMax ComputeMinMax(const InType &input);
  static void CopyInput(const InType &input, OutType &output);
  static void LinearStretch(const InType &input, OutType &output, int min_i, int range);
};
```

### 5.3. Метод `RunImpl`
1. Проверка на пустой вход.
2. `MinMax result = ComputeMinMax(input)`;
3. Если `result.min == result.max`:
    - При `size > threshold_size` → `CopyInput` (параллельно).
    - Иначе последовательное копирование.
4. Иначе → `LinearStretch`.

### 5.4. Реализация вспомогательных функций:
- **Поиск min/max** (`ComputeMinMax`)
    ```cpp
    const size_t num_threads = std::min<size_t>(ppc::util::GetNumThreads(), size);
    const size_t block = size / num_threads;
    std::vector<MinMax> local(num_threads, {255, 0});
    std::vector<std::jthread> threads;
    for (size_t tid = 0; tid < num_threads; ++tid) {
        size_t begin = tid * block;
        size_t end = (tid == num_threads-1) ? size : begin + block;
        threads.emplace_back([&, tid, begin, end]() {
            auto [min_it, max_it] = std::ranges::minmax_element(
                input.begin() + begin, input.begin() + end);
            local[tid] = {*min_it, *max_it};
        });
    }
    ```
    - Создание вектора потоков выполняется внутри отдельного блока, который необходим, чтобы ограничить области видимости этого вектора и потоки корректно завершились до чтения массива `local`.
    - Лямбда захватывает `tid`, `begin`, `end` по значению, а `input` и `local` – по ссылке.

- **Копирование** (`CopyInput`) – аналогичный макет, содержательно каждый поток копирует свой поддиапазон.
    ```cpp
    threads.emplace_back([&, begin, end]() {
      for (size_t i = begin; i < end; ++i) {
        output[i] = input[i];
      }
    });
    ```
- **Преобразование** (`LinearStretch`) – аналогичный макет, содержательно каждый поток преобразует свой поддиапазон.
    ```cpp
    threads.emplace_back([&, begin, end, min_i, range]() {
      for (size_t i = begin; i < end; ++i) {
        int pixel = static_cast<int>(input[i]);
        int value = (pixel - min_i) * 255 / (range);
        value = std::clamp(value, 0, 255);
        output[i] = static_cast<uint8_t>(value);
      }
    });
    ```

#### Защита от слишком большого числа потоков:
`num_threads = min(GetNumThreads(), size)` – если изображение меньше запрошенного числа потоков, создаётся только size потоков.


## 6. Проверка корректности
Используются те же тестовые наборы, что и для SEQ (см. [seq/report.md](../seq/report.md#5-проверка-корректности)), с применением критерия `CheckRange`:
```cpp
bool CheckRange(const OutType &data) {
  if (data.empty()) return false;
  auto [min_it, max_it] = std::ranges::minmax_element(data);
  return (*min_it == 0 && *max_it == 255) || (*min_it == *max_it);
}
```

#### Используемые тестовые наборы:
- **Валидационные:** если пустой вход, то `ValidationImpl` возвращает `false`.
- **Функциональные:** синтетические изображения 1×1, 2×2, 3×3, 100×100, 500×500 с низкой контрастностью, то `CheckRange` должно быть истинно (min=0, max=255).
- **Однородные изображения:** 10×10 и 501×501, заполненные нулями, тогда `CheckRange` истинно (min==max).
- **Реальное изображение:** `data/grayimg.jpg`, то после обработки `CheckRange` истинно.

#### Дополнительные проверки:
- Использование `std::jthread` гарантирует, что все потоки завершатся до того, как будут использованы локальные результаты.
- Запуск с разным числом потоков (2,4,6,12,24) даёт одинаковые выходные данные.

#### Запуск тестов:
```bash
./bin/ppc_func_tests --gtest_filter="*Otcheskov*stl*"
```

## 9. Выводы
Ручная реализация на `std::jthread` показала лучшее ускорение (до 6.57×) и эффективность (до 55%) на 12 потоках по сравнению с OpenMP и TBB.

#### Удобство STL:
- Полный контроль над разбиением данных – простое статическое разбиение эффективно для равномерной нагрузки.
- Отсутствие накладных расходов на динамическое планирование задач (нет work stealing, нет рекурсивных разбиений).
- Использование `std::jthread` упрощает управление жизненным циклом потоков (автоматический `join`).

#### Ограничения:
- Для нерегулярных данных статическое разбиение может привести к дисбалансу, тогда как OpenMP/TBB адаптируются.
- В коде больше механической работы (ручное разбиение, локальные массивы, объединение результатов) – выше вероятность ошибок.
- На очень маленьких изображениях порог для параллельного копирования приходится подбирать вручную.

#### Заключение:
Ручное управление потоками через STL – мощный инструмент, когда требуется максимальная производительность и предсказуемость на однородных данных. Для задачи линейной растяжки гистограммы оно даёт отличное ускорение, а использование `std::jthread` делает код безопасным.
