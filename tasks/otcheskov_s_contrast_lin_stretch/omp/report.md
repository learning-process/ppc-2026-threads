## 1. Контекст

В последовательной версии (SEQ) вся вычислительная нагрузка ложится на одно ядро.

Для ускорения обработки больших изображений (например, 10000 × 10000 пикселей)
целесообразно распараллелить два основных этапа:  

- поиск глобальных минимума и максимума яркости;  
- преобразование каждого пикселя по формуле линейного растяжения.  

OpenMP позволяет легко организовать параллельное выполнение циклов на общей памяти без изменения структуры данных.

## 2. Постановка задачи

Полная постановка задачи приведена в [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- Вход: `std::vector<uint8_t>` – пиксели полутонового изображения.
- Выход: `std::vector<uint8_t>` той же длины – контрастированное изображение.
- Алгоритм:
  1. Найти `min` и `max` во входном векторе.
  2. Если `min == max` – скопировать вход в выход.
  3. Иначе для каждого пикселя `p`:
     - $output[i]=\frac{(input[i] - min_I) * (255 - 0)}{max_I-min_I}$
     - Привести к `[0,255]`.
- OMP-версия должна давать результат, идентичный SEQ (проверяется через инварианты `CheckRange`),
и обеспечивать ускорение за счёт параллельной обработки.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).

Основные этапы:

1. Поиск глобальных `min` и `max` во входном векторе.
2. Если `min == max` – копирование входного вектора в выходной.
3. Иначе – преобразование каждого пикселя по формуле.

## 4. Схема распараллеливания

### 4.1 Области кода

- **Поиск глобальных min/max**
- **Копирование однородного изображения** (при достаточно большом размере)
- **Основной цикл преобразования**

### 4.2. Переменные: shared / private / reduction

#### 4.2.1. Поиск min/max

```cpp
#pragma omp parallel for default(none) shared(input, size) reduction(min : min_val) reduction(max : max_val)
for (size_t i = 0; i < size; ++i) {
  min_val = std::min(input[i], min_val);
  max_val = std::max(input[i], max_val);
}
```

- `default(none)` – требует явного указания класса каждой переменной, что исключает случайные `shared` и потенциальные гонки.
- `shared(input, size)` – все потоки читают одни и те же входные данные и размер; запись отсутствует.
- `reduction(min : min_val)` и `reduction(max : max_val)` – каждый поток использует локальные копии `min_val` и `max_val`.
После цикла значения объединяются операциями `min` и `max`.
  - Без редукции возникла бы гонка при одновременной записи в общие переменные.

#### 4.2.2. Копирование однородного изображения

```cpp
#pragma omp parallel for if (input.size() > threshold_size) default(none) shared(input, output, size)
for (size_t i = 0; i < size; ++i) {
  output[i] = input[i];
}
```

- `if(...)` – запускает параллельную версию только если размер превышает 1 000 000 элементов.
При малых изображениях накладные расходы на создание потоков перевешивают выигрыш.
- `shared(input, output, size)` – все потоки читают `input` и `size`,
пишут в свои непересекающиеся элементы `output` (гонки нет, так как индексы i различны).

#### 4.2.3. Основной цикл преобразования

```cpp
#pragma omp parallel for default(none) shared(input, output, min_i, range, size)
for (size_t i = 0; i < size; ++i) {
  int pixel = static_cast<int>(input[i]);
  int value = (pixel - min_i) * 255 / range;
  output[i] = static_cast<uint8_t>(std::clamp(value, 0, 255));
}
```

- `shared`— данные (`input`, `min_i`, `range`, `size`) только читаются.
Запись в `output` выполняется по разным индексам:
каждый поток обрабатывает свою часть массива.
- `private` переменные отсутствуют явно, но итерационная переменная `i` по умолчанию `private`.

### 4.3. Выбор schedule

Во всех трёх секциях `schedule` явно не указан, т.е. используется параметр `auto`,
где разработчики компилятора сами определяют, какой тип расписания используется.
В случае компилятора MSVC моей версии, используется статическое расписание.

- При статическом распределении итерации делятся на непрерывные блоки (чанки) примерно одинакового размера,
каждый поток получает один такой блок.

- Для моей задачи это эффективно, потому что:
  - Дисбаланс нагрузки отсутствует, действия над частями изображения будут однородны по сложности.
  - Статическое расписание минимизирует накладные расходы на динамическое распределение итераций.
  - Работа над каждым элементом осуществляется за константное время (пара арифметических операций)
- Для поиска min/max и копирования изображения такое решение также отличное.

### 4.4. Барьеры

- В конце каждой области `parallel for` есть неявный барьер.
Главный поток ждёт завершения всех потоков.
Это необходимо, так как min/max должны быть вычислены до основного цикла.
- В случае `parallel for if(false)` барьера нет, так как код выполняется последовательно.

## 5. Детали реализации

### 5.1. Файлы

- `omp/include/ops_omp.hpp` – объявление класса `OtcheskovSContrastLinStretchOMP`.
- `omp/src/ops_omp.cpp` – реализация методов.

### 5.2. Структура класса

```cpp
class OtcheskovSContrastLinStretchOMP : public BaseTask {
 public:
  static constexpr ppc::task::TypeOfTask GetStaticTypeOfTask() {
    return ppc::task::TypeOfTask::kOMP;
  }
  explicit OtcheskovSContrastLinStretchOMP(const InType &in);

 private:
  bool ValidationImpl() override;
  bool PreProcessingImpl() override;
  bool RunImpl() override;
  bool PostProcessingImpl() override;
};
```

#### Изменения относительно SEQ

- Изменены секции кода в `RunImpl`

| Секция | SEQ | OMP |
| -------------- | --- | --- |
| Поиск min/max | `std::ranges::minmax_element` | Параллельный цикл с `reduction(min, max)` |
| Копирование при однородном изображении | `output = input` | Параллельное копирование с `parallel for if(size>1_000_000)` |
| Преобразование изображения | Последовательный цикл преобразования | Параллельный цикл `parallel for` без доп. синхронизаций |

#### Устранённые риски гонок

- При параллельном поиске min/max без редукции потоки одновременно писали бы в `min_val`/`max_val`,
то есть была бы гонка данных. Редукция устраняет её.
- При копировании и преобразовании каждый поток работает со своей уникальной группой индексов `i`
(определяемой статическим распределением), поэтому запись в `output` не конфликтует.

## 6. Проверка корректности

Используются те же тестовые наборы,
что и для SEQ (см. [seq/report.md](../seq/report.md#5-проверка-корректности)), с применением критерия `CheckRange`:

```cpp
bool CheckRange(const OutType &data) {
  if (data.empty()) return false;
  auto [min_it, max_it] = std::ranges::minmax_element(data);
  return (*min_it == 0 && *max_it == 255) || (*min_it == *max_it);
}
```

### 6.1. Используемые тестовые наборы

- **Валидационные:** пустой вход → `ValidationImpl` возвращает `false`.
- **Функциональные:** синтетические изображения 1×1, 2×2, 3×3, 100×100, 500×500
с низкой контрастностью → `CheckRange` истинно (min=0, max=255).
- **Однородные изображения:** 10×10 и 501×501, заполненные нулями → `CheckRange` истинно (min==max).
- **Реальное изображение:** `data/grayimg.jpg` → после обработки `CheckRange` истинно.

### 6.2. Дополнительные проверки

- При `PPC_NUM_THREADS=1` выход OMP-версии идентичен выходу SEQ (запуск с одним потоком и сравнения с эталонным результатом).
- При разных значениях `PPC_NUM_THREADS` (2, 4, 6, 12) выходные векторы совпадают между собой,
что подтверждает отсутствие гонок и детерминизм.

Таким образом, корректность OMP-реализации доказана через выполнение инвариантов алгоритма.

## 9. Выводы

OMP-версия даёт существенный выигрыш по времени (до 5.94× на 12 потоках) по сравнению с последовательной.
Однако при увеличении размера изображения и числа потоков будет деградация производительности,
так как возникнет упор в память:

- Основное время будет уходить не на арифметические операции,
а на перемещение данных между оперативной памятью и кэшем процессора.

### 9.1. Выигрыш использования OpenMP достигается

- при использовании числа потоков, не превышающего количество физических ядер.
- при равномерной нагрузке (все пиксели обрабатываются одинаково).
- При правильно подобранном размере изображения так, чтобы не было упора в память.

### 9.2. Ограничения подхода

- При малых размерах изображения (условно меньше 1 млн пикселей)
накладные расходы на создание потоков и редукцию ухудшают эффективность.
- Гиперпоточность не даёт двукратного прироста на целочисленных вычислениях из-за разделения исполнительных ресурсов ядра.
