## 1. Контекст

TBB (oneTBB) – это библиотека параллельного программирования на основе задач,
которая автоматически управляет пулом потоков и балансировкой нагрузки.

Для задачи линейного растяжения гистограммы TBB предоставляет более высокий уровень абстракции по сравнению с OpenMP:
декомпозиция на задачи, рекурсивное разбиение диапазонов, автоматическая настройка количества чанков.
Наиболее интересно в данной задаче разбиение диапазонов и динамическое распределение задач.

## 2. Постановка задачи

Полная постановка задачи приведена в [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- Вход: `std::vector<uint8_t>` – пиксели полутонового изображения.
- Выход: `std::vector<uint8_t>` той же длины – контрастированное изображение.
- Алгоритм:
  1. Найти `min` и `max` во входном векторе.
  2. Если `min == max` – скопировать вход в выход.
  3. Иначе для каждого пикселя `p`:
     - $output[i]=\frac{(input[i] - min_I) * (255 - 0)}{max_I-min_I}$
     - Привести к `[0,255]`.
- TBB-версия должна давать результат, идентичный SEQ (проверяется через инварианты `CheckRange`),
и обеспечивать ускорение за счёт параллельной обработки.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с SEQ ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).

Основные этапы:

1. Поиск глобальных `min` и `max` во входном векторе.
2. Если `min == max` – копирование входного вектора в выходной.
3. Иначе – преобразование каждого пикселя по формуле.

Оба этапа (`parallel_reduce` для поиска минимума/максимума, `parallel_for` для копирования/преобразования) являются итеративно-независимыми.

## 4. Схема распараллеливания

### 4.1. Выбранные примитивы TBB

- `tbb::parallel_reduce` – для параллельного поиска глобального минимума и максимума.
- `tbb::parallel_for` – для параллельного копирования и для основного цикла преобразования.

Оба примитива работают по принципу **рекурсивного разбиения диапазона** на подзадачи,
которые распределяются между worker-потоками.
Кража задач (work stealing) обеспечивает автоматическую балансировку.

### 4.2. Диапазон работы

Для представления непрерывного диапазона индексов используется `tbb::blocked_range<size_t>(0, size)`, где `size = input.size()`.

- `blocked_range` – это шаблонный класс, хранящий начало, конец и опционально размер зерна (grainsize).
- TBB разбивает исходный диапазон на **субдиапазоны** (чанки), передавая их в лямбда-функции.
- В коде диапазон создаётся без указания третьего параметра (grainsize), что означает **автоматический выбор зерна** библиотекой.

### 4.3. Размер зерна

**Явно не задан** – используется перегрузка `parallel_for` / `parallel_reduce` без аргумента `grainsize`.

- По умолчанию TBB подбирает grainsize эвристически:
обычно старается, чтобы в каждом чанке было не менее нескольких сотен итераций,
чтобы окупить накладные расходы на создание и планирование задач.
- Экспериментальная проверка показала,
что для изображений 100 млн пикселей производительность не чувствительна к grainsize в диапазоне от 1000 до 100000.
Автоматический выбор библиотеки даёт время, сопоставимое с ручной настройкой.

### 4.4. Разбиение

Во всех распараллеливаемых областях кода явно указан `tbb::static_partitioner{}`.

- `static_partitioner` разбивает диапазон один раз на этапе запуска на фиксированное число чанков (приблизительно равных).
- Поскольку обработка каждого пикселя требует константного времени и нагрузка равномерна,
статическое разбиение минимизирует накладные расходы на динамическое перераспределение.

### 4.5. Управление конкуренцией

1. Ограничение количества worker-потоков через `task_arena`:

    ```cpp
    tbb::task_arena arena(ppc::util::GetNumThreads());
    arena.execute([&] {
        // параллельные вызовы
    });
    ```

    - `task_arena` создаёт пул, в котором могут одновременно выполняться не более `GetNumThreads()` потоков.
    - Все параллельные примитивы (`parallel_reduce`, `parallel_for`) запускаются внутри `arena.execute()`,
    что гарантирует, что они будут использовать именно этот ограниченный пул.
    - `task_arena` не создаёт потоки заново при каждом вызове, а использует глобальный пул TBB,
    просто ограничивая количество активных **worker'ов** внутри этой арены.
    - Используется вместо `global_control::max_allowed_parallelism` для явной изоляции параллельных регионов.
2. Внешнее задание числа потоков:
    - `ppc::util::GetNumThreads()` читает переменную окружения `PPC_NUM_THREADS` и возвращает запрошенное значение.
    - Это число передаётся в конструктор `task_arena`.
    Таким образом, можно явно ограничить максимальную конкуренцию без изменения кода.

### 4.6. Редукция для min/max

```cpp
result = tbb::parallel_reduce(
    tbb::blocked_range<size_t>(0, input.size()),
    MinMax{},  // начальное значение
    [&](const tbb::blocked_range<size_t>& r, MinMax local) {
        for (size_t i = r.begin(); i != r.end(); ++i) {
            local.min = std::min(local.min, input[i]);
            local.max = std::max(local.max, input[i]);
        }
        return local;
    },
    [](const MinMax& a, const MinMax& b) -> MinMax {
        return {std::min(a.min, b.min), std::max(a.max, b.max)};
    },
    tbb::static_partitioner{}
);
```

- Каждый **worker** получает свой экземпляр `local` (копию начального значения) и обрабатывает поддиапазон.
- Функция слияния вызывается для объединения частичных результатов.
- **Гонок нет** – нет общих переменных, требующих атомарных операций.

### 4.7. Обработка однородного изображения и порог копирования

```cpp
const size_t threshold_size = 1000000;
if (size > threshold_size) {
    CopyInput(input, output, arena);
} else {
    for (size_t i = 0; i < size; ++i) output[i] = input[i];
}
```

- Порог выбран эмпирически:
при размере менее 1 млн пикселей накладные расходы на создание задач `parallel_for` могут превысить выигрыш.

## 5. Детали реализации

### 5.1. Файлы

- `tbb/include/ops_tbb.hpp` – объявление класса OtcheskovSContrastLinStretchTBB.
- `tbb/src/ops_tbb.cpp` – реализация методов.

### 5.2. Структура класса

```cpp
class OtcheskovSContrastLinStretchTBB : public BaseTask {
 public:
  static constexpr ppc::task::TypeOfTask GetStaticTypeOfTask() {
    return ppc::task::TypeOfTask::kTBB;
  }
  explicit OtcheskovSContrastLinStretchTBB(const InType &in);

 private:
  bool ValidationImpl() override;
  bool PreProcessingImpl() override;
  bool RunImpl() override;
  bool PostProcessingImpl() override;

  struct MinMax {
    uint8_t min{255};
    uint8_t max{0};
  };

  static MinMax ComputeMinMax(const InType &input, tbb::task_arena &arena);
  static void CopyInput(const InType &input, OutType &output, tbb::task_arena &arena);
  static void LinearStretch(const InType &input, OutType &output, int min_i, int range, tbb::task_arena &arena);
};
```

### 5.3. Метод `RunImpl`

1. Проверка на пустой вход.
2. Получение размера.
3. Создание `task_arena` с числом потоков из `ppc::util::GetNumThreads()`.
4. Вычисление `MinMax` через `ComputeMinMax`.
5. Если `min == max`:
   - При `size > 1 000 000` – параллельное копирование `CopyInput`.
   - Иначе – последовательное копирование.
6. Иначе – вычисление `min_i`, `range` и вызов `LinearStretch`.

### 5.4. Реализация вспомогательных функций

- Все они принимают `tbb::task_arena&` и выполняют свою работу внутри arena.`execute(...)`.
- `ComputeMinMax` использует `parallel_reduce` с `static_partitioner`.
- `CopyInput` и `LinearStretch` используют `parallel_for` с `static_partitioner`.
- Тело цикла `LinearStretch` выполняет преобразование:

    ```cpp
    int value = (static_cast<int>(input[i]) - min_i) * 255 / range;
    output[i] = static_cast<uint8_t>(std::clamp(value, 0, 255));
    ```

### 5.5. Особенности работы с памятью

- Выходной вектор выделяется один раз в `PreProcessingImpl` с размером, равным входному.
- Никакой дополнительной синхронизации для записи в `output` не требуется, так как индексы `i` в каждом чанке уникальны.

## 6. Проверка корректности

Используются те же тестовые наборы,
что и для SEQ (см. [seq/report.md](../seq/report.md#5-проверка-корректности)), с применением критерия `CheckRange`:

```cpp
bool CheckRange(const OutType &data) {
  if (data.empty()) return false;
  auto [min_it, max_it] = std::ranges::minmax_element(data);
  return (*min_it == 0 && *max_it == 255) || (*min_it == *max_it);
}
```

### 6.1. Используемые тестовые наборы

- **Валидационные:** если пустой вход, то `ValidationImpl` возвращает `false`.
- **Функциональные:** синтетические изображения 1×1, 2×2, 3×3, 100×100, 500×500 с низкой контрастностью,
то `CheckRange` должно быть истинно (min=0, max=255).
- **Однородные изображения:** 10×10 и 501×501, заполненные нулями, тогда `CheckRange` истинно (min==max).
- **Реальное изображение:** `data/grayimg.jpg`, то после обработки `CheckRange` истинно.

### 6.2. Дополнительные проверки

- Ручная проверка для одного потока (`GetNumThreads() == 1`) – результат совпадает с SEQ.
- Для 2, 4, 6, 12, 24 потоков результаты (минимаксные значения) одинаковы, что подтверждает корректность редукции.

### 8.1. Комментарий о балансировке и накладных расходах

- **До 6 workers:** (количество физических ядер) ускорение растёт почти линейно (эффективность >60%).
Это ожидаемо, так как каждый **worker** выполняется на отдельном физическом ядре без конкуренции за исполнительные ресурсы.
- **На 12 workers:** задействуется технология Hyper-Threading для увеличивая число логических ядер до 12.
Технология позволяет выполнять два потока на одном физическом ядре. Это повышает загрузку исполнительных конвейеров,
но кэш делится между потоками.
Из-за конкуренции за кэш начинают проявляться ограничения памяти.
- **На 24 workers:** мы превышаем число логических ядер и происходит oversubscription,
то есть операционная система и планировщик TBB вынуждены переключать контексты, что приводит к росту накладных расходов,
вследствие чего время и ускорение ухудшились по сравнению с прошлым результатом.

## 9. Выводы

### 9.1. Удобство TBB

- TBB удобен в более сложных задачах с нерегулярной нагрузкой,
где автоматическая балансировка даёт преимущество перед статическим распределением OpenMP,
а также когда требуется изолировать параллельные регионы через task_arena или управлять конкуренцией более гибко.
- При смешанных нагрузках (например, несколько уровней параллелизма) TBB позволяет избежать oversubscription.

### 9.2. Ограничения и накладные расходы

- Для равномерных вычислений вроде линейного растяжения гистограммы TBB может уступать OpenMP из-за больших накладных расходов на рекурсивную декомпозицию.
- Явное указание `static_partitioner` уменьшает, но не устраняет полностью накладные расходы.
- Задача ограничена пропускной способностью памяти: на 12 потоках ускорение не превышает 4.5×.

#### Заключение

TBB является жизнеспособной альтернативой OpenMP и STL, особенно когда важна абстракция задач и гибкость,
но для чисто цикловых вычислений на общих данных OpenMP и STL показывают лучшую производительность.
