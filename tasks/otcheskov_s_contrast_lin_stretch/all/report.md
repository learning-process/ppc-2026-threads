## 1. Контекст
Гибридная версия (MPI + потоки) необходима для распараллеливания задачи на кластерах с распределённой памятью, где каждый узел может иметь несколько ядер.
- MPI обеспечивает обмен данными между независимыми процессами (узлами).
- STL (`std::jthread`) используется для внутриузлового распараллеливания на всех ядрах одного процесса.

Такая иерархическая схема позволяет обрабатывать очень большие изображения, которые не помещаются в память одного узла, и эффективно использовать все уровни параллелизма.


## 2. Постановка задачи
Полная постановка задачи приведена в [`seq/report.md`](../seq/report.md#2-постановка-задачи).

- Вход: `std::vector<uint8_t>` (пиксели полутонового изображения) – хранится только на процессе с рангом 0.
- Выход: `std::vector<uint8_t>` той же длины – собирается на процессе 0.
- Алгоритм:
  1. Найти `min` и `max` во входном векторе.
  2. Если `min == max` – скопировать вход в выход.
  3. Иначе для каждого пикселя `p`:
     - $output[i]=\frac{(input[i] - min_I) * (255 - 0)}{max_I-min_I}$
     - Привести к `[0,255]`.
- Гибридная версия должна давать результат, идентичный SEQ, и обеспечивать масштабируемость как по числу MPI-процессов, так и по числу потоков внутри каждого процесса.


## 3. Базовый алгоритм
Базовый алгоритм полностью совпадает с SEQ ([см. seq/report.md](../seq/report.md#3-базовый-алгоритм)).

Основные этапы:
1. Поиск глобальных `min` и `max` во входном векторе.
2. Если `min == max` – копирование входного вектора в выходной.
3. Иначе – преобразование каждого пикселя по формуле.

## 4. Межпроцессная схема
### 4.1. Роли процессов
- **Ранг 0 (главный процесс):**
  - Хранит исходное изображение (GetInput()).
  - Выполняет MPI_Scatterv для распределения частей изображения всем процессам.
  - После обработки собирает результат через MPI_Gatherv в GetOutput().

- **Остальные процессы (ранги 1 … size-1):**
  - Получают свою локальную часть данных.
  - Обрабатывают её независимо.
  - Отправляют обработанную часть главному процессу.

### 4.2. Распределение данных
Исходное изображение разбивается на непрерывные блоки (по количеству процессов) с помощью статического равномерного разбиения:
```cpp
int base = global_size / size_;
int rem = global_size % size_;
for (int i = 0; i < size_; ++i) {
  counts[i] = base + (i < rem ? 1 : 0);
  displs[i] = (i * base) + std::min(i, rem);
}
```
- `counts[i]` – размер блока для процесса i.
- `displs[i]` – смещение в глобальном массиве.

### 4.3. Используемые MPI-вызовы
- `MPI_Bcast` – для рассылки:
    - результата валидации (is_valid_) от процесса 0 всем остальным.
    - глобального размера изображения (global_size).
- `MPI_Scatterv` – распределение исходных данных: каждый процесс получает свой блок `local_input`.
- `MPI_Allreduce` – поиск глобальных `min` и `max` по всем процессам:
    - `MPI_MIN` и `MPI_MAX` объединяют локальные результаты.
    - Результат (`global_min`, `global_max`) становится известен каждому процессу.
- `MPI_Gatherv` – сбор обработанных блоков обратно на процесс 0 в правильном порядке (с учётом смещений).

**Синхронизация:** неявно используются в коллективных вызовах. 

### 4.4. Обработка однородного изображения
Если глобальный минимум равен максимуму (`global_min == global_max`), то каждый процесс копирует свой локальный блок в local_output. Главный процесс собирает копии. Это работает без дополнительных коммуникаций.


## 5. Внутрипроцессная схема
Внутри каждого MPI-процесса используется ручное управление потоками через `std::jthread` (полный функционал расписан в [STL-версии](../stl/report.md#4-схема-распараллеливания)).

### 5.1. Разбиение локального диапазона
Локальный вектор `local_input` размера `local_size` разбивается на `num_threads` поддиапазонов:
```cpp
const size_t num_threads = std::min<size_t>(ppc::util::GetNumThreads(), local_size);
const size_t block = local_size / num_threads;
```
- Статическое разбиение, последний поток получает остаток.
- Каждый поток работает с непрерывной областью памяти – хорошая локальность.

### 5.2. Параллельные этапы внутри процесса
#### 5.2.1. Поиск локальных min/max (`ComputeMinMax`)
- Каждый поток вычисляет min и max на своём поддиапазоне через std::ranges::minmax_element.
- Результаты сохраняются в массив local[tid].
- После завершения всех потоков (автоматический join через деструктор std::jthread) выполняется последовательное объединение локальных результатов.
- Потоки не конкурируют за общие данные (пишут в разные ячейки).

#### 5.2.2. Параллельное копирование (`CopyInput`)
При однородном изображении каждый поток копирует свой блок из `local_input` в `local_output`.
- Используется тот же шаблон разбиения.
- Потоки не синхронизируются между собой – каждый пишет в свою часть выходного буфера.

#### 5.2.3. Параллельное преобразование (`LinearStretch`)
- Каждый поток применяет формулу растяжения к элементам своего поддиапазона и записывает результат в `local_output`.
- Все потоки читают один и тот же `local_input` (без записи) – безопасно.

### 5.3. Управление потоками и автоматический `join`
- Используется `std::jthread`, который вызывает `join()` в деструкторе.
- В `ComputeMinMax` потоки создаются внутри отдельного блока, чтобы вектор `threads` уничтожился до объединения локальных результатов.
- В `CopyInput` и `LinearStretch` отдельный блок не нужен – деструктор вызывается при выходе из функции.

### 6.3. Использование MPI-вызовов
Ниже приведены ключевые фрагменты кода с вызовами MPI-функций, их назначение и место в логике алгоритма.

#### 6.3.1. Валидация – `MPI_Bcast`:
```cpp
if (rank_ == 0) {
  is_valid_ = !GetInput().empty();
}
MPI_Bcast(&is_valid_, 1, MPI_C_BOOL, 0, MPI_COMM_WORLD);
return is_valid_;
```
- `MPI_Bcast` рассылает флаг `is_valid_` от процесса 0 всем остальным.
- Все процессы узнают, пуст ли вход, чтобы принять единое решение о продолжении вычислений. Без этого процессы с рангом ≠ 0 не знали бы результата валидации.

#### 6.3.2. Передача глобального размера – `MPI_BCAST`
```cpp
int global_size = 0;
if (rank_ == 0) {
  global_size = static_cast<int>(input.size());
}
MPI_Bcast(&global_size, 1, MPI_INT, 0, MPI_COMM_WORLD);
```
- Второй MPI_Bcast передаёт размер исходного изображения всем процессам.
- Каждый процесс узнаёт общий размер, чтобы вычислить свои смещения и количество элементов для Scatterv/Gatherv.

#### 6.3.3. Распределение данных – `MPI_Scatterv`
```cpp
MPI_Scatterv(rank_ == 0 ? input.data() : nullptr,
             counts.data(), displs.data(), MPI_UINT8_T,
             local_input.data(), static_cast<int>(local_size), MPI_UINT8_T,
             0, MPI_COMM_WORLD);
```
- Данный вызов позволяет передавать данные блоками разного размера и смещениями.
- Процесс 0 рассылает свои данные, остальные получают свои блоки.
- Каждый процесс получает только ту часть изображения, которую будет обрабатывать.

#### 6.3.4. Поиск глобальных min/max – `MPI_Allreduce`
```cpp
uint8_t global_min{};
uint8_t global_max{};

MPI_Allreduce(&local.min, &global_min, 1, MPI_UINT8_T, MPI_MIN, MPI_COMM_WORLD);
MPI_Allreduce(&local.max, &global_max, 1, MPI_UINT8_T, MPI_MAX, MPI_COMM_WORLD);
```
- `MPI_Allreduce` выполняет глобальную редукцию и результат рассылает всем процессам.
- `MPI_MIN` – операция взятия минимума, `MPI_MAX` – максимума.
- Каждый процесс узнаёт глобальные минимум и максимум для применения формулы линейной растяжки к своему локальному блоку. Без `Allreduce` пришлось бы сначала собрать локальные минимумы на процессе 0, а затем разослать обратно – `Allreduce` делает это за один шаг.

#### 6.3.5. Сбор результата – `MPI_Gatherv`
```cpp
MPI_Gatherv(local_output.data(), static_cast<int>(local_size), MPI_UINT8_T,
            rank_ == 0 ? output.data() : nullptr,
            counts.data(), displs.data(), MPI_UINT8_T,
            0, MPI_COMM_WORLD);
```
- `MPI_Gatherv` – обратная операция к `Scatterv`. Процесс 0 собирает все локальные блоки в правильном порядке (с учётом размера блока и смещения).
- Нужен, чтобы выходное изображение было восстановлено на процессе 0 для дальнейшего использования.

### 6.4. Потенциально узкие места
- Последовательные `MPI_Allreduce` – два отдельных вызова увеличивают задержку; можно было бы объединить `min` и `max` в одну структуру и использовать пользовательскую операцию, но для простоты оставлено как есть.
- `MPI_Scatterv` / `MPI_Gatherv` – на больших объёмах данных могут стать узким местом.
- `STL-потоки` – статическое разбиение оптимально, но при сильном дисбалансе (которого здесь нет) потребовалась бы динамическая балансировка.


## 7. Проверка корректности
Используются те же тестовые наборы, что и для SEQ (см. [seq/report.md](../seq/report.md#5-проверка-корректности)), с применением критерия `CheckRange`:
```cpp
bool CheckRange(const OutType &data) {
  if (data.empty()) return false;
  auto [min_it, max_it] = std::ranges::minmax_element(data);
  return (*min_it == 0 && *max_it == 255) || (*min_it == *max_it);
}
```

#### Используемые тестовые наборы:
- **Валидационные:** пустой вход → `ValidationImpl` возвращает `false`.
- **Функциональные:** синтетические изображения 1×1, 2×2, 3×3, 100×100, 500×500 с низкой контрастностью → `CheckRange` истинно (min=0, max=255).
- **Однородные изображения:** 10×10 и 501×501, заполненные нулями → `CheckRange` истинно (min==max).
- **Реальное изображение:** `data/grayimg.jpg` → после обработки `CheckRange` истинно.

#### Дополнительные проверки:
- При `size_ = 1` и `num_threads = 1` результат идентичен SEQ.
- При любых комбинациях `ranks × threads` выходной вектор на процессе 0 одинаков – проверено сравнением для нескольких конфигураций.

#### Согласованность между процессами:
- Все процессы получают одинаковые `global_min` и `global_max` через `MPI_Allreduce`.
- Каждый процесс обрабатывает свой блок независимо; сборка через `Gatherv` сохраняет порядок пикселей.

## 10. Выводы
#### Гибридная схема MPI+STL оправдана в следующих случаях:
- Изображение очень большое и не помещается в память одного узла – распределение данных между узлами обязательно.
- Доступен кластер с быстрой сетью, где накладные расходы на `Scatterv`/`Gatherv` не доминируют.
- Необходимо использовать все ядра нескольких узлов – гибридный подход даёт лучшее масштабирование, чем чистый MPI (когда каждый MPI-процесс загружает одно ядро), потому что внутри узла потоки работают с общей памятью и не требуют дополнительных коммуникаций.

#### Гибридная версия неэффективна в след. ситуациях:
- На одном узле чистая STL-версия даёт лучшее время без оверхеда на передачу данных.
- При очень большом числе процессов и малом размере задачи – накладные расходы на коммуникацию становятся сопоставимы с вычислениями.
- При `ranks * threads` >> `физических ядер` (oversubscription) эффективность падает до неприемлемых значений.
