## 1. Назначение

ALL-версия предназначена для запуска задачи в MPI-окружении. Она сохраняет общий контракт задачи: на вход подается
ориентированный граф с неотрицательными весами, на выходе получается вектор кратчайших расстояний от стартовой вершины.

Отчет добавлен отдельным pull request от кода ALL. Он не содержит файлов `all/include` и `all/src`, поэтому не
конфликтует с pull request, где реализована сама технология. После объединения обеих веток директория `all` будет
содержать и исходный код, и отчет.

## 2. Схема работы

Реализация использует MPI для синхронизации входных данных и результата между процессами:

1. Rank 0 проверяет исходный граф и передает общий флаг валидности через `MPI_Bcast`.
2. В preprocessing rank 0 упаковывает список ребер в линейный буфер.
3. Метаданные графа (`n`, `source`, количество ребер) рассылаются всем процессам.
4. Упакованный список ребер также передается всем процессам.
5. Каждый процесс строит одинаковый список смежности.
6. Rank 0 запускает алгоритм Дейкстры и получает итоговый вектор расстояний.
7. Размер и содержимое результата рассылаются всем процессам через `MPI_Bcast`.

Такая схема делает поведение pipeline одинаковым для всех rank и упрощает интеграцию с тестовым фреймворком курса.

## 3. Алгоритм вычисления

Вычислительное ядро остается классическим алгоритмом Дейкстры с приоритетной очередью:

- расстояния инициализируются значением `kInfinity`;
- стартовая вершина получает расстояние `0`;
- из очереди извлекается вершина с минимальным текущим расстоянием;
- для всех исходящих ребер выполняется релаксация;
- устаревшие записи очереди отбрасываются.

Алгоритм корректен для графов с неотрицательными весами. Отрицательные веса отклоняются на этапе валидации.

## 4. Межпроцессное взаимодействие

В ALL-версии используются следующие MPI-вызовы:

- `MPI_Comm_rank` для определения rank процесса;
- `MPI_Bcast` для распространения флага валидации;
- `MPI_Bcast` для передачи метаданных графа;
- `MPI_Bcast` для передачи упакованных ребер;
- `MPI_Bcast` для передачи размера и содержимого выходного вектора.

Полноценное распределение вычислений по вершинам в данной версии не используется, потому что последовательный выбор
минимальной вершины в алгоритме Дейкстры создает сильную глобальную зависимость. MPI применяется для корректного
запуска задачи в режиме ALL и синхронизации состояния между процессами.

## 5. Детали реализации

Основные вспомогательные функции:

- `PackEdges` преобразует список `std::tuple<Vertex, Vertex, Weight>` в линейный буфер целых чисел.
- `UnpackEdges` восстанавливает список ребер из переданного буфера.
- `BuildAdjacency` строит список смежности из количества вершин и списка ребер.
- `DijkstraSeq` выполняет сам поиск кратчайших путей.

Состояние задачи хранит:

- `adjacency_` — список смежности;
- `vertices_count_` — количество вершин;
- `source_` — стартовую вершину.

## 6. Проверка корректности

ALL-реализация проверяется теми же функциональными сценариями, что и остальные технологии:

- базовый граф с несколькими вариантами путей;
- граф с недостижимой вершиной;
- граф со стартовой вершиной, отличной от нулевой.

Для каждого случая результат сравнивается с заранее известным вектором расстояний. Локально functional-тесты были
запущены под MPI с 1, 2, 3 и 4 процессами.

## 7. Результаты

Локальный performance-тест использовал разреженный граф из `3000` вершин. Время ALL-версии:

| Режим         | MPI-процессов | Время, сек |
|---------------|---------------|------------|
| task_run      | 4             | 0.000015   |
| task_pipeline | 4             | 0.000397   |

Значения приведены для WSL2-контейнера с GCC 14.2.0. Из-за малого размера тестового графа основную долю времени
составляют накладные расходы pipeline и MPI-синхронизации.

## 8. Вывод

ALL-версия корректно интегрирует задачу в MPI-режим тестового фреймворка и возвращает одинаковый результат на всех
процессах. Для рассматриваемого графа она не ускоряет вычисления, так как сам алгоритм Дейкстры остается
последовательным на rank 0, а MPI добавляет стоимость передачи данных. Основная ценность реализации — поддержка
технологии ALL и единый интерфейс с остальными вариантами задачи.
