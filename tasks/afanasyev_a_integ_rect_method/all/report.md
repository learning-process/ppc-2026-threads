## 1. Контекст

Переносится вычисление трёхмерного интеграла методом прямоугольников из последовательной версии в
параллельную реализацию, комбинирующую STL (std::thread) и TBB (parallel_reduce)

## 2. Постановка задачи

Вычисление трёхмерного интеграла функции exp(-(x^2 + y^2 + z^2)) по единичному кубу методом прямоугольников

## 3. Базовый алгоритм (SEQ)

1 - Инициализация
   1.1 Получаем n из входных данных
   1.2 Вычисляем шаг сетки h = 1.0 / n
   1.3 Создаем массив индексов idx[k_dim] = {0, 0, 0}
   1.4 Создаем массив координат x[k_dim] = {0, 0, 0}
   1.5 Обнуляем сумму sum = 0.0
2 - Обход всех узлов сетки
   2.1 Для текущего набора индексов вычисляем координаты центров ячеек x[dim] = (idx[dim] + 0.5) × h
   2.2 Вычисляем значение подынтегральной функции exp(-(x^2 + y^2 + z^2))
   2.3 Значение добавляем к сумме
   2.4 Инкрементируем индексы с переносом разряда
     2.4.1 Увеличивается idx[0]
     2.4.2 Если idx[0] == n, сброс в 0 и увеличение idx[1]
     2.4.3 Если idx[1] == n, сброс в 0 и увеличение idx[2]
     2.4.4 Если idx[2] == n, цикл завершается
3 - Получение финального результата
   3.1 Объём ячейки volume = h^3
   3.2 Результат result = sum × volume

## 4. Схема распараллеливания (ALL)

Вся итеративная часть распараллеливается благодаря комбинированию STL и TBB:

Двухуровневая декомпозиция:

Первый уровень — STL: разделение по внешнему измерению i, потоки создаются через std::thread;
каждый поток получает диапазон i от start до end, разбиение учитывает remainder для равномерной
загрузки.

Второй уровень — TBB: внутренняя параллелизация для каждого потока; внутри каждого STL-потока
используется tbb::parallel_reduce, TBB дополнительно распараллеливает итерации по i внутри диапазона,
что даёт вложенный параллелизм (nested parallelism).

## 5. Детали реализации

all/include/ops_all.hpp — объявление класса AfanasyevAIntegRectMethodALL
all/src/ops_all.cpp — реализация

```cpp
const int num_threads = std::max(1, ppc::util::GetNumThreads());
const int worker_count = std::min(num_threads, n);

std::vector<std::thread> workers;
std::vector<double> local_sums(worker_count, 0.0);

const int chunk_size = n / worker_count;
const int remainder = n % worker_count;
int start_i = 0;

for (int thread_id = 0; thread_id < worker_count; ++thread_id) {
  const int end_i = start_i + chunk_size + (thread_id < remainder ? 1 : 0);
  workers.emplace_back([thread_id, start_i, end_i, n, h, &local_sums]() {
    local_sums[thread_id] = tbb::parallel_reduce(
      tbb::blocked_range<int>(start_i, end_i), 0.0,
      [&](const tbb::blocked_range<int> &range, double sum) {
        for (int i = range.begin(); i < range.end(); ++i) {
          sum += ComputeSumForI(i, n, h);
        }
        return sum;
      }, std::plus<>());
  });
  start_i = end_i;
}

for (auto &worker : workers) {
  worker.join();
}

const double sum = std::accumulate(local_sums.begin(), local_sums.end(), 0.0);
```

Устранены сразу несколько рисков гонки:
1 - Гонка sum — двухуровневая защита: локальные суммы для каждого STL-потока + parallel_reduce для TBB
2 - Гонка x — `std::array<double, 3> x` создаётся внутри ComputeSumForI, локальна для каждого вызова
3 - Гонка индексов — i, j, k — локальные переменные, приватны для каждой итерации
4 - Гонка local_sums — каждый поток записывает в свой уникальный элемент массива
5 - Автоматическая синхронизация — TBB управляет внутренней синхронизацией, STL управляет внешней

## 6. Проверка корректности

Результат идентичен последовательной версии, включая граничные случаи: проверка n > 0 в
ValidationImpl(), корректная обработка remainder при неравномерном разделении STL-потоков,
правильное масштабирование результата через умножение на volume = h^k_dim, worker_count
ограничен значением n (нельзя создать больше потоков, чем итераций по i).

## 9. Выводы

ALL реализация — компромиссное решение, объединяющее сильные стороны STL (контроль распределения и
синхронизации) и TBB (разделение нагрузки). Она показывает хорошие результаты по сравнению с другими
параллельными реализациями, но при малом размере задачи и малом числе потоков всё ещё уступает OMP;
начиная с 6 потоков может её обгонять.
