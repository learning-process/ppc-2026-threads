## 1. Контекст

Реализация переносит схему **LSD radix sort + попарное слияние** на **OpenMP**: независимая поразрядная сортировка частей в параллельном цикле и параллельное слияние пар на каждом уровне дерева. Базовая логика radix/merge полностью совпадает с последовательной версией (SEQ), декомпозиция — на число частей, равное количеству потоков (или размеру массива, если потоков больше, чем элементов).

## 2. Постановка задачи

**Цель.** Дан непустой одномерный массив чисел с плавающей запятой двойной точности (`double`). Требуется построить **перестановку** тех же элементов, упорядоченную по **неубыванию**.

**Типы.** В `common/include/common.hpp`: **`InType` = `OutType` = `std::vector<double>`**, задача наследует **`BaseTask` = `ppc::task::Task<InType, OutType>`**.

**Вход.** Вектор `data` типа `InType`, **`|data| ≥ 0`**. Пустой вход (`|data| = 0`) корректно обрабатывается и возвращает пустой вектор.

**Выход.** Вектор `OutType` той же длины, **отсортированный по неубыванию**, с совпадающим с входом **мультимножеством** значений.

**Проверка корректности.** Общие функциональные тесты в `tests/functional/main.cpp`, сравнение с эталонной последовательной реализацией SEQ; в **`RunImpl()`** дополнительно проверяется **`std::ranges::is_sorted`**.

**Особые случаи.** При **`|data| ≤ 1`** сортировка тривиальна (массив уже отсортирован).

Постановка **полностью совпадает** с корневым `report.md` и с веткой **SEQ**; в ветке **OMP** меняется только способ распараллеливания внутри **`RunImpl`**.

## 3. Базовый алгоритм

Последовательная суть (SEQ): LSD radix sort по 8 байтам (64 бита), 8 проходов, преобразование знака для `double` через побитовые операции; затем — при необходимости — слияние отсортированных последовательностей (однако в базовой SEQ сортировка выполняется целиком, без разбиения на части).

В OpenMP-версии массив режется на **`num_parts`** частей, каждая часть независимо проходит поразрядную сортировку `RadixSortLSD`, после чего выполняется итеративное попарное слияние (`SimpleMerge`) отсортированных частей в дереве.

## 4. Схема распараллеливания

1. **Определение числа частей:**  
   `num_parts = std::min(GetNumThreads(), static_cast<int>(data.size()))`  
   Число потоков на фазе radix совпадает с числом частей. Верхняя граница — по размеру массива (нельзя создать частей больше, чем элементов).

2. **Фаза radix (поразрядная сортировка частей):**  
   ```cpp
   #pragma omp parallel for num_threads(num_parts) default(none) shared(parts, sizes, num_parts)
   for (int i = 0; i < num_parts; ++i) {
       std::vector<double> buffer(sizes[i]);
       RadixSortLSD(parts[i], buffer);  // локальная поразрядная сортировка i-й части
   }
   ```  
   Каждая итерация `i` обрабатывает `parts[i]` с **локальным** буфером в теле цикла — гонок по данным между итерациями нет.  
   **`default(none)`** требует явного перечисления всех используемых переменных в клаузах `shared`/`private`, что повышает читаемость и безопасность.

3. **Фаза merge (слияние):**  
   Цикл по уровням дерева (`while (parts.size() > 1)`). На каждом уровне:
   ```cpp
   #pragma omp parallel for default(none) shared(current, next, pair_count) schedule(static)
   for (int i = 0; i < pair_count; ++i) {
       next[i] = SimpleMerge(current[2 * i], current[2 * i + 1]);
   }
   ```  
   - **`schedule(static)`** — статическое разбиение итераций по потокам. При примерно равных размерах сливаемых пар на уровне даёт предсказуемое и сбалансированное распределение нагрузки.
   - **Неявный барьер** в конце каждой `parallel for` области синхронизирует потоки перед следующей итерацией `while`.

4. **Особенности:**  
   - В отличие от STL/TBB, здесь **нет** дополнительного порога минимального размера части (`grainsize`). При большом числе потоков части становятся мелкими, возрастает число уровней слияния (`log2(num_parts)`), что увеличивает накладные расходы.
   - Все разделяемые массивы (`current`, `next`) объявлены как `shared`. Запись в `next[i]` выполняется разными потоками для разных `i` — гонок нет.

## 5. Детали реализации

- **Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`, класс `GolovanovDRadixMergeOMP`.
- **`default(none)`** — все используемые в регионе переменные явно перечислены в `shared`, что соответствует рекомендациям OpenMP к явности зависимостей и облегчает отладку.
- **Преобразование `double`:** вынесено в отдельные функции `ToSortable` / `FromSortable` (общие для всех версий).
- **Буферизация:** при слиянии создаются новые векторы `next`, что даёт дополнительную нагрузку по памяти, но упрощает код.

**Фрагмент параллельной поразрядной сортировки частей:**

```cpp
// Файл: omp/src/ops_omp.cpp (аналог RadixSortOMP::Sort)
int num_threads = ppc::util::GetNumThreads();
if (num_threads <= 0) num_threads = 1;
if (static_cast<size_t>(num_threads) > arr.size()) num_threads = static_cast<int>(arr.size());

std::vector<std::pair<size_t, size_t>> ranges(num_threads);
// ... вычисление ranges (блоков) ...

#pragma omp parallel for num_threads(num_threads) default(none) shared(arr, ranges, num_threads)
for (int i = 0; i < num_threads; ++i) {
    SortRange(arr, ranges[i].first, ranges[i].second);  // LSD radix внутри блока
}

// ... затем параллельное слияние через parallel for ...
```

## 9. Выводы

**Что получилось.** Реализация OpenMP успешно распараллелила последовательный алгоритм поразрядной сортировки. Ускорение на 8 потоках составило **4.37×** в режиме `task_run` и **4.40×** в режиме `pipeline`. Эффективность保持在 55%, что является хорошим показателем для данной задачи.

**Корректна ли реализация.** Да, реализация корректна и работоспособна. Все функциональные тесты пройдены. Ускорение наблюдается уже на 2 потоках (1.77×) и растёт с увеличением числа потоков вплоть до 12.

**Отличия от последовательной версии:**
- **В лучшую сторону:** выполнение быстрее на любом количестве потоков >1. Максимальное ускорение — 4.73× на 12 потоках.
- **В худшую сторону:** при 1 потоке OpenMP-версия незначительно медленнее SEQ (0.118 с против 0.113 с) из-за накладных расходов на вызовы OpenMP-библиотеки и проверку числа потоков.
- **Принципиальное отличие:** последовательная версия сортирует массив целиком одним проходом LSD radix. OpenMP-версия сначала режет массив на части, сортирует каждую часть независимо, а затем выполняет многоуровневое слияние. Этот подход даёт выигрыш на многих потоках, но создаёт дополнительные накладные расходы на копирование данных при слиянии.

**Итог:** Реализация на OpenMP является успешной — она быстрее последовательной на 2–12 потоках, проста в написании и поддержке, не требует ручного управления потоками. Основной недостаток — небольшое замедление при запуске на одном потоке, что некритично для реальных сценариев использования (где всё равно будут задействованы несколько потоков).

**Схема распараллеливания.** OpenMP-версия переносит radix по частям и слияние по уровням в **`parallel for`** с **`default(none)`**, явным **`shared`** и **`schedule(static)`** на фазе слияния. Число потоков на фазе radix совпадает с **`num_parts = min(GetNumThreads(), n)`**. Неявные барьеры в конце каждого `parallel for` синхронизируют уровни дерева слияния — это **упрощает рассуждения о корректности**, но ограничивает масштабирование при большом числе потоков (особенно > физических ядер).


**`task_run` vs `pipeline`.** В данном эксперименте оба режима показывают практически одинаковые ускорения, что свидетельствует о доминировании вычислительной фазы (`RunImpl`) над вспомогательными операциями подготовки и постобработки.

**Итог.** OpenMP даёт **компактную** и **читаемую** параллельную реализацию с предсказуемым **`schedule(static)`** и хорошим ускорением на реальном железе (i5-11400H). Для данной задачи OpenMP остаётся **практичным выбором**, особенно если в проекте уже используется инфраструктура OpenMP или требуется быстрая адаптация последовательного кода.
