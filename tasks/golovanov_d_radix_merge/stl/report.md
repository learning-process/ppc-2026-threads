## 1. Контекст

Реализация переносит схему **LSD radix sort + попарное слияние** на ручное управление
потоками через **`std::thread`**. В отличие от OpenMP, здесь нет автоматического
планировщика задач — программист самостоятельно определяет разбиение данных на блоки,
создаёт потоки, ожидает их завершения и организует слияние результатов. Такой подход
даёт максимальный контроль над распределением нагрузки, но требует более тщательной
работы с синхронизацией и памятью.

## 2. Постановка задачи

**Цель.** Дан непустой одномерный массив чисел с плавающей запятой двойной точности (`double`).
Требуется построить **перестановку** тех же элементов, упорядоченную по **неубыванию**.

**Типы.** В `common/include/common.hpp`: **`InType` = `OutType` = `std::vector<double>`**,
задача наследует **`BaseTask` = `ppc::task::Task<InType, OutType>`**.

**Вход.** Вектор `data` типа `InType`, **`|data| ≥ 0`**. Пустой вход корректно обрабатывается.

**Выход.** Вектор `OutType` той же длины, **отсортированный по неубыванию**.

**Проверка корректности.** Функциональные тесты, сравнение с SEQ; в **`RunImpl()`**
проверяется **`std::ranges::is_sorted`**.

**Особые случаи.** При **`|data| ≤ 1`** сортировка тривиальна.

## 3. Базовый алгоритм

Последовательная суть (SEQ): LSD radix sort по 8 байтам (64 бита), 8 проходов,
преобразование знака для `double`.

В STL-версии массив режется на **`num_threads`** блоков, каждый блок сортируется независимо
в отдельном потоке, после чего выполняется итеративное попарное слияние отсортированных
блоков (снова с созданием потоков на каждом уровне).

## 4. Схема распараллеливания

1. **Определение числа потоков:**  
   `num_threads = std::min(GetNumThreads(), static_cast<int>(data.size()))`

2. **Разбиение на блоки (BuildRanges):**  
   Массив делится на `num_threads` непрерывных диапазонов примерно равного размера.  
   Блоки хранятся как пары `(begin_index, end_index)`.

3. **Фаза radix (параллельная сортировка блоков):**  
   Для каждого блока создаётся `std::thread`, который вызывает `SortRange(arr, left, right)`.  

   ```cpp
   std::vector<std::thread> workers;
   for (const auto &range : ranges) {
       workers.emplace_back([&arr, range]() {
           RadixSortSTL::SortRange(arr, range.first, range.second);
       });
   }
   for (auto &worker : workers) worker.join();
   ```  

   Все потоки работают независимо над разными участками памяти — гонок данных нет.

4. **Копирование отсортированных блоков (CopyParts):**  
   После сортировки каждый блок копируется в отдельный вектор `parts[i]`.
   Это необходимо для последующего слияния, так как слияние «на месте» потребовало бы
   сложного управления памятью.

5. **Фаза merge (параллельное слияние):**  
   Цикл по уровням дерева (`while (parts.size() > 1)`). На каждом уровне:
   - Определяется количество пар для слияния: `pair_count = parts.size() / 2`
   - Создаются потоки — по одному на пару (или меньше, если пар больше, чем потоков)
   - Каждый поток сливает свою пару векторов в новый вектор `next[i]`
   - После завершения всех потоков выполняется `join`
   - Нечётный «хвостовой» элемент (если есть) переносится в `next` без изменений
   - `parts = std::move(next)` — переход на следующий уровень

   **Важное отличие от TBB/OpenMP:** здесь нет встроенного планировщика, который
   автоматически балансировал бы нагрузку. Если размеры сливаемых пар сильно
   различаются, одни потоки могут завершить работу раньше других.

6. **Завершение:** После завершения слияния `parts[0]` содержит полностью отсортированный
   массив, который перемещается в `arr`.

## 5. Детали реализации

- **Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`,
  класс `GolovanovDRadixMergeSTL`.
- **Функция `GetThreadCount(size)`:** определяет число потоков с учётом размера массива
  (не больше элементов).
- **Функция `BuildRanges(size, num_threads)`:** вычисляет границы блоков.
- **`SortParts(arr, ranges)`:** запускает потоки для поразрядной сортировки блоков.
- **`CopyParts(arr, ranges)`:** копирует блоки в отдельные векторы.
- **`MergeStep(parts)`:** один уровень слияния — создаёт потоки, каждый сливает свою пару.

**Фрагмент фазы слияния:**

```cpp
// Файл: radix_sort_stl.cpp
std::vector<std::vector<double>> MergeStep(const std::vector<std::vector<double>> &parts) {
    const size_t pair_count = parts.size() / 2;
    std::vector<std::vector<double>> next((parts.size() + 1) / 2);
    std::vector<std::thread> workers;
    workers.reserve(pair_count);

    for (size_t i = 0; i < pair_count; ++i) {
        workers.emplace_back([&parts, &next, i]() {
            next[i] = RadixSortSTL::Merge(parts[2 * i], parts[(2 * i) + 1]);
        });
    }

    for (auto &worker : workers) {
        worker.join();
    }

    if ((parts.size() % 2U) != 0U) {
        next.back() = parts.back();
    }

    return next;
}
```

## 9. Выводы

**Что получилось.** Реализация STL успешно распараллелила последовательный алгоритм.
Ускорение на 8 потоках составило **4.73×** в режиме `task_run` и **4.76×** в режиме
`pipeline`. Эффективность — 59–60%, что является хорошим показателем для задачи сортировки.

**Корректна ли реализация.** Да, реализация корректна и работоспособна. Все функциональные
тесты пройдены. Ручное управление потоками позволило добиться ускорения, сопоставимого
с OpenMP и TBB.

**Отличия от последовательной версии:**

- **В лучшую сторону:** выполнение значительно быстрее на 2–12 потоках.
  Максимальное ускорение — 5.17× на 12 потоках (эффективность 43%).
- **В худшую сторону:** при 1 потоке STL-версия незначительно медленнее SEQ
  (0.116 с против 0.113 с) из-за накладных расходов на создание и проверку структур
  для потоков (даже когда поток фактически один).

**Специфика реализации:**

- Нет автоматического планировщика — потоки создаются и уничтожаются вручную.
- Разбиение на блоки — статическое (фиксированные диапазоны).
- Слияние — с созданием новых потоков на каждом уровне.
- Копирование данных (`CopyParts`) создаёт дополнительную нагрузку на память,
  но упрощает код слияния.

**Итог:** Реализация на `std::thread` является успешной. Она быстрее последовательной
версии на всех протестированных конфигурациях (2–12 потоков). Ручное управление потоками
даёт полный контроль над распределением нагрузки, но требует больше кода по сравнению
с OpenMP. Для данной задачи STL-версия показывает результаты, близкие к OpenMP и TBB.
