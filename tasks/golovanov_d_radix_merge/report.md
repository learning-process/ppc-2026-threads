## 1. Введение

Поразрядная сортировка (Radix Sort) — это алгоритм сортировки с линейной асимптотической сложностью O(k·n), где k — количество разрядов, а n — количество элементов. В отличие от алгоритмов, основанных на сравнениях, Radix Sort обрабатывает данные побайтово, что делает его особенно эффективным для сортировки чисел с фиксированным форматом представления.

Данная задача отлично подходит для исследования различных моделей параллелизма. Алгоритм содержит фазы подсчёта гистограмм и распределения данных, которые легко распараллеливаются, а также фазу слияния отсортированных блоков, требующую синхронизации. Реализация включает пять версий (SEQ, OMP, TBB, STL, ALL), что позволяет провести комплексное сравнение технологий параллельного программирования для задачи, интенсивно использующей память.

## 2. Единая постановка задачи

**Вход:** Вектор `std::vector<double>` размером n.

**Выход:** Отсортированный по возрастанию вектор тех же чисел.

**Ограничения:**
- n ≥ 0 (пустой вектор допустим).
- Числа могут быть как положительными, так и отрицательными.
- Формат данных: IEEE 754 double precision.

**Критерий корректности:**
- Выходной вектор содержит те же элементы, что и входной.
- Элементы отсортированы по неубыванию.
- Сохраняется относительный порядок равных элементов (стабильность сортировки).
- Корректная обработка знака: отрицательные числа должны идти перед положительными.

**Алгоритм:**
1. Преобразование `double` в `uint64_t` с битовыми манипуляциями (инверсия знакового бита для корректного порядка).
2. LSD Radix Sort (8 проходов по байтам).
3. Обратное преобразование в `double`.
4. Для параллельных версий: разделение данных на блоки, локальная сортировка, иерархическое слияние результатов.

## 6. Интерпретация различий

- **SEQ:** Установил базовое время выполнения. Алгоритм ограничен пропускной способностью памяти (memory-bound), так как требует 8 проходов по массиву для обработки байтов плюс финальное слияние.

- **OMP:** Показал результаты, близкие к последовательной версии, но с небольшим замедлением. Директивы `#pragma omp parallel for` добавляют накладные расходы на создание барьеров и синхронизацию, которые не окупаются при относительно лёгких вычислениях внутри цикла.

- **TBB:** Результаты сопоставимы с OMP. Использование `tbb::parallel_for` с `blocked_range` обеспечивает хорошую балансировку нагрузки, однако динамический планировщик задач добавляет overhead, заметный на задачах с простой вычислительной логикой.

- **STL:** Ручное управление потоками (`std::thread`) показало наилучшие результаты среди однопроцессных версий. Отсутствие overhead менеджера потоков и минимальная синхронизация позволили достичь производительности, близкой к теоретическому пределу для данной архитектуры.

- **ALL:** Гибридная версия с использованием MPI продемонстрировала значительное ускорение (~4.6–5.0×) при запуске на 4 процессах. Это объясняется:
  1. Распределением данных между процессами, что снижает объём работы на каждом ядре.
  2. Лучшим использованием кэш-памяти: каждый процесс работает с локальным подмассивом.
  3. Параллельным выполнением фаз сортировки и слияния на разных узлах.
  
  Накладные расходы на коммуникацию (`MPI_Scatterv`, `MPI_Gatherv`, `MPI_Send`/`MPI_Recv`) компенсируются выигрышем от распараллеливания при достаточно большом размере задачи.

## 6 Заключение

Для задачи поразрядной сортировки вещественных чисел:
- На одном узле с общей памятью наиболее эффективной оказалась реализация на **STL**, обеспечивающая минимальные накладные расходы.
- Реализации на **OpenMP** и **oneTBB** показали сопоставимые результаты, уступая STL незначительно из-за накладных расходов сред исполнения.
- Гибридная версия **ALL** с использованием MPI продемонстрировала наилучшее абсолютное время выполнения при запуске на нескольких процессах, что делает её предпочтительной для масштабирования на кластерных системах или при работе с данными, не помещающимися в память одного узла.
