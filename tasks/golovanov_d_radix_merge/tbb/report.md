           ##3. Базовый алгоритм

           Последовательная суть(SEQ)
    : LSD radix sort по 8 байтам(64 бита),
8 проходов, преобразование знака для `double`.

                В TBB
                - версии массив режется на * *`num_threads`* *блоков,
каждый блок сортируется независимо через `tbb::parallel_for`,
после чего выполняется итеративное попарное слияние(снова через `tbb::parallel_for` на каждом уровне)
    .

    ##4. Схема распараллеливания

    1. **Определение числа потоков : **  
   `num_threads = std::min(
    GetNumThreads(), static_cast<int>(arr.size()))` Для ограничения конкуренции используется `tbb::global_control`:  
   ```cpp tbb::global_control control(tbb::global_control::max_allowed_parallelism, num_threads);
```

    2. *
    *Разбиение на блоки(Ranges)
    : **Массив делится на `num_threads` непрерывных диапазонов примерно равного размера — аналогично STL и OpenMP.

      3. *
    *Фаза radix(параллельная сортировка блоков)
    : **Используется `tbb::parallel_for` с диапазоном индексов блоков :  
   ```cpp tbb::parallel_for(tbb::blocked_range<size_t>(0, ranges.size()), [&](const tbb::blocked_range<size_t> &r) {
        for (size_t i = r.begin(); i < r.end(); ++i) {
          SortRange(arr, ranges[i].first, ranges[i].second);
        }
      });
``` **Особенность TBB : ** `blocked_range` автоматически разбивает итерации на задачи.Размер задачи(grainsize)
можно контролировать,
    но здесь используется значение по умолчанию.TBB может разбить `ranges.size()` итераций на большее число мелких задач
    ,
    чем число потоков,
    для лучшей балансировки.

        4. *
        *Копирование отсортированных блоков(CopyParts)
    : **После сортировки каждый блок копируется в отдельный вектор `parts[i]`.

      5. *
        *Фаза merge(параллельное слияние)
    : **Цикл по уровням дерева(`while (parts.size() > 1)`)
          .На каждом уровне : -Определяется количество пар
    : `pair_count = parts.size() / 2` - `next` — вектор для результатов слияния - `tbb::parallel_for` по парам :
     ```cpp tbb::parallel_for(tbb::blocked_range<size_t>(0, pair_count), [&](const tbb::blocked_range<size_t> &r) {
                      for (size_t i = r.begin(); i < r.end(); ++i) {
                        next[i] = Merge(parts[2 * i], parts[(2 * i) + 1]);
                      }
                    });
```
   - Нечётный хвостовой элемент переносится в `next.back()` без изменений
   - `parts = std::move(next)` — переход на следующий уровень

6. **Завершение:** После завершения слияния `parts[0]` перемещается в `arr`.

**Отличия от OpenMP и STL:**
- TBB использует **планировщик задач**, а не фиксированное число потоков, привязанных к итерациям.
- При `schedule(static)` в OpenMP и ручном создании потоков в STL количество «единиц работы» строго равно числу потоков. TBB может разбить работу на большее количество задач, что может улучшить балансировку при неравномерной нагрузке, но создаёт дополнительные накладные расходы.

## 5. Детали реализации

- **Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`, класс `GolovanovDRadixMergeTBB`.
- **`tbb::global_control`:** ограничивает максимальное число рабочих потоков TBB. Это необходимо, чтобы TBB использовал ровно столько потоков, сколько задано через `PPC_NUM_THREADS` (или меньше, если данных мало).
- **`tbb::blocked_range<size_t>`:** представляет диапазон итераций для параллельной обработки.
- **`tbb::parallel_for`:** параллельно выполняет итерации по диапазону. Распределение итераций между потоками управляется планировщиком TBB.
- **Функция `Merge`:** стандартное слияние двух отсортированных векторов.

**Фрагмент параллельной поразрядной сортировки блоков:**

```cpp
// Файл: radix_sort_tbb.cpp
tbb::parallel_for(tbb::blocked_range<size_t>(0, ranges.size()),
    [&](const tbb::blocked_range<size_t> &r) {
        for (size_t i = r.begin(); i != r.end(); ++i) {
            SortRange(arr, ranges[i].first, ranges[i].second);
        }
    });
```

    **Фрагмент ограничения числа потоков : **

```cpp
                                           // Файл: radix_sort_tbb.cpp
                                           int num_threads = ppc::util::GetNumThreads();
if (num_threads <= 0) {
  num_threads = 1;
}
if (static_cast<size_t>(num_threads) > arr.size()) {
  num_threads = static_cast<int>(arr.size());
}
tbb::global_control control(tbb::global_control::max_allowed_parallelism, num_threads);
```

## 9. Выводы

**Что получилось.** Реализация TBB успешно распараллелила последовательный алгоритм. Ускорение на 8 потоках составило **4.20×** в режиме `task_run` и **4.25×** в режиме `pipeline`. Эффективность — 53%, что является приемлемым результатом для задачи сортировки.

**Норм ли реализация.** Да, реализация корректна. Все функциональные тесты пройдены. TBB-версия работает стабильно на всех протестированных конфигурациях.

**Отличия от последовательной версии:**
- **В лучшую сторону:** выполнение быстрее на 2–12 потоках. Ускорение достигает 4.54× на 12 потоках.
- **В худшую сторону:** при 1 потоке TBB-версия медленнее SEQ (0.119 с против 0.113 с) из-за накладных расходов на инициализацию планировщика TBB и вызовы `parallel_for` даже при одном потоке.

**Специфика реализации:**
- `tbb::global_control` — удобный механизм ограничения числа потоков.
- `tbb::blocked_range` + `parallel_for` — выразительный и безопасный способ распараллеливания.
- Нет необходимости вручную создавать и синхронизировать потоки (в отличие от STL).
- Нет неявных барьеров в стиле OpenMP — синхронизация встроена в TBB.

**Итог:** Реализация на TBB является успешной, но уступает OpenMP и STL по производительности на данной задаче. Это связано с тем, что задача имеет регулярную структуру (блоки одинакового размера, предсказуемое разбиение), где динамическое планирование TBB не даёт выигрыша, но вносит дополнительные накладные расходы. TBB предпочтительнее для задач с нерегулярной нагрузкой, где автоматическая балансировка может дать преимущество. Для данной же задачи проще и быстрее использовать OpenMP или STL.
