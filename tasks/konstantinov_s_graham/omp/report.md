## 1. Контекст

Версия на OpenMP разрабатывалась с целью ускорить линейные фазы алгоритма Грэхема (`O(n)`) за счёт распределения работы
между потоками. Базовым эталоном производительности служит последовательная реализация (`≈ 0.510 с` на `n = 2 250 000` точках).

---

## 2. Постановка задачи

Та же, что и для SEQ: по множеству `n` точек на плоскости построить выпуклую оболочку методом Грэхема.
В тестах производительности используется сетка `1500 × 1500 = 2 250 000` точек.

---

## 3. Базовый алгоритм

Алгоритм идентичен последовательной версии и состоит из пяти этапов:

1. **RemoveDuplicates** — дедупликация точек с допуском `kKEps`: `O(n log n)`.
2. **FindAnchorIndex** — линейный поиск точки с минимальной `y` (при равенстве — минимальной `x`): `O(n)`.
3. **CollectAndSortIndices** — сборка индексов и сортировка по полярному углу от опорной точки
через векторное произведение: `O(n log n)`.
4. **AllCollinearWithAnchor** — проверка вырожденного случая (все точки коллинеарны): `O(n)`.
5. **BuildHullFromSorted** — построение оболочки стеком: `O(n)`.

Суммарная сложность определяется сортировкой — `O(n log n)`.

---

## 4. Схема распараллеливания

Параллелизм введён в два `O(n)` этапа. Сортировка и построение стека оставлены последовательными,
так как требуют строгого глобального порядка обработки данных.

### 4.1. Поиск опорной точки (`FindAnchorIndex`)

Реализована схема «локальный минимум + критическая секция»: каждый поток ищет наилучшую точку в своём диапазоне,
атем обновляет глобальный результат.

```cpp
size_t idx = 0;
#pragma omp parallel default(none) shared(xs, ys, idx)
{
    size_t local_idx = idx;

#pragma omp for nowait
    for (size_t i = 1; i < xs.size(); ++i) {
        if (ys[i] < ys[local_idx] - kKEps ||
            (std::abs(ys[i] - ys[local_idx]) < kKEps &&
             xs[i] < xs[local_idx] - kKEps))
            local_idx = i;
    }

#pragma omp critical
    {
        if (ys[local_idx] < ys[idx] - kKEps ||
            (std::abs(ys[local_idx] - ys[idx]) < kKEps &&
             xs[local_idx] < xs[idx] - kKEps))
            idx = local_idx;
    }
}
```

Директива `default(none)` предотвращает случайный захват переменных, `nowait` убирает
лишний барьер внутри `for` — синхронизация всё равно обеспечивается `#pragma omp critical` и завершением блока `parallel`.

### 4.2. Заполнение массива индексов (`CollectAndSortIndices`)

Инициализация вектора `idxs` (размер `n − 1`) распараллелена директивой `#pragma omp parallel for`.
Каждый поток независимо заполняет свою часть: если `i < anchor_idx` — записывает `i`,
если `i > anchor_idx` — записывает `i` в позицию `i − 1`.

```cpp
std::vector<size_t> idxs(xs.size() - 1);
#pragma omp parallel for default(none) shared(xs, idxs, anchor_idx)
for (size_t i = 0; i < xs.size(); ++i) {
    if (i < anchor_idx)      idxs[i]     = i;
    else if (i > anchor_idx) idxs[i - 1] = i;
}
```

### 4.3. Проверка коллинеарности (`AllCollinearWithAnchor`)

Линейный проход по отсортированным индексам распараллелен с флагом `result`, защищённым `#pragma omp atomic write`.
Ранняя остановка не гарантирована (потоки продолжают работу даже после нахождения неколлинеарной тройки),
но результат остаётся корректным.

```cpp
bool result = true;
#pragma omp parallel for default(none) shared(xs, ys, sorted_idxs, anchor_idx, result)
for (size_t i = 1; i < sorted_idxs.size(); ++i) {
    if (std::abs(CrossVal(xs, ys, anchor_idx,
                          sorted_idxs[0], sorted_idxs[i])) > kKEps) {
#pragma omp atomic write
        result = false;
    }
}
```

## 9. Анализ результатов

Причина отсутствия ускорения — структура алгоритма.
По закону Амдала теоретическое ускорение ограничено долей распараллеленного кода:

```math
S = 1 / (f_{seq} + f_{par} / p)
```

Оценка долей на данном наборе:

- `RemoveDuplicates` (сортировка + unique): `≈ 40%` — последовательная;
- `CollectAndSortIndices` (сортировка): `≈ 40%` — последовательная;
- `FindAnchorIndex` + заполнение `idxs` + `AllCollinearWithAnchor`: `≈ 20%` — распараллеленные.

При `p = 4` теоретический потолок ускорения: `1 / (0.80 + 0.20 / 4) ≈ 1.18`.
На практике его не достичь из-за накладных расходов OpenMP.

## 10. Выводы

OMP-версия корректна, но не даёт ускорения: параллелизуются лишь линейные фазы,
тогда как доминирующая сортировка остаётся последовательной.
Для получения реального прироста производительности необходима параллельная сортировка.
