## 1. Контекст

Версия на `std::thread` реализует параллелизм вручную, без сторонних библиотек: используется только
стандартная библиотека C++. Это позволяет явно управлять декомпозицией данных, размером блоков и числом потоков,
но требует написания вспомогательной инфраструктуры там, где TBB и OMP предоставляют готовые примитивы.
Базовый эталон — последовательная реализация (`≈ 0.510 с`).

---

## 2. Постановка задачи

Та же, что и для SEQ: по множеству `n` точек на плоскости построить выпуклую оболочку методом Грэхема.
В тестах производительности используется сетка `1500 × 1500 = 2 250 000` точек.

---

## 3. Базовый алгоритм

Алгоритм идентичен последовательной версии и состоит из пяти этапов:

1. **RemoveDuplicates** — дедупликация точек с допуском `kKEps`: `O(n log n)`.
2. **FindAnchorIndex** — линейный поиск опорной точки: `O(n)`.
3. **CollectAndSortIndices** — сборка индексов и сортировка по полярному углу: `O(n log n)`.
4. **AllCollinearWithAnchor** — проверка вырожденного случая: `O(n)`.
5. **BuildHullFromSorted** — построение оболочки стеком: `O(n)`.

---

## 4. Схема распараллеливания

Общий паттерн во всех параллельных фазах: вычислить число потоков, нарезать данные на блоки, запустить `std::thread`
на каждый блок, выполнить `join`, объединить локальные результаты.

Число потоков определяется вспомогательной функцией `GetThreadCount`:

```cpp
size_t KonstantinovAGrahamSTL::GetThreadCount(size_t n) {
    const size_t hw = std::max<size_t>(1,
        static_cast<size_t>(std::thread::hardware_concurrency()));
    return std::min(hw, n);
}
```

Это гарантирует, что не создаётся больше потоков, чем точек, и не превышается аппаратный параллелизм.

### 4.1. Поиск опорной точки (`FindAnchorIndex`)

Каждый поток вызывает `FindLocalAnchor` на своём диапазоне индексов и сохраняет результат в `local_results[thread_idx]`.
После `join` всех потоков главный поток линейно проходит `local_results` и выбирает глобальный минимум.

```cpp
const size_t block_size = (xs.size() + thread_count - 1) / thread_count;

for (size_t thread_idx = 0; thread_idx < thread_count; ++thread_idx) {
    const size_t begin = thread_idx * block_size;
    const size_t end   = std::min(xs.size(), begin + block_size);
    workers.emplace_back([&, thread_idx, begin, end]() {
        local_results[thread_idx] = FindLocalAnchor(xs, ys, begin, end);
    });
}
for (auto &w : workers) w.join();

size_t best = local_results[0];
for (size_t t = 1; t < thread_count; ++t)
    if (IsLowerAnchor(xs, ys, local_results[t], best)) best = local_results[t];
```

Гонок нет: потоки пишут в разные позиции `local_results`, слияние выполняется в одном потоке.

### 4.2. Заполнение массива индексов (`CollectAndSortIndices`)

Диапазон `[0, n)` разбивается на блоки,
каждый поток вызывает `FillIndexRange` — ту же функцию, что и в последовательном случае:

```cpp
void KonstantinovAGrahamSTL::FillIndexRange(
        std::vector<size_t> &idxs, size_t begin, size_t end, size_t anchor_idx) {
    for (size_t i = begin; i < end; ++i) {
        if (i < anchor_idx)      idxs[i]     = i;
        else if (i > anchor_idx) idxs[i - 1] = i;
    }
}
```

Параллельный запуск оформлен в `FillIndicesParallel`.
После завершения потоков применяется стандартная `std::ranges::sort` — однопоточная, как и в SEQ-версии.

### 4.3. Проверка коллинеарности (`AllCollinearWithAnchor`)

Используется `std::atomic<bool> result{true}`.
Каждый поток вызывает `CheckCollinearRange` на своём диапазоне;
при нахождении несоллинеарной тройки записывает `false` атомарной операцией:

```cpp
std::atomic<bool> result{true};

workers.emplace_back([&, begin, end]() {
    if (!CheckCollinearRange(xs, ys, anchor_idx, sorted_idxs, begin, end))
        result.store(false);
});
```

Ранняя остановка на уровне всей задачи не реализована (потоки не проверяют флаг `result` в процессе работы), однако
внутри `CheckCollinearRange` есть `return false` при первом несоответствии, что сокращает работу каждого потока.

### 4.4. Последовательные фазы

`RemoveDuplicates` и `BuildHullFromSorted` оставлены последовательными по тем же причинам, что и в других реализациях.
`std::ranges::sort` в `CollectAndSortIndices` также выполняется в одном потоке.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Функции `IsLowerAnchor`, `FindLocalAnchor`, `FillIndexRange`, `CheckCollinearRange` выделены как отдельные методы класса,
что делает тело каждого потока компактным и легко тестируемым.
Все `std::thread` создаются с `workers.reserve(thread_count)`, избегая лишних перевыделений памяти.

В параллельных фазах применяется явная проверка: если `thread_count <= 1` или данных меньше,
чем `thread_count * 2`, фаза выполняется последовательно — без создания потоков. Это предотвращает деградацию на малых входах.

```cpp
if (thread_count <= 1 || xs.size() < (thread_count * 2)) {
    return FindLocalAnchor(xs, ys, 0, xs.size());
}
```

Результаты STL-версии близки к OMP: ускорение фактически отсутствует,
с небольшим ростом времени при увеличении числа потоков. Это объясняется тем, что, как и в OMP-версии,
параллелизуются только `O(n)` фазы, а доминирующая `O(n log n)` сортировка остаётся однопоточной.

По сравнению с TBB STL показывает чуть более высокое время:
накладные расходы на ручное управление потоками (`std::thread` + `join`) несколько выше, чем у планировщика задач TBB.

## 9. Выводы

STL-реализация демонстрирует корректный и переносимый подход к написанию многопоточного кода без внешних зависимостей.
Явная декомпозиция данных и пороговая защита от малых входов делают код устойчивым, однако ускорения добиться не удаётся
по той же причине, что и в OMP: узкое место — однопоточная сортировка.
Для данной задачи `std::thread` уступает TBB как по простоте кода, так и по производительности.
