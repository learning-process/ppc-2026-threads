## 1. Контекст

Версия ALL объединяет два уровня параллелизма: межпроцессный (MPI) и внутрипроцессный (TBB).
Входные данные распределяются между MPI-процессами, каждый процесс строит локальную выпуклую оболочку своего
подмножества точек с применением TBB,
затем локальные оболочки объединяются на процессе 0 финальным проходом алгоритма Грэхема. Это единственная реализация,
демонстрирующая реальное ускорение: `≈ 3.4×` относительно SEQ на 4 процессах.

---

## 2. Постановка задачи

Та же, что и для SEQ: по множеству `n` точек на плоскости построить выпуклую оболочку методом Грэхема.
В тестах производительности используется сетка `1500 × 1500 = 2 250 000` точек.

---

## 3. Базовый алгоритм

Алгоритм на каждом процессе идентичен TBB-версии:

1. **RemoveDuplicates** — дедупликация точек: `O(n log n)`.
2. **FindAnchorIndex** (`tbb::parallel_reduce`) — поиск опорной точки: `O(n)`.
3. **CollectAndSortIndices** (`tbb::parallel_for` + `tbb::parallel_sort`) — сортировка по полярному углу: `O(n log n)`.
4. **AllCollinearWithAnchor** (`tbb::parallel_reduce`) — проверка вырождения: `O(n)`.
5. **BuildHullFromSorted** — построение стеком: `O(n)`.

Вся логика вынесена в вспомогательную функцию `BuildHullFromCoords`,
которая вызывается как для локальных подмножеств, так и для итогового слияния оболочек.

---

## 4. Схема распараллеливания

### 4.1. Рассылка размера задачи

Процесс 0 рассылает общий размер входного массива всем остальным процессам через `MPI_Bcast`.
Тип `uint64_t` выбран для переносимости между платформами с разным `size_t`.

```cpp
MPI_Bcast(&total_size_u64, 1, MPI_UINT64_T, 0, MPI_COMM_WORLD);
```

Если `total_size == 0` или `proc_num == 1`, задача решается процессом 0 полностью, результат рассылается остальным.

### 4.2. Распределение точек (`ScatterInput`)

Точки распределяются между процессами блоками примерно равного размера.
Остаток от деления `total_size % proc_num` распределяется по одной точке на первые `remainder` процессов:

```cpp
const size_t base      = total_size / static_cast<size_t>(proc_num_);
const size_t remainder = total_size % static_cast<size_t>(proc_num_);

for (int rank = 0; rank < proc_num_; ++rank) {
    const size_t amount = base + (std::cmp_less(rank, remainder) ? 1U : 0U);
    counts[rank] = static_cast<int>(amount);
    displs[rank] = static_cast<int>(offset);
    offset += amount;
}
```

Рассылка выполняется отдельно для `xs` и `ys` через `MPI_Scatterv`.

### 4.3. Локальное построение оболочки

Каждый процесс вызывает `BuildHullFromCoords` на своём подмножестве точек.
Внутри функции используется полный TBB-конвейер: `parallel_reduce` для поиска опорной точки,
`parallel_for` + `parallel_sort` для сортировки, `parallel_reduce` для проверки коллинеарности.

```cpp
const tbb::global_control gc(
    tbb::global_control::max_allowed_parallelism,
    ppc::util::GetNumThreads());
```

### 4.4. Сбор локальных оболочек (`GatherLocalHull`)

Каждый процесс отправляет размер своей локальной оболочки через `MPI_Gather`,
затем процесс 0 собирает координаты всех вершин через `MPI_Gatherv`:

```cpp
MPI_Gather(&local_size, 1, MPI_INT,
           counts.data(), 1, MPI_INT, 0, MPI_COMM_WORLD);

MPI_Gatherv(local_xs.data(), local_size, MPI_DOUBLE,
            gathered_xs.data(), counts.data(), displs.data(),
            MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

Координаты `x` и `y` передаются двумя отдельными вызовами `MPI_Gatherv`.

### 4.5. Финальное слияние и рассылка результата

Процесс 0 вызывает `BuildHullFromCoords` на объединении всех локальных оболочек — это корректно,
так как выпуклая оболочка объединения множеств является подмножеством объединения их оболочек.
Результат рассылается всем процессам через `MPI_Bcast` (сначала размер, затем координаты).

## 5. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Ключевые архитектурные решения:

- `BuildHullFromCoords` — единая функция без состояния, вызываемая и на локальных данных,
и на объединении оболочек, что устраняет дублирование логики.
- Конструктор класса инициализирует `proc_rank_` и `proc_num_` и заполняет входные данные только на процессе 0;
остальные процессы получают данные через MPI.
- Тип `uint64_t` для передачи размеров гарантирует корректность при `n > 2^31` на 64-битных системах.
- `FillIndicesParallel` в ALL-версии реализована через `tbb::parallel_for` по заранее построенным диапазонам.

ALL — единственная реализация с ощутимым ускорением: `3.35×` при 4 процессах.
Высокая эффективность (`83.7%`) объясняется тем, что объём коммуникаций (передача оболочек,
а не всего массива точек на финальный шаг) значительно меньше объёма вычислений.
С ростом числа процессов эффективность снижается из-за роста накладных расходов MPI и уменьшения локального подмножества.

## 9. Выводы

Двухуровневая декомпозиция MPI+TBB позволила достичь реального ускорения, недоступного однопроцессным реализациям.
Ключевой принцип — алгоритмическая корректность слияния через повторный запуск Грэхема на
объединении локальных оболочек.
Узкими местами остаются последовательные фазы на процессе 0 (финальное слияние, `RemoveDuplicates`),
а также накладные расходы MPI-коммуникаций, что ограничивает масштабируемость при большом числе процессов.
