## 1. Контекст

OpenMP-версия повторяет общий алгоритм SEQ, но сортирует две половины массива параллельно. В этой реализации
распараллеливание сделано не по множеству мелких блоков, а по двум крупным независимым частям. Такой вариант
хорошо совпадает с текущей структурой алгоритма: до слияния левая и правая половины не имеют общих данных.

Перед замером можно было ожидать небольшого выигрыша, потому что половины массива достаточно большие. На
практике оказалось, что двух секций для заметного ускорения мало.

## 2. Постановка задачи

Во всех версиях решается одна и та же задача: нужно отсортировать вектор целых чисел `std::vector<int>` по
неубыванию. На вход может поступить пустой массив, один элемент, уже отсортированная последовательность,
обратный порядок, повторяющиеся значения и отрицательные числа. Валидация в реализации всегда возвращает
`true`, поэтому допустимым считается любой вектор данного типа.

Результатом работы должен быть вектор того же размера, содержащий те же значения, но в отсортированном
порядке. Корректность проверяется двумя способами: функциональные тесты сравнивают выход с заранее заданным
ожидаемым ответом, а `PostProcessingImpl` дополнительно проверяет отсортированность результата.

## 3. Базовый алгоритм

Базовая схема алгоритма одинакова для всех реализаций. Сначала входной массив копируется во внутренний буфер
и, если нужно, дополняется до ближайшей степени двойки значением `std::numeric_limits<int>::max()`. Это нужно,
чтобы две половины имели удобный размер для odd-even merge. После сортировки добавленные элементы удаляются,
поэтому в итоговый ответ они не попадают.

Далее массив делится на две половины. Каждая половина сортируется классической сортировкой Шелла: шаг
начинается с `n / 2` и последовательно делится на два. После этого две отсортированные половины объединяются
через merge Бэтчера: элементы делятся по четности позиций, четные и нечетные подпоследовательности сливаются
отдельно, затем они перемежаются и для соседних нечетных пар выполняется compare-exchange.

## 4. Схема распараллеливания

В `omp/src/ops_omp.cpp` используется `#pragma omp parallel sections`. Создаются две секции: одна сортирует
`left`, другая сортирует `right`.

```cpp
#pragma omp parallel sections default(none) shared(left, right)
{
#pragma omp section
  ShellSort(left);
#pragma omp section
  ShellSort(right);
}
```

Переменные `left` и `right` указаны как `shared`, но каждая секция пишет только в свой вектор. Общей
изменяемой скалярной переменной нет, поэтому `reduction` не нужен. Индексы и временные значения внутри
`ShellSort` являются локальными переменными вызова. Явный `schedule` здесь не используется, потому что это не
`parallel for`, а две секции. После блока `parallel sections` есть неявный барьер: слияние начинается только
после завершения обеих сортировок.

То есть с точки зрения OpenMP код довольно прямой: две работы, потом ожидание, потом обычный merge. Никакой
скрытой балансировки по блокам здесь нет.

## 5. Детали реализации

Файлы реализации:

| Файл | Роль |
| --- | --- |
| `omp/include/ops_omp.hpp` | объявление класса задачи |
| `omp/src/ops_omp.cpp` | алгоритм, OpenMP-секции, post-processing |

`PreProcessingImpl` копирует вход во внутренний буфер. `RunImpl` делает padding, делит массив и запускает две
OpenMP-секции. `PostProcessingImpl` проверяет `std::ranges::is_sorted` и записывает выход.

Гонка данных между секциями не возникает, потому что `left` и `right` - разные объекты `std::vector<int>`.
Общий объект `data_` в этот момент не изменяется. Запись обратно в `data_` происходит уже после завершения
параллельной области.

## 6. Проверка корректности

Корректность проверялась общим набором функциональных тестов из `tests/functional/main.cpp`. OMP-версия
запускается на тех же восьми сценариях, что и SEQ: пустой вход, один элемент, два элемента, обратный порядок,
уже отсортированный массив, смешанные знаки, дубликаты и обычный пример с разной четностью позиций.

В performance-запуске также проверялось, что выход отсортирован и имеет тот же размер, что и вход. Расхождений
с ожидаемым результатом в предоставленном логе нет, тесты прошли.

параллельная работа ограничена двумя секциями.

## 8. Результаты

| Режим | SEQ, c | OMP, c | Speedup vs SEQ | Комментарий |
| --- | ---: | ---: | ---: | --- |
| pipeline | 0.5131675234 | 0.5228806732 | 0.981 | полный pipeline чуть медленнее baseline |
| task_run | 0.4466987710 | 0.4516862236 | 0.989 | вычислительная часть почти равна SEQ |

Результат близок к последовательной версии, но не дает ускорения на данном запуске. Это ожидаемо для такой
декомпозиции: параллельно выполняются только две сортировки половин, а padding, разделение, odd-even merge и
проверка результата остаются последовательными. Плюс у OpenMP есть накладные расходы на создание параллельной
области.

Разница по времени небольшая, но ускорения нет. Эта версия показывает ограничение выбранной схемы:
параллельная область есть, но полезной независимой работы в ней немного.

## 9. Выводы

OMP-реализация корректно повторяет результат SEQ и использует понятную схему с двумя независимыми секциями. На
данном входе она не ускорила программу: `task_run` получил speedup `0.989`, а `pipeline` - `0.981`. Главная
причина - небольшая ширина параллелизма в самой реализации. Для улучшения можно было бы распараллеливать
больше этапов merge или делить массив на большее число блоков, но это уже другая структура алгоритма.
