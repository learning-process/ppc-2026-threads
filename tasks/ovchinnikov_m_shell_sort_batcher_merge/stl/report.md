## 1. Контекст

STL-версия использует ручное создание потоков через `std::thread`. По смыслу она близка к OpenMP-версии: левая
и правая половины массива сортируются независимо, после чего результат сливается тем же odd-even merge
Бэтчера. Разница в том, что жизненный цикл потоков здесь полностью задается в коде.

По смыслу эта версия близка к OMP, но по `task_run` она получилась заметно быстрее. Для схемы из двух крупных
работ ручной запуск потоков оказался достаточно легким.

## 2. Постановка задачи

Во всех версиях решается одна и та же задача: нужно отсортировать вектор целых чисел `std::vector<int>` по
неубыванию. На вход может поступить пустой массив, один элемент, уже отсортированная последовательность,
обратный порядок, повторяющиеся значения и отрицательные числа. Валидация в реализации всегда возвращает
`true`, поэтому допустимым считается любой вектор данного типа.

Результатом работы должен быть вектор того же размера, содержащий те же значения, но в отсортированном
порядке. Корректность проверяется двумя способами: функциональные тесты сравнивают выход с заранее заданным
ожидаемым ответом, а `PostProcessingImpl` дополнительно проверяет отсортированность результата.

## 3. Базовый алгоритм

Базовая схема алгоритма одинакова для всех реализаций. Сначала входной массив копируется во внутренний буфер
и, если нужно, дополняется до ближайшей степени двойки значением `std::numeric_limits<int>::max()`. Это нужно,
чтобы две половины имели удобный размер для odd-even merge. После сортировки добавленные элементы удаляются,
поэтому в итоговый ответ они не попадают.

Далее массив делится на две половины. Каждая половина сортируется классической сортировкой Шелла: шаг
начинается с `n / 2` и последовательно делится на два. После этого две отсортированные половины объединяются
через merge Бэтчера: элементы делятся по четности позиций, четные и нечетные подпоследовательности сливаются
отдельно, затем они перемежаются и для соседних нечетных пар выполняется compare-exchange.

## 4. Схема распараллеливания

Ключевой фрагмент из `stl/src/ops_stl.cpp`:

```cpp
std::thread left_thread([&left]() { ShellSort(left); });
std::thread right_thread([&right]() { ShellSort(right); });
left_thread.join();
right_thread.join();
```

Оба потока создаются до вызова `join`, то есть сортировки действительно могут выполняться одновременно. Если
бы `join` вызывался сразу после создания первого потока, параллелизм фактически потерялся бы. Здесь этого нет:
сначала стартуют `left_thread` и `right_thread`, потом основной поток ожидает их завершения.

Это важный момент для реального параллелизма: если вызвать `join` сразу после создания первого потока, вторая
половина не будет сортироваться одновременно с первой.

Каждый поток работает со своим вектором. `left` и `right` не пересекаются, поэтому `mutex` и `atomic` не
используются. Общий результат `data_` формируется только после `join`, когда оба рабочих потока уже завершили
запись в свои локальные половины.

## 5. Детали реализации

Файлы реализации:

| Файл | Роль |
| --- | --- |
| `stl/include/ops_stl.hpp` | объявление класса STL-задачи |
| `stl/src/ops_stl.cpp` | ручное создание потоков и общий алгоритм |

`ValidationImpl` возвращает `true`. `PreProcessingImpl` переносит вход в `data_`. В `RunImpl` выполняется
padding, запуск двух потоков, слияние и обрезка результата. `PostProcessingImpl` проверяет отсортированность и
записывает выход.

Эта версия не читает `PPC_NUM_THREADS` и не использует `hardware_concurrency()`: число рабочих потоков
зафиксировано кодом как два, потому что алгоритм делит массив ровно на две части.

## 6. Проверка корректности

STL-backend включен в общий набор функциональных тестов. Проверяются пустой вход, один элемент, два элемента,
обратный порядок, уже отсортированный массив, смешанные знаки, дубликаты и обычный массив из восьми элементов.

Так как каждый поток пишет только в свою половину, основных рисков два: забытый `join` перед merge и ошибка
при слиянии. Первый риск закрыт явным ожиданием обоих потоков, второй проверяется сравнением с ожидаемым
выходом и проверкой `std::ranges::is_sorted` в post-processing.

## 8. Результаты

| Режим | SEQ, c | STL, c | Speedup vs SEQ | Комментарий |
| --- | ---: | ---: | ---: | --- |
| pipeline | 0.5131675234 | 0.4373081186 | 1.173 | быстрее последовательного pipeline |
| task_run | 0.4466987710 | 0.3606396046 | 1.239 | лучший результат среди чистых thread-backend-ов в `task_run` |

STL-версия показывает ускорение относительно SEQ в обоих режимах. Для данной реализации это значит, что
стоимость создания двух потоков окупилась параллельной сортировкой половин. При этом линейного ускорения до 2
нет, потому что часть работы остается последовательной: подготовка данных, merge Бэтчера, обрезка padding и
проверка результата.

По этим замерам STL оказался лучшим из чистых потоковых вариантов. Это не значит, что `std::thread` всегда
лучше TBB или OpenMP; просто данная задача хорошо разложилась на две независимые части.

## 9. Выводы

Ручная STL-версия оказалась самой удачной среди OMP, STL и TBB по `task_run` в данном логе: `0.3606396046` c
против `0.4466987710` c у SEQ. Реализация простая и понятная, но масштабируется только до двух потоков, потому
что параллельная часть жестко привязана к двум половинам массива. Для дальнейшего роста нужно менять разбиение
на большее число блоков.
