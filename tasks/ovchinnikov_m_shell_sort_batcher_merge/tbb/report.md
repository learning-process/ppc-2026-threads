## 1. Контекст

TBB-версия использует `tbb::parallel_invoke` для запуска независимых частей алгоритма. В этой реализации нет
`parallel_for`, диапазонов блоков и `blocked_range`: параллельно выполняются несколько отдельных функций,
между которыми нет общей записи в одни и те же данные.

## 2. Постановка задачи

Во всех версиях решается одна и та же задача: нужно отсортировать вектор целых чисел `std::vector<int>` по
неубыванию. На вход может поступить пустой массив, один элемент, уже отсортированная последовательность,
обратный порядок, повторяющиеся значения и отрицательные числа. Валидация в реализации всегда возвращает
`true`, поэтому допустимым считается любой вектор данного типа.

Результатом работы должен быть вектор того же размера, содержащий те же значения, но в отсортированном
порядке. Корректность проверяется двумя способами: функциональные тесты сравнивают выход с заранее заданным
ожидаемым ответом, а `PostProcessingImpl` дополнительно проверяет отсортированность результата.

## 3. Базовый алгоритм

Базовая схема алгоритма одинакова для всех реализаций. Сначала входной массив копируется во внутренний буфер
и, если нужно, дополняется до ближайшей степени двойки значением `std::numeric_limits<int>::max()`. Это нужно,
чтобы две половины имели удобный размер для odd-even merge. После сортировки добавленные элементы удаляются,
поэтому в итоговый ответ они не попадают.

Далее массив делится на две половины. Каждая половина сортируется классической сортировкой Шелла: шаг
начинается с `n / 2` и последовательно делится на два. После этого две отсортированные половины объединяются
через merge Бэтчера: элементы делятся по четности позиций, четные и нечетные подпоследовательности сливаются
отдельно, затем они перемежаются и для соседних нечетных пар выполняется compare-exchange.

## 4. Схема распараллеливания

Фрагмент из `tbb/src/ops_tbb.cpp`:

```cpp
const auto parallelism = static_cast<std::size_t>(std::max(1, ppc::util::GetNumThreads()));
const tbb::global_control control(tbb::global_control::max_allowed_parallelism, parallelism);
tbb::parallel_invoke([&]() { ShellSort(left); }, [&]() { ShellSort(right); });
```

Конкуренция ограничивается через `tbb::global_control::max_allowed_parallelism`, значение берется из
`ppc::util::GetNumThreads()`. Это связывает реализацию с инфраструктурой курса и переменной окружения, если
она задана runner-ом.

В этой реализации не используются `blocked_range`, `grainsize` и partitioner. `parallel_invoke` запускает
заранее заданные независимые задачи, поэтому размер работы задается самой схемой алгоритма: две сортировки,
две операции split и два merge.

Такое разбиение довольно крупное. Код остается простым, но балансировка получается слабее, чем при делении
массива на много небольших блоков.

Еще один фрагмент из merge:

```cpp
tbb::parallel_invoke([&]() { SplitByParity(left, left_even, left_odd); },
                     [&]() { SplitByParity(right, right_even, right_odd); });

tbb::parallel_invoke([&]() { merged_even = MergeSorted(left_even, right_even); },
                     [&]() { merged_odd = MergeSorted(left_odd, right_odd); });
```

Каждая ветка пишет в свой локальный объект, поэтому явные блокировки не нужны.

## 5. Детали реализации

Файлы реализации:

| Файл | Роль |
| --- | --- |
| `tbb/include/ops_tbb.hpp` | объявление класса TBB-задачи |
| `tbb/src/ops_tbb.cpp` | `parallel_invoke`, ограничение TBB-параллелизма и общий алгоритм |

`ValidationImpl` всегда возвращает `true`. `PreProcessingImpl` копирует вход. `RunImpl` выполняет padding,
разбиение, параллельную сортировку половин и merge. `PostProcessingImpl` проверяет отсортированность.

С точки зрения потокобезопасности важны два момента: `left` и `right` сортируются отдельно, а промежуточные
векторы `left_even`, `right_even`, `merged_even`, `merged_odd` не имеют пересекающихся записей между задачами.
После завершения `parallel_invoke` результат собирается в основном потоке.

## 6. Проверка корректности

TBB-backend входит в общий набор функциональных тестов из `tests/functional/main.cpp`. Проверяются пустой
массив, один элемент, два элемента, обратный порядок, уже отсортированный массив, отрицательные значения,
дубликаты и обычный пример.

В performance-тесте проверяется, что выходной вектор имеет размер входа и отсортирован. По предоставленному
запуску оба режима TBB прошли успешно.

## 8. Результаты

| Режим | SEQ, c | TBB, c | Speedup vs SEQ | Комментарий |
| --- | ---: | ---: | ---: | --- |
| pipeline | 0.5131675234 | 0.4806813050 | 1.068 | полный pipeline быстрее SEQ |
| task_run | 0.4466987710 | 0.4610113894 | 0.969 | вычислительная часть немного медленнее SEQ |

Разница между режимами получилась неодинаковой: в `task_run` накладные расходы `parallel_invoke` не окупились,
а в полном pipeline итоговое время оказалось меньше, чем у SEQ. Поэтому по этим данным нельзя просто сказать,
что TBB-версия быстрее или медленнее всегда. Результат зависит от того, какой режим измеряется.

## 9. Выводы

TBB-версия корректна и аккуратно ограничивает параллелизм через `global_control`. По `pipeline` она получила
speedup `1.068`, но по `task_run` уступила SEQ со speedup `0.969`. Для этой задачи `parallel_invoke` удобен и
читаем, однако он не раскрывает весь потенциал TBB. Более сильная версия могла бы делить массив на большее
число блоков и использовать `parallel_for`, но текущий код реализует именно двухветочную схему.
