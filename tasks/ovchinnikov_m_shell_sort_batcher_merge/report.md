## 1. Введение

В работе сравниваются пять реализаций одной задачи сортировки: последовательная версия, OpenMP,
ручные потоки STL, oneTBB и гибридная версия ALL. Во всех случаях на вход подается
`std::vector<int>`, а результатом должен быть тот же набор элементов, упорядоченный
по неубыванию.

Базовый алгоритм общий: массив дополняется до степени двойки, делится на две половины,
половины сортируются сортировкой Шелла, после чего объединяются odd-even merge Бэтчера.
Различия между технологиями находятся в том, как выполняется сортировка половин и, для ALL,
как данные распределяются между MPI-процессами.

В сравнении оцениваются именно эти реализации, а не предельные возможности технологий.
Например, OMP здесь получает только две независимые секции, поэтому по нему нельзя делать
общий вывод для всех OpenMP-программ.

## 2. Единая постановка задачи

Входные данные: вектор целых чисел. Ограничений на знаки, повторяемость и начальный порядок
элементов нет. Пустой вектор и вектор из одного элемента считаются корректными входами.

Выходные данные: вектор того же размера, отсортированный по неубыванию. Критерий корректности
в performance-тесте - совпадение размера и `std::ranges::is_sorted`. В функциональных тестах
результат сравнивается с заранее заданным ожидаемым ответом.

## 3. Реализации

| Backend | Основная идея |
| --- | --- |
| SEQ | две половины сортируются последовательно, затем выполняется merge Бэтчера |
| OMP | две половины сортируются через `#pragma omp parallel sections` |
| STL | создаются два `std::thread`, каждый сортирует свою половину |
| TBB | независимые фазы запускаются через `tbb::parallel_invoke` |
| ALL | MPI делит массив между rank-ами, локально chunk сортируется через OpenMP sections, затем идет tree merge |

Важно, что OMP, STL и TBB здесь не делят массив на много блоков. Они используют естественное
деление алгоритма на две половины. Поэтому сравнение показывает эффективность именно этой
декомпозиции, а не предельные возможности технологий.

Это главное ограничение сравнения. Для более сильного ускорения стоило бы отдельно менять
разбиение массива, а не только backend.

## 4. Методика эксперимента

Замеры выполнялись в WSL на Windows 11.

| Параметр | Значение |
| --- | --- |
| CPU | 13th Gen Intel(R) Core(TM) i7-13700H |
| RAM | 16 GB DDR4 |
| GPU | GeForce RTX 4060 Laptop GPU |
| ОС | Windows 11, WSL |
| Размер входа | `N = 10000000` |
| Вход performance-теста | убывающая последовательность от `N` до `1` |
| Команда запуска | `mpirun -np 4 ./ppc_perf_tests` |

Performance-тесты запускались в двух режимах:

| Режим | Что измеряется |
| --- | --- |
| `task_run` | основное выполнение `Run` после подготовки входа |
| `pipeline` | полный цикл задачи с preprocessing и postprocessing |

Speedup считается как `T_seq / T_backend` отдельно для каждого режима. Efficiency в общей таблице
не нормируется жестко для всех backend-ов, потому что число рабочих единиц задано по-разному:
SEQ использует один поток, OMP/STL/TBB фактически работают с двумя крупными ветвями, а ALL
использует 4 MPI-rank-а и OpenMP sections внутри rank-а.

Отдельное значение `PPC_NUM_THREADS` в выводе запуска не фиксировалось. Поэтому в общей таблице
приведены времена и speedup, а эффективность по числу потоков не считается как точная величина.

## 5. Проверка корректности

Функциональные тесты находятся в `tests/functional/main.cpp`. Они включают восемь сценариев:
пустой массив, один элемент, два элемента, обратный порядок, уже отсортированный массив,
смешанные знаки, дубликаты и обычный пример с восьмью элементами.

Все backend-ы подключены к одному набору тестов через инфраструктуру курса. Это важно: OMP,
STL, TBB и ALL не проверяются отдельными, более мягкими тестами, а проходят через тот же набор
ожидаемых ответов. В performance-тесте дополнительно проверяется размер выхода и отсортированность.

## 6. Результаты `task_run`

Baseline для режима `task_run`: SEQ = `0.4466987710` c.

| Backend | Время, c | Speedup vs SEQ | Примечание |
| --- | ---: | ---: | --- |
| SEQ | 0.4466987710 | 1.000 | последовательный baseline |
| OMP | 0.4516862236 | 0.989 | две OpenMP-секции, немного медленнее SEQ |
| STL | 0.3606396046 | 1.239 | лучший `task_run` среди всех версий |
| TBB | 0.4610113894 | 0.969 | `parallel_invoke`, накладные расходы не окупились |
| ALL | 0.4068686020 | 1.098 | 4 MPI-rank-а, локально OpenMP sections |

В `task_run` лучшей оказалась STL-версия. Ее преимущество объясняется просто: создаются ровно
два потока для двух независимых половин, без более тяжелого runtime-планирования. ALL тоже
быстрее SEQ, но коммуникации и дерево слияния не позволяют ей обойти STL в вычислительном режиме.

Это объясняется достаточно простой схемой: две большие половины сортируются двумя потоками,
без дополнительного планировщика задач.

## 7. Результаты `pipeline`

Baseline для режима `pipeline`: SEQ = `0.5131675234` c.

| Backend | Время, c | Speedup vs SEQ | Примечание |
| --- | ---: | ---: | --- |
| SEQ | 0.5131675234 | 1.000 | полный последовательный pipeline |
| OMP | 0.5228806732 | 0.981 | близко к SEQ, но без ускорения |
| STL | 0.4373081186 | 1.173 | стабильный выигрыш относительно SEQ |
| TBB | 0.4806813050 | 1.068 | быстрее SEQ в полном цикле |
| ALL | 0.3897978068 | 1.316 | лучший pipeline-результат |

В полном pipeline лучший результат показала ALL-версия. Вероятно, на большом входе распределение
данных между четырьмя rank-ами перекрывает стоимость MPI-обменов. STL остается сильной и простой
версией, а TBB показывает умеренный выигрыш. OMP находится почти на уровне SEQ, но слегка проигрывает.

При этом ALL нельзя считать лучшей версией для всех случаев. Время получилось минимальным
в этом запуске, но за ним стоят пересылки, broadcast и отдельная логика дерева слияния.

## 8. Интерпретация различий

SEQ дает устойчивую базовую линию. Она не имеет накладных расходов на потоки и MPI,
но сортирует обе половины последовательно.

OMP использует две секции. Это корректная и простая модель, однако ширина параллелизма слишком
мала. Когда остается последовательное слияние, выигрыш от двух секций может быть полностью
съеден накладными расходами OpenMP.

STL вручную запускает два потока и вызывает `join` после старта обоих. Для текущей задачи это
оказалось удачным решением: overhead невелик, а две половины достаточно большие, чтобы
параллельная сортировка окупилась.

TBB использует `parallel_invoke` и `global_control`. Это аккуратная реализация, но она тоже имеет
всего несколько независимых задач. В `task_run` TBB немного проиграл SEQ, зато в pipeline оказался
быстрее.

ALL использует MPI-распределение и дерево слияния. Это наиболее сложная версия: у нее есть цена
`MPI_Send`, `MPI_Recv`, `MPI_Bcast` и локального OpenMP. При этом на данном входе она дала лучший
pipeline. Вывод тут не в том, что ALL всегда лучше, а в том, что на крупном убывающем массиве
межпроцессное разбиение оказалось полезным.

## 9. Репродуцируемость

Сборка проекта выполняется стандартным CMake-процессом курса. Пример команд:

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

Запуск функциональных тестов:

```bash
./build/bin/ppc_func_tests --gtest_filter='*ovchinnikov_m_shell_sort_batcher_merge*'
```

Запуск performance-тестов, использованный для приведенных чисел:

```bash
mpirun -np 4 ./ppc_perf_tests
```

Дополнительная стабилизация, например фиксация CPU governor или `taskset`, отдельно не указывалась.
Поэтому числа стоит воспринимать как локальные измерения на конкретном ноутбуке и окружении WSL.
Для более строгого сравнения можно было бы повторить серию замеров и брать медиану, но
предоставленный лог уже дает общую картину.

## 10. Заключение

По `task_run` лучшей стала STL-версия: `0.3606396046` c и speedup `1.239`. По полному
`pipeline` лучший результат у ALL: `0.3897978068` c и speedup `1.316`. OMP в текущей
двухсекционной реализации ускорения не дал, а TBB показал смешанный результат: немного хуже SEQ
в `task_run`, но лучше в `pipeline`.

Главное ограничение сравнения - все версии используют достаточно грубое разбиение. Для OMP, STL
и TBB параллелизм в основном сводится к двум половинам, а не к большому числу независимых блоков.
Поэтому дальнейшее улучшение стоит искать в более мелком разбиении массива и параллельном слиянии,
но это уже потребует переделки алгоритма, а не просто замены технологии.

В целом результаты показывают поведение конкретной реализации. На этих данных лучше всего
смотрятся STL для `task_run` и ALL для `pipeline`, но при другом разбиении массива картина могла бы
измениться.
