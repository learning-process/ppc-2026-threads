- Задача: сортировка целочисленного массива методом Шелла с последующим odd-even merge Бэтчера

## 1. Контекст

Последовательная версия является базовой для всей работы. Она не использует потоки или MPI и нужна как эталон
корректности и как точка отсчета для ускорения остальных реализаций. Все версии решают одну и ту же задачу: на
вход поступает `std::vector<int>`, на выходе должен быть вектор того же размера, отсортированный по
неубыванию.

SEQ в этой работе важен не только как самая простая версия, но и как контрольная точка. Если в ней есть
ошибка, то сравнение остальных реализаций теряет смысл, даже если они показывают меньшее время.

## 2. Постановка задачи

Во всех версиях решается одна и та же задача: нужно отсортировать вектор целых чисел `std::vector<int>` по
неубыванию. На вход может поступить пустой массив, один элемент, уже отсортированная последовательность,
обратный порядок, повторяющиеся значения и отрицательные числа. Валидация в реализации всегда возвращает
`true`, поэтому допустимым считается любой вектор данного типа.

Результатом работы должен быть вектор того же размера, содержащий те же значения, но в отсортированном
порядке. Корректность проверяется двумя способами: функциональные тесты сравнивают выход с заранее заданным
ожидаемым ответом, а `PostProcessingImpl` дополнительно проверяет отсортированность результата.

## 3. Базовый алгоритм

Базовая схема алгоритма одинакова для всех реализаций. Сначала входной массив копируется во внутренний буфер
и, если нужно, дополняется до ближайшей степени двойки значением `std::numeric_limits<int>::max()`. Это нужно,
чтобы две половины имели удобный размер для odd-even merge. После сортировки добавленные элементы удаляются,
поэтому в итоговый ответ они не попадают.

Далее массив делится на две половины. Каждая половина сортируется классической сортировкой Шелла: шаг
начинается с `n / 2` и последовательно делится на два. После этого две отсортированные половины объединяются
через merge Бэтчера: элементы делятся по четности позиций, четные и нечетные подпоследовательности сливаются
отдельно, затем они перемежаются и для соседних нечетных пар выполняется compare-exchange.

Фрагмент из `seq/src/ops_seq.cpp`:

```cpp
const std::size_t original_size = data_.size();
const std::size_t padded_size = NextPowerOfTwo(original_size);
data_.resize(padded_size, std::numeric_limits<int>::max());

auto [left, right] = SplitInHalf(data_);

ShellSort(left);
ShellSort(right);

data_ = BatcherOddEvenMerge(left, right);
data_.resize(original_size);
```

Этот код показывает основной pipeline последовательной версии: padding, деление пополам, две сортировки Шелла
и финальное слияние.

Отдельно отмечу маленькую практическую деталь: padding значением `int::max()` удобен, но после merge его
обязательно надо убрать. Иначе на тестах с обычными числами ошибка могла бы не сразу броситься в глаза.

## 4. Детали реализации

Основная реализация находится в `seq/src/ops_seq.cpp`, интерфейс - в `seq/include/ops_seq.hpp`. Общие типы
`InType`, `OutType`, `TestType` и `BaseTask` заданы в `common/include/common.hpp`.

`PreProcessingImpl` копирует входной вектор во внутренний буфер `data_`. `RunImpl` выполняет сортировку и
слияние. `PostProcessingImpl` проверяет результат через `std::ranges::is_sorted`, после чего записывает данные
в `GetOutput()`. Для массивов размера меньше двух запуск сортировки пропускается, потому что такие данные уже
отсортированы.

Средняя сложность сортировки Шелла зависит от последовательности шагов, но для данного отчета важнее
практическое сравнение с другими backend-ами на одном и том же входе. Слияние выполняет линейные проходы по
подпоследовательностям, а дополнительная память расходуется на две половины, промежуточные векторы четных и
нечетных элементов и итоговый буфер слияния.

## 5. Проверка корректности

Функциональные тесты находятся в `tests/functional/main.cpp`. Они проверяют восемь случаев:

| Сценарий | Смысл проверки |
| --- | --- |
| Empty | пустой вход |
| Single | один элемент |
| TwoElements | минимальный нетривиальный случай |
| Reverse | обратный порядок |
| Sorted | уже отсортированный массив |
| MixedSigns | отрицательные и положительные числа |
| Duplicates | одинаковые элементы |
| EvenOddLength | смесь четных и нечетных позиций |

Для SEQ результат сравнивается с ожидаемым вектором. Дополнительно в `PostProcessingImpl` есть проверка
отсортированности. Это не заменяет функциональные тесты, но полезно как локальная защита от некорректного
результата.

Замеры производительности выполнялись в WSL на Windows 11.

| Параметр | Значение |
| --- | --- |
| CPU | 13th Gen Intel(R) Core(TM) i7-13700H |
| RAM | 16 GB DDR4 |
| GPU | GeForce RTX 4060 Laptop GPU |
| ОС | Windows 11, запуск тестов в WSL |
| Размер perf-входа | `N = 10000000` |
| Данные perf-входа | убывающая последовательность от `N` до `1` |
| Команда замера | `mpirun -np 4 ./ppc_perf_tests` |

В performance-тесте используются два режима: `task_run`, где измеряется в основном `Run`, и `pipeline`, где
учитывается полный цикл задачи.

## 7. Результаты

| Режим | Время, c | Speedup | Efficiency | Комментарий |
| --- | ---: | ---: | ---: | --- |
| pipeline | 0.5131675234 | 1.000 | 1.000 | baseline полного pipeline |
| task_run | 0.4466987710 | 1.000 | 1.000 | baseline вычислительной части |

SEQ не обязан быть самым быстрым во всех режимах, но именно он задает честную базу сравнения. Ускорение
остальных версий дальше считается относительно соответствующего SEQ-замера: отдельно для `task_run` и отдельно
для `pipeline`.

Эти два режима лучше рассматривать отдельно, потому что `pipeline` захватывает больше служебной работы.
Разница между ними заметная и влияет на выводы по TBB и ALL.

## 8. Выводы

Последовательная версия корректно обрабатывает пустые, малые и обычные входы, а также задает общий
алгоритмический каркас для OMP, STL, TBB и ALL. На входе `N = 10000000` время `task_run` составило
`0.4466987710` c, а полный pipeline занял `0.5131675234` c. Эти числа используются как baseline, поэтому любые
выводы об ускорении в сводном отчете нормируются именно на них.
