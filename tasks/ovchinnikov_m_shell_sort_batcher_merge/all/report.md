## 1. Контекст

ALL-версия является гибридной. На межпроцессном уровне она использует MPI: rank 0 подготавливает данные,
активные процессы получают свои части массива, сортируют их локально и затем объединяют результат деревом. На
внутрипроцессном уровне локальная сортировка использует OpenMP-секции для двух половин локального блока.

Эта версия состоит из двух уровней. Сначала MPI распределяет данные по процессам, затем каждый процесс
сортирует свой кусок почти так же, как OMP-версия.

Эта версия отличается от OMP/STL/TBB тем, что результат должен быть согласован между MPI-процессами. Поэтому в
отчете важно рассматривать не только потоки, но и роли rank-ов, пересылки и broadcast итогового массива.

## 2. Постановка задачи

Во всех версиях решается одна и та же задача: нужно отсортировать вектор целых чисел `std::vector<int>` по
неубыванию. На вход может поступить пустой массив, один элемент, уже отсортированная последовательность,
обратный порядок, повторяющиеся значения и отрицательные числа. Валидация в реализации всегда возвращает
`true`, поэтому допустимым считается любой вектор данного типа.

Результатом работы должен быть вектор того же размера, содержащий те же значения, но в отсортированном
порядке. Корректность проверяется двумя способами: функциональные тесты сравнивают выход с заранее заданным
ожидаемым ответом, а `PostProcessingImpl` дополнительно проверяет отсортированность результата.

## 3. Базовый алгоритм

Базовая схема алгоритма одинакова для всех реализаций. Сначала входной массив копируется во внутренний буфер
и, если нужно, дополняется до ближайшей степени двойки значением `std::numeric_limits<int>::max()`. Это нужно,
чтобы две половины имели удобный размер для odd-even merge. После сортировки добавленные элементы удаляются,
поэтому в итоговый ответ они не попадают.

Далее массив делится на две половины. Каждая половина сортируется классической сортировкой Шелла: шаг
начинается с `n / 2` и последовательно делится на два. После этого две отсортированные половины объединяются
через merge Бэтчера: элементы делятся по четности позиций, четные и нечетные подпоследовательности сливаются
отдельно, затем они перемежаются и для соседних нечетных пар выполняется compare-exchange.

## 4. Межпроцессная схема

Основные MPI-вызовы:

| MPI-вызов | Роль |
| --- | --- |
| `MPI_Comm_rank` | определить номер процесса |
| `MPI_Comm_size` | определить число запущенных процессов |
| `MPI_Bcast` | разослать размер входа и итоговый результат |
| `MPI_Send` / `MPI_Recv` | раздать чанки и собрать дерево слияния |

Фрагмент из `all/src/ops_all.cpp`:

```cpp
const int active_processes = LargestPowerOfTwoNotGreaterThan(process_count);
std::size_t padded_size = NextPowerOfTwo(original_size);
while (std::cmp_less(padded_size, active_processes)) {
  padded_size <<= 1;
}

const int chunk_size = static_cast<int>(padded_size / static_cast<std::size_t>(active_processes));
```

Число активных процессов выбирается как наибольшая степень двойки, не превосходящая `MPI_Comm_size`. Это
нужно, чтобы дерево слияния строилось проще и без лишних случаев на каждом шаге.

Rank 0 дополняет данные, копирует себе первый chunk и отправляет остальные активным процессам. После локальной
сортировки вызывается `TreeMerge`: процессы с подходящими rank-ами отправляют свои отсортированные данные
соседу ниже по дереву, а rank 0 в конце содержит общий отсортированный результат.

Дерево слияния здесь полезно тем, что rank 0 не собирает все куски совсем вручную один за другим. Но цена
обменов все равно остается, особенно если размер входа уменьшить.

## 5. Внутрипроцессная схема

Локальная сортировка каждого chunk-а повторяет OMP-подход. Чанк делится на две половины, затем половины
сортируются в OpenMP-секциях:

```cpp
#pragma omp parallel sections default(none) shared(left, right)
{
#pragma omp section
  {
    ShellSort(left);
  }
#pragma omp section
  {
    ShellSort(right);
  }
}
```

Таким образом, конфигурация ALL в предоставленном запуске имеет вид `4 ranks x OpenMP sections`. Точное число
фактически выданных OpenMP потоков зависит от окружения запуска, но полезная работа в коде ограничена двумя
секциями на rank.

## 6. Детали реализации

Файлы реализации:

| Файл | Роль |
| --- | --- |
| `all/include/ops_all.hpp` | объявление гибридной задачи |
| `all/src/ops_all.cpp` | MPI-распределение, локальная OMP-сортировка, дерево merge |

`ValidationImpl` рассылает флаг валидности через `MPI_Bcast`. `PreProcessingImpl` очищает выход. `RunImpl`
выполняет всю MPI-логику, локальную сортировку и финальный broadcast результата. `PostProcessingImpl`
проверяет размер и отсортированность.

Явного `MPI_Barrier` в коде нет. Синхронизация возникает за счет блокирующих `MPI_Send` / `MPI_Recv` и
коллективного `MPI_Bcast`. Это нормально для данной схемы: дерево merge само задает порядок обменов.

## 7. Проверка корректности

ALL-backend включен в общий набор функциональных тестов. Проверяются те же восемь входов: пустой, один
элемент, два элемента, reverse, sorted, mixed signs, duplicates и even/odd length.

Дополнительный риск ALL-версии - согласованность результата на разных rank-ах. В коде после слияния rank 0
рассылает итоговый массив через `MPI_Bcast`, а затем каждый процесс записывает `GetOutput() = result`. Поэтому
post-processing может проверять результат локально на каждом rank-е.

## 9. Результаты

| Режим | SEQ, c | ALL, c | Speedup vs SEQ | Конфигурация |
| --- | ---: | ---: | ---: | --- |
| pipeline | 0.5131675234 | 0.3897978068 | 1.316 | `4 ranks x OMP sections` |
| task_run | 0.4466987710 | 0.4068686020 | 1.098 | `4 ranks x OMP sections` |

ALL ускоряет оба режима относительно SEQ. Особенно заметен выигрыш в `pipeline`, где speedup составил `1.316`.
При этом эффективность по общему числу аппаратно занятых workers оценивать нужно осторожно: есть MPI-обмены,
дерево слияния, broadcast результата и OpenMP-секции внутри каждого rank-а. Эти издержки не видны в чистых
OMP/STL/TBB версиях в таком же виде.

Этот результат не стоит переносить на любые размеры входа. На `N = 10000000` ALL выглядит хорошо, а на
маленьких массивах MPI может дать слишком большие накладные расходы.

## 10. Выводы

Гибридная версия показала лучший `pipeline` среди всех backend-ов в предоставленном запуске: `0.3897978068` c.
В `task_run` она тоже быстрее SEQ, но уступает STL. Сильная сторона ALL - распределение массива между четырьмя
rank-ами. Слабая сторона - цена коммуникаций и финального дерева merge. На больших входах эта схема может быть
полезной, но на малых размерах MPI-обмены легко съедят выигрыш, это важно иметь ввиду.
