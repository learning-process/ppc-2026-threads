## 1. Контекст

Эта реализация представляет собой параллельную версию алгоритма Фокса с использованием директив OpenMP. Цель – добиться ускорения за счёт распараллеливания вычислений на многоядерном процессоре, сохранив при этом кэш-локальность, присущую блочной версии. OpenMP выбран из-за простоты добавления параллелизма в существующий последовательный код.

## 2. Постановка задачи

Постановка задачи полностью идентична последовательной версии (см. `seq/report.md`). Отличие заключается в способе достижения результата: вычисления должны выполняться параллельно с использованием нескольких потоков.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с последовательной версией (блочный алгоритм Фокса). Основное изменение – распараллеливание циклов.

## 4. Схема распараллеливания

**Стратегия:** Декомпозиция по данным. Внешние циклы по блокам (`bk`, `bi`, `bj`) распараллеливаются.

**Распределение работы:**

1. Внешний цикл по `bk` (стадия алгоритма Фокса) **не распараллеливается**, так как итерации зависимы.
2. Двумерный цикл по блокам `bi` и `bj` (обход всех блоков результирующей матрицы) **распараллеливается** с помощью `#pragma omp parallel for collapse(2)`.

**Ключевые директивы и переменные:**

```cpp
#pragma omp parallel for collapse(2) default(none) shared(matrix_a, matrix_b, output, n, block_size, num_blocks, bk)
for (size_t bi = 0; bi < num_blocks; ++bi) {
    for (size_t bj = 0; bj < num_blocks; ++bj) {
        size_t broadcast_block = (bi + bk) % num_blocks;
        size_t i_start = bi * block_size;
        size_t i_end = std::min(i_start + block_size, n);
        size_t j_start = bj * block_size;
        size_t j_end = std::min(j_start + block_size, n);
        size_t k_start = broadcast_block * block_size;
        size_t k_end = std::min(k_start + block_size, n);
        ProcessBlock(matrix_a, matrix_b, output, n, i_start, i_end, j_start, j_end, k_start, k_end);
    }
}
```

`default(none)`: Заставляет явно указать атрибуты всех переменных, что повышает безопасность и читаемость.

`shared(...)`: Все основные массивы являются общими, так как потоки записывают в разные части результирующей матрицы (разные блоки).

`collapse(2)`: Объединяет два вложенных цикла в один параллельный, что обеспечивает лучший баланс загрузки, особенно когда `num_blocks` не очень велико.

**Синхронизация:**

В функции `ProcessBlock` используется `#pragma omp atomic` при обновлении элемента результирующей матрицы:

```cpp
#pragma omp atomic
output[(i * n) + j] += sum;
```

Это необходимо, потому что разные блоки могут обновлять один и тот же элемент C[i][j] во время разных стадий bk алгоритма Фокса.

**Управление потоками:** Количество потоков задается через переменную окружения `OMP_NUM_THREADS`.

**Распределение итераций:** Используется статическое распределение (schedule(static) по умолчанию), так как нагрузка между блоками равномерна. `collapse(2)` объединяет циклы `bi` и `bj` для лучшего баланса.

**Атрибуты переменных:**

- `shared(matrix_a, matrix_b, output, n, block_size, num_blocks, bk)` — все потоки читают одни и те же данные и записывают в разные блоки output.
- `private` переменные отсутствуют явно, так как все локальные переменные внутри цикла (`broadcast_block`, `i_start`, и т.д.) автоматически являются приватными для каждого потока.

## 5. Детали реализации

**Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

**Ключевые изменения относительно SEQ:**

1. Добавлена директива `#include <omp.h>`.
2. В `RunImpl` добавлен адаптивный выбор `block_size` в зависимости от числа потоков:

```cpp
size_t block_size = 64;
if (num_threads > 1) {
    block_size = std::max(static_cast<size_t>(32), n / static_cast<size_t>(num_threads * 2));
    block_size = std::min(block_size, static_cast<size_t>(128));
}
```

Это сделано для того, чтобы количество блоков было хотя бы в 2-4 раза больше числа потоков, обеспечивая гибкое распределение нагрузки.
3. Распараллеливание циклов, как описано в п.4.

**Метод:** Результат выполнения OMP-версии сравнивается с результатом SEQ-версии на идентичных входных данных.

## 9. Выводы

- Реализация на OpenMP демонстрирует отличную масштабируемость на физических ядрах. Ускорение близко к идеальному для больших вычислительных задач (n >= 1024) при использовании до 6 потоков.
- Использование `collapse(2)` и адаптивного размера блока позволило добиться хорошего баланса загрузки.
- Преимущество от использования логических потоков сверх физических ядер минимально (менее 10% прироста при удвоении числа потоков), что характерно для ресурсоёмких вычислительных задач, насыщающих ALU.
- `atomic` директива в `ProcessBlock` вносит небольшое замедление. В теории её можно избежать, если гарантировать, что на каждой стадии `bk` каждый блок `C` обновляется только одним потоком, но текущая реализация проще и достаточно эффективна.
- Прирост производительности ограничен 6 физическими ядрами процессора.
- OpenMP-версия является отличным кандидатом для производства, предоставляя значительное ускорение (до ~7.7x на 12 логических потоках) при минимальных изменениях в коде.
