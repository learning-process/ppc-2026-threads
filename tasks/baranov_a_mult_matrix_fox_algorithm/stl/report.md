## 1. Контекст

Данная реализация представляет собой параллельную версию алгоритма Фокса с использованием стандартной библиотеки потоков C++ (`std::thread`). В отличие от OpenMP, где параллелизм добавляется через директивы компилятора, здесь потоки создаются и управляются вручную. Это даёт более тонкий контроль над распределением работы, но требует явного разбиения данных и управления синхронизацией.

## 2. Постановка задачи

Постановка задачи полностью идентична последовательной версии (см. `seq/report.md`). Отличие заключается в способе достижения результата: вычисления выполняются параллельно с использованием нескольких потоков, создаваемых через `std::thread`.

## 3. Базовый алгоритм

Базовый алгоритм полностью совпадает с последовательной версией (блочный алгоритм Фокса). Основное изменение – замена последовательных циклов на параллельное выполнение с ручным разбиением работы между потоками.

## 4. Схема распараллеливания

**Стратегия:** Декомпозиция по данным. Работа разбивается на независимые блоки, которые распределяются между потоками.

**Два подхода к распараллеливанию:**

1. **StandardMultiplication (наивное умножение):**
   - Строки результирующей матрицы распределяются между потоками.
   - Каждый поток вычисляет свои строки независимо.
   - Формула разбиения: `chunk_size = (n + num_threads - 1) / num_threads`

```cpp
void ParallelRowMultiplication(const std::vector<double> &matrix_a, const std::vector<double> &matrix_b,
                               std::vector<double> &output, size_t n, size_t start_i, size_t end_i) {
  unsigned int num_threads = std::thread::hardware_concurrency();
  if (num_threads == 0) num_threads = 4;

  std::vector<std::thread> threads;
  size_t chunk_size = (end_i - start_i + num_threads - 1) / num_threads;

  for (unsigned int tid = 0; tid < num_threads; ++tid) {
    size_t i_start = start_i + (tid * chunk_size);
    size_t i_end_local = std::min(i_start + chunk_size, end_i);
    if (i_start >= end_i) break;

    threads.emplace_back(
        [&, i_start, i_end_local]() { MultiplyRowRange(matrix_a, matrix_b, output, n, i_start, i_end_local); });
  }

  for (auto &thread : threads) thread.join();
}
```

2. **FoxBlockMultiplication (блочный алгоритм):**
   - Блоки результирующей матрицы распределяются между потоками.
   - Для каждой стадии `block_k` создаётся пул потоков, обрабатывающих блоки.
   - Используется линейная индексация блоков для упрощения распределения.

```cpp
void ParallelBlockProcessing(const std::vector<double> &matrix_a, const std::vector<double> &matrix_b,
                             std::vector<double> &output, size_t n, size_t block_k, size_t num_blocks,
                             size_t block_size, const std::vector<size_t> &block_indices) {
  unsigned int num_threads = std::thread::hardware_concurrency();
  if (num_threads == 0) num_threads = 4;

  std::vector<std::thread> threads;
  size_t chunk_size = (block_indices.size() + num_threads - 1) / num_threads;

  for (unsigned int thread_id = 0; thread_id < num_threads; ++thread_id) {
    size_t start_idx = thread_id * chunk_size;
    size_t end_idx = std::min(start_idx + chunk_size, block_indices.size());
    if (start_idx >= block_indices.size()) break;

    threads.emplace_back([&, start_idx, end_idx]() {
      ProcessBlockRange(matrix_a, matrix_b, output, n, block_k, num_blocks, block_size, block_indices, start_idx, end_idx);
    });
  }

  for (auto &thread : threads) thread.join();
}
```

**Синхронизация:**

В данной реализации **отсутствует** явная синхронизация (мьютексы, атомарные операции) при записи в результирующую матрицу. Это возможно благодаря тому, что:

- В `StandardMultiplication` каждый поток пишет в разные строки (разные диапазоны `i`).
- В `FoxBlockMultiplication` каждый поток также пишет в разные блоки, которые не пересекаются.

Таким образом, гонки данных отсутствуют по конструкции.

**Управление потоками:** Количество потоков определяется через `std::thread::hardware_concurrency()`. Это значение фиксируется на время выполнения программы.

**Порядок выполнения:**

1. Сначала создаются все потоки (`threads.emplace_back(...)`).
2. Каждый поток выполняет свою часть работы независимо.
3. **После создания всех потоков** вызывается `thread.join()` для каждого в отдельном цикле. Это гарантирует, что все потоки работают параллельно, а не последовательно.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

**Ключевые особенности реализации:**

1. **Определение числа потоков:**

```cpp
unsigned int num_threads = std::thread::hardware_concurrency();
if (num_threads == 0) num_threads = 4;
```

Используется аппаратное число потоков CPU (для Ryzen 5 5600X это 12). Если определение невозможно, используется значение по умолчанию (4).

2. **Разбиение работы:**

```cpp
size_t chunk_size = (total_work + num_threads - 1) / num_threads;
```

Работа делится на примерно равные части, последний поток получает остаток.

3. **Запуск и ожидание потоков:**

```cpp
threads.emplace_back([&, i_start, i_end_local]() { 
    MultiplyRowRange(matrix_a, matrix_b, output, n, i_start, i_end_local); 
});
// ...
for (auto &thread : threads) thread.join();
```

Каждый поток запускается с лямбда-функцией, захватывающей необходимые данные по значению (кроме больших массивов, которые передаются по ссылке).

4. **Особенности FoxBlockMultiplication:**

- Предварительная генерация индексов блоков (`block_indices`) для упрощения распределения.
- На каждой стадии `block_k` создаётся новый пул потоков.
- Отсутствие синхронизации: каждый поток работает со своим набором блоков.

В FoxBlockMultiplication потоки создаются заново для каждой из `num_blocks` стадий, что создаёт дополнительную нагрузку.

## 9. Выводы

- STL-версия демонстрирует хорошее ускорение (до ~7.4x на 12 потоках), но уступает OpenMP (7.7x) из-за overhead'а на создание потоков.
- Ручное управление потоками даёт полный контроль над распределением работы, но требует больше кода и внимания к корректности.
- Отсутствие синхронизации (мьютексов/атомарных операций) достигнуто за счёт тщательного разбиения данных на непересекающиеся диапазоны.
- Основной недостаток: создание нового пула потоков на каждой стадии `block_k` в `FoxBlockMultiplication` (вместо переиспользования потоков).
- Потенциальное улучшение: реализовать пул потоков, который создаётся один раз и переиспользуется для всех стадий алгоритма.
- STL-версия является хорошим компромиссом между производительностью и контролем, но для максимальной производительности на данной задаче предпочтительнее OpenMP.
