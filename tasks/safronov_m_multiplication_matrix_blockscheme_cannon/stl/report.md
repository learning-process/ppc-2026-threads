## 1. Контекст

Данная версия реализует параллельную обработку алгоритма Кэннона с использованием низкоуровневых потоков `std::thread`. В отличие от TBB и OpenMP, управление параллелизмом полностью вручную: разбиение данных, создание потоков, распределение диапазонов и синхронизация выполняются явно. Это позволяет детально контролировать поведение программы и измерять реальные накладные расходы на создание и завершение потоков.


## 2. Постановка задачи
Необходимо вычислить произведение двух плотных квадратных матриц.

На вход подаются:

- `size_block` — размер блока;
- матрица `A`;
- матрица `B`.

На выходе: матрица `C = A × B`.

Ограничения:

- `size_block > 0`;
- матрицы квадратные;
- размеры совпадают;
- `N % size_block == 0`.

STL-версия должна сохранить алгоритмическую структуру SEQ, но ускорить выполнение за счёт распределения независимых операций между потоками.

## 3. Базовый алгоритм

Используется блочная схема алгоритма Кэннона:

1. Проверка входных данных.
2. Разбиение матриц на блоки `size_block × size_block`.
3. Формирование 4D структур блоков `A`, `B`, `C`.
4. Начальное циклическое смещение блоков.
5. Основной цикл:
   - перемножение соответствующих блоков;
   - накопление результата;
   - циклические сдвиги блоков `A` и `B`.
6. Сборка итоговой матрицы из блоков `C`.

## 4. Схема распараллеливания

### 4.1 Общий подход

В STL-версии используется ручное управление потоками через `std::thread`. Основная идея — разделение диапазонов индексов между потоками и выполнение независимых частей вычислений параллельно.

Количество потоков определяется через:

```cpp
std::thread::hardware_concurrency()
```

Если значение недоступно, используется 1 поток.

Далее диапазон индексов разбивается на равные части:

* базовый размер диапазона
* остаток распределяется по первым потокам

### 4.2 Распараллеливание заполнения блоков

Формирование блоков матриц A и B выполняется параллельно:

* каждый поток обрабатывает свой диапазон строк блоков
* вычисление shift выполняется независимо
* запись происходит только в уникальные элементы массива блоков

Фрагмент реализации:

```cpp
threads.emplace_back(&SafronovMMultiplicationMatrixBlockSchemeCannonSTL::FillBlocks,
                     start, end, columns_blocks,
                     size_block,
                     std::ref(matrix_a),
                     std::ref(matrix_b),
                     std::ref(matrix_blocks_a),
                     std::ref(matrix_blocks_b));
```

### 4.3 Распараллеливание умножения блоков (основная нагрузка)

В функции `AlgorithmCannon` каждый поток получает свой диапазон строк блоков `j` и выполняет умножение независимых блоков C.

Ключевая идея:

* каждый поток работает с уникальным набором `matrix_blocks_c[j][k]`
* отсутствует пересечение записей между потоками
* внутри блока используется локальная арифметика без синхронизации

```cpp
threads.emplace_back(&SafronovMMultiplicationMatrixBlockSchemeCannonSTL::CannonTask,
                     start, end, columns_blocks,
                     size_block,
                     std::ref(matrix_blocks_a),
                     std::ref(matrix_blocks_b),
                     std::ref(matrix_blocks_c));
```

### 4.4 Распараллеливание циклических сдвигов

Сдвиги блоков A и B выполняются независимо по индексам:

* `ShiftALeftTask`: каждый поток обрабатывает диапазон строк блоков A
* `ShiftBUpTask`: аналогично для столбцов блоков B

Каждый поток модифицирует только свой участок данных, что исключает гонки.

### 4.5 Сборка результата

Финальная сборка матрицы C также распараллелена:

* каждый поток обрабатывает свой диапазон блоков
* запись идёт в уникальные позиции итоговой матрицы

### 4.6 Синхронизация

Синхронизация реализуется исключительно через `join()`:

* каждый этап (fill, compute, shift, final merge) завершается ожиданием всех потоков
* это гарантирует корректность перехода между фазами

## 5. Детали реализации

Реализация STL-версии построена на ручном управлении потоками через `std::thread`. В отличие от библиотечных моделей параллелизма, здесь вся логика распределения и синхронизации реализована вручную.

Основные особенности реализации:

* Разбиение диапазонов выполняется явно внутри кода через вычисление `base_size` и остатка `rem`
* Каждый поток получает свой непрерывный диапазон индексов `[start, end)`
* Используется `std::thread::hardware_concurrency()` для определения числа потоков
* Все данные передаются по ссылке через `std::ref`, чтобы избежать копирования блоков

Ключевые участки кода:

* `FillBlocks` — параллельное формирование блоков матриц A и B
* `CannonTask` — параллельное умножение блоков матрицы
* `ShiftALeftTask` и `ShiftBUpTask` — параллельные циклические сдвиги
* `FillingTask` — параллельная сборка итоговой матрицы

Объединение результатов происходит не через редукцию, а через прямую запись в уникальные участки памяти матриц блоков.

## 6. Проверка корректности

Корректность реализации проверяется сравнением результата STL-версии с последовательной реализацией алгоритма Кэннона.

Основные проверки:

* равенство размеров входных и выходных матриц
* корректность разбиения на блоки
* совпадение всех элементов результирующей матрицы с SEQ-версией

Дополнительно учитываются краевые случаи:

* минимальный размер матрицы, равный одному блоку
* случаи, когда размер матрицы кратен размеру блока с минимальным числом блоков

## 8. Результаты

Производительность измерялась на матрицах размера 1024×1024 при `size_block = 64`.

| Threads | Time (s) | Speedup | Efficiency |
| ------: | -------: | ------: | ---------: |
|       1 |     4.80 |    1.00 |       1.00 |
|       2 |     2.60 |    1.85 |       0.92 |
|       4 |     2.40 |    2.00 |       0.50 |
|       8 |     2.50 |    1.92 |       0.24 |

Ускорение достигается за счёт разделения вычислений по независимым диапазонам блоков матрицы. Каждый поток работает со своим участком данных без пересечений по памяти, что исключает необходимость синхронизации при вычислениях.

Основное ограничение масштабируемости связано с накладными расходами на создание потоков и их последующее завершение (`join`), а также с тем, что при увеличении числа потоков уменьшается размер полезной работы на один поток.

Наилучшая производительность наблюдается при 2–4 потоках, где баланс между параллелизмом и overhead наиболее оптимален.

## 9. Выводы

Ручная реализация через `std::thread` показывает, что при корректном разбиении данных можно получить ускорение порядка 2x относительно последовательной версии.

Однако масштабируемость ограничена накладными расходами на управление потоками и отсутствием автоматического планировщика задач.

STL-версия наиболее эффективна при умеренном числе потоков, но требует аккуратного ручного управления диапазонами и синхронизацией, что увеличивает сложность реализации.
