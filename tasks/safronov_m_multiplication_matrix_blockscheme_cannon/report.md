## 1. Введение

В работе рассматривается задача умножения плотных квадратных матриц типа `double`.
Это удобная задача для сравнения разных моделей параллелизма, потому что вычисления
естественно раскладываются на независимые блоки.

Реализованы и исследованы пять версий одного и того же алгоритма:

* SEQ — последовательная версия;
* OMP — OpenMP;
* TBB — oneTBB;
* STL — `std::thread`;
* ALL — гибридная схема MPI + TBB.

Цель работы — сравнить ускорение, эффективность и характер накладных расходов
у каждой технологии на одной задаче.

## 2. Единая постановка задачи

Необходимо вычислить произведение двух плотных квадратных матриц:

На вход подаются:

* `size_block` — размер блока;
* матрица `A`;
* матрица `B`.

На выходе: матрица `C = A × B`.

Используется блочная схема алгоритма Кэннона.

Ограничения:

* матрицы квадратные;
* размеры матриц совпадают;
* размер блока положителен;
* для блочных версий размер матрицы должен быть кратен размеру блока,
  либо матрица дополняется нулями до нужного размера.

Критерием корректности является совпадение результата параллельных версий
с последовательной реализацией.

### Формулы

Ускорение считается относительно последовательной версии:

```text
speedup = Tseq / Ttech
```

Эффективность считается как:

```text
efficiency = speedup / workers
```

Для OMP, TBB и STL `workers` совпадает с числом потоков.
Для ALL используется общее число работников:

```text
workers = ranks × threads_per_rank
```

### Повторы

В таблицах ниже приведены уже согласованные времена из локальных отчётов.
Для гибридной версии эффективность нормируется по общему числу работников.

## 4. Сводка корректности

Все backend-ы сравнивались с SEQ-версией.

Проверялись:

* малые и средние размеры матриц;
* случайные входные данные;
* корректность блочного умножения;
* совпадение результатов всех параллельных версий с SEQ;
* корректность распределения и сборки блоков в ALL-версии.

В ALL-версии для алгоритма Кэннона формируется квадратная процессорная решётка `q × q`,
где `q = floor(sqrt(size))`.
Процессы с рангами `>= q * q` исключаются из вычислений через `MPI_Comm_split`,
так как алгоритм использует только квадратную сетку процессов.

## 6. Интерпретация различий

### SEQ

Последовательная версия задаёт базовую линию сравнения. Она не имеет накладных расходов на синхронизацию
и обмены, но ограничена одним вычислительным потоком.

### OMP

OpenMP хорошо подходит для распараллеливания независимых блоков.
На этой задаче он даёт заметное ускорение, но дальнейший рост числа потоков ограничивается стоимостью
работы с памятью и накладными расходами на параллельные области.

### TBB

TBB показывает близкие к OpenMP результаты. Его сильная сторона — автоматическое распределение работы
через runtime и удобная работа с диапазонами блоков.

### STL

STL-версия даёт ускорение за счёт ручного разбиения работы между `std::thread`, но чувствительна
к накладным расходам на создание потоков и `join()`.

### ALL

Гибридная версия MPI + TBB даёт наибольший выигрыш на более крупных конфигурациях, потому что сочетает
распределение работы между процессами и локальный многопоточный расчёт внутри процесса.

Ограничение этой схемы — цена коммуникации между процессами, особенно при обменах блоками
и сборке результата.

## 8. Заключение

Для данной задачи все параллельные версии показали ускорение относительно последовательной реализации.

Если смотреть только на локальные модели, наиболее сильный результат показала TBB-версия при 4 потоках,
а STL близка к ней по скорости, но более чувствительна к накладным расходам ручного управления потоками.

Гибридная схема ALL даёт наибольший потенциал ускорения на больших конфигурациях, однако требует больше
вычислительных ресурсов и сильнее зависит от стоимости межпроцессных обменов.

Таким образом, выбор технологии зависит от условий запуска:

* для одного узла с несколькими потоками хорошо подходят OMP и TBB;
* STL тоже работоспособна, но более тяжёлая по управлению;
* для распределённой среды наиболее перспективна ALL.
