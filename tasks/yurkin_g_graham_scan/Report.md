1. Introduction
Постановка задачи — построение выпуклой оболочки множества точек в 2D. Цель реализации: корректно и детерминированно возвращать вершины оболочки в порядке обхода; для параллельных версий — ускорять подготовительные этапы (сортировка/подготовка) без изменения семантики результата.

2. Problem Statement
Вход: std::vector<Point> (Point { double x, double y }).
Выход: std::vector<Point> — вершины выпуклой оболочки в порядке обхода.
Требования и ограничения:

Обработка пустого входа и случаев с 1–2 точками.

Удаление точных дубликатов.

Корректная обработка коллинеарных точек.

Детерминированный результат (порядок и набор вершин).

3. Baseline Algorithm (Sequential)
Алгоритм (монотонная цепочка / Graham monotone chain):

Сортировка точек по x, затем по y.

Последовательное удаление точных дубликатов (один проход).

Если число точек ≤ 2 — вернуть вход.

Построение нижней (lower) цепочки: проход по отсортированным точкам, для каждой точки пока последние две точки в lower и текущая образуют не левый поворот — pop_back, затем push_back.

Построение верхней (upper) цепочки: аналогично, но в обратном порядке по точкам.

Объединение lower и upper без дублирования крайних точек → итоговый hull.

4. Parallelization Schemes (кратко)
Общая идея: финальный проход (формирование lower/upper и объединение) оставлен последовательным для детерминизма; параллелизация применяется в подготовительных этапах (в основном сортировка).

SEQ / STL: последовательная реализация; std::ranges::sort / std::sort используется для сортировки; std::ranges::reverse_view для обратного прохода.

OMP: при наличии поддержки execution policies используется параллельная сортировка (std::sort с std::execution::par) для ускорения сортировки; также демонстрационные блоки с OpenMP (подсчёт потоков) в модуле ALL. Синхронизация через reduction и локальные структуры; финальная сборка — последовательная.

TBB: tbb::parallel_sort применяется в PreProcessingImpl и RunImpl для ускорения сортировки; tbb::parallel_for используется как упражнение в ALL.

ALL: вычисляет детерминированный hull последовательно, затем выполняет лёгкие параллельные упражнения (OpenMP счётчик, std::thread spawn/join, TBB параллельный цикл, MPI barrier) без изменения результата.

Классы и интерфейсы: каждая реализация (YurkinGGrahamScanSEQ/OMP/STL/TBB/ALL) наследует общий интерфейс (методы ValidationImpl, PreProcessingImpl, RunImpl, PostProcessingImpl) и использует GetInput() / GetOutput() для доступа к данным.

Особенности реализации:

Ориентация и вычисления выполняются в повышенной точности (long double) для устойчивости.

Удаление дубликатов реализовано последовательным проходом после сортировки.

В RunImpl каждая версия создаёт локальную копию входа, сортирует (опционально параллельно), строит lower и upper, затем объединяет в hull.

В ALL: после вычисления hull выполняются демонстрационные параллельные упражнения (OpenMP reduction, std::thread spawn/join, TBB parallel_for, MPI barrier) — все они не модифицируют hull.

Крайние случаи: пустой вход, 1 точка, 2 точки, все точки коллинеарны — обработаны корректно.

Память: векторы резервируются (reserve) по размеру входа для уменьшения количества аллокаций.

8. Results Discussion (кратко)
Корректность: все реализации возвращают эквивалентный набор вершин (проверено функциональными тестами). Удаление дубликатов и обработка коллинеарных точек реализованы последовательно и корректно.

Производительность: параллелизация сортировки (OMP с execution policy, TBB) даёт выигрыш на подготовительных этапах; однако финальный стековый проход остаётся последовательным и ограничивает масштабируемость (Amdahl). TBB показывает хорошую производительность при 4 потоках в примере. ALL добавляет накладные расходы и служит для интеграционных проверок, а не для ускорения вычисления оболочки.

Узкие места: финальная стадия построения lower/upper и объединения; при очень больших n возможна выгода от более сложной параллельной схемы (разделение по блокам + слияние частичных оболочек), но это выходит за рамки текущей реализации.

9. Conclusions
Реализации SEQ/STL/OMP/TBB/ALL корректны и детерминированы.

OMP и TBB дают ускорение на подготовительных этапах; реальный выигрыш зависит от реализации сортировки и накладных расходов.

ALL полезен для интеграционного тестирования и демонстрации поддержки технологий, но не оптимизирует сам алгоритм построения оболочки.

Для дальнейшего ускорения финального этапа требуется более сложная параллельная стратегия (разбиение данных, построение частичных оболочек и их слияние).
