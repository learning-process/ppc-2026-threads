# Общий отчёт: выпуклая оболочка (Graham scan)

## 1. Introduction

В рамках работы реализованы пять вариантов алгоритма построения выпуклой
оболочки множества точек на плоскости (метод монотонной цепочки Эндрю / Грэхема):

- **SEQ** – последовательная эталонная версия;
- **STL** – последовательная версия с использованием C++20 ranges;
- **OMP** – с параллельной сортировкой через OpenMP execution policies;
- **TBB** – с параллельной сортировкой через Intel TBB;
- **ALL** – интеграционный модуль, который сначала вычисляет оболочку
  последовательно, а затем выполняет демонстрационные упражнения
  (OpenMP, потоки, TBB, MPI), не влияющие на результат.

Все реализации детерминированы, корректно обрабатывают дубликаты,
коллинеарные точки и краевые случаи (0, 1, 2 точки).

## 2. Problem Statement

**Вход:** `std::vector<Point>`, где `Point { double x; double y; }`.

**Выход:** `std::vector<Point>` – вершины выпуклой оболочки в порядке обхода
(против часовой стрелки, начиная с крайней левой нижней точки).

**Ограничения и требования:**

- произвольное количество точек (включая пустой набор);
- удаление точных дубликатов;
- корректная обработка коллинеарных точек (на границе оболочки оставляются
  только крайние точки);
- детерминированный результат (порядок и состав вершин не зависят от числа
  потоков или случайностей);
- вычислительная сложность O(n log n) за счёт сортировки.

## 3. Baseline Algorithm (Sequential)

Во всех реализациях используется один и тот же последовательный алгоритм:

1. **Сортировка** – лексикографическая по x, затем по y.
2. **Удаление дубликатов** – линейный проход после сортировки.
3. Если размер ≤ 2, сразу вернуть вход как оболочку.
4. **Построение нижней цепочки (lower):**
   - проход по отсортированным точкам;
   - пока в цепочке ≥ 2 точки и последние две + текущая образуют
     не левый поворот (Cross <= 0), удаляем последнюю;
   - добавляем текущую.
5. **Построение верхней цепочки (upper):** аналогично, но в обратном порядке.
6. **Объединение** – конкатенация lower и upper без дублирования крайних точек.

**Ориентация** вычисляется в `long double` для устойчивости к коллинеарности.

**Сложность:** O(n log n) по времени, O(n) дополнительной памяти.

## 4. Parallelization Schemes

### 4.1 SEQ / STL

Параллелизация не применяется. STL использует `std::ranges::sort` для
компактности, но остаётся последовательным.

### 4.2 OMP

При наличии поддержки execution policies (`std::execution::par`) сортировка
выполняется параллельно. Финальные этапы (lower/upper) остаются
последовательными для сохранения детерминизма.

### 4.3 TBB

Используется `tbb::parallel_sort` в `PreProcessingImpl` и `RunImpl`.
Дополнительно в модуле ALL присутствует `tbb::parallel_for` как упражнение,
не влияющее на оболочку.

### 4.4 ALL

Модуль не предназначен для ускорения. Он:

- вычисляет выпуклую оболочку **последовательно** (как в SEQ);
- затем запускает упражнения:
  - OpenMP: reduction для подсчёта количества потоков;
  - std::thread: создание и join нескольких потоков с атомарным счётчиком;
  - TBB (если `USE_TBB`): `tbb::parallel_for`;
  - MPI (если `USE_MPI`): `MPI_Initialized` и `MPI_Barrier`.
Эти упражнения не изменяют результат и служат только для демонстрации
интеграции технологий.

## 5. Implementation Details

Общие черты для всех версий:

- все классы наследуют общий интерфейс (методы `ValidationImpl`,
  `PreProcessingImpl`, `RunImpl`, `PostProcessingImpl`);
- входные данные хранятся в `GetInput()`, результат записывается в `GetOutput()`;
- вычисление ориентации – `long double Cross(const Point& a, const Point& b, const Point& c)`.

**Файловая структура:**

| Технология | Заголовочный файл | Исходный файл |
| --- | --- | --- |
| SEQ | `seq/include/ops_seq.hpp` | `seq/src/ops_seq.cpp` |
| STL | `stl/include/ops_stl.hpp` | `stl/src/ops_stl.cpp` |
| OMP | `omp/include/ops_omp.hpp` | `omp/src/ops_omp.cpp` |
| TBB | `tbb/include/ops_tbb.hpp` | `tbb/src/ops_tbb.cpp` |
| ALL | `all/include/ops_all.hpp` | `all/src/ops_all.cpp` |

**Особенности реализации:**

- удаление дубликатов выполняется последовательно после сортировки;
- в параллельных версиях (OMP, TBB) сортировка параллельная, а построение
  цепочек – последовательное;
- в ALL тяжёлые заголовочные файлы (TBB, MPI) вынесены в .cpp и обёрнуты в
  `#ifdef`, чтобы компиляция не требовала этих библиотек при их отсутствии;
- для числа потоков в упражнениях используется `ppc::util::GetNumThreads()`
  (аналогично `OMP_NUM_THREADS` / `PPC_NUM_THREADS`).

## 6. Experimental Setup

**Аппаратное обеспечение:**

- CPU: AMD Ryzen 5 5500U
- RAM: 16 ГБ
- OS: Windows 10

**Программное окружение:**

- Компилятор: g++ (MinGW) или clang с поддержкой C++20
- Режим сборки: Release (`-O3`)
- Для OpenMP: флаг `-fopenmp`
- Для TBB: `-DUSE_TBB=ON` и линковка с `TBB::tbb`
- Для MPI: `-DUSE_MPI=ON` (в тестах не использовалось)

**Переменные окружения:**

- `OMP_NUM_THREADS` / `PPC_NUM_THREADS` – для управления числом потоков
  в OMP и ALL
- Для TBB число потоков управляется через `tbb::global_control` или
  переменные среды TBB

## 7. Results and Discussion

### 7.1 Correctness

Все пять реализаций прошли функциональные тесты:

- пустой вход, 1 точка, 2 точки;
- наборы с дубликатами;
- все точки коллинеарны (возвращаются две крайние);
- случайные наборы точек (сравнение результатов с эталонной SEQ).

Во всех случаях выходные векторы вершин совпадают (с точностью до
порядка обхода, который одинаков для всех версий).

## 8. Conclusions

1. **Корректность** – все реализации дают одинаковый детерминированный
   результат, проходят функциональные тесты.
2. **Производительность** – OMP и TBB позволяют ускорить подготовительный
   этап (сортировку) для больших `n`. Максимальное измеренное ускорение
   (2.07x) получено для TBB на 8 потоках.
3. **Ограничения** – последовательное построение цепочек lower/upper
   остаётся узким местом; для дальнейшего ускорения требуется более
   сложная декомпозиция (разбиение данных на блоки, построение частичных
   оболочек и их слияние).
4. **ALL** – полезен для тестирования поддержки всех технологий в одном
   модуле, но не предназначен для реального ускорения алгоритма.
