## 1. Введение

### 1.1. Описание задачи

Задача линейной растяжки гистограммы заключается в преобразовании диапазона яркостей изображения для улучшения его
визуального контраста. Математически это выражается формулой:
$$f(x) = (x - min) \cdot \frac{max\_val - min\_val}{max - min} + min\_val$$

Алгоритм выполняется в два этапа:

1. **Анализ (Reduction):** Поиск глобальных экстремумов (минимума и максимума) среди всех пикселей изображения.
2. **Преобразование (Map):** Применение формулы линейного отображения к каждому пикселю.

## 2. Единая постановка задачи

Для всех исследуемых технологий (SEQ, OMP, TBB, STL, MPI+OpenMP) используется единая математическая и программная
постановка задачи, что обеспечивает объективность сравнения.

### 2.1. Входные данные

- **Тип данных:** Одномерный массив (`std::vector<unsigned char>`), представляющий яркости пикселей полутонового
  изображения.
- **Диапазон значений:** $[0, 255]$ для каждого элемента (8-битное представление).

### 2.2. Выходные данные

- **Тип данных:** Модифицированный массив того же размера и типа (`std::vector<unsigned char>`).
- **Содержание:** Пиксели, значения которых пересчитаны таким образом, чтобы исходный диапазон яркостей $[min, max]$ был
  растянут до целевого $[min\_val, max\_val]$ (по умолчанию $[0, 255]$).

### 2.3. Математическая модель

Преобразование каждого пикселя $x$ выполняется по формуле:
$$f(x) = \text{clamp}\left( (x - min) \cdot \frac{max\_val - min\_val}{max - min} + min\_val, \, 0, \, 255 \right)$$
Где:

- $min, max$ — минимальная и максимальная яркости, найденные во **всем** входном массиве.
- $min\_val, max\_val$ — границы целевого диапазона (0 и 255).
- Если $min == max$ (однотонное изображение), выходной массив должен быть идентичен входному во избежание деления на
  ноль.

### 2.4. Ограничения

1. **Инвариантность:** Алгоритм должен быть детерминированным.
2. **Память:** Алгоритм должен минимизировать количество дополнительных аллокаций.
3. **Валидация:** Входной вектор не должен быть пустым.

### 2.5. Критерий корректности

Результат работы любой параллельной реализации признается корректным только в том случае, если он **побитово совпадает**
с результатом последовательной версии (SEQ). Проверка осуществляется путем сравнения выходных векторов:
$\forall i \in [0, N-1]: \text{output}_p[i] == \text{output}_{seq}[i]$ округления или некорректной синхронизации в
критических секциях

## 6. Интерпретация различий

### 6.1. SEQ

SEQ задает точку отсчета (`T_seq = 0.013265` с) и демонстрирует ключевые характеристики задачи: низкую арифметическую
интенсивность, линейный доступ к памяти и независимость пикселей.

### 6.2. OMP

**Сильные стороны:** Минимальные изменения кода (всего 2 директивы), эффективный механизм `reduction`, предсказуемое
статическое планирование.

**Слабые стороны:** Падение эффективности до $E=0.33$ на 8 потоках из-за неспособности адаптироваться к гетерогенной
архитектуре P-cores/E-cores. Накладные расходы на создание параллельных регионов снижают эффективность даже на 2 потоках
($E=0.57$ против $0.92$ у STL).

### 6.3. TBB

**Роль grain size:** Автоматический выбор размера блока (по умолчанию) избавляет от ручной настройки, но платой являются
дополнительные вызовы планировщика.

**Роль runtime:** Механизм _work stealing_ теоретически должен сглаживать дисбаланс между P- и E-ядрами. Практически это
дает выигрыш на 8 потоках ($S=2.88$ против $2.65$ у OMP), но не позволяет обогнать STL ($S=3.00$).

### 6.4. STL

**Цена ручного управления:** Увеличенный объем кода , риск ошибок, отсутствие динамической балансировки.

**Выгода:** Минимальные накладные расходы

**Почему STL выиграл:** Задача статическая и предсказуемая. Любой планировщик создает избыточную сложность. Прямой
подход STL оказывается наиболее эффективным.

### 6.5. ALL

**Цена коммуникации:**

- `MPI_Scatterv`/`MPI_Gatherv` требуют двойного копирования данных через сетевой стек (даже на одном узле).
- `MPI_Allreduce` заставляет все процессы ждать самого медленного.

**Когда ALL оправдан:**

- Данные не помещаются в RAM одного узла.
- Высокая вычислительная сложность на пиксель (коммуникации незаметны на фоне вычислений).
- Реальный многоузловой кластер с высокоскоростной сетью.

### 6.6. Сводная таблица

| Технология | Ускорение (max) | Эффективность (на оптимуме) | Сложность кода | Когда выбирать                         |
| :--------- | :-------------: | :-------------------------: | :------------: | :------------------------------------- |
| **SEQ**    |      1.00       |            1.00             |     Низкая     | Проверка корректности                  |
| **OMP**    |      2.65       |            0.33             |   **Низкая**   | Быстрое прототипирование               |
| **TBB**    |      2.88       |            0.36             |    Средняя     | Динамические/гетерогенные задачи       |

## 8. Заключение

### 8.1. Лучшая версия и условия применимости

**Абсолютный лидер по производительности — STL Threads.** STL показал максимальное ускорение ($S=3.00$) и лучшую
эффективность ($E=0.92$ на 2 потоках) благодаря минимальным накладным расходам. Ручное управление оказалось эффективнее
автоматического для статической, предсказуемой задачи.

### 8.2. Ограничения сравнения

- Результаты получены на конкретном CPU (Intel Core i5-13420H).
- Задача Memory-bound; для CPU-bound задач соотношение может измениться.
- Размер изображения фиксирован ($4096 \times 4096$).

### 8.3. Что можно улучшить

- Для TBB подобрать оптимальный `grainsize` вместо автоматического.
- Для STL использовать пул потоков для устранения overhead создания/уничтожения.
- Для ALL применить неблокирующие MPI-вызовы для перекрытия коммуникаций и вычислений.

### 8.4. Итог

Для линейной растяжки гистограммы на одном узле **STL Threads — оптимальный выбор**. OMP и TBB уступают из-за overhead
автоматического управления. ALL на данной задаче неэффективен. Выбор технологии должен определяться характером задачи и
целевой платформой.
