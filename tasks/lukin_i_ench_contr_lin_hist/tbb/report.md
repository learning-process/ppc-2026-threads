### Почему выбрана такая схема

Использование `tbb::blocked_range` в сочетании с лямбда-выражениями позволяет эффективно реализовать параллелизм по

## 3. Базовый алгоритм

Последовательная версия (SEQ) реализует классический алгоритм линейного растяжения гистограммы в два этапа:

1. **Поиск экстремумов:** Выполняется полный итерационный обход входного массива для определения минимального (`min`) и
   максимального (`max`) значений яркости среди всех пикселей изображения.
2. **Линейное преобразование:** На основе найденных границ вычисляется коэффициент масштабирования
   $scale = 255.0 / (max - min)$. Затем для каждого пикселя вычисляется новое значение по формуле:
   $f(x) = (x - min) \cdot scale$

Подробное описание и инварианты алгоритма приведены в [seq/report.md](../seq/report.md).

## 4. Схема распараллеливания

Для эффективного переноса алгоритма линейной растяжки гистограммы на параллельную архитектуру с использованием
библиотеки **Intel oneTBB** была выбрана схема с разделением данных (Data Parallelism). Реализация опирается на
высокоуровневые шаблоны, которые автоматически управляют распределением нагрузки.

### Использованные примитивы

Алгоритм реализован в два этапа, для каждого из которых выбран соответствующий примитив:

1. **tbb::parallel_reduce:** Используется на первом этапе для поиска минимального и максимального значений яркости.
    - **Почему:** Данный примитив позволяет эффективно выполнять операцию reduction. Он разделяет входной массив на
      части, вычисляет локальные экстремумы в каждом потоке (через лямбда-выражение) и затем объединяет их. Это
      исключает необходимость в использовании блокировок или атомарных операций, предотвращая состояние гонки.
2. **tbb::parallel_for:** Используется на втором этапе для применения формулы трансформации к каждому пикселю.
    - **Почему:** Операции над пикселями полностью независимы, что делает `parallel_for` идеальным инструментом для
      параллельной итерации по массиву данных.

### Range

В обоих случаях работа с данными описывается с помощью класса **tbb::blocked_range<>**.

- **Диапазон:** От `0` до `size` (общее количество пикселей).
- **Принцип:** TBB рекурсивно делит этот диапазон на поддиапазоны, пока не будет достигнут предел дробления, после чего
  итерации внутри поддиапазона выполняются последовательно.

### Grainsize

В данной реализации `grainsize` явно не задавался (используется значение по умолчанию - **1**).

- **Обоснование:** Поскольку тело цикла на этапе трансформации содержит легкие арифметические операции, планировщик TBB
  самостоятельно определяет оптимальный чанк для минимизации накладных расходов. При объеме данных $4096 \times 4096$
  (16 млн элементов) автоматический расчет grainsize позволяет эффективно сбалансировать нагрузку без риска чрезмерного
  дробления задач.

### Partitioner

Используется **tbb::auto_partitioner**.

- **Почему:** Он динамически анализирует загрузку ядер процессора и прекращает деление диапазона, когда дополнительные
  задачи не принесут выигрыша в производительности. Это особенно важно для систем с гибридной архитектурой, где P-ядра и
  E-ядра имеют разную производительность.

### Ограничение конкуренции

Конкуренция за ресурсы в реализации контролируется следующими механизмами:

1. **Локальные копии данных:** В `parallel_reduce` каждый поток работает со своей локальной парой `MinMax`, что
   исключает конкуренцию за общую память на этапе накопления.
2. **Отсутствие явных барьеров:** Синхронизация между этапом поиска экстремумов и этапом трансформации обеспечивается
   естественным завершением вызова `parallel_reduce`.
3. **Work Stealing:** Использование планировщика TBB по умолчанию позволяет системе автоматически перераспределять
   задачи, если одно ядро (например, E-core) отстает от других, предотвращая эффект «длинного хвоста» ожидания.

## 5. Детали реализации

Реализация алгоритма на базе TBB сосредоточена в файлах `tbb/include/ops_tbb.hpp` и `tbb/src/ops_tbb.cpp`. Основное
внимание уделено корректной работе с памятью и минимизации синхронизационных задержек.

### 1. Работа с локальными аккумуляторами

Для поиска глобального минимума и максимума используется шаблон `tbb::parallel_reduce`.

- **Локальные переменные:** Внутри лямбда-выражения (body) для каждого поддиапазона `tbb::blocked_range` создаются
  локальные копии `local_min` и `local_max`.
- **Отсутствие Race Condition:** Поскольку каждый поток работает со своим набором локальных переменных внутри
  выделенного ему чанка данных, исключается конкуренция за запись в общую память. Это значительно быстрее использования
  атомарных операций (`std::atomic`), так как не вызывает инвалидацию кэш-линий.

### 2. Объединение результатов

После завершения обработки своих поддиапазонов, TBB выполняет иерархическое слияние результатов:

- **Сигнатура:** Используется вторая лямбда-функция: `[](MinMax left, MinMax right) -> MinMax`.
- **Логика:** Операция слияния сводится к вычислению `std::min` от левого и правого минимумов и `std::max` от
  соответствующих максимумов.
- **Завершение:** Финальный результат типа `std::pair<unsigned char, unsigned char>` возвращается в основной поток и
  присваивается переменным `min` и `max`.

```cpp
    auto result = tbb::parallel_reduce(tbb::blocked_range<int>(0, size), MinMax(min, max),
                                     [&](const tbb::blocked_range<int> &r, MinMax local) {
        unsigned char local_min = local.first;
        unsigned char local_max = local.second;

        for (int i = r.begin(); i < r.end(); i++) {
        unsigned char curr = input[i];
        local_max = std::max(local_max, curr);
        local_min = std::min(local_min, curr);
        }

        return MinMax(local_min, local_max);
    }, [](MinMax left, MinMax right) {
        return MinMax(std::min(left.first, right.first), std::max(left.second, right.second));
  });
```

### 3. Линейное растяжение

Второй этап алгоритма реализует непосредственно модификацию яркости пикселей на основе вычисленных ранее экстремумов.
Для обеспечения максимальной производительности используется шаблон **tbb::parallel_for**.

- **Масштабирование:** Коэффициент `scale` вычисляется один раз перед запуском параллельного региона как
  `255.0f / (max - min)`. Это позволяет избежать дорогостоящей операции деления внутри цикла, заменяя её умножением.
- **Параллельная итерация:** Примитив `tbb::parallel_for` принимает диапазон `tbb::blocked_range<int>(0, size)`, который
  планировщик динамически разбивает на части. Каждый поток обрабатывает свой независимый участок изображения.
- **Отсутствие зависимостей:** Поскольку результат вычисления для каждого пикселя записывается в отдельную ячейку
  выходного вектора `output`, в данной фазе полностью отсутствуют состояния гонки и необходимость в какой-либо
  синхронизации.
- **Типизация:** Для предотвращения переполнения промежуточные расчеты выполняются в типе `float`, после чего результат
  приводится к `unsigned char`.

```cpp
  float scale = 255.0F / static_cast<float>(max - min);

  tbb::parallel_for(tbb::blocked_range<int>(0, size), [&](const tbb::blocked_range<int> &r) {
    for (int i = r.begin(); i < r.end(); i++) {
      output[i] = static_cast<unsigned char>((static_cast<float>(input[i]) - min) * scale);
    }
  });
```

### 4. Работа с памятью и кэш-эффективность

При реализации учитывались особенности доступа к памяти:

- **Линейный доступ:** Внутри `parallel_for` и `parallel_reduce` обход массива `input` и запись в `output` производятся
  строго последовательно (stride-1 access). Это позволяет механизму _hardware prefetcher_ заранее загружать данные в
  L1-кэш.
- **Минимизация копирования:** Входные данные передаются в лямбда-выражения по ссылке (`const auto &input`), что
  исключает лишние аллокации.

### 5. Типизация и безопасность

Для удобства передачи промежуточных результатов в `tbb/src/ops_tbb.cpp` определен псевдоним типа:

## 9. Выводы

В ходе реализации и тестирования алгоритма линейной растяжки гистограммы с использованием библиотеки **Intel oneTBB**
были сделаны следующие выводы о применимости данной технологии.

### Когда TBB удобнее и эффективнее

1. **Высокоуровневая абстракция:** TBB значительно удобнее при работе в современной экосистеме C++. Использование
   лямбда-выражений и объектов-функций позволяет интегрировать параллелизм непосредственно в логику классов, сохраняя
   типизацию и инкапсуляцию, в отличие от директив OpenMP.
2. **Динамическая балансировка:** Благодаря механизму _task stealing_ (кража задач) и планировщику `auto_partitioner`,
   TBB лучше справляется с нагрузкой в условиях, когда операционная система параллельно выполняет другие задачи. Это
   делает его предпочтительным для пользовательских приложений (GUI, редакторы).
3. **Безопасная редукция:** Шаблон `parallel_reduce` предоставляет элегантный и безопасный способ объединения локальных
   результатов (min/max) без использования явных блокировок (`mutex`) или атомарных операций, что критично для
   поддержания высокой пропускной способности памяти.

### Когда накладные расходы (Overhead) мешают

1. **Малые объемы данных:** На изображениях низкого разрешения (например, менее $512 \times 512$) время, затрачиваемое
   на создание графа задач, разбиение диапазона `blocked_range` и дескрипторы потоков, превышает время самих вычислений.
   В таких случаях последовательная (SEQ) версия оказывается быстрее.
2. **Задачи, ограниченные памятью (Memory Bound):** Как показал анализ результатов (Раздел 8), при обработке простых
   массивов (где на один байт данных приходится мало арифметических операций), основной ограничитель — это шина памяти,
   а не процессор. В этой ситуации сложная логика планировщика TBB создает дополнительный overhead, который не может
   быть компенсирован вычислительной мощностью ядер.
3. **Конкуренция за кэш:** При большом количестве потоков накладные расходы на поддержание когерентности кэшей и слияние
   результатов редукции (`join`) начинают снижать эффективность, что особенно заметно на 8 и более потоках.

### Итоговое резюме

Библиотека TBB является мощным инструментом для распараллеливания обработки изображений, обеспечивая отличный баланс
между производительностью и качеством кода. Однако для достижения максимального ускорения в задачах типа _Memory Bound_
необходимо тщательно выбирать размер обрабатываемых блоков и избегать чрезмерного дробления задач.
