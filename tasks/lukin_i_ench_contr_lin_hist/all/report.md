## 1. Контекст

Реализация алгоритма на базе гибридной технологии (**MPI + OpenMP**) представляет собой наиболее масштабируемое решение,
предназначенное для работы в распределенных вычислительных средах, таких как кластеры или суперкомпьютерные системы.

### Почему требуется гибридная версия (ALL)

1. **Масштабируемость за пределы одного узла:** Стандартные библиотеки (STL, TBB, OpenMP) ограничены ресурсами одной
   вычислительной машины (общей памятью). Гибридная версия использует **MPI** для объединения нескольких узлов в единую
   вычислительную сеть.

2. **Эффективное использование многоядерных узлов:** В современных кластерах каждый узел обладает многоядерным
   процессором. Использование только MPI (один процесс на ядро) создает избыточные накладные расходы на передачу
   сообщений внутри узла. Гибридный подход использует **OpenMP** внутри каждого MPI-процесса, что позволяет потокам
   разделять общую память узла, минимизируя копирование данных и нагрузку на сетевой интерфейс.

3. **Адаптация под топологию суперкомпьютера:** Данная версия является универсальной: она может запускаться как на одном
   многоядерном ПК (используя 1 MPI-ранг и много OpenMP-потоков), так и на мощном кластере, гибко подстраиваясь под
   конфигурацию `ranks × threads` для достижения максимальной пропускной способности.

## 2. Постановка задачи

В данной работе рассматривается задача повышения контрастности полутонового изображения с использованием алгоритма
линейной растяжки гистограммы. Основная цель - реализовать высокопроизводительную версию алгоритма, способную работать в
распределенных системах с использованием гибридной модели программирования **MPI + OpenMP**.

    - Распределение данных между узлами кластера средствами **MPI**.
    - Внутриузловое распараллеливание вычислений средствами **OpenMP**.
    - Сборка результирующего изображения на головном узле.

## 3. Базовый алгоритм

Последовательная версия (SEQ) реализует классический алгоритм линейного растяжения гистограммы в два этапа:

1. **Поиск экстремумов:** Выполняется полный итерационный обход входного массива для определения минимального (`min`) и
   максимального (`max`) значений яркости среди всех пикселей изображения.
2. **Линейное преобразование:** На основе найденных границ вычисляется коэффициент масштабирования
   $scale = 255.0 / (max - min)$. Затем для каждого пикселя вычисляется новое значение по формуле:
   $f(x) = (x - min) \cdot scale$

Подробное описание и инварианты алгоритма приведены в [seq/report.md](../seq/report.md).

## 4. Межпроцессная схема

Гибридная реализация (ALL) организует вычисления по схеме **Master-Worker**, где один процесс координирует работу, а все
остальные (включая Master) участвуют в распределенных вычислениях.

### 4.1 Роли процессов (MPI Ranks)

- **Rank 0 (Master):**
  - Отвечает за чтение исходного изображения (в `PreProcessingImpl`).
  - Рассчитывает размеры блоков для каждого участника.
  - Инициирует распределение данных и финальный сбор результатов.
  - Выполняет запись итогового изображения (в `PostProcessingImpl`).
- **Rank 1...n (Workers):**
  - Принимают свою порцию данных от Master.
  - Выполняют параллельные вычисления внутри узла (с помощью OpenMP).
  - Передают локально обработанные фрагменты обратно Master-процессу.

### 4.2 Распределение данных (Partitioning)

Для работы с произвольными размерами изображений используется алгоритм деления вектора на части, максимально близкие по
размеру.

1. Вычисляется базовый размер блока: `vec_size_ / proc_count`.
2. Остаток (`vec_size_ % proc_count`) распределяется по одной единице между первыми процессами.
3. На основе этих данных формируются массивы `send_counts` (количество элементов для каждого ранга) и `displs` (смещения
   в исходном векторе).

### 4.3 Используемые MPI-вызовы

В реализации задействованы следующие функции стандарта MPI:

- **`MPI_Scatterv`:** Используется для распределения сегментов исходного изображения от Master ко всем процессам. В
  отличие от простого `Scatter`, версия `v` позволяет передавать блоки разного размера (важно для обработки остатка).
- **`MPI_Allreduce`:** Применяется дважды для нахождения глобальных экстремумов:
  - С операцией `MPI_MIN` для поиска глобального минимума среди локальных минимумов рангов.
  - С операцией `MPI_MAX` для поиска глобального максимума. Результат становится доступен сразу всем процессам, что
      позволяет им одновременно перейти к фазе трансформации пикселей.
- **`MPI_Bcast`:** Используется для передачи сигнала об однотонном изображении (`max == min`). Если изображение не
  требует контрастирования, Master рассылает исходные данные всем процессам.
- **`MPI_Gatherv`:** Служит для сбора обработанных сегментов изображения обратно в результирующий вектор на
  Master-процессе после завершения вычислений.

Эта схема минимизирует объем пересылаемых данных: вместо пересылки всего изображения на каждый узел, процессы
обмениваются только необходимыми фрагментами и парой байт для синхронизации экстремумов.

## 5. Внутрипроцессная схема

В гибридной реализации каждый MPI-процесс использует возможности многоядерности узла для ускорения локальных вычислений.

### 5.1 Технология многопоточности

Внутри каждого ранга для параллелизации вычислений на общей памяти используется технология **OpenMP**. Она была выбрана
за счет минимальных накладных расходов при работе с циклами и встроенной поддержки операций редукции, что критично для
алгоритма линейной растяжки.

### 5.2 Распределение работы между потоками

Работа между потоками внутри процесса распределяется автоматически с помощью директивы `#pragma omp parallel for`.

- **Стратегия декомпозиции:** Каждый процесс получает от Master-узла фрагмент изображения размером `local_size`. OpenMP
  разбивает этот локальный массив на итерационные пространства (chunks) и распределяет их между доступными потоками в
  рамках одного процесса.
- **Область видимости:** Вектор `local_vec` и его размер объявляются как `shared`, что позволяет всем потокам процесса
  иметь прямой доступ к обрабатываемым данным без лишнего копирования.

### 5.3 Объединение локальных результатов (Reduction)

Процесс поиска экстремумов и трансформации данных внутри ранга организован следующим образом:

1. **Поиск экстремумов:** Для эффективного поиска минимума и максимума используется механизм **OpenMP Reduction**:

    ```cpp
    #pragma omp parallel for reduction(min : local_min) reduction(max : local_max)
    for (int i = 0; i < local_size; i++) {
        local_min = std::min(local_min, local_vec[i]);
        local_max = std::max(local_max, local_vec[i]);
    }
    ```

    Это позволяет каждому потоку искать экстремумы в своем диапазоне данных независимо, а затем объединять их в итоговые
    переменные `local_min` и `local_max` без использования явных блокировок (mutex), что исключает состояние гонки.

2. **Трансформация пикселей:** После того как глобальные значения `min` и `max` получены через MPI-синхронизацию, потоки
   приступают к модификации пикселей:

    ```cpp
    #pragma omp parallel for
    for (int i = 0; i < local_size; i++) {
        output[i] = static_cast<unsigned char>(
            static_cast<float>(local_vec[i] - min) * scale
        );
    }
    ```

    Здесь используется параллельный цикл по данным без зависимостей по итерациям, что обеспечивает максимально возможную
    скорость выполнения на уровне вычислительных ядер.

### 5.4 Синхронизация

Синхронизация между потоками OpenMP внутри процесса происходит неявно в конце параллельных регионов. Это гарантирует,
что локальные экстремумы будут полностью вычислены перед вызовом `MPI_Allreduce`, а выходной вектор будет полностью
сформирован перед вызовом `MPI_Gatherv`

## 6. Детали реализации

**Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Гибридная реализация сочетает в себе механизмы MPI и многопоточность OpenMP, что требует строгого порядка выполнения
операций для обеспечения корректности данных.

### 6.1 Использование коммуникационных примитивов MPI

В коде реализована четкая последовательность обмена данными и синхронизации между процессами:

1. **`MPI_Scatterv` (Распределение):** Выполняется в начале `RunImpl`. Master-процесс (rank 0) распределяет фрагменты
   исходного вектора всем участникам. Использование версии `v` позволяет корректно обрабатывать ситуацию, когда размер
   изображения не кратен количеству процессов.

```cpp
  const int part = vec_size_ / proc_count;
  const int reminder = vec_size_ % proc_count;

  int offset = 0;
  for (int i = 0; i < proc_count; i++) {
    sendcounts[i] = part + (i < reminder ? 1 : 0);
    offsets[i] = offset;
    offset += sendcounts[i];
  }

  int local_size = sendcounts[proc_rank];
  OutType local_vec(local_size);

  MPI_Scatterv(proc_rank == 0 ? GetInput().data() : nullptr, sendcounts.data(), offsets.data(), MPI_UNSIGNED_CHAR,
  local_vec.data(), local_size, MPI_UNSIGNED_CHAR, 0, MPI_COMM_WORLD);
```

2. **`MPI_Allreduce` (Глобальная синхронизация):** Используется дважды после завершения локальных OpenMP-циклов поиска
   экстремумов.
    - Первый вызов с `MPI_MIN` определяет минимальную яркость во всем изображении.
    - Второй вызов с `MPI_MAX` определяет максимальную. Это критическая точка синхронизации: все процессы блокируются до
      тех пор, пока каждый не предоставит свои локальные данные.

```cpp
  unsigned char min = 255;
  unsigned char max = 0;

  MPI_Allreduce(&local_min, &min, 1, MPI_UNSIGNED_CHAR, MPI_MIN, MPI_COMM_WORLD);
  MPI_Allreduce(&local_max, &max, 1, MPI_UNSIGNED_CHAR, MPI_MAX, MPI_COMM_WORLD);
```

3. **`MPI_Bcast` (Обработка крайнего случая):** Если обнаружено, что `max == min` (однотонное изображение),
   Master-процесс транслирует сигнал и исходные данные всем процессам, после чего выполнение завершается.

```cpp
  if (max == min) {
    if (proc_rank == 0) {
      GetOutput() = GetInput();
    }
    OutType output = GetOutput();
    MPI_Bcast(output.data(), vec_size_, MPI_UNSIGNED_CHAR, 0, MPI_COMM_WORLD);
    GetOutput() = output;
    return true;
  }
```

4. **`MPI_Gatherv` (Сбор результатов):** Финальная операция в `RunImpl`. Обработанные пиксели собираются со всех
   процессов обратно в результирующий вектор на Master-узле.

```cpp
MPI_Allgatherv(output.data(), local_size, MPI_UNSIGNED_CHAR, GetOutput().data(), sendcounts.data(), offsets.data(), MPI_UNSIGNED_CHAR, MPI_COMM_WORLD);
```

5. **`MPI_Barrier` (Неявный):** Явные вызовы `MPI_Barrier` в коде отсутствуют, так как функции `Allreduce`, `Scatterv` и
   `Gatherv` являются коллективными операциями и включают в себя встроенный механизм барьерной синхронизации.

### 6.2 Потенциальные узкие места

Несмотря на высокую степень параллелизма, в реализации существуют факторы, ограничивающие производительность:

- **Сетевые задержки:** Использование `MPI_Allreduce` в середине алгоритма заставляет все процессы ждать завершения
  самого медленного участника.
- **Дисбаланс нагрузки:** В гибридной модели время работы процесса зависит от самого медленного OpenMP-потока внутри
  него.
- **Последовательные участки:** Операции ввода-вывода (чтение изображения из файла и аллокация памяти), выполняемые
  Master-процессом в `PreProcessingImpl`, являются строго последовательными и не масштабируются при увеличении числа
  узлов.
- **Copy Overhead:** Передача данных между адресными пространствами процессов (через сетевой стек) значительно
  медленнее, чем доступ к общей памяти, что делает гибридную версию менее эффективной, чем чистый OpenMP, при запуске на
  одной машине.

## 7. Проверка корректности

Для обеспечения надежности гибридной реализации (**ALL**) был проведен комплекс тестов, направленных на верификацию
точности вычислений и стабильности обмена данными в распределенной среде.

### 7.1 Сравнение с SEQ

Основным критерием корректности является полная идентичность выходных данных при одинаковых входных векторах.

### 7.2 Набор тестов

Для валидации использован тот же набор функциональных тестов, что и для SEQ-версии, что позволяет подтвердить
инвариантность алгоритма:

1. **Тесты на различных размерах:**
    - Малые изображения ($10 \times 10$, $128 \times 128$).
    - Средние и большие изображения ($512 \times 512$, $1024 \times 1024$).
    - Проверка корректности распределения итераций при размерах, не кратных количеству потоков.

2. **Характерные и краевые случаи:**
    - **Empty Input:** Проверка срабатывания валидации на пустой вектор.
    - **Constant Image (min == max):** Тест на изображениях с одинаковой яркостью всех пикселей. Проверяется отсутствие
      деления на ноль и корректность возврата исходного массива.

### 7.3 Согласованность результата между рангами

Ключевым этапом проверки стала верификация согласованности (consistency) глобального состояния:

- **Синхронизация экстремумов:** Благодаря `MPI_Allreduce`, каждый процесс на этапе трансформации пикселей оперирует
  абсолютно идентичными значениями $min$ и $max$. Это исключает возникновение «швов» или разрывов яркости на границах
  блоков данных при сборке через `MPI_Gatherv`.
- **Целостность сборки:** Финальная проверка на Master-процессе после вызова `MPI_Gatherv` подтвердила, что порядок
  следования сегментов данных от различных рангов соответствует исходному порядку в векторе, а границы сегментов не
  перекрываются и не имеют пропусков.

Таким образом, гибридная реализация полностью удовлетворяет требованиям корректности и готова к масштабированию на
кластерных системах.

| 4 (MPI)   |   2 (OMP)   |     8     |  0.053236   |     0.69      |       0.09       |

_Где:_

- $R$ — количество MPI-процессов (рангов).
- $T$ — количество OpenMP-потоков на один ранг.
- $P = R \times T$ — общее количество вычислительных единиц.
- $S = T_{seq} / T_p$ — коэффициент ускорения.
- $E = S / P$ — эффективность использования ресурсов.

### Анализ результатов и масштабируемости

1. **Отрицательное ускорение и оверхед:** На обеих размерностях задачи при малом числе вычислителей наблюдается
   замедление относительно последовательной версии ($S < 1$). Это наглядно демонстрирует значительные накладные расходы
   гибридной модели. Время, затрачиваемое на инициализацию MPI-окружения, создание параллельных областей OpenMP и
   коллективные операции, превышает выигрыш от распараллеливания алгоритма с низкой вычислительной плотностью.

2. **Масштабируемость данных:** С увеличением размера задачи до $8192 \times 8192$ время выполнения ожидаемо выросло,
   однако эффективность ($E$) упала еще сильнее. Это подтверждает, что задача является **Memory Bound**. При росте
   объема данных возрастает объем межпроцессорных пересылок, а конкуренция за шину памяти между ядрами и процессами
   становится критическим фактором, не позволяющим достичь положительного ускорения даже на 8 воркерах.

3. **MPI vs OpenMP (баланс гибридизации):** При фиксированном $P=8$ конфигурации ведут себя по-разному в зависимости от
   объема данных:
    - На $4096^2$: вариант `2R x 4T` быстрее за счет использования преимуществ общей памяти (OpenMP) внутри процесса.
    - На $8192^2$: вариант `4R x 2T` показал лучший результат (0.053 сек). Это может указывать на то, что при больших
      объемах данных разделение задачи на большее количество изолированных MPI-процессов позволяет лучше утилизировать
      кэш-память или эффективнее распределять работу планировщиком ОС, несмотря на рост затрат на коммуникацию.

4. **Критическая точка эффективности:** Минимальная эффективность ($0.08–0.13$) зафиксирована при использовании 8
   воркеров. Использование гибридной архитектуры (P-cores + E-cores) в совокупности с задержками сетевого интерфейса MPI
   делает синхронизацию в точках `MPI_Allreduce` чрезвычайно дорогой — быстрые ядра вынуждены простаивать в ожидании
   медленных потоков и завершения передачи пакетов данных.

**Вывод:** Гибридная реализация ALL в данных условиях является избыточной. Ее применение оправдано только на
суперкомпьютерных узлах при обработке массивов данных, многократно превышающих объем оперативной памяти одного узла, где
невозможно использовать общую память.

## 10. Выводы

В ходе выполнения работы была разработана и исследована гибридная реализация алгоритма линейной растяжки гистограммы,
сочетающая межпроцессное взаимодействие (**MPI**) и внутриузловое распараллеливание (**OpenMP**). Анализ полученных
результатов позволяет сделать следующие выводы:

1. **Область применимости гибридной схемы:** Эксперименты показали, что для изображений стандартного разрешения
   ($4096^2$ и $8192^2$) гибридная схема является избыточной и демонстрирует отрицательное ускорение ($S < 1$) или
   крайне низкую эффективность ($E \approx 0.1$). Использование **ALL** оправдано только в следующих случаях:
    - **Объем данных:** Когда размер входных данных многократно превышает объем оперативной памяти одного
      вычислительного узла.
    - **Распределенная среда:** При работе на кластерных системах, где MPI является единственным способом объединения
      вычислительных мощностей разных серверов.

2. **Анализ накладных расходов (Overheads):** Основным сдерживающим фактором стали высокие накладные расходы на
   коммуникацию. В алгоритмах с низкой вычислительной плотностью (таких как линейная обработка векторов) время,
   затрачиваемое на вызовы `MPI_Scatterv`, `MPI_Allreduce` и `MPI_Gatherv`, сопоставимо или даже превышает время самих
   вычислений. Это делает синхронизацию между процессами «бутылочным горлышком».

3. **Влияние топологии (MPI vs OpenMP):** Сравнение конфигураций при одинаковом общем числе потоков ($P=8$) подтвердило
   преимущество использования общей памяти. Конфигурация с меньшим числом MPI-рангов и большим числом OpenMP-потоков
   внутри них (`2R x 4T`) показала лучшую производительность, чем дробление на множество изолированных MPI-процессов, за
   счет более быстрого обмена данными и отсутствия лишних сетевых транзакций.

4. **Аппаратные и системные ограничения:** Низкая эффективность на 8 вычислительных единицах обусловлена не только
   программной архитектурой, но и физическими ограничениями тестовой системы:
    - **Memory Bound:** Задача ограничена пропускной способностью шины памяти, которая перегружается при одновременных
      запросах от множества ядер и процессов.
    - **Дисбаланс ядер:** Наличие P- и E-ядер в процессоре Intel Core i5-13420H вносит асимметрию: MPI-процессы,
      выполняющиеся на медленных ядрах, задерживают всю глобальную синхронизацию в точках `MPI_Allreduce`.

**Заключение:** Гибридная реализация является наиболее сложной в разработке и сопровождении. Она обеспечивает
максимальную масштабируемость «в ширину», но требует значительно больших объемов вычислительной нагрузки для компенсации
коммуникационных затрат. Для локальной обработки изображений на одном многоядерном узле более целесообразным остается
использование чистых реализаций на OpenMP или TBB.
