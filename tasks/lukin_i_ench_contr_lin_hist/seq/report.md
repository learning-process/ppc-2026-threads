## 1. Контекст

Данная работа посвящена реализации алгоритма линейного растяжения гистограммы для улучшения контрастности полутоновых
изображений. Последовательная версия (SEQ) реализует базовый конвейер курса и служит эталоном для проверки корректности
и оценки ускорения параллельных версий. Она необходима для фиксации базового времени выполнения (baseline) и инвариантов
алгоритма, на которые будут опираться все последующие оптимизации.

## 2. Постановка задачи

**Цель работы:** Реализовать алгоритм повышения контрастности изображения путем линейного растяжения его гистограммы.

**Входные данные:**

- `input_data` (тип `InType` — `std::vector<unsigned char>`): Одномерный массив яркостей пикселей изображения. Значения
  каждого элемента находятся в диапазоне $[0, 255]$.

**Выходные данные:**

- `output_data` (тип `OutType` — `std::vector<unsigned char>`): Модифицированный массив яркостей того же размера, что и
  входной.

**Ограничения:**

- Входной вектор не должен быть пустым (проверяется в `ValidationImpl`).

**Крайние случаи:**

- **Однотонное изображение:** Если минимальное и максимальное значения яркости во входном векторе равны ($min = max$),
  во избежание деления на ноль алгоритм возвращает копию исходных данных.
- **Пустой вход:** Метод `ValidationImpl` возвращает `false`, предотвращая выполнение алгоритма.

## 3. Базовый алгоритм

Алгоритм выполняет перераспределение интенсивностей пикселей из исходного узкого диапазона $[min, max]$ в полный
доступный диапазон $[0, 255]$.

### Пошаговое описание

1. **Поиск экстремумов:** Обход входного вектора для нахождения минимального ($min$) и максимального ($max$) значений
   яркости пикселей.
2. **Проверка на вырожденность:** Если $max == min$ (изображение однотонное), алгоритм завершает работу, возвращая копию
   исходных данных.
3. **Вычисление коэффициента:** Расчет множителя масштабирования по формуле: $$scale = \frac{255.0}{max - min}$$
4. **Линейная трансформация:** Обход входного вектора и вычисление нового значения для каждого пикселя:
   $I_{new} = (I_{old} - min) \cdot scale$

### Сложность

- **Асимптотика по времени:** $O(n)$, где $n$ — количество пикселей в изображении. Алгоритм требует двух
  последовательных проходов по данным.
- **Асимптотика по памяти:** $O(n)$ для хранения выходного вектора результатов. Дополнительная память сверх этого
  (константы для $min, max, scale$) составляет $O(1)$.

### Инварианты и критерий корректности

- **Инвариант диапазона:** После обработки все значения в выходном векторе должны лежать в интервале $[0, 255]$.
- **Критерий корректности:** Пиксель с минимальной исходной яркостью всегда принимает значение $0$, а пиксель с
  максимальной — $255$ (при условии, что $max > min$).

## 4. Детали реализации

**Файлы:** `seq/include/ops_seq.hpp`, `seq/src/ops_seq.cpp`

Реализация алгоритма разбита на стандартные этапы конвейера `Task`:

### Описание методов

1. **`ValidationImpl()`**:
    - Проверяет входные данные на корректность перед началом вычислений.
    - **Условие:** Возвращает `true`, если входной вектор не пуст (`!GetInput().empty()`). В противном случае выполнение
      прерывается.

    ```cpp
    bool LukinITestTaskSEQ::ValidationImpl() {
        return !(GetInput().empty());
    }
    ```

2. **`PreProcessingImpl()`**:
    - Хотя основная аллокация памяти под результат происходит в конструкторе
      (`GetOutput() = OutType(GetInput().size())`), данный метод подтверждает готовность к работе, возвращая `true`.

    ```cpp
    bool LukinITestTaskSEQ::PreProcessingImpl() {
        return true;
    }
    ```

3. **`RunImpl()`**:
    - **Поиск экстремумов:** Используются `std::ranges::min_element` и `std::ranges::max_element` для определения
      текущего диапазона яркостей.
    - **Логика трансформации:** Рассчитывается коэффициент `scale`. Затем в цикле `for` выполняется линейное
      преобразование каждого пикселя. Все вычисления проводятся в типе `float` для сохранения точности, после чего
      результат приводится к `unsigned char`.

    ```cpp
    bool LukinITestTaskSEQ::RunImpl() {
        auto min_it = std::ranges::min_element(GetInput().begin(), GetInput().end());
        auto max_it = std::ranges::max_element(GetInput().begin(), GetInput().end());

        unsigned char min = *min_it;
        unsigned char max = *max_it;

        if (max == min)  // Однотонное изображение
        {
            GetOutput() = GetInput();
            return true;
        }

        float scale = 255.0F / static_cast<float>(max - min);

        int size = static_cast<int>(GetInput().size());

        for (int i = 0; i < size; i++) {  // Линейное растяжение
            GetOutput()[i] = static_cast<unsigned char>(static_cast<float>(GetInput()[i] - min) * scale);
        }

        return true;
    }
    ```

4. **`PostProcessingImpl()`**:
    - В данной реализации не требует дополнительных действий и всегда возвращает `true`.

    ```cpp
    bool LukinITestTaskSEQ::PostProcessingImpl() {
        return true;
    }
    ```

### Обработка исключительных и граничных случаев

- **Однотонное изображение ($max == min$):** Внутри `RunImpl` добавлена проверка. Если все пиксели имеют одинаковую
  яркость, разность `max - min` будет равна нулю. Чтобы избежать неопределенности и деления на ноль, алгоритм просто
  копирует входной вектор в выходной и завершает работу.
- **Пустой вход:** Случай обрабатывается на уровне `ValidationImpl`, что гарантирует безопасность обращения к элементам
  вектора в основном цикле.
- **Приведение типов:** Использование `static_cast<unsigned char>` при финальном расчете обеспечивает корректное сужение
  типа из `float` после выполнения арифметических операций.

## 5. Проверка корректности

Для подтверждения правильности работы алгоритма используется набор функциональных тестов.

### Методика сравнения результатов

В каждом тесте создается эталонный результат, с которым сравнивается выходной вектор метода `RunImpl`. Сравнение
реализовано в методе `CheckTestOutputData`:

- Проводится повторный (независимый от основного кода) расчет ожидаемых значений яркости для каждого пикселя.
- Выполняется поэлементная проверка: `output_data[i] == expected_value`.
- Если хотя бы один пиксель не совпадает, тест считается проваленным.

```cpp
  bool CheckTestOutputData(OutType &output_data) final {
    auto min_it = std::ranges::min_element(input_data_.begin(), input_data_.end());
    auto max_it = std::ranges::max_element(input_data_.begin(), input_data_.end());

    unsigned char min = *min_it;
    unsigned char max = *max_it;
    if (max == min) {
      return true;
    }

    float scale = 255.0F / static_cast<float>(max - min);

    int size = static_cast<int>(input_data_.size());

    for (int i = 0; i < size; i++) {
      auto expected_value = static_cast<unsigned char>(static_cast<float>(input_data_[i] - min) * scale);
      if (output_data[i] != expected_value) {
        return false;
      }
    }

    return true;
  }
```

### Использованные функциональные тесты

Тестирование реализовано с помощью фреймворка **Google Test** в файле `main.cpp`. Входные данные генерируются для
различных сценариев:

1. **Тесты на различных размерах (Image Sizes):**
    - Проверка корректности на изображениях размером $128 \times 128$, $256 \times 256$ и $512 \times 512$. Это
      гарантирует, что алгоритм стабильно работает с разными объемами данных и корректно обрабатывает границы циклов.

2. **Крайние случаи (Edge Cases):**
    - **Однотонное изображение ($32 \times 32$):** Проверяется сценарий, в котором все пиксели имеют одинаковую яркость.
      Тест подтверждает, что алгоритм не допускает деления на ноль и возвращает корректный результат (исходное
      изображение).

В файле `main.cpp` данные формируются в зависимости от параметра теста:

- **Квадратные изображения:** Для размеров $128, 256, 512$ создаются векторы с циклическим заполнением.
- **Специальные случаи:** Для теста на **однотонное изображение** ($32 \times 32$) генерируется вектор, заполненный
  одним и тем же значением.

```cpp
void SetUp() override {
    TestType image_size = std::get<static_cast<std::size_t>(ppc::util::GTestParamIndex::kTestParams)>(GetParam());
    int count = static_cast<int>(std::pow(image_size, 2));

    input_data_.resize(count);

    if (image_size == 32) {
        for (int i = 0; i < count; i++) {
            input_data_[i] = 128;  // однотонное изображение
        }
    } else {
        for (int i = 0; i < count; i++) {
            input_data_[i] = 80 + (i % 81);  // [80,160] - как на обычных фото
        }
    }
}
```

## 8. Выводы

Последовательная версия (SEQ) алгоритма линейного растяжения гистограммы является ключевым компонентом проекта по
следующим причинам:

1. **Эталон корректности:** Ввиду отсутствия параллельных примитивов и состояний гонки (race conditions), данная
   реализация гарантирует математически точный результат согласно алгоритму. Все параллельные версии (OMP, TBB, STL,
   ALL) обязаны выдавать идентичный результат, что проверяется через механизмы валидации.
2. **Точка отсчета (Baseline):** Время выполнения последовательной версии на задаче размером $4096 \times 4096$
   ($0.013265$ сек) принимается за единицу. Относительно этого числа будет рассчитываться ускорение ($S = T_s / T_p$) и
   эффективность ($E = S / p$) всех оптимизированных решений.
3. **Определение узких мест:** Анализ SEQ-версии показал, что основным потребителем ресурсов является линейное
   преобразование яркости в цикле. Это подтверждает целесообразность использования параллелизма данных (data
   parallelism) в этой части кода.
4. **Стабильность:** Последовательный код менее чувствителен к специфическим настройкам планировщиков потоков, что
   позволяет использовать его как надежный индикатор производительности самого вычислительного узла.
