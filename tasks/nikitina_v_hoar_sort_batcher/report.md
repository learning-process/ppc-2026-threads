## 1. Введение

Задача быстрой сортировки массивов является классическим полигоном для исследования параллельных архитектур.
Базовый алгоритм (сортировка Хоара) обладает высокой степенью пространственной локальности, но содержит
жесткие зависимости по данным: сдвиг указателей на каждой итерации зависит от предыдущего шага. Для
устранения этих зависимостей применяется сеть четно-нечетного слияния Бэтчера. Она позволяет разбить исходный
массив на независимые сегменты, сортировать их локально, а затем сливать по строгой детерминированной
математической модели. Такая архитектура делает задачу идеальной для кросс-технологического сравнения:
она позволяет оценить как поведение легковесных внутриузловых потоков (OpenMP, oneTBB, STL), так и цену
сетевой коммуникации в гибридных распределенных моделях (MPI + потоки).

## 2. Единая постановка задачи

Необходимо реализовать алгоритм модификации одномерного массива целых чисел.

- **Входные данные:** Вектор `std::vector<int>` произвольного размера.
- **Выходные данные:** Модифицированный вектор того же размера, отсортированный по неубыванию.
- **Ограничения:** Сортировка должна выполняться "на месте". Использование системного
  стека (рекурсии) запрещено во избежание ошибки `Stack Overflow` на вырожденных данных.
- **Критерий корректности:** Строгое выполнение предиката `std::ranges::is_sorted` для выходного
  массива во всех тестовых сценариях.

## 3. Единая методика эксперимента

Для обеспечения честного и воспроизводимого сравнения технологий (backend-ов) все замеры
производились в единой изолированной среде.

**Окружение:**

- **CPU:** Intel Core i7-12700H
- **RAM:** 16 GB
- **OS:** Windows 11
- **Compiler:** GCC 14
- **Build type:** CMake, профиль `Release`

**Данные и параметры:**

- **Генерация:** Псевдослучайная детерминированная последовательность (seed = 42) в диапазоне от -100 000 до 100 000.
- **Размер задачи:** 500 000 элементов.
- **Нормировка метрик:**
  - `Speedup` (Ускорение) = $T_{seq} / T_{parallel}$.
  - `Efficiency` (Эффективность) = $Speedup / W$, где $W$ — общее число вычислителей. Для
    технологий с общей памятью $W = threads$, для гибридной модели $W = ranks \times threads\_per\_rank$.
- **Агрегация:** Для каждого режима выполнялись многократные прогоны в рамках GTest-инфраструктуры
  `BaseRunPerfTests`. В таблицу вынесены медианные значения для фазы `task_run` (чистое время
  алгоритма без учета инициализации тестового раннера).

## 4. Сводка корректности

Корректность всех 5 backend-ов оценивалась путем сопоставления результатов их работы с эталонным
последовательным алгоритмом (SEQ). Проверка производилась с помощью единого набора функциональных
тестов (`tests/functional/main.cpp`).

Покрыты следующие сценарии:

1. **Пустые структуры (n = 0, n = 1):** Ограничение применимости — алгоритмы должны совершать
   ранний выход, избегая деления на ноль при вычислении чанков.
2. **Уже отсортированные данные (включая обратный порядок):** Проверка устойчивости к худшим
   случаям (максимальное количество операций `swap` в схеме Хоара). Итеративный подход полностью
   исключил переполнение стека.
3. **Некратные размеры массива:** Поскольку сеть Бэтчера требует, чтобы количество блоков было
   степенью двойки, проверена механика Padding-а. Массивы дополнялись фиктивными элементами `INT_MAX`,
   которые затем безопасно отсекались на этапе `PostProcessingImpl`. Все backend-ы (включая гибридную
   пересылку MPI) прошли этот тест без искажения исходных данных.

## 6. Интерпретация различий

- **SEQ (Baseline):** Демонстрирует выдающуюся производительность (~2 миллисекунды на 500 000
  элементов). Такая скорость делает задачу крайне чувствительной к любым накладным расходам
  многопоточных фреймворков.
- **STL (`std::thread`):** Показал наихудший результат (замедление в 5 раз относительно SEQ).
  Это классическая иллюстрация цены сырых системных вызовов: постоянное создание `std::thread` и
  ожидание `join` внутри циклов сети Бэтчера занимает значительно больше времени, чем сама сортировка
  чанков памяти.
- **OMP:** Продемонстрировал парадоксальное поведение. На 2 потоках наблюдается феномен
  сверхлинейного ускорения (4.35x), вызванный тем, что разделенные чанки идеально помещаются
  в L2/L3 кэш процессора. Однако при переходе на 4 потока происходит жесткая деградация (0.86x).
  Причина — динамическое выделение памяти во временные буферы внутри `CompareSplit`. Множественные
  аллокации из 4 потоков вызывают блокировки (сериализацию) в системном аллокаторе кучи ОС.
- **TBB:** Оказался **самым стабильным и масштабируемым** решением на 4 потоках (ускорение 1.74x).
  В отличие от STL, oneTBB использует предварительно созданный пул потоков и `auto_partitioner`.
  Планировщик TBB смог частично нивелировать задержки на аллокацию памяти благодаря умной балансировке,
  обойдя OpenMP на максимальной аппаратной конкурентности.
- **ALL (MPI + Threads):** Ожидаемо показал замедление (ускорение 0.46x). На массиве в 500 000
  элементов издержки на упаковку данных, передачу через сокеты ОС и двунаправленные пересылки
  `MPI_Sendrecv` на порядки превышают время полезных процессорных тактов.

## 7. Репродуцируемость

Команды сборки (из корня проекта):

```bash
cmake -S . -B build -D USE_FUNC_TESTS=ON -D USE_PERF_TESTS=ON -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

Команды запуска функциональных тестов:

```bash
export PPC_NUM_THREADS=4
export PPC_NUM_PROC=2
python3 scripts/run_tests.py --running-type threads --counts 1 2 4
python3 scripts/run_tests.py --running-type processes --counts 2
```

Получение основных замеров производительности (на примере OMP и ALL):

```bash
# Тесты с общей памятью (OMP, TBB, STL)

./build/bin/ppc_perf_tests --gtest_filter="*RunPerfTests/*nikitina*omp_enabled*"

# Гибридные тесты (MPI + Потоки)
export PPC_NUM_PROC=2
export PPC_NUM_THREADS=4
./build/bin/ppc_perf_tests --gtest_filter="*RunPerfTests/*nikitina*all_enabled*"
```

## 8. Заключение

Выполненная лабораторная работа позволила провести комплексное кросс-технологическое исследование
на пяти различных архитектурных парадигмах. Полученные результаты наглядно демонстрируют
фундаментальное правило высокопроизводительных вычислений: **распараллеливание ради
распараллеливания на малых данных приводит к катастрофическому падению производительности.**

Подводя итоги сравнения технологий, можно сделать следующие инженерные выводы:

1. **Победа высокоуровневых абстракций (TBB):**
   На конфигурации из 4 потоков библиотека **oneTBB** оказалась единственной технологией,
   показавшей реальное ускорение (1.74x). Встроенный пул потоков и алгоритмы `work-stealing`
   успешно справились с накладными расходами операционной системы, доказав превосходство
   задачно-ориентированной архитектуры над прямым управлением потоками.

2. **Ограничения OpenMP и цена памяти:**
   Технология OpenMP показала феноменальную работу с кэшем на 2 потоках (ускорение 4.35x), но наглядно
   продемонстрировала проблему "узких мест". Динамическое выделение локальных векторов в фазе слияния
   Бэтчера привело к сериализации на уровне системного аллокатора при 4 потоках, полностью уничтожив
   алгоритмический выигрыш (падение до 0.86x). Это доказывает, что параллельный код требует строгой
   оптимизации работы с кучей.

3. **Фатальная цена ручного управления (STL):**
   Использование сырых `std::thread` без архитектуры пула задач потерпело полное фиаско. Замедление в
   5 раз (эффективность 5%) доказывает теорию: системные вызовы `clone` и `join` в Linux/Windows
   слишком дороги для вложенных циклов. В промышленном коде `std::thread` недопустимо использовать
   "на лету" внутри горячих циклов вычислений.

4. **Гибридные и распределенные системы (ALL / MPI):**
   Разработанная гибридная модель показала замедление 0.46x. Это классическая иллюстрация ограничений
   кластерных технологий: для массивов размером 2 МБ (500 000 `int`) цена пересылки через пайпы/сокеты
   ОС в десятки раз превышает стоимость тактов CPU. MPI оправдывает себя только там, где данные
   исчисляются гигабайтами.

**Векторы развития:**
Для дальнейшего улучшения масштабируемости необходимо:

- Полностью переписать этап компаратора `CompareSplit` на работу без динамической аллокации
  (использовать in-place слияние или статически предвыделенные буферы).
- Увеличить размер сортируемого массива до 10–50 миллионов элементов, что выведет время
  полезной работы за пределы статистического шума и системных издержек.
