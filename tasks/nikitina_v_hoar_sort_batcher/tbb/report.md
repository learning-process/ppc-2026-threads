## 1. Контекст

Технология TBB выбрана для данной задачи, поскольку она предоставляет парадигму параллелизма на основе
абстрактных задач, а не прямого управления системными потоками OS.
Алгоритм быстрой сортировки и сеть слияния Бэтчера характеризуются множеством независимых операций. В процессе
локальной сортировки разные блоки массива могут иметь различную степень изначальной упорядоченности. Это
приводит к дисбалансу вычислительной нагрузки: один блок может быть отсортирован быстрее другого. Библиотека
TBB идеально решает эту проблему благодаря встроенному планировщику с кражей задач.
Если один рабочий поток освобождается раньше времени, он автоматически забирает необработанные задачи из
очередей других, более загруженных потоков. Это обеспечивает равномерную утилизацию всех ядер процессора без
необходимости писать ручную логику балансировки. Кроме того, TBB оптимизирован для работы в C++, использует
современные парадигмы (лямбда-выражения) и минимизирует ложное разделение кэша при правильном выборе
размера зерна (grain size).

## 2. Постановка задачи

Необходимо разработать задачно-ориентированную параллельную реализацию алгоритма быстрой сортировки с
четно-нечетным слиянием Бэтчера с использованием библиотеки oneTBB. В качестве эталона для проверки
корректности и замера ускорения (baseline) выступает базовая последовательная реализация (SEQ).
Ключевые требования:

- Базовая сортировка блоков выполняется "на месте", однако для параллельного слияния (функция CompareSplit)
  допускается использование временных локальных буферов (O(N) дополнительной памяти на шаг сети) для
  обеспечения бесконфликтного слияния.
- Использовать высокоуровневые примитивы библиотеки TBB для распараллеливания как этапа локальной
  сортировки блоков, так и этапа слияния.
- Обеспечить корректную работу с массивами произвольного размера, применяя механизмы дополнения фиктивными
  элементами для соответствия требованиям сети Бэтчера.

## 3. Базовый алгоритм

Архитектура решения состоит из двух последовательных вычислительных фаз:

1. **Фаза локальной сортировки (QuickSort Hoare):**

Массив логически разделяется на независимые блоки (чанки). К каждому блоку применяется последовательный
алгоритм быстрой сортировки со схемой разбиения Хоара. Эта схема использует два встречных указателя, что
минимизирует количество обменов памяти и оптимизирует использование кэша L1/L2. Для исключения переполнения
системного стека на вырожденных данных алгоритм реализован итеративно с использованием
`std::vector<std::pair<int, int>>` в качестве стека границ.

2. **Фаза слияния (Batcher Odd-Even Merge):**

Сеть Бэтчера применяется для объединения отсортированных блоков в единый упорядоченный массив. Классическая
сеть Бэтчера требует, чтобы количество блоков было степенью двойки. Для этого исходный массив предварительно
дополняется максимальными значениями `std::numeric_limits<int>::max()` до размера, кратного выбранному
количеству блоков. На каждом шаге сети формируются независимые пары блоков, которые объединяются
специализированной функцией `CompareSplit`. По завершении всех шагов слияния добавленные фиктивные элементы
отсекаются.

## 4. Схема распараллеливания

Для распараллеливания алгоритма выбран примитив алгоритмического уровня `tbb::parallel_for`. Декомпозиция
работы разделена на два этапа.

**Листинг 1. Этап локальной сортировки:**

```cpp
// tbb/src/ops_tbb.cpp
tbb::parallel_for(tbb::blocked_range<int>(0, t), [&](const tbb::blocked_range<int> &r) {
  for (int i = r.begin(); i != r.end(); ++i) {
    QuickSortHoare(output, offsets[i], offsets[i + 1] - 1);
  }
});
```

**Листинг 2. Этап независимых компараторов сети Бэтчера:**

```cpp
// tbb/src/ops_tbb.cpp
tbb::parallel_for(tbb::blocked_range<int>(0, num_pairs), [&](const tbb::blocked_range<int> &r) {
  for (int idx = r.begin(); idx != r.end(); ++idx) {
    int block_a = pairs[idx].first;
    int block_b = pairs[idx].second;
    CompareSplit(output, offsets[block_a], offsets[block_a + 1] - offsets[block_a], 
                 offsets[block_b], offsets[block_b + 1] - offsets[block_b]);
  }
});
```

**Анализ задачно-ориентированной архитектуры:**

- **Примитив и диапазоны работы:** Выбран алгоритм `tbb::parallel_for`. В качестве пространства итераций
  передается `tbb::blocked_range<int>`. В первой фазе диапазон задается от `0` до `t` (количества блоков),
  во второй - от `0` до `num_pairs` (количества независимых пар компараторов на текущем шаге сети Бэтчера).
- **Grainsize (размер зерна):** Третий параметр grainsize явно не задан. В этом случае TBB использует
  эвристическое разбиение диапазона совместно с auto_partitioner.
  - В фазе сортировки блоков количество итераций `t` мало и равно уровню конкуренции (числу ядер). Воркеры
    сразу забирают по одному блоку, и накладные расходы на создание задач минимальны.
  - В фазе слияния `num_pairs` также невелико. Значение `grainsize = 1` позволяет рантайму максимально гибко
    распределять пары между воркерами.
- **Partitioner (планировщик):** Явный partitioner не передан, что заставляет TBB использовать
  `auto_partitioner`. Он опирается на эвристики и механизм work-stealing, автоматически разбивая
  `blocked_range` на подзадачи (chunks), если в пуле есть свободные воркеры. Это устраняет необходимость
  ручного подбора размера чанка и балансирует нагрузку.
- **Контроль конкуренции:** Пул потоков ограничивается на уровне тестового раннера через объект
  `tbb::global_control::max_allowed_parallelism` (который считывает `PPC_NUM_THREADS`). Внутри алгоритма
  этот лимит проверяется через `tbb::this_task_arena::max_concurrency()`. На основе этого числа
  вычисляется `t` - максимальная степень двойки, не превышающая доступное число потоков. Это гарантирует,
  что алгоритм не создаст избыточное число блоков.

## 5. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

Ключевой особенностью реализации на TBB является строгий контроль областей памяти для предотвращения
состояния гонки (data race) и минимизации ложного разделения кэша.

**Управление памятью и границами (Padding):**

Чтобы алгоритм Бэтчера работал корректно с массивами произвольной длины, применяется техника Padding.
Вычисляется количество недостающих элементов `pad`, и в конец вектора добавляются значения "бесконечности".

1. Листинг: Дополнение массива фиктивными элементами

```cpp
int pad = (t - (orig_n % t)) % t;
for (int i = 0; i < pad; ++i) {
  output.push_back(std::numeric_limits<int>::max());
}
int n = orig_n + pad;
std::vector<int> offsets(t + 1, 0);
int chunk = n / t;
for (int i = 0; i <= t; ++i) {
  offsets[i] = i * chunk;
}
```

Такой подход гарантирует, что каждый из `t` блоков будет иметь строго одинаковый размер `chunk`. Массив
`offsets` содержит точные индексы начала каждого блока. Каждая задача сортировки работает только в пределах
собственного диапазона массива `[offsets[i], offsets[i+1]-1]`. Благодаря обработке крупных непрерывных
диапазонов памяти вероятность false sharing снижается, поскольку потоки преимущественно работают с
независимыми участками массива.

**Слияние блоков (CompareSplit):**

Вспомогательная функция CompareSplit выполняет локальное слияние двух отсортированных блоков. Она объединяет
два отсортированных блока `left_block` и `right_block` во временные буферы, производит классическое слияние
с проверкой `left_block[p1] <= right_block[p2]` и записывает результат обратно в исходный массив по индексам
`write1` и `write2`.

2. Листинг: Вызов параллельного слияния (BatcherMergePhase)

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, num_pairs), [&](const tbb::blocked_range<int> &r) {
  for (int idx = r.begin(); idx != r.end(); ++idx) {
    int block_a = pairs[idx].first;
    int block_b = pairs[idx].second;
    CompareSplit(output, offsets[block_a], offsets[block_a + 1] - offsets[block_a], 
                 offsets[block_b], offsets[block_b + 1] - offsets[block_b]);
  }
});
```

По завершении метода `RunImpl` фиктивные элементы отсекаются с помощью `output.resize(orig_n);`.

## 6. Проверка корректности

Сравнение результатов выполнения алгоритма производится с последовательной реализацией (SEQ). Проверка
осуществляется средствами фреймворка Google Test. Основным критерием корректности является сохранение
инварианта упорядоченности структуры данных `std::ranges::is_sorted(output_data)`.

Тестовые сценарии охватывают:

1. **Граничные размеры (`orig_n <= 1`):** Метод `RunImpl` содержит явный guard-блок. Если элементов 1 или 0,
   алгоритм сразу возвращает `true`, не создавая задачи TBB и не выделяя память под `offsets`.

2. **Произвольные размеры массивов:** Функциональные тесты проверяют массивы, размер которых не является
   степенью двойки. Логика Padding со `std::numeric_limits<int>::max()` и последующим `resize` доказывает
   свою корректность: фиктивные "бесконечности" всегда всплывают в конец массива при сортировке и безопасно
   усекаются, не повреждая полезные данные.

3. **Худшие случаи:** Обратно отсортированные данные успешно перевариваются локальной функцией
   `QuickSortHoare` без переполнения стека благодаря вектору `stack`.

**Комментарий о balancing и overhead:**
Замеры демонстрируют ускорение 1.46 на 2 потоках и 1.74 на 4 потоках. В отличие от легковесной потоковой
модели, TBB-версия испытывает значительное влияние overhead (накладных расходов).
Узким местом в данной реализации является не сам планировщик TBB, а сложная архитектура подготовки данных:
выделение дополнительной памяти под padding (`push_back`), создание и копирование временных буферов
`left_block` и `right_block` внутри компаратора `CompareSplit`. Эти аллокации памяти занимают существенное
время. Балансировщик TBB отлично распределяет задачи, но он не способен ускорить линейные операции
выделения памяти системным аллокатором.

## 9. Выводы

Технология oneTBB предоставляет мощный и удобный интерфейс parallel_for вместе с blocked_range, что позволяет
абстрагироваться от ручного управления пулом потоков и избежать жесткой привязки архитектуры к конкретному
железу. Использование auto_partitioner и размера зерна по умолчанию показало себя корректным выбором для
тяжеловесных задач локальной сортировки.

Однако эксперимент наглядно показал, когда overhead мешает. Для объединения блоков сети Бэтчера в функции
CompareSplit алгоритму приходится копировать память во временные вектора. На небольшом массиве в 500 000
элементов цена этих аллокаций памяти съедает значительную долю выигрыша от параллелизма, в результате чего
эффективность на 4 потоках падает до 44%.

TBB значительно удобнее потоковых парадигм при работе с графами задач и неравномерными вычислениями. Для
раскрытия полного потенциала этой реализации в реальных условиях требуется либо переписать CompareSplit без
аллокаций промежуточных буферов (с использованием tbb::scalable_allocator или чистого in-place merge), либо
существенно увеличить размер входного вектора (до миллионов элементов), чтобы доля полезных вычислений
многократно превысила штрафы на работу с динамической памятью.
