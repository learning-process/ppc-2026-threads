## 1. Контекст

Гибридная схема (MPI + STL) является отражением реальной архитектуры современных суперкомпьютеров, которые
представляют собой кластеры вычислительных узлов. Каждый узел имеет общую память и множество многоядерных
процессоров, а сами узлы связаны друг с другом через сеть.

Использование чистого MPI на всех ядрах кластера ведет к избыточному потреблению оперативной памяти (каждый
MPI-процесс требует своих буферов) и перегрузке сетевого интерфейса локальными пересылками. С другой стороны,
чистые потоковые парадигмы (OpenMP, TBB, STL) не способны выйти за пределы одного физического узла. Гибридная
реализация решает эту проблему: MPI используется исключительно для распределения крупных блоков данных между узлами
кластера и коммуникации на этапах сети Бэтчера, а внутри каждого процесса std::thread используется для
параллельной предварительной сортировки локальных диапазонов внутри MPI-процесса.

## 2. Постановка задачи

Необходимо разработать двухуровневую гибридную параллельную реализацию алгоритма сортировки. В качестве эталона
производительности и корректности (baseline) выступает базовая последовательная реализация (SEQ).

Ключевые требования:

- Спроектировать межпроцессную схему обмена данными (Scatter/Gather).
- Адаптировать сеть слияния Бэтчера для работы поверх коммуникаторов MPI.
- Внутри каждого MPI-ранга использовать `std::thread` для локальной сортировки.
- Обеспечить строгую согласованность и отсутствие взаимоблокировок при пересылках.

## 3. Базовый алгоритм

В основе гибридной реализации лежат два классических алгоритма, адаптированных под иерархическую модель
параллелизма (где процессы взаимодействуют по сети, а потоки — в общей памяти узла):

1. **Фаза внутриузловой локальной сортировки (Алгоритм Хоара):**
Вместо рекурсии, которая может привести к переполнению стека на обратно отсортированных данных, используется
итеративная реализация быстрой сортировки Хоара. В качестве структуры для отслеживания нерешенных задач
применяется локальный `std::vector<std::pair<int, int>> stack`. Этот алгоритм работает исключительно с
локальным буфером оперативной памяти.

2. **Фаза межузловой сети слияния (Бэтчер):**
Классическая сеть четно-нечетного слияния Бэтчера генерирует математически строгую последовательность
компараторов. В базовом последовательном алгоритме компаратор сравнивает два элемента. В гибридной адаптации
каждый "компаратор" сравнивает и сливает **целые блоки памяти**, принадлежащие двум разным MPI-процессам.
Сеть строится на основе двух вложенных циклов шагов: `p_step` (степени двойки, определяющие размах подсетей)
и `k_step` (определяющие дистанцию обмена).

## 4. Межпроцессная схема

Взаимодействие между процессами (MPI) используется исключительно для крупноблочного распределения данных
и финального объединения результатов.

**Инициализация и распределение данных:**

- Узел `Rank 0` читает общий размер задачи `total_n` и рассылает его всем процессам через широковещательный
  вызов `MPI_Bcast`.
- Чтобы разделить данные поровну, размер локального буфера рассчитывается как
  `chunk = (total_n + mpi_size - 1) / mpi_size`.
- Недостающие элементы в буфере `send_buffer` на `Rank 0` заполняются "бесконечностью"
  (`std::numeric_limits<int>::max()`).
- Разделение происходит коллективной операцией `MPI_Scatter`.

**Сеть Бэтчера на уровне процессов:**
Самая ресурсоемкая межпроцессная логика сосредоточена в функции `BatcherExchange`. На каждом шаге Бэтчера
процессы вычисляют свои парные индексы `r1` и `r2`. Если текущий ранг совпадает с одним из них, он вступает
в обмен:

Листинг 1: Определение MPI-пар для обмена

```cpp
void BatcherExchange(std::vector<int> &local_data, int mpi_rank, int mpi_size, int p_step, int k_step) {
  for (int j_idx = k_step % p_step; j_idx + k_step < mpi_size; j_idx += (k_step << 1)) {
    for (int i_idx = 0; i_idx < std::min(k_step, mpi_size - j_idx - k_step); ++i_idx) {
      int r1 = j_idx + i_idx;
      int r2 = j_idx + i_idx + k_step;
      if (r1 / (p_step << 1) == r2 / (p_step << 1)) {
        if (mpi_rank == r1) {
          MpiCompareSwap(local_data, r2, true);
        } else if (mpi_rank == r2) {
          MpiCompareSwap(local_data, r1, false);
        }
      }
    }
  }
}
```

**Безопасный компаратор (Deadlock-free Swap):**
Для предотвращения взаимной блокировки (когда оба процесса вызывают `MPI_Send` и ждут ответа) реализована
функция `MpiCompareSwap` с использованием атомарного `MPI_Sendrecv`.

Листинг 2: Логика распределенного компаратора

```cpp
void MpiCompareSwap(std::vector<int> &local_arr, int neighbor, bool keep_low) {
  int size = static_cast<int>(local_arr.size());
  std::vector<int> neighbor_arr(size);
  MPI_Sendrecv(local_arr.data(), size, MPI_INT, neighbor, 0, neighbor_arr.data(), size, MPI_INT, neighbor, 0,
               MPI_COMM_WORLD, MPI_STATUS_IGNORE);
               
  std::vector<int> full_merge(static_cast<size_t>(size) * 2);
  std::ranges::merge(local_arr, neighbor_arr, full_merge.begin());

  if (keep_low) {
    std::copy(full_merge.begin(), full_merge.begin() + size, local_arr.begin());
  } else {
    std::copy(full_merge.begin() + size, full_merge.end(), local_arr.begin());
  }
}
```

*Пояснение:* Оба взаимодействующих процесса получают данные соседа. Затем каждый локально делает слияние
(`std::ranges::merge`). Процесс с меньшим рангом (`keep_low = true`) оставляет себе первую половину
отсортированного массива, а сосед (`keep_low = false`) — вторую.

## 5. Внутрипроцессная схема

Каждый MPI-процесс должен отсортировать свой `local_data` до входа в сеть обменов Бэтчера. Эта задача
перекладывается на внутриузловой параллелизм `std::thread`.

**Динамическое выделение потоков:**
Процесс запрашивает количество доступных аппаратных потоков ОС с помощью `std::thread::hardware_concurrency()`
и передает это значение во вспомогательную функцию `ThreadedSort`.

**Распределение задач и слияние:**
В отличие от межпроцессного MPI-уровня, здесь потоки работают с единым массивом, разделенным на логические сегменты:

Листинг: Многопоточная сортировка внутри MPI-ранга

```cpp
void ThreadedSort(std::vector<int> &arr, int num_threads) {
  int size = static_cast<int>(arr.size());
  if (size <= 1) return;
  
  int active_threads = std::min(num_threads, size / 2);
  std::vector<std::thread> threads;
  int chunk_size = size / active_threads;
  
  for (int iter = 0; iter < active_threads; ++iter) {
    int start = iter * chunk_size;
    int end = (iter == active_threads - 1) ? size - 1 : (start + chunk_size - 1);
    threads.emplace_back([&arr, start, end]() { QuickSortHoare(arr, start, end); });
  }
  for (auto &thr : threads) thr.join();

  QuickSortHoare(arr, 0, size - 1);
}
```

*Пояснение:*

1. `local_data` делится на `active_threads` частей. Лямбда-функция захватывает границы `start` и `end` по
значению и вызывает быструю сортировку независимо в каждом потоке. Барьером выступает цикл `thr.join()`.
2. После завершения работы всех потоков, массив состоит из нескольких отсортированных кусков, но в целом
не упорядочен. Поэтому вызывается финальный проход `QuickSortHoare(arr, 0, size - 1)` в главном потоке
для окончательного упорядочивания локального блока перед выходом в MPI-сеть.

Листинг 1. Иерархия коллективных вызовов (фрагмент `RunImpl`)

*Пояснение:* Строгий порядок коллективных операций заменяет явные барьеры `MPI_Barrier`. Коллективные операции
MPI_Scatter и MPI_Gather требуют участия всех процессов коммуникатора и тем самым обеспечивают согласованное
распределение и сбор данных между этапами алгоритма. Дополнительный MPI_Barrier в реализации не используется.

Листинг 2. Распределенный компаратор `MpiCompareSwap`:

```cpp
void MpiCompareSwap(std::vector<int> &local_arr, int neighbor, bool keep_low) {
  int size = static_cast<int>(local_arr.size());
  std::vector<int> neighbor_arr(size);
  MPI_Sendrecv(local_arr.data(), size, MPI_INT, neighbor, 0, 
               neighbor_arr.data(), size, MPI_INT, neighbor, 0,
               MPI_COMM_WORLD, MPI_STATUS_IGNORE);
               
  std::vector<int> full_merge(static_cast<size_t>(size) * 2);
  std::ranges::merge(local_arr, neighbor_arr, full_merge.begin());

  if (keep_low) {
    std::copy(full_merge.begin(), full_merge.begin() + size, local_arr.begin());
  } else {
    std::copy(full_merge.begin() + size, full_merge.end(), local_arr.begin());
  }
}
```

*Пояснение:* Это главная математическая инновация гибридной версии. Процесс обменивается своим блоком
с процессом `neighbor` с помощью безопасного двунаправленного вызова `MPI_Sendrecv`. Оба процесса локально
сливают два отсортированных блока в единый массив `full_merge` функцией `std::ranges::merge`. Затем один
процесс забирает себе младшую половину (`keep_low == true`), а второй — старшую.

**Потенциальные узкие места:**

1. **Перемещение памяти:** `MpiCompareSwap` требует выделения памяти под `neighbor_arr` и двойного размера
`full_merge` на каждом шаге Бэтчера.
2. **Внутрипроцессная досортировка:** В `ThreadedSort` финальное объединение чанков делается через вызов
`QuickSortHoare` поверх всего массива, что создает дополнительную вычислительную нагрузку O(N log N),
тогда как специализированное линейное слияние отсортированных блоков могло бы быть эффективнее.

## 7. Проверка корректности

Сравнение производилось с эталоном (SEQ) и чистыми backend-ами.
Проверка на согласованность результата между рангами гарантируется архитектурой: `Rank 0` собирает данные
через `MPI_Gather` и именно он присваивает финальный результат.

Важной деталью корректности гибридной версии является строгий контроль областей памяти и очистка мусора
на вспомогательных узлах. Это реализовано в `PostProcessingImpl`:

Листинг: Постобработка и очистка памяти

```cpp
bool HoareSortBatcherALL::PostProcessingImpl() {
  int mpi_rank = 0;
  MPI_Comm_rank(MPI_COMM_WORLD, &mpi_rank);

  if (mpi_rank != 0) {
    data_.clear();
  }

  GetOutput() = data_;
  return true;
}
```

Также успешно пройдены критические случаи:

1. **Размер 0:** Вызов `if (total_n == 0) return true;` синхронизирован через предварительный `MPI_Bcast`,
что предотвращает деление на ноль при вычислении `chunk`.
2. **Некратные размеры (Padding):** Тесты подтверждают, что добавление `INT_MAX` в неполные куски на этапе
`MPI_Scatter` не ломает алгоритм. На последнем этапе в главном потоке `gather_buffer.resize(total_n);`
успешно отсекает все эти фиктивные значения.

## 9. Результаты

Замеры `task_run` проводились на массиве из 500 000 элементов. Базовое время (SEQ) = `0.002168` сек.
Эффективность рассчитывалась как ускорение, деленное на `total_workers`.

| Ranks | Threads_per_rank | Total Workers | Время (сек) | Ускорение (Speedup) | Эффективность (Efficiency) |
| ----- | ---------------- | ------------- | ----------- | ------------------- | -------------------------- |
| 1     | 1 (SEQ baseline) | 1             | 0.002123    | 1.00                | N/A                        |
| 2     | 2                | 4             | 0.004776    | 0.44 (замедление)   | 0.11                       |
| 2     | 4                | 8             | 0.004643    | 0.46 (замедление)   | 0.06                       |

**Комментарий о цене коммуникации и синхронизации:**
Результаты показывают, что гибридная версия уступает последовательной реализации по времени выполнения:
ускорение остаётся ниже 1 даже при увеличении общего числа вычислителей до `2 × 4 = 8 workers`. Для задачи
такого масштаба (500 000 элементов) подобное поведение является ожидаемым.

Основные накладные расходы связаны не с самой сортировкой, а с организацией гибридного выполнения:
межпроцессными MPI-коммуникациями (`MPI_Scatter`, `MPI_Sendrecv`, `MPI_Gather`), синхронизацией процессов
и потоков, а также дополнительными копированиями памяти внутри `MpiCompareSwap`. Кроме того, каждый шаг
распределённой сети Бэтчера требует выделения временных буферов `neighbor_arr` и `full_merge`, что создаёт
дополнительный overhead на динамическое управление памятью.

При переходе от конфигурации `2 × 2` к `2 × 4` наблюдается лишь незначительное улучшение времени
(`0.004776 → 0.004643` сек), поскольку стоимость коммуникаций и синхронизации начинает доминировать
над полезной вычислительной нагрузкой.

## 10. Выводы

Разработанная гибридная схема успешно доказала свою архитектурную корректность: удалось подружить сложную
математику распределенного слияния Бэтчера с внутриузловыми потоками `std::thread`. Реализованный
`MpiCompareSwap` является надежным решением для распределенной сортировки.

Однако замеры производительности наглядно очерчивают границы применимости гибридных схем: они совершенно
не оправданы для задач малого и среднего объема, которые могут полностью поместиться в оперативной памяти
и кэше одного узла (как наши 500 000 элементов). Гибридная схема (MPI + Threads) раскрывает свой потенциал
исключительно на сверхбольших данных (десятки и сотни миллионов элементов), распределенных между физически
разными машинами кластера, где цена сетевого обмена нивелируется колоссальными объемами параллельных
вычислений.
