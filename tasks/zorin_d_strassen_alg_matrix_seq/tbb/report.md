## 1. Контекст

У Штрассена хорошо видны независимые ветви, поэтому oneTBB здесь выглядит вполне уместно.

## 2. Постановка задачи

Задача та же, что и в `SEQ`: умножить две квадратные матрицы `double` алгоритмом Штрассена. В качестве эталона используется последовательная версия из [seq/report.md](/C:/Users/N1lr/ppc-2026-threads/tasks/zorin_d_strassen_alg_matrix_seq/seq/report.md).

## 3. Базовый алгоритм

Базовая логика не меняется:

- при необходимости матрицы дополняются до степени двойки;
- результат строится через семь промежуточных произведений `M1..M7`;
- для малых размеров включается обычное блочное умножение;
- после вычисления padding отбрасывается.

В этой реализации порог перехода к внутренней последовательной части также равен `kCutoff = 256`.

## 4. Схема распараллеливания

В отличие от классических примеров с `parallel_for`, здесь естественнее использовать не диапазоны, а набор независимых задач. Поэтому в коде выбран примитив `oneapi::tbb::parallel_invoke`.

Что именно делается:

- каждая из ветвей для `M1..M7` оформлена как отдельная лямбда;
- `parallel_invoke` запускает их параллельно и сам дожидается завершения;
- затем уже выполняется сборка блоков `C11..C22`.

Что важно по методике:

- `blocked_range`, `grainsize` и `partitioner` здесь специально не используются, потому что эта версия построена не на `parallel_for`, а на `parallel_invoke`;
- контроль конкуренции есть: число рабочих потоков ограничивается через `oneapi::tbb::global_control::max_allowed_parallelism`;
- это сделано, чтобы не раздувать конкуренцию и не получать шум в сравнении с другими backend-ами.

То есть технический фокус этого backend-а не в делении диапазона, а в управлении независимыми задачами верхнего уровня.

## 5. Детали реализации

Файлы:

- `tasks/zorin_d_strassen_alg_matrix_seq/tbb/include/ops_tbb.hpp`
- `tasks/zorin_d_strassen_alg_matrix_seq/tbb/src/ops_tbb.cpp`

По сравнению с `SEQ`:

- добавлен `StrassenTopTbb(...)`;
- верхний уровень вычислений `M1..M7` обёрнут в `parallel_invoke`;
- число потоков ограничивается объектом `global_control`;
- внутренняя последовательная часть оставлена в `StrassenSeqImpl(...)`.

С точки зрения данных всё осталось почти так же, как в `SEQ`: матрицы лежат в линейных векторах, подматрицы адресуются через указатели и `stride`, а для промежуточных результатов создаются отдельные буферы.

`TBB` оказался удобным именно там, где есть набор крупных независимых вычислений. Для этой задачи `parallel_invoke` смотрится логично и не требует вручную управлять каждым потоком.
