## 1. Контекст

Эта версия нужна была, чтобы проверить, как задача выглядит
без OpenMP и без отдельного runtime вроде TBB, а только
на стандартных средствах C++. По сути это попытка сделать
"чистую" потоковую реализацию на базе стандартной библиотеки.

## 2. Постановка задачи

Нужно вычислить произведение двух квадратных матриц `double`
алгоритмом Штрассена. По смыслу задача полностью совпадает с `SEQ`,
а baseline для сравнения остаётся тем же.

Важно сразу зафиксировать: хотя в методичке этот backend называется
`STL` и часто обсуждается рядом со `std::thread`, в моей реализации
конкретно используются `std::async` и `std::future`. Поэтому здесь нет
ручного `join()`, а синхронизация идёт через `future.get()`.

## 3. Базовый алгоритм

Математическая часть снова не меняется:

- разбиение матриц на четыре квадранта;
- вычисление `M1..M7`;
- сборка `C11..C22`;
- padding до степени двойки;
- переход к обычному блочному умножению на малых размерах.

Порог переключения на внутреннюю последовательную часть здесь такой же,
как в `OMP` и `TBB`: `kCutoff = 256`.

## 4. Схема распараллеливания

Граница распараллеливания проходит на верхнем уровне
вычисления `M1..M7`.

Что делает каждая рабочая единица:

- каждая ветвь `M1..M7` запускается через
  `std::async(std::launch::async, ...)`;
- каждая асинхронная задача пишет результат
  в свой собственный буфер `m1..m7`;
- после запуска всех задач главный поток вызывает `get()`
  для каждого `future`;
- только после этого собирается итоговая матрица.

Почему здесь не нужны mutex или atomic:

- каждая задача пишет только в свой локальный контейнер;
- записи в общую память нет;
- синхронизация нужна только в момент ожидания завершения задач.

То есть в этой версии карта "работник -> данные" очень простая:
одна асинхронная ветвь соответствует одному промежуточному произведению.

## 5. Детали реализации

Файлы:

- `tasks/zorin_d_strassen_alg_matrix_seq/stl/include/ops_stl.hpp`
- `tasks/zorin_d_strassen_alg_matrix_seq/stl/src/ops_stl.cpp`

Особенности реализации:

- используется `std::async` с политикой `std::launch::async`,
  чтобы задачи действительно могли стартовать параллельно;
- синхронизация выполняется через `f1.get() ... f7.get()`;
- внутренняя последовательная часть остаётся в `StrassenSeqImpl(...)`;
- входные данные перед запуском так же дополняются
  до степени двойки.

Проверялось:

- совпадение с `SEQ`;

- корректность padding и обратного обрезания результата;
- отсутствие расхождений по точности относительно эталонного
  обычного умножения.

Отдельной проверки на гонки через санитайзеры я не проводил,
но по структуре данных здесь задачи разделены по независимым буферам.
