## 1. Вступление

В этой работе я реализовал умножение плотных квадратных матриц типа `double` с помощью
алгоритма Штрассена. Кроме последовательной версии, были сделаны реализации для `OMP`,
`TBB`, `STL` и итоговая версия `ALL`.

Смысл алгоритма Штрассена в том, что при разбиении матриц на четыре блока он делает не
8 умножений блоков, как обычное блочное умножение, а 7. Из-за этого теоретически он быстрее
обычного алгоритма `O(n^3)`, но на практике это заметно только на достаточно больших матрицах.
На маленьких размерах дополнительные сложения, выделения временных массивов и запуск потоков
могут съесть весь выигрыш.

Ожидаемый результат работы - корректная реализация всех вариантов и сравнение их времени
в одинаковых условиях запуска.

## 2. Постановка задачи

Нужно вычислить произведение двух квадратных матриц:

```text
C = A * B
```

где `A`, `B` и `C` имеют размер `n x n`, а элементы матриц имеют тип `double`.

Входные данные:

- размер матрицы `n`;
- массив элементов матрицы `A`;
- массив элементов матрицы `B`.

Выходные данные:

- массив элементов матрицы `C`.

В реализации учитываются следующие случаи:

- размер `n` может быть не степенью двойки;
- для алгоритма Штрассена матрицы дополняются нулями до ближайшей степени двойки;
- после вычисления лишние строки и столбцы отбрасываются;
- для маленьких блоков используется обычное блочное умножение, потому что рекурсивный Штрассен на них работает невыгодно.

## 3. Последовательная версия (SEQ)

В последовательной версии используется рекурсивный алгоритм Штрассена.
Матрицы разбиваются на четыре подматрицы:

```text
A = | A11 A12 |, B = | B11 B12 |
    | A21 A22 |      | B21 B22 |
```

Дальше считаются семь промежуточных произведений:

```text
M1 = (A11 + A22) * (B11 + B22)
M2 = (A21 + A22) * B11
M3 = A11 * (B12 - B22)
M4 = A22 * (B21 - B11)
M5 = (A11 + A12) * B22
M6 = (A21 - A11) * (B11 + B12)
M7 = (A12 - A22) * (B21 + B22)
```

Потом из них собираются четыре блока результата:

```text
C11 = M1 + M4 - M5 + M7
C12 = M3 + M5
C21 = M2 + M4
C22 = M1 - M2 + M3 + M6
```

Сначала реализацию проще было сделать через явное копирование всех подматриц.
Такой вариант понятнее, но он создаёт много временных векторов. После этого основная часть
была переделана на работу с указателями и `stride`, чтобы не копировать исходные блоки матриц
на каждом уровне рекурсии.

Для базового случая используется блочное классическое умножение. Порог перехода к нему нужен,
потому что на малых размерах рекурсия Штрассена даёт слишком много накладных расходов.

## 4. Схема параллелизации

Главная идея распараллеливания одинаковая для всех параллельных версий:
произведения `M1..M7` независимы и могут вычисляться одновременно.
После их вычисления требуется синхронизация, потому что сборка блоков `C11..C22`
использует результаты всех нужных промежуточных матриц.

В `OMP`-версии независимые произведения запускаются как OpenMP-задачи.
Для директив OpenMP явно указаны области видимости переменных через `default(none)` и `shared(...)`.
Это важно для совместимости с MSVC и clang.

В `TBB`-версии используется `oneapi::tbb::parallel_invoke`.
Число потоков ограничивается через `oneapi::tbb::global_control`, чтобы не получать случайное
переподписывание потоков.

В `STL`-версии часть произведений запускается через `std::async(std::launch::async, ...)`,
а ожидание результата выполняется через `std::future::get()`.

В `ALL`-версии совмещены несколько технологий:

- `TBB` запускает часть независимых произведений через `parallel_invoke`;
- `STL` запускает часть произведений через `std::async`;
- `OpenMP` используется в базовом блочном умножении.

`ALL` не обязан быть самым быстрым вариантом. Его задача - показать совместную работу технологий.
При таком подходе появляется риск `oversubscription`, поэтому параллелизм ограничен верхними
уровнями алгоритма и не запускается бесконтрольно в каждой рекурсии.

## 5. Детали реализации

Основные файлы реализации:

- `tasks/zorin_d_strassen_alg_matrix_seq/common/include/common.hpp` — общие типы входа и выхода;
- `tasks/zorin_d_strassen_alg_matrix_seq/seq/src/ops_seq.cpp` — последовательная версия;
- `tasks/zorin_d_strassen_alg_matrix_seq/omp/src/ops_omp.cpp` — OpenMP-версия;
- `tasks/zorin_d_strassen_alg_matrix_seq/tbb/src/ops_tbb.cpp` — TBB-версия;
- `tasks/zorin_d_strassen_alg_matrix_seq/stl/src/ops_stl.cpp` — STL-версия;
- `tasks/zorin_d_strassen_alg_matrix_seq/all/src/ops_all.cpp` — комбинированная ALL-версия;
- `tasks/zorin_d_strassen_alg_matrix_seq/tests/functional/main.cpp` — функциональные тесты;
- `tasks/zorin_d_strassen_alg_matrix_seq/tests/performance/main.cpp` — тесты производительности.

Важные детали:

- матрицы хранятся в одномерных `std::vector<double>`;
- доступ к элементу выполняется как `row * n + col`;
- для подматриц используются указатели и `stride`;
- если `n` не степень двойки, вход дополняется нулями;
- после вычисления результат копируется обратно в размер `n x n`;
- временные матрицы нужны для сумм, разностей и произведений `M1..M7`;
- для малых блоков используется классическое блочное умножение.

При разработке была практическая проблема с OpenMP: MSVC требовал явно указать переменные
в `shared(...)`, если используется `default(none)`. Также при локальной сборке важно не смешивать
Debug и Release в одной папке, иначе появляются ошибки линковки по `_ITERATOR_DEBUG_LEVEL`
и `RuntimeLibrary`.

### 7.2 Производительность

Финальные числа нужно заносить после запуска тестов производительности на одной и той же машине,
в одной и той же Release-сборке. Если часть запусков выполнялась в Debug или после смешивания
конфигураций, такие результаты лучше не использовать.

Таблица ниже заполнена по запуску `mpiexec -n 1`, так как для этой задачи сравниваются
потоковые реализации внутри одного процесса. В качестве времени взято значение `task_run`,
потому что оно было выведено для всех реализаций.

| Mode | Count | Time, s | Speedup | Efficiency |
| --- | ---: | ---: | ---: | ---: |
| seq | 1 | 1.057882 | 1.00 | N/A |
| omp | 8 | 0.363791 | 2.91 | 36.4% |
| tbb | 8 | 0.427708 | 2.47 | 30.9% |
| stl | 8 | 0.404511 | 2.62 | 32.7% |
| all | 8 | 0.419864 | 2.52 | 31.5% |

Ускорение считается так:

```text
Speedup = T_seq / T_parallel
```

Эффективность считается так:

```text
Efficiency = Speedup / Count * 100%
```

По результатам видно, что все параллельные версии быстрее последовательной. Лучшее время в этом
запуске показала `OMP`-реализация. `STL`, `TBB` и `ALL` тоже дают ускорение, но немного уступают
из-за накладных расходов на запуск задач и работу планировщиков. Для `ALL` это ожидаемо, потому
что там вместе используются несколько механизмов параллелизма, и часть времени уходит на их
координацию.

Также был выполнен запуск через `mpiexec -n 8`. Для этой задачи он не является основным режимом
сравнения, потому что реализации используют потоковый параллелизм, а не распределяют данные между
MPI-процессами. В этом запуске времена стали хуже, что можно объяснить конкуренцией нескольких
процессов за одни и те же ресурсы процессора.

## 8. Выводы

В результате была реализована одна и та же задача для нескольких технологий параллелизма.
Сам алгоритм Штрассена во всех версиях остался тем же: отличие в основном в том, как запускаются
независимые произведения `M1..M7` и где используется базовое блочное умножение.

Наиболее понятной для реализации оказалась `OMP`-версия, потому что независимые задачи хорошо
ложатся на OpenMP tasks. `TBB` удобен для такого типа задач, но требует отдельного управления
числом потоков. `STL`-версия проще по зависимостям, но менее предсказуема по управлению потоками.
`ALL`-версия полезна как итоговая интеграция, но требует осторожности из-за возможного
переподписывания потоков.

Главный практический вывод: для алгоритма Штрассена важно не просто добавить потоки, а ограничить
глубину параллелизма и вовремя переходить к обычному умножению блоков.
