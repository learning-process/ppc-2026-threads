## 1. Введение

Разреженные матрицы встречаются в физическом моделировании, машинном обучении
и алгоритмах на графах. Их особенность: подавляющее число элементов равно нулю,
поэтому хранить их полностью нецелесообразно.

Формат CCS (Compressed Column Storage) хранит только ненулевые элементы,
упорядоченные по столбцам.

Последовательный алгоритм умножения разреженных матриц в формате CCS обрабатывает столбцы результирующей матрицы один за
другим. При больших размерах матриц это начинает занимать много времени, причем столбцы можно обрабатывать независимо.
Библиотека Intel TBB предлагает следующий подход:
задачи делятся на поддиапазоны, а планировщик динамически балансирует
нагрузку через механизм похищения задач (work stealing). Это особенно
выгодно при неравномерной загрузке столбцов, когда одни столбцы матрицы B
значительно плотнее других.

Данная работа описывает параллельную реализацию умножения разреженных
матриц в формате CCS средствами Intel TBB. Алгоритм повторяет двухфазный
подход версии OpenMP, но управление потоками и локальными данными
возложено на инструменты TBB: `tbb::parallel_for` и
`tbb::enumerable_thread_specific`.

## 2. Постановка задачи

Требуется вычислить произведение **C = A × B**, где:

- **A** — разреженная матрица размером `n × k` в формате CCS;
- **B** — разреженная матрица размером `k × m` в формате CCS;
- **C** — результирующая разреженная матрица размером `n × m` в формате CCS.

Требования к реализации:

- Параллельное вычисление с помощью `tbb::parallel_for`.
- Входные данные передаются как `std::tuple<CCS, CCS>`, результат записывается в `OutType` (`CCS`).
- Строки внутри каждого столбца результата упорядочены по возрастанию.
- Валидация входных данных: `a.m == b.n` (число столбцов A равно числу строк B).

## 3. Алгоритм

### 3.1. Двухфазный подход

Параллельная запись в общий выходной массив требует заранее известных смещений для каждого столбца. В последовательной
версии это невозможно, потому что число ненулевых в каждом столбце C определяется только в процессе вычисления. Для
решения проблемы алгоритм разбивается на два прохода.

**Фаза 1 (MatMultPhase1):** подсчёт числа ненулевых элементов в каждом столбце C без вычисления значений. По результатам
строится префиксная сумма — массив `col_ind` итоговой матрицы.

**Фаза 2 (MatMultPhase2):** вычисление значений и запись непосредственно в выходные массивы `c.row` и `c.value` по
заранее известным смещениям из `col_ind`.

### 3.2. Маркер с меткой времени (stamp)

Для отслеживания посещённых строк используется массив `was` размером `a.n`, общий для всего времени жизни потока. Вместо
явного сброса (`memset`) каждый столбец `j` получает уникальную метку (stamp):

- Фаза 1: `stamp = j`, диапазон `[0, b.m)`.
- Фаза 2: `stamp = b.m + j`, диапазон `[b.m, 2·b.m)`.

Строка `i` считается посещённой, если `was[i] == stamp`. Диапазоны не пересекаются, поэтому один и тот же массив `was`
переиспользуется в обеих фазах без повторной инициализации.

### 3.3. Сложность

Сильно зависит от степени разреженности входных матриц.

| Характеристика | Оценка |
| --- | --- |
| Вычислительная (обе фазы) | O(nnz(B) × avg\_col\_nnz(A)) |
| Дополнительная память (на поток) | O(n) |
| Последовательный участок | O(b.m) — префиксная сумма |

## 4. Схема распараллеливания

TBB разбивает диапазон столбцов `[0, b.m)` на блоки (`blocked_range`) и
распределяет их между потоками пула через work-stealing планировщик.
Число потоков определяется автоматически по числу логических ядер.

    ┌──────────────────────────────────────────────────────────────┐
    │  tbb::parallel_for(blocked_range(0, b.m))                    │
    │  tls_was: enumerable_thread_specific                         │
    │                                                              │
    │  ┌──────────┬──────────┬──────────┬──────────┐              │
    │  │ chunk 0  │ chunk 1  │   ...    │ chunk N  │  потоки      │
    │  │ Phase 1  │ Phase 1  │  Phase 1 │ Phase 1  │              │
    │  └──────────┴──────────┴──────────┴──────────┘              │
    │              work stealing при дисбалансе                    │
    └──────────────────────────────────────────────────────────────┘
               │  барьер (конец parallel_for)
               ▼
    ┌──────────────────────────────────────────────────────────────┐
    │  Последовательно: префиксная сумма col_nnz → col_ind        │
    │  Выделение c.value, c.row                                    │
    └──────────────────────────────────────────────────────────────┘
               │
               ▼
    ┌──────────────────────────────────────────────────────────────┐
    │  tbb::parallel_for(blocked_range(0, b.m))                    │
    │  tls_was (переиспользуется), tls_accum, tls_rows             │
    │                                                              │
    │  ┌──────────┬──────────┬──────────┬──────────┐              │
    │  │ chunk 0  │ chunk 1  │   ...    │ chunk N  │  потоки      │
    │  │ Phase 2  │ Phase 2  │  Phase 2 │ Phase 2  │              │
    │  └──────────┴──────────┴──────────┴──────────┘              │
    └──────────────────────────────────────────────────────────────┘

Каждый поток пишет в непересекающиеся позиции `c.value` и `c.row`,
определённые через `c.col_ind[j]`. Гонки данных нет.

## 5. Детали реализации

### 5.1. Архитектура

Файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

### ValidationImpl

Проверка условия `a.m == b.n`.

### PreProcessingImpl

Все входные данные уже готовы.

### RunImpl

Фаза 1 → префиксная сумма → фаза 2.

### PostProcessingImpl

Результат уже в готовом виде; корректность проверяется в тестовых файлах.

### 5.2. enumerable_thread_specific

Ключевое отличие от OpenMP-версии — хранение thread-local данных через
`tbb::enumerable_thread_specific<T>` (ETS). ETS создаёт локальную копию
вектора по требованию: при первом вызове `.local()` данным потоком. Это исключает
лишние аллокации для потоков, которые не участвуют в конкретном вызове
`parallel_for`.

| Объект ETS | Тип элемента | Фаза | Назначение |
| --- | --- | --- | --- |
| `tls_was` | `vector<size_t>`, init = `max` | 1 и 2 | Маркер посещённых строк |
| `tls_accum` | `vector<double>`, init = `0.0` | 2 | Вещественный аккумулятор |
| `tls_rows` | `vector<size_t>`, init = `{}` | 2 | Список посещённых строк столбца |

`tls_was` объявляется до фазы 1 и переиспользуется в фазе 2 — `stamp`
двух фаз не пересекаются, а новые потоки, если они появятся во второй
фазе, получат копию, инициализированную значением `max`.

### 5.3. Прямая запись в результат

В фазе 2 каждый поток пишет в `c.value[c.col_ind[j] .. c.col_ind[j+1]-1]` напрямую, без промежуточных буферов. Это
исключает лишние аллокации и дополнительный проход копирования.

## 6. Тесты

Корректность работы алгоритма проверялась функциональными тестами, а производительность — перфоманс-тестами.

### 6.1. Функциональные тесты

Матрица 6×6 с 12 ненулевыми элементами умножается сама на себя. Результат вычислен вручную и переведён в CCS — это
эталонное значение матрицы C.

### 6.2. Перфоманс-тесты

Создаются две ленточные матрицы A и B. Для проверки корректности используется метод умножения справа на произвольный
вектор:

`(A × B) × x = A × (B × x)`

Если результирующие векторы совпадают с учётом погрешности машинной арифметики, матрицы равны.

накладными расходами на создание потоков, эмуляцией через `Docker`, а также большей "борьбой" потоков за общий L3-кэш и
шину данных.

## Приложение

### Фаза 1 — `MatMultPhase1`

    inline void MatMultPhase1(size_t j, const CCS &a, const CCS &b,
                              std::vector<size_t> &was,
                              std::vector<size_t> &col_nnz) {
      size_t count = 0;
      for (size_t k = b.col_ind[j]; k < b.col_ind[j + 1]; ++k) {
        const size_t b_row = b.row[k];
        for (size_t zc = a.col_ind[b_row]; zc < a.col_ind[b_row + 1]; ++zc) {
          const size_t a_row = a.row[zc];
          if (was[a_row] != j) {
            was[a_row] = j;
            ++count;
          }
        }
      }
      col_nnz[j] = count;
    }

### Фаза 2 — `MatMultPhase2`

    inline void MatMultPhase2(size_t j, const CCS &a, const CCS &b, CCS &c,
                              size_t stamp, std::vector<size_t> &was,
                              std::vector<double> &accum,
                              std::vector<size_t> &rows) {
      rows.clear();
      for (size_t k = b.col_ind[j]; k < b.col_ind[j + 1]; ++k) {
        const double b_val = b.value[k];
        const size_t b_row = b.row[k];
        for (size_t zc = a.col_ind[b_row]; zc < a.col_ind[b_row + 1]; ++zc) {
          const size_t a_row = a.row[zc];
          accum[a_row] += a.value[zc] * b_val;
          if (was[a_row] != stamp) {
            was[a_row] = stamp;
            rows.push_back(a_row);
          }
        }
      }
      std::ranges::sort(rows);
      size_t write = c.col_ind[j];
      for (const size_t i : rows) {
        c.row[write] = i;
        c.value[write] = accum[i];
        accum[i] = 0.0;
        ++write;
      }
    }
