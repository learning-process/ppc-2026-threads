## 1. Введение

Разреженные матрицы встречаются в физическом моделировании, машинном обучении
и алгоритмах на графах. Их особенность: подавляющее число элементов равно нулю,
поэтому хранить их полностью нецелесообразно.

Формат CCS (Compressed Column Storage) хранит только ненулевые элементы,
упорядоченные по столбцам.

Последовательный алгоритм умножения разреженных матриц в формате CCS обрабатывает столбцы результирующей матрицы один за
другим. При больших размерах матриц это начинает занимать много времени, причем столбцы можно обрабатывать независимо.

Версии OpenMP, TBB, STL распараллеливают вычисления в рамках одного
узла — числом потоков, равным числу логических ядер процессора. Для
задач, требующих масштабирования за пределы одного узла кластера, применяют
MPI: каждый процесс получает свою часть данных и работает независимо.

Совмещение MPI и OpenMP даёт двухуровневый параллелизм: MPI разделяет
работу между процессами, а OpenMP распараллеливает её
внутри каждого процесса. Данная работа описывает такую реализацию
умножения разреженных матриц в формате CCS.

## 2. Постановка задачи

Требуется вычислить произведение **C = A × B**, где:

- **A** — разреженная матрица размером `n × k` в формате CCS;
- **B** — разреженная матрица размером `k × m` в формате CCS;
- **C** — результирующая разреженная матрица размером `n × m` в формате CCS.

Требования к реализации:

- Межпроцессное взаимодействие через MPI; внутрипроцессная параллельность через OpenMP.
- Входные данные передаются как `std::tuple<CCS, CCS>`, результат записывается в `OutType` (`CCS`).
- Строки внутри каждого столбца результата упорядочены по возрастанию.
- Валидация входных данных: `a.m == b.n` (число столбцов A равно числу строк B).

## 3. Алгоритм

### 3.1. Двухуровневое разбиение задачи

**Уровень 1 — MPI:** столбцы матрицы B делятся между процессами. Каждый
процесс умножает A на свою часть B и получает соответствующие столбцы C.

**Уровень 2 — OpenMP:** внутри каждого процесса вычисление своей части
ведётся параллельно по потокам двухфазным методом.

### 3.2. Балансировка нагрузки между процессами

Так как столбцы матрицы B имеют разное число ненулевых элементов, то
и вычислительные затраты на них будут разные. Таким образом простое равномерное разбиение столбцов по числу процессов
давало бы не самый оптимальный результат.

Функция `EstimateColumnCost` оценивает стоимость j-го столбца как суммарное
число ненулевых элементов в соответствующих столбцах A:

    cost(j) = Σ (a.col_ind[b.row[k]+1] - a.col_ind[b.row[k]])
                для k в [b.col_ind[j], b.col_ind[j+1])

Функция `BuildBalancedStarts` строит разбиение по префиксным суммам
весов так, чтобы суммарная вычислительная сложность работы каждого процесса была примерно
одинакова.

### 3.3. Двухфазный подход внутри процесса

Тот же, что в версии OpenMP. Небольшое отличие в том, что временная метка `stamp` фазы 2 использует `local_cols + j`
(а не `b.m + j`), так как каждый процесс работает со своим локальным
диапазоном столбцов.

- Фаза 1: stamp = `j`, диапазон `[0, local_cols)`.
- Фаза 2: stamp = `local_cols + j`, диапазон `[local_cols, 2·local_cols)`.

### 3.4. Сложность

| Характеристика | Оценка |
| --- | --- |
| Вычислительная (на процесс) | O(nnz(B\_local) × avg\_col\_nnz(A)) |
| Дополнительная память (на поток) | O(n) |

## 4. Схема распараллеливания

    Процесс 0 (rank=0)                  Процессы 1..P-1
    ┌────────────────────────────┐       ┌────────────────────┐
    │ EstimateColumnCost для B   │       │                    │
    │ BuildBalancedStarts        │       │                    │
    └────────────┬───────────────┘       └────────────────────┘
                 │ MPI_Bcast(col_starts)
                 │ BcastCCS(A)           ──────────────────────►
                 │ ScatterB(B_local)     ──────────────────────►
                 ▼
    ┌────────────────────────────────────────────────────────────┐
    │  Каждый процесс независимо:                                │
    │                                                            │
    │  #pragma omp parallel  (Phase 1)                           │
    │  ┌─────────┬─────────┬─────────┬─────────┐                │
    │  │ thread0 │ thread1 │   ...   │ threadN │                │
    │  │ Phase 1 │ Phase 1 │ Phase 1 │ Phase 1 │                │
    │  └─────────┴─────────┴─────────┴─────────┘                │
    │         │ барьер                                           │
    │         ▼                                                  │
    │  Префиксная сумма → local_c.col_ind                        │
    │         │                                                  │
    │  #pragma omp parallel  (Phase 2)                           │
    │  ┌─────────┬─────────┬─────────┬─────────┐                │
    │  │ thread0 │ thread1 │   ...   │ threadN │                │
    │  │ Phase 2 │ Phase 2 │ Phase 2 │ Phase 2 │                │
    │  └─────────┴─────────┴─────────┴─────────┘                │
    └────────────────────────────────────────────────────────────┘
                 │ GatherC(local_c → c)  ◄─────────────────────
                 │ PostProcessingImpl: MPI_Bcast(C)
                 ▼
    Результат C на всех процессах

Гонки данных нет: внутри процесса каждый OMP-поток пишет в
непересекающийся диапазон `local_c`, определённый через `local_c.col_ind[j]`.
Между процессами данные не пересекаются по определению разбиения.

## 5. Детали реализации

### 5.1. Архитектура

Файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

#### ValidationImpl

Проверка условия `a.m == b.n` только на процессе 0; остальные возвращают `true`.

#### PreProcessingImpl

Все входные данные уже готовы.

#### RunImpl

Балансировка → `BcastCCS(A)` → `ScatterB` → OMP Phase 1 →
префиксная сумма → OMP Phase 2 → `GatherC`.

#### PostProcessingImpl

`MPI_Bcast` итоговой матрицы C с корневого процесса на все остальные.

### 5.2. MPI-коммуникации

| Операция | Функция | Направление |
| --- | --- | --- |
| Рассылка матрицы A | `BcastCCS` | root → все |
| Рассылка разбиения | `MPI_Bcast(col_starts)` | root → все |
| Раздача части B | `ScatterB` | root → каждому |
| Сбор частей C | `GatherC` | каждый → root |
| Рассылка C | `PostProcessingImpl` | root → все |

`ScatterB` и `GatherC` реализованы через `MPI_Send`/`MPI_Recv`,
а не через `MPI_Scatter`/`MPI_Gather`, так как части матрицы B имеют
разный размер после балансировки.

### 5.3. Thread-local данные (OMP-уровень)

Идентичны версии OpenMP: каждый поток объявляет `was`, `accum`, `rows`
внутри `#pragma omp parallel`. Инициализация — один раз при входе в параллельную секцию.

| Буфер | Размер | Фаза | Назначение |
| --- | --- | --- | --- |
| `was` | `a.n` | 1 и 2 | Маркер посещённых строк |
| `accum` | `a.n` | 2 | Вещественный аккумулятор |
| `rows` | динамический | 2 | Список посещённых строк столбца |

## 6. Тесты

Корректность работы алгоритма проверялась функциональными тестами, а производительность — перфоманс-тестами.

### 6.1. Функциональные тесты

Матрица 6×6 с 12 ненулевыми элементами умножается сама на себя. Результат вычислен вручную и переведён в CCS — это
эталонное значение матрицы C.

### 6.2. Перфоманс-тесты

Создаются две ленточные матрицы A и B. Для проверки корректности используется метод умножения справа на произвольный
вектор:

`(A × B) × x = A × (B × x)`

Если результирующие векторы совпадают с учётом погрешности машинной арифметики, матрицы равны.

    python3 run_tests.py --running-type=performance

Очень важно добавить: export OMPI_MCA_hwloc_base_binding_policy=none

мы хотим создать 3 потока на процесс, но без этой команды они все создадутся на одном
логическом ядре и прироста производительности от параллелизма не будет.

## 8. Результаты экспериментов

Параметры запуска:

- Размеры матриц: 20000 × 20000
- Ширина ленты: 100

Строка `1 proc / 1 threads` соответствует результату последовательной версии.

Берется среднее время за 3 прогона. Входные данные при каждом запуске одинаковые.

| proc | threads | time (ms) | speedup | efficiency |
| --- | --- | --- | --- | --- |
| 1 | 1 | 2516 | 1.00 | 100% |
| 1 | 2 | 1584 | 1.59 | 79.42% |
| 2 | 1 | 1652 | 1.52 | 76% |
| 2 | 2 | 933 | 2.70 | 67.5% |
| 2 | 3 | 879 | 2.86 | 47.67% |
| 3 | 2 | 789 | 3.19 | 53.17% |

Можем заметить, что на локальной машине MPI масштабируется лучше, чем OpenMP. В задачах умножения разреженных матрицы
часто происходят ситуации, которые называются random access: данные сильно разбросаны по памяти, в следствии
чего `was[a_row]`, `accum[a_row]`, переходы по `col_ind` — почти всегда cache misses.
Скорость ограничена пропускной способностью памяти. B этих условиях MPI оказывается эффективнее OpenMP, поскольку
устраняет конкуренцию за кэш, также ещё усугубляет ситуацию использование эмулирования с помощью `Docker`.
Наибольшее ускорение даёт конфигурация — 3 proc / 2 threads.

## 9. Выводы

- Реализован двухуровневый параллельный алгоритм умножения разреженных
  матриц в формате CCS: MPI — межпроцессный уровень, OpenMP — межпоточный.
- Корректность подтверждена тестами с учётом погрешности `floating-point` вычислений.
- Балансировка нагрузки между процессами выполняется на основе оценки
  стоимости каждого столбца, что может дать хороший прирост производительности при неравномерной разреженности B.
- Двухфазный подход сохраняется без изменений; позволяет избежать гонки данных при параллельной записи в выходной
  массив.
- На 3 процессах по 2 потока достигнуто ускорение в 3.19 раз при эффективности 53.17%.
- Уменьшение эффективности при росте числа потоков: бутылочным горлышком становится пропускная способность шины данных и
  локальность данных в разреженной алгебре, а также `BcastCCS(A)` — это ограничивает масштабируемость при больших A.

## Приложение

Балансировка нагрузки (процесс 0):

    std::vector<size_t> weights(b.m, 0);
    for (size_t j = 0; j < b.m; ++j)
      weights[j] = EstimateColumnCost(a, b, j);
    col_starts = BuildBalancedStarts(weights, world_size);
    MPI_Bcast(col_starts.data(), world_size + 1, MPI_INT, 0, MPI_COMM_WORLD);

Двухфазное вычисление на каждом процессе:

    #pragma omp parallel num_threads(...) default(none) \
        shared(a, local_b, col_nnz, local_cols)
    {
      std::vector<size_t> was(a.n, std::numeric_limits<size_t>::max());
    #pragma omp for schedule(static)
      for (size_t j = 0; j < local_cols; ++j)
        MatMultPhase1(j, a, local_b, was, col_nnz);
    }

    // префиксная сумма → local_c.col_ind

    #pragma omp parallel num_threads(...) default(none) \
        shared(a, local_b, local_c, local_cols)
    {
      std::vector<size_t> was(a.n, std::numeric_limits<size_t>::max());
      std::vector<double> accum(a.n, 0.0);
      std::vector<size_t> rows;
    #pragma omp for schedule(static)
      for (size_t j = 0; j < local_cols; ++j)
        MatMultPhase2(j, a, local_b, local_c, local_cols + j, was, accum, rows);
    }

    GatherC(c, local_c, col_starts, world_rank, world_size);
