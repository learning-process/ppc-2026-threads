## 1. Введение

Поразрядная сортировка (Radix Sort) является эффективным алгоритмом, который не использует сравнения
и позволяет достичь линейного времени работы. Применение библиотеки Intel Threading Building Blocks (TBB)
позволяет реализовать этот алгоритм с использованием высокоуровневых параллельных конструкций, таких как
задачи и параллельные циклы, что упрощает масштабирование на многоядерных системах.

Целью данной работы является реализация параллельной версии поразрядной сортировки с использованием TBB
и анализ её производительности.

## 2. Постановка задачи

Необходимо реализовать параллельный алгоритм поразрядной сортировки (LSD) для 32-битных целых чисел.

- **Входные данные**: Вектор `std::vector<int32_t>`.
- **Выходные данные**: Отсортированный вектор.
- **Особенности**: Использование `tbb::parallel_for`, `tbb::parallel_reduce` и механизмов рекурсивного
  распараллеливания задач для слияния.

## 3. Описание алгоритма (последовательный)

Последовательная база алгоритма включает:

1. Поиск минимального и максимального элементов для определения диапазона.
2. Сдвиг всех значений в положительную область (нормализация), чтобы корректно обрабатывать разряды.
3. Поразрядная сортировка (LSD) с использованием стабильной сортировки подсчетом для каждого десятичного разряда.
4. Обратный сдвиг к исходным значениям.

## 4. Схема распараллеливания

Параллельная реализация на TBB использует следующие подходы:

1. **Параллельный поиск**: Минимум находится через `tbb::parallel_reduce`, что эффективнее последовательного
   прохода на больших данных.
2. **Параллельная трансформация**: Нормализация массива (сдвиг) выполняется через `tbb::parallel_for`.
3. **Рекурсивное разделение**: Массив делится на блоки. Каждый блок сортируется локально.
4. **Параллельное слияние**: Слияние отсортированных блоков организовано в виде дерева задач. На каждом этапе
   слияния используется `tbb::parallel_invoke` или аналогичные механизмы для одновременной обработки
   независимых пар блоков.

- **Использование конструкций TBB**:
  - `tbb::blocked_range<int32_t>(0, n)`: Применяется в `tbb::parallel_reduce` для автоматического и эффективного разбиения индексов исходного массива на непрерывные поддиапазоны, обрабатываемые отдельными потоками.
  - `grainsize и партиционер`: Для высокоуровневых конструкций `tbb::parallel_for` (нормализация массива) и `tbb::parallel_reduce` (поиск минимума) размер зерна (`grainsize`) не задавался вручную. Вместо этого используется стандартный `tbb::auto_partitioner`, который динамически определяет оптимальный размер порции данных на основе текущей загрузки ядер процессора, минимизируя накладные расходы.
  - `Рекурсивный grain_size`: Для параллельного древовидного слияния блоков вручную определён порог отсечения `grain_size = 4096`. Если размер обрабатываемого подмассива превышает это значение, задача рекурсивно делится пополам и подзадачи запускаются параллельно через `tbb::parallel_invoke`. При достижении размера $\le 4096$ создание новых задач прекращается, и блок сортируется последовательно внутри текущего потока.
  - `Контроль конкуренции`: Явное ограничение количества потоков внутри кода через `tbb::global_control` или `tbb::task_arena` не накладывается. Библиотека TBB автоматически инициализирует пул потоков, соответствующий количеству доступных логических ядер процессора (в данном случае 6 ядер). На этапе тестирования производительности управление количеством задействованных ресурсов контролируется извне с помощью переменной окружения `PPC_NUM_THREADS`.

## 5. Детали реализации

- **Класс**: `VotincevDRadixMergeSortTBB`.
- **Локальная сортировка**: Метод `LocalRadixSort` выполняет классический LSD Radix Sort над выделенным
  диапазоном памяти.
- **Слияние**: Функция `Merge` объединяет два соседних отсортированных участка во временный буфер.
- **Особенности TBB**: Вместо явного управления потоками (как в OMP) используется планировщик задач TBB,
  который автоматически балансирует нагрузку.
- **Параметр grain_size=4096**:
Если количество элементов больше 4096 - используется tbb::parallel_invoke для деления задачи пополам.
Как только размер блока становится <=4096 - задачи перестают создаваться и происходит просто сортировка блока в потоке.
Для параллельного цикла инициализации и parallel_reduce применяется неявный tbb::auto_partitioner, который динамически 
адаптирует размер подзадач под текущую загрузку ядер процессора.
В функции рекурсивного слияния контроль зернистости (grain_size = 4096) осуществляется вручную
(предотвращая избыточное ветвление задач (tbb::parallel_invoke) на малых объемах данных)

## 8. Заключение

Параллельная реализация на базе TBB показала результаты, сопоставимые с OpenMP. Основное преимущество TBB
заключается в более гибком планировании задач, что потенциально дает лучшую масштабируемость на системах
с неравномерной нагрузкой. Ускорение в 1.84 раза на 4 потоках ограничено затратами на выделение памяти
под временные буферы и операциями слияния.

// параллельная сортировка слиянием(сортировка + слияние)
void VotincevDRadixMergeSortTBB::ParallelRadixMergeSort(uint32_t *data, int32_t left, int32_t right, uint32_t *temp) {
  const int32_t grain_size = 4096;  // порог для перехода на последовательную сортировку

  if (right - left <= grain_size) {
    LocalRadixSort(data + left, data + right);
    return;
  }

  int32_t mid = left + ((right - left) / 2);

  // рекурсивно запускаются две задачи в параллель
  tbb::parallel_invoke([&] { ParallelRadixMergeSort(data, left, mid, temp); },
                       [&] { ParallelRadixMergeSort(data, mid, right, temp); });

  // слияние результатов
  Merge(data, temp, left, mid, right);

  // копируем в основной массив
  std::copy(temp + left, temp + right, data + left);
}

Пояснение:
grain_size = 4096:
Если элементов в массиве больше, чем 4096 - tbb::parallel_invoke рекурсивно ветвит выполнение на 2 подзадачи.
Как только размер достаточно малый - вызывается LocalRadixSort.
