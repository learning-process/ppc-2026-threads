## 1. Контекст

Исходная задача заключается в численном вычислении кратных интегралов методом трапеций. Последовательная версия служит
эталоном корректности и базой для измерения ускорения параллельных реализаций.

В многомерном случае интегрирование по прямоугольному параллелепипеду может быть сведено к суммированию значений функции
в узлах равномерной сетки с весовыми коэффициентами, зависящими от положения узла (внутренний, на грани, на ребре, в
вершине).

TBB-реализация использует задачно-ориентированную модель: вычислительная нагрузка разбивается на независимые задачи,
которые динамически распределяются между рабочими потоками. Такой подход позволяет эффективно задействовать все
доступные вычислительные ресурсы и адаптироваться к изменениям загрузки системы без ручного управления потоками.

## 2. Постановка задачи

Задача полностью аналогична описанной в последовательной версии (`seq/report.md`).

**Входные данные**: структура `Integral`, содержащая:

- `lo`, `hi` — векторы нижних и верхних пределов интегрирования;
- `steps` — вектор количества шагов разбиения по каждому измерению;
- `f` — интегрируемая функция, заданная как `std::function<double(const std::vector<double>&)>`;
- `dim` — размерность пространства.

**Выходные данные**: приближённое значение кратного интеграла типа `double`.

Вычисление выполняется по многомерной составной формуле трапеций: область разбивается на прямоугольные ячейки, интеграл
выражается взвешенной суммой значений функции в узлах равномерной сетки с последующим умножением на произведение шагов.

Целью TBB-реализации является получение того же численного результата, что и в последовательном алгоритме, но за меньшее
время за счёт распараллеливания вычислений.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги (детальное описание приведено в `seq/report.md`):

1. Для каждого измерения i вычисляется шаг сетки.

2. Организуется полный перебор всех узлов сетки в лексикографическом порядке. В последовательной версии используется
   собственный итератор в методе `IterateGridPoints`, который инкрементирует d-мерный массив индексов в
   лексикографическом порядке (аналог счётчика с переменным основанием).

3. Для каждого узла:

- определяются его координаты;
- подсчитывается количество граничных координат;
- вычисляется вес;
- значение функции в узле умножается на вес и добавляется к общей сумме.

4. Результат суммы умножается на произведение шагов, что даёт приближённое значение интеграла.

Временная сложность SEQ: O(N). Память SEQ ~ O(dim)\*const.

Именно этот алгоритм подлежит распараллеливанию в TBB-версии. Причём для параллельного выполнения эффективнее
использовать линейную индексацию узлов вместо многомерного итератора.

## 4. Схема распараллеливания

**Выбранный примитив.** Для распараллеливания задачи выбран `tbb::parallel_reduce`. Этот шаблон предназначен для
вычисления ассоциативных операций (в данном случае — сложения) над диапазоном элементов. Он автоматически разбивает
диапазон на поддиапазоны, распределяет их между потоками, выполняет локальную редукцию в каждом потоке и затем
объединяет результаты. Использование `parallel_reduce` аналогично в том числе использованному OpenMP
`#pragma omp parallel for` с редукцией (`reduction`), показавшему хорошую эффективность и масштабируемость на разном
количестве потоков.

**Диапазон работы.** Диапазон узлов в n-мерном пространстве задаётся как непрерывный линейный интервал целочисленных
индексов от 0 до total_nodes - 1 с помощью `tbb::blocked_range<int>`. Каждый узел сетки получает уникальный линейный
идентификатор, что позволяет разбить всю совокупность узлов на непересекающиеся фрагменты. Восстановление многомерных
координат и весов производится внутри обработчика поддиапазона по линейному индексу.

**Grain size (размер зёрна).** В реализации было решено не задавать явный `grainsize`. TBB использует значение по
умолчанию (обычно `grainsize = 1` для диапазонов небольшого размера, либо автоматическое вычисление, чтобы число
поддиапазонов примерно равнялось числу потоков, умноженному на некоторый коэффициент). Для данной задачи, где каждая
итерация имеет одинаковую вычислительную сложность, зерно по умолчанию даёт хороший баланс, поскольку слишком мелкие
зёрна увеличивают накладные расходы на планирование, слишком крупные — могут привести к дисбалансу при неравномерной
загрузке системы. Было экспериментально установлено, что для сетки размером 1 млн узлов автоматический выбор TBB
обеспечивает эффективное распараллеливание (см. раздел 8).

**Partitioner.** В реализации было решено использовать партиционер по умолчанию (`tbb::auto_partitioner`), который
адаптивно выбирает стратегию разбиения в зависимости от числа доступных потоков и динамики выполнения. В отличие от
статического разбиения, `auto_partitioner` может перераспределять оставшиеся поддиапазоны при преждевременном завершении
некоторых потоков, что улучшает балансировку нагрузки. Необходимо заметить, что хотя для редукционной задачи, где все
итерации имеют одинаковую стоимость, `auto_partitioner` эквивалентен `simple_partitioner`, тем не менее текущий вариант
оставляет возможность для адаптации в более сложных сценариях при необходимости расширения.

**Управление конкуренцией.** Фактическое число рабочих потоков TBB контролируется через переменную окружения
`TBB_NUM_THREADS`, которая задаётся перед запуском приложения. Важно отметить, что TBB не создаёт потоков в момент
вызова parallel_reduce, а использует заранее запущенный пул потоков (worker threads). Ограничение `TBB_NUM_THREADS`
задаёт максимальное количество потоков в этом пуле, которое может быть использовано для выполнения задач. В
экспериментах (см. раздел 8) значения 1, 2, 4, 8, 16, 32 были заданы именно через `PPC_NUM_THREADS`.

Все разделяемые данные (`lo`, `hi`, `steps`, `dim`, `f`, а также предвычисленные `h` и `sizes`) доступны потокам только
для чтения — они не модифицируются в процессе вычислений, поэтому гонки при их использовании отсутствуют. Единственная
переменная, которая изменяется в данном случае - это также накапливаемая сумма, для которой для безопасного
параллельного обновления был применён tbb::parallel_reduce: каждый поток получает собственную локальную копию суммы, а
после завершения обработки всех поддиапазонов результаты атомарно объединяются. Таким образом, в конце параллельной
области `parallel_reduce` выполняется неявное объединение локальных сумм с помощью бинарной операции. Никаких
дополнительных барьеров или critical секций при этом не требуется, а все разделяемые данные, кроме аккумулятора,
доступны только для чтения, поэтому гонки отсутствуют.

## 5. Детали реализации

**Файлы реализации:**

- `tbb/include/ops_tbb.hpp` — заголовочный файл, содержащий объявление класса
  `TsibarevaEIntegralCalculateTrapezoidMethodTBB`, наследующего `BaseTask`.
- `tbb/src/ops_tbb.cpp` — реализация методов класса, включая `RunImpl()` с вызовом `tbb::parallel_reduce`.

**Отличия от последовательной версии (SEQ):**

1. Линейная индексация узлов. Вместо многомерного итератора используется плоский индекс `node` от 0 до total_nodes-1.
   Координаты и вес восстанавливаются в момент обработки с помощью операций деления и взятия остатка. Это позволяет
   тривиально разбить диапазон между задачами TBB.
2. Использование `parallel_reduce`. Цикл накопления суммы заменён вызовом `tbb::parallel_reduce`, который внутренне
   распараллеливает итерации и автоматически выполняет редукцию.
3. Выделение обработки диапазона в отдельную функцию: для повышения читаемости код, обрабатывающий один поддиапазон,
   вынесен в метод `ComputeSumNode`, что упрощает тестирование и анализ.

**Локальные аккумуляторы и объединение результатов.** В лямбда-функции, передаваемой в `parallel_reduce`, используется
параметр `local_sum` — локальная сумма для текущего поддиапазона. После обработки всех узлов поддиапазона функция
возвращает обновлённое значение. TBB обеспечивает, что каждый поток получит свою копию `local_sum`, а после завершения
всех поддиапазонов вызовет функцию объединения `std::plus<>()`, которая сложит результаты потоков.

Заметим, что описанные детали и особенности реализации по большей части могут быть отражены в коде:

```cpp
double res_sum = tbb::parallel_reduce(
    tbb::blocked_range<int>(0, total_nodes), 0.0,
    [&](const tbb::blocked_range<int> &r, double local_sum) {
        for (int node = r.begin(); node != r.end(); ++node) {
            int rem = node;
            double node_w = 1.0;
            std::vector<double> point(dim);
            for (int i = dim - 1; i >= 0; --i) {
                int idx = rem % sizes[i];
                rem /= sizes[i];
                if (idx == 0 || idx == steps[i]) node_w *= 0.5;
                point[i] = lo[i] + idx * h[i];
            }
            local_sum += node_w * f(point);
        }
        return local_sum;
    },
    std::plus<>()
);
```

**Работа с памятью.** Для хранения координат точки в каждой итерации создаётся локальный вектор `point` размера `dim`.
Это приводит к многократным выделениям памяти, что может влиять на производительность. Однако в силу малой размерности
(обычно 2 или 3) и эффективного управления памятью в стандартной библиотеке компилятора, накладные расходы не являются
критичными. Заметим, что все параметры интегрирования (`lo`, `hi`, `steps`, `dim`, `f`) передаются в лямбду по ссылке
(`[&]`) и доступны только для чтения. Это безопасно и эффективно, так как TBB не делает копий этих данных.

**Предвычисление параметров перед распараллеливанием**. Как было упомянуто изначально при описании базового алгоритма
(см. `seq/ report.md`) задача хорошо поддаётся не только распараллеливанию вычисления локальных сумм, но и выделению
предварительных вычислительных работ в отдельные блоки. В частности, в реализации все вспомогательные величины, не
зависящие от конкретного узла сетки, вычисляются до вызова parallel_reduce: для каждого измерения заранее определяются
шаг `h[i] = (hi[i] - lo[i]) / steps[i]` и размер сетки `sizes[i] = steps[i] + 1`, а также общее количество узлов
total_nodes. Такой подход исключает повторное вычисление одних и тех же значений внутри каждой итерации, снижает
нагрузку на критические ресурсы и улучшает локальность данных. Благодаря предвычислению параллельная область занимается
только восстановлением координат по линейному индексу, взвешиванием и вызовом подынтегральной функции, что максимально
приближает TBB-версию к эффективности OpenMP при сохранении задачно-ориентированной модели.

## 6. Проверка корректности

Корректность TBB-реализации проверялась путём сравнения её результатов с результатами последовательной версии (SEQ) на
наборе тестов, описанных в `seq/report.md`. Для каждого теста вычислялось приближённое значение интеграла с одинаковыми
входными параметрами (пределы интегрирования, шаги разбиения, функция), после чего сравнивались выходные значения.

**Критерии идентичности:** Абсолютная погрешность между результатами SEQ и TBB не превышает 10^-12. Это подтверждает,
что различия в порядке суммирования не вносят значимой ошибки.

**Набор тестов:**

| Тест       | Размерность | Функция   | Область     | Шаги  | Ожидаемое значение |
| ---------- | ----------- | --------- | ----------- | ----- | ------------------ |
| Simple2D   | 2           | x^2 + y^2 | [0,1]^2     | 100^2 | 0.666667           |
| Constant2D | 2           | 5         | [0,2]x[0,3] | 50^2  | 30.0               |
| Simple3D   | 3           | x + y + z | [0,1]^3     | 50^3  | 1.5                |
| Constant3D | 3           | 3         | [0,2]^3     | 40^3  | 24.0               |

**Дополнительные проверки:** Для каждого теста TBB-версия запускалась с 1, 2, 4, 8, 16, 32 потоками. Во всех случаях
результат совпадал с SEQ-эталоном в пределах заданной погрешности. Это подтверждает отсутствие гонок и корректность
работы редукции независимо от разбиения итераций. Проверены граничные случаи (шаг разбиения = 1 по всем измерениям) и
функции с различным масштабом значений.

Все тесты пройдены успешно. TBB-реализация воспроизводит результат последовательного алгоритма с точностью до ошибок
округления.

## 8. Результаты производительности

Размерность сетки 100 на 100 на 100 (1 млн) узлов. Время было измерено и усреднено для каждого количества рабочих
потоков по 3-5 запускам. Ускорение было посчитано относительно времени на одном потоке (0.1194 с), а эффективность - как
ускорение / количество потоков.

| Потоки (workers) | Время (сек) | Ускорение | Эффективность |
| ---------------- | ----------- | --------- | ------------- |
| 1                | 0.1194      | 1.00x     | 100%          |
| 2                | 0.0457      | 2.61x     | 131%          |
| 4                | 0.0357      | 3.34x     | 84%           |
| 8                | 0.0237      | 5.04x     | 63%           |
| 16               | 0.0183      | 6.53x     | 41%           |
| 32               | 0.0145      | 8.22x     | 26%           |

Достигнуто суперлинейное ускорение на 2 потоках (2.61x), связанное с эффектами кэширования. Причём ускорение продолжает
расти до 32 потоков (8.22x), хотя на 32 потоках эффективность падает до 26%. Это объясняется ограниченным числом
физических ядер (8) и накладными расходами на переключение контекста при использовании гиперпоточности.

По сравнению с OpenMP (ускорение 3.62x на 8 потоках, 4.20x на 32 потоках), TBB демонстрирует значительно лучшую
масштабируемость при большом числе потоков, что связано с динамическим распределением задач.

## 9. Выводы

Для TBB-реализации было успешно произведено распараллеливание вычисления многомерного интеграла методом трапеций с
использованием задачно-ориентированной модели. Выбранный примитив `parallel_reduce` эффективно подошёл для редукционной
задачи и позволил избежать использования явных критических секций, а использование диапазона `blocked_range<int>`
обеспечивает хороший баланс нагрузки и масштабируемость без ручной настройки.

TBB демонстрирует хорошую масштабируемость по сравнению с OpenMP (ускорение 8.22x против 4.20x на 32 потоках), что
объясняется динамическим распределением задач и адаптивным разбиением.

Основным недостатком является зависимость от сторонней библиотеки, что может усложнить переносимость кода на платформы
без поддержки TBB. Однако для многопоточных вычислений на x86-64 архитектурах это не является критическим ограничением.
