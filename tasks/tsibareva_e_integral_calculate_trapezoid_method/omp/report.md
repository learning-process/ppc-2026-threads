## 1. Контекст

Исходная задача заключается в численном вычислении кратных интегралов методом трапеций. Последовательная версия служит
эталоном корректности и базой для измерения ускорения параллельных реализаций.

В многомерном случае интегрирование по прямоугольному параллелепипеду может быть сведено к суммированию значений функции
в узлах равномерной сетки с весовыми коэффициентами, зависящими от положения узла (внутренний, на грани, на ребре, в
вершине).

OpenMP-реализация использует модель общей памяти: множество узлов сетки равномерно распределяется между потоками, каждый
поток независимо вычисляет свою часть суммы с учётом весов. Для этого будет применяться директива
`#pragma omp parallel for` с редукцией (`reduction`) для накопления глобальной суммы. Такой подход позволяет эффективно
задействовать все ядра процессора, минимизируя накладные расходы на синхронизацию и передачу данных.

## 2. Постановка задачи

Задача полностью аналогична описанной в последовательной версии (`seq/report.md`).

**Входные данные**: структура `Integral`, содержащая:

- `lo`, `hi` — векторы нижних и верхних пределов интегрирования;
- `steps` — вектор количества шагов разбиения по каждому измерению;
- `f` — интегрируемая функция, заданная как `std::function<double(const std::vector<double>&)>`;
- `dim` — размерность пространства.

**Выходные данные**: приближённое значение кратного интеграла типа double.

Вычисление выполняется по многомерной составной формуле трапеций: область разбивается на прямоугольные ячейки, интеграл
описывается взвешенной суммой значений функции в узлах равномерной сетки с последующим умножением на произведение шагов.

Целью OpenMP-реализации является получение того же численного результата, что и в последовательном алгоритме, но за
меньшее время за счёт распараллеливания вычислений на общей памяти в многопоточном режиме вычисления.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги (детальное описание приведено в `seq/report.md`):

1. Для каждого измерения i вычисляется шаг сетки.

2. Организуется полный перебор всех узлов сетки в лексикографическом порядке. В последовательной версии используется
   собственный итератор в методе `IterateGridPoints`, который инкрементирует d-мерный массив индексов в
   лексикографическом порядке (аналог счётчика с переменным основанием).

3. Для каждого узла:
    - определяются его координаты;
    - подсчитывается количество граничных координат;
    - вычисляется вес;
    - значение функции в узле умножается на вес и добавляется к общей сумме.

4. Результат суммы умножается на произведение шагов, что даёт приближённое значение интеграла.

Временная сложность SEQ: O(N). Память SEQ ~ O(dim)\*const.

Именно этот алгоритм подлежит распараллеливанию в OpenMP-версии. Причем заметим, что как алгоритм будет подвержен
распараллеливанию вычислений, так и для узлов может быть осуществлено предвычисление координат, значений функции, весов.

## 4. Схема распараллеливания

**Выбор области для распараллеливания**. В последовательной версии основное время выполнения приходится на цикл перебора
всех узлов сетки. Причём заметим, что вычисление вклада каждого узла на деле независимо: для получения суммы достаточно
сложить взвешенные значения функции, причём порядок сложения не влияет на результат. Таким образом, в OpenMP-реализации
был распараллелен цикл от `node` = 0 до `node` < `total_nodes`, для которого каждая итерация является полностью
автономной - вычисляет координаты точки по индексу, определяет вес и добавляет произведение `weight` \* `f(point)` к
локальной сумме данного потока.

**Анализ зависимостей данных**. Зависимости между итерациями в данной реализации отсутствуют. Чтение общих данных (`lo`,
`hi`, `steps`, `h`, `sizes`, `dim`) должно быть произведено только для вычисления координат и весов, то есть эти данные
не изменяются в процессе работы. Следовательно, "гонок" не возникает при условии корректного использования механизма
редукции для накопления глобальной суммы. Поскольку механизм редукции использован предоставленный стандартными
библиотеками OpenMP, можно считать, что зависимости между итерациями не будут накоплены.

**Управление переменными**. В директиве `#pragma omp parallel for` явно указаны атрибуты:

```cpp
#pragma omp parallel for default(none) shared(dim, h, sizes, total_nodes) reduction(+ : global_sum)
```

А именно, предположено следующее разделение переменных:

- default(none) — требует явного перечисления всех переменных, используемых внутри параллельной области.
- shared(dim, h, sizes, total_nodes) — данные переменные только читаются всеми потоками. Они не изменяются в процессе
  работы, поэтому общий доступ является безопасным и эффективным.
- reduction(+ : global_sum) — одной из основных особенностей параллельного алгоритма является, что переменная global_sum
  накапливает сумму взвешенных значений функции. Каждый поток получает свою локальную копию global_sum,
  инициализированную нулём. После завершения цикла все локальные суммы складываются в исходную global_sum (за счёт
  операции "+"), что полностью исключает гонки при параллельном суммировании.

Переменные, объявленные внутри тела цикла (node, remainder, node_weight, point, idx), являются локальными для каждого
потока и не требуют явного указания private.

**Выбор schedule**. Использован schedule по умолчанию: для parallel for - schedule(static), что означает, что
итерационный интервал делится на непрерывные блоки примерно равного размера, и каждый блок статически закрепляется за
строго определённым потоком. static подходит для данной задачи, поскольку вычислительная нагрузка на каждую итерацию
практически одинакова, что позволяет предсказать равномерную загрузку потоков.

Заметим, что отсутствие необходимости в динамическом перераспределении итераций снижает накладные расходы на
синхронизацию.

**Барьеры**. В конце параллельной области parallel for существует неявный барьер: все потоки ожидают завершения самого
медленного, прежде чем продолжить выполнение после цикла, что необходимо, потому что последующие операции (умножение на
результирующую высоту и запись в GetOutput()) должны использовать окончательное значение global_sum, полученное после
полной редукции. Накладные расходы на данный барьер являются незначительными по сравнению с вычислительной работой,
выполняемой потоками. Более того, барьер не ограничивает масштабируемость, так как синхронизация происходит один раз за
весь расчёт. Заметим, что никаких дополнительных critical секций или atomic операций не потребовалось, поскольку
механизм reduction полностью решает проблему суммирования.

## 5. Детали реализации

**Файлы реализации:**

- `omp/include/ops_omp.hpp` — заголовочный файл, содержащий объявление класса
  `TsibarevaEIntegralCalculateTrapezoidMethodOMP`,
- наследующего `BaseTask`.
- `omp/src/ops_omp.cpp` — реализация методов класса, включая `RunImpl()` с OpenMP-директивой.

**Отличия от последовательной версии (SEQ):**

**Способ перебора узлов сетки.**  
Ранее в последовательной версии применялся собственный итератор `IterateGridPoints`, который последовательно перебирал
узлы в лексикографическом порядке, модифицируя d-мерный массив индексов. При переходе к параллельной реализации важно
было обеспечить независимость итераций друг от друга, поскольку они будут выполняться одновременно разными потоками. Для
этого каждому потоку необходимо передать уникальный набор узлов, причём без использования разделяемых итераторов,
которые потребовали бы синхронизации.

Поскольку общее количество узлов известно до начала вычислений, а итерации могут быть пронумерованы от 0 до N-1 (имеют
непрерывную индексацию), то в данном случае, для многопоточного параллельного выполнения оказалось более целесообразным
реализовать линейную индексацию. Идея была основана на том, что разделяемые итераторы с расходами на синхронизацию имели
бы большие затраты эффективности. Гораздо проще и логически удобнее оказалось использовать заранее известное число
узлов, которое может быть равномерно распределено между потоками. В данном случае, вместо хранения и модификации
многомерного счётчика каждый поток самостоятельно восстанавливает координаты узла по его линейному индексу с помощью
операций деления и взятия остатка. Такой подход:

- полностью исключает гонки за счёт отсутствия разделяемого состояния между потоками;
- позволяет применить стандартную директиву `#pragma omp parallel for`, распределяющую непрерывные диапазоны индексов
  между потоками;
- упрощает код и делает его более предсказуемым для компилятора.

В остальном алгоритм OpenMP реализации основан на базовом SEQ-алгоритме решения задачи и имеет общий вид:

```cpp
bool TsibarevaEIntegralCalculateTrapezoidMethodOMP::RunImpl() {
  int dim = GetInput().dim;

  // Предвычисление количества узлов и весовых коэффициентов, обнуление переменных

#pragma omp parallel for default(none) shared(dim, h, sizes, total_nodes) reduction(+ : global_sum)
  for (int node = 0; node < total_nodes; ++node) {
    int remainder = node;
    double node_weight = 1.0;
    std::vector<double> point(dim);

    for (int i = dim - 1; i >= 0; --i) {
      int idx = remainder % sizes[i];
      remainder /= sizes[i];
      if (idx == 0 || idx == GetInput().steps[i]) {
        node_weight *= 0.5;
      }
      point[i] = GetInput().lo[i] + (idx * h[i]);
    }
    global_sum += node_weight * GetInput().f(point);
  }

  // Умножение локальных сумм на "высоты" конкретных измерений из числа шагов разбиения

  GetOutput() = global_sum * res_h;
  return true;
}
```

**Устранение рисков гонок:**

Одной из особенностей OpenMP-алгоритма выполнения задачи является логически корректное использование переменных. Код
имеет ровно одну разделяемую переменную global_sum, которая изменяется в цикле и обрабатывается через механизм редукции,
накапливая глобальную сумму вместо множества локальных с одной точкой сбора. Каждый поток имеет собственную локальную
копию `global_sum`, что полностью исключает состояние гонки при параллельном суммировании.

Остальные разделяемые переменные предполагают только чтение и не модифицируются внутри параллельной области, поэтому
имеют безопасный доступ.

Были устранены некоторые потенциальные риски, которые появляются при модификации участков кода относительно SEQ версии
при распараллеливании алгоритма:

- Был произведён анализ рисков относительно хранимой функции f: если бы она модифицировала разделяемые данные, это
  привело бы к гонкам. Заметим, что подынтегральная функция является линейной и непрерывной, а следовательно имеет
  только входные и выходные данные в соответствии со свойством функционального отображения зависимой переменной.
  Следовательно, данные не меняет и к гонкам данных не приводит.
- Для предоствращения случайного создания неопределённых `shared` переменных использование `default(none)` вынуждает
  явно перечислить все переменные. В конце `parallel for` использован неявный барьер, который гарантирует, что результат
  редукции полностью сформирован до умножения на произведение шагов. Это корректно и не вносит дополнительных накладных
  расходов, кроме однократной синхронизации.

## 6. Проверка корректности

Корректность OpenMP-реализации проверялась путём сравнения её результатов с результатами последовательной версии (SEQ)
на наборе тестов, описанных в `seq/report.md` и генерируемых в `common.h` . Для каждого теста вычислялось приближённое
значение интеграла с одинаковыми входными параметрами (пределы интегрирования, шаги разбиения, функция), после чего
сравнивались выходные значения.

Абсолютная погрешность между результатами SEQ и OMP не превышает 10^(-12). Малая погрешность объясняется тем, что в
обеих реализациях используются идентичные арифметические операции над числами с плавающей запятой, причём единственное
возможное различие связано с порядком суммирования, однако в рамках проведённых тестов различия оказались на уровне
машинной точности.

Для каждого теста OpenMP-версия запускалась с 1, 2, 4, 8, 16, 32 потоками. Во всех случаях результат совпадал с
SEQ-эталоном в пределах заданной погрешности. Это подтверждает отсутствие гонок и корректность работы редукции
независимо от разбиения итераций между потоками.

Все тесты пройдены успешно. OpenMP-реализация воспроизводит результат последовательного алгоритма с точностью до ошибок
округления, что подтверждает корректность распараллеливания.
