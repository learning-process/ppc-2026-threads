## 1. Введение

Задача численного интегрирования методом трапеций в многомерном пространстве является классическим примером вычисления:
вклад каждого узла сетки не зависит от других узлов, что делает её идеальным полигоном для сравнения разных моделей
параллелизма: от последовательного эталона до гибридного MPI + OpenMP. В данном отчёте обобщаются результаты пяти
реализаций:

- **SEQ** представляет из себя последовательную версию эталона корректности.
- **OMP** предполагает распараллеливание через директивы OpenMP (общая память).
- **TBB** реализует задачно-ориентированное распараллеливание.
- **STL** имеет ручное управление потоками с помощью `std::thread`.
- **ALL** вляется гибридом MPI+OpenMP, сочетающим межпроцессное и внутрипроцессное распараллеливание.

Задача хорошо поддаётся распараллеливанию, что было подробнее описано в локальных отчётах. Выявим сильные и слабые
стороны каждой технологии именно для данной вычислительной задачи и дадим рекомендации по выбору оптимального решения в
зависимости от масштаба и доступных ресурсов.

## 2. Единая постановка задачи

**Входные данные:** структура `Integral`, содержащая векторы нижних (`lo`) и верхних (`hi`) пределов интегрирования,
вектор количества шагов разбиения (`steps`), интегрируемую функцию (`f`) и размерность (`dim`).

**Выходные данные:** приближённое значение кратного интеграла в формате double, вычисленное по многомерной составной
формуле трапеций: взвешенная сумма значений функции в узлах равномерной сетки при этом умножается на произведение шагов.

**Ограничения:** `dim ≥ 1`, `lo[i] < hi[i]`, `steps[i] > 0`, также предположено, что функция должна являться
непрерывной.

**Критерий корректности:** абсолютная погрешность между результатом параллельной версии и SEQ не должна превышать
10^-12. Заметим, что все реализации прошли этот критерий на наборе функциональных тестов, что более подробно описано в
локальных отчётах.

## 3. Единая методика эксперимента

**Аппаратное окружение:**

- Процессор: AMD Ryzen 7 5700U (8 физических ядер, 16 логических)
- RAM: 8 ГБ
- ОС: Windows 10 (64-bit)

**Программное окружение:**

- Компилятор: MSVC (Visual Studio 2022), сборка Release
- MPI-реализация: Microsoft MPI (для ALL)
- Библиотека TBB: oneTBB (встроенная в VS 2022)
- Переменные окружения: `PPC_NUM_THREADS` (для OMP/STL/TBB), `PPC_NUM_PROC` (для MPI в ALL)

**Размер задачи:** для всех замеров использовалась трёхмерная сетка `100×100×100` (1 млн узлов). Время последовательной
версии на этом размере было получено и округлено до 0.1194 секунд.

**Методика измерений:** каждый замер был повторён от 2 до 5 раз, после чего было зафиксировано среднее арифметическое.

Формула ускорения была предположена в виде `speedup = T_seq / T_par`, где T_seq - время выполнения последовательной
версии, T_par - параллельной.

Формула подсчета эффективности выполнения в общем виде может выглядеть так: `efficiency = speedup / count_par`, где
count*par - число исполнителей (потоков, процессов), причём для ALL количество работников = `ранги × потоков*на_ранг`.

## 4. Сводка корректности

Все пять реализаций прошли одинаковый набор функциональных тестов. Была получена погрешность относительно аналитических
значений, не превышающая 10^-4, а расхождение между любой параллельной версией и SEQ при этом имеет точность 10^-12.
Дополнительно были проверены граничные случаи (шаг разбиения = 1, пустые диапазоны, разное число потоков). Ограничений
применимости для размерности `dim ≤ 4` не выявлено, что обусловлено особенностью реализации базового алгоритма и
организации структуры входных данных, а именно - хранением функции непосредственно в качестве поля `Integral`.

## 6. Интерпретация различий

Экспериментальным путём, отображённым в таблице (см. п.5) было получено следующее.

SEQ-реализация алгоритма обеспечивает корректный эталонный baseline, показывающий время 0.1194 с. на 1 млн. узлов,
причем при увеличении числа узлов время растёт линейно.

OMP-реализация алгоритма обеспечивает при этом простоту реализации, хорошее ускорение до 4 потоков (2.56x), однако далее
эффективность падает из-за неявных барьеров и накладных расходов на статическое распределение. На 8 потоках получено
ускорение 3.62x (эффективность 45%). Гиперпоточность (16+ потоков) не дала выигрыша из-за ограниченного числа физических
ядер.

TBB-реализация показывает лучшее ускорение среди однопроцессных многопоточных решений. На 8 потоках – 5.04x, на 32
потоках – 8.22x. Основной причиной этого можно считать динамическое распределение задач (`auto_partitioner`), а также
эффективную работа с кэшем. Суперлинейное ускорение на 2 потоках (2.61x) может также свидетельствовать о положительном
влиянии разбиения на локальность данных.

Было получено, что STL-реализация при использовании ручного управления потоками даёт ускорение, близкое к OpenMP, но с
большими накладными расходами на создание/уничтожение потоков, что становится заметно на 32 потоках – ускорение 4.30x
против 8.22x у TBB. Статическое разбиение диапазонов может потенциально приводить к дисбалансу при высоком числе
потоков.

ALL-реализация, как гибрид MPI+OpenMP, показала наилучшее абсолютное время - 0.0209 с., которое было достигнуто на
конфигурации 2 ранга × 4 потока (ускорение 5.71x). При увеличении числа рангов без роста числа потоков на ранг (8 рангов
× 1 поток) наблюдается падение ускорения до 3.02x из-за затрат на межпроцессную коммуникацию. Гибридная схема
превосходит чистый MPI (который может обеспечить примерно 3x на 8 процессах) и чистый OpenMP (5.04x на 8 потоках),
однако уступает TBB на 8 работниках.

## 7. Репродуцируемость

**Сборка проекта**. Проект был собран при использовании системы CMake, основные команды можно видеть ниже.

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

А также для запуска функциональных тестов был использован python-скрипт предопределённого runner-а:

```bash
python scripts/run_tests.py --running-type=functional
```

А тестирование производительности было произведено при использовании того же предопределённого скрипта и при выставлении
вручную переменных окружения. Для SEQ/OMP/STL/TBB справедлива последовательность команд:

```bash
set PPC_NUM_THREADS=<N>
python scripts/run_tests.py --running-type=performance
```

Для ALL справедлива последовательность команд:

```bash
set PPC_NUM_PROC=<ranks>
set OMP_NUM_THREADS=<threads_per_rank>
mpiexec -np %PPC_NUM_PROC% build/bin/run_tests.exe --running-type=performance --task=integral_calc
```

## 8. Заключение

Экспериментально и аналитически в ходе выполнения проекта было определено следующее.

Для одноузловых вычислений с числом узлов до 10 млн и числом потоков ≤ 8 лучшим выбором является реализация **TBB**,
обеспечивающая простоту, высокую масштабируемость, а также автоматическую балансировку.

**OpenMP** является целесообразным решением после TBB, если она недоступна или требуется минимум зависимостей.

**STL** может быть оправдан при необходимости полного контроля над потоками и данными, в том числе вручную.

**Гибридная версия ALL** может быть рекомендована при наличии кластера из нескольких узлов (за счёт MPI) и многоядерных
процессоров на каждом узле (за счёт OpenMP). Для одного узла ALL не даёт преимуществ перед TBB, и имеет дополнительные
сложности реализации и поддержки.

В общем случае, лучшие показатели эффективности были достигнуты для TBB и ALL-реализаций алгоритма, а средний показатель
эффективности среди обобщения экспериментальных замеров достигает 63.8%.
