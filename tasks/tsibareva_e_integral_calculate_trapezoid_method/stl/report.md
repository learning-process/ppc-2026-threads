## 1. Контекст

Исходная задача заключается в численном вычислении кратных интегралов методом трапеций. Последовательная версия служит
эталоном корректности и базой для измерения ускорения параллельных реализаций.

В многомерном случае интегрирование по прямоугольному параллелепипеду может быть сведено к суммированию значений функции
в узлах равномерной сетки с весовыми коэффициентами, зависящими от положения узла (внутренний, на грани, на ребре, в
вершине).

STL-реализация использует ручное управление потоками на основе `std::thread`. В отличие от OpenMP, где распараллеливание
достигается директивами компилятора, здесь требуется явно разделить диапазон узлов между потоками, запустить рабочие
функции, дождаться завершения всех потоков и объединить локальные результаты. Такой подход даёт полный контроль над
разбиением данных и синхронизацией.

## 2. Постановка задачи

Задача полностью аналогична описанной в последовательной версии (seq/report.md).

**Входные данные**: структура Integral, содержащая:

- lo, hi — векторы нижних и верхних пределов интегрирования;
- steps — вектор количества шагов разбиения по каждому измерению;
- f — интегрируемая функция, заданная как std::function<double(const std::vector< double > &)>;
- dim — размерность пространства.

**Выходные данные**: приближённое значение кратного интеграла типа double.

Вычисление выполняется по многомерной составной формуле трапеций: область разбивается на прямоугольные ячейки, интеграл
аппроксимируется взвешенной суммой значений функции в узлах равномерной сетки с последующим умножением на произведение
шагов.

Целью STL-реализации является получение того же численного результата, что и в последовательном алгоритме, но за меньшее
время за счёт ручного распараллеливания вычислений на нескольких потоках с использованием стандартной библиотеки.

## 3. Базовый алгоритм

Последовательный алгоритм (SEQ) выполняет следующие шаги (детальное описание приведено в `seq/report.md`):

1. Для каждого измерения i вычисляется шаг сетки.

2. Организуется полный перебор всех узлов сетки в лексикографическом порядке. В последовательной версии используется
   собственный итератор в методе IterateGridPoints, который инкрементирует d-мерный массив индексов в лексикографическом
   порядке (был реализован аналог счётчика с переменным основанием).

3. Для каждого узла:

- определяются его координаты;
- подсчитывается количество граничных координат;
- вычисляется вес;
- значение функции в узле умножается на вес и добавляется к общей сумме.

4. Результат суммы умножается на произведение шагов, что даёт приближённое значение интеграла.

Временная сложность SEQ: O(N). Память SEQ ~ O(dim)\*const.

Именно этот алгоритм подлежит распараллеливанию в STL-версии. Как и в других параллельных реализациях, используется
линейная индексация узлов и предвычисление шагов и размеров сетки.

## 4. Схема распараллеливания

Была предположена следующая схема распараллеливания с учетом ручного разделения данных.

**Разбиение данных.** Общее количество узлов total_nodes вычисляется до запуска потоков. Диапазон узлов [0, total_nodes)
делится на num_threads непрерывных поддиапазонов примерно равного размера. Для этого вычисляется базовое количество
узлов на поток `nodes_per_thread = total_nodes / num_threads` и остаток `remainder_nodes = total_nodes % num_threads`.
Первые remainder_nodes потоков получают на один узел больше. Каждый поток обрабатывает строго свой поддиапазон
независимо от остальных.

**Что делает каждый поток.** Каждый поток выполняет функцию `MWork`, которая принимает идентификатор потока, границы
поддиапазона, предвычисленные массивы sizes и h, а также ссылку на вектор partial_sums. Внутри worker происходит
линейный обход от start до end, для каждого узла восстанавливаются многомерные координаты и вычисляется вес. Результат
накапливается в локальной переменной local_sum. По окончании обработки local_sum записывается в partial_sums[thread_id].

**Локальные результаты.** Для хранения частичных сумм используется `std::vector partial_sums` размера
`num_threads`. Каждый поток записывает свой результат в отдельный элемент вектора. Поскольку разные потоки пишут в
разные ячейки памяти (индексы различны), гонки данных не возникает, а следовательно мьютексы или атомарные операции не
требуются.

**Создание и завершение потоков.** Потоки создаются в цикле с помощью `std::thread(worker, ...)`. Созданные потоки сразу
начинают выполнение. После запуска всех потоков вызывается `t.join()` для каждого потока, что гарантирует завершение
всех вычислений перед тем, как основной поток продолжит выполнение.

**Объединение результатов.** После `join` все потоки завершены, и вектор `partial_sums` содержит полные суммы для
каждого поддиапазона. Основной поток суммирует элементы `partial_sums` в переменную `global_sum`. Затем global_sum
умножается на произведение шагов, и результат сохраняется в выходную переменную.

**Синхронизация.** В данной реализации используются только неявные барьеры — вызовы `join`. Было принято решение, что
никаких критических секций, мьютексов, условных переменных или атомарных операций не требуется, поскольку каждый поток
работает с независимыми данными (свои индексы, своя локальная сумма, своя ячейка в partial_sums). Это минималистичное и
эффективное решение для задачи без зависимостей между итерациями.

## 5. Детали реализации

**Файлы реализации:**

- stl/include/ops_stl.hpp — заголовочный файл, содержащий объявление класса
  TsibarevaEIntegralCalculateTrapezoidMethodSTL, наследующего BaseTask.
- stl/src/ops_stl.cpp — реализация методов класса, включая RunImpl() с ручным управлением потоками.

**Отличия от последовательной версии (SEQ):**

**Линейная индексация и предвычисление.** Как и в OpenMP- и TBB-версиях, для удобства распараллеливания была применена
линейная индексация узлов. До запуска потоков вычисляются шаги h[i] и размеры сетки sizes[i], а также общее количество
узлов total_nodes. Эти данные передаются во все потоки только для чтения.

**Ручное разбиение диапазона.** В отличие от OpenMP, где распределение итераций осуществляется автоматически директивой
parallel for, в STL-версии были явно вычислены начальный и конечный индексы для каждого потока, примерно следующим
образом:

```cpp
int nodes_per_thread = total_nodes / num_threads;
int remainder_nodes = total_nodes % num_threads;
int start = 0;
for (int tid = 0; tid < num_threads; ++tid) {
  int end = start + nodes_per_thread + (tid < remainder_nodes ? 1 : 0);
  threads[tid] = std::thread(worker, tid, start, end);
  start = end;
}
```

Причем заметим, что вычислительная логика вынесена в метод MWork, что позволяет чётко разделить код, выполняемый в
параллельных потоках, и основной код.

**Локальные суммы и объединение**. Вместо редукции, предоставляемой OpenMP, используется вектор partial_sums. После
завершения потоков результаты вручную суммируются в основном потоке, примерный код реализации можно видеть ниже.

```cpp
bool TsibarevaEIntegralCalculateTrapezoidMethodSTL::RunImpl() {
  int dim = GetInput().dim;
  std::vector<double> h(dim);
  std::vector<int> sizes(dim);

  // Предвычисление и подготовка данных

  const int num_threads = std::max<int>(1, std::min<int>(ppc::util::GetNumThreads(), total_nodes));
  std::vector<std::thread> threads(num_threads);
  std::vector<double> partial_sums(num_threads, 0.0);

  auto worker = [&](int thread_id, int start, int end) {
    MWork(thread_id, start, end, sizes, h, partial_sums, dim);
  };

  int nodes_per_thread = total_nodes / num_threads;
  int remainder_nodes = total_nodes % num_threads;
  int start = 0;
  for (int tid = 0; tid < num_threads; ++tid) {
    int end = start + nodes_per_thread + (tid < remainder_nodes ? 1 : 0);
    threads[tid] = std::thread(worker, tid, start, end);
    start = end;
  }

  for (auto &t : threads) {
    t.join();
  }

  // Глобальное суммирование и домножение на "высоты" трапеций

  GetOutput() = global_sum * res_h;
  return true;
}
```

**Устранение рисков гонок:**

Заметим, что каждый поток пишет только в свою ячейку `partial_sums[thread_id]`. Разные потоки никогда не обращаются к
одной и той же позиции вектора, поэтому гонки отсутствуют. Все разделяемые данные (`lo`, `hi`, `steps`, `h`, `sizes`,
`dim`, `f`) доступны только для чтения, а потоки не используют глобальные переменные. Кроме того, благодаря вызовам
`join` после завершения всех потоков не возникает проблем с обращением к локальным переменным основного потока.

## 6. Проверка корректности

Корректность STL-реализации проверялась путём сравнения её результатов с результатами последовательной версии (SEQ) на
наборе тестов, описанных в `seq/report.md`. Для каждого теста вычислялось приближённое значение интеграла с одинаковыми
входными параметрами, после чего сравнивались выходные значения.

Абсолютная погрешность между результатами SEQ и STL не превышает 10⁻¹². Это объясняется тем, что в обеих реализациях
используются идентичные арифметические операции над числами с плавающей запятой.

Для каждого теста STL-версия запускалась с 1, 2, 4, 8, 16, 32 потоками. Во всех случаях результат совпадал с
SEQ-эталоном в пределах заданной погрешности. Это подтверждает отсутствие гонок и корректность ручного разбиения
диапазонов.

Все тесты пройдены успешно. STL-реализация воспроизводит результат последовательного алгоритма с точностью до ошибок
округления, что подтверждает корректность распараллеливания.

## 9. Выводы

STL-реализация позволяет вручную управлять разбиением данных и потоками, давая полный контроль над параллельным
выполнением. Она корректно воспроизводит результат SEQ и демонстрирует хорошее ускорение до 4.85x на 16 потоках. Однако
при увеличении числа потоков сверх физического числа ядер ускорение перестаёт расти из-за накладных расходов на
управление потоками.

Ручная реализация с `std::thread` требует аккуратного разбиения диапазонов, использования локальных буферов для
частичных сумм и явного `join`. Преимущество такого подхода — полная прозрачность и отсутствие неявных барьеров. В
рамках данной задачи ручное распараллеливание оказалось эффективным, хотя уступает TBB по масштабируемости.
