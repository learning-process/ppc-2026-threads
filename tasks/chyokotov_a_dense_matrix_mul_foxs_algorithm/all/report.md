## 1. Контекст
Гибридная версия требуется для масштабирования на кластеры с многоядерными узлами. 
MPI обеспечивает межпроцессное взаимодействие между узлами, а TBB — эффективное распараллеливание 
внутри каждого узла. Такой подход позволяет решать задачи, не помещающиеся в память одного узла, и 
достигать более высокой масштабируемости.

## 2. Постановка задачи
Умножение квадратных матриц A×B = C с использованием блочного алгоритма Фокса. 
Baseline: последовательная версия seq/ops_seq.cpp.

## 3. Базовый алгоритм
Последовательная версия реализует блочный алгоритм:
1. Разбивает матрицы на блоки размером sqrt(n)
2. В трёх вложенных циклах по блокам перемножает соответствующие блоки
3. Внутренняя функция Matmul() выполняет стандартный тройной вложенный цикл для перемножения блоков
 с накоплением результата в выходной матрице

Асимптотика: O(n^3) по времени, O(n^2) по памяти.

## 4. Межпроцессная схема
### Какие роли у rank-ов.
- Rank 0 — главный процесс: читает входные данные, распределяет блоки, собирает результат
- Остальные rank-ы — рабочие процессы: получают блоки матриц, выполняют умножение, отправляют 
результат
- Используется коммуникатор MPI_COMM_WORLD с разбиением на активные процессы (color = 0) и
неактивные (color = MPI_UNDEFINED), в случае если размер матрицы не является квадратом

### Как распределяются данные.
Процессы организуются в решетку q × q, где q = floor(sqrt(size)). Каждый процесс получает:
- Блок матрицы A размера block_size × block_size
- Блок матрицы B размера block_size × block_size

### Какие MPI-вызовы используются и зачем.

| Вызов          | Назначение                                           |
|----------------|------------------------------------------------------|
| MPI_Comm_rank  | Определение ранга текущего процесса                  |
| MPI_Comm_size  | Определение общего числа процессов                   |
| MPI_Bcast      | Рассылка размера матрицы n и финального результата   |
| MPI_Send       | Отправка блоков A и B от rank 0 рабочим              |
| MPI_Recv       | Приём блоков рабочими                                |
| MPI_Sendrecv   | Одновременная отправка и приём блоков между блоками  |
| MPI_Comm_split | Создание коммуникатора только для активных процессов |
| MPI_Comm_free  | Освобождение коммуникатора                           |

## 5. Внутрипроцессная схема
### Выбранная технология
TBB — используется для параллельного умножения блоков внутри каждого MPI-процесса.

### Распределение работы между потоками
```cpp
// file ./all/src/ops_all.cpp
tbb::parallel_for(tbb::blocked_range2d<int>(0, block_size, 0, block_size),
    [&](const tbb::blocked_range2d<int> &range) {
        // параллельное выполнение
    });
```
- Диапазон: двумерный blocked_range2d по строкам и столбцам блока
- Разбиение: автоматическое (TBB сама делит блок на подблоки)
- Grainsize: используется авто-разбиение

### Объединение локальных результатов
Локальные результаты накапливаются непосредственно в c_block (блок результата для данного 
процесса). Благодаря тому, что parallel_for разбивает итерации так, что разные потоки работают с 
разными строками i и столбцами j выходного блока, гонок данных нет — атомарность не требуется.

## 6. Детали реализации
### Файлы
all/include/ops_all.hpp — объявление класса с методами для паддинга, распределения, алгоритма Фокса
all/src/ops_all.cpp — реализация гибридного алгоритма

### Точки синхронизации
- MPI_Bcast(&n, ...) — барьерная синхронизация: все процессы ждут, пока rank 0 разошлёт размер 
матрицы
- MPI_Bcast(flat_result.data(), ...) — барьерная синхронизация: все процессы получают результат
- MPI_Sendrecv — точка-точка синхронизация: процесс ждёт обмена с соседями

### Потенциальные узкие места
- MPI_Sendrecv в цикле q шагов, каждый требует коммуникации
- Паддинг - дополнительные нулевые элементы увеличивают объём вычислений

## 7. Проверка корректности
Результат ALL-версии сравнивается с ожидаемым результатом в функциональных тестах аналогичных для 
последовательной версии с точностью 0.0001

### Проверка согласованности между rank-ами
- MPI_Bcast гарантирует, что все процессы имеют одинаковое значение n
- MPI_Comm_split изолирует неактивные процессы (если size не является полным квадратом)
- MPI_Sendrecv обеспечивает корректный обмен блоками между соседями
- MPI_Bcast результата финально синхронизирует выходную матрицу

## 10. Выводы
Гибридная схема оправдана, когда задача не помещается в память одного узла или требует 
масштабирования на кластер — межпроцессное взаимодействие через MPI обеспечивает распределение 
данных между узлами, а TBB эффективно утилизирует все ядра внутри каждого узла. Однако гибридная 
схема не оправдана для задач, которые комфортно размещаются в памяти одного узла и имеют регулярную
вычислительную нагрузку — в этом случае чистый TBB или OpenMP дадут лучшее ускорение без 
дополнительных накладных расходов на MPI-коммуникацию и синхронизацию.
