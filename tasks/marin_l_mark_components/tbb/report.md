## 1. Контекст

Версия на oneTBB нужна для той же задачи маркировки компонент связности, но в модели
task-based parallelism. В этой реализации основная работа раскладывается на независимые
полосы изображения и на линейные проходы по плоским буферам, а распределение этих
диапазонов между worker-потоками выполняет runtime oneTBB.

## 2. Постановка задачи

Требуется построить матрицу меток компонент связности для бинарного изображения.
Вход, выход, ограничения и критерий корректности совпадают с последовательной версией
из
[ppc-2026-threads/tasks/marin_l_mark_components/seq/report.md].
В качестве эталона корректности и baseline по времени
используется `SEQ`.

## 3. Базовый алгоритм

Последовательная логика остается двухпроходной:

1. на первом проходе формируются временные метки и связи между ними через `union-find`;
2. после этого объединяются эквивалентные метки;
3. на втором проходе временные метки заменяются на корни множеств и затем
   компактизируются в плотный диапазон номеров компонент.

В TBB-версии эта схема сохраняется, но изображение заранее переводится в плоский буфер,
разбивается на полосы по строкам, а независимые полосы и линейные участки памяти
обрабатываются через `parallel_for`.

## 4. Схема распараллеливания

Выбранный примитив oneTBB:

- основной примитив - `oneapi::tbb::parallel_for`;
- для распределения работы используются `oneapi::tbb::blocked_range<int>`.

Какие области кода распараллеливаются:

- заполнение массива `parent` во `InitializeParents(...)`;
- преобразование входного двумерного изображения в плоский буфер в `PreProcessingImpl`;
- первый проход по полосам в `RunImpl`;
- второй проход применения уже скомпактизированных меток в `ApplyCompactedLabels(...)`;
- преобразование плоского буфера меток обратно в двумерную матрицу в `PostProcessingImpl`.

Как определяется диапазон работы:

- для проходов по строкам используется `blocked_range<int>(0, height_)`;
- для проходов по полосам используется `blocked_range<int>(0, setup.num_stripes)`;
- для инициализации массива родителей используется `blocked_range<int>(0, total_max_labels)`.

Выбор размера полос:

- число полос рассчитывается в `BuildStripeSetup(...)` как
  `min(height, oneapi::tbb::this_task_arena::max_concurrency() * 2)`;
- затем число полос дополнительно ограничивается так, чтобы на одну полосу приходилось
  не меньше `kMinRowsPerStripe = 64` строк;
- такой выбор уменьшает overhead на слишком мелкие задачи и одновременно дает runtime
  достаточно независимых диапазонов для балансировки.

`grainsize` и `partitioner`:

- явный `grainsize` в `blocked_range` не задается;
- явный `partitioner` в коде не используется;
- поэтому детальное разбиение диапазонов выполняется стандартной политикой oneTBB.

Ограничение конкуренции:

- явный `global_control::max_allowed_parallelism` в этой реализации не используется;
- фактическая верхняя оценка конкуренции берется через
  `oneapi::tbb::this_task_arena::max_concurrency()`;
- именно это значение участвует в выборе числа полос.

Минимальный иллюстративный фрагмент:

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, setup.num_stripes), [&](const tbb::blocked_range<int> &range) {
  for (int stripe = range.begin(); stripe != range.end(); ++stripe) {
    LabelStripe(binary_flat_, labels_flat_, parent, setup.stripe_bounds, setup.stripe_base_label, stripe_max_used,
                width_, stripe);
  }
});
```

Этот фрагмент показывает центральную идею реализации: полосы изображения считаются
независимыми задачами первого прохода, а runtime oneTBB сам распределяет их между
worker-потоками.

## 5. Детали реализации

Файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

Основные внутренние структуры:

- `binary_flat_` - плоский буфер бинарного изображения;
- `labels_flat_` - плоский буфер временных и итоговых меток;
- `labels_out_` - двумерная матрица результата;
- `StripeSetup` - описание полос: границы полос, базовые диапазоны временных меток
  и суммарное число допустимых временных меток.

Ключевые этапы реализации:

- `ValidationImpl` проверяет непустой прямоугольный бинарный вход;
- `PreProcessingImpl` проверяет размеры, ограничение на число пикселей, выделяет
  плоские буферы и параллельно копирует входное изображение;
- `RunImpl` строит полосную схему, инициализирует `parent`, параллельно маркирует полосы,
  затем последовательно объединяет граничные связи и строит компактные метки;
- `PostProcessingImpl` параллельно переводит плоский буфер меток обратно в двумерную
  матрицу результата.

Локальные структуры и объединение результатов:

- отдельные полосы получают собственные диапазоны временных меток через
  `stripe_base_label`, поэтому разные задачи не создают одну и ту же новую метку;
- массив `stripe_max_used` хранит верхнюю границу реально использованных меток по полосам;
- после завершения первого прохода функция `MergeStripeBorders(...)` объединяет связи
  между соседними полосами;
- функция `BuildCompactedLabels(...)` последовательно строит итоговое отображение
  `временная метка -> компактная метка`;
- затем `ApplyCompactedLabels(...)` параллельно применяет уже готовое отображение
  к плоскому буферу меток.

Работа с памятью:

- в реализации не используются специальные TBB-аллокаторы вроде
  `cache_aligned_allocator` или `scalable_allocator`;
- корректность достигается не за счет дополнительных аллокаторов, а за счет того, что
  полосы работают по непересекающимся участкам `labels_flat_` и заранее разведенным
  диапазонам временных меток.

## 6. Проверка корректности

Корректность TBB-версии проверяется сравнением с тем же эталоном, что и у `SEQ`.
В `tests/functional/main.cpp` результат сравнивается не по конкретным числам меток,
а по структуре компонент после нормализации через `NormalizeLabels(...)`.

Использованные функциональные тесты:

- `SingleBlob` на `5x5`;
- `TwoBlocks` на `8x10` и `4x6`;
- `Checker` на `7x7`;
- `Diagonal` на `12x12`;
- `Random` на `15x10` с фиксированным seed.

Характерные случаи:

- `SingleBlob` проверяет, что одна сплошная область остается одной компонентой;
- `TwoBlocks` проверяет независимую маркировку двух разделенных областей;
- `Checker` подтверждает, что диагональные касания не образуют общую компоненту,
  так как используется 4-связность;
- `Random` проверяет устойчивость результата на детерминированном разреженном входе.

В performance-тесте дополнительно проверяется общая валидность результата:

- размеры матрицы меток совпадают с размерами входного изображения;
- фоновым пикселям соответствуют нулевые метки;
- единичные пиксели получают положительные метки.

Управление числом worker-потоков:

- в тестовой инфраструктуре используется переменная `PPC_NUM_THREADS`;
- oneTBB в этой задаче опирается на runtime task arena;
- значение `this_task_arena::max_concurrency()` влияет на число полос, которое

## 8. Результаты

Медианы сняты по схеме `1` прогревочный запуск + `5` рабочих запусков.
Для расчета `speedup` и `efficiency` использовался baseline
`SEQ = 0.2053026600`.

| workers | median time, s | speedup | efficiency |
| ------: | -------------: | ------: | ---------: |
| 1       | 0.1516082400   | 1.35    | 1.35       |
| 2       | 0.0912920200   | 2.25    | 1.12       |
| 4       | 0.0585032000   | 3.51    | 0.88       |
| 8       | 0.0463202000   | 4.43    | 0.55       |
| 16      | 0.0443097600   | 4.63    | 0.29       |

По масштабируемости видно, что TBB-версия дает устойчивое ускорение относительно
baseline, а лучший результат на предоставленных замерах достигается при `16` worker-потоках.
При этом рост перестает быть близким к линейному после `8` потоков.

Основные причины ограничения масштабируемости:

- этап `MergeStripeBorders(...)` остается последовательным;
- построение массива `compacted` в `BuildCompactedLabels(...)` тоже выполняется
  последовательно;
- несколько полных проходов по большим плоским буферам делают реализацию
  чувствительной к пропускной способности памяти.

## 9. Выводы

oneTBB хорошо подходит для этой задачи, когда основную работу можно выразить как набор
независимых диапазонов по строкам и полосам изображения. `parallel_for` с
`blocked_range` позволяет достаточно естественно оформить такие проходы и передать
балансировку runtime.

На этой задаче TBB-версия показывает устойчивое ускорение относительно baseline и
достаточно ровную динамику при увеличении числа worker-потоков. Дальнейший рост
ограничивается уже не самим примитивом `parallel_for`, а последовательными этапами
слияния граничных связей и компактизации меток, а также memory-bound характером
нескольких проходов по большим буферам.
