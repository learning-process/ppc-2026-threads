## 1. Введение

Задача состоит в маркировке компонент связности на бинарном изображении по 4-связности.
Она хорошо подходит для сравнения разных моделей параллелизма, потому что сочетает
несколько типов работы одновременно: регулярные линейные проходы по большим массивам,
локально независимую обработку поддиапазонов и этапы последующего объединения
эквивалентных меток.

Для этой задачи в проекте реализованы пять backend-ов:

- `SEQ` как эталон корректности и baseline по времени;
- `OMP` как полосная data-parallel реализация на OpenMP;
- `TBB` как task-based реализация на oneTBB;
- `STL` как ручная реализация на `std::thread`;
- `ALL` как гибридная MPI + oneTBB схема.

## 2. Единая постановка задачи

Входные данные: бинарное изображение `Image = std::vector<std::vector<int>>`, где каждый
пиксель принимает только значения `0` или `1`.

Выходные данные: матрица меток `labels` той же размерности. Фоновые пиксели должны
иметь метку `0`, а каждый пиксель переднего плана должен получить положительную метку
своей 4-связной компоненты.

Ограничения:

- изображение не должно быть пустым;
- все строки должны иметь одинаковую длину;
- вход должен быть бинарным;
- число пикселей не должно превышать `100000000`.

Критерий корректности:

- нулевые пиксели остаются нулевыми;
- все единичные пиксели получают положительные метки;
- пиксели одной 4-связной компоненты имеют одну и ту же метку;
- результат эквивалентен эталонной разметке из functional-тестов после нормализации
  номеров компонент.

## 3. Единая методика эксперимента

Окружение:

- CPU: 12th Gen Intel(R) Core(TM) i5-12450H 2.00 GHz;
- аппаратные потоки: 12;
- RAM: 16.0 GB (доступно 15.7 GB);
- OS: Windows 10 Pro 22H2, build 19045.3271;
- build type: `Release`;
- технологии: `seq`, `omp`, `tbb`, `stl`, `all` включены в `settings.json`;
- замеры в итоговой таблице получены на этой машине и приведены в виде медианы
  `task_run`.

Переменные окружения:

- `PPC_NUM_THREADS` задает число потоков для thread-based backend-ов;
- `PPC_NUM_PROC` задает число MPI-процессов для `ALL`.

Источник входных данных:

- functional-тесты используют детерминированные сценарии;
- performance-тесты строят случайное бинарное изображение размером `4096x4096`
  с вероятностью единичного пикселя `0.3`;
- генерация фиксирована через `std::seed_seq{height, width, 0x4D415249U}`.

Что сравнивается:

- в сводной таблице приведены замеры режима `task_run`;
- для `OMP`, `TBB`, `STL` рабочей единицей считается число потоков;
- для `ALL` отдельно фиксируются `ranks`, `threads_per_rank` и
  `total_workers = ranks * threads_per_rank`.

Как считаются метрики:

- `speedup = T_seq / T_backend`;
- для `SEQ` принимается `workers = 1`;
- для `OMP`, `TBB`, `STL`:
  `efficiency = speedup / threads`;
- для `ALL`:
  `efficiency = speedup / total_workers`.

Агрегация результатов:

- baseline `SEQ` и все параллельные backend-ы представлены медианой `task_run`;
- в локальных отчетах и в этой сводке используется один и тот же baseline:
  `SEQ = 0.2053026600`.

## 4. Сводка корректности

Все backend-ы сравнивались с одной и той же эталонной логикой из
`tests/functional/main.cpp`.

Использованные функциональные сценарии:

- `SingleBlob` на `5x5`;
- `TwoBlocks` на `8x10` и `4x6`;
- `Checker` на `7x7`;
- `Diagonal` на `12x12`;
- `Random` на `15x10` с фиксированным seed.

Эталон строится функцией `ComputeReferenceLabels(...)`, а затем сравнение выполняется
по структуре компонент после нормализации меток через `NormalizeLabels(...)`.
Это важно, потому что конкретные номера компонент могут отличаться, но сама
компонентная структура должна совпадать.

Ограничения применимости backend-ов:

- `SEQ` используется как базовый эталон;
- `OMP`, `TBB`, `STL` зависят от выбранного числа потоков;
- `ALL` дополнительно зависит от конфигурации `ranks × threads_per_rank` и включает
  издержки MPI-коммуникаций.

## 6. Интерпретация различий

`SEQ`.
Последовательная версия задает честный baseline для всех сравнений. Она не несет
издержек runtime-планировщиков и межпроцессных обменов, поэтому именно ее время
используется в знаменателе для `speedup`.

`OMP`.
OpenMP показывает лучший итоговый результат на предоставленных замерах. Это связано
с тем, что основная работа хорошо ложится на полосную data-parallel схему с
`parallel for` и `schedule(static)`, а overhead OpenMP при таком характере проходов
оказывается низким. При этом после `8` потоков рост замедляется, потому что часть
алгоритма остается последовательной, а несколько проходов по большим массивам
упираются в память.

`TBB`.
TBB показывает более плавное и предсказуемое ускорение, чем чисто последовательный
baseline, но уступает OMP и STL по абсолютному времени. Это выглядит закономерно для
task-based runtime: разбиение на диапазоны и автоматическая балансировка удобны,
но выигрыш ограничивается теми же последовательными стадиями объединения меток
и memory-bound характером проходов.

`STL`.
Ручная реализация на `std::thread` дает очень сильный выигрыш уже на малом числе потоков
и по абсолютному времени оказывается близка к TBB, а на части конфигураций даже лучше.
Здесь помогает полный контроль над блочным разбиением и порядком `create -> work -> join`.
При этом после `8` потоков наблюдается плато: рост числа потоков уже почти не меняет
медиану, потому что начинают доминировать последовательные этапы и накладные расходы
на ручное управление worker-потоками.

`ALL`.
Гибридная версия показывает самую сложную картину. На умеренных конфигурациях
`2 × 2` и `4 × 4` она все же дает ускорение относительно baseline, но масштабируется
хуже чистых thread-based backend-ов. Это естественно: помимо локальной маркировки,
она платит за `MPI_Scatterv`, `MPI_Allgather`, `MPI_Gatherv`, `MPI_Bcast`, а также
за глобальное объединение границ и компактизацию меток на `rank 0`.

## 7. Репродуцируемость

Команды сборки:

```powershell
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

Команды запуска functional-тестов:

```powershell
.\build\bin\ppc_func_tests.exe --gtest_filter="*marin_l_mark_components_seq_enabled*"
$env:PPC_NUM_THREADS='8'
.\build\bin\ppc_func_tests.exe --gtest_filter="*marin_l_mark_components_omp_enabled*"
.\build\bin\ppc_func_tests.exe --gtest_filter="*marin_l_mark_components_tbb_enabled*"
.\build\bin\ppc_func_tests.exe --gtest_filter="*marin_l_mark_components_stl_enabled*"
mpiexec -n 4 .\build\bin\ppc_func_tests.exe --gtest_filter="*marin_l_mark_components_all_enabled*"
```

Команды получения основных performance-замеров:

```powershell
$env:PPC_NUM_THREADS='8'
.\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_marin_l_mark_components_omp_enabled*"
.\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_marin_l_mark_components_tbb_enabled*"
.\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_marin_l_mark_components_stl_enabled*"
mpiexec -n 4 .\build\bin\ppc_perf_tests.exe --gtest_filter="*task_run_marin_l_mark_components_all_enabled*"
```

## 8. Заключение

Для этой задачи лучшей на предоставленных замерах можно считать `OMP`-версию:
она показывает минимальное медианное время и наилучший `speedup` относительно
последовательного baseline.

`TBB` и `STL` тоже дают устойчивое ускорение и хорошо подходят для данной задачи,
но уступают лучшей OpenMP-конфигурации по абсолютному времени. `ALL` методически
интересна как гибридная схема, но на рассматриваемом размере входа и данных
конфигурациях сильнее чувствительна к стоимости коммуникаций и глобальной синхронизации.

Ограничения сравнения:

- итоговые числа относятся к одной конкретной машине;
- в сводке приведен режим `task_run`, а не полный сравнительный набор всех
  возможных performance-режимов;
- для `ALL` эффективность зависит от выбранной нормировки и здесь считается
  по `total_workers = ranks * threads_per_rank`.

Что можно было бы улучшить дальше:

- уменьшить последовательную часть объединения полос и глобальной компактизации;
- исследовать другие конфигурации `ranks × threads` для `ALL`;
- дополнительно построить графики ускорения и эффективности по тем же данным.

## 10. Приложение

В локальных отчетах приведены короткие фрагменты реального кода для каждого backend-а:

- `seq/report.md` - двухпроходная последовательная схема;
- `omp/report.md` - полосный `parallel for` на OpenMP;
- `tbb/report.md` - `parallel_for` по `blocked_range`;
- `stl/report.md` - ручной цикл `create -> work -> join`;
- `all/report.md` - MPI-распределение строк и локальная TBB-обработка.

Этого достаточно, чтобы связать сводный отчет с конкретными реализациями без
дублирования больших кодовых листингов.
