## 1. Контекст

Версия на `std::thread` нужна для ручной thread-based реализации той же задачи маркировки
компонент связности. В отличие от runtime-подходов OpenMP и oneTBB, здесь распределение
диапазонов, запуск worker-потоков, ожидание завершения и обработка ошибок организуются
явно в коде задачи.

## 2. Постановка задачи

Требуется построить матрицу меток компонент связности для бинарного изображения.
Вход, выход, ограничения и критерий корректности совпадают с последовательной версией
из
[ppc-2026-threads/tasks/marin_l_mark_components/seq/report.md].
Эталоном корректности и baseline по времени
считается `SEQ`.

## 3. Базовый алгоритм

Последовательная логика остается двухпроходной:

1. на первом проходе формируются временные метки и отношения эквивалентности
   через `union-find`;
2. после этого объединяются граничные связи между полосами;
3. на втором проходе временные метки заменяются на корни множеств и затем
   компактизируются в плотный диапазон.

STL-версия сохраняет эту схему, но переводит изображение в плоский буфер и делит работу
на независимые блоки строк и полосы, которые вручную распределяются между потоками.

## 4. Схема распараллеливания

Как данные делятся на диапазоны:

- вспомогательная функция `ParallelForBlocks(total_items, worker_count, fn)` разбивает
  диапазон `[0, total_items)` на почти равные непрерывные поддиапазоны;
- границы вычисляются формулой
  `begin = (worker * total_items) / total_workers`,
  `end = ((worker + 1) * total_items) / total_workers`;
- число рабочих потоков ограничивается как
  `min(max(1, worker_count), total_items)`.

Что делает каждый поток:

- при инициализации `parent_` поток заполняет свой непрерывный блок индексов;
- при копировании входного изображения поток копирует свой диапазон строк в плоский
  буфер `binary_`;
- на первом проходе поток обрабатывает выделенный диапазон полос и маркирует только
  соответствующие строки изображения;
- на втором проходе поток переписывает свой диапазон элементов `labels_flat_`,
  заменяя временные метки на компактные;
- при формировании результата поток переносит свой диапазон строк в двумерную матрицу
  `labels_`.

Где хранятся локальные результаты:

- отдельные локальные суммы или локальные буферы результата не создаются;
- изоляция достигается тем, что каждый поток работает только с непересекающимся
  диапазоном строк, полос или индексов;
- для первого прохода разные полосы получают заранее разведенные диапазоны временных
  меток через `stripe_base_label_`.

Как и когда выполняется `join`:

- запуск потоков инкапсулирован в `RunParallelWorkers(...)`;
- сначала создаются worker-потоки с индексами `1..N-1`;
- затем поток `0` выполняет работу прямо в вызывающем потоке;
- после этого вызывается цикл `join()` по всем созданным `std::thread`;
- таким образом, `join` находится после стадии `create -> work`, а не внутри цикла
  создания потоков, поэтому реальный параллелизм сохраняется.

Какая синхронизация используется:

- `atomic`, `condition_variable` и дополнительные `mutex` для данных алгоритма
  не используются;
- единственный `std::mutex` в `RunParallelWorkers(...)` нужен только для безопасной
  записи `std::exception_ptr`, если один из потоков завершился с исключением;
- защита записи в основные структуры алгоритма не нужна, потому что потоки работают
  по непересекающимся диапазонам, а межполосовые связи объединяются позже
  последовательно в `MergeStripeBorders()`.

Минимальный иллюстративный фрагмент:

```cpp
for (int worker = 1; worker < total_workers; ++worker) {
  threads.emplace_back([&, worker]() {
    fn(worker, total_workers);
  });
}

fn(0, total_workers);

for (auto &thread : threads) {
  if (thread.joinable()) {
    thread.join();
  }
}
```

Этот фрагмент показывает ключевую инженерную часть реализации: потоки сначала
создаются, затем выполняют работу параллельно, и только после этого основной поток
дожидается их завершения через `join()`.

## 5. Детали реализации

Файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Основные внутренние структуры:

- `binary_` - плоский буфер бинарного изображения;
- `labels_flat_` - плоский буфер временных и итоговых меток;
- `labels_` - двумерная матрица результата;
- `parent_` - массив `union-find`;
- `stripe_bounds_` - границы полос по строкам;
- `stripe_base_label_` - базовые диапазоны временных меток по полосам;
- `stripe_used_label_end_` - фактически использованные границы меток по полосам;
- `root_to_compact_` - отображение временной метки в компактную итоговую метку.

Карта `поток -> диапазон`:

- при `ParallelForBlocks(height_, requested_threads, ...)` поток получает свой диапазон строк;
- при `ParallelForBlocks(stripe_count_, stripe_count_, ...)` поток получает свой диапазон полос;
- при `ParallelForBlocks(labels_flat_.size(), stripe_count_, ...)` поток получает свой
  непрерывный диапазон индексов плоского буфера.

Схема объединения результатов:

- на первом проходе каждая полоса маркируется независимо и заполняет свой диапазон
  `labels_flat_`;
- затем `MergeStripeBorders()` последовательно объединяет компоненты, пересекающие
  границы соседних полос;
- `BuildCompactedLabels(...)` строит общее отображение временных меток в компактные;
- после этого параллельный второй проход применяет уже готовое отображение ко всему
  `labels_flat_`;
- в `ConvertLabelsToOutput()` плоский буфер переносится в двумерную матрицу результата.

Жизненный цикл задачи:

- `ValidationImpl` проверяет непустой прямоугольный бинарный вход;
- `PreProcessingImpl` вычисляет размеры, число полос, диапазоны меток, выделяет
  буферы и параллельно копирует вход;
- `RunImpl` вызывает `FirstPassSTL()`, `MergeStripeBorders()` и `SecondPassSTL()`;
- `PostProcessingImpl` формирует итоговую двумерную матрицу меток и записывает ее
  в `OutType`.

## 6. Проверка корректности

Корректность STL-версии проверяется тем же способом, что и у `SEQ`: в функциональных
тестах результат сравнивается с эталоном `ComputeReferenceLabels(...)` по структуре
компонент после нормализации меток.

Использованные функциональные тесты:

- `SingleBlob` на `5x5`;
- `TwoBlocks` на `8x10` и `4x6`;
- `Checker` на `7x7`;
- `Diagonal` на `12x12`;
- `Random` на `15x10` с фиксированным seed.

Крайние и характерные случаи:

- `SingleBlob` проверяет одну сплошную компоненту;
- `TwoBlocks` проверяет две разделенные компоненты;
- `Checker` проверяет, что диагональные касания не объединяются при 4-связности;
- `Random` проверяет устойчивость на детерминированном разреженном изображении.

Проверка на отсутствие гонок на уровне логики реализации:

- потоки не записывают в один и тот же диапазон `labels_flat_`;
- потоки не создают общие новые метки, потому что полосы имеют разные диапазоны
  временных меток;
- межполосовые зависимости обрабатываются отдельно после завершения параллельного
  первого прохода;
- запись в `thread_error` защищена `std::mutex`, поэтому исключения из потоков
  не приводят к гонке при фиксации ошибки.

## 8. Результаты

Медианы сняты по схеме `1` прогревочный запуск + `5` рабочих запусков.
Для расчета `speedup` и `efficiency` использовался baseline
`SEQ = 0.2053026600`.

| threads | median time, s | speedup | efficiency |
| ------: | -------------: | ------: | ---------: |
| 1       | 0.0925431600   | 2.22    | 2.22       |
| 2       | 0.0579059600   | 3.55    | 1.77       |
| 4       | 0.0462193200   | 4.44    | 1.11       |
| 8       | 0.0442453800   | 4.64    | 0.58       |
| 16      | 0.0442680200   | 4.64    | 0.29       |

По динамике видно, что ручная thread-based версия хорошо ускоряется до диапазона
`8–16` потоков, после чего дальнейший рост числа потоков уже практически не дает
дополнительного выигрыша.

Основные причины этого поведения:

- каждый параллельный этап требует явного запуска и последующего `join` потоков;
- этап `MergeStripeBorders()` остается последовательным;
- построение `root_to_compact_` в `BuildCompactedLabels(...)` тоже остается
  последовательным;
- несколько проходов по большим плоским буферам делают реализацию чувствительной
  к пропускной способности памяти.

## 9. Выводы

Реализация на `std::thread` показала, что задача хорошо раскладывается на ручное
блочное распараллеливание по строкам и полосам изображения. Такой подход дает полный
контроль над стратегией разбиения, порядком `create -> work -> join` и обработкой ошибок
из worker-потоков.

На этой задаче STL-версия дает устойчивое ускорение относительно baseline и достигает
лучших значений в области `8–16` потоков. Дальнейшее масштабирование ограничивается
как последовательными этапами объединения полос, так и накладными расходами ручного
управления потоками и memory-bound характером нескольких полных проходов по изображению.
