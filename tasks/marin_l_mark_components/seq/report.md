## 1. Контекст

Задача состоит в маркировке компонент связности на бинарном изображении.
Последовательная версия нужна как эталон корректности для всех параллельных
backend-ов и как baseline по времени, относительно которого затем считаются
ускорение и эффективность.

## 2. Постановка задачи

Входные данные: `InType`, содержащий бинарное изображение
`Image = std::vector<std::vector<int>>`, где каждый пиксель принимает только
значения `0` или `1`.

Выходные данные: `OutType`, содержащий матрицу меток `labels` той же размерности.
Фоновые пиксели должны иметь метку `0`, а каждый пиксель переднего плана должен
получить положительную метку своей компоненты связности.

Ограничения:

- изображение не должно быть пустым;
- все строки должны иметь одинаковую длину;
- изображение должно быть бинарным;
- число пикселей не должно превышать `100000000`.

Крайние случаи:

- изображение размера `1x1`;
- полностью заполненная область;
- несколько раздельных компонент;
- шахматный и диагональный паттерны;
- детерминированный случайный бинарный вход.

Критерий корректности:

- фон остается нулевым;
- все единичные пиксели получают положительные метки;
- пиксели одной 4-связной компоненты получают одну и ту же метку;
- результат эквивалентен эталонной маркировке, полученной обходом в глубину.

## 3. Базовый алгоритм

Используется двухпроходный алгоритм маркировки компонент связности с
дизъюнктными множествами.

Шаги алгоритма:

1. Изображение копируется во внутренний буфер `binary_`, а матрица меток
   `labels_` инициализируется нулями.
2. На первом проходе изображение просматривается слева направо и сверху вниз.
   Для каждого единичного пикселя рассматриваются только левый и верхний соседи.
3. Если оба соседа равны нулю, создается новая метка.
4. Если ненулевой только один сосед, пиксель получает его метку.
5. Если оба соседа имеют разные ненулевые метки, пиксель получает минимальную
   из них, а сами метки объединяются в структуре `union-find`.
6. После завершения первого прохода все временные метки сжимаются до корней.
7. На втором проходе каждая метка заменяется на корень своего множества, после
   чего выполняется дополнительная компактизация: корни перенумеровываются в
   плотный диапазон `1, 2, 3, ...`.

Асимптотика по времени: `O(H * W * α(H * W))`, где `H * W` - число пикселей, а
`α` - обратная функция Аккермана для `union-find`. На практике это близко к
линейной сложности `O(H * W)`.

Асимптотика по памяти: `O(H * W)`, так как хранятся бинарное изображение,
матрица меток и массив родителей для временных меток.

Инварианты и критерий корректности:

- после первого прохода каждый единичный пиксель имеет временную метку;
- массив `parent` хранит эквивалентность временных меток;
- после второго прохода эквивалентные метки сведены к одному компактному номеру;
- нулевые пиксели никогда не получают положительную метку.

## 4. Детали реализации

Файлы:

- `seq/include/ops_seq.hpp`
- `seq/src/ops_seq.cpp`

Назначение стадий жизненного цикла задачи:

- `ValidationImpl` проверяет, что входное изображение не пусто, все строки
  имеют одинаковую длину и содержат только `0` и `1`;
- `PreProcessingImpl` копирует входное изображение во внутренний буфер,
  проверяет ограничение на число пикселей и выделяет матрицу меток;
- `RunImpl` последовательно вызывает `FirstPass()` и `SecondPass()`;
- `PostProcessingImpl` упаковывает итоговую матрицу меток в `OutType`.

Ключевые внутренние функции:

- `ProcessPixel(...)` реализует логику первого прохода для одного пикселя;
- `UnionLabels(...)` и `FindRoot(...)` поддерживают структуру `union-find`;
- `FirstPass()` формирует временные метки и эквивалентности;
- `SecondPass()` заменяет временные метки на компактные итоговые номера.

Обработанные исключительные и граничные случаи:

- пустой вход отклоняется до запуска алгоритма;
- небинарные значения отклоняются в `ValidationImpl`;
- изображения с разной длиной строк отклоняются в `ValidationImpl`;
- слишком большой размер изображения отклоняется в `PreProcessingImpl`;
- изображение без единичных пикселей корректно возвращает матрицу нулей.

Минимальный иллюстративный фрагмент:

```cpp
bool MarinLMarkComponentsSEQ::RunImpl() {
  FirstPass();
  SecondPass();
  return true;
}

void MarinLMarkComponentsSEQ::FirstPass() {
  for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
      ProcessPixel(binary_, labels_, parent, row, col, next_label);
    }
  }
}
```

Этот фрагмент показывает основную идею реализации: сначала изображение проходит
первичную маркировку с накоплением эквивалентностей, а затем отдельным проходом
получает окончательные компактные метки компонент.

## 5. Проверка корректности

Корректность проверяется сравнением результата с эталонной разметкой,
полученной функцией `ComputeReferenceLabels(...)` в
`tests/functional/main.cpp`. Эталон строится обходом в глубину по 4-связности.
Сравнение выполняется не по конкретным номерам меток, а по структуре компонент
после нормализации меток функцией `NormalizeLabels(...)`.

Использованные функциональные тесты:

- `SingleBlob` на размере `5x5`;
- `TwoBlocks` на размерах `8x10` и `4x6`;
- `Checker` на размере `7x7`;
- `Diagonal` на размере `12x12`;
- `Random` на размере `15x10` с фиксированным seed.

Характерные примеры:

- для `SingleBlob` все единичные пиксели должны получить одну общую метку;
- для `TwoBlocks` две раздельные прямоугольные области должны получить две
  разные метки;
- для `Checker` каждая изолированная единица образует отдельную компоненту,
  поскольку используется 4-связность.

Дополнительно в performance-тесте проверяется общая валидность результата:

- размеры матрицы меток совпадают с размером входного изображения;
- нулевые пиксели не получают положительных меток;
- единичные пиксели не остаются без компоненты.

Самый дорогой фрагмент baseline-версии - первый проход по изображению, потому
что именно там для каждого единичного пикселя выполняется анализ соседей и, при
необходимости, объединение временных меток в `union-find`. Второй проход тоже
линеен, но обычно дешевле, так как только заменяет уже найденные метки на
корневые и компактные значения.

параллельных runtime и межпоточной синхронизации. Она детерминирована,
полностью покрыта функциональными тестами и дает baseline-время `p = 1`, которое
далее используется как знаменатель для расчета `speedup` и `efficiency` во всех
параллельных отчетах.
