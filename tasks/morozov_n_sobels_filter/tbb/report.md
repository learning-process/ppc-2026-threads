## 1. Контекст

В задачах обработки изображений часто требуется выделение границ объектов. Для решения данной задачи используется
оператор Собеля, который позволяет вычислить градиент яркости изображения и определить области резкого изменения
интенсивности пикселей.

В данном отчете рассматривается распараллеливание алгоритма фильтра Собеля при помощи технологии oneTBB. В отличие от
OpenMP, где распараллеливание выражается через директивы компилятора, oneTBB использует задачно-ориентированную модель.
Работа описывается через диапазоны данных, а runtime oneTBB самостоятельно создает и распределяет задачи между
worker-потоками.

Последовательная версия алгоритма используется как эталон корректности и базовая точка отсчета при анализе
производительности TBB-реализации.

## 2. Постановка задачи

Требуется выделить ребра на изображении с использованием оператора Собеля.

Входные / выходные данные: изображение в оттенках серого с высотой `H` и шириной `W`.

В программе данные представлены структурой `Image`:

```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

Каждый пиксель хранит значение яркости в диапазоне `[0; 255]`.

Полное описание математической основы алгоритма приведено в отчете последовательной версии `seq/report.md`.

---

## 3. Базовый алгоритм

Последовательный алгоритм выполняет проход по всем внутренним пикселям изображения и вычисляет для каждого новое
значение градиента яркости.

Крайние пиксели изображения не обрабатываются, так как для вычисления оператора Собеля используется окрестность `3x3`.

Для вычисления используются два ядра Собеля:

- ядро по оси `X`;
- ядро по оси `Y`.

Для каждого пикселя вычисляются:

- градиент по оси `X`;
- градиент по оси `Y`;
- итоговое значение градиента:

`sqrt(gradX^2 + gradY^2)`

Полученное значение ограничивается диапазоном `[0; 255]` и записывается в результирующее изображение.

Каждый пиксель вычисляется независимо от остальных, так как использует только данные исходного изображения. Это свойство
является основой для распараллеливания алгоритма.

Асимптотика последовательного алгоритма по времени: `O(H * W)`.

Асимптотика по памяти: `O(H * W)`, так как создается результирующее изображение того же размера, что и исходное.

## 4. Схема распараллеливания

Основой распараллеливания является следующий инвариант:

- результирующее значение пикселя `i, j` зависит только от окрестности `3x3` исходного изображения.

Из этого следует, что вычисления различных пикселей независимы между собой:

1. Исходное изображение не изменяется во время работы алгоритма;
2. Каждый элемент результирующего изображения записывается только один раз;
3. Одновременная запись нескольких задач в один и тот же элемент массива невозможна.

### Выбранный примитив oneTBB

Для распараллеливания используется примитив:

`oneapi::tbb::parallel_for`

Данный примитив выбран потому, что алгоритм представляет собой независимую обработку элементов двумерной области
изображения. Для каждого внутреннего пикселя выполняется одинаковая операция вычисления градиента.

Использование `parallel_reduce` не требуется, так как в алгоритме отсутствует глобальная свертка результата.
Использование `task_group` также избыточно, поскольку работа естественно описывается регулярным двумерным диапазоном.

### Диапазон работы

В TBB-реализации используется двумерный диапазон:

`oneapi::tbb::blocked_range2d<size_t>`

Он описывает область внутренних пикселей изображения:

- строки: от `1` до `img.height - 1`;
- столбцы: от `1` до `img.width - 1`.

Таким образом, из обработки исключается граница изображения толщиной в 1 пиксель, как и в последовательной реализации.

Логически весь прямоугольник внутренних пикселей разбивается oneTBB на поддиапазоны. Каждый поддиапазон передается в
тело `parallel_for`, где выполняются обычные вложенные циклы по строкам и столбцам.

### Grain size

В реализации `blocked_range2d` создается без явного указания grain size:

```cpp
oneapi::tbb::blocked_range2d<size_t>(
    1, img.height - 1,
    1, img.width - 1)
```

Это означает, что используются значения grain size по умолчанию для строк и столбцов.

В данной задаче это допустимо, так как:

- вычисление одного пикселя имеет малую, но одинаковую стоимость;
- все пиксели обрабатываются одинаковым образом;
- отсутствуют области изображения, требующие принципиально другого объема вычислений;
- oneTBB может самостоятельно делить диапазон на задачи.

При этом слишком маленький grain size потенциально может увеличивать накладные расходы на создание и планирование задач.
Для больших изображений эти расходы компенсируются большим количеством вычислений. Для маленьких изображений overhead
TBB может быть сравним со временем самой фильтрации, поэтому ускорение может быть небольшим или отсутствовать.

### Partitioner

Явный partitioner в реализации не передается.

Это означает, что используется поведение `parallel_for` по умолчанию. В этом случае разбиение диапазона и балансировка
задач выполняются средствами runtime oneTBB.

Для данной задачи это подходит, так как нагрузка равномерная: каждый внутренний пиксель требует одинакового количества
операций свертки. Поэтому нет необходимости вручную задавать специальную стратегию балансировки.

### Ограничение конкуренции

Фактическая конкуренция ограничивается с помощью:

`oneapi::tbb::global_control`

В реализации используется параметр:

`oneapi::tbb::global_control::max_allowed_parallelism`

и значение:

`ppc::util::GetNumThreads()`

Важно, что `max_allowed_parallelism` не создает строго заданное количество потоков для конкретного цикла, а ограничивает
максимальный уровень параллелизма, который может использовать runtime oneTBB. То есть реализация задает верхнюю границу
числа worker-потоков, доступных для выполнения задач.

## 5. Детали реализации

Основные изменения относительно последовательной версии находятся в файле:

```text
omp/src/ops_tbb.cpp
```

В TBB-версии был изменен только метод `Filter()`.

```cpp
void MorozovNSobelsFilterTBB::Filter(const Image &img) {
  oneapi::tbb::global_control control(oneapi::tbb::global_control::max_allowed_parallelism, ppc::util::GetNumThreads());

  oneapi::tbb::parallel_for(oneapi::tbb::blocked_range2d<size_t>(1, img.height - 1, 1, img.width - 1),
                            [&](const oneapi::tbb::blocked_range2d<size_t> &range) {
    for (size_t id_y = range.rows().begin(); id_y != range.rows().end(); ++id_y) {
      for (size_t id_x = range.cols().begin(); id_x != range.cols().end(); ++id_x) {
        const size_t pixel_id = (id_y * img.width) + id_x;
        result_image_.pixels[pixel_id] = CalculateNewPixelColor(img, id_x, id_y);
      }
    }
  });
}
```

### Метод Filter

Главное отличие TBB-версии от последовательной реализации находится в методе `Filter`.

В последовательной версии используются два вложенных цикла. В TBB-версии вся двумерная область внутренних пикселей
передается в `parallel_for` как `blocked_range2d`.

Каждая задача получает свой поддиапазон строк и столбцов. Внутри этого поддиапазона выполняется обычный последовательный
обход пикселей.

### Работа с памятью

Исходное изображение `img` передается в `Filter` как константная ссылка и во время выполнения алгоритма не изменяется.

Результат записывается в поле класса:

`Image result_image_`

Массив `result_image_.pixels` заранее выделяется в конструкторе класса и имеет размер `height * width`.

Каждая задача записывает данные только в элементы, соответствующие своим координатам пикселей. Поскольку разные пиксели
имеют разные индексы в массиве, гонка данных при записи не возникает.

Дополнительные локальные аккумуляторы для объединения результатов не требуются, так как в алгоритме нет операции
редукции.

Специальные аллокаторы oneTBB, такие как `cache_aligned_allocator` или `scalable_allocator`, в данной реализации не
используются. Это допустимо, так как основные данные уже заранее размещены в векторах, а во время параллельной обработки
не выполняется интенсивное динамическое выделение памяти.

### Возможное ложное разделение кэша

Теоретически при записи соседних пикселей разными задачами может возникать ложное разделение кэша, так как значения
пикселей хранятся последовательно в памяти. Однако каждая операция записи выполняется один раз, а основная часть работы
связана с чтением окрестности пикселя и вычислением свертки.

В данной реализации специальных мер против false sharing не добавлялось. Для более тонкой оптимизации можно было бы
подбирать grain size так, чтобы задачи обрабатывали более крупные прямоугольные блоки изображения, уменьшая вероятность
частого разделения одной cache line между разными worker-потоками.

## 6. Проверка корректности

Проверка корректности TBB-версии производилась сравнением результата с последовательной реализацией алгоритма.

Последовательная версия используется как baseline, так как она проще по структуре и не содержит параллельного
выполнения.

Для тестирования использовались:

- изображения размером `3x3`, `5x5`, `7x7`, `9x9`;
- заранее подготовленные тестовые паттерны;
- попиксельное сравнение результатов.

Дополнительно проверялись:

- корректность обработки границ изображения;
- корректность обработки внутренних пикселей;
- совпадение результата TBB-версии с SEQ-версией;
- корректность работы при различном значении `ppc::util::GetNumThreads()`;
- отсутствие зависимости результата от порядка выполнения задач.

Так как каждый пиксель результата зависит только от исходного изображения, а исходное изображение не изменяется,
результат TBB-версии должен совпадать с результатом последовательной версии при любом допустимом уровне параллелизма.

Все функциональные тесты были успешно пройдены.

Формулы для расчета:

`Speedup = T_seq / T_tbb`

`Efficiency = Speedup / Workers`

где:

- `T_seq` — время последовательной версии;
- `T_tbb` — время TBB-версии;
- `Workers` — максимальное число worker-потоков, разрешенное через `global_control`.

При увеличении числа worker-потоков ожидается уменьшение времени выполнения алгоритма, так как обработка разных пикселей
независима.

Основные факторы, ограничивающие масштабируемость:

- накладные расходы oneTBB runtime;
- разбиение диапазона на задачи;
- слишком маленький grain size на небольших изображениях;
- ограниченная пропускная способность памяти;
- возможное ложное разделение кэша при записи соседних пикселей;
- стоимость чтения окрестности `3x3` для каждого пикселя.

Так как в реализации не используется явный partitioner, балансировка выполняется runtime oneTBB. Для данной задачи это
приемлемо, потому что нагрузка между пикселями равномерная.

## 9. Выводы

TBB-реализация фильтра Собеля эффективно использует независимость вычислений для разных пикселей изображения.

Для распараллеливания был выбран примитив `oneapi::tbb::parallel_for`, так как алгоритм представляет собой регулярный
обход двумерной области данных. Диапазон работы описан через `oneapi::tbb::blocked_range2d<size_t>`, что позволяет
oneTBB делить изображение на прямоугольные поддиапазоны.

Количество параллельно выполняемых worker-потоков ограничивается через
`oneapi::tbb::global_control::max_allowed_parallelism`. При этом важно, что данный механизм задает верхнюю границу
параллелизма, а не вручную создает фиксированное количество потоков для конкретного цикла.

TBB-версия удобна для данной задачи, потому что:

- алгоритм имеет регулярную структуру;
- вычисления разных пикселей независимы;
- отсутствует необходимость в редукции;
- не требуется ручная синхронизация;
- runtime oneTBB берет на себя планирование задач.

На больших изображениях TBB-реализация должна давать ускорение относительно последовательной версии, так как объем
вычислений существенно превышает накладные расходы на создание и планирование задач.

На маленьких изображениях ускорение может быть незначительным или отсутствовать, потому что overhead oneTBB может быть
сопоставим со временем самой фильтрации.

Основными ограничениями масштабируемости являются накладные расходы runtime, выбор grain size, пропускная способность
памяти и возможное ложное разделение кэша при записи в соседние элементы результирующего изображения.
