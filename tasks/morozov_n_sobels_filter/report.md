## 1. Введение

В задачах связанных с обработкой изображений нередко требуется выделять конкретные объекты. Многие алгоритмы нахождения
объектов на изображении требуют выделенных границ объекта. Основной вопрос как же выделить сами границы. Данную задачу
решает оператор Собеля.

Задачи обработки изображения - работа с большими массивами чисел, следовательно параллельная обработка поможет увеличить
эффективность вычислений. Последовательная версия - эталон корректности для будущих параллельных алгоритмов и точка
отсчета времени выполнения операции над изображением.

В данном отчёте обобщаются результаты пяти реализаций:

- **SEQ** представляет из себя последовательную версию эталона корректности.
- **OMP** предполагает распараллеливание через директивы OpenMP (общая память).
- **TBB** реализует задачно-ориентированное распараллеливание.
- **STL** имеет ручное управление потоками с помощью `std::thread`.
- **ALL** вляется гибридом `MPI+OpenMP`, сочетающим межпроцессное и внутрипроцессное распараллеливание.

## 2. Единая постановка задачи

Требуется выделить ребра на изображении с использованием оператора Собеля.

**Входные / выходные данные :** изображение в оттенках серого с высотой `H` и шириной `W`.

В программе данные такого типа описаны структурой `Image`:

```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

## 3. Единая методика эксперимента

**Аппаратное окружение:**

- Процессор: Apple m2 (8 ядер)
- RAM: 16 ГБ
- ОС: Docker Ubuntu 24.04 / Host MacOs 15.7.3

**Программное окружение:**

- Компилятор: g++ v13.3.0, сборка Release
- MPI-реализация: OpenMPI MPI (для ALL)
- Переменные окружения: `PPC_NUM_THREADS` (для OMP/STL/TBB), `PPC_NUM_PROC` (для MPI в ALL)

**Размер задачи:** для всех замеров использовалось сгенерированное изображение размером `8000 x 8000` пикселей

**Методика измерений:** каждый замер был повторён от 2 до 5 раз, после чего было зафиксировано среднее арифметическое.

Формула ускорения была предположена в виде `speedup = T_seq / T_par`, где `T_seq` - время выполнения последовательной
версии, `T_par` - параллельной.

Формула подсчета эффективности выполнения в общем виде может выглядеть так: `efficiency = speedup / threads`, где
`threads` - число потоков.

Во всех версиях алгоритма результаты обработки совпали с последовательной реализацией. Различий в вычисленных значениях

## 6. Интерпретация различий

Последовательная реализация SEQ в данном отчёте используется как базовый вариант для сравнения. Она показывает время
выполнения алгоритма без параллельной обработки и задаёт baseline, относительно которого рассчитывались ускорение и
эффективность остальных backend-ов. Для изображения размером `8000 x 8000` пикселей время SEQ составило `0.2336` с.

Реализация OMP показала хорошее масштабирование на данной задаче. При переходе от 2 к 4 потокам время выполнения
уменьшилось с `0.1112` с до `0.0590` с, а ускорение выросло с `2.10x` до `3.96x`. Эффективность при 4 потоках составила
`99%`, что говорит о близком к линейному ускорении. Такой результат объясняется тем, что оператор Собеля хорошо
распараллеливается по пикселям изображения: вычисление значения для одного пикселя почти не зависит от вычислений для
других пикселей. При этом OpenMP позволяет достаточно просто распределить независимые участки изображения между
потоками.

TBB-реализация также дала устойчивое ускорение относительно SEQ, однако в представленных замерах она немного уступила
OMP и STL. При 2 потоках время составило `0.1257` с, ускорение — `1.86x`, эффективность — `93%`. При 4 потоках время
уменьшилось до `0.0672` с, ускорение составило `3.48x`, эффективность — `87%`. Более низкие значения могут быть связаны
с дополнительными накладными расходами runtime TBB на управление задачами и разбиение работы. Для данной задачи
вычисления имеют достаточно регулярную структуру, поэтому преимущества динамического распределения задач проявляются не
так сильно, как в задачах с неравномерной нагрузкой.

STL-реализация с ручным управлением потоками показала один из лучших результатов среди однопроцессных backend-ов. На 2
потоках время выполнения составило `0.1096` с, ускорение — `2.31x`, эффективность — `116%`. На 4 потоках время
уменьшилось до `0.0588` с, ускорение составило `3.97x`, а эффективность — `99%`. Результат практически совпадает с
OpenMP на 4 потоках и незначительно превосходит его по времени. Это может быть связано с удачным ручным разбиением
изображения на независимые диапазоны строк и низкими накладными расходами в данном конкретном запуске. При этом STL
требует более сложной ручной организации потоков, чем OpenMP.

Для ALL-реализации в сводной таблице пункта 5 не указаны значения времени выполнения, ускорения и эффективности. Поэтому
на основании текущих агрегированных данных невозможно корректно сравнить гибридную версию MPI+OpenMP с остальными
backend-ами. Для полноценной интерпретации ALL необходимо добавить в таблицу фактические результаты замеров: количество
MPI-процессов, число потоков на процесс, общее число workers, время выполнения, ускорение относительно SEQ и
эффективность. Только после этого можно оценивать, компенсирует ли выигрыш от гибридного распараллеливания накладные
расходы на межпроцессное взаимодействие.

В целом по имеющимся данным задача выделения рёбер оператором Собеля хорошо подходит для параллельной обработки. Лучшие
результаты среди заполненных строк таблицы показали STL и OMP на 4 потоках: их ускорение приблизилось к `4x`, а
эффективность составила около `99%`. TBB также показала стабильное ускорение, но в данном эксперименте оказалась немного
менее эффективной. Основная причина близких результатов OMP и STL заключается в регулярности алгоритма: изображение
можно разделить на независимые участки, а объём вычислений для разных частей примерно одинаков.

## 7. Репродуцируемость

**Сборка проекта**. Проект был собран при использовании системы CMake, основные команды можно видеть ниже.

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
```

Тестирование производительности было произведено при выставлении вручную переменных окружения и запуска тестов. Для
SEQ/OMP/STL/TBB справедлива последовательность команд:

```bash
export PPC_NUM_THREADS=<N>
./ppc_func_tests --gtest_filter="*morozov_n_sobels_filter*"
```

Для ALL справедлива последовательность команд:

```bash
set PPC_NUM_PROC=<ranks>
set OMP_NUM_THREADS=<threads_per_rank>
mpirun -n <N> --allow-run-as-root './ppc_perf_tests' --gtest_filter="*morozov_n_sobels_filter*
```

## 8. Заключение

В ходе работы были сопоставлены несколько реализаций алгоритма выделения рёбер на изображении с использованием оператора
Собеля: SEQ, OMP, TBB, STL и ALL. Последовательная версия SEQ использовалась как эталон корректности и базовая точка для
расчёта ускорения.

По результатам представленных замеров можно сделать вывод, что для данной задачи наиболее эффективными среди заполненных
результатов оказались реализации STL и OMP на 4 потоках. STL показала минимальное время выполнения `0.0588` с и
ускорение `3.97x`, а OMP показала очень близкий результат — `0.0590` с и ускорение `3.96x`. Обе реализации
продемонстрировали эффективность около `99%`, что говорит о хорошем масштабировании алгоритма на 4 потоках.

OpenMP можно считать наиболее практичным вариантом для данной задачи, так как он сочетает высокую производительность,
простоту реализации и минимальные изменения исходного последовательного алгоритма. STL также показала высокую
производительность, но требует ручного управления потоками и более аккуратного разделения данных, поэтому такая
реализация сложнее в сопровождении.

TBB в данном эксперименте уступила OMP и STL по времени выполнения и эффективности, однако всё равно обеспечила
значительное ускорение относительно SEQ. Эту реализацию можно считать пригодной для задачи, особенно если в дальнейшем
потребуется более гибкое управление задачами или обработка менее равномерных данных.

По ALL-версии окончательный вывод сделать нельзя, так как в сводной таблице отсутствуют численные результаты для этого
backend-а. Потенциально гибридная схема MPI+OpenMP может быть полезна для очень больших изображений или запуска на
распределённых вычислительных системах, однако для подтверждения этого необходимо добавить результаты измерений и
отдельно оценить влияние коммуникационных расходов.

Таким образом, на текущем наборе данных лучшими вариантами для обработки изображения оператором Собеля являются OMP и
STL. Если выбирать реализацию для практического использования, предпочтительнее OMP, так как она показывает почти такое
же время, как STL, но проще по структуре кода. Для дальнейшего улучшения отчёта необходимо дополнить результаты
ALL-версии и, при возможности, провести дополнительные замеры на изображениях разных размеров, чтобы оценить
масштабируемость backend-ов не только на одном размере задачи.
