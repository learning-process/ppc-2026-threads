## 1. Контекст

В задачах обработки изображений часто требуется выделение границ объектов. Для решения данной задачи применяется
оператор Собеля, позволяющий вычислить градиент яркости изображения и определить области резкого изменения интенсивности
пикселей.

В данном отчете рассматривается распараллеливание алгоритма фильтра Собеля при помощи стандартной библиотеки потоков C++
(`std::thread`). Использование многопоточности позволяет ускорить обработку больших изображений за счет независимого
вычисления различных областей изображения.

Последовательная версия алгоритма используется как эталон корректности и базовая точка отсчета при анализе
производительности STL-реализации.

---

## 2. Постановка задачи

Требуется выделить ребра на изображении с использованием оператора Собеля.

Входные / выходные данные: изображение в оттенках серого с высотой `H` и шириной `W`.

В программе данные представлены структурой `Image`:

```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

Каждый пиксель хранит значение яркости в диапазоне `[0; 255]`.

Полное описание постановки задачи и математической основы алгоритма приведено в `seq/report.md`.

---

## 3. Базовый алгоритм

Последовательный алгоритм выполняет проход по всем внутренним пикселям изображения и вычисляет для каждого нового
значение градиента яркости.

Для вычисления используются два ядра Собеля:

- по оси `X`;
- по оси `Y`.

После вычисления сверток определяется итоговый градиент:

```text
sqrt(gradX^2 + gradY^2)
```

Полученное значение ограничивается диапазоном `[0; 255]` и записывается в результирующее изображение.

Каждый пиксель вычисляется независимо от остальных, так как использует только данные исходного изображения и не изменяет
соседние пиксели.

---

## 4. Схема распараллеливания

Основой распараллеливания является следующий инвариант:

- результирующее значение пикселя `i, j` зависит только от окрестности `3x3` исходного изображения.

Из этого следует, что вычисления для различных пикселей независимы между собой:

1. Исходное изображение не изменяется во время работы алгоритма;
2. Каждый поток записывает результат только в собственный диапазон строк результирующего изображения;
3. Одновременная запись нескольких потоков в одну ячейку памяти невозможна.

### Разделение данных

Изображение разбивается между потоками по строкам. Каждый поток получает непрерывный диапазон строк изображения для
обработки.

Количество потоков определяется через:

```cpp
const int k_num_threads = ppc::util::GetNumThreads();
```

Размер диапазона рассчитывается следующим образом:

- вычисляется базовое количество строк на поток;
- оставшиеся строки распределяются между первыми потоками.

### Работа потоков

Каждый поток выполняет метод `Filter`, в который передаются:

- ссылка на исходное изображение;
- начальная строка обработки;
- количество строк.

Поток независимо вычисляет новые значения пикселей только в пределах своего диапазона.

### Хранение результатов

Результаты обработки сохраняются в общее поле класса:

```cpp
Image result_image_
```

Дополнительная синхронизация записи не требуется, так как каждый поток работает с уникальным диапазоном строк
результирующего изображения.

### Синхронизация потоков

После запуска всех рабочих потоков главный поток ожидает завершения их работы при помощи `join()`:

```cpp
for (auto &thread : threads) {
    if (thread.joinable()) {
      thread.join();
    }
}
```

Только после завершения всех потоков результирующее изображение считается полностью сформированным.

## 5. Детали реализации

Основные изменения относительно последовательной версии находятся в файле:

```text
stl/src/ops_stl.cpp
```

В STL-версии были изменены:

- метод `RunImpl()`;
- сигнатура метода `Filter()`.

### Метод `RunImpl`

Внутри `RunImpl()`:

1. Получаются входные данные через `GetInput()`;
2. Вычисляются диапазоны строк для потоков;
3. Создаются рабочие потоки через `std::thread`;
4. Каждый поток запускает метод `Filter`;
5. Главный поток ожидает завершения всех рабочих потоков через `join`.

Фрагмент создания потоков:

```cpp
for (int i = 0; i < k_num_threads; i++) {
    size_t num_rows = rows_per_thread +
        (std::cmp_less(i, remaining_rows) ? 1 : 0);

    if (num_rows > 0) {
      threads.emplace_back(
        [this, &input, current_start, num_rows]() {
          this->Filter(input, current_start, num_rows);
        });
    }

    current_start += num_rows;
}
```

### Метод `Filter`

Метод `Filter()` выполняет обработку только заданного диапазона строк изображения:

```cpp
void MorozovNSobelsFilterSTL::Filter(
    const Image &img,
    size_t start_row,
    size_t num_rows)
```

Внутри метода выполняется проход по пикселям выделенного диапазона и вычисление новых значений через
`CalculateNewPixelColor()`.

### Вычисление нового пикселя

Метод `CalculateNewPixelColor()` полностью совпадает с последовательной версией алгоритма:

- выполняется свертка ядрами Собеля;
- вычисляется градиент;
- результат ограничивается диапазоном `[0; 255]`.

Так как вычисления для каждого пикселя независимы, дополнительная синхронизация потоков внутри алгоритма не требуется.

Формулы для расчета:

`Speedup = T_seq / T_stl`

`Efficiency = Speedup / Threads`

где:

- `T_seq` — время последовательной версии;
- `T_stl` — время STL-версии;
- `Threads` — количество потоков, созданных через `std::thread`.

Основные факторы, которые могут ограничивать масштабируемость STL-версии:

- накладные расходы на создание и завершение потоков;
- необходимость ожидания всех потоков через `join()`;
- ограниченная пропускная способность памяти;
- возможное ложное разделение кэша при записи соседних пикселей;
- влияние планировщика операционной системы.

В данной реализации дополнительные механизмы синхронизации (`mutex`, `atomic`, `condition_variable`) не используются,
потому что каждый поток записывает результат только в свой диапазон строк. Это уменьшает накладные расходы и делает
ручную многопоточную реализацию эффективной для данной задачи.

## 9. Выводы

Ручная реализация на `std::thread` показала, что задача фильтра Собеля хорошо распараллеливается при разбиении
изображения по строкам.

Главное преимущество STL-версии заключается в полном контроле над созданием потоков и распределением диапазонов данных.
В реализации явно задается, какие строки обрабатывает каждый поток, а главный поток ожидает завершения всех рабочих
потоков через `join()`.

STL-версия эффективна для данной задачи, потому что:

- вычисления разных пикселей независимы;
- изображение можно разбить на непересекающиеся диапазоны строк;
- запись результата выполняется в разные элементы массива;
- не требуется защита общих данных через `mutex` или `atomic`;
- отсутствует редукция или сложное объединение локальных результатов.

При этом ручное управление потоками требует большей аккуратности, чем OpenMP или TBB. Необходимо самостоятельно
рассчитать диапазоны строк, создать потоки, передать им корректные границы обработки и дождаться завершения всех
потоков.

На больших изображениях накладные расходы на создание потоков компенсируются объемом вычислений. На маленьких
изображениях STL-версия может быть менее выгодной, потому что стоимость создания и синхронизации потоков может быть
сравнима со временем самой фильтрации.
