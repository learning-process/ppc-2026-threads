## 2. Постановка задачи
Требуется выделить ребра на изображении с использованием оператора Собеля.

Входные / выходные данные: изображение в оттенках серого с высотой `H` и шириной `W`. 

В программе данные такого типа описаны структурой `Image`:
```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

## 3. Базовый алгоритм
В последовательной версии алгоритма происходил обход по всем пикселям входного изображения с вычислением градиента относительно окружающих его пикселей. Новый пиксель записывался в результирующее изображение на месте старого.

## 4. Схема распараллеливания
Основой для распараллеливания алгоритма является инвариант:
- Результирующее значение пикселя `i,j` зависит от его окрестных (матрица 3х3) пикселей исходного изоражения.

Из данного инварианта можно понять, что операции по вычислению нового значения пикселя путем свертки матриц и сохранение его в результирующее изображение между двумя различными пикселями полностью независимы, потому что:
1. Исходное изображение неизменно => окрестность пикселя не будет затронута другим потоком
2. Адреса двух пикселей в матрице изображения различны => в момент записи 2 различных потока не могут перетереть записи друг друга

В результате такого исследования схема распараллеливания имеет следующий вид: 
1. Цикл разбиения изображения разбивается на строки между потоками
2. Каждый поток изолированно вычисляет свой кусок изображения
3. По завершению своей работы отдельный поток ждет пока завершатся остальные
4. Как только работу завершили все, потоки кроме 0 освобождаются
5. 0 поток продолжает выполнение программы

**Распределение переменных**
- default(none) - обозначает что все переменные не указанные явно в клаузах (shared, private и т. д.), не получают автоматически никакого статуса
- shared: img (исходное изображение), [косвенно] result_image_
- private: this, индексы i, j

Важное уточнение:
Для полей класса в OpenMP действуют особые правила. Поле result_image_ доступно через указатель this, и this по умолчанию считается firstprivate в методе класса. Это означает, что каждый поток получает свою копию указателя this, но все эти копии указывают на один и тот же объект в памяти.

Для данной схемы требуется барьер по окончании работы цикла.

Для распараллеливания цикла использовалась дирректива:
```cpp
#prama omp parallel for schedule(static) default(none) shared(img) num_threads(ppc::util::GetNumThreads())
```
`schedule` был выбран с парметром `static` по следующим соображениям:
- Вычислительная сложность между итерациями в среднем  не изменяется, поэтому статическое распредление даст выйгрыш на этапе компиляции, тогда как динамическое явного выйгрыша в данной ситуации не имеет

## 5. Детали реализации

только название класса `MorozovNSobelsFilterOMP`,

`omp/src/ops_omp.cpp`
Относительно `Seq` версии изменен только метод `Filter` -> добавлена #pragma omp parallel for
```cpp
void MorozovNSobelsFilterOMP::Filter(const Image &img) {
#pragma omp parallel for schedule(static) default(none) shared(img) num_threads(ppc::util::GetNumThreads())
  for (size_t id_y = 1; id_y < img.height - 1; id_y++) {
    for (size_t id_x = 1; id_x < img.width - 1; id_x++) {
      size_t pixel_id = (id_y * img.width) + id_x;
      result_image_.pixels[pixel_id] = CalculateNewPixelColor(img, id_x, id_y);
    }
  }
}
```

Основной риск гонок был убран при помощи распредления итераций внешего цикла между потоками прагмой. Итерации внутреннего цикла защищены от гонок распределением переменных (описанного пунктов выше).

Требующийся барьер для корректного продолжения работы программы одним потоком обеспецивает сама прагма.
