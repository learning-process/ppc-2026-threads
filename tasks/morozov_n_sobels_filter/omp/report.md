## 1. Контекст

В задачах обработки изображений часто требуется выделение границ объектов. Для решения данной задачи используется
оператор Собеля, позволяющий вычислить градиент яркости изображения и определить области резкого изменения интенсивности
пикселей.

В данном отчете рассматривается распараллеливание алгоритма фильтра Собеля при помощи технологии OpenMP. Использование
OpenMP позволяет ускорить обработку изображения за счет распределения вычислений между несколькими потоками.

Последовательная версия алгоритма используется как эталон корректности и базовая точка отсчета при анализе
производительности OpenMP-реализации.

## 2. Постановка задачи

Требуется выделить ребра на изображении с использованием оператора Собеля.

Входные / выходные данные: изображение в оттенках серого с высотой `H` и шириной `W`.

В программе данные представлены структурой `Image`:

```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

Каждый пиксель хранит значение яркости в диапазоне `[0; 255]`.

Полное описание математической основы алгоритма приведено в `seq/report.md`.

---

## 3. Базовый алгоритм

Последовательный алгоритм выполняет проход по всем внутренним пикселям изображения и вычисляет для каждого новое
значение градиента яркости.

Для вычисления используются два ядра Собеля:

- по оси `X`;
- по оси `Y`.

Для каждого пикселя вычисляются:

- градиент по оси `X`;
- градиент по оси `Y`;
- итоговое значение градиента:

```text
sqrt(gradX^2 + gradY^2)
```

Полученное значение ограничивается диапазоном `[0; 255]` и записывается в результирующее изображение.

Каждый пиксель вычисляется независимо от остальных, так как использует только данные исходного изображения.

---

## 4. Схема распараллеливания

Основой распараллеливания алгоритма является следующий инвариант:

- результирующее значение пикселя `i, j` зависит только от окрестности `3x3` исходного изображения.

Из этого следует, что вычисления различных пикселей независимы между собой:

1. Исходное изображение не изменяется во время работы алгоритма;
2. Каждый поток записывает результат только в собственные элементы результирующего изображения;
3. Одновременная запись нескольких потоков в одну ячейку памяти невозможна.

### Разделение работы между потоками

Распараллеливание выполняется по строкам изображения.

Итерации внешнего цикла распределяются между потоками при помощи директивы:

```cpp
#pragma omp parallel for
```

Каждый поток получает собственный диапазон строк изображения и независимо выполняет вычисления для принадлежащих ему
пикселей.

### Распределение переменных

Для распараллеливания использовалась следующая директива:

```cpp
#pragma omp parallel for schedule(static) \
default(none) shared(img) \
num_threads(ppc::util::GetNumThreads())
```

Используемые клаузы:

- `default(none)` — требует явного указания области видимости переменных;
- `shared(img)` — исходное изображение является общим для всех потоков;
- `schedule(static)` — статическое распределение итераций между потоками;
- `num_threads(...)` — задает количество потоков.

Переменные циклов `id_x` и `id_y` являются private автоматически, так как объявлены внутри циклов.

### Работа с полями класса

Поле класса:

```cpp
Image result_image_
```

используется всеми потоками совместно через указатель `this`.

В OpenMP указатель `this` внутри метода класса является `firstprivate`, однако все его копии указывают на один и тот же
объект класса. Благодаря тому, что каждый поток записывает данные только в собственный диапазон строк результирующего
изображения, гонки данных не возникают.

### Выбор schedule(static)

Параметр `schedule(static)` был выбран по следующим причинам:

- вычислительная нагрузка между итерациями распределена равномерно;
- стоимость вычисления каждого пикселя примерно одинакова;
- статическое распределение уменьшает накладные расходы runtime OpenMP.

Использование `schedule(dynamic)` в данной задаче не дает существенного преимущества, так как отсутствует дисбаланс
нагрузки между потоками.

### Синхронизация

После завершения работы директивы `parallel for` OpenMP автоматически выполняет неявный барьер синхронизации.

Это гарантирует, что все потоки завершат обработку изображения до продолжения выполнения программы главным потоком.

## 5. Детали реализации

Основные изменения относительно последовательной версии находятся в файле:

```text
omp/src/ops_omp.cpp
```

В OpenMP-версии был изменен только метод `Filter()`.

### Метод Filter

Внутри метода была добавлена директива OpenMP для распараллеливания внешнего цикла:

```cpp
void MorozovNSobelsFilterOMP::Filter(const Image &img) {
#pragma omp parallel for schedule(static) default(none) \
shared(img) num_threads(ppc::util::GetNumThreads())

  for (size_t id_y = 1; id_y < img.height - 1; id_y++) {
    for (size_t id_x = 1; id_x < img.width - 1; id_x++) {
      size_t pixel_id = (id_y * img.width) + id_x;
      result_image_.pixels[pixel_id] =
          CalculateNewPixelColor(img, id_x, id_y);
    }
  }
}
```

Каждый поток независимо вычисляет значения пикселей своего диапазона строк и записывает результат в результирующее
изображение.

### Предотвращение гонок данных

Основной риск гонок данных устраняется следующим образом:

- итерации внешнего цикла распределяются между потоками OpenMP;
- каждый поток работает только со своим диапазоном строк;
- запись нескольких потоков в один и тот же элемент массива невозможна.

Дополнительные механизмы синхронизации (`critical`, `atomic`, `mutex`) не требуются.

## 8. Результаты

Таблица результатов содержит:

- количество потоков;
- время выполнения;
- ускорение (`speedup`);
- эффективность (`efficiency`).

| Backend | Конфигурация | Время, с | Ускорение | Эффективность |
| ------- | ------------ | -------- | --------- | ------------- |
| SEQ     | 1            | 0.2336   | 1.00x     | 100%          |
| OMP     | 2 потока     | 0.1112   | 2.10x     | 105%          |
| OMP     | 4 потока     | 0.0590   | 3.96x     | 99%           |

Формулы для расчета:

`Speedup = T_seq / T_omp`

`Efficiency = Speedup / Workers`

где:

- `T_seq` — время последовательной версии;
- `T_omp` — время TBB-версии;
- `Workers` — максимальное число worker-потоков.

При увеличении числа worker-потоков ожидается уменьшение времени выполнения алгоритма, так как обработка разных пикселей
независима.

При увеличении числа потоков наблюдается уменьшение времени выполнения алгоритма за счет независимости вычислений для
различных областей изображения.

Основными ограничениями масштабируемости являются:

- накладные расходы OpenMP runtime;
- синхронизация потоков на барьере;
- ограничение пропускной способности памяти при обработке больших изображений.

Наилучший выигрыш достигается на изображениях большого размера, где вычислительная нагрузка значительно превышает
накладные расходы на создание и синхронизацию потоков.

Основным ограничением масштабируемости являются накладные расходы OpenMP runtime и ограниченная пропускная способность
