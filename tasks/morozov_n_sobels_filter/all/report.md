## 1. Контекст

В задачах обработки изображений часто требуется выделение границ объектов. Для решения данной задачи используется
оператор Собеля, позволяющий вычислить градиент яркости изображения и определить области резкого изменения интенсивности
пикселей.

В данном отчете рассматривается гибридная реализация алгоритма фильтра Собеля с использованием MPI и OpenMP.

MPI используется для распределения изображения между несколькими процессами, а OpenMP применяется внутри каждого
процесса для параллельной обработки локального фрагмента изображения несколькими потоками.

Последовательная версия алгоритма используется как эталон корректности и базовая точка отсчета при анализе
производительности ALL-реализации.

## 2. Постановка задачи

Требуется выделить ребра на изображении с использованием оператора Собеля.

Входные / выходные данные: изображение в оттенках серого с высотой `H` и шириной `W`.

В программе данные представлены структурой `Image`:

```cpp
struct Image {
  size_t height;
  size_t width;
  std::vector<uint8_t> pixels;
};
```

Каждый пиксель хранит значение яркости в диапазоне `[0; 255]`.

Полное описание постановки задачи и математической основы алгоритма приведено в `seq/report.md`.

---

## 3. Базовый алгоритм

Последовательный алгоритм выполняет проход по всем внутренним пикселям изображения и вычисляет для каждого новое
значение градиента яркости.

Для вычисления используются два ядра Собеля:

- по оси `X`;
- по оси `Y`.

После вычисления сверток определяется итоговый градиент:

`sqrt(gradX^2 + gradY^2)`

Полученное значение ограничивается диапазоном `[0; 255]` и записывается в результирующее изображение.

Каждый пиксель вычисляется независимо от остальных, так как использует только данные исходного изображения и не изменяет
соседние пиксели. Это свойство позволяет распараллелить алгоритм как между MPI-процессами, так и между потоками внутри
каждого процесса.

---

## 4. Межпроцессная схема

В ALL-версии изображение распределяется между MPI-процессами по строкам.

Каждый процесс получает свой непрерывный диапазон строк изображения. Номер текущего процесса и общее количество
процессов определяются в конструкторе класса при помощи:

`MPI_Comm_rank(MPI_COMM_WORLD, &rank_)`

`MPI_Comm_size(MPI_COMM_WORLD, &size_)`

### Разделение строк между процессами

Для разбиения изображения используется метод `SplitRows`.

Метод принимает:

- общее количество строк изображения;
- номер процесса;
- ссылку на начальную строку;
- ссылку на количество строк.

Разбиение выполняется следующим образом:

- вычисляется базовое количество строк на процесс;
- остаток от деления распределяется между первыми процессами;
- для каждого процесса определяется начальная строка и количество строк.

Такое разбиение позволяет равномерно распределить изображение между процессами даже в случае, если высота изображения не
делится на количество процессов без остатка.

### Передача размеров изображения

В методе `PreProcessingImpl()` процесс с рангом `0` получает размеры исходного изображения.

После этого ширина и высота изображения передаются всем процессам при помощи:

`MPI_Bcast(&width, 1, MPI_UINT64_T, 0, MPI_COMM_WORLD)`

`MPI_Bcast(&height, 1, MPI_UINT64_T, 0, MPI_COMM_WORLD)`

Это необходимо, чтобы каждый процесс мог самостоятельно вычислить свой диапазон строк и подготовить локальное
изображение.

### Halo-области

Оператор Собеля использует окрестность `3x3`, поэтому для вычисления пикселей на границах локального блока процессу
нужны соседние строки из других блоков.

Для этого при распределении данных используется halo-область размером в одну строку:

- если процесс обрабатывает не первую часть изображения, ему передается одна дополнительная строка сверху;
- если процесс обрабатывает не последнюю часть изображения, ему передается одна дополнительная строка снизу.

Это позволяет корректно вычислять градиент для строк, находящихся на границах блоков между MPI-процессами.

### Рассылка данных

Рассылка изображения выполняется в методе `SendImageDataFromZeroProc`.

Процесс с рангом `0`:

1. Для каждого процесса вычисляет диапазон строк через `SplitRows()`;
2. Добавляет необходимые halo-строки;
3. Определяет смещение в исходном изображении;
4. Копирует свой локальный блок в `local_image_`;
5. Остальным процессам отправляет их блоки через `MPI_Send`.

Остальные процессы принимают свои локальные блоки через `MPI_Recv`.

## 5. Внутрипроцессная схема

После получения локального фрагмента изображения каждый MPI-процесс выполняет обработку своего блока.

Внутри процесса используется OpenMP. Распараллеливание выполняется по строкам локального изображения.

Основной инвариант остается тем же:

- результирующее значение пикселя `i, j` зависит только от окрестности `3x3` исходного изображения.

Из этого следует, что вычисления разных пикселей независимы:

1. Локальное исходное изображение `local_image_` во время фильтрации не изменяется;
2. Каждый поток записывает результат только в собственные элементы `local_result`;
3. Одновременная запись нескольких потоков в одну ячейку памяти невозможна.

### OpenMP-распараллеливание

В методе `Filter()` используется директива:

`#pragma omp parallel for schedule(static) default(none) shared(img, local_result, start_row, end_row) num_threads(ppc::util::GetNumThreads())`

Используемые клаузы:

- `default(none)` — требует явно указывать область видимости переменных;
- `shared(img, local_result, start_row, end_row)` — исходный и результирующий локальные блоки общие для потоков;
- `schedule(static)` — строки распределяются между потоками статически;
- `num_threads(ppc::util::GetNumThreads())` — количество потоков берется из инфраструктуры курса.

Выбор `schedule(static)` оправдан тем, что вычислительная нагрузка для строк изображения распределена равномерно: каждый
внутренний пиксель обрабатывается одним и тем же набором операций свертки.

### Границы обработки локального блока

В методе `RunImpl()` задаются границы обработки:

`start_row = 1`

`end_row = local_image_.height - 1`

Это связано с тем, что первая и последняя строки локального изображения могут быть halo-строками. Они нужны для
вычислений, но не должны попадать в итоговый результат как самостоятельные строки данного процесса.

## 6. Детали реализации

Основные файлы реализации:

- `all/include/ops_all.hpp`;
- `all/src/ops_all.cpp`.

В классе `MorozovNSobelsFilterALL` хранятся:

- `rank_` — ранг текущего MPI-процесса;
- `size_` — общее количество MPI-процессов;
- `local_rows_` — количество строк исходного изображения, закрепленных за процессом;
- `start_row_` — начальная строка процесса в глобальном изображении;
- `local_image_` — локальный блок изображения с учетом halo-строк;
- `result_image_` — локальный результат обработки.

### `ValidationImpl`

Метод `ValidationImpl()` проверяет согласованность входного и результирующего изображения:

- высота входного изображения совпадает с высотой результата;
- ширина входного изображения совпадает с шириной результата;
- количество пикселей во входном изображении совпадает с количеством пикселей в результате.

### `PreProcessingImpl`

В методе `PreProcessingImpl()` выполняется подготовка данных:

1. Процесс `0` получает размеры глобального изображения;
2. Размеры изображения рассылаются всем процессам через `MPI_Bcast`;
3. Каждый процесс вычисляет свой диапазон строк через `SplitRows`;
4. Рассчитывается наличие верхней и нижней halo-строки;
5. Выделяется память под локальное изображение;
6. Процесс `0` рассылает блоки изображения;
7. Остальные процессы принимают свои блоки через `MPI_Recv`.

### `RunImpl`

В методе `RunImpl()` выполняется основная обработка:

1. Подготавливается локальный результирующий блок;
2. Запускается метод `Filter()` для локального изображения;
3. Локальные результаты собираются в глобальный результат при помощи `CollectResult()`.

### `Filter`

Метод `Filter()` выполняет OpenMP-обработку локального блока изображения.

Внешний цикл по строкам распараллелен, внутренний цикл проходит по пикселям строки. Для каждого пикселя вызывается
`CalculateNewPixelColor()`.

Так как каждый поток записывает результат только для конкретного пикселя локального результата, дополнительные механизмы
синхронизации (`critical`, `atomic`, `mutex`) не требуются.

### `CalculateNewPixelColor`

Метод `CalculateNewPixelColor()` совпадает по логике с последовательной версией алгоритма:

- выполняется свертка по ядру Собеля для оси `X`;
- выполняется свертка по ядру Собеля для оси `Y`;
- вычисляется итоговый градиент;
- значение ограничивается диапазоном `[0; 255]`;
- результат возвращается как `uint8_t`.

Для защиты от выхода за границы изображения используется `std::clamp`.

### `CollectResult`

Метод `CollectResult()` собирает результат с MPI-процессов.

Каждый процесс отправляет только свои основные строки без halo-областей.

Процесс с рангом `0`:

1. Создает глобальное результирующее изображение;
2. Копирует свой локальный результат;
3. Принимает результаты от остальных процессов через `MPI_Recv`;
4. Записывает собранное изображение в `GetOutput()`.

Остальные процессы отправляют свои локальные результаты процессу `0` через `MPI_Send`.

### `PostProcessingImpl`

В методе `PostProcessingImpl()` итоговое изображение рассылается всем процессам при помощи:

`MPI_Bcast(out.pixels.data(), static_cast<int>(out.pixels.size()), MPI_UNSIGNED_CHAR, 0, MPI_COMM_WORLD)`

Это нужно для того, чтобы после завершения задачи все процессы имели согласованный результат в `GetOutput()`.

## 10. Выводы

ALL-реализация позволила совместить межпроцессный и внутрипроцессный параллелизм при обработке изображения оператором
Собеля.

MPI используется для распределения изображения между процессами по строкам. Для корректной обработки границ блоков
передаются halo-строки, необходимые для вычисления свертки `3x3`.

OpenMP используется внутри каждого MPI-процесса для параллельной обработки локального блока изображения.
Распараллеливание по строкам является корректным, так как значение каждого результирующего пикселя зависит только от
исходного изображения и не зависит от значений соседних результирующих пикселей.

Основное преимущество ALL-подхода проявляется на больших изображениях, где объем вычислений достаточно велик по
сравнению с накладными расходами на коммуникацию между процессами.

Основными ограничениями гибридной реализации являются передача данных между MPI-процессами, работа с halo-областями,
сборка результата на процессе `0` и финальная синхронизация результата между всеми процессами.
