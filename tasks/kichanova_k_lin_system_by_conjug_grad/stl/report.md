## 1. Контекст

Параллельная реализация метода сопряжённых градиентов с ручным управлением потоками через `std::thread`,
организованными в пул потоков (ThreadPool).
Эта версия демонстрирует прямой контроль над распределением работы и синхронизацией
без использования высокоуровневых библиотек (OpenMP, TBB).

## 2. Постановка задачи

Решение СЛАУ Ax = b с симметричной матрицей A со всеми положительными собственными значениями методом сопряжённых градиентов.
SEQ-версия (seq/report.md) служит baseline корректности и производительности.

## 3. Базовый алгоритм

Вычисляет решение СЛАУ Ax = b итерационным методом: на каждом шаге умножает матрицу на вектор,
считает скалярные произведения и обновляет приближение, пока норма невязки не станет меньше заданной точности.
Параллелизм реализован через разбиение диапазонов итераций между потоками пула.

## 4. Схема распараллеливания

**Архитектура:** ThreadPool. Создаётся единственный экземпляр `ThreadPool::Instance()`,
инициализируемый числом потоков, равным `ppc::util::GetNumThreads()`.
Потоки запускаются один раз в конструкторе и ожидают задачи в очереди, защищённой `std::mutex` и `std::condition_variable`.

**Разбиение данных:**

- `ParallelFor(n, func)`: диапазон `[0, n)` делится на num_threads равных частей.
Последний поток получает остаток от деления. Каждая часть отправляется в пул как отдельная задача,
главный поток ожидает завершения через `std::future::get()`.
- `ParallelReduce(n, func)`: аналогичное разбиение,
каждый поток возвращает частичную сумму через `std::future<double>`, результаты суммируются последовательно в главном потоке.

**Синхронизация:**

- `std::mutex` + `std::condition_variable` защищают очередь задач в пуле.
- `std::future::get()` в `ParallelFor/ParallelReduce` блокирует главный поток
  до завершения всех задач - аналог join для пула.
- Между вычислительными ядрами синхронизация неявная:
  следующий `ParallelFor/ParallelReduce` вызывается только после возврата из предыдущего.

**Отличие от наивного `std::thread`:** потоки создаются один раз и переиспользуются через очередь задач.
Это исключает накладные расходы на создание и присоединение (create/join) на каждой итерации,
приближая ThreadPool по эффективности к рантаймам OpenMP и TBB.

**Ключевой фрагмент (ParallelFor):**

```cpp
void ParallelFor(int n, const std::function<void(int, int)>& func) {
  if (n <= 0) return;
  int num_threads = static_cast<int>(threads_.size());
  if (num_threads == 0) { func(0, n); return; }
  std::vector<std::future<void>> futures;
  futures.reserve(num_threads);
  int chunk_size = n / num_threads;
  int remainder = n % num_threads;
  for (int thread_idx = 0; thread_idx < num_threads; ++thread_idx) {
    int start = thread_idx * chunk_size;
    int end = start + chunk_size;
    if (thread_idx == num_threads - 1) end += remainder;
    futures.push_back(Enqueue([func, start, end]() {
      if (start < end) func(start, end);
    }));
  }
  for (auto& future : futures) future.get();
}
```

Потоки получают непрерывные блоки строк матрицы (разбиение по строкам), что оптимально для row-major формата хранения.
Цикл `for (auto& future : futures) future.get()` выполняет роль join -
вызывается после запуска всех задач, обеспечивая реальный параллелизм.

## 5. Детали реализации

**Файлы:** stl/include/ops_stl.hpp, stl/src/ops_stl.cpp

**Функции**:

- `ComputeDotProduct` — скалярное произведение через ParallelReduce.
- `ComputeMatrixVectorProduct` — матрично-векторное умножение через ParallelFor.
- `UpdateSolution`, `UpdateResidual`, `UpdateSearchDirection` — поэлементные обновления через ParallelFor.

**Карта поток -> диапазон:** при n = 5000 и 4 потоках:
поток 0 получает строки `[0, 1250)`,
поток 1 — `[1250, 2500)`,
поток 2 — `[2500, 3750)`,
поток 3 — `[3750, 5000)`.
При 8 потоках — чанки по 625 строк.

**Схема объединения результатов:** в ParallelReduce каждый поток вычисляет частичную сумму,
главный поток суммирует их через `future::get()` и сложение double — O(num_threads), пренебрежимо мало.

**Потокобезопасность:**

- Запись в разные индексы массивов (непересекающиеся диапазоны) — гонки отсутствуют.
- Очередь задач защищена `std::mutex`.
- `condition_variable` оповещает потоки о новых задачах.
- Деструктор `ThreadPool` устанавливает флаг `stop_ = true` и вызывает `join()` для всех потоков.

## 6. Проверка корректности

Корректность проверялась сравнением с SEQ для всех 12 функциональных тестов
(единичная, диагональная, трёхдиагональная, случайная SPD матрицы).
Расхождения в выходных данных не наблюдались при любом числе потоков (1, 2, 4, 8, 14).
Устойчивость проверена повторными запусками.
`ThreadSanitizer` при отдельных запусках гонок не выявил.
Крайние случаи (n = 0, ||r^(0)|| < epsilon, |(p, Ap)| < 10^(-30)) обработаны идентично SEQ.

`ThreadPool` не имеет механизма перехвата работы, поэтому дисбаланс одного потока вызывает простой остальных.

## 9. Выводы

Ручная реализация с пулом потоков конкурентоспособна (до 4.63), но уступает TBB (8.23) из-за отсутствия механизма
перехвата работы и накладных расходов на мьютекс при каждой операции с очередью.
`ThreadPool` — инженерно грамотное решение, превосходящее наивные `std::thread` с create/join на каждой итерации,
но уступающее специализированным библиотекам по эффективности на задачах с частой синхронизацией.
