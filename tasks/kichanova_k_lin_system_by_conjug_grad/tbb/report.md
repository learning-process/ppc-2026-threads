## 1. Контекст

Параллельная реализация метода сопряжённых градиентов с использованием Intel oneTBB.
В отличие от OpenMP с жёсткой моделью fork-join, TBB предлагает задачно-ориентированный подход
с рекурсивным разбиением диапазонов и перехватом работы (work-stealing), что уменьшает накладные расходы на синхронизацию.

## 2. Постановка задачи

Решение СЛАУ Ax = b с симметричной матрицей A со всеми положительными собственными значениями методом сопряжённых градиентов.
SEQ-версия (seq/report.md) служит baseline корректности и производительности.

## 3. Базовый алгоритм

Вычисляет решение СЛАУ Ax = b итерационным методом: на каждом шаге умножает матрицу на вектор,
считает скалярные произведения и обновляет приближение, пока норма невязки не станет меньше заданной точности.
Параллелизм введён с использованием `tbb::parallel_for` и `tbb::parallel_reduce` на уровне отдельных вычислительных ядер.

## 4. Схема распараллеливания

**Примитивы TBB:**

- `tbb::parallel_for` - для операций без редукции (матрично-векторное умножение, обновление векторов)
- `tbb::parallel_reduce` - для скалярных произведений и нормы невязки с автоматическим объединением частичных сумм
- `tbb::blocked_range<int>` - одномерный диапазон итераций.

**Разбиение диапазона и размер зерна (grainsize):**

- `ComputeMatrixVectorAndUpdateTBB`: grainsize = 64.
  Мелкое зерно позволяет механизму перехвата работы эффективно балансировать нагрузку.
- `ComputeDotProduct`, `ComputeResidualNorm`: grainsize = 1024.
  Крупное зерно для лёгких операций уменьшает накладные расходы на создание и объединение частичных сумм.
- `InitializeVectors`, `UpdateSolutionAndResidual`, `UpdateDirection`: grainsize = 1024 (лёгкие поэлементные операции).

**Разбиение (partitioner):** auto_partitioner (по умолчанию).
Автоматически выбирает между simple_partitioner и affinity_partitioner в зависимости от размера зерна и числа потоков.

**Контроль параллелизма**: число рабочих потоков управляется `tbb::global_control::max_allowed_parallelism`,
устанавливаемым инфраструктурой курса через PPC_NUM_THREADS.

Ключевое отличие от OMP - слияние операций, матрично-векторное умножение и обновление решения выполняются в одном parallel_for:

```cpp
void ComputeMatrixVectorAndUpdateTBB(const std::vector<double>& a, const std::vector<double>& v, std::vector<double>& result, int n, double alpha, std::vector<double>& x, const std::vector<double>& p) {
  const auto stride = static_cast<size_t>(n);
  tbb::parallel_for(tbb::blocked_range<int>(0, n, 64),
    [&](const tbb::blocked_range<int>& range) {
      for (int i = range.begin(); i < range.end(); ++i) {
        double sum = 0.0;
        const double* a_row = &a[static_cast<size_t>(i) * stride];
        for (int j = 0; j < n; ++j) sum += a_row[j] * v[j];
        result[i] = sum;
        x[i] += alpha * p[i];
      }
    });
}
```

Это уменьшает число точек синхронизации и повторных чтений данных из памяти.
Аналогично слиты обновления x и r в `UpdateSolutionAndResidual`.

## 5. Детали реализации

**Файлы:** tbb/include/ops_tbb.hpp, tbb/src/ops_tbb.cpp

**Функции:**

- `ComputeMatrixVectorAndUpdateTBB` - матрично-векторное умножение + обновление x.
- `ComputeResidualNorm` - ||r||_2 через parallel_reduce.
- `ComputeDotProduct` - скалярное произведение через parallel_reduce.
- `InitializeVectors` - параллельная инициализация r, p.
- `UpdateSolutionAndResidual` - слитное обновление x и r.
- `UpdateDirection` - обновление направления p.

**Локальные аккумуляторы:** parallel_reduce создаёт приватную local_sum для каждого потока,
объединение через `std::plus<double>()` без явных блокировок.

**Память:** все данные разделяемые.
Запись - в непересекающиеся индексы, ложное разделение кэша (false sharing) не критично при grainsize >= 64.

## 6. Проверка корректности

Корректность проверялась сравнением с SEQ для всех 12 функциональных тестов
(единичная, диагональная, трёхдиагональная, случайная SPD матрицы).
Расхождения в выходных данных не наблюдались при любом числе потоков (1, 2, 4, 8, 14).
Для случайной матрицы финальная невязка всегда меньше epsilon * sqrt(n). Устойчивость проверена повторными запусками.

## 8. Результаты

| Mode        | Count | Time_pipeline, s | Time_task, s |Speedup | Efficiency |

TBB демонстрирует наилучшее масштабирование - монотонный рост от 1.97x до 8.23x.
Слияние операций в одном parallel_for сокращает число барьеров, а механизм перехвата работы эффективно балансирует нагрузку.
Эффективность выше 100% на 1-4 потоках объясняется кэш-эффектами.
Даже при 14 потоках сохраняется 59% эффективности.

## 9. Выводы

TBB-версия превосходит OMP благодаря слиянию операций и механизму перехвата работы.
При большом числе потоков накладные расходы TBB минимальны (8.23x ускорение).
TBB рекомендуется для вычислительно насыщенных задач с частыми барьерами синхронизации,
где накладные расходы OpenMP становятся критичными.
