## 1. Контекст

Гибридная параллельная реализация метода сопряжённых градиентов, использующая одновременно OpenMP и Intel oneTBB.
OpenMP применяется для крупноблочного разделения работы между потоками на верхнем уровне,
а TBB — для тонкой балансировки внутри каждого блока через механизм перехвата работы (work-stealing).
Дополнительно применена ручная векторизация матрично-векторного умножения (развёртка цикла на 8)
и слияние операций обновления решения и невязки.

## 2. Постановка задачи

Решение СЛАУ Ax = b с симметричной матрицей A со всеми положительными собственными значениями методом сопряжённых градиентов.
SEQ-версия (seq/report.md) служит baseline корректности и производительности.

## 3. Базовый алгоритм

Вычисляет решение СЛАУ Ax = b итерационным методом:
на каждом шаге умножает матрицу на вектор, считает скалярные произведения и обновляет приближение,
пока норма невязки не станет меньше заданной точности.
Параллелизм введён на двух уровнях: OpenMP-потоки делят крупные блоки данных,
TBB обрабатывает поддиапазоны внутри каждого блока.

## 4. Межпроцессная схема

Межпроцессное взаимодействие в данной реализации отсутствует.
Все вычисления выполняются в рамках одного процесса с использованием общей памяти. MPI-вызовы не используются.

## 5. Внутрипроцессная схема

**Двухуровневый параллелизм (OpenMP + TBB):**

1. Верхний уровень (OpenMP): `#pragma omp parallel for` с `num_threads` потоками.
Диапазон `[0, n)` делится на `num_threads` крупных блоков.
Используется `schedule(static)` для равномерного распределения,
кроме матрично-векторного умножения, где `schedule(dynamic, 1)` для балансировки.
2. Нижний уровень (TBB): внутри каждого блока `tbb::parallel_for` или `tbb::parallel_reduce`
обрабатывает поддиапазоны с настраиваемым размером зерна (grainsize).
Это позволяет work-stealing планировщику TBB перераспределять работу между потоками при дисбалансе.

**Примитивы TBB:**

- `tbb::parallel_for` — для матрично-векторного умножения, обновления векторов, инициализации.
- `tbb::parallel_reduce` — для скалярного произведения с локальными аккумуляторами.

**Размер зерна (grainsize):**

- `MatrixVectorProductHybrid`: grainsize = 32 (мелкое зерно, так как операция вычислительно тяжёлая — O(n) на строку).
- `DotProductHybrid`: grainsize = 256 (среднее зерно, лёгкая операция).
- `UpdateSolutionAndResidualHybrid`, `UpdateDirectionHybrid`: grainsize = 512
  (крупное зерно, очень лёгкие поэлементные операции).
- `InitializeVectorsHybrid`: grainsize = 1024 (крупное зерно, однократная инициализация).

**Слияние операций:** обновление решения (`x[i] += alpha * p[i]`) и обновление невязки (`r[i] -= alpha * ap[i]`)
выполняются в одном цикле `UpdateSolutionAndResidualHybrid`,
что сокращает число проходов по данным и улучшает локальность кэша.

**Ручная векторизация:** в `MatrixVectorProductHybrid` внутренний цикл
развёрнут на 8 итераций для генерации SIMD-инструкций компилятором.

**Синхронизация:**

- Неявный барьер OpenMP в конце каждого `#pragma omp parallel for`.
- `tbb::parallel_reduce` автоматически объединяет частичные суммы.
- Запись в разные индексы массивов (непересекающиеся блоки) не требует дополнительной синхронизации.

**Ключевой фрагмент (матрично-векторное умножение):**

```cpp
void MatrixVectorProductHybrid(const std::vector<double>& a, const std::vector<double>& v, std::vector<double>& result, int n) {
  if (n <= 0) return;
  const auto stride = static_cast<size_t>(n);
  const double* v_ptr = v.data();
  const int num_threads = omp_get_max_threads();

  #pragma omp parallel for schedule(dynamic, 1)
  for (int t = 0; t < num_threads; ++t) {
    int start = t * (n / num_threads);
    int end = (t == num_threads - 1) ? n : (t + 1) * (n / num_threads);
    if (start >= end) continue;

    tbb::parallel_for(tbb::blocked_range<int>(start, end, 32),
      [&](const tbb::blocked_range<int>& range) {
        for (int i = range.begin(); i < range.end(); ++i) {
          const double* row = &a[i * stride];
          double sum = 0.0;
          int j = 0;
          for (; j <= n - 8; j += 8) {
            sum += row[j] * v_ptr[j];
            sum += row[j+1] * v_ptr[j+1];
            sum += row[j+2] * v_ptr[j+2];
            sum += row[j+3] * v_ptr[j+3];
            sum += row[j+4] * v_ptr[j+4];
            sum += row[j+5] * v_ptr[j+5];
            sum += row[j+6] * v_ptr[j+6];
            sum += row[j+7] * v_ptr[j+7];
          }
          for (; j < n; ++j) sum += row[j] * v_ptr[j];
          result[i] = sum;
        }
      });
  }
}
```

OpenMP-потоки получают крупные блоки строк.
Внутри блока TBB разбивает диапазон на чанки по 32 строки и балансирует их между потоками OpenMP через work-stealing.
Развёртка цикла на 8 позволяет компилятору использовать SIMD-инструкции.

**Ключевой фрагмент (скалярное произведение):**

```cpp
double DotProductHybrid(const std::vector<double>& a, const std::vector<double>& b, int n) {
  if (n <= 0) return 0.0;
  double result = 0.0;
  const int num_threads = omp_get_max_threads();

  #pragma omp parallel for reduction(+: result) schedule(static)
  for (int t = 0; t < num_threads; ++t) {
    int start = t * (n / num_threads);
    int end = (t == num_threads - 1) ? n : (t + 1) * (n / num_threads);
    if (start >= end) continue;

    double local_sum = tbb::parallel_reduce(
      tbb::blocked_range<int>(start, end, 256), 0.0,
      [&](const tbb::blocked_range<int>& range, double sum) {
        for (int i = range.begin(); i < range.end(); ++i) {
          sum += a[i] * b[i];
        }
        return sum;
      },
      [](double x, double y) { return x + y; });
    result += local_sum;
  }
  return result;
}
```

OpenMP делит диапазон на крупные блоки, каждый поток вычисляет свою частичную сумму
через `tbb::parallel_reduce` с локальными аккумуляторами.
Результаты суммируются через `reduction(+: result)` OpenMP.

## 6. Детали реализации

**Файлы:** all/include/ops_all.hpp, all/src/ops_all.cpp
**Функции:**

- `DotProductHybrid` — скалярное произведение (OpenMP + TBB `parallel_reduce`).
- `MatrixVectorProductHybrid` — матрично-векторное умножение с развёрткой цикла (OpenMP + TBB `parallel_for`).
- `UpdateSolutionAndResidualHybrid` — слитное обновление x и r (OpenMP + TBB `parallel_for`).
- `UpdateDirectionHybrid` — обновление направления p (OpenMP + TBB `parallel_for`).
- `InitializeVectorsHybrid` — инициализация r и p (OpenMP + TBB `parallel_for`).

**Узкие места:**

- Двойной параллелизм (OpenMP + TBB) создаёт дополнительные накладные расходы на вложенные параллельные регионы.
Однако однократное создание потоков OpenMP и переиспользование worker-потоков TBB минимизирует этот эффект.

Гибридная версия (OpenMP + TBB) показывает монотонный рост ускорения от 4.42 (1 поток) до 7.36 (14 потоков).
Двухуровневая схема (OpenMP для крупных блоков, TBB для тонкой балансировки)
обеспечивает лучшее масштабирование по сравнению с чистым OMP (4.34).
Слияние операций (x и r в одном проходе) и ручная векторизация дают дополнительный прирост.
При 14 потоках эффективность 53% — выше, чем у OMP (31%), но ниже, чем у чистого TBB (59%).
Накладные расходы на вложенный параллелизм частично компенсируются улучшенной локальностью данных.
Гибридная схема оправдана для задач, где нужно совместить крупноблочное разделение с тонкой балансировкой.

## 10. Выводы

Гибридная ALL-версия (OpenMP + TBB) достигает наилучшего баланса между абсолютным временем (0.0807 с)
и масштабируемостью (7.36, 53% эффективности на 14 потоках).
Двухуровневый параллелизм эффективнее чистого OpenMP, но уступает чистому TBB по эффективности на большом числе потоков
из-за накладных расходов на вложенные параллельные регионы.
Версия рекомендуется для задач среднего размера, где важна и производительность на малом числе потоков, и масштабируемость.
