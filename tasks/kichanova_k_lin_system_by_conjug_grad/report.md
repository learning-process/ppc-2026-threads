## 1. Введение

Данный сводный отчёт обобщает результаты сравнения пяти реализаций метода сопряжённых градиентов для решения
СЛАУ Ax = b с симметричной положительно определённой матрицей A.
Метод сопряжённых градиентов выбран как показательная задача для сравнения моделей параллелизма:
содержит естественный параллелизм (независимые строки в матрично-векторном умножении),
но требует частой синхронизации (скалярные произведения, обновления векторов) на каждой итерации.

## 2. Единая постановка задачи

**Вход:** матрица A размера n x n, вектор правой части b размера n, размерность n, точность epsilon.

**Выход:** вектор x размера n — приближённое решение.

**Критерий корректности:** ||b - Ax||_2 < epsilon * sqrt(n).

**Ограничения:** n > 0, матрица A строго симметрично положительно определённая, размерности согласованы.

## 3. Единая методика эксперимента

**Окружение:**

- CPU: Intel Core Ultra 5 225U (14 ядер/14 потоков, Meteor Lake)
- RAM: 32.0 GiB DDR5
- OS: Ubuntu 24.04.3 LTS, Linux 6.17.0-22-generic
- Компилятор: GCC 14.2.0
- CMake: Release-сборка (-D CMAKE_BUILD_TYPE=Release)

**Переменные окружения:** PPC_NUM_THREADS (1, 2, 4, 8, 14).

**Данные:** n = 5000, трёхдиагональная матрица А
(главная диагональ = 4.0, поддиагонали = -1.0), b = (1, 1, ..., 1)^T, epsilon = 10^(-8).

**Расчёт ускорения:** S(p) = t_seq / t(p), где t_seq — среднее время SEQ по пяти запускам.

**Расчёт эффективности:** E(p) = S(p) / p * 100%.

**Количество запусков:** 10 запусков с разным числом потоков (1, 2, 4, 8, 14).
Каждый запуск содержит оба режима (pipeline и task_run) для всех backend-ов.

## 6. Интерпретация различий

**SEQ** - эталонный baseline.
Среднее время 0.5931 с (task_run).
Основное время (около 95%) тратится на матрично-векторное умножение (двойной цикл 5000x5000, 25 млн операций на итерацию).
Число итераций для данной задачи: около 270-300. Эта версия используется как знаменатель для расчёта всех ускорений.

**OMP** - ускорение 4.14 уже на 1 потоке за счёт эффектов кэш-памяти (сверхлинейное ускорение).
Однако модель fork-join с неявными барьерами ограничивает масштабирование:
ускорение практически не растёт с числом потоков (4.33 на 14 потоках).
Причина: между матрично-векторным умножением (O(n^2)) и лёгкими операциями (O(n)) все потоки синхронизируются,
создавая простои при малейшем дисбалансе.
Эффективность падает с 414% до 31%.
Сильная сторона: простота реализации директивами.
Слабая: накладные расходы на частые барьеры.

**TBB** - лидер по масштабированию: монотонный рост от 1.96 до 8.27.
Ключевые факторы: слияние операций (матрично-векторное умножение + обновление `x`)
в одном `parallel_for` сокращает число барьеров;
механизм перехвата работы (work-stealing) эффективно балансирует нагрузку;
`parallel_reduce` с локальными аккумуляторами уменьшает накладные расходы по сравнению с глобальной редукцией OpenMP.
При 14 потоках сохраняется 59% эффективности - лучший показатель среди всех backend-ов.
Сильная сторона: задачно-ориентированная модель с малыми накладными расходами.
Слабая сторона: требует внешней библиотеки.

**STL** - конкурентоспособен (до 4.63 на 8 потоках), но нестабилен: провал до 2.53 на 4 потоках.
Причина: при n=5000 и 4 потоках остаток от деления (5000 mod 4 = 2) создаёт дисбаланс,
два потока получают на одну строку больше, и остальные простаивают в `future::get()`.
`ThreadPool` переиспользует потоки, но не имеет механизма перехвата работы.
На 8 потоках балансировка улучшается (5000/8=625, без остатка).
При 14 потоках накладные расходы на очередь задач начинают доминировать.
Сильная сторона: полный контроль без внешних зависимостей. Слабая сторона: ручное управление балансировкой.

**ALL (гибрид OpenMP + TBB)** - лучший баланс между абсолютным временем и масштабируемостью.
Двухуровневый параллелизм: OpenMP делит крупные блоки между потоками (`schedule(static)/dynamic`),
TBB обрабатывает поддиапазоны внутри каждого блока с work-stealing.
Слияние операций (`x` и `r` обновляются в одном проходе) и ручная векторизация
(развёртка цикла на 8) дают дополнительный прирост.
Ускорение растёт от 4.41 (1 поток) до 7.35 (14 потоков).
Эффективность 53% на 14 потоках - выше, чем у OMP (31%), но ниже, чем у TBB (59%).
Накладные расходы на вложенный параллелизм (OpenMP создаёт параллельные регионы,
внутри которых TBB управляет своими worker-потоками) частично компенсируются улучшенной локальностью данных.
Гибридная схема оправдана для задач, где нужно совместить крупноблочное разделение OpenMP с тонкой балансировкой TBB.

**Сравнение:**

- TBB лучше для систем с большим числом ядер (8.27x, эффективность 59%).
- ALL лучше для получения минимального абсолютного времени (0.0807 с, 7.35x).
- OMP проще всего в реализации, но не масштабируется.
- STL не требует внешних библиотек, но уступает в эффективности.

## 8. Заключение

**Лучший выбор технологии зависит от приоритетов:**

- Минимальное время выполнения: ALL (0.0807 с, 7.35) - гибрид OpenMP+TBB с векторизацией.
- Лучшее масштабирование: TBB (8.27 на 14 потоках, 59% эффективности),
  для систем с большим числом ядер или больших размеров задачи.
- Простота разработки: OMP (4.33) - минимальные изменения кода относительно SEQ.
- Независимость от внешних библиотек: STL (4.63) - только стандартная библиотека C++.

**Ограничения сравнения:**

- Использовался один тип матрицы (трёхдиагональная симметричная положительно определённая матрица).
  Для плотных матриц относительная производительность может измениться.
- Измерения на одном экземпляре CPU. Для статистической значимости требуется больше запусков.

**Возможные улучшения:**

- OMP: внедрение nowait для перекрытия вычислений.
- TBB: эксперименты с grainsize и affinity_partitioner.
- STL: реализация очереди с перехватом работы.
