## 1. Контекст

Параллельная реализация метода сопряжённых градиентов с использованием OpenMP.
Цель — ускорить вычисления относительно последовательной версии за счёт распараллеливания матрично-векторного умножения,
скалярных произведений и операций вида y = alpha * x + y.

## 2. Постановка задачи

Решение СЛАУ Ax = b с симметричной матрицей A со всеми положительными собственными значениями методом сопряжённых градиентов.
SEQ-версия (seq/report.md) служит baseline корректности и производительности.

## 3. Базовый алгоритм

Вычисляет решение СЛАУ Ax = b итерационным методом:
на каждом шаге умножает матрицу на вектор, считает скалярные произведения и обновляет приближение,
пока норма невязки не станет меньше заданной точности.
Параллелизм введён на уровне пяти вычислительных ядер:
`ComputeDotProductOMP`, `ComputeMatrixVectorProductOMP`, `UpdateSolutionOMP`, `UpdateResidualOMP`, `UpdateSearchDirectionOMP`.

## 4. Схема распараллеливания

**Параллельные регионы:** каждая из пяти функций (`ComputeDotProductOMP`, `ComputeMatrixVectorProductOMP`,
`UpdateSolutionOMP`, `UpdateResidualOMP`, `UpdateSearchDirectionOMP`) содержит директиву `#pragma omp parallel for`.

**Область видимости переменных:**

- shared: входные векторы a, b (или v), размер n, параметры alpha, beta, шаг stride.
- private: счётчики циклов i, j, локальная сумма sum.
- reduction(+: result): в `ComputeDotProductOMP` для атомарного накопления скалярного произведения без гонки данных.
- default(none): во всех директивах — требует явного указания всех переменных, страхует от пропуска shared/private.

**Выбран schedule(static):** равномерное распределение итераций между потоками.
Выбран потому, что матрично-векторное умножение имеет сбалансированную нагрузку:
каждая строка требует O(n) операций, все потоки получают поровну работы.

**Синхронизация:**

- Неявный барьер в конце каждого `#pragma omp parallel for` гарантирует
  завершение операции всеми потоками перед переходом к следующему шагу.
- reduction(+: result) создаёт приватную копию result для каждого потока и атомарно суммирует их в конце,
  гонки на аккумуляторе исключены.
- Итерации внутри parallel for независимы (разные индексы массивов), дополнительная синхронизация не требуется.

**Ключевой фрагмент (матрично-векторное умножение):**

```cpp
void ComputeMatrixVectorProductOMP(const std::vector<double>& a, const std::vector<double>& v, std::vector<double>& result, int n) {
  const auto stride = static_cast<size_t>(n);
  #pragma omp parallel for default(none) shared(a, v, result, n, stride) schedule(static)
  for (int i = 0; i < n; ++i) {
    double sum = 0.0;
    const double* a_row = &a[static_cast<size_t>(i) * stride];
    for (int j = 0; j < n; ++j) {
      sum += a_row[j] * v[j];
    }
    result[i] = sum;
  }
}
```

default(none) заставляет явно перечислить все переменные — `shared(a, v, result, n, stride)`.
Локальная sum и счётчики i, j автоматически приватны. schedule(static) делит итерации поровну.
Барьер в конце директивы синхронизирует потоки перед следующим шагом алгоритма (скалярное произведение).

## 5. Детали реализации

Файлы: omp/include/ops_omp.hpp, omp/src/ops_omp.cpp

**Изменения относительно SEQ:**

- Все пять вычислительных функций обёрнуты в #pragma omp parallel for
- Инициализация векторов r и p распараллелена:

  ```cpp
  #pragma omp parallel for default(none) shared(r, b, p, n) schedule(static)
  for (int i = 0; i < n; i++) {
    r[i] = b[i];
    p[i] = r[i];
  }
  ```

- Все разделяемые данные — read-only (a, b, alpha, beta)
  либо доступны по непересекающимся индексам (`result[i]`, `x[i]`, `r[i]`, `p[i]`)
  
**Устранение рисков гонок:**

- Гонка на sum в матрично-векторном умножении исключена локальностью переменной
- Гонка на аккумуляторе скалярного произведения исключена reduction(+: result)
- Запись в разные индексы массивов не требует синхронизации

## 6. Проверка корректности

Корректность проверялась сравнением с SEQ для всех 12 функциональных тестов

Расхождения в выходных данных не наблюдались при любом числе потоков (1, 2, 4, 8, 14).

| Mode        | Count | Time_pipeline, s | Time_task, s |Speedup | Efficiency |

OpenMP-версия достигает максимального ускорения (~4.5x) уже на 2 потоках за счёт эффектов кэш-памяти (сверхлинейное ускорение).
Дальнейший рост числа потоков не улучшает время из-за неявных барьеров в конце каждого параллельного региона.
Между матрично-векторным умножением (O(n^2)) и лёгкими операциями (O(n))
все потоки синхронизируются, что создаёт простои при дисбалансе.
Эффективность падает с 413% до 33% при переходе на 14 потоков.

## 9. Выводы

OpenMP даёт значительный выигрыш (4.3x) уже на одном потоке благодаря улучшенной локальности данных.
Однако модель fork-join с неявными барьерами ограничивает масштабируемость:
увеличение числа потоков свыше 4 не даёт практически значимого прироста.
Выигрыш ограничен накладными расходами на синхронизацию между вычислительными ядрами.
