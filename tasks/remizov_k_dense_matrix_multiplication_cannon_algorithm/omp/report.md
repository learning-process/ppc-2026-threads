## 1. Введение

Умножение плотных матриц — ключевая операция в вычислительной математике, требующая значительных
ресурсов при больших размерах. Алгоритм Кэннона изначально спроектирован для параллельных систем, но
его блочная структура позволяет эффективно применять различные технологии параллелизма на общей
памяти. OpenMP (Open Multi-Processing) — стандарт параллельного программирования для многопоточных
систем с разделяемой памятью, использующий директивы компилятора. В данной работе реализована
параллельная версия алгоритма Кэннона с использованием OpenMP для ускорения вычислений на
многоядерном процессоре.

## 2. Постановка задачи

**Входные данные**:

- `block_dim` — размер блока (положительное целое);
- `matrix_a`, `matrix_b` — квадратные плотные матрицы размера \(N \times N\) с элементами `double`.

**Условия корректности**:

- \(N\) делится на `block_dim` без остатка;
- матрицы квадратные, непустые.

**Выходные данные**:

- `matrix_c` — результат умножения `matrix_a * matrix_b`.

## 3. Последовательный алгоритм (основа)

Базовый алгоритм описан в отдельном отчёте (SEQ). Он включает:

- Разбиение на блоки размера `block_dim × block_dim`;
- Начальный циклический сдвиг блоков;
- Основной цикл из `blocks = N / block_dim` итераций:
  - умножение соответствующих пар блоков;
  - циклический сдвиг строк `A` влево и столбцов `B` вверх;
- Сборку результата из блоков.

## 4. Параллельная схема (OpenMP)

В OpenMP-реализации распараллелены наиболее вычислительно ёмкие этапы:

1. **Инициализация блоков (`InitializeBlocks`)**  
   Двумерные циклы по `i` и `j` (индексы блоков) охвачены директивой
   `#pragma omp parallel for collapse(2) schedule(static)`. Это позволяет параллельно заполнять все
   блоки `A` и `B` с учётом начального сдвига. Внутренние циклы по `bi`, `bj` остаются
   последовательными, так как их размер (`block_dim`) может быть невелик.

2. **Основной цикл (`RunCannonCycle`)**  
   На каждом шаге `step` умножение блоков распараллеливается:

   ```cpp
   #pragma omp parallel for collapse(2) schedule(dynamic) \
       default(none) shared(...)
   for (int i = 0; i < block_count; ++i)
     for (int j = 0; j < block_count; ++j)
       MultiplyBlock(a_blocks[i][j], b_blocks[i][j], c_blocks[i][j], block_size);
   ```

   - `collapse(2)` объединяет два цикла в один параллельный, создавая
     `block_count × block_count` итераций.
   - `schedule(dynamic)` выбрана для балансировки загрузки, так как время умножения блоков может
     варьироваться из-за состояния кэша.
   - Само умножение блока (`MultiplyBlock`) выполняется последовательно тройным циклом.

3. **Сборка результата (`AssembleOutput`)**  
   Аналогично инициализации, двумерные циклы по блокам параллелятся директивой
   `#pragma omp parallel for collapse(2) schedule(static)`.

4. **Сдвиги блоков (`ShiftBlocksLeft`, `ShiftBlocksUp`)**  
   В данной реализации сдвиги оставлены последовательными, так как их вычислительная сложность
   невелика по сравнению с умножением блоков. При необходимости их также можно распараллелить, но
   это дало бы незначительный выигрыш.

### 5.2 Ключевые особенности

- Для компиляции с OpenMP в CMake добавляется флаг `-fopenmp`.
- Использование `#pragma omp parallel for` с явным указанием `shared` и `default(none)` для
  устранения предупреждений и контроля видимости переменных.
- Функция `MultiplyBlock` не содержит OpenMP-директив, так как распараллеливание по блокам уже даёт
  достаточно задач (например, при `block_count = 16` имеем 256 независимых умножений блоков). Это
  предотвращает вложенный параллелизм, который может привести к избыточным накладным расходам.

### 5.3 Управление потоками

Количество потоков определяется средой выполнения OpenMP (обычно через `OMP_NUM_THREADS`).
В экспериментах использовалось значение по умолчанию, равное 8 (логические ядра процессора
Intel i5-1155G7).

## 7. Результаты и обсуждение

### 7.1 Корректность

Все 8 функциональных тестов для OpenMP-версии успешно пройдены (см. вывод `ppc_func_tests`).
Результаты сравнения с эталонными значениями выполняются с точностью `1e-10`. Таким образом,
параллельная реализация сохраняет численную корректность.

### 7.2 Производительность

Результаты нагрузочного теста (усреднённые по pipeline и task_run):

| Технология | Время (среднее), мс | Ускорение (относительно SEQ) | Эффективность (на 8 потоках) |
|:-----------|--------------------:|-----------------------------:|-----------------------------:|
| SEQ        | 3842                | 1.00                         | N/A                          |
| OMP        | 810                 | 4.74                         | 59.3 %                       |

**Анализ**:

- Ускорение 4.74 раза на 8 логических потоках (4 физических ядра с Hyper-Threading) — отличный
  результат, близкий к теоретическому пределу для данного алгоритма. Эффективность 59% объясняется
  наличием последовательных частей (сдвиги, сборка, инициализация) и накладными расходами на создание
  параллельных областей.
- Основной вклад в ускорение даёт параллельное выполнение 4096 умножений блоков (16×16×16). Благодаря
  `collapse(2)` все 256 пар блоков на каждом шаге распределяются между потоками равномерно.
- По сравнению с последовательной версией, OpenMP-реализация задействует все ядра и существенно
  сокращает время выполнения (с 3.84 с до 0.81 с).
- Сдвиги блоков не распараллелены, но их доля в общем времени мала (менее 5%).

### 7.3 Масштабируемость

На процессоре с 4 физическими ядрами дальнейшее увеличение числа потоков (свыше 8) не даст выигрыша
из-за насыщения вычислительных ресурсов. Использование всех логических ядер уже обеспечивает
практически максимально достижимое ускорение для данной задачи.

## Приложение (пример OpenMP-директивы)

```cpp
// Параллельное умножение всех пар блоков (фрагмент RunCannonCycle)
#pragma omp parallel for collapse(2) schedule(dynamic) default(none) \
    shared(a_blocks, b_blocks, c_blocks, block_size, block_count)
for (int i = 0; i < block_count; ++i) {
  for (int j = 0; j < block_count; ++j) {
    MultiplyBlock(a_blocks[i][j], b_blocks[i][j], c_blocks[i][j], block_size);
  }
}
```cpp
