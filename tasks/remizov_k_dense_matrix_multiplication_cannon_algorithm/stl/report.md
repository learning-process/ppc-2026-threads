## 1. Введение

Умножение плотных матриц является одной из ключевых операций вычислительной математики. Для ускорения
вычислений на многопроцессорных системах с общей памятью можно использовать потоки стандартной
библиотеки C++ (`std::thread`). В отличие от высокоуровневых фреймворков (OpenMP, TBB), `std::thread`
предоставляет полный контроль над созданием и синхронизацией потоков, но требует ручного распределения
работы. В данной работе реализована версия алгоритма Кэннона, использующая `std::thread` для
параллельного выполнения вычислений, и проведён анализ её производительности.

## 2. Постановка задачи

**Входные данные**:

- `block_dim` — размер блока (целое положительное);
- `matrix_a`, `matrix_b` — квадратные плотные матрицы размера \(N \times N\) с элементами типа `double`.

**Условия корректности**:

- \(N\) делится на `block_dim` без остатка;
- матрицы квадратные и непустые.

**Выходные данные**:

- `matrix_c` — результат умножения `matrix_a * matrix_b`.

## 3. Последовательный алгоритм (база)

Базовый алгоритм подробно описан в отчёте SEQ. Ключевые этапы:

- Разбиение матриц на блоки размера `block_dim × block_dim`.
- Начальный циклический сдвиг блоков.
- Основной цикл из `blocks = N / block_dim` итераций: умножение всех пар блоков, затем циклический
  сдвиг строк блоков \(A\) влево и столбцов блоков \(B\) вверх.
- Сборка результата из блоков.

## 4. Параллельная схема (STL)

STL-реализация использует ручное управление потоками через `std::thread`. Для унификации
распараллеливания циклов созданы вспомогательные шаблонные функции `ParallelFor` (одномерный диапазон)
и `ParallelFor2D` (двумерный диапазон).

### 4.1 Функция `ParallelFor`

```cpp
template <typename IndexType, typename Func>
void ParallelFor(IndexType begin, IndexType end, const Func &func) {
  const std::size_t num_threads = std::max(1U, std::thread::hardware_concurrency());
  // Разбиение диапазона [begin, end) на равные части
  // Запуск потоков, каждый выполняет func для своей части
  // Ожидание завершения всех потоков
}
```cpp

- Количество потоков определяется через `hardware_concurrency()` (на целевой системе — 8).
- Диапазон разбивается на `num_threads` непересекающихся интервалов примерно одинаковой длины.
- Каждый поток выполняет `func` для всех элементов своего интервала.

### 4.2 Функция `ParallelFor2D`

Для двумерного случая (строки `[rows_begin, rows_end)`, столбцы `[cols_begin, cols_end)`) диапазон
линеаризуется в одномерный индекс:

```cpp
int total = (rows_end - rows_begin) * (cols_end - cols_begin);
ParallelFor(0, total, [&](int linear_idx) {
  int i = rows_begin + linear_idx / cols;
  int j = cols_begin + linear_idx % cols;
  func(i, j);
});
```cpp

Таким образом, любой двумерный цикл может быть распараллелен через `ParallelFor2D`.

### 4.3 Распараллеленные этапы

- **Инициализация блоков** (`InitializeBlocks`): используется `ParallelFor2D` по индексам блоков
  `(i, j)`. Каждый поток заполняет свою группу блоков `A` и `B` с учётом начального сдвига.
- **Основной цикл** (`RunCannonCycle`): на каждом шаге умножение всех пар блоков выполняется через
  `ParallelFor2D` по `(i, j)`. Внутри `MultiplyBlock` остаётся последовательным (тройной цикл).
- **Сдвиги блоков**:
  - `ShiftBlocksLeft` (сдвиг строк A влево) распараллелен через `ParallelFor` по строкам `i`.
  - `ShiftBlocksUp` (сдвиг столбцов B вверх) — через `ParallelFor` по столбцам `j`.
- **Сборка результата** (`AssembleOutput`): `ParallelFor2D` по блокам `(i, j)`.

### 4.4 Синхронизация

Потоки не разделяют изменяемые данные (каждый поток работает со своим набором блоков или с разными
элементами массивов), поэтому явные блокировки не требуются. Единственная точка синхронизации —
вызов `join()` для всех потоков после завершения параллельной секции.

### 5.2 Ключевые особенности кода

- Вспомогательные функции `ParallelFor` и `ParallelFor2D` определены в анонимном пространстве имён
  внутри `ops_stl.cpp`, чтобы не загрязнять глобальное пространство.
- Количество потоков вычисляется один раз в начале выполнения `ParallelFor` (но может быть
  перевычислено при каждом вызове). Для стабильности можно вычислить глобально, но в текущей
  реализации это не критично.
- Используется `std::move` при циклических сдвигах блоков для эффективного перемещения четырёхмерных
  векторов (без глубокого копирования).

### 5.3 Управление потоками

Число потоков равно `std::thread::hardware_concurrency()`, что на Intel i5-1155G7 даёт 8 логических
потоков. При необходимости количество потоков можно изменить, задав переменную окружения или
модифицировав код.

### 7.1 Корректность

STL-версия успешно прошла все 8 функциональных тестов. Сравнение результатов с эталонными значениями
выполнялось с точностью `1e-10`. Это подтверждает правильность реализации и отсутствие ошибок
синхронизации.

### 7.2 Производительность

Результаты нагрузочного теста (усреднены по pipeline и task_run):

| Технология | Время (среднее), мс | Ускорение (относительно SEQ) | Эффективность (на 8 потоках) |
|:-----------|--------------------:|-----------------------------:|-----------------------------:|
| SEQ        | 3842                | 1.00                         | N/A                          |
| STL        | 874                 | 4.40                         | 55.0 %                       |

**Анализ**:

- Ускорение 4.40 раза на 8 логических потоках — очень хороший результат, лишь немного уступающий
  OpenMP (4.74). Эффективность 55% объясняется накладными расходами на создание потоков
  (`std::thread` создаётся и уничтожается при каждом вызове `ParallelFor`, в отличие от OpenMP, где
  пул потоков переиспользуется).
- Основной выигрыш достигается за счёт параллельного выполнения 256 умножений блоков на каждом шаге
  (всего 4096 умножений). Благодаря равномерному разбиению диапазонов в `ParallelFor2D` нагрузка
  распределяется достаточно равномерно.
- Сдвиги блоков также распараллелены, что даёт небольшой дополнительный выигрыш.
- По сравнению с OpenMP, STL немного медленнее (~874 мс против ~810 мс). Причины:
  - Создание и присоединение потоков на каждом вызове `ParallelFor` (в основном цикле это 16 вызовов
    для умножений + 15 для сдвигов) создаёт дополнительные накладные расходы.
  - В OpenMP потоки создаются один раз при входе в параллельную область, что снижает задержки.
  - Линеаризация двумерных диапазонов через `ParallelFor2D` требует целочисленного деления и взятия
    остатка, что добавляет небольшие затраты.

### 7.3 Сравнение с другими технологиями

- **SEQ**: STL быстрее в 4.4 раза.
- **OMP**: STL уступает ~8% по времени, но демонстрирует схожую масштабируемость.
- **TBB**: STL значительно превосходит TBB (ускорение 4.40 против 1.04) из-за отсутствия избыточных
  накладных расходов планировщика.
- **ALL**: STL быстрее комбинированной версии (874 мс против 1384 мс).

В работе создана параллельная реализация алгоритма Кэннона с использованием стандартной библиотеки
потоков C++ (`std::thread`). Корректность подтверждена функциональными тестами. На нагрузочном тесте

даёт полный контроль над управлением потоками и не требует дополнительных библиотек.

Основной недостаток — накладные расходы на создание и уничтожение потоков при каждом параллельном
вызове. Это можно улучшить, реализовав постоянный пул потоков (например, через `std::async` или
собственный пул). Тем не менее, для большинства задач полученная производительность является вполне
приемлемой.

STL-версия показала себя как надёжная и достаточно эффективная альтернатива OpenMP, особенно в средах,
где установка дополнительных библиотек (TBB) нежелательна.

## Приложение (пример `ParallelFor2D` из кода)

```cpp
template <typename Func>
void ParallelFor2D(int rows_begin, int rows_end, int cols_begin, int cols_end, const Func &func) {
  const int rows = rows_end - rows_begin;
  const int cols = cols_end - cols_begin;
  const int total = rows * cols;
  if (total <= 0) return;

  ParallelFor(0, total, [&](int linear_idx) {
    int i = rows_begin + (linear_idx / cols);
    int j = cols_begin + (linear_idx % cols);
    func(i, j);
  });
}
```cpp
