## 1. Введение

Умножение плотных матриц — фундаментальная операция, требующая эффективной параллелизации для больших
размеров. Intel Threading Building Blocks (TBB) — библиотека C++ для параллельного программирования на
общей памяти, основанная на задачах (task-based). В отличие от OpenMP, TBB предоставляет более
высокоуровневые абстракции и автоматическое распределение нагрузки с помощью work-stealing. В данной
работе реализована версия алгоритма Кэннона с использованием TBB, проведена оценка производительности
и выявлены особенности её поведения на реальном оборудовании.

## 2. Постановка задачи

**Входные данные**:
- `block_dim` — размер блока (целое положительное);
- `matrix_a`, `matrix_b` — квадратные плотные матрицы размера \(N \times N\) с элементами `double`.
- Условия корректности: \(N\) кратно `block_dim`, матрицы не пусты и квадратны.

**Выходные данные**:
- `matrix_c` — результат умножения `matrix_a * matrix_b`.

## 3. Последовательный алгоритм (база)

Базовый алгоритм (описан в отчёте SEQ) включает:
- Разбиение на блоки размера `block_dim × block_dim`.
- Начальный циклический сдвиг блоков.
- Основной цикл из `blocks = N / block_dim` итераций: умножение блоков, затем циклический сдвиг
  строк `A` влево и столбцов `B` вверх.
- Сборку результата из блоков.

## 4. Параллельная схема (TBB)

В TBB-реализации распараллелены все основные вычислительные этапы с использованием шаблонов
`tbb::parallel_for` и `tbb::blocked_range2d`.

**4.1 Инициализация блоков (`InitializeBlocks`)**  
Двумерные циклы по индексам блоков `i`, `j` заменены на:
```cpp
tbb::parallel_for(tbb::blocked_range2d<int>(0, block_count, 0, block_count),
  [&](const tbb::blocked_range2d<int> &r) {
    for (int i = r.rows().begin(); i != r.rows().end(); ++i)
      for (int j = r.cols().begin(); j != r.cols().end(); ++j) {
        // заполнение блоков A[i][j] и B[i][j]
      }
  });
```
Каждая задача обрабатывает непрерывный прямоугольный поддиапазон блоков.

**4.2 Основной цикл (`RunCannonCycle`)**  
На каждом шаге `step` умножение всех пар блоков распараллеливается аналогичным `blocked_range2d`:
```cpp
tbb::parallel_for(tbb::blocked_range2d<int>(0, block_count, 0, block_count),
  [&](const tbb::blocked_range2d<int> &r) {
    for (int i = r.rows().begin(); i != r.rows().end(); ++i)
      for (int j = r.cols().begin(); j != r.cols().end(); ++j)
        MultiplyBlock(a_blocks[i][j], b_blocks[i][j], c_blocks[i][j], block_size);
  });
```
При этом `MultiplyBlock` выполняется последовательно (тройной цикл).

**4.3 Сдвиги блоков**  
- `ShiftBlocksLeft` (сдвиг строк блоков A влево) распараллелен по строкам:
  ```cpp
  tbb::parallel_for(0, block_count, [&](int i) {
    // циклический сдвиг строки i
  });
  ```
- `ShiftBlocksUp` (сдвиг столбцов блоков B вверх) — по столбцам:
  ```cpp
  tbb::parallel_for(0, block_count, [&](int j) {
    // циклический сдвиг столбца j
  });
  ```

**4.4 Сборка результата (`AssembleOutput`)**  
Аналогично инициализации используется `tbb::parallel_for` с `blocked_range2d`.

### 5.2 Ключевые особенности
- Для работы TBB требуется библиотека `tbb` и соответствующие флаги линковки.
- Количество потоков управляется TBB автоматически, по умолчанию используется `tbb::global_control`
  или переменная окружения `TBB_NUM_THREADS`.
- При `block_count = 16` (матрица 1024×1024, блок 64) количество задач на шаге:
  `block_count * block_count = 256`. TBB разбивает их на более мелкие подзадачи для work-stealing.
- Внутреннее умножение блоков не распараллелено, чтобы избежать чрезмерной гранулярности (каждый блок
  64×64 даёт 4096 операций умножения-сложения — достаточно для задачи, но не слишком много).

### 5.3 Управление потоками
В экспериментах использовались настройки TBB по умолчанию, что обычно приводит к задействованию всех
логических ядер (8 на Intel i5-1155G7).

## 7. Результаты и обсуждение

### 7.1 Корректность

TBB-версия успешно прошла все 8 функциональных тестов. Результаты умножения совпадают с эталонными
с точностью `1e-10`. Это подтверждает правильность реализации алгоритма с использованием TBB.

### 7.2 Производительность

Результаты нагрузочного теста (среднее по pipeline и task_run):

| Технология | Время (среднее), мс | Ускорение (относительно SEQ) | Эффективность (на 8 потоках) |
|:-----------|--------------------:|-----------------------------:|-----------------------------:|
| SEQ        | 3842                | 1.00                         | N/A                          |
| TBB        | 3704                | 1.04                         | 13.0 %                       |

**Анализ низкой производительности TBB**:

TBB показала практически такое же время, как последовательная версия (ускорение 1.04). Это
неожиданный результат, который требует объяснения. Возможные причины:

1. **Накладные расходы на планировщик задач**  
   Каждый шаг основного цикла создаёт 256 задач (умножение блоков). Планировщик TBB может разбивать
   их на множество мелких подзадач, что приводит к высоким накладным расходам на переключение
   контекста и распределение работы. Для 256 задач общее время выполнения может быть сопоставимо с
   последовательным, если вычислительная нагрузка на задачу мала (блок 64×64 — всего 4096 операций,
   что на современном CPU выполняется за микросекунды). В таких условиях накладные расходы TBB могут
   доминировать.

2. **Отсутствие вложенного параллелизма**  
   В TBB-реализации умножение каждого блока выполняется последовательно. Это сделано намеренно,
   чтобы избежать чрезмерной гранулярности, но именно эта гранулярность оказалась слишком мелкой для
   эффективной работы TBB. Для сравнения: OpenMP распараллеливает блоки через `collapse(2)`, создавая
   256 итераций, но работа планировщика OpenMP значительно легче, чем у TBB.

3. **Неэффективное использование кэша из-за четырёхмерной структуры блоков**  
   Представление блоков в виде четырёхмерного вектора приводит к непоследовательному доступу к памяти.
   TBB может усугублять проблему из-за перераспределения блоков между потоками, что увеличивает
   промахи кэша.

4. **Параметры планировщика по умолчанию**  
   TBB автоматически настраивает гранулярность, но для такого количества маленьких задач (256)
   оптимальным было бы объединение нескольких умножений блоков в одну задачу. Без ручной настройки
   `blocked_range2d` с указанием `grain_size` производительность остаётся низкой.

5. **Аппаратные особенности процессора**  
   Intel i5-1155G7 имеет 4 физических ядра с Hyper-Threading. TBB может создавать до 8 потоков, но при
   решении 256 задач на каждом шаге возникает интенсивная конкуренция за ресурсы кэша и планировщика.
   Возможно, на процессорах с большим количеством ядер TBB показала бы лучшие результаты, но в данном
   случае накладные расходы перевешивают выигрыш.

**Дополнительное замечание**:  
В отличие от OpenMP и STL, TBB не показала ускорения даже на уровне 2×. Это указывает на то, что
текущая реализация не подходит для данного паттерна вычислений без дополнительной настройки.

## 8. Заключение

В работе создана параллельная реализация алгоритма Кэннона с использованием Intel TBB. Корректность
подтверждена функциональными тестами. Однако производительность оказалась крайне низкой: ускорение
составило всего 4% по сравнению с последовательной версией. Основные причины — высокие накладные
расходы планировщика TBB при большом количестве мелких задач (умножение блоков 64×64) и отсутствие
настройки гранулярности.

Для улучшения производительности TBB-версии можно предложить следующие меры:
- Увеличить размер блока (например, 128×128), чтобы каждая задача выполняла больше работы.
- Вручную указать `grain_size` в `blocked_range2d` для группировки нескольких блоков в одну задачу.
- Распараллелить внутреннее умножение блоков (например, с использованием TBB-алгоритмов или
  вложенных `parallel_for`).
- Перейти от четырёхмерного вектора к плоскому массиву для лучшей локальности данных.

Таким образом, TBB не является оптимальным выбором для данного алгоритма без дополнительной адаптации,
в отличие от OpenMP, показавшего высокую эффективность.
