## 1. Введение

Умножение плотных матриц — базовая операция линейной алгебры, широко используемая в научных расчётах, машинном обучении, обработке сигналов и многих других областях. Классический алгоритм умножения имеет кубическую сложность, а при работе с матрицами больших размеров возникает необходимость в эффективной параллельной реализации. Алгоритм Кэннона (Cannon's algorithm) изначально разработан для распределённых вычислительных систем с топологией решётки, но его блочная структура позволяет успешно применять его и в многопоточных средах.

Цель данной работы — реализовать умножение квадратных плотных матриц по схеме Кэннона с использованием различных средств параллелизма (последовательная версия, OpenMP, Intel TBB, стандартная библиотека потоков C++ и комбинированная реализация), провести функциональное тестирование и оценить производительность полученных решений на реальном оборудовании.

## 2. Постановка задачи

Входные данные:
- `block_dim` — размер блока (положительное целое число);
- `matrix_a`, `matrix_b` — квадратные матрицы размера `N × N` (заданы как `vector<vector<double>>`).

Условия корректности:
- `matrix_a` и `matrix_b` не пусты;
- `matrix_a` и `matrix_b` квадратные, имеют одинаковый размер `N`;
- `N` делится на `block_dim` без остатка.

Выходные данные:
- `matrix_c` — результат умножения `matrix_a * matrix_b` (также матрица `N × N`).

Требуется реализовать алгоритм Кэннона с блочным разбиением и обеспечить его корректную работу для всех поддерживаемых технологий параллелизации.

## 3. Базовый (последовательный) алгоритм

В основе всех реализаций лежит блочный алгоритм Кэннона, который в последовательном исполнении не даёт выигрыша в скорости, но сохраняет ту же структуру данных, что и параллельные версии, что упрощает сравнение.

**Шаги алгоритма:**

1. **Разбиение** — исходные матрицы `A` и `B` делятся на блоки размера `block_dim × block_dim`. Количество блоков по каждому измерению: `blocks = N / block_dim`.
2. **Начальный сдвиг** — каждый блок `A[i][j]` циклически сдвигается влево на `i` позиций (внутри строки), а каждый блок `B[i][j]` циклически сдвигается вверх на `j` позиций (внутри столбца). Фактически для блока `(i,j)` в матрице `A` выбирается исходный блок с индексами `(i, (i+j) mod blocks)`, а для `B` — `((i+j) mod blocks, j)`.
3. **Основной цикл** — повторяется `blocks` раз:
   - Для каждой пары блоков `(i,j)` выполняется умножение блоков: `C[i][j] += A[i][j] * B[i][j]`.
   - После каждого шага, кроме последнего, все строки блоков `A` сдвигаются влево на одну позицию, а все столбцы блоков `B` — вверх на одну позицию (циклически).
4. **Сборка** — из блоков-результатов `C[i][j]` формируется итоговая матрица.

Умножение двух блоков размера `block_dim × block_dim` выполняется стандартным тройным вложенным циклом сложностью `O(block_dim^3)`.

## 4. Схема параллелизации

В данной работе реализованы пять вариантов параллельного алгоритма. Все они используют декомпозицию по блокам, однако отличаются способом координации потоков.

### 4.1 Последовательная версия (SEQ)
Полностью повторяет базовый алгоритм без какой-либо параллелизации. Используется как эталон для расчёта ускорения.

### 4.2 OpenMP (OMP)
- **Умножение блоков**: директива `#pragma omp parallel for collapse(2)` охватывает циклы по `i` и `j` для всех пар блоков. Каждое умножение блока выполняется последовательно.
- **Инициализация блоков и сборка результата**: аналогично используются `#pragma omp parallel for collapse(2)`.
- **Сдвиги блоков** остались последовательными (не распараллелены, так как их доля в общем времени мала).

### 4.3 Intel TBB (TBB)
- **Умножение блоков**: `tbb::parallel_for` с `tbb::blocked_range2d<int>` распределяет пары `(i,j)` по потокам.
- **Сдвиги**: `tbb::parallel_for` по строкам (для сдвига влево) и по столбцам (для сдвига вверх).
- **Инициализация и сборка**: также используют `tbb::parallel_for` с двумерным диапазоном.
- Внутри каждого блока умножение остаётся последовательным.

### 4.4 STL (std::thread)
- Реализована вспомогательная функция `ParallelFor`, которая разбивает диапазон итераций на равные части и запускает `std::thread` для каждой части. Количество потоков определяется как `hardware_concurrency()`.
- Аналогичная функция `ParallelFor2D` линеаризует двумерный диапазон и вызывает `ParallelFor`.
- **Умножение блоков** выполняется через `ParallelFor2D` (параллельны только вызовы умножения блоков, само умножение блока последовательное).
- **Сдвиги**: `ParallelFor` по строкам / столбцам.
- **Инициализация и сборка**: `ParallelFor2D`.

### 4.5 Комбинированная (ALL)
Содержит элементы всех предыдущих подходов (OpenMP, TBB, STL). Конкретно:
- `MultiplyBlock` использует OpenMP для распараллеливания внутренних циклов самого блока.
- Сдвиги реализованы через `std::thread`.
- Цикл Кэннона, инициализация и сборка используют `tbb::parallel_for`.

Такой гибрид призван объединить сильные стороны каждой технологии, но может приводить к избыточным накладным расходам.

## 5. Детали реализации

### 5.1 Структура проекта
Проект организован по технологии ‹‹конвейер›› (pipeline) и содержит следующие каталоги:

```
remizov_k_dense_matrix_multiplication_cannon_algorithm/
├── common/               – общие типы и определения
├── seq/                  – последовательная версия
├── omp/                  – версия с OpenMP
├── tbb/                  – версия с Intel TBB
├── stl/                  – версия на std::thread
├── all/                  – комбинированная версия
└── tests/                – функциональные и нагрузочные тесты
```

Каждая реализация предоставляет свой класс, унаследованный от `BaseTask` (см. `common.hpp` и `task.hpp`).

### 5.2 Основные методы класса
- `ValidationImpl` — проверяет корректность входных данных.
- `PreProcessingImpl` — очищает выходной вектор.
- `RunImpl` — выполняет умножение (блочное разбиение, инициализация, цикл Кэннона, сборка).
- `PostProcessingImpl` — всегда возвращает `true`.

### 5.3 Вспомогательные статические методы
- `MultiplyBlock` — умножение двух блоков (в последовательной и TBB/STL версиях — тройной вложенный цикл, в OMP и ALL — с OpenMP-прагмами).
- `InitializeBlocks` — заполнение блоков A и B с учётом начального сдвига (также распараллелено).
- `RunCannonCycle` — основной цикл алгоритма.
- `ShiftBlocksLeft` / `ShiftBlocksUp` — циклические сдвиги строк/столбцов блоков.
- `AssembleOutput` — сборка итоговой матрицы из блоков C.

### 5.4 Особенности работы с памятью
Для хранения блоков используется четырёхмерный вектор:
```cpp
std::vector<std::vector<std::vector<std::vector<double>>>>
```
Такая структура удобна для индексации `blocks[i][j][bi][bj]`, но приводит к фрагментации памяти и потенциальным кэш-промахам. Однако для целей сравнения реализаций это приемлемо, так как все версии используют одинаковое представление.

Размер блоков фиксирован и задаётся пользователем. По условию `N` всегда кратно `block_dim`.

### 7.2 Производительность

Ниже представлены результаты нагрузочного тестирования для матрицы 1024×1024. Время указано в миллисекундах (среднее арифметическое из двух запусков: pipeline + task_run). Количество потоков = 8 (все логические ядра процессора). Ускорение рассчитано относительно SEQ.

| Технология | Время (среднее), мс | Ускорение | Эффективность (на 8 потоках) |
|------------|---------------------|-----------|------------------------------|
| SEQ        | 3842                 | 1.00      | N/A                          |
| OMP        | 810                  | 4.74      | 59.3 %                       |
| STL        | 874                  | 4.40      | 55.0 %                       |
| ALL        | 1384                 | 2.78      | 34.7 %                       |
| TBB        | 3704                 | 1.04      | 13.0 %                       |

**Наблюдения и анализ:**

- **OpenMP** показывает наилучшее ускорение (почти в 4.7 раза). Это объясняется тем, что директива `collapse(2)` равномерно распределяет все пары блоков между потоками, а накладные расходы на создание параллельных областей минимальны. Сдвиги блоков остаются последовательными, но их доля в общем времени невелика. Ускорение 4.74 при 8 потоках даёт эффективность около 60% — хороший результат для задачи с последовательными фрагментами и интенсивным обменом данными через кэш.
- **STL** (ручное управление потоками) даёт близкое к OpenMP ускорение (~4.4). Небольшое отставание может быть связано с более «тяжёлыми» потоками (создаются заново на каждый вызов `ParallelFor`) или с менее эффективным выравниванием загрузки.
- **Комбинированная версия (ALL)** заметно медленнее (ускорение 2.78). Причина — избыточность: умножение блоков дополнительно распараллелено по OpenMP, что приводит к появлению вложенного параллелизма (блоки распределяются через TBB, а внутри каждого блока — ещё и OpenMP-потоки). Это вызывает конкуренцию за ресурсы и увеличивает накладные расходы на синхронизацию.
- **TBB** продемонстрировала неожиданно низкую производительность, практически равную последовательной версии. Вероятные причины: 
  - Отсутствие внутреннего распараллеливания умножения блока (в то время как в OpenMP оно есть, а в STL — параллелизм по блокам, которых достаточно, 16×16 = 256 задач). 
  - Накладные расходы на `blocked_range2d` для такого количества задач (256) могут быть значительными, особенно при мелких блоках (64×64). 
  - Возможно, также повлияла планировка задач TBB (work-stealing), которая в данном случае не даёт преимущества из-за высокой регулярности и малого размера задачи.
  - Дополнительная причина — процессор i5-1155G7 имеет только 4 физических ядра; TBB, возможно, неудачно распределяет 8 логических потоков, что приводит к излишним переключениям контекста.

**Масштабируемость**: С учётом того, что максимальное ускорение достигнуто на 8 потоках и составляет 4.74, можно предположить, что дальнейшее увеличение числа потоков не приведёт к росту производительности из-за ограничений алгоритма (последовательные сдвиги, задержки доступа к памяти). Потолок ускорения находится в районе 5–6 раз на данном оборудовании.

## 8. Заключение

В ходе работы были реализованы пять вариантов алгоритма Кэннона для умножения плотных матриц: последовательный, с использованием OpenMP, Intel TBB, стандартной библиотеки потоков C++ и комбинированный. Все реализации успешно проходят функциональные тесты. Анализ производительности на реальном процессоре Intel Core i5-1155G7 (4 ядра / 8 потоков) показал, что наилучшее ускорение достигается с OpenMP (в 4.74 раза) и STL (в 4.40 раза). TBB в текущей реализации не даёт выигрыша по сравнению с последовательным кодом, что требует дополнительной оптимизации (например, распараллеливания внутреннего умножения блока или изменения стратегии разбиения). Комбинированная версия оказалась неэффективной из-за накладных расходов на взаимодействие разных технологий параллелизма.

Таким образом, для задач умножения матриц по схеме Кэннона на многопроцессорных системах с общей памятью наиболее предпочтительными являются OpenMP или прямые вызовы `std::thread`. Использование TBB требует более тщательной настройки гранулярности задач.

## Приложение (краткий фрагмент кода)

```cpp
// Пример параллельного умножения блоков с помощью OpenMP
void RemizovKDenseMatrixMultiplicationCannonAlgorithmOmp::RunCannonCycle(
    Block4D &a_blocks, Block4D &b_blocks, Block4D &c_blocks,
    int block_size, int block_count) {
  for (int step = 0; step < block_count; ++step) {
#pragma omp parallel for collapse(2) schedule(dynamic)
    for (int i = 0; i < block_count; ++i) {
      for (int j = 0; j < block_count; ++j) {
        MultiplyBlock(a_blocks[i][j], b_blocks[i][j], c_blocks[i][j], block_size);
      }
    }
    if (step < block_count - 1) {
      ShiftBlocksLeft(a_blocks, block_count);
      ShiftBlocksUp(b_blocks, block_count);
    }
  }
}
```
