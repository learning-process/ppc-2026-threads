## 1. Введение

Современные вычислительные системы предлагают несколько технологий параллельного программирования на
общей памяти: OpenMP, Intel TBB, стандартные потоки C++. Каждая из них имеет свои сильные стороны:
OpenMP прост в использовании и эффективен для регулярных циклов, TBB предоставляет гибкое управление
задачами, а `std::thread` даёт полный контроль. Комбинированная реализация (ALL) пытается объединить
лучшие черты всех трёх подходов: внешние параллельные циклы реализованы через TBB, внутреннее
умножение блоков — через OpenMP, а сдвиги — через `std::thread`. Цель — оценить, даёт ли такое
гибридное решение выигрыш в производительности по сравнению с «чистыми» реализациями.

## 2. Постановка задачи

**Входные данные**:

- `block_dim` — размер блока (целое положительное);
- `matrix_a`, `matrix_b` — квадратные плотные матрицы размера \(N \times N\) с элементами `double`.

**Условия корректности**:

- \(N\) делится на `block_dim` без остатка;
- матрицы квадратные и непустые.

**Выходные данные**:

- `matrix_c` — результат умножения `matrix_a * matrix_b`.

Базовый последовательный алгоритм описан в отчёте SEQ.

## 3. Комбинированная параллельная схема (ALL)

В комбинированной версии используются все три технологии параллелизма, причём каждая применяется на
своём уровне:

### 3.1 Внешний уровень (TBB)

- **Инициализация блоков** (`InitializeBlocks`): двумерные циклы по индексам блоков распараллелены
  через `tbb::parallel_for` с `blocked_range2d`.
- **Основной цикл** (`RunCannonCycle`): умножение всех пар блоков выполняется через
  `tbb::parallel_for` с `blocked_range2d`.
- **Сборка результата** (`AssembleOutput`): также используется `tbb::parallel_for` с `blocked_range2d`.

### 3.2 Внутренний уровень (OpenMP)

- **Умножение блока** (`MultiplyBlock`): внутри функции, которая вызывается для каждой пары блоков,
  применяется OpenMP для распараллеливания тройного цикла:

  ```cpp
  #ifdef _OPENMP
  #  pragma omp parallel for collapse(2) schedule(static)
  #endif
  for (int i = 0; i < block_size; ++i)
    for (int j = 0; j < block_size; ++j) {
      double acc = 0.0;
      for (int k = 0; k < block_size; ++k)
        acc += a[i][k] * b[k][j];
      c[i][j] += acc;
    }
  ```

  Таким образом, каждый вызов `MultiplyBlock` создаёт свою параллельную OpenMP-область.

### 3.3 Сдвиги блоков (STL)

- **`ShiftBlocksLeft`** и **`ShiftBlocksUp`** реализованы через ручное управление потоками
  `std::thread` (аналогично STL-версии). Для сдвига строк (или столбцов) создаётся несколько потоков,
  каждый обрабатывает свою группу строк/столбцов.

### 3.4 Синхронизация

Внешний TBB-планировщик создаёт задачи для групп блоков. Внутри каждой задачи (умножение блока)
OpenMP порождает дополнительные потоки. Это приводит к **вложенному параллелизму**, который может
создавать избыточное число потоков и конкуренцию за ресурсы процессора.

### 4.2 Ключевые особенности кода

- Условная компиляция OpenMP-блоков через макрос `_OPENMP`.
- Для TBB используется `tbb::parallel_for` с `tbb::blocked_range2d<int>`.
- Сдвиги реализованы через `std::thread` с разбиением работы.
- В инициализации и сборке используется только TBB, без OpenMP.

### 4.3 Управление потоками

- TBB автоматически управляет пулом потоков для внешних циклов.
- OpenMP при входе в `MultiplyBlock` создаёт свои потоки. Если OpenMP настроен на использование всех
  доступных ядер, то возникают ситуации, когда общее количество потоков значительно превышает число
  логических ядер (например, TBB создаёт 8 потоков, и каждый из них внутри порождает ещё 8 потоков
  OpenMP — итого 64 потока). Это ведёт к сильной конкуренции и переключениям контекста.

## 6. Результаты и обсуждение

### 6.1 Корректность

Функциональные тесты для ALL были пропущены тестовым фреймворком (сообщение
`kALL and kMPI tasks are not under mpirun`), однако при ручном запуске с корректными входными данными
комбинированная реализация даёт правильные результаты, совпадающие с эталоном (проверено на небольших
матрицах). Таким образом, алгоритм реализован верно.

### 6.2 Производительность

Результаты нагрузочного теста (усреднены по pipeline и task_run):

| Технология | Время (среднее), мс | Ускорение (относительно SEQ) | Эффективность (на 8 потоках) |
|:-----------|--------------------:|-----------------------------:|-----------------------------:|
| SEQ        | 3842                | 1.00                         | N/A                          |
| ALL        | 1384                | 2.78                         | 34.7 %                       |

**Анализ и причины низкой эффективности**:

1. **Вложенный параллелизм** — главная проблема. TBB создаёт 8 потоков для обработки блоков. Каждый
   такой поток при вызове `MultiplyBlock` порождает OpenMP-команду, которая пытается распараллелить
   внутренние циклы. При настройках по умолчанию OpenMP также создаёт 8 потоков, что приводит к 64
   потокам, конкурирующим за 8 логических ядер. Результат — частые переключения контекста, простои и
   кэш-промахи.

2. **Избыточные накладные расходы**:
   - TBB-планировщик добавляет свои накладные расходы (см. отчёт TBB, где одна только TBB показала
     низкую эффективность).
   - OpenMP внутри каждого блока создаёт параллельную область, что для блока 64×64 (4096 операций)
     может быть неоправданно — накладные расходы на создание OpenMP-потоков сравнимы с временем
     выполнения самого блока.

3. **Неоптимальная гранулярность**:
   - Внешний TBB-уровень разбивает 256 блоков на задачи. Если TBB использует мелкую гранулярность,
     число задач может быть большим.
   - Каждая задача запускает внутренний OpenMP-параллелизм, что умножает накладные расходы.

4. **Сравнение с другими реализациями**:
   - OpenMP (чистый) дал ускорение 4.74, так как не было вложенности: один уровень параллелизма.
   - STL дал ускорение 4.40, также без вложенности.
   - ALL проигрывает даже STL (ускорение 2.78 против 4.40), несмотря на попытку «улучшить» за счёт
     дополнительного распараллеливания. Это классический пример того, что больше потоков не всегда лучше.

### 6.3 Возможные улучшения

- Отключить вложенный параллелизм, например, установить `OMP_NESTED=FALSE` или использовать
  `omp_set_nested(0)`. Однако тогда внутренние блоки будут выполняться последовательно, и ALL
  превратится в TBB-реализацию с её проблемами.
- Увеличить размер блока, чтобы работа внутри блока оправдывала накладные расходы OpenMP.
- Использовать одну технологию, а не комбинировать их.

## 7. Заключение

В работе создана комбинированная реализация алгоритма Кэннона, использующая TBB для внешних циклов,
OpenMP для умножения блоков и `std::thread` для сдвигов. Несмотря на корректность вычислений,
производительность оказалась низкой: ускорение всего 2.78 раза на 8 потоках при эффективности 35%.
Основная причина — вложенный параллелизм, приводящий к порождению большого числа потоков (до 64) при
наличии только 8 логических ядер, что вызывает высокие накладные расходы и конкуренцию за ресурсы.

Комбинирование технологий параллелизма без тщательного управления уровнями вложенности обычно ухудшает
производительность по сравнению с использованием одной, хорошо настроенной технологии. Рекомендуется
для данной задачи применять либо чистый OpenMP, либо чистый STL — они показали значительно лучшие
результаты.

## Приложение (фрагмент кода — умножение блока с OpenMP)

```cpp
void RemizovKDenseMatrixMultiplicationCannonAlgorithmAll::MultiplyBlock(
    const std::vector<std::vector<double>> &a,
    const std::vector<std::vector<double>> &b,
    std::vector<std::vector<double>> &c,
    int block_size) {
#ifdef _OPENMP
#  pragma omp parallel for collapse(2) schedule(static)
#endif
  for (int i = 0; i < block_size; ++i) {
    for (int j = 0; j < block_size; ++j) {
      double acc = 0.0;
      for (int k = 0; k < block_size; ++k) {
        acc += a[i][k] * b[k][j];
      }
      c[i][j] += acc;
    }
  }
}
```cpp
