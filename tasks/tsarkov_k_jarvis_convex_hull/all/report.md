# 1. Контекст

В данной работе была реализована гибридная версия алгоритма Джарвиса с использованием MPI и потокового распараллеливания внутри процесса.

Гибридная схема позволяет одновременно использовать:

- межпроцессное распараллеливание;
- внутрипроцессное распараллеливание.

Такой подход применяется в задачах высокопроизводительных вычислений, где вычисления распределяются между несколькими process rank-ами, а внутри каждого процесса дополнительно используются потоки.

В данной реализации MPI используется для организации работы между rank-ами, а внутри процесса применяются OpenMP, oneTBB и STL threads.

# 2. Постановка задачи

Необходимо реализовать гибридную версию алгоритма построения выпуклой оболочки множества точек на плоскости.

Полная постановка задачи, ограничения и описание входных данных приведены:

- в `report.md`;

Последовательная версия используется как baseline для проверки корректности и вычисления ускорения.

# 3. Базовый алгоритм

Последовательная версия алгоритма Джарвиса:

- находит крайнюю левую точку;
- выполняет поиск следующей вершины оболочки;
- на каждой итерации выполняет полный проход по множеству точек;
- завершает построение после возврата в стартовую точку.

Наиболее затратной частью алгоритма является поиск следующей точки оболочки.

# 4. Межпроцессная схема

## Использование MPI

Для организации межпроцессного взаимодействия используется:

```cpp
MPI_Comm_rank
```

Данный вызов определяет rank текущего процесса внутри:

```cpp
MPI_COMM_WORLD
```

Каждый process rank выполняет собственную часть вычислений.

## Роли rank-ов

В реализации используется модель с несколькими MPI process rank-ами.

Каждый rank:

- получает доступ к набору точек;
- выполняет локальные вычисления;
- участвует в синхронизации результатов.

Отдельный управляющий rank в реализации не выделяется - все rank-и выполняют одинаковую вычислительную работу.

## Распределение данных

Набор точек логически разделяется между process rank-ами.

Каждый rank обрабатывает собственную часть диапазона точек при поиске следующей вершины оболочки.

Такой подход позволяет уменьшить объём вычислений, выполняемых одним процессом.

## Используемые MPI-вызовы

В реализации используются:

- `MPI_Comm_rank`
- `MPI_Comm_size`
- `MPI_Barrier`

## MPI_Comm_rank

Используется для получения номера текущего rank-а.

## MPI_Comm_size

Используется для получения общего количества process rank-ов.

## MPI_Barrier

Используется для синхронизации процессов.

```cpp
MPI_Barrier(MPI_COMM_WORLD);
```

Барьер гарантирует, что все process rank-и завершили текущий этап вычислений перед продолжением работы.
Это необходимо для согласованности вычислений между процессами.

# 5. Внутрипроцессная схема

## Используемые thread-технологии

Внутри process rank-а используются:

- OpenMP;
- oneTBB;
- STL threads.

Каждая технология выполняет локальное распараллеливание поиска следующей точки оболочки.

## Разделение работы между потоками

Внутри rank-а диапазон точек дополнительно делится между потоками.

Каждый поток:

- обрабатывает собственный диапазон;
- вычисляет локальный лучший результат;
- не изменяет данные других потоков.

## Локальные результаты

Для каждого потока используется собственный локальный объект:

```cpp
BestPointState
```

После завершения вычислений локальные результаты объединяются внутри process rank-а.

## Объединение результатов

Сначала объединяются результаты потоков внутри процесса.
После этого выполняется синхронизация MPI process rank-ов.

В ALL-версии реализованы:

- OpenMP backend;
- oneTBB backend;
- STL threads backend.

Выбор backend-а зависит от конфигурации запуска.

## MPI_Barrier

Синхронизация процессов выполняется через:

```cpp
MPI_Barrier(MPI_COMM_WORLD);
```

Барьер используется после завершения локальных вычислений process rank-ов.

## Потенциальные узкие места

Основными узкими местами реализации являются:

- MPI-коммуникации;
- синхронизация между rank-ами;
- запуск нескольких процессов;
- overhead потокового runtime внутри rank-а.

На небольших размерах задач стоимость синхронизации и коммуникаций становится сопоставимой со стоимостью самих вычислений.
