## 1. Контекст

В данной работе последовательная версия алгоритма Джарвиса была адаптирована с использованием `std::thread`.

В отличие от OpenMP и oneTBB, стандартная библиотека C++ требует ручного управления потоками,
распределения диапазонов работы и синхронизации.

Данная реализация позволяет более подробно контролировать схему распараллеливания и
показывает стоимость ручного управления потоками.

## 2. Постановка задачи

Необходимо реализовать thread-based версию алгоритма построения выпуклой оболочки множества точек на плоскости.

Полная постановка задачи, ограничения и описание входных данных приведены в `seq/report.md`.

SEQ-версия используется как эталон корректности и baseline для вычисления ускорения.

## 3. Базовый алгоритм

Последовательная версия алгоритма Джарвиса:

- находит крайнюю левую точку;
- выполняет последовательный поиск следующей вершины оболочки;
- на каждой итерации выполняет полный проход по множеству точек;
- завершает построение после возврата в стартовую точку.

Наиболее затратным этапом является поиск следующей точки оболочки.

## 4. Схема распараллеливания

### Разделение диапазонов

Множество точек разбивается между потоками на независимые диапазоны.

Размер диапазона вычисляется по формуле:

```cpp
chunk_size =
    (range_size + actual_thread_count - 1) /
    actual_thread_count;
```

Каждый поток получает собственный диапазон:

```cpp
thread_range_begin
thread_range_end
```

### Работа потоков

Каждый поток:

- выполняет поиск локальной лучшей точки;
- обрабатывает только собственный диапазон;
- не изменяет данные других потоков.

Потоки создаются через:

```cpp
std::thread
```

### Пример создания потоков

```cpp
std::vector<std::thread> threads;

for (std::size_t thread_index = 0;
     thread_index < actual_thread_count;
     ++thread_index) {
  threads.emplace_back([&, thread_index] {
    local_best_states[thread_index] =
        FindBestStateInRange(
            unique_points,
            current_point_index,
            thread_ranges[thread_index].first,
            thread_ranges[thread_index].second);
  });
}
```

### Локальные результаты

Для каждого потока используется собственный локальный объект:

```cpp
std::vector<BestPointState> local_best_states
```

Каждый поток записывает результат только в собственный элемент массива.

После завершения всех потоков локальные результаты объединяются в общий результат.

### Join потоков

Ожидание завершения потоков выполняется после создания всех `std::thread`.

```cpp
for (std::thread& thread : threads) {
  thread.join();
}
```

Вызов `join()` расположен отдельно от цикла создания потоков.
Это позволяет избежать сериализации вычислений.

### Синхронизация

В реализации не используются:

- mutex;
- atomic;
- condition_variable.

Гонки данных отсутствуют, так как:

- каждый поток работает только со своим диапазоном;
- запись выполняется только в собственный локальный элемент;
- основной массив точек используется только для чтения.

### Изменения относительно SEQ

По сравнению с последовательной версией были добавлены:

- ручное разбиение диапазонов;
- создание `std::thread`;
- локальные результаты потоков;
- объединение результатов после завершения вычислений.

Основная логика алгоритма Джарвиса сохранена.

### Карта поток → диапазон

Каждый поток получает собственный диапазон индексов:

```cpp
[thread_range_begin, thread_range_end)
```

Диапазоны не пересекаются.

### Объединение результатов

После завершения всех потоков выполняется последовательное объединение локальных лучших состояний.

Для объединения используется функция:

```cpp
SelectBetterState
```

STL-реализация показала стабильные результаты, однако значительного ускорения получить не удалось.

Основными причинами ограничения производительности стали:

- стоимость создания потоков;
- overhead `join()`;
- ручное управление потоками;
- сравнительно небольшой размер задачи.

Наиболее заметное улучшение времени выполнения наблюдалось при использовании 4 потоков.

## 9. Выводы

Реализация на `std::thread` позволила вручную контролировать процесс распараллеливания алгоритма Джарвиса.

Основными особенностями реализации стали:

- ручное разбиение диапазонов;
- явное создание и завершение потоков;
- локальные результаты потоков;
- отсутствие сложной синхронизации.

По сравнению с OpenMP и TBB реализация на STL требует большего количества служебного кода и ручного управления потоками.

Основным ограничением производительности стал overhead создания и синхронизации потоков на небольших размерах задач.

На более крупных наборах данных и при более длительных вычислениях STL-версия
могла бы показывать более заметное ускорение.
