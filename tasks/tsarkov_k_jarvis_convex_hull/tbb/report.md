## 1. Контекст

В данной работе последовательная версия алгоритма Джарвиса была адаптирована для библиотеки oneTBB.

Для алгоритма Джарвиса характерен многократный перебор большого количества точек при поиске следующей вершины оболочки. 
Такой участок хорошо подходит для task-based распараллеливания, которое предоставляет oneTBB.

TBB позволяет автоматически распределять задачи между worker-потоками и 
выполнять балансировку нагрузки средствами runtime-системы.

## 2. Постановка задачи

Необходимо реализовать TBB-версию алгоритма построения выпуклой оболочки множества точек на плоскости.

Полная постановка задачи, ограничения и описание входных данных приведены в `seq/report.md`.

Последовательная версия используется как baseline для проверки корректности и вычисления ускорения.

## 3. Базовый алгоритм

Последовательная версия алгоритма Джарвиса:

- находит крайнюю левую точку;
- выполняет поиск следующей вершины оболочки;
- на каждой итерации выполняет полный проход по множеству точек;
- завершает построение после возврата в стартовую точку.

Наиболее затратным этапом алгоритма является поиск следующей точки оболочки.

## 4. Схема распараллеливания

### Используемый примитив

В реализации использовался:

```cpp
oneapi::tbb::parallel_reduce
```

Данный примитив позволяет:

- разбивать диапазон данных между worker-потоками;
- вычислять локальные результаты независимо;
- объединять результаты после завершения вычислений.

Для алгоритма Джарвиса это удобно, так как каждая точка может проверяться независимо от остальных.

### Диапазон работы

Для разделения данных использовался:

```cpp
oneapi::tbb::blocked_range<std::size_t>
```

Диапазон содержит индексы точек, которые должны быть обработаны worker-потоком.

Каждый поток выполняет поиск локальной лучшей точки внутри собственного диапазона.

### Пример распараллеливания

```cpp
oneapi::tbb::parallel_reduce(
    oneapi::tbb::blocked_range<std::size_t>(range_begin, range_end),
    initial_state,
    [&](const oneapi::tbb::blocked_range<std::size_t>& range,
        BestPointState local_state) {
      return FindBestStateInRange(
          unique_points,
          current_point_index,
          range.begin(),
          range.end(),
          local_state);
    },
    [&](const BestPointState& left,
        const BestPointState& right) {
      return SelectBetterState(
          unique_points,
          current_point_index,
          left,
          right);
    });
```

### Grainsize

В реализации использовался grainsize по умолчанию.

Для рассматриваемых размеров задач ручной подбор grainsize не дал бы заметного выигрыша, 
так как объём вычислений сравнительно небольшой.

При слишком маленьком grainsize возрастал бы overhead runtime-системы на управление задачами.

### Partitioner

Использовался стандартный `partitioner` oneTBB.

Автоматическое разбиение диапазонов оказалось достаточным, 
так как все итерации имеют примерно одинаковую вычислительную стоимость.

Дополнительная настройка partitioner для данной задачи не потребовалась.

### Ограничение конкуренции

Ограничение числа worker-потоков выполнялось через:

```cpp
oneapi::tbb::global_control
```

Использовался параметр:

```cpp
oneapi::tbb::global_control::max_allowed_parallelism
```

Данный механизм ограничивает максимальное количество worker-потоков runtime-системы TBB.

Количество потоков задавалось через переменную окружения:

```powershell
$env:PPC_NUM_THREADS=<N>
```

### Синхронизация

В реализации отсутствуют:

- mutex;
- atomic;
- critical sections.

Каждый worker-поток вычисляет локальный результат независимо.

Объединение локальных результатов выполняется на этапе reduction после завершения обработки диапазонов.

## 5. Детали реализации

Используемые файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

### Изменения относительно SEQ

По сравнению с последовательной версией были добавлены:

- `parallel_reduce`;
- `blocked_range`;
- локальные состояния потоков;
- reduction-функция объединения результатов.

Основная логика алгоритма Джарвиса при этом сохранена.

### Локальные аккумуляторы

Каждый worker-поток использует собственный локальный объект:

```cpp
BestPointState
```

Это позволяет избежать гонок данных при выборе следующей точки оболочки.

После завершения вычислений локальные состояния объединяются reduction-функцией.

### Работа с памятью

Основной массив точек используется только для чтения, 
поэтому дополнительная синхронизация доступа к памяти не требуется.

Ложного разделения кэша (false sharing) в данной реализации практически не возникает, 
так как worker-потоки работают с независимыми локальными объектами.

## 9. Выводы

oneTBB позволил реализовать task-based распараллеливание алгоритма Джарвиса без ручного управления потоками.

Основными преимуществами реализации стали:

- автоматическое распределение задач;
- встроенная балансировка нагрузки;
- удобная модель reduction;
- отсутствие ручной синхронизации.

Основным ограничением производительности стал overhead runtime-системы oneTBB и стоимость task scheduling 
на небольших размерах задач.

На более крупных наборах данных и при большем количестве вычислений на worker TBB-реализация 
могла бы показывать более заметное ускорение.
