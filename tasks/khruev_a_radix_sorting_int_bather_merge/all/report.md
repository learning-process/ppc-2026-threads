## 1. Контекст

Требуется гибридная реализация (MPI + OpenMP), объединяющая параллелизм с распределенной памятью и параллелизм с общей памятью.

## 2. Постановка задачи

Масштабировать алгоритм на кластерную архитектуру. Ограничения: сеть Бэтчера требует строгого контроля размеров массивов (ограничение до $2^k$).

## 3. Базовый алгоритм

Сохраняется концепция: независимая сортировка половин и последующее слияние Бэтчера.

## 4. Межпроцессная схема (MPI)

- **Распределение:** Процесс 0 добивает массив до $2^k$ и делит его на чанки между `active_procs` (число активных процессов выравнивается вниз до степени двойки). Передача идет через `MPI_Send` / `MPI_Recv`.
- **Дерево слияния:** Используется `TreeMergeHelper`. Процессы попарно объединяют данные: один отсылает свой отсортированный кусок, другой принимает, выполняет `OddEvenMerge` на объединенном локальном массиве и переходит на следующий уровень дерева.

## 5. Внутрипроцессная схема (OMP)

После получения локального чанка (в `LocalSortHelper`), процесс разбивает его пополам. Для независимой сортировки вызывается `#pragma omp parallel sections`, а локальное слияние выполняется через `#pragma omp parallel for`. Таким образом утилизируются ядра внутри узла.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

```cpp
// File: all/src/ops_all.cpp
void KhruevARadixSortingIntBatherMergeALL::TreeMergeHelper(int rank, int active_procs, int chunk_size,
                                                           std::vector<int> &local_data) {
  for (int step = 1; step < active_procs; step *= 2) {
    if (rank % (2 * step) == 0) {
      int sender = rank + step;
      int recv_count = chunk_size * step;
      std::vector<int> recv_data(recv_count);

      MPI_Recv(recv_data.data(), recv_count, MPI_INT, sender, 0, MPI_COMM_WORLD, MPI_STATUS_IGNORE);
      local_data.insert(local_data.end(), recv_data.begin(), recv_data.end());
      OddEvenMerge(local_data, local_data.size());
    } else {
      int receiver = rank - step;
      int send_count = static_cast<int>(local_data.size());
      MPI_Send(local_data.data(), send_count, MPI_INT, receiver, 0, MPI_COMM_WORLD);
      break;
    }
  }
}

```

`TreeMergeHelper`: после локальной сортировки процесс, не кратный текущему шагу слияния, отправляет свои данные принимающему процессу и выходит из цикла агрегации.

## 10. Выводы

Гибридная схема позволила достичь абсолютно лучшего времени среди всех реализаций (0.0266 с) и максимального ускорения 1.74x при конфигурации 3 процесса × 2 потока. Однако результаты наглядно показывают проблему переподписки ядер: на 4-ядерном процессоре конфигурации с 8 логическими работниками (4х2 или 2х4) работают медленнее из-за жесткой конкуренции за кэш и накладных расходов `MPI`. Межпроцессная коммуникация оправдывает себя только на больших объемах данных и при грамотном маппинге процессов на физические ядра.
