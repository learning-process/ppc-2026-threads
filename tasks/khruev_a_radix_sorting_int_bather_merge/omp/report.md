## 1. Контекст

Перенос последовательной реализации на технологию OpenMP для ускорения вычислений на системах с общей памятью (многоядерные процессоры).

## 2. Постановка задачи

Необходимо распараллелить эталонный алгоритм из `seq`, сохранив его семантику и корректность на тех же входных данных. Ограничения и граничные случаи аналогичны SEQ-версии.

## 3. Базовый алгоритм

Алгоритм дополняет массив до степени двойки, сортирует левую и правую половины поразрядной сортировкой, а затем объединяет их через сеть слияния Бэтчера.

## 4. Схема распараллеливания

Распараллеливание выполнено на двух уровнях:

1. **Сортировка половин:** Используется директива `#pragma omp parallel sections` для независимой сортировки левой и правой частей массива.
2. **Слияние Бэтчера:** Циклы внутри шагов сети слияния распараллелены директивой `#pragma omp parallel for`. Так как на каждом шаге итерации независимы (каждый поток читает и пишет строго в свои индексы `i` и `j`), гонки данных (Data Races) отсутствуют. Используется `default(none)` для строгого контроля видимости переменных.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for shared(a, n, po) default(none)
for (size_t i = po; i < n - po; i += 2 * po) {
  for (size_t j = 0; j < po; ++j) {
    CompareExchange(a, i + j, i + j + po);
  }
}
```

Фрагмент показывает распараллеливание этапа слияния.
Переменные `a`, `n`, `po` передаются как `shared`, так как массив `a` модифицируется безопасно, а `n` и `po` только читаются. Неявный барьер в конце области `parallel for` гарантирует, что следующий шаг слияния не начнется до завершения текущего.
