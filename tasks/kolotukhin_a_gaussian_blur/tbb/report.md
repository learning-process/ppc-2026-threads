## 1. Контекст

Эта реализация использует библиотеку `Intel oneTBB` для параллельной обработки изображения.
В отличие от OpenMP, TBB предлагает задачно-ориентированный подход с автоматическим разбиением работы на задачи
(tasks) и кражей работы для балансировки нагрузки. Цель – оценить, насколько продвинутый планировщик
TBB может улучшить масштабируемость.

## 3. Базовый алгоритм

## 4. Схема распараллеливания

### Декомпозиция данных

Выбран примитив TBB `tbb::parallel_for` с двумерным диапазоном.

```cpp
// Фрагмент файла: tbb/src/ops_tbb.cpp
tbb::parallel_for(
    tbb::blocked_range2d<int>(0, img_height, 0, img_width),
    [&](const tbb::blocked_range2d<int> &range) {
        for (int row = range.rows().begin(); row < range.rows().end(); row++) {
            for (int col = range.cols().begin(); col < range.cols().end(); col++) {
                // Вычисление гауссова размытия для одного пикселя
            }
        }
    }
);
```

### Компоненты

- `blocked_range2d<int>(0, img_height, 0, img_width)`:
  - Двумерный диапазон, который будет рекурсивно делиться на поддиапазоны.
  - Разбиение происходит как по строкам, так и по столбцам, что создаёт более мелкие блоки работы
  и улучшает балансировку.
  - Размер блока по умолчанию автоматически подбирается планировщиком.
- **Grain size и Partitioner:** явный размер блока не задан, `auto_partitioner` автоматически определяет степень
дробления, балансируя накладные расходы на создание задач и равномерность загрузки worker'ов.
- **Work-stealing:** планировщик TBB использует кражу работы: если поток завершил свои задачи, он может забрать
задачу из очереди другого потока. Это особенно эффективно на гибридных архитектурах, где скорость выполнения
может различаться.
- **Контроль конкуренции:** количеством рабочих потоков управляют `PPC_NUM_THREADS` и глобальные настройки TBB.

## 5. Детали реализации

Файлы: `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

### Структура класса `KolotukhinAGaussinBlureTBB`

- `ValidationImpl()` аналогично `SEQ` проверяет размеры.
- `PreProcessingImpl()` выделяет память под выходной буфер.
- `RunImpl()` содержит вызов `tbb::parallel_for` с лямбда-функцией.
- `PostProcessingImpl()` пустой, поскольку синхронизация гарантируется завершением `parallel_for`.

### Потокобезопасность

Каждый поток работает с уникальным блоком пикселей, записывая результаты в непересекающиеся области
выходного буфера. Гонки данных исключены без использования мьютексов или атомарных операций.

### Анализ результатов

- **Накладные расходы TBB**

При 1 потоке время (0.2987 с) в 1.5 раза хуже SEQ (0.2021 с) –
возникает из-за инициализации планировщика, создания пула потоков и разбиения диапазона.

- **Характер масштабирования**

Ускорение монотонно растёт до 16 потоков (4.82x).

- **Пик и спад**

Максимальное ускорение достигается при 16 потоках (4.82x). При 32 потоках наблюдается спад до 4.64x – число
потоков превышает физическое количество ядер, накладные расходы на переключение контекста и синхронизацию
начинают доминировать.

- **Эффективность**

Держится на уровне 68-69% вплоть до 4 потоков, затем снижается до 46.0% при 10 потоках и резко падает
до 14.5% при 32 потоках. Это подтверждает memory-bound природу задачи.

- **Двумерное разбиение**

`blocked_range2d` улучшает локальность данных в кэше по сравнению с построчной обработкой,
что позволяет TBB дольше удерживать высокую эффективность при росте числа потоков.

- **Оптимальные конфигурации**
  - 4 потока – пик эффективности (68.0%) при ускорении 2.72x.
  - 7 потоков – хороший баланс ускорения (4.06x) и эффективности (58.0%).

## 9. Выводы

- TBB обеспечивает наилучшее ускорение среди всех рассмотренных реализаций – до **4.82x** на 16 потоках.

- **Work-stealing критичен для гибридных архитектур**

Способность TBB динамически перераспределять работу между потоками компенсирует разницу в
производительности ядер, обеспечивая монотонный рост ускорения без провалов.

- **Двумерное разбиение улучшает локальность данных**

Блочная обработка изображения через `blocked_range2d` эффективнее использует кэш-память,
чем построчная, что частично смягчает эффект memory wall.

- **Предел масштабирования**

При 32 потоках ускорение снижается до 4.64x – накладные расходы на избыточное
число потоков (превышающее физические ядра) начинают доминировать над выигрышем от параллелизма.
