## 3. Межпроцессная схема (MPI)

### Распределение данных

Изображение делится на горизонтальные полосы по количеству MPI-процессов.
Первые `remainder` процессов получат по одной дополнительной строке.
Каждый процесс получает дополнительные halo-строки (по одной сверху или снизу) для корректной обработки границ.

```cpp
// Фрагмент файла: all/src/ops_all.cpp
void KolotukhinAGaussinBlurALL::DistributeWork() {
  int rows_per_process = global_height_ / proc_count_;
  int remainder = global_height_ % proc_count_;

  // Вычисление локальной высоты с halo-зонами
  local_height_ = (rank_ < remainder) ? rows_per_process + 1 : rows_per_process;
  local_height_ = (rank_ == 0 || rank_ == last_handler) 
                  ? local_height_ + 1   // Верхний или нижний процесс: +1 halo
                  : local_height_ + 2;  // Средний процесс: +2 halo (сверху и снизу)

  // ...
  if (rank_ == 0) {
    SendWorkData(rows_per_process, remainder);  // Root отправляет данные
  } else {
    ReceiveWorkData();  // Остальные принимают
  }
}
```

### Halo-зоны

Для процесса, обрабатывающего строки `[start_row, end_row)`:

- Принимаются строки `[start_row-1, end_row+1]` (с учётом границ изображения).
- После фильтрации отправляются только строки `[start_row, end_row)` без halo-зон.

### Коммуникационные операции MPI

- `MPI_Comm_rank` / `MPI_Comm_size` – определение ранга и числа процессов.
- `MPI_Bcast` – широковещательная рассылка размеров изображения.
- `MPI_Send` / `MPI_Recv` – передача полос изображения между процессами.
- `MPI_Allreduce` – проверка валидности на всех процессах.

## 4. Внутрипроцессная схема (OpenMP)

Каждый MPI-процесс обрабатывает свою полосу изображения, используя OpenMP для распараллеливания:

```cpp
// Фрагмент файла: all/src/ops_all.cpp (функция ApplyGaussianBlur)
#pragma omp parallel for collapse(2) schedule(static) default(none) \
    shared(src_data, dst_data, width, height, start_row, end_row, kKernel, kSum)
  for (int row = start_row; row < end_row; row++) {
    for (int col = 0; col < width; col++) {
      // Стандартное вычисление гауссова размытия
    }
  }
```

### Особенности

- Распараллеливание применяется только к локальной полосе строк.
- Количество потоков: `PPC_NUM_THREADS` на каждый MPI-процесс.
- Общее количество worker'ов: `ranks × threads_per_rank`.

## 5. Детали реализации

**Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

### Жизненный цикл задачи

- Инициализация MPI (`MPI_Comm_rank`, `MPI_Comm_size`).
- `ValidationImpl()` проверяет размер изображения, `MPI_Allreduce` для глобальной валидации.
- `PreProcessingImpl()` делает `MPI_Bcast` размеров, выделение памяти, рассылку данных(`DistributeWork()`).
- `RunImpl()` применяет фильтр гаусса к изображению `ApplyGaussianBlur()` с OpenMP.
- `PostProcessingImpl()` собирает обработанные полосы на root-процессе `GatherResults()`.

### Сбор результатов

```cpp
void KolotukhinAGaussinBlurALL::GatherResultsRoot(int rows_per_process, int remainder) {

    std::vector<std::uint8_t> src_data(...);

}
```

### Анализ результатов

- **Базовый оверхед MPI**

При 1 процессе и 1 потоке время (0.3286 с task_run) в 1.6 раза хуже SEQ (0.2021 с).
Это требуют инициализация MPI-окружения, логика распределения (`DistributeWork`), копирование данных в halo-буферы.

- **Масштабирование внутри одного процесса (1 ранг)**

Ускорение практически не растёт с увеличением числа потоков – 1.14x при 3 потоках,
1.17x при 4, и далее плато 1.14-1.18x вплоть до 32 потоков.
MPI-оверхед доминирует над вычислительным временем: 32 потока не компенсируют затраты на коммуникационную логику.
Рост числа потоков после 4 не даёт значимого эффекта – задача упирается в накладные расходы MPI.

- **Эффект двух процессов**

При 2 процессах ускорение достигает 2.33x (2×4, 2×5), что существенно лучше,
чем 1 процесс с тем же числом потоков (1×8: 1.16x vs 2×4: 2.33x).
Каждый MPI-процесс работает с половиной изображения – меньший объём данных лучше помещается в кэш,
а коммуникационные издержки (`MPI_Send`/`MPI_Recv` при раздаче и сборе) фиксированы
и не растут с числом потоков внутри процесса.

- **Насыщение при 2 процессах**

Ускорение стабилизируется на уровне 2.26-2.33x при 4+ потоках на процесс и не растёт вплоть до 16 потоков на процесс
(32 total workers). Дальнейшее наращивание потоков не даёт выигрыша – узким местом становятся коммуникации и сборка
результатов.

- **Причина низкой эффективности**

Размер изображения слишком мал для MPI. Время передачи данных между процессами сопоставимо с временем их обработки.
Halo-зоны добавляют ~1-2% лишних данных. На одном узле гибридный подход избыточен.

- **Оптимальная конфигурация**

2 процесса × 2-3 потоков дают ускорение ~2.30x при достаточно высокой эффективности.

## 9. Выводы

- **Накладные расходы MPI критичны для данной задачи**

При размере изображения 7680×4320 время на коммуникации сопоставимо с временем вычислений.
Это ограничивает ускорение величиной 2.33x даже при 32 потоках.

- **Распределение данных по процессам эффективнее внутрипроцессного распараллеливания**

Конфигурация 2×4 показывает ускорение 2.33x, в то время как 1×8 – лишь 1.16x.
Деление данных между MPI-процессами улучшает локальность (меньше данных на процесс – лучше кэш),
тогда как OpenMP-потоки внутри одного процесса конкурируют за общую память без выигрыша от распределённой обработки.
