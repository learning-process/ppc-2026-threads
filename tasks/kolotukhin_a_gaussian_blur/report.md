## 1. Введение

Данная работа посвящена реализации и сравнению различных технологий параллельного программирования
на примере задачи размытия изображения по Гауссу (Gaussian Blur) с ядром 3×3.
Фильтр Гаусса обладает регулярной вычислительной структурой: одинаковые операции над независимыми
элементами данных (пикселями), что делает данную задачу подходящей для сравнения разных моделей параллелизма.

## 2. Постановка задачи

### Формат данных

Типы данных в задаче описаны [здесь](../common/include/common.hpp) и представляют следующие типы:

- **Input:** кортеж `(pixel_data, img_width, img_height)`, где `pixel_data` – вектор `uint8_t` (оттенки серого)
размера `img_width × img_height`.
- **Output:** вектор `uint8_t` такого же размера, содержащий результирующее изображение в оттенках серого.

### Ядро свёртки (3x3)

```text
1 2 1
2 4 2 / 16
1 2 1
```

### Ограничения

- Входное изображение должно быть непустым (`img_width × img_height > 0`).
Проверка происходит на этапе чтения изображения при формировании теста
(`SetUp()` в [тестах производительности](../tests/performance/main.cpp)).
- Размер входного вектора пикселей должен строго соответствовать произведению
полученных `img_width` и `img_height`. Проверка проходит в `ValidationImpl()`.
- Входное изображение для обработки должно быть в оттенках серого (`uint8_t`).
- Для пикселей на границах используется отражение – координаты за пределами изображения
приводятся к ближайшим допустимым координатам.

### Критерий корректности

Поэлементное совпадение результата с эталонным выходом для заданного входа.

## 6. Интерпретация различий

### 6.1. SEQ – что показывает baseline

Время выполнения последовательной версии (T_seq = 0.2021 с на изображении 7680×4320) является точкой
отсчета и диагностическим инструментом. Оно показывает:

- Алгоритм хорошо ложится в кэш-память. При последовательном обходе изображения строка за строкой данные
подгружаются в L1/L2 кэш с высокой пространственной локальностью. Это делает задачу memory-bound.
- Любая параллельная версия, чтобы превзойти SEQ, должна обеспечить прирост скорости, превышающий затраты
на создание потоков, синхронизацию и коммуникации.
- Даже при идеальной параллелизации без оверхеда, 32-поточная система дала бы ускорение не более 32x.
Однако из-за memory wall реальный предел для данной задачи на этой архитектуре составляет 4-5x,
что подтверждается результатами TBB (4.82x).

**Вывод из SEQ**: baseline показывает, что задача параллелизуется, но не суперлинейно, и оверхед
синхронизации должен быть минимизирован.

### 6.2. OMP – сильные и слабые стороны на данной задаче

#### Сильные стороны OpenMP на данной задаче

- Одна директива `#pragma omp parallel for collapse(2)` превращает последовательный код в параллельный
без изменения логики.
- При небольшом числе потоков (5) достигается 3.14x ускорение при 62.8% эффективности – лучший показатель
среди всех технологий при том же числе потоков.
- Нет оверхеда на инициализацию – пул потоков создается один раз в начале программы.

#### Слабые стороны OpenMP (проявляются именно на этой задаче)

- Статическое распределение итераций (`schedule(static)`) не учитывает гибридную архитектуру процессора.
В конфигурации текущей машины ядра процессора имеют разную производительность.
При 6 потоках один поток, попадая на более медленное ядро,
замедляет всю группу. Это объясняет, почему ускорение на 6 потоках (3.06x) хуже, чем на 5 (3.14x).
- Отсутствие work-stealing, если один поток завершил свою часть работы, он простаивает, пока другие еще работают.
- Эффективность падает после 5-6 потоков. С 5 до 10 потоков эффективность падает
с 62.8% до 28.0% (потеря 55% относительной эффективности, что является самым резким падением среди всех технологий).

**Вывод из OMP**: OpenMP оптимален для задач с равномерной нагрузкой на однородных процессорах.

### 6.3. TBB – роль grain size и runtime

TBB превосходит OpenMP на 31.8% при 7 потоках (4.06x против 3.08x):

- **Grain size (размер задачи на поток)**. TBB автоматически подбирает **grain size** через **auto_partitioner**.
- **Work-stealing (кража задач)**. Когда ядро завершает свои блоки, оно берет невыполненные задачи из очереди
другого ядра. Это компенсирует разницу в производительности ядер.
- TBB создает потоки один раз и переиспользует их. При перезапусках программы (pipeline) появляется экономия
при инициализации.

#### Недостатки TBB для данной задачи

- При однократном запуске обработки вызов `parallel_for` тратит время с на создание пула потоков.
- oneTBB не входит в стандарт C++.
- Для малых изображений прирост TBB над OpenMP незначителен.

### 6.4. STL – цена ручного управления потоками

STL показывает ускорение 2.74x на 10 потоках – хуже, чем OMP (2.80x) и значительно хуже TBB (4.60x).

#### Возможные причины

- В режиме task_run создается 10 потоков через `std::thread`, каждый требует системного вызова и выделения стека.
- В pipeline-режиме (многократный вызов) проблема усугубляется – потоки создаются и уничтожаются на каждую итерацию.
- Отсутствие work-stealing. Как и OpenMP, STL использует статическое распределение:
каждый поток получает блок индексов `[begin, end)`. Разница в скорости ядер приводит к простою.
- Размер блока жестко задан как `total_pixels / num_threads`. Это приводит к:
  - потере пространственной локальности (строки не соседствуют в линейном массиве);
  - увеличению количества промахов кэша.

#### Преимущества STL

- Не требует внешних библиотек (только C++17 и выше)
- Полное управление созданием, привязкой и приоритетами потоков

**Вывод из STL**: для текущей задаче произошла потеря ~40% ускорения относительно TBB
(2.74x против 4.60x на 10 потоках).

### 6.5. ALL – цена коммуникации и выигрыш гибридности

#### Реализация коммуникаций

Root-процесс (rank 0) в цикле `for (int dest = 0; dest < proc_count_; dest++)` формирует для каждого процесса буфер
с halo-зонами и отправляет через `MPI_Send`. Остальные процессы принимают данные через `MPI_Recv`.
Аналогично при сборе результатов – каждый процесс отправляет свою обработанную полосу root-процессу.

Почему ALL проигрывает потоковым версиям на одном узле:

| Конфигурация | Workers | Время (с) | Speedup | Отставание от TBB             |
|--------------|---------|-----------|---------|-------------------------------|
|   ALL 2×4    |    8    |   0.0865  |  2.33x  | в 1.87 раза (TBB: 0.0463 с)   |
|   ALL 2×7    |   14    |   0.0893  |  2.26x  | в 1.80 раза (TBB 7: 0.0497 с) |

Для двух процессов

- Индивидуальная рассылка данных. Root-процесс отправляет `MPI_Send` процессу 1.
- Индивидуальный сбор результатов. Процесс 1 отправляет обработанную полосу root-процессу
- Формирование extended_data (дополнительные строки для границ)

**Вывод из ALL**: коммуникации занимают ~23-25% от времени ALL (0.0865 с).
Остальная разница с TBB объясняется отсутствием work-stealing и последовательной обработкой данных.

### 6.6. Сравнительная таблица характеристик в текущей задаче

| Характеристика    |  SEQ   |  OMP   |   TBB   | STL   | ALL   |
|-------------------|--------|--------|---------|-------|-------|
| Макс. ускорение   |  1.00x |  3.14x |**4.82x**| 3.36x | 2.33x |
| Пик эффективности |  –     |  62.8% |**68.0%**| 50.0% | 29.1% |
| Строки кода       |  1.0x  |  1.03x |**1.03x**| 1.63x | 4.30x |
| Распред. память   |  Нет   |  Нет   |  Нет    | Нет   |**Да** |
| Work-stealing     |  Нет   |  Нет   |  **Да** | Нет   | Нет   |

#### Резюме

- **SEQ** задает реалистичный upper bound (~5) для ускорения
- **OMP** прост, но страдает от статического распределения на гибридных CPU
- **TBB** побеждает за счет work-stealing и адаптивного grain size
- **STL** платит за портативность потерей производительности и сложностью кода
- **ALL** сложна и неэффективна на одном узле, но незаменима для кластеров

## 8. Заключение

### 8.1. Лучшая версия в зависимости от условий

| Условия эксплуатации                       | Рекомендуемая версия | Ключевое преимущество                         |
|--------------------------------------------|:--------------------:|-----------------------------------------------|
| Один узел, максимальная производительность |         TBB          | grain size, work-stealing для гибридных CPU   |
| Один узел, простота разработки             |         OMP          | Ускорение одной директивой, минимум изменений |
| Один узел, запрещены внешние библиотеки    |         STL          | Только стандартная библиотека C++17           |
| Многоузловой кластер / большие данные      |      ALL (MPI+)      | Распределённая память                         |

Лучшая производительность на одном узле: **TBB**\
Лучший баланс «производительность / сложность»: **OMP**

### 8.2. Ограничения сравнения

#### 1. ALL неэквивалентна потоковым версиям

- Разные модели памяти (распределённая vs общая)
- Разная «стоимость» единицы параллелизма (MPI-процесс vs поток)
- Измерения на одном узле – наихудший сценарий для ALL
- Результаты могут измениться для больших изображений
- Не исследовано масштабирование ALL на 4+ узлах

#### 2. Только оттенки серого

- Цветные изображения (RGB) изменят соотношение память / вычисления

### 8.3. Направления улучшения

1. Для **OpenMP** заменить `schedule(static)` на `schedule(dynamic, chunk)` для балансировки
2. Для **ALL** реализовать перекрытие коммуникаций и вычислений (non-blocking MPI) для скрытия латентности
3. Для **STL** добавить пул потоков вместо создания при каждом вызове
4. Для **TBB** настроить `grain size` через `tbb::blocked_range2d(..., grain_size)`
5. **Общее**: поддержка цветных изображений (RGB каналов) – изменит memory-bound характер
