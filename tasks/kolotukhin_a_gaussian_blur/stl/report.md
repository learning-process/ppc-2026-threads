## 4. Схема распараллеливания

### Декомпозиция данных

Изображение обрабатывается по 1D-диапазону индексов пикселей: `[0, total_pixels)`,
где `total_pixels = img_width * img_height`. Диапазон делится на `num_threads` непрерывных блоков примерно
равного размера.

```cpp
// Фрагмент файла: stl/src/ops_stl.cpp
void GetThreadRange(size_t tid, size_t total, size_t num_t, 
                    size_t &begin, size_t &end) {
  size_t chunk = total / num_t;
  begin = tid * chunk;
  end = (tid == num_t - 1) ? total : begin + chunk;
}
```

### Создание и управление потоками

```cpp
// Фрагмент файла: stl/src/ops_stl.cpp
const int num_threads = ppc::util::GetNumThreads();
std::vector<std::thread> threads;
threads.reserve(num_threads);

for (int tid = 0; tid < num_threads; tid++) {
    threads.emplace_back([&, tid]() {
        std::size_t begin = 0, end = 0;
        GetThreadRange(tid, total_pixels, num_threads, begin, end);
        
        for (std::size_t idx = begin; idx < end; idx++) {
            int row = static_cast<int>(idx / img_width);
            int col = static_cast<int>(idx % img_width);
            // ... вычисление для пикселя (col, row) ...
            output[idx] = result;  // Прямая запись в выходной буфер
        }
    });
}

// Синхронизация: дождаться завершения всех потоков
for (auto &th : threads) {
    th.join();
}
```

### Компоненты

- **`GetThreadRange(tid, total, num_t, begin, end)`**
  - Делит линейный диапазон `[0, total_pixels)` на `num_threads` непрерывных блоков равного размера.
  - Каждый поток получает свой уникальный диапазон `[begin, end)`, где все индексы принадлежат только ему.
  - Последний поток забирает остаток (`end = total`), что гарантирует обработку всех пикселей без пропусков.

- **Создание потоков**
  - Потоки создаются в цикле через `emplace_back` с лямбда-функцией.
  - Каждый поток получает `tid` по значению (копия), а все данные захватывает по ссылке `[&, tid]`.
  - `std::thread` при создании выполняет системный вызов для запуска нового потока ОС.

- **Распределение работы**
  - Каждый поток сам вычисляет свой диапазон индексов и обрабатывает его независимо.
  - Отсутствует динамическое перераспределение.

- **Запись результатов**
  - Прямая запись в `output[idx]` без блокировок. Поскольку диапазоны не пересекаются, гонки данных исключены.

- **Синхронизация**
  - Цикл `join()` выполняется после создания всех потоков, что обеспечивает параллельное выполнение.
  - `join()` блокирует главный поток до завершения каждого рабочего потока, гарантируя,
  что все вычисления закончены перед выходом из `RunImpl()`.

- **Контроль конкуренции**
  - Число потоков задаётся через `ppc::util::GetNumThreads()`, которое читает `PPC_NUM_THREADS`.
  - Аналог grainsize задаётся неявно: размер блока = `total_pixels / num_threads`.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

### Структура класса `KolotukhinAGaussinBlureSTL`

- `GetThreadRange()` выделяет диапазоны; гарантирует, что последний поток обрабатывает все оставшиеся
индексы (включая остаток от деления).
- `ValidationImpl()` гарантирует корректность входных данных.
- `PreProcessingImpl()` корректно выделяет память до начала вычислений.
- `RunImpl()` вызывает `join()` для всех потоков – гарантирует завершение перед выходом..
- `PostProcessingImpl()` пустой – синхронизация гарантируется завершением `parallel_for`.

### Потоки

- При каждом вызове `RunImpl()` создаются новые потоки.
- Использование `[&, tid]` в лямбде: `&` захватывает все внешние переменные по ссылке, `tid` – по значению.

### Анализ результатов

- **Накладные расходы при 1 потоке**

Время 0.3812 с почти вдвое хуже SEQ (0.2021 с). Причина в стоимости создания `std::thread`:
системный вызов, выделение стека, запуск планировщика ОС.

- **Рост ускорения**

Стабильный рост до 5 потоков (2.50x), затем замедление.
При 6-8 потоках ускорение почти не меняется (2.51x -> 2.59x),
что указывает на насыщение пропускной способности памяти.

- **Выход за пределы физических ядер**

При 16 потоках ускорение 3.04x, при 32 – 3.36x.
Рост продолжается даже после исчерпания аппаратных ядер процессора,
но эффективность падает до 19.0% и 10.5% соответственно. Это типично для memory-bound задач.

- **Статическое разбиение**

Диапазон делится на равные блоки без учёта фактической загрузки потоков.
На гибридной архитектуре быстрые P-ядра простаивают после завершения своего блока, ожидая медленные E-ядра.

- **Оптимальная конфигурация**

5-6 потоков – лучший баланс ускорения (2.50x-2.51x) и эффективности (50.0%-41.8%).

## 9. Выводы

- Ручное управление потоками через `std::thread` обеспечивает ускорение до **3.36x** (32 потока),
однако практический интерес представляет диапазон до физических ядер: **2.51x** при 6 потоках с эффективностью 41.8%.

- **Статическое разбиение**

Нет адаптации к гибридной архитектуре процессора: быстрые P-ядра простаивают, ожидая завершения E-ядер.

- **1D-разбиение по линейному индексу**

Разбиение соответствует физической структуре данных (одномерный массив) и не создаёт дополнительных
накладных расходов на адресацию.
