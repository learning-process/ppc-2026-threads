## 1. Контекст

Данная реализация использует OpenMP для параллельной обработки пикселей изображения.
Цель – получить ускорение вычислений по сравнению с последовательной версией `SEQ`,
задействовав несколько ядер процессора с минимальными изменениями в коде.
Из исходного алгоритма `SEQ` перенесены все функции.

## 2. Постановка задачи

Задача полностью идентична задаче, описанной в [отчете SEQ](../seq/report.md#2-постановка-задачи).

### Эталонное значение времени (baseline) для сравнения

- `T(seq, task_run)` = **0.2021 с**
- `T(seq, pipeline)` = **0.2086 с**

## 3. Базовый алгоритм

Базовый алгоритм идентичен [последовательной версии](../seq/report.md#3-базовый-алгоритм):
вложенный цикл с применением ядра Гаусса 3×3 к каждому пикселю.

Отличие – введение директивы OpenMP для автоматического распараллеливания.

## 4. Схема распараллеливания

### Область распараллеливания

Двумерный цикл: `for (row = 0; row < img_height; row++) { for (col = 0; col < img_width; col++) { ... } }`.

### Директива OpenMP

```cpp
// Фрагмент файла: omp/src/ops_omp.cpp
#pragma omp parallel for collapse(2) schedule(static) default(none) \
    shared(pixel_data, img_width, img_height, output, kKernel, kSum)
  for (int row = 0; row < img_height; row++) {
    for (int col = 0; col < img_width; col++) {
      // Вычисление гауссова размытия для одного пикселя
    }
  }
```

### Параметры директивы

- `collapse(2)` объединяет два внешних вложенных цикла в одно пространство итераций
размером `img_height × img_width`. Это увеличивает гранулярность параллелизма и улучшает балансировку нагрузки.
- `schedule(static)` – итерации делятся на непрерывные блоки равного размера и распределяются между потоками до
начала выполнения. Это оптимальный выбор для данной задачи, так как объём работы на каждой итерации одинаков
(9 операций), и нет необходимости в динамическом перераспределении.
- `default(none)` требует явного указания области видимости для всех переменных,
что предотвращает случайные ошибки доступа.

### Распределение данных

- `shared`: входное (`pixel_data`) и выходное (`output`) изображения,
параметры (`img_width`, `img_height`) и константы (`kKernel`, `kSum`).
Доступ к ним осуществляется на чтение (кроме `output`, куда каждый поток пишет в свою уникальную область).
- `private (неявно)`: индексы циклов `row`, `col`, переменная `acc`.
Локальная переменная `acc` внутри параллельной области автоматически становится приватной для каждого потока.

### Синхронизация

Явная синхронизация отсутствует. Каждый поток пишет в уникальные ячейки выходного вектора `output`, гонка данных
исключена. Неявный барьер в конце параллельной области гарантирует завершение всех потоков перед выходом.

### Анализ результатов

- **Накладные расходы OpenMP**

При 1 потоке время (0.3194 с) в 1.6 раза хуже SEQ (0.2021 с) из-за создания параллельной области,
неявного барьера и управления пулом потоков.

- **Рост ускорения**

Почти линейный рост до 5 потоков (пик эффективности 3.14x при 64.9%).
При 6-10 потоках ускорение колеблется в диапазоне 2.62-3.08x – рост прекращается из-за насыщения
пропускной способности памяти. Спад при 8 потоках (2.62x) указывает на возможное попадание части
потоков на E-ядра гибридного процессора, где статическое распределение нагрузки создаёт дисбаланс.

- **Нестандартные конфигурации (16 и 32 потока)**

Ускорение достигает 4.15x на 32 потоках, но эффективность падает до 12.9%.
Дальнейший рост обеспечивается не столько параллелизмом, сколько лучшей утилизацией кэша
при сверхдроблении диапазона и Hyper-Threading, однако цена в виде падения эффективности
делает такие конфигурации неоправданными для данной задачи.

- **Memory wall**

Эффективность падает с 63.0% (1 поток) до 28.0% (10 потоков). Задача **memory-bound**:
9 чтений на 1 запись при простых вычислениях. Пропускная способность памяти – основное узкое место после 5 потоков.

## 9. Выводы

### 1. Достигнутое ускорение

OpenMP позволяет ускорить обработку изображения 7680×4320
в 3.14 раза на 5 потоках и до 4.15 раза на 32 потоках относительно последовательной версии.

### 2. Оптимальная конфигурация

5 потоков обеспечивают наилучший баланс между ускорением (3.14x) и эффективностью (62.8%).
Дальнейшее наращивание потоков даёт минимальный прирост ускорения при резком падении эффективности.

### 3. Характер масштабирования

Почти линейный рост до 5 потоков подтверждает, что задача хорошо параллелизуется. Прекращение роста после
5-6 потоков указывает на достижение предела пропускной способности памяти.

### 4. Простота реализации

Ускорение получено добавлением одной директивы `#pragma omp parallel for collapse(2)` без изменения алгоритмической
логики.
