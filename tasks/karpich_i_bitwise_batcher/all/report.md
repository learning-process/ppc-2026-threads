## 1. Контекст

Данная версия алгоритма ориентирована на гибридные кластерные системы с распределенной и
общей памятью. Использование MPI позволяет распределить данные по узлам кластера, а внутри
каждого узла вычислительная нагрузка распараллеливается с помощью OpenMP.

## 2. Постановка задачи

Требуется отсортировать вектор целых 32-битных знаковых чисел с применением гибридной схемы.
Baseline — последовательная версия алгоритма (SEQ).

## 3. Базовый алгоритм

Последовательная логика включает локальную поразрядную сортировку. В распределенной версии
к этому добавляются коммуникации (рассылка и сбор данных).

## 4. Межпроцессная схема

- **Роли rank-ов:** Процесс с рангом 0 выступает в качестве управляющего узла.
  Он хранит исходный массив и генерирует сеть компараторов Бэтчера.
- **Распределение данных:** Исходный массив равномерно разбивается и пересылается на все
  вычислительные узлы с помощью `MPI_Scatterv`. Каждый процесс получает локальный
  блок. В силу специфики сети Бэтчера для работы компараторов часто требуется доступ ко всему
  массиву. Поэтому после локальной сортировки блоков процессы собирают отсортированные части
  в единый массив функцией `MPI_Allgatherv`.
- **Применение компараторов:** На каждом шаге слияния Бэтчера процессы распределяют работу
  по слою компараторов. Для получения глобально консистентного состояния массива
  выполняется `MPI_Allreduce` или `MPI_Allgatherv` после каждого слоя/группы слоев.
- Для барьерной синхронизации процессов перед замером времени используется `MPI_Barrier`.

## 5. Внутрипроцессная схема

- Внутри каждого rank-а для распараллеливания циклов применяется технология OpenMP.
- Локальная поразрядная сортировка, полученная через `MPI_Scatterv`, дополнительно ускоряется
  потоками с помощью директивы `#pragma omp parallel for` .
- Обход локального куска слоя компараторов также распределяется между OpenMP-потоками узла.
- Объединение локальных результатов осуществляется неявно по завершении `omp parallel` блоков
  перед вызовом MPI коммуникаций.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
Узким местом реализации является интенсивный сетевой обмен. Однако, код демонстрирует классическую
парадигму MPI: выделение активного коммуникатора, рассылка данных (`Scatter`), локальное
вычисление и сбор данных (`Gather`).

Фрагмент из `RunImpl`, показывающий межпроцессный обмен:

```cpp
  MPI_Comm active_comm = MPI_COMM_NULL;
  int color = (rank < num_tasks) ? 1 : MPI_UNDEFINED;
  MPI_Comm_split(MPI_COMM_WORLD, color, rank, &active_comm);

  if (active_comm != MPI_COMM_NULL) {
    int chunk_size = padded / num_tasks;
    std::vector<int> local_data(chunk_size);

    MPI_Scatter(data_.data(), chunk_size, MPI_INT, local_data.data(), chunk_size, MPI_INT, 0, active_comm);
    RadixSort(local_data); //внутри вызывается #pragma omp parallel для ускорения
    MPI_Gather(local_data.data(), chunk_size, MPI_INT, data_.data(), chunk_size, MPI_INT, 0, active_comm);

    if (rank == 0) {
      MergeChunks(data_, chunk_size, padded);
      data_.resize(n);
    }
    MPI_Comm_free(&active_comm);
  }
```

Этот листинг показывает, как создается суб-коммуникатор (`active_comm`) только для тех
процессов, которые действительно участвуют в вычислениях. Нулевой ранг раздает куски данных,
каждый процесс локально их сортирует (задействуя внутри OpenMP потоки), после чего
отсортированные блоки собираются обратно на корневом процессе для финального слияния Бэтчера.

## 7. Проверка корректности

Сравнение проводилось с эталонной версией (SEQ) на малых и средних размерах массивов.
Дополнительно проверялась согласованность результата между rank-ами перед окончательным

пройдены.

## 10. Выводы

Для алгоритма поразрядной сортировки со слиянием Бэтчера распределенная схема
крайне неэффективна на малых и средних объемах данных. Вычислительная польза от
распараллеливания полностью нивелируется издержками на частые MPI коммуникации
(`Scatter`, `Gather`, `Allgather`), необходимые для работы сети компараторов.
Гибридная схема оправдана только на сверхкрупных массивах, когда время пересылки
окажется меньше времени вычислений.
