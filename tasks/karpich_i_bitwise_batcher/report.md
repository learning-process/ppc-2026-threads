# Поразрядная сортировка для целых чисел с четно-нечетным слиянием Бэтчера

- Студент: Карпич Иван Валерьевич, группа 3823Б1ПР4
- Вариант: 18
- Локальные отчеты: `seq/report.md`, `omp/report.md`, `tbb/report.md`, `stl/report.md`, `all/report.md`

## 1. Введение

Поразрядная сортировка является эффективным алгоритмом сортировки без сравнений
за линейное время. В данной задаче классическая поразрядная сортировка распараллеливается
с помощью декомпозиции на блоки и последующего объединения блоков с помощью параллельной
сети четно-нечетного слияния Бэтчера. Это позволяет сравнить подходы работы с потоками
и процессами на алгоритме, обладающем специфическими требованиями к барьерной синхронизации.

## 2. Единая постановка задачи

Задан вектор целых 32-битных знаковых чисел. Требуется отсортировать этот вектор по
возрастанию. Для сортировки внутри отдельных блоков (выделяемых потокам или процессам)
должна применяться поразрядная сортировка (побайтово). Для глобального объединения блоков
должна применяться сеть слияния Бэтчера. Критерием корректности является совпадение
отсортированного массива с эталоном `std::sort`.

## 3. Единая методика эксперимента

- **Окружение**: Опыты проводились на Apple M2 (8 ядер), 16 GB RAM, OS macOS/Linux,
  компилятор GCC 13.3.0, сборка `Release` (C++20, CMake).
- **Данные**: Данные генерировались случайно, представляя собой вектора, включающие
  различные диапазоны чисел, в том числе отрицательные.
- Размеры задач подбирались автоматически тестирующим фреймворком `ppc_perf_tests`.
- Замеры выполнялись для числа вычислителей `p` в {1, 2, 4, 6}.
- Ускорение (`Speedup`) считается как `T_seq(1) / T_backend(p)`.
- Эффективность (`Efficiency`) вычисляется как `(Speedup / p) * 100%`.
- Все числа представляют собой агрегированные (усредненные) результаты повторных
  прогонов внутри `ppc_perf_tests`.

## 4. Сводка корректности

Все `backend`-версии детерминированно прошли функциональные тесты `Google Test`. Ошибок с
горизонтами памяти и гонок данных выявлено не было.
Тесты включали сортировку пустых массивов, массивов из 1 элемента, массивов с дубликатами и
знаковыми числами. Использование знаковых чисел потребовало специального сдвига на старшей
итерации Radix Sort.

Общая таблица времени работы (в секундах) для режима `task_run`. Для `all` под `p`

| SEQ        | 0.7766   | -        | -        | -        | 1.0             | -                   |
| MPI        | 0.7979   | 0.8210   | 1.4724   | 1.4708   | 0.527           | 13.2%               |
| OMP        | 0.0470   | 0.0311   | 0.0276   | 0.0337   | 28.14           | 703.5%              |
| TBB        | 0.9875   | 0.8366   | 0.7052   | 0.6857   | 1.101           | 27.5%               |

## 6. Интерпретация различий

- **SEQ** показывает ровный базовый уровень линейного времени на локальных циклах без
  сетевого обмена и барьеров.
- **OMP** показывает высокую производительность на этой задаче, так как
  `#pragma omp parallel for` эффективно сработал на локальных циклах поразрядной сортировки.
- **TBB** и **STL** страдают от overhead-а на барьерную синхронизацию (в `STL` — явные
  барьеры, в `TBB` — неявные через завершение задач) при переходе между слоями
  сети Бэтчера. Построение дерева задач на тысячах слоев сбивает масштабируемость,
  поэтому ускорение с трудом достигает `~1.13`.
- **ALL (MPI)** показывает резкую деградацию при добавлении процессов. Причина:
  сеть Бэтчера требует обмена элементами по всему массиву. Это вынуждает процессы постоянно
  передавать массивы сообщений через `MPI_Allgather`, что во много раз дороже самих
  арифметических сравнений.

## 7. Репродуцируемость

- **Сборка**:

  ```bash
  cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
  cmake --build build --parallel
  ```

- **Запуск тестов (потоки)**:

  ```bash
  $env:PPC_NUM_THREADS="k"; .\build\bin\ppc_perf_tests --gtest_filter="*karpich_i_bitwise_batcher*"
  где k - количество потоков
  ```

- **Запуск тестов (процессы)**:

  ```bash
  mpiexec -n k .\build\bin\ppc_perf_tests --gtest_filter="*karpich_i_bitwise_batcher*"
  где k - количество процессов
  ```

## 8. Заключение

Наилучшую производительность на системах с общей памятью показала технология OpenMP.
Реализация алгоритма сортировки с сетью Бэтчера для распределенных систем (MPI)
оказалась малоэффективной из-за высокой связности алгоритма (плотности барьеров
и необходимости коммуникации на каждом слое слияния). Для MPI требуются алгоритмы с
крупноблочной обработкой данных и редкими фазами синхронизации.
