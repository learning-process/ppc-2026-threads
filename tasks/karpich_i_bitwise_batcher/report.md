## 1. Введение

Поразрядная сортировка (Radix Sort) является алгоритмом сортировки без сравнений, который позволяет сортировать числа за
линейное время, что выгодно отличает его от традиционных сортировок со временем выполнения `O(N log N)`. Целью данной
работы является разработка параллельной версии алгоритма поразрядной сортировки с использованием четно-нечетного слияния
Бэтчера для распределенных и многопоточных вычислительных сред. Ожидаемый результат - получение ускорения при запуске
программы на нескольких вычислительных ядрах и процессах по сравнению с последовательной версией.

## 2. Постановка задачи

Задан большой вектор целых 32-битных чисел (обозначение его длины как `N`). Требуется отсортировать этот вектор по
возрастанию. Ограничения и условия:

1. Для сортировки элементов внутри отдельных блоков использовать поразрядную сортировку, проходящую по каждому байту
   числа от младшего к старшему (Least Significant Digit). В качестве устойчивой сортировки внутри должна использоваться
   сортировка подсчетом.
2. Для объединения отсортированных блоков в многопоточной/многопроцессорной реализации применять четно-нечетное слияние
   Бэтчера.
3. Алгоритм должен корректно работать с любыми размерностями массива.
4. Входные и выходные данные представлены массивами (векторами).

## 3. Описание последовательного алгоритма

Последовательный алгоритм представляет собой классическую поразрядную сортировку целых чисел по основанию 256. Так как
размер 32-битного числа составляет 4 байта, алгоритм включает ровно 4 итерации:

1. Заводится буферный массив такого же размера для промежуточной записи результата.
2. Цикл по байтам: от 0 до 3.
3. На каждой итерации:
    - Создается массив счетчиков частот (размера 256), заполняемый нулями.
    - Выполняется проход по исходному массиву; для каждого числа вычисляется значение текущего байта, и соответствующий
      счетчик инкрементируется.
    - Массив частот преобразуется в массив префиксных сумм, чтобы индекс счетчика указывал на стартовую позицию для
      вставки элементов с данным значением байта.
    - Выполняется обратный проход по исходному массиву для расстановки элементов в буферный массив в правильном порядке.
    - Буферный массив становится исходным для следующей итерации. Так как последовательная сортировка полностью
      сортирует весь массив, слияние по алгоритму Бэтчера для нее не требуется. Вычислительная сложность алгоритма
      составляет `O(B * N)`, где `B` - количество байт.

## 4. Схема распараллеливания

Декомпозиция задачи для получения параллельного алгоритма осуществляется путем разделения данных на равные блоки,
локальной обработки и итогового слияния.

1. **Декомпозиция.** Входной массив разбивается на `P` подмассивов, где `P` - количество потоков или процессов.
2. **Локальная сортировка.** Каждый узелосуществляет локальную поразрядную сортировку своего подмассива алгоритмом из
   пункта 3 абсолютно независимо от остальных.
3. **Обмен и планирование слияния (сеть Бэтчера).** По завершении сортировки начинается фаза слияния отсортированных
   блоков. Строится послойный набор парных компараторов Бэтчера. Передача данных и применение этих компараторов зависит
   от используемой технологии реализации:
    - **Гибридный подход (MPI + OpenMP)**: Данные кросс-узлово распределяются с помощью парадигмы передачи сообщений
      (используются функции `MPI_Scatter` и `MPI_Gather`). Внутри каждого отдельного узла/процесса локальная поразрядная
      сортировка и применение компараторов на каждом слое сети Бэтчера дополнительно ускоряются за счет потоков
      процессора с использованием директив `#pragma omp parallel for`. По окончании всех шагов Бэтчера нулевой процесс
      собирает отсортированные части в единый конечный массив.

    - **OpenMP (OMP)**: Так как используется общая память, физической пересылки данных (на уровне сети) нет. Потокам
      выделяются соответствующие диапазоны индексов. При обходе сети Бэтчера выполняемые на текущем слое сравнения
      блоков распределяются между доступными потоками с помощью `#pragma omp parallel for`. Синхронизация между
      последовательными слоями сети осуществляется через неявные барьеры циклов `omp for`.

    - **Threading Building Blocks (TBB)**: В TBB для распараллеливания применяются алгоритмы, такие как
      `tbb::parallel_for`, работающие на основе планировщика задач с балансировкой нагрузки (work-stealing). Индексы
      разбиваются на блоки классом `tbb::blocked_range`, позволяя равномерно нагружать потоки как при подсчете частот,
      так и при итерации по списку компараторов Бэтчера. Синхронизация реализуется неявно через завершение вызовов
      параллельных конструкций между шагами сети.

    - **C++ Standard Thread Library (STL)**: Выполняется контроль пула потоков через классы `std::thread`. Массив
      разделяется на равные отрезки. Каждый созданный поток обрабатывает элементы своего отрезка во время локальной
      сортировки. На стадии слияния пары для сравнения также балансируются между потоками по индексам. Синхронизация
      потоков между слоями алгоритма Бэтчера осуществляется с помощью барьеров, реализуемых средствами стандартной
      библиотеки std::thread.

В результате после успешного прохождения всех слоев потоками или процессами локальные части складываются в один большой
полностью отсортированный вектор.

## 5. Детали реализации

- **Корректность знаковых чисел**. Так как числа знаковые, то при сортировке их побайтово на самой старшей итерации
  необходимо учитывать знак. Сделать это можно либо сместив все числа на константу при сортировке, либо применяя
  специальный сдвиг к старшему знаковому биту.
- **Нечетное число потоков**. Сеть Бэтчера в классическом виде строится для числа потоков/узлов, являющегося степенью
  двойки. В практической реализации может использоваться обобщение на произвольное число потоков или создание пустых
  блоков, которые затем отфильтровываются.
- **Потребление памяти**. Для локальных сортировок нужно выделение дополнительного вектора того же размера, что может
  быть критично.

## 8. Заключение

В ходе данной лабораторной работы были реализованы и протестированы последовательная и несколько параллельных версий
поразрядной сортировки с использованием сети четно-нечетного слияния Бэтчера: MPI, OpenMP, TBB и STL.

Анализ полученных результатов позволяет сделать следующие выводы:

1. **OpenMP (OMP)**: Эта реализация показала высокую производительность по сравнению с последовательной версией (SEQ),
   превосходя ее даже на одном вычислительном потоке более чем в 16 раз (`0.047 с` против `0.776 с`). Это может
   свидетельствовать об использовании в OMP-версии более эффективного вложенного алгоритма сортировки, либо об
   отсутствии ряда накладных расходов. При дальнейшем увеличении числа потоков (до 4) наблюдается дополнительное
   ускорение (время падает с `0.047 с` до `0.027 с`), но на 6 потоках время начинает немного возрастать.
2. **Threading Building Blocks (TBB) и стандартные потоки (STL)**: Эти технологии с общей памятью показывают ожидаемое
   поведение. На малом числе потоков (1-2 потока) накладные расходы на диспетчеризацию задач, создание потоков и
   выделение памяти приводят к тому, что время работы сопоставимо с последовательной версией (а в случае TBB даже больше
   на одном потоке). Однако при увеличении числа потоков до 4-6 наблюдается положительное параллельное ускорение по
   сравнению с базовой реализацией (`Speedup` ~ `1.13` для TBB и `1.05` для STL). Небольшой уровень ускорения и низкая
   эффективность объясняются тем, что сеть слияния Бэтчера требует частой барьерной синхронизации при переходе между
   слоями, что существенно снижает общий потенциал параллелизма.
3. **Распределенная память (гибридная MPI + OpenMP)**: Реализация MPI показывает значительное проседание в
   производительности при увеличении числа процессов (от `0.79 с` на 1 процессе до `1.47 с` на 4 и 6 процессах). Это
   вызвано тем, что сеть слияния Бэтчера требует крайне интенсивного сетевого обмена (полной пересылки массивов или их
   частей между всеми узлами/процессами). Для данного объема элементов вычислительная польза от параллельной локальной
   сортировки блоков полностью нивелируется огромными затратами времени на коммуникации `MPI_Scatter/MPI_Gather`

Таким образом, для алгоритма поразрядной сортировки с подходом слияния Бэтчера технологии с общей памятью оказались
более предпочтительными благодаря быстрому обмену между потоками через кэш и оперативную память. Для MPI реализации
необходимо избегать частых коммуникаций или применять подходы с существенно большим объемом данных на узел для
перекрытия времени связи вычислениями.

## Приложение

Ниже представлены ключевые фрагменты исходного кода из гибридной реализации MPI + OpenMP (`ops_all.cpp`),
демонстрирующие использование этих технологий:

**1. Подсчет частот с использованием OpenMP (`CountingPass`)** Внутри каждого распределенного блока сортировка
выполняется параллельно:

```cpp
void CountingPass(std::vector<int> &arr, std::vector<int> &buffer, int n, int shift) {
  std::vector<int> count(256, 0);
  int num_threads = omp_get_max_threads();
  std::vector<std::vector<int>> local_counts(num_threads, std::vector<int>(256, 0));

#pragma omp parallel default(none) shared(arr, shift, local_counts, n)
  {
    int tid = omp_get_thread_num();
#pragma omp for
    for (int i = 0; i < n; i++) {
      local_counts[tid][(arr[i] >> shift) & 0xFF]++;
    }
  }

  for (int ti = 0; ti < num_threads; ti++) {
    for (int i = 0; i < 256; i++) {
      count[i] += local_counts[ti][i];
    }
  }
  // следующий блок кода...
}
```

**2. Применение компараторов сети Бэтчера внутри блоков (`ApplyComparatorNetwork`)** Перестановка элементов выполняется
параллельно потоками на независимых индексах:

```cpp
void ApplyComparatorNetwork(std::vector<int> &arr, const std::vector<std::vector<std::pair<int, int>>> &levels) {
  for (int lvl = static_cast<int>(levels.size()) - 1; lvl >= 0; lvl--) {
    const auto &level = levels[lvl];
    int level_size = static_cast<int>(level.size());
#pragma omp parallel for default(none) shared(arr, level, level_size)
    for (int i = 0; i < level_size; ++i) {
      int aa = level[i].first;
      int bb = level[i].second;
      if (arr[aa] > arr[bb]) {
        std::swap(arr[aa], arr[bb]);
      }
    }
  }
}
```
