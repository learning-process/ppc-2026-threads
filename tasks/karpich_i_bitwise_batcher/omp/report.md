## 1. Контекст

В данной задаче используется OpenMP для распараллеливания поразрядной сортировки и
последующего четно-нечетного слияния Бэтчера. OpenMP выбран как стандарт для систем
с общей памятью, позволяющий распараллелить вычисления без явного управления потоками.

## 2. Постановка задачи

Требуется отсортировать вектор целых 32-битных знаковых чисел по возрастанию с использованием
поразрядной сортировки и слияния Бэтчера. Последовательная версия считается эталоном
корректности и базовой линией производительности. Подробное описание в `seq/report.md`.

## 3. Базовый алгоритм

Последовательная версия выполняет 4 прохода поразрядной сортировки (по числу байт) над всем
массивом, вычисляя счетчики и перенося элементы в буферный массив.

## 4. Схема распараллеливания

Входной массив разбивается на равные подмассивы (по числу потоков `OMP_NUM_THREADS`).
Каждый поток независимо сортирует свой блок локальной поразрядной сортировкой:

- Для подсчета частот заводится массив локальных счетчиков (размера `num_threads x 256`),
  каждый поток инкрементирует свою строку счетчиков. Переменные счетчиков являются локальными
  (`private` на уровне логики или `shared` двумерный массив без пересечений индексов),
  поэтому гонки данных не возникает. Для редукции счетчиков используются вложенные циклы.
- После локальной сортировки выполняется слияние блоков сетью Бэтчера. Сеть генерирует пары
  компараторов по слоям.
- Перестановка элементов (компараторы на одном слое) выполняется независимо: применяется
  директива `#pragma omp parallel for default(none) shared(...)`. Так как компараторы на одном
  слое оперируют непересекающимися парами индексов, гонки данных исключены.
- Неявный барьер в конце области `#pragma omp for` обеспечивает синхронизацию между слоями
  сети Бэтчера: потоки не перейдут к следующему слою, пока все сравнения на текущем не завершатся.
- Для балансировки нагрузки выбран `schedule(static)`, поскольку работа по перестановке
  элементов на слое равномерно распределена.

## 5. Детали реализации

Файлы: `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`
Изменения относительно SEQ:

- В `RunImpl` добавлено разбиение массива на блоки.
- Локальная сортировка в блоках вынесена в отдельный метод, который распараллеливается с помощью
  `#pragma omp parallel for`.
- Реализовано применение сети компараторов с многопоточной перестановкой.

Ниже представлен листинг, показывающий, как именно `OpenMP` применяется к локальной сортировке
отдельных блоков исходного массива. Здесь каждый поток обрабатывает свой `chunk` данных,
поэтому гонки данных не происходит.

```cpp
  int chunk = (n + num_thr - 1) / num_thr;
  int padded = chunk * num_thr;
  data_.resize(padded, std::numeric_limits<int>::max());

  auto &data_ref = data_;
#pragma omp parallel for default(none) shared(data_ref, chunk, num_thr) num_threads(num_thr)
  for (int tid = 0; tid < num_thr; tid++) {
    RadixSortRange(data_ref, tid * chunk, (tid + 1) * chunk);
  }
```

Директива `default(none)` заставляет программиста явно указывать разделяемые (`shared`) и
приватные переменные, что предотвращает случайные гонки. Переменная `tid` приватная по
умолчанию как индекс цикла.

## 6. Проверка корректности

Корректность версии OMP проверялась сравнением с SEQ для всех функциональных тестов, включая
минимальные и граничные размеры, а также массивы с отрицательными числами. Расхождения в
выходных данных не наблюдались; дополнительно была проверена устойчивость результата при
числе потоков 1, 2, 4 и 6.

## 9. Выводы

OpenMP дал значительный выигрыш за счет оптимизации работы с памятью в локальных блоках
и распределения компараторов. Однако выигрыш от распараллеливания ограничен overhead-ом на
синхронизацию: сеть Бэтчера требует частой неявной барьерной синхронизации при переходе
между многочисленными слоями компараторов.
