\## 1. Контекст



Задача относится к обработке изображений, представленных одномерным массивом `float` в построчном порядке. Для каждого пикселя выполняется сглаживание separable-вариантом ядра Гаусса 3x3: сначала горизонтальный проход с весами `\[1, 2, 1]`, затем вертикальный проход с теми же весами. На границах изображения используется нормировка по доступным соседям: для первого/последнего столбца делитель равен `3`, для внутренних пикселей — `4`; аналогично обрабатываются первая и последняя строки.



Горизонтальное разбиение означает, что независимой единицей работы является строка изображения. Каждый проход записывает разные строки выходного буфера, поэтому синхронизация между рабочими потоками внутри одного прохода не требуется; барьер нужен только между горизонтальным и вертикальным проходами, так как второй проход читает временный массив целиком.



\## 2. Постановка задачи



\*\*Вход:\*\* кортеж `(image, width, height)`, где `image` — вектор яркостей пикселей типа `float`, `width` — ширина изображения, `height` — высота изображения.



\*\*Выход:\*\* вектор `float` размера `width \* height` со сглаженным изображением.



\*\*Ограничения и корректность входа:\*\*



\- изображение не должно быть пустым;

\- ожидается согласованность размеров: `image.size() == width \* height`;

\- для корректного обращения к соседям в текущей реализации практически предполагаются `width >= 2` и `height >= 2`;

\- пиксели хранятся в row-major порядке: элемент `(x, y)` находится по индексу `y \* width + x`.



\## 3. Базовый алгоритм



1\. Выделить временный массив `temp` размера `width \* height`.

2\. Для каждой строки выполнить горизонтальную свертку:

&#x20;  - левый край: `(2 \* p\[x] + p\[x + 1]) / 3`;

&#x20;  - правый край: `(p\[x - 1] + 2 \* p\[x]) / 3`;

&#x20;  - внутренний пиксель: `(p\[x - 1] + 2 \* p\[x] + p\[x + 1]) / 4`.

3\. Для каждой строки результирующего массива выполнить вертикальную свертку по `temp`:

&#x20;  - верхний край: `(2 \* temp\[y] + temp\[y + 1]) / 3`;

&#x20;  - нижний край: `(temp\[y - 1] + 2 \* temp\[y]) / 3`;

&#x20;  - внутренний пиксель: `(temp\[y - 1] + 2 \* temp\[y] + temp\[y + 1]) / 4`.

4\. Вернуть заполненный выходной массив.



\*\*Сложность:\*\* `O(width \* height)`, так как каждый пиксель обрабатывается константное число раз.  

\*\*Память:\*\* `O(width \* height)` для временного массива `temp` и `O(width \* height)` для выходного массива.



\## 4. Детали реализации



В каталоге задачи представлены пять реализаций одного алгоритма:



\- `seq` — базовая последовательная версия с двумя вложенными циклами по строкам и столбцам;

\- `omp` — OpenMP-версия, распараллеливающая оба прохода директивой `parallel for` по строкам;

\- `tbb` — TBB-версия, использующая `tbb::parallel\_for` по диапазону строк;

\- `stl` — версия на стандартной библиотеке C++ с `std::async`, `std::future` и циклическим распределением строк;

\- `all` — комбинированная версия: TBB для горизонтального прохода и OpenMP для вертикального прохода.



Во всех вариантах входной вектор копируется в объект задачи при создании, выходной буфер очищается, затем в `RunImpl()` выделяются `out` и `temp`. Математические функции фильтрации отличаются только именами с суффиксом технологии, что упрощает проверку эквивалентности реализаций.



\## 5. Проверка корректности



Корректность проверяется функциональными тестами Google Test. Тесты строят детерминированные квадратные изображения размеров `4`, `8`, `16`, `32` и `60/64`, вычисляют эталон последовательной двухпроходной сверткой и сравнивают каждый элемент результата с допустимой погрешностью `1e-5`. Производительные тесты дополнительно используют изображение `512 x 512` и сравнивают результат с тем же эталонным алгоритмом с погрешностью `1e-4`.



\## 6. Экспериментальная среда и методика



Измерения предполагается выполнять через штатный исполняемый файл `ppc\_perf\_tests` в конфигурации Release. Для параллельных режимов число рабочих потоков задается настройками фреймворка PPC и/или переменными окружения, после чего фиксируется время выполнения чистой стадии `RunImpl()` на детерминированном изображении `512 x 512`.



\*\*Определения метрик:\*\*



\- `Time` — среднее время выполнения алгоритма в секундах;

\- `Speedup = T\_seq / T\_p`, где `T\_seq` — время последовательной версии, `T\_p` — время параллельной версии;

\- `Efficiency = Speedup / Workers`.

\## 8. Выводы



Алгоритм хорошо подходит для горизонтального разбиения, потому что строки в каждом проходе записываются независимо. Последовательная версия служит эталоном, OpenMP и TBB используют готовые планировщики потоков, STL-версия явно создает асинхронные задачи, а ALL-версия показывает совместимость TBB и OpenMP при фазовом разделении вычислений. На больших изображениях ожидается ускорение параллельных версий, ограниченное пропускной способностью памяти и накладными расходами на запуск задач/барьеры.
