## 4. Детали реализации

OpenMP-версия сохраняет структуру базового алгоритма,
но распараллеливает оба прохода директивой `#pragma omp parallel for`.
Внешний цикл идет по строкам изображения, поэтому каждая итерация
записывает независимый диапазон элементов `temp` или `out`.

Между двумя директивами OpenMP находится неявный барьер: вертикальный
проход начинается только после полного заполнения временного массива.
В директивах используется `default(none)` и явное перечисление общих
переменных, что снижает риск неявного захвата данных.

## 5. Проверка корректности

Корректность проверяется функциональными тестами Google Test.
Тесты строят детерминированные квадратные изображения размеров
`4`, `8`, `16`, `32` и `60/64`, вычисляют эталон последовательной
двухпроходной сверткой и сравнивают каждый элемент результата с
допустимой погрешностью `1e-5`. Производительные тесты дополнительно
используют изображение `512 x 512` и сравнивают результат с тем же
эталонным алгоритмом с погрешностью `1e-4`.

## 6. Экспериментальная среда и методика

Измерения предполагается выполнять через штатный исполняемый файл
`ppc_perf_tests` в конфигурации Release. Для параллельных режимов
число рабочих потоков задается настройками фреймворка PPC и/или
переменными окружения, после чего фиксируется время выполнения чистой
стадии `RunImpl()` на детерминированном изображении `512 x 512`.

**Определения метрик:**

- `Time` — среднее время выполнения алгоритма в секундах;
  - `Speedup = T_seq / T_p`, где `T_seq` — время последовательной версии,
   `T_p` — время параллельной версии;
  - `Efficiency = Speedup / Workers`.
