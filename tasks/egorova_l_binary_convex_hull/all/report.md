## 1. Введение

Для обработки изображений огромных размеров со множеством независимых объектов часто
требуется распределенная архитектура вычислений. В данном отчете описывается гибридный
алгоритм распараллеливания, сочетающий распределенную память (MPI) для масштабирования
по узлам кластера и общую память (OpenMP) для максимальной утилизации ядер внутри каждого
отдельного вычислительного узла.

## 2. Постановка задачи

**Формальная постановка задачи:**
Дано бинарное изображение `width * height`. Требуется найти все независимые по 4-связности
компоненты и построить для них выпуклые контуры (выпуклые оболочки), исключив переполнения
типов данных.

## 3. Базовый алгоритм (последовательный)

Алгоритм состоит из последовательного выделения компонент (обход графа в ширину/глубину)
и построения их контуров с помощью лексикографической сортировки и алгоритма Эндрю.
Временная сложность построения контура для $N$ точек составляет $O(N \log N)$.

## 4. Схема распараллеливания (MPI + OpenMP)

Используется гибридная схема декомпозиции, решающая проблему разрывов фигур на границах
блоков:

1. **Распределение данных (MPI):** Процесс-мастер рассылает метаданные и само изображение
   (`MPI_Bcast`) всем остальным узлам.
2. **Блочно-строчная декомпозиция:** Изображение логически делится на горизонтальные полосы
   (`chunk = height / p_count`).
3. **Локальный поиск с выходом за границы:** Каждый MPI-процесс обходит свою локальную
   полосу. Алгоритм BFS может «выходить» за пределы локальной полосы по глобальным индексам
   для извлечения полных фигур, пересекающих границы блоков.
4. **Многопоточность в узле (OMP):** Найденные локальные компоненты обрабатываются внутри
   каждого процесса через `#pragma omp parallel for schedule(dynamic)`.
5. **Сборка (MPI_Allgatherv):** Построенные локальные оболочки сериализуются в одномерный
   буфер и собираются со всех процессов.
6. **Слияние (DSU):** На каждом узле применяется алгоритм системы непересекающихся множеств
   (Disjoint Set Union). Если Bounding Box локальных оболочек пересекаются, их точки
   объединяются, и оболочка перестраивается.

## 5. Детали реализации

- **Структура:** Класс `BinaryConvexHullALL` (`ops_all.cpp`).
- **Сериализация:** Функция `SerializeHulls` преобразует вложенную структуру
  `std::vector<std::vector<Point>>` в плоский целочисленный массив `std::vector<int>`
  для эффективной передачи по сети средствами MPI.

## 7. Результаты и обсуждение

### 7.1. Корректность

Алгоритм успешно решает краевую проблему распределенной обработки изображений (разрыв
фигур на границах блоков). Применение алгоритма DSU гарантирует корректное обнаружение
коллизий и склеивание разорванных кусков фигур в единый монолитный контур. Тесты GTest
успешно пройдены.

### 7.2. Сводная производительность

Результаты масштабирования гибридной версии (MPI + OMP) на основе метрики `task_run`
(базовое последовательное время `0.00915` сек):

| Режим   | MPI / OMP | Общее число ядер | Время, сек | Ускорение | Эффективность |
|---------|-----------|------------------|------------|-----------|---------------|
| seq     | 1 / 1     | 1                | 0.00915    | 1.00      | N/A           |
| mpi+omp | 2 / 2     | 4                | 0.00491    | 1.86      | 46.5%         |
| mpi+omp | 4 / 2     | 8                | 0.00488    | 1.87      | 23.4%         |
| mpi+omp | 2 / 4     | 8                | 0.00642    | 1.42      | 17.7%         |

Как видно из таблицы, гибридная версия быстро упирается в «потолок» производительности.
При переходе от конфигурации (2 MPI / 2 OMP) к (4 MPI / 2 OMP) время не уменьшается,
а при (2 MPI / 4 OMP) алгоритм начинает работать медленнее.

Тестовое изображение 1500x1500 является относительно «маленьким». Время, затрачиваемое на
пересылку массивов (`MPI_Bcast`, `MPI_Allgatherv`) и последующий однопоточный этап слияния
оболочек (DSU), существенно превышает вычислительный выигрыш от распараллеливания самого
построения контуров.

**Сводный вывод по всему циклу работ:** Для задач поиска компонент связности на изображениях
среднего размера оптимальным решением являются внутриузловые пулы потоков с динамической
балансировкой (OpenMP `dynamic` или STL с атомарными счетчиками). Использование MPI
целесообразно исключительно для гигантских матриц (Out-of-Core вычисления), которые
физически не помещаются в оперативную память одного узла.
