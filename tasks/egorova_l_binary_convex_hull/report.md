## 1. Введение

В данной работе сравниваются пять реализаций алгоритма поиска выпуклых оболочек на
бинарном изображении: последовательная (SEQ), OpenMP, TBB, ручное управление потоками
(STL) и гибридная MPI+OpenMP (ALL). Задача выбрана как представительный пример нерегулярного
параллелизма: компоненты связности имеют разный размер, а значит, требуют динамической
балансировки нагрузки.

## 2. Единая постановка задачи

### Входные данные

- `width`, `height` — размеры изображения.
- `data` (`std::vector<uint8_t>`) — одномерный массив пикселей; значение `0` — фон,
  остальные значения — объект.

### Выходные данные

- `std::vector<std::vector<Point>>` — массив выпуклых оболочек каждой связной компоненты
  (список вершин, упорядоченных по контуру).

### Ограничения

- Связность пикселей определяется по 4-связной топологии.
- Вычисления векторных произведений защищены от переполнения (`int64_t`).

Критерий корректности: поэлементное лексикографическое сравнение выходных оболочек с
эталонным результатом последовательной реализации.

## 3. Единая методика эксперимента

- **CPU:** 13th Gen Intel(R) Core(TM) i5-13420H
  (8 физических ядер / 12 логических потоков, 2.10 GHz).
- **RAM:** 16.0 ГБ, скорость 4267 MT/s.
- **OS:** Linux (Docker-контейнер на базе Windows 11 64-разрядная).
- **Компилятор:** GCC / Clang, тип сборки `Release`.
- **Данные:** Синтетическое бинарное изображение 1500×1500 пикселей, сетка изолированных
  объектов 20×20 с шагом 100 пикселей.

### Метрики

- Ускорение: $S(p) = T_{seq} / T_{parallel}$
- Эффективность: $E(p) = S(p) / p \times 100\%$

### Процедура измерений

- Метрика: `task_run` (чистое время выполнения без учёта ввода/вывода).
- Базовое последовательное время: `0.00915` сек.

## 4. Сводка корректности

### Функциональные тесты (Google Test)

Покрытые случаи:

- Пустое изображение.
- Одиночная точка.
- Линия (вырожденная оболочка, 1–2 точки).
- Изолированные квадраты и прямоугольники.
- Г-образные фигуры.
- Фигуры, пересекающие границы блоков (актуально для ALL).

Все реализации успешно проходят полный набор тестов.

## 6. Интерпретация различий

| Backend | Сильные стороны                        | Слабые стороны                                    |
|---------|----------------------------------------|---------------------------------------------------|
| SEQ     | Базовый эталон, простота кода          | Не использует параллелизм                         |
| OMP     | Лучшая эффективность (до 90.5%)        | Ограничен одним узлом                             |
| TBB     | Удобный API, work-stealing планировщик | Мьютексы съедают выигрыш от параллелизма          |
| STL     | Lock-free балансировка (`std::atomic`) | Накладные расходы `std::thread` на мелких задачах |
| ALL     | Масштабирование на кластер, DSU-склейка| Высокая цена MPI-коммуникаций                     |

**Примечание по STL:** Метрика `task_run` фиксирует только `RunImpl()`. Тяжёлый
последовательный поиск компонент вынесен в `PreProcessingImpl()` и не учитывается в
замере, что и даёт кажущееся сверхлинейное ускорение на 2 потоках. Падение скорости при
4 потоках объясняется тем, что для малого числа мелких компонент накладные расходы на
создание и join `std::thread` превышают полезные вычисления.

**Примечание по TBB:** Ключевой узкой точкой стал этап параллельного сбора координат
пикселей через `tbb::blocked_range2d` с массивом `std::mutex`. Частые конкурентные
блокировки почти полностью нивелируют выигрыш от Work-Stealing планировщика.

**Примечание по ALL:** При переходе от конфигурации (2×2) к (4×2) ускорение практически
не растёт, а конфигурация (2×4) даже медленнее. Время на `MPI_Bcast` / `MPI_Allgatherv`
и однопоточный DSU-этап слияния оболочек превышает вычислительный выигрыш на данных такого
размера.

## 7. Репродуцируемость

### Команды сборки

```bash
mkdir build && cd build
cmake -DCMAKE_BUILD_TYPE=Release ..
make -j$(nproc)
```

```bash

mpirun --allow-run-as-root -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*seq*"

mpirun --allow-run-as-root -x OMP_NUM_THREADS=2 -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*omp*"
mpirun --allow-run-as-root -x OMP_NUM_THREADS=4 -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*omp*"

# TBB (2 и 4 потока)
mpirun --allow-run-as-root -x TBB_NUM_THREADS=2 -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*tbb*"
mpirun --allow-run-as-root -x TBB_NUM_THREADS=4 -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*tbb*"

# STL (2 и 4 потока)
mpirun --allow-run-as-root -x OMP_NUM_THREADS=2 -n 1 \
  ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*stl*"
mpirun --allow-run-as-root -x OMP_NUM_THREADS=4 -n 1 \
```

## 8. Заключение

**OpenMP** показал наилучшую реальную эффективность среди всех реализаций: ускорение 1.81×
при 90.5% эффективности на 2 потоках и 2.37× на 4 потоках. Динамическое планирование
`schedule(dynamic)` успешно нивелирует дисбаланс вычислительной сложности разных компонент.

**TBB** продемонстрировал наименьший прирост производительности (1.20× на 4 потоках) из-за
высоких накладных расходов на мьютексы при попиксельном сборе точек компонент. Алгоритм
остаётся корректным, но выбранная схема синхронизации неоптимальна для данной задачи.

**STL** (lock-free через `std::atomic`) показал лучшее ускорение на 2 потоках в рамках
измеряемого `RunImpl()`, однако полная картина с учётом `PreProcessingImpl()` сопоставима
с OMP. На 4 потоках накладные расходы на `std::thread` становятся узким местом.

**ALL (MPI+OMP)** эффективно решает алгоритмическую проблему склейки разорванных на
границах объектов через DSU. Однако для изображений размером 1500×1500 коммуникационные
расходы MPI перевешивают вычислительный выигрыш. Применение оправдано только при работе с
данными, не помещающимися в память одного узла.

**Общий вывод:** для задач нерегулярного параллелизма на одном узле оптимальным выбором
является OpenMP с динамическим планированием. MPI целесообразен исключительно для задач,
требующих масштабирования за пределы одного узла (Out-of-Core вычисления).
