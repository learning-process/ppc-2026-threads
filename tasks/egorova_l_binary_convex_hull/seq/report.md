# Поиск выпуклой оболочки на бинарном изображении

## 1. Введение

Задача нахождения выпуклой оболочки (Convex Hull) для объектов на бинарном изображении
является классической в области компьютерного зрения и цифровой обработки изображений.
Выпуклая оболочка позволяет компактно описать форму геометрических объектов, оценить их
размеры, ориентацию и другие морфологические признаки. Целью данной работы является
реализация эффективного алгоритма локализации связных компонент бинарного изображения,
построения для каждой из них выпуклой оболочки и исследование производительности алгоритма.

## 2. Постановка задачи

**Формальная постановка задачи:**
Дано бинарное изображение в виде одномерного массива `std::vector<uint8_t> data` размером
`width * height`. Пиксели со значением `0` представляют собой фон, а пиксели со значением,
отличным от `0`, представляют объекты.

**Входные данные:**

- `width` (int): ширина изображения.
- `height` (int): высота изображения.
- `data` (std::vector<uint8_t>): массив пикселей.

**Выходные данные:**

- Массив выпуклых оболочек `std::vector<std::vector<Point>>`, где каждая оболочка задается
  списком вершин, упорядоченных по контуру.

**Ограничения:**

- Для определения связности пикселей используется 4-связная топология
  (соседи по горизонтали и вертикали).
- Координаты точек и вычисления векторных произведений защищены от переполнения типов
  (используется `int64_t`).

## 3. Базовый алгоритм (последовательный)

Последовательный алгоритм решения задачи разделен на два ключевых этапа:

1. **Выделение компонент связности (Connected Component Labeling):**
   Производится последовательный обход пикселей изображения. При обнаружении пикселя объекта,
   который еще не был размечен (не имеет присвоенного идентификатора в массиве `labels`),
   запускается итеративный алгоритм обхода в глубину (DFS) с использованием явного стека
   `std::stack<Point>`. Все пиксели текущей компоненты извлекаются в вектор.
2. **Построение выпуклой оболочки (Convex Hull):**
   Для множества точек каждой компоненты применяется модифицированный алгоритм Эндрю
   (Monotone Chain).
   - Точки предварительно сортируются лексикографически: сначала по `x`, затем по `y`.
   - На основе косого произведения векторов `CrossProduct` последовательно строятся нижняя
     и верхняя части оболочки. Невалидные точки удаляются из стека оболочки.

## 4. Схема распараллеливания

В последовательной версии (SEQ) распараллеливание отсутствует. Данная версия служит эталоном
(baseline) для оценки корректности и вычисления метрик ускорения параллельных реализаций.

## 5. Детали реализации

- **Структура кода:** Задача реализована в классе `BinaryConvexHullSEQ`.
- **Управление памятью:** Использование `hull.reserve(n + 1)` и
  `component_points.reserve(100)` минимизирует количество повторных аллокаций памяти.
- **Оптимизация:** Перемещение готовых оболочек в результирующий вектор выполняется с
  помощью `std::move`. Алгоритм корректно обрабатывает вырожденные компоненты (1-2 точки).

- **Аппаратное обеспечение (Hardware):**

    (8 физических ядер / 12 логических потоков, базовая частота 2.10 GHz).
  - RAM: 16.0 ГБ (доступно в системе: 15.7 ГБ, скорость 4267 MT/s).
- **Операционная система (OS):** Linux (внутри контейнера Docker
  на базе хост-системы Windows 11 64-разрядная).
- **Инструментарий (Toolchain):** Компилятор GCC / Clang, тип сборки: `Release`.
- **Окружение (Environment):** Запуск производился в изолированном контейнере разработки
  (VS Code Dev Containers / Docker workspace).
- **Данные (Data):** Синтетическое изображение размером 1500x1500 пикселей, содержащее
  сетку из изолированных объектов размером 20x20 пикселей с шагом 100 пикселей.
- **Команды сборки и запуска:**

  ```bash
  mkdir build && cd build
  cmake -DCMAKE_BUILD_TYPE=Release ..
  make -j$(nproc)
  mpirun --allow-run-as-root -n 1 ./bin/ppc_perf_tests --gtest_filter="*egorova_l*seq*"
  ```

## 8. Заключение

Последовательный алгоритм успешно спроектирован. Использование обхода на базе стека
предотвращает переполнение системного стека вызовов, а алгоритм Эндрю обеспечивает
стабильную сложность $O(N \log N)$ для каждой фигуры. Полученные временные характеристики
(около 9 мс на изображение 2.25 Мп) формируют прочный фундамент для исследования
параллельного ускорения.
