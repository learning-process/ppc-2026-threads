# Поиск выпуклой оболочки на бинарном изображении

- Студент: Егорова Лариса Алексеевна, группа 3823Б1ФИ1
- Технология: OMP
- Вариант: 30

## 1. Введение

Данный отчет посвящен реализации параллельной версии алгоритма поиска выпуклой оболочки
с использованием технологии разделяемой памяти OpenMP. Задача сводится к поиску независимых
связных компонент на бинарном изображении и построению их контуров.

## 2. Постановка задачи

**Формальная постановка задачи:**
Дано бинарное изображение `width * height`. Требуется найти все независимые по 4-связности
компоненты и построить для них выпуклые контуры (выпуклые оболочки), исключив переполнения
типов данных.

**Входные/Выходные данные:**

- Вход: `width`, `height`, массив `data`.
- Выход: Массив выпуклых оболочек `std::vector<std::vector<Point>>`
  (координаты вершин контуров компонент).

## 3. Базовый алгоритм (последовательный)

Алгоритм включает последовательный обход пикселей изображения для локализации связных
компонент (с помощью BFS/DFS) и последующее применение алгоритма Эндрю (Monotone Chain)
для вычисления выпуклых оболочек. Временная сложность алгоритма Эндрю для каждой
компоненты составляет $O(N \log N)$ за счет лексикографической сортировки точек.

## 4. Схема распараллеливания

Реализация на OpenMP применяет распараллеливание на уровне обработки независимых компонент
связности:

- **Разделение данных:** Поиск компонент на изображении (`FindComponents`) выполняется
  последовательно (на базе `std::queue` и 4-связности), формируя массив независимых
  векторов точек `components`.
- **Декомпозиция:** Каждая связная компонента обрабатывается независимо в параллельном
  цикле `#pragma omp parallel for`.
- **Планирование (Scheduling):** Используется динамическое планирование
  `schedule(dynamic, 1)`. Это критически важно, так как время построения оболочки зависит
  от размера компоненты, и статическое распределение привело бы к сильному дисбалансу
  нагрузки (load imbalance) между ядрами процессора.
- **Синхронизация:** Запись готовой оболочки в общий выходной массив `output` защищена
  директивой `#pragma omp critical`, предотвращающей состояние гонки (data race).

## 5. Детали реализации

- **Структура кода:** Класс `BinaryConvexHullOMP` (`ops_omp.cpp`).
- Локальная переменная `hull` в цикле объявляется внутри тела, гарантируя её
  потокобезопасность (приватность для каждого потока).
- Количество потоков устанавливается системно через `omp_set_num_threads(omp_get_max_threads())`
  или через переменные окружения.
- Оптимизация памяти достигается за счет вызова `std::move` при переносе готового контура
  в глобальный массив.

## 6. Конфигурация эксперимента

- **Аппаратное обеспечение (Hardware):**
  - CPU: 13th Gen Intel(R) Core(TM) i5-13420H
    (8 физических ядер / 12 логических потоков, базовая частота 2.10 GHz).
  - RAM: 16.0 ГБ (доступно в системе: 15.7 ГБ, скорость 4267 MT/s).
- **Операционная система (OS):** Linux (внутри контейнера Docker
  на базе хост-системы Windows 11 64-разрядная).
- **Инструментарий (Toolchain):** Компилятор GCC / Clang с поддержкой OpenMP,
  тип сборки: `Release`.
- **Окружение (Environment):** Переменная среды `OMP_NUM_THREADS` принимала значения 2 и 4
  для управления количеством задействованных потоков OpenMP.
- **Данные (Data):** Синтетическое изображение размером 1500x1500 пикселей, содержащее
  сетку из изолированных объектов размером 20x20 пикселей.
- **Команды сборки и запуска:**

  ```bash
  # Сборка аналогична базовой. Запуск для 2 и 4 потоков:
  mpirun --allow-run-as-root -x OMP_NUM_THREADS=2 -n 1 \
    ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*omp*"
  mpirun --allow-run-as-root -x OMP_NUM_THREADS=4 -n 1 \
    ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*omp*"
  ```

## 7. Результаты и обсуждение

### 7.1. Корректность

Все функциональные тесты Google Test на проверку граничных случаев (одиночные линии, пустые
изображения, Г-образные фигуры) и перфоманс-тесты успешно проходят, доказывая
эквивалентность параллельных вычислений последовательному эталону.

### 7.2. Производительность

Результаты масштабирования OpenMP (на основе метрики `task_run`):

| Режим | Число потоков | Время, сек | Ускорение | Эффективность |
|-------|---------------|------------|-----------|---------------|
| seq   | 1             | 0.00915    | 1.00      | N/A           |
| omp   | 2             | 0.00505    | 1.81      | 90.5%         |
| omp   | 4             | 0.00386    | 2.37      | 59.3%         |

Эффективность распараллеливания на 2 потоках достигает нормальных значений (~90%). При
переходе на 4 потока ускорение продолжает расти (в 2.37 раза быстрее базовой версии), но
эффективность утилизации ядер предсказуемо снижается до ~59%.

## 8. Заключение

OpenMP-версия продемонстрировала высокую эффективность для изображений с множеством
независимых фигур. Использование `schedule(dynamic)` успешно нивелирует дисбаланс
вычислительной сложности разных объектов. Алгоритм показывает масштабируемость, однако её
предел физически ограничен нераспараллеленным этапом предварительной обработки изображения.
