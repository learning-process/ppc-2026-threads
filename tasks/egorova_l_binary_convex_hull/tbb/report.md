# Поиск выпуклой оболочки на бинарном изображении

- Студент: Егорова Лариса Алексеевна, группа 3823Б1ФИ1
- Технология: TBB
- Вариант: 30

## 1. Введение

В данном отчете представлена реализация параллельного поиска выпуклых оболочек с
использованием библиотеки Intel oneAPI Threading Building Blocks (TBB). Основной фокус
направлен на распараллеливание не только вычисления оболочек, но и этапа сбора пикселей
компонент.

## 2. Постановка задачи

**Формальная постановка задачи:**
Дано бинарное изображение `width * height`. Требуется найти все независимые по 4-связности
компоненты и построить для них выпуклые контуры (выпуклые оболочки), исключив переполнения
типов данных.

## 3. Базовый алгоритм (последовательный)

Алгоритм требует локализации связных пикселей (обход графа) и вычисления контуров
(алгоритм Эндрю).

## 4. Схема распараллеливания

Архитектура TBB-решения разделена на три логических этапа, два из которых параллельны:

1. **Последовательная разметка (`LabelComponents`):** Однопоточный BFS-обход изображения
   по индексам. Создается массив `labels`.
2. **Параллельный сбор точек:** Изображение разбивается на двумерные блоки с помощью
   `tbb::blocked_range2d<int>`. Потоки конкурентно обходят свои блоки и распределяют
   пиксели по корзинам. Для синхронизации используется вектор `std::mutex`.
3. **Параллельное построение оболочек:** Применяется `tbb::parallel_for` по массиву
   собранных компонент с использованием встроенного планировщика Work-Stealing.

## 5. Детали реализации

- **Структура:** Класс `BinaryConvexHullTBB` (`ops_tbb.cpp`).
- Использование `std::scoped_lock` с массивом индивидуальных мьютексов применяется для
  защиты векторов с точками от состояния гонки во время сбора координат.

## 6. Конфигурация эксперимента

- **Аппаратное обеспечение (Hardware):**
  - CPU: 13th Gen Intel(R) Core(TM) i5-13420H
    (8 физических ядер / 12 логических потоков, базовая частота 2.10 GHz).
  - RAM: 16.0 ГБ (доступно в системе: 15.7 ГБ, скорость 4267 MT/s).
- **Операционная система (OS):** Linux (внутри контейнера Docker
  на базе хост-системы Windows 11 64-разрядная).
- **Инструментарий (Toolchain):** Библиотека Intel oneAPI TBB, компилятор C++ с поддержкой
  стандарта C++17, тип сборки: `Release`.
- **Окружение (Environment):** Переменная среды `TBB_NUM_THREADS` принимала значения 2 и 4
  для явного ограничения размера пула потоков TBB.
- **Данные (Data):** Синтетическое изображение размером 1500x1500 пикселей, содержащее
  сетку из изолированных объектов размером 20x20 пикселей.
- **Команды сборки и запуска:**

  ```bash
  mpirun --allow-run-as-root -x TBB_NUM_THREADS=2 -n 1 \
    ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*tbb*"
  mpirun --allow-run-as-root -x TBB_NUM_THREADS=4 -n 1 \
    ./build/bin/ppc_perf_tests --gtest_filter="*egorova_l*tbb*"
  ```

## 7. Результаты и обсуждение

### 7.1. Корректность

Алгоритм успешно проходит все функциональные тесты. Двумерный сплиттинг изображения в TBB
не нарушает целостность данных благодаря корректной блокировке мьютексами при слиянии
векторов.

### 7.2. Производительность

Результаты масштабирования TBB (на основе метрики `task_run`):

| Режим | Число потоков | Время, сек | Ускорение | Эффективность |
|-------|---------------|------------|-----------|---------------|
| seq   | 1             | 0.00915    | 1.00      | N/A           |
| tbb   | 2             | 0.00874    | 1.05      | 52.5%         |
| tbb   | 4             | 0.00761    | 1.20      | 30.0%         |

Результаты показывают сильную деградацию параллельной эффективности (ускорение всего 1.2
раза на 4 потоках). Использование массива `std::mutex` для блокировки при сборе точек
(`CollectPoints`) внутри 2D-цикла создает колоссальные накладные расходы на синхронизацию.
Потоки большую часть времени проводят в ожидании освобождения мьютексов. Накладные расходы
TBB-планировщика в связке с тяжелыми системными блокировками практически полностью съедают
выигрыш от распараллеливания.

## 8. Заключение

Библиотека TBB успешно интегрирована, алгоритм корректен. Однако эксперимент наглядно
доказал, что применение «тяжелых» блокировок (mutex) во внутренних узких циклах
(per-pixel iteration) недопустимо для достижения высокой производительности.
