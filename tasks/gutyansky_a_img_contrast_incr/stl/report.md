## 3. Базовый алгоритм

Алгоритм тот же, но каждый из двух этапов выполняется в отдельном пуле потоков с ручным разбиением диапазона.

## 4. Схема распараллеливания

**Разбиение данных:**

- Количество потоков = `ppc::util::GetNumThreads()`.
- Размер чанка `chunk_sz = (sz + num_threads - 1) / num_threads` - таким образом данные распределяеются равномерно
  (разница между количеством обрабатываемых итераций максимум 1).
- Каждый поток получает непрерывный отрезок `[from, to)`.

**Поиск min/max:**

- Создаётся вектор `local_res` для хранения пар `(min,max)` каждого потока.
- Потоки обновляют свои локальные переменные, затем после синхронизации объединяются результаты.

**Преобразование:**

- Аналогично, каждый поток обрабатывает свой диапазон.
- Запись в выходной массив независима (разные индексы), синхронизация не требуется.
- Потоки синхронизируются в конце обработки для получения полного результата.

В stl версии главный поток не включен в обработку и лишь выполняет распределение задач и синхронизацию.

## 5. Детали реализации

Файлы: `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Функция `ComputeBounds`:

```cpp
// Поиск минимального и максимального элемента во входном массиве
std::vector<std::pair<uint8_t, uint8_t>> local_res(num_threads, {255, 0});
for (uint32_t tid = 0; tid < num_threads; tid++) {
    size_t from = tid * chunk_sz;
    size_t to = std::min(sz, from + chunk_sz);
    threads.emplace_back([&local_res, &input, from, to, tid]() {
        uint8_t local_min = 255, local_max = 0;
        for (size_t i = from; i < to; ++i) {
            local_min = std::min(local_min, input[i]);
            local_max = std::max(local_max, input[i]);
        }
        local_res[tid] = {local_min, local_max};
    });
}

// Синхронизация потоков для получения результатов из всех чанков
WaitAll(threads);
// ...частичные результаты объединяются в lower_bound и upper_bound


// Выполняется преобразование данных
for (unsigned int tid = 0; tid < num_threads; tid++) {
    size_t from = tid * chunk_sz;
    size_t to = std::min(from + chunk_sz, sz);

    threads.emplace_back([&output, &input, from, to, lower_bound, delta]() {
      for (size_t idx = from; idx < to; ++idx) {
        uint16_t old_value = input[idx];
        uint16_t new_value = (kMaxUint8 * (old_value - lower_bound)) / delta;
        output[idx] = static_cast<uint8_t>(new_value);
      }
    });
}

// Происходит синхронизация потоков. После синхронизации в выходном массиве находится полный ответ на задачу
WaitAll(threads);
```

На 1 и 2 потоках наблюдаем сверхускорение. Вероятно, компилятор смог дополнительно оптимизировать какие-то части кода.
Если смотреть на результат локально, а не относительно SEQ версии, то получаем меньшее ускорение,
чем было получено на OMP и TBB. Это может быть связано с накладными расходами по созданию и синхронизации потоков.
Также после выхода на плато ускорения в х3.5 (как и везде) примерно на 12 потоках, происходит ухудшение на 16.
Это также связано с возрастающими накладными расходами.

## 9. Выводы

Ручное управление потоками даёт приемлемое ускорение, но требует больше кода и менее эффективно на больших числах потоков.
std::thread подходит для задач с предсказуемой равномерной нагрузкой, где важно жёсткое управление привязкой.
