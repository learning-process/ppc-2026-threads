## 1. Контекст

Гибридная версия использует MPI для распределения данных между процессами и OpenMP для внутреннего параллелизма
внутри каждого процесса. Это позволяет масштабироваться на кластерах с общей памятью на узле.

## 2. Постановка задачи

(См. `seq/report.md`, раздел 2)  

Дополнительно: данные могут быть распределены между несколькими MPI-процессами.

## 3. Базовый алгоритм

Алгоритм не меняется, но этапы распределены, в связи с этим:

- Каждый процесс выполняет операции на собственной частью данных.
- Требуется обмен данными для распределения данных, вычисления мин./макс. элементов, сбора данных.

## 4. Межпроцессная схема

- Ранг 0 отвечает за раздачу исходных данных и сборку результата.
- Используются коллективные операции: `Bcast`, `Scatterv`, `Allreduce`, `Allgatherv`.

## 5. Внутрипроцессная схема

Внутри каждого процесса применяется OpenMP (как в версии OMP) для:

- параллельного поиска локальных min/max (редукция);
- параллельного преобразования или копирования.

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

Распределение:

```cpp
size_t chunk_size = sz / p_count;
size_t remainder_size = sz % p_count;
size_t local_count = chunk_size + (rank < remainder_size ? 1 : 0);
// ...вычисление send_counts и displs (locs)...

MPI_Scatterv(GetInput().data(), send_counts.data(), locs.data(), MPI_UINT8_T, local_data.data(), local_count, MPI_UINT8_T, 0, MPI_COMM_WORLD);

// Вычисление минимального и максимального элементов
#pragma omp parallel for default(none) shared(local_data, local_count) \
    reduction(min : local_lower_bound) reduction(max : local_upper_bound)
for (size_t i = 0; i < local_count; i++) { ... }

MPI_Allreduce(&local_lower_bound, &lower_bound, 1, MPI_UINT8_T, MPI_MIN, MPI_COMM_WORLD);
MPI_Allreduce(&local_upper_bound, &upper_bound, 1, MPI_UINT8_T, MPI_MAX, MPI_COMM_WORLD);

// ...преобразование данных...

// Объединение данных
MPI_Allgatherv(local_data.data(), local_count, MPI_UINT8_T, GetOutput().data(), send_counts.data(), locs.data(), MPI_UINT8_T, MPI_COMM_WORLD);

```

## 7. Проверка корректности

Функциональные тесты запускаются с разным числом процессов (например, 2, 4). Результаты совпадают с SEQ. Проверены случаи,
когда размер входного массива не кратен числу процессов.

## 10. Выводы

Гибридная схема оправдана, когда данных очень много и они не помещаются в память одного узла, либо при работе на кластере.
На одной машине с общей памятью чистая OpenMP/TBB даёт лучшее время из-за меньших коммуникационных затрат.
