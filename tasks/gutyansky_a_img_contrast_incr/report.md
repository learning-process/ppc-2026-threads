## 1. Введение

Задача линейного растяжения гистограммы (увеличение контраста) широко применяется в обработке изображений.
Она хорошо подходит для сравнения различных моделей параллелизма, так как:

- Данные независимы (каждый элемент обрабатывается отдельно).
- Требуется одна глобальная редукция (min/max).
- Вычислительная нагрузка равномерна.

Однако:

- Количество вычислений по отношению к работе с памятью невелико,
  потому максимальное ускорение ограничивается пропускной способностью памяти.

Сравниваются пять реализаций: SEQ (последовательная), OMP (OpenMP), TBB (Threading Building Blocks),
STL (std::thread), ALL (MPI + OpenMP).

## 2. Единая постановка задачи

Задача состоит в линейном растяжении гистограммы яркости полутонового изображения (или любого одномерного массива байт).

**Входные данные**: вектор `uint8_t` произвольной длины (например, пиксели изображения).  
**Выходные данные**: вектор той же длины, каждый элемент преобразован по следующему алгоритму:

- Найти минимальное `min` и максимальное `max` значения во входном массиве.
- Если `max == min` (все значения одинаковы), выходной массив копирует входной.
- Иначе для каждого элемента `x`:  
  `new = (255 * (x - min)) / (max - min)` (деление целочисленное).

**Ограничения**: входной массив не пуст.
**Крайние случаи**: все пиксели равны, массив из одного элемента, максимальный разброс (0..255).

**Критерий корректности:** последовательная (SEQ) реализация и все параллельные реализации алгорита должны
проходить все функциональные тесты.

## 6. Интерпретация различий

- **SEQ** – базовый эталон, время = 825 мс.
- **OMP** – среднее ускорение (3.4x на 16 потоках), минимальные изменения в коде.
- **TBB** – близок к OMP, но слегка лучше в случае компилятора MSVC.
- **STL** – лучшая эффективность в сравнении с SEQ, но накладные расходы
  большие и эффективность падает быстрее из-за накладных расходов.
- **ALL** – на 4 процессах и 2 потоках достигает 1.66x, значитаельно уступает чистой OpenMP при работе
  на одном узле из-за коммуникаций. Преимущество проявится при масштабировании на несколько узлов кластера.

## 8. Заключение

OpenMP дает хорошее ускорение при минимальных усилиях.

TBB дает сопоставимое с OpenMP ускорение, однако требует чуть больше изменений кода, при этом дает независимость от компилятора.

Ручное управление потоками через STL дает больше контроля над распределением задач,
но проигрывает в количестве накладных расходов. Реализация хорошо работающего алгоритма планирования задач потребует
большого количества трудозатрат.

Гибридная версия ALL необходима только при работе на распределённом кластере с несколькими узлами.
