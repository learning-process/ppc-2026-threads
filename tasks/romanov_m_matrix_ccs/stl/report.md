## 1. Контекст

Разработка параллельной версии вычислительного ядра с использованием
низкоуровневых примитивов стандартной
библиотеки C++ (`std::thread`) обусловлена необходимостью глубокого инженерного
анализа ручного управления
потоками. В отличие от высокоуровневых абстракций (OpenMP или Intel oneTBB),
которые скрывают от разработчика
внутренние механизмы планирования, реализация на базе `std::thread` требует
самостоятельного проектирования
декомпозиции данных, распределения нагрузки и координации жизненного цикла
вычислительных единиц.

Данный подход позволяет оценить «чистую» производительность потоков операционной
системы и зафиксировать цену
накладных расходов на создание и слияние нитей без влияния сторонних
рантайм-планировщиков.

## 2. Постановка задачи

Задача состоит в реализации параллельного алгоритма умножения двух разреженных
вещественных матриц $C = A \times B$ в формате сжатого хранения по столбцам
(CCS) средствами стандартной библиотеки потоков C++.

Последовательный алгоритм SEQ принимается в качестве неизменяемого эталона
корректности и базовой линии производительности (baseline). Разрабатываемая
STL-version обязана выдавать детерминированный результат, полностью
тождественный эталону, осуществлять статическое распределение столбцов по
потокам и корректно управлять их жизненным циклом.

## 3. Базовый алгоритм

Последовательная версия реализует алгоритм Густавсона, основанный на линейной
комбинации столбцов левой матрицы `A`.
Алгоритм итерируется по столбцам `j` правой матрицы `B`. Для каждого столбца
создаётся плотный
вектор-аккумулятор, куда накапливаются произведения элементов столбца `B` на
соответствующие столбцы `A`.
После завершения обработки столбца `B_j` значения в аккумуляторе, превосходящие
порог точности (`<= 10^{-12}`),
упаковываются в финальные сжатые массивы столбца `C_j`.

## 4. Схема распараллеливания

**Разбиение данных на диапазоны (Декомпозиция):**
Параллельный алгоритм реализует ручное блочное распределение нагрузки по
столбцам матрицы $B$. Общий диапазон индексов от 0 до b.cols_num делится на
непрерывные блоки (chunks).

Размер базового блока рассчитывается по формуле chunk_size = b.cols_num /
num_threads. Для предотвращения потери данных из-за целочисленного деления,
остаток диапазона (remainder = b.cols_num % num_threads) распределяется по
принципу «плюс один столбец» между первыми remainder потоками.

**Поведение потока и локализация данных:**
Каждый поток выполняет последовательный обход выделенного ему непрерывного
диапазона столбцов [start, end) в рамках изолированной лямбда-функции worker.
Защита от гонок за данные (Data Races) достигнута на архитектурном уровне за
счет полной локализации записи результатов вычислений.

Потоки не используют блокирующие примитивы (std::mutex), атомики (std::atomic)
или условные переменные, так как каждый поток записывает вычисленные векторы
столбцов в строго закрепленные за ним ячейки независимых внешних контейнеров
temp_vals[j] и temp_rows[j].

**Жизненный цикл потоков (Порядок Create -> Work -> Join):**
Контроль выполнения организован по строгой двухфазной схеме. В первой фазе
главный поток порождает пустой вектор threads и в рамках последовательного цикла
осуществляет запуск всех вычислителей через метод threads.emplace_back(worker,
current_start, current_end).

Во второй фазе главный поток блокируется и последовательно ожидает завершения
каждой рабочей нити с помощью вызова t.join(). Вызовы join() вынесены в
отдельный, изолированный цикл, выполняющийся строго после завершения создания
всех потоков. Это критически важно: вызов join() внутри цикла создания привел бы
к немедленной блокировке и перевел бы алгоритм в неявный последовательный режим
вычислений.

## 5. Детали реализации

Файлы: stl/include/ops_stl.hpp, stl/src/ops_stl.cpp

```cpp
// File: stl/src/ops_stl.cpp
size_t chunk_size = b.cols_num / num_threads;
size_t remainder = b.cols_num % num_threads;
size_t current_start = 0;

for (unsigned int i = 0; i < num_threads; ++i) {
  size_t current_end = current_start + chunk_size + (i < remainder ? 1 : 0);
  if (current_start < current_end) {
    threads.emplace_back(worker, current_start, current_end);
  }
  current_start = current_end;
}

for (auto &t : threads) {
  if (t.joinable()) {
    t.join();
  }
}
```

**Схема объединения локальных результатов:**
После того как управление проходит барьер слияния всех потоков t.join(),
результирующая матрица гарантированно заполнена локальными векторами столбцов.
Главный поток выполняет последовательный обход контейнеров, вычисляет префиксные
суммы для формирования массива смещений c.col_ptrs, резервирует память (reserve)
и выполняет линейную вставку (insert) данных из temp_vals и temp_rows в
финальные плоские массивы формата CCS.

## 6. Проверка корректности

Корректность параллельной STL-реализации проверяется в рамках параметризованных
тестов.

1. Тест базового перемножения квадратных матриц 2х2

2. Тест на крайний случай — умножение на нулевую матрицу

3. Тест перемножения неквадратных прямоугольных матриц

Метод CheckTestOutputData осуществляет сверку результирующих векторов значений с
эталоном с точностью до $1e-9$. Тесты успешно подтверждают полную идентичность
результатов STL-версии и последовательного baseline.

## 7. Экспериментальная среда

```bash
python scripts/run_tests.py --running-type="performance"
```

Анализ экспериментальных данных выявил слабую масштабируемость ручной потоковой

последовательной версии (коэффициент ускорения $1.027$, эффективность —
$12.8\%$).

Такое характер изменений обусловлен двумя факторами:

1. Высокая цена системных вызовов.

2. Дисбаланс статического разбиения.

## 9. Выводы

Ручная реализация параллелизма на базе std::thread подтвердила свою надежность и
потокобезопасность благодаря архитектурному разнесению промежуточных областей
памяти для записи результатов, что позволило полностью отказаться от блокирующих
примитивов синхронизации.

С точки зрения производительности эксперимент доказал, что статическое блочное
разбиение диапазона без механизмов динамической балансировки нагрузки (таких как
динамический планировщик в OpenMP или кража задач work-stealing в oneTBB) не
способно обеспечить устойчивое масштабирование на разреженных структурах малого
объема. Использование низкоуровневых потоков C++ сопровождается высокими
издержками на управление их жизненным циклом. Данный подход экономически
оправдан только при обработке сверхбольших регулярных массивов данных, где
полезное время вычислений внутри потока на порядки превосходит системные затраты
на создание и слияние нитей исполнения.
