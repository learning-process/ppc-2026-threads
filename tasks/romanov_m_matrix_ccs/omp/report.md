## 1. Контекст

Перенос вычислительного ядра из последовательной версии (SEQ) в параллельную
среду OpenMP обусловлен необходимостью
сокращения времени выполнения ресурсоемкой операции умножения разреженных матриц
больших размерностей. Несмотря на
алгоритмическую оптимизацию под формат CCS (сжатие по столбцам), базовая
обработка выполняет итерации по столбцам
результирующей матрицы последовательно, что при больших размерах приводит к
значительным временным затратам.

Использование OpenMP позволяет задействовать многоядерную архитектуру
современных процессоров с общей памятью
(Shared Memory CPU) для одновременного независимого вычисления нескольких
столбцов результирующей матрицы. Это даёт
существенное ускорение при минимальном усложнении кодовой базы по сравнению с
ручным управлением потоками.

## 2. Постановка задачи

Задача состоит в реализации параллельной версии умножения двух разреженных
вещественных матриц $C = A \times B$
в формате CCS с использованием OpenMP. Последовательная реализация
(`seq/report.md`) является эталоном корректности
и базовой линией производительности (baseline). Параллельная версия должна
выдавать математически тождественные
результаты и демонстрировать устойчивое сокращение времени расчёта при
увеличении числа доступных ядер.

## 3. Базовый алгоритм

Последовательная версия реализует алгоритм Густавсона, основанный на линейной
комбинации столбцов левой матрицы `A`.
Алгоритм итерируется по столбцам `j` правой матрицы `B`. Для каждого столбца
создаётся плотный вектор-аккумулятор,
куда накапливаются произведения элементов столбца `B` на соответствующие столбцы
`A`. После обработки столбца `B_j`
значения в аккумуляторе, превосходящие порог точности (`<= 10^{-12}`),
упаковываются в сжатые массивы столбца `C_j`.

## 4. Схема распараллеливания

В качестве параллельной области выбрана внешняя итерационная петля по столбцам
правой матрицы $B$ (переменная цикла j от 0 до b.cols_num). Вычисления для
каждого столбца результирующей матрицы полностью независимы от вычислений для
других столбцов.

**Спецификация области данных и атрибутов переменных:**
Для жесткого контроля области видимости переменных использована директива
default(none). Это обязывает явно задекларировать атрибут каждого
идентификатора, защищая код от неявного и некорректного распределения переменных
компилятором по умолчанию.

shared(a, b, temp_vals, temp_rows): входные матрицы a и b доступны только для
чтения и разделяются между всеми потоками. Временные контейнеры верхнего уровня
temp_vals и temp_rows распределены как shared, так как каждый поток на итерации
j осуществляет запись строго в свою ячейку памяти (temp_vals[j] и temp_rows[j]),
что исключает пересечение потоков и возникновение гонки за данные.

**Локализация данных (Private) и отсутствие Reduction:**
Все промежуточные структуры данных, необходимые для вычисления отдельного
столбца, вынесены за пределы параллельного цикла OpenMP внутрь тела
вспомогательной статической функции MultiplyColumn. Благодаря этому векторы
accumulator, row_mask и active_rows автоматически размещаются в изолированных
стеках каждого отдельного потока выполнения.

Поскольку сборка результирующих данных осуществляется не в общую
переменную-сумматор, а распределяется по персональным независимым ячейкам общего
вектора векторов (temp_vals[j]), использование дорогостоящих директив reduction,
atomic или critical не требуется. Это минимизирует накладные расходы на
синхронизацию.

**Выбор планировщика (Schedule):**
Для распределения итераций между потоками выбран динамический планировщик —
schedule(dynamic). Специфика разреженных матриц заключается в том, что плотность
(число ненулевых элементов) в разных столбцах матрицы $B$ может кардинально
отличаться. Статическое планирование (static) разделило бы итерации на равные
порции, что привело бы к дисбалансу нагрузки (одни потоки завершили бы работу
мгновенно на пустых столбцах, а другие долго обрабатывали бы плотные).
Динамический планировщик выдает итерации потокам "по требованию" (по одной
итерации по умолчанию), гарантируя равномерную утилизацию ядер процессора.

**Барьеры:**
В соответствии со спецификацией OpenMP, в конце параллельного региона (#pragma
omp parallel) существует неявный барьер синхронизации. Все потоки обязаны
дождаться завершения обработки последней итерации цикла, прежде чем управление
будет передано в последовательный участок кода, выполняющий окончательную сборку
структуры CCS. Данный барьер безопасен и необходим, так как последующий код
сборки зависит от полноты заполнения temp_vals и temp_rows.

## 5. Детали реализации

Файлы: omp/include/ops_omp.hpp, omp/src/ops_omp.cpp
**Изменения относительно SEQ:**
В отличие от последовательной версии, где запись вычисленных ненулевых элементов
производилась сразу напрямую в результирующие одномерные векторы c.vals и
c.row_inds (что в параллельной среде вызвало бы непреодолимую гонку данных из-за
динамического изменения размеров векторов через push_back), в OpenMP-версии
вычисления разделены на два этапа:

1. **Параллельный этап:** Потоки параллельно вызывают функцию MultiplyColumn,
заполняя предварительно выделенные ячейки временных контейнеров:

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel num_threads(ppc::util::GetNumThreads()) default(none) shared(a, b, temp_vals, temp_rows)
  {
#pragma omp for schedule(dynamic)
    for (size_t j = 0; j < b.cols_num; ++j) {
      MultiplyColumn(j, a, b, temp_vals[j], temp_rows[j]);
    }
  }
```

2. **Последовательный этап:** На основе размеров заполненных локальных векторов
в один поток вычисляется финальное смещение столбцов c.col_ptrs, резервируется
необходимый объем памяти в c.vals и c.row_inds, после чего данные копируются из
temp_vals и temp_rows в итоговые плоские массивы CCS.

## 6. Проверка корректности

Корректность параллельной OMP-реализации проверяется в рамках параметризованных
тестов.

1. Тест базового перемножения квадратных матриц 2х2

2. Тест на крайний случай — умножение на нулевую матрицу

3. Тест перемножения неквадратных прямоугольных матриц

Метод CheckTestOutputData осуществляет сверку результирующих векторов значений с
эталоном с точностью до $1e-9$. Тесты успешно подтверждают полную идентичность
результатов OMP-версии и последовательного baseline.

## 7. Экспериментальная среда

- Управление потоками: Количество запускаемых потоков жестко контролируется
  программным фреймворком через функцию ppc::util::GetNumThreads(), которая
  транслирует значение переменной среды PPC_NUM_THREADS (или OMP_NUM_THREADS) в

```bash
python scripts/run_tests.py --running-type="performance"
```

## 8. Результаты

| Число потоков | Время выполнения (сек) | Коэффициент ускорения | Эффективность
|
| --- | --- | --- | --- |
| 1 (baseline) | 0.542 | 1 | 100% |
| 2 | 0.437 | 1.240 | 62% |
| 4 | 0.402 | 1.348 | 33.7% |
| 8 | 0.588 | 0.922 | 11.5% |

**Масштабируемость:**  Алгоритм демонстрирует крайне слабую масштабируемость.
Максимальное ускорение достигается на 4 потоках и составляет всего $1.348$ при
низкой эффективности в $33.7\%$.
**Падение на 8 потоках:** На 8 потоках время выполнения ($0.588$ сек) превышает
время последовательной версии ($0.542$ сек). Коэффициент ускорения падает ниже
единицы ($0.922$), что означает замедление программы.
**Причина неэффективности:** Такой характер падения эффективности для
std::thread при статическом блочном разбиении (chunks) свидетельствует о том,
что накладные расходы на ручное создание, диспетчеризацию и слияние потоков
(threads.emplace_back и t.join()) превосходят полезное время вычислений, либо
потоки столкнулись с сильным дисбалансом нагрузки (Load Imbalance), ожидая один
наиболее плотный столбец матрицы.
