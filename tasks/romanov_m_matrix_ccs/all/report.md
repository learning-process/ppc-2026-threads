## 1. Контекст
Создание гибридной параллельной версии (ALL), объединяющей интерфейс передачи сообщений (MPI) и многопоточный рантайм Intel oneTBB, обусловлено необходимостью масштабирования вычислений за рамки одного физического компьютера. Последовательные и чисто многопоточные версии ограничены ресурсами общей памяти (Shared Memory) одного узла.

Гибридный подход реализует иерархическую модель параллелизма: распределенная память (уровень узлов кластера) управляется процессами MPI, а общая память (внутри каждого узла) эффективно утилизируется задачами TBB. Это позволяет обрабатывать разреженные матрицы колоссальных порядков, которые физически невозможно разместить в оперативной памяти одной вычислительной машины.

## 2. Постановка задачи
Задача заключается в проектировании и реализации двухуровневого параллельного алгоритма умножения двух разреженных вещественных матриц $C = A \times B$ в формате сжатого хранения по столбцам (CCS).

Последовательный алгоритм SEQ принимается в качестве неизменяемого эталона корректности и базовой линии производительности (baseline). Разрабатываемая ALL-version обязана выдавать детерминированный результат, полностью тождественный эталону, осуществлять статическое распределение столбцов по потокам и корректно управлять их жизненным циклом.

## 3. Базовый алгоритм
Вычислительное ядро базируется на алгоритме Густавсона (линейная комбинация столбцов). Вместо классического умножения строки на столбец, алгоритм последовательно обрабатывает столбцы правой матрицы $B$. Для каждого столбца его ненулевые элементы извлекаются и используются в качестве коэффициентов масштабирования для соответствующих столбцов левой матрицы $A$. Все промежуточные результаты накапливаются в плотный локальный вектор-аккумулятор, снабженный массивом масок строк. В конце обработки столбца элементы, модуль которых превышает порог точности $10^{-12}$, упорядочиваются по строкам и формируют итоговый сжатый столбец.

## 4. Межпроцессная схема
**Роли рангов (Rank Roles):**
В алгоритме применяется выделенная координирующая роль для процесса с рангом 0 (Master). Процесс-мастер отвечает за получение исходных матриц, инициализацию общей координационной среды, сборку локальных результатов от рабочих узлов и финальное построение результирующей матрицы CCS. Процессы с рангами rank > 0 (Workers) функционируют исключительно в качестве изолированных вычислителей.

**Распределение данных:**
Разбиение матрицы осуществляется по столбцам правой матрицы $B$. Общее число столбцов total_cols делится на количество MPI-процессов size. Вычисление границ диапазонов (start_col, end_col) производится вручную с учетом остатка (remainder = total_cols % size). Первые remainder процессов получают порцию на один столбец больше, что гарантирует идеальную статическую балансировку межпроцессной нагрузки.

**MPI-коммуникации:**
Перед началом этапа вычислений главный процесс осуществляет коллективную широковещательную рассылку размерностей матриц и их физических CCS-структур (col_ptrs, row_inds, vals) всем процессам-воркерам через примитив MPI_Bcast. После завершения локальных вычислений воркеры отправляют свои результаты обратно мастеру. Сборка реализуется через точечные обмены: процессы вызывают MPI_Send, а мастер последовательно принимает сообщения через MPI_Recv в рамках метода MasterCollect.

## 5. Внутрипроцессная схема
Как только каждый MPI-процесс получает свои глобальные границы индексов столбцов [start_col, end_col), управление передается на внутрипроцессный уровень параллелизма. Внутри адресного пространства процесса обработка выделенного локального подмножества столбцов local_count распараллеливается по ядрам процессора с помощью tbb::parallel_for.

```cpp
// File: all/src/ops_all.cpp
if (local_count > 0) {
  tbb::parallel_for(0, local_count, [&](int i) {
    MultiplyColumn(static_cast<size_t>(start_col) + i, a_mat, b_mat, 
                   local_v[static_cast<size_t>(i)],
                   local_r[static_cast<size_t>(i)]);
  });
}
```

Планировщик TBB автоматически распределяет итерации i по внутренним легковесным задачам, используя механизм кражи задач (work-stealing) для балансировки нагрузки на ядрах конкретного узла. Локальные промежуточные результаты накапливаются во внутренних векторах векторов local_v и local_r, полностью исключая взаимные блокировки потоков.

## 6. Детали реализации
Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp
**Преодоление кроссплатформенных ограничений типов данных:**
Критически важной деталью реализации является обеспечение переносимости кода. Тип size_t имеет разную разрядность в зависимости от компилятора и операционной системы (например, 4 байта в MSVC под Windows для unsigned long и 8 байт под Linux). Прямая передача массивов size_t через стандартные типы MPI неизбежно провоцирует критические ошибки несоответствия типов (mpi-type-mismatch) или обрезания сообщений (Message Truncation).

Для решения этой проблемы реализованы промежуточные функции-упаковщики BroadcastSizeTVector, SendSizeTVector и RecvSizeTVector. Они осуществляют явное копирование данных в буфер фиксированного 64-битного типа std::vector<uint64_t>, который гарантированно занимает 8 байт на элемент и безопасно передается через системный тип MPI_UINT64_T на любой платформе.

**Точки синхронизации:**
Синхронизация между узлами обеспечивается коллективными операциями MPI_Bcast при обмене метаданными и финальными размерами вычисленной матрицы $C$. Перед выходом из метода RunImpl вызывается явная барьерная синхронизация MPI_Barrier(MPI_COMM_WORLD), гарантирующая, что все процессы завершили работу с коммуникационной средой и готовы к переходу к следующей задаче.
