## 1. Контекст

Создание гибридной параллельной версии (ALL), объединяющей интерфейс передачи
сообщений (MPI) и
многопоточный рантайм Intel oneTBB, обусловлено необходимостью масштабирования
вычислений за рамки
одного физического компьютера. Последовательные и чисто многопоточные версии
ограничены ресурсами
общей памяти (Shared Memory) одного узла.

Гибридный подход реализует иерархическую модель параллелизма: распределенная
память (уровень
узлов кластера) управляется процессами MPI, а общая память (внутри каждого узла)
эффективно
утилизируется задачами TBB. Это позволяет обрабатывать разреженные матрицы
колоссальных порядков,
которые физически невозможно разместить в оперативной памяти одного
вычислительного машинного узла.

## 2. Постановка задачи

Задача заключается в проектировании и реализации двухуровневого параллельного
алгоритма умножения
двух разреженных вещественных матриц $C = A \times B$ в формате сжатого хранения
по столбцам (CCS).

Последовательный алгоритм SEQ принимается в качестве эталона корректности и
базовой линии
производительности (baseline). Разрабатываемая ALL-version обязана выдавать
детерминированный результат,
полностью тождественный эталону, осуществлять статическое распределение столбцов
по потокам и корректно
управлять их жизненным циклом.

## 3. Базовый алгоритм

Вычислительное ядро базируется на алгоритме Густавсона (линейная комбинация
столбцов). Вместо
классического умножения строки на столбец, алгоритм последовательно обрабатывает
столбцы правой
матрицы `B`. Для каждого столбца его ненулевые элементы извлекаются и
используются в качестве
коэффициентов масштабирования для соответствующих столбцов левой матрицы `A`.
Все промежуточные
результаты накапливаются в плотный локальный вектор-аккумулятор, снабженный
массивом масок строк.
В конце обработки столбца элементы, модуль которых превышает порог точности
`10^{-12}`,
упорядочиваются по строкам и формируют итоговый сжатый столбец.

## 4. Межпроцессная схема

**Роли рангов (Rank Roles):**

В алгоритме применяется выделенная координирующая роль для процесса с рангом 0
(Master). Процесс-мастер
отвечает за получение исходных матриц, инициализацию координационной среды,
сборку локальных
результатов от рабочих узлов и финальное построение результирующей матрицы CCS.
Процессы с рангами
`rank > 0` (Workers) функционируют исключительно в качестве изолированных
вычислителей.

**Распределение данных:**

Разбиение матрицы осуществляется по столбцам правой матрицы `B`. Общее число
столбцов `total_cols`
делится на количество MPI-процессов `size`. Вычисление границ диапазонов
(`start_col`, `end_col`) производится
вручную с учётом остатка (`remainder = total_cols % size`). Первые `remainder`
процессов получают порцию на
один столбец больше, что гарантирует более равномерную статическую балансировку
нагрузки.

**MPI-коммуникации:**

Перед началом этапа вычислений главный процесс осуществляет широковещательную
рассылку размерностей
матриц и их CCS-структур (`col_ptrs`, `row_inds`, `vals`) всем
процессам-воркерам через `MPI_Bcast`.
После завершения локальных вычислений воркеры отправляют свои результаты обратно
мастеру. Сборка
реализуется через точечные обмены: процессы вызывают `MPI_Send`, а мастер
последовательно принимает
сообщения через `MPI_Recv` в методе `MasterCollect`.

## 5. Внутрипроцессная схема

Как только каждый MPI-процесс получает свои глобальные границы индексов столбцов
`[start_col, end_col)`, управление передаётся на внутрипроцессный уровень
параллелизма. Внутри адресного
пространства процесса обработка выделенного локального подмножества столбцов
`local_count`
распараллеливается по ядрам процессора с помощью `tbb::parallel_for`.

```cpp
// File: all/src/ops_all.cpp
if (local_count > 0) {
  tbb::parallel_for(0, local_count, [&](int i) {
    MultiplyColumn(static_cast<size_t>(start_col) + i, a_mat, b_mat,
                   local_v[static_cast<size_t>(i)],
                   local_r[static_cast<size_t>(i)]);
  });
}
```

Планировщик TBB автоматически распределяет итерации `i` по внутренним
легковесным задачам, используя
механизм кражи задач (work-stealing) для балансировки нагрузки на ядрах
конкретного узла. Локальные
промежуточные результаты накапливаются во внутренних векторах `local_v` и
`local_r`, полностью исключая
взаимные блокировки потоков.

## 6. Детали реализации

Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp
**Преодоление кроссплатформенных ограничений типов данных:**
Критически важной деталью реализации является обеспечение переносимости кода.
`size_t` имеет разную
разрядность в зависимости от компилятора и ОС (например, 4 байта в MSVC под
Windows и 8 байт под Linux).
Прямая передача массивов `size_t` через стандартные типы MPI может провоцировать
ошибки несоответствия
типов (mpi-type-mismatch) или обрезания сообщений (Message Truncation).

Для решения этой проблемы реализованы функции-упаковщики `BroadcastSizeTVector`,
`SendSizeTVector` и
`RecvSizeTVector`. Они копируют данные в буфер фиксированного 64-битного типа
`std::vector<uint64_t>`,
который гарантированно занимает 8 байт на элемент и безопасно передаётся через
тип `MPI_UINT64_T`.

**Точки синхронизации:**
Синхронизация между узлами обеспечивается коллективными операциями `MPI_Bcast`
при обмене метаданными и
финальными размерами вычисленной матрицы `C`. Перед выходом из `RunImpl`
вызывается явная барьерная
синхронизация `MPI_Barrier(MPI_COMM_WORLD)`, гарантирующая завершение работы
всех процессов.

## 7. Проверка корректности

Корректность параллельной ALL-реализации проверяется в рамках параметризованных
тестов.

1. Тест базового перемножения квадратных матриц 2х2

2. Тест на крайний случай — умножение на нулевую матрицу

3. Тест перемножения неквадратных прямоугольных матриц

Метод `CheckTestOutputData` выполняет покомпонентное сравнение вещественных
значений результирующего
массива с эталоном с точностью до $10^{-9}$. Все тесты успешно пройдены на
гибридной конфигурации,
подтверждая корректность схемы сбора векторов в `MasterCollect`.

Эксперимент показал реальное ускорение: максимальный результат достигнут при
конфигурации 4 Ranks × 2
Threads, где время сократилось до $0.227$ сек ($S = 2.388$). На матрицах
небольшого объёма
($1000 \times 1000$, плотность $1\%$) конфигурация с большим числом
MPI-процессов (4×2) оказалась
эффективнее, чем та, где делается упор на потоки (2×4). Разбиение на ранги
уменьшило локальный объём
данных на узле и улучшило локальность кэша.

Падение эффективности на большом числе воркеров (до $24.5\% - 29.8\%$)
иллюстрирует высокую цену
межпроцессного взаимодействия и накладных расходов. На матрицах малого порядка
время вычислений мало,
поэтому затраты на упаковку `size_t` в `uint64_t` и их передачу через
`MPI_Bcast` начинают доминировать
над полезной работой. Кроме того, рантайм oneTBB добавляет накладные расходы на
управление микрозадачами.

## 10. Выводы

Разработанная гибридная версия успешно объединила MPI и Intel oneTBB,
продемонстрировав корректность
и переносимость типов данных благодаря предварительной буферизации векторов.
Локальные замеры подтвердили,
что для небольших разреженных матриц иерархический параллелизм избыточен —
латентность сети и системные
вызовы при обмене данными существенно снижают эффективность утилизации ядер.

Гибридная схема оправдана при обработке сверхбольших разреженных массивов на
кластерах, которые нельзя
разместить в памяти одного узла. В таких сценариях объём полезной работы внутри
потока компенсирует
накладные расходы на сетевой интерконнект, синхронизацию и сбор результатов
мастером.
