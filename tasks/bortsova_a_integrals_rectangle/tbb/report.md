## 1. Контекст

TBB-версия алгоритма использует задачно-ориентированную модель параллелизма с автоматическим балансированием нагрузки. В
отличие от OpenMP, где распределение итераций фиксировано на этапе входа в параллельную область, TBB динамически
перераспределяет работу между потоками через механизм work-stealing, что особенно эффективно при неравномерной нагрузке.

## 2. Постановка задачи

Постановка задачи идентична последовательной версии (см. `seq/report.md`). Baseline для измерения ускорения —
последовательная версия с временем выполнения 0.225 секунды.

## 3. Базовый алгоритм

Алгоритм метода средних точек для вычисления многомерного интеграла остается неизменным. Параллелизация достигается
через примитив `tbb::parallel_reduce`.

## 4. Схема распараллеливания

### Выбор примитива

Используется `tbb::parallel_reduce` — паттерн для параллельного вычисления редукции. Он оптимален для задач, где нужно:

- Обработать большой диапазон данных
- Вычислить агрегированное значение (сумма, произведение, минимум и т.д.)
- Обеспечить эффективное объединение частичных результатов

Другие варианты:

- `tbb::parallel_for` — не подходит, так как требует явного управления локальными аккумуляторами
- `tbb::task_group` — избыточен для данной структуры вычислений

### Определение диапазона работы

```cpp
tbb::blocked_range<int64_t> range(0, total_points);
```

Диапазон охватывает все `total_points = N^n` точек сетки. TBB представляет его как полуинтервал `[0, total_points)`.

### Grain size

Grain size (размер зерна) определяет минимальный размер блока работы, который не будет дальше делиться. В реализации
используется значение по умолчанию, которое TBB вычисляет автоматически на основе размера диапазона.

**Влияние grain size:**

- **Слишком малый:** Высокий overhead на создание и управление задачами, частые переключения контекста
- **Слишком большой:** Плохая балансировка нагрузки, простаивающие потоки
- **Автоматический:** TBB использует эвристику, основанную на размере диапазона и числе доступных потоков

Для данной задачи с равномерной нагрузкой автоматический выбор оптимален.

### Partitioner

Используется partitioner по умолчанию — `auto_partitioner`. Он автоматически выбирает стратегию разбиения диапазона на
основе характеристик задачи:

- Начинает с крупных блоков для минимизации overhead
- Динамически уменьшает размер блоков при обнаружении дисбаланса
- Использует work-stealing для перераспределения работы между потоками

Другие варианты:

- `simple_partitioner` — статическое разбиение, аналогично OpenMP `schedule(static)`
- `affinity_partitioner` — оптимизирует локальность кэша, полезен для повторяющихся вычислений

Для данной задачи `auto_partitioner` обеспечивает лучший баланс между overhead и эффективностью.

### Ограничение конкуренции

Число рабочих потоков контролируется через `tbb::global_control`:

```cpp
tbb::global_control control(tbb::global_control::max_allowed_parallelism, num_threads);
```

Это ограничивает максимальное число одновременно активных worker-потоков. В отличие от OpenMP, где `OMP_NUM_THREADS`
задает точное число потоков, TBB может использовать меньше потоков, если задача не требует полного параллелизма.

### Структура parallel_reduce

```cpp
double sum = tbb::parallel_reduce(
    tbb::blocked_range<int64_t>(0, total_points),
    0.0,  // начальное значение
    [&](const tbb::blocked_range<int64_t>& r, double local_sum) {
        // Фаза накопления: обработка блока [r.begin(), r.end())
        for (int64_t idx = r.begin(); idx != r.end(); ++idx) {
            // вычисление многомерных индексов
            // формирование точки
            local_sum += func_(point);
        }
        return local_sum;
    },
    std::plus<>()  // Фаза объединения: сложение частичных сумм
);
```

**Фаза накопления (reduction body):**

Каждый поток обрабатывает свой блок индексов и вычисляет локальную сумму. Блоки независимы, гонки отсутствуют.

**Фаза объединения (join):**

TBB автоматически объединяет частичные суммы через `std::plus<>`. Объединение выполняется иерархически (дерево
редукции), что минимизирует синхронизацию.

## 5. Детали реализации

**Файлы:**

- `tbb/include/ops_tbb.hpp` — объявление класса `BortsovaAIntegralsRectangleTBB`
- `tbb/src/ops_tbb.cpp` — реализация методов

**Изменения относительно SEQ:**

1. Замена простого цикла `for` на `tbb::parallel_reduce`
2. Добавление `tbb::global_control` для ограничения числа потоков
3. Использование лямбда-функций для определения тела редукции

**Локальные аккумуляторы:**

TBB автоматически управляет локальными копиями аккумулятора `local_sum` для каждого блока работы. Программисту не нужно
явно создавать вектор локальных сумм, как в STL-версии.

**Объединение результатов:**

Объединение выполняется через `std::plus<>`, который TBB вызывает автоматически при слиянии результатов двух блоков. Это
обеспечивает потокобезопасность.

**Работа с памятью:**

- `midpoints_` — общий вектор, только чтение, безопасен для параллельного доступа
- `point` — локальная переменная внутри лямбды, создается в стеке каждого потока
- Временные переменные для индексов — также локальные

TBB не требует специальных аллокаторов (`cache_aligned_allocator`) для данной задачи, так как отсутствует false sharing
(каждый поток работает с независимым диапазоном индексов).

## 6. Проверка корректности

**Метод сравнения:**

Результаты TBB-версии сравниваются с результатами последовательной версии с относительной погрешностью `1e-3`.

**Набор тестов:**

Все 10 функциональных тестов проходят успешно. Детальное описание см. в корневом `report.md`.

**Проверка на границах:**

1. **Малое число точек** (`N = 1`) — TBB корректно обрабатывает случай, когда диапазон не делится
2. **Большое число точек** (`N = 200`) — проверяет корректность иерархического объединения частичных сумм
3. **Различные размерности** — универсальность параллелизации

**Детерминированность:**

TBB гарантирует детерминированность результата при фиксированном числе потоков и одинаковом порядке объединения
частичных сумм. Повторные запуски дают идентичные результаты.

## 8. Результаты

**Baseline (SEQ):** 0.225 секунды

### Режим task_run

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.239      | 0.94      | 94.0%         |
| 2       | 0.118      | 1.91      | 95.5%         |
| 5       | 0.052      | 4.33      | 86.6%         |
| 7       | 0.037      | 6.08      | 86.9%         |

### Режим task_pipeline

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.238      | 0.95      | 95.0%         |
| 2       | 0.118      | 1.91      | 95.5%         |
| 5       | 0.050      | 4.50      | 90.0%         |
| 7       | 0.039      | 5.77      | 82.4%         |

**Комментарий о балансировке и overhead:**

1. **Близость к SEQ на 1 потоке:** TBB с одним потоком показывает время 0.239 сек, что практически совпадает с SEQ
(0.225 сек). Разница ~6% — это overhead TBB runtime на управление задачами. В отличие от OpenMP, TBB не дает
дополнительных оптимизаций компилятора при одном потоке.

2. **Отличная масштабируемость 2-5 потоков:**
   - 2 потока: ускорение 1.91x, эффективность 95.5% — почти идеальное масштабирование
   - 5 потоков: ускорение 4.33x, эффективность 86.6% — хорошая балансировка
   - Высокая эффективность указывает на минимальный overhead и равномерное распределение работы

3. **Хорошее масштабирование на 7 потоках:** Ускорение 6.08x при эффективности 86.9% — лучший результат среди всех
потоковых технологий. TBB эффективно использует доступные ядра благодаря work-stealing.

4. **task_run vs task_pipeline:** Оба режима показывают близкие результаты, что говорит о стабильности TBB runtime.

**Преимущества TBB:**

- **Автоматическая балансировка:** Work-stealing обеспечивает равномерную загрузку потоков даже при небольших вариациях
времени выполнения итераций
- **Низкий overhead:** Эффективность 86-95% на всех конфигурациях
- **Масштабируемость:** Линейный рост производительности до 7 потоков
- **Простота использования:** Не требует явного управления локальными аккумуляторами и синхронизацией

**Узкие места:**

- Небольшой overhead TBB runtime (~6%) заметен на малых задачах
- При очень большом числе потоков (>10) возможно снижение эффективности из-за конкуренции за память
- Для задач с очень неравномерной нагрузкой может потребоваться настройка grain size

## 9. Выводы

TBB-версия алгоритма демонстрирует наилучшую масштабируемость среди всех потоковых технологий.
Для данной задачи вычисления многомерного интеграла TBB является оптимальным выбором благодаря:

1. Высокой эффективности (86-95%)
2. Отличной масштабируемости (линейный рост до 7 потоков)
3. Простоте реализации (один вызов `parallel_reduce`)
4. Автоматической балансировке нагрузки

TBB показывает лучшие результаты по сравнению с OpenMP и STL на конфигурациях с 5-7 потоками, что делает его
предпочтительным выбором для многоядерных систем.
