## 1. Контекст

STL-версия алгоритма демонстрирует ручное управление потоками через стандартную библиотеку C++. В отличие от OpenMP и
TBB, где распределение работы и синхронизация выполняются автоматически, здесь программист явно контролирует создание
потоков, разбиение данных, объединение результатов и завершение потоков. Это дает максимальный контроль, но требует
большей аккуратности в реализации.

## 2. Постановка задачи

Постановка задачи идентична последовательной версии (см. `seq/report.md`). Baseline для измерения ускорения —

## 3. Базовый алгоритм

Алгоритм метода средних точек остается неизменным. Параллелизация достигается через явное создание потоков и ручное
разбиение диапазона индексов.

## 4. Схема распараллеливания

### Разбиение данных на диапазоны

Общий диапазон индексов `[0, total_points)` делится на `num_threads` равных (или почти равных) поддиапазонов:

```cpp
int64_t chunk_size = total_points / num_threads;
int64_t remainder = total_points % num_threads;

for (int t = 0; t < num_threads; ++t) {
    int64_t start = t * chunk_size + std::min(static_cast<int64_t>(t), remainder);
    int64_t end = start + chunk_size + (t < remainder ? 1 : 0);
    // поток t обрабатывает диапазон [start, end)
}
```

**Обработка остатка:**

Если `total_points` не делится нацело на `num_threads`, первые `remainder` потоков получают на одну итерацию больше. Это
обеспечивает равномерное распределение работы с максимальной разницей в 1 итерацию между потоками.

### Что делает каждый поток

Каждый поток выполняет идентичную работу на своем диапазоне:

1. Получает границы `[start, end)` своего диапазона
2. Инициализирует локальную сумму нулем
3. Перебирает индексы от `start` до `end - 1`
4. Для каждого индекса:
   - Вычисляет многомерные индексы через операции деления и остатка
   - Формирует точку из предвычисленных координат
   - Вычисляет значение функции
   - Добавляет к локальной сумме
5. Записывает локальную сумму в выделенную ячейку вектора `partial_sums[t]`

### Локальные результаты

Для хранения частичных сумм используется вектор:

```cpp
std::vector<double> partial_sums(num_threads, 0.0);
```

Каждый поток записывает свой результат в отдельную ячейку по индексу потока. Это исключает гонки, так как:

- Каждая ячейка принадлежит ровно одному потоку
- Запись выполняется только один раз в конце работы потока
- Чтение выполняется главным потоком только после `join()` всех рабочих потоков

### Порядок create → work → join

**Создание потоков:**

```cpp
std::vector<std::thread> threads;
threads.reserve(num_threads - 1);

for (int t = 1; t < num_threads; ++t) {
    threads.emplace_back([&, t, start, end] {
        // работа потока t
        partial_sums[t] = ComputePartialSum(start, end);
    });
}
```

Создается `num_threads - 1` рабочих потоков. Главный поток также выполняет свою часть работы (диапазон потока 0), что
избегает простоя.

**Работа главного потока:**

```cpp
partial_sums[0] = ComputePartialSum(start_0, end_0);
```

Главный поток обрабатывает свой диапазон параллельно с рабочими потоками.

**Завершение потоков:**

```cpp
for (auto& thread : threads) {
    thread.join();
}
```

Главный поток ожидает завершения всех рабочих потоков. `join()` блокирует выполнение до тех пор, пока соответствующий
поток не завершится. Только после этого безопасно читать `partial_sums`.

**Важно:** В отличие от антипаттерна из примера `example_threads`, где `join()` вызывается внутри цикла создания потоков
(что сериализует работу), здесь создание и завершение разделены, что обеспечивает истинный параллелизм.

### Синхронизация

**Отсутствие явной синхронизации во время работы:**

Потоки работают с независимыми диапазонами индексов и записывают результаты в независимые ячейки `partial_sums`. Это
исключает необходимость в `std::mutex`, `std::atomic` или других примитивах синхронизации во время вычислений.

**Синхронизация через join:**

`std::thread::join()` обеспечивает барьер памяти (memory barrier), гарантирующий, что все записи, выполненные потоком до
его завершения, видны главному потоку после `join()`.

**Объединение результатов:**

После завершения всех потоков главный поток последовательно суммирует частичные результаты:

```cpp
double sum = 0.0;
for (double partial : partial_sums) {
    sum += partial;
}
```

Это выполняется последовательно и не требует синхронизации.

## 5. Детали реализации

**Файлы:**

- `stl/include/ops_stl.hpp` — объявление класса `BortsovaAIntegralsRectangleSTL`
- `stl/src/ops_stl.cpp` — реализация методов

**Карта поток → диапазон:**

| Поток | Начало (start)                                      | Конец (end)                                         |
|-------|-----------------------------------------------------|-----------------------------------------------------|
| 0     | 0                                                   | chunk_size + (0 < remainder ? 1 : 0)                |
| 1     | end[0]                                              | end[0] + chunk_size + (1 < remainder ? 1 : 0)       |
| ...   | ...                                                 | ...                                                 |
| t     | t * chunk_size + min(t, remainder)                  | start[t] + chunk_size + (t < remainder ? 1 : 0)     |

**Схема объединения локальных результатов:**

1. Каждый поток записывает свою частичную сумму в `partial_sums[t]`
2. Главный поток после `join()` всех потоков последовательно суммирует элементы `partial_sums`
3. Итоговая сумма умножается на объем элементарного прямоугольника

**Метод ComputePartialSum:**

Для снижения когнитивной сложности и избежания дублирования кода вычисление частичной суммы вынесено в отдельный метод:

```cpp
double ComputePartialSum(int64_t start, int64_t end) const {
    double local_sum = 0.0;
    std::vector<double> point(dimension_);
    
    for (int64_t idx = start; idx < end; ++idx) {
        // вычисление многомерных индексов
        // формирование точки
        local_sum += func_(point);
    }
    
    return local_sum;
}
```

Этот метод вызывается как главным потоком, так и рабочими потоками.

**Захват переменных в лямбде:**

```cpp
threads.emplace_back([&, t, start, end] { ... });
```

- `[&]` — захват по ссылке для доступа к `partial_sums`, `midpoints_`, `func_` и другим членам класса
- `t, start, end` — захват по значению для гарантии, что каждый поток получает правильные границы своего диапазона

## 6. Проверка корректности

**Метод сравнения:**

Результаты STL-версии сравниваются с результатами последовательной версии с относительной погрешностью `1e-3`.

**Набор тестов:**

Все 10 функциональных тестов проходят успешно. Детальное описание см. в корневом `report.md`.

**Крайние случаи:**

1. **Число потоков больше числа точек:** Если `num_threads > total_points`, некоторые потоки получают пустые диапазоны и
сразу завершаются с нулевой суммой.

2. **Один поток:** При `num_threads = 1` не создается дополнительных потоков, главный поток обрабатывает весь диапазон.

3. **Неравномерное деление:** Алгоритм разбиения корректно обрабатывает случаи, когда `total_points % num_threads != 0`.

**Проверка на отсутствие гонок:**

1. **ThreadSanitizer:** Запуск с `-fsanitize=thread` (на GCC/Clang) не выявляет гонок данных.

2. **Ручная проверка:** Каждый поток работает с независимым диапазоном индексов и записывает в независимую ячейку
`partial_sums`.

3. **Детерминированность:** Повторные запуски с одинаковым числом потоков дают идентичные результаты.

## 8. Результаты

**Baseline (SEQ):** 0.225 секунды

### Режим task_run

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.226      | 1.00      | 100.0%        |
| 2       | 0.114      | 1.97      | 98.5%         |
| 5       | 0.047      | 4.79      | 95.8%         |
| 7       | 0.042      | 5.36      | 76.6%         |

### Режим task_pipeline

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.228      | 0.99      | 99.0%         |
| 2       | 0.115      | 1.96      | 98.0%         |
| 5       | 0.049      | 4.59      | 91.8%         |
| 7       | 0.043      | 5.23      | 74.7%         |

**Комментарий о накладных расходах create/join:**

1. **Минимальный overhead на 1 потоке:** STL-версия с одним потоком показывает время 0.226 сек, практически идентичное
SEQ (0.225 сек). Разница <1% — это минимальные накладные расходы на создание вектора `partial_sums` и вызов
`ComputePartialSum`.

2. **Отличное масштабирование 2-5 потоков:**
   - 2 потока: ускорение 1.97x, эффективность 98.5% — почти идеальное масштабирование
   - 5 потоков: ускорение 4.79x, эффективность 95.8% — отличная балансировка
   - Высокая эффективность указывает на минимальный overhead создания потоков и равномерное распределение работы

3. **Снижение эффективности на 7 потоках:** Эффективность падает до 76.6% (task_run) и 74.7% (task_pipeline). Причины:

- **Overhead создания/завершения потоков:** Создание 6 дополнительных потоков и вызов `join()` для каждого требует
   времени
  - **Конкуренция за ресурсы:** При 7 потоках возникает конкуренция за кэш и пропускную способность памяти
  - **Малое время выполнения:** При времени ~0.04 сек overhead становится заметным

4. **Сравнение с TBB:** STL показывает лучшие результаты на 5 потоках (4.79x vs 4.33x), но уступает на 7 потоках. Это
связано с тем, что TBB использует work-stealing для балансировки, а STL — статическое разбиение.

5. **task_run vs task_pipeline:** Оба режима показывают близкие результаты с небольшим преимуществом task_run (~2-5%).

**Преимущества ручного управления:**

- Полный контроль над разбиением данных
- Минимальный overhead на малом числе потоков
- Предсказуемое поведение (статическое разбиение)
- Отсутствие зависимости от внешних библиотек

**Недостатки:**

- Overhead создания/завершения потоков заметен на большом числе потоков
- Отсутствие автоматической балансировки нагрузки
- Больше кода по сравнению с OpenMP и TBB
- Необходимость явного управления локальными результатами

## 9. Выводы

1. **Высокая эффективность на малом числе потоков:** 95-98% на 2-5 потоках — лучший результат среди всех технологий.

2. **Простота и прозрачность:** Явное управление потоками делает код более понятным и предсказуемым.

3. **Минимальный overhead:** При правильном разбиении данных и отсутствии синхронизации во время работы overhead
минимален.

4. **Ограничения масштабируемости:** На большом числе потоков (7+) overhead создания/завершения потоков и конкуренция за
ресурсы снижают эффективность.

Для данной задачи вычисления многомерного интеграла STL является хорошим выбором при использовании 2-5 потоков,
обеспечивая эффективность 95-98% и полный контроль над параллелизмом.
