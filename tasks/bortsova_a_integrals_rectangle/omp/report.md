## 1. Контекст

OpenMP-версия алгоритма переносит вычислительно интенсивный цикл перебора точек сетки из последовательной реализации в
параллельную область с автоматическим распределением итераций между потоками. OpenMP выбран для демонстрации
директивного подхода к распараллеливанию, при котором минимальные изменения кода дают значительное ускорение.

## 2. Постановка задачи

Задача полностью совпадает с постановкой в последовательной версии (см. `seq/report.md`). Входные и выходные данные,
ограничения и критерии корректности остаются неизменными. Baseline для измерения ускорения — последовательная версия с
временем выполнения 0.225 секунды на тестовой задаче.

## 3. Базовый алгоритм

Алгоритм идентичен последовательной версии: метод средних точек для вычисления многомерного интеграла. Основное отличие
— параллелизация цикла перебора `N^n` точек сетки.

## 4. Схема распараллеливания

### Параллельная область

Распараллеливается основной цикл в методе `RunImpl()`, который перебирает все точки сетки по линейному индексу от `0` до
`total_points - 1`, где `total_points = N^n`.

### Директива OpenMP

```cpp
#pragma omp parallel for default(none) shared(total_points, midpoints_, dimension_, steps_, func_) \
    reduction(+:sum) schedule(static)
for (int64_t idx = 0; idx < total_points; ++idx) {
  // вычисление многомерных индексов
  // формирование точки
  // вычисление значения функции
  sum += func_(point);
}
```

### Атрибуты переменных

**shared:**

- `total_points` — общее число точек, константа для всех потоков
- `midpoints_` — предвычисленные координаты средних точек, только чтение
- `dimension_` — размерность задачи, константа
- `steps_` — число шагов разбиения, константа
- `func_` — указатель на функцию, только чтение

**private (неявно):**

- `idx` — индекс цикла, каждый поток обрабатывает свой диапазон
- `point` — локальный вектор координат точки
- Временные переменные для вычисления многомерных индексов

**reduction:**

- `sum` — аккумулятор суммы значений функции. OpenMP автоматически создает локальную копию для каждого потока и
объединяет их в конце параллельной области через операцию сложения.

### Выбор schedule

Используется `schedule(static)` — статическое распределение итераций. Диапазон индексов делится на равные блоки по числу
потоков. Это оптимально для данной задачи, так как:

- Все итерации имеют одинаковую вычислительную сложность
- Отсутствует дисбаланс нагрузки
- Минимальные накладные расходы на распределение работы
- Хорошая локальность данных (последовательные индексы обрабатываются одним потоком)

Альтернативы `schedule(dynamic)` или `schedule(guided)` не дают преимуществ, так как нагрузка равномерна.

### Синхронизация

**Неявный барьер:**

В конце параллельной области `#pragma omp parallel for` существует неявный барьер. Все потоки ожидают завершения своих
итераций, затем OpenMP выполняет редукцию локальных сумм в общую переменную `sum`. Только после этого управление
переходит к следующей инструкции.

**Отсутствие гонок:**

- Каждый поток работает с независимым диапазоном индексов
- Локальные переменные (`point`, временные индексы) создаются в стеке каждого потока
- Общие данные (`midpoints_`, `func_`) используются только для чтения
- Запись в `sum` защищена механизмом `reduction`

## 5. Детали реализации

**Файлы:**

- `omp/include/ops_omp.hpp` — объявление класса `BortsovaAIntegralsRectangleOMP`
- `omp/src/ops_omp.cpp` — реализация методов

**Изменения относительно SEQ:**

Единственное изменение — добавление директивы `#pragma omp parallel for` перед основным циклом в `RunImpl()`. Методы
`ValidationImpl()`, `PreProcessingImpl()`, `PostProcessingImpl()` остаются идентичными последовательной версии.

**Управление числом потоков:**

Число потоков задается через переменную окружения `PPC_NUM_THREADS`, которая автоматически экспортируется также как
`OMP_NUM_THREADS`. OpenMP использует это значение для создания команды потоков.

**Устранение рисков гонок:**

1. **Атрибут `default(none)`** требует явного указания области видимости для всех переменных, что исключает случайное
разделение изменяемых данных.

2. **Reduction** гарантирует потокобезопасное накопление суммы без необходимости явных `atomic` или `critical` секций.

3. **Константность общих данных** (`midpoints_`, `func_`) исключает конфликты записи.

## 6. Проверка корректности

**Метод сравнения:**

Результаты OpenMP-версии сравниваются с результатами последовательной версии. Используется относительная погрешность
`1e-3`.

**Набор тестов:**

Все 10 функциональных тестов из `seq/report.md` проходят успешно для OpenMP-версии. Детальное описание тестов см. в
корневом `report.md`.

**Характерные случаи:**

1. **Малое число точек** (`N = 1`) — проверяет корректность при отсутствии реального параллелизма
2. **Большое число точек** (`N = 200`) — проверяет отсутствие ошибок накопления при параллельном суммировании
3. **Различные размерности** (1D, 2D, 3D) — проверяет универсальность распараллеливания

Все тесты детерминированы и дают идентичные результаты при повторных запусках с одинаковым числом потоков.

## 8. Результаты

**Baseline (SEQ):** 0.225 секунды

### Режим task_run

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.034      | 6.62      | 662.0%        |
| 2       | 0.032      | 7.03      | 351.5%        |
| 5       | 0.033      | 6.82      | 136.4%        |
| 7       | 0.033      | 6.82      | 97.4%         |

### Режим task_pipeline

| Потоков | Время, сек | Ускорение | Эффективность |
|---------|------------|-----------|---------------|
| 1       | 0.033      | 6.82      | 682.0%        |
| 2       | 0.037      | 6.08      | 304.0%        |
| 5       | 0.036      | 6.25      | 125.0%        |
| 7       | 0.036      | 6.25      | 89.3%         |

**Комментарий о масштабируемости:**

1. **Аномально высокое ускорение на 1 потоке:** OpenMP-версия показывает ускорение ~6.6x даже с одним потоком по
сравнению с SEQ. Это связано с дополнительными оптимизациями компилятора, которые активируются при использовании
OpenMP-директив (векторизация, улучшенное выравнивание данных).

2. **Слабое масштабирование 2-7 потоков:** Увеличение числа потоков с 2 до 7 не дает значительного прироста
производительности. Время остается в диапазоне 0.032-0.037 секунды. Это указывает на наличие узких мест:

- **Ограничение пропускной способности памяти:** При большом числе потоков возникает конкуренция за доступ к кэшу и
   оперативной памяти
  - **Overhead редукции:** Объединение локальных сумм требует синхронизации
- **Малое время выполнения:** При времени ~0.03 сек накладные расходы на создание потоков и барьеры становятся заметными

3. **task_run быстрее task_pipeline:** Режим `task_run` стабильно показывает лучшие результаты, так как не имеет
дополнительных накладных расходов на управление конвейером.

**Узкие места:**

- Редукция суммы в конце параллельной области
- Конкуренция за кэш L3 при большом числе потоков
- Относительно малый размер задачи (0.03 сек) делает overhead заметным

## 9. Выводы

OpenMP-версия алгоритма демонстрирует:

1. **Значительное ускорение:** Даже с одним потоком достигается ускорение ~6.6x благодаря оптимизациям компилятора,
активируемым OpenMP.

2. **Простоту реализации:** Добавление одной директивы `#pragma omp parallel for` с правильными атрибутами переменных
обеспечивает корректное распараллеливание.

3. **Ограниченную масштабируемость:** Увеличение числа потоков с 2 до 7 не дает пропорционального роста
производительности из-за ограничений памяти и overhead синхронизации.

Для данной задачи OpenMP является хорошим выбором благодаря простоте реализации и хорошему ускорению.
