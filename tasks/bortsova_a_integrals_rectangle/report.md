## 1. Введение

Численное интегрирование многомерных функций методом прямоугольников является вычислительно интенсивной задачей с
экспоненциальным ростом сложности при увеличении размерности. Данный отчет представляет сравнительный анализ пяти
реализаций алгоритма: последовательной (SEQ) и четырех параллельных (OpenMP, Intel TBB, std::thread, MPI+OpenMP).

**Цель работы:** Исследовать эффективность различных технологий параллельного программирования для задачи вычисления
многомерных интегралов и определить оптимальные области применения каждой технологии.

**Структура отчетов:**

Для каждой технологии подготовлен детальный отчет:

- `seq/report.md` — последовательная версия (baseline)
- `omp/report.md` — OpenMP версия
- `tbb/report.md` — Intel TBB версия
- `stl/report.md` — std::thread версия
- `all/report.md` — гибридная MPI+OpenMP версия

Данный документ является сводным отчетом, объединяющим результаты всех реализаций.

## 2. Постановка задачи

**Входные данные:**

- Подынтегральная функция `f(x_1, x_2, ..., x_n)` — функция n переменных
- Нижние границы интегрирования `a_i` для каждой размерности `i = 1..n`
- Верхние границы интегрирования `b_i` для каждой размерности `i = 1..n`
- Число шагов разбиения `N`

**Выходные данные:**

- Приближенное значение многомерного интеграла

**Ограничения:**

- Размерность `n >= 1`
- Число шагов `N > 0`
- Для каждой размерности: `a_i < b_i`
- Функция должна быть определена на всей области интегрирования

Детальная постановка с обработкой крайних случаев приведена в `seq/report.md`.

## 3. Базовый алгоритм

Метод прямоугольников (метод средних точек) аппроксимирует многомерный интеграл суммой значений функции в средних точках
элементарных n-мерных прямоугольников.

**Основные шаги:**

1. Для каждой размерности `i` вычисляется шаг разбиения: `h_i = (b_i - a_i) / N`
2. Вычисляется объем элементарного прямоугольника: `V = h_1 * h_2 * ... * h_n`
3. Предвычисляются координаты средних точек для каждой размерности
4. Перебираются все `N^n` точек сетки
5. Для каждой точки вычисляется значение функции и добавляется к сумме
6. Результат: `I ~= V * суммаf(x)`

**Сложность:** `O(N^n)` — экспоненциальная зависимость от размерности

Детальное описание алгоритма, инвариантов и критериев корректности приведено в `seq/report.md`.

## 4. Обзор реализаций

Все реализации используют единую архитектуру на основе `BaseTask` с методами `ValidationImpl`, `PreProcessingImpl`,
`RunImpl`, `PostProcessingImpl`. Различия заключаются только в способе параллелизации основного вычислительного цикла в
`RunImpl`.

### 4.1 SEQ — Последовательная версия

**Описание:** Эталонная реализация без параллелизма. Простой цикл перебора всех `N^n` точек.

**Особенности:**

- Детерминированность и воспроизводимость
- Минимальные накладные расходы
- Baseline для измерения ускорения

**Детали:** См. `seq/report.md`

### 4.2 OMP — OpenMP

**Описание:** Директивное распараллеливание через `#pragma omp parallel for` с автоматической редукцией суммы.

**Особенности:**

- Минимальные изменения кода (одна директива)
- Автоматическое управление потоками
- Статическое распределение итераций (`schedule(static)`)
- Неявный барьер и автоматическая редукция

**Детали:** См. `omp/report.md`

### 4.3 TBB — Intel Threading Building Blocks

**Описание:** Задачно-ориентированная модель с `tbb::parallel_reduce` и автоматической балансировкой через
work-stealing.

**Особенности:**

- Динамическое распределение работы
- Автоматический выбор grain size
- Иерархическое объединение частичных результатов
- Высокая эффективность использования ядер

**Детали:** См. `tbb/report.md`

### 4.4 STL — std::thread

**Описание:** Ручное управление потоками через стандартную библиотеку C++.

**Особенности:**

- Явное создание и завершение потоков
- Ручное разбиение диапазона на поддиапазоны
- Локальные аккумуляторы для каждого потока
- Полный контроль над параллелизмом

**Детали:** См. `stl/report.md`

### 4.5 ALL — MPI + OpenMP

**Описание:** Двухуровневая гибридная параллелизация: межпроцессный уровень (MPI) и внутрипроцессный уровень (OpenMP).

**Особенности:**

- Распределение работы между MPI-процессами
- Дополнительное распараллеливание внутри каждого процесса через OpenMP
- Коллективные операции `MPI_Reduce` и `MPI_Bcast`
- Масштабирование на кластерных системах

**Детали:** См. `all/report.md`

## 5. Единая методика тестирования

### 5.1 Проверка корректности

**Функциональные тесты:** 10 тестовых случаев, покрывающих различные сценарии:

1. **Постоянная функция 1D** — `f(x) = 5` на `[0, 10]`, `N = 100`
   - Ожидаемый результат: `50.0`
   - Проверяет базовую корректность для простейшего случая

2. **Линейная функция 1D** — `f(x) = 2x` на `[0, 5]`, `N = 100`
   - Ожидаемый результат: `25.0`
   - Проверяет корректность для линейной зависимости

3. **Квадратичная функция 1D** — `f(x) = x²` на `[-1, 1]`, `N = 100`
   - Ожидаемый результат: `~= 0.667`
   - Проверяет работу с отрицательными границами

4. **Произведение переменных 2D** — `f(x,y) = xy` на `[0,1]²`, `N = 100`
   - Ожидаемый результат: `0.25`
   - Проверяет двумерное интегрирование

5. **Сумма переменных 2D** — `f(x,y) = x + y` на `[0,1]²`, `N = 100`
   - Ожидаемый результат: `1.0`
   - Проверяет аддитивность

6. **Постоянная функция 3D** — `f(x,y,z) = 1` на `[0,1]³`, `N = 50`
   - Ожидаемый результат: `1.0`
   - Проверяет трехмерный случай

7. **Сумма переменных 3D** — `f(x,y,z) = x + y + z` на `[0,1]³`, `N = 50`
   - Ожидаемый результат: `1.5`
   - Проверяет сложную трехмерную функцию

8. **Сумма квадратов 2D** — `f(x,y) = x² + y²` на `[0,1]²`, `N = 100`
   - Ожидаемый результат: `~= 0.667`
   - Проверяет нелинейные функции

9. **Один шаг разбиения** — `f(x) = x` на `[0,2]`, `N = 1`
   - Ожидаемый результат: `2.0`
   - Проверяет крайний случай минимального разбиения

10. **Квадратичная функция с большим диапазоном** — `f(x) = x²` на `[0,10]`, `N = 200`
    - Ожидаемый результат: `~= 333.33`
    - Проверяет масштабируемость

**Критерий корректности:** Относительная погрешность <= 1e-3 при сравнении с эталонными значениями.

**Результат:** Все 10 тестов проходят успешно для всех пяти реализаций.

**Режимы тестирования:**

2. **task_pipeline** — выполнение через конвейер задач

Во всех экспериментах режим `task_run` показывает лучшие результаты благодаря отсутствию накладных расходов на
управление конвейером.

**Команды запуска:**

```powershell

cd build\bin
```

## 7. Сравнительный анализ технологий

### 7.1 Интерпретация различий

**OpenMP:**

- **Преимущества:** Минимальные изменения кода, автоматическое управление потоками, дополнительные оптимизации
компилятора
- **Недостатки:** Слабое масштабирование при увеличении числа потоков с 2 до 7, ограничение пропускной способности
памяти
- **Оптимальная конфигурация:** 2 потока (ускорение 7.03x)
- **Причина аномального ускорения на 1 потоке:** Компилятор активирует векторизацию и другие оптимизации при наличии
OpenMP-директив

**Intel TBB:**

- **Преимущества:** Лучшая масштабируемость (86-95% эффективности), автоматическая балансировка через work-stealing,
стабильная производительность
- **Недостатки:** Небольшой overhead runtime (~6%) на малых задачах
- **Оптимальная конфигурация:** 7 потоков (ускорение 6.08x, эффективность 86.9%)
- **Почему лучше других на 7 потоках:** Динамическое перераспределение работы компенсирует небольшие вариации времени
выполнения

**std::thread:**

- **Преимущества:** Минимальный overhead на малом числе потоков, полный контроль, отличная эффективность (95-98% на 2-5
потоках)
- **Недостатки:** Overhead создания/завершения потоков заметен при 7+ потоках, отсутствие автоматической балансировки
- **Оптимальная конфигурация:** 5 потоков (ускорение 4.79x, эффективность 95.8%)
- **Когда выбирать:** Малое число потоков (2-5), равномерная нагрузка, отсутствие зависимостей от внешних библиотек

**MPI + OpenMP:**

- **Преимущества:** Масштабирование на кластерных системах, хорошая эффективность при правильном балансе процессов и
потоков
- **Недостатки:** Коммуникационные издержки (`MPI_Reduce`, `MPI_Bcast`), сложность реализации, избыточный параллелизм
при большом числе работников
- **Оптимальная конфигурация:** 2 процесса *7 потоков (ускорение 7.03x) или 7 процессов* 1 поток (ускорение 5.92x,
эффективность 84.6%)
- **Когда выбирать:** Кластерные системы, задачи с временем выполнения >1 сек (чтобы коммуникационные издержки были
незначительны)

### 7.2 Общие наблюдения

1. **Ограничение пропускной способности памяти:** При большом числе потоков (>5) все технологии сталкиваются с
конкуренцией за доступ к памяти, что ограничивает масштабируемость.

2. **Малый размер задачи:** Время выполнения ~0.03-0.05 сек делает overhead создания потоков и синхронизации заметным.
Для более крупных задач (время >1 сек) относительный overhead снизится.

3. **Режим task_run vs task_pipeline:** Во всех технологиях task_run показывает лучшие результаты (на 2-10% быстрее)
благодаря отсутствию накладных расходов на управление конвейером.

4. **Физические vs логические ядра:** Система имеет 16 физических ядер и 22 логических процессора. Эффективность резко
падает при числе работников >16 из-за гиперпоточности.
