## 1. Контекст

Гибридная версия алгоритма демонстрирует двухуровневую параллелизацию: межпроцессный параллелизм через MPI и
внутрипроцессный параллелизм через OpenMP. Такой подход позволяет эффективно использовать ресурсы современных
вычислительных систем, где узлы кластера содержат многоядерные процессоры. MPI обеспечивает масштабирование на уровне
процессов (узлов), а OpenMP — на уровне потоков внутри каждого процесса.

## 2. Постановка задачи

Постановка задачи идентична последовательной версии (см. `seq/report.md` и корневой `report.md`). Baseline для измерения
ускорения — последовательная версия с временем выполнения 0.225 секунды.

## 3. Базовый алгоритм

Алгоритм метода средних точек остается неизменным. Параллелизация выполняется на двух уровнях:

1. **Межпроцессный уровень (MPI):** Общее пространство точек делится между MPI-процессами
2. **Внутрипроцессный уровень (OpenMP):** Каждый процесс дополнительно распараллеливает свою часть работы между потоками

## 4. Межпроцессная схема (MPI)

### Роли rankов

Все MPI-процессы выполняют одинаковую роль — вычисление частичной суммы на своем диапазоне индексов. Нет выделенного
master-процесса, который только координирует работу. Это обеспечивает максимальную утилизацию ресурсов.

**Особая роль rank 0:**

- Собирает частичные суммы от всех процессов через `MPI_Reduce`
- Выполняет финальное умножение на объем элементарного прямоугольника
- Рассылает итоговый результат всем процессам через `MPI_Bcast`

### Распределение данных

Общий диапазон индексов `[0, total_points)` делится между процессами аналогично разбиению в STL-версии:

```cpp
int rank, size;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

int64_t chunk_size = total_points / size;
int64_t remainder = total_points % size;

int64_t start = rank * chunk_size + std::min(static_cast<int64_t>(rank), remainder);
int64_t end = start + chunk_size + (rank < remainder ? 1 : 0);
```

Каждый процесс получает примерно равную часть работы с максимальной разницей в 1 итерацию.

### MPI-коммуникации

**MPI_Reduce:**

```cpp
double local_sum = /* вычисление на диапазоне [start, end) */;
double global_sum = 0.0;

MPI_Reduce(&local_sum, &global_sum, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);
```

- Собирает частичные суммы `local_sum` от всех процессов
- Выполняет операцию `MPI_SUM`
- Результат `global_sum` доступен только на процессе 0
- Это коллективная операция — все процессы должны вызвать её одновременно

**MPI_Bcast:**

```cpp
if (rank == 0) {
    result = global_sum * volume;
}

MPI_Bcast(&result, 1, MPI_DOUBLE, 0, MPI_COMM_WORLD);
```

- Процесс 0 рассылает итоговый результат всем процессам
- После этого все процессы имеют одинаковое значение `result`
- Необходимо для корректной работы тестовой инфраструктуры, которая проверяет результат на всех процессах

**Зачем нужны вызовы:**

- `MPI_Reduce` — объединяет частичные результаты без необходимости явной передачи сообщений между всеми парами процессов
- `MPI_Bcast` — обеспечивает согласованность результата на всех процессах
- Ручная передача сообщений была бы сложнее и медленнее

## 5. Внутрипроцессная схема (OpenMP)

### Применение OpenMP внутри каждого процесса

Каждый MPI-процесс распараллеливает свой диапазон `[start, end)` между потоками через OpenMP:

```cpp
double local_sum = 0.0;

#pragma omp parallel for default(none) shared(start, end, midpoints_, dimension_, steps_, func_) \
    reduction(+:local_sum) schedule(static)
for (int64_t idx = start; idx < end; ++idx) {
    // вычисление многомерных индексов
    // формирование точки
    local_sum += func_(point);
}
```

### Разделение работы между потоками

Внутри каждого процесса OpenMP делит диапазон `[start, end)` на блоки по числу потоков. Например, если процесс 0
обрабатывает индексы `[0, 1000000)` с 4 потоками:

- Поток 0: `[0, 250000)`
- Поток 1: `[250000, 500000)`
- Поток 2: `[500000, 750000)`
- Поток 3: `[750000, 1000000)`

### Объединение локальных результатов

1. **Внутри процесса:** OpenMP автоматически объединяет частичные суммы потоков через `reduction(+:local_sum)`
2. **Между процессами:** `MPI_Reduce` объединяет `local_sum` от всех процессов

## 6. Детали реализации

**Файлы:**

- `all/include/ops_all.hpp` — объявление класса `BortsovaAIntegralsRectangleALL`
- `all/src/ops_all.cpp` — реализация методов

### Где стоят MPI-вызовы

**Инициализация (в тестовой инфраструктуре):**

```cpp
MPI_Init(&argc, &argv);
```

Вызывается один раз в начале программы.

**В методе RunImpl:**

1. `MPI_Comm_rank` — получение номера процесса
2. `MPI_Comm_size` — получение общего числа процессов
3. Вычисление `local_sum` с OpenMP
4. `MPI_Reduce` — сбор частичных сумм на процессе 0
5. Умножение на объем (только на процессе 0)
6. `MPI_Bcast` — рассылка результата всем процессам

**Финализация (в тестовой инфраструктуре):**

```cpp
MPI_Finalize();
```

Вызывается один раз в конце программы.

### Потенциальные узкие места

1. **Коммуникационные издержки:**
   - `MPI_Reduce` требует синхронизации всех процессов
   - `MPI_Bcast` требует передачи данных от процесса 0 всем остальным
   - При малом размере задачи время коммуникации может превышать время вычислений

2. **Дисбаланс нагрузки:**
   - Если `total_points % size != 0`, некоторые процессы получают на 1 итерацию больше
   - При большом числе процессов и малом числе точек некоторые процессы могут получить пустые диапазоны

3. **Конкуренция за ресурсы:**

- При большом числе процессов и потоков одновременно (например, 7 процессов * 7 потоков = 49 работников) возникает
   конкуренция за кэш и память
  - Overhead создания потоков в каждом процессе

4. **Барьеры:**
   - `MPI_Reduce` и `MPI_Bcast` содержат неявные барьеры — все процессы ожидают самого медленного
   - OpenMP `parallel for` также содержит неявный барьер в конце

## 7. Проверка корректности

**Метод сравнения:**

Результаты гибридной версии сравниваются с результатами последовательной версии с относительной погрешностью `1e-3`.

**Набор тестов:**

Все 10 функциональных тестов проходят успешно для различных конфигураций `ranks * threads`. Детальное описание см. в
корневом `report.md`.

**Проверка на согласованность между rank-ами:**

После `MPI_Bcast` все процессы имеют одинаковое значение результата. Тестовая инфраструктура проверяет результат на
каждом процессе независимо.

**Характерные случаи:**

1. **1 процесс * 1 поток:** Должно совпадать с SEQ
2. **1 процесс * N потоков:** Должно совпадать с чистым OpenMP
3. **N процессов * 1 поток:** Проверяет корректность MPI-коммуникаций
4. **N процессов * M потоков:** Проверяет взаимодействие двух уровней параллелизма

### Комментарий о цене коммуникации и синхронизации

**Анализ по конфигурациям:**

1. **1 процесс * N потоков:** Эквивалентно чистому OpenMP. Эффективность 68-96% — хорошие результаты без
коммуникационных издержек MPI.

2. **N процессов * 1 поток:** Чистый MPI без внутрипроцессного параллелизма.
   - 2 процесса: эффективность 98% — отличное масштабирование
   - 5 процессов: эффективность 83% — хорошо
   - 7 процессов: эффективность 85% — хорошо
   - Коммуникационные издержки минимальны благодаря одному `MPI_Reduce` и одному `MPI_Bcast`

3. **Малое число процессов * среднее число потоков (2*2, 2*5):**
   - Эффективность 54-91% — хороший баланс между двумя уровнями параллелизма
   - Лучшее абсолютное время: 0.032 сек при 2*7 (ускорение 7.03x)

4. **Большое число процессов * большое число потоков (5*5, 7*7):**
   - Эффективность падает до 12-21%
   - Причины:
     - **Избыточный параллелизм:** 25-49 работников на 16 физических ядер
     - **Конкуренция за ресурсы:** Кэш, память, ядра CPU
     - **Overhead:** Создание потоков в каждом процессе + MPI-коммуникации
     - **Гиперпоточность:** Логические процессоры менее эффективны, чем физические ядра

**Оптимальные конфигурации:**

- **Лучшее время:** 2 процесса * 7 потоков = 0.032 сек (ускорение 7.03x)
- **Лучшая эффективность:** 2 процесса * 1 поток = 98% (но медленнее по абсолютному времени)
- **Баланс:** 2 процесса * 5 потоков = 0.041 сек, эффективность 55%

**Цена коммуникации:**

При малом размере задачи коммуникационные издержки `MPI_Reduce` и `MPI_Bcast` становятся заметными. Для более крупных
задач (время выполнения >1 сек) относительная цена коммуникации снижается.

## 10. Выводы

Гибридная MPI + OpenMP версия алгоритма демонстрирует:

- Лучшая конфигурация: 2 процесса * 7 потоков (ускорение 7.03x)
- Гибридная схема дает преимущество только при правильном выборе баланса между процессами и потоками
- Для одноузловой системы чистый TBB (ускорение 6.08x при 7 потоках) проще в реализации и близок по производительности
