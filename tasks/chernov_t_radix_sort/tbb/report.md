## 1. Контекст
Версия с oneTBB (Threading Building Blocks) выбрана для демонстрации задачно-ориентированной модели
параллелизма. В отличие от OpenMP с его директивами, TBB предоставляет высокоуровневые примитивы
(`parallel_for`, `parallel_invoke`, `combinable`), которые автоматически балансируют нагрузку и
минимизируют оверхед на синхронизацию. Цель — оценить эффективность runtime-библиотеки TBB
для задачи поразрядной сортировки.

## 2. Постановка задачи
(Идентична последовательной версии — см. [seq/report.md](seq/report.md))
**Вход:** `std::vector<int>` произвольной длины.
**Выход:** Отсортированный по неубыванию вектор.
**Ограничения:** Корректная обработка 32-битных знаковых целых, линейная сложность.

## 3. Базовый алгоритм
Поразрядная сортировка LSD с основанием 256:
1. Преобразование знака через `x ^ 0x80000000`.
2. 4 прохода по байтам: гистограмма → префиксные суммы → стабильное рассеивание.
3. Обратное преобразование знака.
(Подробности — в `seq/report.md`)

В TBB-версии добавлен параллелизм на двух уровнях: `parallel_invoke` для двух половин массива и `parallel_for` + `combinable` для внутренних гистограмм.

## 4. Схема распараллеливания

**Выбранные примитивы oneTBB:**

| Примитив | Где используется | Зачем |
|----------|-----------------|-------|
| `tbb::parallel_invoke` | В `RunImpl()` для параллельной сортировки `left` и `right` | Две независимые подзадачи выполняются параллельно с автоматическим балансированием |
| `tbb::parallel_for` + `blocked_range` | Внутри `RadixSortLSD` для параллельной обработки элементов массива | Регулярный цикл по индексам массива идеально ложится на эту конструкцию |
| `tbb::combinable` | Для локальных гистограмм в каждом проходе поразрядной сортировки | Избегает гонок данных при подсчёте частот без явных `mutex`/`atomic` |

**Ключевой фрагмент (параллельный подсчёт гистограммы):**
```cpp
// File: tbb/src/ops_tbb.cpp
tbb::combinable<std::vector<int>> local_counts([]() { 
    return std::vector<int>(kRadix, 0); 
});

tbb::parallel_for(tbb::blocked_range<size_t>(0, n),
                  [&local_counts, &temp, shift](const tbb::blocked_range<size_t>& r) {
    auto& my_count = local_counts.local();
    for (size_t i = r.begin(); i != r.end(); ++i) {
        int digit = static_cast<int>((temp[i] >> shift) & 0xFFU);
        ++my_count[static_cast<size_t>(digit)];
    }
});

// Объединение локальных гистограмм в глобальную
std::vector<int> count(kRadix, 0);
local_counts.combine_each([&count](const std::vector<int>& c) {
    for (int digit_idx = 0; digit_idx < kRadix; ++digit_idx) {
        count[static_cast<size_t>(digit_idx)] += c[static_cast<size_t>(digit_idx)];
    }
});
```

**Диапазон работы и grainsize:**
- Используется `tbb::blocked_range<size_t>(0, n)` **без явного указания `grainsize`**.
- Это означает, что TBB применяет `auto_partitioner` по умолчанию — адаптивный алгоритм, который динамически делит диапазон на чанки в зависимости от доступности потоков и нагрузки.
- **Почему не указан `grainsize` явно:**
  - Для задачи поразрядной сортировки стоимость обработки одного элемента мала, но фиксированный `grainsize` мог бы привести к дисбалансу или избыточному оверхеду.
  - `auto_partitioner` автоматически находит баланс между параллелизмом и накладными расходами.

**Partitioner:**
- По умолчанию используется `tbb::auto_partitioner` (не указан явно в коде).
- Он адаптируется к количеству доступных воркеров и характеру нагрузки, что удобно для задач с нерегулярным временем выполнения итераций.

**Контроль конкуренции:**
- Число воркеров TBB ограничивается через переменную окружения `TBB_NUM_THREADS` (или `PPC_NUM_THREADS`, которая экспортируется как `TBB_NUM_THREADS` в инфраструктуре курса).
- В коде не используется `tbb::global_control::max_allowed_parallelism`, поэтому TBB по умолчанию использует `std::thread::hardware_concurrency()` воркеров.
- Для воспроизводимости экспериментов в отчёте фиксируется `PPC_NUM_THREADS=2` или `4`.

**Синхронизация и избегание гонок:**
- `tbb::combinable` предоставляет каждому потоку **локальную копию** гистограммы (`std::vector<int>(kRadix, 0)`), поэтому операция `++my_count[digit]` выполняется без синхронизации.
- Метод `combine_each` последовательно (в одном потоке) объединяет все локальные гистограммы в глобальную — это безопасно, так как выполняется после завершения всех параллельных задач.
- Не используются `mutex`, `atomic` или `critical` — вся синхронизация делегирована примитивам TBB.

## 5. Детали реализации
**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

**Изменения относительно SEQ:**
- В `RadixSortLSD` добавлены вызовы `tbb::parallel_for` для параллельной обработки элементов при преобразовании знака и подсчёте гистограмм.
- Использован `tbb::combinable<std::vector<int>>` для локальных гистограмм вместо одного глобального массива.
- В `RunImpl` заменён последовательный вызов сортировки половин на `tbb::parallel_invoke`.

**Работа с памятью:**
- Временные буферы `temp`, `buffer`, `local_counts` выделяются через `std::vector`, что обеспечивает автоматическое управление памятью (RAII).
- `tbb::combinable` внутри использует аллокатор по умолчанию; для продвинутой оптимизации можно было бы использовать `tbb::cache_aligned_allocator`, чтобы снизить эффект ложного разделения кэша (false sharing), но в данной задаче это не критично из-за малого размера гистограммы (256 `int` ≈ 1 КБ).

**Объединение результатов:**
- Локальные гистограммы объединяются через `combine_each`, который последовательно суммирует значения по каждому разряду.
- После вычисления глобальных префиксных сумм выполняется последовательное рассеивание (так как оно требует атомарного обновления позиций, а параллельная версия усложнила бы код без значимого выигрыша).

## 8. Результаты

**Базовое время SEQ:** `0.450 с` (20 млн элементов).

**Зависимость времени от числа воркеров TBB:**

| Воркеры (TBB) | Время (с) | Ускорение | Эффективность |
|---------------|-----------|-----------|---------------|
| 1 | 0.450 | 1.00× | 100% |
| 2 | 0.235 | 1.91× | 95.5% |
| 4 | 0.158 | 2.85× | 71.3% |
| 6 | 0.132 | 3.41× | 56.8% |
| 12 | 0.128 | 3.52× | 29.3% |

**Наблюдения и анализ:**
- Ускорение растёт почти линейно: с 1 до 4 воркеров — почти линейный выигрыш, далее — насыщение.
- **Почему эффективность падает на 12 воркерах:**
  1. **Оверхед на управление задачами:** `auto_partitioner` создаёт множество мелких чанков, увеличивая накладные расходы на планирование.
  2. **Конкуренция за память:** Алгоритм `memory-bandwidth bound` — больше воркеров → больше одновременных обращений к памяти.
  3. **Последовательные участки:** Вычисление префиксных сумм и рассеивание выполняются последовательно (закон Амдала).
  4. **Balancing:** `auto_partitioner` хорошо справляется с балансировкой при 2–6 воркерах, но при 12 оверхед на создание задач начинает доминировать.

## 9. Выводы

**Когда TBB удобнее:**
- При 2–4 воркерах получено ускорение ~1.9–2.85× при минимальных изменениях кода.
- Примитивы `parallel_for` и `combinable` позволили распараллелить задачу без ручного управления потоками и явной синхронизации.
- `auto_partitioner` автоматически находит баланс между параллелизмом и оверхедом.

**Когда overhead мешает:**
1. **Мелкие чанки:** При большом числе воркеров `auto_partitioner` создаёт много мелких задач, увеличивая накладные расходы.
2. **Последовательные участки:** Фаза префиксных сумм и рассеивания не распараллелена, что ограничивает максимальное ускорение.
3. **Память как узкое место:** Алгоритм упирается в пропускную способность памяти.
