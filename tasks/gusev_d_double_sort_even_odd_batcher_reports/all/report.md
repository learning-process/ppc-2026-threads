---

## 1. Введение

ALL-версия объединяет идеи предыдущих реализаций. В ней используются MPI для запуска
в распределённой среде, OpenMP для сортировки локальных блоков, потоки стандартной
библиотеки для обработки раундов слияния и Intel OneTBB для отдельных внутренних
операций сравнения и обмена.

---

## 2. Постановка задачи

Требуется отсортировать массив `double`, сохранив корректность на одном процессе и при
запуске через MPI. Каждый процесс должен обработать свою часть входных данных, после
чего результаты должны быть собраны и объединены в общий отсортированный массив.

---

## 3. Архитектура распределённых вычислений (уровень MPI)

MPI используется как внешний уровень распределения данных:

1. Корневой процесс определяет размер входного массива.
2. Размер передаётся остальным процессам через `MPI_Bcast`.
3. Данные распределяются между процессами через `MPI_Scatterv`.
4. Каждый процесс сортирует локальный фрагмент.
5. Локально отсортированные фрагменты собираются через `MPI_Gatherv`.
6. Корневой процесс выполняет итоговое слияние блоков.
7. Итоговый результат рассылается обратно через `MPI_Bcast`, чтобы проверки были
   корректны на всех MPI-процессах.

---

## 4. Архитектура общей памяти

Внутри каждого процесса используется комбинированная схема:

- OpenMP сортирует блоки локального массива;
- `std::thread` распределяет независимые пары блоков между worker-потоками;
- TBB ускоряет крупные наборы compare-exchange внутри чётно-нечётного слияния.

Такой подход позволяет задействовать все четыре ранее реализованные идеи, но сохранить
единую структуру алгоритма.

---

## 5. Синхронизация

На уровне MPI синхронизация задаётся коллективными операциями `MPI_Bcast`,
`MPI_Scatterv` и `MPI_Gatherv`. На уровне общей памяти блоки и пары блоков не
пересекаются по записи, поэтому основная защита требуется только для передачи исключений
из worker-потоков. Для этого используется `std::exception_ptr` и `std::mutex`.

---

## 6. Детали реализации

Основные файлы:

- `all/include/ops_all.hpp`
- `all/src/ops_all.cpp`

В `RunImpl()` задаётся общее число потоков, создаётся `tbb::global_control`, определяется
MPI-контекст, затем выполняются `ScatterInput`, локальная сортировка `SortLocal` и
финальное `GatherAndMerge`.

Локальная сортировка сохраняет базовую поразрядную сортировку `double` с преобразованием
битового ключа. Слияние использует padding до степени двойки и сеть сравнений Бэтчера.

---

## 7. Проверка корректности

ALL-версия проверяется как в однопроцессном режиме, так и при запуске через `mpiexec`.
Результат сравнивается с эталонной сортировкой стандартной библиотеки. Дополнительно
проверяются разные размеры входа, нечётное число блоков, повторы, экстремальные значения
и знаковые нули.

---

## 8. Экспериментальная среда и результаты

Типовые команды:

```bash
PPC_NUM_THREADS=4 PPC_NUM_PROC=1 ./ppc_func_tests --gtest_filter="*_all_*"
mpiexec -n 2 ./ppc_func_tests --gtest_filter="*_all_*"
PPC_NUM_THREADS=4 PPC_NUM_PROC=1 ./ppc_perf_tests --gtest_filter="*_all_*"
mpiexec -n 2 ./ppc_perf_tests --gtest_filter="*_all_*"
```

### Тесты производительности

Для ALL-версии performance-тесты особенно важны, потому что в одной реализации
сочетаются MPI, OpenMP, STL threads и TBB. Проверка выполняется в двух конфигурациях:
контрольный запуск на одном процессе и распределённый запуск через `mpiexec`.

| Конфигурация | Что оценивается |
| --- | --- |
| `PPC_NUM_PROC=1` | поведение комбинированной реализации без межпроцессного обмена |
| `mpiexec -n 2` | распределение входа, локальная сортировка, сбор и итоговое слияние |
| `pipeline` | полный путь данных через MPI и этапы задачи |
| `task_run` | основной участок сортировки, обмена и слияния |

На крупных массивах `double` проверяется, что время выполнения фиксируется для всей
комбинированной схемы, а результат после сборки совпадает с эталонной сортировкой.
Это позволяет оценивать производительность без потери требований к корректности.

Реальные замеры выполнены в Release-сборке локально на Windows/MSVC. Тестовые данные -
случайный массив из 32 768 элементов в однопроцессной конфигурации ALL. Ускорение
считается относительно запуска с одним worker-потоком.

| workers | time (s) | speedup | efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.008642 | 1.00 | 1.00 |
| 2 | 0.008676 | 1.00 | 0.50 |
| 4 | 0.011281 | 0.77 | 0.19 |
| 8 | 0.011312 | 0.76 | 0.10 |

| Уровень | Назначение |
| --- | --- |
| MPI | распределение входных данных между процессами |
| OpenMP | сортировка локальных блоков |
| STL threads | параллельная обработка пар слияния |
| TBB | крупные compare-exchange этапы |

---

## 9. Выводы

ALL-реализация является наиболее сложной, потому что объединяет несколько моделей
параллелизма. При этом структура остаётся управляемой: MPI отвечает за процессы,
OpenMP и STL - за независимые блоки и пары, а TBB применяется там, где есть много
однотипных операций сравнения.
