## 1. Введение

В OMP-версии последовательный алгоритм адаптирован для общей памяти. OpenMP используется
для параллельной сортировки независимых блоков массива и для обработки пар блоков на
этапах слияния.

---

## 2. Постановка задачи

Необходимо отсортировать массив `double`, распределив работу между несколькими потоками.
Результат должен совпадать с последовательной версией и не зависеть от числа потоков.

Условия корректности:

- каждый поток работает только со своим диапазоном данных;
- размеры блоков корректно вычисляются при некратном делении массива;
- исключения из параллельной области не теряются;
- итоговый массив полностью отсортирован.

---

## 3. Архитектура параллельного решения

Входной массив делится на блоки по числу доступных OpenMP-потоков. Каждый блок копируется
во временный буфер и сортируется локальной поразрядной сортировкой. После локальной
сортировки начинается серия раундов слияния: соседние блоки объединяются попарно,
при нечётном числе блоков последний блок переносится в следующий раунд без изменений.

---

## 4. Схема распараллеливания

Параллельная сортировка блоков выполняется с помощью `#pragma omp parallel for`.
На этапе слияния OpenMP также используется для независимых пар блоков одного раунда.
Так как разные пары записываются в разные элементы массива следующего уровня, гонки
данных отсутствуют.

Слияние внутри пары сохраняет логику Бэтчера: данные разбиваются на чётные и нечётные
группы, группы сливаются, затем выполняется финальная корректировка соседними
сравнениями.

---

## 5. Детали реализации

Основные файлы:

- `omp/include/ops_omp.hpp`
- `omp/src/ops_omp.cpp`

Для определения числа блоков используется `omp_get_max_threads()`. Границы блоков
вычисляются формулами через целочисленное деление, поэтому все элементы покрываются
ровно один раз. Для безопасной обработки ошибок в параллельных областях используется
сохранение первого `std::exception_ptr` с последующим повторным выбрасыванием после
завершения OpenMP-региона.

---

## 6. Проверка корректности

Результат сравнивается с эталонной сортировкой стандартной библиотеки. Тесты проверяют
случаи малых массивов, больших случайных наборов, повторов, экстремальных значений,
знаковых нулей и нечётного числа блоков.

---

## 7. Экспериментальная среда

Проверка выполнялась в Release-сборке. Число потоков задаётся через переменную окружения
`PPC_NUM_THREADS`, которую использует тестовая инфраструктура проекта.

```bash
PPC_NUM_THREADS=4 ./ppc_func_tests --gtest_filter="*_omp_*"
PPC_NUM_THREADS=4 ./ppc_perf_tests --gtest_filter="*_omp_*"
```

### Тесты производительности

Производительность OMP-версии проверяется через `ppc_perf_tests` при фиксированном
числе потоков. Для сравнения с SEQ важны два режима тестовой инфраструктуры:
`pipeline`, где измеряется полный жизненный цикл задачи, и `task_run`, где отдельно
оценивается основной этап сортировки и слияния.

| Параметр | Значение |
| --- | --- |
| Исполняемый файл | `ppc_perf_tests` |
| Управление потоками | `PPC_NUM_THREADS` |
| Основные режимы | `pipeline`, `task_run` |
| Контроль результата | сравнение с `std::ranges::sort` |

В performance-наборе используются крупные массивы `double`, чтобы накладные расходы
OpenMP не перекрывали полезную работу алгоритма. Отдельно проверяется, что ускорение
достигается на сортировке блоков и на независимых парах слияния.

Реальные замеры выполнены в Release-сборке локально на Windows/MSVC. Тестовые данные -
случайный массив из 8 192 элементов. Ускорение считается относительно запуска с одним
worker-потоком.

| workers | time (s) | speedup | efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.384590 | 1.00 | 1.00 |
| 2 | 0.064457 | 5.97 | 2.98 |
| 4 | 0.094789 | 4.06 | 1.01 |
| 8 | 0.118846 | 3.24 | 0.40 |

---

## 8. Результаты

На больших массивах OMP-версия ускоряет этап локальной сортировки блоков и слияние
независимых пар. На небольших данных эффект ограничен накладными расходами на запуск
параллельных областей.

| Компонент | Распараллеливание |
| --- | --- |
| Сортировка блоков | OpenMP `parallel for` |
| Слияние пар блоков | OpenMP `parallel for` |
| Слияние внутри пары | чётно-нечётное слияние Бэтчера |
| Performance-тесты | пройдены |

---

## 9. Выводы

OpenMP хорошо подходит для данной задачи благодаря простой схеме разбиения массива на
независимые блоки. Основная сложность реализации связана не с самим разбиением, а с
аккуратной обработкой исключений и корректным переносом последнего блока при нечётном
числе частей.

---

## Приложение

Типовая последовательность проверки:

```bash
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
./ppc_func_tests --gtest_filter="*_omp_*"
PPC_NUM_THREADS=4 ./ppc_perf_tests --gtest_filter="*_omp_*"
```
