## 1. Введение

В отчёте рассматривается последовательная версия поразрядной сортировки для массива
вещественных чисел типа `double`. Эта реализация используется как базовая: по ней
удобно проверять корректность параллельных вариантов и сравнивать накладные расходы
других технологий.

---

## 2. Постановка задачи

Требуется отсортировать входной вектор `double` по неубыванию. Алгоритм должен корректно
работать с отрицательными и положительными значениями, нулём разных знаков, повторяющимися
элементами и массивами произвольной длины.

Условия корректности:

1. Результат совпадает с результатом стандартной сортировки.
2. Количество элементов после сортировки не меняется.
3. Пустой массив и массив из одного элемента обрабатываются без специальных ошибок.

---

## 3. Описание алгоритма

Основная сортировка выполняется поразрядно. Для `double` нельзя напрямую сортировать
исходные биты как беззнаковое целое число, поэтому сначала используется преобразование
битового представления в сортируемый ключ. Положительные числа получают смещённый знак,
а отрицательные инвертируются, что сохраняет правильный числовой порядок.

После этого выполняются восемь проходов по байтам 64-битного ключа. На каждом проходе
строится массив счётчиков на 256 корзин, вычисляются префиксные суммы и элементы
переносятся во вспомогательный буфер.

Для объединения отсортированных частей применяется чётно-нечётное слияние Бэтчера:
элементы разделяются по чётным и нечётным позициям, соответствующие группы сливаются,
после чего выполняется финальная серия сравнений соседних элементов.

---

## 4. Детали реализации

Основные файлы реализации:

- `seq/include/ops_seq.hpp`
- `seq/src/ops_seq.cpp`

Внутри `RunImpl()` входной массив разбивается на две части. Каждая часть сортируется
поразрядной сортировкой, затем две отсортированные последовательности объединяются
с помощью чётно-нечётного слияния. Предварительная обработка копирует входные данные,
а постобработка записывает результат в выходной буфер задачи.

---

## 5. Проверка корректности

Корректность проверяется сравнением с `std::ranges::sort`. Отдельно покрываются случаи
с пустым массивом, одним элементом, уже отсортированными и обратно отсортированными
данными, повторяющимися значениями, экстремальными числами и знаковыми нулями.

---

## 6. Экспериментальная среда

Для проверки использовалась сборка Release и стандартный набор тестов проекта:

```bash
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build --parallel
```

Функциональные тесты запускаются через `ppc_func_tests`, тесты производительности -
через `ppc_perf_tests`.

### Тесты производительности

Для последовательной реализации performance-тесты нужны как контрольная точка для
остальных технологий. В `ppc_perf_tests` запускаются оба стандартных режима:

| Режим | Что проверяется |
| --- | --- |
| `pipeline` | полный цикл `Validation` + `PreProcessing` + `Run` + `PostProcessing` |
| `task_run` | время основного этапа сортировки после подготовки входных данных |

На вход подаются большие массивы `double` со смешанными положительными и отрицательными
значениями. После каждого измерения результат дополнительно сравнивается с эталоном,
поэтому performance-тесты проверяют не только время выполнения, но и корректность
полученного порядка.

Реальные замеры выполнены в Release-сборке локально на Windows/MSVC. В качестве
табличного результата взят режим `task_run` для случайного массива из 3 000 элементов:

| workers | time (s) | speedup | efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.005650 | 1.00 | 1.00 |

## 7. Результаты

SEQ-версия не имеет накладных расходов на создание потоков и синхронизацию. На малых
массивах она является наиболее стабильной по времени, а на больших массивах служит
контрольной точкой для оценки ускорения OMP, STL, TBB и ALL.

| Проверка | Результат |
| --- | --- |
| Сравнение с `std::ranges::sort` | пройдено |
| Пустой и единичный массив | пройдено |
| Повторяющиеся элементы | пройдено |
| Отрицательные и положительные `double` | пройдено |
| Performance-тесты `pipeline` и `task_run` | пройдены |

---

## 8. Выводы

Последовательная реализация задаёт корректную базу для всей серии работ. Главная
особенность алгоритма состоит в преобразовании `double` в сортируемый ключ, без которого
поразрядная сортировка давала бы неверный порядок для отрицательных значений.
