## 1. Введение

В работе реализованы и исследованы пять вариантов поразрядной сортировки вещественных
чисел типа `double` с чётно-нечётным слиянием Бэтчера. Последовательная версия SEQ
используется как базовая, а параллельные версии применяют OpenMP, потоки стандартной
библиотеки C++, Intel OneTBB и комбинированную схему ALL.

Главная особенность задачи состоит в том, что для корректной поразрядной сортировки
`double` недостаточно напрямую обрабатывать исходные биты числа. Перед сортировкой
битовое представление преобразуется в ключ, который сохраняет числовой порядок для
отрицательных и положительных значений.

---

## 2. Единая постановка задачи

Требуется отсортировать входной массив `double` по неубыванию.

Для всех реализаций должны выполняться одинаковые требования:

1. Результат совпадает с `std::ranges::sort`.
2. Сохраняется количество элементов и их кратность.
3. Корректно обрабатываются пустые массивы, один элемент, повторы, отрицательные числа,
   экстремальные значения и знаковые нули.
4. Размер массива может быть не кратен числу потоков или блоков.
5. Параллельные реализации не должны допускать гонок данных и потери исключений.

---

## 3. Краткое описание реализаций

| Технология | Основная идея |
| --- | --- |
| SEQ | Последовательная поразрядная сортировка и слияние блоков без потоков |
| OMP | Сортировка блоков и обработка независимых пар слияния через OpenMP |
| STL | Ручное распределение работы между `std::thread` |
| TBB | Параллельная обработка блоков и этапов слияния средствами Intel OneTBB |
| ALL | Комбинация MPI, OpenMP, STL threads и TBB в одной реализации |

Во всех версиях сохраняется единая структура алгоритма: входные данные разбиваются на
части, каждая часть сортируется поразрядно, затем отсортированные блоки объединяются
раундами слияния Бэтчера.

## 4. Методология эксперимента

### Аппаратно-программная среда

- CPU: Intel(R) Core(TM) Ultra 9 185H
- RAM: 31.4 GB
- OS: Windows 11
- Компилятор: MSVC 19.44.35220
- CMake: 4.2.0-rc2
- Тип сборки: Release

### Параметры тестирования

- Основной измеряемый участок: `Run` или стандартный режим `task_run`.
- `speedup = time(1 worker) / time(N workers)`.
- `efficiency = speedup / workers`.
- Для SEQ используется один worker, поэтому speedup и efficiency равны 1.00.
- Для ALL в таблице приведена однопроцессная конфигурация с разным числом worker-потоков.

Так как реализации создавались в отдельных ветках и их performance-наборы отличаются по
размеру входа, в сводной таблице указаны реальные размеры тестовых массивов для каждой
технологии.

### Команды сборки и запуска

```bash
cmake -S . -B build -D CMAKE_BUILD_TYPE=Release
cmake --build build --config Release --parallel
./ppc_func_tests --gtest_filter="*gusev_d_double_sort_even_odd_batcher*"
./ppc_perf_tests --gtest_filter="*gusev_d_double_sort_even_odd_batcher*"
```

Для ALL дополнительно используется запуск через MPI:

```bash
mpiexec -n 2 ./ppc_perf_tests --gtest_filter="*_all_*"
```

## 5. Проверка корректности

Функциональные тесты для каждой технологии сравнивают результат работы алгоритма с
эталонной сортировкой стандартной библиотеки. Отдельно проверяются уже отсортированные
и обратно отсортированные массивы, случайные данные, повторы, массивы с разными знаками,
экстремальные значения и случаи, где число элементов не делится ровно на число блоков.

В STL и ALL дополнительно проверяется безопасная передача исключений из рабочих потоков.
В ALL также проверяется корректность работы при запуске через MPI.

## 7. Интерпретация результатов

SEQ-версия является базовой и не содержит накладных расходов на создание потоков,
синхронизацию или планирование задач.

OMP показала наибольшее ускорение на локальном наборе данных. Значение efficiency выше
1.00 для двух потоков связано с малым временем теста и влиянием кэша и измерительной
погрешности, поэтому результат следует рассматривать как признак успешного
распараллеливания, а не как идеальную масштабируемость алгоритма.

STL-версия на двух потоках дала небольшое ускорение, но при дальнейшем увеличении числа
worker-потоков накладные расходы на создание и синхронизацию потоков стали больше
выигрыша от параллельной сортировки блоков.

TBB на данном размере входа показала высокие накладные расходы. Это объясняется тем, что
планировщик задач и внутренние параллельные этапы становятся выгодными только при более
крупных объёмах однотипной работы.

ALL является самой тяжёлой по инфраструктуре реализацией. В ней присутствуют затраты на
MPI-контекст, OpenMP, `std::thread` и TBB, поэтому на локальном однопроцессном запуске
ускорение ниже, чем у более простых реализаций. Такой вариант полезнее для демонстрации
комбинированной модели параллелизма.

---

## 8. Заключение

Все пять реализаций решают одну и ту же задачу и используют общий алгоритмический подход:
преобразование `double` в сортируемый ключ, поразрядная сортировка блоков и последующее
чётно-нечётное слияние Бэтчера.

На локальных тестах лучше всего проявила себя OMP-реализация, а SEQ остаётся стабильной
контрольной версией. STL, TBB и ALL корректно работают, но на выбранных размерах входа
их накладные расходы заметно влияют на итоговое время.
