## 1. Введение

STL-версия использует потоки стандартной библиотеки C++. В отличие от OpenMP и TBB,
здесь распределение работы и ожидание завершения потоков выполняются явно через
`std::thread`.

---

## 2. Постановка задачи

Требуется реализовать многопоточную сортировку массива `double` без стороннего
планировщика задач. Реализация должна корректно завершать все потоки, не допускать
гонок данных и возвращать тот же результат, что и последовательная версия.

---

## 3. Описание алгоритма

Алгоритм состоит из трёх основных этапов:

1. Входной массив делится на блоки.
2. Блоки сортируются поразрядной сортировкой для `double`.
3. Отсортированные блоки объединяются раундами чётно-нечётного слияния Бэтчера.

Для слияния используется временный массив элементов с признаком padding. Это позволяет
довести размер объединяемых частей до степени двойки и применить регулярную сеть
сравнений.

---

## 4. Схема распараллеливания

Функция запуска по индексам распределяет независимые задачи между несколькими
`std::thread`. Один и тот же механизм используется для сортировки блоков и для обработки
пар блоков в раундах слияния.

Число потоков ограничивается не только значением `PPC_NUM_THREADS`, но и реальным
`std::thread::hardware_concurrency()`. Это защищает реализацию от лишнего oversubscription
в CI, где extended-тесты запускаются с большим числом потоков.

---

## 5. Детали реализации

Основные файлы:

- `stl/include/ops_stl.hpp`
- `stl/src/ops_stl.cpp`

Для передачи исключений из рабочих потоков используется общий `std::exception_ptr`,
защищённый `std::mutex`. После `join()` первое сохранённое исключение повторно
выбрасывается в основном потоке. Такой подход позволяет не завершать программу аварийно,
если ошибка возникла внутри worker-потока.

---

## 6. Проверка корректности

Тесты сравнивают результат с `std::ranges::sort`. Дополнительно проверяются повторные
запуски `Run()` до `PostProcessing()`, неизменность снимка входных данных после
`PreProcessing()`, сохранение кратности элементов и разные значения `PPC_NUM_THREADS`.

---

## 7. Экспериментальная среда и результаты

Для локальных проверок использовалась Release-сборка. Отдельно запускались обычные
и extended-наборы потоков:

```bash
scripts/run_tests.py --running-type="threads" --counts 1 2 3 4
scripts/run_tests.py --running-type="threads" --counts 5 7 11 13
```

### Тесты производительности

Для STL-версии обязательно запускается `ppc_perf_tests`, потому что ручное управление
`std::thread` может дать как ускорение на больших массивах, так и заметные накладные
расходы на малых. Измеряются два стандартных сценария: полный `pipeline` и отдельный
`task_run`.

| Параметр | Значение |
| --- | --- |
| Исполняемый файл | `ppc_perf_tests` |
| Управление потоками | `PPC_NUM_THREADS` |
| Проверяемые режимы | `pipeline`, `task_run` |
| Что сравнивается | время выполнения и совпадение с эталонной сортировкой |

Performance-тесты выполняются на больших входных массивах с числами разных знаков.
Такой набор показывает, что ограничение числа worker-потоков не ломает корректность и
не создаёт лишнего oversubscription в расширенных CI-проверках.

Реальные замеры выполнены в Release-сборке локально на Windows/MSVC. Тестовые данные -
случайный массив из 32 768 элементов. Ускорение считается относительно запуска с одним
worker-потоком.

| workers | time (s) | speedup | efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.003873 | 1.00 | 1.00 |
| 2 | 0.003641 | 1.06 | 0.53 |
| 4 | 0.004900 | 0.79 | 0.20 |
| 8 | 0.005738 | 0.67 | 0.08 |

| Проверка | Результат |
| --- | --- |
| Функциональные тесты STL | пройдены |
| Extended-набор потоков | пройден |
| Performance-тесты STL | пройдены |
| clang-format | пройден |
| clang-tidy | пройден |

---

## 8. Выводы

STL-реализация даёт полный контроль над потоками, но требует большего внимания к
жизненному циклу worker-потоков и передаче исключений. Ограничение числа потоков
реальными ресурсами машины делает версию устойчивее в разных CI-окружениях.
