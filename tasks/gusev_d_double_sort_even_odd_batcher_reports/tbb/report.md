---

## 1. Введение

В TBB-версии параллелизм организован средствами Intel OneTBB. Эта технология берёт на
себя планирование задач и балансировку нагрузки между рабочими потоками.

---

## 2. Постановка задачи

Необходимо отсортировать массив `double`, используя TBB для параллельной обработки
независимых участков алгоритма. Итоговый результат должен совпадать с SEQ-версией.

---

## 3. Описание алгоритма

Общая структура совпадает с остальными реализациями: массив делится на блоки, каждый
блок сортируется поразрядно, затем блоки объединяются по раундам. Для корректной
поразрядной сортировки `double` применяется преобразование значения в сортируемый
64-битный ключ.

Чётно-нечётное слияние Бэтчера используется для объединения двух отсортированных
последовательностей. Часть операций внутри слияния также может выполняться параллельно.

---

## 4. Схема распараллеливания

TBB применяется на нескольких уровнях:

- `tbb::parallel_for` сортирует блоки входного массива;
- `tbb::parallel_for` обрабатывает независимые пары блоков при слиянии;
- `tbb::parallel_invoke` используется для независимых ветвей внутри подготовки слияния;
- `tbb::global_control` ограничивает максимальный параллелизм значением из тестовой среды.

Для маленьких диапазонов предусмотрены последовательные ветви, чтобы не тратить время
на создание слишком мелких задач.

---

## 5. Детали реализации

Основные файлы:

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

В реализации используются пороги `kMinParallelElements` и `kMinParallelPairs`. Они
отделяют случаи, где параллельный запуск оправдан, от малых массивов, где быстрее
выполнить операцию обычным циклом.

---

## 6. Проверка корректности

Функциональные тесты проверяют совпадение с `std::ranges::sort`, сохранение количества
элементов, корректную обработку отрицательных чисел, повторов, экстремальных значений
и массивов, размер которых не кратен числу потоков.

---

## 7. Экспериментальная среда и результаты

Проверка выполняется через стандартные тестовые исполняемые файлы проекта:

```bash
PPC_NUM_THREADS=4 ./ppc_func_tests --gtest_filter="*_tbb_*"
PPC_NUM_THREADS=4 ./ppc_perf_tests --gtest_filter="*_tbb_*"
```

### Тесты производительности

Производительность TBB-версии проверяется тем же `ppc_perf_tests`, что и остальные
варианты. Тестовая среда задаёт число рабочих потоков через `PPC_NUM_THREADS`, а внутри
реализации это ограничение передаётся в `tbb::global_control`.

| Режим | Назначение |
| --- | --- |
| `pipeline` | измеряет полный проход задачи вместе с подготовкой и записью результата |
| `task_run` | измеряет основной параллельный участок сортировки и слияния |

В performance-тестах используются достаточно большие массивы, чтобы проявился эффект
планировщика TBB. После каждого запуска выходной массив сверяется с эталонной
сортировкой, поэтому ускорение не рассматривается отдельно от корректности результата.

Реальные замеры выполнены в Release-сборке локально на Windows/MSVC. Тестовые данные -
случайный массив из 32 768 элементов. Ускорение считается относительно запуска с одним
worker-потоком.

| workers | time (s) | speedup | efficiency |
| ---: | ---: | ---: | ---: |
| 1 | 0.004498 | 1.00 | 1.00 |
| 2 | 0.544448 | 0.0083 | 0.0041 |
| 4 | 0.652617 | 0.0069 | 0.0017 |
| 8 | 0.939697 | 0.0048 | 0.0006 |

| Этап | Используемый механизм |
| --- | --- |
| Разбиение и сортировка блоков | `tbb::parallel_for` |
| Независимые ветви слияния | `tbb::parallel_invoke` |
| Ограничение числа потоков | `tbb::global_control` |

---

## 8. Выводы

TBB-версия удобна тем, что не требует ручного управления потоками. Планировщик хорошо
подходит для набора независимых блоков и пар слияния, а пороги для малых задач помогают
уменьшить накладные расходы.
