## 1. Контекст

STL‑версия реализована с использованием нативных потоков `std::thread`. В отличие от
OpenMP и TBB, здесь всё управление потоками (создание, разбиение диапазонов,
синхронизация) выполняется вручную. Такая реализация даёт полный контроль над
распараллеливанием, но требует особой аккуратности: важно не допустить гонок данных
и правильно распределить нагрузку.

## 2. Постановка задачи

Полная постановка (вход, выход, ограничения) приведена в [seq/report.md](seq/report.md).
STL‑версия решает ту же задачу умножения матриц алгоритмом Штрассена.
Baseline для ускорения — время последовательной версии (`0.039966 с` для матрицы 512×512).

## 3. Базовый алгоритм

Базовый алгоритм полностью повторяет последовательную версию (итеративный стек
фреймов, выравнивание нулями, вычисление 7 промежуточных матриц и сборка).
Порог переключения на базовое умножение (`kBaseCaseSize`) также установлен в 128
(как в OMP и TBB). Отличие — все вспомогательные операции (копирование блоков,
сложение, вычитание, базовое умножение) выполняются с помощью вспомогательной
функции `RunParallel`, которая вручную распределяет итерации по потокам.

## 4. Схема распараллеливания

**Как данные делятся на диапазоны.**  
В функции `RunParallel(begin, end, threshold, func)` общее число итераций
`total = end - begin` разбивается на `num_threads` примерно равных непрерывных
диапазонов. Размер чанка вычисляется как `chunk_size = total / num_threads`.
Последний поток получает оставшиеся итерации. Если `total < threshold`,
распараллеливание не применяется (работа выполняется последовательно в текущем
потоке). Порог `kParallelThreshold = 65536` выбран, чтобы избежать накладных
расходов на создание потоков для очень маленьких объёмов.

**Что делает каждый поток.**  
Каждый поток выполняет одну и ту же лямбда-функцию `func` для своего диапазона
индексов. В зависимости от вызова это может быть:

- Копирование строки матрицы при выравнивании.
- Элементарное сложение/вычитание.
- Вычисление строки в `BaseMultiply` (умножение матриц).
- Разбиение или сборка блоков.

**Где хранятся локальные результаты.**  
Поскольку каждый поток пишет в непересекающиеся области памяти (разные индексы
результирующих векторов), локальные буферы не нужны. Например, в `BaseMultiply`
результат `res` является общим вектором, но каждая строка `i` обрабатывается
только одним потоком, поэтому гонок нет. Аналогично в `SplitMatrix` и
`CombineMatrix` — разные потоки работают с разными строками результирующих матриц.

**Как и когда выполняется `join`.**  
Потоки создаются в цикле, сразу после создания каждый поток начинает выполнение.
Затем, после завершения цикла создания, вызывается `join` для каждого потока.
Такой подход гарантирует, что все потоки отработают параллельно. Код:

```cpp
std::vector<std::thread> threads;
for (std::size_t i = 0; i < num_threads; ++i) {
    threads.emplace_back([...] { ... });
}
for (auto &t : threads) t.join();
```

Синхронизация не требуется, так как каждый поток обрабатывает свой уникальный
диапазон индексов и не конфликтует с другими. Никакие `mutex`, `atomic` или
`critical section` не используются — это преимущество независимых по данным операций.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`.

Функция `RunParallel` вычисляет для каждого потока `current_start` и `current_end`:

```cpp
std::size_t chunk_size = total / num_threads;
for (std::size_t i = 0; i < num_threads; ++i) {
    std::size_t current_start = begin + i * chunk_size;
    std::size_t current_end = (i == num_threads - 1) ? end : current_start + chunk_size;
    threads.emplace_back([current_start, current_end, &func]() {
        for (std::size_t j = current_start; j < current_end; ++j) func(j);
    });
}
```

**Схема объединения локальных результатов.**  
Поскольку локальные результаты отсутствуют, объединение не требуется. Каждый поток
напрямую записывает в итоговые векторы (например, `res` в `BaseMultiply` или
`c11`/`c12`/... в сборке).

## 8. Результаты

Для каждого количества потоков выполнено по 3–4 запуска, в таблице указана
**медиана** времени `task_run`. Baseline (`T_seq`) — 0.039966 с
(из `seq/report.md`).

| Потоки | Время (с) (медиана) | Ускорение (S) | Эффективность (E) |
|:-------|--------------------:|--------------:|------------------:|
| 1 (SEQ)| 0.039966            | 1.00x         | 100.0%            |
| 2      | 0.025827            | 1.55x         | 77.5%             |
| 4      | 0.026539            | 1.51x         | 37.7%             |
| 6      | 0.025874            | 1.54x         | 25.7%             |

**Комментарий о накладных расходах `create`/`join` и масштабируемости:**

- Ускорение на 2 потоках составляет лишь 1.55x. Это объясняется накладными расходами на
  создание и запуск потоков (`std::thread` требует системных вызовов), а также
  дополнительными копированиями данных при разбиении диапазонов.
- При переходе на 4 потока ускорение не растёт, а даже немного падает (1.51x).
  Это говорит о том, что алгоритм упирается в пропускную способность памяти,
  а ручное разбиение вносит дополнительную задержку.
- На 6 потоках ускорение остаётся на том же уровне (1.54x), то есть
  дополнительные потоки не дают выигрыша, а только увеличивают накладные
  расходы на создание и синхронизацию.
- Сравнение с OMP и TBB (ускорение 2.1–2.4x на 2–4 потоках) показывает,
  что ручное управление потоками в данной задаче менее эффективно.

## 9. Выводы

Реализация на `std::thread` продемонстрировала, что ручное распараллеливание
регулярных операций с независимыми данными приводит к большим дополнительным затратам.
Ускорение не превысило 1.55x на любом количестве потоков, а увеличение числа потоков свыше 2 не дало прироста.
Это показывает, что для задачи умножения матриц по алгоритму Штрассена автоматические библиотеки
(OpenMP, TBB) значительно удобнее и эффективнее.
