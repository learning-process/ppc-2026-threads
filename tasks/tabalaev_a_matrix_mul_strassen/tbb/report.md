## 1. Контекст

Последовательная версия (SEQ) реализует алгоритм Штрассена итеративно со стеком фреймов.
В TBB-версии используется задачно-ориентированная модель библиотеки oneTBB для распараллеливания
ресурсоёмких операций (сложение, вычитание, разбиение на блоки, базовое умножение, сборка
результата). TBB удобен благодаря автоматическому управлению задачами, эффективному
планировщику и возможности адаптивного разбиения диапазонов, что позволяет получить хорошую
масштабируемость на многоядерных системах без ручного управления потоками.

## 2. Постановка задачи

Полная постановка задачи (входные данные, выходные данные, ограничения, критерии корректности)
приведена в [seq/report.md](seq/report.md). TBB-версия решает ту же задачу умножения плотных
матриц произвольных размеров с использованием алгоритма Штрассена, но с распараллеливанием
вспомогательных операций.

## 3. Базовый алгоритм

Базовый алгоритм полностью повторяет последовательную версию (см. раздел 3 в
[seq/report.md](seq/report.md)): выравнивание матриц нулями до степени двойки, итеративное
разбиение на блоки, вычисление 7 промежуточных матриц $P_1 \dots P_7$, сборка результирующих
блоков $C_{11} \dots C_{22}$ и обрезка до исходных размеров. Порог переключения на базовое
кубическое умножение `kBaseCaseSize` установлен в 128 (как и в OMP‑версии), что уменьшает
глубину рекурсии и количество выделений памяти по сравнению с SEQ (где порог был 32).

## 4. Схема распараллеливания

В TBB‑реализации используется **параллелизм по данным** с помощью алгоритма `tbb::parallel_for`.
Ниже перечислены ключевые аспекты.

- **Выбранный примитив:**  
  `tbb::parallel_for` для независимых итераций (копирование элементов, сложение/вычитание,
  базовое умножение, сборка блоков). `parallel_reduce` не требуется, так как каждый поток
  пишет в непересекающиеся ячейки памяти.

- **Диапазон работы:**  
  В большинстве вызовов используется упрощённая форма `tbb::parallel_for(first, last, body)`,
  где `first` и `last` — целочисленные границы диапазона. TBB автоматически разбивает этот
  диапазон на поддиапазоны (чанки) и распределяет их по потокам. Явный `blocked_range` не
  используется.

- **Grainsize:**  
  Явное значение `grainsize` не задано, используется значение по умолчанию (автоматический
  выбор). При этом в коде присутствует порог `kParallelThreshold = 262144` (размер вектора):
  для операций с меньшим объёмом данных распараллеливание отключается. Это позволяет избежать
  накладных расходов на создание задач для очень мелких блоков.

- **Partitioner:**  
  Не указан явно, используется планировщик по умолчанию.

- **Ограничение конкуренции:**  
  Число рабочих потоков контролируется через переменную окружения `PPC_NUM_THREADS`.

- **Синхронизация:**  
  `tbb::parallel_for` не требует явных барьеров — после завершения цикла все задачи гарантированно
  выполнены. Гонки данных отсутствуют, так как каждый поток работает с уникальным набором индексов.

## 5. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`.

Изменения относительно SEQ:

- Все вспомогательные методы (`Add`, `Subtract`, `SplitMatrix`, `CombineMatrix`, `BaseMultiply`)
  переписаны с использованием `tbb::parallel_for` для крупных блоков (где размер вектора
  `n * n >= kParallelThreshold`). Для мелких блоков используется последовательный код.
- В `PreProcessingImpl` и `RunImpl` распараллелено копирование исходных матриц в выровненные
  буферы и обратное копирование результата.
- В `StrassenMultiply` сохранена итеративная логика со стеком фреймов, но операция сборки
  семи матриц $P_1 \dots P_7$ в блоки `c11`–`c22` выполняется параллельно.

**Работа с памятью и локальные аккумуляторы:**

- Локальные аккумуляторы не используются, так как все операции являются поэлементными и не
  требуют редукции. Результаты записываются напрямую в выходные векторы.

Пример фрагмента кода (базовое умножение с `parallel_for` по строкам):

```cpp
// File: tbb/src/ops_tbb.cpp
std::vector<double> TabalaevAMatrixMulStrassenTBB::BaseMultiply(
    const std::vector<double> &mat_a, const std::vector<double> &mat_b, size_t n) {
  std::vector<double> res(n * n, 0.0);
  tbb::parallel_for(static_cast<size_t>(0), n, [&](size_t i) {
    for (size_t k = 0; k < n; ++k) {
      double temp = mat_a[(i * n) + k];
      if (temp == 0.0) continue;
      for (size_t j = 0; j < n; ++j) {
        res[(i * n) + j] += temp * mat_b[(k * n) + j];
      }
    }
  });
  return res;
}
```

## 8. Результаты

Замеры выполнены для матрицы 512×512 в режиме `task_run`. В качестве эталонного времени
`T_seq` используется значение последовательной версии из [seq/report.md](seq/report.md) – **0.039966 с**.

| Потоки (workers) | Время (с) (медиана) | Ускорение (S) | Эффективность (E) |
|:-----------------|--------------------:|--------------:|------------------:|
| 1 (SEQ)          | 0.039966            | 1.00x         | 100.0%            |
| 2                | 0.018895            | 2.12x         | 106.0%            |
| 4                | 0.016463            | 2.43x         | 60.7%             |
| 6                | 0.018200            | 2.20x         | 36.7%             |

**Комментарий о масштабируемости.**  
На двух потоках наблюдается лучшее ускорение (2.12x), что объясняется улучшенной
локальностью кэша и меньшей глубиной рекурсии (порог `kBaseCaseSize = 128` против 32 в SEQ).
Четыре потока дают максимальное ускорение 2.43x при эффективности 61%. При шести потоках время возрастает (0.0182 с),
ускорение падает до 2.20x, что свидетельствует о растущих накладных расходах планировщика TBB.

## 9. Выводы

TBB-версия показала ускорение до 2.43× на 4 потоках, что сопоставимо с OpenMP. Задачно-ориентированный подход упрощает код,
но накладные расходы планировщика заметны на мелких блоках (для них введён порог kParallelThreshold). Дальнейшее ускорение
на 6 потоках ограничено памятью и накладными расходами планировщика.
