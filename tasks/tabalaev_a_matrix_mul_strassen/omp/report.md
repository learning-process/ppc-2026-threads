## 1. Контекст

Последовательная версия (SEQ) реализует алгоритм Штрассена итеративно со стеком фреймов,
обеспечивая корректный baseline. В данной реализации **OpenMP** используется для
параллельного выполнения ресурсоёмких поэлементных операций над матрицами (сложение,
вычитание, копирование блоков, сборка результирующих блоков). Это позволяет ускорить
вычислительно тяжёлую часть алгоритма на многоядерных процессорах без изменения общей
логики декомпозиции.

## 2. Постановка задачи

Полная постановка задачи (входные данные, выходные данные, ограничения, критерии
корректности) приведена в [seq/report.md](seq/report.md). OMP‑версия решает ту же задачу
умножения плотных матриц произвольных размеров с использованием алгоритма Штрассена, но с
распараллеливанием вспомогательных операций.

## 3. Базовый алгоритм

Базовый алгоритм полностью повторяет последовательную версию (см. раздел 3 в
[seq/report.md](seq/report.md)): выравнивание матриц нулями до степени двойки,
итеративное разбиение на блоки, вычисление 7 промежуточных матриц $P_1 \dots P_7$,
сборка результирующих блоков $C_{11} \dots C_{22}$ и обрезка до исходных размеров.
Единственное отличие – порог переключения на базовое кубическое умножение
`kBaseCaseSize` увеличен с 32 (в SEQ) до 128, чтобы уменьшить накладные расходы на
порождение потоков для мелких подзадач.

## 4. Схема распараллеливания

В отличие от распараллеливания дерева подзадач (каждая из 7 рекурсий), здесь применяется
**параллелизм по данным** на уровне поэлементных операций с матрицами.

- **Область кода:**  
  Распараллелены следующие независимые циклы:
  - Копирование блоков при разбиении (`PushStrassenSubtasks`).
  - Покоординатное сложение/вычитание матриц (`Add`, `Subtract`).
  - Объединение результатов в блоки `c11`, `c12`, `c21`, `c22` (стадия сборки).
  - Базовое кубическое умножение `BaseMultiply` (при $n \le 128$).

- **Переменные (`shared` / `private`):**  
  Во всех директивах `#pragma omp parallel for` используется `default(none)`, затем явно
  перечисляются `shared` (общие массивы, размеры) и `private` (неявно – индексы циклов).
  Например, в сборке блоков:
  
  ```cpp
  #pragma omp parallel for default(none) shared(p, c11, c12, c21, c22, sz)
  ```

  Переменные `i` и `j` являются локальными для каждого потока.

- **Reduction / atomic / critical:**  
  Не требуются. Каждый поток записывает результат в уникальную ячейку целевой матрицы
  (индексы `i` и `j` полностью определяют положение), поэтому гонки данных отсутствуют.

- **Выбор `schedule`:**  
  Используется неявный `schedule(static)`. Операции над матрицами имеют
  регулярную структуру (все итерации одинаково трудоёмки), статическое распределение
  даёт минимальные накладные расходы и хорошую локальность данных.

- **Объединение циклов (`collapse`):**  
  Для двумерных циклов копирования блоков (например, заполнение `a11`, `b12` и т.д.)
  применена директива `collapse(2)`. Это объединяет два вложенных цикла в одно
  итерационное пространство, позволяя планировщику OpenMP более равномерно распределить
  работу.

- **Барьеры:**  
  После каждого `parallel for` действует **неявный барьер** – все потоки синхронизируются,
  гарантируя, что все элементы матрицы вычислены до перехода к следующему шагу алгоритма.
  Явные барьеры (`#pragma omp barrier`) не используются.

## 5. Детали реализации

**Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`.

Изменения относительно SEQ:

- Все вспомогательные методы (`Add`, `Subtract`, `PushStrassenSubtasks`,
  `CombineStrassenResults`, `BaseMultiply`) переписаны с использованием OpenMP-директив.
- В `CombineStrassenResults` (стадия сборки) четыре формулы для `c11`, `c12`, `c21`, `c22`
  объединены в один параллельный цикл, что улучшает локальность данных и уменьшает
  количество барьеров.
- Порог `kBaseCaseSize` увеличен с 32 до 128, чтобы снизить накладные расходы на
  создание потоков для мелких задач.

**Риски гонок и их устранение:**

- Потенциальная гонка могла бы возникнуть при одновременной записи в одну ячейку
  результирующей матрицы из разных потоков, но в реализации каждый поток
  обрабатывает непересекающийся набор индексов. Все операции – это независимые присваивания или
  чтения/записи в разные ячейки.
- Для операций сложения/вычитания матриц каждый поток работает со своей парой
  `(i, j)`, поэтому `reduction` не нужен.

Пример распараллеленной сборки блоков (фрагмент из `omp/src/ops_omp.cpp`):

```cpp
// File: omp/src/ops_omp.cpp (CombineStrassenResults, stage == 8)
#pragma omp parallel for default(none) shared(p, c11, c12, c21, c22, sz)
for (size_t i = 0; i < sz; ++i) {
    c11[i] = p[0][i] + p[3][i] - p[4][i] + p[6][i];
    c12[i] = p[2][i] + p[4][i];
    c21[i] = p[1][i] + p[3][i];
    c22[i] = p[0][i] - p[1][i] + p[2][i] + p[5][i];
}
```

## 8. Результаты

Замеры выполнены для матрицы 512×512. В качестве baseline `T_seq` взято время
последовательной версии из [seq/report.md](seq/report.md) – 0.039966 с (режим `task_run`).

| Потоки (T) | Время (с) | Ускорение (S) | Эффективность (E) |
|:-----------|----------:|--------------:|------------------:|
| 1 (SEQ)    | 0.039966  | 1.00x         | 100.0%            |
| 2          | 0.018662  | 2.14x         | 107.0%            |
| 4          | 0.015324  | 2.61x         | 65.2%             |
| 6          | 0.015139  | 2.64x         | 44.0%             |

**Комментарий:**

- **Лучшее ускорение** на 2 потоках (S > 2) объясняется двумя факторами:
  1) Увеличенный порог `kBaseCaseSize = 128` по сравнению с 32 в SEQ – для матрицы
     512×512 это приводит к меньшему числу рекурсивных вызовов и
     меньшим накладным расходам на выделение памяти.
  2) При статическом разбиении циклов каждый поток работает со своим непрерывным диапазоном памяти,
     что снижает количество кэш-промахов.
- При переходе с 2 на 4 потока ускорение растёт нелинейно (2.14 → 2.61), эффективность
  падает до 65%.
- Дальнейшее увеличение до 6 потоков даёт прирост всего 0.03 (2.61 → 2.64),
  эффективность падает до 44%.

## 9. Выводы

OpenMP обеспечил хороший выигрыш в производительности для поэлементных операций
алгоритма Штрассена. Наиболее эффективное использование ресурсов достигается при
2–4 потоках (ускорение 2.1–2.6x). Дальнейшее увеличение числа потоков нецелесообразно
из-за возрастающих накладных расходов.

**Когда OpenMP дал выигрыш:**  

- Для всех размеров матриц, где объём вычислений достаточен, чтобы окупить затраты на
  порождение потоков (начиная с 256×256).
- Особенно эффективен на регулярных операциях (сложение, вычитание, копирование),
  где статическое распределение и `collapse(2)` дают почти линейное ускорение.

**Чем ограничен выигрыш:**  

- Узким местом является выделение/освобождение памяти для
  промежуточных блоков.
