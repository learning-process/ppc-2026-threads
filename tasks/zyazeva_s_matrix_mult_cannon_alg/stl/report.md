## 1. Контекст

STL-версия реализует алгоритм Кэннона с использованием исключительно
стандартной библиотеки C++ (`std::thread`) без сторонних runtime-библиотек
параллелизма. Весь механизм распределения работы между потоками реализован
вручную через вспомогательную функцию `ParallelFor`, которая создаёт пул
потоков, делит диапазон задач на равные блоки и ожидает завершения каждого
потока через `join`.

## 2. Постановка задачи

**Входные данные:**

- `sz` — сторона квадратной матрицы (тип `size_t`);
- `m1` — матрица A, построчно, размер `sz * sz`;
- `m2` — матрица B, аналогично.

**Выходные данные:** результирующая матрица C = A·B, построчно,
размер `sz * sz`.

**Ограничения:**

- `sz > 0` — нулевой размер недопустим;
- `m1.size() == sz * sz` и `m2.size() == sz * sz` — проверяется в
  `ValidationImpl`;
- элементы — `double`, переполнение не рассматривается.

**Крайние случаи:**

- матрица 1×1 — корректно обрабатывается основным циклом;
- пустые векторы при `sz > 0` — отклоняются валидацией.

Число потоков определяется через `std::thread::hardware_concurrency()`
в момент создания пула.

## 3. Базовый алгоритм

Структура алгоритма Кэннона идентична OMP-версии: разбиение на `g x g`
блоков (где `g = sqrt(sz)`), начальное выравнивание, `g` шагов умножения
со сдвигом, сборка результата. Условие применимости:
`g > 1 && g*g == sz && sz % g == 0`. При несоблюдении —
fallback к `RegularMultiplication`.

## 4. Схема распараллеливания

`ParallelFor`:

```cpp
// File: stl/src/ops_stl.cpp — ParallelFor
void ParallelFor(size_t count,
                 const std::function<void(size_t)>& func) {
  size_t threads_count = std::thread::hardware_concurrency();
  if (threads_count == 0) threads_count = 4;
  threads_count = std::min<size_t>(threads_count, count);

  std::vector<std::thread> threads(threads_count);
  size_t block_size = count / threads_count;
  size_t remainder  = count % threads_count;
  size_t begin = 0;

  for (size_t th = 0; th < threads_count; ++th) {
    size_t end = begin + block_size + (th < remainder ? 1 : 0);
    threads[th] = std::thread([begin, end, &func]() {
      for (size_t i = begin; i < end; ++i) func(i);
    });
    begin = end;
  }

  for (auto& th : threads) th.join();
}
```

**Карта поток → блок данных:**

- Диапазон `[0, count]` делится на `threads_count` частей.
- Первые `remainder` потоков получают `block_size + 1` итераций,
  остальные — `block_size`. Распределение без потери итераций.
- Каждый поток работает со своим непересекающимся поддиапазоном;
  захват `[begin, end, &func]` передаёт счётчики по значению, исключая
  race condition на переменных счётчика.
- `join` вызывается после запуска всех потоков.

**Синхронизация:**

- Барьер реализован через цикл `join` в конце `ParallelFor`.
- Общих накопителей нет: `func(i)` для разных `i` пишет в
  непересекающиеся области памяти. `atomic` и `mutex` не требуются.
- Между шагами Кэннона `ParallelFor` завершается (барьер через `join`)
  до начала следующего шага.

**Параллелизуемые области:**

- `InitializeBlocks` — разрезка матриц на блоки;
- `AlignBlocks` — начальное выравнивание;
- `CannonStep` — умножение блоков на каждом шаге;
- `Shift` — циклический сдвиг блоков;
- `Assemble` — сборка результата;
- `RegularMultiplication` (fallback) — параллельный проход по строкам.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

Весь вспомогательный код вынесен в анонимное пространство имён.
`MulBlock` реализует перемножение одного блока в формате ijk, что
обеспечивает хорошую локальность кэша при доступе к блоку B.

```cpp
// File: stl/src/ops_stl.cpp — Shift
void Shift(std::vector<std::vector<double>>& a,
           std::vector<std::vector<double>>& b,
           std::vector<std::vector<double>>& na,
           std::vector<std::vector<double>>& nb,
           size_t g) {
  ParallelFor(g * g, [&](size_t id) {
    size_t i = id / g;
    size_t j = id % g;
    na[id] = a[(i * g) + ((j + 1) % g)];
    nb[id] = b[(((i + 1) % g) * g) + j];
  });
}
```

Записи идут в `na[id]` и `nb[id]` — разные блоки для разных `id`,
гонок нет. Чтения из `a` и `b` — только чтение, безопасны.
После `Shift` (= после `join`) выполняется `aa = std::move(na)` —
однопоточная операция, конкуренции нет.

Потенциальная неэффективность: `std::function<void(size_t)>` добавляет
overhead на динамическую диспетчеризацию по сравнению с шаблонным
лямбда-подходом TBB или compile-time OpenMP директивами.

## 6. Проверка корректности

STL-версия сравнивалась с SEQ для всех функциональных тестов, включая
размеры, при которых fallback активируется (N не является полным
квадратом). Корректность `join`-барьера проверена: ни одна запись в
вектор результата не происходит до завершения всех потоков текущего
шага. Расхождений не наблюдалось. Оба режима производительности
запускались и уложились в лимиты.

## 9. Выводы

`std::thread` позволяет реализовать параллельный алгоритм Кэннона без
зависимостей от сторонних библиотек. Ручное разбиение диапазонов в
`ParallelFor` прозрачно и воспроизводимо, однако создание и уничтожение
потоков при каждом вызове `ParallelFor` является потенциальным источником
накладных расходов при большом числе шагов. Использование `std::function`
добавляет overhead на динамическую диспетчеризацию. Масштабируемость
ограничена теми же условиями, что и у OMP: N должен быть полным
квадратом.
