## 1. Контекст

TBB-версия реализует алгоритм Кэннона с использованием
задачно-ориентированного подхода oneTBB. Вместо явного управления потоками
каждая операция над блоком формулируется как задача, а диспетчеризацию берёт
на себя runtime oneTBB. Дополнительно применяется
`tbb::cache_aligned_allocator` для блочных данных, что снижает ложное
разделение кэша при параллельном доступе к соседним блокам.

## 2. Постановка задачи

**Входные данные**
(`InType = std::tuple<size_t, std::vector<double>, std::vector<double>>`):

- `sz` — сторона квадратной матрицы (тип `size_t`);
- `m1` — матрица A, хранимая построчно в одномерном
  `std::vector<double>` размером `sz * sz`;
- `m2` — матрица B, аналогично.

**Выходные данные** (`OutType = std::vector<double>`):
результирующая матрица `C = A·B`, построчно, размер `sz * sz`.

**Ограничения:**

- `sz > 0` — нулевой размер считается недопустимым;
- `m1.size() == sz * sz` и `m2.size() == sz * sz` —
  несоответствие размеров блокируется в `ValidationImpl`;
- элементы — `double`, переполнение не рассматривается.

**Крайние случаи:**

- матрица `1×1` — корректно обрабатывается основным циклом;
- передача пустых векторов (`sz > 0`, но `size == 0`) —
  отклоняется валидацией.

Размер сетки блоков (`grid_size`) определяется динамически через
`tbb::this_task_arena::max_concurrency()`, а не фиксируется на этапе
компиляции.

## 3. Базовый алгоритм

Структура алгоритма Кэннона такая же, как в OMP-версии:
начальное выравнивание блоков, `g` шагов умножения с последующим циклическим
сдвигом карт индексов (`map_a`, `map_b`), сборка результата.

Ключевое отличие — смещение блоков реализовано через индексные карты
(`map_a[id]`, `map_b[id]`), а не через физическое копирование блоков на
каждом шаге, что уменьшает объём выделяемой памяти.

## 4. Схема распараллеливания

`tbb::parallel_for` по диапазону от `0` до `total_blocks`
(размер `size_t`).

```cpp
// File: tbb/src/ops_tbb.cpp — PerformCannonStep
tbb::parallel_for(static_cast<size_t>(0), total_blocks,
    [&](size_t id) {
        PerformCannonStepForBlock(
            id, a, b, c, map_a, map_b, block_area, bs);
    });
````

Здесь `id` — глобальный индекс блока в сетке.
Функция `PerformCannonStepForBlock` читает блоки из `a` и `b`
по индексам `map_a[id]` и `map_b[id]`
и записывает накопленный результат в `c[id]`.

Поскольку каждый `id` уникален и разные блоки результата
не пересекаются, синхронизации не требуется.

**Диапазон и grainsize:**

Диапазон `[0, total_blocks)` передаётся напрямую,
без явного `blocked_range`.
oneTBB самостоятельно выбирает разбиение на чанки;
`auto_partitioner` (по умолчанию) адаптивно подбирает размер chunk-а
в зависимости от числа задач в очереди,
что подходит для однородной нагрузки.

**Контроль конкуренции:**

Максимальное число worker-потоков определяется через
`tbb::this_task_arena::max_concurrency()` и используется
для вычисления `grid_size` (`FindGridSize`).

Явного `global_control` не создаётся;
runner курса управляет числом потоков через переменную окружения
`PPC_NUM_THREADS`.

```cpp
// File: tbb/src/ops_tbb.cpp — FindGridSize
size_t FindGridSize(int sz) {
  const auto max_threads = tbb::this_task_arena::max_concurrency();
  const int root = static_cast<int>(std::sqrt(max_threads));

  for (int k = root; k >= 1; --k) {
    if (sz % k == 0) {
      return static_cast<size_t>(k);
    }
  }

  return 1;
}
```

Функция ищет наибольший делитель `sz`,
не превышающий `√max_threads`.

Это гарантирует, что блоки делят матрицу ровно,
даже если `max_threads` не является полным квадратом.

**Работа с памятью:**

```cpp
using AlignedVector =
    std::vector<double, tbb::cache_aligned_allocator<double>>;
```

Блочные данные (`a`, `b`, `c`) хранятся в векторах
с `cache_aligned_allocator`,
что выравнивает начало каждого вектора по границе строки кэша
и снижает ложное разделение при параллельной записи
в соседние блоки.

**Индексные карты** (`map_a`, `map_b`) хранят исходные индексы блоков
для каждого шага и обновляются последовательно (`UpdateMaps`)
между шагами, поскольку обновление зависит
от текущего состояния карт.

## 5. Детали реализации

**Файлы:**

- `tbb/include/ops_tbb.hpp`
- `tbb/src/ops_tbb.cpp`

Весь вспомогательный код
(инициализация блоков, Cannon-шаг, сборка результата)
вынесен в анонимное пространство имён,
что предотвращает коллизии имён с другими реализациями.

`MultiplyBlockRow` содержит проверку:

```cpp
if (a_val == 0.0) {
    continue;
}
```

Это раннее исключение нулевых элементов,
которое ускоряет разреженные матрицы.

`InitializeBlocks` и `AssembleResult`
также реализованы через `tbb::parallel_for`
по всем блокам:

```cpp
// File: tbb/src/ops_tbb.cpp — InitializeBlocks
tbb::parallel_for(static_cast<size_t>(0), total_blocks,
    [&](size_t id) {
        InitializeBlock(
            id, m1, m2, a, b,
            gs, bs, sz, block_area);
    });
```

Каждый `id` записывает в уникальный диапазон:

```text
[id * block_area, (id + 1) * block_area]
```

Гонок данных не возникает.

## 9. Выводы

TBB позволил реализовать алгоритм Кэннона
с минимальным явным управлением параллелизмом:
все блочные операции сводятся к `tbb::parallel_for`
по линейному диапазону блоков.

Использование `cache_aligned_allocator`
уменьшает конкуренцию за строки кэша.

Механизм индексных карт вместо физического копирования блоков
сокращает объём аллокаций на каждом шаге.

Ограничения масштабируемости аналогичны OMP:
размер сетки ограничен `√max_threads`,
а обновление индексных карт между шагами
выполняется последовательно.
