## 1. Введение

Задача — произведение двух квадратных вещественных матриц A и B
размера N×N. Алгоритм Кэннона является примером блочного параллельного
умножения матриц: он разбивает матрицы на блоки, назначает каждый блок
отдельной рабочей единице и на каждом шаге выполняет локальное умножение
с циклическим сдвигом блоков.

Пять реализаций — SEQ, OMP, TBB, STL, ALL — применяют одну и ту же
вычислительную схему с разными механизмами распараллеливания, что
позволяет честно сравнивать именно накладные расходы и выигрыш каждой
технологии, а не разные алгоритмы.

## 2. Единая постановка задачи

**Входные данные:**

- `sz` — сторона квадратной матрицы (`size_t`);
- `m1` — матрица A, построчно, размер `sz x sz`;
- `m2` — матрица B, построчно, размер `sz x sz`.

**Выходные данные** (`OutType = std::vector<double>`):
матрица C = A·B, построчно, размер `sz x sz`.

**Ограничения:** `sz > 0`; `m1.size() == sz * sz`;
`m2.size() == sz * sz`. Проверяются в `ValidationImpl` каждого backend-а.

**Критерий корректности:** результат совпадает с результатом SEQ
на тех же входных данных (поэлементное сравнение с допуском `double`).

**Ограничения применимости алгоритма Кэннона:**

| Backend | Условие применения Кэннона | Fallback |
| --- | --- | --- |
| SEQ | — | тройной цикл (всегда) |
| OMP | `sqrt(num_threads)` — целое, `sz % sqrt(num_threads) == 0` | RegularMultiplication (OMP) |
| TBB | `sz % FindGridSize(sz) == 0` | сетка 1×1 |
| STL | `g = sqrt(sz)` — целое, `g > 1`, `sz % g == 0` | RegularMultiplication (STL) |
| ALL | `sqrt(mpi_size_)` — целое, `sz % sqrt(mpi_size_) == 0` | RegularMultiplication (OMP) |

## 3. Единая методика эксперимента

| Параметр | Значение |
| --- | --- |
| CPU | AMD Ryzen 7 8845HS w/ Radeon 780M Graphics |
| RAM | 32 ГБ |
| OS | Windows 11 Pro |
| Compiler | g++ / clang++ (Release) |
| CMake build type | Release |

**Переменные окружения:** `PPC_NUM_THREADS` (= `OMP_NUM_THREADS` для
OMP- и ALL-версий), `PPC_NUM_PROC` (число MPI-рангов для ALL).
Автоматически 16 потоков.

**Генерация данных:** входные матрицы генерируются тестовой
инфраструктурой курса (`MakeAllPerfTasks`); конкретный размер N
фиксируется runner-ом.

**Формула ускорения:**

```text
Speedup = T_seq / T_backend
```

где `T_seq` — время SEQ-версии в соответствующем режиме.

**Формула эффективности:**

```text
Efficiency = Speedup / workers
```

где `workers` — число рабочих единиц: для OMP/TBB/STL — число потоков;
для ALL — `ranks x threads_per_rank`.

**Агрегирование:** единичные значения из одного запуска тестовой
инфраструктуры курса. Для повышения воспроизводимости рекомендуется
3–5 повторов с медианой.

**Режимы тестирования:** `pipeline` (полный пайплайн
`Validation → PreProcessing → Run → PostProcessing`) и `task_run`
(только фаза `Run`). Результаты двух режимов не смешиваются.

## 4. Сводка корректности

Все пять backend-ов сравнивались с SEQ на едином наборе функциональных
тестов из `tests/functional/main.cpp`. Тесты включают:

- матрицы 1×1 и 2×2 (ручная проверка);
- единичные и нулевые матрицы (инвариант I·A = A, 0·A = 0);
- матрицы произвольного размера (сравнение с SEQ-результатом).

Расхождений не наблюдалось ни для одного backend-а. Ветки fallback
тестировались отдельно при соответствующих размерах и числах рабочих
единиц. Оба режима производительности уложились в установленные лимиты.

## 6. Интерпретация различий

**SEQ** реализует классический тройной цикл O(N^3) без оптимизаций.
Доступ к столбцу матрицы B является неудобным для кэша, поэтому
SEQ-время систематически хуже, чем при блочной реорганизации данных.
Параллельные версии выигрывают не только от многопоточности, но и
частично от блочной структуры алгоритма Кэннона.

**OMP** показывает ускорение 4.09x / 2.62x при 16 потоках.
Распараллеливание через `parallel for collapse(2)` покрывает
инициализацию блоков, выравнивание, Cannon-шаги и сборку результата.
Ограничение: между шагами Кэннона аллоцируются новые векторы блоков
(`new_aligned_a`, `new_aligned_b`), что снижает ускорение в `task_run`.
Разница между режимами (4.09x vs 2.62x) объясняется различиями в
условиях кэша при повторных запусках.

**TBB** показывает результаты, близкие к OMP (3.78x / 2.85x).
Ключевое отличие — использование индексных карт (`map_a`, `map_b`)
вместо физического копирования блоков, что снижает объём аллокаций
на каждом шаге. `cache_aligned_allocator` уменьшает ложное разделение
кэша. В режиме `task_run` TBB незначительно опережает OMP (0.197 с
против 0.214 с), что согласуется с меньшим объёмом аллоцируемой памяти.

**STL** показывает наилучший результат среди однопроцессных реализаций
в режиме `task_run` (3.38x, 0.166 с). `ParallelFor` создаёт потоки на
каждую параллельную операцию и синхронизирует их через `join`, что при
равномерной нагрузке оказывается эффективным. В режиме `pipeline`
результаты STL и OMP практически совпадают (0.173 с против 0.174 с).

**ALL (MPI + OMP)** даёт ускорение 9.89x / 8.33x. MPI распределяет
блоки по процессам, OMP ускоряет вспомогательные операции внутри ранга.
Коммуникационные издержки на `MPI_Bcast` и `MPI_Sendrecv_replace`
перекрываются вычислительным выигрышем. Последовательная рассылка блоков
через `MPI_Send` на ранге 0 — архитектурное ограничение при большом
числе процессов.

Близость результатов OMP, TBB и STL в режиме `pipeline` (0.173–0.189 с)
объясняется тем, что все три реализации упираются в одно и то же
ограничение — размер сетки блоков `g x g` и объём памяти для хранения
промежуточных блоков.

## 8. Заключение

Лучшим по абсолютной производительности является гибридная версия ALL
(MPI + OMP) с ускорением ~9.9x / ~8.3x. Среди однопроцессных реализаций
STL показала лучший результат в режиме `task_run` (3.38x), OMP и STL —
сопоставимые результаты в режиме `pipeline` (4.09x / 4.12x).

При наличии нескольких MPI-процессов ALL является очевидным выбором;
при однопроцессном запуске STL или OMP практически равнозначны, а TBB
уступает им незначительно.
