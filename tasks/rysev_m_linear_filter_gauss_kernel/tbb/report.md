## 1. Контекст

Параллельная реализация линейной фильтрации изображений с использованием Intel oneTBB. В отличие от OpenMP с жёсткой моделью fork-join, TBB предлагает задачно-ориентированный подход с рекурсивным разбиением диапазонов и перехватом работы (work-stealing), что уменьшает накладные расходы на синхронизацию и улучшает балансировку нагрузки при обработке строк изображения.

## 2. Постановка задачи

Решение задачи фильтрации изображения ядром Гаусса 3×3 (RGB) с обработкой краёв путём пропуска выходящих за границу пикселей. SEQ-версия служит эталоном корректности и производительности.

**Входные данные:** целое число `GetInput()` – размер квадратного изображения (или 0 для загрузки реального `pic.ppm`), генерируются случайные пиксели RGB.

**Выходные данные:** отфильтрованное изображение и сумма пикселей для верификации.

## 3. Базовый алгоритм

Последовательная версия: для каждого канала (0..2) и для каждой строки и столбца вычисляется свёртка с ядром 3×3. Вычислительная сложность – O(H×W×9) на канал. Основная нагрузка – внутренний цикл по 9 элементам ядра. Параллелизм вводится на уровне обработки строк внутри каждого канала с помощью `tbb::parallel_for`.

## 4. Схема распараллеливания

**Примитив TBB:** `tbb::parallel_for` — для параллельной обработки строк внутри одного канала.

**Диапазон работы:** строки от 0 до `rows-1`. Разбиение выполняется автоматически планировщиком TBB с использованием `auto_partitioner` (по умолчанию).

**Размер зерна (grainsize):** не задан явно; TBB использует эвристику, основанную на числе итераций и количестве потоков. Для типичных размеров изображения (≥1024×1024) это даёт хорошую балансировку.

**Partitioner:** `auto_partitioner`. Планировщик TBB рекурсивно разбивает диапазон строк на поддиапазоны и распределяет их между потоками. При дисбалансе (например, из-за различий во времени обработки строк на краях изображения) применяется механизм work-stealing – поток, завершивший свою работу, «ворует» задачи из очереди другого потока.

**Контроль параллелизма:** число рабочих потоков управляется через `tbb::global_control::max_allowed_parallelism`, устанавливаемое инфраструктурой курса через переменную окружения `PPC_NUM_THREADS`.

**Ключевой фрагмент (код из `ops_tbb.cpp`):**
```cpp
void RysevMGaussFilterTBB::ApplyKernelToChannel(int channel, int rows, int cols) {
  const auto &input = input_image_;
  auto &output = output_image_;
  int channels = channels_;

  tbb::parallel_for(0, rows, [&](int row) {
    for (int col = 0; col < cols; ++col) {
      float sum = ComputePixelValue(row, col, channel, rows, cols, channels, input);
      std::size_t out_idx = (static_cast<std::size_t>(row) * cols + col) * channels + channel;
      output[out_idx] = static_cast<uint8_t>(std::clamp(sum, 0.0F, 255.0F));
    }
  });
}
```
Функция ApplyKernelToChannel вызывается последовательно для каждого канала (0,1,2). Внутри неё tbb::parallel_for разбивает строки на задачи. Каждая задача обрабатывает одну строку (из-за отсутствия указания grainsize, TBB может объединять строки в блоки, если считает это эффективным). Такой подход даёт хорошую балансировку, особенно когда строки содержат разное количество граничных пикселей.

**Синхронизация:**
    Неявная синхронизация после tbb::parallel_for гарантирует завершение всех задач перед возвратом.
    Запись в output – разные строки, коллизий нет.
     Отсутствие явных барьеров снижает накладные расходы по сравнению с OpenMP.

## 5. Детали реализации

**Файлы:** `tbb/include/ops_tbb.hpp`, `tbb/src/ops_tbb.cpp`

**Функции:**
- `ApplyKernelToChannel` – применяет фильтр к одному каналу с помощью `tbb::parallel_for`.
- `RysevMGaussFilterTBB::RunImpl` – последовательно вызывает `ApplyKernelToChannel` для трёх каналов.

**Особенности TBB-реализации:**
- Нет слияния операций (каналы обрабатываются последовательно). Однако внутри канала используется детализированное распараллеливание по строкам.
- grainsize не задан – полагаемся на автоматическую настройку TBB.
- Память: разделяемая, записи в непересекающиеся индексы.
- Отсутствие редукций (нет глобальных сумм в основном цикле, только в PostProcessing, который выполняется последовательно). 

**Потенциальные улучшения (не реализованы в данной версии):**
- Параллельная обработка каналов с помощью `tbb::parallel_for` по каналам.
- Слияние циклов для улучшения локальности кэша.

## 6. Проверка корректности

Корректность проверена сравнением с SEQ-версией на следующих тестах:
- Постоянное изображение (все пиксели 128).
- Горизонтальный градиент.
- Случайные изображения размером 128×128, 256×256, 512×512.
- Реальное изображение `pic.ppm`.

Для всех тестов выходные суммы пикселей совпадают с эталоном при любом числе потоков (1, 2, 4, 8, 14). Визуально отфильтрованные изображения идентичны. TBB-версия корректно обрабатывает края, поскольку `ComputePixelValue` проверяет границы независимо от распараллеливания.

## 9. Выводы

TBB-версия демонстрирует высокую масштабируемость (ускорение до 8.59× на 14 потоках) благодаря задачно-ориентированной модели и механизму перехвата работы. Даже на одном потоке накладные расходы минимальны (0.99×). В отличие от OpenMP, TBB автоматически балансирует нагрузку при обработке строк, что особенно полезно для изображений с неравномерной сложностью (например, на краях). 

Ограничения: последовательная обработка каналов (по три канала) не использует TBB для внешнего уровня параллелизма, что могло бы дать дополнительный прирост на системах с числом потоков >3. Также без явного указания grainsize для очень малых изображений (менее 256×256) создание большого количества задач (по одной на строку) может привести к снижению производительности. 

Рекомендуется использовать TBB-реализацию для обработки изображений размером не менее 512×512, где преимущества work-stealing полностью раскрываются.
