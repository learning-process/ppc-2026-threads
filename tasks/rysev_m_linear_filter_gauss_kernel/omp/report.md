## 1. Контекст

Параллельная реализация линейной фильтрации изображений с использованием OpenMP. Цель — ускорить обработку относительно последовательной версии за счёт распараллеливания независимых вычислений над строками изображения внутри каждого цветового канала. 

## 2. Постановка задачи

Решение задачи фильтрации изображения ядром Гаусса 3×3 (RGB) с обработкой краёв путём пропуска выходящих за границу пикселей. SEQ-версия служит базой для проверки корректности и оценки ускорения.

**Входные данные:** целое число `GetInput()` – размер квадратного изображения (или 0 для загрузки реального `pic.ppm`), генерируются случайные пиксели RGB.

**Выходные данные:** отфильтрованное изображение и сумма пикселей для верификации.

## 3. Базовый алгоритм

Последовательная версия: для каждого канала (0..2) и для каждой строки и столбца вычисляется свёртка с ядром 3×3. Вычислительная сложность – O(H×W×9) на канал. Основная нагрузка – внутренний цикл по 9 элементам ядра. Параллелизм вводится на уровне обработки строк внутри канала.

## 4. Схема распараллеливания

**Параллельный регион:** функция `ApplyKernelToChannel` содержит директиву `#pragma omp parallel for` перед циклом по строкам.

**Область видимости переменных:**
- `shared`: `rows, cols, channels, input, output, channel`
- `private`: `row` (итератор цикла, автоматически приватный)

**Выбран `schedule(static)`** – равномерное распределение строк между потоками, так как каждая строка требует одинакового количества операций (9 умножений на пиксель). Дисбаланс отсутствует.

**Синхронизация:**
- Неявный барьер в конце `parallel for` гарантирует, что все строки обработаны перед переходом к следующему каналу.
- Записи в `output` происходят по непересекающимся индексам (разные строки), гонки данных исключены.
- Чтение `input` – только для чтения, безопасно.

**Ключевой фрагмент (код из `ops_omp.cpp`):**
```cpp
void RysevMGaussFilterOMP::ApplyKernelToChannel(int channel, int rows, int cols) {
  const auto &input = input_image_;
  auto &output = output_image_;
  int channels = channels_;

#pragma omp parallel for default(none) shared(rows, cols, channels, input, output, channel)
  for (int row = 0; row < rows; ++row) {
    for (int col = 0; col < cols; ++col) {
      float sum = ComputePixelValue(row, col, channel, rows, cols, channels, input);
      std::size_t out_idx = (static_cast<std::size_t>(row) * cols) + col;
      out_idx = (out_idx * channels) + channel;
      output[out_idx] = static_cast<uint8_t>(std::clamp(sum, 0.0F, 255.0F));
    }
  }
}
```

## 5. Детали реализации

**Файлы:** `omp/include/ops_omp.hpp`, `omp/src/ops_omp.cpp`

**Изменения относительно SEQ:**
- Цикл по строкам внутри `ApplyKernelToChannel` распараллелен с помощью `#pragma omp parallel for`.
- Все разделяемые данные – `input`, `output` – доступны только для чтения или по непересекающимся индексам.
- Гонки данных отсутствуют: каждый поток пишет в свои строки выходного изображения.

**Устранение рисков гонок:**
- Локальная переменная `sum` в `ComputePixelValue` приватна по определению.
- Доступ к `output[out_idx]` – разные `out_idx` для разных строк.
- Не требуется `reduction`, так как нет глобального аккумулятора.

## 6. Проверка корректности

Корректность проверена сравнением с SEQ-версией на следующих тестах:
- Постоянное изображение (все пиксели 128).
- Горизонтальный градиент.
- Случайные изображения размером 128×128, 256×256, 512×512.
- Реальное изображение `pic.ppm` (при наличии).

Для всех тестов выходные суммы пикселей совпадают с эталоном при любом числе потоков (1, 2, 4, 8, 14). Визуально отфильтрованные изображения идентичны.

## 9. Выводы

OpenMP-версия обеспечивает ускорение до 7.7 на 14 потоках для изображения 1024×1024. Модель распараллеливания «по строкам» хорошо масштабируется, пока число строк не станет слишком маленьким. Благодаря отсутствию глобальных аккумуляторов синхронизация требуется только один раз на канал (неявный барьер). 

Ограничения: при малых размерах изображения (меньше 512×512) накладные расходы на создание потоков и распределение строк могут свести ускорение к нулю или сделать его отрицательным. Для достижения максимальной производительности рекомендуется обрабатывать изображения размером не менее 1024×1024.
