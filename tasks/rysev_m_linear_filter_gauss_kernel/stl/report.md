## 1. Контекст

Параллельная реализация линейной фильтрации изображений с использованием средств стандартной библиотеки C++ – `std::thread`. В отличие от OpenMP, TBB и гибридных подходов, здесь потоки создаются и уничтожаются для каждого канала изображения. Это демонстрирует базовый уровень параллелизма без дополнительных зависимостей, но с потенциально высокими накладными расходами на создание/присоединение потоков.

## 2. Постановка задачи

Решение задачи фильтрации изображения ядром Гаусса 3×3 (RGB) с обработкой краёв путём пропуска выходящих за границу пикселей. SEQ-версия служит эталоном корректности и производительности.

**Входные данные:** целое число `GetInput()` – размер квадратного изображения (или 0 для загрузки реального `pic.ppm`), генерируются случайные пиксели RGB.

**Выходные данные:** отфильтрованное изображение и сумма пикселей для верификации.

## 3. Базовый алгоритм

Последовательная версия: для каждого канала (0..2) и для каждой строки и столбца вычисляется свёртка с ядром 3×3. Вычислительная сложность – O(H×W×9) на канал. Параллелизм в STL-версии вводится на уровне обработки строк внутри каждого канала с помощью ручного создания потоков.

## 4. Схема распараллеливания

**Примитивы:** `std::thread` – создание и присоединение потоков.

**Разбиение данных:** диапазон строк `[0, rows)` делится на `num_threads` непрерывных блоков. Количество потоков определяется как `std::thread::hardware_concurrency()`, но ограничивается числом строк (при малых изображениях). 
- Размер блока: `rows_per_thread = rows / num_threads`
- Остаток распределяется между первыми `remainder` потоками по одной дополнительной строке.

**Карта поток → строки:** 
- Поток 0: `[0, rows_per_thread + (0 < remainder ? 1 : 0))`
- Поток 1: следующий блок, и т.д.

**Создание потоков:** внутри `ApplyKernelToChannel` для каждого потока создаётся лямбда-функция, которая обрабатывает назначенные строки. Потоки запускаются немедленно.

**Синхронизация:** главный поток вызывает `worker.join()` для каждого потока – ожидает завершения всех потоков перед обработкой следующего канала. Никаких дополнительных блокировок не требуется, так как каждый поток пишет в непересекающиеся строки выходного массива.

**Ключевой фрагмент (код из `ops_stl.cpp`):**
```cpp
void RysevMGaussFilterSTL::ApplyKernelToChannel(int channel, int rows, int cols) {
  const auto &input = input_image_;
  auto &output = output_image_;
  int channels = channels_;

  unsigned int num_threads = std::thread::hardware_concurrency();
  if (num_threads == 0) num_threads = 1;
  if (rows > 0) num_threads = std::min(num_threads, static_cast<unsigned int>(rows));

  std::vector<std::thread> workers;
  int rows_per_thread = rows / static_cast<int>(num_threads);
  int remainder = rows % static_cast<int>(num_threads);
  int start_row = 0;

  for (unsigned int thread_idx = 0; thread_idx < num_threads; ++thread_idx) {
    int end_row = start_row + rows_per_thread + (thread_idx < remainder ? 1 : 0);
    workers.emplace_back([&, start_row, end_row]() {
      for (int row = start_row; row < end_row; ++row)
        for (int col = 0; col < cols; ++col)
          // вычисление свёртки
    });
    start_row = end_row;
  }
  for (auto &worker : workers) if (worker.joinable()) worker.join();
}
```
**Особенности STL-подхода в данной реализации:**
    Потоки создаются заново для каждого канала (3 раза). Это приводит к значительным накладным расходам на создание/уничтожение потоков, особенно на малых изображениях.
    Отсутствует пул потоков (в отличие от ThreadPool из примера сопряжённых градиентов). Каждый вызов ApplyKernelToChannel порождает новую группу потоков.
    Балансировка нагрузки статическая (фиксированное разбиение строк). При различной вычислительной сложности строк (например, на краях строки обрабатываются быстрее из-за пропуска элементов ядра) возможен дисбаланс.

## 5. Детали реализации

**Файлы:** `stl/include/ops_stl.hpp`, `stl/src/ops_stl.cpp`

**Функции:**
- `ApplyKernelToChannel` – создаёт потоки, разбивает строки, ожидает завершения.
- `RysevMGaussFilterSTL::RunImpl` – последовательно вызывает `ApplyKernelToChannel` для трёх каналов.

**Управление потоками:**
- Количество потоков определяется через `std::thread::hardware_concurrency()`. На тестовой системе (Intel Core Ultra 5 225U) это 14.
- При `rows < num_threads` число потоков ограничивается числом строк.
- Потоки получают захват `[&, start_row, end_row]` – все данные разделяемые, но каждый поток работает только со своими строками.

**Память и гонки:**
- Запись в `output_image_` – по разным индексам (разные строки), гонок нет.
- Чтение `input_image_` – только для чтения, безопасно.
- Потоки не используют блокировки, что хорошо для производительности, но требует строгого разбиения по непересекающимся диапазонам.

**Потенциальные проблемы:**
- Создание и присоединение потоков на каждый канал (3 раза). Для изображения 1024×1024 и 14 потоков это означает создание 14×3 = 42 потоков, что заметно дороже однократного создания пула.
- При малом размере изображения (например, 256×256) число строк меньше числа потоков, и часть потоков не создаётся, эффективность падает.
- Отсутствие механизма перехвата работы (work-stealing) – самый медленный поток определяет общее время.

## 6. Проверка корректности

Корректность проверена сравнением с SEQ-версией на следующих тестах:
- Постоянное изображение (все пиксели 128).
- Горизонтальный градиент.
- Случайные изображения размером 128×128, 256×256, 512×512.
- Реальное изображение `pic.ppm`.

Для всех тестов выходные суммы пикселей совпадают с эталоном при любом числе потоков (1, 2, 4, 8, 14). Визуально отфильтрованные изображения идентичны. Корректность обработки краёв сохраняется, так как `ComputePixelValue` проверяет границы независимо от распараллеливания.

## 9. Выводы

STL-версия с ручным управлением потоками даёт заметный выигрыш по сравнению с последовательной (до 6.25× на 14 потоках), но уступает специализированным библиотекам (OpenMP, TBB, гибрид). Основные недостатки:
1. Потоки создаются заново для каждого канала (3 раза), что приводит к накладным расходам на создание/уничтожение.
2. Статическое разбиение строк без work-stealing – при неравномерной нагрузке (например, из-за границ изображения) возможен простой потоков.
3. Отсутствие пула потоков – в отличие от идеального `ThreadPool` из примера сопряжённых градиентов.

**Преимущества:** минимальная зависимость (только стандартная библиотека), полный контроль над параллелизмом, отсутствие блокировок. STL-подход может быть оправдан для встраиваемых систем или сред, где OpenMP/TBB недоступны. Однако для большинства задач фильтрации изображений рекомендуется использовать TBB или OpenMP как более производительные и удобные.
