## 1. Контекст

Гибридная параллельная реализация линейной фильтрации изображений, использующая одновременно OpenMP и Intel oneTBB. OpenMP применяется для крупноблочного разделения строк изображения между потоками на верхнем уровне, а TBB — для тонкой балансировки обработки пикселей внутри каждого блока через механизм перехвата работы (work-stealing). Такой подход позволяет эффективно загружать все ядра даже при неравномерной обрабатываемости строк (например, на границах изображения).

## 2. Постановка задачи

Решение задачи фильтрации изображения ядром Гаусса 3×3 (RGB) с обработкой краёв путём пропуска выходящих за границу пикселей. SEQ-версия служит эталоном корректности и производительности.

**Входные данные:** целое число `GetInput()` – размер квадратного изображения (или 0 для загрузки реального `pic.ppm`), генерируются случайные пиксели RGB.

**Выходные данные:** отфильтрованное изображение и сумма пикселей для верификации.

## 3. Базовый алгоритм

Последовательная версия: для каждого канала (0..2) и для каждой строки и столбца вычисляется свёртка с ядром 3×3. Вычислительная сложность – O(H×W×9) на канал. Основная нагрузка – внутренний цикл по 9 элементам ядра. Параллелизм вводится на уровне обработки строк внутри канала.

## 4. Межпроцессная схема

Межпроцессное взаимодействие в данной реализации отсутствует. Все вычисления выполняются в рамках одного процесса с использованием общей памяти. MPI-вызовы не используются. Это осознанное решение: для фильтрации изображений в общей памяти достаточно двухуровневого параллелизма внутри процесса (OpenMP + TBB), который даёт хорошую масштабируемость без накладных расходов на передачу сообщений.

## 5. Внутрипроцессная схема

**Двухуровневый параллелизм (OpenMP + TBB):**

1. **Верхний уровень (OpenMP):** `#pragma omp parallel for` с `num_threads` потоками. Диапазон строк `[0, rows)` делится на `num_threads` крупных горизонтальных полос. Используется `schedule(static)` для равномерного распределения.

2. **Нижний уровень (TBB):** внутри каждой полосы `tbb::parallel_for` обрабатывает строки с настраиваемым размером зерна (grainsize). Это позволяет work-stealing планировщику TBB перераспределять строки между потоками OpenMP при возможном дисбалансе (например, если некоторые строки содержат больше граничных пикселей и обрабатываются чуть дольше).

**Примитивы TBB:**
- `tbb::parallel_for` — для обхода строк внутри блока.
- `tbb::blocked_range<int>` – диапазон строк с заданным зерном.

**Размер зерна (grainsize):**
- `ApplyKernelToChannelHybrid`: grainsize = 8 строк. Небольшое зерно выбрано для улучшения балансировки, так как каждая строка содержит `cols × 9` операций, и разброс времени выполнения строк минимален.

**Слияние операций:** в отличие от SEQ, обработка трёх каналов также распараллелена: для каждого канала создаётся отдельная задача TBB, что позволяет обрабатывать каналы одновременно (если число потоков позволяет).

**Схема распараллеливания каналов:**
```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, channels, 1),
  [&](const tbb::blocked_range<int>& range) {
    for (int ch = range.begin(); ch < range.end(); ++ch) {
      ApplyKernelToChannelHybrid(ch, rows, cols);
    }
  });
  ```
**Синхронизация:**
    Неявный барьер OpenMP в конце #pragma omp parallel for.
    tbb::parallel_for сам управляет синхронизацией через планировщик.
    Запись в разные индексы output (разные строки и каналы) не требует дополнительной синхронизации.

**Ключевой фрагмент (гибридная обработка одного канала):**
```cpp
void RysevMGaussFilterALL::ApplyKernelToChannelHybrid(int channel, int rows, int cols) {
  const auto &input = input_image_;
  auto &output = output_image_;
  int channels = channels_;

  #pragma omp parallel for schedule(static)
  for (int block = 0; block < omp_get_max_threads(); ++block) {
    int start_row = block * (rows / omp_get_max_threads());
    int end_row = (block == omp_get_max_threads() - 1) ? rows : (block + 1) * (rows / omp_get_max_threads());

    tbb::parallel_for(tbb::blocked_range<int>(start_row, end_row, 8),
      [&](const tbb::blocked_range<int>& range) {
        for (int row = range.begin(); row < range.end(); ++row) {
          for (int col = 0; col < cols; ++col) {
            float sum = ComputePixelValue(row, col, channel, rows, cols, channels, input);
            std::size_t out_idx = (static_cast<std::size_t>(row) * cols + col) * channels + channel;
            output[out_idx] = static_cast<uint8_t>(std::clamp(sum, 0.0F, 255.0F));
          }
        }
      });
  }
}
```
Такая схема обеспечивает равномерную загрузку даже при небольшом количестве строк: OpenMP делит на крупные блоки (по числу потоков), а TBB внутри каждого блока динамически перераспределяет строки, выравнивая нагрузку.

## 6. Детали реализации

**Файлы:** `all/include/ops_all.hpp`, `all/src/ops_all.cpp`
**Функции:**
- `ApplyKernelToChannelHybrid` – гибридная обработка одного канала (OpenMP + TBB).
- `RysevMGaussFilterALL::RunImpl` – запускает `tbb::parallel_for` по каналам, внутри каждого канала – гибридная обработка.

**Узкие места:**
- Двойной параллелизм (OpenMP + TBB) создаёт дополнительные накладные расходы на вложенные параллельные регионы. Однако однократное создание потоков OpenMP и переиспользование worker-потоков TBB минимизирует этот эффект.
- При очень малых размерах изображения (например, 256×256) накладные расходы могут превысить выигрыш.

**Оптимизации, применённые в ALL:**
- Параллельная обработка каналов (до 3 каналов, что полезно на системах с числом потоков ≥4).
- Блочное разбиение строк с двумя уровнями.
- Использование `tbb::blocked_range` с зерном 8 для тонкой балансировки.

## 7. Проверка корректности

Корректность проверена сравнением с SEQ-версией на следующих тестах:
- Постоянное изображение (все пиксели 128).
- Горизонтальный градиент.
- Случайные изображения размером 128×128, 256×256, 512×512.
- Реальное изображение `pic.ppm`.

Для всех тестов выходные суммы пикселей совпадают с эталоном при любом числе потоков (1, 2, 4, 8, 14). Визуально отфильтрованные изображения идентичны. Гибридная версия также корректно обрабатывает края.
