## 1. Контекст

Необходимость создания гибридной версии алгоритма обусловлена архитектурными особенностями современных
высокопроизводительных вычислительных систем (HPC), объединяющих многоядерные узлы в общую распределенную сеть. Чистые
модели внутриузлового распараллеливания (такие как OpenMP, TBB или STL) ограничены ресурсами одного процессора и общей
шиной памяти. В то же время классическая модель MPI при запуске по одному процессу на каждое ядро порождает избыточные
накладные расходы на межпроцессное взаимодействие (IPC) и дублирование локальных структур данных. Гибридный подход
позволяет совместить крупнозернистый межпроцессный параллелизм для распределения данных по узлам сети и легковесный
внутрипроцессный параллелизм на базе асинхронных задач для эффективной утилизации ядер внутри каждого отдельного ранга.

## 2. Постановка задачи

- **Входные данные:** Объект `std::vector<int>` размера $N$, изначально доступный на корневом процессе (`rank == 0`).
- **Выходные данные:** Объект `std::vector<int>` размера $N$, собранный на корневом процессе, элементы которого
  переставлены в порядке неубывания.
- **Базовый эталон (Baseline):** В качестве эталона для валидации, расчета ускорения ($Speedup$) и эффективности
  ($Efficiency$) используется последовательная версия алгоритма ($T_1 = 278.56$ мс), описанная в
  [seq/report.md](../seq/report.md). Дополнительно проводится методическое сопоставление результатов с чистым
  внутриузловым backend-ом на базе STL.

## 3. Базовый алгоритм

В качестве вычислительного ядра последовательной версии используется поразрядная сортировка (Radix Sort) по байтам.
Внутри каждого разряда (байта) применяется распределяющая сортировка подсчетом (Counting Sort).

## 4. Межпроцессная схема

Межпроцессный уровень абстракции реализован средствами библиотеки MPI и организует распределенное взаимодействие между
независимыми процессами в рамках глобального коммуникатора `MPI_COMM_WORLD`.

### Роли процессов (Ranks)

- **Корневой процесс (`rank == 0`):** Выступает в роли диспетчера и управляющего узла. Производит первичную валидацию,
  вычисляет параметры глобального распределения данных, осуществляет физическую нарезку и рассылку блоков, а на
  финальном этапе собирает отсортированные части в результирующий контейнер.
- **Рабочие процессы (`rank > 0`):** Являются вычислительными узлами. Получают свою порцию данных, активируют
  внутрипроцессное вычислительное ядро и принимают участие в глобальных фазах обмена сети слияния Бэтчера.

### Распределение данных

Глобальный массив распределяется сбалансированно. Функция `CalculateDistribution` рассчитывает массивы размеров
`send_counts` и смещений `displs` по формуле базового размера `N / size` с распределением остатка `N % size` по первым
рангам. Физический обмен данными между адресными пространствами процессов выполняется с помощью функции `MPI_Scatterv`,
что гарантирует бесконфликтную доставку сегментов произвольного размера на каждый ранг.

### Используемые MPI-вызовы и их назначение

1. `MPI_Comm_rank` и `MPI_Comm_size` — вызываются на старте метода `RunImpl` для инициализации топологии, получения
   уникального идентификатора текущего процесса в коммуникаторе и общего числа запущенных процессов.
2. `MPI_Scatterv` — блочно распределяет исходный вектор из адресного пространства `rank == 0` по локальным векторам
   `local_data` всех процессов в коммуникаторе.
3. `MPI_Sendrecv` — применяется внутри метода `NetworkMergeAndSplit` для организации парного обмена между процессами.
   Первый вызов обменивается размерами локальных векторов для аллокации буфера `partner_data`, второй — осуществляет
   непосредственный одновременный прием и передачу элементов сортируемых массивов, исключая возникновение дедлоков.
4. `MPI_Gatherv` — выполняет редукцию данных, собирая отсортированные локальные массивы обратно в результирующий вектор
   в адресном пространстве корневого процесса.

## 5. Внутрипроцессная схема

После получения процессом своего сегмента данных `local_data` в методе `LocalSortPhase` активируется внутренний уровень
иерархического параллелизма.

### Используемая внутрипроцессная технология

Внутри каждого процесса параллелизм реализуется на базе стандартных механизмов C++ STL: `std::async` с политикой
принудительного запуска `std::launch::async` и объектов синхронизации `std::future`.

### Разделение работы между потоками

Локальный массив `local_data` внутри процесса нарезается на подоблоки, количество которых эквивалентно числу доступных
аппаратных потоков (`std::thread::hardware_concurrency`). По аналогии с глобальным этапом, каждый подоблок передается в
асинхронную задачу:

```cpp
void LevonychevIRadixBatcherSortALL::LocalSortPhase(std::vector<int> &local_data) {
  int num_threads = static_cast<int>(std::thread::hardware_concurrency());
  int loc_n = static_cast<int>(local_data.size());

  std::vector<std::vector<int>> blocks(num_threads);
  int current_pos = 0;
  for (int i = 0; i < num_threads; ++i) {
    int b_size = (loc_n / num_threads) + (i < (loc_n % num_threads) ? 1 : 0);
    if (b_size > 0) {
      blocks[i].assign(local_data.begin() + current_pos, local_data.begin() + current_pos + b_size);
      current_pos += b_size;
    }
  }

  std::vector<std::future<void>> sort_futures;
  for (auto &b : blocks) {
    if (!b.empty()) {
      sort_futures.push_back(std::async(std::launch::async, [&b]() { LocalRadixSort(b); }));
    }
  }
  for (auto &f : sort_futures) {
    f.wait();
  }

  LocalBatcherMerge(blocks);

  local_data.clear();
  for (const auto &b : blocks) {
    local_data.insert(local_data.end(), b.begin(), b.end());
  }
}
```

Потоки работают изолированно, осуществляя побайтовую сортировку `LocalRadixSort` над своими независимыми буферами.

### Объединение локальных результатов внутри ранга

Сборка подоблоков в единый отсортированный массив ранга осуществляется методом `LocalBatcherMerge`, реализующим
локальную итерационную сеть Бэтчера. Вызовы компараторов `CompareAndMergeBlocks` на каждом внутреннем слое `k` также
оборачиваются в `std::async`. Общая память `blocks` разделяется по ссылкам без блокировок (lock-free), поскольку
топология индексов гарантирует отсутствие пересечения пар на одном шаге.

## 6. Детали реализации

Гибридная схема находится в файлах `all/include/ops_all.hpp` и `all/src/ops_all.cpp`.

### Точки синхронизации и барьеры

Синхронизация в коде строго иерархична и разделена на два уровня:

1. **Внутрипроцессные барьеры:** Реализованы на уровне потоков внутри каждого ранга с помощью вызовов блокирующего
   ожидания фьючерсов:

```cpp
  std::vector<std::future<void>> sort_futures;
  for (auto &b : blocks) {
    if (!b.empty()) {
      sort_futures.push_back(std::async(std::launch::async, [&b]() { LocalRadixSort(b); }));
    }
  }
  for (auto &f : sort_futures) {
    f.wait();
  }
```

А также на этапе внутрипроцессного слияния блоков:

```cpp
  std::vector<std::future<void>> m_futures;

  for (int j = k % pr; j <= n_blocks - 1 - k; j += 2 * k) {
    for (int i = 0; i < std::min(k, n_blocks - j - k); ++i) {
      int i1 = j + i;
      int i2 = j + i + k;

      if ((i1 / (pr * 2)) == (i2 / (pr * 2))) {
        m_futures.push_back(
            std::async(std::launch::async, [&blocks, i1, i2]() { CompareAndMergeBlocks(blocks[i1], blocks[i2]); }));
      }
    }
  }

  for (auto &f : m_futures) {
    f.wait();
  }
```

Эти циклы фиксируют завершение расчетов текущего вычислительного слоя или фазы компараторов перед переходом к следующему
этапу внутри адресного пространства конкретного процесса.

2. **Межпроцессные барьеры:** Реализованы на уровне коммуникатора для синхронизации независимых процессов операционной
   системы:

```cpp
  for (int pr = 1; pr < size; pr <<= 1) {
    for (int k = pr; k > 0; k >>= 1) {
      GlobalBatcherStep(local_data, rank, size, pr, k);
      MPI_Barrier(MPI_COMM_WORLD);
    }
  }
```

Метод `MPI_Comm_rank` возвращает ранг процесса в коммуникаторе, а `MPI_Barrier` блокирует вызвавший процесс до тех пор,
пока в барьер не войдут все участники. Согласно стандарту MPI, данный вызов блокирует выполнение процесса до тех пор,
пока абсолютно все процессы коммуникатора не войдут в эту точку. В логике нашего алгоритма эта жесткая синхронизация
критически необходима на глобальном этапе. Межпроцессная сеть слияния Бэтчера строится на последовательной итерационной
смене фаз работы компараторов. Преждевременный запуск обмена данными через `MPI_Sendrecv` на следующем слое `k`
процессами, которые опередили остальные ранги, привел бы к перемешиванию некорректных (еще не подготовленных соседом)
данных, возникновению состояния гонки на сетевом уровне и полному нарушению инварианта упорядоченности сети Бэтчера.

### Потенциальные узкие места

- **Сетевой оверхед обменов:** Метод `NetworkMergeAndSplit` выполняет пересылку полных объемов данных между парами
  процессов через `MPI_Sendrecv`. На больших размерах массивов $N$ пропускная способность сетевого интерфейса становится
  главным сдерживающим фактором.
- **Частые аллокации памяти:** Внутри `NetworkMergeAndSplit` на каждом шаге сети динамически аллоцируется временный
  вектор `merged` размера двух блоков, что нагружает системный аллокатор, приводит к фрагментации кучи и вынужденным
  простоям вычислительных ядер.

## 7. Проверка корректности

Проверка корректности гибридного backend-а выполняется с помощью автоматизированных тестов:

1. **Функциональная валидация:** Результаты работы на смешанных, сугубо отрицательных и пустых структурах данных
   покомпонентно сопоставляются с последовательной версией.
2. **Проверка согласованности рангов:** Проверяется, что после выполнения `MPI_Gatherv` итоговый массив на корневом
   процессе удовлетворяет отношению `GetOutput()[i - 1] <= GetOutput()[i]`. Топология глобальной сети Бэтчера
   гарантирует миграцию элементов в нужные ранги, что исключает рассинхронизацию данных между процессами.

_Комментарий к результатам:_ Экспериментальные данные демонстрируют стабильный, но умеренный уровень ускорения
вычислительного ядра, стабилизирующийся в районе **$1.25\text{x} - 1.32\text{x}$** относительно последовательного
эталона. Такое поведение алгоритма на массиве из 16 млн элементов обусловлено балансом между вычислительным выигрышем и
системными издержками:

1. **Положительный эффект внутрипроцессного распараллеливания:** Достижение стабильного ускорения (в пике до 1.32x)
   обеспечивается за счет работы фазы `LocalSortPhase`. Разделение данных внутри каждого ранга на потоки через
   `std::async` выполняется в рамках общей памяти эффективно. Это позволяет утилизировать ядра процессора Core i7-9700k
   на этапе побайтовой сортировки подоблоков.
2. **Влияние накладных расходов IPC при локальном запуске:** Поскольку эмуляция распределенных процессов происходит на
   одной физической машине, вызовы `NetworkMergeAndSplit` задействуют механизмы IPC операционной системы. Постоянная
   пересылка массивов данных (16 млн `int` занимают около 64 МБ) между изолированными адресными пространствами через
   `MPI_Sendrecv` создает стабильный оверхед на копирование памяти в ОЗУ, что удерживает кривую ускорения от линейного
   роста.
3. **Ограничение пропускной способности шины памяти (Memory Bandwidth Bound):** Поразрядная сортировка критична к
   скорости подсистемы памяти из-за регулярного чтения и записи больших объемов данных. При одновременной работе 4
   рангов по 2 потока (суммарно 8 воркеров) возникает конкуренция за единый контроллер памяти. Насыщение шины приводит к
   тому, что ядра CPU частично простаивают в очередях на транзакции памяти, из-за чего эффективность падает до 16.1%.
4. **Сдерживающий фактор барьерной синхронизации:** Специфика сети Бэтчера обязывает вызывать
   `MPI_Barrier(MPI_COMM_WORLD)` после каждого шага компараторов. Неравномерность выделения квантов времени
   планировщиком ОС для тяжелых процессов вынуждает более быстрые ранги ожидать отстающие на барьере, что объясняет
   небольшое проседание скорости на конфигурации $4 \times 2$ (до 215.93 мс) по сравнению с $3 \times 2$.

## 10. Выводы

1. Разработанная гибридная схема ALL (MPI + std::async) демонстрирует полную функциональную корректность и обеспечивает
   умеренное масштабирование с выходом на эффективное время **$211.03\text{ мс}$** (ускорение $1.32\text{x}$) при
   задействовании гибридной конфигурации воркеров.
2. Применение гибридной модели MPI + Threads в пределах одного многоядерного узла не дает преимущества перед чистыми
   внутриузловыми технологиями (такими как TBB или OpenMP). Необходимость изоляции данных между процессами вынуждает
   использовать примитивы `MPI_Scatterv` и `MPI_Sendrecv`, которые в данных условиях приводят к избыточному копированию
   векторов внутри оперативной памяти.
3. Алгоритмическая природа сети Бэтчера с жестким барьерным ожиданием `MPI_Barrier` на каждом внутреннем слое выступает
   основным ограничителем роста.
