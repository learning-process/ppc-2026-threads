## 1. Введение

Задача высокопроизводительной сортировки больших массивов данных является классической вычислительной проблемой,
эффективность решения которой определяет скорость работы СУБД, поисковых движков и систем обработки сигналов. Комбинация
поразрядной сортировки (Radix Sort) и сети сортировки Бэтчера (Batcher's Odd-Even Merge) представляет собой идеальный
полигон для исследования иерархического параллелизма и сравнения различных моделей программирования. Данный алгоритм
сочетает в себе фазу локальной обработки данных с высокой степенью независимости (идеально для внутриузлового
параллелизма) и фазу итерационного слияния блоков, требующую жесткой координации, барьерной синхронизации и парных
обменов (что позволяет детально исследовать накладные расходы различных сред исполнения от OpenMP до MPI).

## 2. Единая постановка задачи

- **Входные данные:** Объект `std::vector<int>` размера $N$.
- **Выходные данные:** Объект `std::vector<int>` размера $N$, содержащий те же элементы, переставленные в порядке
  неубывания.
- **Ограничения:** Алгоритм должен корректно обрабатывать знаковые целые числа (включая отрицательные значения, ноль и
  дубликаты), а также устойчиво работать на граничных тестах (массивы нулевого и единичного размера).
- **Критерий корректности:** Выходной вектор должен удовлетворять предикату упорядоченности: output[i - 1] <= output[i]
  для всех элементов, а его компонентный состав должен быть пермутацией исходного.

## 3. Единая методика эксперимента

### Аппаратное и программное окружение

- **CPU:** Intel Core i7-9700k
- **RAM:** 16 GB
- **OS:** Ubuntu 24.04 LTS
- **Compiler:** GCC 13.3.0
- **Build type:** Release

### Переменные окружения и параметры запуска

- Для OpenMP: количество потоков регулировалось через `OMP_NUM_THREADS`.
- Для TBB: управление пулом рабочих потоков осуществлялось через `tbb::global_control`.
- Для STL: количество асинхронных тасок `std::async` определялось размером контейнера блоков.
- Для ALL (MPI + STL): конфигурация задавалась через `mpirun` с флагом `-n` в сочетании с внутренним ограничением
  потоков.

### Метрики и методика измерений

Тестирование проводилось на фиксированном размере задачи $N = 16\,777\,216$ ($16\text{М}$) элементов. Данные
генерировались псевдослучайным образом с использованием `std::mt19937` в диапазоне знаковых целых чисел. Для каждого
backend-а выполнялось по $10$ независимых запусков вычислительного ядра `task_run`, после чего вычислялось медианное
время работы в миллисекундах ($\text{мс}$).

Метрики вычислялись по следующим формулам относительно последовательного baseline ($T_1 = 278.56\text{ мс}$):

- **Ускорение ($Speedup$):** $$S_p = \frac{T_1}{T_p}$$ где $T_p$ — медианное время работы параллельной версии.
- **Эффективность ($Efficiency$):** $$E_p = \frac{S_p}{W} \times 100\%$$ где $W$ — общее число выделенных работников
  ($Total\ Workers$). Для модели ALL нормировка велась по полной гибридной конфигурации:
  $$W = \text{Ranks} \times \text{Threads}$$

## 4. Сводка корректности

Валидация каждого из параллельных backend-ов (`omp`, `tbb`, `stl`, `all`) осуществлялась через сквозную систему
модульных тестов на базе фреймворка Google Test посредством верификации результатов относительно последовательной версии
(`seq`).

Функциональные проверки включали в себя сопоставление выходных данных на четырех базовых конфигурациях:

- Массивы граничной длины (из одного элемента).
- Наборы, содержащие только положительные целые числа.
- Наборы, содержащие только отрицательные целые числа.
- Смешанные входные датасеты.

Во всех реализациях знаковые числа корректно сортировались за счет побайтовой инверсии старшего бита на последнем байте
типа `int`. Все параллельные технологии детерминированно прошли тесты на согласованность данных. Ограничений
применимости по размеру или профилю входных значений у разработанных backend-ов не выявлено.

## 6. Интерпретация различий

Анализ таблицы и экспериментальных данных позволяет сделать несколько ключевых архитектурных выводов о поведении
поразрядной сортировки с четно-нечетным слиянием Бэтчера в различных параллельных средах.

1. **Влияние накладных расходов на создание параллельных контекстов (Аномалия при $Workers = 1$):** При запуске
   потоковых версий (`OMP`, `TBB`, `STL`) на одном рабочем потоке наблюдается падение производительности по сравнению с
   последовательным baseline (`SEQ` `task_run` $= 278.56\text{ мс}$). Время выполнения увеличивается до
   $291.20\text{ мс}$, $285.50\text{ мс}$ и $289.70\text{ мс}$ соответственно. Это классический оверхед на инициализацию
   многопоточного рантайма, выделение внутренних дескрипторов задач и создание инфраструктуры синхронизации внутри ядра,
   которые не окупаются при отсутствии реального масштабирования на фазе локальной сортировки.

2. **Особенности масштабирования внутриузловых backend-ов (`OMP`, `TBB`, `STL`):** При изменении временных показателей
   на $4$ и $8$ потоках профили рантаймов разделились, выявив особенности управления потоками в ОС:
    - В `OMP` фиксированные барьеры между итерационными слоями компараторов заставляют потоки простаивать, ожидая самый
      медленный участок. Минимальное время достигается на $4$ потоках ($212.64\text{ мс}$), после чего на $8$ потоках
      производительность падает из-за накладных расходов на синхронизацию.
    - `TBB` на $4$ потоках работает за $208.76\text{ мс}$, а на $8$ потоках
      зафиксировал стагнацию ($209.44\text{ мс}$). Это показывает, что при глубоком дроблении сети компараторов Бэтчера
      накладные расходы самого рантайма oneTBB на менеджмент мелких задач начинают конкурировать с полезной
      вычислительной работой.
    - В `STL` асинхронное порождение задач политикой `std::launch::async` показало наилучшую масштабируемость на
      максимальном числе воркеров, достигнув абсолютного минимума времени ($200.00\text{ мс}$ на $8$ потоках). На
      большом количестве ядер ($8$ потоков) высокая вычислительная нагрузка фазы локальной побайтовой сортировки смогла
      полностью компенсировать системный оверхед ОС на создание фьючерсов и управление ресурсами `std::future` на этапе
      слияния Бэтчера.

3. **Специфика распределенной памяти на локальном узле (`ALL`):** Гибридная версия `MPI + STL` показала стабильные
   результаты, но не смогла продемонстрировать линейное ускорение, зафиксировав время $211.03\text{ мс}$ на конфигурации
   из $3$ процессов (`Ranks`) по $2$ потока. Межпроцессное слияние блоков на одном физическом узле связано с отправкой
   сообщений через вызовы `MPI_Sendrecv`. В условиях отсутствия реальной сети и разделяемой физической памяти эти вызовы
   превращаются в избыточное внутрисистемное копирование буферов через механизмы IPC операционной системы. Глобальные
   вызовы `MPI_Barrier`, координирующие этапы слияния Бэтчера между рангами, дополнительно увеличивают время простоя
   вычислителей.

4. **Аппаратные лимиты архитектуры:** Поскольку процессор Intel Core i7-9700k обладает $8$ физическими ядрами без
   поддержки Hyper-Threading, распределение ресурсов происходит линейно. В отличие от OpenMP и TBB, которые уперлись в
   пропускную способность шины памяти и барьеры уже на $4$ потоках, рантайм STL смог утилизировать все $8$ независимых
   ядер для достижения пикового ускорения $1.39\text{x}$. Тем не менее, общая пологая кривая эффективности ($17.4\%$ на
   $8$ потоках) наглядно иллюстрирует закон Амдала: на последних шагах сети Бэтчера, где блоки сливаются в единый
   массив, алгоритм неизбежно теряет параллелизм.

## 8. Заключение

1. **Наилучшая конфигурация:** Лидером по чистой производительности в рамках одного узла является асинхронная модель
   **STL** ($Время = 200.00\text{ мс}$, $Ускорение = 1.39\text{x}$ на $8$ потоках). При полной утилизации физических
   ядер процессора параллелизм на этапе независимой побайтовой обработки перевесил накладные расходы на создание потоков
   через `std::async`.
2. **Архитектурные ограничения:** Все внутриузловые решения (`OMP`, `TBB`, `STL`) ограничены пропускной способностью
   общей шины памяти многоядерной ЭВМ (`memory-bound` алгоритм) и законом Амдала на финальных этапах слияния блоков.
   Потенциал распределенной гибридной модели `ALL` искусственно сдерживается накладными расходами на копирование данных
   через механизмы IPC операционной системы, имитирующие сетевой интерконнект в рамках одного физического ОЗУ.
3. **Направления улучшения:** Для дальнейшего развития программного комплекса целесообразно:
    - Перенести гибридную версию `ALL` на реальный вычислительный кластер со скоростной межузловой сетью (например,
      InfiniBand) для раскрытия потенциала разделения адресных пространств.
    - Внедрить кастомные или специализированные аллокаторы памяти, чтобы полностью исключить системный оверхед на
      динамическое выделение временных векторов в точках слияния компараторов Бэтчера.
    - Заменить стандартные последовательные процедуры слияния на полностью параллельный алгоритм слияния (_Parallel
      Merge_) на стыках независимых блоков.
