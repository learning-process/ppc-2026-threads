## 1. Контекст

Параллельная реализация поразрядной сортировки с использованием стандартных механизмов многопоточности C++ Standard
Library (STL) представляет собой этап адаптации алгоритма под парадигму явного управления асинхронными задачами. В
отличие от декларативного OpenMP или ориентированного на автоматическое воровство задач (Work-stealing) Intel oneTBB,
реализация на STL использует связку `std::async` и `std::future`.

## 2. Постановка задачи

- **Входные данные:** Объект `std::vector<int>` размера $N$.
- **Выходные данные:** Объект `std::vector<int>` размера $N$, содержащий те же элементы, переставленные в порядке
  неубывания.
- **Базовый эталон (Baseline):** В качестве эталона для функциональной валидации граничных условий, вычисления точных
  метрик ускорения ($Speedup$) и оценки эффективности распараллеливания используется последовательная версия алгоритма,
  детально описанная в [seq/report.md](../seq/report.md). Все замеры производительности STL-версии сопоставляются с
  временными метриками, зафиксированными для последовательной версии ($T_1 = 278.56$ мс).

## 3. Базовый алгоритм

В качестве вычислительного ядра последовательной версии используется поразрядная сортировка (Radix Sort) по байтам.
Внутри каждого разряда (байта) применяется распределяющая сортировка подсчетом (Counting Sort).

## 4. Схема распараллеливания

Распараллеливание алгоритма в рамках STL строится на базе явного порождения асинхронных задач с помощью
`std::launch::async`.

### 1. Этап параллельной локальной сортировки блоков

Вся вычислительная нагрузка фазы поразрядной сортировки распределяется в методе `ParallelRadixPhase`:

```cpp
void LevonychevIRadixBatcherSortSTL::ParallelRadixPhase(std::vector<std::vector<int>> &blocks) {
  std::vector<std::future<void>> futures;
  futures.reserve(blocks.size());

  for (auto &block : blocks) {
    futures.push_back(std::async(std::launch::async, [&block]() { RadixSortSequential(block); }));
  }
  for (auto &f : futures) {
    f.wait();
  }
}
```

- **Какая область кода параллелится:** Цикл осуществляет итерацию по всем сформированным подмассивам. Для каждого блока
  порождается отдельная асинхронная задача, внутри которой последовательно выполняется `RadixSortSequential(block)`.
- **Разделение переменных (Data Sharing):**
  - Контейнер `blocks` разделяется между асинхронными задачами. Поскольку каждая лямбда-функция захватывает по ссылке
      строго свой уникальный объект `block`, гонки за данными на этапе записи отсутствуют.
  - Массив фьючерсов является приватным для вызывающего (главного) потока и используется для накопления дескрипторов.

### 2. Этап параллельного шага слияния Бэтчера

Параллельный вызов компараторов на шаге сети слияния реализован в методе `BatcherMergeStep`:

```cpp
void LevonychevIRadixBatcherSortSTL::BatcherMergeStep(std::vector<std::vector<int>> &blocks, int p, int k) {
  const int n_blocks = static_cast<int>(blocks.size());
  std::vector<std::future<void>> futures;
  futures.reserve(static_cast<size_t>(n_blocks));

  for (int j = k % p; j <= n_blocks - 1 - k; j += 2 * k) {
    for (int i = 0; i < std::min(k, n_blocks - j - k); ++i) {
      int idx1 = j + i;
      int idx2 = j + i + k;

      if ((idx1 / (p * 2)) == (idx2 / (p * 2))) {
        futures.push_back(std::async(std::launch::async, [&blocks, idx1, idx2]() {
          MergeAndSplit(blocks.at(static_cast<size_t>(idx1)), blocks.at(static_cast<size_t>(idx2)));
        }));
      }
    }
  }

  for (auto &f : futures) {
    f.wait();
  }
}
```

- **Какая область кода параллелится:** Внутренние циклы вычисляют индексы пар блоков (`idx1` и `idx2`), подлежащих
  сравнению на текущем слое сети. Вызов `MergeAndSplit` для разрешенных топологией пар оборачивается в `std::async`.
- **Разделение переменных (Data Sharing):**
  - Индексы `idx1` и `idx2` передаются в лямбду по значению (копируются), обеспечивая изоляцию контекста.
  - Контейнер `blocks` передается по ссылке. Математические свойства топологии Бэтчера гарантируют, что на
      фиксированном шаге `k` множества индексов `{idx1, idx2}` для всех одновременно выполняемых тасок не пересекаются.

### Механизмы синхронизации

Синхронизация между потоками реализована без использования примитивов взаимоисключения (мьютексов):

1. **Барьер через фьючерсы:** В конце методов `ParallelRadixPhase` и `BatcherMergeStep` выполняется явный итерируемый
   цикл по массиву `futures`, вызывающий блокирующий метод `f.wait()`. Текущий поток приостанавливает выполнение до тех
   пор, пока абсолютно все асинхронные задачи слоя не завершат модификацию памяти.
2. Это формирует жесткие барьеры синхронизации, гарантирующие, что фаза слияния не начнется до завершения сортировки, и
   слои сети Бэтчера выполняются строго последовательно.

## 5. Детали реализации

Специфика STL-версии сосредоточена в файлах `stl/include/ops_stl.hpp` and `stl/src/ops_stl.cpp`.

### Изменения относительно последовательной версии (SEQ)

1. **Явное разделение на векторы:** В отличие от SEQ, работающей с единым массивом, в STL-версии данные физически
   копируются в `std::vector<std::vector<int>>`. Это необходимо для упрощения передачи независимых ссылок в потоки.
2. **Асинхронные обертки:** Последовательные циклы заменены на диспетчеризацию через `std::async`.
3. **Объединение через `MergeAndSplit`**: Вместо поштучного переключения компараторов над индексами элементов, в STL
   слияние реализовано крупными блоками с привлечением алгоритма `std::ranges::merge` и последующим расщеплением.

### Устранение рисков гонок за данными (Data Races)

1. На локальном этапе потоки полностью изолированы, так как работают в рамках выделенных им наборов данных
   `blocks.at(i)`.
2. На этапе слияния корректность записи обеспечивается барьером `f.wait()`. Следующий слой компараторов сети не начнет
   вычисляться, пока не завершился предыдущий, что исключает наложение чтений и записей на одни и те же субвекторы.

Валидация STL-реализации осуществляется через набор функциональных тестов в файле `tests/functional/main.cpp`.

### Сравнение с последовательной версией (SEQ)

1. **Проверка эквивалентности:** Результаты сортировки произвольных массивов STL-версией покомпонентно совпадают с

2. **Проверка предикатом:** Выходной массив проверяется на упорядоченность встроенной функцией валидации, подтверждая
   отсутствие логических ошибок при блочном слиянии.

### Набор тестов и характерные случаи

1. **Массив из 1 элемента (`one_element`):** `input_data_ = {5}`. Обрабатывается в `RunImpl` мгновенно через возврат
   исходного вектора, предотвращая деление на ноль в `GetNumBlocks`.
2. **Положительные числа (`only_positive`):** Проверяет базовую корректность побайтового сдвига и блочного деления.
3. **Отрицательные числа (`only_negative`):** Подтверждает корректность работы знакового инвертирования (`b ^= 0x80`)
   при параллельном вызове `RadixSortSequential` в разных потоках.
4. **Смешанные данные (`mixed`):** Валидирует сеть Бэтчера на предмет корректной миграции чисел между блоками через
   вызовы `MergeAndSplit`.

### Характерные примеры входа и ожидаемого выхода

- Вход (Смешанные): `[-170, 45, 75, -90, 2, 24, -802, 66]`
- Выход (Смешанные): `[-802, -170, -90, 2, 24, 45, 66, 75]`

## 9. Выводы

1. Параллельная реализация на базе механизмов STL (`std::async`) обеспечивает корректное распараллеливание с выходом на
   эффективное время **$200.00\text{ мс}$** при задействовании всех 8 физических ядер процессора.
2. Главным ограничением эффективности STL-версии является оверхед подсистемы управления потоками. В отличие от OpenMP и
   TBB, использующих пул готовых потоков, частые вызовы `std::async` на каждом слое слияния принуждают ОС тратить время
   на менеджмент ресурсов, что снижает общую выгоду от параллелизма.
3. Алгоритмическое ограничение сети Бэтчера проявляется здесь наиболее остро: барьерное ожидание через `f.wait()`
   вынуждает родительский поток последовательно засыпать и просыпаться на каждом внутреннем шаге `k`, увеличивая
   простои.
4. Потребление памяти и промахи кэша остаются высокими, поскольку метод `MergeAndSplit` выполняет глубокое копирование и
   выделение временных векторов `merged` на подэтапах слияния, нагружая общую шину данных.
