## 1. Контекст

Параллельная реализация поразрядной сортировки с использованием библиотеки Intel oneTBB (Threading Building Blocks)
представляет собой этап адаптации алгоритма под парадигму параллелизма, основанную на задачах (Task-based parallelism).
В отличие от OpenMP, ориентированного на жесткое разделение итераций между потоками операционной системы, TBB использует
динамическое планирование (Work-stealing) и концепцию диапазонов (`blocked_range`). Это позволяет эффективнее
балансировать нагрузку на многоядерных архитектурах при выполнении гибридных схем сортировки и итерационного сетевого
слияния Бэтчера.

## 2. Постановка задачи

- **Входные данные:** Объект `std::vector<int>` размера $N$.
- **Выходные данные:** Объект `std::vector<int>` размера $N$, содержащий те же элементы, переставленные в порядке
  неубывания.
- **Базовый эталон (Baseline):** В качестве эталона для функциональной валидации граничных условий, вычисления точных
  метрик ускорения ($Speedup$) и оценки эффективности распараллеливания используется последовательная версия алгоритма,
  детально описанная в [seq/report.md](../seq/report.md). Все замеры производительности TBB-версии сопоставляются с
  временными метриками, зафиксированными для последовательной версии.

## 3. Базовый алгоритм

В качестве вычислительного ядра последовательной версии используется поразрядная сортировка (Radix Sort) по байтам.
Внутри каждого разряда (байта) применяется распределяющая сортировка подсчетом (Counting Sort).

## 4. Схема распараллеливания

Распараллеливание алгоритма в TBB реализовано с помощью высокоуровневых шаблонов параллельных циклов, заменяющих явную
потоковую декомпозицию OpenMP.

### 1. Этап локальной сортировки (Блочная декомпозиция через blocked_range)

Локальная сортировка кусков данных запускается в методе `RunImpl()` через шаблонный цикл с автоматическим разбиением
диапазона:

```cpp
tbb::parallel_for(tbb::blocked_range<int>(0, n, grain), [&](const tbb::blocked_range<int> &r) {
    int left = r.begin();
    int right = r.end();

    std::vector<int> local_block(GetOutput().begin() + left, GetOutput().begin() + right);
    for (size_t i = 0; i < sizeof(int); ++i) {
      CountingSort(local_block, i);
    }
    std::ranges::copy(local_block, GetOutput().begin() + left);
  });
```

- **Какая область кода параллелится:** Конструкция разделяет индексное пространство от `0` до `n` на блоки размера
  `grain = std::max(1, n / num_threads)`. Внутри лямбда-функции каждый поток получает свой поддиапазон `r`, вычисляет
  локальные `left` и `right`, копирует элементы в `local_block`, побайтово сортирует их и с помощью `std::ranges::copy`
  возвращает в исходный вектор.
- **Разделение переменных (Data Sharing):**
  - Переменные `n` и `grain` захватываются лямбда-выражением по значению или ссылке как доступные только для чтения.
  - Вектор `GetOutput()` захватывается по ссылке. Безопасность обеспечивается тем, что итераторы `begin() + left` и
      `begin() + right` для разных задач гарантированно не пересекаются в памяти.
  - Объект `std::vector<int> local_block` создается внутри тела лямбды, что автоматически делает его изолированным
      (приватным) для каждой выполняемой задачи.

### 2. Этап четно-нечетного слияния Бэтчера

Параллельное применение компараторов реализовано в итеративном методе `BatcherMergeIterative()`:

```cpp
tbb::parallel_for(0, num_iters, [&](int i) {
   int j = (k % pv) + (i * (2 * k));
   BatcherCompareRange(arr, j, k, p2);
});
```

- **Какая область кода параллелится:** Внутренний цикл по индексу `i` от `0` до вычисленного количества независимых
  итераций `num_iters`. Внутри тела цикла вычисляется смещение `j` для каждого конкретного компаратора, после чего
  вызывается `BatcherCompareRange`.
- **Разделение переменных (Data Sharing):**
  - Параметры сети слияния (`arr`, `k`, `pv`, `p2`) захватываются лямбда-функцией. Модифицируемый вектор `arr`
      разделяется между задачами. Благодаря математическим свойствам сети Бэтчера, вычисленные индексы `idx1` и `idx2`
      внутри `BatcherCompareRange` для разных `i` никогда не совпадают, что предотвращает конфликты параллельного
      доступа.

### Механизмы синхронизации

Явные примитивы синхронизации (мьютексы, атомики, критические секции) в коде отсутствуют:

1. Шаблон `tbb::parallel_for` выполняет скрытый внутренний барьер — управление не вернется в вызывающий поток до тех
   пор, пока все порожденные задачи текущего цикла не завершат работу. Это гарантирует завершение локального этапа перед
   переходом к слиянию, а также строгую последовательность слоев компараторов в сети Бэтчера.
2. Локальные данные полностью изолированы, а запись в глобальный массив происходит по непересекающимся индексам, что
   делает алгоритм lock-free.

## 5. Детали реализации

Специфика реализации параллельной версии TBB сосредоточена в файлах `tbb/include/ops_tbb.hpp` и `tbb/src/ops_tbb.cpp`.

### Изменения относительно последовательной версии (SEQ)

1. **Использование `tbb::blocked_range`**: Вместо стандартных итераторов или единого вызова распределяющей сортировки
   для всего массива в `RunImpl()` применяется разбиение диапазона с заданным шагом `grain`, оптимизирующее
   распределение данных под управление планировщика TBB.
2. **Task-based декомпозиция**: Последовательные циклы заменены на конструкции `tbb::parallel_for` с лямбда-функциями.
   Это позволяет TBB использовать внутренний пул потоков и балансировку методом воровства задач (Work-stealing).
3. **Расчет итераций слияния**: На этапе Бэтчера шаг цикла `j += 2 * k` заменен на расчет абсолютного количества
   итераций `num_iters`. Это необходимо для плоского индексирования диапазона от `0` до `num_iters` внутри
   `tbb::parallel_for`, так как TBB требует линейных инкрементов в базовых параллельных циклах.

### Устранение рисков гонок за данными (Data Races)

1. **Изоляция таблиц счетчиков**: Метод `CountingSort` содержит деструктивный для параллелизма инкремент
   `++count[value_of_byte]`. Риск гонки полностью нивелирован за счет того, что сортировка подсчетом вызывается над
   изолированным вектором `local_block`, создаваемым внутри лямбды для каждого поддиапазона отдельно.
2. **Исключение race conditions на слиянии**: Архитектура сети Бэтчера спроектирована так, что на шаге `k` любые два
   индекса `idx1` и `idx2`, передаваемые в компаратор, зависят от уникального шага итерации `i`. Разные задачи
   линеаризованного цикла `tbb::parallel_for` гарантированно пишут в изолированные ячейки памяти вектора `arr`.

## 6. Проверка корректности

Валидация параллельной TBB-реализации алгоритма осуществляется путем комплексного тестирования и сопоставления
результатов со свойствами эталонной версии. Код тестов находится в файле `tests/functional/main.cpp`.

### Сравнение с последовательной версией (SEQ)

Для подтверждения математической точности параллельной схемы TBB применяется двухэтапный контроль:

1. **Функциональная эквивалентность:** Выходные данные TBB-версии на идентичных входных наборах покомпонентно совпадают
   с результатами работы последовательного алгоритма `SEQ`. Это доказывает, что квантование диапазона и перестройки в
   сети слияния не искажают данные.
2. **Проверка инварианта:** Итоговый массив валидируется предикатом `CheckTestOutputData`, который гарантирует
   выполнение условия неубывания элементов для всей последовательности.

### Набор тестов и характерные случаи

Параллельные задачи активируются на наборах данных различной структуры, что позволяет проверить корректность межпоточной
синхронизации и обработки знаковых разрядов планировщиком TBB:

1. **Размер $N = 1$ (`one_element`):** `input_data_ = {5}`. Граничный случай минимальной длины. Метод `RunImpl`
   совершает тривиальный выход без запуска параллельного планировщика `tbb::parallel_for`, предотвращая неопределенное
   поведение.
2. **Только положительные числа (`only_positive`):** `input_data_ = {170, 45, 75, 90, 2, 24, 802, 66}`. Тестирует
   базовое распределение поддиапазонов между задачами TBB и корректность сборки отсортированных частей через компараторы
   Бэтчера в беззнаковом диапазоне.
3. **Только отрицательные числа (`only_negative`):** `input_data_ = {-170, -45, -75, -90, -2, -24, -802, -66}`.
   Подтверждает стабильность работы маски инверсии старшего байта в условиях, когда локальная поразрядная сортировка
   выполняется параллельно в контексте разных задач на изолированных участках памяти.
4. **Смешанные данные (`mixed`):** `input_data_ = {-170, 45, 75, -90, 2, 24, -802, 66}`. Наиболее критический сценарий
   для сети Бэтчера. Проверяет, что разнознаковые элементы, изначально распределенные по разным поддиапазонам задач,
   корректно мигрируют в нужные позиции общего вектора на этапе параллельного слияния.

### Характерные примеры входа и ожидаемого выхода

1. _Только положительные числа (`only_positive`):_
    - Вход: `[170, 45, 75, 90, 2, 24, 802, 66]`
    - Выход: `[2, 24, 45, 66, 75, 90, 170, 802]`
2. _Только отрицательные числа (`only_negative`):_
    - Вход: `[-170, -45, -75, -90, -2, -24, -802, -66]`
    - Выход: `[-802, -170, -90, -75, -66, -45, -24, -2]`
3. _Смешанные данные (`mixed`):_
    - Вход: `[-170, 45, 75, -90, 2, 24, -802, 66]`
    - Выход: `[-802, -170, -90, 2, 24, 45, 66, 75]`

Во всех тестовых конфигурациях при запуске на 1, 2 и 4 потоках параллельная TBB-версия успешно проходит автоматическую
валидацию. Состояния гонки (data races) и неопределенное поведение на стыках объединяемых блоков отсутствуют.

_Примечание к результатам:_

1. **Эффективность на 1 потоке:** Время выполнения на одном воркере (285.50 мс) незначительно уступает последовательному
   baseline (278.56 мс). Это связано с внутренними накладными расходами планировщика TBB на генерацию, создание объектов
   `blocked_range` и холостой обход сети Бэтчера. При этом накладные расходы чуть ниже, чем у OpenMP, за счет более
   легких абстракций задач.
2. **Масштабируемость и стагнация:** Пик производительности достигается на 4 потоках с ускорением в **1.33x**. Это
   несколько превосходит результат OpenMP-версии благодаря умному планированию задач (Work-stealing), однако общая
   тенденция к стагнации сохраняется. Локальный этап ускоряется хорошо, но стадия сети слияния Бэтчера выступает строгим
   ограничителем из-за частых синхронизаций между слоями компараторов.

## 9. Выводы

1. Параллельная реализация на базе Intel oneTBB демонстрирует корректное масштабирование с пиком ускорения
   **$1.33\text{x}$** на 4 потоках, обходя OpenMP-версию за счет динамического распределения задач и оптимизированного
   управления пулом потоков.
2. Ключевым сдерживающим фактором для TBB, как и для OpenMP, остается пропускная способность подсистемы памяти. Операции
   сравнения и перестановки элементов в компараторах Бэтчера выполняются нелокально по всему объему общего вектора. Это
   порождает массовые промахи кэша (cache-misses), нивелируя выгоду от эффективного динамического планирования задач на
   большом числе ядер.
