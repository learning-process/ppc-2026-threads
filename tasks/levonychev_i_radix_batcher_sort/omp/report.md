## 1. Контекст

Параллельная реализация поразрядной сортировки с использованием OpenMP представляет собой первый этап ускорения базового
алгоритма на многопроцессорных системах с общей памятью (Shared Memory). Использование гибридной схемы — разделение
данных на блоки, локальная сортировка подсчетом и последующее параллельное слияние через сеть Бэтчера — позволяет
эффективно распределить вычислительную нагрузку между ядрами процессора и оценить масштабируемость алгоритма при
совместном доступе к памяти. Из последовательной версии перенесен код побайтовой сортировки.

## 2. Постановка задачи

- **Входные данные:** Объект `std::vector<int>` размера $N$.

- **Выходные данные:** Объект `std::vector<int>` размера $N$, содержащий те же элементы, переставленные в порядке
  неубывания.
- **Базовый эталон (Baseline):** В качестве эталона для функциональной валидации граничных условий, вычисления точных
  метрик ускорения ($Speedup$) и оценки эффективности распараллеливания используется последовательная версия алгоритма,
  детально описанная в [seq/report.md](../seq/report.md). Все замеры производительности OpenMP-версии сопоставляются с
  временными метриками, зафиксированными для последовательной версии.

## 3. Базовый алгоритм

В качестве вычислительного ядра последовательной версии используется поразрядная сортировка (Radix Sort) по байтам.
Внутри каждого разряда (байта) применяется распределяющая сортировка подсчетом (Counting Sort).

## 4. Схема распараллеливания

Алгоритм распараллеливания разделен на два независимых этапа: локальную независимую сортировку блоков данных и
последующее итерационное слияние через сеть Бэтчера.

### 1. Этап локальной сортировки (Блочная декомпозиция)

Параллельная область инициализируется в методе `RunImpl()` с помощью базовой директивы:

```cpp
#pragma omp parallel default(none) shared(num_threads, block_size, n, std::ranges::copy) num_threads(num_threads)
  {
    int tid = omp_get_thread_num();
    int left = tid * block_size;
    int right = (tid == num_threads - 1) ? n : (tid + 1) * block_size;

    if (left < right) {
      std::vector<int> local_block(GetOutput().begin() + left, GetOutput().begin() + right);
      for (size_t i = 0; i < sizeof(int); ++i) {
        CountingSort(local_block, i);
      }
      std::ranges::copy(local_block, GetOutput().begin() + left);
    }
  }
```

- **Какая область кода параллелится:** Весь блок внутри фигурных скобок, где каждый поток вычисляет свои локальные
  индексы `left` и `right`, копирует выделенный ему сегмент во внутренний вектор `local_block`, вызывает для него 4
  итерации последовательной поразрядной сортировки `CountingSort`, а затем записывает результат обратно в общий вектор
  `GetOutput()`.
- **Разделение переменных (Data Sharing):**
  - **Shared (Общие):** `num_threads`, `block_size`, `n` — неизменяемые управляющие параметры; `GetOutput()` — целевой
      массив, куда потоки пишут результаты. Каждый поток пишет строго в свой диапазон индексов `[left, right)`, гонки за
      данными отсутствуют.
  - **Private (Локальные):** Переменные `tid` (идентификатор потока), `left` и `right` (границы блока) неявно являются
      приватными, так как объявлены внутри параллельной области. Объект `std::vector<int> local_block` создается на
      стеке каждого потока индивидуально, изолируя данные.
- **Барьеры:** В конце этой параллельной области существует **неявный барьер синхронизации OpenMP** (implicit barrier).
  Ни один поток не выйдет из блока, пока все остальные потоки не завершат локальную сортировку и не скопируют свои
  данные обратно. Это критически важно, так как этап слияния Бэтчера не может начаться на неотсортированных сегментах.

### 2. Этап четно-нечетного слияния Бэтчера

Вычислительное ядро сети слияния находится в методе `BatcherMergeIterative()` и распараллеливается с помощью директивы:

```cpp
#pragma omp parallel for schedule(static) default(none) shared(n, pv, p2, k, arr) num_threads(threads)
      for (int j = k % pv; j < n - k; j += 2 * k) {
        BatcherCompareRange(arr, j, k, p2);
      }
```

- **Какая область кода параллелится:** Внутренний цикл по переменной `j`, который инициирует работу компараторов
  `BatcherCompareRange` на текущем шаге сети слияния.
- **Разделение переменных (Data Sharing):**
  - **Shared (Общие):** `arr` (модифицируемый вектор данных), `n` (размер вектора), а также итерационные переменные
      внешних циклов `pv`, `p2`, `k`. Они являются общими, так как их значения только читаются внутри параллельного
      цикла.
  - **Private (Локальные):** Индекс цикла `j` автоматически является приватным для каждого потока (`private(j)`).
- **Выбор планировщика (Schedule):** Выбран тип планирования **`schedule(static)`**. На каждой фазе работы компараторов
  Бэтчера объем вычислений внутри одной итерации строго постоянен (сравнение двух элементов и возможная перестановка).
  Статическое распределение (`static`) минимизирует накладные расходы на работу планировщика OpenMP, деля итерации между
  потоками на равные порции еще на этапе компиляции/старта цикла.
- **Барьеры:** Директива параллельного цикла содержит **неявный барьер** в конце итерации. Потоки обязаны
  синхронизироваться перед переходом к следующей итерации внешнего цикла по переменной `k` (убывающий шаг компараторов),
  поскольку топология сети слияния Бэтчера требует строгого порядка выполнения слоев сравнения.

### Механизмы синхронизации (Reduction / Atomic / Critical)

В данной параллельной реализации механизмы `reduction`, `atomic` или `critical` секции **не требуются**:

1. В алгоритме отсутствуют операции агрегации данных (подсчет сумм, минимумов и т.д.), что исключает необходимость в
   `reduction`.
2. Дизайн сети слияния Бэтчера гарантирует, что на любой фиксированной итерации параллельного цикла компараторы работают
   с непересекающимися парами индексов элементов. Потоки никогда не совершают одновременную запись в одну и ту же ячейку
   памяти, что делает алгоритм потокобезопасным (lock-free) и избавляет от необходимости использовать ресурсоемкие
   `atomic` или `critical`.

## 5. Детали реализации

Специфика реализации параллельной версии OpenMP сосредоточена в файлах `omp/include/ops_omp.hpp` и
`omp/src/ops_omp.cpp`.

### Изменения относительно последовательной версии (SEQ)

1. **Метод `RunImpl()`**: В отличие от последовательной версии, где вся память вектора последовательно обрабатывалась
   единым вызовом `CountingSort`, здесь введена блочная декомпозиция. Входной массив `GetOutput()` разделяется на
   непересекающиеся отрезки `[left, right)` в соответствии с количеством выделенных потоков `num_threads`.
2. **Локальный буфер**: Каждый поток оперирует изолированным локальным вектором `local_block`, к которому
   последовательно применяется 4 прохода поразрядной сортировки по байтам. По завершении локальной сортировки потоки
   синхронизированно перезаписывают свои сегменты в результирующий массив.
3. **Добавление сети слияния Бэтчера**: Добавлены статические методы `BatcherMergeIterative` и `BatcherCompareRange`.
   Они реализуют итеративный алгоритм четно-нечетного слияния для объединения локально упорядоченных блоков в единый
   выходной массив, чего не требовалось в базовой `SEQ` версии.

### Устранение рисков гонок за данными (Data Races)

1. **Изоляция на этапе локальной сортировки**: Поразрядная сортировка подсчетом требует разделяемого массива счетчиков
   `count`. Если бы потоки параллельно вызывали `CountingSort` для общего массива, возникла бы деструктивная гонка за
   данными при инкременте `++count[value_of_byte]`. Риск полностью устранен за счет выделения индивидуального вектора
   `local_block` на стеке каждого потока. Вектор `GetOutput()` на данном этапе модифицируется строго по изолированным
   индексам `[left, right)`, уникальным для каждого `tid`.
2. **Корректность параллельного цикла слияния**: В методе `BatcherMergeIterative` параллельный цикл распределяет вызовы
   компараторов. Использование директивы `default(none)` гарантирует, что все переменные (`n`, `pv`, `p2`, `k`, `arr`)
   явно передаются как `shared`, так как их значения во внешних циклах неизменны.
3. **Безопасность компараторов**: Математическая топология четно-нечетной сети Бэтчера гарантирует, что при
   фиксированных параметрах шага `k` итерации цикла `j` генерируют пары индексов `(i, j)`, которые никогда не
   пересекаются между параллельными потоками. Таким образом, операции сравнения и перестановки элементов в памяти
   выполняются без блокировок, и риск одновременной записи нескольких потоков в одну ячейку полностью исключен.

## 6. Проверка корректности

Валидация параллельной OpenMP-реализации алгоритма осуществляется путем тестирования и сопоставления результатов с
базовым решением. Код тестов находится в файле `tests/functional/main.cpp`.

### Сравнение с последовательной версией (SEQ)

Для подтверждения математической точности параллельной схемы (включая блочную декомпозицию и работу сети слияния
Бэтчера) в рамках тестирования применяется двухэтапный контроль:

1. **Функциональная эквивалентность:** Выходные данные OpenMP-версии на идентичных входных наборах покомпонентно
   совпадают с результатами работы последовательного алгоритма `SEQ`. Это доказывает, что параллельное слияние не
   нарушает логику сквозного упорядочения.
2. **Проверка инварианта:** Итоговый массив валидируется предикатом `CheckTestOutputData`, который гарантирует
   выполнение условия неубывания элементов для всей последовательности.

### Набор тестов и характерные случаи

Параллельные потоки активируются на наборах данных различной структуры, что позволяет проверить корректность межпоточной
синхронизации и обработки знаковых разрядов:

1. **Размер $N = 1$ (`one_element`):** `input_data_ = {5}`. Граничный случай минимальной длины. Проверяет корректность
   работы условий декомпозиции, когда размер блока меньше или равен единице. Алгоритм совершает тривиальный выход без
   запуска параллельных регионов.
2. **Только положительные числа (`only_positive`):** `input_data_ = {170, 45, 75, 90, 2, 24, 802, 66}`. Тестирует
   базовое распределение индексов между рабочими потоками и корректность сборки отсортированных частей через компараторы
   Бэтчера в беззнаковом диапазоне.
3. **Только отрицательные числа (`only_negative`):** `input_data_ = {-170, -45, -75, -90, -2, -24, -802, -66}`.
   Подтверждает стабильность работы маски инверсии старшего байта в условиях, когда локальная поразрядная сортировка
   выполняется параллельно разными потоками на изолированных участках памяти.
4. **Смешанные данные (`mixed`):** `input_data_ = {-170, 45, 75, -90, 2, 24, -802, 66}`. Наиболее критический сценарий
   для сети Бэтчера. Проверяет, что разнознаковые элементы, изначально распределенные по разным потокам, корректно
   мигрируют в нужные позиции общего вектора на этапе параллельного слияния.

### Характерные примеры входа и ожидаемого выхода

1. _Только положительные числа (`only_positive`):_
    - Вход: `[170, 45, 75, 90, 2, 24, 802, 66]`
    - Выход: `[2, 24, 45, 66, 75, 90, 170, 802]`
2. _Только отрицательные числа (`only_negative`):_
    - Вход: `[-170, -45, -75, -90, -2, -24, -802, -66]`
    - Выход: `[-802, -170, -90, -75, -66, -45, -24, -2]`
3. _Смешанные данные (`mixed`):_
    - Вход: `[-170, 45, 75, -90, 2, 24, -802, 66]`
    - Выход: `[-802, -170, -90, 2, 24, 45, 66, 75]`

Во всех тестовых конфигурациях при запуске на 1, 2 и 4 потоках параллельная OpenMP-версия успешно проходит
автоматическую валидацию. Состояния гонки (data races) и неопределенное поведение на стыках объединяемых блоков
отсутствуют.

## 9. Выводы

1. Параллельная реализация OpenMP на базе гибридной схемы (локальная блочная сортировка + сеть Бэтчера) способна давать
   выигрыш по времени на многоядерных системах, однако масштабируемость алгоритма жестко ограничена его топологией.
   Пиковое ускорение составило **$1.31\text{x}$** на 4 рабочих потоках, после чего наступает стагнация
   производительности.
2. Локальный этап алгоритма (блочное разделение данных и независимый вызов `CountingSort`) параллелится с идеальной
   линейной эффективностью, так как потоки работают со своими изолированными буферами на стеке и не конкурируют за общую
   память.
3. Основное ограничение производительности и падение эффективности (до $32.7\%$ на 4 потоках и ниже на большем числе
   воркеров) связаны с фазой слияния Бэтчера. Частые неявные барьерные синхронизации между слоями компараторов
   заставляют потоки простаивать в ожидании друг друга, сводя на нет выигрыш от параллелизма при росте числа $p$.
4. Дополнительным сдерживающим фактором на этапе слияния является деградация локальности кэш-памяти: компараторы Бэтчера
   выполняют хаотичные и нелокальные перестановки элементов на стыках блоков в рамках единого общего вектора, что
   приводит к массовым промахам кэша (cache-misses) и взаимной блокировке ядер на общей шине памяти.
