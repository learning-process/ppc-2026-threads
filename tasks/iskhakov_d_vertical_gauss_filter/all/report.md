## 1. Контекст

Гибридная версия объединяет межпроцессное взаимодействие (MPI) и внутрипроцессное распараллеливание (OpenMP).
Изображение делится на вертикальные полосы между MPI‑процессами, а внутри каждого процесса обработка своей полосы
дополнительно распараллеливается с помощью OpenMP‑потоков. Это позволяет задействовать как распределённую память
(несколько процессов), так и общую память (несколько ядер) для максимального ускорения.

**ограничения** и **крайние случаи**


## 3. Базовый алгоритм

Последовательный алгоритм тот же, что и в SEQ: свёртка 3×3 с ядром Гаусса и зеркальным отражением границ
Асимптотика по времени O(N), по памяти O(N). Подробнее см. `seq/report.md` (пункт 3)


## 4. Межпроцессная схема

- **Распределение данных:** MPI‑процессы делят изображение по столбцам. Каждый процесс вычисляет свой диапазон столбцов
`[start_col, end_col)` на основе общего числа процессов `size`, номера процесса `rank` и параметров `cols_per_proc`,
`remainder`.

- **Роли процессов:**

  1. **Процесс 0 (главный):** после локальных вычислений собирает локальные блоки всех процессов (включая свой собственный)
  в общий результирующий массив `global_result`. Затем рассылает полный результат всем процессам через `MPI_Bcast`.
  2. **Остальные процессы:** вычисляют свою полосу и отправляют её процессу 0 с помощью `MPI_Send`.

- **Коммуникации:** используются `MPI_Send` / `MPI_Recv` для передачи локальных блоков от всех процессов к корневому
процессу 0, и один `MPI_Bcast` для распространения полного результата.
- **Синхронизация:** явный `MPI_Bcast` гарантирует, что все процессы получат итоговое изображение и продолжат работу
согласованно.


## 5. Внутрипроцессная схема

- Внутри каждого MPI‑процесса обработка своей полосы выполняется функцией `ProcessLocalBlock`, которая
распараллеливает внешний цикл по столбцам той же директивой `#pragma omp parallel for`.
- Все переменные, включая локальный результат `local_result`, объявлены как `shared`, запись ведётся в
уникальные индексы, поэтому дополнительная синхронизация не требуется.
- Число OpenMP‑потоков внутри процесса задаётся через `PPC_NUM_THREADS` (вызов `omp_set_num_threads`).


## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`


Основные шаги `RunImpl`:

1. Определяются `rank` и `size` через `MPI_Comm_rank` / `MPI_Comm_size`.
2. Вычисляются границы вертикальной полосы для текущего процесса.
3. Выделяется локальный буфер `local_result` размером `local_cols * height`.
4. Устанавливается число OpenMP‑потоков (`omp_set_num_threads`).
5. Вызывается `ProcessLocalBlock`, где внешний цикл по столбцам внутри полосы распараллелен с помощью
`#pragma omp parallel for`.
6. Функция `GatherAndBroadcast`:

   - Процесс 0 копирует свою полосу в `global_result` (`CopyOwnBand`), затем принимает блоки от всех остальных
  процессов (`ReceiveOtherBands`).
   - Остальные процессы отправляют свой `local_result` процессу 0 (`SendOwnBand`).
   - После этого все процессы получают полный результат через `MPI_Bcast`.

7. Итоговое изображение сохраняется в `GetOutput().data`.

Функция зеркального отражения `IskhakovDGetPixelMirrorAll` идентична остальным версиям, изменено только последняя
часть имени.

## 9. Результаты

Измерения проводились на изображении 8192×8192 пикселей. Базовое время SEQ = 1.79 с
(ускорение = 1, эффективность = 100%).

| Процессов | Потоков/процесс | Всего workers | Время, с | Ускорение | Эффективность |
| --------- | --------------- | ------------- | -------- | --------- | ------------- |
| 1 | 4 | 4 | 2.34 | 0.77 | 19.2% |
| 2 | 4 | 8 | 1.44 | 1.24 | 15.5% |
| 4 | 4 | 16 | 0.95 | 1.88 | 11.8% |
| 2 | 8 | 16 | 1.51 | 1.19 | 7.4% |
| 1 | 16 | 16 | 2.62 | 0.68 | 4.3% |

Ускорение: S = T_seq / T_p. Эффективность: E = S / total_workers × 100%.

Даже лучшая конфигурация (4 процесса × 4 потока) даёт ускорение лишь 1.88, что значительно
хуже чистого OpenMP (2.67) или TBB (2.91). Причины:

- явные пересылки (`MPI_Send`/`MPI_Recv` + `MPI_Bcast`) увеличивают накладные расходы, которые
не компенсируются параллелизмом;
- выделение и копирование локальных буферов также добавляет время.
С ростом числа процессов и потоков эффективность резко падает. Гибридная схема на одном компьютере
для данной задачи нецелесообразна.


## 10. Выводы

- Гибридная версия (MPI + OpenMP) на одном узле **не дала преимущества** над чистыми реализациями с общей памятью
(OMP, TBB). Максимальное ускорение составило 1.88×, в то время как OMP достиг 2.67×, а TBB — 2.91×.
- Основная причина — высокая цена межпроцессных коммуникаций и синхронизации, которые сводят на нет выигрыш от
параллелизма на одном компьютере.
- Таким образом, для фильтрации изображений предпочтительнее использовать технологии общей памяти (OpenMP или TBB),
а гибридный MPI+OpenMP отложить для действительно распределённых систем.
