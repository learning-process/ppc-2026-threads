## 1. Контекст

Гибридная версия объединяет два уровня параллелизма:

- Межпроцессный уровень (MPI) — распределение столбцов результирующей матрицы между независимыми процессами
- Внутрипроцессный уровень (OpenMP) — распараллеливание обработки столбцов внутри каждого процесса
Такой подход позволяет эффективно использовать как распределённую память (через MPI),
так и разделяемую память внутри узла (через OpenMP).
Это особенно актуально для суперкомпьютерных кластеров, где каждый узел имеет несколько ядер,
а узлы объединены сетью.
Последовательная версия (`seq/report.md`) служит эталоном корректности, параллельные версии
(`omp/report.md`, `tbb/report.md`, `stl/report.md`) — для сравнения с чисто потоковыми подходами.

## 2. Постановка задачи

### Входные данные

Две разреженные матрицы A и B в формате CCS.
Структура SparseMatrixCCS:

- `values` — массив ненулевых значений (тип double);
- `row_indices` — массив индексов строк для каждого значения;
- `col_ptrs` — массив указателей на начало каждого столбца (размер `cols + 1`);
- `rows` — количество строк;
- `cols` — количество столбцов.

### Выходные данные

Результирующая матрица C = A × B в формате CCS.

### Ограничения

- Количество столбцов матрицы A должно равняться количеству строк матрицы B
(причем количество строк и столбцов матрицы должно быть неотрицательно);
- Все индексы строк находятся в диапазоне `[0, rows-1]`;
- Указатели `col_ptrs` являются неубывающей последовательностью
- Первый указатель `col_ptrs[0] = 0`;
- Последний указатель `col_ptrs[cols] = общему количеству ненулевых элементов`.

## 3. Базовый алгоритм

Последовательная версия вычисляет произведение матриц по следующей схеме. Для каждого столбца j матрицы B:

- Обход ненулевых элементов столбца j, получение пар (k, `B[k][j]`);
- Для каждого k обход ненулевых элементов столбца k матрицы A;
- Накопление в `temp[i]: A[i][k] × B[k][j]`;
- Сохранение ненулевых значений из temp в результирующую матрицу.
Детальное описание последовательного алгоритма приведено в `seq/report.md`.

## 4. Межпроцессная схема

### Роли процессов

- Все процессы — каждый процесс получает свою порцию столбцов результирующей матрицы и независимо вычисляет их значения;
- Процесс с рангом 0 — выполняет сборку глобального результата:
  - Собирает локальные результаты от всех процессов через `MPI_Gather` и `MPI_Gatherv`;
  - Агрегирует количества ненулевых элементов через `MPI_Reduce`;
  - Распространяет итоговую матрицу через `MPI_Bcast`.

### Распределение данных

Столбцы матрицы B (и результирующей матрицы) распределяются между процессами по блочной схеме:

```cpp
int total_cols = b.cols;
int cols_per_proc = total_cols / size;
int remainder = total_cols % size;
int start_col = (rank * cols_per_proc) + std::min(rank, remainder);
int end_col = start_col + cols_per_proc + (rank < remainder ? 1 : 0);
```

### Используемые MPI-вызовы

| Вызов           | Место использования          | Назначение                                                   |
|-----------------|------------------------------|--------------------------------------------------------------|
| MPI_Comm_rank   | Начало MultiplyMatricesMPIOMP| Определение ранга текущего процесса                          |
| MPI_Comm_size   | Начало MultiplyMatricesMPIOMP| Определение общего числа процессов                           |
| MPI_Gather      | GatherAndBroadcastResult     | Сбор количества ненулевых элементов от всех процессов к 0    |
| MPI_Gatherv     | GatherAndBroadcastResult     | Сбор массивов значений и индексов от всех процессов          |
| MPI_Reduce      | GatherAndBroadcastResult     | Суммирование количеств ненулевых элементов по столбцам       |
| MPI_Bcast       | GatherAndBroadcastResult     | Рассылка итоговой матрицы от процесса 0 всем остальным       |

## 5. Внутрипроцессная схема

Внутри каждого MPI-процесса используется OpenMP для распараллеливания обработки назначенных столбцов.

Работа внутри процесса (обработка столбцов от start_col до end_col) распределяется с использованием динамического планирования:

```cpp
#pragma omp parallel for default(none) shared(a, b, start_col, end_col, temp_columns, nnz_counts, epsilon) \
    schedule(dynamic, 4)
for (int j = start_col; j < end_col; ++j) {
    // обработка столбца j
}
```

Параметры:

- `schedule(dynamic, 4)` — динамическое планирование с `chunk size = 4`
- Динамический тип выбран из-за неравномерности разреженности (разные столбцы могут требовать разного времени обработки)
- `Chunk size = 4` подобран экспериментально для балансировки нагрузки и минимизации overhead

### Объединение локальных результатов

Внутри процесса локальные результаты объединяются в два этапа:

1. Вычисление и подсчёт (`ComputeColumnBlock`):

- Каждый поток обрабатывает свои столбцы;
- Записывает временные векторы и количества ненулевых элементов;
- Отсутствует синхронизация (разные столбцы = разные позиции в массивах).

2. Построение локального CCS (`BuildLocalResult`):

- Сначала строится массив `local_col_ptrs` (префиксные суммы от `nnz_counts`);
- Затем потоки параллельно заполняют `local_values` и `local_row_indices`;
- Каждый поток записывает в непересекающиеся диапазоны (свои столбцы).

## 6. Детали реализации

Файлы: `all/include/ops_all.hpp`, `all/src/ops_all.cpp`

### Основные методы

`ValidationImpl()` — проверка корректности входных данных:

- Валидность структуры CCS обеих матриц
- Совместимость размерностей (A.cols == B.rows)

`PreProcessingImpl()` — подготовка к вычислениям:

- Создание результирующей матрицы C с размерами A.rows × B.cols

`RunImpl()` — выполнение умножения:

- Вызов `MultiplyMatrices(A, B)`

`PostProcessingImpl()` — завершающие действия:

- В данной реализации не требуется (возвращает true)

### Вспомогательные методы

`IsMatrixValid()` — проверка валидности CCS формата (реализация ограничений из 2):

- Размеры матрицы неотрицательны;
- Размер `col_ptrs` равен `cols + 1`;
- Количество значений равно количеству индексов строк;
- Первый указатель `col_ptrs[0] = 0`;
- Последний указатель `col_ptrs[cols] = nnz` (где `nnz(X)` — количество ненулевых элементов в матрице X);
- Указатели неубывают и неотрицательны;
- Индексы строк в допустимом диапазоне.

`ComputeColumnBlock()` — параллельное вычисление блока столбцов.

`BuildLocalResult()` — преобразование временных векторов в локальный CCS формат.

`GatherAndBroadcastResult()` — сбор и распространение результата.

`MultiplyMatricesMPIOMP()` — главная гибридная функция.

```cpp

    schedule(dynamic, 4)

    // вычисление столбца j

```

В функции BuildLocalResult — второй проход:

```cpp

for (int j = 0; j < local_cols; ++j) {
    // заполнение CCS для столбца j
}
```

Гибридная версия корректна — результаты совпадают с последовательной версией для всех функциональных тестов

### Когда гибридная схема оправдана

- Крупные задачи — размер матриц > 1000×1000 (на 700×700 эффект уже заметен, но масштабирование ограничено);
- Высокие требования к абсолютной производительности — ALL даёт наилучшее абсолютное ускорение среди всех реализаций.

### Не оправдана, когда

- Малые и средние задачи (< 500×500) — overhead MPI может превысить выигрыш;
- Одноузловые вычисления — чистые потоковые версии (особенно OpenMP) дают лучшее соотношение производительности к сложности;
- Сильно неравномерная разреженность — статическое распределение столбцов между MPI-процессами может быть неэффективным;
- Ограниченные вычислительные ресурсы — если доступно только 2-4 ядра, OpenMP или TBB будут эффективнее.
