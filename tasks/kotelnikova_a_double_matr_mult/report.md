## 1. Введение

Умножение разреженных матриц является одной из ключевых операций в вычислительной линейной алгебре.
Оно широко применяется в методах конечных элементов, решении систем линейных уравнений, обработке графов,
машинном обучении и других областях, где возникают матрицы большой размерности с преимущественно нулевыми элементами.
Данная задача особенно подходит для сравнения разных моделей параллелизма по следующим причинам:

- Высокая степень независимости — вычисления для разных столбцов результирующей матрицы полностью независимы;
- Неравномерная нагрузка — разреженная структура создаёт неравномерное распределение вычислительной работы;
- Различные стратегии балансировки — можно применить статическое и динамическое распределение;
- Масштабируемость — задача хорошо параллелится как на уровне одного узла, так и на кластере.

В работе реализованы и сравнены пять вариантов:
SEQ — последовательная эталонная версия
OMP — распараллеливание с помощью OpenMP
TBB — распараллеливание с помощью oneTBB
STL — ручное управление потоками через std::thread
ALL — гибридная версия (MPI + OpenMP)

## 2. Единая постановка задачи

### Входные данные

Две разреженные матрицы A и B в формате CCS.
Структура SparseMatrixCCS:

- `values` — массив ненулевых значений (тип double);
- `row_indices` — массив индексов строк для каждого значения;
- `col_ptrs` — массив указателей на начало каждого столбца (размер `cols + 1`);
- `rows` — количество строк;
- `cols` — количество столбцов.

### Выходные данные

Результирующая матрица C = A × B в формате CCS.

### Ограничения

- Количество столбцов матрицы A должно равняться количеству строк матрицы B (причем количество строк
и столбцов матрицы должно быть неотрицательно);
- Все индексы строк находятся в диапазоне `[0, rows-1]`;
- Указатели `col_ptrs` являются неубывающей последовательностью
- Первый указатель `col_ptrs[0] = 0`;
- Последний указатель `col_ptrs[cols] = общему количеству ненулевых элементов`.

### Крайние случаи

- Нулевая матрица (отсутствие ненулевых элементов);
- Матрицы с размерностью 0×0;
- Единичная матрица;
- Диагональные матрицы;
- Прямоугольные матрицы (когда `A.cols != A.rows`).

### Критерий корректности

Результат умножения считается корректным, если:

- Размеры матрицы C равны `A.rows × B.cols`;
- При преобразовании в плотный формат значения совпадают с эталонным умножением плотных матриц;
- Допустимая погрешность для double: `ε = 1e-10`.

### Метрики производительности

Speedup (ускорение):
`Speedup = T_seq / T_parallel`
где T_seq — время выполнения последовательной версии, T_parallel — время выполнения параллельной версии.
Efficiency (эффективность):
`Efficiency = (Speedup / Workers) × 100%`
где Workers — количество задействованных вычислительных единиц (потоков для OMP/TBB/STL, процессов для ALL).

### Процедура измерений

## 6. Интерпретация различий

### SEQ

Последовательная версия является эталоном корректности и базой для расчёта ускорения.
Её время выполнения (0.247 с) для матриц 700×700 с плотностью 10% отражает вычислительную сложность
задачи ~2.5 млн операций умножения.

### OMP

Сильные стороны:

- Простота реализации — достаточно добавить несколько директив `#pragma omp`;
- Динамическое планирование (`schedule(dynamic,8)`) эффективно балансирует неравномерную нагрузку;
- Хорошее масштабирование от 2 к 4 потокам;
- Низкие накладные расходы/

### TBB

Сильные стороны:

- Более тонкий контроль гранулярности через grain_size;
- Результаты сопоставимы с OpenMP.
Слабые стороны:
- Больший overhead на малых задачах;
- simple_partitioner() менее гибок, чем schedule(dynamic) в OpenMP.

### STL

Сильные стороны:

- Нет внешних зависимостей;
- Полный контроль над потоками.
Слабые стороны:
- Статическое разбиение неэффективно при неравномерной нагрузке;
- Отсутствие масштабирования при переходе от 2 к 4 потокам (9.70→9.92);
- Высокая сложность корректной реализации.

### ALL

Сильные стороны:

- Наилучшее абсолютное ускорение (13.08× на 4 процессах);
- Возможность использования нескольких узлов.
Слабые стороны:
- Высокая сложность реализации;
- Дублирование матриц A и B во всех процессах;
- Накладные расходы на MPI-коммуникации.

## 8. Заключение

В рамках данной работы успешно реализованы и протестированы пять вариантов умножения разреженных матриц.
Все реализации корректны и проходят функциональные тесты. Анализ производительности показал, что:

- OpenMP даёт лучшее соотношение производительности к сложности реализации;
- TBB показывает сопоставимые результаты с OpenMP при большей переносимости;
- STL страдает от статического разбиения и отсутствия масштабирования;
- ALL показывает наилучшее абсолютное ускорление, но требует распределённой памяти.
Для большинства практических задач на одном узле рекомендуется использовать OpenMP.
Гибридная версия оправдана только при работе с очень крупными матрицами или в кластерных средах.
