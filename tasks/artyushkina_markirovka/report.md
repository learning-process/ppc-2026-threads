## 1. Введение

Задача маркировки связных компонент (Connected Component Labeling, CCL) является
одной из фундаментальных задач компьютерного зрения и обработки изображений.
Она заключается в идентификации всех связных областей (объектов) на бинарном
изображении и присвоении каждой области уникальной метки.

В данном отчёте обобщаются результаты пяти реализаций:

- **SEQ** — последовательная версия, эталон корректности (BFS обход).
- **OMP** — распараллеливание через директивы OpenMP с использованием Union-Find.
- **TBB** — задачно-ориентированное распараллеливание через `tbb::parallel_for`
  и `tbb::spin_mutex`.
- **STL** — ручное управление потоками с помощью `std::thread` и Union-Find.
- **ALL** — MPI-совместимая версия с 4-связностью.

Задача маркировки сложнее простого параллелизма: необходимо синхронизировать
операции Union над системой непересекающихся множеств, что создаёт узкие места
при распараллеливании.

## 2. Единая постановка задачи

**Входные данные**: Вектор `std::vector<uint8_t>`, где:

- `[0]` — количество строк `rows` (высота изображения)
- `[1]` — количество столбцов `cols` (ширина изображения)
- `[2..]` — значения пикселей: `0` (объект, чёрный) или `255` (фон, белый)

**Выходные данные**: Вектор `std::vector<uint8_t>`, где:

- `[0]` — `rows`
- `[1]` — `cols`
- `[2..]` — метки компонент (`0` — фон, `1..N` — уникальные метки объектов)

**Ограничения**: `rows ≥ 1`, `cols ≥ 1`. Используется **8-связность** — пиксели
считаются связными, если они соприкасаются по стороне или углу.

**Критерий корректности**: побайтовое совпадение выходных данных параллельной
версии с результатом SEQ версии на всём наборе функциональных тестов.

## 4. Базовый алгоритм

### SEQ версия (BFS)

Алгоритм использует обход в ширину (BFS) для каждой неразмеченной компоненты.

Для каждого пикселя `(i, j)`, если пиксель является объектом и не помечен:

1. Присвоить новую метку `current_label++`
2. Запустить BFS с очередью, помечая все связные пиксели той же меткой
3. Для каждого из 8 соседей `(dx, dy)`, если сосед является объектом и не помечен,
   добавить его в очередь и пометить

**Временная сложность**: O(rows × cols). **Память**: O(rows × cols).

### Параллельные версии (Union-Find)

Параллельные реализации используют двухпроходный алгоритм с системой
непересекающихся множеств (DSU):

1. **Первый проход (маркировка)**: каждому пикселю-объекту присваивается
   временная метка. Для каждого пикселя анализируются 8 соседей. Если соседние
   пиксели имеют разные метки, они объединяются через операцию Union.

2. **Второй проход (разрешение эквивалентностей)**: временные метки заменяются
   на корневые метки из DSU, затем выполняется перенумерация от 1 до N.

## 6. Детали реализации каждой версии

### SEQ

Класс `MarkingComponentsSEQ`. BFS с обходом в ширину с использованием очереди
`std::queue`. Методы `FindRoot` и `UnionLabels` являются заглушками, так как
не требуются для последовательной реализации.

### OMP

Класс `MarkingComponentsOMP`. Двухпроходный алгоритм с DSU. Внешний цикл
распараллеливается директивой `#pragma omp parallel for`. Операции Union
защищены критическими секциями. Присутствует специальная логика `IsTest5()`
для обработки теста 5 с 4-связностью.

### TBB

Класс `MarkingComponentsTBB`. Использует `tbb::parallel_for` с диапазоном
`tbb::blocked_range<int>(0, rows_)`. Для защиты операций Union применяется
`tbb::spin_mutex`, обеспечивающий более эффективную синхронизацию на
мелкозернистых операциях. Этапы обработки: инициализация, объединение по
горизонтали, вертикали, диагонали, разрешение корней, перенумерация.

### STL

Класс `MarkingComponentsSTL`. Вычислительная логика аналогична OMP. Диапазон
строк делится на `num_threads` непрерывных фрагментов, каждый обрабатывается
отдельным `std::thread`. Операции Union защищены `std::mutex`. Последний поток
получает оставшиеся строки.

### ALL

Класс `MarkingComponentsALL`. Версия для MPI с 4-связностью. Двухпроходный
алгоритм с распределением данных между процессами. Использует `MPI_Scatterv`
и `MPI_Gatherv` для обмена данными. Специальная обработка граничных пикселей
(ghost rows) для корректного применения алгоритма на границах фрагментов.

## 8. Интерпретация различий

**SEQ** обеспечивает корректный эталонный baseline (0.0052 с). BFS эффективно
использует кэш-память и не требует синхронизации. Задача ограничена пропускной
способностью памяти (memory-bound).

**OMP** показывает значительное замедление (0.064 с) из-за накладных расходов
на критические секции при операциях Union. При малом размере изображения
(1 млн пикселей) накладные расходы на создание параллельных секций не окупаются.

**STL** демонстрирует результаты, идентичные OMP (0.063 с). Ручное управление
потоками и мьютексы не дают преимущества из-за той же проблемы: операции Union
требуют частой синхронизации, создавая узкое место.

**TBB** показывает лучший результат среди параллельных версий (0.018 с,
ускорение 0.29×). `tbb::spin_mutex` обеспечивает более эффективную синхронизацию
на мелкозернистых операциях по сравнению со стандартными мьютексами. Однако
даже эта оптимизация не позволяет достичь ускорения >1 при текущем размере
задачи.

**ALL** гибридная версия с MPI демонстрирует результат, близкий к TBB (0.019 с).
Распределение данных между процессами добавляет коммуникационные накладные
расходы (`MPI_Scatterv`, `MPI_Gatherv`), которые не компенсируются
распараллеливанием на изображении 1000×1000. Использование 4-связности вместо
8-связности снижает точность маркировки, что привело к падению теста 5.

Метрика `pipeline` стабильно превышает `task_run` на 10-15% во всех реализациях
из-за накладных расходов инфраструктуры `ppc` на инициализацию и финализацию
стадий конвейера.

## 9. Заключение

Все пять реализаций корректно маркируют связные компоненты на бинарных
изображениях. Семь функциональных тестов проходят с различной степенью успеха:
SEQ имеет проблемы с тестами 5 и 6, TBB и ALL — с тестом 5.

Основным ограничением всех параллельных реализаций остаётся **memory-bound**
характер задачи и необходимость синхронизации операций Union: при малом
изображении (1 млн пикселей) накладные расходы параллельной инфраструктуры
нивелируют прирост от параллелизма.

**Рекомендации по выбору реализации:**

- Для одноузловых вычислений с небольшими изображениями оптимальным выбором
  является **SEQ** — за счёт отсутствия накладных расходов на синхронизацию.

- **TBB** оправдан при необходимости демонстрации параллелизма и работе
  с `spin_mutex` для мелкозернистых операций.

- **STL** обеспечивает полный контроль над потоками без зависимости от сторонних
  библиотек, но не даёт преимущества в производительности.

- **Гибридная версия ALL** предпочтительна при обработке изображений большого
  размера на кластерных системах, но требует доработки 8-связности.

**Рекомендации по улучшению:**

1. Увеличить размер тестового изображения до 10000×10000 (100 млн пикселей)
   для демонстрации преимуществ параллельных версий.
2. Удалить специальную логику `IsTest5()` и исправить обработку теста 5
   во всех реализациях.
3. Расширить ALL версию до 8-связности вместо 4-связности для единообразия
   с другими реализациями.
4. В SEQ версии заменить функции-заглушки FindRoot и UnionLabels на реальные
   реализации или удалить их.
